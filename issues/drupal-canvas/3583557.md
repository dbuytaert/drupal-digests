## Summary

The Canvas editor's Pages list panel (left sidebar) and top navigation panel previously capped page listings at 50 results. This change adds infinite scroll to both locations: when a user scrolls to the bottom of either list, the next batch of pages loads automatically and appends to the existing list. The feature builds on the offset-based pagination API introduced in [#3502691](https://www.drupal.org/node/3502691). A secondary sort by entity ID was also added on the backend to guarantee stable pagination when multiple pages share the same creation timestamp.

## Impact

Affects module maintainers.

- **New API:** New `ContentListResult` interface (`{ items, totalCount }`) replaces the plain `ContentStub[]` return type of `useGetContentListQuery`; callers must read `data.items` instead of `data`.

## Technical details

The backend fix in `ApiContentControllers::getContentList` adds a secondary `.sort($id_key)` after the primary `.sort($revision_created_field_name, 'DESC')`. Without this, pages with identical `revision_created` values could be returned in varying database-dependent orders across paginated requests, causing duplicates or missing items.

On the frontend the main structural change is in `contentApi` (RTK Query). The `getContentList` endpoint return type changed from `ContentStub[]` to a new `ContentListResult` interface (`{ items: ContentStub[], totalCount: number | null }`). Three new RTK Query options enable client-side cache merging across paginated calls: `serializeQueryArgs` keys the cache by `entityType-search` only (ignoring `offset`), `merge` appends new items to the cache when `offset > 0` and replaces them when `offset === 0`, and `forceRefetch` triggers a new network request whenever the query args change.

Cache invalidation on delete and create was replaced with manual cache management to avoid losing accumulated items. The delete mutation uses `contentApi.util.updateQueryData` to optimistically filter the deleted item from all matching cached queries and decrements `totalCount`, with rollback on failure. The create mutation force-refetches from `offset: 0` after success rather than invalidating the `Content/LIST` tag.

The new `usePaginatedContentList` hook encapsulates `offset` state, debounces the search term (300 ms), resets `offset` to 0 when the search changes, and computes `hasMore` (true only when not searching and `items.length < totalCount`). It exposes `items`, `hasMore`, and `handleLoadMore` to callers. Both the `Pages` and `PageInfo` containers replaced their direct `useGetContentListQuery` + `useDebounce` calls with this hook.

The new `InfiniteScrollObserver` React component uses `IntersectionObserver` with a 100 px `rootMargin` to detect when a 1 px sentinel `div` enters the viewport, then calls `onLoadMore`. It is rendered conditionally at the bottom of both the `PageList` component and the `Navigation` component when `hasMore` is true.

Callers that only needed items (`useSmartRedirect`, `UnpublishedChanges`) were updated to read `data?.items` from the new response shape. Cypress tests that previously waited for a re-fetch after deletion were updated to remove those waits since deletion no longer triggers a network request. Playwright tests were added for both panels using `page.route` to mock paginated API responses, avoiding the need to create 50+ real pages.

## Upgrade

If your custom code calls `useGetContentListQuery` and reads the result directly as an array, update it to read `data.items`:

```ts
// Before
const { data: pageItems } = useGetContentListQuery({ entityType: 'canvas_page' });

// After
const { data } = useGetContentListQuery({ entityType: 'canvas_page' });
const pageItems = data?.items ?? [];
```
