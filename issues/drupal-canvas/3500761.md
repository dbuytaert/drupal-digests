## Summary

Drupal Canvas code components (in-browser-editable JSX components) currently cannot import third-party npm packages. This spike issue investigated whether import maps and a public ES module CDN could make that possible without a bundler. The team proved the approach viable and identified a simpler path than originally proposed: bare npm specifiers in the component source are rewritten to absolute esm.sh CDN URLs at compile time, with a static import map covering the React/Preact aliases. A predefined set of bundled packages was landed in a related issue ([#3508734](https://www.drupal.org/node/3508734)) as an interim step; support for arbitrary npm imports is being implemented in [#3560197](https://www.drupal.org/node/3560197) using the approach validated here.

## Impact

Affects module maintainers.

- **New API:** Code components will be able to use bare npm import specifiers (e.g. `import * as X from "@radix-ui/react-accordion"`) which Canvas rewrites to esm.sh URLs at compile time.

## Technical details

The original proposal called for using `@jspm/generator` in the browser to resolve transitive dependencies and generate a full import map, storing the resolved package list in `js_imports` and `js_import_dependencies` properties on the `JsComponent` config entity. A key discovery during the spike changed this plan: esm.sh rewrites all internal import specifiers within the packages it serves so that every dependency is also fetched from esm.sh. This means a component importing `@radix-ui/react-accordion` does not need its entire transitive dependency tree enumerated in the import map; only the top-level specifier needs to be present.

The simplified approach replaces a bare specifier like `import * as Accordion from "@radix-ui/react-accordion"` with `import * as Accordion from "https://esm.sh/@radix-ui/react-accordion?external=react,react-dom"` at compile time, alongside a static import map that maps `react`, `react-dom`, and `preact` to the Preact compat layer also served from esm.sh. This avoids runtime dependency resolution entirely and removes the need to store resolved package lists on the config entity for the initial implementation.

Parsing import statements from JSX source was tested with several tools. `@jspm/generator`'s HTML parser rejected JSX syntax; `acorn` combined with `acorn-jsx` also failed on the sample component. `babylon` (the parser underlying Babel) succeeded and additionally provided syntax validation as a side-effect. A practical constraint also surfaced: browsers do not allow an existing page's import map to be mutated at runtime, so injecting an import map into a live preview requires an iframe boundary. CDN configurability and per-component version locking were explicitly deferred as follow-up work.

### Collaboration

balintbrews identified that esm.sh auto-rewrites all internal dependency imports within the packages it serves, which eliminated the need for `@jspm/generator` at serve time and made the full `js_imports`/`js_import_dependencies` config-entity proposal premature. This single observation collapsed a multi-step dependency-resolution pipeline into a simple URL substitution, and balintbrews backed it with a working POC (https://github.com/balintbrews/poc-import-from-esm-sh) demonstrating Preact, a React compat layer, and `@radix-ui/react-accordion` all imported from esm.sh via an import map. larowlan contributed the earlier POC showing that `babylon` could parse JSX import statements and feed them to `@jspm/generator` to produce import maps, and flagged the runtime immutability constraint on import maps that implies iframe sandboxing for live previews.
