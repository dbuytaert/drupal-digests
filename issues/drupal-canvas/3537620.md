## Summary

In the Canvas (Experience Builder) code editor, when defining a component prop as a "List" type (string or integer), authors previously could only enter raw values for the enum options. Those raw values were also shown as-is to end users in the Experience Builder UI. This change adds a separate "Label" field for each option in the Props tab, allowing authors to provide a human-readable, translatable display string distinct from the stored value. Existing enum props without explicit labels continue to function: the label simply displays as the raw value. Duplicate values within the same prop are now flagged inline with a "Value must be unique." error.

## Impact

Affects module maintainers.

- **API change:** `CodeComponentProp.enum` changed from `string[]` to `Array<CodeComponentPropEnumItem>`; callers must destructure `{ value, label }` instead of using the string directly
- **New API:** New `CodeComponentPropEnumItem` interface (`{ label: string; value: string | number }`) exported from `src/types/CodeComponent.ts`

## Technical details

The internal representation of enum options changed from a flat `string[]` to an array of `CodeComponentPropEnumItem` objects (`{ label: string; value: string | number }`), defined in the new exported interface in `src/types/CodeComponent.ts`. `CodeComponentPropSerialized` also gains a typed `'meta:enum'` field (`Record<value, label>`), formalising the schema format introduced in issue [#3516602](https://www.drupal.org/node/3516602).

`serializeProps` now emits `meta:enum` entries as `[value, label]` pairs (previously it emitted `[value, value]`, making the label field redundant). `deserializeProps` reads `meta:enum` first when present and falls back to the plain `enum` array for backwards compatibility, converting each entry to a `CodeComponentPropEnumItem`.

The `EnumValuesForm` React component in `FormPropTypeEnum` renders a two-column row per option ("Value" and "Label"). A `touched` boolean array tracks which label fields have been manually edited; untouched labels auto-mirror the value as the user types, matching the common case where value and label are the same. Once a label is manually changed, auto-sync stops for that row. Duplicate-value detection is handled by `getDuplicateValueIndices`, which highlights the value fields in red and shows the inline error. The section heading was also renamed from "Values" to "Options".

## Upgrade

Any TypeScript code that imports `CodeComponentProp` and accesses its `enum` field must be updated. The type changed from `string[]` to `Array<CodeComponentPropEnumItem>`.

```ts
// Before
const values: string[] = prop.enum ?? [];

// After
import type { CodeComponentPropEnumItem } from '@/types/CodeComponent';
const values: CodeComponentPropEnumItem[] = prop.enum ?? [];
// Access individual entries as item.value and item.label
```

Serialized component JSON that uses `meta:enum` to store labels distinct from values will now be round-tripped correctly. Previously, `serializeProps` always wrote the value as the label (e.g. `"Not Started": "Not Started"`); after this change it writes the author-supplied label (e.g. `"Not Started": "Pending"`).
