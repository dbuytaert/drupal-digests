## Summary

The Canvas CLI now helps projects migrate from the deprecated `CANVAS_COMPONENT_DIR` environment variable to the new `canvas.config.json` configuration file. When the CLI detects that a project is still using the legacy environment variable, it displays a warning and offers to automatically create or update the configuration file with the `componentDir` setting. This prevents existing projects from breaking after the configuration approach changed. In non-interactive mode, the CLI provides clear manual migration instructions instead of prompting.

## Impact

No upgrade or configuration changes required.

## Technical details

The CLI now runs a migration check before executing any command through a `preAction` hook. The `handleLegacyComponentDirMigration()` function detects when `CANVAS_COMPONENT_DIR` is set in the environment but `componentDir` is not defined in `canvas.config.json`. It warns the user that the environment variable is deprecated.

For the current CLI run, it preserves backward compatibility by setting both `componentDir` and `deprecatedComponentDir` in the runtime config. This ensures commands continue to work while migration is in progress.

In interactive mode, the function prompts the user to create or update `canvas.config.json` with the `componentDir` value from the environment variable. If the user confirms, it writes the configuration file, either creating a new one or merging the setting into an existing file. If the user declines or cancels, it shows a message explaining how to migrate manually later.

In non-interactive mode (when the `--yes` flag is passed), the function skips prompts entirely and displays manual migration instructions. A guard flag prevents the migration logic from running multiple times in the same process. Tests can reset this flag using `resetLegacyComponentDirMigrationForTests()`.
