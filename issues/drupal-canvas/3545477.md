## Summary

The Canvas AI page builder was extremely slow when building context for AI prompts that needed the full component list. Each call to `CanvasAiPageBuilderHelper::getAllComponentsKeyedBySource()` invoked the component-list controller directly, bypassing Drupal's caching stack entirely. A prompt that triggered five such calls took roughly 76 seconds (about 15 seconds per call). The fix routes these lookups through a proper HTTP subrequest so that Dynamic Page Cache can serve subsequent calls, dropping latency to roughly 62ms for two calls.

## Impact

No upgrade or configuration changes required.

## Technical details

The old implementation injected `ApiConfigControllers` directly and called `::list(Component::ENTITY_TYPE_ID)` on it, which bypassed the full Drupal HTTP kernel and therefore got no caching benefit. The fix replaces that dependency with `HttpKernelInterface` and `RequestStack`, then constructs a `Symfony\Component\HttpFoundation\Request` targeting the `canvas.api.config.list` route with `_format=json` set on its attributes. `HttpKernelInterface::handle()` is called with `HttpKernelInterface::SUB_REQUEST`, which runs the request through the full kernel stack including Dynamic Page Cache. The subrequest copies cookies and server variables from the parent request to preserve session context. A bare `catch (\Exception)` wraps the call and returns an empty array on any failure, preventing AI errors from surfacing as unhandled exceptions.
