## Summary

In the Canvas module, components placed inside `ContentTemplate`s and `PageRegion`s were not always rendered in the order in which they were arranged in the editor. A typical symptom was that placing a child component into a slot caused its parent container (e.g. a two-column block) to jump to the first position on the page, displacing components that should appear above it. This critical bug is now fixed; the rendered order of components matches the order defined in the layout editor.

## Impact

No upgrade or configuration changes required.

## Technical details

The bug was in `ComponentTreeItemListInstantiatorTrait::generateComponentTreeKeys()`, which converts a flat component tree (as sent by the client) into a keyed array used for config storage and rendering. The method uses `\Drupal\Component\Graph\Graph::searchAndSort()` to order components by depth in the tree. After sorting, it iterated over the resulting graph and assigned integer keys to top-level components (those without a `parent_uuid`) using an `$outer_delta` counter that incremented in the order the sorted graph returned vertices. Because `Graph::searchAndSort()` assigns `weight` values based on depth-first traversal and does not guarantee stable ordering among top-level siblings, the reconstructed key array placed top-level items in graph-traversal order rather than original input order.

The fix records each top-level component's original input position before sorting, via a `$top_level_delta` counter that increments only for items with no parent. After sorting, the code uses `$graph['delta']` (the pre-sort position) as the key for top-level items instead of the post-sort iteration counter. Child components, whose keys are already derived from their parent's key and slot position, are unaffected by this change.

The root cause was a missed edge case in the original introduction of `generateComponentTreeKeys()` in [#3526127](https://www.drupal.org/node/3526127). Two new `#[TestWith]` dataset cases were added to `ConfigWithComponentTreeTestBase::testComponentTreeKeyOrder` to cover the previously failing scenarios: one with a simple two-level tree where the second top-level item has a single child, and one reproducing the real-world scenario (reported by a user) where child items appear before their parent in the flat input array.
