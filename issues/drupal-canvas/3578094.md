# #3578094: Fix REST API violation of the Canvas Page PATCH operation: auto-save operations should be auto-save prefixed

## Summary

The Canvas module's publish and unpublish API endpoint violated REST principles and lacked consistency with other auto-save operations. The endpoint operated on auto-save storage but was located at a path intended for the external Canvas Page API. This created confusion about which operations were stateful versus stateless and blocked work on the proper external API.

## Impact

No upgrade or configuration changes required.

## Technical details

The PATCH operation for publishing and unpublishing Canvas pages accepted a `clientInstanceId` parameter and saved to auto-save storage, making it stateful. However, it used the path `/canvas/api/v0/content/canvas_page/{canvas_page}`, which was needed for a truly stateless external Canvas Page API being developed in [#3575144](https://www.drupal.org/node/3575144).

The fix moved the entire PATCH operation from `ApiContentControllers::patch()` to a new `ApiContentAutoSaveControllers::patch()` controller and changed the route path to `/canvas/api/v0/content/auto-save/canvas_page/{canvas_page}`. This aligns with other auto-save endpoints like `canvas.api.config.auto-save.patch` which already used the `auto-save` path prefix.

The change also extracted the homepage checking logic from `ApiContentControllers` into a new `HomePageHelper` service, improving code organization and allowing the logic to be shared between controllers. The frontend JavaScript and all link generation logic were updated to use the new endpoint path. OpenAPI documentation was reorganized to reflect that this endpoint manages auto-save storage, not direct entity updates.
