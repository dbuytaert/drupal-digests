## Summary

Content Templates and Page Regions in Canvas can now be translated using Drupal core's Config Translation UI (when the `canvas_dev_translation` feature-flag module is enabled). Previously, a component instance's `inputs` were stored as a JSON blob and declared `type: ignore` in config schema, making them completely opaque to the translation system. They are now stored as structured YAML arrays, and only the genuinely translatable inputs (plain strings, HTML strings, and URIs for SDC/code components; `label` for block components) are offered for translation. Inputs sourced from an `EntityFieldPropSource` or `HostEntityUrlPropSource` are automatically marked non-translatable, because the underlying entity field handles its own translation. An update path is provided to migrate existing config entities from JSON-blob storage to the new array format.

## Impact

Affects module maintainers and site owners.

- **Database update:** Four `canvas_post_update_0016_*` functions convert `inputs` from JSON blobs to PHP arrays for `Pattern`, `PageRegion`, `ContentTemplate`, and `field_config` config entities.
- **Config change:** `canvas.component_tree_node.inputs` schema type changes from `type: ignore` to `type: mapping` with class `ComponentInputsMapping`; exported YAML for all config entities with component trees changes from JSON-blob `inputs` strings to structured YAML arrays.
- **Deprecation:** Config entities that still store `inputs` as a JSON-encoded string trigger `E_USER_DEPRECATED` (deprecated in `canvas:1.4.0`, removed in `canvas:2.0.0`); running the update path via `drush updb` resolves this automatically.

## Technical details

The root change is in `canvas.schema.yml`: `canvas.component_tree_node.inputs` is promoted from `type: ignore` to `type: mapping` with `class: '\Drupal\canvas\Config\Schema\ComponentInputsMapping'`. The new `ComponentInputsMapping extends Mapping` class generates its full mapping definition dynamically in its constructor by reading the parent component instance's `component_id`, `component_version`, and actual `inputs` values, then delegating to `ComponentInputs::resolveConfigSchemaMapping()`. That static method is the single source of truth for translatability for both config entity translation (this class) and content entity translation (`ComponentInputs::getTranslatableInputKeys()`). Translatability is determined by two dimensions: shape-based (string/HTML/URI props are translatable; integers, booleans, and structured prop sources are not) and value-based (if the stored value is a `StaticPropSource` it is translatable; if it is an `EntityFieldPropSource` or `HostEntityUrlPropSource` it is not, regardless of prop shape). Prior art for the `class:`-powered dynamic schema is `JsonSchemaObject`, already used for code component prop validation.

For the Config Translation UI form, translatable SDC/code component props receive `form_element_class: CanvasStaticPropSourceFieldWidget`. This single class extends `FormElementBase` and delegates to `StaticPropSource::getWidget()` to conjure the same field widget the Canvas editor uses, handling both single-property field types (e.g. `StringItem` → scalar) and multi-property types (e.g. `TextLongItem` → `{value, format}`, `LinkItem` → `{uri, title, options}`) without additional per-field-type classes. On form submission `CanvasStaticPropSourceFieldWidget::setConfig()` calls `StaticPropSource::optimizeExplicitInput()` to produce the same compacted representation used by the default translation. Block component instances forward the native `block.settings.<plugin_id>` config schema, inheriting core's block translatability directly. Two additional `@internal` form-element classes — `CanvasComponentTreeItemListFormElement` and `CanvasComponentTreeItemInputsMappingFormElement` — wrap `config_translation`'s `ListElement` to strip distracting `<details>` wrappers and to ensure omitted optional inputs are still presented for translation.

To satisfy `config_translation`'s requirement for an `edit-form` link template, placeholder routes `entity.content_template.edit_form` and `entity.page_region.edit_form` are registered, served by `NoUiConfigController::placeholder`. The `canvas_dev_translation` feature-flag module's previous `MakeComponentTreeTranslatable` hook (which used `hook_config_schema_info_alter` to add `translatable: true` at runtime) is replaced by `ConfigTranslationSupportHooks`, which uses `hook_entity_type_alter` to inject the `edit-form` link templates so `config_translation` can generate its overview link.

`ComponentTreeConfigEntityBase` now overrides `preCreate()` and `set('component_tree', …)` to call the new `componentTreeInstancesInputsMustBeArrays()` static method, which JSON-decodes any string-valued `inputs` key. This prevents `ConfigSchemaChecker` and `ValidKeysConstraint` from rejecting entities whose `inputs` were previously set via `ComponentTreeItem` (which stores `ComponentInputs` as a JSON string internally). `CanvasConfigUpdater` gains `needsConfigEntityWithComponentTreeInputsAsArrays()` (which also emits `E_USER_DEPRECATED` for stale configs) and `updateConfigEntityWithComponentTreeInputsAsArrays()`. The step 1 blocker for this work was [#3586342](https://www.drupal.org/node/3586342), which introduced `ComponentInstanceInputsConfigSchemaGeneratorInterface`. The known limitation that reordering component instances breaks existing language overrides is tracked in [#3582464](https://www.drupal.org/node/3582464).

## Upgrade

After updating, run the update path to convert stored JSON-blob `inputs` to structured arrays across all config entity types:

```bash
drush updb
```

Four post-update functions run automatically:
- `canvas_post_update_0016_pattern_component_inputs_must_be_arrays`
- `canvas_post_update_0016_page_region_component_inputs_must_be_arrays`
- `canvas_post_update_0016_content_template_component_inputs_must_be_arrays`
- `canvas_post_update_0016_component_tree_field_default_value_inputs`

After running updates, re-export configuration. The `inputs` key in component tree nodes will change format in exported YAML:

```yaml
# Before (canvas 1.3 and earlier):
component_tree:
  - uuid: fd141b6e-c57a-4355-9885-a8b2e60627d6
    component_id: sdc.canvas_test_sdc.heading
    component_version: 8c01a2bdb897a810
    inputs: '{"text":"Hello world","style":"primary","element":"h1"}'

# After (canvas 1.4+):
component_tree:
  - uuid: fd141b6e-c57a-4355-9885-a8b2e60627d6
    component_id: sdc.canvas_test_sdc.heading
    component_version: 8c01a2bdb897a810
    inputs:
      text: Hello world
      style: primary
      element: h1
```

Custom code that reads `inputs` directly from a config entity's raw data and expects a JSON string must be updated to handle a PHP array. Code that sets `component_tree` via `ComponentTreeConfigEntityBase::set('component_tree', …)` or `::setComponentTree()` continues to work: JSON blobs are automatically decoded to arrays on write.
