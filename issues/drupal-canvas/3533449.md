## Summary

Before this fix, any user who could open the Drupal Canvas editor could also open the AI agent panel, regardless of whether they held the `use Drupal Canvas AI` permission. The permission check existed on the back-end AI function-call plugins but was never enforced in the front-end: the AI button appeared for all Canvas users. This fix makes the server pass a per-user permission flag to JavaScript, and the Topbar component now checks that flag before rendering the AI button. Additionally, a stray capitalization inconsistency in the permission machine name (`ai` vs `AI`) was corrected.

## Impact

Affects site owners.

- **Permission change:** Permission machine name corrected from 'use Drupal Canvas ai' to 'use Drupal Canvas AI'; users must hold this permission for the AI panel button to appear in the Canvas editor.

## Technical details

The root cause was that `aiExtensionAvailable` in `drupalSettings.canvas` only indicated whether the `canvas_ai` module was installed, with no per-user access signal. The `Topbar` React component relied on this flag alone to decide whether to render the "Open AI Panel" button.

The fix wires a server-side permission check into `CanvasAiHooks::jsSettingsAlter()` (a `#[Hook('js_settings_alter')]` implementation). When `aiExtensionAvailable` is truthy, the hook now also injects `settings['canvas']['permissions']['useCanvasAi']` as a boolean derived from `AccountInterface::hasPermission()`. The `Topbar` component is updated to require both `aiExtensionAvailable` and `permissions.useCanvasAi === true` before setting `hasAiExtensionAvailable`.

A follow-up commit (a6f39e75) tightened the guard condition in `jsSettingsAlter` from a strict `=== TRUE` comparison to `!empty(...)`, preventing PHP warnings about an undefined `canvas` array key in contexts where the Canvas base module has not yet populated that settings key (exposed by `SetAIGeneratedComponentStructureTest`).

As part of this work, a new `CanvasAiPermissions` constants class was introduced with `USE_CANVAS_AI = 'use Drupal Canvas AI'`, and all inline permission strings throughout the module (routes, function-call plugins, and tests) were migrated to use it. The permission machine name itself was also corrected from the previous lowercase `use Drupal Canvas ai` to `use Drupal Canvas AI`.

## Upgrade

If you have assigned the old `use Drupal Canvas ai` permission to any role, you will need to reassign it using the corrected name `use Drupal Canvas AI` after updating. The route access checks and front-end visibility gate now both use the new name.
