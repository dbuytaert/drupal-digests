## Summary

Site owners using Canvas Workbench can now upload images to a Drupal site via the Canvas CLI. A new `POST /canvas/api/v0/media/{media_type}/upload` endpoint accepts a multipart image upload and creates a file entity and media entity in one round trip, returning the entity ID, UUID, and resolved image prop values ready for use in Canvas page components. This is marked as an unstable v0 API intended for the Canvas CLI rather than the Canvas UI, and it unblocks CLI push/pull for pages that contain image component instances (see [#3578452](https://www.drupal.org/node/3578452)).

## Impact

Affects module maintainers and site owners.

- **Config change:** New per-image-media-type OAuth2 scopes (`canvas:media:{media_type_id}:create`) are created on `canvas_oauth` install and via post-update `canvas_oauth_post_update_0003_media_image_scopes`.
- **New API:** New `POST /canvas/api/v0/media/{media_type}/upload` endpoint (unstable v0) accepts a multipart image upload and returns the created media entity's `id`, `uuid`, and `inputs_resolved` fields.

## Technical details

The core design decisions were settled by wim leers in comment #2: use Drupal core's `FileUploadHandler` for all file-handling and security logic (consistent with how REST and JSON:API handle uploads), and bundle file creation and media entity creation into a single request to avoid the state management brittleness of two separate round trips.

The new `ApiMediaControllers::upload()` method in `canvas` module handles the request. It validates that the target media type uses the `image` source plugin, resolves file extension and size validators from the source field definition via `FileUploadLocationTrait`, wraps the incoming `UploadedFile` in `FormUploadedFile`, and delegates to `FileUploadHandlerInterface::handleFileUpload()`. If upload validation passes, it creates a media entity via entity storage, validates it with `TypedDataInterface::validate()`, saves it, and returns a `201` JSON response containing `id`, `uuid`, and `inputs_resolved`. The `inputs_resolved` payload contains `src` (with the `?alternateWidths` query parameter), `alt`, `width`, and `height`, matching the Canvas image prop shape.

The route `canvas.api.media.upload` is defined at `POST /canvas/api/v0/media/{media_type}/upload` with access controls `_canvas_authentication_required` and `_entity_create_access: 'media:{media_type}'`. The `{media_type}` path parameter is upcasted to a `media_type` entity. The route is special-cased in `CanvasOauthAuthenticationProvider::applies()` so that OAuth tokens are accepted for it (media is not a Canvas-provided entity type and therefore not covered by the existing config-entity routing logic).

OAuth support is provided by a new `MediaScopesHelper` class in `canvas_oauth`, which creates a `canvas:media:{media_type_id}:create` OAuth2 scope (mapped to the `create {media_type_id} media` permission) for every media type whose source plugin is `image`. Scopes are created on module install and via `canvas_oauth_post_update_0003_media_image_scopes`. The follow-up issue [#3582492](https://www.drupal.org/node/3582492) tracks honoring per-field alt/title requiredness settings.

### Collaboration

Wim leers flagged in comment #2 that file upload is security-sensitive territory, drawing on his prior work on core's file upload unification in [#1927648](https://www.drupal.org/node/1927648) and [#2940383](https://www.drupal.org/node/2940383). He required use of `FileUploadHandler` and argued against two-step uploads (first file, then media entity) as both slower and more fragile. That recommendation directly shaped the final design: a single multipart request that creates both entities in one go. penyaskito then opened the MR and worked through several rounds of security-focused review with wim leers before landing the change.
