## Summary

In the Canvas Page Builder, when editing a Content Template, linking a prop field to a dynamic source (such as "Relative URL") and then unlinking it caused the field to render incorrectly as a split fieldset instead of restoring its original single-input widget. This has been fixed. Additionally, the server now detects and explicitly rejects corrupted `StaticPropSource` data rather than silently rendering a broken form.

## Impact

Affects module maintainers.

- **API change:** `ComponentSourceInterface::buildComponentInstanceForm()` `$component` parameter changed from `?Component $component = NULL` to the non-nullable `Component $component`; custom implementations must update their signatures.

## Technical details

The bug lived in `LinkedFieldBox.tsx`. When a field was unlinked, the component restored only `expression` and `sourceType` from `propData` but omitted `sourceTypeSettings`. This sent a `StaticPropSource` to the server that was structurally different from the shape stored in the `Component` config entity, causing the server to render the wrong field widget. The one-line client fix adds `sourceTypeSettings: propData.sourceTypeSettings` to the payload built during the unlink action.

On the server side, `GeneratedFieldExplicitInputUxComponentSourceBase::buildComponentInstanceForm()` was hardened with a new guard: before building the form for each prop, it compares the incoming `StaticPropSource` against `getDefaultStaticPropSource()`. If the shapes differ, it throws a `NotAcceptableHttpException` (HTTP 406) with a message identifying the affected component, version, and prop name. This is intentionally a developer-facing error: it should never appear for UI-authored content but surfaces immediately in tests and recipe/default-content authoring if data is sent in the wrong shape.

The `buildComponentInstanceForm()` signature in `ComponentSourceInterface` and all implementations (`Personalization`, `BlockComponent`, `Fallback`, `InvalidSlots`, `GeneratedFieldExplicitInputUxComponentSourceBase`) changed the `$component` parameter from `?Component $component = NULL` to the non-nullable `Component $component`, removing a type-system loophole that an earlier `assert($component instanceof Component)` call had been papering over. Test setup in `CanvasTestSetup` was also simplified: prop inputs previously given as fully-expanded `StaticPropSource` arrays (with explicit `sourceType`, `expression`, `sourceTypeSettings` keys) are now expressed as plain scalars or minimal arrays, matching how the server-side component discovery populates the full shape automatically. The `link` field test values were also updated to include the `options` key and `link_type` instance setting to match the actual `StaticPropSource` shape.

## Upgrade

If you implement `ComponentSourceInterface` or extend `ComponentSourceBase` in custom code, update the `buildComponentInstanceForm()` signature to remove the nullable type and default value:

```php
// Before
public function buildComponentInstanceForm(
  array $form,
  FormStateInterface $form_state,
  ?Component $component = NULL,
  string $component_instance_uuid = '',
  array $inputValues = [],
  ?EntityInterface $entity = NULL,
  array $settings = []
): array { ... }

// After
public function buildComponentInstanceForm(
  array $form,
  FormStateInterface $form_state,
  Component $component,
  string $component_instance_uuid = '',
  array $inputValues = [],
  ?EntityInterface $entity = NULL,
  array $settings = []
): array { ... }
```

If you have test fixtures or recipes that pass fully-expanded `StaticPropSource` arrays (with `sourceType`, `expression`, `sourceTypeSettings`) for props that use a different shape than the `Component` config entity records, those will now trigger a `406 Not Acceptable` response. Update such fixtures to match the shape the component's discovery stores, or use plain scalar/array values and let the server derive the full `StaticPropSource` automatically.
