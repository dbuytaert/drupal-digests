## Summary

Canvas 1.x now supports Drupal 11.3. This update ensures Canvas continues working when Drupal core updates from 11.2 to 11.3. A new CI job tests every Canvas change against both Drupal 11.2 (the minimum supported version) and 11.3, preventing future compatibility breaks. The work involved fixing test failures caused by upstream changes in Drupal core and Symfony, particularly around block label settings and deprecated PHP/Symfony methods.

## Impact

No upgrade or configuration changes required.

## Technical details

The MR added a new CI job testing Canvas against Drupal 11.3 alongside the existing 11.2 tests. The majority of changes (150 files, +764/-230 lines) are test updates forced by upstream changes. Three upstream changes drove most test modifications. First, the `label_display` block setting changed from `type: label` to `type: string` with `NotBlank` and `Choice` constraints in Drupal core. This altered component version hashes because Canvas uses block settings' config schema in version generation. Tests needed updates to use dynamic version lookups via `Component::load()->getActiveVersion()` instead of hardcoded hashes. Second, Symfony 7.4 deprecated passing constraint options as positional arguments. All constraint instantiation now requires associative arrays: `['StringSemantics' => StringSemanticsConstraint::PROSE]` became `['StringSemantics' => ['semantic' => StringSemanticsConstraint::PROSE]]`. Third, PHPUnit attribute `#[RunTestsInSeparateProcesses]` became required on more test classes to prevent cross-test contamination. Production code changes were minimal: updating `ShapeMatchingHooks`, field type overrides, and constraint validators to use the new Symfony constraint syntax. The `ComponentInputsEvolutionTest` required special handling with version-specific hardcoded hashes using `match` statements to support both 11.2 and 11.3. The CI configuration now pins `gitlab_templates` to `1.12.x-latest` to avoid surprise breaks when upstream template defaults change.

## Collaboration

This issue spanned two months with 34 comments and contributions from phenaproxima, penyaskito, wim leers, and effulgentsia. phenaproxima drove the bulk of test fixes, hitting a wall on hash validation in `ComponentInputsEvolutionTest` where the expected hash was opaque. penyaskito provided the breakthrough in comment #24, realizing that component version hashes depend on config schema and differ between 11.2 and 11.3, requiring version-specific hardcoded values using `match`. The team prioritized this work after discovering Canvas CI was "failing HARD" when `gitlab_templates` unexpectedly bumped default Drupal from 11.2.8 to 11.3.1, breaking all in-progress merge requests. The solution: always test both versions rather than manual-only 11.3 tests, preventing future regressions. wim leers identified two systemic issues warranting follow-ups: absence of validation when constructing test cases, and the block ComponentSource storing unnecessary `label`/`label_display` data despite hiding it from users.

## Lessons learned

- When testing a library across multiple dependency versions, test infrastructure should validate every commit against all supported versions, not just the minimum. Regressions otherwise go undetected until an upstream version change forces compatibility work, creating a large batch of breaking changes that could have been addressed incrementally. *(via penyaskito, wim leers)*
- When config schema changes in a dependency (like adding constraints), any code that hashes or validates configuration will produce different results even if the actual data values are unchanged. Hash-based versioning systems must account for schema evolution separate from data evolution. *(via penyaskito, wim leers)*
- When test assertions compare opaque generated values like hashes, hardcoding expected values makes tests brittle across dependency versions. Either compute expected values from the same inputs the system uses, or assert relationships between values rather than exact matches. *(via phenaproxima, penyaskito)*
- When hiding configuration fields from users but restoring default values before save, storing those defaults wastes space and creates maintenance burden. Omit hidden fields from storage entirely and populate them just-in-time when needed by downstream systems. *(via wim leers)*
- When supporting a library across multiple dependency versions, changes to test infrastructure (like CI job configurations) should run all supported versions on every commit. Manual or on-demand testing leads to version-specific regressions accumulating unnoticed, creating large integration problems when the minimum version is eventually bumped. *(via wim leers, penyaskito)*
