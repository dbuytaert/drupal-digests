## Summary

The Canvas content template editor can now link a component prop to an entity field. Previously, component instance forms in content templates only supported static inputs. Now each prop that has a compatible entity field shows a link icon next to its label. Clicking the icon opens a dropdown of matching fields; selecting one replaces the input widget with a badge showing the linked field name and an unlink button. This connects the `DynamicPropSource` infrastructure that already existed to a working editor UI.

## Impact

Affects module maintainers.

- **New API:** `ContentTemplate::getTargetEntityDataDefinition()`, `ContentTemplate::createEmptyTargetEntity()`, and `DynamicPropSource::label()` added; `FieldForComponentSuggester::structureSuggestionsForResponse()` extracted as a public static method.

## Technical details

On the server side, `GeneratedFieldExplicitInputUxComponentSourceBase::buildComponentInstanceForm()` now branches on whether `$entity` is a `ContentTemplate`. When it is, the method calls `FieldForComponentSuggester::suggest()` for every prop and passes the results through the newly extracted static helper `FieldForComponentSuggester::structureSuggestionsForResponse()` (previously that formatting logic lived inline in `ApiUiContentTemplateControllers`). For each prop that already carries a `DynamicPropSource`, the widget array is entirely replaced with a new `linked_prop_source` render element (`LinkedPropSource` class). For linkable-but-unlinked props, a `#prop_link_data` key is attached to the widget, which the Stark theme's `canvas_stark_preprocess_field_multiple_value_form()` copies into label attributes so the frontend `PropLinker` component can read it. Because field widgets need a real `FieldableEntityInterface` for context, `ContentTemplate::createEmptyTargetEntity()` is introduced to create a throw-away empty entity of the template's target type and bundle for that purpose. `ContentTemplate::getTargetEntityDataDefinition()` was also added to return the typed-data definition needed by the suggester. `DynamicPropSource::label()` is a new method that walks the field definitions for the expression's entity type and returns the human-readable field label, used by `LinkedPropSource` to populate the badge title.

On the frontend (TypeScript/React), two new components handle the UI. `PropLinker` renders as a link icon beside the field label and opens a Radix UI `DropdownMenu` listing field suggestions; selecting one dispatches `updateExistingComponentValues` with both `values` and a new optional `sources` map so the source expression is stored alongside the resolved value. `LinkedFieldBox` replaces the entire widget when a prop is already linked, showing the field label and an unlink button that calls `patchComponent` to revert to the static default. Both components are registered in `twigToJSXComponentMap` and exposed as Twig themes (`prop_linker`, `linked_field_box`) declared in `canvas_stark_theme()`. A bug found during review required updating both the `layoutModel` store and the `formStateSlice` serialized `form_canvas_props` value when a prop is linked; `setFieldValue` was extended with an `updateSource` flag to handle this. The `Attribute` TypeScript type was also widened to include `object` to accommodate `prop_link_data` being passed through Drupal attributes.

`FieldForComponentSuggester` is now injected into `GeneratedFieldExplicitInputUxComponentSourceBase` (and its concrete subclass `SingleDirectoryComponent`) so the form builder can reuse the same suggestion and formatting logic as the HTTP API endpoint from [#3510896](https://www.drupal.org/node/3510896).
