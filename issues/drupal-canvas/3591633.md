# #3591633: Move CLI sync settings to canvas.config.json, enable everything by default

- **Project:** Drupal Canvas
- **Type:** Normal feature request
- **Status:** Fixed
- **Discussion:** [#3591633](https://git.drupalcode.org/project/canvas/-/work_items/3591633) · 1 contributor · 4 comments

## Summary

The Canvas CLI `push` and `pull` commands now read their sync preferences (whether to include pages, content templates, and global regions) from a `sync` key in `canvas.config.json` rather than from environment variables. All three content types are now enabled by default, so a project with no `sync` key set will automatically include pages, content templates, and regions in every push/pull operation. The old `CANVAS_INCLUDE_PAGES`, `CANVAS_INCLUDE_CONTENT_TEMPLATES`, and `CANVAS_INCLUDE_REGIONS` environment variables continue to work but emit deprecation warnings and the CLI offers to migrate them to `canvas.config.json` automatically.

## Impact

Affects module maintainers and site owners.

- **Config change:** New optional `sync` key in `canvas.config.json` controls whether pages, contentTemplates, and regions are included in CLI push/pull; all three default to true.
- **Deprecation:** `CANVAS_INCLUDE_PAGES`, `CANVAS_INCLUDE_CONTENT_TEMPLATES`, `CANVAS_INCLUDE_REGIONS` environment variables and `--include-pages`/`--include-content-templates`/`--include-regions` CLI flags deprecated in favour of `canvas.config.json` sync settings and `--no-*` flags.

## Technical details

The `sync` object is now the canonical source of truth for per-content-type inclusion. It is read from `canvas.config.json` by `readConfiguredSyncSettings()` in `packages/cli/src/config.ts`, with the `DEFAULT_CANVAS_CONFIG.sync` object in `packages/discovery/src/config.ts` providing `{ pages: true, contentTemplates: true, regions: true }` as defaults. If a `sync` key is absent from `canvas.config.json`, the defaults apply; if it is present, each key overrides the corresponding default.

The legacy env vars are handled by `LEGACY_SYNC_ENV_SETTINGS` and the `applySyncOptionAliasesAndWarnings` migration helper. When the env vars are still set, the CLI still honors them (feeding them through `setConfig`) but warns the user and, if confirmed interactively, writes the equivalent `sync.*` keys back to `canvas.config.json`. The `--include-pages`, `--include-content-templates`, and `--include-regions` CLI flags have similarly been deprecated in favor of negative flags (`--no-pages`, `--no-content-templates`, `--no-regions`) for disabling sync at runtime, because the positive forms are now no-ops when everything defaults to enabled.

## Upgrade

Update `canvas.config.json` to include the `sync` key and remove the old environment variables:

```jsonc
// canvas.config.json
{
  "sync": {
    "pages": true,
    "contentTemplates": true,
    "regions": true
  }
}
```

Remove these environment variables from `.env` and CI pipelines (they are now deprecated):
```
CANVAS_INCLUDE_PAGES
CANVAS_INCLUDE_CONTENT_TEMPLATES
CANVAS_INCLUDE_REGIONS
```

To selectively disable sync for one content type, set the value to `false` in `canvas.config.json`, or pass `--no-pages` / `--no-content-templates` / `--no-regions` on the command line. The previously preferred flags `--include-pages=false` etc. are deprecated.
