## Summary

Canvas CLI now discovers and builds Code Components automatically from any directory structure. The new build command finds components by scanning for component metadata files, bundles third-party dependencies and local imports with Vite, compiles JavaScript and CSS, and generates a manifest tracking all assets. This replaces the manual per-component workflow with automatic discovery and optimized production builds. The old build command remains available as build-d but is deprecated.

## Impact

No upgrade or configuration changes required.

## Technical details

The new `build` command implements automatic component discovery using the `@drupal-canvas/discovery` package. It scans for `component.yml` or `[name].component.yml` files and matches them with JavaScript and CSS entry files following naming conventions (index or named patterns).

Build workflow: 1) Discovery scans the configured directory and validates components. 2) Import analysis walks the dependency graph from component entry files, categorizing imports as third-party packages, alias imports (`@/` prefix), or relative imports. 3) Vendor bundling uses Vite to create optimized bundles for third-party packages in `dist/vendor/` with code splitting and tree-shaking. Packages already provided by Drupal Canvas (React, Preact, etc.) are marked as external. 4) Local import bundling processes `@/` alias imports—JavaScript modules are bundled, CSS files are copied, and image/SVG/font assets are copied preserving extensions. 5) Individual components are compiled with SWC (now supporting TypeScript/TSX syntax). 6) Tailwind CSS is generated by extracting class name candidates from all components. 7) A manifest file maps import specifiers to their bundled output paths.

Configuration uses two files: `canvas.config.json` (committed, defines codebase structure like `componentDir`, `aliasBaseDir`, `outputDir`) and `.env` (environmental secrets like OAuth credentials). The Vite build configuration uses the new `@drupal-canvas/vite-compat` plugin that provides alias resolution, SVGR for SVG imports, and handles side-effect CSS imports.

The deprecated `build-d` command preserves the old workflow where components are selected individually and built without Vite bundling.

### Collaboration

The issue evolved significantly from prototype to committed implementation over one month. balintbrews opened with a detailed spec and proof-of-concept, then extracted shared discovery and Vite compatibility packages while building Canvas Workbench in parallel ([#3573867](https://www.drupal.org/node/3573867)). hooroomoo implemented the build command, followed by design discussions between hooroomoo, mglaman, and balintbrews about manifest structure. They decided to drop component tracking from the manifest (comments #8-12) since components can be discovered dynamically and the manifest should focus on vendor and local assets. Testing revealed edge cases around CSS subpath imports and same-named files in different directories, which shaped the final import resolution logic.

## Lessons learned

- When building a manifest for a compilation process, track only the artifacts that cannot be rediscovered through convention. If components can be found by scanning for metadata files (e.g., *.yml), exclude them from the manifest and track only vendor dependencies and imported assets that lack discoverability conventions. *(via mglaman, hooroomoo, balintbrews)*
