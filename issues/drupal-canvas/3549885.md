# #3549885: Prevent deleting Code Components if there are usages in forward revisions

- **Project:** Drupal Canvas
- **Type:** Critical bug report
- **Version:** 1.x-dev
- **Status:** Fixed

- **Followers:** 5

## Summary

Deleting a Code Component in Canvas now also checks whether that component is referenced in any forward (pending, non-default) revision, such as content staged inside a Workspace. Previously, only auto-save storage and default revisions were checked; forward revisions were silently ignored. This meant that publishing a workspace after deleting a component it used would cause those pages to render the component fallback instead of the intended content. The fix prevents the deletion from being permitted when the component is in use in any latest revision.

## Impact

No upgrade or configuration changes required.

## Technical details

The delete access check lives in `VisibleWhenDisabledCanvasConfigEntityAccessControlHandler::checkAccess()`. Before this fix it ran two `AccessResult::forbiddenIf()` calls chained with `->orIf()`: one querying `RevisionAuditEnum::AutoSave` (auto-save storage) and one querying `RevisionAuditEnum::Default` (current/default revisions). A third `->orIf()` check using `RevisionAuditEnum::Latest` was added, which translates to `$query->latestRevision()` inside `ComponentAudit::getContentRevisionIdsUsingComponentIds()`. This covers any non-default, pending revision (most notably content that has been edited inside a Workspace but not yet published). The ordering is intentional: auto-save is checked first because it is cheapest; default revision next; latest revision last, as it is the most expensive and only relevant when the preceding checks pass.
