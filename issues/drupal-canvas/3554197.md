## Summary

Canvas content templates now support linking multiple-cardinality field widgets, such as a Tags taxonomy reference field, to component properties that accept arrays. Previously the linker button did not appear next to these fields at all, making it impossible to connect multi-value fields to components expecting a list of values. After linking, the component preview correctly renders all values from the field.

## Impact

No upgrade or configuration changes required.

## Technical details

Multi-value field widgets in Drupal render their field label as an `<h4>` element inside a table header, not through the standard `#title_display` render element. The existing linker logic in the `canvas_stark` theme only attached to elements that had `#title_display`, so it never fired for multi-value fields.

The fix extends `canvas_stark_preprocess_field_multiple_value_form()` in `themes/canvas_stark/canvas_stark.theme`. When `#prop_link_data` is present on the field element and `$variables['multiple']` is truthy, the first table header cell is replaced with a container render array that wraps both the original `<h4>` and a new `prop_linker` theme element. A matching CSS rule `.canvas-linked-prop-label-wrapper--multiple-values` applies flexbox alignment. No changes were needed to the React layer or to `GeneratedFieldExplicitInputUxComponentSourceBase.php`.

The solution avoids rendering a duplicate linker on each individual row because [#3550848](https://www.drupal.org/node/3550848) added a `titleDisplay !== 'invisible'` guard in `DrupalFormElementLabel.tsx`. Drupal core sets `#title_display => 'invisible'` for individual items in multi-value fields (in `WidgetBase.php` lines 207-213), so the React linker component suppresses itself on those rows and renders only once in the table header.

An earlier approach in MR !679 blocked duplicate linkers by stopping `propLinkData` propagation inside `GeneratedFieldExplicitInputUxComponentSourceBase.php`. After [#3550848](https://www.drupal.org/node/3550848) landed, that logic became redundant. tedbow investigated this and documented the analysis in comment #18, confirming that MR !278's theme-only approach was sufficient, then closed MR !679. End-to-end test coverage was added in `tests/src/Playwright/templates-linking.spec.ts`, backed by an expanded `template_basic_setup` recipe that adds an unlimited-cardinality `field_multiple_tags` entity reference field and fixture content with two taxonomy terms, verifying both values render in the component preview after linking.
