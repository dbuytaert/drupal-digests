## Summary

Canvas now supports linking multiple-value fields (like a tags field that can store several taxonomy terms) to component properties. Previously, the linker button did not appear for fields that allow multiple values because they render their label differently than single-value fields. This feature enables content editors to connect multi-value entity fields to components that accept arrays, making it possible to display lists of tags, categories, or other multi-value content in custom components.

## Impact

No upgrade or configuration changes required.

## Technical details

Multi-value field widgets in Drupal render their labels as an `<h4>` element inside a table header, unlike single-value fields that use standard `#title_display` rendering. The existing linker logic only attached to elements with `#title_display`, so it never appeared for multi-value fields.

The fix adds logic in `canvas_stark_preprocess_field_multiple_value_form()` to detect when `#prop_link_data` is available on a multiple-value field element. When present, it reconstructs the table header to wrap both the existing `<h4>` and a new linker element in a container with flexbox styling.

A key dependency is [#3550848](https://www.drupal.org/node/3550848), which added a `titleDisplay !== 'invisible'` check in `DrupalFormElementLabel.tsx`. This prevents duplicate linkers from appearing on individual rows within multi-value fields. Drupal core sets `#title_display => 'invisible'` for individual items in multi-value fields (see `WidgetBase.php` lines 207-213), so the linker only renders once in the table header where `titleDisplay` is not invisible.

The solution is theme-layer only because the React component already handles the `titleDisplay` filtering correctly.

### Collaboration

This issue took four months from opening to commit, with significant complexity around finding the right approach. An earlier attempt made changes to `GeneratedFieldExplicitInputUxComponentSourceBase.php` to prevent duplicate linkers on each row of multi-value fields. However, during development, [#3550848](https://www.drupal.org/node/3550848) was committed with a more general solution that made those changes unnecessary.

The git history was accidentally lost through force-pushing, causing confusion during review. tedbow investigated why the earlier approach was no longer needed and documented the analysis in #17, confirming that the `titleDisplay !== 'invisible'` check from [#3550848](https://www.drupal.org/node/3550848) already solved the duplicate linker problem. This allowed the final solution to be simpler, requiring only theme-layer changes.

The issue also benefited from [#3546760](https://www.drupal.org/node/3546760) landing during development, which added end-to-end template tests. This enabled proper test coverage to be added, including verification that both values from a multi-value field appear in the component preview.
