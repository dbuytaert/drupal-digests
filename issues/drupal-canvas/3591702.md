# #3591702: Fatal error (AssertionError on dev env) when a component instance's image references a deleted file

- **Project:** Drupal Canvas
- **Type:** Critical bug report
- **Version:** 1.x-dev
- **Status:** Fixed
- **Discussion:** [#3591702](https://git.drupalcode.org/project/canvas/-/work_items/3591702) · 1 contributor · 3 comments

## Summary

Opening a Canvas editor page that contains a component with an image prop pointing at a deleted file no longer produces a 500 error. Previously, the editor would fatal with an `AssertionError` and log repeated failures whenever a referenced file or media entity had been removed, for example after a content reset or partial re-import. After this fix, the affected component renders without its image and the rest of the page continues to load normally. The empty render is also correctly cache-tagged so it is automatically invalidated if the deleted entity is ever restored.

## Impact

No upgrade or configuration changes required.

## Technical details

The root cause was in `ComputedUrlWithQueryString`, which evaluates an image prop expression into a URL and then asserts `\is_string($url->value)`. When the referenced file had been deleted, `Evaluator::evaluate()` returned a `GeneratedUrl` whose `value` was `NULL`: the field item existed (so the `is_required` guard in `Evaluator` did not fire), but the entity no longer loaded. That `NULL` flowed directly into the `assert()`, fataling on any environment with assertions enabled.

The fix adds an explicit `NULL` check before the assertion in `ComputedUrlWithQueryString`. When `$url->value === NULL` and the prop expression implements `ReferencePropExpressionInterface`, the method resolves the raw `EntityReference` typed-data property, delegates to the new `TypedDataHelper::getDeletedReferencedEntityCacheability()` helper, attaches the resulting cache tags to the partially-built `GeneratedUrl`, and returns early without touching the `assert()`.

The deleted-entity cacheability logic that previously lived inline in `Evaluator` (with a comment about it being "imperfect") was extracted into the new public static method `TypedDataHelper::getDeletedReferencedEntityCacheability(EntityReference $reference): CacheableDependencyInterface`. Both call sites, `ComputedUrlWithQueryString` and `Evaluator`, now use this single helper. The cache tag produced is `entity_type_id:id`, which is the default for most entity types per `EntityBase::getCacheTagsToInvalidate()`; the code notes this is imperfect for entity types that override that method. A new kernel test `ComputedUrlWithQueryStringTest::testHandlesBrokenReference()` verifies that accessing `src_with_alternate_widths` on a field item whose file has been deleted returns an empty `GeneratedUrl` carrying the deleted file's cache tags, without throwing.
