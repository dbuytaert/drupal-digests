## Summary

Sites using the Canvas layout builder could encounter a fatal assertion failure during component version updates when a required prop (such as an entity reference field) existed in both the old and new component versions, already had a user-provided value, but had no example value defined in the component metadata. The updater now validates existing inputs against the target version before deciding whether to apply a default, and correctly skips the assertion for props whose existing values are already valid.

## Impact

No upgrade or configuration changes required.

## Technical details

The root cause was in `GeneratedFieldExplicitInputUxComponentInstanceUpdater::updateComponentInstanceProps()`. The call to `getDefaultExplicitInput()` and its `assert(array_key_exists('value', ...) && ... !== NULL)` ran unconditionally at the top of the required-prop loop for every required prop, before any check of whether a default was actually needed. The default is only needed for new props, for optional-to-required transitions without an existing value, or for existing inputs that are invalid in the new version. When a prop existed in both versions and already had a valid user input, the assertion still fired and failed if the component metadata provided no example value.

The fix restructures the method in two ways. First, it calls `$target_source->validateComponentInput()` up front against the projected input set to collect per-prop violations in the target version; this produces an `$invalid_prop_names` map used throughout. Second, the loop continues past a required prop (without touching `getDefaultExplicitInput()`) when the prop's existing input is present and passes validation. The `getDefaultExplicitInput()` call and its assertion are reached only when a default is genuinely required.

The stale-input removal step was also broadened: instead of only removing props absent from the `from`-to-`to` delta, it now removes any inputs keyed to props not present in `to_props` at all. This handles pre-1.1 "garbage" inputs (inputs that accumulated for props that never existed in any schema version). For such garbage inputs the validity check determines the outcome: if the garbage value is valid for a newly added required prop it is preserved; if invalid it is replaced with the prop's example default. The commit also adds a new data-provider case (`"Required prop stays required while a new optional prop is added"`) and a dedicated `testUpdateWithGarbageInput()` test with two scenarios (garbage value valid vs. invalid for the incoming prop) to cover these paths.

### Collaboration

The fix took several weeks to land because of a substantive design question about "garbage" inputs. longwave simplified the initial MR by proposing an early-bail structure rather than moving the assertion into branches. wim leers then flagged missing test coverage in `GeneratedFieldExplicitInputUxComponentInstanceUpdaterTest` and separately raised the concern that treating a pre-existing garbage value as valid simply because it was present could silently preserve corrupt data. penyaskito weighed in with a view that accepting some data-loss risk was acceptable to make progress. After further discussion on the MR, the team converged on using `validateComponentInput()` to make the validity judgment explicitly, satisfying wim leers's concern without treating presence as a proxy for correctness.
