## Summary

When mapping entity data to a `type: integer` prop in a Canvas ContentTemplate, the `DynamicPropSource` suggestion list previously included every integer-typed entity field regardless of its meaning. This produced dozens of nonsensical suggestions such as node creation timestamps, user last-login times, and file sizes appearing as candidates for props intended to hold image dimensions or counts. After this fix, UNIX timestamp fields are excluded from the default integer suggestions. A component that genuinely needs a UNIX timestamp integer can still opt in by explicitly declaring `minimum: -2147483648` and `maximum: 2147483648` in its JSON schema prop definition.

## Impact

No upgrade or configuration changes required.

## Technical details

The shape-matching system determines which entity field instances are compatible with a given SDC prop's JSON schema shape. For an unrestricted `type: integer` prop, `JsonSchemaType::toDataTypeShapeRequirements()` previously returned `FALSE` (match any integer field). It now returns a negated `DataTypeShapeRequirement`: match any integer field that does NOT carry Drupal core's `Range` constraint with `min: '-2147483648'` / `max: '2147483648'`. That Range constraint is exactly what `TimestampItem` declares for its `value` property, making timestamp fields the excluded class.

To support negation, `DataTypeShapeRequirement` gained a new `negate` constructor parameter (defaulting to `FALSE`). The matching logic in `JsonSchemaFieldInstanceMatcher` was updated to apply XOR with `negate` when evaluating a requirement. A plain `type: integer` prop with explicit `minimum`/`maximum` values matching the timestamp range continues to match timestamp fields as before, so components that deliberately need UNIX timestamps can declare this intent in their schema.

A second infrastructure change in `ShapeMatchingHooks::entityBaseFieldInfoAlter()` forward-ports the Drupal core fix from [#3521088](https://www.drupal.org/node/3521088): it copies the `constraints` from the `timestamp` base field definition onto the `changed` and `created` base fields, which were missing the Range constraint in Drupal versions prior to 11.3. Without this, `created` and `changed` fields would still slip through the exclusion filter on older core versions.

## Upgrade

SDC components that declare a bare `type: integer` prop and previously relied (intentionally or accidentally) on timestamp fields being suggested will no longer see those suggestions. To explicitly allow UNIX timestamp inputs for an integer prop, add the timestamp range to the prop schema:

```yaml
# In your component's *.component.yml or JSON schema
properties:
  my_timestamp_prop:
    type: integer
    minimum: -2147483648
    maximum: 2147483648
```
