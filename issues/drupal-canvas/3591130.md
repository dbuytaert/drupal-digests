## Summary

When a Canvas component is renamed via the CLI, the operation is actually a delete followed by a create. Any `Folder` config entity that listed the old component in its `items` retained the stale name after the delete, because Canvas's dependency tracking did not know the connection existed. A later operation on that folder (such as a PATCH) would fail with a 422 "Unprocessable Content" error and a message like "The 'canvas.component.js.media-feature-grid' config does not exist." The fix adds proper config-dependency declarations to `Folder` so that Drupal's built-in dependency resolution removes deleted components from folder items automatically. Existing sites need to run database updates to recompute the previously missing dependencies.

## Impact

Affects site owners.

- **Database update:** New post-update function resaves all `Folder` entities to compute previously missing config dependencies; run `drush updb` after updating.

## Technical details

The root cause is that `Folder` never overrode `calculateDependencies()`. It inherited `ConfigEntityBase::preDelete()` and `ConfigEntityBase::onDependencyRemoval()` unchanged, but those methods can only act on dependencies that have already been declared. Because no dependencies were declared, deleting or renaming a `JavaScriptComponent` (or any other config entity stored in a folder) left the `Folder`'s `items` array untouched. The fix in !1133 implements `Folder::calculateDependencies()` to register each item in `items` as a config dependency. Once those dependencies are present in the config graph, `ConfigEntityBase::onDependencyRemoval()` can fire and strip the deleted entry from `items` without any custom delete-hook logic. A post-update function (modeled on `canvas_post_update_0002_intermediate_component_dependencies_in_patterns()` and its companion `UpdateIntermediateDependenciesUpdateTest`) was added to resave all existing `Folder` entities so their dependency metadata is computed and stored. This MR (!1135) contains only the corresponding test in `CanvasConfigEntityHttpApiTest`, asserting that after an HTTP DELETE of a component, the folder's `items` list is empty. A related follow-up issue, [#3591475](https://www.drupal.org/node/3591475), tracks the additional behavior of deleting the `Folder` itself when its last dependency is removed.

## Upgrade

After updating the module, run database updates to recompute folder dependencies on existing sites:

```sh
drush updb
# or
vendor/bin/drush updb
```

Without this step, existing `Folder` config entities will continue to lack declared dependencies and the bug will persist for them even after the code update.
