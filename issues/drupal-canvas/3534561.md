## Summary

Canvas pages that contain component trees referencing other entities (media, files, and similar content entities) can now be exported and re-imported using Drupal core's default content exporter, which was introduced in Drupal 11.3. Previously there was no way for the exporter to discover which entities a Canvas component tree depended on, making site templates that ship Canvas-ified content either incomplete or broken on import. With this change, exporting a Canvas page with `--with-dependencies` will automatically pull in all transitively referenced content entities, and the entity references stored in component inputs are serialized as UUIDs so they survive transport to a different site where serial IDs differ. The integration is inactive on Drupal 11.2 and earlier and requires no configuration.

## Technical details

The fix introduces `DefaultContentSubscriber` in `src/EventSubscriber/DefaultContentSubscriber.php`, which listens to three events from core's `Drupal\Core\DefaultContent` namespace: `PreExportEvent`, `PreImportEvent`, and `PreEntityImportEvent`. The subscriber is registered conditionally in `CanvasServiceProvider::alter()` only when `Drupal\Core\DefaultContent\Exporter` exists, keeping the module compatible with Drupal 11.2.

On export, the subscriber registers a callback for field items of type `field_item:component_tree` via `PreExportEvent::setCallback()`. That callback calls `ComponentTreeItem::calculateFieldItemValueDependencies()` on the host entity to resolve the full set of content dependencies, then loads each referenced entity by UUID and passes it to `ExportMetadata::addDependency()`. For entity-reference prop inputs stored as static prop sources, the serialized `target_id` is replaced with a special `CANVAS_ENTITY_REFERENCE` key holding `target_uuid` and `target_type`. Using a dedicated key rather than the conventional `target_uuid` avoids ambiguity at import time: the importer can identify Canvas entity references without inspecting component prop sources. The `component_version` field is stripped from the export so that version changes on the destination site do not cause import failures, provided required props are still satisfied.

On import, `preImport()` caches the component-tree field map once at the start of the batch, and `preEntityImport()` walks each component tree field in the incoming YAML data, calling `processComponentInputOnImport()` for any prop input that carries the `CANVAS_ENTITY_REFERENCE` key. That helper loads the entity by UUID and replaces the marker with `target_id`. The export callback also normalizes uncollapsed prop-source syntax (the long `sourceType`/`value`/`expression` form) to its collapsed scalar equivalent, ensuring imported pages pass validation. A new functional test `DefaultContentExportImportTest` covers the full round-trip for media, file, and code-component inputs and is run in CI only against Drupal 11.3 via a dedicated `phpunit (>11.2-only features)` job. The second commit renames the abstract test base class `GeneratedFieldExplicitInputUxComponentSourceBaseTest` to `GeneratedFieldExplicitInputUxComponentSourceBaseTestBase` to suppress a PHPUnit 11.3 warning about abstract classes whose names end in `Test`. Related: [#3544604](https://www.drupal.org/node/3544604) (dependency calculation for intermediate entities in reference chains).
