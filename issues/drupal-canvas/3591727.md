# #3591727: Deleting optional props from an SDC/code component causes validation errors in config-defined component instances: regression for monolingual sites, bug for multilingual sites

- **Project:** Drupal Canvas
- **Type:** Major task
- **Status:** Fixed
- **Discussion:** [#3591727](https://git.drupalcode.org/project/canvas/-/work_items/3591727) · 1 contributor · 3 comments

## Summary

When an optional prop was removed from an SDC or code component (triggering a new component version), existing component tree instances that referenced the old version would produce config validation errors, even though those instances were otherwise untouched and completely valid. On multilingual sites the same root cause also broke prop translatability for component instances at old versions. This is now fixed. A one-time post-update hook backfills the new per-prop `derived_schema_metadata` onto all existing `Component` config entities; automatic component instance updating then handles migrating instances to the current version as before.

## Impact

Affects site owners.

- **Database update:** Post-update hook `canvas_post_update_0021_store_prop_derived_schema_metadata` backfills `derived_schema_metadata` on all existing `Component` config entities.
- **Config change:** New `derived_schema_metadata` key (with optional `string_shape` sub-key) added to each prop entry inside `type: canvas.json_schema_props`; all `Component` config exports will include it after the update.

## Technical details

The root cause was that `JsonSchemaPropsComponentInstanceInputsConfigSchemaGenerator::getConfigSchemaMapping()` and `ComponentInputsTranslatablesExtractor` both called `getMetadata()` on the live component source to determine which props exist and which are translatable. The live source only knows about the currently deployed JSON schema: a prop removed from the live SDC or code component is simply absent from `getMetadata()`, so the generated config schema mapping dropped the entry for it, causing config validation to reject any stored instance that still referenced that prop under the old component version.

The fix stores a new `derived_schema_metadata` key per prop inside each version's `prop_field_definitions` (under `type: canvas.json_schema_props`). Its only current sub-key is `string_shape`: the translation-relevant subset of the prop's JSON schema at the time that version was created (plain string, multi-line `pattern`, rich text `contentMediaType`/`x-formatting-context`, or URI-esque `format`). The new static `PropShape::getTranslatableStringShape()` extracts that subset from a raw or normalized JSON schema, and returns `NULL` for non-translatable shapes such as integers or string enums. This replaces the removed static `isTranslatableShape()` on `JsonSchemaPropsComponentInstanceInputsConfigSchemaGenerator`.

`getConfigSchemaMapping()` now iterates `prop_field_definitions` (versioned, config-stored data) instead of calling `getMetadata()`, so a removed or changed prop keeps its mapping entry for every version that still names it. Translatability is determined by the new `JsonSchemaPropsComponentSourceBase::isExplicitInputTranslatable()`, which reads `derived_schema_metadata['string_shape']` from the versioned config and additionally checks that the prop is not backed by an entity reference. Inputs that are populated in a stored instance but have no field definition for that version (for example, props whose shape is not storable) now receive a non-translatable `type: ignore` entry rather than being silently dropped.

`derived_schema_metadata` is deliberately excluded from the version hash. `ComponentSourceBase` gains a new `settingsAffectingVersionHash()` hook that subclasses can override to strip purely derived data; `JsonSchemaPropsComponentSourceBase` overrides it to remove `derived_schema_metadata` before hashing, so backfilling or recomputing this field never changes existing component version ids.

The post-update hook `canvas_post_update_0021_store_prop_derived_schema_metadata` runs `CanvasConfigUpdater::updatePropFieldDefinitionsWithDerivedSchemaMetadata()` on all existing `Component` config entities. For each version it can still derive the shape from the live implementation (active version only); past versions whose live schema is no longer available receive an empty `derived_schema_metadata` and are treated as non-translatable until the instance is updated to the active version. A side change makes props with a `minLength` JSON schema constraint non-storable (returning `NULL` from `JsonSchemaType`), because length constraints interact poorly with translation.

## Upgrade

No application code changes are required. After running database updates, the post-update hook `canvas_post_update_0021_store_prop_derived_schema_metadata` will backfill `derived_schema_metadata` on all `Component` config entities automatically. If you export config after updating, every `Component` config file will gain `derived_schema_metadata: {}` (or `derived_schema_metadata: {string_shape: {…}}`) under each prop in `prop_field_definitions`. Commit those exports. No changes to SDC `.component.yml` files or code component definitions are needed.
