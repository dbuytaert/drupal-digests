## Summary

Canvas content templates can now link to the entity being displayed. A new prop source type, `host-entity-url`, generates the canonical absolute URL of the host entity and wires directly to any URI-typed prop in a component tree. Previously there was no way to do this inside a component tree; the workaround in Drupal CMS was a custom computed `url` field on nodes. That workaround can now be removed.

## Impact

Affects module maintainers.

- **New API:** New `sourceType: 'host-entity-url'` value usable in `ContentTemplate` component tree config to populate any `type: string, format: uri|uri-reference|iri|iri-reference` prop with the host entity's canonical absolute URL.

## Technical details

The new `HostEntityUrlPropSource` class (in `Drupal\canvas\PropSource`) serializes as `{"sourceType": "host-entity-url"}` and evaluates by calling `$host_entity->toUrl('canonical')->setAbsolute()->toString(TRUE)->getGeneratedUrl()`, returning an absolute URL that respects path aliases. It is permitted exclusively in `ContentTemplate` component trees: the `canvas.content_template.*.*.*` config schema allows it, while `canvas.component_tree`, `field.value.component_tree`, and `canvas.pattern.*` all explicitly list it under `absence`. The `ComponentTreeMeetsRequirements` constraint was updated to recognize the new `sourceTypePrefix` so that validation errors are actionable.

The shape-matching layer in `FieldForComponentSuggester` was extended to track `host_entity_urls` alongside `instances` and `adapters`. A new static `matchHostEntityUrlPropSources()` method matches `HostEntityUrlPropSource` for any prop shape that is `type: string` with a `format` of `uri`, `uri-reference`, `iri`, or `iri-reference`, subject to two exclusions: shapes that carry a `contentMediaType` restriction (e.g., image props) and shapes whose `x-allowed-schemes` list excludes both `http` and `https`. Matched suggestions are placed at synthetic depth `-1` in `structureSuggestionsForResponse()`, so they appear first in the UI before the field hierarchy. `PropShape::getType()` was added as a helper to support this check. The `FieldForComponentSuggester` return type was also updated: `instances` now holds `DynamicPropSource` objects (wrapping the previous `StructuredDataPropExpressionInterface` expressions) rather than raw expressions.

Follow-up issues track configuring relative vs. absolute URLs and alternative link templates ([#3551455](https://www.drupal.org/node/3551455)), a UI bug where clicking the suggestion had no effect ([#3555068](https://www.drupal.org/node/3555068)), and linking to referenced entities ([#3555413](https://www.drupal.org/node/3555413)). The `FieldForComponentSuggester` rename to `PropSourceSuggester` is tracked in [#3523446](https://www.drupal.org/node/3523446).

## Upgrade

If you previously worked around the missing feature by adding a computed `url` field to nodes (as Drupal CMS's helper module did), you can replace that workaround. In your content template config, use `sourceType: 'host-entity-url'` for any URI-typed prop:

```yaml
inputs:
  href:
    sourceType: 'host-entity-url'
```

Note that `host-entity-url` is only valid inside a `ContentTemplate` component tree. Using it in a `Pattern`, `PageRegion`, or field-value component tree will produce a validation error.

### Collaboration

The issue started when phenaproxima reported being unable to link to the displayed entity from a Canvas content template, and described the computed `url` field shim used in Drupal CMS as a stop-gap. wim leers laid out four implementation options in a detailed comment: (1) the computed field approach already in MR !96, (2) bringing a subset of `AdaptedPropSource` support into Canvas, (3) a computed `canonical_url` property on the ID field's field type, and (4) a dedicated `HostEntityUrlPropSource` class. He opened MR !199 implementing option 4 and made the case that it was the cleanest path given existing infrastructure. After MR !96 was closed and !199 merged as the first commit, wim leers immediately flagged in the thread that the merge was premature: the UI suggestion code was commented out with a `@todo`, tests and documentation were incomplete, and shape-matching support was missing. phenaproxima addressed tests and docs in a follow-up MR !222 (the second commit), while wim leers contributed the shape-matching changes and documentation for section 3.1.4 and 3.2.2 in `docs/shape-matching.md`.
