## Summary

Exporting a canvas page that contains a Drupal view block or any other block component (not an SDC or JavaScript component) crashed the `content:export` command with a `TypeError` in `PropSource::parse()`. The export would abort entirely before producing output. After this fix, block component instances are skipped during prop-source processing and the export completes successfully.

## Impact

No upgrade or configuration changes required.

## Technical details

The export callback in `DefaultContentSubscriber` iterated over every component's inputs and passed each one to `PropSource::parse()`, which requires an array. Block components store their configuration as plain scalar values (e.g., `'label' => ''`, `'level' => 1`) rather than the structured array format prop sources expect, so the call immediately threw a `TypeError`.

The fix adds an `instanceof GeneratedFieldExplicitInputUxComponentSourceBase` guard inside the loop. Only SDC and JavaScript component sources extend that base class and therefore support the prop-source model. When the component source is not an instance of that class, the loop skips prop-source processing with `continue`. A `@todo` in the committed code documents that the preferred long-term replacement is `ComponentInputs::getPropSourcesUsingExpressionClass(ReferenceFieldTypePropExpression::class)`, but that approach is blocked on [#3566720](https://www.drupal.org/node/3566720).

The functional test `DefaultContentExportImportTest` was extended with a `block.system_menu_block.main` component instance (UUID `ce05b065-00e7-43c4-8808-ac757de1c98a`) to cover this code path, and the `assertSame` on the process exit code now surfaces the command output on failure for easier diagnosis.
