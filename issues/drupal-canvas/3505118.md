## Summary

The Experience Builder editor's status badge previously showed only "Published" or "Draft" based on the raw Drupal `status` field, which was misleading: it could show "Published" for a page that had never actually been published. The badge now reflects the real editing state. It shows "Draft" (blue) for a newly created page that has never been published, "Changed" (yellow) when there are pending autosaved changes on a page that was previously published, and "Published" (green) when the published version matches what is in the editor. The badge also updates immediately when edits are made, instead of waiting up to ten seconds for the next polling cycle.

## Impact

Affects module maintainers.

- **New API:** New `isNew` and `isPublished` boolean fields added to the `GET /xb/api/layout/{entityTypeId}/{entityId}` response; both are now required properties in the OpenAPI schema.

## Technical details

The back-end change adds two new boolean fields, `isNew` and `isPublished`, to the `GET /xb/api/layout/{entityTypeId}/{entityId}` API response. `isPublished` is taken directly from `EntityPublishedInterface::isPublished()`. `isNew` uses a heuristic: the entity is considered new when its current label still matches the default title generated at creation time (`ApiContentControllers::defaultTitle()`). To make that default title available in both the creation path and the newness check, it was extracted into a new public static method `ApiContentControllers::defaultTitle(EntityTypeInterface $entity_type)`, which returns a `TranslatableMarkup` of `'Untitled @singular_entity_type_label'`. The `ApiLayoutController::get()` method now requires its entity parameter to implement `EntityPublishedInterface` (the type hint was narrowed to `FieldableEntityInterface&EntityPublishedInterface`). Both fields are read from the autosave data when a draft exists, or from the live entity otherwise. The OpenAPI spec in the project was updated to document both fields as required properties on the response schema.

On the front end, a new `PageStatus` React component replaces the inline badge previously in `PageInfo`. It reads `isNew` and `isPublished` from the `useGetLayoutByIdQuery` result and combines them with `useGetAllPendingChangesQuery` (checked via a new `findInChanges()` utility) to pick the correct badge. A `useEffect` in `UnpublishedChanges` now triggers an immediate pending-changes refetch whenever the layout, model, or entity form fields change, eliminating the previous need for 11-second timeouts in Cypress tests. After a successful publish, `useLazyGetLayoutByIdQuery` is called to refresh `isNew`/`isPublished` for the current entity. `PublishReview` gains an `isFetching` prop that disables the publish button and shows "Please wait" while the layout is being re-fetched.

Test infrastructure was consolidated: `ApiLayoutControllerGetTest`, `ApiLayoutControllerPatchTest`, and `ApiLayoutControllerPostTest` now all extend a shared `ApiLayoutControllerTestBase`, which also provides a `filterLayoutForPost()` helper to strip the new read-only `isNew`/`isPublished` fields before posting layout data back to the server. A `testStatusFlags()` kernel test covers the three `isNew`/`isPublished` combinations.
