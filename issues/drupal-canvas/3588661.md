## Summary

The `canvas build` and `canvas push` commands now use a shared Vite-based build pipeline. This unlocks Vite-resolved import patterns that were not previously available in local Code Component builds: static asset imports (images, video, audio, fonts), SVG files imported as URLs, and SVG files imported as React components using the `?react` suffix. Three additional packages (`@drupal-api-client/json-api-client`, `drupal-jsonapi-params`, and `swr`) are now listed as built-in bundled packages available to Code Components. A new `canvas pull --include-brand-kit` and `canvas push --include-brand-kit` flow lets you manage Brand Kit fonts from a local `canvas.brand-kit.json` file alongside Code Components. The documentation for Code Components has been substantially reorganized, with new pages covering local codebase setup, imports and assets, Brand Kit, and component metadata.

## Impact

Affects module maintainers and site owners.

- **Config change:** `componentDir` default changed from `process.cwd()` to `src/components`; must now be inside `aliasBaseDir`
- **New API:** New `buildCanvasProject()` function in `packages/cli/utils/build-project` unifies CLI build and push pipelines

## Technical details

The core change replaces the multi-step build pipeline in `packages/cli` with a single `buildCanvasProject()` call imported from `utils/build-project`. Previously, `canvas build` called `analyzeAndBundleImports`, `buildComponent` (per component), `buildTailwindForComponents`, and `generateManifest` as discrete steps. These are now unified under `buildCanvasProject()`, which returns a structured `BuildResult` carrying `componentResults`, `tailwindResult`, `vendorImportCount`, and `localImportCount`. Vite owns the browser graph, artifact output, vendor chunk splitting, and asset manifest entries.

`canvas push` now calls `pushBuiltComponents` instead of `buildAndPushComponents`. The push path no longer rebuilds components from source; it consumes the pre-assembled component payloads from the build result. Dependency ordering during upload is handled by inspecting `importedJsComponents` on each payload. Duplicate asset files referenced by multiple manifest entries are now uploaded once and the same URI is reused for each entry.

On the configuration side, `componentDir` defaults changed from `process.cwd()` to `"src/components"` for both the Workbench and the CLI `canvas.config.json` schema. The documentation also notes that `componentDir` must now be inside `aliasBaseDir`. Projects that relied on the old `process.cwd()` default or placed components outside `aliasBaseDir` need to add an explicit `componentDir` entry to `canvas.config.json`.

In `CanvasBuilder.php`, three entries were appended to the built-in packages list: `json_api_client` (`@drupal-api-client/json-api-client`), `drupal_jsonapi_params`, and `swr`. A pre-existing malformed code fence in the `FormattedText` example (missing opening backtick) was also corrected. The packages documentation file hash in `CanvasPackagesDocsFileHashTest` was updated to match the new content.

## Upgrade

If your project uses a `canvas.config.json` that does not explicitly set `componentDir`, the default has changed from the project root (`process.cwd()`) to `src/components`. Add an explicit entry if your components live elsewhere:

```json
{
  "componentDir": "./components"
}
```

The `componentDir` path must be inside `aliasBaseDir` (default `src`). If your component directory is outside `aliasBaseDir`, move it or adjust both settings:

```json
{
  "aliasBaseDir": ".",
  "componentDir": "./components"
}
```

To use static asset or SVG React imports, import them directly from the local codebase and push with the CLI. For TypeScript projects using `?react` SVG imports, add the SVGR client types to a Vite environment file:

```ts
/// <reference types="vite/client" />
/// <reference types="vite-plugin-svgr/client" />
```

To opt in to Brand Kit font sync:

```bash
npx canvas pull --include-brand-kit
npx canvas push --include-brand-kit
```
