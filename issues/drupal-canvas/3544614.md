## Summary

When using the Canvas AI page builder agent to place components that have required image props, the agent would repeatedly fail validation and eventually report "The job is not solvable." This happened because the AI was supplying image data in the correct client format (an object with `src`, `alt`, `width`, `height`), but the validator was feeding those raw props directly to the server-side typed data layer without conversion, triggering a `LengthException` that caused the image value to be dropped and the required-field check to fail. The fix converts AI-supplied props through the proper client-to-server transformation before validation, and updates the agent prompt to instruct the AI to use each image prop's resolved default value verbatim rather than inventing image URLs.

## Impact

No upgrade or configuration changes required.

## Technical details

The core problem was in `AiResponseValidator::convertToComponentTreeData`, which set the `inputs` array on a component tree item directly from the raw `componentData['props']` YAML. For most prop types the client and server representations match, but for image props the component source applies a transformation via `ComponentSourceInterface::clientModelToInput`. Without that conversion, `TypedData::validate` threw a `LengthException` ("Field item list length is a lie"), the image value was unset by `validateComponentInput`, and the required-field constraint fired.

The fix loads the component entity, calls `$component->normalizeForClientSide()->values` to obtain the `propSources` map, builds a `$clientModel` with `source` and `resolved` keys, then runs `$source->clientModelToInput()` to produce properly converted inputs. These are used as the `inputs` array in place of the raw props.

A companion fix in `CanvasAiPageBuilderHelper::processSdc` changes how the `default` value for each prop is reported to the agent. Previously it read from the raw SDC YAML (`$prop_details['default']` or `$prop_details['examples'][0]`), which for image props gives an unresolved relative path. Now it first checks `$client_normalized["propSources"][$prop_name]["default_values"]["resolved"]`, which contains the fully resolved path that `clientModelToInput` will accept without error.

The agent's system prompt (`canvas_page_builder_agent.ai_agent.yml`) was also updated: image handling changed from "NEVER include `src` or `alt` props" to a requirement that the AI copy the prop's exact default object (all keys and values) into its YAML output. The `entity_type` guard was moved from a freeform preamble block into a numbered Step 1 in the agent's sequential workflow, and the `entity_type` context value is now injected directly into the prompt string rather than being fetched via a separate `get_entity_context` information tool (which was removed).
