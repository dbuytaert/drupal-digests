## Summary

When a Canvas component developer removes a prop or slot from a component definition, pages that still use the old component version are now automatically upgraded when a content author opens them in Canvas. Orphaned prop values are deleted from the component instance, and child component instances placed in removed slots are removed from the component tree. If a prop or slot is removed and then re-added before the page is edited, component instances that were not edited in the meantime retain their old data and resume working normally. Bulk upgrading all existing instances is not part of this change; cleanup happens only when a page is opened for editing.

## Impact

No upgrade or configuration changes required.

## Technical details

Previously, `GeneratedFieldExplicitInputUxComponentInstanceUpdater::canUpdate()` classified removing props and removing slots as unsafe changes, returning `FALSE` and blocking automatic version upgrades entirely. This issue reclassifies both as safe changes.

The `update()` method now performs cleanup before bumping the stored `component_version`. For removed props, it calls `array_diff_key()` between the from-version and to-version `prop_field_definitions` arrays and calls `$component_instance->setInput()` with the orphaned keys stripped. For removed slots, it collects removed slot names via `array_diff_key()` on the slot definitions, then calls `ComponentTreeItemList::filter()` to remove any `ComponentTreeItem` whose `slot` matches a removed slot name and whose `parentUuid` matches the component instance being upgraded. A private `getPropDefinitions()` helper was extracted to avoid loading version settings twice across `canUpdate()` and `update()`.

`ComponentSourceManager::updateComponentInstances()` now returns a `bool` instead of `void`. `ApiLayoutController::buildRegion()` uses that return value: when the component tree is modified during a GET request (because auto-upgrade removed orphaned children), it calls `$this->autoSaveManager->saveEntity($entity)` immediately, so the cleaned-up tree is stored as an auto-save. Without this, a subsequent PATCH from the editor would load the published (unmodified) tree and re-introduce the orphaned children. A `@todo` in the second commit ([#3498525](https://www.drupal.org/node/3498525)) notes that the auto-save path currently handles only `ComponentTreeEntityInterface` entities and will need generalizing to all `FieldableEntityInterface` types. This builds on the auto-upgrade foundation laid in [#3567413](https://www.drupal.org/node/3567413).

### Collaboration

wim leers raised a significant concern: deleting children in removed slots could silently destroy carefully crafted content, and proposed an alternative "rescue" mechanism that would move orphaned children into a visible sibling container so the author could decide what to keep. lauriii confirmed the simpler deletion approach was correct and intentional, drawing explicit parallels to how Drupal's Layout Builder and paragraph fields already handle orphaned content: stop rendering it, then clean it up, without attempting to rescue it. He also noted that because upgrades only happen on edit there is already a recovery window if a developer removes a slot by mistake. wim leers accepted the decision. A later thread surfaced a UX concern from lauriii that the auto-save created during the GET request would make the page appear "changed" to the author unexpectedly; penyaskito created follow-up issues ([#3574253](https://www.drupal.org/node/3574253) and [#3574257](https://www.drupal.org/node/3574257)) to add a toast notification informing editors when their component tree was auto-updated.
