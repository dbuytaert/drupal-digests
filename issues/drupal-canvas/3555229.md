## Summary

Canvas now uses a clearer name for a key internal concept. The class that retrieves data from entity fields is now called `EntityFieldPropSource` instead of `DynamicPropSource`. The old "dynamic" prefix was vague and made the codebase harder to understand. The new name immediately tells developers that this prop source pulls values from fields on entities. This change improves Canvas maintainability and makes onboarding new contributors easier. All existing Canvas sites will automatically upgrade through a combination of an update hook and just-in-time conversion when loading old data.

## Impact

Affects module maintainers.

- **API change:** `DynamicPropSource` renamed to `EntityFieldPropSource`

## Technical details

The `DynamicPropSource` class has been renamed to `EntityFieldPropSource`, along with updates to all references throughout Canvas. This includes configuration schema, documentation, test fixtures, API responses, and internal constants. The `PropSource` enum now uses `EntityField` instead of `Dynamic` as its primary case value.

For backward compatibility, the system handles the legacy `dynamic` prefix in three ways. First, `PropSource::parse()` detects the old prefix and triggers a deprecation warning pointing to the change record. Second, the `canvas_post_update_0013` update hook loads and re-saves all `ContentTemplate` config entities, which automatically converts any stored `dynamic` references to `entity-field`. Third, just-in-time conversion continues to work for any exported config in recipes or default content that still uses the old prefix.

The `PropSource::Dynamic` enum case remains temporarily to support this backward compatibility, but is marked for removal before Canvas 2.0. All config schema has been updated to expect `entity-field` while still permitting (but deprecating) `dynamic` during the transition period. The `ComponentTreeMeetsRequirementsConstraint` validator correctly handles both prefixes as equivalent.

## Upgrade

Site owners using Canvas 1.x will see a deprecation notice if any content templates still use the old `dynamic` prop source type. Run database updates to automatically convert all content templates to use `entity-field`. After the update completes, re-export any Canvas content templates to update your configuration files and recipes. The deprecation will be removed in Canvas 2.0.

## Collaboration

This issue originated in October 2024 as part of a larger cleanup effort to improve Canvas naming consistency. The discussion highlighted the tension between perfect naming and backward compatibility. The team ultimately chose a hybrid approach: an update path for stored config plus just-in-time conversion for exported config. This balances clean migration for existing sites with minimal disruption to recipes and site templates. The collaboration involved four contributors over three months, with careful attention to testing both the update path and recipe scenarios. The final implementation includes comprehensive test coverage for both upgrade paths and ensures no sites are left with unconverted data.

## Lessons learned

- When renaming a plugin type that affects stored configuration, implement both an update path to migrate existing data and a just-in-time transformation for externally-managed configuration like recipes. This ensures both installed sites and newly-applied exported configuration continue to work. *(via wim leers, phenaproxima, penyaskito)*
- When under time pressure, shipping with pragmatic test coverage that enables rapid iteration is better than missing deadlines or shipping without tests. Refactoring painful tests later with high confidence beats writing tests from scratch with low confidence. *(via wim leers)*
