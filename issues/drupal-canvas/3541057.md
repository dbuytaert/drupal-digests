## Summary

Content templates in the Canvas page builder can have component props linked to entity fields using `DynamicPropSource`. Previously those field values were not resolved or rendered when the editor fetched or updated the layout via the API, so the preview always showed stale or empty values. This change makes all three layout API operations (GET, PATCH, POST) pass the `$preview_entity` through to prop resolution, so the editor preview correctly reflects actual field values (for example, a heading wired to a node's title field now shows the real title in the Canvas editor).

## Impact

Affects module maintainers.

- **New API:** ContentTemplate::build() now accepts a second `bool $isPreview` parameter (default FALSE); callers that previously relied on the single-argument signature continue to work without changes.

## Technical details

The core problem was that `ApiLayoutController::get()`, `patch()`, and `post()` all received a `$preview_entity` parameter but never forwarded it to the code that resolves component inputs. The fix threads `$preview_entity` through a call chain: `buildRegion()` → `ComponentTreeItemList::getClientSideRepresentation()` → `buildLayoutAndModel()` → `ComponentSourceInterface::getExplicitInput()`. All implementations of `getExplicitInput()` (which is `@internal`) gained the optional `?FieldableEntityInterface $host_entity` parameter. In `GeneratedFieldExplicitInputUxComponentSourceBase::getExplicitInput()`, the logic that detects a "dangling" component tree (one belonging to a config entity like `ContentTemplate`, where `$item->getRoot() === $item->getParent()`) now substitutes `$host_entity` in place of the item's own entity, enabling `DynamicPropSource` expressions to evaluate against the correct fieldable entity.

`ContentTemplate::build()` gained a `bool $isPreview` parameter so the preview flag is correctly propagated through `toRenderable()`. `buildPreviewRenderable()` in `ApiLayoutController` was simplified to take only `$entity` and `$preview_entity`, using `$entity->build($preview_entity, isPreview: TRUE)` for `ContentTemplate`s and the existing `componentTreeLoader` path for fieldable entities.

The PATCH path was substantially refactored: instead of loading the last stored JSON blob and mutating it by UUID, it now uses `getAutoSavedVersionIfAvailable()` to load the auto-saved entity, `getEntityWithComponentInstance()` to find which entity owns the targeted component instance (the edited entity or a `PageRegion`), and `updateComponentInstance()` to apply the client model directly to the entity's component tree. This also removed the old recursive `componentInstanceExistInComponentData()` search and the `getLastStoredData()` / `getRegionForComponentInstance()` methods. A related bug was fixed where `buildLayoutAndModel()` failed to pass `$host_entity` to recursive calls for slot children, causing nested components with dynamic props to fail.

The POST path was similarly aligned with PATCH: it now validates required body keys with `BadRequestHttpException` rather than assertions, extracts entity mutation into the new `updateEntity()` method, and uses `getAutoSavedVersionIfAvailable()` for both the entity and page regions. A condition in `GeneratedFieldExplicitInputUxComponentSourceBase::clientModelToInput()` was narrowed to restore the omitted `value` field only for `StaticPropSource` prefixes, preventing incorrect restoration during PATCH for dynamic props.
