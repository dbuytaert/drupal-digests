## Summary

Canvas code components can now declare a first-class `content-entity-reference` prop type. Instead of using a bare `type: object` and relying on implicit inference from `dataDependencies.entityFields`, component developers use `$ref: json-schema-definitions://canvas.module/content-entity-reference` to explicitly signal that a prop receives a content entity. At render time, Canvas resolves the referenced entity, evaluates all declared entity field expressions against it, and delivers a keyed payload to the component. The concrete entity type and bundle are derived at runtime from `dataDependencies.entityFields` (the single source of truth) and are never duplicated in the persisted prop schema.

## Impact

Affects module maintainers.

- **New API:** New prop shape `json-schema-definitions://canvas.module/content-entity-reference`; new public methods `JavaScriptComponent::getContentEntityReferenceProps()` and `::getEntityFieldExpressions()`; new `conditions` property on `SequenceKeysMustMatchConstraint`; new `BetterEntityDataDefinition::getEntityType()` method.

## Technical details

The new shape `json-schema-definitions://canvas.module/content-entity-reference` is added to `schema.json` and registered in `canvas.json_schema.yml` alongside `image` and `video`. The `JsonSchemaObjectRef` enum gains a `ContentEntityReference` case.

`JavaScriptComponent::toSdcDefinition()` now projects `x-allowed-entity-type-id` and, when the target entity type has bundles, `x-allowed-bundle` from `dataDependencies.entityFields` into the returned SDC definition array. The stored config entity is never mutated; the projection is local and idempotent. Two new public methods support this: `::getContentEntityReferenceProps()` returns the subset of props that carry this shape (matched by normalized schema, not raw definition), and `::getEntityFieldExpressions()` returns the declared field expression strings for a given prop.

At render time, `JsComponent::getExplicitInput()` iterates populated content-entity-reference props, reads the already-resolved `EvaluationResult` from the parent call, and evaluates each entity field expression against the resolved `FieldableEntityInterface`. Dangling references, empty fields, and unresolvable sources are silently skipped. Developer-facing payload keys are computed by `JsComponent::generateKeyForExpression()`: entity key names (e.g., `label`) take precedence over raw field names; reference traversal segments are joined with `$` (valid in JS/TS identifiers but not in Drupal field machine names, preventing collisions).

`ContentEntityReferenceObjectConstraint` is registered as the JSON Schema `object` constraint in `CanvasServiceProvider`. It validates that `x-allowed-entity-type-id` resolves to a known content entity type and that `x-allowed-bundle` is present if and only if the entity type has bundles. The same logic is exposed as `::validateMetadataSchema()` for use at metadata-derivation time by `ComponentMetadataRequirementsChecker`, which also enforces that content-entity-reference props are never `required` and never carry `examples`. The ad-hoc "entity-ref props cannot be required" check previously in `JsComponentHasValidAndSupportedSdcMetadataConstraintValidator` is removed, fully subsumed by this path.

`SequenceKeysMustMatchConstraint` gains a `conditions` property. In `canvas.schema.yml`, this is used to narrow the `entityFields` key-subset check so it only matches props that carry the `content-entity-reference` `$ref`, closing a `@todo` left by [#3585298](https://www.drupal.org/node/3585298).

`EntityFieldPropSourceMatcher::matchContentEntityReferenceShape()` (private) matches host-entity reference fields to a content-entity-reference prop by comparing the field's referenceable entity type and bundle against `x-allowed-entity-type-id` / `x-allowed-bundle` from the projected schema.

`JsonSchemaType::computeStorablePropShape()` now handles this shape, returning a `StorablePropShape` with field type `entity_reference`, widget `entity_reference_autocomplete`, and `target_type` storage setting derived from `x-allowed-entity-type-id`; `target_bundles` is added to the instance settings when `x-allowed-bundle` is present.

## Upgrade

Code component developers declare a content-entity-reference prop in their component's `props` definition:

```yaml
article:
  title: Interesting article
  type: object
  $ref: json-schema-definitions://canvas.module/content-entity-reference
```

Rules that apply to this prop type:
- The prop must **not** be listed in `required`; referenced entities may be deleted or absent.
- Do **not** add `x-allowed-entity-type-id` or `x-allowed-bundle` to the persisted prop definition; `JavaScriptComponent::toSdcDefinition()` injects these from `dataDependencies.entityFields` at runtime.
- Do **not** add `examples`; the entity value is resolved at runtime.

The prop receives a keyed object payload at render time. Keys follow entity key names where available (e.g., `label`) or field machine names otherwise. Reference traversal uses `$` as a separator (e.g., `image$url`).

Callers integrating with `JavaScriptComponent` can now use:
- `JavaScriptComponent::getContentEntityReferenceProps(): array` — returns the prop name-to-definition map for entity-reference props.
- `JavaScriptComponent::getEntityFieldExpressions(string $prop_name): list<string>` — returns the entity field expression strings for a given entity-reference prop.

The new optional `conditions` property on `SequenceKeysMustMatchConstraint` filters which elements of the target sequence are considered when checking key membership:

```yaml
SequenceKeysMustMatch:
  propertyPathToSequence: props
  conditions:
    type: object
    "$ref": "json-schema-definitions://canvas.module/content-entity-reference"
```
