## Summary

Canvas code components can now declare a `content-entity-reference` prop, allowing a component to receive a resolved Drupal content entity object at render time. Previously, entity-reference semantics were only implicit (inferred from `dataDependencies.entityFields` without a formal prop type), making props indistinguishable from plain `type: object` props and blocking tooling, validation, and proper shape routing. With this change a component author declares a prop using `$ref: json-schema-definitions://canvas.module/content-entity-reference`, selects which entity type and which fields to expose via `dataDependencies.entityFields`, and the component receives a structured payload of those field values keyed by developer-friendly names. Cardinality-1 entity references are supported; multi-cardinality is a tracked follow-up.

## Impact

Affects module maintainers.

- **New API:** New public methods `JavaScriptComponent::getContentEntityReferenceProps()`, `::getEntityFieldExpressions()`, `BetterEntityDataDefinition::getEntityType()`, `JsComponent::generateKeyForExpression()`; new optional `conditions` on `SequenceKeysMustMatchConstraint`; new `content-entity-reference` well-known prop shape.

## Technical details

The new well-known prop shape `json-schema-definitions://canvas.module/content-entity-reference` is added to `schema.json` and registered in `canvas.json_schema.yml` alongside `image` and `video`. It is represented as the new `JsonSchemaObjectRef::ContentEntityReference` enum case (`@internal`).

The developer-facing prop definition deliberately omits `x-allowed-entity-type-id` and `x-allowed-bundle`. Those keys are derived at runtime: `JavaScriptComponent::toSdcDefinition()` projects them into the returned SDC definition array from `dataDependencies.entityFields` (the single source of truth), leaving the stored config entity untouched. The new public methods `JavaScriptComponent::getContentEntityReferenceProps()` and `::getEntityFieldExpressions()` expose the entity-reference subset of props and their field expression lists respectively.

`ComponentMetadataRequirementsChecker` is extended to validate content-entity-reference props: they must not be required (the referenced entity may be absent or deleted), `x-allowed-entity-type-id` must resolve to a known content entity type, `x-allowed-bundle` is required if and only if the target entity type has bundles, and `examples` are forbidden. The new `ContentEntityReferenceObjectConstraint` (registered as the JSON Schema `object` constraint via `CanvasServiceProvider`) enforces the same rules at prop-value validation time, with matching error messages.

`SequenceKeysMustMatchConstraint` gains an optional `conditions` map. The `entityFields` config schema constraint now uses it to require that every `entityFields` key corresponds to a `content-entity-reference`-shaped prop (and vice versa), closing the `@todo` left by [#3585298](https://www.drupal.org/node/3585298).

`JsonSchemaType::computeStorablePropShape()` returns a `StorablePropShape` for entity-reference shapes: field type `entity_reference`, widget `entity_reference_autocomplete`, with `target_type` in storage settings and optionally `target_bundles` via the `default` handler. `EntityFieldPropSourceMatcher::matchContentEntityReferenceShape()` matches host-entity reference fields whose target entity type and bundle satisfy the shape.

At render time, `JsComponent::getExplicitInput()` reuses the parent-resolved `EvaluationResult` for each populated entity-reference prop, evaluates every declared `entityFields` expression on the referenced `FieldableEntityInterface`, and assembles a payload keyed by `JsComponent::generateKeyForExpression()`. That method converts field names to entity keys where applicable (e.g. `title` → `label`) and uses `$` as the separator for reference sub-expressions, since `$` is not valid in a Drupal field machine name but is a valid JavaScript identifier character. Dangling references, empty host fields, or non-`FieldableEntityInterface` values are silently skipped. The ad-hoc "entity-ref props cannot be required" check previously in `JsComponentHasValidAndSupportedSdcMetadataConstraintValidator` is removed, subsumed by the new `ComponentMetadataRequirementsChecker` branch. `BetterEntityDataDefinition::getEntityType()` is added as a new public method to eagerly probe entity-type resolution.

## Upgrade

No changes are required to existing code. The new prop type is additive only; no config update hook is needed.

To declare a content-entity-reference prop in a code component:

```yaml
# In dataDependencies.entityFields (single source of truth for target type + bundle):
dataDependencies:
  entityFields:
    article:
      - 'entity:node:article/title'
      - 'entity:node:article/field_image'

# In props (omit x-allowed-entity-type-id / x-allowed-bundle here;
# they are projected by toSdcDefinition() at runtime):
props:
  type: object
  properties:
    article:
      title: 'Featured article'
      type: object
      $ref: json-schema-definitions://canvas.module/content-entity-reference
```

The component receives a payload such as `{ label: '...', field_image: ... }` for the `article` prop at render time.
