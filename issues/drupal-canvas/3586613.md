## Summary

The Canvas module now supports a `content-entity-reference` prop type for JavaScript (code) components. A component author can declare a prop with `$ref: json-schema-definitions://canvas.module/content-entity-reference`, signalling that Canvas will inject a Drupal content entity object at render time. The referenced entity's field data (title, body, image, and so on) is specified through `dataDependencies.entityFields` and delivered to the component as a keyed payload. This enables patterns such as an "Article Card" component that automatically reflects an entity's current field values without manual copying. At runtime the concrete target entity type and bundle are projected into the SDC definition from `dataDependencies.entityFields`; the stored config entity is never mutated. Only single-cardinality entity references are supported in this release; multi-cardinality is tracked in a follow-up.

## Impact

Affects module maintainers.

- **API change:** Removed the ad-hoc 'entity-ref props cannot be required' check from `JsComponentHasValidAndSupportedSdcMetadataConstraintValidator`; that rule is now enforced by `ComponentMetadataRequirementsChecker` for the content-entity-reference shape specifically.
- **New API:** New `JavaScriptComponent::getContentEntityReferenceProps()` and `::getEntityFieldExpressions()` public methods; new `BetterEntityDataDefinition::getEntityType()` public method; new optional `conditions` property on `SequenceKeysMustMatchConstraint`; new `content-entity-reference` well-known prop shape `json-schema-definitions://canvas.module/content-entity-reference`.

## Technical details

The `content-entity-reference` definition is added to `schema.json` and listed as an allowed `$ref` choice in `canvas.json_schema.yml`, alongside the existing `image` and `video` shapes. The `@internal` `JsonSchemaObjectRef` enum gains a `ContentEntityReference` case and an `isContentEntityReference()` helper that normalizes the prop schema before comparing.

`JavaScriptComponent` receives two new public methods: `getContentEntityReferenceProps()` returns the subset of a component's props that carry this shape (compared by normalized shape, not raw definition); `getEntityFieldExpressions(string $prop_name)` returns the declared field-expression strings from `dataDependencies.entityFields` for a given prop. `toSdcDefinition()` is extended to project `x-allowed-entity-type-id` and, when the target entity type has bundles, `x-allowed-bundle` into the returned array from the single source of truth in `dataDependencies.entityFields`; the stored config entity is never touched, so calls are idempotent. `JsComponent::getExplicitInput()` is overridden to evaluate each declared entity-field expression against the resolved referenced entity and assemble a payload. Keys are computed by `generateKeyForExpression()`, which maps field machine names to entity keys (e.g. `title` → `label`) and joins reference-chain segments with `$` — valid in JS identifiers but not in Drupal field machine names, preventing collisions.

`ContentEntityReferenceObjectConstraint` (new) is registered as the JSON Schema `object` constraint. It validates `x-allowed-entity-type-id` (must be a known content entity type) and `x-allowed-bundle` (required iff the entity type has bundles, invalid if provided for a bundle-less type) both at metadata-time via the static `validateMetadataSchema()` method and at prop-value-time. `ComponentMetadataRequirementsChecker` is extended to enforce shape-specific rules: content-entity-reference props must not be `required` and must not carry `examples` (the value is resolved at runtime).

`SequenceKeysMustMatchConstraint` gains an optional `conditions` property — a map of key-value pairs that filter which elements in the target sequence are compared before key extraction. This is used in `canvas.schema.yml` to narrow the `entityFields` key-subset check so it only considers props carrying the `content-entity-reference` shape, closing the `@todo` from [#3585298](https://www.drupal.org/node/3585298).

`EntityFieldPropSourceMatcher` gains the private helper `matchContentEntityReferenceShape()`, which matches host-entity reference fields whose target type and bundle satisfy the shape's `x-allowed-entity-type-id` / `x-allowed-bundle` constraints, returning `FieldPropExpression` instances with `propName: 'entity'`. `BetterEntityDataDefinition::getEntityType()` is added as a new public method used by the projection and matching code. The ad-hoc "entity-ref props cannot be required" check previously in `JsComponentHasValidAndSupportedSdcMetadataConstraintValidator` is removed, fully replaced by the `ComponentMetadataRequirementsChecker` branch for this shape.

## Upgrade

To declare a `content-entity-reference` prop in a JavaScript component:

```yaml
# In your component's props definition (developer-facing; no x-allowed-* keys here).
# The concrete entity type and bundle come from dataDependencies.entityFields.
promoted_article:
  title: 'Promoted article'
  type: object
  $ref: json-schema-definitions://canvas.module/content-entity-reference
```

The prop must **not** be marked `required` and must **not** have an `examples` key.

The target entity type and bundle are declared once in `dataDependencies.entityFields`, for example:

```yaml
dataDependencies:
  entityFields:
    promoted_article:
      - 'entity:node:article/title'
      - 'entity:node:article/field_image'
```

At render time the component receives the prop as an object keyed by the terminal field/entity-key names, for example `{ label: '...', field_image: ... }`. Reference-chain expressions use `$` as the segment separator (e.g. `field_image$url`).

If your code calls `JavaScriptComponent::getEntityReferenceProps()` (the name used in an earlier draft), rename it to `JavaScriptComponent::getContentEntityReferenceProps()`.
