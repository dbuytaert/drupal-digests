## Summary

Canvas code components can now declare props that receive a content entity object by using the new `json-schema-definitions://canvas.module/content-entity-reference` well-known prop shape, alongside the existing `image` and `video` shapes. Previously, entity-reference semantics were purely implicit, inferred from `dataDependencies.entityFields` with no declared shape. With this change a component prop marked `$ref: json-schema-definitions://canvas.module/content-entity-reference` is recognized as an entity-reference prop at every layer: config validation, form rendering, shape matching, and runtime data injection. At render time Canvas evaluates the entity field expressions configured in `dataDependencies.entityFields` against the referenced entity and passes the results to the component as a keyed object. Only single-cardinality entity-reference props are supported; multi-cardinality support is tracked separately.

Affects module maintainers.

## Technical details

The new shape is defined in `schema.json` as the `content-entity-reference` definition and registered as a valid `$ref` choice in `canvas.json_schema.yml`. A new `JsonSchemaObjectRef::ContentEntityReference` enum case carries the URI. The developer-facing prop definition deliberately omits `x-allowed-entity-type-id` and `x-allowed-bundle`; those constraints are projected at runtime by `JavaScriptComponent::toSdcDefinition()` from `dataDependencies.entityFields` (the single source of truth), leaving the persisted config entity unchanged.

`ComponentMetadataRequirementsChecker` gains shape-specific validation: entity-reference props must be optional (the referenced entity may be deleted or the host field empty), must carry `x-allowed-entity-type-id` resolving to a known content entity type, must have `x-allowed-bundle` iff the target entity type has bundles, and must not declare `examples` (the value is resolved at runtime). A new `ContentEntityReferenceObjectConstraint` — registered as the JSON Schema `object` constraint class via `CanvasServiceProvider::alter()` — performs the same checks at prop-value validation time.

`JsComponent::getExplicitInput()` is overridden to resolve entity-reference props: it reuses the already-evaluated `EvaluationResult` for each populated prop, calls `Evaluator::evaluate()` on the referenced entity for every declared `entityFields` expression, and assembles a payload keyed by `JsComponent::generateKeyForExpression()`. That helper maps field machine names to entity key names (e.g. `title` → `label`) and uses `$` as the separator for nested reference expressions, producing identifiers valid in both PHP and JavaScript. Dangling references and unresolvable sources are silently skipped.

`EntityFieldPropSourceMatcher` gains `matchContentEntityReferenceShape()`, which returns `FieldPropExpression` instances for host-entity reference fields whose target type and bundle satisfy the schema constraints. `SequenceKeysMustMatchConstraint` receives a new optional `conditions` property — a map of `key => primitive-value` pairs — used to narrow the `entityFields` key check to only `content-entity-reference`-shaped props, closing a `@todo` left by [#3585298](https://www.drupal.org/node/3585298). `PropShape::normalizePropSchema()` is hardened to tolerate unrecognized `type` values without throwing, using `strtolower + array cast` instead of `JsonSchemaType::from()`.

## Upgrade

To declare a content-entity-reference prop in a code component, add a prop using the new `$ref`:

```yaml
# component.component.yml
props:
  type: object
  properties:
    # Bundle-less entity type (e.g. user):
    contributor:
      title: Contributor
      type: object
      $ref: json-schema-definitions://canvas.module/content-entity-reference

    # Bundled entity type — x-allowed-entity-type-id and x-allowed-bundle are
    # injected by Canvas at runtime from dataDependencies.entityFields; do NOT
    # write them in the developer-facing prop definition for code components.
```

The `x-allowed-entity-type-id` and `x-allowed-bundle` keys in the projected SDC definition are derived from `dataDependencies.entityFields` and must not be duplicated in the persisted prop schema. Entity-reference props must be optional (not listed under `required`) and must not declare `examples`.
