# #3591700: Support `content-entity-reference` props in CLI and Workbench

- **Project:** Drupal Canvas
- **Type:** Normal feature request
- **Version:** 1.x-dev
- **Status:** Fixed
- **Discussion:** [#3591700](https://git.drupalcode.org/project/canvas/-/work_items/3591700) · 1 contributor · 4 comments

## Summary

The Canvas module now fully supports `content-entity-reference` (CER) props throughout the developer workflow. Before this change, components that referenced a Drupal content entity via a CER prop could not be previewed reliably in Workbench, pulled content could lose entity reference data, and AI coding agents lacked enough context to work with these props. After this change: the Workbench sidebar shows a live entity picker for each CER prop so developers can select the preview entity per prop; page, region, and content template previews also resolve CER props before rendering; the CLI can pull and push `dataDependencies.entityFields` from `component.yml`; a new `cer-expressions` context provider generates a browsable list of field expressions grouped by entity type and bundle; and a new `cer-preview` context provider resolves and prints the actual prop shape a component will receive at runtime, which AI agents can use to understand what the component code should consume.

## Impact

Affects module maintainers.

- **New API:** New `POST /canvas/api/v0/ui/content-entity-reference/preview/{entity_type}/{entity}` endpoint and new `ApiService` methods `previewContentEntityReferenceFields`, `listContentEntityReferenceEntityTypes`, and `fetchContentEntityReferenceFields`.

## Technical details

On the Drupal side, a new controller action `ApiUiContentEntityReferenceControllers::preview()` is registered at `POST /canvas/api/v0/ui/content-entity-reference/preview/{entity_type}/{entity}`. It accepts a JSON body with an `entityFields` map (prop names to field expression arrays), parses and coalesces those expressions via `JavaScriptComponent::parseAndCoalesceEntityFieldExpressions`, evaluates them against the loaded entity via `JsComponent::buildReferencePayload`, and returns resolved prop values. Access is guarded by the Canvas UI access check, OAuth authentication, and the entity-level `entity.view` permission.

On the CLI side, `pullAgentsContext` was refactored from a single monolithic function into a provider-based system. Three `AgentsContextProvider` implementations live under `agents-context-providers/`: `content-templates` (existing behavior, now the default provider), `content-entity-reference-expressions` (alias `cer-expressions`), and `content-entity-reference-preview` (alias `cer-preview`, not default). The `agents-context` command now requires either a provider name or `--all`; calling it without either prints help and exits. The `cer-expressions` provider calls `listContentEntityReferenceEntityTypes` and `fetchContentEntityReferenceFields` on `ApiService`, walks nested reference fields up to four levels deep via `collectNestedFieldJobs`, and writes `content-entity-reference-expressions.json`. The `cer-preview` provider loads local `component.yml` metadata, reads `dataDependencies.entityFields`, fetches a preview entity suggestion for each CER prop, and calls the new `previewContentEntityReferenceFields` endpoint to print the resolved prop shape. The `pull` task now preserves `dataDependencies.entityFields` when writing local `component.yml` files, and the `Metadata` type gains an optional `dataDependencies` field. The `validate` command is refactored to use a new `validateComponents` function that accepts the full discovery result and an optional `ApiService`; CER prop validation against the live site now happens here when credentials are available.

In the Workbench (`packages/workbench`), `App.tsx` derives the set of CER preview props with `getContentEntityReferencePropPreviews`, which reads `x-allowed-entity-type-id` and `x-allowed-bundle` from each prop's JSON Schema alongside `dataDependencies.entityFields`. A `PreviewEntityPicker` component is rendered in the sidebar for each CER prop. During default (non-mock) component preview rendering, `groupEntityFieldsByProp` collects selected entity IDs, and `fetchResolvedContentEntityReferenceFields` calls the new endpoint for each group; resolved values are merged into the spec with `applyResolvedToElementsOfType`. For page, region, and content template previews, `fetchResolvedContentEntityReferenceFieldsForSpec` walks the full spec and resolves CER props for all components in it.

## Upgrade

If you have automation that calls `npx canvas agents-context` without arguments, update it. The command now shows help and exits when no provider is specified. To reproduce the previous behavior (generating prop sources and view modes), use:

```bash
npx canvas agents-context content-templates
# or run all default providers:
npx canvas agents-context --all
```

To generate the new CER expressions file:

```bash
npx canvas agents-context cer-expressions
```

To inspect resolved CER prop values for a component:

```bash
npx canvas agents-context cer-preview article-card
```

In `component.yml`, CER props must carry both `x-allowed-entity-type-id` and `x-allowed-bundle`, and the prop must not appear in `required`. Add field expressions under `dataDependencies.entityFields.<propName>`:

```yaml
props:
  properties:
    featuredArticle:
      title: Featured article
      type: object
      $ref: json-schema-definitions://canvas.module/content-entity-reference
      x-allowed-entity-type-id: node
      x-allowed-bundle: article
dataDependencies:
  entityFields:
    featuredArticle:
      - ℹ︎␜entity:node:article␝title␞␟value
      - ℹ︎␜entity:node:article␝path␞␟alias
```
