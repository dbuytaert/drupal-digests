## Summary

When the Canvas module is enabled, viewing a node whose content type has no Canvas content template attached causes the node page to serve stale content after edits. Saving a node title change, for example, would not be reflected on the front end until a full cache clear, regardless of whether caches were partially cleared. This affects any site with Canvas installed, even when the page-builder features are not actively used for that content type. The fix is included in the 1.x branch.

## Impact

No upgrade or configuration changes required.

## Technical details

The bug lives in `ContentTemplateAwareViewBuilder`, Canvas's custom entity view builder that replaces Drupal's default node view builder. When no content template exists for a given node, the else-branch of the builder added the `content_template_list` cache tags so the page would re-render as soon as a template was created. The original code did this with `(new CacheableMetadata())->addCacheTags(...)->applyTo($defaults)`. Because `applyTo()` merges the caller's metadata *into* the render array, but a freshly constructed `CacheableMetadata` carries no existing tags, the call effectively discarded all cache metadata that Drupal's core view-builder pipeline had already placed in `$defaults` (tags like `node:1` and `node_view`). Without those tags, Drupal's cache could not invalidate the cached node page when the node was saved.

The fix replaces `new CacheableMetadata()` with `CacheableMetadata::createFromRenderArray($defaults)`, which initialises the metadata object from the existing render array state before adding the template-list tags and writing back. A new kernel test `testAbsentContentTemplateKeepsCacheMetadata()` asserts that a rendered node build carries all three expected tags: `node_view`, `node:1`, and `config:content_template_list`. The regression was introduced by [#3520487](https://www.drupal.org/node/3520487).
