## Summary

The Canvas editor's side panel layout has been reorganised. The "Add" button in the side menu is renamed to "Library" and the separate "Manage Library" button is replaced by a new dedicated "Code" button that opens a standalone Code panel. Folder-based component categorisation, previously only available inside Manage Library, is now integrated directly into the regular Library panel. Code component (JS component) previews, which previously showed only a `@todo` placeholder, now render actual preview markup including any auto-saved changes. The code-editor URL path changed from `/code-editor/code/:id` to `/code-editor/component/:id`; a legacy redirect handles the old path.

## Impact

Affects module maintainers.

- **New API:** New `folders` boolean in the Canvas controller's front-end permission object, checked against `Folder::ADMIN_PERMISSION`

## Technical details

In `JavaScriptComponent::normalizeForClientSide()`, the `preview` key previously returned a render array containing a literal `@todo` string. A new private `buildPreview()` method replaces that placeholder by instantiating the `JsComponent` `ComponentSource` plugin with the entity's `local_source_id`, calling `JsComponent::renderComponent()` with the component's example prop values, and using `AutoSaveManager::getAutoSaveEntity()` so that any auto-saved props and slots override the saved config values. Static class properties (`$componentSourceManagerForPreviews`, `$autoSaveManagerForPreviews`) lazily initialise the required services. The HTTP API response for `GET /canvas/api/v0/config/js_component` now carries `AutoSaveManager::CACHE_TAG` in its cache metadata and returns real `default_markup`, `css`, `js_header`, and `js_footer` values for each code component.

`CanvasController` now exposes a `folders` boolean to the front end alongside existing `codeComponents`, `contentTemplates`, and `publishChanges` flags, checking `Folder::ADMIN_PERMISSION` against the current user.

`CodeComponentItem` no longer issues an individual `useGetCodeComponentQuery` fetch per list item (which caused N+1 requests when opening the Code panel); it now uses the data already provided as props from the collection response. `ComponentList` drops the `{ status: false }` argument from `useGetFoldersQuery()`. The `SidebarFolder` component receives a new `id` prop. `ListItem` is widened to accept a `LibraryItem` union type (`CanvasComponent | Pattern | CodeComponentSerialized`) and handles the new `LayoutItemType.CODE` case.

The front-end router renames the code-editor path from `/code-editor/code/:codeComponentId` to `/code-editor/component/:codeComponentId` and adds a `LegacyCodeEditorRedirect` component that transparently redirects the old path. `data-testid` attributes on library panel tabs are renamed from `canvas-manage-library-*` to `canvas-library-*`. The `canvas_internal_code_components` test module and its fixture config entities are removed; the remaining tests use only `canvas_test_code_components`.

## Upgrade

If you link directly to the Canvas code editor, update URLs from:

```
/canvas/code-editor/code/{component_id}
```

to:

```
/canvas/code-editor/component/{component_id}
```

A legacy redirect is in place, but it should not be relied on permanently.

If you have Playwright or custom automation that references `canvas-manage-library-*-tab-select` or `canvas-manage-library-*-tab-content` selectors, update them to `canvas-library-*-tab-select` and `canvas-library-*-tab-content` respectively.

If you reference `openManageLibraryPanel()` from the `CanvasEditor` test helper, replace it with `openCodePanel()`.
