## Summary

The Canvas module's `auto-saves/pending` API endpoint was not including content entity cache tags (such as `node:1`) in its responses. This meant that when an entity included in the auto-save list was saved or otherwise changed, the cached response for that endpoint would not be invalidated and visitors could receive stale data. Adding `enforceIsNew(FALSE)` when reconstructing entities from auto-save storage data corrects this, so the endpoint now carries the correct cache tags and invalidates properly on entity saves.

## Impact

No upgrade or configuration changes required.

## Technical details

The bug was in `AutoSaveManager::getAllAutoSaveList()`. When building the list of auto-saved entities to return, the method called `$storage->create($entry['data'])` to reconstruct each entity from its stored data. The `create()` method marks the resulting entity as new (i.e., `isNew() === TRUE`). Drupal's content entity system only attaches entity-specific cache tags (e.g. `node:4`) to entities that are not new, because a new entity has no persistent ID. As a result, those cache tags were absent from the `CacheableJsonResponse` produced by the endpoint.

The fix chains `->enforceIsNew(FALSE)` onto the `create()` call. This tells the entity system that the reconstructed object represents an already-persisted record, which causes the standard cache tag metadata (entity type + ID tags like `node:4` and `canvas_page:2`) to be attached when the entity's cacheability is merged into the response.

The accompanying tests in `ApiAutoSaveControllerCacheabilityTest` were extended to assert that a `DynamicPageCache` HIT is converted to a MISS after a node save, and again after manual `Cache::invalidateTags()` with the entity's own cache tags. `ApiAutoSaveControllerTest` was updated to expect the previously missing entity tags in the response's cache metadata.
