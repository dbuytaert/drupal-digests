## Summary

When upgrading Canvas sites from 1.0 to 1.2, existing pages could crash with fatal errors if the stored component trees referenced component versions with different props than the current versions. This happened because Canvas tried to process inputs for props that no longer existed in the stored component version, throwing exceptions during rendering. The fix makes rendering more robust by catching these exceptions during hydration and displaying them gracefully like other component errors, preventing entire pages from crashing while still alerting developers to the underlying data issue.

## Impact

No upgrade or configuration changes required.

## Technical details

Before Canvas 1.1.0, component instances could be saved with inputs for props that didn't exist in the referenced component version. When Canvas 1.1.0 introduced validation to prevent new garbage values ([#3524401](https://www.drupal.org/node/3524401)), it didn't include an update path to clean up existing invalid data. An update path is infeasible because it would need to load potentially millions of component instances across all content entities and revisions.

The bug manifested when `GeneratedFieldExplicitInputUxComponentSourceBase::getExplicitInput()` tried to uncollapse inputs for non-existent props, throwing `OutOfRangeException`. This happened before Canvas's existing exception-handling infrastructure could catch it, because that only protected against exceptions during rendering, not during hydration.

The fix wraps hydration in a try-catch block in `ComponentTreeItemList::getHydratedValue()`. When an exception occurs during `getExplicitInput()`, it's stored in the hydrated component instance data and re-thrown during rendering. This allows the existing `RenderSafeComponentContainer` infrastructure to catch and display it gracefully.

The fix also reorders validation logic to check for garbage values before attempting to optimize inputs, since optimization would fail on garbage data anyway.

## Lessons learned

- When hydrating stored component instances, validate that stored prop values reference props that exist in the loaded component version before processing them. Component versions evolve over time, and stored instances may contain inputs for props that were added, removed, or renamed in newer versions. Filter extraneous inputs during hydration to prevent crashes, and wrap hydration in exception handling to render errors gracefully rather than breaking the entire page. *(via wim leers)*
- When fixing data validation issues, consider that an update path to clean legacy data may be infeasible at scale (e.g., processing millions of stored component instances). Instead, make the system resilient to invalid data by catching exceptions during hydration and rendering, allowing the application to handle legacy data gracefully without requiring bulk data migration. *(via wim leers)*
- When building robust rendering pipelines, distinguish between hydration exceptions (loading and transforming stored data) and rendering exceptions (executing component logic). Catch hydration exceptions before entering the rendering layer so both failure modes can be handled consistently, showing errors inline without crashing the entire response. *(via wim leers)*
