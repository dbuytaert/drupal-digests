## Summary

Experience Builder (canvas) previously validated image prop URLs by matching file extensions with a hardcoded regex that only permitted PNG, GIF, JPG, JPEG, WEBP, and AVIF. This excluded SVG and any other image format. The fix replaces that approach with the JSON Schema `contentMediaType: image/*` annotation, allowing any image MIME type. The same change applies to video props, which previously only allowed MP4. As a side effect, the system now properly maps image string props to the `image` field type rather than a plain `string` field type, which means image props in generated field-based UX components are backed by an image field with the `image_image` widget instead of a plain text field.

## Impact

Affects module maintainers.

- **API change:** JSON schema definitions `image-uri` and `stream-wrapper-image-uri` replace extension-regex `pattern` with `contentMediaType: image/*` and a simplified URL pattern; image and video prop shapes now map to field type `image` with widget `image_image` instead of `string`.

## Technical details

The old `json-schema-definitions://experience_builder.module/image-uri` and `stream-wrapper-image-uri` definitions carried `pattern` regexes that explicitly enumerated case-insensitive extensions. Those are replaced with `contentMediaType: image/*` plus a simplified URL-structure `pattern`. The video prop definition similarly moves from an MP4-only extension regex to `contentMediaType: video/*`.

On the matching side, `JsonSchemaFieldInstanceMatcher` previously called `buildCaseInsensitiveExtensionRegex()` to synthesise extension-based `Regex` constraints for file URI field properties. That method is removed. Instead, a new `fileExtensionsToTargetContentMediaType()` helper uses the `file.mime_type.guesser.extension` service to map a media type's allowed extensions to their wildcard MIME category (e.g. `["jpg","png","svg"]` → `"image/*"`), and attaches the new `UriTargetMediaTypeConstraint` instead.

`UriTargetMediaTypeConstraint` is a no-op validation constraint (modelled on Drupal core's `FullyValidatable`) whose only role is to carry MIME type semantics on a data definition, enabling the shape-matching logic in `mediaLibraryStoragePropShapeAlter` and `JsonSchemaStringFormat::computeStorablePropShape()` to recognise image and video URI props and wire them to the `image` field type with the `image_image` widget.

Supporting changes: `DataTypeShapeRequirement` and `DataTypeShapeRequirements` now implement `\IteratorAggregate`, letting `JsonSchemaType` spread either type uniformly. `StorablePropShape::fieldTypeProp` and `CandidateStorablePropShape::fieldTypeProp` now accept `ReferenceFieldTypePropExpression` in addition to `FieldTypePropExpression|FieldTypeObjectPropsExpression`. `JsonSchemaType::patternToPcre()` extracts the pattern-to-PCRE conversion that was previously inlined. The ordering of `format` vs `pattern` cases in `JsonSchemaType::computeStorablePropShape()` was corrected so that `format` takes precedence when both are present.

## Upgrade

SDC prop schemas for image URLs must replace the old extension regex with `contentMediaType` and a URL-structure pattern:

```yaml
# Before
src:
  type: string
  format: uri-reference
  pattern: '^(/|https?://)?.*\.([Pp][Nn][Gg]|[Gg][Ii][Ff]|[Jj][Pp][Gg]|[Jj][Pp][Ee][Gg]|[Ww][Ee][Bb][Pp]|[Aa][Vv][Ii][Ff])(\?.*)?(#.*)?$'

# After — or use $ref: json-schema-definitions://experience_builder.module/image-uri
src:
  type: string
  format: uri-reference
  contentMediaType: image/*
  pattern: '^(/|https?://)?(?!.*\://)[^\s]+$'
```

Stream-wrapper image URIs:

```yaml
# Before
src:
  type: string
  format: uri
  pattern: '^(?!https?://)[\w\-]+://.*\.([Pp][Nn][Gg]|...)(\?.*)?(#.*)?$'

# After — or use $ref: json-schema-definitions://experience_builder.module/stream-wrapper-image-uri
src:
  type: string
  format: uri
  contentMediaType: image/*
  pattern: '^(?!https?://)[\w\-]+://'
```

Video URL props follow the same pattern with `contentMediaType: video/*`.
