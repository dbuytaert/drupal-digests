## Summary

When building a Code Component in the Canvas theme builder, importing only utility functions from `@/lib/drupal-utils` that carry no `drupalSettings` requirements (such as `sortMenu`) caused the component save to fail with a backend validation error: "This value should not be blank." for `dataDependencies.drupalSettings`. After this fix, such components save successfully.

## Impact

No upgrade or configuration changes required.

## Technical details

The `getDataDependenciesFromAst` function (in the theme builder's JavaScript AST analysis layer) inspects a Code Component's import statements to derive `dataDependencies`. It previously unconditionally spread a `drupalSettings` key into the returned object as part of its reducer loop, even when the accumulated array remained empty because none of the imported symbols (`getSiteData`, `getPageData`, `JsonApiClient`) actually required `drupalSettings`. The empty array then reached the backend, which rejected it because config schema requires the property to be non-blank when present.

The fix extracts the dependency computation into a local variable `computedSettings` first, then uses the conditional spread `...(computedSettings.length > 0 && { drupalSettings: computedSettings })` so the `drupalSettings` key is only included in the returned `DataDependencies` object when it has at least one entry. A dedicated test case was added asserting that importing `sortMenu` from `@/lib/drupal-utils` produces an empty `{}` result with no `drupalSettings` property. The maintainer noted (#comment-16301943) that the backend (`JavaScriptComponent::updateFromClientSide()`) could also be loosened under the Robustness Principle, and opened [#3553427](https://www.drupal.org/node/3553427) to track that separately.
