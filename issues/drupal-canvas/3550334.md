## Summary

Canvas components whose props carry CKEditor-formatted HTML (schema using `contentMediaType: text/html`) were rendering escaped markup on screen instead of the intended HTML. Outputting `{{ text }}` in a component's Twig template produced visible `&lt;p&gt;` tags rather than rendered paragraphs. The natural workaround of adding a `|raw` filter introduced an XSS risk because Canvas is not the only possible rendering context. This is fixed in `1.x` as of the commits associated with this issue. Sites that created component instances before the fix was shipped need to run `drush updatedb` to rewrite affected `Component` config entities; component instances referencing older version hashes may still need manual replacement until the holistic migration path in [#3463996](https://www.drupal.org/node/3463996) is complete.

## Impact

Affects site owners.

- **Security:** HTML props using `contentMediaType: text/html` were double-escaped on output; the `|raw` workaround introduced XSS risk. Fixed by switching from the `value` to the `processed` field property.
- **Database update:** `canvas_post_update_0005_use_processed_for_text_props_in_components` rewrites stored expressions in `Component` config entities from `value` to `processed` for text field types across all stored versions.

## Technical details

The root cause was in `JsonSchemaType::computeStorablePropShape()`, which mapped `contentMediaType: text/html` props to `FieldTypePropExpression('text_long', 'value')` and `FieldTypePropExpression('text', 'value')`. The `value` property is the raw stored string; Twig sees it as an untrustworthy string and HTML-encodes it. Changing both to `'processed'` makes Canvas read the filter-applied `FilteredMarkup` object instead, which Twig recognizes as safe and renders without re-encoding.

Debugging exposed two separate bugs in Drupal core. `TextItemBase::applyDefaultValue()` has an unresolved `@todo` from at least 2013: it always sets `format` to `NULL` rather than using the first entry in the field instance's `allowed_formats` setting. `TextItemBase::generateSampleValue()` has the same blindspot, always falling back to `filter_fallback_format()`. Both meant that new field items for canvas-managed text fields silently received no format, causing them to be processed as plain text. Canvas works around both defects with `CoreBugFixTextItemBaseDefaultValueTrait` and `CoreBugFixTextItemBaseGenerateSampleValueTrait`, applied via `use` to `TextItemOverride`, `TextLongItemOverride`, and `TextWithSummaryItemOverride`. Core issues [#3556506](https://www.drupal.org/node/3556506) and [#3556508](https://www.drupal.org/node/3556508) were opened to fix the underlying problems upstream.

The update path added in the third commit introduces `canvas_post_update_0005_use_processed_for_text_props_in_components()`, which calls `CanvasConfigUpdater::updatePropFieldDefinitionsUsingTextValue()` on every `Component` config entity. That method walks all stored versions oldest-to-newest (so that a newly created active version is also caught), rewrites any `ℹ︎text_long␟value` or `ℹ︎text␟value` expressions to their `processed` counterparts, and generates a new version hash for the active version when it is changed. Past versions are updated in place (they use `type: ignore` for settings in config schema validation). A deprecation warning is triggered for any component entity that still carries the old expression at runtime, referencing change record `https://www.drupal.org/node/3556442`. `Component::preSave()` was also extended to call `updatePropFieldDefinitionsUsingTextValue()` automatically on non-syncing saves, so components updated manually or through discovery are also corrected on write. The same commit introduces `canvas_post_update_0004_*` hooks that collapse previously uncollapsed component inputs on `Pattern`, `PageRegion`, `ContentTemplate`, and `FieldConfig` entities.

## Upgrade

Run database updates after deploying this release:

```bash
drush updatedb
```

`canvas_post_update_0005_use_processed_for_text_props_in_components` rewrites all `Component` config entity versions that stored `ℹ︎text_long␟value` or `ℹ︎text␟value` expressions to use `ℹ︎text_long␟processed` or `ℹ︎text␟processed` instead. No stored field values need to change; only the expression metadata in config is updated.

Component instances in content entities (pages) that reference old version hashes are not automatically migrated by this update path. Until [#3463996](https://www.drupal.org/node/3463996) provides a holistic migration, affected content can be fixed by deleting the old component instance and creating a new one with the same prop values.
