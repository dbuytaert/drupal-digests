## Summary

Drupal Canvas now supports Tailwind CSS plugins in global stylesheets, starting with the bundled `@tailwindcss/typography` plugin. This enables teams to use advanced styling features like prose formatting for content-rich areas without needing custom build processes. The change allows Canvas sites to leverage the broader Tailwind ecosystem for professional design systems and accessible component libraries.

## Impact

No upgrade or configuration changes required.

## Technical details

The Canvas module upgraded the `tailwindcss-in-browser` library from version 0.5.0 to 0.6.0, which adds support for loading Tailwind CSS plugins via the native JavaScript `import()` function and the `@plugin` directive. The `@tailwindcss/typography` plugin is now bundled and available through the global import map.

A new `GlobalImports` service centralizes the import map configuration that was previously defined in `JsComponent::renderComponent()`. The service is injected into `CanvasController` and outputs the import map in the HTML head, making it available both for rendered components and the Canvas UI. The import map includes the new `@tailwindcss/typography` entry pointing to a bundled JavaScript module.

Plugin loading works via full URLs (e.g. `@plugin "https://esm.sh/@tailwindcss/typography@0.5.19"`) or bare specifiers (e.g. `@plugin "@tailwindcss/typography"`) resolved through the import map. The URL-based approach only works in browsers, not in local Node.js development with Canvas CLI, so the bundled approach provides cross-environment compatibility.

The `GlobalImports::getGlobalImports()` method builds the map of JavaScript module paths, and `GlobalImports::getQueryString()` handles cache-busting. This refactoring allows future expansion of the plugin system through the existing `['#attached']['import_maps']` API.

## Lessons learned

- When building abstractions that run in both browser and server environments, API compatibility differences in native functions (like import()) break cross-environment workflows. Restrict feature sets to the intersection of both runtimes, or explicitly document environment-specific limitations. *(via wotnak)*
- When a controller returns an early response (like HtmlResponse), the standard attachment hooks (page_attachments, page_attachments_alter) never fire. To modify attachments on these responses, decorate the attachments processor service instead. *(via wotnak)*
