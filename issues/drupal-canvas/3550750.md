## Summary

Canvas now supports mixing different media types that require structurally different field traversal paths within a single reference field prop expression. Before this change, a video field that needed to accept both a core `VideoFile` media type and an Acquia DAM video type could not be configured, because each requires a different chain of field properties to reach the source URL. The new bundle-specific expression branching lets a single prop expression route to the correct traversal path at evaluation time, based on the actual bundle of the referenced entity. An automatic update path converts all existing `Component` config entities to the new format without disrupting existing component instances.

## Impact

Affects module maintainers and site owners.

- **Database update:** New `canvas_post_update_0011_multi_bundle_reference_prop_expressions()` converts legacy multi-bundle `Component` config entity expressions to the new per-bundle branching format.
- **Deprecation:** Multi-bundle array overloads on `FieldPropExpression`, `FieldTypeObjectPropsExpression`, and `ReferenceFieldTypePropExpression` deprecated in canvas:1.1.0; use per-bundle `withAdditionalBranch()` on `ReferenceFieldTypePropExpression` instead.
- **New API:** New `ReferencedBundleSpecificBranches` class and `withAdditionalBranch()`, `withoutBranch()`, `hasBranch()`, `targetsMultipleBundles()`, `getTargetExpression()` on `ReferenceFieldTypePropExpression`.

## Technical details

The core problem was that `FieldPropExpression` was previously overloaded to handle multiple bundles by accepting arrays of field names and prop names keyed by bundle. This made it impossible for a `ReferenceFieldTypePropExpression` to follow references into bundles that need fundamentally different sub-expressions, not merely different field names.

The fix introduces `ReferencedBundleSpecificBranches`, which holds a map of bundle identifiers (e.g. `entity:media:image`) to their corresponding single-bundle expressions. `ReferenceFieldTypePropExpression` now accepts either a single expression or a `ReferencedBundleSpecificBranches` as its `referenced` parameter. Three new alteration helper methods are added: `withAdditionalBranch()`, `withoutBranch()`, and `hasBranch()`, intended for use in `hook_canvas_storable_prop_shape_alter()` implementations. At runtime, the updated `Evaluator` calls `getTargetExpression($referenced_entity)` when `targetsMultipleBundles()` is true, selecting the correct branch based on the actual entity bundle encountered.

The string representation of multi-branch expressions uses `[branch1][branch2]` syntax, e.g. `ℹ︎entity_reference␟entity␜[␜entity:media:image␝field_media_image␞␟src_with_alternate_widths][␜entity:media:remote_image␝field_media_oembed_image␞␟remote_image_web_uri]`. Parsing helpers (`parseBranches()`, `computeMaxBranchDepth()`) live in `CompoundExpressionTrait`. Branch serialization is validated by the existing `ValidStructuredDataPropExpression` constraint.

`ShapeMatchingHooks::mediaLibraryStorablePropShapeAlter()` is updated to emit a `ReferencedBundleSpecificBranches`-backed `ReferenceFieldTypePropExpression` for both image (object shape) and video (scalar shape) matches. The old per-bundle array overloads in `FieldTypeObjectPropsExpression`, `ReferenceFieldTypePropExpression`, and `FieldPropExpression` are deprecated in canvas:1.1.0; their evaluation, labeling, and dependency calculation logic has already been removed. Only the constructor, `fromString`, and `toString` paths remain to support the update path.

Update path `canvas_post_update_0011_multi_bundle_reference_prop_expressions()` (via `CanvasConfigUpdater::updateMultiBundleReferencePropExpressionToMultiBranch()`) walks all `Component` config entity versions from newest to oldest and rewrites any expression detected by `expressionUsesDeprecatedReference()`. For `ReferenceFieldTypePropExpression` it calls `generateBundleSpecificBranches()`; for `FieldTypeObjectPropsExpression` it calls `liftReferenceAndCreateBranchesIfNeeded()`, which both lifts the shared reference out of the object shape and converts to branches where bundles differ. The active version receives a new version hash; past versions are updated in place under `setSyncing(TRUE)`.

## Upgrade

Implementations of `hook_canvas_storable_prop_shape_alter()` that construct multi-bundle `FieldPropExpression` objects must migrate to the new per-bundle branching API. The deprecated pattern passed arrays of bundle IDs, field names, and prop names:

```php
// DEPRECATED (canvas:1.1.0) — will be removed in canvas:2.0.0
new ReferenceFieldTypePropExpression(
  new FieldTypePropExpression('entity_reference', 'entity'),
  new FieldPropExpression(
    BetterEntityDataDefinition::create('media', ['bundle_a', 'bundle_b']),
    ['bundle_a' => 'field_name_a', 'bundle_b' => 'field_name_b'],
    NULL,
    'src_with_alternate_widths',
  ),
);
```

The replacement uses `withAdditionalBranch()` on an existing `ReferenceFieldTypePropExpression`, which already carries one or more branches built from single-bundle expressions:

```php
// Check whether a branch to alter exists, remove it, and/or add a new one.
$expr = $storable_prop_shape->fieldTypeProp; // ReferenceFieldTypePropExpression
if ($expr->hasBranch('entity:media:vacation_photos')) {
  $expr = $expr->withoutBranch('entity:media:vacation_photos');
}
$expr = $expr->withAdditionalBranch(new FieldPropExpression(
  entityType: BetterEntityDataDefinition::create('media', ['remote_image']),
  fieldName:  'field_media_remote_image',
  delta:      NULL,
  propName:   'some_computed_property',
));
$storable_prop_shape->fieldTypeProp = $expr;
```

See the updated `hook_canvas_storable_prop_shape_alter()` example in `canvas.api.php` and the change record at https://www.drupal.org/node/3563451 for further detail. Existing `Component` config entities are migrated automatically by `canvas_post_update_0011_multi_bundle_reference_prop_expressions()`.
