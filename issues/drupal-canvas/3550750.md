## Summary

Drupal Canvas now allows site builders to mix different types of media in a single field when those media types need different ways to retrieve their content. For example, a video field can now allow both locally hosted videos and videos from external services like Acquia DAM, even though they store their data differently. Previously, this was impossible to configure. An automatic update path converts existing configurations to the new format without disruption.

## Impact

No upgrade or configuration changes required.

## Technical details

Canvas uses "prop expressions" to define how to retrieve values from Drupal fields. Before this change, a reference field expression could only handle multiple bundles if they all used identical field structures. This meant that an entity reference field targeting both a core `VideoFile` media type and an Acquia DAM video media type couldn't work, because they require different expression paths to retrieve the source URL.

This change introduces `ReferencedBundleSpecificBranches` to allow a single `ReferenceFieldTypePropExpression` to contain multiple "branches," each targeting a specific bundle with its own expression. For example, one branch might follow `field_media_video_file` → `entity` → `file:uri` → `url` for core videos, while another branch follows `acquia_dam_video_item` → `src` for Acquia DAM videos.

The new branching infrastructure supports both `ReferenceFieldTypePropExpression` (for field type props) and `ReferenceFieldPropExpression` (for dynamic prop sources). String representations use `[branch1][branch2]` syntax. Three new methods simplify alteration: `withAdditionalBranch()`, `withoutBranch()`, and `hasBranch()`.

The update path `canvas_post_update_0011_multi_bundle_reference_prop_expressions()` automatically converts legacy multi-bundle expressions introduced in [#3530521](https://www.drupal.org/project/experience_builder/issues/3530521) and [#3530533](https://www.drupal.org/project/experience_builder/issues/3530533) to the new branching format. All legacy evaluation and labeling code for the old format has been removed. `Evaluator` now selects the correct branch based on the actual referenced entity's bundle. Dependency calculation and labeling both handle branched expressions. The change record at [#3563451](https://www.drupal.org/node/3563451) documents the new API.

## Lessons learned

- When caching or building data structures at multiple layers, transparent automatic migrations at runtime obscure failures and make debugging harder. Prefer explicit, tested update paths that fail visibly when incomplete. *(via wimleers)*
- When building infrastructure that supports polymorphic data, avoid special-casing multiple variants in a single expression. Instead, design branch structures where each branch targets a single variant, making the system easier to maintain and avoiding combinatorial complexity. *(via wimleers)*
- When introducing better abstractions that obsolete previous approaches, remove the legacy evaluation code immediately after providing an update path. Keeping dead code paths increases maintenance burden without benefit if the update path is trustworthy. *(via wimleers, f.mazeikis)*
