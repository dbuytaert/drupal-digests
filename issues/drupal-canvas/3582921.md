## Summary

The Canvas CLI gains an interactive browser-based login command, `canvas login`, so site administrators and developers can authenticate as their own Drupal user instead of using a shared service-account secret. Running `npx canvas login --site-url https://example.com` opens a browser, prompts the user to log in to Drupal, and stores the resulting tokens locally. Subsequent `canvas push` and `canvas pull` commands use those stored tokens automatically, with no `CANVAS_CLIENT_ID` or `CANVAS_CLIENT_SECRET` environment variables required. A matching `canvas logout` command removes the stored credentials. To use this flow, a Consumer in Drupal admin must have the Authorization Code grant type enabled, PKCE checked, and `http://localhost:4444/callback` registered as a redirect URI.

No upgrade or configuration changes required.

## Technical details

The implementation adds three new files to the `packages/cli` workspace. `src/services/auth.ts` handles the full OAuth 2.0 flow: `discoverAuth()` fetches `/.well-known/oauth-protected-resource` with an `X-Canvas-CLI: 1` header to obtain the authorization server base URL and reads the `X-Consumer-ID` response header to auto-detect the client ID; if that endpoint returns 404 it falls back to Simple OAuth's conventional `/oauth/authorize` and `/oauth/token` paths on the site. It then fetches `/.well-known/openid-configuration` from the auth server to resolve the final `authorization_endpoint` and `token_endpoint`. PKCE helpers (`generateCodeVerifier`, `deriveCodeChallenge` via SHA-256, `generateState`) produce the code verifier and S256 challenge. `waitForCallback(port)` starts a temporary `http.createServer` listener on `http://localhost:PORT/callback` with a 5-minute timeout, validates the returned `state` parameter to guard against CSRF, and resolves with the authorization code. `exchangeCode()` posts the `authorization_code` grant with the code verifier to the token endpoint. `src/lib/token-store.ts` persists the access token, optional refresh token, expiry timestamp, client ID, and token endpoint to `~/.config/drupal-canvas/oauth.json` (written as a JSON file with mode `0o600`, keyed by normalized site URL) using `getTokenEntry`, `setTokenEntry`, and `removeTokenEntry`. `src/commands/login.ts` wires these together into `loginCommand` and `logoutCommand` registered with Commander. The commit diff also updates the module README to document the Consumer configuration steps (enable Authorization Code grant, check Use PKCE, uncheck Is Confidential, register the callback URI) and adds a note that scopes used with `canvas login` must have the Authorization Code grant type enabled.

## Upgrade

To use `canvas login`, configure a Consumer in Drupal admin at `/admin/config/services/consumer`:

1. Enable the **Authorization Code** grant type.
2. Check **Use PKCE?** and leave **Is Confidential?** unchecked.
3. Add `http://localhost:4444/callback` as a **Redirect URI** (use a different port if you pass `--port`).
4. Enable the **Authorization Code** grant type on each scope the CLI will request.

Then run:

```bash
npx canvas login --site-url https://example.com --client-id YOUR-CLIENT-ID
```

Tokens are stored in `~/.config/drupal-canvas/oauth.json`. Existing `client_credentials` setups are unaffected.
