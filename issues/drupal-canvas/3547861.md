## Summary

The Canvas module's Pages listing at `/admin/content/pages` now includes an "Add Page" local action button. Clicking it immediately creates a new, unpublished Canvas page with a default title and redirects the editor straight to that page's edit form, matching the quick-create workflow used elsewhere in Canvas. Before this change, there was no way to create a Canvas page directly from the Pages listing.

## Impact

Affects module maintainers.

- **New API:** New `AddPageController` service and `entity.canvas_page.add_page` route at `/admin/content/pages/add`

## Technical details

Three wiring changes land together. First, a local action definition (`entity.canvas_page.add_page`) is registered in `canvas.links.action.yml` so the "Add Page" button appears on the `entity.canvas_page.collection` route. Second, a new route `entity.canvas_page.add_page` is defined at `/admin/content/pages/add`; it requires the `_entity_create_access: canvas_page` check and a CSRF token (`_csrf_token: TRUE`), and is flagged as an admin route.

Third, the new `Drupal\canvas\Controller\AddPageController` (a final readonly class registered as a service) handles the route via `__invoke()`. It calls `ApiContentControllers::defaultTitle()` to generate the initial title, creates a `canvas_page` entity with `status: FALSE`, saves it immediately, then issues a `RedirectResponse` to `entity.canvas_page.edit_form` for the new entity. The page therefore exists in storage before the editor ever sees the edit form.

A "Pages" menu link under `system.admin_content` is also added in `canvas.links.menu.yml`, giving the collection its own tab in the Content admin area. `CanvasPageListTest` is updated to assert the local action link is present and to cover the tab navigation, and a new kernel test `AddPageControllerTest` verifies access-denied behavior without the create permission, rejection without a valid CSRF token, and a 302 redirect to the editor URL when both checks pass.
