## Summary

When SDC component authors write a `description` field for a prop in their component YAML schema, that text now appears as help text beneath the corresponding form element in the Drupal Canvas editor. Previously the description was silently ignored in the editing UI. This change makes component schemas more self-documenting for content editors, consistent with how UI Suite handles the same JSON Schema field.

## Impact

Affects module maintainers.

- **New API:** New `<canvas-description>` custom element in Twig templates maps to the `InputDescription` React component for rendering prop descriptions in the Canvas editor

## Technical details

On the PHP side, `GeneratedFieldExplicitInputUxComponentSourceBase::buildComponentInstanceForm()` now reads `$component_schema['properties'][$sdc_prop_name]['description']` and routes it through two paths. For static (non-linked) props the value is forwarded to `StaticPropSource::getWidget()` as a new optional `?string $sdc_prop_description = NULL` parameter; `getWidget()` calls `->setDescription()` on the ephemeral field definition so the standard Drupal form element `#description` key is populated. For linked props the description is placed directly on the render array as `#description` and also carried in the `prop_link_data` payload.

On the frontend, a new `InputDescription` React component (`ui/src/components/form/components/drupal/InputDescription.tsx`) centralises description rendering. It accepts a `description` (string, `{content, attributes}` object, or false), a `descriptionDisplay` placement hint, and wraps its children with the description either before or after them. `DrupalFormElement` and `InputBehaviorsCommon` were refactored to delegate to `InputDescription` instead of their own duplicated description markup. `LinkedFieldBox` was also updated to accept and render the description via `InputDescription`.

A new `<canvas-description>` custom element was registered in the Twig-to-JSX component map, pointing at `InputDescription`. Several new canvas_stark Twig templates (`datetime-wrapper.html.twig`, `field-multiple-value-form.html.twig`, `fieldset.html.twig`, `text-format-wrapper.html.twig`) emit `<canvas-description>` so that widget types rendered through these wrappers (date/time, fieldsets, multi-value fields, text format wrappers) also pick up the description correctly.

A targeted `hook_field_widget_single_element_link_default_form_alter` in `ReduxIntegratedFieldWidgetsHooks` prevents the link widget from duplicating its own internal hint text when a prop description is present: it replaces the URI sub-field's `#description` with the prop-level description (or removes it entirely when no description is set), avoiding the verbose default link widget help appearing alongside the prop description.

### Collaboration

The fix had an important detour: mglaman submitted the initial MR, but both wim leers and penyaskito independently noticed that descriptions were not actually appearing in the Canvas editor despite the unit tests passing. wim leers diagnosed the root cause in comment #22: the existing tests exercised the Drupal-side form build but not the `hyperscriptify` layer that converts the form into React. larowlan then added the missing frontend wiring and Cypress tests that actually verified descriptions render in the editor UI. A side discussion between heyyo, effulgentsia, lauriii, and d34dman about whether a single `description` field can serve developers, LLM context, and content editors simultaneously was resolved by lauriii: YAML inline comments handle developer notes, LLM guidance will live elsewhere, and the JSON Schema `description` is appropriate as editorial help text consistent with UI Suite.
