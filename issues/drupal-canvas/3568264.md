## Summary

Code components created through the Canvas UI can now define array-type properties, enabling lists of strings, numbers, booleans, or objects (like multiple images). Previously, only single-value properties were supported. This allows ambitious site builders to create more flexible components without writing PHP code, such as a card carousel with multiple images or a tag list with multiple labels. Array properties support optional maximum item limits and work with all existing property types that Canvas already supports.

## Impact

No upgrade or configuration changes required.

## Technical details

Canvas code components (JavaScriptComponent config entities) store their metadata in YAML, including prop definitions that follow JSON Schema conventions. This issue added support for `type: array` with an `items` schema defining the type of each array element. The implementation validates both single-cardinality props (matching the 5 basic JSON Schema types: boolean, integer, number, string, object) and array-cardinality props (sequences of those types).

The config schema definition uses Drupal's dynamic typing system to validate prop examples based on both the prop's `type` and its `items.type`. This required leveraging `TypedConfigManager::getDefinitionWithReplacements()` with replacement tokens like `[%parent.%parent.type]` and `[%parent.%parent.items.type]` to create 10 distinct example validation schemas (5 basic types × 2 cardinalities).

The `JsonSchemaObject` class was extended to find `$ref` values for both single-object props and array-of-objects props by traversing the parent chain differently for each case. A `maxItems` constraint validates minimum value of 2 (enforcing arrays are actually multi-value). The implementation also worked around a Drupal core cache pollution bug ([#3400181](https://www.drupal.org/node/3400181)) with a temporary `TypedConfigManagerWithCachePollutionFix` class that prevents caching dynamically-typed definitions.
