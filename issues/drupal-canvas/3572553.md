## Summary

Canvas code components can now define and persist props that accept multiple values (array-type props). When a component's props are configured in the Component Builder UI with `allow multiple values` enabled, those settings are correctly saved to the `JavaScriptComponent` config entity and reloaded when the component is revisited in the Canvas editor. This was the backend counterpart to the UI work in [#3571917](https://www.drupal.org/node/3571917) and the schema work in [#3568264](https://www.drupal.org/node/3568264).

## Impact

No upgrade or configuration changes required.

## Technical details

The `JavaScriptComponent` config entity and its `updateFromClientSide()` method now correctly store and round-trip array-type prop schemas sent by the client. A companion commit via MR !706 extended `ComponentMetadataRequirementsChecker::check()` to validate `enum` and `meta:enum` keys on array props, which had previously produced "not a supported key" errors. The schema support for the `array` type itself was handled upstream in [#3568264](https://www.drupal.org/node/3568264).

An earlier iteration of this change included a private `normalizePropsSchema()` method on `JavaScriptComponent` that silently relocated `enum`, `meta:enum`, and `x-translation-context` keys from the array level to the `items` level of each prop, and stripped empty `[[]]` examples, as a server-side workaround for incorrectly structured client payloads. wim leers identified this as technical debt directly opposed to what [#3568264](https://www.drupal.org/node/3568264) had established. MR !742 (the fix diff) removes `normalizePropsSchema()` and its call in `updateFromClientSide()` entirely, along with the corresponding `providerNormalizePropsSchema`/`testNormalizePropsSchema` test coverage, and instead adds clarifying comments in the client-side `serializeProps()` function to document the correct serialization contract for array props (non-empty string check for string-array props, truthy-or-explicit-false for scalars).
