## Summary

The Canvas editor can now correctly save and reload multi-value (array) prop configurations for code components. Previously, selected values for array-type props did not persist after adding a component and returning to its page. As part of the fix, a short-lived server-side normalization workaround that had been added to compensate for incorrectly structured client data was identified as technical debt and removed, leaving the backend clean and free of compensating logic.

## Impact

No upgrade or configuration changes required.

## Technical details

The overall feature work landed in two commits. The first (`0920a02a`) delivered the core persistence changes. The second, MR !742 (`94e27a47`), is the corrective diff provided here: it removes `JavaScriptComponent::normalizePropsSchema()` and the call to it inside `updateFromClientSide()`. That private method had been moving `enum`, `meta:enum`, and `x-translation-context` keys from the array-schema level down into the `items` subschema, and also stripping placeholder `[[]]` examples, to work around clients sending malformed prop data. Wim Leers flagged this as directly opposed to the schema work in [#3568264](https://www.drupal.org/node/3568264) (which added proper `array` type support to `canvas.json_schema.yml`) and as pure technical debt. The correct contract is for the client to submit prop schemas that are already valid, with `enum`/`meta:enum` nested under `items`.

Along with the production code removal, the associated `testNormalizePropsSchema` test method and its `providerNormalizePropsSchema` data provider are removed from `JavascriptComponentTest`, as is the now-unused `ConstraintViolationsTestTrait` use in that class. The only addition in MR !742 is inline comments in the UI serialization function `serializeProps` clarifying the `hasExample` guard logic for array versus non-array props, where string arrays require at least one non-empty value.
