# #3544213: Harden XB content dependency calculation infrastructure to not cause a PHP fatal error when content entities have disappeared whose deletion should have been prevented

## Summary

Deleting a media entity that was referenced inside a Canvas component tree could cause a PHP fatal `TypeError` when anything tried to calculate content dependencies for that component tree, such as running `drush content:export` or certain update hooks. The crash happened because Canvas's dependency calculation code did not defend against the case where a referenced entity no longer exists. This has been hardened so that missing entity references are skipped gracefully rather than causing a fatal error. A related crash in the `ComputedUrlWithQueryString` data type (which produced an `AssertionError` instead of a `TypeError`) was also fixed: it now returns an empty URL with the deleted entity's cache tags so that any cached empty result is invalidated if the entity is ever restored.

## Impact

No upgrade or configuration changes required.

## Technical details

The fatal originated in `ReferenceFieldTypePropExpression::calculateDependencies()` and `ReferenceFieldPropExpression::calculateDependencies()`. Both methods call `EntityReferenceFieldItemListInterface::referencedEntities()` to collect referenced entities and then map over them to build `dependencies['content']`. When a referenced entity has been deleted, earlier code paths could reach this with a `null` entity value; the PHP typed closure `fn (FieldableEntityInterface $entity)` then threw a `TypeError`. The fix switches to relying on `referencedEntities()` returning only fully loadable entities (deleted entries are absent rather than `null`), and adds an `\assert(Inspector::assertAllObjects(..., FieldableEntityInterface::class))` guard to catch regressions in development.

The companion fix (MR 1264) addressed `ComputedUrlWithQueryString`, which encountered the same deleted-entity condition through a different path. When the resolved URL value is `NULL` and the prop expression is a `ReferencePropExpressionInterface`, the class now calls `TypedDataHelper::getDeletedReferencedEntityCacheability()` to produce a `CacheableMetadata` carrying the deleted entity's cache tag (`entity_type_id:id`), then returns the partially-built `GeneratedUrl` instead of asserting. The cache tag ensures the cached empty URL is invalidated if the entity is recreated. Parallel discussion in the thread pointed to the deeper core-side problem of preventing deletion of entities that are active config dependencies, tracked separately in the core issue [#3532615](https://www.drupal.org/node/3532615).
