# #3591698: Release symmetrical translation functionality: drop the `canvas_dev_translation` feature flag module

- **Project:** Drupal Canvas
- **Type:** Critical task
- **Status:** Fixed
- **Discussion:** [#3591698](https://git.drupalcode.org/project/canvas/-/work_items/3591698) · 1 contributor · 3 comments

## Summary

Symmetric translation support for Canvas config entities (content templates and page regions) is no longer a feature-flagged experiment. The functionality previously gated behind the `canvas_dev_translation` sub-module is now built directly into the main `canvas` module. The `canvas_dev_translation` module is marked deprecated and will be removed before Canvas 2.0. Any site, recipe, or extension that installs `canvas_dev_translation` to enable translation should remove that dependency.

## Impact

Affects module maintainers.

- **API change:** `ConfigTranslationSupportHooks` in `canvas_dev_translation` moved to `ConfigTranslationHooks` in the main `canvas` module; `devTranslationMode` removed from `DrupalSettings`.
- **Deprecation:** `canvas_dev_translation` sub-module deprecated; its functionality is now in the main `canvas` module and the sub-module will be removed before Canvas 2.0.

## Technical details

The `ConfigTranslationSupportHooks` class from `Drupal\canvas_dev_translation\Hook` has been moved into the main module and renamed `ConfigTranslationHooks` under `Drupal\canvas\Hook`. Its `entityTypeAlter()` hook, which registers the `edit-form` link templates for `config_translation` and attaches `CanvasConfigEntityTranslationsAreValidConstraint`, is now an instance method (injecting `ModuleHandlerInterface`) and guards on `config_translation` being installed rather than on `canvas_dev_translation`. The `configSchemaInfoAlter()` method runs unconditionally in both the old and new code, because its `field.value.*` schema corrections are required for content translation as well as config translation.

`ContentTranslationHooks::entityTypeAlter()` no longer requires `canvas_dev_translation` to be present alongside `content_translation`; the guard is now `!moduleExists('content_translation')` only, so symmetric-translation constraints are active whenever content translation is enabled.

The `devTranslationMode` key has been removed from the `DrupalSettings` TypeScript interface and from the PHP attachment in `CanvasPageController`. The Topbar component previously showed translation UI only when `devTranslationMode` was true; it now shows it whenever `contentTranslationEnabled` or `configTranslationEnabled` is true. `LanguageConfigOverrideSchemaChecker` updates its reference from `ConfigTranslationSupportHooks::TRANSLATABLE_ENTITY_TYPE_IDS` to `ConfigTranslationHooks::TRANSLATABLE_ENTITY_TYPE_IDS`.

## Upgrade

1. Remove `canvas_dev_translation` from any module install lists, recipes, or test fixtures:

```yaml
# Before (recipe or test fixture)
install:
  - canvas:canvas_dev_translation

# After — simply omit it
```

2. If your code references the old class, update the namespace:

```php
// Before
use Drupal\canvas_dev_translation\Hook\ConfigTranslationSupportHooks;
ConfigTranslationSupportHooks::TRANSLATABLE_ENTITY_TYPE_IDS

// After
use Drupal\canvas\Hook\ConfigTranslationHooks;
ConfigTranslationHooks::TRANSLATABLE_ENTITY_TYPE_IDS
```

3. Remove any frontend checks for `devTranslationMode` in custom JavaScript or TypeScript that reads `drupalSettings.canvas.devTranslationMode`. Translation availability is now reflected directly by `contentTranslationEnabled` or `configTranslationEnabled`.
