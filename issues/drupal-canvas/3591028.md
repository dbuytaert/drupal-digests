## Summary

The Canvas module now validates component prop schemas using JSON Schema Draft-07 on both the PHP backend and the JavaScript frontend. Previously the PHP backend relied on the `justinrainbow/json-schema` library which was limited to Draft-04, while the frontend Ajv validator used a different default dialect. This mismatch meant Draft-07 schema features could be silently ignored or produce inconsistent validation results depending on which side evaluated the schema. Both validators are now explicitly configured to the same Draft-07 dialect.

## Impact

Affects site owners.

- **Requirements:** Minimum `justinrainbow/json-schema` version raised from ^6.6.2 to ^6.8.0

## Technical details

On the PHP side, `CanvasServiceProvider` now calls `$factory->addMethodCall('setDefaultDialect', [DraftIdentifiers::DRAFT_7])` on the `JsonSchema\Constraints\Factory` service definition. This requires the bumped `justinrainbow/json-schema ^6.8.0` dependency in `composer.json` (previously `^6.6.2`), which is the minimum version that introduced Draft-07 support.

On the JavaScript side, a new `createAjv()` factory function in `ui/src/utils/ajv.ts` centralizes Ajv instantiation. It creates an `Ajv` instance with `defaultMeta` explicitly set to `http://json-schema.org/draft-07/schema#`, making Draft-07 the active dialect for any schema that does not declare a `$schema` keyword. The `ajv-formats-draft2019` package is still layered on top via `addDraft2019()` to register extended string format validators (like `idn-email`, `iri`, `duration`) without changing the core dialect. Both the component prop-form validation path and `InputBehaviors` now call `createAjv()` instead of creating their own ad-hoc `new Ajv()` instances.

The issue notes this mirrors the approach Drupal core is taking in [#3352063](https://www.drupal.org/node/3352063) for Single Directory Component prop schemas.
