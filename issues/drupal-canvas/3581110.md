## Summary

The Canvas editor now renders `list_string` and `list_integer` fields with a rich multi-select UI: a React Select control showing selected values as removable chips, a searchable dropdown of allowed values, and automatic enforcement of the field's cardinality setting. This feature is active unconditionally for those field types, without requiring the `canvas_dev_mode` module.

## Impact

No upgrade or configuration changes required.

## Technical details

On the PHP side, `ReduxIntegratedFieldWidgetsHooks` previously gated all multi-value UI handling behind a `canvas_dev_mode` module check. The change adds an `OR` branch: if the field storage type is `list_string` or `list_integer`, the multi-value path runs regardless. Inside that path, `markMultivalueFormElements` is extended to recognize `select` elements (in addition to `textfield`, `number`, `url`, etc.) and stamps them with `#is_multivalue_form = TRUE`, `data-field-label`, and a new `data-cardinality` attribute carrying the raw cardinality integer (–1 for unlimited).

`canvas_stark_theme_suggestions_select_alter` detects `#is_multivalue_form` and pushes `select__multivalue_form` as a theme suggestion, causing Canvas Stark to render the select through a new Twig template (`select--multivalue-form.html.twig`). That template emits a `<drupal-canvas-select--multivalue-form>` custom element, which the frontend component registry maps to the new `DrupalSelectMultivalueForm` React component.

`DrupalSelectMultivalueForm` wraps a hidden native `<select multiple>` (managed by the existing `InputBehaviors` HOC for Redux tracking) alongside a `react-select` multi-select control. When the user picks or removes options, the component updates native select state and fires a synthetic `change` event so `InputBehaviors` picks it up. Cardinality enforcement is handled in `isOptionDisabled`: once the number of local selections reaches the `data-cardinality` value, any unselected option is disabled; `–1` means unlimited.

Two JavaScript fixes accompany the feature. First, `InputBehaviorsEntityForm` now detects `<select multiple>` elements and collects all `selectedOptions` rather than reading `.value` (which only returns the last-clicked item). Second, both `InputBehaviorsEntityForm` and `PageDataFormRenderer` convert array-valued multi-select entries (keyed as `field_name[]`) into indexed keys (`field_name[0]`, `field_name[1]`, ...) before writing to the Redux store or dispatching `setPageData`, preventing a PHP `Array to string conversion` error in `ClientDataToEntityConverter::setEntityFields`. Empty-array serialization is also fixed in `formStateToObject`: when the array is empty an empty-string sentinel is appended, and the deserializer normalizes both `''` and `['']` back to `[]`.

`PropShapeRepository` test coverage is extended with `list_string` and `list_integer` prop shapes (both unlimited and `maxItems`-bounded), each mapping to `options_select` widget and `canvas_load_allowed_values_for_component_prop` storage settings. Cypress entity-form-field-type tests and Playwright multi-value tests cover add, remove, clear-all, cardinality-limit enforcement, and persistence across page reload. This builds on the same multi-value UI pattern introduced for plain text fields in [#3573096](https://www.drupal.org/node/3573096) and integer/number fields in [#3576003](https://www.drupal.org/node/3576003).
