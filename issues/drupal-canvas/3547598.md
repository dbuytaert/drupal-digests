## Summary

The Canvas module's HTTP API for suggesting entity field sources to use in Content Template component props was returning developer-oriented, verbose labels and a flat response structure. Labels read like "Subset of this Article's field_silly_image: alt (1 of 7 props — absent: ...)" and suggestions were not ordered consistently with the site's form display. This change replaces those labels with concise human-readable ones (e.g. just "Title", "Silly image 🤡", or "Silly image 🤡 → Alternative text"), orders suggestions by the entity form display weight, and restructures the API response from a flat keyed object into a nested array tree that the frontend can render as a hierarchical contextual menu without any reprocessing.

## Impact

Affects module maintainers.

- **API change:** The Canvas HTTP API response for `DynamicPropSource` field suggestions changed from a flat keyed object (`{id: {label, source}}`) to a hierarchical array tree (`[{id?, label, source?, items?}]`), with the new `structureSuggestionsForHierarchicalResponse()` method replacing `structureSuggestionsForResponse()` in the controller.
- **New API:** New `Labeler` service (`\Drupal\canvas\PropExpressions\StructuredData\Labeler`) added with `label()` and `flatten()` public methods for generating hierarchical human-readable labels from field expression objects.

## Technical details

Three commits landed in sequence. The first (`9f218c54`) replaced `EntityTypeBundleInfoInterface` with `EntityDisplayRepositoryInterface` in `FieldForComponentSuggester`, added a bucketing pass that groups expression instances by field name using form-display component order, and simplified suggestion labels to drop the entity-bundle prefix and the verbose "absent props" listing. The new label rules are: if the expression uses the field's main property (or a computed property derived from it), the label is just the field label; if it follows a reference, a "Field → Entity type → Referenced field" pattern is used; if only non-main props are used, the label becomes "Field label (only prop-label)".

The second commit (`e8ff828e`) patched a sorting bug: `EntityFormDisplay::getComponents()` returns components without guaranteeing weight order, so `uasort($expected_order, SortArray::sortByWeightElement(...))` was added immediately after the call to produce the correct field ordering.

The third commit (`6d0717b0`) introduced the new `Labeler` service (`\Drupal\canvas\PropExpressions\StructuredData\Labeler`) which centralises label generation using semantical hierarchy markers (Unicode control characters from `StructuredDataPropExpressionInterface`) and a `flatten()` method that replaces those markers with ` → `. Label text now uses the Drupal-registered human label for each field property (e.g. "alt" becomes "Alternative text"), and File entity references are treated as an implementation detail: their type label is omitted from the hierarchy so that "Silly image 🤡 → File → status" becomes "Silly image 🤡 → Status". The three expression classes (`FieldPropExpression`, `FieldObjectPropsExpression`, `ReferenceFieldPropExpression`) each gained a `getHostEntityDataDefinition()` method to support this.

The controller now calls the new `FieldForComponentSuggester::structureSuggestionsForHierarchicalResponse()` instead of `structureSuggestionsForResponse()`. The response schema changed: `StructuredDataForPropShapeSuggestion` (a flat keyed object) was replaced by `StructuredDataForPropShapeHierarchicalSuggestionLevel` (an array), `StructuredDataForPropShapeHierarchicalSuggestionPath` (label + optional items, no selectable choice), and `StructuredDataForPropShapeHierarchicalSuggestionPathWithChoice` (label + id + source + optional items). Each node in the tree may be selectable (has `id` + `source`) or non-selectable (label only, grouping deeper choices). `JsonSchemaFieldInstanceMatcher` was annotated `@internal`, and its previously private `getReferenceDependency()` was made `public` to support `Labeler::usesMainProperty()`.

## Upgrade

Any code consuming the Canvas `DynamicPropSource` suggestions HTTP endpoint must be updated for the new response shape.

Before (flat keyed object):
```json
{
  "text": {
    "256dc2b8dcec7355": {
      "label": "This Article's Title",
      "source": { "sourceType": "dynamic", "expression": "..." }
    },
    "82ec95693bc89080": {
      "label": "Subset of this Article's field_silly_image: alt (1 of 7 props — absent: ...)",
      "source": { "sourceType": "dynamic", "expression": "..." }
    }
  }
}
```

After (hierarchical array tree):
```json
{
  "text": [
    {
      "id": "256dc2b8dcec7355",
      "source": { "sourceType": "dynamic", "expression": "..." },
      "label": "Title"
    },
    {
      "label": "Silly image 🤡",
      "items": [
        {
          "id": "82ec95693bc89080",
          "source": { "sourceType": "dynamic", "expression": "..." },
          "label": "Alternative text"
        }
      ]
    }
  ]
}
```

Nodes with `id` + `source` are selectable choices. Nodes with only `label` + `items` are non-selectable grouping levels. A node may have both `source` and `items` when the field itself is selectable and also has deeper nested choices.

The `FieldForComponentSuggester::getUsedFieldProps()` static method was removed; the equivalent is now `Labeler::getUsedFieldProps($expr, $actual_entity_type_and_bundle)`. The `FieldForComponentSuggester` constructor no longer takes `EntityFieldManagerInterface`; it now requires `EntityDisplayRepositoryInterface` and `Labeler`.
