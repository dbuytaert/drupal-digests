## Summary

A race condition in the Canvas editor could corrupt form state when a user interacted with multi-value field widgets quickly enough to trigger both a Drupal AJAX request and a layout PATCH request simultaneously. Because both requests read and write the same form build ID, one could overwrite the other's changes. The fix introduces bidirectional request serialization entirely in the frontend: AJAX requests now wait for any in-flight layout PATCH to finish before processing, and layout PATCH requests triggered by field blur now wait for any in-flight AJAX to complete before committing form state.

## Impact

No upgrade or configuration changes required.

## Technical details

The race condition had two directions. First, if a layout PATCH (`ApiLayoutController` or `componentInstanceFormApi`) started while a Drupal AJAX response was being processed, the AJAX command could overwrite the form cache with a stale form build ID. Second, if an AJAX request was in flight when a field blur triggered a model patch, the patch could send an out-of-date form build ID.

To prevent AJAX from running while a layout request is active, `pushCanvasLayoutRequest()` and `popCanvasLayoutRequest()` helpers are added to `baseQuery.ts`. They maintain a boolean stack in `drupalSettings.canvas.canvasLayoutRequestInProgress`. Both `componentInstanceFormApi` and `previewApi` call `pushCanvasLayoutRequest()` in `onQueryStarted` and `popCanvasLayoutRequest()` after `queryFulfilled`. When the stack empties, `popCanvasLayoutRequest()` fires a native `canvasLayoutRequestComplete` CustomEvent. `Drupal.Ajax.prototype.eventResponse` is monkey-patched in `ajax.command.customizations.js` to check a new `ajaxCanProceed()` guard before proceeding; if a layout request is in progress the handler registers a `{ once: true }` listener on `canvasLayoutRequestComplete` and defers execution until it fires.

To prevent a layout PATCH from running while AJAX is active, `InputBehaviorsCommon` in `inputBehaviors.tsx` now checks `isAjaxing()` inside the `onBlur`/`onChange` handler that calls `commitFormState()`. If AJAX is in progress, `commitFormState()` is deferred via a `{ once: true }` listener on a new `drupalAjaxStop` CustomEvent. That event is dispatched from `app.tsx` by listening to jQuery's `ajaxStop` event and re-firing it as a native `CustomEvent`, because jQuery's `.trigger()` does not reach native `addEventListener` listeners. The `isAjaxing()` helper (which checks `Drupal.ajax.instances`) was extracted from `app.tsx` into a shared `@/utils/isAjaxing.ts` module.
