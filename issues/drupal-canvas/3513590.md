## Summary

In the Canvas editor, props that are populated by a `DynamicPropSource` (i.e. props whose values are drawn from an entity field via a dynamic expression rather than stored statically) now display a visible label suffix in the component instance form to indicate they are disabled and dynamically driven. More importantly, the dynamic prop value evaluation now actually receives the host entity instead of always receiving `NULL`, so Content Templates can correctly resolve and display field-based prop values when previewing a component against a real entity.

## Impact

Affects module maintainers.

- **API change:** `ComponentSourceInterface::clientModelToInput()` gains a new `?FieldableEntityInterface $host_entity` parameter (between `$client_model` and `$violations`); the interface is `@internal`

## Technical details

The root cause was a `@todo` in `GeneratedFieldExplicitInputUxComponentSourceBase::clientModelToInput()` where `$source->evaluate(NULL, $is_required_prop)` was always called with `NULL` for the host entity, making `DynamicPropSource` expressions unevaluable in any context that provided an entity. The fix threads a `?FieldableEntityInterface $host_entity` parameter through the entire call stack: the `@internal` `ComponentSourceInterface::clientModelToInput()` signature gains the new parameter between `$client_model` and `$violations`, all five concrete implementations (`GeneratedFieldExplicitInputUxComponentSourceBase`, `BlockComponent`, `Fallback`, `InvalidSlots`, and `Personalization`) are updated accordingly, and `ClientServerConversionTrait::clientModelToInput()` (the private static method that drives bulk layout conversion) and `ComponentInstanceForm` both propagate the entity through to the interface call. The `ClientServerConversionTrait` also tightens its own `$entity` parameter from optional-with-default-`NULL` to required-nullable. On the UI side, `GeneratedFieldExplicitInputUxComponentSourceBase` now detects when the uncollapsed prop source is a `DynamicPropSource` and appends a `$label_suffix` containing the dynamic expression to the field widget label before calling `getWidget()`, giving editors a visible signal. This change was a prerequisite for [#3541057](https://www.drupal.org/node/3541057) (rendering `DynamicPropSource`s in `ApiLayoutController` for Content Templates).

## Upgrade

All affected symbols are marked `@internal`. If you have implemented `ComponentSourceInterface` despite that marker, you must add the new parameter:

```php
// Before
public function clientModelToInput(
    string $component_instance_uuid,
    Component $component,
    array $client_model,
    ?ConstraintViolationListInterface $violations = NULL,
): array { … }

// After
public function clientModelToInput(
    string $component_instance_uuid,
    Component $component,
    array $client_model,
    ?FieldableEntityInterface $host_entity,
    ?ConstraintViolationListInterface $violations = NULL,
): array { … }
```
