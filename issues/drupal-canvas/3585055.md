## Summary

The `canvas-workbench` CLI gains a new `preview-build` command in `@drupal-canvas/workbench@0.3.0`. Developers can now export a self-contained HTML preview for a single component or page without running the full Workbench dev server. The output is portable: each HTML file bundles fonts, images, and other local assets inline as base64 data URLs, so the artifact can be moved, shared, or used in review pipelines and downstream tooling without any additional serving infrastructure.

## Impact

Affects module maintainers.

- **New API:** New `preview-build` subcommand and exported functions `buildPreviewArtifact`, `parsePreviewBuildArgs`, `runPreviewBuildCli` in `@drupal-canvas/workbench@0.3.0`.

## Technical details

The CLI entry point (`bin.mjs`) gains a new branch that detects the `preview-build` subcommand and routes it to a dedicated script (`dist/server/src/server/preview-build.mjs`) via a refactored `runChild()` helper, leaving the existing Vite-backed `dev` / `build` path unchanged.

The new `preview-build.mjs` module exports `parsePreviewBuildArgs`, `buildPreviewArtifact`, and `runPreviewBuildCli`. `parsePreviewBuildArgs` validates the `--component-path` / `--page-path` (exactly one required), `--out-dir` (required), and optional `--pretty` flags and returns a typed result object. `buildPreviewArtifact` accepts injectable dependencies (`buildPreviewPayload`, `discover`, `resolveConfig`, `extractMetadata`, `bundleInteractivePreview`) so it can be tested without filesystem side-effects. For component targets it writes `component-default.html` plus zero-padded numbered mock files (`component-mock-01.html`, etc.); for page targets it writes `page-default.html`. A `manifest.json` with portable relative `path` values and `targetType`, `target.hasMocks`, `target.mockCount`, and per-entry labels accompanies every export.

Writes are atomic: all HTML files are prepared in a sibling staging directory first, then the directory is renamed into place. If an error occurs the staging directory is removed and any pre-existing output directory is restored from a backup rename, so a failed build never corrupts previously exported artifacts.

Path constraints are enforced against the configured `componentDir` and `pagesDir` from `canvas.config.json`. Paths outside those roots are rejected with an error message that names the configured directory and explains that preview discovery, mocks, and `@/` module resolution are scoped to those roots.

The companion `preview-payload` module (`preview-payload.mjs`) houses `buildPreviewPayload` and `bundleInteractivePreview`, which use Vite in library mode with `assetsInlineLimit: Infinity` to inline all local assets. The React client entry (`main.tsx`) gains `ensurePreviewRuntimeGlobals()`, which initializes `window.React` and `window.drupalSettings.canvasData.v0.baseUrl` as defaults so exported HTML files work when opened without the Workbench server's runtime scaffold. The command always writes one JSON payload to stdout covering `request`, `target`, `renderMode` (`interactive`), `outDir`, `manifestPath`, summary counts, warnings, and errors.
