## Summary

The `@drupal-canvas/workbench` package gains a new `preview-build` CLI command that exports standalone, self-contained HTML preview artifacts for a single component or page without running the full Workbench dev server. This makes it straightforward to generate portable preview output for review pipelines, testing, or downstream tooling. The package was published as version 0.3.0.

## Impact

Affects module maintainers.

- **New API:** New `canvas-workbench preview-build` CLI command with `--component-path`/`--page-path` and `--out-dir` flags for exporting standalone HTML preview artifacts

## Technical details

The issue adds a `preview-build` subcommand to the `canvas-workbench` CLI. The entry-point script (`canvas-workbench.mjs`) was refactored to extract a `runChild` helper and then branch on the first argument: when it is `preview-build`, the script delegates to a separate server-side binary at `dist/server/src/server/preview-build.mjs`; otherwise it falls through to the existing Vite dev-server path.

The core logic lives in the new `preview-build.ts` module, which exports two public functions: `parsePreviewBuildArgs` (parses `--component-path`/`--page-path`, `--out-dir`, and `--pretty` from `process.argv`) and `buildPreviewArtifact` (performs the actual export). `buildPreviewArtifact` accepts an options object and an injected-dependency bag (`buildPreviewPayload`, `discover`, `resolveConfig`, `extractMetadata`, `bundleInteractivePreview`) to keep it unit-testable. It stages HTML files to a temporary directory and atomically moves them into `--out-dir` only after all writes succeed, leaving existing output untouched on failure.

For component targets the output directory contains `component-default.html`, one `component-mock-NN.html` per mock variant, and `manifest.json`. For page targets it contains `page-default.html` and `manifest.json`. Each HTML file is self-contained: local asset imports (fonts, images) are bundled inline as data URLs, with binary assets such as WOFF2 and PNG files encoded as base64. `manifest.json` is target-specific: component manifests include `entries.default` and `entries.mocks`; page manifests include only `entries.default`. Entry `path` values are relative to the export directory so the bundle is portable. The command prints a JSON payload to stdout with `request`, `target`, `renderMode` (`interactive`), `outDir`, `manifestPath`, summary counts, warnings, and errors.

The client entry point (`main.tsx`) gained `ensurePreviewRuntimeGlobals()`, which sets `window.React`, `window.drupalSettings.canvasData.v0`, and `window.drupalSettings.canvasData.v0.baseUrl` to safe defaults when they are absent. This ensures the standalone exported HTML files have the runtime globals that Canvas components expect even outside a live Drupal site.

Path constraints are enforced: `--component-path` must resolve to a file inside the configured `componentDir` and `--page-path` must resolve inside `pagesDir`, because preview discovery, mocks, and `@/` module resolution are all scoped to those roots. Attempting to target a path outside those roots produces an `artifact_build_failed` error that names the configured directory in its message.

Documentation was added in the Workbench README (a new "Preview build command" section in the configuration reference) and as a new standalone user-docs page at `preview-build/`.
