## Summary

Canvas now blocks editing a component tree directly on an individual node. Any attempt to open the Canvas editor for a node (e.g. `/canvas/editor/node/1`) returns an HTTP 500 with the message "For now Canvas only works if the entity is a canvas_page! Other entity types and bundles must use content templates for now." Sites that previously attached per-node Canvas component trees must use Content Templates instead. A second fix shipped in the same issue: the component tree field now renders identically on the live page and inside the Canvas editor preview, without the default Drupal `<div class="field ...">` wrapper markup. This eliminates CSS selector mismatches (like `.layout--pass--content-medium > *` hitting the wrong elements) that had been causing broken layouts for some themes.

## Impact

Affects front-end developers.

- **Theme change:** New `field--component-tree.html.twig` template overrides the default field template for Canvas component tree fields, removing the `<div class="field ...">` wrapper; themes that override `field--canvas-page--components.html.twig` may find their override no longer called in the Canvas editor preview (tracked in #3550093).

## Technical details

`ComponentTreeLoader::getCanvasFieldName()` now throws `\LogicException` immediately for any entity that is not a `canvas_page`, with a test-only exception for `article` nodes. This short-circuits every Canvas API endpoint that routes through `ComponentTreeLoader`, making it the enforcement point for the per-node editing block.

`ContentTemplateAwareViewBuilder::getBuildDefaults()` was refactored to call `ContentTemplate::loadForEntity($entity, $view_mode)` directly rather than probing for a Canvas field. The `#theme` key (which suppresses `node.html.twig`) is now only unset when an active `ContentTemplate` exists for that entity and view mode. Cache dependencies are now set on the `ContentTemplate` config entity itself (or on its entity-type list cache tags when no template exists), so render caches invalidate correctly when a template is created or toggled. `ContentTemplate::buildMultiple()` was also updated to skip the Canvas-field requirement entirely when no exposed slots are defined, meaning Content Template rendering works without the node type having a Canvas field.

For the field-wrapper fix: `ModuleHooks::theme()` registers `field__component_tree` with `'base hook' => 'field'`, and a new `field--component-tree.html.twig` template renders only `{{ item.content }}` per item, stripping the surrounding `<div class="field field--type-component-tree ...">` wrapper. A new kernel test `ComponentTreeFieldRenderingTest` asserts that `.field--type-component-tree` is absent from both the live page HTML and the Canvas preview API JSON response. E2E Playwright snapshots were updated to reflect that nodes now render inside the full `<article>` element produced by `node.html.twig` when no Content Template is active.

## Upgrade

Sites that were using per-node Canvas component trees (a Canvas field on a node type with individual component trees per node) must migrate to Content Templates. Attempting to use the Canvas editor on a node will now return HTTP 500.

There is no automated migration. The release notes advise manually copying component trees from individual nodes to Page entities using Canvas' copy/paste functionality before updating to this version.
