# #3591684: UI Exception / 500 Error when triggering Preview on Content Templates causes persistent state lock

- **Project:** Drupal Canvas
- **Type:** Critical bug report
- **Status:** Fixed
- **Discussion:** [#3591684](https://git.drupalcode.org/project/canvas/-/work_items/3591684) · 1 contributor · 3 comments

## Summary

When editing Content Templates in the Canvas page builder, the Preview button appeared in the toolbar even when no preview entity had been selected. Clicking it caused a 500 Internal Server Error, and clicking "Exit Preview" afterward left the editor stuck in a broken layout state that required a hard browser refresh to clear. Both error conditions are now resolved: the Preview button is hidden for Content Templates until a preview entity is chosen, and the preview flow uses the correct backend endpoint when a preview entity is available.

## Impact

No upgrade or configuration changes required.

## Technical details

The 500 error originated in `ComponentTreeLoader::getCanvasFieldName()`, which throws a `LogicException` if the entity passed to it is not a `canvas_page` entity type. The generic layout GET route `/canvas/api/v0/layout/{entity_type}/{entity}` routes through this check, so calling it with a Content Template entity inevitably fails. A regression had caused `PreviewControls.tsx` to display the Preview button inside the template editor context even when no `previewEntityId` param was present in the URL, and its "enter preview" click handler navigated to a path that `PagePreview.tsx` backed with the wrong API queries.

The fix has three coordinated parts. First, `PreviewControls.tsx` now returns `null` when `isTemplateContext && !previewEntityId`, hiding the button until a preview entity is selected in the `ContentPreviewSelector` dropdown. Second, when a `previewEntityId` is set, the click handler navigates to `/preview/template/${entityType}/${bundle}/${previewEntityId}/${viewMode}`, and "Exit Preview" uses the `isTemplatePreviewRoute` branch to navigate back to `/template/${entityType}/${bundle}/${viewMode}/${entityId}`, keeping the editor's Redux `EditorFrameContext` state consistent. Third, `PagePreview.tsx` detects the `/preview/template/` prefix via a local `isContentTemplate` flag, skips both `useGetPageLayoutQuery` and the `postPreview` mutation (both of which use the generic layout endpoint), and instead drives the preview iframe via `useGetSnapshotPreviewQuery` with `isTemplate: true`. That query calls the dedicated route `/canvas/api/v0/layout-content-template/${entityType}.${bundle}.${viewMode}/${previewEntityId}`, which is designed to handle non-`canvas_page` entities. The `useTemplateRef` hook centralises the two context signals (`isTemplateContext` from Redux, `isTemplatePreviewRoute` from the URL path) so both `PreviewControls` and other components can branch consistently.
