## Summary

The `canvas validate` CLI command now validates pages in addition to components, and the `canvas push --include-pages` command validates pages before uploading and aborts if any page fails. This makes it easier for developers and AI agents to catch page errors locally before they reach a Drupal site. The change also fixes a long-standing gap where per-component prop validation inside `defineComponentCatalog` was silently skipped for catalogs with more than one component.

## Impact

Affects module maintainers.

- **New API:** New `validatePages()` and `validateElements()` functions in `utils/validate-page.ts` for validating Canvas page files and their element trees against a component catalog.

## Technical details

A new `validatePages()` async function in `utils/validate-page.ts` drives the page validation logic. It reads each discovered page file, validates the file structure against `page-spec.schema.json` using `ajv` v8 with `ajv-formats`, then validates the page elements against a component catalog. `buildElementsValidationContext()` builds that catalog from enabled component metadata via `defineComponentCatalog()`, exposing a pre-built `ElementsValidationContext` that `validateElements()` consumes. `validateElements()` first checks for disabled components and reports them directly, then converts the `AuthoredSpecElementMap` to a json-render spec via `authoredSpecToComponentTree()` and `canvasTreeToSpec()` and calls `catalog.validate()`.

The `drupal-canvas` package (bumped to v0.3.2) received two fixes required by this feature. First, `defineComponentCatalog()` now overrides the default `catalog.validate` method with one built on a Zod discriminated union keyed on `type`, so each element's `props` are checked against its specific component schema. The upstream default fell back to `z.record(z.string(), z.unknown())` for catalogs with two or more components, silently bypassing all prop validation. Second, the new `stripUriRefFormats()` function recursively removes `uri-reference` and `iri-reference` format keywords from component JSON schemas before they are passed to `fromJSONSchema()`. Without this, `fromJSONSchema` maps those formats to `z.url()`, which rejects relative URLs. The stripped paths are re-validated with `new URL(current, 'http://localhost')` to accept relative URLs. A synthetic `canvas:component-tree` entry is also registered in the catalog to handle multi-root trees produced by `canvasTreeToSpec()`.

In the `validate` command, results are now typed as `Result[]`, each entry carrying an `itemType` field ('Component' or 'Page'). The `reportResults` call uses 'Item' as the label, and the underlying `reportResults` utility was fixed to use `headers.length - 1` for the wrapping column rather than a hardcoded index of `2`. In the `push` command, a validation spinner and a call to `validatePages()` are inserted before the page push loop; if any page fails, an error is thrown and the push is aborted. The `discoverCanvasProject` call in validate now passes `pagesRoot: config.pagesDir`.

This issue builds on page push/pull support added in [#3575874](https://www.drupal.org/node/3575874) and the local page file format established by [#3582632](https://www.drupal.org/node/3582632).
