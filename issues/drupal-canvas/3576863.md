## Summary

Canvas sites can now manage Content Templates through the CLI and Workbench, alongside the existing pages and Code Components support. The CLI gains `--include-content-templates` / `CANVAS_INCLUDE_CONTENT_TEMPLATES` flags on `pull` and `push`, and a new `contentTemplatesDir` config key (defaulting to `./content-templates`). The Workbench dev server now discovers `content-templates/*.json` files and renders live previews of them. A new experimental `agents-context` command pulls field binding and view-mode metadata for AI coding agents. Sites need to run database updates to install the new `canvas_content_template` OAuth scope.

## Impact

Affects module maintainers and site owners.

- **Database update:** New post-update hook `canvas_oauth_post_update_0005_canvas_content_template_scope` creates the `canvas_content_template` OAuth2 scope.
- **Config change:** New `contentTemplatesDir` key in `canvas.config.json` (defaults to `./content-templates`) controls where the CLI and Workbench look for content template spec files.
- **New API:** New POST route `canvas.api.layout.content_template_draft` resolves an unsaved content template draft against a preview entity; new `ComponentTree` OpenAPI schema; PATCH now permitted on `ContentTemplate` config entities.

## Technical details

The change touches the PHP backend, the CLI, the Workbench, and the shared tooling packages.

On the PHP side, a new POST route `canvas.api.layout.content_template_draft` is added at `/canvas/api/v0/layout-content-template-draft/{entity_type}/{preview_entity}`, handled by `ApiLayoutController::draftContentTemplate`. It resolves an unsaved content template draft against a preview entity and returns the evaluated prop model. The route requires `administer content templates` and entity view access, and is added to the named-route allowlist in `CanvasOauthAuthenticationProvider` so it can authenticate via simple_oauth. `ContentTemplate::ENTITY_TYPE_ID` is also added to the list of protected config entity types in that provider. The `canvas.api.config.patch` route regex now includes `content_template` so PATCH operations on `ContentTemplate` config entities are permitted. A `canvas_oauth_post_update_0005_canvas_content_template_scope` post-update hook creates the new `canvas_content_template` OAuth2 scope (backed by the `administer content templates` permission), with matching config YAML and an upgrade path test.

On the JavaScript side, a new monorepo package `@drupal-canvas/auth` (`packages/auth`) extracts the shared OAuth token-endpoint helpers (previously living only in the CLI) into a common library shared by both the CLI and the Workbench. It exports `requestOauthToken`, `refreshAccessToken`, `fetchClientCredentialsToken`, `OauthError`, `OauthTokenResponse`, `getTokenEntry`/`setTokenEntry`/`removeTokenEntry`, and related types. The CLI and Workbench both update their `package.json` to depend on `@drupal-canvas/auth` instead of the CLI-local copy.

The OpenAPI spec gains a `PATCH /canvas/api/v0/content-template/{configEntityId}` operation with an `UpdateContentTemplate` schema, a new `POST /canvas/api/v0/layout-content-template-draft/{entityTypeId}/{entityId}` operation, a new `ComponentTree` schema (a flat array of component instance objects with `uuid`, `component_id`, `component_version`, `parent_uuid`, `slot`, `inputs`, and `label`), and `component_tree` added as a required field on the `ContentTemplateLayout` response schema.

## Upgrade

Sites running `canvas_oauth` must run database updates to create the new scope:

```bash
drush updb
```

CLI users who want to sync content templates should add `contentTemplatesDir` to `canvas.config.json`:

```json
{
  "contentTemplatesDir": "./content-templates"
}
```

Then use the new flags:

```bash
npx canvas pull --include-content-templates
npx canvas push --include-content-templates
```

To request the new scope via the environment variable approach, add `canvas:content_template` to `CANVAS_SCOPE`.
