## Summary

Canvas Workbench is a new local developer tool for previewing Drupal Canvas Code Components and pages directly from source code, without a running Drupal backend. Similar to Storybook, it automatically discovers components from your codebase, reads their configuration from `component.yml` files, displays live previews with example values, and supports hot module reload. It is distributed as the `@drupal-canvas/workbench` npm package and can be started with `npx @drupal-canvas/workbench@latest` from your project root. A `canvas.config.json` file is optional and lets you override default paths for component scanning, pages, global CSS, and alias resolution. The name "Workbench" is under active discussion (see [#3576829](https://www.drupal.org/node/3576829)) due to potential confusion with the existing Workbench contrib module.

## Technical details

The commit introduces three new packages and expands the `drupal-canvas` package.

`@drupal-canvas/discovery` provides filesystem discovery via `discoverCodeComponents()` and `loadComponentsMetadata()`. It scans a `scanRoot` for `component.yml` and `*.component.yml` files, applies root `.gitignore` rules, ignores standard build and dependency directories, resolves JavaScript entries by extension precedence (`.ts`, `.tsx`, `.js`, `.jsx`), attaches optional `.css` entries, and discovers pages from `pages/*.json`. Component IDs are generated as SHA1 hashes of the metadata file path. Warnings are emitted for `missing_js_entry`, `duplicate_definition`, and `conflicting_metadata` conditions.

`@drupal-canvas/vite-compat` provides shared Vite compatibility helpers: resolving `@/` alias imports from host components, supporting asset imports, enabling SVGR for SVG components, and handling extensionless directory imports with automatic index file resolution.

`@drupal-canvas/workbench` runs a single Vite dev server that serves both the component browser UI and preview iframes. Discovery middleware exposes component and page data at `/__canvas/discovery` and preview manifests at `/__canvas/preview-manifest`. Component modules are imported through Vite `@fs` URLs and must have a `default` export. Configuration is read from an optional `canvas.config.json` file. A `preview-build` command exports self-contained HTML artifacts for individual component or page targets.

The `drupal-canvas` package gains new exports on the `drupal-canvas/json-render-utils` subpath: `renderSpec()` renders a json-render `Spec` using a `ComponentRegistry`, resolving `$state` expressions and `$cond/$then/$else` conditional logic; `defineComponentRegistry()` builds a registry by dynamically importing component JavaScript entry files; `defineComponentCatalog()` converts `ComponentMetadata` arrays from `@drupal-canvas/discovery` to json-render catalogs using Zod schemas, supporting spec validation and AI prompt generation via `catalog.prompt()`. An internal `normalizeCanvasProps()` function unwraps rich text `{ value, format }` objects and image metadata wrappers before passing props to React components. `specToCanvasTree` now accepts a `PropResolutionContext` and resolves `$state` prop expressions via `resolveElementProps` from `@json-render/core`. `zod` is added as a dependency.

## Upgrade

The json-render utilities `canvasTreeToSpec`, `specToCanvasTree`, and `renderCanvasTree` must now be imported from `drupal-canvas/json-render-utils` instead of `drupal-canvas`.

```js
// Before
import { canvasTreeToSpec, specToCanvasTree, renderCanvasTree } from 'drupal-canvas';

// After
import { canvasTreeToSpec, specToCanvasTree, renderCanvasTree } from 'drupal-canvas/json-render-utils';
```
