## Summary

When you edit a page, template, or region in Canvas, component instances now automatically upgrade to the latest version of their components when it's safe to do so. Safe upgrades include adding optional props, adding slots, making props optional, or changing default values. This means you'll immediately see new features like additional image media types without manual intervention, keeping your component trees up to date as developers improve components.

## Impact

No upgrade or configuration changes required.

## Technical details

This feature introduces automatic component instance version upgrading for component sources using `GeneratedFieldExplicitInputUxComponentSourceBase` (SDCs and code components). A new `ComponentInstanceUpdaterInterface` handles the upgrading logic, with implementations registered via the `updater` parameter in the `ComponentSource` attribute.

The `GeneratedFieldExplicitInputUxComponentInstanceUpdater` implements safe auto-updates by comparing prop and slot definitions between versions. Safe changes include adding optional props or slots, changing props from required to optional, modifying widgets, changing default values, or expanding allowed entity reference target bundles. Unsafe changes that prevent auto-update include removing props/slots, adding required props, changing prop from optional to required, or changing field storage/instance settings (except entity reference target bundle supersets).

`ComponentSourceManager::updateComponentInstances()` iterates through component trees and attempts updates. This runs automatically in `ApiLayoutController::buildRegion()` and `ApiConfigControllers::normalize()`, ensuring component instances reference the active version before rendering forms or previews. `StorablePropShape::fieldDataFitsIn()` validates field data compatibility between versions. The `Component` entity's `getSettings()` and `getSlotDefinitions()` methods now accept an optional version parameter to retrieve historical version data.

## Lessons learned

- When a component schema evolves, automatic upgrades are safe when changes only add optional capabilities or relax constraints (like adding optional props or slots, or making required fields optional) because these never invalidate existing data. Detect these safe patterns and transparently upgrade instances to avoid accumulating technical debt from version fragmentation. *(via effulgentsia, penyaskito, wim leers)*
- When multiple versions of a schema coexist, minimizing version fragmentation improves consistency and reduces maintenance burden. Automatically upgrade to the latest compatible version when loading data for editing, rather than only when explicitly saved, so users immediately benefit from the newest capabilities. *(via wim leers, penyaskito)*
