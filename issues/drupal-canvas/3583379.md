## Summary

On multilingual sites, Canvas component props sourced from entity reference fields were always populated from the referenced entity's default language, ignoring the current page language. A paragraph referencing a translated Media image, for example, would display the English alt text even on a French or Spanish page. This fix makes `Evaluator` resolve every referenced entity to its correct translation before reading field values from it, so component props now reflect the language of the host entity (or the negotiated content language when there is no host entity).

## Impact

Affects module maintainers.

- **API change:** `Evaluator::evaluate()` gained a required fourth parameter `NegotiatedLanguage $language`; all direct callers must be updated.

## Technical details

The root cause was that `Evaluator::doEvaluate()` followed `DataReferenceInterface` to obtain referenced entities but never called `getTranslation()` on them. The fix introduces a new `@internal` value object `NegotiatedLanguage` (in `Drupal\canvas\PropExpressions\StructuredData`) that bundles a `LanguageInterface` instance with its associated cache metadata. It provides two factory methods: `NegotiatedLanguage::matchEntity($entity)` (seeds language from the host entity, inheriting its cache tags) and `NegotiatedLanguage::negotiateFromConfigAndContext()` (reads the active content language from `LanguageManager`, adding `languages:language_content` cache context only on multilingual sites).

`Evaluator::evaluate()` and `Evaluator::doEvaluate()` now require a `NegotiatedLanguage $language` argument that is threaded through the full recursion chain. A new private `Evaluator::resolveTranslation()` helper calls `EntityRepositoryInterface::getTranslationFromContext()` at the three places in `doEvaluate()` where a referenced entity is unwrapped (`ReferenceFieldTypePropExpression` for field-type-based expressions, and both the single-cardinality and multi-cardinality branches of `ReferenceFieldPropExpression`). The helper guards against locked language codes (`und`, `zxx`) and triggers a deprecation when an untranslatable entity type is encountered with such a code.

Entry points seed the language: `EntityFieldPropSource::evaluate()` and `JsComponent` call `NegotiatedLanguage::matchEntity($host_entity)`, so the host node's language drives all downstream reference resolution. `StaticPropSource::evaluate()` does the same when the host entity is translatable, and falls back to `negotiateFromConfigAndContext()` otherwise. `ComputedUrlWithQueryString` uses the field item's own langcode as a `NegotiatedLanguage` (with a `@todo` to use the host entity approach once a related cacheability issue, [#3571785](https://www.drupal.org/node/3571785), is resolved).

The dependency-calculation paths in `ReferenceFieldPropExpression` and `ReferenceFieldTypePropExpression` are also simplified: they now call `EntityReferenceFieldItemListInterface::referencedEntities()` directly instead of going through `Evaluator::evaluate()`, eliminating the need to pass a language at dependency-calculation time.

`Layers.php` (Canvas's architectural constraint file) is updated to permit imports from `Drupal\Core\Language` inside the `Evaluator`/`Labeler` layer.

## Upgrade

Any code calling `Evaluator::evaluate()` directly must now pass a `NegotiatedLanguage` as the fourth argument. `NegotiatedLanguage` is `@internal`, so the intended pattern is to use its factory methods:

```php
use Drupal\canvas\PropExpressions\StructuredData\Evaluator;
use Drupal\canvas\PropExpressions\StructuredData\NegotiatedLanguage;

// Before:
$result = Evaluator::evaluate($entity, $expr, is_required: FALSE);

// After — use the host entity's language:
$result = Evaluator::evaluate($entity, $expr, is_required: FALSE, language: NegotiatedLanguage::matchEntity($entity));

// Or, when there is no host entity, fall back to the negotiated content language:
$result = Evaluator::evaluate($entity, $expr, is_required: FALSE, language: NegotiatedLanguage::negotiateFromConfigAndContext());
```
