## Summary

This is an internal architectural change with no user-facing effect. It lays the groundwork for symmetrical translation of component tree inputs on multilingual Canvas sites. Once the follow-up issues ([#3582478](https://www.drupal.org/node/3582478) for config-defined trees and [#3583684](https://www.drupal.org/node/3583684) for content-defined trees) land, editors will be able to translate individual component inputs in both config entities and content entities using a consistent, per-input translatability model.

## Impact

Affects module maintainers.

- **API change:** `GeneratedFieldExplicitInputUxComponentSourceBase::getExplicitInputDefinitions()` changed from `protected` to `public`; subclasses that override it as `protected` will break.
- **New API:** New `ComponentInstanceInputsConfigSchemaGeneratorInterface` with `getConfigSchemaMapping()` and `refineForInstance()`, new `inputs_config_schema_generator` parameter on `#[ComponentSource]` attribute, new static `ComponentInputs::resolveConfigSchemaMapping()`, and new `ComponentInputs::getTranslatableInputKeys()`.

## Technical details

The core problem is that translating component instance inputs requires knowing which of those inputs are actually translatable, and that knowledge must be consistent between config entity translation (driven by configuration schema discovered by `config_translation`) and content entity translation (driven by `content_translation`). Before this change there was no single source of truth for that.

The solution introduces `ComponentInstanceInputsConfigSchemaGeneratorInterface` with two methods: `getConfigSchemaMapping(ComponentSourceInterface $component_source)` generates a config schema mapping for all inputs of a component version, and `refineForInstance(array $mapping, array $actual_inputs, string $component_id, string $component_version)` adjusts that mapping per-instance (for example, removing `translatable: true` from an input that is populated by a non-static prop source rather than direct user input).

A new `inputs_config_schema_generator` constructor parameter is added to the `ComponentSource` attribute, accepting an FQCN implementing this interface. It defaults to `FallbackComponentInstanceInputsConfigSchemaGenerator`, which emits `type: ignore` for all inputs and marks nothing as translatable. Three concrete implementations are provided:

- `FallbackComponentInstanceInputsConfigSchemaGenerator`: the default, produces `type: ignore` for every input key with no translatability.
- `BlockComponentInstanceInputsConfigSchemaGenerator`: reads the existing `block.settings.*` config schema via `TypedConfigManagerInterface`, strips `provider`, `id`, and `context_mapping`, and returns the native block settings mapping. Block settings are plain values so `refineForInstance()` is a no-op.
- `GeneratedFieldExplicitInputUxComponentInstanceInputsConfigSchemaGenerator`: inspects the normalized JSON schema prop shapes from `GeneratedFieldExplicitInputUxComponentSourceBase::getExplicitInputDefinitions()` and marks a prop as `translatable: true` when its shape is `type: string` (with no `enum`) and its `format` is absent or one of `iri`, `iri-reference`, `uri`, `uri-reference`, or when `contentMediaType` is `text/html`. In `refineForInstance()`, any prop whose stored value is not a `StaticPropSource` has its `translatable` key removed, because only user-authored static values should be translated.

The static method `ComponentInputs::resolveConfigSchemaMapping(string $component_id, string $component_version, array $actual_inputs)` is the single point of truth: it loads the `Component` config entity, resolves the generator class from the plugin definition, instantiates it via `ClassResolverInterface` (to support container injection as in `BlockComponentInstanceInputsConfigSchemaGenerator`), and calls `getConfigSchemaMapping()` then `refineForInstance()`. A new instance method `ComponentInputs::getTranslatableInputKeys()` calls `resolveConfigSchemaMapping()` and filters to keys whose schema definition has `translatable: true` (using an ephemeral `Mapping` object to resolve inherited translatability from core's config schema type hierarchy).

`BlockComponent` and both `GeneratedFieldExplicitInputUx`-based sources (`JsComponent` and the SDC source) have their `#[ComponentSource]` attributes updated to specify their respective generator class.

`GeneratedFieldExplicitInputUxComponentSourceBase::getExplicitInputDefinitions()` is promoted from `protected` to `public` so that `GeneratedFieldExplicitInputUxComponentInstanceInputsConfigSchemaGenerator` can call it from outside the class hierarchy.
