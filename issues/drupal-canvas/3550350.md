## Summary

The Canvas editor now supports panning by holding the Spacebar and dragging with the left mouse button, in addition to the previously available middle-mouse-button drag. The cursor changes to a grab hand when Space is held and a grabbing hand while actively panning, giving clear visual feedback of the current mode. Panning no longer cancels when the mouse crosses over a sidebar panel (the components list or props panel).

## Technical details

The core change is in `EditorFrame`: mouse move and mouse up listeners were moved from React synthetic event handlers on the editor pane element (`onMouseMove`, `onMouseUp`, `onMouseLeave`) to native `document.addEventListener('mousemove'/'mouseup')` handlers registered only while a pan is active and cleaned up on release. This is what prevents the sidebar from interrupting a pan in progress.

The panning start state was switched from `useState` (which would be stale in event handler closures) to a `useRef` (`panningStartRef`) that stores the initial mouse coordinates alongside the initial `scrollLeft`/`scrollTop` of the editor pane. The scroll delta is then computed as `newScrollLeft = startScrollLeft - (currentMouseX - startMouseX)`, fixing a latent bug where the old code conflated mouse position with absolute scroll position.

New `spaceKeyPressed` and `panningMode` state variables drive the CSS classes that set cursor styles. A `useHotkeys('space', ...)` handler with `{ preventDefault: true, eventListenerOptions: { capture: true } }` intercepts the Space key before any other handler, preventing the browser's default page-scroll behavior and a race with the component selection hotkey. `ComponentOverlay` was also updated to use `event.code === 'Space'` (instead of `event.key === ' '`) and guards with `!event.repeat` to avoid repeated triggers. The `.sortableItem` CSS rule that unconditionally set `cursor: grab` on draggable components in the preview was removed, aligning the behavior with tools like Figma and Framer.
