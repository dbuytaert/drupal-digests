## Summary

Experience Builder and Canvas previously matched image and video URI props in SDCs by checking whether the URI ended in a hardcoded list of file extensions (`png`, `gif`, `jpg`, `jpeg`, `webp`, `avif`, `mp4`). This broke on Drupal sites whose image or video fields allowed different formats, meaning years of uploaded images could not be connected to components in the page editor. The fix replaces that approach with standard JSON Schema vocabulary: `contentMediaType: image/*` (or `video/*`) declares the media category, and the new custom extension `x-allowed-schemes` declares valid URI schemes. The built-in `image-uri`, `stream-wrapper-image-uri`, and the video URI schema definitions are already updated. SDC component authors who used `$ref` to those definitions need no changes; authors who copied the old extension-based `pattern` directly into their own component YAML should migrate to the new form.

## Impact

Affects module maintainers.

- **API change:** `UriTargetMediaTypeConstraint::PLUGIN_ID` renamed from `'UriTargetMediaTypeConstraint'` to `'UriTargetMediaType'`; image-uri and stream-wrapper-image-uri `$ref` definitions now emit `contentMediaType` + `x-allowed-schemes` instead of a file-extension `pattern`.
- **New API:** New `UriConstraint` (`Uri`), `UriSchemeConstraint` (`UriScheme`), and `UriTargetMediaTypeConstraint` (`UriTargetMediaType`) Symfony validation constraint plugins in the Canvas module; new `x-allowed-schemes` custom JSON Schema keyword for URI/URI-reference props.

## Technical details

The old shape matching path in `JsonSchemaFieldInstanceMatcher` translated Drupal's `FileExtension` constraint (a space-separated list like `"png gif jpg jpeg webp avif"`) into a case-insensitive regex pattern that was then compared against the regex in the SDC prop's JSON Schema. Any mismatch in the extension list meant no match, so a site that had not yet enabled AVIF would fail to match fields even though AVIF was present in the schema.

The first commit (`9048ab84`, `experience_builder`) introduces `contentMediaType: image/*` (or `video/*`) as the semantic signal for URI prop types, backed by a new no-op `UriTargetMediaTypeConstraint` (`PLUGIN_ID: UriTargetMediaType`). The `fileExtensionsToTargetContentMediaType()` method replaces `buildCaseInsensitiveExtensionRegex()`: it calls `ExtensionMimeTypeGuesser::guessMimeType()` on each allowed extension and reduces the results to the top-level MIME media type (`image`, `video`, etc.). Shape matching in `JsonSchemaStringFormat::toDataTypeShapeRequirements()` and `computeStorablePropShape()` now branches on `contentMediaType` rather than on `pattern`. The `pattern` in that commit was simplified to only validate URI structure (no protocol-other-than-http/https, no whitespace) rather than file extension. `DataTypeShapeRequirement` and `DataTypeShapeRequirements` both implement `\IteratorAggregate` so they can be spread with `...iterator_to_array(...)` when building a `DataTypeShapeRequirements` from a single `DataTypeShapeRequirement`. `JsonSchemaType::patternToPcre()` is extracted to avoid duplicated inline regex escaping.

The second commit (`247bf510`, `canvas`) goes further and removes `pattern` from the URI shape definitions entirely, replacing it with a new `x-allowed-schemes` custom JSON Schema keyword. For browser-accessible image URLs the value is `["http", "https"]`; for stream wrapper URIs it is `["public"]`. Two new Symfony-based validation constraint plugins are added to the Canvas module: `UriConstraint` (`Uri`) validates that the string is a syntactically valid URI or URI-reference (using `JsonSchema\Tool\Validator\UriValidator` and `RelativeReferenceValidator`), and `UriSchemeConstraint` (`UriScheme`) validates an allowlist of URI schemes. `CanvasServiceProvider::alter()` wires a custom `UriSchemeAwareFormatConstraint` into the `ComponentValidator`'s `JsonSchema\Validator` instance so that `x-allowed-schemes` is enforced during SDC prop validation at the JSON Schema level. Field type overrides (`FileUriItemOverride`, `ImageItemOverride`, `LinkItemOverride`, `UriItemOverride`) drop their `Regex` constraints in favour of `UriConstraint` + `UriSchemeConstraint`. Shape matching discriminates image-URI vs stream-wrapper-URI shapes by checking `x-allowed-schemes` rather than comparing literal `pattern` strings. A `UriConstraint::isSupersetOf()` helper allows the matcher to recognize that a field property validated as a plain URI is a valid candidate for a `uri-reference` prop shape.

## Upgrade

SDC component authors who hardcoded a file-extension regex for image or video URI props should migrate to the new form. Using the canonical `$ref` is the recommended approach:

```yaml
# Before (hardcoded extension list — no longer needed)
src:
  type: string
  format: uri-reference
  pattern: '^(/|https?://)?.*\.(png|gif|jpg|jpeg|webp|avif)(\?.*)?(#.*)?$'

# After — option 1: use the built-in $ref
src:
  $ref: json-schema-definitions://canvas.module/image-uri
  type: string   # required until core bug #3352063 is fixed

# After — option 2: write the equivalent schema inline
src:
  type: string
  format: uri-reference
  contentMediaType: image/*
  x-allowed-schemes: [http, https]
```

For stream-wrapper image URIs (e.g. `public://`):

```yaml
src:
  $ref: json-schema-definitions://canvas.module/stream-wrapper-image-uri
  type: string

# Inline equivalent:
src:
  type: string
  format: uri
  contentMediaType: image/*
  x-allowed-schemes: [public]
```

The `UriTargetMediaTypeConstraint` PLUGIN_ID changed from `'UriTargetMediaTypeConstraint'` to `'UriTargetMediaType'`. Any code that referenced the old string constant should use `UriTargetMediaTypeConstraint::PLUGIN_ID` instead.
