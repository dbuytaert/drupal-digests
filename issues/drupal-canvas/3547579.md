## Summary

Canvas now automatically re-evaluates how component props are stored whenever relevant configuration changes. Previously, if no compatible MediaType existed at the time Canvas was installed, components would remain stuck using a plain image field widget even after a site owner later created an appropriate MediaType. Now, once the relevant cache tags are invalidated (for example, when the media type list changes), Canvas recomputes the affected prop shapes and regenerates `Component` config entities automatically, without requiring a manual cache clear or component resave. Module authors implementing the prop-shape alter hook can opt into this mechanism by declaring which cache tags make their hook decisions stale.

## Impact

Affects module maintainers.

- **Deprecation:** `hook_storage_prop_shape_alter` deprecated in canvas:1.0.0; replace with `hook_canvas_storable_prop_shape_alter` by canvas:2.0.0
- **New API:** Hook implementations can call `CandidateStorablePropShape::addCacheTags()` to declare which tags make their prop-shape decision stale, triggering automatic `Component` config entity regeneration

## Technical details

The root cause was that `PropShape::getStorage()` invoked `hook_storage_prop_shape_alter` and discarded its result with no cacheability tracking. Nothing in Canvas knew that the hook's output might need to change in response to external config edits.

The fix introduces `PropShapeRepositoryInterface` with two implementations. `EphemeralPropShapeRepository` is used during component validation and discovery; it calls the alter hook and collects results without writing to cache. `PersistentPropShapeRepository` extends `CacheCollector` and also implements `CacheTagsInvalidatorInterface`; it is registered as a service tagged `cache_tags_invalidator` and `needs_destruction`. The cache entry keyed by `canvas_prop_shape_repository` stores each computed `StorablePropShape` together with the cache tags that governed the decision. When any of those tags is invalidated, `PersistentPropShapeRepository::invalidateTags()` marks the affected shapes for re-resolution. On kernel destruction (`updateCache()`), it recomputes them via `EphemeralPropShapeRepository::getCandidateStorablePropShape()` and, if any shape changed, calls `ComponentSourceManager::generateComponents()` to regenerate `Component` config entities.

`CandidateStorablePropShape` now implements `RefinableCacheableDependencyInterface` (via `RefinableCacheableDependencyTrait`), so hook implementations can call `$storable_prop_shape->addCacheTags([...])` to register the tags that affect their decision. Canvas's own `ShapeMatchingHooks::mediaLibraryStorablePropShapeAlter()` adds `config:media_type_list` to cover the case where new MediaTypes are created after installation.

`PropShape::getStorage()` is removed entirely; all callers now inject and call `PropShapeRepositoryInterface::getStorablePropShape()`. `JsonSchemaType::computeStorablePropShape()` gains a `PropShapeRepositoryInterface` parameter so that `type: array` shapes can look up their item shape's storability through the repository rather than re-invoking the alter hook independently. Both repository classes are marked `@internal`.

## Upgrade

Rename all implementations of `hook_storage_prop_shape_alter` to `hook_canvas_storable_prop_shape_alter` and update the `#[Hook]` attribute accordingly:

```php
// Before
#[Hook('storage_prop_shape_alter')]
public function myStoragePropShapeAlter(CandidateStorablePropShape $storable_prop_shape): void { ... }

// After
#[Hook('canvas_storable_prop_shape_alter')]
public function myStorablePropShapeAlter(CandidateStorablePropShape $storable_prop_shape): void { ... }
```

If your implementation's decision depends on external configuration or state, declare the relevant cache tags so Canvas can automatically re-evaluate the shape when those tags are invalidated:

```php
#[Hook('canvas_storable_prop_shape_alter')]
public function myStorablePropShapeAlter(CandidateStorablePropShape $storable_prop_shape): void {
  // Declare what makes this decision stale.
  $storable_prop_shape->addCacheTags(['config:my_module_type_list']);

  $types = $this->entityTypeManager->getStorage('my_type')->loadMultiple();
  if (!empty($types)) {
    $storable_prop_shape->fieldWidget = 'my_custom_widget';
  }
}
```

The old hook name is deprecated in `canvas:1.0.0` and will be removed in `canvas:2.0.0`.

### Collaboration

The solution that shipped went through a substantive design pivot. mglaman and phenaproxima challenged the original cache-collector approach (MR !145) by pointing out that a cache miss alone cannot trigger recalculation of `prop_field_definitions`, which is stored inside `Component` config entities rather than in any central cache. larowlan proposed a more pragmatic path via `hook_field_config_insert` and `hook_field_config_update` (MR !298), which wim leers rejected as unscalable: there are arbitrarily many reasons a prop shape might change, and handling each case ad hoc would not serve contrib implementors. wim leers then pushed a commit demonstrating how moving prop-shape computation into a cache-tag-aware repository could work for both SDC and code components. MR !298 was closed and the effort consolidated into MR !145. penyaskito took over writing the test coverage and hit a multi-day debugging session caused by an orthogonal bug: `Component::save()` was being called unconditionally during discovery, producing spurious cache-tag invalidations that broke Dynamic Page Cache. That blocker was extracted into [#3556327](https://www.drupal.org/node/3556327). wim leers fixed a small bug in penyaskito's tests, refactored `ReadOnlyPropShapeRepository` into the `Ephemeral`/`Persistent` split with detailed reasoning, and added the Mermaid architecture diagram. penyaskito completed the final review and committed the result.
