## Summary

In the Canvas module, auto-save entries were being stored using `$entity->toArray()`, which includes computed fields. Computed fields are derived at runtime and are not actually stored, so including them in the auto-save snapshot could cause spurious hash mismatches and incorrect conflict signals during the `auto-saves/pending` polling cycle. This MR fixes the serialization step so that computed fields are excluded from auto-save storage and hash comparisons. The one exception is the `path` field type, which Drupal marks as computed but does store, so it is intentionally kept. Cypress tests are also updated to assert that polling the `auto-saves/pending` endpoint continues to return HTTP 200 and not unexpected error codes.

## Impact

No upgrade or configuration changes required.

## Technical details

A new private static method `AutoSaveManager::toStorableArray()` replaces the direct `$entity->toArray()` calls used in two places: `saveEntity()` (where the snapshot and `data_hash` are written) and `normalizeEntity()` (where the entity is diffed for conflicts). For `FieldableEntityInterface` entities the method iterates `getFields(include_computed: TRUE)` and skips any field whose definition returns `isComputed() === true`, unless the field item list is an instance of `PathFieldItemList`. The path field is special-cased because `\Drupal\path\Hook\PathHooks::entityBaseFieldInfo` marks it computed even though it is persisted. For non-fieldable entities the method falls back to `$entity->toArray()` unchanged. Without this fix, any computed field value that happens to differ between two entity loads (e.g. cache-warmed or lazily resolved fields) would produce a different hash and trigger a false conflict detection in the broader conflict-detection work being built in this issue series.
