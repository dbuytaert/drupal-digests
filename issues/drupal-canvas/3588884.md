## Summary

A regression introduced in Canvas 1.4.0 caused every `entity_autocomplete` field inside a component instance form to have its selected value rewritten into the `entity:<type>/<id>` URI scheme, even when that was wrong. Widgets like the Webform block's autocomplete (which expects a raw label/machine-name pair such as `Newsletter Signup (newsletter_signup)`) instead received a value like `entity:node/newsletter_signup`, making it impossible to save any component that contains such a field. This fix restores correct behaviour for all non-link autocomplete widgets while keeping URI resolution working for link widgets.

## Impact

No upgrade or configuration changes required.

## Technical details

The regression came from MR !981 ([#3586589](https://www.drupal.org/node/3586589)), which added a `resolveEntityUri()` call inside `TextFieldAutocomplete.tsx`. That call was gated only on `data-form-id === 'component_instance_form'`, so it applied to every `entity_autocomplete` in a component form, not just link URI fields.

The fix introduces an explicit opt-in marker. `ReduxIntegratedFieldWidgetsHooks::fieldWidgetSingleElementLinkDefaultWidgetFormAlter()` now stamps `data-canvas-resolve-entity-uri="true"` onto the `uri` element of the link widget via `$form['uri']['#attributes']`. In `TextFieldAutocomplete.tsx`, the condition changes from checking `record.target.dataset.formId` to checking `record.target.dataset.canvasResolveEntityUri === 'true'`, so only inputs that were explicitly marked by the PHP hook trigger URI resolution. All other `entity_autocomplete` inputs (config entities, reference fields, etc.) keep their raw autocomplete value untouched. A Cypress test covering a config-entity autocomplete (image style) was added to prevent re-regression.
