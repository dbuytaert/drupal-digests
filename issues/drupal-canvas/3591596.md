## Summary

Sites using the `frontend_editing_paragraphs` submodule alongside `paragraphs_edit` 4.x would hit an exception immediately on clicking the paragraph edit action in the frontend editing sidebar. The submodule previously hard-coded assumptions from paragraphs_edit 3.x (a custom `entity_edit` operation and a `root_parent` build argument) that no longer exist in 4.x. This fix adds runtime version detection so both major versions are handled correctly, and updates the `composer.json` constraint to allow `^3.0 || ^4.0`.

## Impact

No upgrade or configuration changes required.

## Technical details

The entire fix lives in `FrontendEditingParagraphsFormBuilder`. Previously both `buildEditForm` and `buildDeleteForm` unconditionally used the paragraphs_edit 3.x API: the `entity_edit` / `entity_delete` form operations, a `root_parent` build arg passed via `$form_state_additions['build_info']['args']`, and the module-specific routes `paragraphs_edit.edit_form` / `paragraphs_edit.delete_form`.

The fix detects the installed version at runtime by calling `$entityTypeManager->getDefinition('paragraph')->getFormClass('entity_edit')`. If that class is registered, paragraphs_edit 3.x is present and the old code path runs unchanged. If it returns nothing, paragraphs_edit 4.x is assumed: the standard `edit` / `delete` operations are used, no build args are needed (4.x's `ParagraphFormHelperTrait` auto-resolves `root_parent` inside `buildForm()`), and the form action is set to the standard entity routes `entity.paragraph.edit_form` / `entity.paragraph.delete_form`.

One reviewer (csakiistvan) noted that the delete-path version check uses `entity_edit` rather than `entity_delete`, because `frontend_editing_paragraphs` itself registers `entity_delete` unconditionally via `hook_entity_type_build`, which would make an `entity_delete` check unreliable. Using `entity_edit` as the sole discriminator avoids that trap.
