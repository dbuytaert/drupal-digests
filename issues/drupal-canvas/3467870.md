## Summary

Experience Builder (XB) can now handle SDC props declared as `type: array` in JSON Schema. Previously this threw a `LogicException` with "Support for array props is not yet implemented." With this change, array-typed props are mapped to multi-value Drupal fields, so a component such as a sparkline chart (array of integers) or an image gallery (array of image objects with `$ref`) can receive its data from a `StaticPropSource` or be matched against a `DynamicPropSource` field instance on a content entity. Previews of both example SDCs are shown working in the issue. Full editing support for multi-value field widgets is tracked separately in [#3546869](https://www.drupal.org/node/3546869).

## Impact

Affects module maintainers and site owners.

- **Config change:** New optional `cardinality` integer key added to `sourceTypeSettings` in `experience_builder.generated_field_explicit_input_ux`; `default_value` changed from a single typed scalar to a sequence of `field.value.*` items.
- **API change:** `StaticPropSource::generate()` has a new required `$cardinality` parameter in second position; `StaticPropSource::$fieldItem` (FieldItemInterface) replaced by `$fieldItemList` (FieldItemListInterface); `StructuredDataPropExpressionInterface::isSupported()` now requires `FieldItemListInterface` in its union type.
- **New API:** `StorablePropShape` and `CandidateStorablePropShape` gained a `cardinality` property; `SdcPropJsonSchemaType::ARRAY` now returns a fully populated `StorablePropShape` instead of NULL.

## Technical details

`SdcPropJsonSchemaType::computeStorablePropShape()` previously returned `NULL` (with a `@todo`) for `ARRAY` type. It now maps `type: array` JSON Schema props to multi-value field storage: the `items` sub-schema is resolved recursively to find the backing field type, and `maxItems` becomes the field cardinality (`CARDINALITY_UNLIMITED` when omitted). Only schemas whose keys are a subset of `['type', 'items', 'maxItems']` are supported; anything beyond that returns `NULL`.

`StaticPropSource` was refactored from holding a single `FieldItemInterface` (`$fieldItem`, public readonly) to a `FieldItemListInterface` (`$fieldItemList`). The internal factory method `conjureFieldItem()` became `conjureFieldItemList()`, which sets cardinality on the `FieldStorageDefinition` before creating the list. A `$cardinality` parameter was added in second position to `StaticPropSource::generate()` (breaking change). The `evaluate()` method dispatches on cardinality: single-cardinality sources pass the first `FieldItemInterface` to `Evaluator`; multi-cardinality sources pass the entire `FieldItemListInterface`. `getValue()` returns a scalar/object for cardinality 1 and an array for higher cardinality. The `FieldItemList::isEmpty()` quirk with empty-string values (e.g. a `list_string` field where `''` is an allowed value) is explicitly worked around by using `::count()` instead.

`Evaluator::evaluate()` and `doEvaluate()` now accept `FieldItemListInterface` and map over its items when one is passed. `StructuredDataPropExpressionInterface::isSupported()` was broadened from `EntityInterface|FieldItemInterface` to also accept `FieldItemListInterface`; all five concrete implementations (`FieldTypePropExpression`, `FieldTypeObjectPropsExpression`, `FieldPropExpression`, `ReferenceFieldTypePropExpression`, `ReferenceFieldFieldTypePropExpression`) were updated accordingly. `StorablePropShape` and `CandidateStorablePropShape` gained an optional `cardinality` property that is required (not NULL) for array shapes and forbidden on all others.

On the config schema side, `experience_builder.generated_field_explicit_input_ux` gained a new optional `cardinality` integer key in `sourceTypeSettings` (single-cardinality is the default and need not be stored). The `default_value` key changed from a single typed value to a `sequence` of `field.value.*` items, allowing multi-value defaults for array props. `DefaultRelativeUrlPropSource` was refactored to recurse into `type: array` schemas when rewriting example URLs. A pre-existing bug in `FloatItemOverride::generateSampleValue()` (`'values'` key corrected to `'value'`) was fixed in the same changeset.

## Upgrade

**Callers of `StaticPropSource::generate()`** must insert the new `$cardinality` argument in second position. Pass `NULL` to preserve single-cardinality behavior:

```php
// Before:
StaticPropSource::generate($expression, $field_storage_settings, $field_instance_settings);

// After:
StaticPropSource::generate($expression, NULL, $field_storage_settings, $field_instance_settings);
// Or, for an array prop with e.g. maxItems: 5:
StaticPropSource::generate($expression, 5, $field_storage_settings, $field_instance_settings);
```

**Code accessing `StaticPropSource::$fieldItem`** must switch to `$fieldItemList` (type `FieldItemListInterface`). For single-cardinality use, call `->fieldItemList->first()`.

**Implementations of `StructuredDataPropExpressionInterface::isSupported()`** must update the parameter type to include `FieldItemListInterface`:

```php
// Before:
public function isSupported(EntityInterface|FieldItemInterface $entity_or_field): bool;

// After:
use Drupal\Core\Field\FieldItemListInterface;
public function isSupported(EntityInterface|FieldItemInterface|FieldItemListInterface $entity_or_field): bool;
```
