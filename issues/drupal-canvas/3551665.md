## Summary

When building a Canvas content template using an SDC component that has both an image prop and a string prop, setting a static image and then linking the string prop to a dynamic field source caused the static image to silently disappear from the preview and from the saved configuration. This is now fixed. The image is correctly preserved when other props are linked to dynamic sources.

## Impact

No upgrade or configuration changes required.

## Technical details

The bug lived in `_updateExistingComponentValuesForLinking` inside the Canvas Redux slice. When a user linked a prop to a `DynamicPropSource`, that action reconstructed the `resolved` map by spreading `model.resolved` (the server-persisted snapshot) and then the new linking `values`. The problem was that form-state changes, such as a freshly selected static image, had not yet been flushed back into `model.resolved`. So when the spread ran, the image data was absent from `model.resolved` and got overwritten rather than preserved.

The fix reads the live form state from `state.formState['component_instance_form'].values` and passes it through `getPropsValues()` (imported from `@/components/form/formUtil`) together with the current `model` and component transforms. The resulting `propsValues` are spread into `resolved` between `model.resolved` and the incoming `values`, so any prop value held only in form state (the static image) is carried forward before the new dynamic-prop value is applied. wim leers confirmed during investigation that the corrupted second request was sending back the raw component schema defaults for the image prop instead of the user-selected value, which matched this form-state/model sync gap as the root cause.

### Collaboration

wim leers did the detailed forensic work that pinpointed the bug. By capturing two sequential API requests to `/canvas/api/v0/form/component-instance/...` and comparing their `form_canvas_props` payloads, he showed precisely what was being sent to the server: the second request replaced the user's chosen image with the raw component schema defaults, including validation-only fields like `jsonSchema` that have no place in a component instance form. That analysis, shared in comment [#16303266](https://www.drupal.org/node/16303266), gave bnjmnm the clear target needed to write the fix.
