# #3575644: `Regex` constraint on `string_long` field type intended to convey semantics causes "JIT stack limit reached" on long strings; make this pattern a no-op since it allows _anything_

## Summary

Sites using the Canvas module would see a "This value is not valid." error when saving content with a `string_long` field (plain long text) containing more than roughly 15,000 characters. The error gave no useful indication of the cause. This fix eliminates that failure: long multiline text in `string_long` fields now saves without validation errors.

## Impact

No upgrade or configuration changes required.

## Technical details

Canvas overrides `StringLongItem` via `StringLongItemOverride` and attaches a `Regex` constraint with the pattern `/(.|\\r?\\n)*/` to the field's `value` property. The pattern was not meant for validation; it exists purely as a semantic marker so `EntityFieldPropSourceMatcher` can identify multi-line string component props by comparing constraint values against field property constraints.

The problem is that the alternating group `(.|\\r?\\n)` under a greedy `*` quantifier causes PHP's PCRE JIT engine to exhaust its stack on strings longer than roughly 15,000 characters, returning `PREG_JIT_STACKLIMIT_ERROR` (code 6). Symfony's `RegexValidator` treats any `preg_match()` failure as a non-match and adds a violation, producing the opaque error message.

The fix cannot simply replace the pattern, because changing it byte-for-byte would break the constraint-comparison logic in `EntityFieldPropSourceMatcher`. Instead, a new `JitSafeRegexValidator` class extends Symfony's `RegexValidator` and early-returns without calling `preg_match()` when the constraint pattern is exactly `JsonSchemaType::patternToPcre('(.|\r?\n)*')`. This validator is registered in `CanvasServiceProvider` under the key `'\\' . RegexValidator::class` (with a leading backslash) because Drupal's `ConstraintValidatorFactory` resolves validators by class name and `RegexConstraint::validatedBy()` returns the fully qualified name with that prefix. The service is registered as non-shared, matching the requirement for all constraint validators. `StringLongItemOverride` is updated to derive the pattern through `JsonSchemaType::patternToPcre()` rather than a hardcoded string literal, so the two sides remain byte-identical.

A kernel test `StringLongItemOverrideTest::testLongMultilineValueIsValid()` reproduces the original failure by validating an 80,000-character multiline string on a `string_long` field and asserting zero violations. A `@todo` in `JitSafeRegexValidator` marks it for removal once multi-line prop matching is reworked to avoid needing a matches-everything Regex marker altogether (tracked separately).
