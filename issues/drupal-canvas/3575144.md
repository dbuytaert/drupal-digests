## Summary

Canvas now exposes unstable HTTP API endpoints for external tools to create, read, update, and delete Canvas pages with their component trees. These v0 endpoints enable the Canvas CLI tool to synchronize pages between Drupal and local development environments using JSON as the bridge format. The API returns complete page data including nested components with their inputs, slots, and parent relationships. While marked unstable and subject to change, these endpoints follow the structure planned for a future public v1 API.

## Impact

Affects module maintainers.

- **New API:** Four new OAuth scopes (`canvas:page:read`, `canvas:page:create`, `canvas:page:edit`, `canvas:page:delete`) added for Canvas page API access.

## Technical details

The implementation adds three new routes to the existing Canvas HTTP API: GET and PATCH for individual pages (`/canvas/api/v0/content/canvas_page/{canvas_page}`), plus enhanced POST for creating pages with full data or duplicating existing ones. All routes are marked with the `canvas_external_api` option, enabling OAuth authentication via the Canvas OAuth module.

The GET endpoint returns complete page entities including the `components` field serialized as an array of component tree items with decoded JSON inputs. The PATCH endpoint validates incoming data and throws a 409 Conflict error if auto-saved data exists, forcing users to resolve conflicts in the UI first. The POST endpoint now accepts optional page data (title, status, path, components) to create fully populated pages, not just blank pages or duplicates.

Four new OAuth scopes map to standard Canvas page permissions: `canvas:page:read` uses `access content`, while create/edit/delete scopes use their respective permissions. An update hook installs these scopes on existing sites. The `CanvasOauthAuthenticationProvider` was extended to apply to these content entity routes, checking route names rather than just config entity types.

The `ApiContentControllers` class gained private `normalizeWithComponents()` method to serialize component trees consistently across GET, POST, and PATCH responses. This ensures the API returns structured objects for inputs rather than JSON strings, matching consumer expectations. Test coverage includes functional tests for all CRUD operations, OAuth scope tests, and kernel tests for GET/PATCH/POST with various component tree structures.
