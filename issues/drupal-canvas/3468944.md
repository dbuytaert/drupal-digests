## Summary

The built-in `image` SDC that ships with Drupal Canvas (formerly Experience Builder) has been restructured to follow SDC best practices. The redundant nested `image` object prop was removed in favour of flat top-level props (`src`, `alt`, `width`, `height`, `sizes`, `loading`). The template now correctly outputs the auto-injected `attributes` variable, making the component compatible with lazy-loading APIs, ARIA injection, and other systems that attach HTML attributes. Responsive image support was also added via `srcset` generation and a `lazy` loading default. The component is now marked `noUi: true`, signalling that it is a developer building block for inclusion in other SDCs rather than a user-selectable element in the Canvas component browser.

## Impact

Affects module maintainers.

- **API change:** The `canvas:image` SDC prop schema changed from a single nested `image` object to flat top-level props (`src`, `alt`, `width`, `height`, `sizes`, `loading`); callers must update `with` blocks and template references accordingly.

## Technical details

The original `image.component.yml` wrapped all image data inside a single `image` object prop, accessed in the Twig template as `{{ image.src }}`, `{{ image.alt }}`, etc. As noted in the issue thread by pdureau, this created a redundant nesting layer: the component and its sole prop described exactly the same thing. The fix (landed in [#3535453](https://www.drupal.org/node/3535453)) promotes all properties to the top level, so the template accesses `{{ src }}`, `{{ alt }}`, `{{ width }}`, `{{ height }}`, `{{ sizes }}`, and `{{ loading }}` directly.

The Twig template in `components/image/image.twig` was rewritten to output `{{ attributes }}` at the bottom of the `<img>` tag, consuming the `Attribute` object that SDC automatically injects into every component. Two custom Twig filters, `getWidth`/`getHeight` (to derive intrinsic dimensions from the source when not supplied) and `toSrcSet` (to generate a `srcset` attribute), were added alongside `file_url()` to resolve Drupal stream wrapper URIs. The `loading` prop defaults to `lazy` and `sizes` defaults to `auto 100vw`.

The `noUi: true` flag in the component definition tells Canvas to exclude this component from the editor's component browser. A code comment in the YAML explains the deliberate decision not to reuse the `json-schema-definitions://canvas.module/image` JSON Schema reference: since this SDC is composed into other SDCs, flat top-level props give a better developer experience than a single nested object would. The issue discussion also produced a concrete follow-up: defining `attributes` explicitly as a prop in SDC YAML (as several XB components had done) is incorrect because SDC already injects it automatically, and using a PHP namespace as a prop type is not JSON Schema compliant (tracked in [#3470575](https://www.drupal.org/node/3470575)).

## Upgrade

If you have custom SDCs that include `canvas:image` and access props with the old nested pattern, flatten the references:

```twig
{# Before #}
<img src="{{ image.src }}" alt="{{ image.alt }}" width="{{ image.width }}" height="{{ image.height }}">

{# After — props are now top-level #}
{% include 'canvas:image' with { src: ..., alt: ..., width: ..., height: ... } %}
```

Also remove any explicit `attributes` prop definition from your own SDC YAML files if it uses a PHP namespace type such as `Drupal\Core\Template\Attribute`; SDC injects `attributes` automatically and that type string is not JSON Schema compliant.

### Collaboration

pdureau provided the critical technical grounding for the whole issue. In comment #5 they explained that every SDC template already receives an auto-injected `attributes` object, documented why the nested `image` prop was architecturally redundant, and argued that plain HTML elements do not need SDC wrappers at all because Drupal already has `#type => 'html_tag'` renderables. That analysis directly shaped the final component design: flat props, `attributes` output, and the `noUi: true` marker that separates developer building blocks from end-user components. ctrladel added a counterpoint in comment #16 defending the single-prop nesting as a future-proofing technique and noting that explicit `attributes` output felt like an SDC antipattern inherited from Drupal's render layer, a view that lauriii ultimately weighed in on to close the issue.
