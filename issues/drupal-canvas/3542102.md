## Summary

Canvas (Experience Builder) previously allowed content to be published even when a `block` component instance contained invalid field values, such as an entity reference pointing to a non-existent entity. Publishing content in that state would cause the page to crash on render. This bug is now fixed: invalid block component inputs are detected and block publishing until the user corrects the values.

## Impact

No upgrade or configuration changes required.

## Technical details

The root cause was in `BlockComponent::validateComponentInput()`. When a block form contains an `EntityAutocomplete` element and the user enters an invalid entity reference, Drupal's `EntityAutocomplete::validateEntityAutocomplete()` nullifies the submitted value during form validation. The original invalid string entered by the user is lost. When publishing triggers validation outside a form context, there is nothing left to fail against, so the publish proceeds with corrupted data.

The fix moves the `$form_state->setValue($parents, NestedArray::getValue($input, $parents))` call so that the user-provided raw input is restored to form state for *all* validation errors, not only for `#required` field errors as it was before. This means the original (potentially invalid) value is preserved in the temp store so that later calls to `validateComponentInput()` during the publish flow can still detect and reject it.

A guard was also added in the test-only `CanvasTestBlockForm::submitForm()` to handle the case where the restored value has not yet been transformed into a numeric entity ID by `EntityAutocomplete`, preventing a PHP type error during the test's block submit handler. The unit tests were updated to assert two violations (a primitive-type constraint plus the domain-specific message) and a new integration-style test simulates submitting an invalid entity-reference value and confirms violations correctly bubble up through `$page->validate()`.
