## Summary

Canvas now correctly handles symmetric content translation for content entities with component tree fields. Previously, saving a non-default-language translation that omitted non-translatable component inputs (such as `cta1href` or `attributes` in an SDC hero component) caused a `LogicException` or `EntityStorageException` about required properties. The module now stores all inputs on every translation row and synchronizes non-translatable input values from the default translation automatically at write time, so translators and developers do not need to duplicate non-translatable data in each translation.

## Impact

No upgrade or configuration changes required.

## Technical details

Drupal's `content_translation.synchronizer` (`FieldTranslationSynchronizerInterface`) handles synchronizing non-translatable field column groups, including Canvas's `tree` group (`uuid`, `parent_uuid`, `slot`, `component_id`, `component_version`). Non-translatable keys inside the `inputs` JSON blob property of `ComponentTreeItem` are not a column group and therefore fall outside that mechanism, causing `ComponentTreeItem::preSave()` to throw because required inputs were absent.

The fix introduces `ComponentTreeFieldSymmetricalTranslationSynchronizer`, registered in `CanvasServiceProvider` as a Symfony service decorator on `content_translation.synchronizer`. After the standard column-group sync, the decorator's `synchronizeComponentInstanceInputs()` iterates every component tree field in symmetric mode (tree synced, inputs translatable), determines translatable keys via `ComponentInputs::getTranslatableInputKeys()` (which calls `ComponentInputs::resolveConfigSchemaMapping()` to read config schema translatability markers), then writes `array_merge(target_inputs, non_translatable_from_default)` back to each non-default translation's field item. This ensures non-translatable inputs from the default translation always win while preserving the translation's own translatable values.

A special case handles net-new component instances added to the default translation: because the target translation has no prior inputs for that instance (core's `createMergedItem()` may have placed inputs from a different component instance at that delta), the decorator uses the default translation's full input set instead of the (incorrect) target inputs. Rendering, JSON:API, and the `inputs_resolved` computed property all read `inputs` identically for any translation with no merge logic at read time. MR !1219 adds ADR #12 to `docs/adr/` documenting this write-time synchronization architecture and its tradeoffs (including that translatable inputs on newly added component instances are intentionally left empty in non-default translations, requiring manual translator input).
