## Summary

Forms embedded inside block components on Canvas pages were broken. Submitting a form would produce an error message ("Drupal\Core\Form\EnforcedResponseException occurred during rendering of component...") instead of completing the submission. The same problem affected Ajax-based form submissions. This is fixed in Canvas 1.0.0-rc1.

## Impact

No upgrade or configuration changes required.

## Technical details

Canvas' `RenderSafeComponentContainer::renderComponent()` wraps all component rendering in a broad `catch (\Throwable $e)` block intended to prevent one component's failure from crashing an entire page. However, Drupal core's `FormBuilder` uses `EnforcedResponseException` and `FormAjaxException` as control-flow signals during normal form processing, not as error conditions. Catching them caused Canvas to swallow the form's redirect or Ajax response and log a spurious rendering failure instead.

The fix adds a `catch (EnforcedResponseException | FormAjaxException $e) { throw $e; }` clause before the broad `\Throwable` catch in two places: `src/Element/RenderSafeComponentContainer.php` (in `renderComponent()`) and in the `\Drupal\canvas\ComponentTreeItemList` field item list, which has its own parallel exception-handling path. Both catches carry a `@todo` to remove them once the underlying core issue ([#2367555](https://www.drupal.org/node/2367555)) is resolved. A new functional test `BlockComponentFormTest` was added, using core's `\Drupal\block_test\Plugin\Block\TestFormBlock` to verify that a form block renders, submits, and returns a 200 with the expected output when embedded in a Canvas page.

This pattern mirrors what Drupal's Layout Builder already does in `\Drupal\layout_builder\Plugin\Block\FieldBlock`, which added the same `EnforcedResponseException` rethrow as part of [#2995893](https://www.drupal.org/node/2995893).
