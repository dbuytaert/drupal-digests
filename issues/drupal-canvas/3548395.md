## Summary

When building a `ContentTemplate` in Canvas and placing an SDC component, the "linker" control that lets you bind a component prop to a content field was missing for several field widget types: Media Library widgets (image fields), checkbox widgets (boolean fields), `LinkWidget` (URI fields), and `ImageWidget`. Only text-based widgets with simple `<label>` elements showed the linker correctly. This fix makes the linker appear for all affected widget types.

## Impact

No upgrade or configuration changes required.

## Technical details

Two separate root causes were addressed across the two commits.

**First commit (4acf84d7): Media Library fieldset legend.** The Media Library widget renders its label as a `<fieldset>` `<legend>` rather than a `<label>` element. The existing mechanism for injecting prop-linker data targets `#label_attributes` on elements with `#title_display`, which only reaches a `<label>`. The fix introduces `canvas_stark_preprocess_fieldset__media_library_widget()` to encode and forward `#prop_link_data` directly to the Twig template as `prop_link_data`. The `media-library-widget.html.twig` template was updated to use a new `<canvas-drupal-label>` tag (registered as `DrupalFormElementLabel` in `twigToJSXComponentMap`) in place of the plain `<canvas-text>` tag, and it passes the data via a `direct_linker_data` attribute. `DrupalFormElementLabel` was extended to accept `directLinkerData` as an explicit prop alongside the existing `attributes.prop_link_data` path, so the linker renders regardless of which path supplied the data.

**Second commit (d2e5ee66): Recursive prop-link propagation.** For all other broken widget types, the problem was that `#prop_link_data` set on a top-level field element was not reaching nested child elements that own a visible `#title_display`. The old code in `canvas_stark_preprocess_field_multiple_value_form()` tried to patch only the very first child element, which missed compound widgets entirely. The replacement is an `#after_build` callback, `moveSuggestionsToLabel()`, added to the component-instance form when the entity is a `ContentTemplate`. It calls `processElementTreeLinkerLabels()`, which walks the full form element tree recursively: whenever a node carries `#prop_link_data` that data is inherited by the subtree, and any descendant element that has `#title_display` receives the data in its `#label_attributes`, making the linker appear next to whichever label is ultimately rendered.
