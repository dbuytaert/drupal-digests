## Summary

The Canvas module previously discarded all Drupal cache metadata (cache tags, contexts, max-age) when a prop source resolved a value from a referenced entity or computed field. This had two consequences: changing a referenced entity would not invalidate the render cache, so end users would continue seeing stale content; and, more critically, revoking entity or field access would not clear cached output either, meaning unauthorized data could persist in the Render cache, Dynamic Page Cache, Page Cache, or an external reverse proxy until a full cache flush. This fix ensures cacheability flows from every prop source evaluation into the component render array, so access changes take effect immediately.

## Impact

Affects module maintainers and site owners.

- **Security:** Component instances rendering data from referenced entities could serve stale, access-revoked content from cache indefinitely; cacheability is now correctly bubbled from all prop source evaluations.
- **API change:** `AdapterInterface::adapt()` return type changed from `mixed` to `EvaluationResult`; `UrlRewriteInterface::rewriteExampleUrl()` return type changed from `string` to `\Drupal\Core\GeneratedUrl`; `Evaluator::evaluate()` and `PropSourceBase::evaluate()` now return `EvaluationResult` instead of `mixed`.

## Technical details

A new immutable value object, `EvaluationResult` (`Drupal\canvas\PropExpressions\StructuredData\EvaluationResult`), now wraps both the resolved value (`public readonly mixed $value`) and its cache metadata (implementing `CacheableDependencyInterface`). `Evaluator::evaluate()` returns `EvaluationResult` instead of `mixed`. `Evaluator::validateAccess()` now returns `AccessResultInterface` rather than `void`, so the access-check cacheability is incorporated into the result rather than discarded. A private helper `permanentCacheabilityUnlessSpecified()` was added to handle the case where the host entity or field does not itself implement `CacheableDependencyInterface`.

Every `AdapterInterface` implementation (`DayCountAdapter`, `ImageAdapter`, `ImageAndStyleAdapter`, `ImageUriAdapter`, `UnixTimestampToDateAdapter`) was updated to return `EvaluationResult`. Image-loading adapters now add the loaded `FileInterface` entity as a cacheable dependency; `ImageAndStyleAdapter` and `ImageUriAdapter` additionally apply the `url.site` cache context when generating absolute URLs.

`GeneratedFieldExplicitInputUxComponentSourceBase` gained a protected static helper `getResolvedPropsAndCacheability()` that merges the cacheability of every resolved prop value and returns both the raw value array and a combined `CacheableMetadata`. `JsComponent::renderComponent()` and `SingleDirectoryComponent::renderComponent()` now call this helper and apply the merged cacheability to the component render array.

`RenderSafeComponentContainer::handleComponentException()` now accepts a `CacheableMetadata $component_exception_cacheability` argument (derived from the component's render array at the call site) and applies it to the error build. Without this, a crashing component would produce an output with no cacheability, meaning the error could be cached indefinitely, and correct cacheability would never reach callers.

Two `DataType` plugins, `ComputedEntityCanonicalRelativeUrl` and `ComputedUrlWithQueryString`, now implement `CacheableDependencyInterface` via a new internal `ComputedDataTypeWithCacheabilityTrait`. A `discoveryCacheTags` parameter was added to the `ComponentSource` plugin attribute; `SingleDirectoryComponent` declares `['component_plugins']` and `JsComponent` declares `['config:js_component_list']`, making example-value URLs vary correctly when component metadata changes without a corresponding config entity update.

`UrlRewriteInterface::rewriteExampleUrl()` return type changed from `string` to `\Drupal\Core\GeneratedUrl` so URL-rewriting component sources can also carry cacheability metadata.

The two follow-up commits fixed MySQL/MariaDB CI regressions: auto-increment user IDs differed from SQLite because a test was relying on the sequence starting value. The fix was to create the referenceable user with an explicit high ID (`uid: 1103448`) so expectations are database-backend-independent.

## Upgrade

Code implementing `AdapterInterface::adapt()` must wrap the return value in `EvaluationResult`:

```php
// Before:
public function adapt(): mixed {
    return $someValue;
}

// After:
use Drupal\canvas\PropExpressions\StructuredData\EvaluationResult;
use Drupal\Core\Cache\CacheableDependencyInterface; // if needed

public function adapt(): EvaluationResult {
    // Pass a CacheableDependencyInterface as second argument when the adapted
    // value depends on a cacheable object (e.g. a loaded entity).
    return new EvaluationResult($someValue, $cacheableDependency);
    // Or with no extra cacheability:
    return new EvaluationResult($someValue);
}
```

Code implementing `UrlRewriteInterface::rewriteExampleUrl()` must change its return type from `string` to `\Drupal\Core\GeneratedUrl`:

```php
// Before:
public function rewriteExampleUrl(string $url): string {
    return $url;
}

// After:
use Drupal\Core\GeneratedUrl;
use Drupal\Core\Url;

public function rewriteExampleUrl(string $url): GeneratedUrl {
    // For a fully-qualified URL:
    return (new GeneratedUrl())->setGeneratedUrl($url);
    // For a site-relative URL that needs cache tags:
    return Url::fromUri('base:/' . $path)->toString(TRUE)->addCacheTags([...]);
}
```

Code calling `PropSource::evaluate()` or `Evaluator::evaluate()` must access `->value` to get the raw resolved value, and may read cacheability from the returned object:

```php
// Before:
$resolved = $propSource->evaluate($entity, is_required: FALSE);

// After:
$result = $propSource->evaluate($entity, is_required: FALSE);
$resolved = $result->value;
// Optionally bubble cacheability:
CacheableMetadata::createFromObject($result)->applyTo($build);
```
