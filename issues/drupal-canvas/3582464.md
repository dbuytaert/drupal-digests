## Summary

Canvas config-defined component trees (ContentTemplate, PageRegion, Pattern) now use component instance UUIDs as their sequence keys in Drupal's active config storage, replacing the previous position-based keys like `0` or `0:the_body:0`. This means translations applied to a component instance's inputs now survive when that instance is moved to a different position in the tree. Before this fix, moving any component instance would change its config property path and cause Drupal's `LanguageConfigOverride` system to silently orphan or misapply the translation. Sites using Canvas with multilingual content templates or page regions must run the included database updates.

## Impact

Affects site owners.

- **Database update:** Three post_update hooks (0017) migrate Pattern, PageRegion, and ContentTemplate component trees from positional string keys to component instance UUID keys.
- **Config change:** The `canvas.component_tree` config schema sequence type drops `orderby: key`; exported YAML now uses position-prefixed UUID keys (e.g., `0:the_body:0:{uuid}`) transformed by `ComponentTreeConfigEntityTransformer`.

## Technical details

Previously, `ContentTemplate`, `PageRegion`, and `Pattern` each overrode `set()` to call `generateComponentTreeKeys()`, which computed position-encoded sequence keys (e.g., `0:the_body:0`) immediately on value assignment. Drupal's config translation system stores `LanguageConfigOverride` data keyed by property path, so repositioning a component instance changed its path and caused `ConfigFactoryOverrideBase::filterOverride()` to delete the override entirely. A test written during investigation (comment #23) confirmed this behavior is automatic and unavoidable with position-based keys.

The fix consolidates key generation into `ComponentTreeConfigEntityBase::preSave()` via the private method `asDeterministicallyAndTranslatableKeyedComponentTreeSequence()`, which re-keys each sequence item by its `uuid` field using `array_combine(array_column(..., 'uuid'), ...)`. Moving key generation from `set()` to `preSave()` was suggested by tedbow (#17, #18): it means unsaved (auto-saved) entity versions retain the last-saved positional keys in memory, so existing `LanguageConfigOverride` data continues to apply correctly until an entity is formally saved. The per-entity `set()` overrides in `ContentTemplate`, `PageRegion`, and `Pattern` are removed entirely.

To preserve human-readable, diff-friendly exported YAML, a new `ComponentTreeConfigEntityTransformer` event subscriber handles `ConfigEvents::STORAGE_TRANSFORM_EXPORT` and `ConfigEvents::STORAGE_TRANSFORM_IMPORT`. On export, each UUID key is prefixed with position information (e.g., `0:the_body:0:3a76bf4f-9306-43e6-ba8f-cb4b5b6459df`), making tree structure visible in config diffs. On import, `ksort()` recovers the correct ordering even if YAML entries were manually reordered (e.g., during a merge-conflict resolution), then the position prefix is stripped back to bare UUIDs. The `canvas.component_tree` config schema type drops `orderby: key` because ordering is now preserved by value order in active storage. Language override config (stored in non-default collections) is explicitly excluded from both transformations.

Three `post_update` hooks (`canvas_post_update_0017_pattern_component_tree_sequence_keys`, `canvas_post_update_0017_page_region_component_tree_sequence_keys`, `canvas_post_update_0017_content_template_component_tree_sequence_keys`) migrate existing entities via `CanvasConfigUpdater::needsConfigEntityWithComponentTreeSequenceKeysUpdate()`, which also triggers a deprecation when old non-UUID keys are detected. `FieldConfig::$default_value` is explicitly excluded from the migration because it must remain a zero-indexed list. This is Step 3 of a three-step series; Step 1 ([#3586342](https://www.drupal.org/node/3586342)) and Step 2 ([#3582478](https://www.drupal.org/node/3582478)) handled config schema and Config Translation UI work respectively.

## Upgrade

Database updates are required. Run `drush updb` (or equivalent) to apply the three `post_update` hooks.

If you have custom code that reads raw component tree data from a `ComponentTreeConfigEntityBase` subclass and expects the old positional string keys, update it to use UUID keys instead:

```php
// Before (old positional keys):
$tree = $content_template->get('component_tree');
$first = $tree['0'];           // position-based key
$nested = $tree['0:the_body:0']; // nested position-based key

// After (UUID keys):
$tree = $content_template->get('component_tree');
// Keys are now component instance UUIDs:
// e.g. $tree['3a76bf4f-9306-43e6-ba8f-cb4b5b6459df']
// Use array_values() if you only need ordered iteration:
$first = array_values($tree)[0];
```

Exported config YAML (sync storage) will have position-prefixed UUID keys after the next `drush cex`. This is the transformed export format and is correct; do not manually edit these keys. The import transformer restores UUID-only keys on `drush cim`.

### Collaboration

The thread contains a substantive design debate that directly shaped what shipped. wim leers identified the root cause: Drupal's `LanguageConfigFactoryOverride::onConfigSave()` calls `ConfigFactoryOverrideBase::filterOverride()`, which silently deletes a `LanguageConfigOverride` when its keys no longer match the saved config entity — confirmed by a test he wrote on the fly (comment #23). This ruled out approach #13.1 (updating overrides via a `ConfigEvents::SAVE` subscriber) and led to approach #13.3 (UUID keys at rest, position-prefixed keys on export only).

tedbow's contribution in comments #17 and #18 unlocked the chosen approach: by moving key generation from `set()` to `preSave()`, unsaved (auto-saved) versions of entities keep the previous saved keys in memory, meaning existing `LanguageConfigOverride` data applies correctly until a real entity save occurs. wim leers described this as solving two "only-#13.1, not-#13.3" problems simultaneously.

phenaproxima challenged whether position information needed to be in the exported keys at all, suggesting UUID-only export keys would still be alphanumerically sortable. wim leers demonstrated this was wrong: sorting by UUID discards the original position ordering, causing silent data loss for any tree where the UUIDs don't sort in the same order as their intended positions (comment #27).
