## Summary

The Canvas page builder editor can now navigate between different pages and content without triggering a full browser reload. Previously, clicking a different page in the Pages panel or following a publish-error link to a component on another page forced a complete page reload, which lost any transient state such as the list of publish errors. Navigation is now handled by React Router internally, making the transition feel instant. One important behavioral side effect: the undo/redo history is cleared each time the editor navigates to a different entity or template, so edits made on a previous page cannot be undone after switching away. Follow-up issue [#3566074](https://www.drupal.org/node/3566074) tracks a more complete "action/inverse action" undo system that would eventually restore cross-page undo capability.

## Impact

No upgrade or configuration changes required.

## Technical details

The core change is that `LayoutLoader` and `PageDataFormRenderer` now read `entityId` and `entityType` from React Router's `useParams` and pass them as arguments to `useGetPageLayoutQuery` and `useGetPageDataFormQuery` (using `skipToken` when the params are absent). Both components now call `refetch()` in a `useEffect` that depends on those params, so switching the route automatically fetches the new entity's data without unmounting and remounting the whole app.

Navigation calls that previously used `window.location.href` (in `AddTemplateDialog`, `Topbar`, `TemplateRoot`, `TemplateListItem`, `ContentGroup`, `CodeComponentItem`, and others) have been replaced with a centralized `useEditorNavigation` hook. That hook exposes typed navigation functions (`navigateToEditor`, `navigateToTemplateEditor`, `navigateToCodeEditor`) and URL builder functions (`urlForEditor`, `urlForTemplateEditor`, `urlForCodeEditor`) built on React Router's `navigate()` API. `SidebarNode`'s `href` prop was renamed to `to` and the underlying element changed from `<a>` to React Router `<Link>`, and page list items in `ContentGroup` similarly switched to `<Link>`.

A new `useNavigationListener` hook, called from `App.tsx`, monitors the route params `entityType`, `entityId`, `bundle`, and `viewMode`. When any of them changes between renders, it dispatches the undo/redo history clear action. On the Redux side, the `undoable` wrappers for `layoutModelReducer` and `pageDataReducer` now declare a `clearHistoryType` (`@@redux-undo/layoutModel_CLEAR_HISTORY` and `@@redux-undo/pageData_CLEAR_HISTORY`). The existing `undoRedoActionIdMiddleware` was extended to intercept actions matching the `_CLEAR_HISTORY` pattern and dispatch the corresponding `clearUndoRedoHistory` thunk. `setInitialized` dispatch is also now excluded from the undo filter so that loading a new entity's data does not pollute the undo stack.

`DrupalPathWidget` was refactored from a `useRef`-based `autoGenerateOn` flag to explicit state (`userHasEditedPath`, `lastGeneratedPath`, `isInitialized`), preventing unwanted path overwrites when returning to a previously edited entity. `useDrupalBehaviors` gained a `loading` parameter; callers pass the fetching state so that Drupal behaviors are not attached to an incomplete form DOM. A new Playwright test suite in `Routing.spec.ts` covers direct URL visits, the absence of full page reloads (verified by injecting a `window.navigationMarker`), undo stack clearing, and page data form isolation. The follow-up [#3572292](https://www.drupal.org/node/3572292) applied the same SPA navigation patterns to the Publishing UX.
