## Summary

Canvas now lets you navigate between editing different pages and content without full browser reloads. The editor stays open while you switch pages through the Pages panel or navigation menu, preserving your editing context and making workflows feel faster and more fluid. This is especially helpful when working with publish errors across multiple pages or switching between templates. However, undo/redo history is now cleared when you navigate to a different page, which is a temporary trade-off until the undo system is redesigned to work across page boundaries.

## Impact

No upgrade or configuration changes required.

## Technical details

This feature implements single-page application (SPA) navigation within the Canvas editor. Previously, navigating between pages triggered full browser reloads because entity data was only fetched on initial page load. Now `LayoutLoader` watches route parameters (`entityId`, `entityType`) and refetches layout data dynamically when they change, while `useNavigationListener` monitors route changes and clears undo/redo history on navigation.

The implementation faced complexity around undo/redo. When navigating between pages, the undo stack contained complete layout snapshots for the previous page. If a user undid an action after navigating away, the browser would navigate back but then the server would asynchronously return the newer autosaved state, overwriting the undo. The team decided to clear the undo/redo stacks on navigation rather than implement a complex state reconciliation system, which matches the existing full-reload behavior.

New hooks `useEditorNavigation`, `useStableCallback`, and `useSyncTitle` coordinate navigation and state updates. The `componentAndLayoutApi.getPageLayout` query now accepts `{ entityId, entityType }` parameters instead of reading them from global context. The path widget auto-generation logic was rewritten to track user edits versus auto-generated paths, preventing overwrites during navigation.

Navigation now uses React Router's `<Link>` components throughout the UI (sidebar nodes, page list items, code component items), and the preview request waits for Drupal AJAX operations to complete before sending to avoid race conditions.

## Collaboration

Jesse Baker drove this implementation after documenting the undo/redo complexity that blocked the initial approach. The team decided to merge the feature early despite outstanding code review comments so more people could test it in real workflows and surface edge cases. This "merge and iterate" approach prioritized getting the UX improvement into hands quickly, with follow-ups created for the undo/redo redesign and publishing UX improvements. QA testing identified three minor issues before merge, two of which were resolved immediately.

## Lessons learned

- When implementing SPA navigation that swaps state asynchronously, full undo/redo stacks that store complete snapshots become fragile. Navigating to a new view loads fresh data from the server that overwrites any restored undo state, creating race conditions. Either clear the undo stack on navigation or implement action/inverse-action patterns that can survive data reloads. *(via jessebaker)*
- When a large architectural change has been sanity-checked but not exhaustively reviewed, merging early for broader testing can surface issues faster than holding it in review. Ship with known follow-ups documented and be prepared to revert if necessary. *(via hooroomoo)*
