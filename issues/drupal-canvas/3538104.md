## Summary

Canvas can now navigate between editing different pages or content without a full browser reload. This makes moving around the editor feel instant instead of requiring a page refresh each time you switch pages. When you click a page in the navigation panel or follow a link to edit a different component, the interface updates seamlessly without losing your place. However, navigating to a different page now clears the undo/redo history, so you cannot undo changes made to a previous page after switching away from it.

## Impact

No upgrade or configuration changes required.

## Technical details

This change implements single-page application (SPA) navigation in the Canvas editor by refactoring how routes handle entity and template switching. Previously, navigating between entities like `/editor/canvas_page/1` to `/editor/canvas_page/2` triggered a full page reload because layout and page data were only fetched on initial mount.

The implementation adds a `useNavigationListener` hook that monitors route parameter changes and triggers cleanup actions. The `LayoutLoader` and `TemplateLayout` components now use `useParams` to watch `entityId`, `entityType`, `bundle`, and related parameters, and call `refetch()` when these change instead of relying on mount-only behavior.

Navigation utilities are centralized in `useEditorNavigation`, which exposes functions like `navigateToEditor()`, `navigateToTemplateEditor()`, and `navigateToCodeEditor()`. These functions use React Router's `navigate()` API instead of `window.location.href`. The hook also provides `urlFor*` builder functions that components can use with React Router `<Link>` components, replacing hardcoded `href` attributes.

To prevent conflicts with ongoing AJAX requests, the preview request logic in `Preview.tsx` now uses a polling mechanism that checks `isAjaxing()` before dispatching requests, ensuring Drupal form behaviors complete before state updates occur.

The undo/redo system presented a significant challenge. When navigating between pages, the old approach of storing entire layout snapshots in an undo stack would cause the server's auto-saved state to overwrite the undo state asynchronously. The solution adopted here is to clear the undo/redo history whenever the user navigates to a different entity or template. Redux-undo now supports a `clearHistoryType` action, and the `UndoRedoActionCreators` includes a `clearHistory()` method. A new `useSyncTitle` hook invalidates the content list cache after successful preview updates when the title or path alias changes.

The `useDrupalBehaviors` hook now accepts a `loading` parameter to prevent attaching behaviors while data is still being fetched. The `DrupalPathWidget` logic was refactored to track whether the user manually edited the path or if it remains auto-generated from the title, preventing unwanted path overwrites during navigation.

## Collaboration

jessebaker led this complex feature, navigating significant challenges with the undo/redo stack and race conditions. After documenting the problem space in detail (comment #13), the team decided to clear undo/redo history on navigation as an incremental improvement that doesn't regress current behavior, while deferring a proper "action/inverse action" undo system to [#3566074](https://www.drupal.org/project/canvas/issues/3566074). The team merged this early despite incomplete code review to enable broader testing and unblock other large merge requests like [#3562026](https://www.drupal.org/project/canvas/issues/3562026), with follow-up MRs addressing review feedback. ## Lessons learned

- When implementing SPA navigation that loads remote state, asynchronous data fetching can overwrite synchronously restored state (like undo/redo). Load remote data only when the navigation wasn't triggered by a state restoration action, or accept that the navigation invalidates the restoration stack. *(via jessebaker)*
- When implementing undo/redo by storing snapshots of entire application state, the stack size grows unbounded and becomes incompatible with multi-user editing. An action/inverse-action pattern where each operation stores only its reversal scales better and enables collaborative features. *(via jessebaker)*
- When a large architectural change enables early discovery of integration bugs and unblocks other critical work, merge it after basic sanity checks rather than waiting for perfect code review. The increased testing surface and reduced merge conflicts often outweigh the risk of reverting. *(via hooroomoo)*
