## Summary

This META issue defines the requirements, user journeys, designs, and implementation plan for adding conflict detection and resolution to the Canvas module. When a Canvas user has unpublished autosave changes on an entity and that entity is updated outside Canvas (via the Drupal admin UI, CLI, config import, or code), Canvas currently has no way to surface or resolve that conflict. The planned feature will flag conflicting entities in the "Review Changes" panel, prevent publishing of conflicted items until the conflict is resolved, and provide a new side-by-side comparison UI where the user can choose to keep either the published version or their autosaved version. The same side-by-side UI will also be available for non-conflicting changes as a general "review before publish" tool. Two foundational child issues have already landed: [#3588038](https://www.drupal.org/node/3588038) (refactoring the `auto-saves/pending` response to wrap entries in a top-level `data` key) and [#3588546](https://www.drupal.org/node/3588546) (adding content entity cache tags to that endpoint).

## Impact

Affects module maintainers.

- **New API:** New HTTP 409 response shape on `auto-saves/pending` with top-level `data` and `errors` keys; new conflict resolution PATCH endpoint; preview endpoint gains a version-selection parameter

## Technical details

The core challenge is detecting when the entity underlying a Canvas autosave entry has been modified outside Canvas. The plan extends `AutoSaveManager` to store an `original_hash` at autosave-entry creation time; on subsequent reads the hash is compared against the current entity state to detect drift. Config entities (Content Templates, Code Components, Regions, Patterns) use a hash of all properties except `status` and `label`; content entities (Canvas Pages) treat any revision-ID change as a conflict.

When conflicts are detected, the `auto-saves/pending` endpoint returns HTTP 409 with both a `data` array (all entries) and a top-level `errors` array whose items include a `conflict` object carrying `conflictId`, `status`, and `timestamp`. This schema follows JSON:API conventions, a structural change already shipped in [#3588038](https://www.drupal.org/node/3588038). The `original_hash` value itself is never sent to the client.

Conflict resolution will be exposed through a new HTTP PATCH call to the auto-save API supplying a `conflictId` and the chosen resolution. Resolving in favour of the autosaved version marks the conflict resolved without discarding the user's work; resolving in favour of the published version removes the autosave entry entirely. The side-by-side visual comparison will reuse the existing Canvas preview endpoint with a new parameter to request either the autosaved or the published rendering. Entities without meaningful visual previews (e.g. global CSS entities) will fall back to a text-based diff using config-export or default-content-export representations.
