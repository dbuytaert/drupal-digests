## Summary

The Canvas AI component agent received a substantial overhaul that fixes five bugs and adds support for creating components with slots. Previously, the agent could generate broken code by importing from the deprecated `next-image-standalone` package, importing unsupported external npm packages, producing components where a prop and a slot shared the same name, returning an unhelpful hardcoded error string when no component was open for editing, and it had no mechanism to define slots at all. All five issues are resolved in this change. Site builders can now ask Canvas AI to create code components that include slots, and those slots immediately appear in the Canvas editor's Slots panel. The AI will also respond with natural-language guidance when editing is unavailable, instead of a fixed error string.

## Impact

Affects module maintainers and site owners.

- **Config change:** Post-update hook `canvas_ai_post_update_0008_reimport_component_agent` overwrites the stored `ai_agents.ai_agent.canvas_component_agent` system prompt and tool usage limits from module install config.
- **API change:** Three `canvas_ai` tokens removed (`selected_component_required_props`, `menu_fetch_source`, `json_api_module_status`); replaced by a single `component_agent_dynamic_state` token assembled at request time.
- **New API:** `CreateComponent` and `EditComponentJs` function call plugins now accept an optional `slots_metadata` context (JSON array); `GetJsComponent` now returns `props_metadata` and `slots_metadata` in agent-readable format.

## Technical details

The core structural change is a new dynamic prompt system. Instead of evaluating runtime state (which component is open, whether JSON:API is enabled, which menu source is configured) inside a single monolithic system prompt, `CanvasAiPageBuilderHelper::generateComponentAgentDynamicPromptSection()` reads `modules/canvas_ai/src/DynamicPrompts/canvas_component_agent/dynamic_prompt_parts.yml` at request time and assembles only the sections that match the current state. The result is injected via a single new `[canvas_ai:component_agent_dynamic_state]` token. The three former individual tokens `[canvas_ai:selected_component_required_props]`, `[canvas_ai:menu_fetch_source]`, and `[canvas_ai:json_api_module_status]` are removed from `CanvasAiHooks` entirely.

Slots support is implemented through a new `AiGeneratedJsComponentPropsAndSlotsTrait` shared by `CreateComponent`, `EditComponentJs`, and `GetJsComponent`. Both create and edit plugins now accept an optional `slots_metadata` context (a JSON array where each entry has `id`, `name`, and an optional `example`). `transformSlotsMetadata()` validates and converts agent-supplied slot objects into the stored `JavaScriptComponent` slot definition shape, enforcing that each slot's `id` is the strict camelCase of its `name`. When an edit drops existing props or slots, `buildRemovedMetadataWarning()` appends a warning to the tool's success message so the agent can report the change. `GetJsComponent` now also returns `props_metadata` and `slots_metadata` in the agent-readable format using `buildPropsMetadataFromStored()` and `buildSlotsMetadataFromStored()`, so the agent sees the full current state of a component before editing.

On the PHP side, `CanvasBuilder::getSupportedLibraries()` no longer includes the `importing_packages` entry that encouraged importing arbitrary npm packages via CDN. The return value is now JSON-encoded before being placed in the token context (previously it was passed as a PHP array, which rendered as the string `"Array"` in the system prompt). On the JavaScript side, `AiWizard.tsx` adds a `slotsMetadataHandler` that applies `slots_metadata` from AI responses to the Redux code-editor state, and the `selected_component` sent with each request now prefers the URL route parameter `codeComponentId` over the Redux `machineName` to avoid a race condition immediately after component creation. A post-update hook `canvas_ai_post_update_0008_reimport_component_agent` overwrites the stored system prompt and tool usage limits from the module's install config for existing sites.

## Upgrade

**Custom canvas_ai token references:** The tokens `[canvas_ai:selected_component_required_props]`, `[canvas_ai:menu_fetch_source]`, and `[canvas_ai:json_api_module_status]` are removed. Replace any references in custom agent YAML or code with `[canvas_ai:component_agent_dynamic_state]`.

**Customized system prompt:** If you edited the component agent system prompt directly in the Drupal configuration UI, the post-update hook will overwrite those changes with the updated prompt from the module. The recommended way to extend or alter agent behavior going forward is through the Context Control Center or custom event subscribers (as noted in the hook's log warning).

**Slot metadata shape for custom tool integrations:** If custom code calls `ai_agent_create_component` or `ai_agent_edit_component_js`, you can now supply a `slots_metadata` context value:
```json
[
  { "id": "children", "name": "Children", "example": "<p>Place components here</p>" },
  { "id": "footer", "name": "Footer" }
]
```
The `id` must be the strict camelCase of `name` (e.g. `"CTA Content"` → `"ctaContent"`), or the tool returns an error.
