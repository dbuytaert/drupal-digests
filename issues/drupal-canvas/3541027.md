## Summary

A new Templates panel is now available in the Canvas left sidebar. It lists all node bundles (content types) with their assigned templates in an accordion, shows permission-gated contextual menus per bundle ("Edit fields" and "Delete content type" links that open Drupal core pages in a new tab), and provides a delete action per template with a confirmation dialog. An "Add new template" button opens a dialog where a content type and view mode can be selected to create a new template via the Canvas HTTP API. For the 1.0 scope only the "full" (page template) view mode is creatable through this UI. The entire Templates panel, including its sidebar separator, is gated behind the `canvas_dev_mode` module and will not appear unless that module is installed.

## Impact

Affects module maintainers and site owners.

- **Config change:** New `entityTypeLabels` key added to `drupalSettings.canvas` by `BootController`, mapping entity type IDs to bundle label maps.
- **New API:** New RTK Query hooks `useCreateContentTemplateMutation` and `useDeleteContentTemplateMutation` for POST and DELETE on `/canvas/api/v0/config/content_template`.

## Technical details

The sidebar gating is handled in `SideMenu.tsx`: both the Templates button and the separator before it now read `canvasSettings.devMode` (from `drupalSettings`) and set `hidden` accordingly. `TemplateList.tsx` was substantially expanded with two new components: `BundleListItem` now renders a `ContextMenu` and `DropdownMenu` using `UnifiedMenu` with "Edit fields" and "Delete content type" items driven by the new `editFieldsUrl` / `deleteUrl` fields the API returns; `TemplateListItem` wraps each `SidebarNode` in a context menu with a single "Delete template" option that opens a confirmation `Dialog` and calls `deleteContentTemplateMutation`. The "Add new template" form lives in `AddTemplateDialog` inside `Templates.tsx`; it uses `useGetViewModesQuery` to populate the view mode dropdown (filtering the list so only `full` is shown) and `useCreateContentTemplateMutation` to POST to `/canvas/api/v0/config/content_template`. The `createContentTemplate` endpoint name was also added to the exclusion list in `withAutoSavesInjection` so the middleware does not attempt to inject auto-save data into that mutation.

On the PHP side, `ApiConfigControllers` gains two new constructor dependencies: `AccessManagerInterface` and `AccountProxyInterface`. For every bundle in the content template list response, the controller now checks access to `entity.$entity_type_id.field_ui_fields` and `entity.$bundle_entity_type_id.delete_form` for the current user. Both URLs are always present in the response (required properties per the updated OpenAPI spec), but they are `null` when the current user lacks the corresponding permission. The access check results are merged into `$additional_cacheability` via `addCacheableDependency`; this surfaces `config:core.extension` as a new cache tag because route access checks depend on enabled modules. Individual bundle config entity tags (e.g. `config:node.type.article`) are then filtered out since the already-present list cache tag (e.g. `config:node_type_list`) makes them redundant. The second commit (`ee84ac41`) added the missing `else` branches that set the URLs to `null`, the cacheability bubbling, the cache-tag deduplication, and fixed the OpenAPI spec to declare both fields as required rather than optional. `BootController` was also updated to pass `entityTypeLabels` (a map from entity type ID to bundle label map) inside `drupalSettings`, giving the frontend the data it needs to populate the content type dropdown without an extra HTTP request.
