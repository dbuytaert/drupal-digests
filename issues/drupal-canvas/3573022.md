# #3573022: [upstream] Data loss: `drush config:import` deletes config (e.g. code component + component config entities) and bypasses config system data integrity checks

- **Project:** Drupal Canvas
- **Type:** Critical bug report
- **Version:** 1.x-dev
- **Status:** Fixed
- **Discussion:** [#3573022](https://git.drupalcode.org/project/canvas/-/work_items/3573022) · 3 contributors · 18 comments
- **Followers:** 4

## Summary

Running `drush config:import` could silently delete in-use Canvas code components without triggering Canvas's built-in safety mechanisms, leaving dangling references in page content and producing a white screen of death (WSOD) on every page that referenced those components. There was no way to recover without directly editing the database. Canvas now blocks any config import that would delete a component still in use on a page, giving site owners a clear error message and keeping the site intact. Unused components can still be removed via config import.

## Impact

No upgrade or configuration changes required.

## Technical details

The root cause is in Drupal core's `ConfigImporter::importConfig()`, which uses the low-level `Config::delete()` to remove configuration entries rather than calling `ConfigEntityBase::delete()`. That bypass means `ConfigEntityBase::preDelete()` is never invoked, and therefore `Component::onDependencyRemoval()` never runs. That method is what Canvas relies on to switch an in-use component to its fallback source; without it, deleting a `canvas.js_component.*` or `canvas.component.js.*` config entity via `drush config:import` leaves live page content pointing at configuration that no longer exists.

The fix adds `ComponentConfigImportValidator`, a final class extending `ConfigImportValidateEventSubscriberBase`, registered as a service in `canvas.services.yml`. During `ConfigEvents::IMPORT_VALIDATE` it inspects the deletion list for two danger vectors: (A) a `Component` config entity is itself scheduled for deletion, and (B) non-component configuration that a retained `Component` depends on is being deleted (detected via `ConfigManagerInterface::findConfigEntityDependenciesAsEntities()`). For every endangered `Component` the subscriber calls `ComponentAudit::hasUsages()` for both published revisions and auto-save drafts; if either returns `true`, the import is aborted via `ConfigImporter::logError()` with an explanatory message naming the affected component.

This is explicitly a stopgap until core is fixed. The class docblock references two upstream issues: one to route config-import deletions through `onDependencyRemoval()` and one (`#2414951`) that would add this class of detection to core itself, at which point the subscriber could be removed rather than just adjusted.

### Collaboration

wim leers traced the failure through both Drush and Drupal core source code before any patch existed. He identified that `ConfigImporter` calls the storage-layer `Config::delete()` rather than the entity-layer `ConfigEntityBase::delete()`, which is what bypasses `onDependencyRemoval()`. He then found Drupal core's `content_moderation` module using `ConfigEvents::IMPORT_VALIDATE` for a related-but-different purpose and concluded that the same mechanism was the only mitigation Canvas could apply unilaterally. bernardm28's original report confirmed, in response to wim leers' direct question, that both `canvas.js_component.foo` and `canvas.component.js.foo` were wiped together, making the two-vector design of the validator necessary.
