## Summary

Canvas Workbench and the Canvas CLI now treat global regions (header, footer, etc.) as first-class items alongside pages and content templates. Developers place region JSON specs in a `regionsDir` directory, run `canvas pull --include-regions` or `canvas push --include-regions` to two-way sync them with Drupal, and optionally add a `src/layout.jsx` file that wraps Workbench page and content template previews with the surrounding regions. If the layout file is absent, previews continue rendering without surrounding regions, so existing projects are unaffected until they opt in.

## Impact

Affects module maintainers and site owners.

- **Database update:** `canvas_oauth_post_update_0006_page_region_scope` installs a new `canvas:page_region` OAuth scope gated by the `administer page template` permission.
- **New API:** New `regionsDir` and `layoutPath` config keys in `canvas.config.json`; new `--include-regions` CLI flag for `pull`, `push`, and `validate`; new `ApiService` methods `listRegions`, `getRegion`, `createRegion`, `updateRegion`, `deleteRegion` targeting `/canvas/api/v0/config/page_region`; `Region` component exported from `drupal-canvas@^0.4.0`.

## Technical details

Two new keys are added to `canvas.config.json`: `regionsDir` (default `./regions`) and `layoutPath` (default `./src/layout.jsx`). Region spec files in `regionsDir` use the same element tree JSON format as page specs. The filename (without extension) becomes the region machine name; the Drupal theme is implicit and resolved server-side from the default theme when pushing. Push semantics are full-sync: remote regions absent from the local `regionsDir` are deleted.

The `ApiService` class gains `listRegions()`, `getRegion()`, `createRegion()`, `updateRegion()`, and `deleteRegion()` methods, all targeting the `/canvas/api/v0/config/page_region` REST endpoint. The `--include-regions` / `CANVAS_INCLUDE_REGIONS=true` option is wired into the `pull`, `push`, and `validate` commands. A new `createRegionsPullTask()` function and push utilities (`prepareRegions`, `pushRegions`, `collectRegionResults`, `validateRegions`) parallel the existing page and content-template paths.

A new `canvas:page_region` OAuth scope is installed by `canvas_oauth_post_update_0006_page_region_scope`, gated by the `administer page template` permission. The `canvas_oauth` authentication provider now protects `PageRegion` config entities alongside `ContentTemplate`, `BrandKit`, and others.

The layout file must export a default React component accepting `children`. The `Region` component from `drupal-canvas` (now at `^0.4.0`) slots in a region by machine name; regions with no matching spec file render as nothing, and a `fallback` prop can supply a placeholder during development.

Internally, the `authored-element-utils` module is consolidated into a new `authored-elements` module that exposes shared helpers (`componentTreeToAuthoredElementMap`, `authoredElementMapToComponentTree`, `jsonRenderSpecToAuthoredElementMap`) used by all three entity types. A new `ConfigComponentTreePayload` type and `stripNullableKeysForConfigComponentTree()` function handle the fact that config-entity component trees (regions, content templates) reject `null` values for `parent_uuid`, `slot`, and `label`, while page content entities tolerate them. The now-redundant `authoredPropToServer()` pass-through (which was only a no-op for pages/regions) is removed from the content-template utilities.

## Upgrade

After running database updates, the new OAuth scope is available:

```bash
drush updb
```

To opt in to region sync, add to `canvas.config.json`:

```json
{
  "regionsDir": "./regions",
  "layoutPath": "./src/layout.jsx"
}
```

Create region specs in `regions/`:

```json
// regions/header.json
{
  "status": true,
  "elements": {
    "header": {
      "type": "js.header",
      "props": { "darkVariant": true },
      "slots": { "branding": ["header-logo"] }
    },
    "header-logo": {
      "type": "js.logo",
      "props": { "linkToFrontPage": true }
    }
  }
}
```

Create `src/layout.jsx` to wrap Workbench previews with regions:

```jsx
import { Region } from 'drupal-canvas';

export default function Layout({ children }) {
  return (
    <>
      <Region name="header" />
      <main>{children}</main>
      <Region name="footer" />
    </>
  );
}
```

Pull or push regions with the new flag:

```bash
npx canvas pull --include-regions
npx canvas push --include-regions
```

### Collaboration

The design discussion shaped the final approach significantly. effulgentsia proposed three options for how to surface regions in Workbench, and in comment [#16522896](https://www.drupal.org/node/16522896) described a conversation with rikki_iki in which she suggested Option 2 (regions shown as part of page previews rather than in isolation) and specifically proposed using React Router concepts — a `layout` with an `outlet` — rather than Drupal's region terminology in the local codebase. wotnak pushed back in comment [#16582626](https://www.drupal.org/node/16582626), arguing that purely using an outlet without a local concept of regions would make two-way CLI sync difficult, and proposed the hybrid approach that landed: local region JSON specs that sync like pages, combined with a local `layout.jsx` for Workbench rendering. wim leers flagged in comment [#16584111](https://www.drupal.org/node/16584111) that making `PageRegion` implement `CanvasHttpApiEligibleConfigEntityInterface` carried significant security implications and requested it be extracted to a separate blocking issue, which became [#3589801](https://www.drupal.org/node/3589801).
