## Summary

After a Canvas deployment, browsers or edge caches holding old copies of the `clsx.js` or `tailwind-merge.js` JavaScript chunks could throw a runtime error like "The requested module './clsx.js' does not provide an export named 'c'", breaking any Code Component that uses those libraries. This happened because those chunks were loaded via relative paths that bypassed Canvas's import-map cache-busting mechanism. The fix centralizes handling in the `astro-hydration` build so all import-map libraries are always loaded through the import map with proper cache-busting query strings, eliminating the stale-cache errors.

## Impact

No upgrade or configuration changes required.

## Technical details

The `astro-hydration` build bundles `drupal-canvas`, which in turn imports `clsx` and `tailwind-merge`. By default, Rollup created shared chunks for those transitive dependencies using relative path imports (e.g., `import {c as o} from "./clsx.js"`). Import maps only intercept bare specifiers, so relative-path imports bypassed Canvas's cache-busting query strings. A second problem compounded this: Rollup's minified export aliases (e.g., `export {i as c}`) are not stable across builds. A browser serving a cached old chunk with different aliases caused imports to fail with a `SyntaxError`.

An initial fix (commit `13a8f384`) added `noExternal: ['clsx', 'tailwind-merge']` to `packages/drupal-canvas/tsdown.config.ts`, inlining those libraries into the `drupal-canvas/utils` bundle. This was documented in ADR-0007. It resolved the symptom but was considered wrong-level: it required every nested package to carry its own workaround and introduced minor code duplication.

The final approach (commit `91f6ee79`, superseding the first) reverts that change and fixes the problem at its source in `ui/lib/astro-hydration/astro.config.mjs`. The Rollup `external` option is replaced with a function. The function reads `astro-hydration`'s own `package.json` dependencies and filters out build-time-only entries listed under `canvas.buildOnly` (`astro` and `@astrojs/preact`) to arrive at the set of import-map libraries. For each import matching that set, the function bundles it when the import originates directly from `astro-hydration/src/`, when it is an internal import within the same package, or when the parent is a Vite/Rollup virtual wrapper. In all other cases (imports from nested dependencies such as `drupal-canvas`) it marks the import as external, causing Rollup to emit a bare specifier that the import map intercepts with its cache-busting query string. `clsx` and `tailwind-merge` were added as direct dependencies of `astro-hydration`. ADR-0008 documents this decision and supersedes ADR-0007.

### Collaboration

After the first fix was committed and the issue was closed, wotnak proposed a substantially better approach: instead of patching each nested dependency, configure `astro-hydration`'s Rollup `external` option as a function that conditionally marks bundled libraries external based on where the import originates. balintbrews agreed it was better in every way, adopted the idea, and extended it by deriving the library list dynamically from `package.json` rather than hardcoding it, then superseded the first ADR with a new one documenting the revised decision.
