## Summary

In the Canvas editor, users who only have `administer content templates` or `administer code components` permissions were unable to reach the Templates or Manage Library panels because the side menu was not rendered at the `/canvas` root route. They had to first select a page (which could itself trigger a 403 if they lacked page-editing permissions), opening a maze of steps just to reach a panel they had permission to use. After this fix, the side menu is visible as soon as `/canvas` loads, regardless of whether a page is being edited, so users with template or code-component permissions can navigate directly to the relevant panel.

## Impact

No upgrade or configuration changes required.

## Technical details

The root cause was that `SideMenu`, `PrimaryPanel`, and `Dialogs` were only rendered inside specific route elements (`/editor/:entityType/:entityId`, `/template/...`) and not on the index/welcome route at `/canvas`. A new `UiShell` wrapper component was introduced in `AppRoutes` that composes `SideMenu`, `PrimaryPanel`, and `Dialogs` together; every route element (canvas root, editor, template, code editor) now wraps its inner content with `UiShell`, so these shell components are always mounted.

`SideMenu` was refactored to use `useParams()` alongside `useLocation()` to compute `hasActiveEditorFrame`: true only when an entity ID is present in the URL (a page in the editor, or a preview entity in a template route). The "Add" (library) and "Layers" menu items are now hidden when `hasActiveEditorFrame` is false rather than when the code-editor path is detected. A `useEffect` in `SideMenu` also closes those panels automatically if they are active but the editor frame is no longer loaded. Panel-management logic that previously lived in `SideMenu` (switching to `manageLibrary` on code-editor entry) was relocated to a `useEffect` in `MosaicContainer`, which also cleans up on unmount.

The default active panel in `primaryPanelSlice` changed from `'layers'` to an empty string, so no panel opens automatically on the canvas root. `PrimaryPanel` now conditionally renders its header and content only when `activePanel` is truthy. The `localStorage` persistence was narrowed to only remember `'pages'` or `'templates'` as the active panel across navigation; editor-specific panels (`'layers'`, `'library'`) are no longer persisted. Playwright and Cypress test helpers were updated accordingly: a new `goToCanvasRoot()` helper navigates to `/canvas` and waits for `canvas-side-menu` and `canvas-topbar`; tests that previously assumed the Layers panel was open by default now call `openLayersPanel()` explicitly.
