## Summary

The Canvas CLI `push` and `pull` commands now sync page path aliases and meta descriptions in addition to title and components. Previously, when pushing a local page spec, the path was always taken from the remote record and description was never transferred at all. After this change, both fields are stored in the local spec file and round-trip correctly through the API.

## Impact

Affects module maintainers and site owners.

- **Config change:** The authored page spec JSON schema now requires a `path` field alongside `title` and `elements`.
- **New API:** The `Page` interface and `ApiService.createPage`/`updatePage` gain a required `description: string` field; `path` is now `string` (not `string | null`) in those method signatures.

## Technical details

The issue builds on [#3575874](https://www.drupal.org/node/3575874), which first added pages to the CLI commands. Two fields were missing from that initial implementation: `path` and `description`.

On the TypeScript CLI side, the `Page` interface gains a `description: string` field. `pageToAuthoredSpec` is refactored to build a shared `meta` object containing `uuid`, `title`, `path`, and `description`, which is spread into both the empty-components and populated-components return values. `preparePages` reads `description` and `path` from the local JSON spec (both optional, defaulting to `''`). `pushPages` now passes `description` and `page.path` (from the spec file) when calling `ApiService.createPage` and `ApiService.updatePage`, replacing the previous behaviour of using `remotePage.path` on update and a hard-coded empty string on create. The `path` parameter type in both API service methods also tightens from `string | null` to `string`. The JSON schema (`page-spec.schema.json`) adds `path` to the `required` array and adds an optional `description` string property. `parsePageSpec` adds both `path` and `description` to its `allowedTopLevelKeys` set.

On the Drupal PHP side, `ApiContentControllers` adds `description` to the field loop that checks edit access during PATCH handling and passes it through on POST entity creation. A new private method `normalizeWithMetadataAndComponents` wraps the existing `normalizeWithComponents`, appending the `description` field value from the entity (guarded by a `hasField` check and a `view` access check) and moving `links` to the end of the array. All three controller actions (GET single, PATCH, POST/duplicate) switch from `normalizeWithComponents` to `normalizeWithMetadataAndComponents`. The OpenAPI spec adds `description` (type string, "The meta description.") to the page response schema and to the example payloads.
