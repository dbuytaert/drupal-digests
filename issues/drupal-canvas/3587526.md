## Summary

The Canvas module now actively prevents publishing entities that have unresolved conflicts detected in the auto-save layer. When the `canvas_dev_cd` feature-flag module is enabled, selecting and submitting conflicted items for publishing is rejected by the server with an HTTP 409 response, rather than silently allowing the operation to proceed. Items that were already selected before a conflict is detected are automatically unselected in the UI, and a "Resolve conflict" option appears in the per-item dropdown menu.

## Impact

Affects module maintainers.

- **New API:** New `AutoSaveEntityConflictConstraint` validation constraint class and `AutoSaveManager::getUnresolvedConflictForEntity()` method for detecting unresolved auto-save conflicts on Page entities; `AUTO_SAVE_CONFLICT_KEY` constant moved from `ApiAutoSaveController` to `AutoSaveManager`.

## Technical details

The enforcement is split across the GET and POST sides of `ApiAutoSaveController`. On GET, conflict data was already included in the response when `canvas_dev_cd` is enabled. The MR extends this to the POST (publish) path: `getPublishableAutoSaves()` and `getAllAutoSaveList()` are now called with `with_conflicts: $conflict_detection_dev_mode` instead of hardcoded `FALSE`, and a new early-exit check in `post()` returns HTTP 409 with per-item error details if any selected auto-save entry carries a `conflict_id`.

Per-entity validation during the publish loop now routes through a new private method `getConflictAwareContentEntityViolations()`. For `Page` entities in conflict-detection mode this method suppresses Drupal core's `EntityChangedConstraint` violation (which would otherwise block publishing a resolved conflict) and replaces it with the new `AutoSaveEntityConflictConstraint` marker constraint when an unresolved conflict is present. `AutoSaveEntityConflictConstraint` extends `EntityChangedConstraint` and carries the conflict ID as a violation parameter. `ApiExceptionSubscriber::violationToJsonApiStyleErrorObject()` recognises this marker and injects the `conflict_id` and the `ItemEntityUpdatedExternally` error code into the JSON:API-style error meta.

Supporting changes: `AUTO_SAVE_CONFLICT_KEY` (`'conflict_id'`) is moved from `ApiAutoSaveController` to `AutoSaveManager` as the single canonical location; a new `getUnresolvedConflictForEntity(Page)` method is added to `AutoSaveManager`; `ensureEntityIsValid()` is promoted from a static to an instance method so it can delegate to `getConflictAwareContentEntityViolations()`; `EntityConstraintViolationList::findByCodes()` is overridden to avoid a `TypeError` caused by the parent's `new static($violations)` call not matching the subclass constructor signature; and `ApiControllerBase::validateExpectedAutoSaves()` now returns HTTP 409 instead of 422 when the violation set contains the `ItemEntityUpdatedExternally` error code. The OpenAPI spec gains a `409` entry on the publish endpoint and a `422` description.

## Upgrade

No upgrade steps are required for existing sites. The conflict-blocking behaviour is gated entirely behind the `canvas_dev_cd` feature-flag module and is inactive when that module is absent.

Integrators calling the Canvas publish API (`POST /canvas/api/v0/auto-saves`) should now handle `409 Conflict` in addition to `422 Unprocessable Entity`. A 409 response means one or more of the submitted auto-save keys has an unresolved conflict; the body follows the same JSON:API error structure and includes `conflict_id` in each error's `meta` object:

```json
{
  "errors": [
    {
      "detail": "The entity has been updated externally.",
      "source": { "pointer": "<auto-save-key>" },
      "code": 409,
      "meta": {
        "entity_type": "canvas_page",
        "entity_id": "2",
        "label": "Page 2",
        "conflict_id": "<revision-id>",
        "api_auto_save_key": "<auto-save-key>"
      }
    }
  ]
}
```
