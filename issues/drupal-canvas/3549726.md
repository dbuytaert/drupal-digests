## Summary

The `category` field has been removed from Canvas `Component` config entities. It was originally used to automatically place components into UI folders, but its role had gradually eroded and it remained only as a static value stored on each config entity. Folder placement now happens dynamically via a new `ComponentSourceInterface::determineDefaultFolder()` method, so each component source plugin decides its default folder at runtime rather than storing it in configuration. Existing `Component` config entities that still carry `category` will have the property stripped during a database update, and will trigger a deprecation warning before that update runs.

## Impact

Affects module maintainers and site owners.

- **Config change:** `category` key removed from the `canvas.component.*` config schema; post-update `canvas_post_update_0009` strips it from existing Component entities.
- **API change:** `ComponentInterface::getCategory()` removed; `ComponentSourceInterface::determineDefaultFolder(): string` added (default in `ComponentSourceBase` returns `'Other'`); `computeCurrentComponentMetadata()` no longer returns a `category` key.
- **Deprecation:** Any `Component` config entity with a non-null `category` value triggers `E_USER_DEPRECATED` (deprecated in canvas:1.0.2, removed in canvas:2.0.0) until the post-update runs.

## Technical details

The `category` property was declared on `Component` (the config entity class) and its interface (`ComponentInterface`), with a corresponding key in the `canvas.component.*` config schema. It was consumed in `Component::postSave()` to auto-create a `Folder` for new components, and in `Component::preDelete()` to remove deleted components from their folder. The issue also identified a bug: `preDelete()` used the stored `category` to find the folder, but a user could have moved the component to a different folder, making the lookup incorrect.

The fix removes the `category` property and `ComponentInterface::getCategory()` entirely, and adds `ComponentSourceInterface::determineDefaultFolder(): string` to the plugin interface. `ComponentSourceBase` provides a default implementation returning `'Other'`. `BlockComponent` overrides it to return the block plugin definition's `category`, and `SingleDirectoryComponent` returns the SDC's `category` (mapped from its `group` metadata). The `preDelete()` bug is corrected by replacing the category-based folder lookup with `Folder::loadByItemAndConfigEntityTypeId()`, which searches for whichever folder actually contains the component. The `computeCurrentComponentMetadata()` return type on `ComponentCandidatesDiscoveryInterface` is narrowed to `array{'label': string, 'status'?: bool}`, dropping the `category` key.

A post-update function `canvas_post_update_0009_unset_category_property_on_components` iterates all `Component` entities using `ConfigEntityUpdater` and calls `CanvasConfigUpdater::unsetComponentCategoryProperty()` to null out the field, emitting an `E_USER_DEPRECATED` warning for any component that still carries a non-null value.

## Upgrade

**If you implemented `ComponentSourceInterface`**, add `determineDefaultFolder()` to your plugin class:

```php
public function determineDefaultFolder(): string {
    // Return the folder name this source places components into by default.
    return 'My Custom Folder';
}
```

**If you called `Component::getCategory()`**, replace it with `$component->getComponentSource()->determineDefaultFolder()`.

**If you referenced `category` in Component config YAML**, remove the key — existing entries will be cleared by `canvas_post_update_0009_unset_category_property_on_components` and will emit a deprecation until then:

```yaml
# Before
id: sdc.my_module.my-component
source: sdc
source_local_id: my_module:my-component
category: My Category   # <-- remove this line

# After
id: sdc.my_module.my-component
source: sdc
source_local_id: my_module:my-component
```

**If you called `computeCurrentComponentMetadata()`**, stop reading the `category` key from the returned array; it is no longer present.
