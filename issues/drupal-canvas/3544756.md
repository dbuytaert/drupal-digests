## Summary

When a content author cleared an optional SDC component prop (for example, a "CTA text" field) in the Canvas component form, saved, and then reopened the form, the field would reappear pre-filled with the component schema's default value. The cleared value was stored correctly, but the form always fell back to the schema default when a prop was absent from the stored input, so the empty state was never reflected in the UI. After this fix the form correctly shows the field as empty when the author has explicitly cleared it.

## Impact

No upgrade or configuration changes required.

## Technical details

`GeneratedFieldExplicitInputUxComponentSourceBase::buildComponentInstanceForm()` receives its `$inputValues` parameter from `::clientModelToInput()`, not the raw client request. `::clientModelToInput()` intentionally omits optional props that evaluate to `NULL` to avoid storing empty values. The original code handled a missing prop with `$client_model[$sdc_prop_name] ?? $default_prop_sources[$sdc_prop_name]`, which meant that any deliberately-cleared optional prop would silently reinstate the schema default when the form was rendered.

The first commit (e511280f) tightened the fallback: when a prop key is absent from the input array, `$default_value` is now `NULL` rather than the schema default, preventing the unwanted reinstatement. It also added a Playwright regression test that asserts the heading field does not repopulate with the schema default after a page reload.

The second commit (4b62024f) from wim leers refined the fix further after identifying that the intermediate version was still calling `$this->getDefaultExplicitInput()` unnecessarily (since `::uncollapse()` already handles the default-source logic). The final form simply calls `$this->uncollapse($inputValues[$sdc_prop_name] ?? NULL, $sdc_prop_name)`, fully trusting `::clientModelToInput()` to supply correct values for all required props. An `assert()` formalises this contract: a prop may be absent from `$inputValues` only if it is not in the set of required props. The parameter was also renamed from `$client_model` to `$inputValues` across the `@internal` `ComponentSourceInterface` and all its implementations (`BlockComponent`, `Fallback`, `InvalidSlots`, `Personalization`, `GeneratedFieldExplicitInputUxComponentSourceBase`) to reflect that the method receives processed input rather than the raw client payload. The wim leers comment ([#16276565](https://www.drupal.org/node/16276565)) traces the full history: the fallback to schema defaults was introduced in [#3518336](https://www.drupal.org/node/3518336), `::clientModelToInput()` was evolved through [#3528396](https://www.drupal.org/node/3528396), [#3529788](https://www.drupal.org/node/3529788), and [#3537945](https://www.drupal.org/node/3537945), and the two responsibilities were never reconciled until this issue.
