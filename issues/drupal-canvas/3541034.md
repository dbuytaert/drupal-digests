## Summary

The Canvas module can now display and save the component instance form (for editing static component props) when editing a `ContentTemplate` config entity, not just when editing fieldable content entities like pages. Previously the props editor panel was stubbed out with a placeholder component (`DummyPropsEditForm`) for template routes; it now uses the real `ComponentInstanceForm` backed by a dedicated API endpoint. Changes made through the form are reflected live in the template preview.

## Impact

Affects module maintainers.

- **API change:** `ComponentInstanceForm::buildForm()` `$entity` parameter type broadened to `?EntityInterface`; new optional `$preview_entity` parameter added.
- **New API:** New route `canvas.api.form.component_instance.content_template` at `/canvas/api/v0/form/component-instance/content_template/{entity}/{preview_entity}` for retrieving and submitting the component instance form when editing a `ContentTemplate`.

## Technical details

On the backend, a new route `canvas.api.form.component_instance.content_template` was added at `/canvas/api/v0/form/component-instance/content_template/{entity}/{preview_entity}`. It inherits from the existing `canvas.api.form.component_instance` route via YAML anchors and adds a second route parameter, `preview_entity`, resolved by `canvas_content_template_preview_entity`, plus a `_canvas_preview_entity_view_access` requirement. `ComponentInstanceForm::buildForm()` had its `$entity` parameter type broadened from `?FieldableEntityInterface` to `?EntityInterface`, and a new optional `?FieldableEntityInterface $preview_entity` parameter was added. A `$host_entity` local variable now holds either the `$preview_entity` (when `$entity` is a `ContentTemplate`) or `$entity` itself, and is passed to `clientModelToInput()` and `buildComponentInstanceForm()` in place of `$entity`. A bug in `ReduxIntegratedFieldWidgetsHooks::fieldWidgetSingleElementMediaLibraryWidgetFormAlter()` was fixed by switching from `$parameters->get('entity')` (which returns the hydrated entity object) to `$parameters->getRawParameters()`, preventing failures when the component instance form is rendered for templates that have inputs backed by the media library widget.

On the frontend, the placeholder `DummyPropsEditForm` component and `dummyPropsFormApi` RTK Query service were renamed to `ComponentInstanceForm` and `componentInstanceFormApi` respectively. A new `EditorFrameContext` enum (`ENTITY = 'entity'`, `TEMPLATE = 'template'`) was added to `uiSlice`, with a `setEditorFrameContext` action and `selectEditorFrameContext` selector. The `Editor` component now accepts a `context: EditorFrameContext` prop and dispatches `setEditorFrameContext` on mount, so all downstream components (Preview, Topbar, PageInfo, ContextualPanel) can read context from the Redux store instead of parsing the URL pathname. Both `getComponentInstanceForm` and `patchComponent` (preview update) API calls now include a `type` field that selects the correct URL: `/canvas/api/v0/form/component-instance/{entity_type}/{entity_id}` for `entity`, and `/canvas/api/v0/layout-content-template/{entity_type}.{template_bundle}.{template_view_mode}/{entity_id}` (or the corresponding form path) for `template`. The `ContextualPanel` tab label changes to "Template data" in template context, and the `PageDataForm` is suppressed there (a follow-up refactor is noted in [#3546319](https://www.drupal.org/node/3546319)).

## Upgrade

If you have custom code that calls `ComponentInstanceForm::buildForm()` directly or sub-classes `ComponentInstanceForm`, note that the `$entity` parameter type changed from `?FieldableEntityInterface` to `?EntityInterface`, and a new optional parameter `?FieldableEntityInterface $preview_entity` was added:

```php
// Before
public function buildForm(array $form, FormStateInterface $form_state, ?FieldableEntityInterface $entity = NULL): array

// After
public function buildForm(array $form, FormStateInterface $form_state, ?EntityInterface $entity = NULL, ?FieldableEntityInterface $preview_entity = NULL): array
```

If you use the `dummyPropsFormApi` RTK Query service or the `useGetDummyPropsFormQuery` hook, they have been renamed:

```ts
// Before
import { dummyPropsFormApi } from '@/services/dummyPropsForm';
import { useGetDummyPropsFormQuery } from '@/services/dummyPropsForm';

// After
import { componentInstanceFormApi } from '@/services/componentInstanceForm';
import { useGetComponentInstanceFormQuery } from '@/services/componentInstanceForm';
```

The query argument shape also changed: `useGetComponentInstanceFormQuery` now accepts `{ queryString: string; type: 'entity' | 'template' }` instead of a plain `string`.
