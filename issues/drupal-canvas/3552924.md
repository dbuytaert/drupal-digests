## Summary

Canvas content templates were ignoring language context when resolving entity reference fields. When viewing a translated node whose content template mapped a component prop to a field on a referenced entity (for example, the title of a related article), Canvas always rendered the default-language value of the referenced entity rather than the translation matching the current page language. This fix makes the prop-expression evaluator language-aware so all referenced entities are resolved to the correct translation throughout the template rendering chain.

## Impact

No upgrade or configuration changes required.

## Technical details

The core change is adding a `?string $language_code` parameter to `Evaluator::evaluate()` and `::doEvaluate()`. The parameter defaults to `null` and falls back to the site default language inside `evaluate()`, preserving backward compatibility. `DynamicPropSource::evaluate()` now seeds the chain with `$host_entity->language()->getId()`, so the host entity's active translation language flows down through every recursive call.

Two translation-resolution points are added in `doEvaluate()`. First, a new private `evaluateTranslatedReference()` method handles `ReferenceFieldTypePropExpression`: it retrieves the raw field value, checks whether it implements `TranslatableInterface`, has the target language, and swaps to that translation before recursing into the referenced expression. Second, within the `FieldPropExpression` branch, prop values that are not `PrimitiveInterface` instances are now checked for `TranslatableInterface` before being stored in `$result`, again swapping to the requested translation when available. The `language_code` is forwarded to every recursive `self::evaluate()` call so deeply nested references are also resolved correctly.

This MR (!249) was a draft proposal. The maintainer (wim leers) redirected the essential fix to issue [#3583379](https://www.drupal.org/node/3583379), which landed the same infrastructure. MR !249 was subsequently closed and the issue was marked fixed once [#3583379](https://www.drupal.org/node/3583379) merged.
