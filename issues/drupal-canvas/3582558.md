## Summary

Canvas site owners who use TMGMT (Translation Management Tool) to manage config-translation workflows can now translate the text inputs of component instances stored in `ContentTemplate` and `PageRegion` config entities. Inputs populated by static prop sources (plain strings, rich text, URIs, and multi-value sequences) are exposed as translatable segments in the TMGMT job UI. Inputs driven by entity field references or host-entity URL prop sources are correctly excluded. Optional inputs not populated in the default-language template can still receive a translation value. This is Step 4 of a series of issues making Canvas config-defined component trees symmetrically translatable; it requires the `canvas_dev_translation` module to be enabled.

## Impact

Affects module maintainers and site owners.

- **Config change:** New `translatable_uri` config schema type added; `field.value.link` and `field.value.uri` URI properties retyped to `translatable_uri` via `hook_config_schema_info_alter()` in `canvas_dev_translation`.
- **New API:** New `ComponentInputsConfigProcessor` class in `Drupal\canvas\Tmgmt\` (marked `@internal`) registered as `tmgmt_config_processor` for `canvas.content_template.*.*.*` and `canvas.page_region.*` config schema types.

## Technical details

The central addition is `ComponentInputsConfigProcessor` (`Drupal\canvas\Tmgmt\`, `@internal`), which extends TMGMT's `DefaultConfigProcessor`. It is registered on the `canvas.content_template.*.*.*` and `canvas.page_region.*` config schema types via a new `hook_config_schema_info_alter()` implementation in `canvas_dev_translation`'s `ConfigTranslationSupportHooks`.

When `extractTranslatables()` encounters a `ComponentInputsMapping` element, it bypasses standard schema-driven extraction and instead loads the `Component` entity and its `JsonSchemaPropsComponentSourceBase` component source. It then constructs an ad-hoc concrete schema by mapping each prop's field type to the corresponding `field.value.*` config schema definition, handling four cardinality/property-count combinations (single cardinality + single property, multi-cardinality + single property, single cardinality + multi-property, multi-cardinality + multi-property). Props not backed by a `StaticPropSource` are emitted as `type: ignore`. Optional inputs absent from the default translation are injected as `NULL` into `$config_data` so TMGMT's `ArrayElement::getAllKeys()` does not silently skip them; because TMGMT's flattener drops empty strings, the Unicode sentinel `âˆ…` ("empty set") is substituted to force these inputs visible in the translation UI, via the private `extractTranslatablesIncludingEmpty()` method.

Two config schema alterations accompany the processor. `field.value.link`'s `uri` property and `field.value.uri`'s `value` property are retyped to a new `translatable_uri` schema type (a translatable variant of `type: uri`) added by the MR with a `@todo` referencing core issue [#3595275](https://www.drupal.org/node/3595275).

A bug fix in `CanvasStaticPropSourceFieldWidget::setConfig()` was also required to make TMGMT's save path work correctly. An `assert()` requiring values to be delta-keyed lists was removed (TMGMT does not supply delta keys); multi-value fields now strip only the `add_more` pseudo-key. A second bug was fixed where the config override was written to `$base_key` instead of `$subkey` for multi-property field types, and optional empty properties in the base config now skip creating an override rather than writing an empty value.

Functional test coverage is split into two new parallel test classes sharing a `ConfigWithComponentTreeTranslationTestBase` fixture: `ConfigWithComponentTreeConfigTranslationUiTest` (core Config Translation UI) and `ConfigWithComponentTreeTmgmtUiTest` (TMGMT UI). Both assert the same resulting `LanguageConfigOverride`, covering plain prose, rich text (value-only override, format excluded), URI-esque props, multi-cardinality sequences, optional unpopulated props, and excluded non-static prop sources.
