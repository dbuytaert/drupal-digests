## Summary

Running `canvas push --include-pages` could silently overwrite the public URL alias of an existing remote page when the local page file's `path` field did not match the remote alias. This risked breaking live page URLs without any warning. The fix adds a validation step that compares the local and remote path aliases before any write occurs: if they differ, the push is rejected with a descriptive error message. New pages (those without a UUID matching a remote page) are unaffected and can still set any path alias freely.

## Impact

No upgrade or configuration changes required.

## Technical details

A new `page-path-alias-validation.ts` module introduces three exports: `normalizePathAlias()` (trims whitespace), `getPathAliasChange()` (compares normalized local vs. remote aliases and returns an `ExistingPagePathAliasChange` object when they differ), and `formatPagePathAliasChangeError()` (produces the human-readable error string).

`validatePages()` receives a new optional `PageValidationOptions` parameter containing a `remotePageByUuid: Map<string, PageListItem>`. When that map is provided and a page's UUID matches a remote entry, `getPathAliasChange()` is called and any difference is appended as a validation detail with `heading: 'path'`, causing the page to fail validation before any API call is made. The push command now passes the already-fetched `remotePageByUuid` map into `validatePages()`.

A secondary guard is also placed directly inside `pushPages()` at the write boundary: if `getPathAliasChange()` detects a mismatch at that point it throws immediately, ensuring correctness even if `pushPages()` is called without going through the earlier validation step. `normalizePathAlias()` is additionally applied to `page.path` when constructing the `updatePage` payload, so the value sent to the API is always trimmed.
