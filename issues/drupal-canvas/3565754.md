## Summary

Canvas code components can now access metadata about the primary Drupal entity on the current page. The `getPageData()` utility function returns a new `mainEntity` property containing the entity's `uuid`, `bundle`, and `entityTypeId`. When the current route has no associated entity (such as the login page, front page, or the Canvas code editor), `mainEntity` is `null`. A typical use case is a "Related Articles" component that needs to exclude the article currently being viewed from its JSON:API query.

## Impact

Affects module maintainers.

- **New API:** New `mainEntity` property on the `PageData` interface returned by `getPageData()` exposes the current page entity's `uuid`, `bundle`, and `entityTypeId` (or `null` on non-entity routes).

## Technical details

On the PHP side, a new `getCanvasDataMainEntityV0()` method on `CodeComponentDataProvider` walks the current route's parameters in priority order (`preview_entity`, `node`, `entity`, `canvas_page`, then any remaining params) and returns the first `EntityInterface` instance it finds, exposing `bundle()`, `getEntityTypeId()`, and `uuid()`. This data is injected into `drupalSettings.canvasData.v0.mainEntity`. A corresponding `canvasData.v0.mainEntity` asset library entry is registered, and `ComponentSourceHooks` wires it into the attach pipeline (respecting existing overrides via `NestedArray::getValue` null-check, consistent with the pattern used for `pageTitle` and `breadcrumbs`).

On the TypeScript side, the `PageData` interface gains `mainEntity: EntityMetadata | null` where `EntityMetadata` carries `bundle`, `entityTypeId`, and `uuid`. The `getPageData()` function reads `window.drupalSettings?.canvasData?.v0?.mainEntity`. The AST-based data-dependency scanner (`getDataDependenciesFromAst`) now maps `getPageData` calls to `v0.mainEntity` in addition to `v0.breadcrumbs` and `v0.pageTitle`, so components that call `getPageData()` automatically declare the library dependency. The `DrupalSettings` TypeScript interface in the CLI tooling is also updated with the matching shape.

The design deliberately treats `mainEntity` as page-level context (implicit input) rather than a prop (explicit input), since a "Related Articles" component should always exclude the current article automatically rather than requiring the site builder to manually map a UUID prop. The docs include a `RelatedArticlesWrapper` pattern showing how to guard against `null` and wrong entity type before calling hooks, which is essential for components placed in regions that appear on all routes.

## Upgrade

Code components that call `getPageData()` now receive `mainEntity` automatically. No configuration changes are needed. Components placed in regions that appear on routes without a canonical entity must guard against `null`:

```js
import { getPageData } from 'drupal-canvas';

function RelatedArticlesWrapper() {
  const { mainEntity } = getPageData();

  if (
    !mainEntity ||
    mainEntity.entityTypeId !== 'node' ||
    mainEntity.bundle !== 'article'
  ) {
    return null;
  }
  return <RelatedArticles mainEntity={mainEntity} />;
}
```

The TypeScript type for `mainEntity` is `EntityMetadata | null`:

```ts
interface EntityMetadata {
  bundle: string;
  entityTypeId: string;
  uuid: string;
}
```

### Collaboration

The shape of the returned data was influenced by a review comment from effulgentsia, who suggested wrapping the entity fields inside a nested `mainEntity` object (with its own `EntityMetadata` interface) rather than adding a flat `entityUuid` key directly to `PageData`. This makes it easier to add further entity fields later without BC breaks, and is the shape that shipped.

wim leers initially argued that entity data does not belong in `getPageData()` at all, since that function reflects the current route match (which is not always an entity route), and proposed a separate `getEntityRouteData()` function instead. After hooroomoo pushed back on DX grounds, effulgentsia framed the distinction as "context" (implicit, always the current entity) versus "props" (explicit, chosen by the site builder), which persuaded wim leers to accept the `getPageData()` placement provided the docs clearly explain that `mainEntity` is `null` on non-entity routes and that components must guard accordingly.
