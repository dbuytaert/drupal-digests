## Summary

In the Canvas module, the Linker UI allows editors to connect a component prop to a dynamic field source. After a change landed in [#3548395](https://www.drupal.org/node/3548395), selecting most Linker choices had no visible effect: the client kept sending `sourceType: static` to the server instead of `sourceType: dynamic` for the chosen source. This fix corrects the propagation of prop-link data through the render element tree so all Linker selections are correctly recorded and sent.

## Impact

No upgrade or configuration changes required.

## Technical details

The bug was in `GeneratedFieldExplicitInputUxComponentSourceBase::processElementTreeLinkerLabels()`, a recursive method that walks the render array tree and stamps each child element with a `#label_attributes['prop_link_data']` value. The Linker UI reads this data to know which dynamic prop source selection to transmit.

The old implementation used `foreach ($element as $key => &$child)` with a manual `is_array($child) && $key[0] !== '#'` guard to skip non-element keys. Because PHP's `foreach`-by-reference semantics can include integer-keyed entries (since `(string)0)[0]` is `'0'`, which passes the `!== '#'` check) and has known pitfalls around reference aliasing, the `propLinkData` was not reliably reaching all the child elements the Linker reads.

The fix replaces that loop with `foreach (Element::children($element) as $key)` and accesses `$element[$key]` directly. `Element::children()` is the canonical Drupal API for this: it returns only string-keyed, non-`#`-prefixed, array-valued render element children, exactly the set the recursive walk should descend into. A separate but related problem (Linker choices failing when the dynamic source evaluates to `NULL` for an empty optional field) was split into its own issue, [#3548761](https://www.drupal.org/node/3548761).
