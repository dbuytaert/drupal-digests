# #3591584: Publishing a canvas_page via auto-save API drops all non-default-language translations

- **Project:** Drupal Canvas
- **Type:** Critical bug report
- **Version:** 1.x-dev
- **Status:** Fixed
- **Discussion:** [#3591584](https://git.drupalcode.org/project/canvas/-/work_items/3591584) · 3 contributors · 10 comments

## Summary

Sites running the Canvas module with multilingual content were silently losing translations whenever a page was published through the Canvas editor's auto-save flow. Every non-default-language translation would disappear from the published page. This critical data-loss bug is now fixed: publishing via the auto-save API preserves all translations correctly. A related edge case where the `content_moderation` field was being ignored during auto-save publish is also corrected.

## Impact

Affects module maintainers.

- **New API:** `AutoSaveManager::isPersistedComputedField(FieldDefinitionInterface $field_definition): bool` added as a public static method to identify computed fields that are user-editable and persisted outside the entity's own tables (currently `path` and `moderation_state`).

## Technical details

The root cause was in `ApiAutoSaveController`'s publish path. The auto-save snapshot only ever captures the single translation being actively edited. Previously, the controller saved the entity reconstructed directly from that snapshot, which discarded every other translation. The fix loads the authoritative stored entity via `loadUnchanged()`, identifies which translation the snapshot belongs to (via its `langcode`), and then applies only the changed fields from the snapshot onto that specific translation of the loaded entity. All other translations are untouched because the real entity is never discarded.

This approach also fixes a subtler secondary bug: `content_translation`'s `FieldTranslationSynchronizer::synchronizeFields()` requires a valid loaded revision ID to determine the correct synchronization source for untranslatable ("symmetric") field columns. The reconstructed entity lacked this, causing the synchronizer to fall back to a stale translation and potentially reverting structural edits on the default language. Loading the real entity avoids this.

A new public static method, `AutoSaveManager::isPersistedComputedField(FieldDefinitionInterface $field_definition)`, was extracted to identify computed fields that are user-editable and persisted outside the entity's own tables: the `path` field (persisted as `path_alias` entities via `PathItem::postSave()`) and `moderation_state` (persisted as `content_moderation_state` entities). These fields must be carried through the auto-save store and applied on publish; previously only `path` was special-cased. The issue notes that no reliable runtime signal exists to detect these fields generically and links [#3503446](https://www.drupal.org/node/3503446) as the follow-up to remove the hardcoding.

An earlier approach that re-added missing translations back onto the reconstructed entity after the fact was explicitly rejected because it papered over the dropped translations without fixing the symmetric structural-edit corruption.
