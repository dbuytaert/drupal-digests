## Summary

Canvas's "Review changes" panel now surfaces conflict indicators when an auto-save entry conflicts with an external update to its underlying entity. A yellow banner at the top of the panel counts active conflicts and offers a "Resolve N conflict(s)" call-to-action button. Each conflicted row replaces its normal entity-type icon with a red triangle warning icon, disables its checkbox, and shows a "Resolve conflict" option in the row's dropdown menu instead of "View changes." Conflicted rows are automatically excluded from select-all, and any row that becomes conflicted while already selected is silently deselected. This conflict UX is currently gated behind the `canvas_dev_mode` module (dev mode flag) while the full conflict resolution workflow is built out in a follow-on issue.

## Impact

Affects module maintainers.

- **New API:** New `ConflictBanner` component and `isConflictUxEnabled()` utility added to the Canvas React frontend; `PublishReview` accepts new optional `conflictCount` and `onResolveConflict` props.

## Technical details

The `/canvas/api/v0/auto-saves/pending` endpoint can now return HTTP 409 with a `{ data, errors }` envelope alongside the normal flat 200 shape. A new `applyConflictStateFromResponse()` function in `pendingChangesApi` normalizes both shapes: it iterates the `errors` array, indexes entries by `meta.api_auto_save_key` (falling back to `source.pointer`), and marks each matching `PendingChange` with `hasConflict: true` and a `conflict_id` string only when the error carries `CONFLICT_CODE.DETECTED` (new enum value `4`). Non-resolvable codes (`UNEXPECTED = 1`, `EXPECTED = 2`) do not set `hasConflict`. The RTK Query `getAllPendingChanges` endpoint now passes `validateStatus` to accept 409 as a valid response, and `transformResponse` delegates to `applyConflictStateFromResponse`.

The new `isConflictUxEnabled()` utility in `@/features/conflict/conflictUtils` reads `getCanvasSettings()?.devMode` as a feature flag; all conflict-specific rendering is conditional on this flag at runtime. `ConflictBanner` is a standalone React component (with its own CSS module) that renders null when `conflictCount` is zero, shows a pluralised conflict count headline and subtext, and exposes an `onResolveClick` callback and a `disabled` prop for the CTA button.

`ChangeRow` now computes `hasConflict = conflictUxEnabled && !!change.hasConflict` and swaps the `ChangeIcon` for a `ExclamationTriangleIcon` (testid `change-conflict-icon`), disables the checkbox, and gates the "View changes" menu item behind `!hasConflict` while injecting a "Resolve conflict" item when `hasConflict` and `onResolveConflict` are both present. `ChangeGroup` and `PublishReview` both derive a `selectableChanges` list by filtering out conflicted entries when the flag is on, and feed that subset to selection state and the select-all checkbox. A `useEffect` in `PublishReview` watches `changes` for newly conflicted pointers and removes them from `selectedChanges`. The `onResolveConflict` handler in `UnpublishedChanges` currently calls `refetch()` as a placeholder; the full resolution UI is tracked in issue [#3591601](https://www.drupal.org/node/3591601).
