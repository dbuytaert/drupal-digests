## Summary

When editing a component instance inside a Canvas `ContentTemplate`, selecting a `HostEntityUrlPropSource` suggestion (offered for `type: string, format: uri|uri-reference` props) appeared to do nothing. The prop linker UI did not update, and the selected source was never applied. This fix makes the suggestion selection work as expected so site builders can successfully link a component's URL prop to the host entity's URL.

## Impact

No upgrade or configuration changes required.

## Technical details

The bug had two interacting parts. On the PHP side, the `LinkedPropSource` render element asserted that `#linked_prop_source` must be a `DynamicPropSource` instance, and `GeneratedFieldExplicitInputUxComponentSourceBase` only set `$linked_prop_source` when the source was a `DynamicPropSource`. Both were widened to also accept `HostEntityUrlPropSource`.

On the React side, `PropLinker.tsx` determined which suggestion was currently active by reading `fieldSource.expression` after casting the source to `DynamicPropSource`. Because `HostEntityUrlPropSource` carries no `expression` key, the comparison always failed and nothing appeared selected. The fix stops introspecting source objects and instead compares the entire source object as an opaque blob using `JSON.stringify()`. The local variable formerly called `currentExpression` is now `selectedSourceAsString`, and both `isActive` comparisons use `JSON.stringify(item.source) === selectedSourceAsString`. As a side effect, `BasePropSource` (in `layoutModelSlice`) is now exported so `PropLinker.tsx` can type-cast to it instead of the narrower `DynamicPropSource`.
