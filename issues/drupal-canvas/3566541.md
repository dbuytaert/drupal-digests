## Summary

The Code Component Editor in Canvas's Theme builder now includes a "Date and time" option in the Type dropdown when defining props for Single Directory Components. Selecting it lets a content editor populate a date value using a browser calendar widget, or link the prop dynamically to a date field on a node. Two sub-types are available: "Date only" (produces `type: string, format: date`) and "Date and time" (produces `type: string, format: date-time`). The underlying SDC schema had supported both formats since the module's initial release; this change adds the missing UI to actually create them.

## Impact

Affects module maintainers.

- **New API:** New 'Date and time' prop type in the Code Component Editor produces `type: string, format: date` or `type: string, format: date-time` SDC props.

## Technical details

A new entry in `derivedPropTypes.ts` registers the `date` derived type with `displayName: 'Date and time'`. Its `derive` function matches any serialized prop where `type === 'string'` and `format` is either `'date'` or `'date-time'`. The initial state defaults to `format: 'date'`.

A new `FormPropTypeDate` component renders a "Date type" dropdown (Radix UI `Select`) offering "Date only" and "Date and time" choices, followed by an HTML5 `<input type="date">` or `<input type="datetime-local">` for the example value. For the datetime sub-type the component converts between the UTC ISO 8601 string required by the server and the local-time string expected by the `datetime-local` input, using new utility functions `localTimeToUtcConversion` and `utcToLocalTimeConversion` added to `date-utils.ts`. Local state tracks the `datetime-local` value so the input remains stable while conversion happens before dispatch.

`Props.tsx` adds a `case 'date':` branch to the existing switch that renders prop-type-specific form controls, passing `id`, `example`, and `format` to `FormPropTypeDate`. The same commit also threads `format` through to `FormPropTypeLink`, which was previously receiving it as `undefined`. The `Props.test.tsx` vitest suite was significantly refactored (shared `beforeEach` rendering, a reusable `addProp` helper) and extended with a new test covering both date sub-types including timezone conversion.

Two pre-existing bugs surfaced during testing were filed separately: a visual overlap of the calendar icon in Chrome ([#3572934](https://www.drupal.org/node/3572934)) and a `RangeError: Invalid time value` when selecting a datetime on a prop with no example value ([#3573426](https://www.drupal.org/node/3573426)).
