## Summary

The Code Component Editor now lets content editors work with date and datetime fields. Component developers can define props that accept date values, which editors can populate with a calendar widget or link to date fields on content. This enables use cases like event listings, publication dates, and any component that needs to display temporal information. The dropdown menu for defining prop types now includes a "Date and time" option that supports both date-only and full datetime values, matching the formats already allowed in Single Directory Component schemas.

## Impact

No upgrade or configuration changes required.

## Technical details

This feature adds date and datetime support to the Code Component Editor's prop type system. The `derivedPropTypes` configuration already allowed `type: string, format: date` and `type: string, format: date-time` in component schemas, but the UI had no way to create them.

A new `FormPropTypeDate` component provides the editing interface. It includes a dropdown to toggle between `date` and `date-time` formats, and renders the appropriate HTML input element (`type="date"` or `type="datetime-local"`). The datetime implementation handles timezone conversion: the server stores values as UTC ISO strings like `2026-01-15T12:34:56.789Z`, but the `datetime-local` input requires local time format like `2026-01-15T14:30`. Two utility functions in `date-utils.ts` handle bidirectional conversion: `localTimeToUtcConversion()` and `utcToLocalTimeConversion()`.

The `Props.tsx` component routes date props to the new form component based on the `format` field. The serialization and deserialization logic in `code-editor-utils` already supported these formats since they match the JSON Schema spec. Comprehensive tests in `Props.test.tsx` verify switching between date types, timezone handling, and example value validation.

A change record documents the new functionality: [#3572962](https://www.drupal.org/node/3572962).
