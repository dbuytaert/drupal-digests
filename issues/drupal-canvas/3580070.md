## Summary

When a block plugin's `defaultConfiguration()` returns values that violate its own config schema (for example, returning a string where the schema declares a boolean), the Canvas Components library listing would crash entirely, hiding all other available components. Canvas now validates a block plugin's default configuration against its schema during component discovery. Any block plugin that fails this check is rejected with an explanatory message, and the rest of the component library continues to work normally.

## Impact

No upgrade or configuration changes required.

## Technical details

`BlockComponentDiscovery::checkRequirements()` previously built a typed config data definition from the block's `defaultConfiguration()` but never validated it. The fix enriches that call to `typedConfigManager->createFromNameAndData()` to include `id`, `provider`, `label`, and `label_display` alongside the settings (matching what `computeCurrentComponentMetadata` produces), then calls `$data_definition->validate()` after the existing `FullyValidatable` constraint check passes. Any constraint violations are collected and surfaced as rejection reasons, e.g. `'Block plugin has invalid default setting "crash": This value should be of the correct primitive type.'`

A new test block `CanvasTestBlockInputInvalidDefault` was added to the test suite. It extends `CanvasTestBlockInputValidatableCrash` (changed from `final` to `class` for this purpose) and intentionally returns a string for a field declared as `boolean`. The `BlockComponentTest` kernel test asserts that this plugin is now rejected during discovery. A pre-existing `ComponentInputsEvolutionTest` required a version hash update because the new validation also fires when the test's simulated schema change causes a previously valid block's stored configuration to fail validation, correctly preventing a new component version from being generated.
