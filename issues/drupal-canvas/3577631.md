## Summary

Canvas can now manage custom fonts through a Brand Kit interface. Site builders can upload font files (WOFF2, WOFF, TTF, OTF) and Canvas automatically generates the necessary CSS font-face declarations. Both static and variable fonts are supported, with automatic detection of font variation axes. Uploaded fonts become available globally across all Canvas pages and components. The feature is currently behind a feature flag.

## Impact

Affects module maintainers and site owners.

- **Config change:** New `canvas.brand_kit.global` config entity created on installation; upgraded sites run `canvas_post_update_0014_create_global_brand_kit` to create it
- **Permission change:** New `administer brand kit` permission required to manage fonts and Brand Kit resources (hidden from UI unless `canvas_dev_mode` is enabled)
- **New API:** New `canvas/api/v0/artifacts/upload` endpoint for uploading font files; new `canvas/api/v0/config/brand_kit` endpoints for managing Brand Kit configuration via HTTP API

## Technical details

A new `BrandKit` config entity type stores uploaded font metadata including family name, weight, style, format, and variable font axes. The entity generates CSS `@font-face` rules from this metadata and serves them through the library system as `canvas/brand_kit.{id}` and `canvas/brand_kit.{id}.draft` for auto-save support.

Font files are uploaded via a new generic artifact endpoint at `/canvas/api/v0/artifacts/upload` using streamed binary uploads. The endpoint validates file extensions (js, css, json, map, gif, png, jpg, jpeg, svg, webp, avif, ico, woff, woff2, ttf, eot, otf) and enforces a 10MB size limit. Uploaded files are saved as managed Drupal files in `public://canvas/assets/` with automatic file usage tracking.

The global Brand Kit (ID: `global`) is created during installation or via post-update hook for upgraded sites. It cannot be deleted. The UI and permission are hidden unless `canvas_dev_mode` is enabled. Font files are preloaded using HTML head link tags for performance.

Variable font support includes parsing and storing axis definitions (tag, name, min, max, default values). The schema validates axis uniqueness and range constraints. File usage records ensure uploaded fonts are not prematurely deleted, with separate tracking for auto-saved drafts versus published configuration.

The artifact upload endpoint is reusable for future asset types beyond fonts. Access requires either `administer brand kit` or `administer code components` permissions.
