## Summary

Canvas can now manage fonts through a Brand Kit UI. Site owners can upload font files (both static and variable fonts) and Canvas automatically generates the necessary @font-face CSS rules for use across the site. The Brand Kit feature is behind a feature flag controlled by the canvas_dev_mode module.

## Impact

Affects module maintainers and site owners.

- **Permission change:** New `administer brand kit` permission required to manage fonts (hidden behind canvas_dev_mode feature flag)
- **API change:** New `/canvas/api/v0/artifacts/upload` endpoint for uploading font files; brand kit endpoints added to `/canvas/api/v0/config/brand_kit`
- **New API:** New `BrandKit` config entity type for managing fonts and typography assets

## Technical details

Canvas now provides a Brand Kit system for managing typography assets. A new `BrandKit` config entity stores font metadata including family name, file URI, format (woff2, woff, ttf, otf), weight, style, and for variable fonts, axis definitions (tag, min, max, default values). The global brand kit (`BrandKit::GLOBAL_ID`) is created during installation and cannot be deleted.

Font files are uploaded via a new generic artifact upload endpoint (`canvas.api.artifacts.upload`) that accepts binary streams and creates managed file entities in `public://canvas/assets/`. The endpoint validates file extensions and enforces a 10MB size limit. The `ArtifactUploadAccessCheck` requires either the `administer brand kit` or `administer code components` permission.

Font metadata is stored in the brand kit's `fonts` property. When the entity saves, `BrandKit::buildGeneratedCss()` generates @font-face rules by iterating over font entries and constructing CSS declarations. The generated CSS is prepended to any manually authored CSS in the brand kit. `BrandKit::buildFontFaceRule()` constructs each rule with font-family, src (with format label mapping), font-weight, and font-style declarations.

File lifecycle management uses Drupal's file usage system. `BrandKit::syncFontFileUsage()` tracks which files are referenced by comparing old and new URI lists, adding usage records for new files and removing records for deleted files. The system distinguishes between published config (`BrandKit::FILE_USAGE_TYPE`) and auto-saved drafts (`BrandKit::AUTO_SAVE_FILE_USAGE_TYPE`). When a brand kit is deleted, `BrandKit::postDelete()` clears all file usage records.

Auto-save support required extending the existing auto-save infrastructure. `BrandKit::syncAutoSaveFileUsage()` and `BrandKit::clearAutoSaveFileUsage()` manage draft file usage separately from published config. `ApiConfigAutoSaveControllers::patch()` was updated to call these methods when saving auto-save entities. When an auto-save is deleted or published, the draft file usage records are cleared so unused files become temporary.

The brand kit integrates with Canvas's library system. `LibraryHooks::libraryInfoBuild()` creates two library entries per brand kit: a production library (e.g. `canvas/brand_kit.global`) and a draft library (e.g. `canvas/brand_kit.global.draft`). Code components depend on the global brand kit library. `JsComponent::buildSlottableComponentTree()` adds font preload hints to the page by calling `getFontPreloadLinks()`, which generates `<link rel="preload" as="font">` elements for each font.

Publishing workflow was updated in `ApiAutoSaveController::publish()`. When publishing code components, the system now checks whether both the global asset library and global brand kit have unpublished changes. If either has unpublished changes, publishing is blocked with `ErrorCodesEnum::GlobalAssetNotPublished`.

The `administer brand kit` permission is hidden from the permissions page when canvas_dev_mode is disabled via `canvas_form_alter()`. `BrandKitAccessControlHandler::checkAccess()` prevents deletion of the global brand kit. The Canvas UI receives the user's brand kit permission status in the `permissions.brandKit` flag.

Routing changes added `brand_kit` to the `canvas_config_entity_type_id` requirement on all config entity API routes. OpenAPI schema was extended with `BrandKit` and `NewBrandKit` schemas documenting the font entry structure including read-only computed properties (`url`, `variantType`). A new OAuth scope `canvas:brand_kit` was added requiring the `administer brand kit` permission.

The upgrade path in `canvas_post_update_0014_create_global_brand_kit()` installs the brand_kit entity type definition if needed and creates the default global brand kit entity.
