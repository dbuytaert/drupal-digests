## Summary

Canvas now ships a dedicated Brand Kit config entity for managing site fonts. A single "global" brand kit is created automatically on installation or after running database updates. Site owners with the new `administer brand kit` permission can upload static and variable font files through the Theme Builder UI; Canvas automatically generates `@font-face` CSS declarations and loads them on the site frontend and in the Canvas preview. The feature is currently gated behind `canvas_dev_mode`, so the permission is hidden from the standard permissions page on production sites until that module is enabled.

## Impact

Affects module maintainers and site owners.

- **Database update:** Run `canvas_post_update_0014_create_global_brand_kit` to install the `brand_kit` entity type and create the global brand kit; run `canvas_oauth_post_update_0002_canvas_brand_kit_scope` to install the new OAuth scope.
- **Config change:** New `canvas.brand_kit.global` config entity created on install or post-update; `fonts` field removed from `canvas.asset_library.global`.
- **Permission change:** New `administer brand kit` permission added; hidden from the permissions UI unless `canvas_dev_mode` is enabled.
- **New API:** New `/canvas/api/v0/artifacts/upload` endpoint and `/canvas/api/v0/config/brand_kit` CRUD endpoints; new `canvas:brand_kit` OAuth scope in `canvas_oauth`.

## Technical details

The `BrandKit` config entity (`canvas.brand_kit.*`) is defined in `src/Entity/BrandKit.php` with `CanvasAssetStorage` and a new `BrandKitAccessControlHandler`. Its config schema constrains `id` to the regex `/^global$/`, meaning only the single global instance is allowed. Each font entry stores `id` (UUID), `family`, `uri`, `format`, `weight`, `style`, and an optional `axes` list for variable-font axis definitions (`wght`, `wdth`, `slnt`, etc.). `BrandKit::getCss()` generates `@font-face` rules server-side from these entries and merges them with any compiled CSS from the auto-save system.

Font management was removed from `AssetLibrary` entirely (the `fonts` field and related `getFonts`/`setFonts` methods are gone from that class). That field had been introduced in the unreleased [#3576865](https://www.drupal.org/node/3576865) and was never shipped to users.

The internal routing (`canvas.routing.yml`) adds `brand_kit` to the allowed entity-type regex on the auto-save get/patch routes, the CSS and JS auto-save endpoints, and the config list/get/patch/post/delete routes. A new `/canvas/api/v0/artifacts/upload` route is backed by `ArtifactController::upload`, which accepts binary font streams via `Content-Disposition` filename header, saves them as Drupal managed file entities, and returns `fid`, `uri`, and `url`. Access to this endpoint requires either `administer brand kit` or `administer code components` (via the new `ArtifactUploadAccessCheck`). `BrandKitAccessControlHandler::checkCreateAccess()` returns `AccessResult::forbidden` to prevent API callers from creating additional brand kit entities, and deletion of the global brand kit is also blocked.

File usage is tracked via `BrandKit::syncFontFileUsage` and `clearFontFileUsage`, and the auto-save layer calls `BrandKit::syncAutoSaveFileUsage` / `clearAutoSaveFileUsage` to keep draft file references consistent. The publish-consistency check in `ApiAutoSaveController` was extended to include the global brand kit alongside the global asset library, so attempting to publish code components while the brand kit has unpublished changes returns a `GlobalAssetNotPublished` error.

`canvas_post_update_0014_create_global_brand_kit` installs the entity type if needed and creates the default global brand kit with an enforced `canvas` module dependency. A companion `canvas_oauth_post_update_0002_canvas_brand_kit_scope` installs a new `canvas:brand_kit` OAuth scope tied to the `administer brand kit` permission.

## Upgrade

Run database updates to install the global brand kit entity and the new OAuth scope:

```bash
drush updb
```

If you use `canvas_oauth` and need external API clients to access brand kit data, grant them the `canvas:brand_kit` scope (maps to the `administer brand kit` permission).

The `fonts` field is no longer part of `canvas.asset_library.global`; if you have custom code that called `AssetLibrary::getFonts()` or `AssetLibrary::setFonts()`, those methods no longer exist and you must migrate to `BrandKit::getFonts()` / `BrandKit::setFonts()` on `BrandKit::load(BrandKit::GLOBAL_ID)`.
