## Summary

When configuring Content Templates in Canvas, entity reference fields pointing to User entities (such as a node's "Author" or "Revision user" fields) were missing from the list of available data sources for `type: string` component props. After this fix, the User `name` field is correctly recognized as a prose string and appears in shape matching results. This means site builders can now wire text props in a Hero or similar component directly to the author's name, revision user's name, and any other user reference field on nodes, media, files, or canvas pages.

## Impact

No upgrade or configuration changes required.

## Technical details

Canvas's shape matching system assigns a `StringSemantics` constraint to string-typed field properties so it can identify which fields are prose text and therefore compatible with `type: string` component props. This constraint is normally applied automatically by the `StringItemOverride` plugin, which intercepts fields using the standard `StringItem` class. The `User` entity type, however, declares its `name` base field using the `string` field type but overrides the item class with `UserNameItem` (see `User::baseFieldDefinitions()`). Because `UserNameItem` is not `StringItem`, `StringItemOverride` never fires, leaving `name` without any `StringSemantics` annotation and therefore invisible to shape matching.

The fix adds an explicit condition in `ShapeMatchingHooks::entityBaseFieldInfoAlter()` for `entity_type->id() === 'user'`, calling `$fields['name']->addPropertyConstraints('value', ['StringSemantics' => StringSemanticsConstraint::PROSE])` to manually restore the prose semantics that the class override discarded. A `@todo` comment marks this workaround for removal once `StringItemOverride` is merged into core. Separately, a previously identical workaround for the `taxonomy_term` `name` field was removed from the same method.

The cascading effect in the test fixtures is substantial: once `entity:user::name::value` is correctly tagged, shape matching traverses every entity reference field pointing at `User` (node `uid`, `revision_uid`; media `uid`, `revision_user`; file `uid`; canvas_page `owner`, `revision_user`) and surfaces the user name as a reachable string prop source across all of them.
