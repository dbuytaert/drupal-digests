## Summary

The Canvas module in Drupal CMS 1.1.0 caused a fatal error when visiting the display management page for view modes not controlled by Canvas templates, such as Teaser or Search index. The error occurred because Canvas's custom routing controller did not properly delegate to the standard form controller when no Canvas template existed. This regression broke the ability to configure any view mode that wasn't using Canvas, especially when Layout Builder was enabled.

## Impact

No upgrade or configuration changes required.

## Technical details

The Canvas module overrides the route controller for entity view mode display forms via `ViewModeDisplayController`. When a view mode has an associated Canvas content template, it redirects to the Layout Builder interface. When no template exists, it should fall back to the standard entity view display form.

The bug occurred in the fallback path. The original code manually instantiated the form object using `EntityTypeManagerInterface::getFormObject()`, retrieved the entity from the route match, and passed it to `FormBuilderInterface::getForm()`. This approach failed when Layout Builder was enabled because Layout Builder's route subscriber alters the controller and form, but the manual instantiation bypassed that integration.

The fix changes `ViewModeDisplayController` to inject and delegate to `HtmlEntityFormController` instead. When no Canvas template exists, it calls `HtmlEntityFormController::getContentResult()`, passing the request and route match. This properly invokes the standard entity form controller chain, allowing Layout Builder and other form alterations to work correctly.

The test coverage adds `testLayoutBuilderCompatibility()` which installs Layout Builder, creates a view mode with Layout Builder enabled, and verifies the display page loads without errors.
