## Summary

A bug in Canvas 1.1.0 caused fatal errors when administrators tried to configure display settings for view modes not managed by Canvas (like Teaser or Search index). The crash occurred specifically when Layout Builder was installed, preventing access to the Manage Display pages for these view modes. This has been fixed so all view mode configuration pages now work correctly regardless of whether Canvas manages them.

## Impact

No upgrade or configuration changes required.

## Technical details

The `ViewModeDisplayController` in the Canvas module was incompletely handling view modes that don't have a corresponding `ContentTemplate`. When no template existed, the controller attempted to manually build the entity view display form by retrieving the form object and entity from the route match, then passing it to `FormBuilderInterface`.

This approach failed when Layout Builder was enabled because Layout Builder's route subscriber (`LayoutBuilderRoutes::onAlterRoutes`) modifies the route to use a custom form class (`LayoutBuilderEntityViewDisplayForm`). That form expects specific initialization that the manual form building process didn't provide, resulting in a null reference error when `getLayoutBuilderUrl()` was called.

The fix refactors `ViewModeDisplayController::__invoke()` to use `HtmlEntityFormController` instead of manually building the form. This service properly handles all entity form routing, including Layout Builder's customizations. The controller now simply delegates to `HtmlEntityFormController::getContentResult()` when no Canvas template exists, ensuring the correct form is built with proper initialization.

A new test method `testLayoutBuilderCompatibility()` verifies that view modes with Layout Builder enabled but no Canvas template can be accessed without errors.
