## Summary

Canvas now validates language config overrides (`LanguageConfigOverride`) for its config entity types (`ContentTemplate` and `PageRegion`) against config schema and constraints. Drupal core has never validated these overrides beyond checking that values are scalars, arrays, or null, meaning an invalid translation (for example, a boolean stored where a URI is expected) could silently corrupt a site. Canvas fills this gap with two complementary mechanisms: a test-time event subscriber that throws a `ConfigException` when an invalid override is saved, and a runtime entity constraint that surfaces violations through the standard entity validation API.

## Impact

Affects module maintainers.

- **New API:** New `CanvasConfigEntityTranslationsAreValidConstraint` (plugin ID `CanvasConfigEntityTranslationsAreValid`) constraint added to `ContentTemplate` and `PageRegion` config entity types; `ConfigTranslationSupportHooks::TRANSLATABLE_ENTITY_TYPE_IDS` public constant added.

## Technical details

Drupal core's `LanguageConfigOverride::save()` only calls `StorableConfigBase::validateValue()`, which rejects objects and resources but ignores schema. Core's `ConfigSchemaChecker` (the test-time strict validator) explicitly skips non-default collections, because overrides are partial and cannot be validated alone. The fix merges the override data onto the base config via `NestedArray::mergeDeepArray()` before validating, replicating what `Config::setOverriddenData()` does at read time.

Two mechanisms are added. First, `LanguageConfigOverrideSchemaChecker` (`@internal`, test-only) is an `EventSubscriberInterface` that listens to `LanguageConfigOverrideEvents::SAVE_OVERRIDE`. When a save targets a Canvas config entity (identified by `canvas.content_template.` or `canvas.page_region.` config name prefixes), it merges the override onto the base config and validates via `TypedConfigManagerInterface`. Violations result in a `ConfigException`. It is registered as an event subscriber in both `CanvasKernelTestBase` and `FunctionalTestBase`, mirroring how core registers `ConfigSchemaChecker`. Second, `CanvasConfigEntityTranslationsAreValidConstraint` (plugin ID `CanvasConfigEntityTranslationsAreValid`) is added to `ContentTemplate` and `PageRegion` entity types in `hook_entity_type_alter()` via `ConfigTranslationSupportHooks`. Its validator (`CanvasConfigEntityTranslationsAreValidConstraintValidator`) iterates over all configured languages, merges each language's override onto the base config, validates via typed config, and surfaces violations through the entity validation API. The canonical list of translatable Canvas config entity types is centralized in `ConfigTranslationSupportHooks::TRANSLATABLE_ENTITY_TYPE_IDS`.

Both mechanisms are explicitly temporary workarounds for the long-standing core issue [#2270399](https://www.drupal.org/node/2270399) (a `@todo` in `LanguageConfigOverride::save()` for over a decade), and carry `@todo Remove when ...` comments pointing to that issue.

## Upgrade

Tests that save `LanguageConfigOverride` values for Canvas config entities (`ContentTemplate`, `PageRegion`) now fail with a `ConfigException` if the merged result (base config + override) violates config schema or constraints. Ensure all such saves in tests use valid values. Example of what now throws:

```php
// Saving a non-URI string into a URI-typed prop now throws ConfigException
// in test environments (where LanguageConfigOverrideSchemaChecker is active):
$this->saveConfigEntityTranslation($template, 'nl', $nl_values_with_invalid_uri);
// → ConfigException: Validation of the "canvas.content_template...." LanguageConfigOverride failed: ...
```

At runtime (non-test), the `CanvasConfigEntityTranslationsAreValidConstraint` constraint on `ContentTemplate` and `PageRegion` means that calling `$entity->validate()` will now return violations for each language whose override produces an invalid merged config.
