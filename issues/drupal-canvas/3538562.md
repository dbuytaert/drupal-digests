## Summary

When the `canvas_stark` theme is not installed or not enabled, Canvas's contextual edit forms now display a clear warning message instead of rendering broken or empty. Previously, if the install hook that enables `canvas_stark` failed to fire (for example due to a recipes issue), the affected UI would appear buggy in ways that were hard to trace back to the missing theme. The warning makes the root cause immediately visible.

## Impact

No upgrade or configuration changes required.

## Technical details

Both `EntityFormController::form()` and `ComponentInstanceForm::buildForm()` now declare a `ThemeHandlerInterface` constructor dependency. At the top of each method, before any form building occurs, the code calls `$this->themeHandler->themeExists('canvas_stark')` and checks `listInfo()['canvas_stark']->status`; if either check fails, the method returns early with a plain `#markup` render array containing a human-readable warning. The kernel test `PageMetatagIntegrationTest::testSeoSettingsForm()` was updated in the same commit to install `canvas_stark` via `ThemeInstallerInterface` before constructing the controller, and to pass the `ThemeHandlerInterface` instance explicitly to the `EntityFormController` constructor, since the new guard would otherwise cause the test to hit the early-return path and miss the SEO form assertions.

## Upgrade

If you instantiate `EntityFormController` or `ComponentInstanceForm` directly (e.g., in a test), you must now pass a `ThemeHandlerInterface` instance:

```php
// Before
$controller = new EntityFormController($autoSaveManager, $requestStack);

// After
$themeHandler = $container->get('theme_handler');
$controller = new EntityFormController($autoSaveManager, $requestStack, $themeHandler);
```

For `ComponentInstanceForm`, the `create()` factory method already wires this automatically from the container, so no change is needed for normal service instantiation.
