## Summary

On a fresh Standard profile install, Drupal Canvas now enables only six core block plugin components in the Library by default: Main navigation, Footer, Breadcrumb, Messages, Powered by Drupal, and Site branding. All other components sourced from core block plugins (such as User login, Recent content, Tabs, Page title, Who's new, Administration menu, and others) are disabled by default. Site builders can re-enable any of them individually through the Manage Library screen. Additionally, JavaScript (code) components no longer appear under a spurious "@todo" folder in the Library.

## Impact

Affects module maintainers.

- **API change:** `ComponentInterface::getCategory()` return type widened to `string|TranslatableMarkup|null`; `category` field removed from the `canvas/api/v0/config/component` HTTP API response and the `BaseComponent` TypeScript interface.

## Technical details

The `BlockManager::getDefinitions()` override was extended to programmatically disable any block sourced from core or a core-packaged module, unless the block's plugin ID is listed in the new `BlockManager::BLOCKS_TO_KEEP_ENABLED` constant. The allowed list contains exactly six IDs: `system_powered_by_block`, `system_branding_block`, `system_breadcrumb_block`, `system_messages_block`, `system_menu_block:main`, and `system_menu_block:footer`. A special case handles Views-provided blocks: a view block is disabled only if its underlying view is tagged `default` (indicating it was created by core), while custom or contributed view blocks remain enabled.

The `JsComponent` source plugin previously set `category` to the string `'@todo'` when creating component config entities, which caused a visible "@todo" folder in the Library. This is now set to `NULL`, suppressing folder creation entirely until the proper solution tracked in [#3541364](https://www.drupal.org/node/3541364) lands.

As a side effect of allowing `NULL` categories, the `category` config schema field was changed from `type: required_label` to `type: label` with `nullable: true` and a `NotBlank` constraint (allowing `null` but not an empty string). `ComponentInterface::getCategory()` return type was widened from `string|TranslatableMarkup` to `string|TranslatableMarkup|null`, and `Component::postSave()` now checks `$category === NULL` rather than `empty($category)` to decide whether to create a Folder. The `category` field was also removed from the `canvas/api/v0/config/component` HTTP API response entirely, and the `BaseComponent` TypeScript interface was updated to match. The related follow-up for removing `category` from `Component` config entities altogether is tracked in [#3549726](https://www.drupal.org/node/3549726).

## Upgrade

**`ComponentInterface::getCategory()` return type is now nullable:**

```php
// Before
public function getCategory(): string|\Drupal\Core\StringTranslation\TranslatableMarkup;

// After
public function getCategory(): string|\Drupal\Core\StringTranslation\TranslatableMarkup|null;
```

Callers that assume a non-null return must add a null check.

**`category` is no longer present in the `/canvas/api/v0/config/component` HTTP API response.** Any front-end or integration code reading `component.category` from that endpoint will need to be updated.

**`BlockManager::BLOCKS_TO_KEEP_ENABLED`** is a new public constant listing the six core block plugin IDs that remain enabled by default. Tests or integrations that expect all core block components to be enabled must explicitly re-enable them, e.g.:

```bash
drush cset canvas.component.block.system_menu_block.account status 1 -y
```
