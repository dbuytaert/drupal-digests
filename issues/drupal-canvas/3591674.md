# #3591674: Explicitly support `comment` as a referenceable content entity type (stop omitting it from the content entity types endpoint)

- **Project:** Drupal Canvas
- **Type:** Normal feature request
- **Version:** 1.x-dev
- **Status:** Fixed
- **Discussion:** [#3591674](https://git.drupalcode.org/project/canvas/-/work_items/3591674) · 1 contributor · 4 comments

## Summary

In the Canvas module, the content entity types API endpoint now correctly lists `comment` as a referenceable entity type for users who hold `access comments` and have view access to the parent entity type (e.g., `access content` for comments on nodes). Previously, `comment` was silently absent from the picker even when permissions were in place, because the access check always failed on the stub entity used for eligibility testing.

## Impact

No upgrade or configuration changes required.

## Technical details

Canvas evaluates whether an entity type/bundle is referenceable by creating an unsaved stub entity and running a `'view'` access check on it via `getBundleViewAccess()`. `CommentAccessControlHandler::checkAccess()` calls `$entity->getCommentedEntity()->access('view')`, which throws on a stub that has no commented entity. The error was caught by `getBundleViewAccess()` and treated as forbidden, so `comment` was never included.

The fix is in `ApiUiContentEntityReferenceControllers::createBundleStub()`. After creating a `comment` stub, the method now loads the comment type's `targetEntityTypeId` and creates a normalized stub of that parent type (using the same activation/publish normalization extracted into the new `normalizeStub()` helper), then injects it as the `entity_id` of the comment stub. This gives the access handler a valid parent to check. If the parent entity type has no bundles yet, or its provider module has been uninstalled leaving a dangling `target_entity_type_id`, the method returns `NULL` and the comment bundle is skipped cleanly. The `$inject_commented_entity` parameter defaults to `TRUE` but is passed as `FALSE` for the recursive parent-stub call to prevent infinite recursion on comment-on-comment setups.

`getBundleViewAccess()` remains as a last-resort catch for unexpected handler failures, but is no longer the primary path for comment. The test `testContentEntityTypesCatchesStubAccessThrow()` (which previously asserted `comment` was absent) is replaced by three tests: `testCommentIsReferenceableWithPermissions()` asserts `comment` appears in the listing and its fields endpoint returns 200; `testCommentOmittedWithoutAccessCommentsPermission()` and `testCommentOmittedWithoutParentTypeViewAccess()` pin the conjunction-gate behavior.
