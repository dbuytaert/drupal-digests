## Summary

Drupal Canvas now supports more flexible component metadata files. Components can now use either the traditional index-style metadata (`component.yml` in a component directory) or the new named-style metadata (e.g., `button.component.yml` alongside other components in a shared directory). The validation rules have been updated to accommodate both styles while maintaining consistency checks. The older, stricter validation rules remain available for the deprecated `build-d` command to ensure backward compatibility.

## Impact

No upgrade or configuration changes required.

## Technical details

This change updates the `@drupal-canvas/eslint-config` package to support the new flexible component discovery introduced in [#3571534](https://www.drupal.org/project/canvas/issues/3571534). The original `required` ESLint config has been renamed to `requiredDeprecated` and is now used exclusively by the deprecated `npx canvas build-d` command.

A new `required` config was created with relaxed rules. Three strict rules were removed from the new config and marked as deprecated: `component-files`, `component-imports`, and `component-no-hierarchy`. These rules enforced the old rigid structure where each component lived in its own directory with specific file naming requirements.

The remaining rules were updated to support both index-style metadata (`component.yml` in a dedicated component directory) and named-style metadata (e.g., `button.component.yml` in a shared directory). The `component-dir-name` rule now validates that the `machineName` matches either the directory name (index-style) or the filename prefix before `.component.yml` (named-style). The `component-exports` rule was refactored to use a new `isComponentEntrypoint()` utility that correctly identifies entry files for both styles.

TypeScript file extensions (`.ts`, `.tsx`) were added to the file patterns across all config variants, expanding beyond the original JavaScript-only support.
