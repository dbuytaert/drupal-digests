## Summary

The Canvas module's `getSiteData()` function now exposes the active theme's logo and favicon through a new `themeAssets` field. Code components can access `getSiteData().themeAssets.logo.url` and `getSiteData().themeAssets.favicon.url` (plus `favicon.mimeType`) without reaching outside Canvas's data layer. This is a data-layer addition with no admin UI changes.

## Impact

Affects module maintainers.

- **New API:** New `themeAssets` field on the `SiteData` type returned by `getSiteData()`, exposing `logo.url`, `favicon.url`, and `favicon.mimeType` from the active theme; backed by the new `canvas/canvasData.v0.themeAssets` Drupal library and `CodeComponentDataProvider::getCanvasDataThemeAssetsV0()`.

## Technical details

Canvas exposes Drupal data to JavaScript code components through the `drupalSettings.canvasData.v0.*` namespace, with each data slice backed by a dedicated Drupal asset library and a PHP provider method. Before this change, `getSiteData()` covered branding (site name, slogan, home URL) and `baseUrl` but had no canonical way to supply theme logo or favicon paths.

The fix adds a new slice following the existing pattern:

**PHP (`CodeComponentDataProvider`):** `getCanvasDataThemeAssetsV0()` reads `theme_get_setting('logo.url')`, `theme_get_setting('favicon.url')`, and `theme_get_setting('favicon.mimetype')`, returning them under the `v0.themeAssets` key.

**PHP (`ComponentSourceHooks`):** `'canvas/canvasData.v0.themeAssets' => 'getCanvasDataThemeAssetsV0'` is added to `ASSET_LIBRARY_METHOD_MAPPING`. The `jsSettingsAlter()` conditional block populates the setting only when `themeAssets.logo.url` is still `NULL`, preserving any value set earlier and allowing downstream overrides via `array_replace_recursive`.

**Asset library (`canvas.libraries.yml`):** A new `canvasData.v0.themeAssets` library entry is declared with `drupalSettings` defaults (`logo.url: null`, `favicon.url: null`, `favicon.mimeType: null`). The parent `canvasData.v0` library's dependency list and the `canvas.js_component.*` config schema `Choice` constraint are both updated to include `v0.themeAssets` (with the full list alphabetically ordered, which also corrects the previous ordering of `v0.jsonapiSettings`).

**TypeScript (`drupal-utils.ts`):** A `ThemeAssets` interface is defined with `logo: { url: string }` and `favicon: { url: string; mimeType: string }`. `SiteData` gains a `themeAssets: ThemeAssets` field, and `getSiteData()` reads `window.drupalSettings?.canvasData?.v0?.themeAssets` with a safe fallback. The `DrupalSettings` type definition is also extended to match.
