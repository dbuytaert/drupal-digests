## Summary

Canvas kernel tests were verbose and hard to understand because they each had to install dozens of modules and configure systems before running. A new `CanvasKernelTestBase` class now handles common setup automatically, cutting over 900 lines of repetitive code across the test suite. This makes existing tests simpler to read and future tests faster to write, improving developer experience for Canvas contributors without changing any public APIs or functionality.

## Impact

No upgrade or configuration changes required.

## Technical details

Canvas already used kernel tests extensively because they run faster than functional tests. However, every kernel test needed to manually declare 15-30 modules in `$modules`, call `installConfig()` and `installEntitySchema()` multiple times, and enable strict config schema checking despite Canvas being a contrib module. This resulted in hundreds of lines of boilerplate.

The new `CanvasKernelTestBase` class extends `KernelTestBase` and automatically installs all of Canvas's direct and indirect dependencies (including `system`, `user`, `block`, `editor`, `ckeditor5`, `filter`, `text`, `datetime`, `image`, `link`, `media_library`, `options`, `path`, `file`, `media`, `path_alias`, `views`, and test components). It enables `$strictConfigSchema` by default and installs the `stark` theme plus essential config like date formats needed by Canvas fields.

The class deliberately does not install entity schemas, allowing tests to opt in only to the schemas they need. This avoids the performance overhead of installing and tearing down entity tables for every test. This decision was explicitly documented in the class docblock after discussion in the merge request.

Over 60 existing kernel tests were converted to extend `CanvasKernelTestBase`, removing their redundant module lists and setup code. The diff shows +322, −992 lines. A dozen tests that use `CanvasTestSetup` (a test fixture designed for end-to-end tests) were left unconverted and marked with `@todo` comments pointing to a follow-up issue.

Notable exceptions include `ModuleInstallationTest` and `RecipeSubscriberTest`, which must test Canvas installation itself and therefore cannot use a base class that pre-installs Canvas. These are documented with inline comments explaining why they use `KernelTestBase` directly.

Phenaproxima opened the issue and created the initial implementation after recognizing that Canvas kernel tests were becoming difficult to maintain. Wim Leers completed the majority of the merge request (authoring over 50% of the final changes) after the issue stalled, converting dozens of tests to the new base class and documenting edge cases discovered during conversion. The collaboration pattern of one contributor starting an improvement and another finishing it when bandwidth allowed proved effective for large-scale refactoring across a 60+ test suite.

## Lessons learned

- When creating a base test class that installs modules, installing entity schemas can significantly slow down the test suite. Install only module configuration and let individual tests opt into schema installation as needed. *(via phenaproxima, wim leers)*
- When refactoring test infrastructure with broad impact, limit scope to foundational improvements that unblock future work. Defer dependent technical debt cleanup to follow-up issues rather than expanding scope indefinitely. *(via wim leers)*
