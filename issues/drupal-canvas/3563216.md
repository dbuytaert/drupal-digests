## Summary

Canvas kernel tests required a lot of repetitive setup code that made them verbose and hard to understand. A new base test class now handles the common setup automatically, making tests easier to write and maintain. Over 60 existing tests were updated to use this base class, removing nearly 1,000 lines of boilerplate code while keeping the same test coverage.

## Impact

## Technical details

The new `CanvasKernelTestBase` class extends `KernelTestBase` and provides standardized setup for Canvas kernel tests. It automatically installs all of Canvas's direct and indirect module dependencies (like `ckeditor5`, `editor`, `filter`, `media_library`, `image`, `link`, and the `canvas_test_sdc` test module). It also enables strict config schema validation by default, which is normally disabled for contributed modules but critical for Canvas since the project makes extensive use of validation.

The base class intentionally does not install any content entity schemas, not even Canvas's own `canvas_page` entity type. Tests must opt in to installing only the schemas they need. This keeps tests fast by avoiding unnecessary schema installation overhead. The class documents this explicitly so developers understand when to use it versus `KernelTestBase`.

Over 60 existing kernel tests were converted to extend `CanvasKernelTestBase` instead of `KernelTestBase`. This removed their `$modules` arrays and `setUp()` boilerplate, cutting 992 lines of code while adding 322 lines (mostly the new base class and documentation). Tests that cannot yet be converted due to their use of `CanvasTestSetup` now have TODO comments pointing to [#3531679](https://www.drupal.org/project/canvas/issues/3531679).

## Lessons learned

- When creating a shared test base class that installs dependencies, installing entity schemas universally can significantly slow test suites. Let individual tests install only the schemas they need, even if it means some repetition. *(via phenaproxima, wim leers)*
- When refactoring test infrastructure, scope the work to establish the foundation first. Converting all existing tests and addressing related technical debt can happen in follow-up issues once the base pattern is proven. *(via wim leers)*
