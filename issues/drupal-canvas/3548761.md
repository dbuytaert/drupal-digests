## Summary

When building a Canvas content template, linking an optional SDC prop to an entity field that happens to be empty on the preview entity now works correctly. Previously the linking process would silently fail to complete: the empty field caused the prop to be dropped from the values array that the server used to recognize dynamic prop sources, so the prop appeared unlinked even after the user confirmed the choice. This was a critical bug that broke the `ContentTemplate` linking workflow for any non-required field that had no value.

## Impact

Affects module maintainers.

- **API change:** `StructuredDataPropExpressionInterface::isSupported()` renamed to `validateSupport()` with return type changed from `bool` to `void`; all implementing classes must be updated.

## Technical details

The root cause was in `GeneratedFieldExplicitInputUxComponentSourceBase::clientModelToInput()`. After calling `PropSource::parse($prop_source)`, the code passed the result directly to `$source->evaluate($host_entity, $is_required_prop)`. For a `DynamicPropSource` pointing to an empty optional field, `evaluate()` returns `NULL`, which caused that prop's entry to be omitted from the input array. Later, `buildComponentInstanceForm()` looked up `$inputValues[$sdc_prop_name]` to determine whether a prop is dynamic; finding nothing, it could not reconstruct the dynamic link.

The fix adds an early branch in `clientModelToInput()`: when `PropSource::parse()` produces a `DynamicPropSource`, the method skips `evaluate()` entirely. Instead it asserts that a `$host_entity` is present (throwing `\InvalidArgumentException` otherwise), calls `$source->expression->validateSupport($host_entity)` to confirm the expression is structurally compatible with the entity type and bundle, and then stores the source via `$this->collapse($source, $prop)` before continuing. Runtime value evaluation is not needed to persist a dynamic linkage, so bypassing it is both correct and necessary.

Two related changes accompanied the fix. First, `StructuredDataPropExpressionInterface::isSupported()` was renamed to `validateSupport()`, with its return type changed from `bool` to `void`. All six implementing classes (`FieldObjectPropsExpression`, `FieldPropExpression`, `FieldTypeObjectPropsExpression`, `FieldTypePropExpression`, `ReferenceFieldPropExpression`, `ReferenceFieldTypePropExpression`) were updated to drop the `return TRUE` statements. The method's contract is now accurate: it validates by throwing `\DomainException` on incompatibility and does nothing on success. Second, `DynamicPropSource::$expression` was widened from `private` to `public` readonly so that `clientModelToInput()` can reach it to call `validateSupport()`.

New test coverage was added in `SingleDirectoryComponentTest::testClientModelToInput()` (with a data provider covering: dynamic source without a host entity, dynamic source with an empty non-required prop, dynamic source with a non-empty prop, and a bundle-mismatched expression) and in `ComponentInstanceFormTest::testDynamicProps()` for the full form-request integration path. A shared helper `getFormCanvasPropsForComponent()` was extracted to eliminate duplication between the two test classes.

## Upgrade

Any class implementing `StructuredDataPropExpressionInterface` must rename `isSupported()` to `validateSupport()` and change its return type from `bool` to `void`:

```php
// Before
public function isSupported(EntityInterface|FieldItemInterface|FieldItemListInterface $entity_or_field): bool {
    if (/* incompatible */) {
        throw new \DomainException('...');
    }
    return TRUE;
}

// After

}
```
