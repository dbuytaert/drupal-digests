## Summary

Code (JS) components in Drupal Canvas can now be used as sources of dynamic prop suggestions when building ContentTemplates. Before this change, the `/api-ui/content-template/suggestions/…` endpoint returned a 400 error ("Code components are not supported yet.") for any component not backed by an SDC plugin. That restriction is lifted: all component sources that declare their input schema via `ComponentMetadata` now get the same Drupal Field-powered dynamic prop suggestion UX that SDC components already had.

## Impact

No upgrade or configuration changes required.

## Technical details

The root problem was that `GeneratedFieldExplicitInputUxComponentSourceBase` held a single abstract method `getSdcPlugin()` (public) that forced every subclass to supply a live SDC plugin instance for all metadata operations, even simple things like reading the prop schema or slot definitions. Because SDC's `ComponentValidator::validateProps()` genuinely requires a full plugin object, the class had no clean way to separate "I only need metadata" from "I need the whole plugin."

The fix splits the responsibility. The public `getSdcPlugin()` is removed and replaced by two methods: `getComponentPlugin()` (protected abstract, called only inside `validateComponentInput()` where the full plugin object is unavoidable) and `getMetadata()` (public concrete, lazily returns `ComponentMetadata` from the plugin and caches it). Every internal call that previously accessed `$this->getSdcPlugin()->metadata->…` now calls `$this->getMetadata()->…` instead.

The prop-enumeration logic (`PropShape::getComponentProps()` and `PropShape::getComponentPropsForMetadata()`) is removed from `PropShape` and promoted to a public static method `GeneratedFieldExplicitInputUxComponentSourceBase::getComponentInputsForMetadata(string $plugin_id, ComponentMetadata $metadata)`. This decouples prop iteration from having a live plugin object.

`FieldForComponentSuggester::suggest()` gains an explicit `ComponentMetadata $component_metadata` parameter in place of the internal `ComponentPluginManager::find()` call, so callers now pass metadata directly. `ComponentSourceInterface` gains `getSourceSpecificComponentId()`, which `ComponentSourceBase` implements by reading `$this->getConfiguration()['local_source_id']`; the `Fallback` and `InvalidSlots` plugins override it to return an empty string. In `ApiUiContentTemplateControllers`, the `instanceof SingleDirectoryComponent` guard and its 400 response are removed, using `$source->getSourceSpecificComponentId()` and `$source->getMetadata()` instead of `$source->getSdcPlugin()->getPluginId()`. `ComponentPropExpression::$componentName` is renamed to `$sourceSpecificComponentId` and the class is annotated `@internal`. All touched classes (`ComponentSourceInterface`, `ComponentSourceBase`, `PropShape`, `FieldForComponentSuggester`) carry `@internal`, so these are internal restructuring changes that happen to produce a user-observable behavior improvement.

## Upgrade

All affected symbols are `@internal`. If you referenced them directly, note the following shifts:

- `getSdcPlugin()` (public) is gone; use `getMetadata()` for metadata access and `getComponentPlugin()` (now protected) only when a full SDC plugin instance is required.
- `PropShape::getComponentProps()` and `PropShape::getComponentPropsForMetadata()` are removed; call `GeneratedFieldExplicitInputUxComponentSourceBase::getComponentInputsForMetadata($plugin_id, $metadata)` instead.
- `FieldForComponentSuggester::suggest()` now requires a `ComponentMetadata` argument after `$component_plugin_id`:
```php
// Before
$suggester->suggest($component_plugin_id, $entity_data_definition);

// After
$suggester->suggest($component_plugin_id, $source->getMetadata(), $entity_data_definition);
```
- `ComponentPropExpression::$componentName` is renamed to `$sourceSpecificComponentId`.
- Custom `ComponentSourceInterface` implementations that extend `ComponentSourceBase` inherit `getSourceSpecificComponentId()` automatically; plugins that do not have a `local_source_id` in their config (e.g. fallback-style plugins) must override it to return `''`.
