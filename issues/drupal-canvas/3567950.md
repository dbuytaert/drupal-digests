## Summary

Canvas now enforces a coding standard requiring global PHP functions to be called with a leading backslash (e.g. `\sprintf()` instead of `sprintf()`). This allows PHP to apply compile-time optimizations for improved performance. The change started with two commonly used functions (`assert()` and `sprintf()`) to avoid disrupting existing work. Additional functions will be converted gradually through follow-up issues. This aligns with a similar proposal for Drupal core and ensures consistency across the codebase.

## Impact

No upgrade or configuration changes required.

## Technical details

The Slevomat Coding Standard rule `FullyQualifiedGlobalFunctions` was added to Canvas's PHPCS configuration. When global PHP functions are called with a fully-qualified namespace (e.g. `\sprintf()`), the PHP compiler can apply optimizations at compile time. The list of functions that benefit from this optimization is documented in the PHP source code.

The initial implementation targets only `\assert()` and `\sprintf()` to minimize merge conflicts with in-progress work. The PHPCS rule is configured with an `include` property listing these two functions. The codebase was updated using simple find-and-replace operations: ` sprintf(` to ` \sprintf(` and `(sprintf(` to `(\sprintf(`, with similar changes for `assert()`.

Follow-up issues were created to gradually extend this pattern to other commonly used global functions like `in_array()`, `array_key_exists()`, `array_map()`, `array_keys()`, and type-checking functions. The team decided against using `ReferenceUsedNamesOnly` because it would require verbose `use function` statements at the top of every file. The fully-qualified approach is more consistent with existing Canvas conventions and Drupal core's direction.

## Lessons learned

- When calling PHP global functions from namespaced code, use fully-qualified names (e.g., \sprintf() instead of sprintf()). Modern PHP can apply compile-time optimizations to functions like array_key_exists(), in_array(), is_array(), and others, but only when they are fully-qualified. This provides performance benefits without changing behavior. *(via wim leers, larowlan, catch)*
- When adopting a new code style convention that requires widespread changes, introduce it gradually to avoid disrupting active merge requests. Start with the most commonly used functions, create follow-up issues for remaining work, and provide clear guidance for novice contributors to help with incremental adoption. *(via wim leers)*
- When code reviews consistently address the same convention, automate it with coding standards tools. Automated feedback removes noise from the review process and increases development velocity by establishing consistency without requiring human intervention. *(via wim leers)*
