## Summary

Canvas now enforces a coding standard that requires certain PHP global functions to be called with a leading backslash, like `\sprintf()` instead of `sprintf()`. This change improves performance because modern PHP can optimize fully-qualified global function calls at compile time. The standard currently applies to `\assert()` and `\sprintf()`, with plans to expand to more functions incrementally. This automated check reduces noise during code review and ensures consistency across the codebase.

## Technical details

The `SlevomatCodingStandard.Namespaces.FullyQualifiedGlobalFunctions` PHPCS rule was added to `phpcs.xml` with an initial configuration targeting only `assert()` and `sprintf()`. This addresses a long-standing inconsistency where some parts of Canvas used fully-qualified global functions while others did not, creating confusion during code review.

The change is motivated by PHP compile-time optimizations: when global functions are fully-qualified with a leading backslash, the PHP compiler can apply several performance improvements for certain built-in functions. The specific functions that benefit are documented in the PHP source code. Related follow-up issues were created to gradually expand the rule to other high-frequency functions like `in_array()`, `array_key_exists()`, `array_map()`, `array_keys()`, and type-checking functions.

The implementation updates all existing occurrences of `assert()` and `sprintf()` throughout Canvas modules, entity handlers, controllers, and tests to use the fully-qualified form. The diff shows systematic replacements across approximately 180 files. This approach allows the standard to be enforced without disrupting all existing merge requests at once. Additional functions will be added incrementally through novice-friendly follow-up issues.

## Lessons learned

- When calling PHP global functions from namespaced code, fully-qualify the function name with a leading backslash (e.g., \sprintf() instead of sprintf()). Modern PHP applies compile-time optimizations to fully-qualified global function calls that it cannot apply to non-qualified calls, improving performance without requiring code changes beyond the naming convention. *(via wim leers, larowlan)*
- When adopting a coding standard that requires mass changes across a codebase, roll it out incrementally by function frequency rather than all at once. Start with the most commonly used functions to maximize impact while minimizing disruption to existing work, then create follow-up issues for remaining cases. This reduces merge conflicts and allows the team to adjust to the new convention gradually. *(via wim leers)*
