## Summary

When a site builder submits a prompt in the Drupal Canvas AI Wizard, the chat panel now shows live progress updates for each agent that executes rather than a blank wait. Each step appears with a spinning indicator while running and a green checkmark when it finishes. Steps use plain language (e.g. "Thinking", "Building the page", "Generate a title") rather than internal agent identifiers. The progress area is updated in place so the chat is not flooded; once the main response arrives the final result is appended normally. Intermediate progress steps are not stored in chat history and do not reappear on page reload.

## Impact

Affects module maintainers and site owners.

- **Requirements:** Dev dependency on `drupal/ai_agents` raised from `^1.1.2` to `^1.2@beta`.
- **New API:** New `/admin/api/canvas/ai-progress` GET endpoint returns per-agent execution status; the main `/admin/api/canvas/ai` endpoint now requires a `request_id` field in the POST body.

## Technical details

The feature is built on a polling architecture using the `AiAgentStatusPollerServiceInterface` introduced in `ai_agents` 1.2. When a prompt is submitted, the frontend generates a unique `request_id` and attaches it to the POST body. The `CanvasBuilder` controller (injecting `ai_agents.agent_status_poller`) passes this ID to the agent via `setProgressThreadId()` and enables detailed tracking for three event types via `setDetailedProgressTracking()`: `AiAgentStatusItemTypes::Started`, `AiAgentStatusItemTypes::TextGenerated`, and `AiAgentStatusItemTypes::Finished`.

A new route `canvas_ai.get_progress` at `/admin/api/canvas/ai-progress` (protected by CSRF token and the `use Drupal Canvas AI` permission) is handled by the new `CanvasBuilder::getAiProgress()` method. It calls `AiAgentStatusPollerServiceInterface::getLatestStatusUpdates()`, maps the returned events into items keyed by agent ID, and returns a JSON payload with an `items` array (each item carrying `id`, `type`, `name`, `description`, `status`, and `generated_text`) plus an `is_finished` flag. User-friendly descriptions are resolved by `getAgentDescription()` via a static map from agent IDs such as `canvas_ai_orchestrator`, `canvas_page_builder_agent`, and `canvas_title_generation_agent` to translated strings.

On the frontend, `startPolling()` in `AiWizard.jsx` fires every two seconds (max 500 polls). It maintains an `itemStatuses` Map and renders a `<div>` block into the chat via `chatEl.addMessage()` on first receipt and `chatEl.updateMessage()` on subsequent polls, tracking the message index to update in place. Inline CSS injected via `auxiliaryStyle` provides `.aiLoader` (a CSS-only spinning border animation) and `.aiCompletedIcon` (a bordered circle with a checkmark pseudo-element). The main `/admin/api/canvas/ai` fetch is no longer `await`ed; its resolved value is stored as `pendingResponse` and processed only after the polling signals completion, ensuring the final LLM reply appears after all progress steps. The dev dependency on `drupal/ai_agents` is raised from `^1.1.2` to `^1.2@beta` to access the new poller service.
