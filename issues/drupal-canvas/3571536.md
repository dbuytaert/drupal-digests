## Summary

The Canvas CLI now ships a `push` command (`npx canvas push`) that handles the full workflow of getting local code components onto a Drupal site in one step. It discovers components, compiles Tailwind CSS, bundles third-party vendor dependencies and local alias imports, uploads components as `js_component` config entities, and uploads build artifacts as managed files tracked through the global `asset_library`. Fonts support is deferred to a separate issue. The command is marked experimental and supports `--dir` and `--yes` flags for non-interactive (CI/CD) use.

## Impact

Affects module maintainers and site owners.

- **Requirements:** Canvas module now requires the `drupal:file` module as a dependency.
- **Config change:** New optional `imports`, `assets`, and `shared` sequences added to the `canvas.asset_library.*` config schema, each entry mapping an import specifier to a `public://` file URI.
- **New API:** New `POST /canvas/api/v0/artifacts/upload` endpoint handled by `ApiArtifactController` stores build artifacts as managed File entities.

## Technical details

The push workflow is split into two API interactions to avoid concurrent write conflicts on the `asset_library` config entity. First, artifact files (vendor bundles, local alias bundles, shared chunks) are uploaded in parallel to `POST /canvas/api/v0/artifacts/upload`, handled by the renamed `ApiArtifactController`. Each file is stored as a managed File entity via Drupal's file system and tracked with `file_usage`. Once all uploads complete, a separate sync request updates the `asset_library` manifest in one atomic write.

The `canvas.asset_library.*` config schema gains three new optional sequences: `imports`, `assets`, and `shared`. Each entry is typed as `canvas.asset_library_manifest_entry`, a mapping of `name` (the import specifier or chunk path) and `uri` (constrained to the `public://` scheme). These entries are used by Drupal to build the browser import map when code components are rendered. The canvas module now declares a dependency on `drupal:file` to support managed File entity creation in `ApiArtifactController`.

On the CLI side, the previously separate `collectImports`, `bundleVendorDependencies`, and `bundleLocalAliasImports` calls are consolidated into a single `analyzeAndBundleImports` utility, used by both the `build` and `push` commands. The `build` command is updated to pass `sharedChunks` into `generateManifest`. The `canvas_oauth` `CanvasOauthAuthenticationProvider` is updated to apply OAuth authentication to the artifact upload route. All `canvas_ai` kernel tests are migrated from core's `KernelTestBase` to `CanvasKernelTestBase` to reduce module list duplication and ensure consistent setup.

A known limitation discussed in the thread: because `asset_library` config entities now reference managed File entities, these file references do not travel with a standard Drupal config export/import. Sites using the CLI alongside config sync must include a CLI push step in their deployment pipeline. The same underlying problem exists for `ContentTemplate` entities referencing media files (see [#3532574](https://www.drupal.org/node/3532574)).

## Upgrade

No changes are required for existing code. The new `imports`, `assets`, and `shared` keys on `asset_library` config entities are optional (`requiredKey: false`). Sites not using the CLI are unaffected.

Developers adopting the CLI `push` command should note that build artifacts are now uploaded as managed files and tracked in the global `asset_library`. A `config:export` taken after a push will include file URI references that will not resolve on environments where the artifacts have not been pushed:

```bash
# Deploy to another environment: run push there too
npx canvas push --yes
```
