## Summary

The Canvas CLI now includes a `push` command that uploads components, global CSS, and build artifacts from your local codebase to Drupal in one step. This streamlines the workflow for developers managing code components outside the browser-based editor. The command discovers components, builds Tailwind CSS, bundles third-party dependencies and local imports, and uploads everything including the manifest that maps import specifiers to file locations. This replaces the need to run build, upload, and asset sync steps separately.

## Impact

No upgrade or configuration changes required.

## Technical details

The `push` command orchestrates the full workflow: component discovery, Tailwind CSS compilation, dependency analysis and bundling, component upload, global asset library update, and artifact synchronization.

The implementation splits artifact handling into three categories. Component source and compiled output are stored directly in `js_component` config entities. Global CSS and JS go into the `asset_library` config entity. Vendor dependencies, local alias imports, and shared chunks are uploaded as managed File entities and tracked in the asset library's `imports`, `assets`, and `shared` manifest properties.

The artifact upload process runs in parallel for throughput. Files are streamed individually to `/canvas/api/v0/artifacts/upload`, which creates managed File entities tracked via `file_usage`. Once all files are uploaded, the CLI sends a synchronization request to update the asset library manifest. This two-step approach avoids concurrent writes to the config entity during parallel uploads.

The `asset_library` config schema now includes `imports`, `assets`, and `shared` sequences. Each entry maps an import specifier or chunk name to a managed file URI. When components reference third-party packages or local aliases, Drupal resolves these using the manifest to generate the browser import map.

Local alias imports are now bundled with content hashes in their filenames (e.g. `utils-abc123.js`). Multiple files with the same base name from different directories are disambiguated with numeric suffixes in the Vite entry name. The Vite manifest maps source paths to hashed output filenames, which the CLI uses to build the final import map.

The CLI's test suite was refactored to use `CanvasKernelTestBase` instead of core's `KernelTestBase`, ensuring consistent module setup and reducing duplication.

### Collaboration

The issue took six weeks from opening to commit with active collaboration among eight contributors. Early discussion clarified that the `pull` command would be deferred to a future issue, narrowing scope to the push workflow.

The review process involved multiple architectural questions. One key simplification came when the team realized the manifest synchronization endpoint could be eliminated by moving file usage logic into a config entity presave hook, though the two-endpoint approach was kept to avoid concurrent write issues during parallel uploads.

A significant portion of the discussion addressed how this change interacts with Drupal's config sync workflow. The team acknowledged that File entities referenced from config entities complicate traditional config deployment, similar to challenges with default content. The decision was to accept this limitation for now, noting that sites using the CLI can include it in their deployment pipeline.
