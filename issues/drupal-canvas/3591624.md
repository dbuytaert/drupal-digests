# #3591624: Block validation is broken for any block without a default value for `label_display`

- **Project:** Drupal Canvas
- **Type:** Critical bug report
- **Version:** 1.x-dev
- **Status:** Fixed
- **Discussion:** [#3591624](https://www.drupal.org/node/3591624) · 4 contributors · 9 comments

## Summary

A critical bug in the Canvas module caused block validation to fail for any block plugin that did not provide a default value for `label_display` in its configuration. This meant such blocks were incorrectly excluded from the Canvas component library and could not be used as Canvas components. The bug was introduced by the stricter block validation added in MR #781 and was breaking all Drupal CMS CI pipelines. Sites using Canvas with block plugins that rely on the inherited default for `label_display` were affected.

## Impact

No upgrade or configuration changes required.

## Technical details

The bug was introduced in `BlockComponentDiscovery::checkRequirements()` when MR #781 added stricter block settings validation. That method constructs a dummy config array to pass to `TypedConfigManagerInterface::createFromNameAndData()` for the `block.settings.*` schema, so it can validate the block plugin's default configuration. The dummy array included `'label_display' => FALSE` (a PHP boolean), but the `block.settings.*` config schema requires `label_display` to be the string `'0'` (representing "hidden") rather than a boolean. Passing `FALSE` produced a type violation, causing the validation to report errors for every block that did not explicitly override `label_display` in `defaultConfiguration()`. The correct value `'0'` was already used elsewhere in the same file at the point where real component metadata is built (line 169 of `BlockComponentDiscovery.php`). The fix brings the dummy config in `checkRequirements()` in line with that existing correct usage by changing `'label_display' => FALSE` to `'label_display' => '0'`.
