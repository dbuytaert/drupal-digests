## Summary

Sites using the Canvas module's code components with `image` or `video` prop types would get a 500 error ("Extraneous JSON Schema information detected") when saving a page that relied on the default prop values. This happened even when the schema was technically correct, because the comparison logic only sorted the top-level JSON Schema keys and not the nested `properties` sub-keys. The fix applies a recursive key sort before comparing, so key ordering differences inside nested objects no longer trigger the false-positive error.

## Impact

No upgrade or configuration changes required.

## Technical details

`DefaultRelativeUrlPropSource::parse()` in `src/PropSource/DefaultRelativeUrlPropSource.php` validates that a stored JSON Schema contains no extraneous information by comparing it against a minimal normalized form. The comparison used shallow `ksort()` calls on both schemas, but the `properties` sub-object for multi-key types like `video` (which has `src` and `poster` keys, each with several attributes such as `format`, `pattern`, `contentMediaType`) could have its nested keys in a different order depending on how they were stored. The shallow sort left those inner key orders unsorted, so a strict `!==` comparison would fail even when the schemas were semantically equivalent.

The fix replaces the two `ksort()` calls with a new private static `recursiveKsort(array &$array)` method that sorts every level of the array recursively. A `@todo` is added to remove this local helper once Canvas requires Drupal 11.3, which will ship `SortArray::recursiveKsort()` via core issue [#3556987](https://www.drupal.org/node/3556987). A kernel test assertion was added to `PropSourceTest` that deliberately reverses both the top-level `properties` key order and the attribute key order within a single property, then asserts that `PropSource::parse()` still returns a valid `DefaultRelativeUrlPropSource` instance.
