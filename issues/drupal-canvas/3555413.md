## Summary

When editing a Canvas `ContentTemplate`, site builders working with link props (props typed `type: string, format: uri-reference`) can now link directly to the canonical URL of any referenced entity, not just the host entity. For example, on an article node template the prop source suggester now offers options like "Authored by → URL" (the author's user profile page) and "Revision user → URL", in addition to the existing "Canonical absolute URL" for the host node itself. This makes it straightforward to build components that link tags, authors, related media, or other referenced entities to their canonical pages.

## Impact

Affects module maintainers.

- **New API:** New `url` computed field property on `EntityReferenceItem` (via `CoreFeatureEntityReferenceItemAddPropertiesTrait`) returns the canonical relative URL of the referenced entity as a `ComputedEntityCanonicalRelativeUrl` Typed Data value.

## Technical details

The implementation adds a computed `url` field property to `EntityReferenceItem::propertyDefinitions()` by extracting the property-definition logic from `EntityReferenceItemOverride` into a new `CoreFeatureEntityReferenceItemAddPropertiesTrait`. The trait now registers two properties: the existing `target_uuid` and a new `url` property backed by the new `ComputedEntityCanonicalRelativeUrl` data type. `ComputedEntityCanonicalRelativeUrl` extends `Uri`, walks up the Typed Data parent chain to fetch the referenced entity, calls `toUrl('canonical')`, and performs combined view + URL access checks mirroring `EntityReferenceLabelFormatter`. The property is only registered when the target entity type has a `canonical` link template.

Because a referenced entity may be inaccessible to the current user, the `url` property's `getValue()` returns a `MaybeUrl` value object (a new `@internal` class combining a `GeneratedUrl` and an `AccessResultInterface` with full cache metadata). When the prop is required and evaluation yields `NULL` due to access denial, `Evaluator::evaluate()` now throws a `CacheableAccessDeniedHttpException` carrying the access-result cache metadata and reason string. For optional props the evaluator simply returns `NULL`. This access pathway reuses the existing `DynamicPropSource` Typed Data traversal with no new suggestion infrastructure.

`FieldPropExpression::calculateDependenciesForProperty()` was extended to accept a `$targeted_delta` parameter so that dependency tracking for the new computed property (which implements `DependentPluginInterface`) only records content dependencies for the specific delta being targeted, rather than all deltas. A separate one-character bugfix was made in `UriTargetMediaTypeConstraint::isValid()`, where the regex `/\w+\/w+/` was corrected to `/\w+\/\w+/`.

### Collaboration

The design reaching this commit was non-trivial. An earlier implementation had landed a `HostEntityUrlPropSource` limited to the host entity ([#3545859](https://www.drupal.org/node/3545859)), and this follow-up issue initially proposed renaming it to `EntityUrlPropSource` and teaching it to traverse arbitrary reference chains. Wim Leers raised detailed objections: depth-of-indirection semantics were unspecified, the label collision for multi-cardinality tag fields was unresolved, and the traversal infrastructure would need a major rewrite. After a synchronous meeting between lauriii, wim leers, phenaproxima, and effulgentsia, the team agreed on a hybrid: keep `HostEntityUrlPropSource` unchanged for the host entity, and instead expose canonical URLs as a computed field property on entity reference fields, reusing the existing `DynamicPropSource` machinery with zero new traversal code. Wim Leers summarized the rationale in comment #14: adding a computed field *property* to a field type is far safer than adding a computed *base field* to every entity type, because field type property definitions are rarely overridden by contrib, making the approach feasible for eventual upstream contribution to Drupal core.
