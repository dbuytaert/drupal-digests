# #3549124: Confirm overriding local global CSS file with the download command

- **Project:** Drupal Canvas
- **Type:** Normal feature request
- **Version:** 1.x-dev
- **Status:** Fixed
- **Discussion:** [#3549124](https://git.drupalcode.org/project/canvas/-/work_items/3549124) · 1 contributor · 2 comments
- **Followers:** 2

## Summary

The Canvas CLI tool previously fetched `global.css` from Canvas silently in the background during `build` and `upload`, and the `download` command would overwrite the local `global.css` without warning. This issue, resolved via [#3560957](https://www.drupal.org/node/3560957), introduces explicit control over global CSS syncing. The `download` and `upload` commands now list "Global CSS" as a selectable option in interactive mode (pre-selected by default). The `build` command switches to a local-first strategy: it reads the local `global.css` when it exists instead of always pulling from the remote API. New `--skip-css` and `--css-only` flags for both `download` and `upload` give non-interactive and CI/CD users precise control over whether and what CSS is transferred.

## Impact

Affects module maintainers.

- **New API:** New `--skip-css` and `--css-only` flags on the `download` and `upload` CLI commands; `getGlobalCss()` and `downloadGlobalCssInBackground()` utility functions added to `build-tailwind.ts`

## Technical details

The core change is a `getGlobalCss(useLocal: boolean = true)` function added to `build-tailwind.ts`. When `useLocal` is true and a local `global.css` exists under the configured `CANVAS_COMPONENT_DIR`, it is read from disk; otherwise the function falls back to `downloadGlobalCssInBackground()`, which calls `apiService.getGlobalAssetLibrary()` without touching the local file. Both `buildComponent()` in `build.ts` and `buildTailwindForComponents()` in `build-tailwind.ts` now accept a `useLocalGlobalCss` parameter (default `true`) and pass it through to `getGlobalCss`, replacing their previous direct calls to `apiService.getGlobalAssetLibrary()` for CSS.

The `component-selector.ts` module gains a `GLOBAL_CSS_SELECTOR` sentinel string and a `determineGlobalCssSelection()` helper. Both `selectLocalComponents()` and `selectRemoteComponents()` return an `includeGlobalCss` boolean in their result types (`LocalSelectorResult` and `RemoteSelectorResult`). In interactive mode, a "Global CSS" option is inserted immediately after "All components" and pre-selected when `globalCssDefault` is true. The `--css-only` path short-circuits component discovery entirely and returns an empty component list with `includeGlobalCss: true`.

The `download` command now fetches components and global CSS in parallel via `Promise.all` (unless `--css-only` is set, in which case only CSS is fetched), and writes the local `global.css` only when `includeGlobalCss` is true. The `upload` command skips the global CSS upload when `includeGlobalCss` is false, and passes `includeGlobalCss` through `getBuildAndUploadResults` to control which CSS source is used for the Tailwind build. An inline `@todo` workaround that re-fetched the existing `css.original` from the API before every upload (to avoid overwriting it with an empty value) was removed, since `getGlobalCss()` now provides the correct local source.

## Upgrade

CLI users and CI/CD scripts that call `npx canvas build`, `npx canvas download`, or `npx canvas upload` should note the behavior changes:

- `build` and `upload` now use the **local** `global.css` by default instead of fetching CSS from Canvas on every run. Ensure your local `global.css` is up to date before building.
- `download` now includes global CSS in the download by default. Use `--skip-css` to preserve the old skip behavior.
- New flags for `download` and `upload`:

```bash
# Skip global CSS during download or upload
npx canvas download --all --skip-css
npx canvas upload --all --skip-css

# Transfer only global CSS, skip components
npx canvas download --css-only
npx canvas upload --css-only
```

- `--skip-css` and `--css-only` cannot be combined; doing so throws an error.

### Collaboration

The design was shaped by a focused exchange between balintbrews and lauriii early in the thread. lauriii questioned whether silently compiling with the remote CSS when a user skips the global CSS upload would produce confusing "side effects," particularly in multi-user environments where one person edits via CLI and another edits in the browser code editor. In response, balintbrews updated the issue summary to adopt lauriii's preferred behavior: when global CSS upload is skipped, the build falls back to fetching the remote CSS in the background without touching the local file, making any differences in compiled output an explicit consequence of the user's choice rather than a hidden surprise.
