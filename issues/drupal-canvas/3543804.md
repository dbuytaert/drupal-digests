## Summary

The Manage Library panel in the Canvas (Experience Builder) module received several UI behavior fixes. When a new code component is created, the panel now automatically switches to the Code tab. When a user clicks on a code component from any tab, the Code tab stays selected rather than reverting to a default. Folders created under the Code tab are now visible even when no code components have been added yet. Components listed in the Manage Library panel are also correctly non-draggable, matching the context where dragging is not meaningful.

## Impact

No upgrade or configuration changes required.

## Technical details

The root cause of the tab-persistence and auto-navigation problems was that `ManageLibrary` used a local `defaultValue` on `Tabs.Root`, meaning the active tab was not tracked anywhere persistent. The fix lifts the active tab into Redux: a new `manageLibraryTab: string | null` field and `setManageLibraryTab` action are added to `primaryPanelSlice`, with a matching `selectManageLibraryTab` selector. `ManageLibrary` now reads from and writes to this slice via `onValueChange`. When `AddCodeComponentDialog` successfully creates a component it dispatches both `setManageLibraryTab('code')` and `setActivePanel('manageLibrary')`; `AddToComponentsDialog` does the same for `'components'`.

The folder visibility bug was in `folderfyComponents`: the early-return guard was `!folders || !components` (either null aborts), changed to `(!folders && !components)` (both null required to abort). `CodeComponentList` also extended its empty-state callout condition to check `!folderEntries.length` in addition to no code components, so an empty folder list no longer suppresses the callout prematurely.

The `DisplayContext` React context (`manage-library` / `component-library` / `layout`) is removed entirely. `ListItem`, `PatternNode`, `ExposedJsComponent`, and `CodeComponentList` previously read from it to decide draggability and dropdown visibility; they now call `useAppSelector(selectActivePanel)` and compare against `'manageLibrary'` directly. The `includeDropdown` prop on `SidebarNode` is removed; callers now control dropdown rendering by passing `null` for `dropdownMenuContent` when no dropdown is wanted.

## Upgrade

If your custom code consumes `DisplayContext` or `useDisplayContext` from `@/components/sidePanel/DisplayContext`, that module is deleted. Replace with `useAppSelector(selectActivePanel)` from `@/features/ui/primaryPanelSlice` and compare against the `'manageLibrary'` string.

If your code passes `includeDropdown` to `SidebarNode`, that prop is removed. Control dropdown visibility by passing `null` or a valid element as `dropdownMenuContent` instead.

```tsx
// Before
<SidebarNode includeDropdown={false} dropdownMenuContent={<Menu />} />

// After
<SidebarNode dropdownMenuContent={null} />
```
