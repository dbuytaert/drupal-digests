## Summary

Component previews in Safari were broken due to authentication issues. When hovering over code components in the library, Safari would show an HTTP 401 error and authentication prompt instead of the preview. This was caused by iframes lacking the authentication context needed for assets that require session information. The fix changes how previews are rendered so they inherit the current session, making previews work correctly in Safari and other browsers.

## Impact

No upgrade or configuration changes required.

## Technical details

The component preview system creates iframes to display component markup. Previously, it generated HTML content, created a blob URL with `URL.createObjectURL()`, and set that as the iframe's `src` attribute. In Safari, iframes loaded from blob URLs do not share the authentication context of the parent page. Code components include assets that require authentication (like `api/v0/auto-save` endpoints), causing 401 errors.

The fix switches from `src` with a blob URL to `srcDoc` with inline HTML content. Iframes using `srcDoc` inherit the parent page's security context and session, allowing authenticated requests to succeed. The solution also converts relative asset URLs (in `<link>`, `<script>`, and `<canvas-island>` tags) to absolute URLs so they load correctly.

An additional enhancement watches for size changes in components with `<canvas-island>` elements. These can change dimensions after hydration, so an interval checks every 250ms for up to 5 seconds and recalculates scaling if the size changes. This ensures previews remain properly scaled even after client-side hydration completes.
