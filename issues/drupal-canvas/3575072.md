## Summary

Component preview tooltips in Safari were broken due to authentication failures. When hovering over code components in the component library, Safari would display an HTTP authentication dialog instead of the preview. This happened because code components request assets that require authentication, but the iframe used for previews did not have access to the current user session. The fix changes how preview content is loaded into the iframe so authentication works correctly across all browsers.

## Impact

No upgrade or configuration changes required.

## Technical details

The component preview system previously used a Blob URL created from the preview HTML and passed it to the iframe's `src` attribute. This approach caused the iframe to load in a context without the parent page's authentication session. When code components tried to fetch authenticated assets (like auto-save API endpoints), Safari blocked them with a 401 error and showed an authentication prompt.

The solution switches from `src` with a Blob URL to using the `srcDoc` attribute directly with the HTML content. This keeps the iframe in the same authentication context as the parent page, allowing authenticated requests to succeed.

The change also required making asset paths absolute instead of relative, since `srcDoc` iframes handle base URLs differently than `src` iframes. A helper function now converts relative paths in `<link>`, `<script>`, and `<canvas-island>` elements to absolute URLs before rendering.

For code components with `<canvas-island>` elements that can change size after hydration, the implementation adds an interval that monitors for dimension changes over 5 seconds and re-applies scaling logic when detected.
