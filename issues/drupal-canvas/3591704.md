# #3591704: Only show default-language entities when reviewing auto-saved (pending) changes

- **Project:** Drupal Canvas
- **Type:** Critical task
- **Version:** 1.x-dev
- **Status:** Fixed
- **Discussion:** [#3591704](https://git.drupalcode.org/project/canvas/-/work_items/3591704) · 1 contributor · 5 comments

## Summary

On multilingual Canvas sites, the pending-changes review list now shows one row per entity (its default-language version) instead of one row per translation. Acting on a non-default-translation row was already broken: discarding it would delete the default language instead, and publishing a subset of translations would corrupt the unselected ones. Collapsing to a single default-language row per entity prevents editors from hitting those broken paths until langcode-aware endpoints are added in a follow-up ([#3591703](https://www.drupal.org/node/3591703)). A separate bug where Canvas with the metatag module installed could recurse infinitely when reconstructing a draft entity is also fixed in this commit.

## Impact

Affects site owners.

- **Config change:** New `canvas.auto_save.entity_memory_cache` service added to `canvas.services.yml`; existing auto-save entries written before this update lack `is_default_translation` and are treated as default-translation entries (safe fallback).

## Technical details

The `ApiAutoSaveController` GET handler now filters the pending-changes list with `array_filter` to drop any entry where `is_default_translation` is falsy before building the response.

To support that filter, `AutoSaveManager::saveEntity()` now writes a new `is_default_translation` boolean alongside the existing `langcode` key when persisting an auto-save entry. Storing it at write time (when the real entity object is available) avoids an expensive `loadUnchanged()` call at read time, since whether a given langcode is the default is entity-specific and cannot be determined from the langcode string alone. A `@phpstan-type AutoSaveEntry` type alias documents the full shape of these stored arrays.

Entity reconstruction was extracted from `getAutoSaveEntity()` and `getAllAutoSaveList()` into a new private `createEntityFromAutoSaveEntry()` method. This method calls `enforceIsNew(FALSE)` and, for revisionable entities, `updateLoadedRevisionId()`, removing duplicated post-reconstruction code from `ApiLayoutController` and `EntityFormController`. It also calls `ContentEntityBase::setDefaultTranslationEnforced()` when the entity is translatable, using the new `is_default_translation` flag (defaulting to `TRUE` for older entries that predate the field).

The infinite-recursion fix changes the constructor-injected `$cache` from `cache.static` (a serializing backend) to a new `canvas.auto_save.entity_memory_cache` service backed by `LruMemoryCache`. Serializing a live entity object triggers `__sleep()`, which causes metatag's computed field to evaluate the `[current-page:title]` token, which calls `ApiLayoutController::getLabel()`, which calls `getAutoSaveEntity()` again. The LRU cache stores the object reference directly without serialization, breaking the cycle. The cache is bounded to 100 entries to be safe in long-running server processes such as FrankenPHP or RoadRunner workers.
