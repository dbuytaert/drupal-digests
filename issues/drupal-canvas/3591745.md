# #3591745: Enabling content translation of Canvas pages results in an unsupported state (asymmetrical translations) by default: force the known valid state if the field is marked as translatable

- **Project:** Drupal Canvas
- **Type:** Critical bug report
- **Version:** 1.x-dev
- **Status:** Fixed
- **Discussion:** [#3591745](https://git.drupalcode.org/project/canvas/-/work_items/3591745) · 1 contributor · 12 comments

## Summary

When site builders enabled content translation on Canvas pages, the standard Drupal "Content language and translation" settings form presented checkboxes for both the component input values and the component tree, allowing asymmetrical configurations that Canvas does not support. This fix enforces the only valid state: component input values are translatable, the component tree is shared across languages. The form now hides the column-group checkboxes and renames the field label to "Component input values" to clarify what is actually being translated. A post-update hook (`canvas_post_update_0022`) repairs any site that previously reached an invalid configuration. Going forward, the correct state is also set automatically on module install and rejected at config import time.

## Impact

Affects site owners.

- **Database update:** Post-update hook `canvas_post_update_0022_enforce_symmetrical_canvas_page_components_translation` creates or corrects the `components` base field override to enforce symmetrical translation settings.
- **Config change:** New `core.base_field_override.canvas_page.canvas_page.components` config object is created on install and enforced by a post-update hook; config schema constraints and a config import validator reject the asymmetrical `translation_sync` combination.

## Technical details

The `components` base field of the `canvas_page` entity type stores both the component tree (which instances exist, in what order) and per-instance input values. Content translation's `FieldTranslationSynchronizer` uses the `translation_sync` third-party setting on a `BaseFieldOverride` to decide which column groups to synchronize across translations. The only supported combination is `inputs => 'inputs'` (translatable) and `tree => '0'` (shared). Previously the UI exposed checkboxes for both, so any user could accidentally enable the unsupported asymmetrical mode.

Enforcement is now layered across four surfaces. First, `ModuleHooks::afterBuildCanvasPageLanguageSettings()` removes the `components` column-group sub-checkboxes from the form, renames the field label, and `ModuleHooks::validateCanvasPageLanguageSettings()` (a new form validator) hard-codes the symmetrical values into form state before the settings are saved. Second, a `hook_modules_installed` implementation in `ContentTranslationHooks` calls `ComponentTreeFieldSymmetricalTranslationSynchronizer::ensureSymmetricalCanvasPageComponents()` to create or update the `BaseFieldOverride` whenever either `canvas` or `content_translation` is installed while the other is already present. Third, a new config schema file `canvas_symmetrical_translations_only.schema.yml` defines `Choice` constraints on the `translation_sync` mapping, making the valid state machine-checkable. Fourth, `SymmetricalTranslationsConfigImportValidator` (registered as an event subscriber on `ConfigImporterEvent`) validates the staged override before any config import and logs an error to abort the import if the `tree` column group is set to translatable. The post-update hook `canvas_post_update_0022` handles four distinct cases: no `content_translation` installed (no-op), an existing override with invalid settings (corrected), an existing override already valid (no-op), and `content_translation` present but no override yet (created). All of this is marked `@todo Remove in` issue [#3571130](https://www.drupal.org/node/3571130), which tracks a planned core fix that would make the form-level workaround unnecessary.

## Upgrade

Sites running Canvas with `content_translation` enabled should run database updates to trigger `canvas_post_update_0022`, which will create or fix the `core.base_field_override.canvas_page.canvas_page.components` config object:

```bash
drush updatedb
```

If you manage config with Config Management (CMI/recipes), be aware that importing a staged override with `translation_sync.tree` set to `'tree'` (the asymmetrical value) will now be rejected at import time with a message containing "Canvas Pages support only symmetrical translations". Correct the staged config to `translation_sync: { inputs: inputs, tree: '0' }` before importing.
