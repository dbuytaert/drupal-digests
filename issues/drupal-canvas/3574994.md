## Summary

The right sidebar in Drupal Canvas (where component properties are edited) can now be resized by dragging a handle. Users can drag the handle to adjust the sidebar width between 320px and 640px, and the chosen width is saved in localStorage so it persists across reloads and sessions. The sidebar collapses automatically when no component is selected in the template editor or when interactive mode is active, and expands back to the saved width when a component is selected again.

## Impact

No upgrade or configuration changes required.

## Technical details

The implementation ships in three commits. The first introduces the `allotment` library to replace `react-resizable-panels` throughout the codebase and implements `Allotment`-based pane splitting in both `Editor.tsx` and `CodeEditorUi.tsx`. A new `layoutStorage.ts` utility (`getLayoutItem` / `setLayoutItem`) wraps `localStorage` with error handling so the UI falls back to defaults when storage is unavailable.

The third commit reworks the main editor sidebar after manual QA revealed that `Allotment` stores pane sizes as percentages: toggling the left sidebar changes the total container width, which causes the right sidebar to visually shift. The fix introduces a new `EditorLayout` component (`EditorLayout.tsx`) that replaces `Allotment` for the main editor split. `EditorLayout` uses a custom pointer-event drag implementation: `handlePointerDown` captures the pointer, `pointermove` computes the new pixel width by subtracting `e.clientX` from the layout container's right edge, and `pointerup` releases capture and persists the final value. A companion `editorLayoutStorage.ts` stores and loads the width as a plain pixel integer (`SIDEBAR_MIN_PX` 320px, `SIDEBAR_MAX_PX` 640px, default 320px). The right column hides with a CSS `width: 0; transition: width 0.15s ease-out` rather than `display: none`, so the collapse animates smoothly. `AppRoutes` now renders `EditorLayout` instead of `Editor` directly; `Editor` is stripped back to its original role of managing context and rendering `EditorFrame`.

The code editor (`CodeEditorUi`) continues to use nested `Allotment` components for its horizontal and vertical splits. Panel sizes are persisted separately under the key `canvas-code-editor-layout`. The `ContextualPanel` CSS switches from a fixed `width: var(--sidebar-right-width)` to `flex: 1; min-width: 0` so it fills its `EditorLayout`-managed column. The `--sidebar-right-width` CSS variable is removed from the global token set. Playwright and Cypress tests are updated to account for the new visibility transitions: `waitForContextualPanel()` is called after component selection, and the `previewReady()` command switches its opacity assertion from `.canvasEditorFrameScalingContainer` to `[data-testid="canvas-editor-frame"]`.

During manual verification, mayur-sose found that toggling the left sidebar caused the right sidebar width to change (test case 2.0 fail). jessebaker reviewed the report and initially decided it was acceptable behavior, noting that the extra space was simply shared proportionally and fixing it would require significant work for little benefit. lauriii disagreed, pushed back on the issue thread, and submitted a third MR that switched from percentage-based `Allotment` sizing to a custom pixel-based drag implementation.
