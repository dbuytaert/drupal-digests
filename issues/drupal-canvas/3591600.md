## Summary

The `ai_plus` module replaced its HTML injection pattern for communicating tool results back to the browser with a proper Drupal AJAX command pipeline. Previously, AI tool actions queued raw HTML strings (for example, open-link anchor tags) that were injected directly into DeepChat messages. Now, tool actions queue typed `CommandInterface` objects through a `UiCommandQueue` service. These commands are serialized as a `drupal_commands` key in the DeepChat API JSON response and executed by the browser using Drupal's standard AJAX command system. The polling and instant-refresh auto-reload mechanism is also replaced by new `RefreshComponentsController` and `DeferredProcessingController` endpoints. The module-level config setting `refresh_mode` is removed and replaced by `placeholder_media_id`.

## Impact

Affects module maintainers and site owners.

- **Config change:** `ai_plus.settings` key `refresh_mode` (string) removed; replaced by `placeholder_media_id` (nullable integer)
- **API change:** `ChatResponseQueue::add(string $html)` replaced by `UiCommandQueue::add(CommandInterface $command)` and `UiCommandQueue::addRendered(array $command)`

## Technical details

The old `ChatResponseQueue` service accumulated raw HTML strings; `DeepChatMessagePrepender` (via `hook_deepchat_prepend_message`) flushed them into chat response text. The "open link" action specifically injected `<a class="ai-plus-open-link">` HTML that a `MutationObserver` in the DeepChat shadow DOM would detect and process.

The new design introduces `UiCommandQueue`, which accepts `CommandInterface` instances via `add()` or pre-rendered arrays via `addRendered()`. A new `DeepChatResponseListener` event subscriber listens on `KernelEvents::RESPONSE`, targets only the `ai_chatbot.api` route, and injects the rendered command list as `drupal_commands` into the outgoing JSON. The client-side `ai-plus-command-processor.js` wraps DeepChat's `responseInterceptor` to extract `drupal_commands` and execute them through `Drupal.ajax().success()`. The open-link action becomes the new `OpenLinkCommand` PHP class (renders `{'command': 'aiPlusOpenLink', 'url': ...}`) handled by `Drupal.AjaxCommands.prototype.aiPlusOpenLink` in `ai-plus-open-link.js`.

The auto-refresh subsystem is overhauled: `ChangeCheckController` and the `ai_plus.change_check` route are removed; `ai_plus.refresh_components` (POST) replaces them via `RefreshComponentsController`. Two new routes `ai_plus.deferred_processing` and `ai_plus.deferred_retry` are added for `DeferredProcessingController`. A `DeferredProcessorManager` plugin collector and `ImageGenerationProcessor` handle async image-generation tasks. The `ai_plus_post_update_add_refresh_mode` update hook is deleted. JavaScript files are now injected into the `ai_chatbot/deepchat` library via `hook_library_info_alter` rather than through a standalone `open_link` library. A new `Drupal.AiPlus.log()` utility (`ai-plus-logging.js`) enables opt-in debug logging via `localStorage`.

## Upgrade

**1. Replace `ChatResponseQueue` calls with `UiCommandQueue`:**

Old code:
```php
// Injected raw HTML into the chat response.
$this->chatResponseQueue->add('<a href="' . $url . '" class="ai-plus-open-link">' . $title . '</a>');
```

New code:
```php
use Drupal\ai_plus\Ajax\OpenLinkCommand;

// Queue a typed AJAX command instead.
$this->uiCommandQueue->add(new OpenLinkCommand($url, $title));
```

**2. Config schema change in `ai_plus.settings`:**

Old:
```yaml
refresh_mode: polling
```

New:
```yaml
placeholder_media_id: 1
```

Remove any code reading `ai_plus.settings:refresh_mode`. The `ai_plus_post_update_add_refresh_mode` update hook has been deleted, so sites that have not yet run it will skip it cleanly.

**3. Removed routes and libraries:**

- `ai_plus.change_check` route no longer exists; replace references with `ai_plus.refresh_components` (POST) or the new deferred routes.
- The `ai_plus/open_link` Drupal library is removed; any explicit `#attached` references to it must be dropped.
