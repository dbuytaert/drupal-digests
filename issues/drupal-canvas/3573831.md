## Summary

Canvas code components now support a new "entity reference" prop type that lets a component declare it wants to receive live data from a specific Drupal content entity (node, user, taxonomy term, etc.). Instead of duplicating entity data by hand or relying on automated JSON:API lists with no curation control, a content author can handpick exactly which entities to feature, arrange them freely in Canvas, and have fields such as title, price, or image stay automatically in sync with the source entity. This unblocks common use cases like a hand-curated promotional page that still reflects live product prices, or a content template that passes the host entity directly into a "card"-style code component.

## Impact

Affects module maintainers.

- **New API:** New `content-entity-reference` prop type for `JavaScriptComponent` config entities; field selections stored under `dataDependencies.entityFields.<propName>` as `EntityFieldBasedPropExpressionInterface` expressions.

## Technical details

This commit lands ADR #11 (`docs/adr/0009-entity-reference-props-in-code-components.md`), the formal architectural decision for the feature. The implementation itself is spread across child issues: [#3586613](https://www.drupal.org/node/3586613) added the `content-entity-reference` well-known prop shape to the JSON schema, [#3587374](https://www.drupal.org/node/3587374) added `HostEntityPropSource` so a content template can resolve the rendering host entity into such a prop, and [#3574857](https://www.drupal.org/node/3574857) added the client-side transform for the `entity_reference_autocomplete` widget.

The ADR specifies that each "entity reference" prop is expressed in config as `type: object, $ref: json-schema-definitions://canvas.module/content-entity-reference` on a `JavaScriptComponent` config entity, and the selected field data is stored under `dataDependencies.entityFields.<propName>` as a list of `EntityFieldBasedPropExpressionInterface` expressions. At render time these expressions are evaluated against the referenced entity object to produce an object-shaped value with keys derived from field names (using entity keys as aliases where applicable, e.g. `owner` instead of `uid` for nodes). When the referenced entity is absent or deleted the prop value is `null`, and code components must handle this case. Because `JavaScriptComponent` is a config entity, it can express config dependencies on the target entity type and bundle, allowing import validation to check whether a code component's entity reference props are satisfiable on the importing site. This capability is explicitly out of scope for SDCs, which are plugins rather than config and cannot carry such dependencies.

### Collaboration

Pierre Dureau (pdureau) raised a concern in the thread that coupling components to entity data contradicts the SDC philosophy of "dumb", stateless, design-system-native UI building blocks decoupled from Drupal's data model. Wim Leers responded by agreeing explicitly that entity reference props must never be added to SDCs, citing the structural reason: SDCs are plugins and cannot declare config dependencies on bundles or fields, while `JavaScriptComponent` config entities can. He committed an update to the ADR to close the question definitively, replacing language that had been too open-ended about future SDC support.
