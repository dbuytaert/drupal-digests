## Summary

The Canvas UI was incorrectly sending a `value` field when linking dynamic or host entity URL data sources to component properties. This caused the server to receive unnecessary data and in some cases triggered errors. The UI was originally built around static prop sources, and dynamic sources were added later without fully refactoring the linking mechanism. This fix updates the prop linker to only send the source reference, not a value placeholder, making the data flow cleaner and preventing the server from receiving malformed requests.

## Impact

No upgrade or configuration changes required.

## Technical details

The `PropLinker` component was sending `{[propName]: 'noop'}` in the `values` object whenever linking any prop source type, even though only `StaticPropSource` needs a `value` field. `DynamicPropSource` and `HostEntityUrlPropSource` only require a `source` reference since their values are computed server-side.

The fix introduces a new action `_linkPropToEntityValue` that replaces `_updateExistingComponentValuesForLinking` specifically for prop linking scenarios. This new action only updates the `source` field in the model and explicitly removes any resolved value for the linked prop by deleting it from the `resolved` object before sending to the server.

The refactored code checks that a valid evaluated component model exists and that the prop exists in the component metadata before proceeding. It then sends only the updated `source` and cleaned `resolved` object to the `updateComponent` endpoint, eliminating the unnecessary `value` field that was causing issues.

## Lessons learned

- When sending data to an API that only accepts application/x-www-form-urlencoded, JavaScript undefined values are serialized as the string literal "undefined" rather than being omitted. Either omit the key-value pair entirely from the object before serialization, or switch to application/json where undefined values are automatically omitted during JSON.stringify(). *(via jessebaker)*
- When building UI features around new abstractions (like DynamicPropSource) on top of a system designed for older patterns (like StaticPropSource), bolting on new functionality without refactoring creates technical debt. Under aggressive deadlines, this is understandable and necessary, but the underlying assumptions should be refactored once time permits. *(via wim leers)*
