# #3591699: Entity-reference component inputs (e.g. image src) are exposed as translatable, breaking rendering of translated config

- **Project:** Drupal Canvas
- **Type:** Critical bug report
- **Version:** 1.x-dev
- **Status:** Fixed
- **Discussion:** [#3591699](https://git.drupalcode.org/project/canvas/-/work_items/3591699) · 2 contributors · 7 comments

## Summary

On multilingual Canvas sites, components whose inputs are backed by entity references (such as an image populated via a file or image field reference) would fail to render in non-default languages. The config translation workflow stored an empty override value for those inputs because they appeared translatable by schema shape. At render time that empty override shadowed the real entity reference, producing a Twig `RuntimeError` and a blank component while the rest of the page loaded normally. This fix ensures that only genuinely author-entered (static) inputs are extracted for translation; entity-reference-backed inputs are excluded from the translation workflow entirely.

## Impact

No upgrade or configuration changes required.

## Technical details

Translatability was derived from a component input's JSON Schema shape in `JsonSchemaPropsComponentInstanceInputsConfigSchemaGenerator`. A URI-typed image `src` looks like a translatable string at schema level, so it was marked translatable even when populated by an `EntityFieldPropSource`. The translation UI had no widget for it, but TMGMT and `config_translation` still wrote an empty override, which shadowed the base reference at render time.

The fix introduces `ComponentInputsConfigProcessor` in `src/Tmgmt/ComponentInputsConfigProcessor.php`. This class extends `DefaultConfigProcessor` and overrides `extractTranslatables()`. When the schema element is a `ComponentInputsMapping`, the processor bypasses the existing schema and builds a "concrete schema" from scratch by inspecting the actual underlying field type (`field.value.*`) for each prop. Inputs populated by a `StaticPropSource` get a concrete type derived from `field.value.*`; inputs from any other source (entity field, host entity URL, etc.) get `type: ignore` and are skipped. The processor is registered in `canvas_dev_translation` via `hook_config_schema_info_alter()`, which adds a `tmgmt_config_processor` key to the schema definitions for `canvas.content_template.*.*.*` and `canvas.page_region.*`.

The same hook alters the core schema definitions `field.value.link` and `field.value.uri` to replace their URI sub-properties with a new `translatable_uri` type, making URI-esque inputs from `link` and `uri` field types correctly extractable by TMGMT.

`CanvasStaticPropSourceFieldWidget::setConfig()` received two corrections: multi-cardinality field submissions now strip the `add_more` pseudo-submit key by filtering to integer-keyed deltas only, and individual sub-key overrides are now stored at the granular `$subkey` path rather than being collapsed to `$base_key`. The "null base plus empty translation value" case is also treated as equal, preventing unnecessary overrides for optional properties.

The existing functional test `testContentTemplateConfigTranslationUi()` in `TranslationTest` was extracted into `ConfigWithComponentTreeConfigTranslationUiTest` and a new parallel class `ConfigWithComponentTreeTmgmtUiTest`, both sharing the fixture setup in `ConfigWithComponentTreeTranslationTestBase`. Coverage now spans plain prose, rich prose, URI-esque inputs, multi-cardinality sequences, optional unpopulated props, and non-static prop sources, for both the `config_translation` UI path and the TMGMT workflow.
