## Summary

The payload delivered to JS code components for `content-entity-reference` props is restructured. Instead of a flat dictionary keyed by field names joined with a `$` separator (e.g., `uid$name`), each resolved entity now arrives as a nested object that mirrors the reference structure, with a `__type` key set to the entity's bundle. Reference expressions produce nested sub-objects rather than compound flat keys. Code components that destructure the old flat payload will need to update to the new nested shape. Two new internal HTTP API endpoints also land to power the upcoming entity reference selection UI.

## Impact

Affects module maintainers.

- **API change:** `JsComponent::generateKeyForExpression()` (public static) and `JsComponent::CONTENT_ENTITY_REFERENCE_KEY_SEPARATOR` are removed; the runtime payload for `content-entity-reference` props changes from a flat `$`-keyed dict to a nested object with a `__type` bundle key.

## Technical details

The public static method `JsComponent::generateKeyForExpression()` and the constant `JsComponent::CONTENT_ENTITY_REFERENCE_KEY_SEPARATOR` (`'$'`) are removed. Their replacement is the private `JsComponent::buildReferencePayload()`, which takes the resolved entity wrapped in an `EvaluationResult` and the list of `EntityFieldBasedPropExpressionInterface` instances declared for the prop. It groups reference expressions by their referencer field, evaluates each group by recursing into the referenced entity, and assembles a nested object. Every entity object in the result carries `__type` set to `$entity->bundle()`. Scalar and object-prop leaves are hoisted directly onto the entity object using `getDeveloperFacingKey()` rather than the old `$`-joined compound keys. The cacheability of every traversed entity is composed into the returned `EvaluationResult`.

`JavaScriptComponent` gains `expandEntityFields()` and `coalesceEntityFields()` (both `private static`), implementing the expand/coalesce contract from ADR [#0005](https://www.drupal.org/node/0005). Before the `dataDependencies` object is sent to the client, `expandEntityFields()` unpacks stored `FieldObjectPropsExpression` entries into per-property atomic leaf strings via `Coalescer::expand()`. On save, `coalesceEntityFields()` re-groups the leaf strings the client returns into `FieldObjectPropsExpression` form via `Coalescer::coalesce()`, keeping the client from needing to understand expression merging.

Five new config-schema validation constraints enforce the invariants the selection UI requires: `EntityFieldExpressionsMustBeIdempotent` (a stored expression must reproduce exactly after an expand/coalesce round trip via `Coalescer`), `EntityFieldExpressionsSameFieldMustBeCoalesced` (multiple expressions on the same host entity field must be stored as a single `FieldObjectPropsExpression`), `EntityFieldExpressionMustNotTargetInternalProperty` (blocks non-computed internal fields and field properties, matching the same rule applied by `ApiUiContentEntityReferenceControllers::buildFieldEntry()`), `MultiValuedFieldNotSupported` (temporary, rejects fields with cardinality other than 1), and `MultiTargetBundleReferenceNotSupported` (temporary, rejects reference fields that target more than one bundle). Both temporary constraints carry a `@todo` referencing their respective follow-up issues.

The two new HTTP endpoints live in `ApiUiContentEntityReferenceControllers`, which is marked `@internal`. `listContentEntityTypes()` iterates all content entity type definitions, creates an unsaved stub per bundle, runs `view` access against the stub, and returns only bundles the current user can view. `listFields()` returns per-field descriptors: each entry includes the field name, label, pickable field properties with pre-built expression strings, and for reference fields a `targetBundles` map and a `hasChildren` flag. An optional `?parent=[expression]` query parameter chains the response onto an existing reference path so the UI can lazily expand reference fields without constructing expression strings. Stubs are normalized before the access check (users activated, files given a `public://` URI, other content entities published) to avoid false denials from handlers that require a valid entity state.
