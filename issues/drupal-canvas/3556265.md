## Summary

Canvas pages can now be unpublished after being published, allowing editors to temporarily hide pages from visitors without deleting them. The unpublish action is staged through auto-save and takes effect when publishing changes, matching Canvas's existing workflow. Draft pages (never published) automatically publish when changes are published, while previously published pages can be unpublished or re-published. The homepage cannot be unpublished. This feature works alongside the Trash module for complementary workflows and is designed to be compatible with future Workspaces integration.

## Impact

No upgrade or configuration changes required.

## Technical details

Canvas pages implement `EntityPublishedInterface` with a boolean `status` field, but need to distinguish three states: draft (never published), published, and unpublished (previously published). Rather than adding a stored field, the implementation uses `AutoSaveManager::entityIsConsideredNew()` to determine if a page is a draft based on whether it still has the default "Untitled page" title. This works because Canvas creates all pages through the UI and controls the initial title.

A new PATCH endpoint (`canvas.api.content.patch`) allows updating a page's published status through auto-save. When a user clicks "Unpublish page," the auto-save entry is updated with `status = 0`. When "Review changes" is published, the unpublish action takes effect on the live entity. The same workflow works in reverse for publishing an unpublished page.

The operation links (unpublish/publish) use core's `disable`/`enable` link relation types, which are semantically appropriate for entities implementing `EntityPublishedInterface`. Homepage unpublishing is prevented both for the current homepage and any staged homepage configuration.

Published pages with auto-saved unpublish operations show a "🕦Unpublish" indicator with clock icon. Unpublished pages (non-drafts) show an "Unpublished" badge and display a "Publish page" action. Draft pages show neither action since they use the main Publish button. This approach avoids Views plugins and complex migrations, keeping the implementation compatible with future Workspaces integration where draft pages would exist as published entities within workspace.

## Lessons learned

- When a boolean field must distinguish between three states (e.g., draft/published/archived), check if two states share the same boolean value and can be differentiated by metadata like revision history or default field values. This avoids adding storage fields that complicate future migrations to workflow systems like Workspaces or Content Moderation. *(via tedbow, lauriii)*
- When building custom workflow features that will eventually be replaced by established systems (like Workspaces or Content Moderation), minimize schema changes and stored state. Compute state from existing data when possible, so the migration path involves removing code rather than migrating data and deprecating fields. *(via tedbow, catch, lauriii)*
- When implementing stop-gap solutions before a planned architectural change, prioritize the simplicity of the eventual migration over feature completeness. A smaller, easily-removable implementation is preferable to a comprehensive solution that creates technical debt and complex deprecation paths. *(via catch, lauriii, tedbow)*
