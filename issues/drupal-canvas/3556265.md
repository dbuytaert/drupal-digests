## Summary

Canvas Pages can now be unpublished after being published. Previously, there was no way to take a published page offline while keeping it editable. This feature lets content editors hide pages from visitors by marking them as unpublished through the Canvas editor. The unpublish action is staged through the same auto-save and "Review changes" workflow that Canvas uses for other edits. Draft pages that have never been published work differently from unpublished pages, and the interface shows the correct status for each. The homepage cannot be unpublished to prevent breaking the site. This feature is built to avoid conflicts with future Workspaces integration planned for later in 2026.

## Impact

No upgrade or configuration changes required.

## Technical details

Canvas Pages now support an "Unpublish" action that allows published pages to be taken offline. This is implemented through a new PATCH endpoint (`canvas.api.content.patch`) that updates the auto-save entry with the new published status. When a user clicks "Unpublish page" in the editor, it stages the status change in auto-save rather than immediately updating the live entity. The change takes effect when the user publishes through the "Review changes" workflow, consistent with how other Canvas edits work.

The implementation distinguishes between three states without adding a new field: published pages (status = 1), unpublished pages (status = 0 and previously published), and draft pages (status = 0 and never published). Draft detection relies on the existing `AutoSaveManager::entityIsConsideredNew()` logic, which checks if the page title is still the default "Untitled page." This avoids adding database fields that would complicate the planned migration to Workspaces.

The PATCH endpoint validates that only the `status` field is being updated and prevents unpublishing the homepage by checking both current and staged homepage configuration. Auto-save cache tags ensure the UI updates immediately when status changes are staged. The navigation UI shows badges for unpublished pages and pages pending unpublish, with different indicators for saved versus unsaved status changes.

The implementation avoids Views plugins initially proposed in earlier merge requests, keeping the solution simpler and reducing technical debt for the Workspaces migration.

## Lessons learned

- When caching happens at multiple layers, invalidating an inner component doesn't help if an outer cache still serves stale data. Avoid adding fields that represent derived state when that state can be computed from existing data, especially if the derived field will become obsolete when migrating to a different subsystem. Computing state dynamically (like distinguishing draft from archived by checking revision history) avoids data migration complexity and conflicting systems during transitions. *(via tedbow, lauriii)*
- When building a temporary solution before a planned migration to a different subsystem, minimize changes that will require deprecation and removal. Evaluate whether existing contributed modules solve the use case adequately, even if imperfectly, rather than building custom implementations that create technical debt during the migration period. *(via catch, amateescu, tedbow)*
