## Summary

The Canvas UI boot response was silently discarding cache metadata from entity-create access checks. On sites where entity create access is modified by custom modules (via access hooks or custom access control handlers), the Canvas UI could serve stale access information because the `HtmlResponse` did not vary by those access results. After this fix, the response correctly propagates all cache tags and contexts from every entity-type/bundle `createAccess()` check, so the UI is properly invalidated when access changes.

## Impact

No upgrade or configuration changes required.

## Technical details

`CanvasController::getContentEntityCreateOperations()` called `createAccess($bundle)` with its boolean return, discarding the full `AccessResult` object and any cache tags or contexts it carried. The private method is renamed `getAllContentEntityCreateLinks()` and now calls `createAccess($bundle, return_as_object: TRUE)`. Allowed results are wrapped in `CanvasResourceLink` objects and collected into a `CanvasResourceLinkCollection`; denied and neutral results still have `addCacheableDependency($access)` called on the collection so their metadata is not lost. The collection itself is then passed to `HtmlResponse::addCacheableDependency()` in `__invoke`, ensuring the full access cacheability bubbles through. The collection also carries explicit cache tags `entity_field_info`, `entity_bundles`, and `entity_types` to invalidate on structural definition changes. A new `CanvasUriDefinitions::LINK_REL_CREATE` constant is added. The `$content_entity_create_operations` array fed to the client's `contentEntityCreateOperations` key is reconstructed from the link collection's `label` attributes, keeping the client data contract unchanged.

### Collaboration

wim leers objected to the initial MR approach of passing cacheability out-of-band, calling it "a code smell, a risk and an anti-pattern," and particularly flagged the risk given the security implications of access checks in the controller that boots the Canvas UI. He pushed for cacheability to be associated directly with the data it describes via `CanvasResourceLinkCollection`. penyaskito raised three concrete concerns about using the link collection directly: it would change the client data contract, require bundle URL string handling on the client, and the create URL was not actually needed. wim leers reworked his proposed approach to avoid any client-side changes, reconstructing the existing array format from the link collection server-side, which resolved all three constraints.
