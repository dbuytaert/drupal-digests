# #3591782: Content template editor preview shows a stale snapshot instead of live edits

- **Project:** Drupal Canvas
- **Type:** Major bug report
- **Version:** 1.x-dev
- **Status:** Fixed
- **Discussion:** [#3591782](https://git.drupalcode.org/project/canvas/-/work_items/3591782) · 1 contributor · 3 comments

## Summary

In the Canvas module's content template editor, triggering a full-page preview and then returning to the editor left the editor preview frame frozen on the snapshot captured during that preview. Any subsequent edits (such as linking a component prop to an entity field or typing in a text field) were applied to the model but were not visible in the preview until an unrelated action forced a full re-render. This made it appear as though edits had no effect. The fix ensures the stale snapshot is cleared as soon as the content template editor establishes its own live content, so the preview frame reflects current edits immediately.

## Impact

No upgrade or configuration changes required.

## Technical details

The preview HTML selector `selectPreviewHtml` evaluates `snapshotHTML || html`: a non-empty `snapshotHTML` takes precedence over the live editor `html`. The snapshot is populated when the user opens the full-page preview and was only cleared on the `canvas_page` layout path (fixed earlier in [#3591684](https://www.drupal.org/node/3591684)) and for language previews on unmount. The content template editor's own two data-loading paths, `getContentTemplateLayout` in `componentAndLayoutApi` and the component-update handler in `previewApi`, never dispatched `setSnapshotHTML('')`, so a leftover snapshot permanently masked all subsequent live updates for a plain content template preview, and a late-arriving preview response could re-populate the snapshot even after an unmount cleanup, causing intermittent failures in the `Templates - General › Add templates to page` Playwright test.

The fix adds `dispatch(setSnapshotHTML(''))` immediately before `dispatch(setHtml(html))` in both affected RTK Query handlers: in the `onQueryStarted` success path of the content template layout endpoint in `componentAndLayoutApi`, and in the equivalent component-update path in `previewApi`. A new Playwright test reproduces the exact failure sequence (open template, preview, exit preview, link a prop to the entity Title field, assert the preview `<h1>` shows the live resolved value) to prevent regression.
