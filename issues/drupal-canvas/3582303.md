## Summary

The Canvas CLI now supports using a pre-issued access token via the `CANVAS_ACCESS_TOKEN` environment variable, in addition to the existing OAuth client credentials flow. When this variable is set, the CLI uses the token directly as a Bearer token and skips the `client_credentials` flow entirely. This allows the CLI to work with authorization systems beyond Simple OAuth. When the variable is not set, the CLI continues to use the existing client credentials flow with `CANVAS_CLIENT_ID`, `CANVAS_CLIENT_SECRET`, and `CANVAS_SCOPE`.

## Impact

Affects module maintainers and site owners.

- **Config change:** New `CANVAS_ACCESS_TOKEN` environment variable to provide pre-issued Bearer token
- **New API:** `ApiOptions` interface has new optional `accessToken` parameter

## Technical details

The Canvas CLI previously required OAuth `client_credentials` flow to obtain an access token. Some sites use authorization systems beyond Simple OAuth and need to provide a specific bearer token directly.

A new optional `accessToken` parameter was added to the `ApiOptions` interface. When provided, the `ApiService` constructor sets it directly as `this.accessToken` and adds it to the client's default Authorization header as `Bearer ${accessToken}`, skipping the OAuth flow.

The `createApiService()` helper function checks for `process.env.CANVAS_ACCESS_TOKEN`. When present and truthy, it creates an `ApiService` with the token and passes empty strings for `clientId`, `clientSecret`, and `scope`.

The `refreshAccessToken()` method now throws an error if `!this.clientId || !this.clientSecret`, preventing refresh attempts when using a static token. The 401 error handler checks `!this.clientId` to determine whether to show an access token error message or a client credentials error message.

Tests verify that the static token is used directly without hitting the OAuth endpoint, that it's sent as a Bearer token on all requests, and that 401 errors produce the appropriate error message.
