## Summary

The Canvas module now exposes a new HTTP API endpoint, `GET /canvas/api/v0/site-data`, that returns the site-level `canvasData.v0` payload (base URL, branding, theme assets, and JSON:API settings). Previously, this data was only available when Drupal rendered a full page with the relevant asset libraries attached, which meant Canvas Workbench running as a local Vite dev server had no way to call `getSiteData()` with live site data. With this change, the `@drupal-canvas/vite-plugin` fetches the endpoint at build start using an OAuth Bearer token and injects the response into `window.drupalSettings.canvasData.v0` before any component code runs, so the existing `getSiteData()` function works in Workbench without modification. The scope is intentionally limited to site-level data; page-level data (breadcrumbs, page title, main entity) is deferred to a follow-up issue.

## Technical details

The new route `canvas.api.site_data` is registered at `GET /canvas/api/v0/site-data` in `canvas.routing.yml` with the same access requirements as other read-only Canvas API routes: `_canvas_authentication_required: TRUE`, `_canvas_ui_access: 'TRUE'`, and `canvas_external_api: true`. The controller `Drupal\canvas\Controller\SiteDataController` (marked `@internal`) calls the four `CodeComponentDataProvider` methods already used by `ComponentSourceHooks::jsSettingsAlter()` (`getCanvasDataThemeAssetsV0`, `getCanvasDataJsonApiSettingsV0`, `getCanvasDataBrandingV0`, `getCanvasDataBaseUrlV0`) and merges them via `NestedArray::mergeDeep`. It returns a `CacheableJsonResponse` with cache contexts for `url.site` and dependencies on `system.site`, `system.theme`, `system.theme.global`, and the active theme's settings config.

OAuth coverage is added by appending `canvas.api.site_data` to the `$named_routes` array in `CanvasOauthAuthenticationProvider::applies()`, which explicitly enumerates routes; without this the OAuth provider would not authenticate requests to the new route.

On the TypeScript side, `@drupal-canvas/vite-plugin` gains a `buildStart()` hook that reads an access token (first from the `CANVAS_ACCESS_TOKEN` environment variable, then from the token store via `getTokenEntry()` from `@drupal-canvas/auth`). If a valid token is found, it fetches the endpoint and caches the response in `canvasApiData`. The `transformIndexHtml` hook then merges `canvasApiData` into the `canvasData.v0` object, resolving relative theme asset paths (logo and favicon URLs starting with `/`) to absolute URLs by prepending the site base URL. When `canvasApiData` is present, the static `jsonapiPrefix` option and `CANVAS_JSONAPI_PREFIX` env var are no longer used, since `jsonapiSettings` comes from the API response. The endpoint shape is also added to the module's OpenAPI spec.
