## Summary

Canvas now has a server-to-client notification system that lets Canvas extensions surface messages to authenticated users without a page reload. Notifications appear in two places: an Activity Center dropdown (bell icon in the top bar) and as auto-dismissing toast messages in the top-right corner. Five types are supported: processing, success, info, warning, and error. The entire feature is currently gated behind `drupalSettings.canvas.devMode`.

## Impact

Affects module maintainers and site owners.

- **Database update:** Two new tables added: `canvas_notifications` and `canvas_notification_reads`.
- **New API:** New `NotificationService` PHP class and `GET /canvas/api/v0/notifications` / `POST /canvas/api/v0/notifications/read` REST endpoints for creating and polling notifications from Canvas extensions.

## Technical details

The backend stores notifications in two new database tables: `canvas_notifications` (id, type, key, title, message, timestamp, actions) and `canvas_notification_reads` (uid, notification_id, timestamp). A `NotificationService` PHP class handles creation and retrieval. Each notification gets a UUID on creation; an optional `key` field groups related notifications so that creating a new notification with a given key automatically deletes all prior error, warning, and processing notifications for that key, enabling clean state transitions such as `processing -> success` or `error -> processing` on retry.

Two REST endpoints are exposed: `GET /canvas/api/v0/notifications` returns up to 25 recent non-processing notifications plus all active processing notifications with per-user `hasRead` state; `POST /canvas/api/v0/notifications/read` marks a list of notification IDs as read for the current user. A cron hook replaces stale processing notifications (older than 30 minutes) with a timeout error and purges all notifications and read entries older than 30 days.

The frontend uses RTK Query for polling with adaptive frequency: 2 seconds when processing notifications are present, 10 seconds normally, and 5 minutes when the browser tab is not focused. A Redux slice manages UI state. Marking notifications as read uses an optimistic update. The Activity Center is a Radix Popover sorted as: processing at top, then unread errors and warnings (newest first), then everything else chronologically. Toast notifications appear for any notification with a timestamp after page load, auto-dismiss after 15 seconds, and can be dismissed early (which also marks the notification as read).

The design was shaped substantially in the comment thread: justafish proposed the data structure and a full OpenAPI schema including a `key`-based dismissal mechanism and a separate read endpoint. wim leers pushed back on a dedicated polling endpoint (preferring consolidation with the existing autosave poll) and raised concerns about information disclosure for globally scoped notifications; the final Phase 1 design accepted a dedicated polling endpoint and global-only notifications as acknowledged trade-offs. Notifications in Phase 1 are broadcast to all authenticated users; per-user or per-role scoping is noted as a future enhancement. The implementation was delivered through child issues [#3514033](https://www.drupal.org/node/3514033), [#3550161](https://www.drupal.org/node/3550161), and [#3574257](https://www.drupal.org/node/3574257).

### Collaboration

justafish drove the API design by posting a detailed client-side data structure proposal and a complete OpenAPI schema for both the GET and POST endpoints, which became the basis for the final implementation. wim leers raised substantive concerns: whether polling should be consolidated with the existing autosave endpoint, how cross-session read state should be tracked (UUID vs timestamp), and whether global notifications created an information disclosure risk.
