## Summary

Canvas was unconditionally saving `Component` config entities every time plugin definitions were re-cached, even when nothing about the component had actually changed on disk. This caused two categories of bugs: unnecessary database writes on every cache clear, and dangerous entity CRUD operations inside the `UpdateKernel` (which Drupal's own update rules explicitly forbid). This fix centralises all `Component` config entity creation and saving into a single `ComponentSourceManager::generateComponents()` method that skips entirely when running in an update kernel or during config sync, and only calls `Component::save()` when a computed version hash or unversioned metadata has actually changed.

## Impact

Affects module maintainers.

- **API change:** The `ComponentSource` plugin attribute now requires a `discovery` field: either a `ComponentCandidatesDiscoveryInterface` FQCN or `FALSE`; `VersionedConfigEntityInterface::isVersionedProperty()` added.
- **Deprecation:** `ComponentSourceInterface::checkRequirements()` marked for removal in #3561265; migrate logic to a `ComponentCandidatesDiscoveryInterface` implementation.

## Technical details

Before this change, both a `plugin.manager.block` decorator (`BlockManager`) and a `plugin.manager.sdc` decorator (`ComponentPluginManager`) overrode `setCachedDefinitions()` to blindly call `Component::save()` whenever plugin definitions were re-cached. `JavascriptComponentStorage::doPostSave()` maintained its own parallel create/update logic. During `UpdateKernel`, the NullBackend forces re-discovery on every access, so these save calls fired unconditionally — violating Drupal's rule that entity CRUD is never safe inside an update hook.

The fix introduces `ComponentCandidatesDiscoveryInterface`, a new `@internal` interface that separates "find candidates and check requirements" from plugin instance concerns. Each component source type that auto-discovers gets a dedicated discovery class: `BlockComponentDiscovery` for Drupal block plugins, `JsComponentDiscovery` for JS components; sources with no auto-discovery (such as `Fallback` and `Personalization`) set `discovery: FALSE`. The `ComponentSource` plugin attribute gains a `discovery` field to carry this value. The deleted `BlockManager` decorator (191 lines) and the scattered save logic in `JavascriptComponentStorage` and `SingleDirectoryComponent` are replaced entirely by `ComponentSourceManager::generateComponents()`.

`generateComponents()` guards on an `$isUpdateKernel` flag (set in the constructor by checking `$kernel instanceof UpdateKernel`) and on `ConfigInstallerInterface::isSyncing()`, returning early in both cases. For existing components it computes a `$version` hash and compares it against `getActiveVersion()`; it also checks each item in `computeCurrentComponentMetadata()` against the stored value. `Component::save()` is called only when `$needs_update` is `TRUE`. The `VersionedConfigEntityInterface` gains `isVersionedProperty()` to support this per-property comparison. `ComponentSourceInterface::checkRequirements()` is marked `@todo 🚨 Remove` and is tracked in [#3561265](https://www.drupal.org/node/3561265). `ComponentStatusController` no longer forces plugin rediscovery on page load; `RecipeSubscriber` and `ComponentSourceHooks::rebuild()` both delegate directly to `ComponentSourceManager::generateComponents()` rather than calling `getDefinitions()` on individual plugin managers. `ComponentPluginManager` is now wired as a class override via `CanvasServiceProvider::alter()` instead of as a named service decorator.

## Upgrade

Any module that declares a custom `ComponentSource` plugin must add the new `discovery` attribute parameter. Use `FALSE` for sources that ship their components as config in `config/install`, or provide a FQCN implementing `ComponentCandidatesDiscoveryInterface` for sources that auto-discover:

```php
// Source with no auto-discovery (ships Component config entities in config/install):
#[ComponentSource(
  id: 'my_source',
  label: new TranslatableMarkup('My Source'),
  supportsImplicitInputs: FALSE,
  discovery: FALSE,    // ← add this
)]

// Source with auto-discovery:
#[ComponentSource(
  id: 'my_source',
  label: new TranslatableMarkup('My Source'),
  supportsImplicitInputs: TRUE,
  discovery: MySourceDiscovery::class,    // ← add this
)]
```

`ComponentSourceInterface::checkRequirements()` is slated for removal in [#3561265](https://www.drupal.org/node/3561265); if your plugin overrides it, plan to migrate that logic to a `ComponentCandidatesDiscoveryInterface` implementation.
