## Summary

Canvas content authors can now map "List (string)" fields to `type: string` component props in content templates. Previously this was blocked because `list_string` values are machine names (e.g. `"be_nl"`, `"light"`, `"red"`) classified as structured strings, incompatible with prose-typed component props. The fix takes a different approach: instead of passing the stored machine-name value, it passes the human-readable label configured in the field's allowed values. For example, a field storing `"dark"` would supply `"Dark mode"` to a component. The label is translated to the current content language.

## Impact

Affects module maintainers.

- **New API:** New `CoreFeatureListStringItemAddPropertiesTrait` for contributed `list_string` subclasses to expose a computed `label` property compatible with Canvas shape matching

## Technical details

The implementation adds `ListStringItemOverride` (extending core's `ListStringItem`) and registers it in `ShapeMatchingHooks` for the `list_string` field type. A new `CoreFeatureListStringItemAddPropertiesTrait` attaches a computed read-only `label` property to `list_string` field items. The computed property is backed by the `ListStringItemLabel` data type plugin (`computed_list_string_label`), which extends `StringData`, calls `$list_item->getPossibleOptions()` to resolve the label for the stored key, and adds `languages:language_content` cache context plus the field storage definition as a cacheable dependency. The `label` property carries `StringSemanticsConstraint::PROSE` (making it eligible for `type: string` matching), while the `value` property receives a `'is source for' => 'label'` setting so that `Labeler` can recognize the two as linked when evaluating prop expressions.

`StaticPropSource` was updated to count only non-computed properties when deciding whether a field item's value should be serialized as a flat scalar or as a keyed array. Before this change, the count included computed properties, which would have broken `list_string` (and also changed behaviour for `datetime`) by treating a single-stored-property field as multi-property. The serialization format for such fields consequently changed: `datetime` values previously stored as `{"value":"2020-04-16"}` are now stored as the bare scalar `"2020-04-16"`. A backward-compatibility layer in `StaticPropSource::denormalizeValue` accepts both formats on read; re-saving a static prop source migrates it to the new format automatically.

`PropSourceSuggester` also drops the `array_multisort` depth-based reordering of suggestions, relying on form display weight order instead, which corrected a misalignment in test expectations for image and user-reference fields.

## Upgrade

If you have saved `StaticPropSource` values for `datetime` or any other field type that has exactly one stored property alongside computed properties, those values were previously serialized as a one-key array and are now stored as a plain scalar. Existing serialized values are read correctly by the BC layer, but will be written back in the new format on next save. No manual migration is needed. If your code extends `StaticPropSource` or asserts on its serialized JSON, update the assertion:

```php
// Before (format prior to #3548749):
// 'value' => ['value' => '2020-04-16']

// After (new format):
// 'value' => '2020-04-16'
```

If you have a contributed `list_string` subclass and want it to expose the `label` computed property for Canvas mapping, apply the trait:

```php
use Drupal\canvas\Plugin\Field\FieldTypeOverride\CoreFeatureListStringItemAddPropertiesTrait;

class MyCustomListStringItem extends ListStringItem {
  use CoreFeatureListStringItemAddPropertiesTrait;
}
```
