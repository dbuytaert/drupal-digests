## Summary

The Manage Display admin page for node content types now shows a "New: Design with Drupal Canvas" call-to-action banner when no Canvas content template exists yet for the given bundle. The banner links directly into the Canvas editor to start building a template. It is shown only to users who have permission to create Canvas content templates. A follow-up bug fix corrected an earlier access control mistake that caused the banner to appear for users with general Canvas UI access even when they lacked permission to create content templates.

## Impact

No upgrade or configuration changes required.

## Technical details

`FieldUiHooks::formEntityViewDisplayEditFormAlter()` in `src/Hook/FieldUiHooks.php` implements `hook_form_entity_view_display_edit_form_alter` and injects the CTA via the Messenger service using the custom message type `canvas_cta`, rendered through the `canvas_cta` theme template defined in `ModuleHooks`. The CTA fires only when three conditions are all met: the current route is `entity.entity_view_display.node.default`, no `ContentTemplate` entities exist for the current bundle (queried without access checking), and the current user passes `createAccess()` on the `ContentTemplate` access control handler. A duplicate-message guard checks `$this->messenger->messagesByType('canvas_cta')` before proceeding to prevent the banner from appearing more than once.

The initial implementation checked access via `CanvasUiAccessCheck->access($currentUser)`, which tests general Canvas UI access rather than the specific right to create content templates. This caused the banner to appear for users who could open Canvas but could not create content templates. The fix removes `CanvasUiAccessCheck` and `AccountInterface` from the `FieldUiHooks` constructor entirely, replacing the guard with `$this->entityTypeManager->getAccessControlHandler(ContentTemplate::ENTITY_TYPE_ID)->createAccess()`. wim leers noted the initial MR !497 attempt also used `update` access rather than `create` access, which was corrected by vishalkhode.

A new kernel test class `FieldUiHooksTest` in `tests/src/Kernel/FieldUiHooksTest.php` uses a `DataProvider` to cover: user with permission and no existing template (CTA shown), user with permission and an existing template (CTA suppressed), wrong route (CTA suppressed), and user without permission (CTA suppressed). The test constructs `FieldUiHooks` directly with mocked `RouteMatchInterface` and `MessengerInterface` collaborators and a real `EntityTypeManagerInterface` from the kernel container.

This issue is a follow-up to [#3551709](https://www.drupal.org/node/3551709), which added a link from Manage Display back into Canvas when a template is already active for a view mode.
