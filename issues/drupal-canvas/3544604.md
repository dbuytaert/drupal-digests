## Summary

A bug in the Canvas module caused component dependency calculation to skip intermediate entities in reference chains. For example, a page component referencing a Media item that itself references a File would record a dependency on the File but omit the Media entity entirely. This left those entities vulnerable to accidental deletion without the expected constraint warnings, breaking Canvas's "Audit" functionality and data integrity guarantees. The bug was fixed in the dependency logic, and four database update hooks correct the incomplete dependency records stored in existing sites.

## Impact

Affects site owners.

- **Database update:** Four new `canvas_post_update_0002_*` hooks recalculate and persist corrected component dependencies on Pattern, PageRegion, ContentTemplate, and FieldConfig entities that use reference field prop expressions.

## Technical details

The bug lived in `calculateDependencies()` on both `ReferenceFieldPropExpression` and `ReferenceFieldTypePropExpression`. When a host entity was provided, both classes evaluated the `referencer` expression to obtain referenced entities, then called `$this->referenced->calculateDependencies()` without passing those referenced entities as the new host context. This meant only the leaf-level dependencies (from the final `referenced` expression) and the top-level referencer's static dependencies were collected; the intermediate entity's own dependency chain was dropped. Additionally, `ReferenceFieldTypePropExpression` had a stray unconditional `NestedArray::mergeDeep($dependencies, $this->referenced->calculateDependencies())` at the return point that further confused the content-aware vs. content-unaware paths.

The fix corrects both classes symmetrically: `$this->referencer->calculateDependencies($host_entity)` now propagates the host context into the referencer; and when evaluated referenced entities are available, the code iterates over each `$referenced_content_entity` and calls `$this->referenced->calculateDependencies($referenced_content_entity)`, capturing the full intermediate dependency chain. A fallback path calls `$this->referenced->calculateDependencies()` (content-unaware) when `$host_entity` is `NULL` or when evaluation yields no referenced entities. The stray unconditional call in `ReferenceFieldTypePropExpression` is removed.

The update path adds `CanvasConfigUpdater::needsIntermediateDependenciesComponentUpdate()`, which inspects a component tree entity's inputs for any `ReferenceFieldPropExpression` or `ReferenceFieldTypePropExpression` via the new `ComponentInputs::getPropSourcesUsingExpressionClass()` helper. Four `canvas_post_update_0002_*` hooks use `ConfigEntityUpdater` to batch-process `Pattern`, `PageRegion`, `ContentTemplate`, and `FieldConfig` entities, recalculating and persisting their corrected dependencies. `ContentTemplate::preSave()` and `PageRegion::preSave()` also gained guards to recalculate dependencies when `isSyncing()` is true, ensuring config import does not re-introduce stale dependency data. The bug was originally introduced in [#3457504](https://www.drupal.org/node/3457504).

## Upgrade

Run database updates after upgrading Canvas:

```shell
drush updb
```

The four hooks `canvas_post_update_0002_intermediate_component_dependencies_in_patterns`, `..._in_page_regions`, `..._in_content_templates`, and `..._in_field_config_component_trees` will batch-update all affected config entities to record the previously missing intermediate entity dependencies.
