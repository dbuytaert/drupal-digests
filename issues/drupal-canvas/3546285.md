## Summary

When using the Canvas editor's undo and redo controls (or keyboard shortcuts Ctrl+Z / Ctrl+Y / Cmd+Shift+Z), the editor now automatically navigates back to the URL where the original action occurred. Before this change, undo and redo reverted state but left the user on whatever URL they were currently viewing, which could be confusing when edits had been made across different components or editor panels. Undo and redo keyboard shortcuts also now work correctly when focus is inside a form field, and they suppress the browser's native undo behavior in favor of Canvas's own Redux-managed history.

## Impact

Affects module maintainers.

- **API change:** `selectUndoType` and `selectRedoType` selectors renamed to `selectUndoItem` and `selectRedoItem`; `pushUndo` payload changed from a `UndoRedoType` string to a `UndoRedoStackItem` object containing `targetSlice` and `routeSnapshot`.
- **New API:** New `useRouteSync` hook, `setCurrentRoute` action, `selectCurrentRoute` selector, and `RouteSnapshot` / `UndoRedoStackItem` TypeScript interfaces exported from `uiSlice`.

## Technical details

Each entry in the `undoStack` and `redoStack` (managed by `uiSlice` in the Redux store) previously stored only a `UndoRedoType` string (`'layoutModel'` or `'pageData'`). It now stores a `UndoRedoStackItem` object that adds a `routeSnapshot: RouteSnapshot` field (containing `pathname`, `search`, and `hash`). A new `currentRoute: RouteSnapshot` field is also added to `uiSliceState` and kept up to date via the new `setCurrentRoute` action and `selectCurrentRoute` selector.

The `undoRedoActionIdMiddleware` was updated to read `state.ui.currentRoute` at the moment an undoable action fires and attach it to the `pushUndo` payload. The `performUndoOrRedo` reducer now sets `state.currentRoute` to the route stored in the popped stack item, which is what triggers navigation.

The new `useRouteSync` hook provides bidirectional synchronization between React Router and the Redux store. In the React Router → Redux direction it dispatches `setCurrentRoute` whenever `useLocation` reports a new pathname. In the Redux → React Router direction it calls `useNavigate` whenever `selectCurrentRoute` changes to a value that differs from the current browser location. A `useRef` flag (`isNavigatingProgrammatically`) prevents the two effects from feeding back into each other.

The `historyEraser` reducer enhancer was updated to check for `action.payload.targetSlice` instead of comparing the payload string directly, since `pushUndo` now receives an object. The two public selectors `selectUndoType` and `selectRedoType` were renamed to `selectUndoItem` and `selectRedoItem` respectively, and their return types changed from `UndoRedoType | undefined` to `UndoRedoStackItem | undefined`. Callers in `useUndoRedo` now read `.targetSlice` from the returned item. The `useHotkeys` calls for undo/redo were updated with `enableOnFormTags: true` and `preventDefault: true` so that keyboard shortcuts intercept the browser's native undo inside text inputs.

## Upgrade

If your code imports `selectUndoType` or `selectRedoType` from `@/features/ui/uiSlice`, rename the imports and update property access:

```ts
// Before
import { selectUndoType, selectRedoType } from '@/features/ui/uiSlice';
const undoType = selectUndoType(state); // returns UndoRedoType | undefined
const redoType = selectRedoType(state);

// After
import { selectUndoItem, selectRedoItem } from '@/features/ui/uiSlice';
const undoItem = selectUndoItem(state); // returns UndoRedoStackItem | undefined
const redoItem = selectRedoItem(state);
// Access the slice name via .targetSlice:
if (undoItem) dispatch(UndoRedoActionCreators.undo(undoItem.targetSlice));
```

If your code dispatches `pushUndo` directly, update the payload from a string to a `UndoRedoStackItem` object:

```ts
// Before
store.dispatch(pushUndo('layoutModel'));

// After
import type { UndoRedoStackItem } from '@/features/ui/uiSlice';
store.dispatch(pushUndo({
  targetSlice: 'layoutModel',
  routeSnapshot: { pathname: '/your/route', search: '', hash: '' },
}));
```

### Collaboration

jessebaker identified the critical flaw in the first attempted approach (MR !65), where treating navigation itself as an undoable action meant that a user could accidentally wipe the redo stack simply by clicking on a different component in the editor frame. jessebaker then suggested the alternative of saving a route snapshot alongside each stack entry, which became the approach that shipped in MR !141.
