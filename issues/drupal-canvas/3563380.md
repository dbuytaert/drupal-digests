## Summary

When building Canvas content templates for nodes, site builders can now link a component prop declared as `type: string, format: date` to the node's "Authored on" (`created`) or "Changed" (`changed`) timestamp fields. These fields store UNIX timestamps (integers) and previously could not be offered as candidates for date-shaped props. The conversion from integer timestamp to ISO date string is now handled automatically by the existing `unix_to_date` adapter plugin, with no changes required to the UI or to how content templates are stored beyond the addition of an optional `adapter` key in the prop source config.

## Impact

Affects site owners.

- **Config change:** Stored `DynamicPropSource` entries in `ContentTemplate` config entities may now include an optional `adapter` string key naming an adapter plugin ID such as `unix_to_date`.

## Technical details

The core change is a new optional adapter capability on `DynamicPropSource`. The constructor gains a nullable `$adapter` parameter of type `AdapterInterface`; a `\LogicException` is thrown at construction time if the provided adapter has more than one input, restricting this path to simple single-input transforms. A `withAdapter(string $adapter_plugin_id)` factory method creates a new `DynamicPropSource` instance with the adapter instantiated via `AdapterManager`. During `evaluate()`, if an adapter is present and the raw `EvaluationResult` is non-null, the sole input is set and `adapt()` is called, wrapping the result in a new `EvaluationResult`. The `toArray()` / `fromArray()` round-trip gains an optional `adapter` string key holding the plugin ID, and `calculateDependencies()` now declares the module providing the adapter plugin, derived from its plugin definition.

In `PropSourceSuggester::getSuggestionsForNode()`, hard-coded logic (annotated `@todo` pointing to [#3563960](https://www.drupal.org/node/3563960) for future generalization) pushes `created` and `changed` — each wrapped in a `DynamicPropSource` carrying the `unix_to_date` adapter — into the instance candidates list whenever the target prop schema is exactly `['type' => 'string', 'format' => 'date']` and the host entity type is `node`.

The existing `@internal` `AdaptedPropSource` class, which supports multiple inputs and adapter chaining, was deliberately kept separate. The new feature intentionally avoids that class to contain the scope: `DynamicPropSource` now handles the narrow case of a single adapter with a single input, while `AdaptedPropSource` retains the broader (and riskier) multi-input, chainable form.

A new test SDC `canvas_test_sdc:date` with a `type: string, format: date` prop and a `caption` string prop was added, together with a test-only Twig template, JS behavior using `Intl.DateTimeFormat`, and CSS. The `NodeTemplatesTest` kernel test was extended to assert that the `created` timestamp is rendered as the expected ISO date string through the full content template rendering pipeline.

## Upgrade

No breaking changes. If custom code constructs `DynamicPropSource` instances directly and needs to apply an adapter, use the new factory method:

```php
$source = (new DynamicPropSource($expression))->withAdapter('unix_to_date');
```

Stored `ContentTemplate` config may now contain an optional `adapter` key in a `DynamicPropSource` entry:

```yaml
inputs:
  date:
    sourceType: dynamic
    expression: 'ℹ︎␜entity:node␝created␞␟value'
    adapter: unix_to_date
```

Existing config without the `adapter` key continues to work unchanged. No update hook is needed.
