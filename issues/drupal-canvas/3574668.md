## Summary

Component props marked as required now automatically receive sensible example values when the required toggle is activated. If someone clears that example value, the interface displays a validation error explaining that required props need example values. This prevents confusion about configuration requirements and helps authors create valid component definitions on the first attempt. The system provides appropriate defaults for each data type, such as "Example text" for text fields or "0" for numbers.

## Impact

No upgrade or configuration changes required.

## Technical details

This change addresses a usability gap in the component props configuration form. Previously, users could mark a prop as required without realizing they also needed to provide an example value, leading to invalid configurations.

The implementation introduces a `useRequiredProp` custom hook that tracks when the required toggle changes state. When activated on a prop with no example, the hook triggers a callback to prefill an appropriate default based on the prop type. Default values are defined in `DEFAULT_EXAMPLES` (text: "Example text", integer: "0", date: "2026-01-25", etc.) and `DEFAULT_ENUM_OPTIONS` for list types.

Each form component (text fields, formatted text, links, dates, and enums) was updated to accept a `required` prop, integrate the hook, and display validation errors. The error state is managed locally within each form component and synchronized with user input: clearing an example on a required prop shows the error, filling it hides the error, and toggling required off clears the error.

Type changes on required props also trigger prefilling of the appropriate default for the new type. For list types without existing options, the system creates a default option alongside the example value.
