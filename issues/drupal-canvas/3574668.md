## Summary

Component props marked as required must have example values, but the interface didn't communicate this constraint or help users meet it. Users could create invalid component definitions without realizing the problem. The system now automatically fills in example values when a prop is marked required and displays validation errors if the example field is cleared, making the requirement visible and preventing invalid configurations.

## Impact

No upgrade or configuration changes required.

## Technical details

The issue addresses a usability gap in the component props editor where required props need example values but nothing indicated this constraint. The solution introduces a `useRequiredProp` hook that tracks the required state and example value, automatically prefilling sensible defaults when required is toggled on. Default examples vary by prop type: text gets "Example text", numbers get "0", dates get "2026-01-25", and list types get a default option if none exists.

When users clear an example value on a required prop, the UI now shows "A required prop must have an example value" in red below the field and marks the input with error styling. The error clears when required is toggled off or when a valid value is provided. For enum/list types, the system prefills both an option and selects it as the example. When changing prop types on a required prop, the system updates to the appropriate default for the new type.

The hook uses refs to track previous required state and stabilize dynamic dependencies, triggering the prefill callback only when required transitions from false to true while the example is empty. All major prop type components now pass the required flag and integrate this validation.
