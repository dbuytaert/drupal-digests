## Summary

Drupal Canvas previously misread JSON Schema semantics for multi-value (`type: array`) component props, equating `required: [prop]` with `minItems: 1`. This caused the UI to block removing the last item even when an empty array was perfectly valid, and made it impossible to use `minItems: 1` explicitly in SDCs. The fix enforces correct JSON Schema semantics: a required array prop that lacks `minItems: 1` is now Canvas-ineligible. When `minItems: 1` is present alongside `required`, Canvas correctly enforces non-emptiness in both the form UI and API validation. The code component editor automatically adds `minItems: 1` when "Required" is toggled on for an array prop, so manual YAML editing is only needed for SDCs.

## Impact

Affects module maintainers.

- **API change:** Required `type: array` props in SDCs must now include `minItems: 1` to be Canvas-eligible; bare `required` without `minItems: 1` makes an array prop Canvas-ineligible.

## Technical details

The root cause was a false comment in `JsonSchemaType.php` claiming `required` and `minItems: 1` are equivalent. This led two places to block all `minItems` values entirely: `JsonSchemaType.php` and `EntityFieldPropSourceMatcher.php`. Both now allow `minItems: 1` specifically (all other values still return `NULL`/`[]` because Drupal Field API has no concept of minimum cardinality greater than 1).

`ComponentMetadataRequirementsChecker` gained two new validation rules: a required `type: array` prop without `minItems: 1` is rejected with a message "Multiple-cardinality prop X is required, but does not specify `minItems: 1`"; an optional `type: array` prop that does specify `minItems` is also rejected. This makes `required` array props without `minItems: 1` Canvas-ineligible rather than silently over-enforcing in the UI.

In `GeneratedFieldExplicitInputUxComponentSourceBase`, the call to `setRequired(TRUE)` on the Drupal field widget now only fires when the array prop has both `required` and `minItems: 1`. Without `minItems: 1`, `$is_required` is forced to `FALSE`, so the form shows no required asterisk and the user can remove all items. No JavaScript changes were needed: `isRemoveButtonEnabled()` already read the `.form-required` CSS class generated by the PHP side.

On the TypeScript side, `codeEditorSlice.ts`'s `toggleRequired` action now sets `prop.minItems = 1` when enabling required on an array prop and deletes `prop.minItems` when disabling it. `serializeProps` in `utils.ts` outputs `minItems` when it equals `1`, and `deserializeProps` reads it back. `CodeComponentProp` and `CodeComponentPropSerialized` in `CodeComponent.ts` gained `minItems?: number`.

A new test SDC `sparkline_no_min` (required array, no `minItems`) is now listed as Canvas-ineligible, and `sparkline_min_1` (required array with `minItems: 1`) is eligible and enforced. Existing required array props in several test SDCs had `minItems: 1` added to bring them into compliance with the new rule.

## Upgrade

**SDC authors:** Any `type: array` prop listed in `required` must now also declare `minItems: 1`. Without it, Canvas will report the component as ineligible.

```yaml
# Before (Canvas-ineligible after this fix):
props:
  type: object
  required:
    - items
  properties:
    items:
      type: array
      items:
        type: string

# After (Canvas-eligible):
props:
  type: object
  required:
    - items
  properties:
    items:
      type: array
      minItems: 1
      items:
        type: string
```

`minItems` values greater than 1 remain unsupported. Optional array props (not in `required`) must not specify `minItems` at all.

**Code component authors:** No manual change needed. Toggling "Required" on in the Canvas code editor for an array prop now automatically adds `minItems: 1` to the serialized YAML.

### Collaboration

tedbow opened MR !923 after discovering that Canvas was inconsistently enforcing array prop emptiness: the UI blocked removing the last item even when `value: []` was schema-valid, yet the API happily accepted `[]` for required array props. When wim leers reviewed the MR, he raised a pivotal objection: the MR proposed supporting both `minItems: 1` and bare `required` (treating bare-required arrays as optional in the UI), but his reading was that a `type: array` prop marked `required` without `minItems: 1` is semantically meaningless and should simply be Canvas-ineligible. This shifted the scope significantly. They paired to resolve the remaining test failures, at which point tedbow noted that all multi-value prop editing had been gated behind `canvas_dev_mode`, meaning no production code components could have been created with the now-invalid patterns, eliminating the need for an update path.
