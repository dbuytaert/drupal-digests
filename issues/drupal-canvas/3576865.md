## Summary

AssetLibrary entities can now track custom font files. This feature allows fonts to be added programmatically through the API, with font CSS files included in the generated asset library. This is primarily for the CLI code component push commands, though the editor could expose this in the UI later. The fonts are stored as a list of name and URI pairs, and the CSS files are automatically added to the library when rendered.

## Impact

No upgrade or configuration changes required.

## Technical details

AssetLibrary entities gain a new `fonts` property to track custom font CSS files. The property stores a nullable array of entries, each with a `name` and `uri` field pointing to a managed file.

The `AssetLibrary` entity now includes `getFonts()` and `setFonts()` methods. When fonts are set, they are included in the client-side normalization and persisted to configuration. The default value is null, which `getFonts()` normalizes to an empty array.

In `LibraryHooks::libraryInfoBuild()`, font entries are processed using the `FileUrlGeneratorInterface` to generate public URLs from the managed file URIs. These URLs are added to the library definition as theme CSS assets, ensuring they load when the asset library attaches to a page.

Configuration schema defines fonts as a nullable sequence of mappings with `name` and `uri` string properties. The global asset library configuration includes the new `fonts: null` default. Comprehensive test coverage validates saving, loading, clearing, client-side normalization, and library discovery integration.
