# #3591601: Rudimentary conflict resolution UI

- **Project:** Drupal Canvas
- **Type:** Critical feature request
- **Status:** Fixed
- **Discussion:** [#3591601](https://git.drupalcode.org/project/canvas/-/work_items/3591601) · 1 contributor · 3 comments

## Summary

The Canvas module now includes a rudimentary conflict resolution interface. When an auto-saved page has a conflict with a version published externally, editors can open a dedicated side-by-side comparison screen showing the "Published version" and "New version" (auto-save) of each conflicted page. They select which version to keep, click "Resolve conflict," and the module either discards the auto-save (published version wins) or records the resolution via API (auto-save wins). Multiple conflicts are presented as a paginated queue ("Review 1 of 2"), and after all conflicts are resolved a completion screen confirms that everything is up to date. The feature is currently gated behind the `devConflictDetectionMode` Canvas setting.

## Impact

Affects module maintainers.

- **New API:** New `PATCH /canvas/api/v0/content/auto-save/:entityType/:entityId` payload field `resolved_conflict_id` and four-outcome `ConflictResolutionOutcomeEnum` for the server-side conflict resolution handler.

## Technical details

Two new frontend routes are registered in `AppRoutes`: `/conflict/:entityType/:entityId` renders `ConflictResolutionPage` inside `UiShell`, and `/conflict` renders the same component without an entity param to show the "all resolved" state. A companion set of preview routes (`/conflict-preview/:entityType/:entityId` and an optional `/:width` variant) renders `ConflictPreview`, a stripped-down iframe component that fetches layout HTML via `useGetConflictPageLayoutQuery` with a `?version=published` search param to toggle between the published and auto-save renders. `Topbar` was updated to treat `/conflict-preview` as a preview context so the toolbar renders correctly.

`ConflictResolutionPage` drives the resolution flow. On mount it reads the live pending-changes list, snapshots the initial conflict queue with `getConflictQueue` and `findConflictIndex` (from `conflictComparison.ts`), and holds that snapshot stable while live data refreshes. Each conflict entry shows a two-column `PageConflictComparisonView` with "Visual" and "Text" tabs. Choosing "Published version" calls `useDiscardPendingChangeMutation` (a DELETE on `/canvas/api/v0/auto-saves/:entityType/:entityId`). Choosing "New version" calls `useUpdateContentMutation` (a PATCH on `/canvas/api/v0/content/auto-save/:entityType/:entityId`) with `{ resolved_conflict_id: conflict_id }`. Either path invalidates `Layout` and `Content` RTK Query tags and refetches pending changes. Resolution errors surface as a dismissible toast rather than replacing the comparison view.

On the PHP side, `ConflictResolutionOutcomeEnum` models the four outcomes of the PATCH endpoint: `Resolved` (204), `NoAutoSaveItem` (404), `NoActiveConflict` (422), and `ConflictMismatch` (409). A `ConflictMismatch` response includes the current `conflict_id` in its `meta` payload so the client can detect a race condition and retry.

`PublishReview` gained an optional `open` prop for controlled open state. When provided, the component defers open/close decisions to the caller and omits its own `setInternalOpen` call. `UnpublishedChanges` manages this state and handles a new `?reviewChanges=1` query param: on mount it opens the review panel, refetches changes, and replaces the URL to strip the param. `onResolveConflict` in `UnpublishedChanges` now calls `navigate(getConflictRouteForChange(change))` instead of a bare `refetch()`. The pending-changes API type for `PendingResponse.data` was tightened to `Record<string, { data_hash?: string }>`, and `conflict_id` is now extracted from error metadata (`meta.conflict_id`) returned by the pending-changes endpoint.
