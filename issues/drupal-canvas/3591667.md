## Summary

The Canvas module now ships a `canvas:doctor` command (requiring Drupal 11.4's `dr` CLI) that validates all Canvas data and reports integrity problems. It runs nine distinct health checks: four system checks (Canvas update paths executed, config entities that escaped the update path, whether all installed extensions are at tagged releases, and whether every component source plugin supports schema evolution) and five data checks (Canvas config entities, default content revisions, past revisions, forward/draft revisions, and auto-save snapshots). Each check reports a count of healthy vs. failing items, a duration, and optional per-violation details with a prescription for remediation. The command can output a human-readable table or structured JSON. Validation also runs incrementally via cron on sites that never use the CLI, so health records stay warm without requiring manual intervention.

## Impact

Affects module maintainers and site owners.

- **Database update:** New `canvas_health_records` table is created lazily on first write; no update hook is required.
- **New API:** New `canvas:doctor` CLI command (requires Drupal 11.4+) with `--checks`, `--all-checks`, `--no-cache`, `--details`, and `--format=json` options for auditing Canvas data integrity.

## Technical details

The implementation is organized into five `@internal` classes under `src/Health/` plus the command itself in `src/Command/DoctorCommand.php`.

`HealthCheck` is a string-backed PHP enum with nine cases split into system checks (run once per invocation, never cached) and data checks (streamed as generators, results persisted). The `isSystemCheck()` and `findingsAreRisks()` methods are the single source of truth for those splits: adding a new case without updating both exhaustive `match` expressions causes a PHPStan failure.

`Doctor` is the validation engine. `runDataChecks()` yields `CheckItem` structs via generators, which lets `DoctorCommand` display a live progress bar without loading all entities into memory first. `runCronWindow()` accepts an offset and limit, enabling `HealthHooks::cron()` to advance a cursor stored in Drupal state across separate cron runs. Each data check's freshness signal is computed at the moment Doctor visits the item, not when `Inventory` listed it, so paginated results from a previous cron run do not carry stale freshness data.

`HealthRecords` persists results in the `canvas_health_records` database table (keyed on `data_key` + `check`). The table is created lazily on first write rather than via a formal install hook. Freshness is determined by matching two fingerprints: an environment fingerprint (Canvas schema version, count of applied and pending post-updates, xxh64 hash of all installed extension versions) computed by `Environment`, and a data fingerprint per item (cache-tag checksum for config and content entities, a precomputed data hash for auto-save snapshots). If any installed extension reports no release version (`dev`), `HealthRecords::isFresh()` unconditionally returns `false`, disabling all result reuse.

`UpdatesEscapedConfig` detection uses reflection to enumerate `CanvasConfigUpdater::needs*()` methods, filters to those whose first required parameter type-matches the entity being inspected, and calls each. Deprecation emissions from those methods are suppressed during the audit by temporarily calling `setDeprecationsEnabled(FALSE)`. The `DoctorCommand` itself is thin: it delegates all logic to `Doctor` so the same validation can power the planned web UI ([#3588907](https://www.drupal.org/node/3588907) via Drupal Batch API).
