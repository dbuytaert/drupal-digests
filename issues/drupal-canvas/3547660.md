## Summary

In the Canvas visual editor, UI annotation overlays for slots and child components would momentarily snap to the top-left corner of the canvas whenever a component prop was changed. The overlays returned to their correct positions only after the server responded. This issue is now fixed: annotations remain stable in their correct positions throughout the entire real-time update cycle.

## Impact

No upgrade or configuration changes required.

## Technical details

The root cause was in `calculateBoundingRect` (in the DOM geometry utilities). When a real-time prop change triggers a component re-render, the preview iframe briefly removes and re-attaches DOM elements. The function was still including those disconnected elements when computing bounding rectangles. Disconnected elements report their position as 0,0, so the computed offset collapsed to the canvas origin, moving every annotation to the top-left corner.

The fix introduces an `el.isConnected` guard inside `collectElementsWithCalculableSize`: any element not currently attached to the live document is skipped. If every candidate element is filtered out this way, `calculateBoundingRect` now returns `null` instead of attempting to compute a zero-based rect. Matching that, `getDistanceBetweenElements` now has the return type `{ horizontalDistance: number | null; verticalDistance: number | null }` and returns `null` values (instead of `0`) when either bounding rect is `null`. Both call sites in `useSyncPreviewElementOffset` check for `null` distances and bail out early, preserving the annotation's last known valid position until a measurement with live DOM elements is available again.
