## Summary

Two fixes shipped in this commit for the Multiple Email module. First, the `multiple_email.confirmer` service was converted to use Symfony autowiring, so it no longer requires a long list of manually specified arguments in the service container. Second, a bug in the add-email form was corrected: the redirect to the email management page now only fires after the email entity has been successfully saved, rather than unconditionally at the end of the submit handler.

## Impact

No upgrade or configuration changes required.

## Technical details

In `multiple_email.services.yml`, the `multiple_email.confirmer` service received `autowire: true`. The previous definition listed seven explicit arguments (`@config.factory`, `@datetime.time`, `@plugin.manager.mail`, `@language_manager`, `@token`, `@logger.channel.multiple_email`, `@string_translation`). After the change, only `$logger: '@logger.channel.multiple_email'` remains explicit, because a named logger channel cannot be resolved by type alone; all other dependencies are injected automatically. A test-layer stub file that declared a local `LoggerChannelInterface` shim was removed along with this change.

In `MultipleEmailAddForm::submitForm()`, the `$form_state->setRedirect('multiple_email.manage', ...)` call was moved inside the conditional block that wraps the `$entity->save()` and confirm calls. Previously it sat outside that block, so it executed regardless of whether the save branch was entered. Two unused `use Drupal\user\UserInterface` imports were also removed from the entity class and its interface.
