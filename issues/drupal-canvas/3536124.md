## Summary

Drupal Canvas (formerly Experience Builder) now automatically configures the bundled `@drupal-api-client/json-api-client` for use in code components. Previously, component authors had to supply the site base URL and JSON:API prefix when constructing a client; now they can write `new JsonApiClient()` with no arguments and everything is set up from server-provided settings. As a side-effect, the `baseUrl` value in `drupalSettings.xbData.v0` was corrected from a root-relative path (e.g. `/subpath/`) to a full absolute URL (e.g. `https://example.com/subpath`), which is what JavaScript HTTP clients actually need.

## Impact

Affects module maintainers.

- **New API:** New `drupalSettings.xbData.v0.jsonapiSettings` key exposes the JSON:API base path to code components, and `new JsonApiClient()` now works with zero constructor arguments.

## Technical details

Two problems existed in the old wrapper (`ui/lib/astro-hydration/src/lib/jsonapi-client.ts`). First, it read the base URL from `parent.window.location.origin + drupalSettings.path.baseUrl` via a try/catch, which is fragile in iframe contexts. Second, it had no knowledge of the JSON:API base path, so a site that changed the `jsonapi.base_path` service parameter away from `/jsonapi` would silently break.

The fix has three parts. In `CodeComponentDataProvider`, the `baseUrl` calculation was changed from `$request->getBaseUrl() . '/'` to `$request->getSchemeAndHttpHost() . $request->getBaseUrl()`, producing a proper absolute URL. A new method `getXbDataJsonApiSettingsV0()` was added that reads the `jsonapi.base_path` service parameter via the injected `ContainerInterface`; if the JSON:API module is not installed the method returns `null` for `jsonapiSettings`, otherwise it returns `['apiPrefix' => ltrim($jsonapi_base_path, '/')]`. A new `experience_builder/xbData.v0.jsonapiSettings` asset library carries this value into `drupalSettings.xbData.v0.jsonapiSettings`.

The static method `getRequiredXbDataLibraries()` now also detects the string `@drupal-api-client/json-api-client` in a component's compiled JS and automatically attaches both `xbData.v0.baseUrl` and `xbData.v0.jsonapiSettings` libraries, so no manual library declaration is needed from the component author.

The `XbJsonApiClient` wrapper was rewritten to read `clientBaseUrl` from `window.drupalSettings?.xbData?.v0?.baseUrl` and `apiPrefix` from `window.drupalSettings?.xbData?.v0?.jsonapiSettings?.apiPrefix`, with all constructor options still overridable by the caller. If `jsonapiSettings` is explicitly `null` (module absent), the constructor throws a clear error. The `DrupalSettings` TypeScript interface was updated to include the `jsonapiSettings` field.

## Upgrade

Code components that previously read `getSiteData().baseUrl` and expected a root-relative path with a trailing slash will now receive a full absolute URL without a trailing slash:

```js
// Before — root-relative, e.g. "/subpath/"
const { baseUrl } = getSiteData();

// After — absolute URL, e.g. "https://example.com/subpath"
const { baseUrl } = getSiteData();
```

Code components using `@drupal-api-client/json-api-client` can drop constructor arguments:

```js
// Before
import { JsonApiClient } from "@drupal-api-client/json-api-client";
const client = new JsonApiClient(baseUrl, options);

// After — no arguments needed
import { JsonApiClient } from "@drupal-api-client/json-api-client";
const client = new JsonApiClient();
```
