## Summary

The Canvas Stark theme now uses custom HTML elements in Twig templates instead of a separate theme engine to embed React components. This simplifies how forms render in the Canvas UI by making templates more readable and reducing the number of internal processing stages. Instead of JSON configuration files that few people understood, developers now work with familiar Twig syntax that includes custom tags like `<drupal-canvas-form>` which map directly to React components. The change removes approximately 500 lines of theme engine code while achieving the same functionality.

## Impact

No upgrade or configuration changes required.

## Technical details

Canvas previously used a custom "semi-coupled" theme engine that processed Twig templates containing JSON configuration. These JSON files mapped render array properties to React component props, requiring developers to understand both the theme engine's special syntax and its processing pipeline.

The new approach eliminates the theme engine entirely. Twig templates now use custom HTML elements (like `<drupal-canvas-input>` or `<drupal-canvas-textarea>`) that Hyperscriptify converts to React components on the client side. A new `CanvasPropVisitor` Twig node visitor detects when variables appear inside these custom element attributes and automatically wraps them with a `jsx_attributes` filter. This filter converts Attribute objects and arrays to JSON, ensuring React receives properly formatted props.

For props that need to be elements (not strings), a new `slot()` Twig function wraps content in `<element-prop slot="name">` tags. Hyperscriptify maps these slots to React props with element values, leveraging the Web Components slots API pattern.

The visitor includes logic to detect the `children` attribute within custom elements and throws a syntax error, since `children` is reserved in React. It parses the Twig buffer character-by-character to determine context, handling escaped quotes and tracking whether the parser is inside an attribute value.

Templates moved from `process_as_jsx/` directories to regular locations. The media library still uses standard Twig templates (not React) because dialogs need rendered HTML before insertion into the page. Theme suggestion hooks moved from a dedicated `SemiCoupledThemeEngineHooks` class into `canvas_stark.theme`.
