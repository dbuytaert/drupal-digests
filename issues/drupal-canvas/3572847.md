## Summary

Canvas tests now validate all component trees before saving them into content entities. Previously, some test fixtures created invalid component trees without validation, which could hide bugs and lead to incorrect reasoning about Canvas behavior. This gap became apparent during Drupal 11.3 testing. The project added validation to `CanvasTestSetup` and all kernel tests that create component trees, simplified test fixtures by removing unnecessary adapter usage, and extended the custom PHPCS rule to automatically enforce this requirement going forward. This ensures all tests work with valid data and makes future test authoring safer.

## Impact

No upgrade or configuration changes required.

## Technical details

`CanvasTestSetup` creates test fixtures for Cypress end-to-end tests and many kernel tests. It was creating content entities with component trees but skipping validation, allowing invalid data into the test database. Similarly, some kernel tests like `BlockComponentTest::testGetExplicitInput()` created nodes with component trees without validating them first.

The fix adds `self::assertEntityIsValid()` calls throughout the test suite. This new helper method in `CanvasKernelTestBase` validates both content entities and config entities with helpful output when validation fails. All component tree creation in tests now goes through validation before saving.

The implementation also simplified test fixtures. Many tests used complex `StaticPropSource` structures and adapters when simple values would suffice. For example, image inputs changed from full entity reference expressions to just `['target_id' => 4]`. This makes tests more readable while still exercising the same Canvas functionality.

A custom PHPCS rule now detects manual `assertSame([], self::violationsToArray(...))` patterns in `CanvasKernelTestBase` subclasses and requires using `self::assertEntityIsValid()` instead. This enforces consistent validation patterns across the test suite.

The work also introduced `DataProviderWithComponentTreeTrait::populateActiveComponentVersionPlaceholders()` to handle test data that uses placeholder component versions, replacing the more narrowly-focused trait name that only mentioned block components.

## Lessons learned

- When test fixtures create entities, validate them before saving. Invalid test data leads to incorrect reasoning about failures and can hide legitimate bugs in the system under test. *(via wim leers, phenaproxima)*
- When test infrastructure is shared across many tests (like fixtures or setup classes), enforce validation there even for entities tangential to the feature being tested. Simple mistakes in widely-used fixtures skew reasoning across the entire test suite. *(via wim leers, phenaproxima)*
