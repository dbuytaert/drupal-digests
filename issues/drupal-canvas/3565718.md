## Summary

The Canvas module now validates that the `required` flag stored in a Component config entity's `active` version prop field definitions actually matches the requiredness declared by the underlying component source. Previously, incorrect `required` values could be silently saved. Sites using `canvas.generated_field_explicit_input_ux`-backed components where the stored `required` flag disagrees with the real component metadata will now receive validation errors.

## Impact

No upgrade or configuration changes required.

## Technical details

The old `NotNullValueForEveryRequiredSdcPropConstraint` was removed. It was SDC-specific, required `ComponentPluginManager` as a constructor dependency, and operated at the `prop_field_definitions` mapping level. It compared the entire mapping against the SDC plugin's schema in a single pass.

Two source-agnostic constraints were added in its place, both using `ComponentConfigEntityDependentValidatorTrait` to reach the live `Component` config entity during validation:

- `MatchesComponentPropRequirednessConstraint` (`type: bool`): applied directly to the `required` field inside each prop entry in the `canvas.generated_field_explicit_input_ux` config schema. The validator looks up the prop name from the validation context, reads the component source's metadata schema, and checks whether the stored boolean matches membership in the schema's `required` array. Violation message: `The requiredness of the prop "%prop_title" (%prop_machine_name) must match its implementation.`
- `NotNullIfRequiredComponentPropConstraint`: applied to the `default_value` mapping inside each prop entry. It reads the sibling `required` field from the context and raises a violation only when `required` is true but `default_value` is null. Violation message: `The required component prop "%prop_title" (%prop_machine_name) must not be null.`

A shared helper `getComponentSourceFromComponentIfPossible()` was extracted into `ComponentConfigEntityDependentValidatorTrait`. It silently returns `NULL` when the source plugin does not exist or when the component is broken, deferring to other existing constraints (`PluginExists`) in those cases. `ComponentStatusConstraintValidator` was refactored to use this helper, removing duplicated guard logic.

Test fixtures for JS-backed components (`js.canvas_test_code_components_with_props`, `js.canvas_test_code_components_with_slots`) had their `name` prop corrected from `required: false` to `required: true`. The SDC test fixture `sdc.canvas_test_sdc.my-cta` had `href` corrected from `required: false` to `required: true`. Version hashes were regenerated accordingly. The test method `testInvalidWidgetSettings` was renamed to `testInvalidPropFieldDefinition` and extended to assert both violation types.

## Upgrade

No code changes are required for most sites. However, if you have Component config entities (using the `canvas.generated_field_explicit_input_ux` config schema type) where the stored `required` flag for any prop does not match the actual component source metadata, those entities will now fail config validation. Audit affected entities and correct the `required` values, then re-export config.
