## Summary

Canvas component instances are now required to use the exact field type, storage settings, and instance settings defined by the referenced `Component` config entity version. Previously a component instance could deviate from the component version's field type specification. That flexibility is removed: `StaticPropSource` inputs stored in component tree items must always be in the compact "collapsed" format (just the value, not a full `sourceType`/`expression`/`sourceTypeSettings` array). This guarantees consistency across all instances of a given component version, which in turn makes update paths for component version upgrades far more tractable.

## Impact

Affects module maintainers.

- **New API:** New `InvalidComponentInputsPropSourceException` thrown by `ComponentSourceInterface::optimizeExplicitInput()` when a `StaticPropSource` shape does not match the Component entity version's field type.

## Technical details

The enforcement is in `GeneratedFieldExplicitInputUxComponentSourceBase::optimizeExplicitInput()`. When processing a `StaticPropSource`, it now calls `getDefaultStaticPropSource()` and compares shapes via `hasSameShapeAs()`. A mismatch throws the new `InvalidComponentInputsPropSourceException`, which `ValidComponentTreeItemConstraintValidator` catches and converts into a validation violation citing the component ID and version. The `collapse()` method was also hardened: rather than returning an uncollapsed array when the shape does not match, it now throws a `\LogicException` as a programming-error guard (the validation layer should have already rejected the input before `collapse()` is reached with a mismatching source).

Three `getSourceSpecificComponentId()` methods in `BlockComponentDiscovery`, `JsComponentDiscovery`, and `SingleDirectoryComponentDiscovery` had their `str_starts_with()` arguments reversed (haystack and needle were swapped); the commit corrects this as a bug fix alongside the main change.

A new test support module `canvas_test_entity_reference_shape_alter` was added to test the `canvas_storable_prop_shape_alter` hook path where a prop's field type is altered to `entity_reference`. The `ContentTemplateOnDependencyRemovalTest::testRemoveFieldTypeProviderModule()` test was reworked because the old scenario (a deviant static prop source using a third-party field type) is no longer valid; the module-removal test now focuses on a field *config* dependency rather than a component instance dependency. `ApiLayoutControllerTest` (functional) was removed entirely because it tested sending uncollapsed static prop sources through the layout API, which is now a validation error.

## Upgrade

Code that creates or stores component instances programmatically must use the collapsed input format for static prop sources. The field type and expression to use are those defined in the `Component` config entity version.

```php
// Before: uncollapsed static prop source (now rejected by validation)
'heading' => [
  'sourceType' => 'static:field_item:string',
  'value' => 'Hello, world!',
  'expression' => 'ℹ︎string␟value',
],
'cta1href' => [
  'sourceType' => 'static:field_item:uri',
  'value' => 'https://drupal.org',
  'expression' => 'ℹ︎uri␟value',
],

// After: collapsed format matching the Component version's field type
'heading' => 'Hello, world!',
'cta1href' => ['uri' => 'https://drupal.org'],
```

Non-static prop source types (`dynamic`, `host-entity-url`) are unaffected and continue to use the full array format with a `sourceType` key.

If a test or recipe previously supplied a `StaticPropSource` with a field type that differed from the one recorded in the `Component` entity version, you must either update the stored input to match the component version's field type, or create a new component version whose `prop_field_definitions` record the field type your test actually needs.
