## Summary

Canvas's shape-matching system previously required SDC `object` props to carry an explicit `$ref` URI pointing to a well-known Canvas definition (such as `json-schema-definitions://canvas.module/image`) in order to receive the correct field widget (for example, the Media Library image picker). If a component author wrote the equivalent inline JSON Schema schema without `$ref`, or referenced a different URI that resolved to the same schema, Canvas would report that it did not know of a field type or widget for the prop. This fix makes shape matching work on the resolved, normalized form of a schema, so that any prop whose structure is structurally equivalent to a well-known prop shape is now matched correctly, regardless of how that schema was written. As a secondary fix, `meta:enum` keys containing dots are now permitted in SDC props (they were already valid JSON Schema but had been incorrectly rejected); the dot restriction continues to apply only to JavaScript code components, where keys are stored as Drupal config entities that cannot contain dots. Sites running Canvas should run `drush updb` to trigger the SDC rediscovery cache wipe that ships with this change.

## Impact

Affects module maintainers and site owners.

- **Database update:** New `canvas_post_update_0008_rediscover_sdcs()` wipes the SDC plugin manager cache to force re-evaluation of all SDC components.
- **API change:** `ComponentMetadataRequirementsChecker::check()` gained a required fourth parameter `$forbidden_key_characters`; callers must add it.
- **New API:** `PropShape::standardize()` added for resolving and normalizing a raw prop schema to its canonical well-known form; `ComponentPluginManager::resolveJsonSchemaReferences()` made public.

## Technical details

The root cause was that shape matching in `JsonSchemaType::computeStorablePropShape()` and related code matched against raw prop schemas, including the literal `$ref` string. This meant the matching would only succeed when the exact `$ref` URI was present; a structurally equivalent inline schema or a different `$ref` that resolved to the same content would not match.

The fix introduces `PropShape::standardize()`, which resolves `$ref`s via `ComponentPluginManager::resolveJsonSchemaReferences()` (powered by `justinrainbow/json-schema`'s `SchemaStorage::resolveRefSchema()`), normalizes the result, then checks whether it appears in the map returned by the new `getWellKnownPropShapes()`. That private static method scans all installed modules and themes for `/schema.json` files, reads their `$defs` sections, and builds a map from `json-schema-definitions://` URIs to normalized schemas. If the incoming prop schema resolves to a known shape, `standardize()` returns a `PropShape` carrying the canonical `$ref` URI, which the existing matching code then recognizes. For `array` prop shapes, `standardize()` also tries matching the `items` schema. This replaces the three independent hand-rolled `resolveSchemaReferences()` methods that existed in `AdapterBase`, `JsonSchemaFieldInstanceMatcher`, and `PropShape`; all now delegate to the single `ComponentPluginManager::resolveJsonSchemaReferences()`. A workaround `RuntimeException` that had prevented Canvas-namespace `json-schema-definitions://` refs from being resolved by `justinrainbow/json-schema` is removed. The resolved schema's `id` property (set by the JSON Schema library post-resolution) is stripped to avoid spurious URI-format validation errors from `justinrainbow/json-schema` v5.

The `meta:enum` fix works by adding a `$forbidden_key_characters` parameter to `ComponentMetadataRequirementsChecker::check()`. Callers for `SingleDirectoryComponent` pass an empty array (no forbidden characters), while callers for `JsComponent` pass `['.' => '_']`. The dot-to-underscore substitution that had been hardcoded now flows through this parameter. The previously rejected SDC `canvas_test_sdc:component-mismatch-meta-enum` is now a valid, enabled Canvas component.

`canvas_post_update_0008_rediscover_sdcs()` is an intentionally empty update hook whose sole purpose is to wipe the SDC plugin manager cache, forcing re-evaluation of all SDC components under the new matching logic.

## Upgrade

If your module calls `ComponentMetadataRequirementsChecker::check()` directly, add the new required `$forbidden_key_characters` parameter:

```php
// Before:
ComponentMetadataRequirementsChecker::check($id, $metadata, $required_props);

// After (for SDCs, where dots are allowed):
ComponentMetadataRequirementsChecker::check($id, $metadata, $required_props, forbidden_key_characters: []);

// After (for code/JS components, where config entities forbid dots):
ComponentMetadataRequirementsChecker::check($id, $metadata, $required_props, forbidden_key_characters: ['.' => '_']);
```

Run database updates to trigger SDC rediscovery:
```
drush updb
```
