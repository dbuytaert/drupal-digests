## Summary

A new optional submodule, `entity_blueprint_config`, has been added to the Entity Blueprint project. Enabling it extends Entity Blueprint so that config entities (node types, field storage, views, etc.) can be serialized to a blueprint array, deserialized and created or updated from that array, validated structurally and semantically, and manipulated through targeted operations, all using the same patterns as the existing content-entity backend. The module is marked experimental and requires Drupal 10.3 or 11.

## Impact

Affects module maintainers.

- **New API:** New `entity_blueprint_config` submodule adds `ConfigBlueprintBackend`, `ConfigSerializer`, `ConfigDeserializer`, `ConfigOperations` (`ConfigOperationsInterface`), `ConfigSchemaBuilder`, and the `PropertyHandlerInterface` / `PropertyHandlerManager` extension point for serializing config entity properties.

## Technical details

The submodule registers a `ConfigBlueprintBackend` service that implements `BlueprintBackendInterface` at priority 0 (above the content backend at priority -10). Its `supports()` method checks that the entity type implements `ConfigEntityInterface`, so it is selected automatically for all config entity types without any caller changes.

Serialization is handled by `ConfigSerializer`, which calls `$entity->toArray()`, strips system properties (`uuid`, `_core`, `dependencies`, `langcode`) via `ConfigEntityMetadataTrait::$systemProperties`, and delegates plugin collection keys to `PluginCollectionPropertyHandler`. Deserialization is two-phase: `ConfigStructuralValidator` checks required fields and immutability constraints first, then `ConfigDeserializer::hydrateEntity()` calls `$storage->create()` or `$storage->load()` and applies properties; for entities implementing `EntityWithPluginCollectionInterface` it routes plugin data through `DefaultLazyPluginCollection`. `ConfigSchemaBuilder` introspects type definitions from `TypedConfigManagerInterface`, trying progressively more wildcard segments (up to four) to handle config prefixes with varying dot depth.

The `ConfigOperations` class (implementing `ConfigOperationsInterface`) provides targeted mutation methods: `updateProperties`, `addPlugin`, `updatePlugin`, `removePlugin`, `reorderPlugins`, and `batchOperations`. Batch operations apply a list of `{op, ...}` entries sequentially via a `match` dispatch, short-circuiting on the first failure. All methods return a `ConfigBlueprintResult` carrying the unsaved entity on success or collected `BlueprintError` objects on failure; persistence is the caller's responsibility.
