## Summary

The `ai_best_practices` project now ships a new `evals/compare.py` script and expanded `CONTRIBUTING.md` documentation that let contributors measure whether adding or editing a guidance skill actually changes Claude's output in the desired direction. Before this, there was no automated way to distinguish a skill that genuinely improves model behavior from one that merely adds text. Contributors submitting new or modified skills are now expected to include comparison output in their merge request description.

## Impact

Affects module maintainers.

- **New API:** New `evals/compare.py` script for A/B comparison of skill effectiveness, with `--no-baseline`, `--baseline`, `--runs`, `--model`, `--output-dir`, and `--dry-run` flags.

## Technical details

The new `evals/compare.py` supports two modes, controlled by flags. The default mode calls `git show HEAD:skills/...` to retrieve the last committed skill version as a baseline and compares it against the working copy; `--baseline /path/to/old.md` allows an explicit file instead. The `--no-baseline` flag substitutes a no-skill prompt for the baseline, measuring whether the skill adds value at all over the raw model.

Both configurations call `claude -p` with identical flags (`--no-session-persistence`, `--permission-mode auto`, `--output-format json`). The only variable between configs is whether skill content is prepended to the prompt. Grading is fully deterministic: `must_contain_any` and `must_not_contain` string checks on the lowercased response, plus optional `php -l` lint on extracted PHP code blocks via `check_php_lint`. There is no LLM judge. For isolation, `~/.claude/CLAUDE.md` is moved aside during the run and restored afterward so user-level plugins and configuration do not contaminate either config (`--no-isolate` skips this). Pass rates, token counts, and cost are aggregated across runs and saved as `comparison-{timestamp}.json` in the skill's eval directory; these files are excluded from git via a new `.gitignore`.

`CONTRIBUTING.md` gains a detailed A/B comparison section covering both editing and new-skill workflows, how to read the per-case delta table, and a six-step walkthrough for writing an `evals.json` from scratch (directory layout, `static-checks.json`, behavioral case structure, `README.md`, and verification commands). `evals/README.md` also gets a short A/B synopsis linking to the full guide.
