## Summary

Vector search performance on large datasets can now be optimized through pgvector index configuration. When using the Postgres AI VDB provider for Search API, site administrators can choose between exact search (no index) for small datasets or approximate nearest neighbor indexes (HNSW or IVFFlat) for large datasets. The index strategy can be changed at any time through the Search API server configuration form, and the system automatically rebuilds the database index to match. This setting respects the configured similarity metric (cosine, Euclidean, or inner product) and ensures the correct operator class is used.

## Impact

Affects module maintainers.

- **New API:** New `vector_index_strategy` database setting in Postgres provider configuration allows selecting None, HNSW, or IVFFlat pgvector index strategy for Search API servers.

## Technical details

The `PostgresProvider` plugin now exposes a `vector_index_strategy` select element in `buildSettingsForm()` with three options: None, HNSW, and IVFFlat. The strategy is stored in the Search API server configuration at `database_settings.vector_index_strategy`.

Because the upstream `ai_search` module does not call `submitSettingsForm()` on the provider, index application is handled by a `ConfigEvents::SAVE` event subscriber (`VectorIndexConfigSubscriber`) at priority `-10`. This runs after the upstream subscriber creates the collection table, ensuring the table exists before indexing.

The `PostgresPgvectorClient` provides three new methods: `createVectorIndex()`, `dropVectorIndex()`, and `ensureVectorIndex()`. The last compares the current index strategy and similarity metric operator class against the desired configuration and only drops/recreates when needed. Index creation uses the appropriate pgvector operator class (`vector_cosine_ops`, `vector_l2_ops`, or `vector_ip_ops`) based on the configured similarity metric.

When clearing all indexed data via `deleteAllItems()`, the provider automatically re-applies the configured index strategy after the table is recreated.

A new `VectorIndexStrategy` enum defines the three options and provides human-readable labels. A `VectorIndexException` is thrown on indexing failures and caught by the subscriber, which logs errors without blocking the configuration save.

## Upgrade

Site owners using the Postgres AI VDB provider should review their Search API server configuration and select an appropriate vector index strategy:

1. Navigate to **Configuration → Search and metadata → Search API → [Server] → Edit**
2. Under **Database settings**, select a **Vector index strategy**:
   - **None** for small datasets (exact search)
   - **HNSW** for large datasets (recommended)
   - **IVFFlat** as an alternative ANN index
3. Save the configuration

The index will be created or updated automatically. For large collections, indexing may take a few moments. No code changes are required; this is a configuration-only change exposed through the administrative UI.

## Lessons learned

- When an upstream module doesn't expose a lifecycle hook (like submitSettingsForm), you can use a config save event subscriber with lower priority to run logic after the upstream has finished. This ensures dependencies like database tables exist before your code attempts to reference them. *(via nikro)*
- When exposing database index strategies in configuration UI, make the UI metric-aware: if the user changes similarity metric (cosine vs euclidean), automatically recreate the index with the corresponding operator class rather than forcing users to understand the mapping between metrics and operator classes. *(via nikro)*
