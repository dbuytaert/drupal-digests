## Summary

The AI VDB Provider Postgres module now allows you to choose how your vector search data is indexed. You can select between exact search for small datasets or use HNSW or IVFFlat strategies for faster searches on large datasets. This setting is exposed in the Search API server configuration form and can be changed at any time. The module automatically creates or updates the database index when you save your configuration, and it respects your chosen similarity metric (cosine, euclidean, or inner product) when building the index.

## Impact

Affects module maintainers.

- **New API:** New `VectorIndexStrategy` enum and `PostgresPgvectorClient::ensureVectorIndex()` method for managing pgvector index strategies

## Technical details

The `PostgresProvider` plugin now exposes a **Vector index strategy** select element in `buildSettingsForm()` with three options: None (exact search via sequential scan), HNSW (Hierarchical Navigable Small World), and IVFFlat. The selected strategy is persisted in `database_settings.vector_index_strategy` within the Search API server configuration.

Because the upstream `ai_search` module does not invoke provider `submitSettingsForm()` during normal form submission, index application happens via a `ConfigEvents::SAVE` event subscriber at priority `-10`. This runs after the upstream subscriber (priority `0`) that creates the collection table, ensuring the table exists before index operations.

The `PostgresPgvectorClient::ensureVectorIndex()` method compares the current index state (strategy and operator class) against the desired configuration and only drops and recreates the index when they differ. The operator class is derived from the configured similarity metric: `vector_cosine_ops`, `vector_l2_ops`, or `vector_ip_ops`.

A new `VectorIndexStrategy` backed enum provides case definitions, human-readable labels, and form options. `VectorIndexException` is thrown on index operation failures and logged without failing the entire config save. Index changes are also reapplied after clearing all indexed data via `deleteAllItems()`.

## Upgrade

No external code changes are required. This is a new feature exposed through the UI.

If you maintain custom code that directly calls `PostgresPgvectorClient` methods or extends `PostgresProvider`, note that the client now provides public methods `ensureVectorIndex()`, `createVectorIndex()`, `dropVectorIndex()`, and `getVectorIndexStrategy()` for managing pgvector indexes programmatically. The new `VectorIndexStrategy` enum is available at `\Drupal\ai_vdb_provider_postgres\Enum\VectorIndexStrategy`.

## Lessons learned

- When a framework creates resources (tables, indexes) in response to configuration saves but does not expose a submit handler, use a config save event subscriber at a lower priority than the framework's own subscriber. This ensures the framework's creation logic completes before your dependent logic (like index creation) runs, guaranteeing the resource exists before you reference it. *(via nikro)*
- When managing database indexes based on user configuration, detect the current index state before making changes. Drop and recreate only when the desired strategy or operator class differs from what exists, avoiding unnecessary schema operations that can be expensive on large tables. *(via nikro)*
