## Summary

New `input_length_limit` guardrail plugin blocks chat requests that exceed a configurable character or token count before they reach the AI provider API. This provides a safety net against denial-of-wallet attacks (consuming expensive API tokens), context stuffing attacks, and overwhelming the model's context window.

## Impact

Affects module maintainers and site owners.

- **Config change:** New `ai.guardrail.settings.input_length_limit` config schema with `max_length`, `use_tokens`, `tokenizer_model`, `check_all_messages`, and `violation_message` keys.
- **New API:** New `input_length_limit` guardrail plugin blocks chat input exceeding a configurable character or token count.

## Technical details

The AI module's guardrail system had no built-in way to limit the length of user input sent to AI providers. The new `InputLengthLimit` plugin (`src/Plugin/AiGuardrail/InputLengthLimit.php`) extends `AiGuardrailPluginBase` and implements `processInput()` as a pre-generate guardrail.

The plugin only applies to `ChatInput`. Non-chat input types (e.g. `TextToImageInput`) receive a `PassResult` and are skipped. By default, it measures the last user message only. When `check_all_messages` is enabled, it concatenates all `ChatMessage` texts (joined with newlines) and measures the total.

Length measurement supports two modes: character-based counting via `mb_strlen()` (the default), and token-based counting via the `ai.tokenizer` service with a configurable `tokenizer_model` (defaults to `gpt-4`). When the measured length exceeds `max_length`, the plugin returns a `StopResult` with a configurable violation message supporting `@count`, `@max`, and `@unit` placeholders. The `processOutput()` method always returns `PassResult` since this is an input-only guardrail.

The configuration form uses Drupal `#states` to show the `tokenizer_model` field only when `use_tokens` is checked. A bug in the initial implementation used the wrong form element name selector for this visibility toggle, which was fixed during review.

Config schema was added at `ai.guardrail.settings.input_length_limit` with five properties: `max_length` (integer), `use_tokens` (boolean), `tokenizer_model` (string), `check_all_messages` (boolean), and `violation_message` (string). Unit tests cover character limits, token limits, all-messages mode, last-message-only mode, custom violation messages, multibyte characters, and non-chat input passthrough. Kernel tests verify the full guardrail pipeline using `AiGuardrail` and `AiGuardrailSet` config entities with the EchoAI test provider.
