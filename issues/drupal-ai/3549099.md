## Summary

The `gemini_provider` AI module now exposes Gemini API token usage data to callers. After a chat operation, the response object carries input, output, total, reasoning (thoughts), and cached token counts sourced from Gemini's `usageMetadata` field. This makes it possible for other modules, ECA plugins, and custom code to track API consumption for cost analysis, per-user logging, or usage-based workflows without any changes to how existing code calls the provider.

## Impact

Affects module maintainers.

- **New API:** New `setChatTokenUsage()` protected helper on `GeminiProvider` populates `ChatOutput` with a `TokenUsageDto` containing prompt, candidate, total, reasoning, and cached token counts from Gemini's `usageMetadata`.

## Technical details

The original issue proposed adding a `$lastResponse` property and a `getLastResponse()` getter to the low-level `ApiClient` service. The accepted implementation instead used the integration point already provided by the AI Core module: `ChatOutput::setTokenUsage()` paired with `TokenUsageDto`.

In `GeminiProvider` (`src/Plugin/AiProvider/GeminiProvider.php`), which extends `AiProviderClientBase`, the `chat()` method now constructs the `ChatOutput` object first and then calls a new protected helper `setChatTokenUsage(ChatOutput $chat_output, Arrayable $raw_response): ChatOutput`. That helper calls `$raw_response->toArray()` and populates a `TokenUsageDto` with named constructor arguments: `input` from `usageMetadata.promptTokenCount`, `output` from `usageMetadata.candidatesTokenCount`, `total` from `usageMetadata.totalTokenCount`, `reasoning` from `usageMetadata.thoughtsTokenCount`, and `cached` from `usageMetadata.cachedContentTokenCount`, all with `?? NULL` fallbacks. `TokenUsageDto` comes from `Drupal\ai\Dto\TokenUsageDto` and `Arrayable` from `Gemini\Contracts\Arrayable`.

Token usage is only attached when the request is not streamed and not running inside a PHP Fiber, guarded by `!$this->streamed && !\Fiber::getCurrent()`. This avoids trying to read `usageMetadata` from an incomplete streaming response.

A second cleanup commit removed the custom `loadApiKey()` override from `GeminiProvider`. The override called `$this->keyRepository->getKey(...)` directly and would throw a generic PHP error on a missing key; the parent class `AiProviderClientBase` already provides `loadApiKey()` with proper `AiSetupFailureException` handling, so the override was unnecessary and inferior.
