## Summary

AI Automators in the `ai_automators` submodule can now be run in bulk via Views Bulk Operations or any other Drupal action UI. To use this, set the automator's worker type to "Action" when configuring it. The automator will not run during normal entity saves - only when explicitly triggered as an action. If `edit_mode` is disabled on the automator, entities that already have a value for the target field are skipped, which avoids unnecessary AI API calls on existing content.

## Impact

Affects module maintainers and site owners.

- **Config change:** New valid value `'action'` for the `worker_type` property on `ai_automator` config entities; automators with this type are exposed as Drupal actions and do not run during entity save
- **New API:** New `ActionProcessing` process rule plugin (`id: 'action'`) and `RunAutomatorAction` action plugin (`id: 'ai_automators_run'`) with `RunAutomatorActionDeriver` in `ai_automators`

## Technical details

**Background.** `ai_automators` processes fields on entity save using pluggable worker types (called process rules). Previously there was no way to run an automator on demand against a filtered set of existing entities.

**New process rule: `ActionProcessing`.** A new `AiAutomatorProcessRule` plugin (`id: 'action'`, class `ActionProcessing`) implements `AiAutomatorDirectProcessInterface` and returns `FALSE` from `shouldProcessDirectly()`. This prevents the automator from firing during entity presave. It lives in `src/Plugin/AiAutomatorProcess/ActionProcessing.php`.

**Derivative action plugins.** `RunAutomatorAction` (`id: 'ai_automators_run'`) extends `EntityActionBase` and uses `RunAutomatorActionDeriver` to create one derivative per `ai_automator` config entity whose `worker_type` equals `'action'`. Each derivative is labeled `"AI: [automator label]"` and scoped to the automator's entity type. `RunAutomatorActionDeriver` loads all `ai_automator` configs and filters by `worker_type === 'action'` inside `getDerivativeDefinitions()`.

**Cache invalidation.** `AiAutomator::postSave()` and `AiAutomator::postDelete()` call `\Drupal::service('plugin.manager.action')->clearCachedDefinitions()` so derived action definitions stay in sync with config changes without manual cache clears.

**Access and bundle checking.** `RunAutomatorAction::access()` compares the entity's bundle against the automator's target bundle. A mismatch returns `AccessResult::forbidden()` with the automator config entity as a cacheable dependency. Performing this check in `access()` rather than `execute()` means VBO correctly reports bundle-mismatched entities as skipped rather than silently counting them as processed.

**`edit_mode` and recursion prevention.** `execute()` checks whether the target field already has a value and, if `edit_mode` is off, skips the entity with a status message. After calling `generateResponse()`, the entity is saved inside a `ai_automators_entity_can_save_toggle(FALSE)` / `finally { ai_automators_entity_can_save_toggle(TRUE) }` guard to prevent the automator hook from re-triggering. Errors from `generateResponse()` are caught and logged without aborting the batch.

**Naming decision.** The initial implementation named the worker type `bulk_action`. `fago` requested renaming it to `action` because the core action plugin system is general-purpose and not limited to VBO. All class names, plugin IDs, and file names were updated accordingly.

**Deferred scope.** Processor subforms (allowing per-processor configuration UI) were proposed during review but deferred to a separate issue per `fago`'s recommendation.
