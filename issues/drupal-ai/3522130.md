## Summary

Canvas now supports date range fields in component properties. Previously, components using date range elements would trigger a logic exception because the necessary data transformations were missing. This fix enables site builders to use date range widgets when editing component instances, allowing them to specify both start and end dates that flow correctly through the component rendering system. The implementation ensures dates are properly converted to UTC ISO format for storage and display.

## Impact

No upgrade or configuration changes required.

## Technical details

The Canvas system uses field widgets from core to edit component properties in the sidebar. Each widget type requires a client-side transform to convert form values into the format expected by components. The `daterange_default` widget was missing this transform, causing exceptions when components declared date range properties.

The fix adds a `dateRange` transform that processes both `value` and `end_value` fields from the date range widget. It reuses the existing `dateTime` transform logic to handle both date-only and datetime types. The transform extracts date and optional time values from the form structure, converts them to UTC ISO-8601 strings, and returns an object with `value` and `end_value` properties.

The implementation includes null guards aligned with other transforms, a new `canvas.transform.dateRange` library definition, widget mapping registration in `ReduxIntegratedFieldWidgetsHooks::fieldWidgetInfoAlter()`, and comprehensive test coverage including Cypress end-to-end tests that verify autosave PATCH requests contain correct date values. The fix also removes a test override and TODO comment that were blocking `daterange_default` support.
