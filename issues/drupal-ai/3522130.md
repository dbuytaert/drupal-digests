## Summary

When using date range fields in component forms, Canvas now properly transforms the field widget data between the editing interface and the stored values. Previously, attempting to use a date range field widget would cause a logic exception. This enables site builders to use date range fields in their components without errors, making more field types available for component configuration.

## Impact

No upgrade or configuration changes required.

## Technical details

The issue was that `daterange_default` widget had no client-side transform mapping, causing failures when components used date range fields. The fix adds three pieces: a widget mapping in `ReduxIntegratedFieldWidgetsHooks::fieldWidgetInfoAlter()` pointing to a new `dateRange` transform, the transform itself in `ui/src/utils/transforms.ts`, and a corresponding `canvas.transform.dateRange` library definition.

The `dateRange` transform reuses the existing `dateTime` infrastructure. It extracts `value` and `end_value` from the form values, applies the `dateTime` transform to both using the field's `datetime_type` setting (date or datetime), and returns an object with both ISO-formatted values. The transform includes null/undefined guards aligned with other transforms.

Cypress E2E coverage was added to `prop-types.cy.js` enabling the `datetime_range` module and testing the daterange widget with autosave PATCH synchronization. The test handles both direct parameter and encoded JSON payload formats. Unit tests verify UTC/ISO mapping for both date and datetime range types. The change removes the previous test override and todo comment for [#3523379](https://www.drupal.org/node/3523379).
