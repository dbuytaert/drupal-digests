## Summary

The AI Validations module previously used a plain dropdown for selecting a specific AI provider and model in each validation rule, with no way to fall back to the site's configured default provider. This made it impossible to create portable recipes or shared configurations without hardcoding a provider. Each of the four AI FieldValidationRule plugins now uses the AI module's reusable `ai_provider_configuration` form element, which surfaces a "Default" option that automatically falls back to whichever provider the site administrator has configured as the default for the relevant operation type. Existing saved rule configurations using a hardcoded `provider__model` string continue to work without any data migration.

## Impact

Affects module maintainers.

- **New API:** Two new public abstract base classes: `AiConstraintFieldValidationRuleBase` for rule plugins and `AiConstraintValidatorBase` for constraint validators, intended for custom AI validation rule implementations.

## Technical details

Two new abstract base classes consolidate the shared code that was previously duplicated across each plugin. `AiConstraintFieldValidationRuleBase` (in `src/AiConstraintFieldValidationRuleBase.php`) extends `ConstraintFieldValidationRuleBase` and centralizes `ai.provider` service injection, plus the helpers `buildModelDefaultValue()` (decodes a stored `provider__model` string into the structured array the `ai_provider_configuration` element expects as `#default_value`) and `extractStoredModel()` (normalizes the submitted element value back into a `provider__model` string or empty string). Subclasses implement the abstract `getOperationType()` method and may override `getModelConfigKey()` to preserve legacy config key names: `AiTextConstraintFieldValidationRule` and `AiImageConstraintFieldValidationRule` store under `provider`, while `AiImageClassificationConstraintFieldValidationRule` and `AiObjectDetectionConstraintFieldValidationRule` use `model`.

`AiConstraintValidatorBase` (in `src/AiConstraintValidatorBase.php`) extends `ConstraintValidator` and provides three shared helpers used by all four validators: `resolveProviderOption()` resolves the stored `provider__model` option or falls back to `getDefaultProviderForOperationType()` on the `ai.provider` plugin manager; `loadImageFile()` loads a file entity by ID with consistent violation handling; `runBooleanChat()` encapsulates the XTRUE/XFALSE prompt protocol; and `labelMatchesFinder()` applies the exact/contains/substring matching strategies. A previously missed gap was fixed during review: `AiObjectDetectionConstraintValidator` was still reading `$constraint->model` directly instead of calling `resolveProviderOption()`, meaning it could not fall back to the configured default. It now matches the other three validators.

A new `hook_form_FORM_ID_alter()` in `ai_validations.module` (with submit handler `_ai_validations_sync_rule_to_collection()`) works around a latent `field_validation` bug exposed when the AJAX-driven `ai_provider_configuration` element is present: serialization during the AJAX round-trip causes the form controller's plugin instance to diverge from the rule set's plugin collection, silently dropping edits on final save. The workaround uses `ReflectionProperty` to push the form's modified instance back into the collection before `save()`. A `@todo` links to the upstream issue [#3587461](https://www.drupal.org/node/3587461) for removal once the bug is fixed there.

## Upgrade

Developers who have subclassed any of the four rule plugins directly should update their extends clause to use the new base class (or `ConstraintFieldValidationRuleBase` if the AI-specific helpers are not needed). The stored configuration format (`provider__model` string, or empty string for "Default") is unchanged, so no data migration is required.

```php
// Before (example for a custom plugin extending one of the AI rule plugins):
class MyCustomAiRule extends AiTextConstraintFieldValidationRule { ... }

// After (extend the new shared base directly if you need provider config support):
class MyCustomAiRule extends AiConstraintFieldValidationRuleBase {
  protected function getOperationType(): string {
    return 'chat'; // return the relevant operation type
  }
}
```
