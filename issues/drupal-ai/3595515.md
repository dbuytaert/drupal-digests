## Summary

The `ai_validations` module now ships a fifth field validation rule: **AI text classification constraint**. Site owners can attach it to any text or string field and configure an AI provider model, a classification tag to block (e.g. "spam"), a matching strategy (exact, contains, or case-insensitive contains), a minimum confidence threshold (0–1), a "model not available" behavior (skip or fail), and a custom error message. When a content form is submitted, the validator sends the field value to the chosen AI provider's text-classification operation and adds a violation if the returned label matches the configured tag at or above the confidence threshold. This mirrors the behavior of the existing AI image classification rule.

## Impact

Affects module maintainers.

- **New API:** New `AiTextClassificationConstraint`, `AiTextClassificationConstraintValidator`, and `AiTextClassificationConstraintFieldValidationRule` plugin classes; `resolveProviderOption()` in `AiConstraintValidatorBase` gains an optional `$violateOnMissing` bool parameter (default `TRUE`, fully backward-compatible).

## Technical details

Three new plugin files implement the feature. `AiTextClassificationConstraintFieldValidationRule` (plugin id `ai_text_classification_constraint_rule`) extends `AiConstraintFieldValidationRuleBase` and exposes the config form; it returns `'AiTextClassification'` from `getConstraintName()`. `AiTextClassificationConstraint` is the Symfony `@Constraint` class (id `AiTextClassification`) holding properties `message`, `model`, `tag`, `finder`, `minimum`, and `na`. `AiTextClassificationConstraintValidator` extends `AiConstraintValidatorBase`; its `validate()` method wraps the field value in `TextClassificationInput`, calls `$provider->textClassification($input, $model)`, and iterates the normalized result through the shared `labelMatchesFinder()` / confidence comparison logic inherited from the base class.

During development a bug was found: when no `text_classification` provider was configured and the rule was set to `na: skip`, the validator was still raising a violation because `resolveProviderOption()` in `AiConstraintValidatorBase` unconditionally called `addViolation()` on a missing provider. The fix adds a `$violateOnMissing` boolean parameter (default `TRUE`) to `resolveProviderOption()`; the text classification validator passes `$constraint->na !== 'skip'` so the violation is suppressed when graceful degradation is requested.

Test coverage is provided by a new `AiTextClassificationConstraintValidatorTest` Kernel test. Because `AiProviderPluginManager` is declared `final` and cannot be mocked directly, a helper testing module (`ai_validations_test`) was added, containing `ClassifierTestProvider` (extending `EchoProvider`) whose `textClassification()` output is driven by Drupal state. This lets the Kernel test exercise every branch: match above threshold, match below threshold, label mismatch, contains-finder, empty value short-circuit, missing model with `na: fail`, provider exception with `na: fail`, and provider exception with `na: skip`. A FunctionalJavascript test case was also added to assert the selected model persists correctly through the config form save cycle.
