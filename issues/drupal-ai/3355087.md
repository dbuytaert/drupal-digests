## Summary

The Scheduler module now supports entity types that have no bundles, such as custom entities built without a bundle type or entities from contrib modules like `ai_context`. Previously, Scheduler silently dropped any plugin targeting a no-bundle entity type during discovery, making it impossible to schedule publishing on such entities regardless of the plugin definition. After this change, site owners can enable and configure scheduled publishing and unpublishing on no-bundle entity types via a new per-entity-type settings page at `/admin/config/content/scheduler/{entity_type_id}`. Scheduler 2.3.0 includes this feature.

## Impact

Affects module maintainers and site owners.

- **Config change:** New `scheduler.no_bundle_entity_type_settings.*` config objects store publish/unpublish settings for no-bundle entity types, reusing the `scheduler_third_party_settings` schema.
- **New API:** New `SchedulerManager` methods: `hasBundleType()`, `entityTypeNoBundleConfig()`, `getNoBundleEntityTypeSetting()`, `buildSchedulerFormElements()`; new `SchedulerNoBundleSettingsForm` class and `scheduler.no_bundle_settings_form` route.

## Technical details

The root blocker was in `SchedulerPluginManager::findDefinitions()`, which called `$entityType->getBundleEntityType()` and unset any plugin definition that returned falsy. Removing that guard opens discovery to no-bundle types.

With discovery unblocked, several other subsystems needed to branch on whether a bundle type exists. The new `SchedulerManager::hasBundleType()` method encapsulates that check. `getThirdPartySetting()` now delegates to `getNoBundleEntityTypeSetting()` for no-bundle entities, reading from `scheduler.no_bundle_entity_type_settings.{entity_type_id}` config objects rather than third-party settings on a bundle entity. `getEnabledTypes()` returns `[$entityTypeId]` when the no-bundle config enables scheduling, and the publish/unpublish entity queries skip the `typeFieldName()` bundle condition when `typeFieldName()` returns an empty string. Logging was refactored into the protected `getEntityLogContext()` helper to handle both paths without fatal bundle-entity loads.

The new `SchedulerNoBundleSettingsForm` provides a dedicated admin UI registered at the `scheduler.no_bundle_settings_form` route; settings are saved to `scheduler.no_bundle_entity_type_settings.*` config objects, which reuse the existing `scheduler_third_party_settings` config schema. Form-building logic was extracted from `_scheduler_entity_type_form_alter()` into the public `SchedulerManager::buildSchedulerFormElements()` helper to avoid duplication. A submit-time edge case is also handled: form modes registered only via `hook_entity_form_mode_info_alter()` with no backing `entity_form_mode` config entity ("phantom" modes) would cause a fatal in `EntityDisplayBase::calculateDependencies()`; the submit handler now skips such modes. `SchedulerPluginBase::typeFieldName()` returns an empty string for no-bundle types, and `getTypes()` and `entityTypeFormIds()` return early empty arrays when no bundle entity type exists.

A new `scheduler_no_bundle_test` module ships a `scheduler_test_no_bundle` entity type (`EntityTestNoBundlePublishable`) and its `SchedulerTestNoBundleScheduler` plugin. A `SchedulerNoBundleSupportTest` kernel test covers discovery, settings resolution, and the phantom form-mode edge case. The new `SchedulerNoBundleEntitySetupTrait` and `dataNoBundleEntityTypes()` data provider extend existing functional tests to exercise the no-bundle code path alongside bundled entity types.

## Upgrade

To write a Scheduler plugin for a no-bundle entity type, declare a standard `@SchedulerPlugin` annotation omitting any bundle-specific fields. No `typeFieldName()` override is needed; the base class returns `''` automatically.

```php
/**
 * @SchedulerPlugin(
 *   id = "my_entity_scheduler",
 *   label = @Translation("My entity scheduler plugin"),
 *   description = @Translation("Provides scheduler support for my entity type"),
 *   entityType = "my_entity",
 *   dependency = "my_module"
 * )
 */
class MyEntityScheduler extends SchedulerPluginBase implements ContainerFactoryPluginInterface {}
```

Configure settings via the new config object rather than bundle third-party settings:

```bash
drush config-set scheduler.no_bundle_entity_type_settings.my_entity publish_enable 1
drush config-set scheduler.no_bundle_entity_type_settings.my_entity unpublish_enable 1
```

Or visit `/admin/config/content/scheduler/my_entity` in the admin UI. Existing code that reads settings through `SchedulerManager::getThirdPartySetting()` continues to work without changes; the method now routes no-bundle entities to `getNoBundleEntityTypeSetting()` automatically.
