# #3601376: Allow .drupalaibp.json to append or replace the welcome text

- **Project:** Drupal AI

- **Discussion:** [#3601376](https://git.drupalcode.org/project/one_line_installer/-/work_items/3601376) · 1 contributor · 3 comments

## Summary

Projects shipping a `.drupalaibp.json` config can now customize the "you're all set" summary that the one-line installer prints at the end of a successful build. Two optional keys are available: `append_welcome_text` adds extra paragraphs after the standard summary (site URL, admin login, handy commands), and `replace_welcome_text` replaces the entire summary with custom content. Both keys work for Drupal and non-Drupal builds. When both are set, `replace_welcome_text` wins and `append_welcome_text` is ignored. The feature is documented in `docs/drupalaibp-json.md` including a full token and color reference.

## Impact

Affects module maintainers.

- **New API:** Two new `.drupalaibp.json` keys: `append_welcome_text` and `replace_welcome_text`, with `{TOKEN}` and color-code substitution in both.

## Technical details

Both keys are read by the new `_config_read_welcome_text` function in `src/lib/config/load_config.sh` into `APPEND_WELCOME_TEXT` and `REPLACE_WELCOME_TEXT` bash arrays. Each key accepts either a bare string or an array of strings (paragraphs); the reader handles both shapes.

`print_welcome` in `src/lib/steps/print_welcome.sh` checks `REPLACE_WELCOME_TEXT` first: when the array is non-empty it iterates and prints each paragraph through `_welcome_text_substitute`, then returns before any of the standard summary is built. If only `APPEND_WELCOME_TEXT` is set, the standard summary prints in full and the extra paragraphs are appended at the end via the same helper.

The new helper `src/lib/steps/_welcome_text_substitute.sh` performs `{TOKEN}` substitution. Built-in tokens are `{PROJECT_NAME}`, `{SITE_URL}`, `{DRUPAL_USERNAME}`, `{DRUPAL_PASSWORD}`, and `{DRUPAL_LOGIN_URL}`. Any key from the config's `questions` array is also available by its `env` name. Color tokens (`{BOLD}`, `{DIM}`, `{GREEN}`, `{YELLOW}`, `{RED}`, `{RESET}`) expand to ANSI codes only when stdout is a real TTY (gated by `_welcome_text_colors_enabled` which wraps `[[ -t 1 ]]`); they collapse to nothing in piped or redirected output. An unrecognized token is left untouched rather than silently deleted, surfacing typos in the terminal output. Every paragraph receives an automatic trailing `RESET` when colors are enabled so an unclosed color token cannot bleed into subsequent output. Both new keys are registered in `DRUPALAIBP_KNOWN_KEYS` so the unknown-key warning does not fire on valid configs.

## Upgrade

No changes required for existing configs; both keys are optional and the default behavior (standard summary, no appended or replaced text) is unchanged.

To append project-specific instructions after the standard summary:

```json
{
  "append_welcome_text": [
    "Project docs: https://example.com/handbook",
    "Ask in #project-support if you get stuck."
  ]
}
```

To replace the entire summary with custom content (accepts a bare string or array):

```json
{
  "replace_welcome_text": "{BOLD}{GREEN}Welcome to the ACME demo!{RESET} Run `ddev launch` and log in with admin / admin."
}
```

Tokens from `questions[].env` answers are available in both keys, e.g. `{SITE_FLAVOR}` when a question with `"env": "SITE_FLAVOR"` is defined. Unknown tokens are left verbatim in the output rather than silently removed.
