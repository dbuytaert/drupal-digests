# #3601377: Support conditional (depends_on) custom questions in .drupalaibp.json

- **Status:** Fixed

## Summary

The `one_line_installer` project's `.drupalaibp.json` config format now supports conditional questions. A question entry can include a `depends_on` field that gates the prompt on another question's answer. If the condition is not met the question is silently skipped, its answer is set to an empty string in `.ddev/.env`, and the user is never shown the prompt. This is useful for things like "only ask for a Redis TTL when the user opted into Redis" rather than always asking both questions regardless of context.

## Impact

Affects module maintainers and site owners.

- **Config change:** Any `depends_on` in a config's `questions` block disables `.ddev/.env` answer reuse for the entire block on every re-run.
- **New API:** New `depends_on` field in `.drupalaibp.json` question entries, supporting `equals`, `not_equals`, and `in` condition kinds against an earlier question's answer.

## Technical details

The implementation adds `_config_read_depends_on` to `src/lib/config/load_config.sh`. It runs for every question entry in `_config_read_questions` and populates three new parallel arrays: `CONFIG_QUESTION_DEP_ENV`, `CONFIG_QUESTION_DEP_KIND`, and `CONFIG_QUESTION_DEP_VALUES`. Every question gets exactly one entry in these arrays (empty strings when no `depends_on` is present), keeping them parallel to `CONFIG_QUESTION_KEYS`.

Three condition kinds are supported: `equals` (string equality), `not_equals` (string inequality), and `in` (the dependency's answer is one of an array of values; stored internally as a comma-joined string). The `depends_on.env` value must reference a question that appears **earlier** in the same `questions` array. A forward reference, self-reference, a name that matches no earlier question, or a `depends_on` block that lacks all three condition keys triggers a warning at load time and falls back to asking the question unconditionally rather than silently dropping it.

`ask_config_questions` in `src/lib/shared/prompts/ask_config_questions.sh` evaluates the condition against `ANSWERS[dep_env]` before prompting. A failing condition sets `ANSWERS[key]=""` and skips the prompt entirely, even in unattended runs where the question's own `default` would otherwise be used. The `value` pre-answer field short-circuits before `depends_on` is evaluated, so a pre-answered question is never conditionally skipped.

A boolean flag `CONFIG_HAS_CONDITIONAL_QUESTION` is set by `_config_read_depends_on` as soon as any valid `depends_on` is parsed. `ask_config_questions` uses this flag to disable `.ddev/.env` reuse for the **entire** questions block, not just the conditional questions. The rationale documented in `AGENTS.md` and `docs/drupalaibp-json.md` is that a stale saved value for a dependency question could silently gate a dependent question using an answer the user never confirmed in the current run. The cost is that every question in the block, including unconditional and `password`-type ones, is re-prompted on every re-run of that config.

## Upgrade

Config authors add `depends_on` to any question entry:

```json
{
  "questions": [
    { "env": "USE_REDIS", "label": "Use Redis?", "type": "text", "default": "no" },
    {
      "env": "REDIS_TTL",
      "label": "Redis cache TTL (seconds)",
      "type": "text",
      "default": "3600",
      "depends_on": { "env": "USE_REDIS", "equals": "yes" }
    }
  ]
}
```

Other condition kinds:

```json
"depends_on": { "env": "MODE", "not_equals": "prod" }
"depends_on": { "env": "FLAVOR", "in": ["blog", "shop"] }
```

`depends_on.env` must name a question that appears **earlier** in the array. A forward reference or a missing condition key (`equals`/`not_equals`/`in`) produces a warning and the question is asked unconditionally. Adding any `depends_on` to a config disables `.ddev/.env` answer reuse for the whole `questions` block on every re-run.
