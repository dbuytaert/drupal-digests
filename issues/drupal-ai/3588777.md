## Summary

The `canvas_ai_migrations` migrate-site skill now requires a mandatory Phase 3.7 gate between component QA (Phase 3.5) and the CMS upload (Phase 4). Before any content is pushed to the CMS, one `page-story-composer` agent must run per page in the site inventory, producing a Storybook story under `src/stories/pages/` that is populated with verbatim source content and verified to render with zero console errors. Phase 4 is blocked until the `artifact-checker` in `phase-3.7` mode confirms all pages have valid stories. This closes the gap where page-layout problems only surfaced after a full CMS deploy cycle, costing 10+ minutes per iteration.

## Impact

Affects module maintainers.

## Technical details

Three artifacts were added. First, `.claude/agents/page-story-composer.md` defines a new Sonnet-model sub-agent. It accepts a `page` slug, a `content_path` (`docs/migration/content/<slug>.md`), and a `composition_path` (`docs/migration/page-compositions/<slug>.json`). It writes `src/stories/pages/<PageName>.stories.tsx`, using PascalCase of the slug for the filename. The file's first line must be a `// @migrate-site-provenance: {...}` JSON comment carrying `source`, `page`, `generated_by: "page-story-composer"`, and `generated_at`. After writing the story, the agent checks for console errors via `agent-browser console --json`, retrying up to three times. A per-page gate artifact at `docs/migration/phase-3.7-page-<slug>.json` records the final `status: "PASS"` or `"FAIL"`.

Second, `scripts/gates/phase-3.7-validate.mjs` is the gate script called by `artifact-checker mode: phase-3.7`. It reads `docs/migration/page-inventory.json`, checks that each page slug resolves to a corresponding PascalCase story file in `src/stories/pages/`, parses and validates the provenance comment JSON (all four required fields must be present, `generated_by` must equal `"page-story-composer"`, `source` must resolve to a real file, `page` must match the inventory slug), and exits with code 0 on full pass, 1 on any failure, or 3 if the homepage entry is missing (the homepage is detected via `is_homepage: true` in the inventory). Exit code 3 triggers `overall: escalated` in the checklist JSON and blocks Phase 4 immediately without auto-retry.

Third, `.claude/skills/migrate-site/SKILL.md` was updated: `page-story-composer` is added to the agent table (Sonnet, Phase 3.7), and a new Phase 3.7 section sits between Phase 3.5 and Phase 4 with the full gate criteria and the rationale for using provenance-comment detection rather than canvas-starter boilerplate string scanning. The issue body proposed detecting literal strings like "Welcome to Our Site" or "placehold.co", but the implementation uses the provenance comment instead because real client pages can legitimately contain those strings. Both the SKILL.md and `artifact-checker.md` document this design choice explicitly.

## Upgrade

No breaking change to existing phase outputs. Any workflow that dispatches Phase 4 must now first wait for `artifact-checker mode: phase-3.7` to return `overall: PASS`. The checklist artifact lands at `docs/migration/workflow/03.7-page-stories-checklist.json`. If you have an automation that calls Phase 4 directly after Phase 3.5, insert:

```bash
node scripts/gates/phase-3.7-validate.mjs --root .
# exit 0 → proceed to Phase 4
# exit 1 → re-dispatch page-story-composer for failing pages
# exit 3 → homepage missing, surface to user
```

Each story file must start with:

```ts
// @migrate-site-provenance: {"source":"docs/migration/content/<slug>.md","page":"<slug>","generated_by":"page-story-composer","generated_at":"<ISO-8601 UTC>"}
```

Files without this comment will fail the gate regardless of content.
