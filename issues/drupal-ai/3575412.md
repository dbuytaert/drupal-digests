## Summary

A new `document_loader_automator` submodule adds AI Automator types that extract document content from a source field and store it in a destination field automatically, without requiring any AI provider. Once enabled and configured, a content editor or administrator can map a file, media, link, or URL source field to a text or JSON destination field; the extraction runs on entity save, cron, or batch. Eight destination field types are supported: `string`, `string_long`, `text`, `text_long`, `text_with_summary`, `json`, `json_native`, and `json_native_binary`. For formatted text fields, a text format can be pinned in the automator settings to avoid anonymous-user permission issues during cron runs.

Affects module maintainers.

- **New API:** New `InputResolverTrait` in `Drupal\document_loader\Trait\` and `DocumentLoaderManager::getAvailableOutputFormats()` public method added to the core document_loader module.

## Technical details

All eight plugins extend `DocumentLoaderAutomatorBase`, which itself extends the AI Automators `ExternalBase` class. Using `ExternalBase` suppresses the LLM provider and model configuration fields that standard automators display. The `generate()` method iterates each item in the source field, calls the internal `resolveInputData()` dispatcher to produce a `['file_input' => ...]` or `['url' => ...]` array, then passes that to `DocumentLoaderManager::loadFromData()`. The `storeValues()` method handles plain fields (direct `entity->set()`) and formatted text fields (adds a `format` column via `GeneralHelper::calculateTextFormat()` or an explicit configured format). `DocumentLoaderJson` overrides both `generate()` (forces `output_format` to `'json'`) and `storeValues()` (wraps non-JSON content as `{"content": "..."}`); `DocumentLoaderJsonNative` and `DocumentLoaderJsonNativeBinary` extend it unchanged.

Two additions land in the core `document_loader` module alongside the submodule. `InputResolverTrait` (in `Drupal\document_loader\Trait\`) centralizes four resolution helpers: `resolveMediaToInputData()`, `resolveFileToInputData()`, `resolveStringToInputData()`, and `resolveUriToInputData()`. Both `DocumentLoaderAutomatorBase` and the existing `DocumentLoaderAction` class now use this trait, removing the duplicate inline implementations that `DocumentLoaderAction` previously carried. The trait requires consumers to implement `getEntityTypeManager()`.

`DocumentLoaderManager::getAvailableOutputFormats()` is a new public method that iterates all loader plugin definitions, collects their `output_types` arrays into a sorted, deduplicated list, and returns it. The automator's configuration form calls this to populate the output format select dynamically, so any newly installed loader plugin with a new output type appears automatically.

### Collaboration

The issue was originally closed by the author after they decided a Field Widget Actions integration was a better fit for their use case. `marcus_johansson` reopened it months later with a concrete argument: multiple contrib modules (unstructured, ai_simple_pdf_to_text, convertapi) were each writing their own version of the same Document Loader automator, and known production users including a Fortune 500 deployment were already running similar patterns. Centralizing one canonical implementation in Document Loader would eliminate duplication and simplify the dependency tree for the broader AI module ecosystem. `robloach` reviewed the resulting MR and caught two gaps: the text format selector and `storeValues()` format logic covered only `text_long` and `text_with_summary` but not the base `text` field type, and a config schema question about arbitrary loader option keys. `ahmad khader` addressed both: added `text` to all format-aware checks and confirmed that the `plugin_config` schema's `type: ignore` sequence made the arbitrary keys safe.
