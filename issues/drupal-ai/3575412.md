## Summary

A new `document_loader_automator` submodule integrates Document Loader with the AI Automators module, letting site builders automatically extract content from files, media, links, or URL-bearing text fields and write it into a destination field without needing any AI provider. Extraction runs on entity save, cron, or batch. Eight destination field types are supported: `string`, `string_long`, `text`, `text_long`, `text_with_summary`, `json`, `json_native`, and `json_native_binary`. For formatted text destination fields, a specific text format can be pinned in the automator's advanced settings, which avoids permission failures when cron runs as the anonymous user.

## Impact

Affects module maintainers.

- **New API:** New `document_loader_automator` submodule with eight `AiAutomatorType` plugins; new `InputResolverTrait` in `Drupal\document_loader\Trait\` and `DocumentLoaderManager::getAvailableOutputFormats()` public method added to the core `document_loader` module.

## Technical details

All eight plugins extend `DocumentLoaderAutomatorBase`, which extends the AI Automators `ExternalBase` class. Using `ExternalBase` suppresses the LLM provider and model configuration fields that standard automators display, so no AI module dependency is needed at runtime. The `generate()` method iterates each item on the source field, calls `resolveInputData()` to produce a `['file_input' => ...]` or `['url' => ...]` dispatch array (dispatching on the source field type: `file`/`image`, `entity_reference` for media, `link`, or any of the text variants), then passes that to `DocumentLoaderManager::loadFromData()`. The `storeValues()` method handles plain fields with a direct `entity->set()` and formatted text fields (`text`, `text_long`, `text_with_summary`) by appending a `format` column via either a configured `automator_use_text_format` value or `GeneralHelper::calculateTextFormat()`.

`DocumentLoaderJson` overrides `generate()` to force `output_format` to `'json'` regardless of the configured value, and overrides `storeValues()` to wrap any content that is not already valid JSON as `{"content": "..."}`. `DocumentLoaderJsonNative` and `DocumentLoaderJsonNativeBinary` extend `DocumentLoaderJson` unchanged, targeting the `json_native` and `json_native_binary` field types respectively.

The `extraFormFields()` method populates the output format select by calling `DocumentLoaderManager::getAvailableOutputFormats()`, a new public method that collects `output_types` from all installed loader plugin definitions and returns a sorted, deduplicated list. This means any newly installed loader plugin that supports a new output type appears automatically in the automator configuration form. Source resolution logic is centralised in `InputResolverTrait` (in `Drupal\document_loader\Trait\`), which is shared between `DocumentLoaderAutomatorBase` and the existing `DocumentLoaderAction` class, removing previously duplicated inline implementations.

### Collaboration

The issue was originally closed by `unqunq` after deciding that a Field Widget Actions integration was a better fit. `marcus_johansson` reopened it with a concrete argument: multiple contrib modules (Unstructured, ai_simple_pdf_to_text, convertapi) were each writing their own version of the same Document Loader automator, and known production users including a Fortune 500 deployment were already running similar patterns; centralising one canonical implementation in Document Loader would eliminate duplication and simplify the dependency tree for the broader AI module ecosystem. `robloach` reviewed the resulting MR and identified two gaps: the text format selector and `storeValues()` format logic covered `text_long` and `text_with_summary` but not the base `text` field type, and raised a config schema question about arbitrary loader option keys. `ahmad khader` addressed both: added `text` to all format-aware checks and confirmed that the `plugin_config` schema's `type: ignore` sequence made the arbitrary keys safe.
