## Summary

The AI module now provides a dedicated data transfer object (DTO) to help developers request structured JSON responses from AI providers. Previously, developers passed raw arrays to define the shape of AI responses. This worked but had no validation and different providers required different formats. The new DTO validates schema names, checks that required fields are present, and normalizes the array format for provider compatibility. Developers can still pass plain arrays for backwards compatibility, but the DTO approach is now recommended for cleaner code and better error messages. This lays the groundwork for future features like automated schema generation from PHP classes.

## Impact

Affects module maintainers.

- **Deprecation:** `ChatInput::setChatStrictSchema()` and `getChatStrictSchema()` deprecated in ai:1.3.0, removed in ai:2.0.0; set strict on `StructuredOutputSchema` instead
- **New API:** New `StructuredOutputSchema` DTO class for structured output requests (recommended; plain arrays still accepted for backwards compatibility)

## Technical details

The new `StructuredOutputSchema` class provides a validated wrapper for structured output requests. It accepts four properties: name (validated as lowercase letters, numbers, underscores, hyphens only), description (optional), strict (boolean, default false), and json_schema (the actual JSON schema array, must contain a 'properties' key). The class uses Symfony Validator constraints on public properties to validate inputs at construction or when set via setters.

The DTO can be instantiated directly or created from an array via `fromArray()`. The `toArray()` method returns a normalized array with the key 'schema' for the schema content, matching the documented format most providers expect. For backwards compatibility, both the DTO and `ChatInput::setChatStructuredJsonSchema()` accept either 'schema' or 'json_schema' as the key for the schema content, but normalize and store it as 'schema'.

When a `StructuredOutputSchema` object is passed to `ChatInput::setChatStructuredJsonSchema()`, it is converted to the normalized array format and stored. For backwards compatibility, the strict flag is also copied to the now-deprecated `chatStrictSchema` property. Plain arrays are still accepted and are normalized to the same format. The API Explorer was updated to use the new DTO when processing JSON schema form input.

## Upgrade

**For module developers using structured output:**

The recommended approach is now to use the `StructuredOutputSchema` DTO:

```php
use Drupal\ai\Dto\StructuredOutputSchema;

$schema = new StructuredOutputSchema(
  name: 'weather_response',
  description: 'Structured weather data',
  strict: TRUE,
  json_schema: [
    'properties' => [
      'temperature' => ['type' => 'number'],
      'location' => ['type' => 'string'],
    ],
  ],
);

$input->setChatStructuredJsonSchema($schema);
```

Plain arrays still work but will be normalized (the 'json_schema' key is accepted but stored as 'schema'):

```php
$input->setChatStructuredJsonSchema([
  'name' => 'my_schema',
  'strict' => TRUE,
  'schema' => [
    'properties' => ['field' => ['type' => 'string']],
  ],
]);
```

Replace calls to `setChatStrictSchema()` and `getChatStrictSchema()` with the `strict` property on the schema object or array (these methods are deprecated in ai:1.3.0 and removed in ai:2.0.0).

## Lessons learned

- When building an abstraction layer for multiple providers, accept both structured objects and raw arrays at the API boundary, then normalize to a standard format internally. This allows gradual migration to the structured approach while maintaining backward compatibility with existing array-based implementations. *(via marcus_johansson, abhisekmazumdar)*
- When different providers require different input formats for the same concept, create a translation layer in each provider module rather than exposing provider-specific variations in the shared API. The abstraction should enforce a common interface that works across providers, allowing users to switch providers without code changes. *(via marcus_johansson)*
- When introducing structured DTOs to replace arrays, support multiple key names (like 'schema' and 'json_schema') during the transition period. Normalize to a canonical key internally while accepting legacy formats on input to prevent breaking production code. *(via abhisekmazumdar)*
