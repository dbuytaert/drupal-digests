## Summary

The `canvas_ai_migrations` migrate-site skill now has a fully working Phase 4.5 for Canvas content template creation. The Sprint 6 stub agent referenced an admin UI path (`/admin/structure/canvas-templates`) that returns 404 on every probed Acquia Source instance. This refactor replaces it with the real Canvas REST API at `/canvas/api/v0/config/content_template`, using Drupal session cookies rather than OAuth (no OAuth scope exists for this endpoint). One template per content type now serves all current and future nodes of that type automatically, so content editors can publish new events, blog posts, or other dynamic items without any developer intervention or script run per node.

## Impact

No upgrade or configuration changes required.

## Technical details

The `canvas-content-template-agent.md` was fully rewritten around a two-step API pattern. `POST /canvas/api/v0/config/content_template` with a minimum body of `{entityType, bundle, viewMode}` creates an empty draft entity with `status: false`; a subsequent `PATCH /<id>` with `{elements, status: true}` populates and activates it. On a 409 conflict (template already exists), the agent skips the POST and patches directly. All requests use Drupal session cookies from `agent-browser --profile .canvas-browser/` combined with a CSRF token fetched as plain text from `GET /session/token`. The OAuth client-credentials path cannot be used for this endpoint.

Two Nebula discovery endpoints are now wired in before template construction: `GET /canvas/api/v0/ui/content_template/view_modes/node` confirms valid bundle/view-mode combinations, and `GET /canvas/api/v0/ui/content_template/suggestions/prop-sources/node/<bundle>/<configEntityId>` returns ready-to-paste prop source objects so that the agent does not need to manually construct the Unicode-separated entity-field expression syntax.

Phase 4.5 in `SKILL.md` now has two hard preconditions: the `.canvas-browser/` profile must be authenticated (verified at Phase 0.5, with `phase_4_5_blocked` written to `state.jsonl` on failure), and the migrating Drupal user must have the `administer content templates` permission. On a 403 from the API, the agent writes a structured entry to `journals/blocked.md` and halts. The orchestrator gates progression on `payload.overall: pass` in the new JSON checklist at `docs/migration/workflow/04.5-content-template-checklist.json`. A new `content_template_created` state event is defined in `state-events.md`. A new reference file `canvas-api-contracts.md` documents all four Canvas API groups (`js_component`, `asset_library`, `brand_kit`, `content_template`) with their auth requirements, CRUD shapes, and common error codes, sourced from `canvas openapi.yml` and live empirical probes.

## Upgrade

Operators running Phase 4.5 must take two actions before the migration session:

1. Grant the `administer content templates` Drupal permission to the migrating user role. This permission is not assigned to default admin users and may not appear on older Acquia releases' permissions page. Confirm with the Acquia operator.

2. Ensure the `.canvas-browser/` agent-browser profile is authenticated before the migration session starts (via `npm run verify-browser-session`). Phase 4.5 will halt immediately if the session is stale.

There is no admin UI alternative. `/admin/structure/canvas-templates` and `/admin/canvas/templates` both return 404 on Acquia Source; the API at `/canvas/api/v0/config/content_template` is the only path.
