# #3601381: setup-site: positional config source, multiple named configs in a .drupalaibp/ directory, and git ref (branch/MR/fork) support

## Summary

The `drupalaibp` one-line installer (version 2.0.0) gains three related improvements to how a config source is specified. First, the config URL or path can now be passed as a plain positional argument (`drupalaibp https://git.drupalcode.org/project/foo` or `drupalaibp ./team-config`) instead of requiring the `--config-url` or `--config-path` flag; the installer auto-detects whether it is a URL, local directory, or saved global config name. Second, a single source can now carry multiple installer configs as `*.json` files inside a `.drupalaibp/` directory, with automatic selection when only one exists, an interactive menu when several do, and a `--config <name>` flag for non-interactive disambiguation. Third, URL sources can target a specific branch, tag, or GitLab MR via an `@<ref>` suffix (`foo@1.2.x`, `foo@!12`), or via the new `--ref` flag. Breaking change (2.x): the root `.drupalaibp.json` file is no longer recognized anywhere. All configs must now live as `*.json` files directly under a `.drupalaibp/` directory.

## Impact

Affects module maintainers and site owners.

- **Config change:** Breaking (2.x): root `.drupalaibp.json` removed; configs must now live as `*.json` files directly under `.drupalaibp/`; `CONFIG_BASEDIR` now resolves to `<source>/.drupalaibp/` so relative hook and `copy_paths` paths resolve there
- **New API:** New positional `source` argument on `setup-site`, `--config <name>` and `--ref <ref>` flags, and `@<ref>` suffix syntax on URL sources and the `pull` command's `repo` argument

## Technical details

The `setup-site` command's new optional positional `source` argument is classified by `resolve_config_source` (`src/lib/config/resolve_config_source.sh`). Resolution order: an existing local directory routes to `--config-path`; a git URL (by `://` scheme, SCP syntax `user@host:path`, `.git` suffix, or `owner/repo` shorthand expanded to `https://git.drupalcode.org/<owner/repo>`) routes to `--config-url`; otherwise the value is looked up as a saved global config name under `~/.drupalaibp/configs/`. Combining the positional form with an explicit `--config-*` flag is a fatal error.

The root `.drupalaibp.json` convention is replaced by a `.drupalaibp/` directory holding one or more `*.json` config files. `select_config` (`src/lib/config/select_config.sh`) picks which config to load: with exactly one `*.json` it proceeds automatically; with multiple it requires `--config <name>` or prompts interactively via `select_one`; under `ASK_UNATTENDED` it dies naming the available choices. The selector deliberately ignores the `config/` YAML import subdir and hook scripts (`.sh` files), only matching `*.json` files directly under `.drupalaibp/`. `CONFIG_BASEDIR` now points to `<source>/.drupalaibp/` (the dirname of the chosen config file) rather than the source root, so relative paths in `hooks[].script`, `copy_paths`, and `db_import_path` resolve inside `.drupalaibp/`; a `"../"` prefix reaches the repo root.

Git ref support is added via `parse_git_ref` (`src/lib/config/parse_git_ref.sh`), which splits an `@<ref>` suffix off a source string conservatively: plain SCP URLs (`user@host:path` with no second `@`) are left whole; for scheme URLs the split fires only when the remaining source still contains a path after `://`. The extracted ref is consumed by `_git_clone_ref` (`src/lib/config/_git_clone_ref.sh`): a plain branch or tag name uses `git clone --depth 1 -b <ref>`; a `refs/…` refspec or `!<iid>` (expanded to `refs/merge-requests/<iid>/head` for GitLab MRs) does a full clone followed by a shallow `fetch` and `checkout FETCH_HEAD`. The `--ref` flag is an alternative to the `@<ref>` suffix and is accepted on both the positional `source` form and `--config-url`. The `pull` command's `repo` argument also accepts `@<ref>`.

`_migrate_legacy_configs` (`src/lib/config/_migrate_legacy_configs.sh`) performs a one-time best-effort migration of pre-2.x flat saved configs (`~/.drupalaibp/configs/<name>.json`) into the nested layout (`configs/<name>/.drupalaibp/<name>.json`) on load. It is non-fatal: a warning is emitted and the old file is left in place on any failure.

## Upgrade

```bash
# Migrate a root .drupalaibp.json to the 2.x layout:
mkdir -p .drupalaibp
mv .drupalaibp.json .drupalaibp/config.json
# A lone *.json in .drupalaibp/ is picked automatically — no rename required.

# Relative hook and copy_paths paths now resolve inside .drupalaibp/, not the repo root.
# If hooks[].script was "agent-extras.sh" (resolved from root), move the file:
mv agent-extras.sh .drupalaibp/agent-extras.sh
# Or keep it at the root and update the path to "../agent-extras.sh".

# Before (pre-2.x):
drupalaibp --config-url https://git.drupalcode.org/project/foo

# After (2.x) — positional form:
drupalaibp https://git.drupalcode.org/project/foo

# With a branch ref:
drupalaibp https://git.drupalcode.org/project/foo@1.2.x

# With a GitLab MR ref:
drupalaibp https://git.drupalcode.org/project/foo@!12

# With multiple configs in .drupalaibp/ — pick by name:
drupalaibp https://git.drupalcode.org/project/foo --config config2 --yolo
```
