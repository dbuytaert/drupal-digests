# #3601379: Config-url clobbers the scaffold composer.json

- **Project:** Drupal AI
- **Type:** Major bug report
- **Status:** Fixed
- **Discussion:** [#3601379](https://git.drupalcode.org/project/one_line_installer/-/work_items/3601379) · 1 contributor · 2 comments

## Summary

Two bugs in `--config-url` provisioning are fixed in this release. First, when building a fresh Drupal scaffold, the config repo's own `composer.json` was silently overwriting the scaffold's `composer.json`, wiping the `allow-plugins` and `minimum-stability` settings that the installer had just written. This caused subsequent `composer require` calls to fail or refuse to resolve `@dev`/`@beta` packages. The overlay now preserves the scaffold's `composer.json` and `composer.lock`. Second, the `recipes` key in `.drupalaibp.json` has changed meaning: it now only applies recipes via `drush recipe` and never fetches them. Contrib recipes must be listed in both `composer_projects` (to fetch) and `recipes` (to apply). A new `composer_with_all_dependencies` config key passes `--with-all-dependencies` to `composer require` for packages that need their full dependency tree updated.

## Impact

No upgrade or configuration changes required.

## Technical details

In `_copy_config_url_repo` (`src/lib/config/load_config_url.sh`), fresh-scaffold Drupal mode (i.e., `IS_DRUPAL=true` and no `DB_IMPORT`) now stashes `composer.json` and `composer.lock` before running `cp -R "${CONFIG_URL_REPO_DIR}/." .` and then restores them afterward. Existing-codebase/non-Drupal mode is unaffected because there the cloned repo is the site root.

In `step_provision_from_config` (`src/lib/steps/provision_from_config.sh`), `COMPOSER_PROJECTS` and `RECIPES` are now strictly separated: `composer_projects` calls `composer require --no-interaction` (optionally with `--with-all-dependencies` when `COMPOSER_WITH_ALL_DEPS=true`) and never applies anything; `recipes` calls `ddev drush recipe` and never fetches. A `drupal/<name>` recipe entry is resolved to `../recipes/<name>` (the path Composer installs it to), while any other entry (e.g., `core/recipes/standard`) is passed verbatim to drush as a locator. The new `composer_with_all_dependencies` key is read into `COMPOSER_WITH_ALL_DEPS` in `load_config.sh` and defaults to `false`.

## Upgrade

If your `.drupalaibp.json` lists contrib recipes (e.g., `drupal/my_recipe`) under `recipes`, you must now also list them under `composer_projects` so they are fetched first:

```json
{
  "composer_projects": ["drupal/my_recipe"],
  "recipes": ["drupal/my_recipe"]
}
```

Core or already-present recipes (e.g., `core/recipes/standard`) still only need to appear under `recipes`.

If any recipe's dependencies conflict with existing constraints, add the `composer_with_all_dependencies` flag:

```json
{
  "composer_with_all_dependencies": true,
  "composer_projects": ["drupal/my_recipe"],
  "recipes": ["drupal/my_recipe"]
}
```
