# #3528673: Create Document Loader Normalization Layer

## Summary

The AI module lacked a standard way to feed document content (PDFs, spreadsheets, web pages, and so on) into AI pipelines. Any agent or recipe that needed to read a document was forced to hardcode a specific service or library. This issue called for an abstraction layer so site owners can choose their own document-loading backend — just as they choose their LLM provider — without breaking existing recipes or agents.

## Impact

Affects module maintainers.

- **New API:** Document loading abstraction is provided by the standalone `document_loader` module (https://www.drupal.org/project/document_loader) rather than AI Core; sites needing PDF/spreadsheet/web-scraping input for AI pipelines should install and configure that module.

## Technical details

The AI module already normalizes LLM provider calls through a plugin system, but had no equivalent layer for document ingestion. The issue proposed adding a `DocumentLoader` plugin type with a dedicated plugin manager, a `DocumentLoaderType` attribute (carrying `label`, `description`, `document_loader_type`, and `output_type` metadata), and a set of typed interfaces: `DocumentLoaderInput`, `DocumentLoaderOutput`, and `DocumentLoaderType`. Output formats to support were text, markdown, CSV, and HTML. A settings page would let administrators choose a default `DocumentLoader` plugin per document-loader type, mirroring how LLM defaults are configured. The plugin manager would expose a helper to retrieve the default loader for any given type.

Rather than implementing this directly inside the AI module, contributor `unqunq` proposed and built a standalone `document_loader` module (https://www.drupal.org/project/document_loader) to keep the surface area self-contained and independently extensible. `robloach` followed with a `document_loader_pdfparser` submodule backed by `smalot/pdfparser`. The issue was closed by `marcus_johansson` once the standalone module had enough momentum to continue on its own.
