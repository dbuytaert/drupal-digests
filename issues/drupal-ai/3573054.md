## Summary

The AI File to Text module now provides a unified document loader plugin that converts files to text, HTML, or Markdown. Out of the box it supports Word documents, spreadsheets, PDFs, plain text, and Markdown files. Other modules can add support for new file types by registering extractor services without modifying this module. All automators, agent function calls, and the document loader plugin flow through the same extensible architecture. The system automatically adapts when new file types are added, making recipes and agents portable across different document loader implementations.

## Impact

Affects module maintainers.

- **New API:** New `document_loader:file` plugin for converting documents, spreadsheets, and text files to text, HTML, or Markdown. New `ai_file_to_text.extractor` service tag for registering custom file type handlers.

## Technical details

The change introduces `FileDocumentLoader`, a `document_loader` plugin (ID `document_loader:file`) that serves as the single entry point for all file conversions. It delegates to `FileExtractorManager`, which auto-discovers all services tagged `ai_file_to_text.extractor`. Each extractor implements `ExtractorInterface` and declares its `document_loader_type`, supported file extensions, and output formats.

Seven built-in extractors handle ten file extensions across five document types: Word (docx/doc/odt), spreadsheets (xlsx/xls/ods/csv via PhpSpreadsheet), PDF (smalot/pdfparser and optional Poppler), plain text, and Markdown.

The manager collects document loader types and output formats at construction. A `hook_document_loader_info_alter()` implementation patches the cached plugin definition so type-based discovery works for dynamically registered types.

When extractors have different output capabilities (e.g. PDF supports HTML but PowerPoint only text), the alter hook splits the plugin into multiple definitions—one per capability group—ensuring accurate filtering. The manager validates per-extension output format support and falls back to text when an unsupported format is requested.

`SpreadsheetExtractor` replaces separate CSV and ODS extractors, using PhpSpreadsheet for consistent rich-text cell styling, merged cells, and automatic heading detection. AI automators and the `file_to_text` function call now use the document loader plugin instead of calling extractors directly.

## Upgrade

Third-party modules can add support for new file types by creating an extractor class and registering it as a tagged service:

```php
// src/Extractor/PptxExtractor.php
namespace Drupal\your_module\Extractor;

use Drupal\ai_file_to_text\Extractor\ExtractorInterface;

class PptxExtractor implements ExtractorInterface {
  public function isAvailable(): bool {
    return TRUE;
  }

  public function getDocumentLoaderType(): string {
    return 'document_loader_type:presentation';
  }

  public function getSupportedOutputFormats(): array {
    return ['text'];
  }

  public function getSupportedExtensions(): array {
    return ['pptx'];
  }

  public function extractText(string $file_path, string $extension = ''): string {
    // Extraction logic here.
  }

  public function extractHtml(string $file_path, string $extension = '', array $options = []): string {
    return '';
  }
}
```

```yaml
# your_module.services.yml
services:
  your_module.extractor.pptx:
    class: Drupal\your_module\Extractor\PptxExtractor
    tags:
      - { name: ai_file_to_text.extractor }
```

The `FileExtractorManager` automatically discovers the new extractor and registers its type with the document loader plugin system. No changes to `ai_file_to_text` are needed.

Modules using the document loader plugin can now call `getLoaderByType('document_loader_type:presentation')` or process `.pptx` files via the unified `document_loader:file` plugin.

The issue opened with a clear goal: create a plugin architecture for document loading to abstract file conversion across modules. After initial implementation by unqunq with three separate plugins, ahmad khader proposed a unified architecture using a single `FileDocumentLoader` plugin backed by tagged extractor services. This approach better supported extensibility and reduced duplication. The team iterated on capability grouping to handle extractors with different output format support, ensuring accurate plugin discovery without false positives. robloach provided UI feedback on naming and validation. The final solution unified automators, function calls, and the document loader plugin under one extractor manager, with comprehensive unit tests covering dynamic type registration and mixed-capability scenarios.

## Lessons learned

- When building plugins that wrap multiple similar operations, use a single plugin with dynamic dispatch instead of creating separate plugins per operation. Let tagged services handle the variations, and use hook_alter to register their capabilities in the plugin definition. This allows third-party modules to extend functionality by adding services without modifying the plugin system itself. *(via ahmad khader)*
- When designing plugin architectures, place the extension point at the lowest useful level. Let higher-level components (plugins) delegate to lower-level extensible services rather than requiring new plugins for each variation. This creates a clearer separation: plugins define integration points with the framework, while services implement the domain logic that varies. *(via ahmad khader)*
