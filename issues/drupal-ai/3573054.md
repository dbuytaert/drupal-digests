## Summary

The AI File to Text module now integrates with the Document Loader module, providing a unified plugin for extracting content from documents, spreadsheets, and other files. This makes it easier to build automators and agents that handle document processing without caring which specific parser is used. Other modules can extend support to new file types by registering tagged extractor services, eliminating the need to modify AI File to Text directly. Built-in support covers PDF, Word documents, spreadsheets, plain text, and Markdown files with three output formats: text, HTML, and Markdown.

## Technical details

The module now provides a `document_loader:file` plugin that delegates to a `FileExtractorManager`. The manager discovers all services tagged with `ai_file_to_text.extractor`, builds an extension-to-extractor map, and dispatches extraction requests. Each extractor implements `ExtractorInterface` and declares its `document_loader_type`, supported extensions, and output formats (text, html, or both).

Dynamic type registration solves discovery: the manager collects all unique `document_loader_type` strings from registered extractors and exposes them via `getDocumentLoaderTypes()`. A `hook_document_loader_info_alter()` implementation patches the cached plugin definition so `getLoaderByType()` can find the plugin for any type—including types added by third-party modules.

The manager tracks output formats per extension and validates requests. If a caller requests HTML or Markdown but the extractor only supports text, extraction falls back to plain text. The output metadata reports both `requested_format` and `output_format`.

Capability grouping prevents false positives. When extractors have different output capabilities (e.g., PDF supports HTML but PowerPoint only supports text), the alter hook splits the single plugin definition into multiple entries—one per capability group. Each variant has accurate `output_types` that match its `document_loader_types`, so `getLoaderByType('document_loader_type:presentation', 'html')` correctly returns NULL instead of a false match.

The refactor consolidated `CsvExtractor` and `OdsExtractor` into a single `SpreadsheetExtractor` using phpoffice/phpspreadsheet for all tabular formats (XLSX, XLS, ODS, CSV). This provides consistent handling of merged cells, rich text, and cell styling across all spreadsheet types. The AI automator base class and function call plugin now delegate to the document loader plugin instead of calling extractors directly.

Comprehensive unit tests verify dynamic type registration, third-party extractor integration, unavailable extractor exclusion, output format aggregation, per-extension format support, fallback behavior, and capability group splitting.

## Upgrade

Third-party modules can add support for new file types by creating a class implementing `ExtractorInterface` and registering it as a tagged service:

```php
// src/Extractor/PptxExtractor.php
namespace Drupal\your_module\Extractor;

use Drupal\ai_file_to_text\Extractor\ExtractorInterface;

class PptxExtractor implements ExtractorInterface {
  public function isAvailable(): bool {
    return TRUE;
  }

  public function getDocumentLoaderType(): string {
    return 'document_loader_type:presentation';
  }

  public function getSupportedOutputFormats(): array {
    return ['text'];
  }

  public function getSupportedExtensions(): array {
    return ['pptx'];
  }

  public function extractText(string $file_path, string $extension = ''): string {
    // Your extraction logic.
  }

  public function extractHtml(string $file_path, string $extension = '', array $options = []): string {
    return '';
  }
}
```

```yaml
# your_module.services.yml
services:
  your_module.extractor.pptx:
    class: Drupal\your_module\Extractor\PptxExtractor
    tags:
      - { name: ai_file_to_text.extractor }
```

The `FileExtractorManager` automatically discovers the new extractor, and the `document_loader_type:presentation` becomes available via `getLoaderByType()` without any changes to AI File to Text.

## Lessons learned

- When building a plugin system for file processing, use a dispatcher pattern with a single unified plugin that delegates to format-specific handlers tagged as services. This allows third-party modules to extend support for new file types without modifying the core plugin by simply registering a tagged service implementing the handler interface. *(via ahmad khader)*
- When plugin definitions depend on dynamically discovered capabilities (like supported file types from multiple providers), collect those capabilities at manager construction time and use a hook to alter the cached plugin definitions. This enables a single plugin to appear under multiple type-based lookups without hardcoding every variant. *(via ahmad khader)*
