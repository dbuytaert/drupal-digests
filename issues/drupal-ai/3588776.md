## Summary

In the `canvas_ai_migrations` AI-agent workflow, the `storybook-qa` agent was filing visual-mismatch issues without always capturing a built screenshot at filing time. This left the `component-fixer` agent working blind: it had no before-fix baseline to compare against and could report a fix as successful without any visual evidence. Three Sprint 3 commits close this gap by making screenshot evidence mandatory at every stage of the Phase 3.5 QA loop. `storybook-qa` now refuses to write an `issue.md` file unless the required source/built PNG pairs already exist on disk. `component-fixer` now verifies those PNGs are present and non-zero before starting any fix, and aborts with a `BLOCKED` status when they are missing. A new Phase 3.5 gate runs `validate-screenshots.mjs` across all open issue directories and fails the gate if any required PNG is absent or zero-byte.

## Impact

No upgrade or configuration changes required.

## Technical details

The changes touch three agent instruction files and add a new validation script plus tests.

In `.claude/agents/storybook-qa.md`, a new "Issue Filing Protocol (mandatory)" block (lines 98–130) describes a five-step pre-filing sequence: determine the Viewport (`desktop`, `mobile`, or `both`), capture source and built PNGs at the matching element selector via `agent-browser screenshot`, capture computed styles as JSON, then verify every PNG is non-zero before writing `issue.md`. Failure at any capture step causes the agent to emit a `issue_filing_skipped` event to `state.jsonl` and abort filing that issue rather than writing an incomplete record. The required filenames are viewport-driven: `source-desktop.png` / `built-desktop.png` for desktop, `source-mobile.png` / `built-mobile.png` for mobile, both pairs for `both`.

In `.claude/agents/component-fixer.md`, a "Pre-fix screenshot guard (P1-6)" block (lines 69–75) requires the fixer to check the PNG siblings for every path in `issue_paths` before any fix attempt. If any required PNG is missing or zero-byte, the agent emits an `action: fix_blocked` state event to `state.jsonl` and aborts with `BLOCKED: issue <NN-desc> missing screenshot evidence — re-dispatch storybook-qa to re-capture`. The implementation deliberately chose a hard abort rather than the ticket's proposed silent re-dispatch, because a missing baseline indicates Phase 3.5 did not complete properly; silent recovery would mask the upstream failure.

In `.claude/agents/artifact-checker.md`, the Phase 3.5 gate now includes an issue-dir screenshot check. It runs `node scripts/issue-files/validate-screenshots.mjs docs/migration/issues/` against every issue directory whose `issue.md` has `Status: open` or `Status: fixing`. The script (`scripts/issue-files/validate-screenshots.mjs`) reads the `Viewport` field from each `issue.md`, derives the required PNG filenames, and checks existence and non-zero size for each. A non-zero exit fails the gate and pastes the failure list into the artifact summary. Vitest tests for the script live in `tests/issue-files/required-screenshots.test.mjs` and cover pass, missing file, zero-byte, mobile-only viewport, and resolved-issue-skip cases.

## Upgrade

If you are running the `canvas_ai_migrations` Phase 3.5 QA loop, expect the following behavior changes:

1. **`storybook-qa`** will refuse to write `issue.md` when any required PNG is missing. Ensure the agent has access to a running Storybook server and the production site so it can complete captures during filing.

2. **`component-fixer`** will abort with `BLOCKED` (rather than proceeding) when an issue directory is missing its screenshot evidence. Re-dispatch `storybook-qa` to re-capture, then re-invoke `component-fixer`.

3. The Phase 3.5 gate now runs `validate-screenshots.mjs` automatically. To run it manually against an existing issues directory:

```bash
node scripts/issue-files/validate-screenshots.mjs docs/migration/issues/
```

A non-zero exit prints the failure list. Zero exit prints `PASS: all open issues have required screenshots`.

4. Issue file screenshot fields use viewport-driven filename pairs (`source-desktop.png` / `built-desktop.png`, etc.) rather than the `source_screenshot:` / `built_screenshot:` YAML keys described in the ticket's spec. The gate logic keys off the `**Viewport:**` field in `issue.md`, not YAML keys.
