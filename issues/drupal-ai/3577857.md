## Summary

Document loader plugins can now declare their own runtime options via a schema method, and those options are automatically surfaced in every integration point: the Document Loader Explorer form, the Field Widget Action plugin, the AI Automator, and the AI tool deriver. Site builders no longer need to create separate input classes or hardcode loader-specific settings. Options marked `hide_from_llm_by_default` are pre-configured to be hidden from the LLM and locked to their schema defaults on the AI Agent configuration form.

## Impact

Affects module maintainers.

- **New API:** New `getLoaderOptionsSchema()` method on `DocumentLoaderInterface` / `DocumentLoaderBase` allows loader plugins to declare typed runtime options that are automatically rendered in all integration UIs and AI tool schemas.

## Technical details

The core addition is `getLoaderOptionsSchema()` on `DocumentLoaderInterface`, with a default empty-array implementation in `DocumentLoaderBase`. Loader plugins override this to return an associative array of option definitions. Each definition carries `data_type` (`string`, `boolean`, `integer`, `list`), `label`, `description`, `required`, `default_value`, optional `options` (key→label map for select/checkboxes), `multiple`, and the new `hide_from_llm_by_default` flag.

To prevent key collisions when multiple loaders declare options with the same name, all options are namespaced as `{loader_plugin_id}__{option_key}` wherever they cross plugin boundaries. `DocumentLoaderBase::extractOwnOptions()` strips this prefix at runtime, routing only the current loader's options to it. Colons in derivative plugin IDs (e.g. `document_loader:file`) are encoded as `_C_` in form keys and tool schemas to avoid Drupal context definition parsing issues.

`DocumentLoaderTypeFactory::createInput()` gains an `$extra_options` parameter; the factory merges these into the input's `options` array after type-detection, keeping schema-declared fields separate from loader options. Each integration point (FWA's `DocumentLoaderAction`, `DocumentLoaderAutomatorBase`, `DocumentLoaderExplorerForm`, `DocumentLoaderToolDeriver`) calls `collectLoaderOptionsSchemas()` or `getLoaderOptionsSchema()` to build its form section and then extracts submitted values using the `dl__` (automator/FWA) or direct-key (explorer) conventions. The new `DocumentLoaderToolHooks` class implements `hook_form_alter` to pre-select "Force value" and "Hide property" on the AI Agent, Modeler API Wrapper, and AI API Explorer forms for any property with `hide_from_llm_by_default` set.

## Upgrade

Loader plugins that want to expose runtime options implement the new method:

```php
public function getLoaderOptionsSchema(): array {
  return [
    'strategy' => [
      'data_type' => 'string',
      'label' => new TranslatableMarkup('Strategy'),
      'description' => new TranslatableMarkup('Processing strategy.'),
      'required' => FALSE,
      'default_value' => 'auto',
      'options' => ['auto' => 'Auto', 'fast' => 'Fast', 'hi_res' => 'Hi-Res'],
      'hide_from_llm_by_default' => FALSE,
    ],
  ];
}
```

Inside your loader's `loadDocument()` (or equivalent), call the new `extractOwnOptions()` helper to retrieve only your loader's options from the input:

```php
$opts = $this->extractOwnOptions($input->getOptions());
$strategy = $opts['strategy'] ?? 'auto';
```

No changes are needed if you do not need to declare loader-specific options.
