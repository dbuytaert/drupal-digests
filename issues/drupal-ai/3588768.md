## Summary

Phase 6 of the `canvas_ai_migrations` migrate-site skill previously dispatched two LLM-driven subagents (site-configurator, menu-builder) to fill Drupal admin forms with data already present in migration artifacts. Because every field value is known at run time (site name from `plan.md`, menu items from `menu-map.json`, logo from `media-map.md`), the LLM added inference cost and hallucination risk without contributing judgment. The recorded cost was roughly 88 minutes wall-clock for work that scripted automation completes in under five minutes. This change replaces those agents with two new deterministic bash scripts invoked directly from Phase 6, demoting the LLM agents to emergency fallbacks.

## Impact

No upgrade or configuration changes required.

## Technical details

Three new scripts are added to the `scripts/` directory. `acquia-browser-helpers.sh` is a sourced library providing `acquia_session_warmup` (opens the persistent `.canvas-browser/` profile), `acquia_form_post` (fills form fields by CSS selector and submits), and `acquia_classify_redirect` (categorises the post-submit URL as `success`, `sso-waf`, or `unknown`). The success pattern matches `/admin/structure/menu/item/<id>/edit`; the WAF pattern matches `id.acquia.com`, `source.acquia.com`, and the simplesamlphp ACS path.

`configure-site.sh` extracts site name, slogan, and front page from `docs/migration/plan.md` with grep/sed, then submits once to `/admin/config/system/site-settings`. On failure it retries once; if both attempts fail the event is appended to `docs/migration/blocked.md` and the script exits 0. `create-menus.sh` reads `docs/migration/menu-map.json`, checks each item against the live `GET /api/menu_items/<menu>` endpoint for idempotency (matching on label+url tuple), then submits new items to `/admin/structure/menu/manage/<menu>/add` with one retry on non-success. Failed items are also appended to `blocked.md`. Both scripts support `--dry-run` and always exit 0.

CSS selectors used by both scripts are externalised to `references/drupal-admin-selectors.json`, so updating them when the admin theme changes requires editing only that one file. A companion `references/menu-map-schema.md` documents the shape of `menu-map.json` that Phase 1's site-analyzer must emit; it notes that `parent` and `weight` fields in the schema are parsed but not yet wired into the script (silently dropped, manual intervention required for nesting and ordering).

`SKILL.md` Phase 6 is rewritten to invoke the two scripts in sequence and explicitly marks the former LLM agents as fallback-only. The second commit adds vitest tests in `tests/scripts/` covering the classify-redirect logic exhaustively and asserting that the scripts exist and are executable and that Phase 6 references the bash invocations. It also replaces a shell-interpolation call in the existing classify test (`execSync` with a string containing the URL) with `execFileSync` passing the URL as a separate argument, closing a shell injection vector in the test harness.

## Upgrade

No external callers need updating. If you have extended or overridden Phase 6 orchestration logic (e.g. by forking `SKILL.md` or wrapping the agent dispatch), replace those dispatch blocks with direct bash invocations:

```bash
bash scripts/configure-site.sh
bash scripts/create-menus.sh
```

If you relied on the `site-configurator` or `menu-builder` agents as the primary Phase 6 path, they still exist but are now fallback-only; invoking them directly continues to work for emergency or manual-override scenarios.

The CSS selectors for admin forms are now read at runtime from `.claude/skills/migrate-site/references/drupal-admin-selectors.json`. If the Acquia admin theme changes, update that file rather than the scripts themselves.
