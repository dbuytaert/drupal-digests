## Summary

Phase 6 of the `migrate-site` skill previously dispatched two LLM-driven subagents (`menu-builder` and `site-configurator`) to fill Drupal admin forms. Because all required data (site name, slogan, front page, logo, menu structure) is already present in migration artifacts, no LLM inference was needed. Recorded cost was roughly 88 minutes wall-clock for a task that scripting can finish in under 5 minutes. Phase 6 now invokes two deterministic bash scripts instead: `scripts/configure-site.sh` and `scripts/create-menus.sh`. The LLM agents are retained only as documented emergency fallbacks. Both scripts exit with status 0 always; any item that cannot be processed is appended to `docs/migration/blocked.md` for manual follow-up.

## Impact

No upgrade or configuration changes required.

## Technical details

The `SKILL.md` Phase 6 section was rewritten to invoke `bash scripts/configure-site.sh` then `bash scripts/create-menus.sh` in sequence rather than spawning LLM agent tool calls.

`scripts/configure-site.sh` reads `Site name:`, `Slogan:`, and `Front page:` headings from `docs/migration/plan.md` using `grep`/`sed`, then calls shared helpers to submit the Acquia site-settings form at `/admin/config/system/site-settings`. CSS selectors are read at runtime from a new reference file `.claude/skills/migrate-site/references/drupal-admin-selectors.json`, so a change to the Drupal admin theme requires updating only that one file.

`scripts/create-menus.sh` reads `docs/migration/menu-map.json` (a structured JSON file emitted by Phase 1's site-analyzer; schema documented in `references/menu-map-schema.md`). For each menu it fetches existing items via `GET /api/menu_items/<menu>` and skips any `(label, url)` pair already present, making the script idempotent. New items are submitted via the Drupal admin form at `/admin/structure/menu/manage/<menu>/add`.

Both scripts source `scripts/acquia-browser-helpers.sh`, which provides three functions: `acquia_session_warmup` (opens the persistent `.canvas-browser/` profile and confirms authentication), `acquia_form_post` (opens a URL, fills CSS-selector-addressed fields, clicks submit, and returns the final URL), and `acquia_classify_redirect` (categorizes the post-submit URL as `success`, `sso-waf`, or `unknown`). The `success` pattern matches `/admin/structure/menu/item/<id>/edit`; SSO/WAF matches `id.acquia.com`, `source.acquia.com`, or a `simplesamlphp` path. Each script retries once on non-success before writing a structured entry to `blocked.md` and continuing.

Both scripts support a `--dry-run` flag. A shell-injection fix in the existing test helper swapped `execSync` for `execFileSync` when invoking `acquia_classify_redirect` from the Vitest suite. Two new test files were added under `tests/scripts/`: prose-sanity assertions (scripts exist and are executable, `SKILL.md` Phase 6 references them, `menu-builder`/`site-configurator` are demoted to fallback) and unit tests for the redirect classifier covering all three outcome categories.

## Upgrade

No upgrade steps are needed for existing Drupal installations. If you maintain a fork of `canvas_ai_migrations` and customized the Phase 6 agent-dispatch logic in `SKILL.md`, reconcile against the new script-based invocation:

```
# Phase 6 now calls scripts, not agents:
bash scripts/configure-site.sh
bash scripts/create-menus.sh
```

Ensure Phase 1 (`site-analyzer`) emits `docs/migration/menu-map.json` in the shape documented in `references/menu-map-schema.md` (keyed by menu machine name, items with `label`, `url`, optional `parent` and `weight`). If your admin theme changes Drupal form selectors, update `.claude/skills/migrate-site/references/drupal-admin-selectors.json` rather than editing the scripts directly.
