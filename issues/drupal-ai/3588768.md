## Summary

In the `canvas_ai_migrations` module, Phase 6 of the migrate-site skill previously dispatched two LLM agents (menu-builder and site-configurator) to fill out Drupal admin forms. Because the inputs (site name, menu structure, logo) come entirely from known migration artifacts with no ambiguity, the LLM added inference cost, hallucination risk, and unpredictable session length. Measured wall-clock for a real Phase 6 run was 88 minutes; the same work by a deterministic script takes under 5 minutes. This change replaces the default agent dispatch with two bash scripts, and demotes the LLM agents to a documented fallback for manual override only.

## Impact

No upgrade or configuration changes required.

## Technical details

The core change is in `.claude/skills/migrate-site/SKILL.md`: the Phase 6 block that previously instructed the orchestrator to spawn `site-configurator` and `menu-builder` agents in parallel now instructs it to run `bash scripts/configure-site.sh` followed by `bash scripts/create-menus.sh` in sequence. Both scripts exit 0 regardless of outcome; failures are appended to `docs/migration/blocked.md` in a structured format with manual recovery instructions.

Three new script files are introduced. `scripts/acquia-browser-helpers.sh` is a sourced library providing `acquia_classify_redirect`, `acquia_session_warmup`, and `acquia_form_post`. The classifier distinguishes three redirect outcomes after a form submit: `success` (URL matches `/admin/structure/menu/item/<id>/edit`), `sso-waf` (URL contains `id.acquia.com`, `source.acquia.com`, or the simplesamlphp ACS path), and `unknown`. `scripts/configure-site.sh` extracts `Site name:`, `Slogan:`, and `Front page:` from `docs/migration/plan.md` via `grep`/`sed`, reads CSS selectors from `references/drupal-admin-selectors.json`, and submits to `/admin/config/system/site-settings` (the Acquia-specific path; `/admin/config/system/site-information` returns 403). `scripts/create-menus.sh` reads `docs/migration/menu-map.json` (a new artifact that Phase 1's site-analyzer must emit), checks each item for existence via `GET /api/menu_items/<menu>`, and submits new items to `/admin/structure/menu/manage/<menu>/add`. Both scripts support a `--dry-run` flag and retry once on non-success redirects.

CSS selectors are centralized in a new `references/drupal-admin-selectors.json` file rather than hardcoded in each script, so a change in the Drupal admin theme requires editing only one file. A `references/menu-map-schema.md` documents the shape of `menu-map.json` and explicitly notes that the `parent` and `weight` fields are recognized by the schema but currently silently dropped by `create-menus.sh`.

The second commit (a919b624) added the tests that preceded the fix in the commit history and also hardened the existing classifier test to use `execFileSync` with `bash -c script -- url` instead of `execSync` with string interpolation, closing a shell-injection vector in the test helper.

## Upgrade

If you use the canvas_ai_migrations migrate-site skill, Phase 6 now requires `docs/migration/menu-map.json` to be present (emitted by Phase 1). Verify that your Phase 1 site-analyzer produces this file before running Phase 6. The LLM agents remain available as a manual fallback; to invoke them you must explicitly override the default script path documented in SKILL.md.

The `references/drupal-admin-selectors.json` file centralizes form selectors. If your Acquia admin theme differs from the expected selectors, update that file rather than the scripts:

```json
{
  "menu_add": {
    "label": "input[name=\"link[title][0][value]\"]",
    "url":   "input[name=\"link[uri][0][uri]\"]"
  },
  "site_settings": {
    "name":       "input[name=\"site_name\"]",
    "slogan":     "input[name=\"site_slogan\"]",
    "front_page": "input[name=\"site_frontpage\"]"
  }
}
```
