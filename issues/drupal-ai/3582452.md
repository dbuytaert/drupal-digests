## Summary

The AI Search Block configuration form now includes a dedicated **Guardrails** section. Site builders can choose one of three modes: no guardrails, apply an existing saved `ai_guardrail_set` entity, or pick individual pre-generation and post-generation guardrails. The selected guardrail configuration is saved in block config, exported correctly, and is attached to every AI request that block makes, so compatible providers (such as LiteLLM) can enforce the configured safety policies. If guardrail entities are deleted or unavailable, stale IDs are filtered out at request time rather than causing errors. This requires the `ai` module at version 1.4 or later.

## Impact

Affects module maintainers and site owners.

- **Requirements:** `drupal/ai` minimum version raised from `^1.2` to `^1.4`.
- **New API:** New `guardrail_mode`, `guardrail_set`, `pre_generate_guardrails`, `post_generate_guardrails`, and `guardrail_stop_threshold` keys in `block.settings.ai_search_block` config schema.

## Technical details

The `SearchFormBlock` plugin gains a `guardrails` fieldset with a `guardrail_mode` radio group (`none`, `sets`, `individual`). In `sets` mode, a select element lists all `ai_guardrail_set` entities; in `individual` mode, two `checkboxes` elements list all `ai_guardrail` entities for pre-generation and post-generation slots, along with a numeric `guardrail_stop_threshold` field. Empty-state markup (with links to create/manage entities) is shown when no entities exist, preventing confusing or broken UI states. `calculateDependencies()` is implemented on `SearchFormBlock` to record the selected `ai_guardrail_set` and `ai_guardrail` config entities as proper block config dependencies.

The actual guardrail attachment happens in the new private `applyGuardrailSet()` method on `AiSearchBlockHelper`, called just before `ChatInput` is passed to the provider. In `sets` mode it delegates to `AiGuardrailHelper::applyGuardrailSetToChatInput()`. In `individual` mode it loads and validates the selected guardrail IDs, then creates a transient unsaved `AiGuardrailSet` entity (identified by a hashed UUID prefix to avoid collisions), clones the `ChatInput`, and calls `addGuardrailSet()` on the clone. Non-existent guardrail IDs are intersected against actually-loaded entities before the transient set is built. The `@ai.guardrail_helper` service is injected into `AiSearchBlockHelper` via `ai.services.yml`. The `drupal/ai` requirement is bumped from `^1.2` to `^1.4` because `addGuardrailSet()` and `getGuardrailSets()` on `ChatInput` are only available from that version. A complete `block.settings.ai_search_block` config schema was also added, covering all existing and new settings. Kernel test coverage is provided in `AiSearchBlockHelperGuardrailTest` with five test cases covering sets mode, individual mode, empty selection, and stale-ID filtering.

## Upgrade

Sites must update the `drupal/ai` dependency to `^1.4` before enabling this version of `ai_search_block`. In `composer.json`:

```json
"require": {
    "drupal/ai": "^1.4"
}
```

No existing block configuration is affected: the new `guardrail_mode` key defaults to `none`, so existing blocks continue to send requests without guardrails unless explicitly reconfigured.
