## Summary

The "tool" contributed module gained a mechanism for individual tool plugins to declare whether their prerequisites are satisfied. Tools that depend on an external API key, a system library like FFmpeg, or any other runtime condition can now return false from a new check method, causing them to be excluded from the set of executable tools. A default implementation in `ToolBase` returns true so existing plugins require no changes.

## Impact

Affects module maintainers.

- **New API:** New availability-check method on `ToolInterface` (default `true` in `ToolBase`) and `getExecutableDefinitions()` on the plugin manager to retrieve only tools whose prerequisites are met.

## Technical details

The "tool" module manages Drupal AI tool plugins. Before this change there was no standard way for a plugin to signal that it was not ready to run (for example, because a required API key had not been configured or a system dependency was absent).

The fix adds an availability-check method to `ToolInterface` with a default implementation in `ToolBase` that returns `true`, keeping existing plugins unaffected. The design was explicitly modelled on `ImageToolkitInterface::isAvailable()` from Drupal core (see `core/lib/Drupal/Core/ImageToolkit/ImageToolkitInterface.php` line 149), which performs the same role for GD and ImageMagick.

Implementation approach, confirmed by b_sharpe in comment [#16363764](https://www.drupal.org/node/16363764): the pattern mirrors the existing `isDestructive` flag. Rather than removing unavailable tools from all plugin definitions, they are flagged so that UI components such as checkbox pickers can still enumerate and display them (as disabled), while a new `getExecutableDefinitions()` method on the plugin manager filters the list to only those tools that pass the check. Reviewed by nikro (comment [#16415963](https://www.drupal.org/node/16415963)), who confirmed that `getExecutableDefinitions()` correctly excludes a tool configured to return `false`.

The method name evolved during the issue: the original proposal used `isAvailable`, b_sharpe's MR !62 implemented it as `isExecutable`, and michaellander's MR !74 (branch `3555051-poc_checkRequirements`) introduced a further rename that marcus_johansson described as "much better". Without a commit diff the exact final method name on `ToolInterface` cannot be confirmed here; nikro's test comment names `getExecutableDefinitions()` on the manager side.
