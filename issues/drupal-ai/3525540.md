## Summary

When using the Drupal AI module with certain contributed modules that define action plugins with complex configuration schemas, the system could crash with a fatal PHP error. This happened when the AI module tried to discover available actions for AI agents to use, but encountered action configurations it couldn't properly process. The error appeared on administrative pages like the AI Agents settings page and chat explorer. The fix adds defensive checks to skip action configurations that can't be converted to the format AI agents expect, preventing the crash and allowing the AI module to work alongside a wider range of contributed modules.

## Impact

No upgrade or configuration changes required.

## Technical details

The `ActionPluginDeriver` class in the AI module discovers action plugins from contributed modules and converts them into function calls that AI agents can invoke. During this discovery process, the `resolveRootDataType()` method uses PHP reflection to extract data type information from configuration schema definitions.

The crash occurred when `resolveRootDataType()` called `$attributes[0]->newInstance()` on an empty array. This happened with action plugins that had configuration schemas the method couldn't resolve—either because the attributes array was empty, or because the configuration used a `sequence` type that contains multiple values.

The fix makes two changes. First, it adds a guard check in `resolveRootDataType()` to return an empty string when no attributes are found, instead of attempting to access a non-existent array element. Second, it adds special handling for `sequence` type configurations by extracting the inner type from `$info['sequence']['type']` before resolving it. Finally, it skips creating a `ContextDefinition` entirely when the type resolution returns an empty string.

This allows the deriver to gracefully handle action plugins from modules like Action (core), Domain Access, and others that define complex configuration schemas that aren't compatible with AI function calling.

### Collaboration

This issue required significant community effort to diagnose. The initial reporter couldn't reliably reproduce the error, and several contributors tested clean installations without success. The breakthrough came when contributors identified that enabling Drupal core's Action module triggered the error consistently. One contributor then traced the issue to specific configuration schemas in the Domain Access module that use sequence types. The discussion included multiple rounds of testing across different Drupal versions (10.4, 10.5, and 11.x) and AI module branches (1.1.x and 1.2.x), with contributors confirming that related improvements in the 1.2.x branch prevented similar issues there. The maintainer backported the fix to both branches with slightly different implementations due to code divergence between versions.

## Lessons learned

- When deriving plugin definitions from dynamic configuration schemas, validate that referenced type definitions exist before attempting to instantiate them. Return a safe fallback (like the original type string) when schema references are unresolvable to prevent null pointer errors deep in discovery. *(via robloach, norbert-goco, marcus_johansson)*
- When processing configuration schemas with nested data structures (like sequences within mappings), resolve the innermost concrete type rather than the container type. Configuration can contain multi-level structures where the meaningful type for plugin derivation is the leaf element, not the collection. *(via norbert-goco)*
