## Summary

The `canvas_ai_migrations` workflow gains a new Phase 5.5 that deploys global CSS and brand kit settings to Canvas config entities as a deterministic (non-LLM) step. Previously, `global.css` was bundled inside a specific component, meaning any style update required re-uploading the whole component. After this change, global styles live in the Canvas `asset_library` config entity and can be independently PATCH-updated post-migration without touching component bundles. A companion script also patches the global `brand_kit` entity with fonts discovered during migration. Both scripts run via Node.js and are orchestrated between Phase 5 (media) and Phase 6 (site-config) in SKILL.md.

## Impact

No upgrade or configuration changes required.

## Technical details

Two new scripts are added under `scripts/`:

`setup-asset-library.mjs` targets `/canvas/api/v0/config/asset_library/global` using the `canvas:asset_library` OAuth scope. It reads `src/components/global.css`, GETs the endpoint to check existence, then either POSTs (HTTP 201 expected) or PATCHes (HTTP 200 expected). The payload uses `{ id, label, css: { original, compiled } }`. On completion it writes `docs/migration/workflow/05.5-asset-library-checklist.json` conforming to `state-envelope.schema.json` via an atomic `.tmp`-to-rename write.

`setup-brand-kit.mjs` targets `/canvas/api/v0/config/brand_kit/global` using the `canvas:brand_kit` scope. Live API probing during development confirmed that POST on the brand_kit endpoint returns 403 with `"Brand kits cannot be created via the API"` — only the pre-provisioned `global` instance exists per Acquia installation. The script therefore does GET-then-PATCH only. Fonts are read from `docs/migration/discovery/design-tokens.json`; if no fonts key is present, the script exits cleanly with a skip note. PATCH requires the Drupal permission `administer brand kit`; the script detects the 403 message and throws a permission-specific error. The `canvas:brand_kit` OAuth scope is only available on Acquia release 173672 and later; older instances will record `invalid_scope` in the scope-availability manifest and Phase 5.5 brand_kit step is skipped automatically.

Both scripts export a pure function with a `fetchImpl` parameter for fetch-mocked unit tests, and use a CLI guard (`import.meta.url === file://${process.argv[1]}`) so importing the module in tests does not trigger side effects. SKILL.md documents the two-script sequence, OAuth scope requirements, and the POST-disabled brand_kit constraint. `component-builder.md` adds a `## Global CSS handling` note that components ship JS and component-scoped styles only. Two new entries in `references/gotchas.md` cover the brand_kit POST restriction and the `administer brand kit` permission requirement.

## Upgrade

Operators running the `canvas_ai_migrations` workflow should add Phase 5.5 between media upload and site-config phases:

```bash
# Phase 5.5 — Asset Library + Brand Kit Setup
node scripts/setup-asset-library.mjs
node scripts/setup-brand-kit.mjs
```

Ensure the `canvas:asset_library` OAuth scope is granted (it is universally available on Canvas/Acquia). For brand kit setup, ensure the `canvas:brand_kit` scope is available (Acquia release 173672+) and that the migrating user role has the `administer brand kit` Drupal permission. If the permission is absent, `setup-brand-kit.mjs` exits 1 with a descriptive error; coordinate with the Acquia operator before running Phase 5.5.

Global CSS is now read from `src/components/global.css` and should no longer be embedded in any component bundle. The `CANVAS_GLOBAL_CSS_PATH` environment variable can override the default path.
