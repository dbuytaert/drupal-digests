## Summary

The `document_loader` module now exposes two alter hooks that let other modules intercept every document load centrally, without patching individual consumers. A new optional `$caller` parameter on `loadFromInput()` and `loadFromData()` identifies which consumer triggered the load (MDX editor, FWA field widget action, agent tool, explorer form, Drush, or direct API), so hook implementations can apply conditional logic per consumer.

## Impact

Affects module maintainers.

- **API change:** New optional `string $caller = 'api'` parameter added to `DocumentLoaderManager::loadFromInput()` and `loadFromData()`; `ModuleHandlerInterface` added as a new required constructor dependency of `DocumentLoaderManager`.
- **New API:** Two new hooks: `hook_document_loader_pre_load_alter` (fires before loading, mutable: `output_format`, `max_length`, `loader`) and `hook_document_loader_post_load_alter` (fires after loading, mutable: `content`, `format`, `metadata`, `source`, `original_bytes`, `loader_label`).

## Technical details

The core problem was that `DocumentLoaderManager::loadFromInput()` is the single entry point for all document loading but provided no extension point for other modules. The fix adds two `ModuleHandlerInterface::alter()` calls inside `loadFromInput()`, each split into a mutable first parameter and a read-only second parameter to make the hook contract self-documenting.

`hook_document_loader_pre_load_alter(array &$params, array $context)` fires after validation and access checks but before `$loader->load()`. The mutable `$params` array carries `output_format`, `max_length`, and `loader`; the read-only `$context` carries `document_loader_type`, `input`, and `caller`.

`hook_document_loader_post_load_alter(array &$result_data, array $load_context)` fires after the loader runs but before the immutable `DocumentLoaderResult` is constructed. The mutable `$result_data` array carries `content`, `format`, `metadata`, `source`, `original_bytes`, and `loader_label`; `$load_context` is read-only and includes the full load configuration plus `caller`.

Both `loadFromInput()` and `loadFromData()` receive a new optional `string $caller = 'api'` parameter. Existing consumers pass it as a PHP named argument: `DocumentLoaderAction` (FWA) passes `caller: 'fwa'`, `DocumentLoaderTool` passes `caller: 'tool'`, `DocumentLoaderCommands` passes `caller: 'drush'`, and `DocumentLoaderExplorerForm` passes `caller: 'explorer'`.

`ModuleHandlerInterface` is injected into `DocumentLoaderManager` as a new last constructor argument and registered via `@module_handler` in `document_loader.services.yml`. Full hook documentation lives in the new `document_loader.api.php` file. Kernel tests in `DocumentLoaderHooksTest` verify both hooks fire correctly using a `document_loader_test` fixture module that reads override values from Drupal state.

## Upgrade

Existing call sites for `loadFromInput()` and `loadFromData()` continue to work without changes because `$caller` defaults to `'api'`. Pass it as a named argument to distinguish your consumer in hook implementations:

```php
$result = $manager->loadFromInput(
  'document_loader_type:pdf',
  $input,
  $account,
  'markdown',
  5000,
  caller: 'my_module',
);
```

If you have a service that manually constructs or decorates `DocumentLoaderManager`, add `ModuleHandlerInterface` as its final constructor argument:

```php
new DocumentLoaderManager(
  $plugin_manager,
  $config_factory,
  $document_loader_plugin_manager,
  $type_factory,
  $file_system,
  $file_input_normalizer,
  $module_handler, // new
);
```
