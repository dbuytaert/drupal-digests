## Summary

The `gitlab_api` module now supports per-project configuration and webhook-driven automation. A new `GitlabProject` config entity lets site owners register individual GitLab repositories with their own access tokens and webhook secrets (managed via the Key module). Incoming GitLab webhooks are dispatched as Symfony events that ECA workflows can subscribe to, with new ECA event plugins for nine webhook types (push, tag push, issue lifecycle, comment, merge request, pipeline), and new ECA conditions for slash command matching and issue template validation. The `GitlabServer` config entity now stores its authentication token as a Key entity reference rather than a plain string, and update hooks migrate existing tokens.

## Impact

Affects module maintainers and site owners.

- **Database update:** New gitlab_seen_event table for webhook deduplication; update hooks migrate existing server access tokens to Key entities
- **Config change:** GitlabServer config entity renames auth_token to auth_token_key (now a Key entity machine name) and adds label and status fields; new GitlabProject config entity type added
- **New API:** Nine new Symfony events (GitlabPushEvent, GitlabTagPushEvent, GitlabIssueCreatedEvent, GitlabIssueUpdatedEvent, GitlabIssueClosedEvent, GitlabIssueReopenedEvent, GitlabCommentEvent, GitlabMrEvent, GitlabPipelineEvent) and nine ECA event plugin derivatives, two ECA condition plugins (eca_gitlab_api_slash_command, eca_gitlab_api_body_follows_template), and the eca_gitlab_api.slash_command_parser service

## Technical details

A new `GitlabProject` config entity (`gitlab_api.gitlab_project.*`) carries fields for the linked server, the numeric GitLab project ID, the project path, an `access_token_source` toggle (server-level vs. per-project Key entity), a `webhook_secret_keys` sequence of Key entity IDs, a `maintainers` sequence of GitLab usernames, and an enabled `status` flag. `GitlabServer` config schema gains `label` and `status` fields, and its `auth_token` field is renamed to `auth_token_key` to reflect that the stored value is now a Key entity machine name.

A new `WebhookController` handles POST requests at `/gitlab-api/webhook/{data_name}`, resolves the `GitlabProject` entity by machine name, verifies the `X-Gitlab-Token` header against the project's Key entity secret(s), deduplicates events using a `gitlab_seen_event` database table, and dispatches one of nine typed Symfony events: `GitlabPushEvent`, `GitlabTagPushEvent`, `GitlabIssueCreatedEvent`, `GitlabIssueUpdatedEvent`, `GitlabIssueClosedEvent`, `GitlabIssueReopenedEvent`, `GitlabCommentEvent`, `GitlabMrEvent`, and `GitlabPipelineEvent`. All extend `GitlabEventBase` and carry the raw webhook payload and a reference to the `GitlabProject` entity.

The `eca_gitlab_api` submodule provides `GitlabIntegrationEvent` (plugin ID `eca_gitlab_api_event`), derived into nine variants via `GitlabIntegrationEventDeriver`. Each variant exposes a rich `[event:*]` token set pulled from the webhook payload, including `[event:project_id]`, `[event:user_username]`, `[event:payload_json]`, type-specific tokens such as `[event:issue_iid]`, `[event:body]`, `[event:mr_iid]`, and `[event:maintainer_mentions]`. The event configuration form includes a scope radio (all projects or selected) and a project-filter checkbox list; `generateWildcard` encodes the selected project IDs into a comma-separated string that `appliesForWildcard` matches at dispatch time.

Two new ECA conditions ship with the submodule. `SlashCommandCondition` (`eca_gitlab_api_slash_command`) delegates parsing to the new stateless `SlashCommandParser` service (`eca_gitlab_api.slash_command_parser`), which scans comment lines anchored at column zero for `/command arg1 arg2` patterns including namespaced commands (`/ai:needsWork`, `/priority::high`) and kebab names (`/needs-review`). On a match it exposes `[gitlab_slash:command]` and `[gitlab_slash:arg:<name>]` tokens. `BodyFollowsTemplateCondition` (`eca_gitlab_api_body_follows_template`) checks minimum character length, minimum ATX Markdown heading count, and required heading titles, with each check independently optional.

## Upgrade

Existing `GitlabServer` config entities store a raw token in the `auth_token` field. Before running update hooks, create a Key entity for each token and note its machine name. The field is renamed to `auth_token_key` and must now hold a Key entity machine name:

```yaml
# Before
gitlab_api.gitlab_server.my_server:
  auth_token: 'glpat-xxxxxxxxxxxxxxxxxxxx'

# After
gitlab_api.gitlab_server.my_server:
  auth_token_key: 'my_server_access_token'   # Key entity machine name
  label: 'My Server'
  status: true
```

For webhook automation, create a `GitlabProject` config entity for each repository and configure the webhook URL at `https://<your-site>/gitlab-api/webhook/<project-machine-name>` in the GitLab project settings. The webhook secret must match the Key entity referenced in `webhook_secret_keys`.

The `eca_gitlab_api` submodule requires ECA 3.x plus the `eca_tool` bridge module for Tool API plugin support, or ECA 4.x when that is released.
