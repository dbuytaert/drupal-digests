## Summary

The `gitlab_api` module (3.0.x) gains three major capabilities in this release. First, access tokens for GitLab servers are moved from inline config strings to Key module entities, making the Key module a hard dependency. Second, a new `GitlabProject` config entity lets administrators configure multiple projects, each with its own access token and webhook secrets. Third, a public webhook endpoint at `/gitlab-api/webhook/{project}` receives GitLab events (issues, comments, push, merge requests, pipelines) and dispatches typed Symfony events, which the `eca_gitlab_api` submodule exposes as ECA event plugins complete with conditions for slash commands and template validation. Sites upgrading from an earlier 2.x configuration must run `drush updatedb` and, critically, must switch the auto-created Key entities off the `config` provider before running `drush config:export` — otherwise the GitLab access token will be written in plaintext to config YAML and any git commit of that export will expose the secret.

## Impact

Affects module maintainers and site owners.

- **Security:** Update hook 9003 creates Key entities with `key_provider: config` by default; running `drush config:export` before switching to an `environment` or `file` provider will write the GitLab access token in plaintext to config YAML. See UPGRADING.md for remediation steps.
- **Requirements:** `drupal/key ^1.18` is now a hard runtime dependency of `gitlab_api`.
- **Database update:** New `gitlab_seen_event` table with primary key on SHA-256 event ID and an index on `seen_at` for cron-based TTL expiry.
- **Config change:** `gitlab_server` config entity field `auth_token` (inline string) removed; replaced by `auth_token_key` (Key entity ID). New `gitlab_project` config entity type added.
- **Permission change:** New `administer gitlab_project` and `use gitlab_api tools` permissions, both marked `restrict access: true`.
- **New API:** Nine typed Symfony event classes (`GitlabPushEvent`, `GitlabCommentEvent`, etc.), `GitlabProject` config entity, `WebhookController`, `GitLabApiClientFactory`, `SignatureVerifier`, and ECA event/condition plugins in `eca_gitlab_api`.

## Technical details

The new `GitlabProject` config entity (`gitlab_api.gitlab_project.*` schema) holds the GitLab project ID and path, a reference to a parent `GitlabServer`, an `access_token_source` field (server-level or per-project Key entity), one or more `webhook_secret_keys` Key entity IDs, and an optional maintainer username list. The admin UI gains a "Projects" tab alongside "Servers" via new local-tasks and action-links.

The webhook route (`/gitlab-api/webhook/{project}`, POST, `_access: TRUE`) is handled by `WebhookController::receive`, which delegates HMAC verification to the new `gitlab_api.signature_verifier` service and deduplicates payloads using the `gitlab_seen_event` database table. Each row stores a SHA-256 hash of the project ID concatenated with the `X-Gitlab-Event-UUID` header (or a hash of the body when the header is absent). Rows expire after 600 seconds via `hook_cron`. Nine Symfony event classes are fired on verified receipt: `GitlabPushEvent`, `GitlabTagPushEvent`, `GitlabIssueCreatedEvent`, `GitlabIssueUpdatedEvent`, `GitlabIssueClosedEvent`, `GitlabIssueReopenedEvent`, `GitlabCommentEvent`, `GitlabMrEvent`, and `GitlabPipelineEvent`. All extend `GitlabEventBase` and carry the full payload and the originating `GitlabProject` entity.

The `eca_gitlab_api` submodule maps each event to a derived ECA event plugin via `GitlabIntegrationEventDeriver` (base plugin `eca_gitlab_api_event`, derivative IDs `:push`, `:comment`, etc.). Plugins expose payload fields as `[event:*]` tokens and a `[gitlab_project]` entity token. A project scope filter (all or selected IDs) is implemented through `generateWildcard` and `appliesForWildcard`. Two new ECA conditions ship: `SlashCommandCondition`, which matches `/command` lines in comment bodies using the stateless `SlashCommandParser` service and exposes `[gitlab_slash:command]` and `[gitlab_slash:arg:<name>]` tokens on match; and `BodyFollowsTemplateCondition`, which validates minimum character length, ATX Markdown heading count, and required heading titles.

The migration path runs three update hooks: `gitlab_api_update_9001` installs the Key module, `gitlab_api_update_9002` creates `gitlab_seen_event`, and `gitlab_api_update_9003` reads each server's inline `auth_token`, creates a Key entity (ID `gitlab_api_server_<id>_token`, provider `config`) holding the value, writes the Key ID back as `auth_token_key`, and clears the inline field. A `hook_requirements` check surfaces servers where a runtime config override (e.g. `$config[...]['auth_token']` in `settings.php`) outlived the migration, since the override no longer has any effect but still holds the secret on disk.

## Upgrade

Run database updates after deploying 3.0.x:

```bash
drush updatedb
```

This installs the Key module, creates the `gitlab_seen_event` table, and migrates each GitLab server's inline token to a Key entity.

**Before running `drush config:export`**, switch every auto-created Key entity off the `config` provider:

1. Visit `/admin/config/system/keys` and edit each `gitlab_api_server_<id>_token` entity.
2. Change the Key provider to **Environment** (or **File**) and supply the token value via an environment variable or file outside the docroot.
3. Save, then export config. The exported YAML will contain only the provider configuration, not the token value.

If you previously set `auth_token` via `settings.php` or a similar override, that override no longer does anything; remove it. The status report at `/admin/reports/status` lists any servers where such a stale override is detected.

The two new permissions should be reviewed and granted appropriately:

```
administer gitlab_project  # restrict access: true
use gitlab_api tools       # restrict access: true — grants write operations to GitLab via Tool plugins
```
