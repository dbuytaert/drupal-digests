## Summary

The AI Automators module previously ran each automator rule only when the target field was empty (or in edit mode with a changed source). There was no programmatic way to force a rule to re-run on entities that already had values. This change adds a Symfony event, `ai_automator.should_process_field`, that fires after the empty-check logic runs, letting developers override the process/skip decision with a custom event subscriber. A companion interface gives rule plugins a hook to adjust the normalized value before the event fires. Two pre-existing bugs in the base-mode decision logic and original-entity retrieval were also corrected in the same commit.

## Impact

Affects module maintainers.

- **New API:** New `ShouldProcessFieldEvent` (event name `ai_automator.should_process_field`) and `AiAutomatorPostCheckIfEmptyInterface` allow external code to override or adjust automator field processing decisions.

## Technical details

The decision of whether to run an automator rule lives in `AiAutomatorEntityModifier` in two private methods: `tokenShouldSave()` for token-mode rules and `baseShouldSave()` for base-mode rules. Both previously hard-coded their boolean return based directly on the output of `checkIfEmpty()`.

The commit rewires both methods around a three-step pipeline:

1. **Post-check normalization.** A new `applyPostCheckIfEmpty()` helper checks whether the rule implements the new `AiAutomatorPostCheckIfEmptyInterface`. If it does, `postCheckIfEmpty(ContentEntityInterface $entity, array $value, array $automatorConfig)` is called and its return value replaces `$value` (ignored when the return is not an array, or when `checkIfEmpty()` returned a non-array type). `RuleBase` now implements this interface with a passthrough default so all existing rules need no changes; rules not extending `RuleBase` may opt in by implementing the interface.

2. **Initial decision.** A new `isValueEmptyAfterCheck(mixed $value)` helper computes the initial `$shouldProcess` boolean from the normalized value, handling both array and scalar returns from `checkIfEmpty()` without array-access errors on non-arrays.

3. **Event dispatch.** A new `ShouldProcessFieldEvent` (constant `EVENT_NAME = 'ai_automator.should_process_field'`) is constructed with the entity, field definition, and full automator config, then dispatched. Subscribers call `setShouldProcess(TRUE|FALSE)` to override the decision in either direction. The return value of both methods becomes `$event->shouldProcess()`. The event class lives in `modules/ai_automators/src/Event/ShouldProcessFieldEvent.php`.

Two pre-existing bugs were fixed in `baseShouldSave()`. The original three `if`-blocks were partly unreachable (the second and third conditions were strict subsets of the first) and did not correctly implement the stated intent (run when value is empty, or when edit mode is on and the source field changed). This was replaced with a clear two-branch condition. The original-entity retrieval also had a bug: `isset($entity->original) && json_encode(...)` produced a boolean AND result rather than the serialized value. A new `getOriginalEntity()` helper fixes this and adds Drupal 10/11 compatibility, preferring `getOriginal()` (available since Drupal 11.2) and falling back to the magic `$entity->original` property for older versions.

Documentation was added in a new `modules/ai_automators/ai_automators.api.php` and in `README.md`. Unit tests in `AiAutomatorEntityModifierDecisionTest` and `ShouldProcessFieldEventTest` cover event getters/setters, force-process and force-skip overrides, the `postCheckIfEmpty()` hook path, and safe handling of non-array and object returns from `checkIfEmpty()`.

## Upgrade

Rule plugins that extend `RuleBase` automatically inherit the default no-op `postCheckIfEmpty()` and require no changes.

Rule plugins that do **not** extend `RuleBase` may optionally implement the new interface to gain the same hook point:

```php
use Drupal\ai_automators\PluginInterfaces\AiAutomatorPostCheckIfEmptyInterface;
use Drupal\Core\Entity\ContentEntityInterface;

class MyCustomRule implements AiAutomatorTypeInterface, AiAutomatorPostCheckIfEmptyInterface {

  public function postCheckIfEmpty(ContentEntityInterface $entity, array $value, array $automatorConfig = []): array {
    // Return [] to force the automator to treat the field as empty and run.
    return $value;
  }

}
```

To force an automator rule to re-process an already-filled field, subscribe to the new event:

```php
use Drupal\ai_automators\Event\ShouldProcessFieldEvent;
use Symfony\Component\EventDispatcher\EventSubscriberInterface;

final class MyAutomatorSubscriber implements EventSubscriberInterface {

  public static function getSubscribedEvents(): array {
    return [ShouldProcessFieldEvent::EVENT_NAME => 'onShouldProcessField'];
  }

  public function onShouldProcessField(ShouldProcessFieldEvent $event): void {
    if ($event->getEntity()->bundle() === 'article') {
      $event->setShouldProcess(TRUE);
    }
  }

}
```
