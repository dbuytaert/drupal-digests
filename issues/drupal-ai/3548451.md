## Summary

The Drupal AI module now captures and exposes rate limit information returned by OpenAI-compatible providers during non-streamed chat requests. Rate limit data such as maximum requests, remaining requests, and reset times is now available through `getRateLimits()` on the `ChatOutput` object, giving modules a reliable way to inspect provider capacity and surface quota status to users instead of encountering silent failures or opaque exceptions.

## Impact

Affects module maintainers.

- **New API:** New `ChatProviderLimitsDto` DTO and `ChatOutput::setRateLimits()` / `getRateLimits()` methods expose provider rate limit information from non-streamed chat responses.

## Technical details

AI providers such as OpenAI and Anthropic include rate limit headers in chat API responses. Before this change the AI module discarded that data entirely. The work introduced a new DTO, wired it into the OpenAI-based provider base, and fixed `ChatOutput` serialization so the data survives round-trips.

**`ChatProviderLimitsDto`** (`src/Dto/ChatProviderLimitsDto.php`) is a new plain PHP class using `DtoBaseMethodsTrait`. Its constructor accepts six nullable ints as promoted properties: `rateLimitMaxRequests`, `rateLimitMaxTokens`, `rateLimitRemainingRequests`, `rateLimitRemainingTokens`, `rateLimitResetRequests`, and `rateLimitResetTokens` (reset values are stored in seconds). An `empty()` method returns true when all properties are `NULL`; it uses strict `=== NULL` comparisons rather than `empty()` to avoid false positives when a field legitimately holds zero.

**`ChatOutput`** (`src/OperationType/Chat/ChatOutput.php`) gains a private `$rateLimits` property initialized to a default empty `ChatProviderLimitsDto` in the constructor, with `setRateLimits()` and `getRateLimits()` accessors. Both `toArray()` and `fromArray()` were updated to include the `rateLimits` key so that the DTO survives serialization round-trips; `fromArray()` uses `ChatProviderLimitsDto::create()` (provided by `DtoBaseMethodsTrait`) to hydrate the object.

**`OpenAiBasedProviderClientBase`** (`src/Base/OpenAiBasedProviderClientBase.php`) gains three new protected methods and a `$rateLimitHeaders` array property:
- `captureRateLimitHeaders(?MetaInformation $metaInformation)` reads the six rate limit fields from the OpenAI PHP SDK's `MetaInformation` object (available via `$response->meta()`) and stores them in `$rateLimitHeaders`.
- `mapRateLimits()` converts `$rateLimitHeaders` to a `ChatProviderLimitsDto`, or returns `NULL` if no headers were captured. It uses `isset()` guards on each key so non-OpenAI providers that share the base class are unaffected.
- `parseResetTime(string $resetTime)` converts OpenAI's duration strings (e.g. `"1m30s"`, `"6s"`, `"200ms"`) to integer seconds.

In the non-streamed chat path, the initial response is captured as `$initialResponse`, `captureRateLimitHeaders()` is called on `$initialResponse->meta()`, and the resulting DTO is attached to `ChatOutput` via `setRateLimits()` before the output is returned. The streamed path is not yet covered and is noted as a follow-up.

Developer documentation was added to `docs/developers/call_chat.md` and `docs/developers/writing_an_ai_provider.md` explaining how to read rate limits from `ChatOutput` and how provider authors should set them.
