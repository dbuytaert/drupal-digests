## Summary

The Mistral AI provider now dynamically fetches and caches the list of available models instead of using a hardcoded list. Models are sorted alphabetically and filtered based on their actual capabilities like vision support or function calling as reported by the Mistral API. The model list is cached for 24 hours to reduce unnecessary API calls. This means sites using the Mistral provider will automatically see new models as they become available and get more accurate capability matching.

## Technical details

The `MistralProvider::getConfiguredModels()` method was refactored to use the Mistral API's model listing endpoint more effectively. Previously, the method used hardcoded logic to filter models based on name patterns like checking if a model ID starts with "pixtral" for vision support. The Mistral API now returns capability information in each model's data, including `completion_chat`, `vision`, and `function_calling` flags.

The new implementation checks these API-provided capability flags directly. For vision support it checks `$model_capabilities['vision']`, for tools it checks `$model_capabilities['function_calling']`, and for JSON output it verifies `$model_capabilities['completion_chat']` exists. Models are now filtered based on their actual reported capabilities rather than naming conventions.

The method now implements caching using Drupal's cache backend. It builds a cache key from the operation type and requested capabilities, sorted for consistency. The cache stores the filtered model list for 24 hours (86400 seconds). Before making an API call, the method checks the cache and returns cached data if available.

Finally, the filtered model list is sorted alphabetically using `asort()` before being returned and cached. This addresses the issue that the Mistral API returns models in random order without any priority ranking.
