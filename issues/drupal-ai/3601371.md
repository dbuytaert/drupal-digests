# #3601371: Make the build script only move the main script on tags

## Summary

The one-line installer's GitLab Pages publish step now fires only when a Git tag is pushed. Before this change the CI `pages` job ran on every merge to the default branch, meaning the public URL `https://project.pages.drupalcode.org/one_line_installer/drupalaibp` could serve unreleased, in-progress work to anyone running the one-liner. After this change that URL always reflects the most recent tagged release. The workflow for maintainers is now: merge the change, then tag the commit to publish it.

## Impact

No upgrade or configuration changes required.

## Technical details

The `pages` job in `.gitlab-ci.yml` gained a `rules: - if: '$CI_COMMIT_TAG'` guard. The job builds the MkDocs docs site and copies `drupalaibp` alongside it (`cp drupalaibp public/drupalaibp`), then publishes the whole `public/` directory to GitLab Pages.

The entire job is tag-gated rather than just the `cp` line because a Pages deploy replaces the whole site: a docs-only deploy from the default branch would publish a `public/` directory without `drupalaibp` in it, deleting the installer from the live URL until the next tag. Keeping docs and script together and shipping both only on a tag avoids that window.

The `bashly_verify` job still runs on every pipeline (merge or tag), so drift between `src/` and the committed `drupalaibp` is caught continuously. The tag pipeline runs `bashly_verify` as a prerequisite for `pages` (via `needs`), so a drifted script cannot be published even if a tag is pushed without regenerating.
