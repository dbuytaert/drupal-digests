## Summary

The `canvas_ai_migrations` migrate-site skill now runs a "Phase 0.5" preflight check before beginning any migration. It probes every OAuth scope against the Acquia Source instance, classifying each as `confirmed`, `issuance-only`, `denied`, or `unknown`, and writes results to `docs/migration/workflow/scope-availability.json`. Downstream agents read this file to decide whether to use an API-first path or fall back to browser automation. The check is non-blocking: a denied or missing scope never aborts the migration. This eliminates the previous failure pattern where a scope issued a valid token but the corresponding API endpoint did not exist on the target Acquia release, causing agents to burn retry turns mid-migration on a 404 that had no resolution.

## Impact

Affects module maintainers.

- **New API:** New `classifyResponse(status)` export in `check-12-verify-scopes.mjs` mapping HTTP status codes to one of four scope-classification states (`confirmed`, `issuance-only`, `denied`, `unknown`)

## Technical details

The root cause was that OAuth `client_credentials` token issuance and API endpoint availability are independent on Acquia Source. A scope like `menu:administer` could return HTTP 200 from `/oauth/token` while `POST /api/menu_link_content/menu_link_content` returned 404 because the endpoint simply did not exist on that Acquia release. Without a preflight, agents discovered this only deep inside Phase 6.

The fix adds `scripts/preflight/check-12-verify-scopes.mjs`, registered with `severity: 'soft'` in the preflight runner. On each `migrate.sh` invocation it reads `CANVAS_SITE_URL`, `CANVAS_CLIENT_ID`, and `CANVAS_CLIENT_SECRET` from `.env`, then iterates the nine scopes in `scripts/preflight/lib/scope-manifest.json`. For each scope it requests a `client_credentials` token, then probes the associated endpoint with the bearer token. The exported `classifyResponse()` function maps the probe HTTP status to one of four states: `confirmed` (200/201/204), `issuance-only` (404/405, meaning the API does not exist on this release), `denied` (403), or `unknown` (5xx, network error, or null). Three entries in the manifest are marked `_guess` for entity types whose JSON:API paths were not verified against a live instance; a runtime 404 from those probes produces `issuance-only`, which agents already handle by routing to browser automation.

Results are written to `docs/migration/workflow/scope-availability.json` keyed by scope name with `{issuance, endpoint, http_status, probe_url}` per entry. The script also appends an "OAuth scopes on this Acquia release" section to the runtime `docs/migration/gotchas.md` so the orchestrator's context inherits platform state across compaction boundaries. The template copy at `.claude/skills/migrate-site/references/gotchas.md` is never modified at runtime. Agent instruction files for `menu-builder` and `site-configurator` were updated to read `scope-availability.json` before choosing API vs. browser paths, and `SKILL.md` gained a Phase 0.5 block documenting the four states and the "log and continue" policy.
