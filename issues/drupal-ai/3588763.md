## Summary

The `canvas_ai_migrations` skill now runs a "Phase 0.5" OAuth scope verification preflight before the main migration phases begin. It probes every OAuth scope used by the skill against the target Acquia Source instance, writing a `docs/migration/workflow/scope-availability.json` file. Downstream migration agents (menu-builder, site-configurator, and others) read this file to decide whether to use a JSON:API path or fall back to browser automation. The preflight never blocks a migration: if scopes are missing or denied, agents simply use browser paths instead.

## Impact

Affects module maintainers.

## Technical details

The root problem was that some OAuth scopes on Acquia Source mint valid tokens (HTTP 200 from `/oauth/token`) but the corresponding API endpoints do not exist on that release, returning 404 or 405. An agent that assumed a token issuance meant the endpoint was reachable would burn turns on retry/diagnostic loops for an endpoint that simply was not there.

The new `scripts/preflight/check-12-verify-scopes.mjs` script implements two phases per scope: a `client_credentials` token request, then a probe request to the scope's canonical endpoint. The exported `classifyResponse(status)` function maps HTTP status to one of four states: `confirmed` (200/201/204), `issuance-only` (404/405, endpoint absent on this release), `denied` (403, ACL), or `unknown` (5xx, network error, or `null`). Results are written to `docs/migration/workflow/scope-availability.json` keyed by scope name with `{issuance, endpoint, http_status, probe_url}`. A summary section is also appended to the runtime `docs/migration/gotchas.md` so the orchestrator's context carries the platform state across compaction boundaries.

The scope manifest at `scripts/preflight/lib/scope-manifest.json` lists nine scopes: four `canvas_oauth:*` scopes (menu_link_content, taxonomy_term, canvas_page, media) and five `acquia_source:*` scopes (site_settings, content_types, templates, code_components, redirects). The last three are marked `_guess` in the manifest — standard Drupal JSON:API conventions that may not exist on every Acquia release. The check is registered with `severity: 'soft'` in the preflight runner, so it always returns `pass` and never aborts the migration.

Agent prompt files for `menu-builder.md` and `site-configurator.md` were updated with a "Phase 0.5 dependency" section instructing each agent to read `scope-availability.json` before choosing a path, and to treat every scope as `unknown` (and use browser automation) if the file is absent.
