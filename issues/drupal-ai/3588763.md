## Summary

The `canvas_ai_migrations` migrate-site skill now includes a Phase 0.5 "verify-scopes" preflight step. Before the main migration begins, a new script probes every OAuth scope in a manifest: it requests a token via the `client_credentials` grant, then calls the corresponding API endpoint with that token. The result for each scope is recorded as `confirmed`, `issuance-only`, `denied`, or `unknown` in `docs/migration/workflow/scope-availability.json`. Downstream agents (menu-builder, site-configurator, and others) read this file to decide whether to use an API-first path or fall back to browser automation. The migration never aborts because a scope is unavailable; it logs the state and continues. This eliminates the class of failure where an agent burns retries on a 404 endpoint that simply does not exist on the current Acquia Source release, because the OAuth server issued a valid token but the corresponding API was never deployed.

## Technical details

The core problem was that several OAuth scopes on Acquia Source mint a valid token (HTTP 200 from `/oauth/token`) but have no corresponding API endpoint: for example, `menu:administer` issued a token yet `POST /api/menu_link_content/menu_link_content` returned 404. Agents had no way to distinguish "scope configured but endpoint missing" from "scope configured and endpoint reachable," so they looped through retry and diagnostic logic for endpoints that could never succeed.

The fix introduces `scripts/preflight/check-12-verify-scopes.mjs`, a preflight check registered with severity `soft` in the existing runner framework. It exports `classifyResponse(status)`, which maps HTTP status codes to four states: `confirmed` (200/201/204), `issuance-only` (404/405), `denied` (403), and `unknown` (everything else including network errors). The exported `verifyScopes()` function iterates the scope manifest, calls `requestToken()` per scope, then calls `probeEndpoint()` with the issued bearer token. Results are written as a JSON object keyed by scope name to `docs/migration/workflow/scope-availability.json`. A `null` endpoint in the manifest is a deliberate "this API does not exist" marker and is classified as `issuance-only` without making an HTTP call.

The scope manifest lives at `scripts/preflight/lib/scope-manifest.json` and defines nine scopes. Six are marked as stable (verified against a live instance): `canvas_oauth:menu_link_content`, `canvas_oauth:taxonomy_term`, `canvas_oauth:canvas_page`, `canvas_oauth:media`, `acquia_source:site_settings`, and `acquia_source:content_types`. Three (`acquia_source:templates`, `acquia_source:code_components`, `acquia_source:redirects`) are best-effort guesses against standard JSON:API conventions and are annotated with `_guess` markers in the manifest.

`SKILL.md` receives a new "Phase 0.5 — Verify Scopes" block that documents the four classification states and the non-blocking design. The agent instruction files `menu-builder.md` and `site-configurator.md` each gain a "Phase 0.5 dependency" section instructing the agent to read `scope-availability.json` before choosing an API path, and to treat any missing file as all-`unknown` (fall back to browser automation for the entire run). After writing the JSON file, the check also appends an "OAuth scopes on this Acquia release" section to the runtime `docs/migration/gotchas.md` so the classification survives context compaction. The template file at `.claude/skills/migrate-site/references/gotchas.md` is never modified at runtime. Results are session-only; there is no persistent cache across `migrate.sh` invocations.
