## Summary

A critical bug prevented the AI Context module from working immediately after installation on Drupal 10. When users tried to create their first AI Context item, the page crashed with a fatal error about missing scheduler integration fields. The error only occurred on Drupal 10, not Drupal 11, and required visiting a settings page first to trigger initialization. This fix ensures the required field definitions are created automatically before they are needed, making the module work correctly on fresh Drupal 10 installations without manual workarounds.

## Impact

No upgrade or configuration changes required.

## Technical details

The issue was a timing race condition specific to Drupal 10. The `ai_context_item/default` bundle was already registered with the content moderation workflow, and Scheduler was enabled. When users visited `/admin/ai/context/items/add` immediately after installation, Scheduler Content Moderation Integration (SCMI) ran its form hooks before its `publish_state` and `unpublish_state` field storage definitions were materialized. This caused SCMI to dereference null when checking for those fields, triggering a fatal error.

The fix introduces `AiSchedulerContentModerationSync`, a dedicated class that lazily synchronizes SCMI field storage definitions before Scheduler's hooks run. The module now implements `hook_scheduler_hide_publish_date()` and `hook_scheduler_hide_unpublish_date()`, uses `hook_module_implements_alter()` to ensure these run first, and triggers the sync during `hook_modules_installed()` for any relevant module (ai_context, scheduler, or SCMI). The sync operation is idempotent and safe to call multiple times.

The fix only targets Drupal 10, where the issue was reproducible. Drupal 11 did not exhibit this problem. Test coverage was expanded to verify the add form works immediately after install with Scheduler enabled, and existing test workarounds that disabled Scheduler were removed.

## Lessons learned

- When a module's configuration entity type is marked as moderated and integrates with Scheduler, field storage for moderation integration fields (like publish_state and unpublish_state) must be materialized before any form_alter hooks attempt to access those fields. If lazy initialization is unavoidable, eagerly synchronize field storage definitions in your module's hook_form_alter implementation before delegating to third-party integrations, ensuring dependent modules don't dereference null field definitions. *(via scott falconer)*
- When configuration is required for entity forms to render but isn't created during module installation, visiting the entity add form will fail. Ensure default configuration entities are created in hook_install or provide lazy initialization in form_alter before any hook that depends on that configuration runs, so users don't encounter errors on first use. *(via scott falconer, dstorozhuk)*
