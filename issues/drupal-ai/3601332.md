# #3601332: Adopt Bashly (bashly.dev) to generate the installer from a single YAML-defined CLI

## Summary

The one_line_installer project adopted [Bashly](https://bashly.dev/) as its build framework, replacing roughly 750 lines of hand-written bash spread across four files with a single generated script called `drupalaibp`. The committed `drupalaibp` file (around 4,900 lines) is generated from a `src/bashly.yml` CLI definition plus per-command and shared-library partials in `src/lib/`. End users run the same `bash <(curl …)` invocation; they now target the `drupalaibp` script rather than the old `install.sh`. Maintainers who regenerate the script need the `bashly` Ruby gem (version pinned in `.bashly-version`). A CI job (`bashly_verify`) fails if the committed script ever drifts from `src/`.

## Impact

Affects module maintainers.

- **New API:** New `.drupalaibp.json` config schema for unattended, config-driven builds with provisioning, custom questions, and hook scripts at named pipeline events.

## Technical details

The old installer duplicated `step`/`ok`/`warn`/`die` helpers and ANSI color constants across `install.sh`, `install-macos.sh`, `install-linux.sh`, and `github-push.sh`. Bashly solves this by injecting a shared library into the generated bundle. Each shared helper now lives once: `src/lib/shared/colors.sh`, `src/lib/shared/output_helpers.sh`, `src/lib/shared/select_one.sh`, `src/lib/shared/select_many.sh`, and so on.

The CLI is structured as three subcommands defined in `src/bashly.yml`: `setup-site` (the default, covering fresh installs and existing-project detection), `install` (registers the tool as a global `drupal-setup` command in `~/.local/bin`), and `delete` (removes a DDEV project). Automatic `--help`, `--version`, and flag validation come from Bashly's generated argument parser, which requires bash 4. A `src/lib/core/00_require_bash.sh` shim re-execs under a newer bash when macOS ships bash 3.2.

The Bashly adoption also carried a significant feature expansion. A `.drupalaibp.json` config format was added (`src/lib/config/load_config.sh`, `src/lib/config/hooks.sh`) that lets maintainers drive unattended builds from a JSON file declaring profile, coding agents, skill repos, Composer packages, modules, recipes, DDEV add-ons, custom interactive questions, and hook scripts fired at named pipeline events (`ddev_started`, `coding_agent_added`, etc.). The config-export path (`drupalaibp_export`) serializes the interactive choices back to JSON in pure bash without a `jq` runtime dependency. The `--config-url` flag clones an external git repo containing a `.drupalaibp.json` and uses it as the build config.

## Upgrade

Users previously invoking the old `install.sh` entry point should switch to `drupalaibp`:

```bash
# Old (1.0.x)
bash <(curl -fsSL https://git.drupalcode.org/project/one_line_installer/-/raw/1.0.x/install.sh)

# New (2.0.x)
bash <(curl -fsSL https://project.pages.drupalcode.org/one_line_installer/drupalaibp)
```

Maintainers who need to regenerate the committed script require the Bashly gem at the pinned version:

```bash
gem install bashly -v "$(cat .bashly-version)"
bin/generate   # regenerates drupalaibp from src/
```

The `-yolo` flag from 1.0.x is now `--yolo` (double-dash). Config-driven builds use a `.drupalaibp.json` placed in the launch directory; see `docs/config.md` and `docs/examples/` for the full schema.
