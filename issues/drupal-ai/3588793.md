# #3588793: [Sprint 6] Canvas content templates not used for content-type migrations

## Summary

Sites migrated with the `canvas_ai_migrations` migrate-site skill can now support self-service content creation for dynamic content types such as events, blog posts, and news items. Previously, the skill treated all content as one-off Canvas page entities, meaning editors could not create a new event from the Drupal admin UI and have it render with Canvas styles without re-running a migration script. This feature adds a Phase 4.5 to the skill that detects dynamic content types during Phase 1 analysis, then creates Canvas content templates bound to Drupal node bundle display modes (`node.<type>.full`), so any newly created node of a flagged type renders correctly by default.

## Impact

Affects module maintainers.

- **New API:** New `.claude/agents/canvas-content-template-agent.md` adapter agent and Phase 4.5 conditional phase added to the migrate-site skill's `SKILL.md`.

## Technical details

The skill previously used only Canvas page entities (explicit per-item pages created by the migration), ignoring Canvas content templates, which are display-mode-level registrations (`node.<type>.full`) that automatically apply a Canvas component layout to any node of a given bundle. The fix has four parts.

**Phase 1 detection** was added to `site-analyzer.md`: it now flags dynamic content URL patterns (`/blog/*`, `/events/*`, `/news/*`, `/people/*`, paginated date-based paths) and writes `template_needed: true` entries into `plan.md`.

**Phase 4.5 (conditional)** was inserted into `.claude/skills/migrate-site/SKILL.md` between Phase 4 (upload) and Phase 5 (media). It skips entirely when `plan.md` has no `template_needed: true` entries. When it runs, it dispatches the new `canvas-content-template-agent` for each flagged type, verifies rendering by creating a test node via the admin UI, and deletes any legacy per-item Canvas pages created as temporary workarounds (archiving via path-alias change is explicitly avoided because it creates stale `path_alias` entries and permanent 301 redirect traps).

**`canvas-content-template-agent.md`** is a thin adapter agent (Sonnet model, low-effort setting) that reads Acquia Nebula's `canvas-content-templates` SKILL.md for format conventions and field-expression syntax, accepts a content type name and component-to-field mapping from the orchestrator, produces a template JSON file at `content-templates/{entityType}.{bundle}.{viewMode}.json`, validates it with `npx canvas validate`, uploads via browser automation (admin-cookie session, because the `page_template:administer` OAuth endpoints returned 404 during an empirical scope probe on 2026-05-01), and verifies by creating a test node. The agent switches to OAuth once `scripts/verify-scopes.mjs` reports those endpoints are live.

**Nebula as a devDependency** is the structural mechanism: `package.json` gains `"acquia-nebula": "github:acquia/nebula"` and a `postinstall` symlink (`ln -sfn ../../node_modules/acquia-nebula/.agents/skills .claude/reference/nebula`), making all 24 Nebula skills available at `.claude/reference/nebula/` and refreshable via `npm update`.

A `docs/migration/gotchas.md` entry was added noting that dynamic content types require Canvas content templates and that new content requires Phase 4.5, not a manual script. Four tests in `tests/skills/sprint6-skill-architecture.test.mjs` cover the Nebula symlink, agent file, Phase 4.5 wiring, and site-analyzer detection. Commits: 8ef9c5c, 54718ae, 1a9b582.

## Upgrade

No API-breaking changes. Existing migrations that processed only static pages are unaffected (Phase 4.5 skips when `plan.md` has no `template_needed: true` entries).

For projects that have already migrated a site with dynamic content types:
1. Run `npm install` to pull Acquia Nebula as a devDependency and establish the `.claude/reference/nebula/` symlink.
2. Re-run Phase 1 analysis against the source site to populate `template_needed: true` entries in `plan.md`.
3. Run Phase 4.5 manually or re-run the full skill; it will dispatch `canvas-content-template-agent` for each flagged type.
4. Delete any per-item Canvas page entities that were created as workarounds (do **not** archive via path-alias change — see `docs/migration/gotchas.md`).
