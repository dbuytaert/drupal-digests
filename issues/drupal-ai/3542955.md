## Summary

AI agents in Drupal CMS can now be extended through recipes and modules without modifying the original agent configuration. Site builders can install recipes that add specialized capabilities (like a Matomo analytics assistant) by creating override configurations that attach tools, adjust instructions, or refine behavior. These overrides are toggled through the agent edit form and remain intact when the base agent updates, allowing communities to share agent extensions as distributable packages.

## Impact

No upgrade or configuration changes required.

## Technical details

This change introduces an `ai_agent_override` config entity that allows recipes and modules to extend existing AI agents. The `AiAgentOverrideApplier` service applies enabled overrides at runtime when agents are instantiated through `AiAgentManager::createInstance()`, creating modified copies without altering the underlying entity.

Overrides support tool management through `tools_add` and `tools_remove` properties, prompt modification via three strategies (prefix, suffix, or token replacement), and granular control over tool settings and usage limits. Multiple overrides can target the same agent and are processed in weight order, allowing layered customization.

The agent edit form displays active overrides in a dedicated section where administrators can enable or disable them. A visual enforcement summary shows which tools are controlled by overrides using color-coded borders (green for forced-enabled, red for forced-disabled). The agent collection list builder adds an "Overrides" column showing which overrides are active for each agent.

The `AiAgentOverride` entity validates that parent agents exist when `fail_if_parent_missing` is true, throwing `ParentAgentNotFoundException` if validation fails. Tool settings and usage limits are merged recursively when overrides are applied. The `AiAgentCollectionFormAlter` hook class injects override information into the agent listing table using the new Hook attribute system.

## Lessons learned

- When enabling third-party extensions to modify core functionality through configuration, implement an override entity pattern rather than direct modification. This allows extensions to declare their changes declaratively while preserving the ability for the core entity to be updated without losing extension modifications. *(via jibran, marcus_johansson)*
- When designing extensibility systems for distribution ecosystems, prioritize configuration-only solutions over code-based extensions. This enables broader adoption by reducing the technical barrier and making extensions compatible with declarative deployment tools like recipes. *(via marcus_johansson)*
