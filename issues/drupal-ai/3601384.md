# #3601384: Add an opt-in glab (GitLab CLI) extra: install and authenticate to git.drupalcode.org in the web container

## Summary

The `one_line_installer` project gains a new opt-in extra called `glab` that installs the GitLab CLI (pinned at v1.64.0) into the DDEV web container and authenticates it to `git.drupalcode.org` with a Personal Access Token you paste during setup. Once selected, the tool prompts you to create an `api`-scoped PAT at drupal.org, stores it in `.ddev/.env` (chmod 600, gitignored), bakes `glab` into the web image so it survives rebuilds, and writes a DDEV post-start hook that re-authenticates on every container start. After a single `ddev restart`, `glab mr list`, HTTPS `git clone`, and `git push` to `git.drupalcode.org` all work from inside the container. Because the extra requires an interactive token paste, it is never silently installed by `--yolo` or unattended config-driven runs.

## Technical details

The extra lives in `src/lib/extras/extra_glab.sh` and follows the project's `extra_<id>_meta` + `extra_<id>` function-pair convention, so it is auto-discovered by `_extras_discover` and appears in the installer's "Any extra Drupal installations?" multi-select without any other file needing to change.

Several sharp edges are addressed by design. The token is stored as `DRUPALCODE_GITLAB_TOKEN` rather than `GITLAB_TOKEN`: glab treats `GITLAB_TOKEN` as a global override applied to every host, which causes 401s against gitlab.com and makes `glab auth status` exit non-zero even when drupalcode auth works. Feeding the token through `glab auth login --stdin` instead writes it into glab's own per-host config, scoped to `git.drupalcode.org` alone. The extra also strips any pre-existing `GITLAB_TOKEN` or `DRUPALCODE_GITLAB_TOKEN` lines from `.ddev/.env` on re-run to undo contamination from earlier versions.

The DDEV post-start hook in `.ddev/config.glab.yaml` additionally runs `git config --global 'credential.https://git.drupalcode.org.helper' '!glab auth git-credential'`, because `glab auth login --stdin` does not wire the credential helper automatically (only the interactive prompt does). DDEV merges post-start hooks across `config.*.yaml` files, so the hook coexists cleanly with the coding-agent add-ons' own post-start hooks.

Verification uses `ddev exec glab api user` rather than `glab auth status`, because `glab auth status` always probes gitlab.com and reports failure even when `git.drupalcode.org` auth is fully functional. The extra's meta line has no `|default` suffix, so it is never pre-ticked or silently installed by `--yolo`. `_glab_read_token` is factored out into its own function so the bats suite can stub it instead of feeding keystrokes, following the project's "prompts are functions" testing rule.
