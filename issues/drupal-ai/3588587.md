## Summary

The `drupal-devkit` tool now supports harness-native plugin installation for Codex CLI and Gemini CLI users, not just Claude Code. Before this change, the `drupal-devkit harness-plugin install` command had no available backend for Codex or Gemini users even though both harnesses were listed in the README as supported. The `drupal-workflow` marketplace entry is now available under all three harnesses.

## Impact

Affects module maintainers.

- **New API:** Two new plugin backends registered under type keys `"codex-marketplace"` and `"gemini-extension"` in the harness plugin registry

## Technical details

The `HarnessPluginBackend` protocol in `harness_plugins/protocol.py` already defined the interface (`is_available`, `install`, `uninstall`, `list_installed`, `info`). This commit adds two concrete implementations alongside the existing `ClaudeCodePluginBackend`.

`CodexPluginBackend` (type `"codex-marketplace"`) manages plugins by combining a CLI call to `codex plugin marketplace add` to register the upstream marketplace source with direct manipulation of `~/.agents/plugins/marketplace.json`, the file Codex reads on startup. Because individual plugin install/list in Codex is TUI-only as of v0.130+, the backend writes plugin entries directly into that JSON file. If `_register_plugin` raises after a marketplace was added during the same run, the backend attempts a rollback via `codex plugin marketplace remove`; it correctly skips rollback when the marketplace was already present before the run.

`GeminiPluginBackend` (type `"gemini-extension"`) delegates entirely to the `gemini extensions install/uninstall/list` subcommands available in Gemini CLI v0.38+. A notable quirk is that `gemini extensions list -o json` writes its JSON array to stderr rather than stdout, and may prefix it with non-JSON lines such as theme registration notices. The backend handles this with `_parse_json_array`, which strips a leading non-JSON prefix before attempting to decode.

Both backends are registered in `harness_plugins/__init__.py` under their respective type keys in the `BACKENDS` dict, and corresponding marketplace entries (`"drupal-workflow-codex"` and `"drupal-workflow-gemini"`) are added to `default_marketplaces.json` so the registry can resolve them by `harness` + `type`.
