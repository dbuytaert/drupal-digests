## Summary

The `mcp_client` module replaces its underlying transport library, swapping the third-party `swisnl/mcp-client` for the official `modelcontextprotocol/php-sdk` (`mcp/sdk`). The previous library required multiple reflection hacks to work around ReactPHP lifecycle issues that caused "Connection aborted early" errors on Drupal 11.3+. The new SDK is synchronous, built on Symfony HttpClient and Process, and eliminates all of those workarounds. The public interface of `MCPClient` (its `listTools()`, `executeTool()`, and `disconnect()` methods) is unchanged, so tool plugins work as before. Note that `mcp/sdk` is still at version `0.6`; the module maintainer has announced that only beta releases will be made until the upstream SDK reaches 1.0.

## Impact

Affects module maintainers.

- **API change:** `Drupal\mcp_client\Transporter\CustomHeadersHttpTransporter` has been removed; it was public but was an implementation detail of `MCPClient`.

## Technical details

The `swisnl/mcp-client: ^0.5` Composer dependency (and the `minimum-stability: dev` flag it required) is replaced with `mcp/sdk: ^0.6`. `MCPClient` is rewritten to use `\Mcp\Client`, `\Mcp\Client\Transport\HttpTransport`, and `\Mcp\Client\Transport\StdioTransport`. The `CustomHeadersHttpTransporter` class that previously subclassed `StreamableHttpTransporter` to inject custom HTTP headers and override `initializeConnection()` to strip problematic `register_shutdown_function()` calls is deleted entirely. Custom headers now pass through `HttpTransport`'s constructor as a supported first-class argument.

Stderr capture previously relied on `\ReflectionClass` to read the protected `transporter` property of the Swis client. This is replaced by the new `BufferingLogger` class (a `Psr\Log\AbstractLogger` decorator) that wraps any injected logger, retains the last ten log records in memory, and exposes `getStderrErrors()` to extract lines the SDK's `StdioTransport` logs under the message `"Server stderr"`. The `BufferingLogger` instance is created inside `initializeClient()` and passed to both the transport and the SDK `Client::builder()`, so stderr is captured without any reflection.

`listTools()` now follows the MCP pagination cursor in a bounded loop (capped at `MAX_TOOL_PAGES = 100`), treating a `null` or empty-string cursor as terminal. An empty JSON object that the SDK preserves as `\stdClass` (for example an `inputSchema['properties']` of a parameterless tool) is normalized to a plain PHP array via a `json_encode`/`json_decode` round-trip in `normalizeToArray()`. `executeTool()` uses an `instanceof TextContent` check rather than `method_exists()` probing; other content types go through `jsonSerialize()` and the same normalization. `McpToolBase::doExecute()` is updated to look up tools via the `ToolCollection` API (`$server->getTool($tool_name)`) rather than the removed name-keyed `tools` array and the separate `enabled_tools` property. The STDIO transport environment variable handling now merges configured variables over the current parent environment (via `array_merge(getenv(), $config['env'])`) rather than replacing it, preserving `PATH` and other inherited variables.

## Upgrade

The public class `Drupal\mcp_client\Transporter\CustomHeadersHttpTransporter` has been removed. Any code that extended or directly instantiated it must be updated.

Before:
```php
use Drupal\mcp_client\Transporter\CustomHeadersHttpTransporter;
// ...
$transporter = new CustomHeadersHttpTransporter($endpoint, $headers, $logger);
```

After: custom HTTP headers are passed directly through `MCPClient`'s configuration array via `McpClientFactory`, and the transport is created internally. There is no supported replacement class to extend; inject headers through the module's server configuration instead.
