## Summary

CKEditor AI tools (Fix Spelling, Modify with a Prompt) were returning HTTP 403 errors on servers running ModSecurity with the Core Rule Set enabled. The problem was that the browser POST request carried no `Content-Type` header, causing the browser to default to `text/plain`, which ModSecurity rule 960010 rejects. Adding `Content-Type: application/json` to the fetch call fixes the rejection. The fix was cherry-picked to all supported branches (1.1.x, 1.2.x, 1.3.x, 2.0.x).

## Impact

No upgrade or configuration changes required.

## Technical details

The AI CKEditor sub-module provides CKEditor 5 commands implemented in `modules/ai_ckeditor/js/ckeditor5_plugins/aickeditor/src/Commands/AiWriter.js`. The `execute()` method uses the browser `fetch()` API to POST `JSON.stringify(request_parameters)` to the Drupal endpoint at `/api/ai-ckeditor/request/...`. Because no explicit `Content-Type` header was set, the browser treated the body as `text/plain`. ModSecurity's CRS rule 960010 ("Request content type is not allowed by policy") blocks `text/plain` POST bodies, resulting in a 403 before the request ever reached Drupal, with nothing logged in either the Drupal watchdog or the AI module's logging collection.

The fix adds a `headers` object to the `fetch()` options with `"Content-Type": "application/json"`. This is the correct type because the body is already serialised with `JSON.stringify()`. The streamed response from Drupal is unaffected; it uses `text/event-stream` as set server-side in `AiRequest.php` and is not changed by this patch.
