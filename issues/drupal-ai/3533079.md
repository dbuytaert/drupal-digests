## Summary

Drupal Canvas AI can now generate complete page templates from a single prompt, assembling headers, footers, and body content automatically. Users can describe the page they want (like "create a homepage for a pizza shop with header and footer") and the system builds the entire layout using available components. Site builders can configure how different page regions should be used to guide the AI's decisions when placing components across headers, sidebars, footers, and content areas.

## Impact

No upgrade or configuration changes required.

## Technical details

This feature introduces a new AI sub-agent, `canvas_template_builder_agent`, to the Canvas AI orchestrator. The agent generates complete page layouts in YAML format by strategically selecting and arranging components across multiple regions. It receives context about available components via `GetComponentContext`, the current page layout via `GetCurrentLayout`, and region descriptions via a new `[canvas_ai:available_regions]` token.

The new `SetAIGeneratedTemplateData` function call tool processes the YAML output. It validates region names against the current layout, checks component structure integrity using `AiResponseValidator`, and converts the YAML into operations the `addNewComponentToLayout` thunk can execute. The helper function `CanvasAiPageBuilderHelper::processSetTemplateDataToolInput()` handles the conversion by calculating nodePaths for each component based on region indices and component positions.

A new configuration form (`CanvasAIThemeRegionSettingsForm`) at `/admin/config/ai/canvas-ai-settings` allows site builders to provide descriptions for enabled theme regions. These descriptions help guide the AI's component placement decisions. The configuration is stored in `canvas_ai.theme_region.settings` with a fully validatable schema.

The orchestrator prompt was updated to distinguish between the existing `canvas_page_builder_agent` (for incremental additions to existing pages) and the new `canvas_template_builder_agent` (for creating complete pages from scratch). Both agents share identical prop-setting logic, particularly the strict requirement to use exact default values for image props. The Prop Setting Logic sections in both agent system prompts must remain synchronized.

Helper methods `getRegionIndex()` and `getAvailableRegions()` were added to retrieve region information from layouts. The verbose context for the orchestrator now includes instructions to generate page titles and descriptions when they're empty, triggered for both page builder and template builder operations. Status messages distinguish between "Finding components to place" (page builder) and "Designing the page" (template builder).

A post-update hook rebuilds routes since Canvas AI route paths changed from `/admin/config/system/canvas-ai-*` to `/admin/config/ai/canvas-ai-*`.

## Lessons learned

- When designing AI agents that generate structured layouts, provide the agent with contextual configuration about available regions and their purposes. Site builders can describe how regions should be used (e.g., 'Place sidebar menu in this region when generating templates'), enabling the AI to make informed placement decisions without hardcoding assumptions. *(via akhil babu)*
- When building multi-agent AI systems, separate concerns by creating specialized agents for distinct tasks rather than one monolithic agent. An orchestrator can delegate to a template builder for full page generation and a page builder for incremental additions, allowing each agent to focus on its core responsibility with tailored prompts and tools. *(via akhil babu)*
- When AI agents modify persistent data structures like page layouts, validate the AI-generated output before processing it. Structured formats like YAML enable validation of component references, region names, and prop schemas before adding components to the page, preventing invalid states from AI hallucinations. *(via akhil babu)*
- When AI behavior varies significantly across different LLM providers or model versions, document which models exhibit known issues. Testing should verify functionality across the primary supported models, as capabilities like following instructions to avoid certain actions can differ substantially between providers. *(via akhil babu, vipin.mittal18)*
