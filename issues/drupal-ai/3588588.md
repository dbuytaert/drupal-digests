## Summary

The `drupal_devkit` tool now includes a Gemini CLI backend for harness-native plugin installation. Previously, Gemini users could only install skills through the marketplace path; there was no way to install harness-native plugins (the equivalent of what Claude Code users get via `ClaudeCodePluginBackend`). The new backend brings `install`, `uninstall`, and `list` operations to Gemini CLI, closing that gap. Note that only repositories shipping a `gemini-extension.json` manifest are supported, which is a Gemini CLI requirement rather than a devkit limitation.

## Impact

Affects module maintainers.

- **New API:** New `GeminiPluginBackend` class in `harness_plugins/gemini.py` exposes `install`, `uninstall`, and `list_installed` operations for Gemini CLI extensions.

## Technical details

`GeminiPluginBackend` is implemented in `src/drupal_devkit/harness_plugins/gemini.py`, following the same shell-out pattern as `ClaudeCodePluginBackend`. It delegates to the `gemini extensions install`, `gemini extensions uninstall`, and `gemini extensions list` CLI subcommands, parses JSON output, performs idempotent installs, and rolls back on failure. Repos without a `gemini-extension.json` manifest fail at install time with "Configuration file not found" from the Gemini CLI itself. Eleven unit tests in `tests/harness_plugins/test_gemini_backend.py` cover `is_available`, install happy path, already-installed detection, install failure, uninstall, `list_installed` (empty, malformed, and valid JSON with stderr-prefix handling), and `info`.
