## Summary

The `ai_validations` module gains a fifth field validation rule: "AI moderation constraint." Site builders can attach it to any text field and it will call the AI module's native `moderation()` operation on form save, blocking submission when the provider flags the content as harmful. The rule's configuration lets you choose a provider and model, optionally restrict failures to specific moderation categories (e.g. `hate,violence,sexual`), and set a confidence threshold for those categories. If the moderation provider throws an exception, the error is logged and the field fails validation rather than crashing the site.

## Impact

Affects module maintainers.

- **New API:** New `ai_moderation_constraint_rule` field validation rule plugin that screens text fields using the AI module's `moderation()` operation type.

## Technical details

Three new classes implement the constraint following the existing plugin pattern. `AiModerationConstraintFieldValidationRule` (plugin id `ai_moderation_constraint_rule`) provides the admin config form with an `ai_provider_configuration` element for provider/model selection, a free-text `categories` field (comma-separated, blank means any flagged category triggers a violation), and a `threshold` float field (0–1, default 0.5) applied when specific categories are configured. `AiModerationConstraint` is the constraint data class holding `$categories`, `$provider`, and `$threshold`. `AiModerationConstraintValidator` resolves the provider, skips empty/null values using `$data === NULL || (string) $data === ''` (fixing a prior `empty()` false-negative on `0`), then calls `$provider->moderation(new ModerationInput($data), $model)->getNormalized()`. If `isFlagged()` is true and no category filter is set, a violation is added immediately. With categories configured, the validator reads scores from `getInformation()` and adds a violation only when a watched category's score is numeric and exceeds the configured threshold. Provider exceptions are caught, logged via the `ai_validations` logger channel, and treated as a violation.

A separate fix in `AiConstraintValidatorBase::runBooleanChat()` wraps the `$provider->chat()` call in a try/catch that returns `FALSE` on any exception. Previously, providers that reject inputs via their own safety filter (before returning a response) would throw an uncaught exception and produce a 500 page; all three existing chat-based validators (text prompt, image prompt, audio) are affected by this fix.
