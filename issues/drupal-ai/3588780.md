## Summary

The `canvas_ai_migrations` project's component-fixer had no way to verify that a fix actually improved a component's visual fidelity versus introducing a regression. This sprint adds a lightweight per-component revision tracking system: before and after each fix attempt, screenshots and computed styles are captured and compared to the original source baseline. The agent can now determine whether a change was converging (post-fix is more similar to source), diverging (regression), or unchanged (fix missed the right element), and it halts with a revert instruction on diverging results.

## Impact

Affects module maintainers.

- **New API:** New `capture-component.sh` script and `compare-three-way.mjs` aggregator; `component-fixer` and `storybook-qa` agent workflows gain mandatory pre/post capture steps and a `revisions/revisions.jsonl` log per component.

## Technical details

Three new artifacts implement the system. `scripts/visual/capture-component.sh` is a bash script that issues a single `agent-browser batch` call per phase (`source`, `pre`, or `post`). For `source` phase it writes `docs/migration/components/<Name>/source.png` and `source-styles.json`. For `pre`/`post` phases it writes `revisions/<N>-{phase}.png`, `<N>-{phase}-styles.json`, and `<N>-{phase}-diff.json` (a pixel-diff against `source.png`). Batching all four browser actions into one call avoids repeated browser startup overhead.

`scripts/visual/compare-three-way.mjs` is the aggregator. It reads the two diff JSONs and three styles JSONs, converts `diffRatio` (0 = identical) to `pixel_similarity` (1 = identical), classifies `direction` as `converging`/`diverging`/`unchanged` using a 0.02-point band, and emits a JSON object with `revision`, `pixel_similarity_pre`, `pixel_similarity_post`, `direction`, and `css_deltas`. Each `css_deltas` entry carries a `regressed` flag that is `true` only when the pre-fix state matched source and the post-fix state does not, distinguishing regressions from properties that were already wrong.

The agent workflow in `.claude/agents/component-fixer.md` gains Step 3a (call `capture-component.sh --phase pre` before any code change) and Step 5a (call `capture-component.sh --phase post` after upload, then run `compare-three-way.mjs` and append a `revisions.jsonl` line). On `direction: "diverging"` the agent reverts with `git restore` and emits `BLOCKED: regression detected`. `.claude/agents/storybook-qa.md` is updated to call `capture-component.sh --phase source` once when the first issue for a component is filed, seeding the stable `source.png` baseline. The `visual-comparison` skill docs gain a dedicated `Three-way comparison` section documenting the full capture-aggregate-interpret flow. Vitest tests for `compare-three-way.mjs` and bash smoke tests for `capture-component.sh` are included; a new `visual` test suite is registered in `vitest.config.js`.

## Upgrade

No migration of existing components is required. To start tracking revisions for a component that already has issues, run the source-baseline capture once:

```bash
scripts/visual/capture-component.sh --component <Name> --revision 0 --phase source \
  --url <SOURCE_URL> --selector "<SECTION_SELECTOR>"
```

Subsequent `component-fixer` runs will then use `--revision 1`, `--revision 2`, etc. automatically (the agent derives `N` from the line count of `revisions/revisions.jsonl`). Components that have never had a `storybook-qa` issue filed since this change will need the source baseline captured manually before the fixer can produce a `direction` result.
