# #3601361: Replace the experimental tool prompts with a generic, script-driven "Extra Drupal Installations" multi-select

## Summary

The one-line installer now offers all optional Drupal tools through a single "Any extra Drupal installations?" multi-select prompt instead of separate yes/no questions for each tool. Three extras ship today: Drush Token Killer (`dtk`, pre-ticked and installed by default under `--yolo`), Tool API + Toolbelt (`tool_api`, opt-in), and the new Droost AI-agent MCP integration (`droost`, opt-in). Anyone using the old `--tool-api` flag must switch to `--extras tool_api`. Config files should use the new `extra_scripts` array key instead of per-tool booleans. Adding a new optional tool no longer requires editing prompts, globals, the pipeline, or config readers: dropping one file into `src/lib/extras/` and regenerating is all that is needed.

## Impact

Affects module maintainers and site owners.

- **Config change:** `.drupalaibp.json` gains `extra_scripts` (array of ids: `dtk`, `tool_api`, `droost`); the old per-tool boolean keys are no longer read.
- **API change:** The `--tool-api` boolean flag is removed; use `--extras tool_api` (or a comma-separated list) instead.
- **New API:** New `extra_installed <id>` event emitted after each successful extra install; new `src/lib/extras/` convention for adding optional tools as `extra_<id>_meta` + `extra_<id>` function pairs.

## Technical details

The old architecture had a dedicated prompt function (`choose_toolbelt`, `choose_dtk`, `choose_tool_api`), a per-tool global (`TOOL_API`, `TOOLBELT`, `DTK`), a per-tool install step (`step_install_tool_api`), and a per-tool boolean CLI flag (`--tool-api`) for every optional install. Each addition required changes across at least five places.

The new architecture introduces `src/lib/extras/` as a convention. Each extra defines exactly two bash functions: `extra_<id>_meta` (prints `Name|Description`, or `Name|Description|default` to mark it as a default) and `extra_<id>` (performs the install using `step`/`ok`/`warn` and `ddev`/`drush`). The single `_extras_discover` function finds them at runtime via `compgen -A function` scanning the bash function namespace, because the entire script is bundled by Bashly and extras cannot be globbed off disk. `choose_extras` (registered on `ask_questions` at weight 45) builds one `select_many` menu from the discovered extras, pre-ticking defaults via `MENU_PRESELECT`, and sets the `EXTRAS` array of ids. The generic `step_install_extras` (pipeline slot 10, replacing `step_install_tool_api`) iterates `EXTRAS`, calls `extra_<id>`, and emits `extra_installed <id>` on success. Failures warn and continue.

The `--tool-api` boolean flag is replaced by `--extras LIST` (comma-separated ids). Presence of the flag, not its value, overrides the defaults: `--extras ""` is a valid explicit "install nothing." The `.drupalaibp.json` config schema gains `extra_scripts` (array of ids), read into `EXTRAS` by `load_config.sh`. Old per-tool boolean keys (`toolbelt`, `dtk`) are not read; the project documents this as part of its 2.0 schema clean break. `export_config.sh` writes `extra_scripts` with the selected ids rather than expanding packages into `composer_projects`/`install_modules`, so configs no longer need to track package names separately from the extras themselves. A new `extra_installed` event joins the event registry alongside `ddev_addon_added` and `skill_added`.

Also in this commit: the Bashly generator version is pinned to a `.bashly-version` file and a `bin/generate` wrapper enforces the pin, preventing CI failures caused by version drift between local and CI generators.

## Upgrade

If you use the CLI flag, replace `--tool-api` with `--extras tool_api`:

```bash
# Before
drupalaibp --tool-api

# After
drupalaibp --extras tool_api
# Multiple extras (comma-separated)
drupalaibp --extras tool_api,droost
# Explicit "none" (overrides the dtk default)
drupalaibp --extras ""
```

If you have a `.drupalaibp.json` config that listed Tool API packages manually, replace those entries with the `extra_scripts` key:

```json
// Before (manual package listing)
{
  "composer_projects": ["drupal/tool", "drupal/tool_belt"],
  "install_modules": ["tool", "tool_belt", "tool_belt_content", "..."],
  "composer_allow_plugins": ["tbachert/spi"]
}

// After
{
  "extra_scripts": ["tool_api"]
}
```

To add a new optional tool, create one file in `src/lib/extras/` following this pattern, then run `bin/generate`:

```bash
# src/lib/extras/extra_mymodule.sh
extra_mymodule_meta() {
  printf '%s|%s\n' "My Module" "short description shown in the menu"
  # Add |default as a third field to pre-tick it
}

extra_mymodule() {
  ddev exec composer require drupal/mymodule --no-interaction || { warn "Could not require mymodule."; return 1; }
  ddev drush en -y mymodule || warn "Could not enable mymodule."
  ok "My Module installed."
}
```
