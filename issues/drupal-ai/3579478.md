## Summary

This feature integrates LiteLLM Proxy's server-side guardrail system with Drupal's AI module. Site administrators can sync guardrails configured in their LiteLLM proxy into Drupal as config entities, then attach them to guardrail sets. When a request is sent, the guardrail names are included in the API payload so the proxy enforces them server-side. This allows sites to use guardrails like PII masking or prompt injection detection without implementing the logic in Drupal.

## Impact

Affects module maintainers.

- **New API:** New `LiteLlmGuardrail` plugin (type `AiGuardrail`) for pass-through guardrails enforced by LiteLLM Proxy

## Technical details

LiteLLM Proxy exposes a `GET /guardrails/list` endpoint that returns guardrails configured in its `config.yaml`. Each guardrail has a name, type, and enforcement mode (pre_call, post_call, during_call). The proxy enforces guardrails when the request payload includes a `guardrails` array with the names to apply.

The new `LiteLlmGuardrail` plugin (`AiGuardrail` type) is a pass-through plugin. Its `processInput()` and `processOutput()` methods always return `PassResult` because Drupal does not enforce anything itself. The plugin stores only the `guardrail_name` from its configuration.

The sync flow lives in `LiteLlmAiConfigForm::syncGuardrails()`. It calls `LiteLlmAiClient::guardrails()`, which fetches the list from the proxy and extracts the `guardrail_name` field from each entry. For each name, the sync method generates a config entity ID by lowercasing, replacing non-alphanumeric characters with underscores, prepending `litellm_`, and truncating to 64 characters. If an entity with that ID exists, it is updated. Otherwise a new `ai_guardrail` entity is created with plugin type `litellm_guardrail`.

When `LiteLlmAiProvider::chat()` receives a `ChatInput` with a guardrail set, it walks the pre-generate and post-generate guardrails. For each `LiteLlmGuardrail` instance, it extracts the `guardrail_name` from the plugin configuration. The names are deduplicated and injected into `$this->configuration['guardrails']` before delegating to the parent `chat()` method (which passes configuration values into the request payload). The key is then removed from configuration so it does not bleed into subsequent calls.

A `hook_form_alter()` implementation adds a notice to guardrail set forms when LiteLLM guardrails are present, explaining that the pre/post distinction in Drupal is irrelevant because enforcement timing is controlled by the proxy configuration.
