## Summary

The `setup-site` command gains a `--config-path <dir>` flag, a third way to point the installer at a `.drupalaibp.json` alongside the existing `--global-config` (a named file in `~/.drupalaibp/configs/`) and `--config-url` (a git-cloned repo). Passing a local directory path â€” such as a folder checked into the current project â€” loads its root `.drupalaibp.json`, drives the build from it, and resolves sibling hook scripts and `copy_paths` assets relative to that directory. This lets teams share a config by committing it to their repo and passing its directory at build time, without copying the file into each developer's global config store. The flag is local-only and intentionally has no network or URL support; for remote configs, `--config-url` is the appropriate tool.

## Impact

Affects module maintainers.

- **New API:** New `--config-path <dir>` flag on `setup-site` builds from a `.drupalaibp.json` in a local directory, mutually exclusive with `--config-url` and `--global-config`.

## Technical details

The implementation lives in two new files: `src/lib/config/_resolve_config_path_source.sh` (`_resolve_config_path_source`) validates that the argument is an existing directory containing a `.drupalaibp.json`, then sets `RESOLVED_CONFIG_SOURCE` to its absolute path. `src/lib/config/load_config_path.sh` (`load_config_path`) calls that resolver and then hands the resolved path to the existing `drupalaibp_load` machinery, which populates all choice globals and registers config hooks exactly as it does for other load paths. `setup_site_command.sh` dispatches to `load_config_path` when `--config-path` is present and `PROJECT_CONTEXT == none`.

Compared to `--global-config`, `--config-path` sets only `USE_CONFIG=1`, not `YOLO=1`: the config's own `questions` still prompt interactively unless `--yolo` is passed separately. `CONFIG_BASEDIR` is set to the given directory (via the normal `drupalaibp_load` call), so `hooks[].script` and `copy_paths` entries resolve relative to it and are copied into the project by the existing `_copy_hook_scripts`/`_copy_config_paths` machinery. No key restriction, trust gate, or HTTPS fetch is involved; those concerns belong to `--config-url`.

`--config-path` is mutually exclusive with `--config-url` and `--global-config`; passing more than one of the three causes an immediate `die`. The flag is silently rejected in management mode (it only applies when `PROJECT_CONTEXT == none`). The bats suite covers the happy path, `CONFIG_BASEDIR` positioning, a missing `.drupalaibp.json`, and a nonexistent directory in `tests/bats/lib/config/load_config_path.bats` and `_resolve_config_path_source.bats`.
