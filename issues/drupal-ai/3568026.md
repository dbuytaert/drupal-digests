## Summary

The AI Default Settings form at `/admin/config/ai/settings` was redesigned from a fieldset-based layout to a table-based layout. Each AI capability (chat, image generation, etc.) now appears as a compact row with three columns: Capability, Default Provider, and Default Model. The change makes the form easier to scan, preserves all existing provider and model selection functionality including AJAX model loading and a no-JavaScript fallback, and lays the groundwork for additional metadata columns in follow-up work.

## Impact

Affects front-end developers.

- **Theme change:** The AI Default Settings form at `/admin/config/ai/settings` now renders each capability inside a `#type => 'table'` row instead of a `#type => 'fieldset'`; custom CSS targeting the old fieldset structure will need to be updated.

## Technical details

The sole file changed was `src/Form/AiSettingsForm.php` in the AI Core module. In `buildForm()`, a `#type => 'table'` element was added at `$form['default_providers']['table']` with three column headers ("Capability", "Default Provider", "Default Model"). Each operation type that previously occupied its own `#type => 'fieldset'` container was moved into a table row with three cells: a `label` cell using `#markup` to render the capability name as `<strong>` text and its description as a `<span class="form-item__description">`, a `provider` container cell holding the provider `select` element and the no-JS "Choose Model" submit button, and a `model` container cell holding the AJAX-wrapped model `select`. The provider and model `select` elements gained explicit `#name` and `#parents` properties to preserve correct form value submission within the deeper nesting, and `#title_display => 'invisible'` since column headers replace per-element labels. All references to `$form['default_providers'][$operation_type]` in the `loadModels()` AJAX callback were updated to `$form['default_providers']['table'][$operation_type]` to match the new tree. The no-JS fallback path that strips `#ajax` and reveals the submit button was updated in parallel. This issue covers only the structural table conversion; follow-up issue [#3568028](https://www.drupal.org/node/3568028) adds a provider info cell and a YAML-based provider registry on top of this foundation.
