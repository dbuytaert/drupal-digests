## Summary

The `document_loader` module now ships two new optional submodules. `document_loader_mdx` adds a "Load Document" button to the MDX editor toolbar (used by the `ai` module). Clicking it opens a modal where editors can select a media file from the Drupal media library, enter a URL or Drupal file reference, or supply JSON API options (method, headers, body). The loaded content is sanitized for MDX compatibility and inserted directly into the editor without a page reload. `document_loader_media` is a companion submodule that provides the reusable media library opener used by the dialog. Requires AI module 1.4 or higher; the install hook checks for compatibility and uninstalls itself if the requirement is not met.

## Technical details

The `document_loader_mdx` submodule injects a toolbar button using `Drupal.behaviors.documentLoaderMdxButton`. Because the MDX editor is an async React app, the behavior uses a `MutationObserver` to detect when the `.mdxeditor-toolbar` element appears, then inserts a `<button>` imperatively before the last toolbar child (the diff/source toggle group). A second `MutationObserver` on the toolbar watches for `toolbarTitleMode` class changes so the button is hidden in diff/source mode and re-positioned when switching back to rich-text mode.

`DocumentLoaderMdxForm` is a `FormBase` modal served at `/document-loader/mdx/dialog`. It calls `DocumentLoaderManager::discoverSourceCategories()` to dynamically enumerate all available loaders, filters to those that produce `markdown` or `text` output, and renders the active category's fields. For the `file` category it renders a media library widget backed by a custom `document_loader_media.media_library.opener` service; this avoids the `UploadedFile` PHP-serialization problem that would arise from using `#type => 'managed_file'` in an AJAX modal. For URL inputs, `resolveFileUrlInput()` inspects the URL path extension against `DocumentLoaderManager::collectAllSupportedExtensions()` and redirects matching URLs to `file_input` so they are downloaded as files rather than scraped as web pages. The load is dispatched to `DocumentLoaderManager::loadFromData()` with `caller: 'mdx_editor'` (enabled by [#3585786](https://www.drupal.org/node/3585786)). `sanitizeForMdx()` strips HTML comments, raw HTML tags, and image syntax before the content is returned.

The AJAX response uses `FillMdxEditorCommand`, which implements `CommandInterface::render()` returning `command: 'documentLoaderFillMdxEditor'` and the target textarea name. The matching JS handler in `commands.fill_mdx_editor.js` sets `textarea.value`, dispatches `input` and `change` events, and dispatches the `drupal:mdx-fill` CustomEvent that the patched AI module listens for to call `editorRef.current.setMarkdown()` and visually re-render the editor (see [#3581363](https://www.drupal.org/node/3581363)). The modal itself uses a custom `#dl-document-loader-dialog` selector rather than `#drupal-modal` so the nested media library modal does not inadvertently close the parent dialog.

## Upgrade

No changes to existing APIs or configuration. Enable the new submodules to opt in:

```bash
drush en document_loader_media document_loader_mdx
drush cr
```

AI module 1.4 or higher is required. The `document_loader_mdx_install()` hook checks for the `drupal:mdx-fill` string in the compiled `ui/mdxeditor/dist/assets/main.js` and uninstalls the module automatically if the requirement is not satisfied.
