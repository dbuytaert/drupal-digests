## Summary

A new `document_loader_mdx` submodule has been added to the Document Loader project. It places a "Load document" button in the MDX editor toolbar (provided by the `ai` module). Clicking the button opens a modal form where an editor can upload a file, enter a URL or Drupal file reference, optionally supply JSON API options (method, headers, body), and choose an output format (Markdown, plain text, JSON, or CSV). On submit, the document loader fetches and converts the content, then inserts it directly into the MDX editor both for immediate visual display and for persistence on the next form save. The submodule requires a small patch to the `ai` module's React source; instructions and the patch file are shipped inside the module under `patches/`.

## Impact

Affects module maintainers and site owners.

- **Requirements:** Requires a patch to the `ai` module's React source (`patches/ai-mdx-editor-drupal-fill-event.patch`) and a JS bundle rebuild before the submodule functions.
- **New API:** New `FillMdxEditorCommand` PHP AJAX command class and `documentLoaderFillMdxEditor` JS command for injecting content into MDX editor instances from AJAX responses.

## Technical details

The MDX editor is a React application that renders asynchronously and exposes no PHP alter points for adding toolbar buttons. The submodule works around this entirely from the browser side.

`PageHooks::pageAttachments()` (via `hook_page_attachments`) attaches the `mdx_editor_button` library and injects `drupalSettings.documentLoaderMdx.dialogUrl` and `iconPath` on every page. `js/mdx-editor-button.js` registers a `Drupal.behaviors` attach handler that uses `once()` on every `textarea[data-mdxeditor]`. Because React renders the toolbar asynchronously, a `MutationObserver` on the editor wrapper waits for the `.mdxeditor-toolbar` element before calling `insertButton()`. The button is inserted via `toolbar.insertBefore(button, toolbar.lastElementChild)` to sit just before the diff/source toggle group. A second `MutationObserver` on the toolbar detects mode changes (by checking for the `toolbarTitleMode` class) and hides the button in diff and source modes, then re-positions it when rich-text mode is restored. The button mirrors the native toolbar DOM structure and uses a custom DOM tooltip (`dl-mdx-tooltip`) with the same Radix UI CSS custom properties (`--baseText`, `--baseBase`, `--radius-medium`, `--text-xs`) because the React `@radix-ui/react-tooltip` component cannot be instantiated imperatively.

Clicking the button calls `Drupal.ajax({ dialogType: 'modal', url: dialogUrl, submit: { editor_id, textarea_name } }).execute()`, which opens `DocumentLoaderMdxForm`. The form uses a two-step file upload: a plain `#type => 'file'` input plus a separate AJAX "Upload" button (`handleUpload()`) that moves the file to `temporary://`, creates a status-0 `File` entity (required by `FileInputNormalizer`), and stores the URI in form state. This avoids a serialization failure that `#type => 'managed_file'` causes in AJAX modal contexts because `UploadedFile` objects cannot be PHP-serialized into the form cache. Validation warnings (e.g. both file and URL supplied) are stored via `$form_state->set('form_warnings')` rather than `setError()`, which would otherwise render messages on the main page behind the modal. `resolveInputData()` routes the form values to the correct `DocumentLoaderManager::loadFromData()` input key: file upload, API call (when JSON API options are present), Drupal URI/file ID, `.parquet` URL, known document extension URL, or plain URL.

The loaded content passes through `sanitizeForMdx()`, which strips HTML comments, raw HTML tags, and markdown image syntax, because the MDX editor's JSX parser rejects constructs that CommonMark accepts.

On success the PHP side returns an `AjaxResponse` containing `CloseModalDialogCommand`, a `FillMdxEditorCommand`, and a `MessageCommand`. `FillMdxEditorCommand` (in `src/Ajax/`) is a `CommandInterface` that emits `{ command: 'documentLoaderFillMdxEditor', textareaName, content }`. The corresponding JS handler in `js/commands.fill_mdx_editor.js` sets `textarea.value`, dispatches `input` and `change` events, then fires the `drupal:mdx-fill` CustomEvent.

Visually refreshing the editor requires calling `MDXEditorMethods.setMarkdown()`, which is only accessible through a React ref that was internal to `Editor.jsx`. The included patch (`patches/ai-mdx-editor-drupal-fill-event.patch`) adds an `onRef` callback prop to `Editor.jsx` so `main.jsx` can capture the ref, then adds a `drupal:mdx-fill` listener on the textarea that calls `editorRef.current.setMarkdown(event.detail.content)`. This patch was tracked upstream as [#3581363](https://www.drupal.org/node/3581363). The related issue [#3585786](https://www.drupal.org/node/3585786) added pre/post-load hooks to `DocumentLoaderManager::loadFromInput()` that this module depends on for the `caller: 'mdx_editor'` context.

## Upgrade

Enable the new submodule after applying the patch to the `ai` module:

```bash
# 1. Apply the source patch to the ai module
cd web/modules/contrib/ai
patch -p1 < /path/to/document_loader_mdx/patches/ai-mdx-editor-drupal-fill-event.patch

# 2. Rebuild the JS bundle
cd ui/mdxeditor
npm install && npm run build

# 3. Verify the patch is in the built output
grep -c "drupal:mdx-fill" dist/assets/main.js
# Expected: 1

# 4. Enable the module and clear caches
drush en document_loader_mdx && drush cr
```

Re-apply the patch and rebuild whenever the `ai` module is updated via Composer.
