## Summary

The agui chat component gains the ability to switch between conversation threads at runtime, which enables history features in larger AI-powered applications. Developers can now programmatically move a chat session to any past or new thread ID, and the backend (e.g. Agno) will recall the matching conversation history. Two companion utilities on the global `AguiTools` API let integrators restore or discard historical tool call UI after a thread switch.

## Impact

Affects module maintainers.

- **New API:** `AguiChatCore.switchThread(newThreadId, clearUI?)`, `AguiChatCore.threadId` getter, `window.AguiTools.replayToolCall()`, and `window.AguiTools.removeToolCall()` added; `agui:thread:switched` CustomEvent dispatched on `window` after every thread switch

## Technical details

The feature adds three related capabilities across `AguiChatCore` and the `ToolRegistry`.

**Thread switching (`AguiChatCore`)**
`switchThread(newThreadId: string, clearUI = true)` is a new public method on `AguiChatCore` (in `components/chat/src/js/core/index.ts`). It stops any in-progress generation via `stopCurrentRun()`, instantiates a fresh `HttpAgent` using the supplied `threadId`, calls `resetMessageState()` and clears `customRenderedTools`, optionally emits the existing `chat:cleared` event to wipe the UI, and then dispatches a global `agui:thread:switched` `CustomEvent` on `window` with `detail: { threadId }` so external listeners can react. A companion `get threadId(): string` getter exposes the active thread ID at any time.

The `ChatCoreInterface` type (used by tool renderers to interact with the chat without importing the full core) is extended with `switchThread` and `readonly threadId` so renderers can trigger a thread switch themselves.

**Tool call replay (`window.AguiTools.replayToolCall`)**
`replayToolCall(toolCallId, toolName, args?, result?)` is a new method on `ToolRegistry` and the public `AguiToolsAPI`. It renders a tool call directly in `'complete'` status, bypassing the `'pending'` and `'streaming'` intermediate states. This is intended for history restoration: after switching to an older thread, callers replay the stored tool results so the UI reflects the historical output without re-executing the tool. It respects `targetContainerId` and `containerClass` from the registered renderer config, and adds the new container to `activeToolCalls` for lifecycle tracking.

**Tool call removal (`window.AguiTools.removeToolCall`)**
`removeToolCall(toolCallId)` removes a tracked tool call from the DOM and from `activeToolCalls`, calling the renderer's `cleanup` handler if one is defined. A `clearAll()` helper on `ToolRegistry` iterates all active calls and removes each one, used internally when the chat is cleared or a thread switch discards the previous session's tool UI.

All three additions are reflected in the generated type declarations under `components/chat/types/`.
