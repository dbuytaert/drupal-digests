## Summary

The `ai_agents.agent_status_poller` service could not report any information about default information tool invocations. When an agent used a default information tool, the poller returned no status, arguments, or completion data about that call. After this fix, the poller captures and exposes the full execution lifecycle for both standard agent-decided tools and default information tools, enabling use cases such as showing which context was loaded in a chat UI.

## Impact

Affects module maintainers.

- **New API:** `AgentToolPreExecuteEvent` and `AgentToolFinishedExecutionEvent` now carry an `isAgentDecision()` method to distinguish standard tool calls from default information tool invocations.

## Technical details

The `AgentStatusSubscriber` (in the `ai_agents` module) subscribes to `AgentToolPreExecuteEvent` and `AgentToolFinishedExecutionEvent` to feed tool execution data into the `ai_agents.agent_status_poller` service. Before this change, the event dispatch was guarded by an `if ($agent_decision)` check, which silently skipped event dispatch whenever a default information tool was invoked (since those calls are not classified as agent decisions).

Three coordinated changes were made:

1. The `if ($agent_decision)` guard was removed so that `AgentToolPreExecuteEvent` and `AgentToolFinishedExecutionEvent` fire for all tool invocations, including default information tools.

2. The `$agent_decision` value is now passed into both tool events. Consumers can call `isAgentDecision()` on the event object to distinguish a standard agent-decided tool call from a default information tool call.

3. Because default information tools do not carry a persistent tool ID, a unique identifier is generated at dispatch time by combining the tool plugin ID, the runner ID, and the context values. This lets the poller correlate a pre-execute event with its matching finished-execution event.

`AgentStatusSubscriber` is also updated to read and push the arguments passed to default information tools into poller storage, so downstream consumers (such as a chat UI) can inspect what context or parameters the tool received.

A new automated test was added to verify that both events fire correctly for default information tools.

This change was requested as a prerequisite for implementing tool-call visibility in the Canvas AI chat interface (see related issue [#3575780](https://www.drupal.org/node/3575780)).
