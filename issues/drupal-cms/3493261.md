## Summary

When a site owner installs the Blog, Case Studies, Events, News, or Projects recipe in Drupal CMS, they now get a pre-built, published listing page for that content type. Each listing page lives at its canonical URL (`/blog`, `/case-studies`, `/events`, etc.), appears in the main navigation, and displays an intro text area above a paginated grid of content cards. Previously these recipes had no dedicated listing pages: the Views themselves served as page displays, which meant no intro text and no Layout Builder control. Now each listing page is a real `page` node that editors can update through the normal content editing workflow.

## Impact

Affects module maintainers.

- **New API:** New `drupal_cms_page` recipe dependency replaces `drupal_cms_content_type_base` in all content type recipes (Blog, Case Studies, Events, News, Projects)

## Technical details

Each content type recipe previously declared `drupal_cms_content_type_base` as its recipe dependency. That dependency is now replaced by `drupal_cms_page`, which brings in the Page content type needed to create the new listing nodes.

The Views configs were restructured across all affected recipes. Page-type displays (`page`, `page_1`, `page_2`) were removed from the Blog, News, and Events Views. In their place, standardized block displays were added with consistent machine names: `all` (full paginated listing, 15 items, embedded on the listing page), `latest` (3 most recent items with a "All posts" link back to the listing page, for use on the homepage or other pages), and where applicable `past` and `upcoming` (events). Unused `featured` and `recent` block displays were removed from the Blog and News Views. For Case Studies, an entirely new `case_studies` Views config was introduced with `all`, `latest`, and `related` block displays.

Each listing page is shipped as a default content YAML file: a `page` bundle node with a `layout_builder__layout` that holds the moderation control block, an intro text (`field_content`) block, and the corresponding `views_block` (e.g. `views_block:blog-all`) in a single `layout_onecol` section. Menu link default content was updated to link via `target_uuid` to these new nodes rather than pointing to a static `internal:/blog`-style URI. A patch for `drupal/default_content` ([#3160146](https://www.drupal.org/node/3160146)) is applied via `composer.json` to support serializing and importing Layout Builder section data, which is required for these nodes to be correctly imported on recipe install. This was the dependency that blocked the issue on Drupal 11.1.0 (see related issue [#3493287](https://www.drupal.org/node/3493287)).
