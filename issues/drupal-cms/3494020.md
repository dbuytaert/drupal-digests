## Summary

The `launch-drupal-cms.sh` script that ships in the Drupal CMS download archive was completely rewritten to be simpler and more fault-tolerant. The old script used `ddev composer create`, which requires a completely empty directory; anything unexpected in the project folder (such as a macOS `.DS_Store` file created by opening the folder in Finder) caused a hard failure with a confusing error message. The new script replaces that approach with four straightforward DDEV commands: configure DDEV if not already done, start the containers, install Composer dependencies if not already done, and open the site. The script is now re-entrant, meaning that if it fails part-way through a user can simply run it again without manual cleanup.

## Impact

No upgrade or configuration changes required.

## Technical details

The old script used `ddev composer create drupal/cms` to build the project from scratch inside the DDEV container. Because `ddev composer create` requires an empty directory, the script had to move all existing project files into a backup directory (`.ddev/drupal-cms-backup`), run the create command, then restore those files on failure. This made the script complex and brittle: a hidden file like `.DS_Store` left behind by macOS Finder was enough to abort the entire process.

The new script is four lines of active logic. It uses `test -d .ddev || ddev config --project-type=drupal11 --docroot=web` to run DDEV configuration only when needed, then unconditionally calls `ddev start`, then uses `test -f composer.lock || ddev composer install` to install dependencies only when they are missing, and finally calls `ddev launch`. Because each step is guarded by a condition, re-running the script after a partial failure is safe and skips already-completed steps.

As a consequence of this simplification, the hidden `.drupal-cms` marker file (whose sole purpose was to let the old script verify it was operating in a Drupal CMS project) was removed from the project template. The `LaunchScriptTest` PHPUnit test class that exercised the old script's backup-restore logic was also removed entirely. A separate build-script fix in the same commits corrected a `drush site:install` argument from `--uri=installer-cache` to `--sites-subdir=installer-cache`.
