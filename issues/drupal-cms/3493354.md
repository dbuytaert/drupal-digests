## Summary

Running Drupal CMS locally with DDEV previously required the project template to ship a `.ddev/config.yaml` file, which caused `ddev composer create drupal/cms` to overwrite the user's existing DDEV configuration with an outdated and incorrect one. The fix removes the `.ddev` directory entirely from the project template and replaces it with a shell script, `launch-drupal-cms.sh`. Running that script from the project root handles the full DDEV setup automatically: it configures DDEV for the project, clears the directory so `ddev composer create` has a clean slate, installs dependencies, and opens the browser at the Drupal installer. If the project was already set up, the script starts DDEV if the containers are not running and then opens the browser. The README and post-install console message now direct users to `./launch-drupal-cms.sh` instead of `ddev launch`.

## Technical details

The project template previously contained `project_template/.ddev/config.yaml` (a 289-line file with the `drupal` project type and inline documentation) and a `commands/web/bash-alias` file. Both are deleted in this change. In their place the template gains two files: a hidden `.drupal-cms` marker file (used by the script to verify it is running in the correct project root) and `launch-drupal-cms.sh`, a 91-line POSIX shell script.

The script follows this logic: if `$IS_DDEV_PROJECT` is set it exits immediately, since DDEV is already running. It then checks for the `.drupal-cms` marker and for `ddev` in `$PATH`. If `.ddev/config.yaml` does not yet exist, it runs `ddev config --project-type=drupal11 --docroot=web` to create it, backs up key files (`composer.json`, `.drupal-cms`, `launch-drupal-cms.sh`, `README.md`, and `web/`) into `.ddev/drupal-cms-backup`, then calls `ddev composer create $COMPOSER_CREATE` (defaulting to `drupal/cms --stability="RC"`). A `trap 'restore_backup' ERR` handler restores the backup if `composer create` fails, so the project is left in a recoverable state. On success the backup is removed. The script then checks for `web/index.php` to confirm the codebase was laid down, probes container health with `ddev php --version`, starts DDEV if needed, and ends with `ddev launch`.

The dev-only `.ddev/config.yaml` used inside the repository itself is also updated: the project type changes from `drupal` to `drupal11` and `ddev_version_constraint: '>= 1.24.0'` is added. A `LaunchScriptTest` PHPUnit test class is added under `drupal_cms_installer` tests; it copies the project template into a temp directory and exercises the script's success path, marker-file check, missing-DDEV path, config failure, `composer create` failure with backup restoration, and already-running container detection by swapping in a mocked `ddev` shell script.

## Upgrade

Anyone whose documentation, quick-start guides, or automation scripts instruct users to run `ddev launch` after `composer create-project drupal/cms` should update those instructions to `./launch-drupal-cms.sh`. The old step of manually placing or managing a `.ddev/config.yaml` is no longer needed; the script creates DDEV configuration on first run.

### Collaboration

Two review comments materially changed the shipped design. `tim.plunkett` noted that a `.command` extension (intended to let macOS users double-click the script) causes the script to run from the user's home directory rather than the project root, making it fail. `pameeela` independently raised the same concern about the unusual extension. The file was therefore shipped as `launch-drupal-cms.sh`. Separately, `rkoller` manually tested the MR and hit a concrete failure: `ddev composer create` refused to run because `README.md` left over in the project root made the directory non-empty. That test result directly prompted the backup-and-clear mechanism that moves project files aside before the `composer create` step and restores them on failure.
