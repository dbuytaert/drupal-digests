## Summary

Drupal CMS now ships a dedicated route at `/admin/config/development/site-export` (inside the `drupal_cms_helper` module) that exports the current site as a downloadable ZIP file containing a Drupal recipe. The route requires the `administer site configuration` permission and is intentionally not linked anywhere in the UI. Individual site templates, such as the "Blank" starter kit, can expose it by adding their own menu links. Sites installed from the blank starter kit (`drupal_cms_site_template_base`) will have their export automatically seeded from that base template, including scaffold files like `README.md` and CI configuration.

## Impact

Affects module maintainers.

- **New API:** New route `drupal_cms_helper.site_export` at `/admin/config/development/site-export` (marked `@api`); `SiteExporter::export()` gains an optional `?string $base` parameter for specifying a base recipe to seed the export.

## Technical details

The new route `drupal_cms_helper.site_export` (marked `# @api` in routing) maps to `ExportController::exportArchive()` (marked `@internal`). The controller reads the `site_template_base` state key, which the installer's `SiteTemplateForm` stores when a user chooses the `drupal_cms_site_template_base` recipe during setup. That path is forwarded to `SiteExporter::export()` as the new optional `?string $base` parameter.

`SiteExporter::export()` was extended with a `$base` argument (defaulting to `NULL`). When a base path is supplied and is a valid directory, the new private `copyBaseRecipe()` method mirrors the base recipe into the destination (excluding its `content/` directory) and renames `*.example` files by stripping the `.example` suffix, so scaffold templates become active. The logic that previously lived inline in `SiteExportCommand` was consolidated into `copyBaseRecipe()`. The `.htaccess` deletion from the exported `config/` directory was also moved into `SiteExporter::export()` itself, making all callers benefit from it automatically.

An initial implementation added the export as a button on the core config export form via `hook_form_config_export_form_alter`, but was quickly reverted after UX review. The final design uses an invisible standalone route that site templates can surface where appropriate, and drops the earlier custom `PharData` class-exists access check in favor of the standard permission requirement alone.

## Upgrade

Site template authors who call `SiteExporter::export()` directly can now pass a base recipe path as the second argument:

```php
// Before (still works, $base defaults to NULL)
$siteExporter->export($destination);

// After: optionally seed the export from a base recipe
$siteExporter->export($destination, '/path/to/base-recipe');
```

To expose the export route in a custom site template, add a menu link entity pointing to `drupal_cms_helper.site_export`. No code changes are required to use the route; a user with `administer site configuration` can already access `/admin/config/development/site-export` directly.
