# #3493438: Klaro significantly adds to anonymous visitor page weight

## Summary

Out of the box, Drupal CMS no longer loads any Klaro JavaScript or CSS for anonymous visitors. Previously the Klaro consent management library (`klaro-no-css.js`) accounted for more than 90% of the JavaScript served to anonymous users, at 212 KB uncompressed (65 KB compressed). The fix ensures Klaro assets are only loaded when at least one tracked service (such as Google Analytics) is actually activated in the site's configuration. Additionally, the Klaro library itself has been rebuilt: the file was switched from `klaro-no-css.js` to `klaro-no-translations-no-css.js` and recompiled targeting Drupal's browser compatibility defaults rather than very old browsers, reducing it from ~208 KB to ~64 KB when services are active.

## Impact

Affects site owners.

- **Config change:** Klaro JS/CSS only loads on pages where at least one consent-managed service is activated; a default Drupal CMS install with no third-party services enabled sends no Klaro assets to anonymous visitors.

## Technical details

The issue was tracked at the Drupal CMS level and resolved through fixes in several places. Issue [#3493540](https://www.drupal.org/node/3493540) tightened ECA (Event-Condition-Action) configuration so that Klaro JavaScript files are only added to the page when services are actually activated, bringing the default anonymous page to zero Klaro assets. Issue [#3491681](https://www.drupal.org/node/3491681) (Klaro module) swapped the bundled asset from `klaro-no-css.js` to `klaro-no-translations-no-css.js` and recompiled the library against Drupal's baseline browser targets rather than broad legacy-browser polyfill targets, shrinking the asset from ~208 KB to ~64 KB.

A secondary performance symptom noted by catch was that Klaro's `hook_js_settings_alter()` implementation injects a large amount of data into JS settings on every page request, costing ~16 ms even on render-cached pages. This cost is now avoided on pages where no services are active.

Issue [#3494580](https://www.drupal.org/node/3494580) updated Drupal CMS performance tests to assert that a default (no services enabled) anonymous front page loads zero bytes of Klaro JS and CSS.

### Collaboration

catch discovered this issue while profiling PHP performance: Klaro's `hook_js_settings_alter()` was taking ~16 ms on render-cached pages due to injecting a large settings payload, which led to checking the front-end where the library size became apparent. The maintainer of the COOKiES module (anybody) joined to clarify that switching to COOKiES was not the right path and that COOKiES is actually planned to be deprecated in favour of Klaro once feature parity and an upgrade path exist. jurgenhaas confirmed the architectural reasoning behind the Klaro choice (documented in the project's ADR) and agreed the library size needed to be addressed upstream rather than by switching tools.
