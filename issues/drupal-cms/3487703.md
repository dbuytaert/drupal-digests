## Summary

This is a closed proposal (MR !216) that was never merged into Drupal CMS. The intent was to make three contrib event add-ons, `drupal/events_calendar`, `drupal/events_recurring`, and `drupal/events_registration`, more discoverable to site owners by listing them as Composer suggestions in the Events recipe and surfacing them in Project Browser. The MR was ultimately closed without merging: the Drupal CMS Event content type had diverged significantly from the contrib add-ons' content model, breaking compatibility. The project's later shift toward a site-templates architecture removed the motivation to invest further in this direction. Site owners seeking these capabilities can still use the contrib recipes independently in a custom site template.

## Impact

No upgrade or configuration changes required.

## Technical details

The MR made four changes. First, `drupal_cms_events/composer.json` gained a `suggest` block listing `drupal/events_calendar`, `drupal/events_recurring`, and `drupal/events_registration`, plus a `conflict` entry for `drupal/events < 1.1.1` to prevent applying older contrib Events recipes that conflict with the Drupal CMS version. Second, `project_template/composer.json` added all three as hard `require` entries alongside the existing Drupal CMS components. Third, the starter-recipe Project Browser configuration added the three package names to `enabled_sources`. Fourth, `ComponentValidationTest::testSuggestedRecipes()` was added to dynamically read the `suggest` keys from `composer.json` at test time, look up each installed path via `Composer\InstalledVersions::getInstallPath()`, and apply each recipe twice to verify idempotency, skipping gracefully when a package is not installed.

A blocking technical obstacle emerged during review: the contrib add-ons depend on the standalone `drupal/events` recipe, which conflicted with the Drupal CMS Events recipe. The proposed workaround was to use `strict: false` in the contrib recipe (available from Drupal 10.4+ and 11.1+), but that would have required a new minor branch of the upstream contrib package. The conflict was reportedly resolved, but by then the Event content type in Drupal CMS had drifted far enough from the contrib add-ons' data model that full compatibility was no longer assured, and the issue was closed as won't-fix.
