## Summary

The Drupal CMS downloadable zip file previously bundled `composer.lock` and the full `vendor` directory (~70 MB). This meant anyone installing from the zip got dependency versions frozen at the time the release was cut, not the latest compatible versions, even though Drupal CMS does not intentionally pin dependencies. The zip has been changed so that it contains only `composer.json` and a pre-warmed installer cache. Users who download and unzip the archive now need Composer (or DDEV) to run `composer install` before Drupal CMS will work, but they will always start with up-to-date dependencies. A related breakage ([#3493354](https://www.drupal.org/node/3493354), which removed the `.ddev` directory from the template) had already made the old bundled-vendor zip format incompatible with `ddev composer create`, so the fix was applied urgently.

## Impact

No upgrade or configuration changes required.

## Technical details

The entire change lives in `.gitlab-ci.yml`. The old `.build` hidden job template ran `composer create-project` with a full install, producing a `vendor/` tree and `composer.lock` that were then zipped up. This template was replaced with a `.create-project` YAML anchor that passes `--no-install` to `composer create-project`, so the resolved lock file and installed packages are never committed to the artifact.

The CI pipeline was restructured into two clearly separated concerns. A `build test project` job (renamed from `install dev dependencies`) explicitly runs `composer install` after merging dev requirements from `dev.composer.json`, ensuring the test suite still has a fully resolved environment. A new `prime installer cache` deploy job (replacing `export config`) runs `build-cache.sh` inside the installer profile to pre-parse all available recipes and stores the resulting cache directory as a short-lived artifact. The `build project` zip job then consumes that artifact, injects the cache at `$_WEB_ROOT/profiles/drupal_cms_installer/cache`, and builds the archive from the dependency-free project tree. A follow-up commit (`9f42650e`) switched that final step from `composer archive` back to the `zip` utility to ensure the archive extracts into its own directory (per issue [#3493356](https://www.drupal.org/node/3493356)). The `COMPOSER_MIRROR_PATH_REPOS=1` variable was promoted to global scope so both the test and zip pipelines resolve local path repositories consistently.

## Upgrade

Sites that install Drupal CMS by downloading the zip file and extracting it must now run `composer install` (or use `ddev launch`, which invokes Composer automatically) before Drupal CMS will function. The zip no longer ships a `vendor/` directory or `composer.lock`.

He concluded that a fix was unavoidable because issue [#3493354](https://www.drupal.org/node/3493354) (removing `.ddev` from the project template) had already broken the existing zip for `ddev composer create`, and he found a way to preserve the cache pre-warming through the restructured pipeline. pameeela, who filed the report, had pressed for the requirement to be treated as non-negotiable without prescribing the solution, which kept the discussion focused.
