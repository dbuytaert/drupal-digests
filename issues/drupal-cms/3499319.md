# #3499319: Reponsive images should use sizes attribute for container-width sensitive image loading

## Summary

Drupal CMS responsive image styles now use the HTML `sizes` attribute approach instead of mapping a single image style per breakpoint. All aspect-ratio-specific styles (e.g. `16_9_medium`, `3_2_large`, `4_3_wide`) emit a `srcset` of multiple focal-point-cropped WebP candidates and a `sizes: 100vw` hint, letting the browser pick the right resolution automatically. The responsive image styles are also decoupled from the Olivero theme: they now depend on custom breakpoints (`custom.sm` through `custom.xl`) defined and managed through an ECA model, so Olivero no longer needs to be installed just for image configuration to work.

## Impact

No upgrade or configuration changes required.

## Technical details

All `responsive_image.styles.*` configs shipped by `drupal_cms_media` and `drupal_cms_starter` use `image_mapping_type: sizes` under the `custom.sm` breakpoint with `sizes: 100vw`. Each style lists between one and five focal-point-cropped WebP `sizes_image_styles` at different pixel widths; the browser resolves which file to request based on its own viewport arithmetic. Hero styles (`hero_large`, `hero_wide`) continue to use `image_mapping_type: image_style` because they have a fixed layout that spans known breakpoints rather than a fluid container.

The breakpoint group was previously `olivero`, requiring Olivero to be installed alongside the image recipe. MR623 (tracking issue [#3543740](https://www.drupal.org/node/3543740)) introduced an ECA model (`eca.eca.define_breakpoints`) that fires on `eca_render:breakpoints_alter` and injects four `custom.*` breakpoints at runtime via `eca_render_alter_breakpoint` actions. All responsive image styles were updated to reference `breakpoint_id: custom.sm` and `breakpoint_group: custom`, with an enforced config dependency on the ECA model so that deleting the model cascades to delete the styles. The recipe's install list replaced `olivero` with `bpmn_io`, `eca_base`, `eca_render`, `eca_ui`, and `modeler_api`.

A related simplification (part of the same effort) removed the "large" view mode options (`16_9_large`, `1_1_square_large`, etc.) from the CKEditor allowed embedded media display modes, since a narrower content column means "large" in-body images are no longer meaningfully distinct from "medium."

### Collaboration

tonypaulbarker commented that the editor's explicit size choice when embedding an image (small vs. medium in a narrow column) cannot be replaced by the `sizes` attribute alone, because responsive images decide based on viewport/container math rather than editorial intent. This shaped the outcome: the `sizes` approach was adopted for resolution selection within each named style, but the distinct small/medium/large/wide view modes were kept so editors retain control over the display size they intend.
