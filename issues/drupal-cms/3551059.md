## Summary

Drupal CMS 2.x (and 1.2.x) gains a new `drush site:export` command that exports the current site's full configuration and content into a recipe directory. Site template authors can now follow a three-step workflow: install Drupal CMS, configure the site as desired, then run `drush site:export` to produce a ready-to-use recipe that can be shared and applied to fresh installations. This replaces the previously fiddly manual process of hand-crafting a recipe from an existing site.

## Impact

Affects module maintainers.

- **New API:** New `drush site:export` command (with optional `--destination` flag) exports a live Drupal CMS site's config and content into a recipe directory.

## Technical details

The commit adds two new `@internal` classes to the `drupal_cms_helper` module. `SiteExporter` (in `src/SiteExporter.php`) is a service that performs the actual export. `SiteCommands` (in `src/Drush/Commands/SiteCommands.php`) is the Drush command class that wires the exporter to the `site:export` command.

`SiteExporter::export(string $destination)` does the following in sequence. It collects all installed modules and themes (excluding the install profile), writes a `recipe.yml` that lists them under `install`, sets `config.strict: false`, and places simple config objects from the `system.*` and `user.*` namespaces into `config.actions` using the `simpleConfigUpdate` action (because those config objects are guaranteed to exist before any recipe is applied). It then copies all remaining active config to a `config/` subdirectory via an anonymous subclass of `FileStorage` that strips the `uuid` and `_core` keys on every write, and also applies a workaround for the known core bug tracked in [#3002532](https://www.drupal.org/node/3002532) (adding `language.entity.und` and `language.entity.zxx` as dependencies of non-built-in language entities). `core.extension` is always deleted from the exported config directory. It writes a `composer.json` with a `drupal-recipe` type and version constraints (using `^x.y` for stable packages, bare dev strings for dev packages) derived from `Composer\InstalledVersions`. Finally, it calls `ContentExporter::exportWithDependencies()` for every exportable content entity, skipping path aliases, internal entity types, and users with uid 0 or 1.

`SiteCommands::export()` resolves the default destination by reading the root `composer.json`'s `extra.installer-paths` to find the path configured for the `drupal-recipe` type, substituting `site_export` for the `{$name}` placeholder. The destination can be overridden with `--destination`.
