## Summary

The Experience Builder demo handler in Drupal CMS has been converted from a Composer plugin (`drupal/xb_demo`) to a plain Composer script. Previously, the demo lifecycle was managed by a plugin package, which meant a site owner who did not want the XB demo could not simply delete or ignore that plugin without causing Composer autoloader errors. Now the handler is a single file (`ExperienceBuilderDemo.php`) in the project template root, registered as a standard `pre-package-update` Composer script. A site owner who has no interest in the XB demo can delete that one file without any Composer complaints.

## Impact

No upgrade or configuration changes required.

## Technical details

The old implementation lived in `project_template/recipes/xb_demo/src/Plugin.php` as `Drupal\XbDemo\Plugin`, which implemented Composer's `PluginInterface` and `EventSubscriberInterface`. It subscribed to `PackageEvents::PRE_PACKAGE_UPDATE` through the plugin system and required `composer-plugin-api: ^2`, its own `composer.json`, and an `allow-plugins` entry for `drupal/xb_demo`.

The new implementation is a single file `project_template/ExperienceBuilderDemo.php` containing a final class `ExperienceBuilderDemo` with one public static method `onPackageUpdate(PackageEvent $event)`. The project template `composer.json` wires it up via:

```json
"scripts": {
    "pre-package-update": "ExperienceBuilderDemo::onPackageUpdate"
},
"autoload": {
    "classmap": ["ExperienceBuilderDemo.php"]
}
```

The `drupal/xb_demo` path repository, its `allow-plugins` entry, and its `require-dev` dependency are all removed. The direct dependency on `drupal/experience_builder` (`0.x-dev@dev`) is added in its place.

The logic inside `onPackageUpdate` is also tightened: the earlier version called a private `uninstallXb` method and checked `trim($output) === 'Enabled'` to decide whether to uninstall; the new version inlines the logic and instead returns early if `trim($output) === 'Disabled'`, which correctly handles the case where Experience Builder is absent from the module list entirely. The `drupal_cms_starter` recipe also drops `drupal/xb_demo` from its `additional_trusted_composer_plugins` list.

## Upgrade

No code changes are required for most sites. If your project explicitly references `drupal/xb_demo` in a custom `composer.json`, `allow-plugins`, or a recipe, remove those references. If you previously trusted `drupal/xb_demo` as a Composer plugin, remove it from `package_manager.settings.simpleConfigUpdate.additional_trusted_composer_plugins`.
