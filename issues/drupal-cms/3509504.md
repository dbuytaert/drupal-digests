## Summary

Drupal CMS now publishes a second Composer project template, `drupal/cms_cpanel`, specifically for shared-hosting environments like cPanel and Softaculous that cannot install into a relocated document root. The package is automatically generated whenever a new release is tagged and mirrors the regular `drupal/cms` template in every way except that files land directly in the installation directory rather than a `web/` subdirectory. Softaculous updated its Drupal CMS installer to use the new package shortly after the fix shipped.

## Impact

Affects module maintainers.

- **New API:** New `drupal/cms_cpanel` Composer project template published on each tagged release for cPanel and Softaculous shared-hosting installs without a relocated document root

## Technical details

The `ddev tag` script now generates a `cpanel_template/` directory as part of the release process. It copies `project_template/`, rewrites `composer.json` to replace `"web/` path mappings with `"./"` via `sed`, strips the `--docroot=web` flag from `README.md`, sets the package name to `drupal/cms_cpanel` via `composer config name`, and then moves all files from `web/` up to the template root. The `.gitlab-ci.yml` subtree-split configuration gains a new entry pointing `cpanel_template/` at `git@git.drupal.org:project/cms_cpanel.git`, so each tagged commit is automatically pushed to the separate Drupal.org project. CI is configured to skip PHP tests for `cpanel_template` because it is structurally identical to the normal template; the tag script also broadens the component-discovery `find` pattern from `project_template` to `'*_template'` so both templates receive correct version constraints on tagging. As a related fix, the existing `project_template/composer.json` corrects two installer path entries for `drupal-drush` and `drupal-recipe` to use `./drush/` and `./recipes/` rather than relative paths without the explicit `./` prefix. The approach is analogous to core's `drupal/legacy-project` template but without requiring a separate manually-maintained project: the cPanel template is always derived automatically and never diverges from the canonical template except for the document-root difference. The decision to host the split on Drupal.org rather than GitHub came from drumm, who noted that consistent access management across all subtree splits is easier when everything lives in the same place.

### Collaboration

phenaproxima initially proposed hosting the new package on GitHub (under the DA's namespace) so it could be auto-published to Packagist without manual release nodes. drumm redirected that plan, explaining that mixing subtree-split repositories between GitHub and Drupal.org creates long-term access-management problems and that having everything work the same way is much easier to maintain. The package ended up as a proper Drupal.org project pushed via the existing subtree-split CI pipeline.
