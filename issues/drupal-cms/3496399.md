## Summary

The Drupal CMS installer can now recover gracefully from mid-installation interruptions such as timeouts. If the installer is cut off while applying recipes, it previously had no way to know how far it had gotten, potentially forcing a full restart and making a bad situation worse. The installer now tracks which recipes have been successfully applied. On resumption, it skips over those already-applied recipes and continues from where it left off. The tracking state is cleaned up automatically once installation completes, so there is no lasting footprint on the installed site.

## Impact

No upgrade or configuration changes required.

## Technical details

A new `RecipeAppliedSubscriber` event subscriber in the `drupal_cms_installer` module listens to `RecipeAppliedEvent` and appends each successfully applied recipe's directory name to a list stored in Drupal State under the key `drupal_cms_installer.applied_recipes`.

In `drupal_cms_installer_apply_recipes()`, the function now calls `array_diff()` between the full list of recipes chosen by the user and the state-stored list of already-applied recipes. The resulting `$recipes_to_apply` list is what actually gets queued into the batch. If the diff is empty (all chosen recipes are already done), the function returns an empty array immediately, bypassing even the `install_profile_modules()` call. When the installer finishes and uninstalls itself in `drupal_cms_installer_uninstall_myself()`, it deletes the state key via `\Drupal::state()->delete(RecipeAppliedSubscriber::STATE_KEY)`. Both the command-line and interactive install tests assert that this key is absent after a successful install.
