## Summary

A new `drupal_cms_authentication` recipe has been extracted from the Drupal CMS base recipe and published as a standalone, reusable component. It installs the `login_emailusername` module (so users can log in with their email address instead of just their username), sets the site's 403 page to `/user/login` using Drupal core's built-in page configuration, and uses an ECA automation to redirect users back to the login page after they log out. Any Drupal site can now require this recipe directly, without pulling in all of Drupal CMS.

## Impact

Affects module maintainers and site owners.

- **Config change:** Drupal CMS now sets `system.site page.403` to `/user/login` via the new recipe instead of using an ECA event handler.
- **New API:** New `drupal_cms_authentication` recipe installable independently on any Drupal site to enable email login, 403 redirect to login page, and post-logout redirect.

## Technical details

Previously the authentication redirect logic lived entirely in the Drupal CMS base recipe as an ECA model (`auth_redirects`) that handled three flows: 403 errors, post-login redirects to the dashboard, and post-logout redirects to `/user/login`. It also relied on `bpmn_io`, `eca_base`, `eca_misc`, `eca_user`, and `login_emailusername` being declared directly in the base recipe's `composer.json` and `recipe.yml`.

The commit extracts these concerns into a new `drupal/drupal_cms_authentication` Composer package at `recipes/drupal_cms_authentication/`. The base recipe now lists `drupal_cms_authentication` as a recipe dependency instead of declaring those modules itself.

Inside the new recipe, the ECA model is reduced to only the logout flow: on `user:logout`, redirect to `/user/login`. The 403 redirect is dropped from ECA entirely and replaced with a `system.site simpleConfigUpdate` that sets `page.403: '/user/login'`, letting Drupal core handle that behavior natively. The post-login dashboard redirect was removed from the new recipe (the dashboard module's own hard-coded redirect makes it redundant). The recipe is declared idempotent via `config.strict: false`, and the functional test (`RedirectsTest`) confirms this by applying the recipe twice. The 403 assertion was updated to check for HTTP 403 status and the presence of the login form rather than a URL match, because core's built-in 403 page renders the login form in-place without changing the URL.
