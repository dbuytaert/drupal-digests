## Summary

The `drupal_cms_image` recipe's responsive image styles were hardwired to the `olivero` breakpoint group, meaning Olivero had to remain installed and could not be uninstalled without breaking those styles. This was a problem for sites that wanted to evaluate other themes. MR !550 proposed fixing this by introducing a tiny, hidden theme called `drupal_cms_breakpoints` that owns only the breakpoints and nothing else, so the image recipe no longer depends on Olivero. The MR was closed without merging; the issue was ultimately marked fixed because Drupal CMS 2.x moved to an ECA-based approach that defines breakpoints in configuration instead, making them fully customizable and theme-independent.

## Impact

No upgrade or configuration changes required.

## Technical details

Every responsive image style config file in the `drupal_cms_image` recipe had `breakpoint_group: olivero` and `breakpoint_id: olivero.*` references, which forced Olivero to remain installed (Drupal's Breakpoint module only accepts breakpoints defined by modules or themes, not raw config).

The MR proposes a minimal hidden theme: two files (`drupal_cms_breakpoints.info.yml` with `hidden: true` and `drupal_cms_breakpoints.breakpoints.yml`) bundled inside `recipes/drupal_cms_image/assets/`. The `drupal_cms_image` recipe's `composer.json` declares Composer scaffold mappings that copy those two files into `[web-root]/themes/drupal_cms_breakpoints/`. The top-level `composer.json` is updated to add `drupal/drupal_cms_image` to `allowed-packages` in the scaffold config. The recipe's `recipe.yml` then installs `drupal_cms_breakpoints` instead of `olivero`. All `breakpoint_group` and `breakpoint_id` references across every responsive image style config are rewritten from `olivero` / `olivero.sm` / `olivero.md` / etc. to `drupal_cms_breakpoints` / `drupal_cms_breakpoints.sm` / etc.

The breakpoint values themselves (sm at 500 px, md at 700 px, lg at 1000 px, xl at 1300 px, plus nav and grid variants) are borrowed from Olivero, so the actual image rendering behaviour does not change; only the ownership of those breakpoint definitions moves to the new glue theme. A concern raised in the thread was that this scaffold approach ties the recipe to `drupal/cms` projects specifically, which prompted discussion of alternatives including reusing `drupal_cms_olivero` and waiting for the Mediacurrent design-system theme. The eventual fix in 2.x used an ECA model to store breakpoints in configuration rather than in any theme file.
