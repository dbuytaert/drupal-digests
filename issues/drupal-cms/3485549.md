## Summary

When an administrator manually creates a new user account at `/admin/people/create`, the user registration form in Drupal CMS now shows only the fields that are actually relevant to the situation. The "Notify user of new account" checkbox defaults to checked, and while it is checked, the password and account status fields are hidden: there is no point setting a password when the new user will receive an email and set one themselves. If the admin unchecks the notification, both fields reappear. When the form is submitted with notification enabled, a secure random password is generated automatically so the account is created successfully without the admin having to supply one.

## Impact

No upgrade or configuration changes required.

## Technical details

The change adds a new ECA model (`user_register`) shipped as config in the `drupal_cms_authentication` recipe. Two ECA events drive the behaviour. On `form:form_build` for `user-register-form`, a sequence of actions reorders fields (`mail` at weight -10, `name` at -9, `notify` at -8, `pass` at -7), sets the `notify` checkbox default value to `1` (checked), marks the `pass` field as not required (`eca_form_field_require` with `flag: false`), and attaches Drupal `#states` rules via `eca_form_field_add_state` so that both `pass` and `status` become invisible whenever `notify` is checked. On `form:form_validate`, a condition (`Flow_notify_on`) checks whether the `notify` field equals `1`; if so, `eca_token_set_random_value` generates a random 32-character password into a `[password]` token, which is then written into the `pass` field via `eca_form_field_set_value`, and the account `status` field is forced to `1`. The recipe's install list gained `eca_form` to support the form-related ECA plugins, and the `bpmn_io` dependency was updated to `2.1.x-dev` in both `drupal_cms_authentication` and `drupal_cms_starter`. A Cypress test (`authentication.cy.js`) was added to verify the conditional field visibility and the successful account creation flow.

### Collaboration

jurgenhaas proposed using ECA after the issue was opened and then built the initial implementation. During review, jurgenhaas noted a design tension: because ECA removes the `required` attribute from the password field (so the form can submit without one when notification is on), the original plan included an ECA form-validation step to error when notification was off and the password was left empty. pameeela pushed back, arguing that a required-field error on a field not marked as required would be confusing, and suggested simply accepting an empty password in that case. jurgenhaas agreed and removed the validation step. A separate discussion arose over the "Notify user of new account" label: pameeela had committed a label change, but jurgenhaas pointed out that deviating from the core label would break existing translations, so the label was reverted to the core string. phenaproxima wrote the Cypress test to cover the JavaScript-dependent behaviour before the MR was merged.
