## Summary

This proposal adds a `Launch Drupal CMS.command` shell script to the Drupal CMS project template, giving users a way to run Drupal CMS locally without installing DDEV, Docker, PHP, or Composer. Double-clicking the script (on macOS; also works on Linux from the terminal) downloads FrankenPHP and Composer binaries on first run, installs Composer dependencies, starts the FrankenPHP application server, and opens the Drupal installer in a browser. The MR was closed without merging after internal discussion: phenaproxima concluded that rolling a custom launcher would be painful to maintain, and that an existing third-party tool should be officially adopted instead. The approach was left as a proof of concept; the policy discussion continues in [#3476832](https://www.drupal.org/node/3476832).

## Technical details

The script (`project_template/Launch Drupal CMS.command`) is a POSIX shell script placed at the project root. On first run it creates a `bin/` directory (added to `.gitignore`), downloads the appropriate FrankenPHP standalone binary from GitHub releases (`v1.2.5`, detected via `uname` for macOS or Linux) and the Composer PHAR (`2.7.9`), then runs `$FRANKENPHP php-cli $COMPOSER install` to install dependencies when no `composer.lock` is present. It then downloads a `Caddyfile` from Drupal core's repository into the `web/` directory (scaffolded via a patch for [#3437187](https://www.drupal.org/node/3437187)) and starts FrankenPHP with `frankenphp run --pidfile=...`. On macOS it auto-quits Terminal when the script exits via `osascript`. Subsequent runs reuse the already-downloaded binaries. A PID file guards against double-starts. The existing `composer launch` script (which called `quick-start drupal_cms_installer`) is removed from `composer.json` and replaced by instructions to use the new `.command` file. The `.ddev/config.yaml` hook is also updated to check for `composer.lock` rather than the vendor directory as the signal that dependencies are installed.
