# #3577804: The installer should set a state flag to remember what site template was applied

## Summary

During Drupal CMS installation, when a user selects the "Blank" (starter kit) site template on the site template selection form, the installer now stores the chosen template's recipe locator in the `site_template_base` state key. This state flag is deliberately environment-local (state is never ported between environments) and is used by a new export endpoint that lets users with the `administer site configuration` permission download the current site as a recipe ZIP file from `/admin/config/development/site-export`. This lays the groundwork for future contextual guidance in the admin UI. This change was implemented as part of [#3575283](https://www.drupal.org/node/3575283).

## Impact

Affects module maintainers.

- **New API:** New `@api` route `/admin/config/development/site-export` for downloading the current site as a recipe ZIP, and `site_template_base` state key written during installation when the 'Blank' site template is selected.

## Technical details

In `SiteTemplateForm::submitForm()`, a new `StateInterface` dependency is injected. When the submitted choice is `drupal_cms_site_template_base`, the recipe locator path is written to state via `$this->state->set('site_template_base', $locator)` before the recipe is enqueued. Only the "Blank" starter kit triggers this; other site templates (Haven, Byte, etc.) do not set the key.

The state key feeds the new `ExportController::exportArchive()` (an `@internal` controller in `drupal_cms_helper`), which calls `SiteExporter::export($destination, $this->state()->get('site_template_base'))`. `SiteExporter::export()` was refactored to accept an optional `?string $base` parameter: when a base path is provided and the directory exists, the new private `copyBaseRecipe()` method mirrors it to the destination (excluding `content/`) and renames any `*.example` files by stripping the suffix. When no base is set, the directory is prepared from scratch as before. The exporter also now deletes `config/.htaccess` from the output, since the exported recipe is meant to be shared.

A new route `drupal_cms_helper.site_export` at `/admin/config/development/site-export` is marked `# @api` in `drupal_cms_helper.routing.yml`. It requires the `administer site configuration` permission and a custom `exportArchiveAccess()` check that verifies `PharData` is available. The `ExportController` class itself carries the `@internal` marker; only the route path is part of the public API surface.
