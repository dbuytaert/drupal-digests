## Summary

When Drupal CMS is installed with the SEO Tools recipe, the Simple XML Sitemap was effectively empty by default: only the front page appeared, because no content types had sitemap settings configured. This change adds an ECA (Event-Condition-Action) model to the SEO Tools recipe that automatically creates sitemap settings for every node content type. Settings are created when the SEO Tools recipe is first applied (covering all existing content types) and whenever a new content type is created programmatically afterward. Content types created through the admin UI are intentionally excluded, since the sitemap settings form is available there and the UI handles that path directly. Existing per-bundle settings are never overwritten.

## Impact

Affects module maintainers.

- **New API:** New ECA model `node_sitemap_settings` in the SEO Tools recipe automatically creates `simple_sitemap.bundle_settings.default.node.*` config for each node content type on recipe apply and on programmatic content type creation.

## Technical details

The initial proposed fix used the `createForEachIfNotExists` config action against `node.type.*`, but that approach failed because `simple_sitemap.bundle_settings.*` is simple config, not a config entity type, making `getEntityTypeIdByName()` return null and throwing a `ConfigActionException`.

The implemented solution is a new ECA model (`node_sitemap_settings`, added to `recipes/drupal_cms_seo_tools/config/optional/eca.eca.node_sitemap_settings.yml`). It uses two event triggers: a `config:save` event filtered to `node.type.*` new writes outside of config sync and outside the `/admin/structure/types/add` path, and a `drupal:recipe_applied` event for `drupal_cms_seo_tools` that iterates over all existing node bundles. In both paths the model reads the existing `simple_sitemap.bundle_settings.default.node.[bundle]` config and skips the bundle if settings are already present. When settings are absent, it writes `index: 1`, `priority: 0.9`, `changefreq: daily`, and `include_images: 0` via `eca_config_write` actions.

A key implementation detail: the `config:save` event previously lacked `config_name`, `sync_mode`, and `write_mode` filter fields (stored as empty `{}`). ECA 2.1.4 introduced these filter parameters, which prevent the recursive event loop that would otherwise fire when the model itself writes sitemap config and re-triggers the `config:save` listener. The minimum ECA constraint was therefore bumped to `^2.1.4` across all affected recipes (`drupal_cms_seo_tools`, `drupal_cms_content_hub`, `drupal_cms_privacy_basic`, and the root `drupal_cms` composer manifest). The SEO Tools recipe now installs the ECA submodules `eca_base`, `eca_config`, `eca_content`, `eca_misc`, and `eca_user`, and adds `drupal/bpmn_io` and `drupal/eca` as explicit composer requirements.

Test coverage was added in `testAutomaticSitemapSettings()` inside `ComponentValidationTest`, asserting that: settings are created for types existing at recipe-apply time, programmatically created types get settings immediately, UI-created types do not, existing settings survive a recipe reapply, and config sync writes are not intercepted.

### Collaboration

The design went through a meaningful pivot mid-thread. phenaproxima proposed using `createForEachIfNotExists` as a clean declarative solution, but riyas_nr discovered during testing that the approach fails at runtime because `simple_sitemap.bundle_settings.*` config is not a config entity type, causing a `ConfigActionException` with the message "Cannot determine a config entity type." phenaproxima then proposed an ECA-based approach as the alternative.

jurgenhaas took over implementation and uncovered two further problems: the `config:save` event triggered a recursion error when the model itself wrote config, and buggy `[bundle]` token substitution was creating sitemap settings for non-existent node types like `index` and `yoast_seo`. jurgenhaas resolved both by extending ECA with the `config_name`, `sync_mode`, and `write_mode` filter fields in the config:save event plugin, releasing the fixes as ECA 2.1.4 and updating the recipe's constraint accordingly.
