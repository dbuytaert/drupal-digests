## Summary

The Drupal CMS web installer used to freeze for a noticeable amount of time after you clicked "Continue" on the site-name screen. The delay came from the recipe system having to parse a deep, complex tree of Recipe objects before it could build the install batch job. This change eliminates that delay for users who start from the official Drupal CMS ZIP file: all installer recipes are now pre-parsed and serialized during the CI build and shipped inside the ZIP, so the installer loads them from cache instead of re-parsing them. The cache is a write-once, read-once flash bag: once a cached file is consumed it is immediately deleted, sidestepping cache-invalidation problems entirely. Users who start with `composer create-project` will not have a cache and will experience the original slower path, but they are expected to use `drush site:install` which is comparatively fast anyway.

## Impact

No upgrade or configuration changes required.

## Technical details

The core of the change is the new `RecipeLoader` class (`profiles/drupal_cms_installer/src/RecipeLoader.php`). Its static `load()` method replaces direct calls to `Recipe::createFromDirectory()` throughout the installer. On a cache hit it reads the serialized `Recipe` object from a `cache/` directory adjacent to the installer profile, deserializes it, and immediately deletes the cache file. Absolute path references inside the serialized object are made portable by substituting the project root with a `%PROJECT_ROOT%` placeholder at write time; on load, a regex over the raw serialized string replaces that placeholder with `realpath($project_root)` before calling `unserialize()`, so PHP's length-prefixed string format remains valid.

Cache files are written only when the `DRUPAL_CMS_INSTALLER_WRITE_CACHE` environment variable is set. The CI pipeline runs a new shell script `build-cache.sh` (in the web root) that does a full `drush site:install` against a throw-away SQLite database with that variable set and with all available add-on recipes selected (`add_ons=*`). This produces serialized cache files for every selectable recipe. The CI `build project` job then runs the script before assembling the distribution ZIP, so the cache directory is included in the archive that end users download.

Supporting the `add_ons=*` wildcard required a small change to `RecipesForm`: a `valueCallback` was added that converts a comma-separated string or the literal `*` into the checkbox array format that `Checkboxes::valueCallback()` expects, enabling `drush site:install` to drive the form programmatically. The submit buttons also gained an `#op` key so `submitForm()` can distinguish a programmatic submission from a "Skip this step" click without relying on fragile `#array_parents` inspection.
