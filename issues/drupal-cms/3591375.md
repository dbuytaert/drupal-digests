# #3591375: Automatically copy font and color CSS files to Drupal root if a site template sets up a Mercury-based theme by default

- **Project:** Drupal CMS
- **Type:** Normal feature request
- **Status:** Fixed
- **Discussion:** [#3591375](https://www.drupal.org/node/3591375) · 1 contributor · 2 comments

## Summary

When a site template sets a Mercury-based theme as the site default, Drupal CMS Helper now automatically copies that theme's non-minified CSS files (typically `theme.css` and `fonts.css`) from the theme directory into the Drupal web root. Because Canvas does not yet offer a UI for overriding fonts and colors, Mercury-based themes expose these as plain CSS files that developers can edit. Placing copies in the web root gives developers an obvious, version-control-friendly location to make those customizations without touching the theme package itself. The copy operation uses `FileExists::Error`, so it will not overwrite files a developer has already customized. If the web root is not writable, or the files already exist, the copy attempt is silently skipped.

## Impact

Affects module maintainers.

- **New API:** New static `RecipeSubscriber::isMercuryBased(Extension $theme): bool` method for checking if a theme is Mercury or a Mercury sub-theme.

## Technical details

The logic is added to `RecipeSubscriber::onApply()` in `drupal_cms_helper`. After handling the existing front-page alias and user-cache concerns, it now also runs when `$event->recipe->type === 'Site'`. It reads the current default theme from `system.theme` config, calls the new static `RecipeSubscriber::isMercuryBased(Extension $theme)` helper (which returns `true` when the theme machine name is `mercury` or when its `generator` info key starts with `mercury:`), and only proceeds if the web root (`$appRoot`) is writable. It then calls `LibraryDiscoveryInterface::getLibraryByName()` to retrieve the theme's `global` library, filters its `css` entries to those where `minified` is empty/false, and copies each file to `$appRoot . DIRECTORY_SEPARATOR . basename($origin)` via `FileSystemInterface::copy()` with `FileExists::Error`. A caught `FileException` is silently swallowed so the recipe apply does not fail on a pre-existing file. Four new constructor arguments are injected via autowiring: `ThemeExtensionList`, `FileSystemInterface`, `LibraryDiscoveryInterface`, and the `app.root` parameter. A new kernel test class `RecipeSubscriberTest` covers both the Mercury-theme path (expects `fonts.css` and `theme.css` to be present and byte-for-byte identical to the theme source) and the non-Mercury path (expects no CSS files copied).
