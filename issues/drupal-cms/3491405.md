## Summary

Drupal CMS now ships with automated performance regression tests that run in CI. Two scenarios are covered against the `drupal_cms_starter` recipe: an anonymous visitor hitting the `/home` front page with a fully warm cache, and a logged-in content editor doing the same. The tests assert exact counts for database queries, cache gets, cache tag checks, and stylesheet and script counts, plus upper-bound assertions on total CSS and JS byte weight. Any future change that silently adds queries, cache operations, or asset weight will cause CI to fail, making regressions visible at the pull-request stage rather than months later.

## Impact

No upgrade or configuration changes required.

## Technical details

The new test class `Drupal\Tests\drupal_cms_starter\FunctionalJavascript\PerformanceTest` lives in `recipes/drupal_cms_starter/tests/src/FunctionalJavaScript/PerformanceTest.php`. It extends Drupal core's `PerformanceTestBase`, uses `RecipeTestTrait` to apply the `drupal_cms_starter` recipe, and collects telemetry with `collectPerformanceData()`. The anonymous scenario asserts zero queries and exactly two cache gets on a page-cache hit. The editor scenario asserts 59 queries and 241 cache gets; individual query assertions are deferred to a follow-up ([#3493429](https://www.drupal.org/node/3493429)) because the Coffee module generates an extra AJAX request on every page load ([#2453585](https://www.drupal.org/node/2453585)). Both scenarios apply loose upper-bound assertions on stylesheet and script byte totals (under 120 KB CSS and 245 KB JS for anonymous; under 370 KB CSS and 535 KB JS for the editor) with `@todo` markers pointing to Klaro's out-of-the-box asset weight ([#3493438](https://www.drupal.org/node/3493438)).

To make FunctionalJavascript tests runnable in GitLab CI, `.gitlab-ci.yml` gained a reusable `.with-chrome` YAML anchor that provisions a `selenium/standalone-chrome:latest` service, and the `MINK_DRIVER_ARGS_WEBDRIVER` variable was threaded through the `phpunit` invocation. The test runner command also gained `--log-junit` output so that GitLab's test-report tab populates. The test is tagged `@group OpenTelemetry` and `@group #slow` and requires the `apcu` PHP extension; it uses Stark as the default theme to keep assertions independent of Olivero's specific asset set.

While debugging GitLab CI, catch noticed that the front page was issuing a redirect to `/home`, causing Drupal to bootstrap twice. The test was updated to request `/home` directly, and the underlying redirect issue was tracked as [#3493615](https://www.drupal.org/node/3493615). The profiling work also surfaced navigation-bar caching issues addressed in Drupal core ([#3493406](https://www.drupal.org/node/3493406)).

### Collaboration

catch drove the entire implementation and fought through several GitLab CI configuration dead ends before functional JavaScript tests would run in the pipeline at all. After landing a passing green run, phenaproxima raised a substantive question about whether exact-assertion performance tests were too fragile a gate for a project with many unpinned dependencies, suggesting an allowed-to-fail job instead. catch explained the deliberate design: the tests are meant to surface regressions immediately, and adjusting an assertion expectation by one is the intended workflow when a change legitimately alters performance. Phenaproxima opened a follow-up to move the tests into their own CI job ([#3494505](https://www.drupal.org/node/3494505)) as a compromise, and catch opened several more follow-ups for issues the profiling work uncovered ([#3493349](https://www.drupal.org/node/3493349), [#3493406](https://www.drupal.org/node/3493406), [#3493615](https://www.drupal.org/node/3493615), [#3493429](https://www.drupal.org/node/3493429), [#3493430](https://www.drupal.org/node/3493430), [#3496652](https://www.drupal.org/node/3496652)).
