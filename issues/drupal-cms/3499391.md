## Summary

On the Drupal CMS 1.x development branch, developers who clone the repo and try to install additional modules via Project Browser hit a hard Composer error: "Could not scan for classes inside 'ExperienceBuilderDemo.php' which does not appear to be a file nor a folder." The cause is a symlink named `ExperienceBuilderDemo.php` at the project root that Composer's autoloader cannot handle. The proposed fix removes that symlink and registers the real file via `autoload.classmap` in `composer.json`, while also moving `cweagans/composer-patches` to `require-dev` and loosening the `drupal/default_content` constraint to `^2@alpha`. The MR was ultimately closed without merging after the pipeline failed; catch opened a separate follow-up, [#3501190](https://www.drupal.org/node/3501190), to remove the demo script from the project template entirely.

## Impact

## Technical details

The root of the problem is that `ExperienceBuilderDemo.php` at the project root is a symlink pointing to `project_template/ExperienceBuilderDemo.php`. When Composer resolves the `autoload` configuration it follows the project root, encounters the symlink, and cannot classify it as a file or directory, causing the scan to abort. The proposed MR removes the symlink (tracked in version control as a single-line file containing its target path) and adds an explicit `autoload.classmap` entry in `composer.json` pointing directly to `project_template/ExperienceBuilderDemo.php`, bypassing symlink resolution entirely. The same diff moves `cweagans/composer-patches` from `require` to `require-dev`, so the patching plugin is no longer installed on production sites, and tightens the `drupal/default_content` constraint to `^2@alpha` to allow alpha releases. phenaproxima noted after the MR opened that the approach still caused pipeline failures, leaving the issue unresolved by this specific change.
