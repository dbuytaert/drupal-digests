# #3488457: Ensure that Drupal CMS can be updated with automatic updates

## Summary

Drupal CMS has established a formal policy ensuring that stable releases (RC and beyond) never ship with patched dependencies. Because the `automatic_updates` module's `ComposerPatchesValidator` blocks updates when a patcher like `cweagans/composer-patches` is present, shipping patches in stable releases would silently break the ability for non-technical site owners to apply unattended updates. The fix separates development patching (allowed, for CI and internal work) from production releases (prohibited). Site owners can rely on automatic updates working out of the box on any stable Drupal CMS release.

## Impact

No upgrade or configuration changes required.

## Technical details

The root problem is that `package_manager`'s `ComposerPatchesValidator` requires the patcher to be a direct, explicitly managed dependency and blocks updates whenever a patch fails to apply or is ambiguous. If `cweagans/composer-patches` were part of the distributed project template, any recipe-applied patch could silently break update capability for non-developer site owners who did not knowingly install the patcher.

The implementation separates concerns across two composer manifests. The distributed project template (`project_template/composer.json`) contains no reference to `cweagans/composer-patches`. The internal development manifest (`dev.composer.json`) declares `cweagans/composer-patches: ^2` under `require-dev`, keeping it entirely out of production installs. Development-time patches are collected in `patches.json` at the repo root, with a `_readme` key explicitly marking them as "strictly for internal development and CI." The `patches/` directory holds the actual patch files (currently limited to `drupal/core` patches for Umami, date field default value, and PHPUnit recipe discovery).

`CONTRIBUTING.md` codifies the policy in prose: patches are permitted to assist internal development and CI, must never ship in stable releases, and any dependency that still requires a patch when an RC is tagged must be removed from Drupal CMS entirely. Acceptable exit strategies include securing a maintainer commitment backed by granting commit access to a Drupal CMS committer, removing the dependent feature, or polyfilling the functionality via a contributed module. The parallel issue [#3488829](https://www.drupal.org/node/3488829) captured the governance discussion that led to this outcome.
