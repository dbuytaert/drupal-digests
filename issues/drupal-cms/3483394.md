## Summary

A new optional `drupal_cms_privacy_advanced` recipe is proposed for Drupal CMS, intended for sites that allow user account registration and need to meet data protection obligations. When applied, it installs the GDPR contrib module suite and configures it to: mark user account fields as personally identifiable (PII), add an "All your data" tab in user profiles where authenticated users can submit Subject Access Requests (SARs) and data removal requests, provide admins a task queue for processing those requests, and anonymize personal data in SQL database dumps so staging or development environments do not contain real user data. The recipe is designed as an optional complement to the basic privacy recipe. The MR was closed without merging; discussion in the thread shifted toward first extracting the underlying GDPR module components into standalone first-class modules before building the recipe on top of them.

Affects module maintainers and site owners.

- **New API:** New `drupal_cms_privacy_advanced` recipe enables PII field marking, Subject Access Requests, data removal requests, and SQL dump anonymization for user accounts via the GDPR contrib module suite.

## Technical details

The recipe (`recipes/drupal_cms_privacy_advanced`) installs four GDPR module sub-modules: `gdpr`, `gdpr_fields`, `gdpr_dump`, and `gdpr_tasks`. Because these sub-modules do not provide config schema, the recipe cannot use config actions and instead uses `config.strict: false` with raw YAML config files placed in `config/` alongside the recipe.

The `gdpr_fields` config marks the `name`, `mail`, `pass`, and `init` fields on the `user` bundle as PII, each with `rta: inc` (include in SARs output) and `rtf: anonymize` (anonymize on removal requests), using `username_anonymizer`, `email_anonymizer`, or `password_anonymizer` as appropriate. The `gdpr_dump` config maps those same columns in `users_field_data` to the matching anonymizers for SQL dump sanitization. The `gdpr_tasks` config sets the export directory to `private://forget`, which requires a configured Drupal private filesystem to function. The recipe grants authenticated users the `create gdpr tasks`, `view gdpr data summary`, and `view gdpr tasks` permissions. A `ComponentValidationTest` verifies clean apply and idempotency.

Hands-on testing by boromino (comment [#15883565](https://www.drupal.org/node/15883565)) surfaced blockers: exports fail silently when no private file path is configured in `settings.php`, the password hash is exported in SARs output, and there is no guard against processing a removal request for user 1. These issues, combined with concerns about the `gdpr` brand being EU-centric, prompted jurgenhaas to propose (comment [#15959008](https://www.drupal.org/node/15959008)) extracting `gdpr_fields` and `gdpr_tasks` into a standalone "PII API" module with separate, narrowly scoped use-case modules for SARs, JSON:API protection, and render output protection, informed by a conversation with yanniboi, the GDPR sub-module maintainer. The MR was closed pending that upstream work.
