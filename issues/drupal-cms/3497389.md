## Summary

Drupal CMS gains its first purpose-built Single Directory Component (SDC) for Experience Builder: a Heading component styled to match the Olivero design system. Site builders using the XB page editor can place a heading on any XB page, choose the semantic HTML element (h1 through h6), and see the appropriate Olivero-styled typography applied automatically. This makes the XB editing experience viable for content composition with branded headings.

## Impact

Affects module maintainers.

- **New API:** New `drupal_cms_olivero:heading` SDC component available for use in Experience Builder page layouts.

## Technical details

The component lives in `drupal_cms_olivero/components/heading/` and follows the standard SDC layout: a `heading.component.yml` schema, a `heading.css` stylesheet, and a `heading.html.twig` template, plus a thumbnail. The schema declares two required props: `text` (string) and `element` (enum of `h1` through `h6`). The Twig template renders `<{{ element }}{{ attributes.addClass(classes) }}>{{ text }}</{{ element }}>`, applying a base `.heading` class plus a modifier class such as `.heading--h1` driven by the chosen element.

The CSS defines distinct typographic rules for each heading level using Olivero CSS custom properties (`--sp2-5`, `--sp3`, `--sp4`, `--font-size-base`, `--font-size-s`, `--line-height-base`, `--line-height-s`). The default (unmodified) `.heading` style matches the h2 appearance. H5 and H6 add `text-transform: uppercase`. A responsive breakpoint at `43.75rem` is present but the grid-column span to columns 3–10 (from the Figma spec) is left as a `@todo` comment; the layout constraint was considered impractical to apply directly to the heading element without a wrapper.

The XB module ships its own `experience_builder:heading` component; during the review thread, balintbrews noted that XB-provided components can be disabled via config at `/admin/structure/component`, and that work was deferred to a separate issue ([#3498337](https://www.drupal.org/node/3498337)).
