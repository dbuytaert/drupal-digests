## Summary

The Drupal CMS installer now supports paid (premium) site templates alongside free ones. Premium templates display a price and a "Buy for $X" link that opens the vendor's purchase page in a new tab. After purchasing, the user enters a license key through a modal dialog; the installer validates the key against the vendor's Composer repository (and optionally a lightweight server-side validation endpoint), saves it as a bearer token in `auth.json`, and then proceeds with installation as normal. Free templates also receive a "Free" badge for visual parity. The blank starter template is always listed first and pre-selected. Site hosts and testing environments can place a `site-templates.php` file in the site directory to override the remotely curated template list entirely; this is documented as an official extension point.

## Impact

Affects module maintainers.

- **New API:** New `site-templates.php` extension point in the site directory: returning an iterable of template arrays overrides the remote `site-templates.yml` list in the Drupal CMS installer.

## Technical details

The central addition is the `SiteTemplate` final readonly value object (`Drupal\drupal_cms_installer\SiteTemplate`, marked `@internal`). It normalizes both locally present recipes and remotely hosted Composer packages into a single shape, carrying: `name`, `description`, `locator` (file path or Composer package name), `repository`, `authorization`, `links` (as `Link` objects), `price`, `purchaseUrl`, `keyValidationUrl`, and a `getScreenshot()` method that base64-encodes local `.webp` files so they can be served from outside the web root. The static factory `SiteTemplate::createFromRecipe()` wraps an existing `Recipe` object.

`SiteTemplateForm` was refactored to work with `SiteTemplate` instances throughout. The `access_key` field is now a `#type => container` holding one textfield per premium template, each shown conditionally via `#states` when its associated radio is selected. Validation logic moved from `submitForm()` into `validateForm()`: alternate Composer repositories are registered via `ComposerExecutor::execute('repository', 'add', ...)` using the xxh3 hash of the host+port as the repository name, the bearer token is written via `composer config bearer.<host> <key>`, then `composer show --all <package>` confirms the package is visible; on failure the token is unset and a form error is set. The site directory may also contain a `site-templates.php` returning an iterable of template arrays; when present it overrides the remote `site-templates.yml` list entirely and is treated as `@api`.

The `form_element__site_template` Twig template was substantially expanded to render links, price, purchase button, and a "Already purchased? Enter license key" button that opens a `<dialog>` element. The `premium-helper.js` behavior manages the dialog lifecycle, applies a UUID input mask, performs debounced server-side validation (via `Drupal.debounce` and a `fetch()` against `keyValidationUrl`), and updates status indicators (loading spinner, valid checkmark, invalid X). The `site-templates.yml` format was extended with `purchase.price`, `purchase.url`, `purchase.validation_url`, `links`, and `package.authorization` keys. A new `SiteTemplateInstallTest` build test provides end-to-end coverage of the bearer-token flow using an embedded PHP server that simulates a protected Composer repository.

## Upgrade

Site hosts who want to override the installer's template list can create a file at `sites/default/site-templates.php` (or whichever site directory is in use) that returns an iterable of template arrays:

```php
<?php
// sites/default/site-templates.php
return [
  'my_template' => [
    'name' => 'My Template',
    'description' => 'A custom site template.',
    'screenshot' => 'https://example.com/screenshot.webp',
    'package' => [
      'name' => 'vendor/my-template',
      'repository' => 'https://packages.example.com',
      'authorization' => 'bearer',
    ],
    'purchase' => [
      'price' => 99,
      'url' => 'https://example.com/buy',
      'validation_url' => 'https://example.com/validate', // optional
    ],
    'links' => [
      ['text' => 'Demo', 'url' => 'https://example.com/demo'],
    ],
  ],
];
```

When this file exists and returns an iterable, it replaces the remote `site-templates.yml` list entirely. This is an official extension point (`@api`).
