## Summary

When a new entity reference field is created on a Drupal CMS site that uses the `admin_ui` or `content_type_base` recipe, the Tagify tag-picker widget is now automatically selected as the default widget instead of the standard entity reference autocomplete. This brings newly created fields in line with the existing entity reference fields already configured to use Tagify by those recipes.

## Impact

No upgrade or configuration changes required.

## Technical details

Both the `admin_ui` and `content_type_base` recipes previously configured Tagify for existing entity reference fields but did not affect the widget chosen when editors add a new field. The Tagify module exposes a `set_default_widget` configuration key (added in related issue [#3512454](https://www.drupal.org/node/3512454)) that, when `true`, makes Tagify the automatic widget selection for any new supported field.

The initial implementation attempt tried to ship a custom copy of `tagify.settings` config inside each recipe directory. thejimbirch identified that this approach breaks when a recipe also installs the module: Drupal imports the module's own default config first, so a recipe-level config import for the same key fails. The correct approach is to install `tagify` in the recipe's `install` list and then use the `simpleConfigUpdate` config action to patch the single key:

```yaml
config:
  actions:
    tagify.settings:
      simpleConfigUpdate:
        set_default_widget: true
```

This pattern is applied identically in both `admin_ui/recipe.yml` and `content_type_base/recipe.yml`.
