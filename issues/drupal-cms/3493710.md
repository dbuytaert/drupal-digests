## Summary

Completing the Drupal CMS installer in Safari 18.2 (and some Chromium-based browsers) produced a "white screen of death" instead of redirecting to the dashboard. The site was actually installed correctly, and clearing caches manually resolved the error, but users had no way to know that. This fix makes the installer reliably deliver users to the dashboard after setup.

## Impact

No upgrade or configuration changes required.

## Technical details

The root cause was a race between stray browser requests and the installer's container management. Safari 18.2 (and similar browsers) makes extra requests for icon files that do not exist during installation. Unlike the installer's own requests, these stray requests are handled by the standard front controller (`index.php`) with `DrupalKernel` rather than `InstallerKernel`. `DrupalKernel` builds and dumps the service container to cache; `InstallerKernel` intentionally does not. The result is that the cached container reflects the state of Drupal mid-installation, with the `drupal_cms_installer` profile still registered and its hooks (including `drupal_cms_installer_theme_registry_alter()`) still present in the `hook_implementations_map` container parameter (a change introduced in Drupal core's OOP hooks work, [#3442009](https://www.drupal.org/node/3442009)).

When `drupal_cms_installer_uninstall_myself()` runs at the end of the install, it uninstalls the profile. The resulting container rebuild is not dumped (because the installer context disables dumping). The subsequent redirect to `/admin/dashboard/welcome` boots with the stale cached container, which still lists the installer profile's hooks. `drupal_cms_installer_theme_registry_alter()` fires, calls `getPath()` on an extension object that is now null, and throws a fatal error.

The fix adds a single call to `\Drupal::service('kernel')->invalidateContainer()` at the end of `drupal_cms_installer_uninstall_myself()` in `drupal_cms_installer.profile`. This marks the cached container as invalid before the redirect, so the first post-install request rebuilds it from scratch, without the uninstalled profile's hooks.

### Collaboration

The diagnosis that unlocked the fix came from znerol in comment #30. They identified that any stray request handled by `DrupalKernel` during installation would trigger a container dump, and that since the OOP hooks work began storing hook implementations in the container parameter `hook_implementations_map`, a stale container loaded from cache had a much worse effect than before. This explained both why the bug appeared with certain browser versions (Safari 18.2 makes additional requests for icon files that do not exist) and why it did not reproduce in Firefox. phenaproxima then proposed `\Drupal::service('kernel')->invalidateContainer()` as the targeted fix. rushikesh raval quickly tested the approach and confirmed it worked; rkoller ran three consecutive install rounds and verified the WSOD was gone. benjifisher had earlier done the patient work of upgrading macOS specifically to match rkoller's Safari version to confirm reproduction, and opened the MR.
