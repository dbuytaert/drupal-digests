## Summary

Drupal CMS now ships a `drupal_cms_analytics` recipe that sets up the `google_tag` module for website traffic tracking via Google Analytics and Google Tag Manager. Site builders apply the recipe and supply their Google Tag property ID (GT-, G-, AW-, GTM-, or DC- format); the recipe installs the module, imports its configuration, and wires the provided ID into a tracking container. Tracking is automatically suppressed on admin, batch, node editing, and user account pages. This is notably the first Drupal CMS recipe that requires a runtime input from the site builder to function.

## Impact

Affects module maintainers.

- **New API:** New `drupal_cms_analytics` recipe; apply with `drush recipe recipes/drupal_cms_analytics --input=drupal_cms_analytics.property_id=GTM-xxxxxxxx`

## Technical details

The new recipe lives at `recipes/drupal_cms_analytics` and is registered as a Composer path repository (`drupal/drupal_cms_analytics`) requiring `drupal/google_tag ^2.0.7`. Its `recipe.yml` defines a single `input` key, `property_id`, with `NotBlank` and `Regex` constraints validating the recognized tag ID prefixes. At apply time the recipe imports all `google_tag` default config, sets `google_tag.settings:default_google_tag_entity` to `drupal`, and uses a `set` action on `google_tag.container.drupal` to inject the validated ID into `tag_container_ids`. The bundled container config (`google_tag.container.drupal.yml`) ships with `consent_mode: false`, pre-defined `login` and `sign_up` events, and a `request_path` condition that negates tracking on `/admin*`, `/batch*`, `/node/add*`, node and user edit/delete paths. A `ComponentValidationTest` verifies idempotency, that a missing ID fails with "This value should not be blank.", and that a malformed ID fails with "This value is not valid." The `.gitlab-ci.yml` export-config job and the starter's `ComponentValidationTest` were both updated to pass `--input=drupal_cms_analytics.property_id=GTM-123456` when iterating over suggested add-on recipes. A related issue, [#3489649](https://www.drupal.org/node/3489649), was opened in parallel to handle enabling the Klaro privacy apps (GA and GTM) when this recipe is applied.
