## Summary

Drupal CMS sites that install the `drupal_cms_search` recipe now get a fully styled search experience when using the Olivero or `drupal_cms_olivero` theme. Previously the search form was entirely unstyled. The change adds a dark-background header search toggle for wide viewports, a compact mobile search bar, styled autocomplete suggestions, and cleaned-up search-result page styles, all matching the Drupal CMS visual design. The `drupal_cms_search` recipe is also now visible in Project Browser.

## Impact

Affects module maintainers.

- **New API:** Two functions exposed on `Drupal.drupal_cms_olivero`: `searchIsVisible()` and `toggleSearchVisibility(visibility)` for controlling the wide header search panel from external scripts.

## Technical details

The `drupal_cms_olivero` sub-theme received two new block components: `search-wide` (desktop) and `search-narrow` (mobile), each with dedicated CSS, Twig block templates, and SVG search icons. The wide component also ships a JavaScript behavior (`search-wide.js`) that manages toggling a `block-search-wide__wrapper` panel with `is-active`, handles `aria-expanded` on the toggle button, and closes the panel on outside clicks, focus loss, or Escape.

Because jQuery UI autocomplete appends its dropdown to the document body with no reliable DOM link back to the triggering input, `Drupal.behaviors.searchWideForm` works around this by adding the `search-wide-dropdown` CSS class to every `.search-api-autocomplete-search` element when the wide search opens, and removing it on close. This lets the dark-background autocomplete styles apply only when the header search is active. Two helper functions, `Drupal.drupal_cms_olivero.searchIsVisible` and `Drupal.drupal_cms_olivero.toggleSearchVisibility`, are exposed for potential external use.

The `drupal_cms_search` recipe places two separate search block instances instead of one: `block.block.search_form` in the `secondary_menu` region (the animated desktop header panel) and `block.block.search_form_primary` in the `primary_menu` region (the mobile inline form). The dual-block approach preserves correct keyboard focus order, a decision noted in the recipe comments. Additional fixes include adding a `'Search keywords'` label to the search view's exposed filter (previously empty), changing the submit label from "Find" to "Search", extending the `search_api_autocomplete` library to inject a `z-index: 110` override so autocomplete dropdowns appear above the flyout navigation, and adding `hook_preprocess_views_view__search` in `drupal_cms_olivero.theme` to attach the `view-search` library only on the search results view.
