# #3501069: Search broken without search_api_autocomplete

- **Project:** Drupal CMS
- **Type:** Critical bug report
- **Status:** Fixed
- **Discussion:** [#3501069](https://git.drupalcode.org/project/drupal_cms/-/work_items/3501069) · 2 contributors · 5 comments
- **Followers:** 4

## Summary

When a site admin uninstalled `search_api_autocomplete` from a Drupal CMS site that had the Search recipe applied, the site's search stopped working entirely. The Olivero theme's `search-wide.js` threw an uncaught `TypeError` trying to call `.focus()` on a null element, the search dropdown rendered incorrectly, and clicking the search field immediately redirected to the search results page rather than letting the user type. The Search recipe now no longer requires `search_api_autocomplete` or `simple_search_form`. Uninstalling those modules will not break search on sites running Drupal CMS.

## Impact

No upgrade or configuration changes required.

## Technical details

The original `drupal_cms_search` recipe (added in [#3468271](https://www.drupal.org/node/3468271)) configured a `simple_search_form` block with `input_type: search_api_autocomplete`, pointing at the `content_autocomplete` search configuration. Olivero's `search-wide.js` assumed this autocomplete widget was always present in the DOM; its `handleFocus` handler at line 86 called `.focus()` on the autocomplete input element unconditionally, producing `Cannot read properties of null (reading 'focus')` when the module was absent. The dependency was hard-coded with no fallback, making Drupal CMS incompatible with any `search_api` + `simple_search_form` setup that omits `search_api_autocomplete`.

The fix, delivered through issue [#3567584](https://www.drupal.org/node/3567584), replaced the entire search form implementation. `search_api_autocomplete` and `simple_search_form` were removed from the recipe's `install` list and `composer.json` requirements. In their place, `canvas`, `canvas_stark`, `eca_config`, and `eca_user` were added. The search Views display was converted from a routed page at `/search` to a block display (`views_block:search-block`) intended to be placed via Canvas. An ECA model (`eca.eca.search_exclude.yml`) now handles automatically enabling `search_api_exclude` for new content types as they are created. The permission grants for `use search_api_autocomplete for content_autocomplete` on the anonymous and authenticated roles were also removed.

## Upgrade

Existing sites that already applied `drupal_cms_search` retain their previously installed modules and configuration. Re-applying the updated recipe will not automatically remove `search_api_autocomplete` or `simple_search_form`; those must be uninstalled manually if desired. New applications of the recipe will install Canvas and ECA instead, and the search form is rendered as a block rather than appearing at `/search`. The anonymous and authenticated user permissions for `use search_api_autocomplete for content_autocomplete` are no longer granted by the recipe.
