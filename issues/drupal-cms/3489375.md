## Summary

When installing Drupal CMS, the installer now shows a "Create your account" step that requires the site owner to enter an email address and password for the administrator account. Previously, the installer silently generated a random password and set the admin email to `admin@hostname`, showing the credentials in a status message. This meant that if the password was not noted down and later forgotten, the "forgot password" recovery flow could not reach the admin because the auto-generated email was usually unreachable. Now the admin account is tied to a real, user-supplied email from the start.

## Impact

No upgrade or configuration changes required.

## Technical details

The old approach bypassed core's `SiteConfigureForm` by submitting it programmatically inside a custom `drupal_cms_installer_configure_site()` task, using `Random::machineName()` for the password and constructing a placeholder email. That function has been removed entirely.

The replacement is a `drupal_cms_installer_form_install_configure_form_alter()` hook that customises core's existing `install_configure_form` instead of bypassing it. The altered form shows only email and password fields (username is fixed to `admin` via a hidden field, site name flows from the earlier installer step, and the site email defaults to `no-reply@hostname`). Regional settings and update-notification fields are hidden. The form title becomes "Create your account" and the submit button becomes "Finish".

To make this work, the task ordering in `drupal_cms_installer_install_tasks_alter()` was changed: a new `drupal_cms_installer_install_myself` task now runs before `install_configure_form`, installing the `user` module and calling `install_install_profile()` so that the administrator role exists and is assignable to user 1 during that step. Previously, `install_install_profile` and `install_profile_themes` were both skipped.

A helper `_drupal_cms_installer_password_value()` value callback was added to the password element. It smooths over the fact that Drush and the `drupal install` CLI command submit `install_configure_form` programmatically and assume the password field is a `password_confirm` element (which sends `pass1`/`pass2` keys), whereas the new form uses a plain `password` element.

`drupal_cms_installer_uninstall_myself()` was also corrected: instead of manually clearing the `profile` key from `core.extension`, it now calls `ModuleInstallerInterface::uninstall(['drupal_cms_installer'])` and clears the messenger status queue there. An `administrator` role config file (`config/install/user.role.administrator.yml`) was added to the profile so the role is created automatically. The SQLite database path field was additionally moved into the "Advanced options" section of the database form.

### Collaboration

The fix had a notable design pivot. phenaproxima's first approach (MR !261) stored the submitted email and password in cleartext in a temporary file outside the web root, and explicitly flagged this as a security concern in the thread, requesting review from tim.plunkett. After further work, phenaproxima found that reordering the installer tasks and using a form alter hook made the temporary file entirely unnecessary, calling the revised approach "rock solid" from a security standpoint (MR !262, comment [#15884961](https://www.drupal.org/node/15884961)).
