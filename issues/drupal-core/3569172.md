## Summary

On multilingual sites with multiple language switcher blocks, parts of the page could randomly display in the wrong language. This was caused by PHP fiber concurrency where temporary language state changes during access checks in `ConfigurableLanguageManager::getLanguageSwitchLinks()` would leak to other fibers running concurrently. The fix contains these changes within the language switcher logic so they don't affect the rest of the page.

## Impact

No upgrade or configuration changes required.

## Technical details

`ConfigurableLanguageManager::getLanguageSwitchLinks()` temporarily changes `$this->negotiatedLanguages[LanguageInterface::TYPE_CONTENT]` and `$this->negotiatedLanguages[LanguageInterface::TYPE_INTERFACE]` to each link's language before checking URL access. This provides the correct language context for access checks. After all access checks complete, the original languages are restored.

When running in a fiber, calling `$url->access()` can suspend the fiber if entities are loaded as part of the access check. While the fiber is suspended, the temporary language state leaks to other fibers running concurrently, causing parts of the page to render in the wrong language.

The fix uses a "fiber trap" pattern (from [#3565937](https://www.drupal.org/node/3565937)). When `getLanguageSwitchLinks()` is running inside a fiber, the URL access checks are wrapped in child fibers. The child fiber is started and managed with a while loop that resumes it until it terminates. This contains the negotiated language changes within `getLanguageSwitchLinks()` so they don't escape to other fibers. When not running in a fiber, the access check runs normally.

A unit test simulates two fibers running in parallel: one running `getLanguageSwitchLinks()` and one checking the current language. The test forces the first fiber to suspend during `$url->access()` and verifies that the current language in the second fiber remains unchanged.
