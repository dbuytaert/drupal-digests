## Summary

Menu blocks in Drupal always computed the active trail and added the `route.menu_active_trails` cache context, even for menus like footer or top-level navigation where the active trail styling is never used. This caused many near-identical cached variations of those menus. A new optional `ignore_active_trail` configuration property on `SystemMenuBlock` lets site builders opt out of active trail computation for eligible blocks. When set, the menu block skips the active trail lookup and drops the `route.menu_active_trails` cache context, so the block can be served from a single cache entry on every page. The Umami demo profile's main menu, footer, and account menu blocks have been updated to use the new setting, and performance tests show meaningful reductions in queries and cache operations as a result.

## Impact

Affects module maintainers, front-end developers and site owners.

- **Config change:** New optional boolean `ignore_active_trail` key in `block.settings.system_menu_block:*` config schema; only stored when TRUE, absent means active trail is computed.
- **Theme change:** Menu blocks with `ignore_active_trail: true` no longer add `in_active_trail` classes to menu items; themes relying on those classes for styling should not enable this option.
- **New API:** New protected `SystemMenuBlock::shouldSetActiveTrail()` method and new `IgnoreActiveTrailConstraint` config schema constraint (valid only when `level=1` and `expand_all_items=true` or `depth=1`).

## Technical details

**Background:** `SystemMenuBlock::build()` and `getCacheContexts()` always called `menuActiveTrail->getActiveTrailIds()` and added `route.menu_active_trails:<menu_name>` to the cache context. For menus that show the same links on every page (all items expanded, or only one level deep), this created a separate cache entry per active trail variation even though the output was identical.

**New config key:** `ignore_active_trail` is added to the `block.settings.system_menu_block:*` schema as an optional boolean (`requiredKey: false`). It is only stored in config when `TRUE`; absent or unset is treated as `FALSE` (active trail is computed), preserving backward compatibility. No upgrade path or config update function is needed.

**`shouldSetActiveTrail()` method:** A new protected method in `SystemMenuBlock` consolidates the decision. It returns `TRUE` (active trail required) when any of these hold: `ignore_active_trail` is not set, `level` is not 1, or (`expand_all_items` is false and `depth` is not 1). The method is checked in both `build()` and `getCacheContexts()`. The `build()` method was also updated to use static `MenuTreeParameters` (bypassing `getCurrentRouteMenuTreeParameters`) when `level=1` and `depth=1`, a case that previously only applied when `expand_all_items` was true.

**Config schema constraint:** `IgnoreActiveTrailConstraint` and `IgnoreActiveTrailConstraintValidator` are added in `core/modules/system/src/Plugin/Validation/Constraint/`. The constraint (applied to the block settings mapping) rejects `ignore_active_trail: true` unless `level=1` AND (`expand_all_items=true` OR `depth=1`).

**Form:** A new `add_active_trail_class` checkbox is added to the `menu_levels` fieldset. Its value is the inverse of the `ignore_active_trail` config key (positive framing in the UI, negative storage in config). `blockSubmit()` translates the form value: when checked, `ignore_active_trail` is unset from configuration; when unchecked, it is set to `TRUE`. The checkbox uses `#states` to become `required` when the current level/depth/expand_all combination does not allow ignoring the active trail. `blockValidate()` is also added to enforce this server-side, covering AJAX submissions from Layout Builder that bypass browser validation.

**Umami demo profile:** `block.block.umami_account_menu`, `block.block.umami_main_menu`, and `block.block.umami_footer` all have `ignore_active_trail: true` set. The footer block's `depth` was also changed from `null` to `1` to satisfy the constraint.

**Optional key rationale:** Making the key optional avoids the need to update existing block config (including Layout Builder overrides, which have their own complexities tracked in [#3521221](https://www.drupal.org/node/3521221) and [#3530727](https://www.drupal.org/node/3530727)). A follow-up issue can make the key required once a "just-in-time update" API exists.

## Upgrade

No changes are required. Existing menu blocks without `ignore_active_trail` in their config continue to behave as before.

To opt in for a menu block that shows all items or only one level, set the option in the block configuration form ("Add a CSS class to ancestors of the current page" unchecked), or directly in config:

```yaml
# core/profiles/demo_umami/config/install/block.block.umami_footer.yml (example)
settings:
  level: 1
  depth: 1
  expand_all_items: false
  ignore_active_trail: true
```

The constraint requires `level: 1` AND either `expand_all_items: true` OR `depth: 1`. Setting `ignore_active_trail: true` with any other combination fails config schema validation.

### Collaboration

The UI label went through 74 comments and two formal usability team meetings ([#3536866](https://www.drupal.org/node/3536866), [#3540379](https://www.drupal.org/node/3540379)). The original implementation used the negative label "Show the same menu markup on every page". The usability team (benjifisher, rkoller, simohell) argued for a positive framing such as "Add the active-trail class". catch pushed back on introducing the technical "active trail" concept into the UI at all. The team landed on "Add a CSS class to ancestors of the current page" as a compromise: positive framing without requiring users to know what an active trail is. This created a mismatch between the form field name (`add_active_trail_class`) and the config property (`ignore_active_trail`), which is explicitly noted in code comments and handled by inverting the value in `blockSubmit()`. The decision to use `requiredKey: false` rather than an upgrade function came from the complexity of updating Layout Builder block overrides, documented in [#3521221](https://www.drupal.org/node/3521221).
