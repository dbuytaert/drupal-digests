## Summary

Menu blocks can now skip computing the active trail when menus show the same links on every page. Many themes do not style active menu trails in header or footer navigation, yet Drupal still calculates which menu items lead to the current page. This calculation requires database queries and creates separate cache entries for each unique active trail. The new option significantly reduces database queries and cache storage when enabled on menus that display consistently across pages, particularly benefiting sites with large navigation structures.

## Impact

- **Config change:** New optional `ignore_active_trail` boolean in menu block settings allows skipping active trail calculation when menu shows all items expanded or only first level.

## Technical details

The `SystemMenuBlock` plugin now supports an optional `ignore_active_trail` configuration property that skips active trail computation and omits the `route.menu_active_trails` cache context. The feature only applies when the menu block is configured to show all items expanded or when displaying only the first level with a depth of 1, since these configurations produce identical link structures regardless of the current page.

The block configuration form adds a checkbox labeled "Add a CSS class to ancestors of the current page" that controls this behavior. Form states make the checkbox required when active trail calculation is necessary for correct menu rendering. The form element name is `add_active_trail_class` (positive statement) while the configuration key is `ignore_active_trail` (negative statement) to simplify both UI and API logic. The `blockSubmit()` method inverts the form value before saving.

The `shouldSetActiveTrail()` method centralizes the logic determining when active trail processing is required. The `getCacheContexts()` method conditionally includes the active trail context based on this logic. A custom `IgnoreActiveTrailConstraint` validates that the configuration property is only set to TRUE when compatible with other menu settings.

The configuration schema marks `ignore_active_trail` with `requiredKey: false`, allowing gradual adoption without requiring an upgrade path for existing blocks. When unset, the property defaults to FALSE (active trail enabled). The Umami profile's header, footer, and account menu blocks enable this optimization, demonstrating performance improvements in test results with reductions of 12+ queries per request.

## Upgrade

No action required. Existing menu blocks continue calculating active trails as before. To optimize performance, edit menu blocks that show all items expanded or only the first level and uncheck "Add a CSS class to ancestors of the current page" if your theme does not style the active trail differently on each page.

The change record at [#3533514](https://www.drupal.org/node/3533514) provides additional details.

## Lessons learned

- When caching happens at multiple layers, cache contexts that don't affect the output still fragment the cache. For blocks that display the same markup regardless of context (like fully-expanded menus ignoring active trail), omitting unnecessary cache contexts consolidates identical variations into a single cache entry. *(via catch)*
- When adding new configuration properties where NULL and a default value should be treated identically, making the property optional (not required in schema) avoids the complexity of update paths across multiple storage implementations (config entities, Layout Builder overrides, exported config). The system can gradually introduce the property on save while maintaining backward compatibility. *(via catch, godotislate, alexpott)*
- When designing configuration UI, avoid exposing internal technical concepts like 'active trail' directly to users. Frame options in terms of visible outcomes ('show the same markup on every page' or 'add a CSS class to parent links') rather than system implementation details. *(via catch, benjifisher)*
