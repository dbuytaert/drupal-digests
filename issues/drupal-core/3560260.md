## Summary

The oEmbed iframe controller was accidentally preventing responses from being cached due to incorrect cache metadata handling. When displaying remote video embeds from YouTube, Vimeo, and other oEmbed providers, every request would bypass the cache entirely. This caused unnecessary load on external services and slower page performance for visitors. The fix ensures that embedded media responses are now properly cached based on the provider's cache settings, typically improving response times and reducing external API calls.

## Impact

No upgrade or configuration changes required.

## Technical details

The `OEmbedIframeController::render()` method was calling `$response->addCacheableDependency(Url::createFromRequest($request))` to vary responses based on the request URL. However, `Url` does not implement `CacheableDependencyInterface`, so this silently set the cache max age to 0, making all responses uncacheable.

The fix replaces this with explicit `CacheableMetadata` that adds the `url.site` and `url` cache contexts to vary responses appropriately. The response now properly inherits the cache max age from the oEmbed resource itself (typically set by the provider), capped at 5 years maximum.

Error handling was also improved by switching from `BadRequestHttpException` to `CacheableBadRequestHttpException` for validation failures, ensuring that even error responses respect cache metadata. When a `ResourceException` occurs while fetching the resource, the response max age is explicitly set to 0 to prevent caching errors.

Three new kernel tests verify that the response cache max age matches the resource's cache age, that errors are not cached, and that excessively long cache ages are capped at 5 years.
