## Summary

Theme access checks are now slightly faster on every HTML page request. `ThemeAccessCheck` previously called `ThemeHandlerInterface::listInfo()` each time it needed to verify a theme is installed, which triggered at least two cache lookups per request. It now reads the same information directly from the `container.themes` container parameter, which is already in memory, reducing cache gets by two across most page loads.

## Impact

Affects module maintainers.

- **Deprecation:** Passing `ThemeHandlerInterface` to `ThemeAccessCheck::__construct()` is deprecated in drupal:11.4.0 and removed in drupal:12.0.0; pass the `%container.themes%` container parameter array instead.

## Technical details

`ThemeAccessCheck` is an access checker tagged with `_access_theme` in `core.services.yml`. Its `checkAccess()` method is called during theme negotiation to confirm the candidate theme is installed. Previously it depended on `ThemeHandlerInterface` and called `listInfo()` on every check; `listInfo()` reads from the cache and allocates memory, producing measurable overhead visible in xhprof profiles.

The fix switches the service definition in `core.services.yml` from `arguments: ['@theme_handler']` to `arguments: ['%container.themes%']`. The constructor parameter changes from `ThemeHandlerInterface $theme_handler` to an array `$themes`, and `checkAccess()` is reduced to `isset($this->themes[$theme])`. A BC shim in the constructor detects when a non-array (i.e., a legacy `ThemeHandlerInterface`) is passed, emits a `@trigger_error` deprecation notice, and falls back to `\Drupal::getContainer()->getParameter('container.themes')`. This protects sites that have a stale container cache from a hard failure across a deployment.

A subtle timing problem was also addressed. Access checker services are instantiated when the event dispatcher needs them, which happens before any controller method runs. A Nightwatch test support controller installs a new theme and immediately sets it as the admin theme in a single request, then the framework renders a response in that same theme. Because `ThemeAccessCheck` was already constructed before the controller executed, the newly installed theme was absent from its frozen theme list. The workaround, added to the `installTheme()` helper, calls `\Drupal::service('theme.manager')->getActiveTheme($route_match)` before calling `$this->themeInstaller->install()`, forcing theme negotiation to run while the old theme list is still valid. A detailed code comment explains why this ordering matters.

Performance test fixtures (`testGetIndividual`, `testLogin`, `testLoginBlock`) drop by two cache gets per scenario, confirming the reduction comes directly from eliminating the `listInfo()` call. A follow-up issue [#3584064](https://www.drupal.org/node/3584064) was opened to investigate whether the theme access check during negotiation is even necessary.

## Upgrade

Code that constructs `ThemeAccessCheck` directly or subclasses it while passing a `ThemeHandlerInterface` must be updated before Drupal 12.

```php
// Before (deprecated in drupal:11.4.0, removed in drupal:12.0.0)
new ThemeAccessCheck($theme_handler);

// After: pass the container.themes parameter
new ThemeAccessCheck(\Drupal::getContainer()->getParameter('container.themes'));
```

In a service definition, change the argument from `'@theme_handler'` to `'%container.themes%'`.
