## Summary

When editing a node that already had a menu link or a URL alias set, the "Menu settings" and "URL path settings" details panels in the node edit sidebar would auto-expand, distracting editors and behaving inconsistently with other panels such as "Authoring information" and "Promotion options" that always stayed collapsed. Both panels now stay collapsed by default regardless of whether values are already set. Editors can still see at a glance that values exist through the detail summary text and can expand the panel deliberately when they need to change something.

## Impact

No upgrade or configuration changes required.

## Technical details

The node edit form exposes several collapsible `#type => 'details'` elements in the advanced sidebar. Two of them were unconditionally auto-opened when existing data was detected: in `MenuUiHooks::formNodeFormAlter()` (in `core/modules/menu_ui/src/Hook/MenuUiHooks.php`), the "Menu settings" details element carried `'#open' => (bool) $defaults['id']`, which opened the panel whenever a menu link ID was already present. In `PathWidget::formElement()` (in `core/modules/path/src/Plugin/Field/FieldWidget/PathWidget.php`), the "URL path settings" details element carried `'#open' => !empty($items[$delta]->alias)`, opening it whenever a URL alias existed. The fix is a one-line removal in each file: both `#open` assignments are deleted, leaving the `details` elements collapsed by default (the HTML `open` attribute is simply absent). The usability team reviewed the proposed change and reached a clear consensus that all details elements in the advanced sidebar should be consistently collapsed, relying on the detail summaries for situational awareness. A follow-up for "Comment settings" (which also auto-expands but lacks a summary line) was deferred to a separate issue ([#3571616](https://www.drupal.org/node/3571616)). Functional test coverage was added in `MenuUiNodeTest::testMenuNodeFormWidget()` and `PathAliasTest::testNodeAlias()`, using `assertSession()->elementAttributeNotExists('css', 'details.menu-link-form', 'open')` and `assertSession()->elementAttributeNotExists('css', 'details.path-form', 'open')` to assert the `open` attribute is absent after saving nodes with menu and alias values.

### Collaboration

The usability team held a dedicated meeting ([#3568606](https://www.drupal.org/node/3568606)) to review this issue. rkoller wrote a detailed summary: the group considered expanding all panels on node creation but decided against it due to increased vertical height and cognitive load. They concluded that the detail summaries provide sufficient situational awareness and that collapsing all panels by default is the right behavior. rkoller also caught during testing that `comment settings` and `url alias` were still expanding in the MR, prompting smustgrave to expand scope to include the path alias widget. dcam raised a concern that the initial tests were checking the `aria-expanded` attribute set by `details-aria.js` rather than the native HTML `open` attribute; smustgrave switched the assertions accordingly, though the first attempt targeted the wrong element. After a second round of feedback from dcam, smustgrave corrected the selectors and dcam confirmed the result looked good.
