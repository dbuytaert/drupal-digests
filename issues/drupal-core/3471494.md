## Summary

Drupal 11.1 gains a new experimental Icon Management API built directly into Drupal core. Modules and themes can now declare "icon packs" through YAML files, and any code can render icons using a unified render element or Twig function. Three built-in extractor types (path, SVG, SVG sprite) are included. The API is marked experimental and internal, but it establishes a common foundation that replaces the current scattered, hardcoded approach to icons across core and enables contrib modules to build consistent icon integrations on top of a shared standard.

## Impact

Affects module maintainers.

- **New API:** New experimental Icon Management API: `IconPackManager`, `IconExtractorPluginManager`, `IconDefinition`, `IconCollector`, `Icon` render element, `icon()` Twig function, and `hook_icon_pack_alter` in `Drupal\Core\Theme\Icon` and related namespaces; all marked `@internal` and experimental.

## Technical details

Before this change, Drupal core had over 140 icons scattered across many directories and referenced with hardcoded paths in CSS, PHP, and Twig, with no shared API for declaring or rendering icons. Multiple contrib modules were independently duplicating plugin managers, render elements, and Twig functions to fill the gap. The new API, inspired by the contrib `ui_icons` module, was placed directly in `Drupal\Core` (rather than as an experimental module) to avoid the migration overhead experienced when promoting the Single Directory Components experimental module ([#3352256](https://www.drupal.org/node/3352256)).

**Icon pack declaration.** Any module or theme drops an `EXTENSION_NAME.icons.yml` file to register one or more icon packs. Each pack entry specifies a required `extractor` plugin ID and `template` (an inline Twig string), plus optional `config` (passed to the extractor), `settings` (JSON schema scalars that generate a settings form), and a `library` (asset library ID). Declarations are validated against `core/assets/schemas/v1/icon_pack.schema.json`. Packs can be modified via `hook_icon_pack_alter`.

**Plugin managers.** `IconPackManager` (service `plugin.manager.icon_pack`, class `Drupal\Core\Theme\Icon\Plugin\IconPackManager`) handles YAML discovery across modules and themes. `IconExtractorPluginManager` (service `plugin.manager.icon_extractor`, class `Drupal\Core\Theme\Icon\IconExtractorPluginManager`) discovers extractor plugins via PHP `#[IconExtractor]` attributes from `Plugin/IconExtractor` subdirectories and fires `hook_icon_extractor_info_alter`.

**Icon definitions.** `IconDefinition` implements `IconDefinitionInterface` and holds `pack_id`, `icon_id`, `template`, optional `source` (URL or path), optional `group`, and extractor-supplied `data`. Full icon IDs use the `pack_id:icon_id` format. `IconDefinition::getRenderable()` is a static shortcut to build a render array without needing to look up the icon first.

**Extractor plugins.** `IconExtractorBase` (in `Drupal\Core\Theme\Icon`) is the abstract base class; `IconExtractorWithFinder` extends it for file-based extractors, using the `IconFinder` service. `IconFinder` wraps Symfony Finder and supports `{icon_id}` and `{group}` placeholders in source path patterns, relative or absolute paths, and remote URLs. Local file discovery is restricted to `svg`, `png`, and `gif` extensions for security. Three built-in extractors live in `Drupal\Core\Theme\Plugin\IconExtractor`: `PathExtractor` (references image files by URL or path), `SvgExtractor` (reads SVG file content inline), and `SvgSpriteExtractor` (references symbols inside an SVG sprite). Each extractor implements `discoverIcons()` to return a flat array keyed by full icon ID, and `loadIcon()` to reconstruct an `IconDefinitionInterface` object on demand.

**Lazy caching via `IconCollector`.** Rather than loading all icons eagerly, `IconCollector` extends `CacheCollector` and is tagged `needs_destruction` in `core.services.yml`. Individual icons are loaded on first access by invoking the appropriate extractor's `loadIcon()` and stored for persistence. The `needs_destruction` tag causes `::destruct()` to run during kernel teardown, making cache writes additive (merging with any other sets that occurred during the request). Icons are stored in pack definitions as minimal arrays rather than `IconDefinition` objects, reducing cache bin size by roughly 8×.

**Rendering.** The `Icon` render element (`#[RenderElement('icon')]` in `Drupal\Core\Render\Element\Icon`) accepts `#pack_id`, `#icon_id`, and `#settings`. Its `preRenderIcon()` callback looks up the icon via `plugin.manager.icon_pack`, then renders it as an `inline_template` whose context is merged from extractor data, settings, and the resolved `icon_id` and `source`. `IconsTwigExtension` (in `Drupal\Core\Template`) registers the `icon(pack_id, icon_id, settings)` Twig function.

**Settings forms.** `IconExtractorSettingsForm` converts JSON schema scalar settings declared in the YAML definition into Drupal Form API elements: `boolean` to checkbox, `number`/`integer` to number field, `string` to textfield (with `color` format support and `enum` to select).

All classes are tagged `@internal` and marked as experimental.

### Collaboration

A substantive architectural debate arose when matthieuscarset proposed modelling icons as content entities, arguing it would eliminate the need for new plugin infrastructure. nod_ countered that icon pack update functions would be nearly impossible to write correctly across environments, and that content you cannot edit safely is not really content. catch added that content entities cannot be version-controlled in the normal deployment workflow. pdureau completed the argument by mapping icons to Drupal's code/config/content hierarchy: because icons need to be referenceable from content, config, and code alike, they must themselves live at the code layer.

The decision to land the API directly in `Drupal\Core` rather than as an experimental module was made at the Barcelona DrupalCon meeting (recorded by pdureau in the thread) to avoid repeating the multi-step promotion process required by the Single Directory Components experimental module ([#3352256](https://www.drupal.org/node/3352256)).

During the final review, catch identified that the `IconCollector` service was missing its `needs_destruction` tag, added it, and explained why the tag is necessary: without it the `::destruct()` merge step never fires, so concurrent cache writes can silently overwrite each other rather than accumulating additively. larowlan conducted multiple formal review passes across the thread and wrote the initial change record.
