## Summary

Drupal 11.1 ships an experimental Icon Management API baked directly into `Drupal\Core`. Any module or theme can now register one or more icon packs by dropping an `EXTENSION_NAME.icons.yml` file in its root directory. Icons declared this way can be rendered in PHP with the new `icon` render element or in Twig with the `icon()` function. Three built-in extractor types cover the most common cases: path-based image references, inline SVG, and SVG sprites. The goal is to replace the existing situation where Drupal core has over 140 icons referenced with hardcoded paths scattered across CSS, PHP, and Twig, and where contrib modules independently duplicate the same plugin managers, render elements, and Twig functions to fill the gap. All classes are tagged `@internal` and the feature is marked experimental, so its interfaces may still evolve.

## Technical details

The core problem was the absence of any shared contract for declaring or rendering icons. Drupal core's own icons were referenced by hardcoded paths in CSS backgrounds, PHP strings, and Twig templates, with no way for themes to override them and no uniform rendering mechanism for contrib to build on.

Two new plugin systems anchor the API. `IconPackManager` (service `plugin.manager.icon_pack`, class `Drupal\Core\Theme\Icon\Plugin\IconPackManager`) uses YAML discovery to load `EXTENSION_NAME.icons.yml` files from all enabled modules and themes. Each pack entry requires an `extractor` plugin ID and a Twig `template` string; optional keys are `config` (passed to the extractor), `settings` (JSON schema scalars that `IconExtractorSettingsForm` converts to Drupal Form API elements), and `library` (an asset library ID). Pack definitions are validated against `core/assets/schemas/v1/icon_pack.schema.json` and can be modified via `hook_icon_pack_alter`. `IconExtractorPluginManager` (service `plugin.manager.icon_extractor`, class `Drupal\Core\Theme\Icon\IconExtractorPluginManager`) discovers extractor implementations from `Plugin/IconExtractor` subdirectories using the PHP `#[IconExtractor]` attribute and fires `hook_icon_extractor_info_alter`.

`IconDefinition` implements `IconDefinitionInterface` and carries `pack_id`, `icon_id`, `template`, optional `source`, optional `group`, and extractor-supplied `data`. Full icon IDs use the `pack_id:icon_id` format. `IconDefinition::getRenderable()` is a static shortcut for building a render array without first performing a lookup.

Rather than loading all icons eagerly on every request, `IconCollector` extends `CacheCollector` and is registered with the `needs_destruction` tag in `core.services.yml`. Without that tag `::destruct()` never fires at kernel teardown and concurrent cache writes silently overwrite one another instead of accumulating additively; the tag was identified as missing and added during final review. Icons are stored in pack definitions as minimal flat arrays rather than `IconDefinition` objects, reducing the cache bin footprint by roughly 8x. On cache miss, `IconCollector` instantiates the appropriate extractor plugin and calls its `loadIcon()` to reconstruct the `IconDefinitionInterface` object, then persists the result.

Rendering is handled by the `Icon` render element (`#[RenderElement('icon')]` in `Drupal\Core\Render\Element\Icon`), which accepts `#pack_id`, `#icon_id`, and `#settings`. Its `preRenderIcon()` callback looks up the icon via `plugin.manager.icon_pack` and renders it as an `inline_template`, merging extractor data, caller settings, and the resolved `icon_id` and `source` into the template context. `IconsTwigExtension` in `Drupal\Core\Template` registers the `icon(pack_id, icon_id, settings)` Twig function that returns the same render array. Three extractors in `Drupal\Core\Theme\Plugin\IconExtractor` cover built-in needs: `PathExtractor`, `SvgExtractor`, and `SvgSpriteExtractor`. `IconFinder` wraps Symfony Finder and supports `{icon_id}` and `{group}` placeholders in source path patterns; local discovery is restricted to `svg`, `png`, and `gif` extensions for security.

The API was placed directly in `Drupal\Core\Theme\Icon` rather than as an experimental module to avoid repeating the multi-step namespace migration required when the Single Directory Components experimental module was promoted to core ([#3352256](https://www.drupal.org/node/3352256)). A follow-up commit removed a test fixture SVG whose filename contained special characters that broke `git checkout` on Windows.

### Collaboration

A substantive debate arose when matthieuscarset proposed modelling icons as content entities, arguing it would eliminate the need for new plugin infrastructure. nod_ countered that icon pack update functions would be nearly impossible to write correctly across environments, and that content you cannot edit safely is not really content. catch added that content entities cannot be version-controlled in the normal deployment workflow. pdureau completed the argument by mapping icons to Drupal's code/config/content hierarchy: because icons need to be referenceable from content, config, and code alike, they must themselves live at the code layer.

The decision to land the API directly in `Drupal\Core` rather than as an experimental module was made at the Barcelona DrupalCon meeting, recorded by pdureau in the thread, to avoid repeating the multi-step promotion process required by the Single Directory Components module ([#3352256](https://www.drupal.org/node/3352256)).

During final review, catch identified that the `IconCollector` service was missing its `needs_destruction` tag, added it, and explained precisely why the tag is necessary: without it the `::destruct()` merge step never fires, so concurrent cache writes can silently overwrite each other rather than accumulating additively. larowlan conducted multiple formal review passes across the thread and wrote the initial change record draft.
