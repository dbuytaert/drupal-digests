## Summary

Drupal's database Schema layer now routes all DDL statements (CREATE TABLE, ALTER TABLE, DROP TABLE, ADD/DROP INDEX, etc.) through a new protected `Schema::executeDdlStatement()` method instead of calling `Connection::query()` directly. On databases that do not support transactional DDL (MySQL and MariaDB), executing a DDL statement inside an active transaction causes the database engine to silently auto-commit. The new method detects this and immediately calls `TransactionManager::voidClientTransaction()` so that any still-open Drupal `Transaction` objects no longer attempt further SQL transaction operations when they go out of scope. Attempting to call `Transaction::rollBack()` after a DDL statement has already committed the transaction now triggers an `E_USER_WARNING` rather than silently succeeding with no effect. This change affects database driver developers and any code that performs DDL operations inside a transaction; no site-configuration changes are required.

## Impact

Affects module maintainers.

- **API change:** `TransactionManagerInterface::addPostTransactionCallback()` callback `$success` parameter semantics are now formally documented: TRUE for committed or auto-committed-after-DDL, FALSE for rolled back or failed.
- **New API:** New protected `Schema::executeDdlStatement()` method on the abstract Schema base class for use by database driver Schema subclasses.

## Technical details

MySQL (and MariaDB) do not support transactional DDL: issuing any DDL statement (ALTER TABLE, DROP TABLE, CREATE INDEX, etc.) while a transaction is open causes the database engine to commit the current transaction immediately. PostgreSQL and SQLite support transactional DDL and are not affected by the implicit commit, but the code path is now unified for all drivers.

**New `Schema::executeDdlStatement()` method** (`core/lib/Drupal/Core/Database/Schema.php`): A `protected` method on the abstract base class that (1) delegates to `Connection::query()` to run the statement, then (2) checks `Connection::supportsTransactionalDDL()` and `TransactionManager::inTransaction()`. If the driver does not support transactional DDL and a transaction is currently tracked, it calls `TransactionManager::voidClientTransaction()`. This moves the responsibility for voiding the transaction stack into the Schema layer rather than leaving it to reactive detection in driver-level `TransactionManager` implementations.

**Driver-level Schema call-site updates**: Every direct `$this->connection->query()` call executing a DDL statement in `mysql/Schema.php`, `pgsql/Schema.php`, `sqlite/Schema.php`, and the base `Schema.php` (`createTable` loop) was replaced with `$this->executeDdlStatement()`. The method was intentionally left `protected` and on `Schema` rather than being made a public method on `Connection`, to keep it as an implementation detail of the Schema layer.

**`Truncate` query** (`core/lib/Drupal/Core/Database/Query/Truncate.php`): No routing change was needed here because `Truncate::__toString()` already falls back to a `DELETE` when `Connection::inTransaction()` returns true, meaning `TRUNCATE` (a DDL statement) is never issued inside a transaction.

**Rollback-after-DDL handling** in `TransactionManagerBase::rollback()` (`core/lib/Drupal/Core/Database/Transaction/TransactionManagerBase.php`): If `getConnectionTransactionState()` returns `ClientConnectionTransactionState::Voided` at rollback time, the method now sets the state to `ClientConnectionTransactionState::RollbackFailed` and emits an `E_USER_WARNING` ("Transaction::rollBack() failed because of a prior execution of a DDL statement.") instead of silently proceeding. A `@todo` comment notes this will be changed to throw a `TransactionOutOfOrderException` in Drupal 12.0.0.

**Post-transaction callback `$success` semantics**: The documentation on `TransactionManagerInterface::addPostTransactionCallback()` was significantly expanded. The `$success` parameter is now clearly defined: `TRUE` means the transaction was successfully committed (or auto-committed after DDL with no subsequent rollback attempt, i.e., `Voided` state); `FALSE` means the transaction was rolled back or failed. This aligns with the `TransactionManagerBase` callback invocation logic at line 543, which passes `true` for both `Committed` and `Voided` states.

**`Transaction.php` doc cleanup** (`core/lib/Drupal/Core/Database/Transaction.php`): Outdated class-level documentation referring to databases that do not support transactions at all was removed. A `__destruct()` docblock was added clarifying that destruction leads to a `COMMIT` (root item) or `RELEASE SAVEPOINT` (savepoint item).

**New kernel tests** in `DriverSpecificTransactionTestBase.php`: Three new test methods cover the new behavior: `testRollbackAfterDdlStatementForNonTransactionalDdlDatabase()` verifies the user warning and `RollbackFailed` state; `testRootTransactionEndCallbackCalledAfterDdlAndRollbackForTransactionalDdlDatabase()` verifies that on PostgreSQL/SQLite a DDL-then-rollback sequence correctly delivers `$success = FALSE` to callbacks; and `testRootTransactionEndCallbackFailureUponDdlAndRollbackForNonTransactionalDdlDatabase()` verifies the MySQL/MariaDB path where the auto-commit has already occurred. The prerequisite fix for post-transaction callback ordering ([#3440848](https://www.drupal.org/node/3440848)) landed before this issue was committed.

## Upgrade

Custom database drivers that subclass `\Drupal\Core\Database\Schema` should replace direct calls to `$this->connection->query()` for DDL statements with the new `$this->executeDdlStatement()` method:

```php
// Before
$this->connection->query('ALTER TABLE {my_table} ADD COLUMN foo INT');

// After
$this->executeDdlStatement('ALTER TABLE {my_table} ADD COLUMN foo INT');
```

Code that registers post-transaction callbacks via `TransactionManagerInterface::addPostTransactionCallback()` should note the clarified `$success` semantics: `TRUE` now explicitly means the transaction was committed (including auto-committed after DDL with no rollback attempt); `FALSE` means rolled back or failed. No change is needed if the callback already treats `FALSE` as "something went wrong."

Code that calls `Transaction::rollBack()` after executing DDL on MySQL or MariaDB will now trigger an `E_USER_WARNING`. In Drupal 12.0.0 this will become a `TransactionOutOfOrderException`. DDL should not be executed inside a user-managed transaction on non-transactional-DDL databases.

### Collaboration

The design went through three substantive pivots driven by alexpott's review. First, alexpott questioned whether using a separate connection for DDL would sidestep the problem entirely; mondrake explained that MySQL error 1412 ("Table definition has changed, please retry transaction") affects any concurrent active transaction on the server regardless of which connection issues the DDL, ruling that approach out. Second, alexpott pushed back on the original design that surfaced `executeDdlStatement` as a public method on `Connection`, arguing this mixed responsibilities; mondrake reworked it into a `protected` method on the abstract `Schema` class, which also eliminated the need to modify `Truncate`. Third, alexpott and mondrake debated what `$success` should mean for post-transaction callbacks when MySQL auto-commits a transaction mid-DDL and the caller then attempts a rollback. Alexpott argued `$success = FALSE` was correct for a failed rollback scenario; mondrake clarified that `Voided` (DDL happened, no rollback attempted) should remain `TRUE` while `RollbackFailed` (DDL happened, rollback attempted) should be `FALSE`, and the existing line 543 logic already reflected this. The rollback warning was also changed from a deprecation notice to an `E_USER_WARNING` at alexpott's suggestion, reserving the exception for Drupal 12.
