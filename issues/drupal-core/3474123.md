## Summary

The logo-related settings in the Navigation module's `navigation.settings` configuration object have been reorganized from five flat top-level keys (`logo_provider`, `logo_managed`, `logo_max_filesize`, `logo_height`, `logo_width`) into a single nested `logo` mapping, with maximum-dimension constraints grouped under a `logo.max` sub-mapping. This aligns the Navigation module's config schema with patterns used elsewhere in Drupal core such as `system.theme.global`. Existing installations are migrated automatically by a database update function. The module was in beta at the time of this change, so a change record was also required.

## Impact

Affects site owners.

- **Database update:** `navigation_update_11001()` migrates existing `navigation.settings` config from the old flat structure to the new nested mapping.
- **Config change:** Five flat `logo_xxx` keys in `navigation.settings` replaced by a nested `logo` mapping with sub-keys `logo.provider`, `logo.managed`, `logo.max.filesize`, `logo.max.height`, and `logo.max.width`.

## Technical details

The `navigation.settings` config object previously stored all logo-related values as flat top-level keys with a `logo_` prefix. Because the number of logo settings was growing, the issue proposed restructuring them into a nested mapping to match conventions used by similar schemas like `system.theme.global`.

**Schema and install config** (`core/modules/navigation/config/schema/navigation.schema.yml`, `core/modules/navigation/config/install/navigation.settings.yml`): The five flat keys were replaced with a single `logo` mapping. Within `logo`, `provider` and `managed` are direct children, while the three numeric constraints (`filesize`, `height`, `width`) are grouped under a `logo.max` sub-mapping. Both the `logo` and `logo.max` levels gained `ValidKeys: '<infer>'` constraints to enforce strict key validation.

**Update hook** (`core/modules/navigation/navigation.install`): `navigation_update_11001()` reads the old flat keys from `navigation.settings` and calls `setData()` to replace the entire config with the new nested structure, then calls `->save(TRUE)` because the data is known to be correctly typed after the transformation. The use of `save(TRUE)` (trusted save) was explicitly requested by the reviewer.

**PHP code** (`core/modules/navigation/src/Form/SettingsForm.php`, `core/modules/navigation/src/NavigationRenderer.php`): All `->get()` and `->set()` calls were updated to use the new dot-notation paths: `logo.provider`, `logo.managed`, `logo.max.filesize`, `logo.max.height`, `logo.max.width`. The `#config_target` element attribute in `SettingsForm::buildForm()` was similarly updated from `navigation.settings:logo_provider` to `navigation.settings:logo.provider`.

**Tests** (`NavigationLogoTest`, `NavigationSettingsFormTest`): Config key references in test setup and assertions were updated to the new dot-notation paths.

The change was committed to 11.x and 11.0.x, then later cherry-picked to 10.4.x because navigation is experimental and the maintainers decided to keep all navigation changes in sync across active branches until the module reaches stable.

## Upgrade

Any custom code that reads from `navigation.settings` directly must update its config key references. Replace the old flat keys with the new nested dot-notation equivalents:

```php
// Before
$config->get('logo_provider');
$config->get('logo_managed');
$config->get('logo_max_filesize');
$config->get('logo_height');
$config->get('logo_width');

// After
$config->get('logo.provider');
$config->get('logo.managed');
$config->get('logo.max.filesize');
$config->get('logo.max.height');
$config->get('logo.max.width');
```

The same substitution applies to `->set()` calls and any `#config_target` form element attributes referencing these keys. Run `drush updatedb` (or equivalent) to execute `navigation_update_11001()` and migrate existing stored configuration to the new structure.
