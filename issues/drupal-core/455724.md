## Summary

The `check_markup()` function in `filter.module` is deprecated as of Drupal 11.4.0 and will be removed in Drupal 13.0.0. The function has no direct procedural replacement. The recommended approach is to return a `processed_text` render array directly rather than flattening it to a markup string, because flattening discards cacheability metadata (cache tags, contexts, max-age) that filters can attach. Module code that currently calls `check_markup()` and renders the result must be updated before upgrading to Drupal 13.

## Impact

Affects module maintainers.

- **Deprecation:** `check_markup()` in `filter.module` is deprecated in drupal:11.4.0 and removed in drupal:13.0.0; use a `processed_text` render array instead.

## Technical details

The function `check_markup()` was a thin convenience wrapper: it built a `['#type' => 'processed_text', '#text' => ..., '#format' => ..., ...]` render array and immediately flattened it to a `MarkupInterface` string via the renderer. That flattening step is the problem, because it silently drops any cacheability metadata that filter plugins attach during processing.

The commit adds a `@trigger_error()` deprecation notice at the top of `check_markup()` and documents the deprecation in the docblock. There is explicitly no new service or function provided as a replacement; callers are expected to pass the `processed_text` render array further up the stack and let the rendering system handle it.

The one production caller updated in this commit is the Views `MarkupField` handler (`core/modules/views/src/Plugin/views/field/MarkupField.php`), which now returns a `processed_text` render array directly. Test code that needed to assert on string output was updated to use a new test-only trait `Drupal\Tests\filter\Traits\ProcessedTextTestTrait`, whose `processText()` helper replicates what `check_markup()` did but is explicitly documented as unsuitable for production use. The issue was originally filed in 2009 as a renaming request, and was rescoped to a full deprecation as part of the broader meta effort to eliminate core `.module` procedural files ([#3566536](https://www.drupal.org/node/3566536)).

## Upgrade

Replace calls to `check_markup()` with a `processed_text` render array and let Drupal's render pipeline handle it:

```php
// Before (deprecated):
$output = check_markup($text, $format_id, $langcode, $filter_types_to_skip);

// After:
$output = [
  '#type' => 'processed_text',
  '#text' => $text,
  '#format' => $format_id,
  '#langcode' => $langcode,
  '#filter_types_to_skip' => $filter_types_to_skip,
];
```

Return the render array from your method or template variable rather than casting it to a string. If you must obtain a string for an isolated test assertion, use the `Drupal\Tests\filter\Traits\ProcessedTextTestTrait` helper in your test class — but do not use that pattern in production code.

### Collaboration

claudiu.cristea reopened work on the issue in 2025, noting that `check_markup()` had roughly 450 known usages in contributed projects. In a series of comments they weighed whether the logic should move into the `renderer` service, a new dedicated service, or simply be dropped with no replacement. They concluded that the two-line body of `check_markup()` did not warrant a new service and that the correct guidance was to avoid flattening the render array at all, which led to the "no direct replacement" framing that appears in the final deprecation message.
