## Summary

Drupal core's pinned versions of `guzzlehttp/guzzle`, `guzzlehttp/psr7`, and `guzzlehttp/promises` in `core-recommended` were updated to address three security advisories against the Guzzle HTTP client (GHSA-h95v-h523-3mw8, GHSA-wm3w-8rrp-j577, GHSA-f283-ghqc-fg79). Sites running automated vulnerability scanners were failing compliance checks until this update landed. The fix is in both Drupal 10.6.x and 11.3.x.

## Impact

Affects site owners.

- **Security:** Three security advisories against guzzlehttp/guzzle resolved by updating to 7.15.1 and guzzlehttp/psr7 to 2.13.0 in core-recommended.

## Technical details

The commits updated `composer/composer.json` and `composer.lock` in both branches. `guzzlehttp/guzzle` moved from `~7.12.1` to `~7.15.1`, `guzzlehttp/psr7` from `~2.12.1` to `~2.13.0`, and `guzzlehttp/promises` from `~2.5.0` to `~2.5.1`. Because `core-recommended` pins these packages with tilde constraints, the lock file was also updated to the exact new references. The pin on `symfony/polyfill-php80` tightened from `^1.24` to `^1.25` as a side effect of the upstream Guzzle dependency tree. A follow-up issue ([#3612448](https://www.drupal.org/node/3612448)) was opened to widen the `core-recommended` constraints so that future patch-level Guzzle security releases no longer require a core issue, as noted in [#3600889](https://www.drupal.org/node/3600889).

## Upgrade

Sites using `drupal/core-recommended` on 10.6.x or 11.3.x should run `composer update` after pulling the updated core release to receive the patched Guzzle versions. Until the update is applied, the following version alias workaround can be used:

```bash
composer require "guzzlehttp/guzzle:7.15.1 as 7.12.1" "guzzlehttp/promises:2.5.1 as 2.5.0" "guzzlehttp/psr7:2.13.0 as 2.12.1"
```
