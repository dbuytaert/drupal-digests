## Summary

Drupal's Single Directory Components (SDC) system used to silently append `"object"` to the declared type of every component prop. This was an undocumented internal modification that widened each prop's accepted types beyond what the component author specified in their `*.component.yml` schema. That behavior has been removed. Component schemas are now used as written, without Drupal adding `object` as an extra permissible type. If you author SDC components, review your prop schemas: any prop that genuinely accepts a render array or other PHP object should now explicitly declare `"object"` as one of its types. Contrib modules that worked around this behavior (such as UI Patterns and Display Builder) should be able to simplify their code.

## Impact

No upgrade or configuration changes required.

## Technical details

The automatic widening lived in `ComponentMetadata::parseSchemaInfo()` in `core/lib/Drupal/Core/Theme/Component/ComponentMetadata.php`. After processing each prop schema from the component definition file, the method replaced the `type` field with `array_unique([...(array) $type, 'object'])`, unconditionally appending `'object'` regardless of what the author declared. The comment attributed the intent to "deferring rendering in Twig to the render pipeline." According to the original author (Mateu, referenced in the thread by pdureau), it was a remnant from the early CL Components era when there were no slots; render arrays were passed through props because no alternative existed. With slots now available for renderable content, the workaround is no longer needed. The fix is a straightforward deletion of that block. Test fixtures in `ComponentMetadataTest` are updated to reflect that prop types are now plain scalar values such as `'string'` rather than arrays like `['string', 'object']`.

## Upgrade

If you have an SDC component whose prop legitimately accepts a PHP object (including a render array), explicitly add `object` to the `type` field in your `*.component.yml` schema:

```yaml
# Before (implicit — no longer works):
props:
  type: object
  properties:
    my_prop:
      type: string   # Drupal silently accepted objects here too

# After (explicit):
props:
  type: object
  properties:
    my_prop:
      type:
        - string
        - object
```

If you previously worked around this behavior in contrib or custom code (e.g. stripping `object` from a prop's resolved type before validation), that workaround can be removed.
