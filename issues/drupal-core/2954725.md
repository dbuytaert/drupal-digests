## Summary

`AccountInterface::getLastAccessedTime()` is documented to return an `int`, but both `User` and `UserSession` implementations were returning string values sourced directly from database columns. This caused silent failures in strict comparisons (using `===`) and was inconsistent with the method's documented contract. The return values in both implementations are now cast to `int`.

## Impact

No upgrade or configuration changes required.

## Technical details

The root cause is that PHP database drivers return column values as strings by default. `User::getLastAccessedTime()` called `$this->getFieldValue('access', 'value')` and `UserSession::getLastAccessedTime()` returned `$this->access` directly, neither applying any cast.

The fix adds `(int)` casts in both `User::getLastAccessedTime()` (`core/modules/user/src/Entity/User.php`) and `UserSession::getLastAccessedTime()` (`core/lib/Drupal/Core/Session/UserSession.php`). The `@var` docblock on `UserSession::$access` is corrected from `string` to `int`. `AccountProxy::getLastAccessedTime()` is unaffected because it delegates to the underlying account object, so it picks up the fix through the cast on those implementations.

The migrate kernel test `MigrateUserTest::testUser()` was updated to cast `$source->access` before the `assertSame()` call, since the migration source record still carries a string but `getLastAccessedTime()` now guarantees an int. New `testGetLastAccessedTime()` kernel tests were added to `UserEntityTest` and `UserSessionTest` to pin the int-return contract going forward.

## Upgrade

If you were working around this bug by casting the return value yourself, those casts can now be removed:

```php
// Before (workaround)
$lastAccess = (int) $user->getLastAccessedTime();

// After (no cast needed)
$lastAccess = $user->getLastAccessedTime();

// Strict comparisons now work correctly
if ($user->getLastAccessedTime() === 0) { /* user never accessed */ }
```
