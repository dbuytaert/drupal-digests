## Summary

A PHP deprecation notice was triggered whenever the `[comment:homepage]` token was replaced for a comment whose author had not provided a homepage URL. This is the common case for authenticated users, so the notice could fire broadly. The bug is now fixed: an empty string is returned for the token when no homepage is set, which is the correct behavior. A new kernel test was added to cover both the null-homepage and set-homepage cases.

## Impact

No upgrade or configuration changes required.

## Technical details

In `CommentTokensHooks::tokens()`, the `homepage` case called `UrlHelper::stripDangerousProtocols($comment->getHomepage())` unconditionally. `Comment::getHomepage()` reads `$this->get('homepage')->value`, which is `NULL` for an empty URI field. `UrlHelper::stripDangerousProtocols()` passes its argument directly to `strpos()`, which requires a string; passing `NULL` triggers the PHP 8 deprecation "Passing null to parameter #1 ($haystack) of type string is deprecated". The fix stores the return value of `getHomepage()` in a local variable and wraps the call in a truthiness check: `$homepage ? UrlHelper::stripDangerousProtocols($homepage) : ''`. The new `CommentTokenReplaceTest` kernel test (in `core/modules/comment/tests/src/Kernel/CommentTokenReplaceTest.php`) asserts that the token returns an empty string when no homepage is set and the sanitized URL when one is present.
