## Summary

Drupal's recipe system now supports collecting recipe inputs through the Form API, not just the command line. A new `RecipeInputFormTrait` gives any form class three helper methods to render a recipe's required inputs as form elements, validate submitted values, and pass them back to the recipe before it is applied. This is the foundation that allows tools like Project Browser to apply recipes requiring user-supplied values (such as API keys or email addresses) through an interactive UI rather than a terminal prompt.

## Impact

Affects module maintainers and site owners.

- **Config change:** Recipe YAML input definitions now accept an optional `form` key: an associative array of Form API properties (no child elements permitted).
- **API change:** `ValidationFailedException` thrown by `InputConfigurator::collectAll()` now wraps the `TypedDataInterface` object as its value instead of the raw collected value; callers catching this exception must call `->getValue()` on the result of `$e->getValue()` to retrieve the raw value.
- **New API:** New `RecipeInputFormTrait` in `\Drupal\Core\Recipe` with `buildRecipeInputForm()`, `validateRecipeInput()`, and `setRecipeInput()` for use by any form class that needs to collect and apply recipe inputs.

## Technical details

Before this change, recipe inputs could only be provided via the CLI (`drupal recipe FOO --input=name=value`). There was no mechanism for a Drupal form to drive the same input-collection pipeline.

**New trait: `RecipeInputFormTrait`** (`core/lib/Drupal/Core/Recipe/RecipeInputFormTrait.php`)

Three protected methods are provided:

- `buildRecipeInputForm(Recipe $recipe)`: Drives `InputConfigurator::collectAll()` with an anonymous `InputCollectorInterface` implementation that reads each input's data definition for a `form` setting. When that setting is present, the method merges in `#description`, `#default_value`, and forces `#required = TRUE`, then places the element into a `#tree`-structured array keyed by recipe name and input name (e.g. `$form['recipe_name']['input_name']`). Inputs without a `form` setting are silently skipped.
- `validateRecipeInput(Recipe $recipe, array &$form, FormStateInterface $form_state)`: Calls `setRecipeInput()` inside a try/catch for `ValidationFailedException`. Because `InputConfigurator` now throws the exception with the `TypedDataInterface` object rather than the raw value, this method can call `$data->getName()` to resolve the qualified dotted name back to a form element path and call `$form_state->setError()` on the correct element.
- `setRecipeInput(Recipe $recipe, FormStateInterface $form_state)`: Drives `collectAll()` with an anonymous collector that reads values from `$form_state->getValue()` using the same dotted-name splitting logic.

**Changes to `InputConfigurator`** (`core/lib/Drupal/Core/Recipe/InputConfigurator.php`)

Two supporting changes were required. First, `$typedDataManager->create()` now passes `name: "$prefix.$name"` as a named argument so each typed data object carries its fully-qualified name (e.g. `recipe_name.input_key`). Second, `ValidationFailedException` is thrown with the `TypedDataInterface` object instead of the raw collected value, making the name available to `validateRecipeInput()` for precise form error placement. `describeAll()` and `collectAll()` were updated to call `$data->getName()` rather than manually constructing `$this->prefix . '.' . $key`.

**Recipe YAML: new optional `form` key**

`Recipe::parse()` (`core/lib/Drupal/Core/Recipe/Recipe.php`) was extended with a Symfony Validator `Optional` constraint on the `form` key of each input definition. The `form` value must be an associative array with no child elements (checked via `Element::children()`), because recipe inputs are always primitive typed data and nesting is not allowed. A violation message `'Form elements for recipe inputs cannot have child elements.'` is emitted when this rule is broken.

**Tests and fixtures**

A new browser test `RecipeFormInputTest` (`core/modules/system/tests/src/Functional/Form/RecipeFormInputTest.php`) exercises the full lifecycle using a `FormTestRecipeInputForm` test form and the `input_test` fixture recipe (`core/tests/fixtures/recipes/input_test/recipe.yml`). The fixture recipe defines two inputs (`owner` of type `string` with a regex constraint, and `break_stuff` of type `boolean`) each with a `form` key.

The existing `InputTest` kernel test was updated to reflect that `ValidationFailedException::getValue()` now returns a `TypedDataInterface` object; callers must call `->getValue()` on it to get the raw value.

**10.x revert**: The change was committed to 11.x and 11.1.x but was reverted from 10.4.x and 10.5.x. alexpott noted that the constraints for the `form` definition in the recipe needed changes before the feature could land on those branches.

## Upgrade

If you have code that catches `Symfony\Component\Validator\Exception\ValidationFailedException` thrown during recipe input collection and reads the raw value directly, update it:

```php
// Before
catch (ValidationFailedException $e) {
    $rawValue = $e->getValue(); // was the raw collected value
}

// After
catch (ValidationFailedException $e) {
    $data = $e->getValue(); // now a TypedDataInterface object
    if ($data instanceof \Drupal\Core\TypedData\TypedDataInterface) {
        $rawValue = $data->getValue();
    }
}
```

To add form UI support to a recipe input, add a `form` key to the input definition in `recipe.yml`:

```yaml
input:
  recipient:
    data_type: email
    description: 'The email address for feedback submissions.'
    form:
      '#type': email
      '#title': 'Feedback email address'
    default:
      source: config
      config: ['system.site', 'mail']
```

To collect those inputs in a Drupal form class, use the trait:

```php
use \Drupal\Core\Recipe\RecipeInputFormTrait;

class MyForm extends FormBase {
  use RecipeInputFormTrait;

  public function buildForm(array $form, FormStateInterface $form_state): array {
    $recipe = Recipe::createFromDirectory('path/to/my_recipe');
    $form += $this->buildRecipeInputForm($recipe);
    return $form;
  }

  public function validateForm(array &$form, FormStateInterface $form_state): void {
    $this->validateRecipeInput(Recipe::createFromDirectory('path/to/my_recipe'), $form, $form_state);
  }

  public function submitForm(array &$form, FormStateInterface $form_state): void {
    $recipe = Recipe::createFromDirectory('path/to/my_recipe');
    $this->setRecipeInput($recipe, $form_state);
    RecipeRunner::processRecipe($recipe);
  }
}
```

### Collaboration

phenaproxima and alexpott held a Zoom discussion in which they acknowledged a deeper architectural gap: Drupal lacks a single unified system for data input, validation, and presentation that spans Typed Data, config schema, the Form API, and CLI prompts. The patch's approach of embedding Form API element definitions directly into recipe YAML was recognized as pragmatic but inelegant. phenaproxima described it as "sad" compared to a hypothetical system where declaring a string input with constraints would automatically produce appropriate UI in any context. They consciously chose to ship the pragmatic approach with the expectation of future deprecation as Drupal's input handling matures.
