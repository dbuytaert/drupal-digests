# #3565033: Deprecate using numeric literals for 'version' in .info.yml files, require strings

## Summary

Custom modules with unquoted version numbers in `.info.yml` files could cause fatal errors during database updates. When YAML parses `version: 1.0`, it converts it to the float `1`, dropping the trailing `.0`. Code expecting a string would then crash with a `TypeError`. The info parser now casts numeric versions to strings and triggers a deprecation warning. This only affects custom modules; contrib modules hosted on drupal.org always receive version strings from the packaging script.

## Impact

Affects module maintainers.

- **Deprecation:** Using non-string values for `version` in `.info.yml` files deprecated in 11.4.0, fatal error in 13.0.0

## Technical details

`InfoParserDynamic::parse()` reads and validates extension `.info.yml` files. When YAML encounters an unquoted value that looks like a number (`1.0`, `24.0103`), it parses it as a float. The YAML spec also drops insignificant trailing zeros, so `1.0` becomes the integer `1`.

Code elsewhere in core assumed the version key would be a string. In `SystemRequirements.php`, `preg_replace()` expected a string and threw a `TypeError` when given a float.

The fix adds three checks in `InfoParserDynamic::parse()` after the existing `VERSION` constant replacement. First, if the version is non-scalar (an array or object), throw an `InfoParserException`. Second, if the version is a scalar but not a string, trigger a deprecation warning and cast to string. Third, the existing `VERSION` check continues to work. This ensures the version key is always a string.

Test coverage verifies that a quoted string `'1.0'` remains `'1.0'`, an unquoted float `1.1` gets cast to `'1.1'` with the deprecation, and a non-scalar version throws the exception.

A separate backport to 10.6.x and 11.3.x provides the same casting and exception logic without the deprecation warning.

## Upgrade

Wrap version values in single quotes:

```yaml
# Before
version: 1.0

# After
version: '1.0'
```

### Collaboration

The issue began as a simple request to cast floats to strings. During test development, dww discovered that casting alone could not preserve values like `1.0` because the YAML parser had already converted them to `1`. After discussion in Slack, alexpott suggested deprecating non-string versions entirely rather than attempting to recover lost precision. The team created two merge requests: one with the deprecation for 11.4.x and later, and a fix-only backport for 10.6.x and 11.3.x that skips the deprecation warning.
