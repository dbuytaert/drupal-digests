## Summary

Link fields exposed through JSON:API and REST previously returned only the internal URI format (such as `entity:node/5` or `internal:/node/add`), which decoupled front-end applications cannot use directly to render links. A new computed property, `resolvable_uri`, is now included in link field output alongside the existing `uri`, `title`, and `options` properties. It contains the processed, routable URL: relative paths for internal links (for example `/node/5`) and full absolute URLs for external links. The property can also be used to set a link field item's value in PHP, with the internal `uri` and `options` being derived automatically.

## Impact

Affects module maintainers.

- **New API:** New computed `resolvable_uri` property on `LinkItem` (and new `LinkResolvableUriComputed` DataType plugin) that returns the processed, routable URL for a link field item; appears in JSON:API and REST output and can be written to in PHP.

## Technical details

A new `DataType` plugin, `LinkResolvableUriComputed` (plugin id `resolvable_uri`), is registered in `core/modules/link/src/Plugin/DataType/LinkResolvableUriComputed.php`. It extends `Uri` and implements `CacheableDependencyInterface`. Its `getValue()` method calls `$item->getUrl()->toString(TRUE)` on the parent `LinkItem` and caches the resulting `GeneratedUrl`; the `getCacheTags()`, `getCacheContexts()`, and `getCacheMaxAge()` methods delegate to that same object so cacheability metadata is correctly propagated.

`LinkItem::propertyDefinitions()` registers `resolvable_uri` as a computed, non-internal, read-only property using this data type. `LinkItem::onChange()` is added to keep the stored properties in sync when `resolvable_uri` is written directly: it parses the incoming URL, prepends the `internal:` scheme when the path contains no scheme marker, and calls `writePropertyValue()` for `uri` and optionally `options`. `LinkItem::setValue()` is updated to call `onChange('resolvable_uri', FALSE)` when the value array contains a `resolvable_uri` key but no `uri` key, enabling the field to be set entirely through the new property (for example `$entity->field_link = ['resolvable_uri' => '/node/add']`).

The property name went through two earlier iterations. `url` was blocked because Twig resolves `field.0.url` as a call to `LinkItem::getUrl()`, shadowing the method and silently returning `NULL` for any chained access like `field.0.url.routename`. The name was changed to `full_url` in community patches that were widely deployed, then finally to `resolvable_uri` in the MR that landed.

## Upgrade

No action is required for sites that have not applied earlier community patches. Sites or front-end applications that relied on the `url` or `full_url` property from unofficial patches must rename all references to `resolvable_uri`.

```php
// Old (patch-era, never in a Drupal core release):
$entity->field_link->url
$entity->field_link->full_url

// New (Drupal core, 11.x / main):
$entity->field_link->resolvable_uri

// Setting a link field item via the new property:
$entity->field_link->resolvable_uri = 'https://www.drupal.org?test_param=value#main-content';

// Or in an array assignment (uri is derived automatically):
$entity->field_link = ['resolvable_uri' => '/node/add'];
```

JSON:API and REST responses will now include `resolvable_uri` in every link field item. Any client code that deserializes or validates link field shapes may need to be updated to accept the new key.

### Collaboration

The property name produced a significant design dispute. An early reviewer (greg__) reported in comment #22 that the original name `url` silently broke Twig templates: `field.0.url.routename` resolves in Twig to `$item->getUrl()->getRouteName()`, so the new computed property shadowed the existing method and returned `NULL` for any chained access. jibran and lauriii confirmed the conflict. alexpott explicitly blocked commitment in comment #45, pointing to contrib templates already using the pattern and stating "This looks disruptive in its current form." The issue sat at RTBC for over a month before that review.

The name was changed to `full_url` and community patches under that name were widely deployed for years. When dcam reopened the MR in 2025 targeting 11.x, he renamed the property to `resolvable_uri`, citing semantic clarity. dgroene objected in comment #98 that yet another rename was painful for sites already running `full_url` in production. Committer longwave acknowledged the history but agreed that `full_url` was not sufficiently descriptive and committed under the `resolvable_uri` name, noting minor typing issues would be cleaned up separately rather than holding the issue any longer.
