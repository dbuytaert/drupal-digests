## Summary

Drupal's codebase now passes stricter PHPStan type analysis for array parameters. The change fixes 25 type declaration errors where functions expected arrays of strings but received arrays with integers or multidimensional arrays. Most fixes update documentation comments to reflect the actual data structures used. In one case, integer values are explicitly cast to strings before being passed to `str_replace()`. This is part of ongoing work to improve Drupal's static analysis coverage and code quality.

## Technical details

This issue resolved PHPStan violations detected when enabling the `checkFunctionArgumentTypes` parameter. PHPStan flagged 25 locations where `@param` annotations declared `array<string>` or `string[]` but the actual data structures were different.

The fixes fall into two categories. Most changes update `@param` documentation to accurately describe complex array structures. For example, `AnnotatedClassDiscovery::$pluginNamespaces` changed from `string[]` to `array<string, list<string>>` to reflect that it's a keyed array where each value is itself a list of strings. Similarly, test method parameters in `ConfigEntityValidationTestBase` now declare `array<string, array<string, list<string>|string>>|null` to describe nested validation error structures.

The other category involves runtime type casting. In `Result::render()`, the Views result area plugin builds a replacements array for `str_replace()`. Since `str_replace()` expects `array<string>`, integer values like `$start`, `$end`, and `$total` are now cast to strings using `(string)` before assignment. The final `str_replace()` call was also simplified to pass `$replacements` directly instead of calling `array_values()`, since the array is already properly structured.

These changes only affect internal type documentation and do not alter runtime behavior. The fixes improve PHPStan's ability to catch type errors as Drupal moves toward higher analysis levels.
