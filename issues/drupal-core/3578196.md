## Summary

The Navigation module's "Content" menu (labeled "Create" in the sidebar) previously showed hard-coded sub-items for the "Document" and "Image" media types and a "User" creation link in addition to all node types. User testing found this confusing: visitors expected only content-creation shortcuts like pages and blog posts under "Create," and the User entry was easily mistaken for a user-profile type. This change removes the media and user sub-items so the Content menu shows only node type links by default. Sites that relied on those quick-create links for Document, Image, or User should re-add them manually as menu items.

## Impact

Affects site owners.

- **Config change:** Navigation module's Content (Create) menu no longer shows Document, Image media type links or the User creation link by default; only node types appear.

## Technical details

The fix is in `NavigationContentLinks::addMenuLinks()`. The call to `$this->addCreateEntityLinks('media_type', 'entity.media.add_form', $links, ['document', 'image'])`, which hard-coded the allowed media-type IDs, is removed entirely. The block that added the bundleless `navigation.create.user` link pointing to `user.admin_create` with a `weight` of 100 is also removed. Only the `addCreateEntityLinks('node_type', 'entity.node.add_form', $links)` call remains.

The kernel test `NavigationContentLinksTest` drops `MediaTypeCreationTrait` and `UserCreationTrait`, removes the media-type setup steps, and removes the corresponding assertions for `navigation.create.user`, `navigation.content.media_type.document`, `navigation.content.media_type.image`, and the negative assertion for `navigation.content.media_type.special`. The performance test's expected `QueryCount` drops from 264 to 263, reflecting one fewer query without the User link resolution.

## Upgrade

Sites that want to restore the removed links can add them back manually as Navigation menu items. A change record was created at https://www.drupal.org/node/3581445 with instructions for doing so.

### Collaboration

emma horrell reported direct user-testing evidence that drove the decision: participants using the "Create" menu expected only page and post types, found the "User" item confusing because it resembled a profile type, and expected media to be managed through the dedicated Media section of the sidebar, not through "Create." This grounded the consensus that removing the hard-coded entries was the right call rather than a debatable preference.
