## Summary

This plan issue establishes a documented best practice for PHPUnit assertion messages in Drupal core tests. Drupal inherited verbose custom assertion messages from the SimpleTest era, when every assertion displayed a manually written message in the test UI. In PHPUnit those messages only appear when an assertion fails, making most of them redundant alongside PHPUnit's own informative default failure output. The agreed policy is to omit custom messages in ordinary test methods unless the assertion is inside a loop that needs context, and to keep them in custom `assert*()` methods in traits or base test classes and in helper methods. When a message is needed, use the form `{subject} should {verb} {payload}`, for example: "Duration input $name should appear exactly once." The guidance is published as a best practice on the PHPUnit in Drupal handbook page at `https://www.drupal.org/docs/develop/automated-testing/phpunit-in-drupal/drupal-test-coding-standards`. This only affects developers writing or reviewing automated tests.

## Impact

No upgrade or configuration changes required.

## Technical details

Drupal's test assertions originally came from SimpleTest, where every assertion carried a manually written message that was rendered green or red in the SimpleTest UI. When the test suite migrated to PHPUnit, those messages were largely carried over unchanged. PHPUnit only shows custom messages on failure, and it appends them in front of its own default failure message, so many carried-over messages were either redundant, misleading (phrasing a success state when the context is a failure), or from a different era of test design entirely.

The issue worked through several candidate formats before settling on one: (a) positive phrasing ("Field X exists"), which most of core already used as a SimpleTest holdover; (b) negative phrasing ("Field X not found"), which matches the failure context but reads oddly in the test source; (c) "Failed asserting that..." to mirror PHPUnit's own default messages, rejected because PHPUnit always appends its own "Failed asserting that..." line anyway, causing visible repetition; and (d) the `{subject} should {verb} {payload}` form proposed by dww in comment #21, demonstrated with a nested-loop duration-form-element test. The `should` phrasing reads as test documentation when scanning the source and still makes sense when printed at failure time alongside PHPUnit's own diagnostic line, without repetition.

The three cases where a custom message is warranted:
1. Inside a loop, when the loop variable provides context (such as an element name or index) that is not visible in the assertion arguments alone, and a `@dataProvider` refactor is not practical.
2. In a custom `assert*()` method in a trait or base test class, to identify what higher-level condition was being checked when the lower-level assertion fired.
3. In a helper method called from test methods, where the arguments to the helper carry context that should appear in any failure output.

The outcome is documentation rather than enforced code changes. alexpott noted late in the thread that nuance prevents this from being a strict linting rule — for example, an `assertEquals` that exercises case-insensitivity semantics benefits from a message even outside a loop. catch and quietone agreed to publish it as a best practice rather than a coding standard, and quietone placed the page under the PHPUnit in Drupal guide. xjm added editorial revisions emphasizing that unnecessary custom messages can actually obscure debugging by drawing attention away from the assertion arguments and the line number. The related meta issue [#3128573](https://www.drupal.org/node/3128573) drove much of the original momentum, as replacing assertions with more specific PHPUnit methods often required removing stale custom messages as a prerequisite.
