## Summary

After an experimental module or theme is installed, Drupal's status report and module help pages no longer display orange warning-level notices. Those notices are now shown at the blue informational level. The warning shown at install time is unchanged. This reflects the reality that experimental modules in core now always ship with a supported upgrade path, so their ongoing presence on a site is something to be aware of rather than a problem to fix.

## Impact

No upgrade or configuration changes required.

## Technical details

Two runtime-phase locations are changed. In `HelpController::helpPage()`, the `addWarning()` call for modules whose `ExtensionLifecycle::LIFECYCLE_IDENTIFIER` equals `ExtensionLifecycle::EXPERIMENTAL` is replaced with `addMessage()`, downgrading the notice to status level. In the system requirements checker (`checkRequirements()`), the `severity` key for both the `experimental_modules` and `experimental_themes` requirement entries is changed from `RequirementSeverity::Warning` to `RequirementSeverity::Info`, moving them from the "Warnings" group to the "Checked" group on `/admin/reports/status`. These changes apply only to the runtime phase; install and update phase warnings are unaffected. Tests are updated to assert the presence of a `status`-type message rather than a `warning`-type message, and a new browser assertion verifies the status report entry lands in the "Checked" group (`h3#checked`).

### Collaboration

This issue had a notable arc spanning nearly eight years. When japerry filed it in 2017, xjm opposed it directly, posting a screenshot of a site broken beyond recovery after an 8.2→8.3 upgrade with Content Moderation installed, and argued that sites were under-warned about experimental modules, not over-warned. That opposition was well-founded at the time and effectively stalled the issue. By 2023, catch revisited it and noted that core's policies had fundamentally changed: experimental modules no longer ship in tagged releases without a supported upgrade path, which removed the original rationale for the warning. With that blocker gone, ptmkenny returned to the issue in 2025 after encountering the warning again on a fresh install, created an MR based on the existing patch, added the screenshots and positive test assertion that catch had asked for in comment #58, and pushed it to final review. dcam contributed clearer before/after screenshots and confirmed the install-phase warning was still intact, and quietone coordinated release manager agreement before the issue was committed.
