## Summary

`HelpTopicsSyntaxTest` was taking over 19 minutes to run in CI because it was a full functional (browser) test that repeatedly read and wrote `State` storage through a helper Twig extension. The test has been converted to a kernel test and the `#slow` group tag removed; it now completes in roughly 9 to 12 seconds.

## Impact

No upgrade or configuration changes required.

## Technical details

The test previously extended `BrowserTestBase` and depended on a dedicated test support module, `help_topics_twig_tester`, which provided `HelpTestTwigExtension` and `HelpTestTwigNodeVisitor`. The node visitor tracked rendering state (which processing mode to apply, current chunk counts) by reading and writing `\Drupal::state()` on every Twig node traversal. That pattern meant dozens of database round-trips per topic render, and the problem was amplified further by the pending changes in [#3496257](https://www.drupal.org/node/3496257).

The fix moves the test to `KernelTestBase` and implements `NodeVisitorInterface` directly on `HelpTopicsSyntaxTest` itself. State is now tracked in plain PHP properties: `$this->manner`, `$this->maxChunk`, and `$this->returnChunk`. The Twig extension is registered inline via an anonymous class inside `register(ContainerBuilder $container)`, injecting `$this` as the node visitor. The entire `help_topics_twig_tester` module is deleted. The `drupalGet()` page-load calls inside `verifyTopic()` are also removed, since `HelpTopicTest::verifyHelp()` already provides a smoke test of the controller. A stale PHPStan `$ignoreErrors` entry for `HelpTestTwigNodeVisitor::setStateValue()` is also cleaned up. The simplification during merge also collapsed the separate `chunk_count` and `max_chunk` tracking into a single `$this->maxChunk` counter incremented inline in `leaveNode()`.
