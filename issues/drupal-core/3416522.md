## Summary

Site installation and module enabling in Drupal 11.2 are significantly faster because the dependency injection container is now rebuilt as few times as possible when installing multiple modules, instead of once per module. Modules can declare `container_rebuild_required: false` in their `.info.yml` to opt into batched installation; all core modules already carry this flag. The batch size defaults to 20 modules per operation and is tunable via the `core.multi_module_install_batch_size` setting. Real-world testing showed site-install times drop substantially and peak memory use can fall dramatically (for example, from 252 MB to 70 MB on the standard profile).

## Impact

Affects module maintainers and site owners.

- **Config change:** New `core.multi_module_install_batch_size` entry readable via `Settings::get()` controls how many modules are installed per batch operation (default 20).
- **API change:** `ConfigInstallerInterface::installDefaultConfig()` gains a new `DefaultConfigMode $mode` parameter; `checkConfigurationToInstall()` now accepts `string|array $name`; protected `ConfigInstaller` methods `findPreExistingConfiguration()` and `findDefaultConfigWithUnmetDependencies()` have new optional array parameters.
- **New API:** New `\Drupal\Core\Config\DefaultConfigMode` enum controls which phase of default config to install; new `container_rebuild_required` key in module `.info.yml`; new `core.multi_module_install_batch_size` site setting (default 20).

## Technical details

The root problem was that `ModuleInstaller::install()` called `updateKernel()` (a full container rebuild) for every module in the list, even when the module registered no new container services. This dominated install time and inflated peak memory because each rebuild re-parses all service definitions and plugin attributes.

`install()` now reads the `container_rebuild_required` key from each module's parsed `.info.yml` data and partitions the list into sequential groups. A new private helper `doInstall(array $module_list, ...)` processes each group. It performs a single `updateKernel()` call at the start of the group, then runs the install steps in four ordered passes: (1) simple config only (`DefaultConfigMode::InstallSimple`) for all modules, (2) config entities (`DefaultConfigMode::InstallEntities`) and `hook_install()` per module, (3) per-module optional config (`DefaultConfigMode::Optional`), (4) site-wide optional config from all installed modules' `config/optional` directories (`DefaultConfigMode::SiteOptional`). A final `updateKernel([])` call at the end of a multi-module group resets static caches without a full rebuild. `hook_modules_installed()` is now fired once after all groups in the original call complete, not per group. A group boundary is forced when a module has `container_rebuild_required: true` or when the key is absent (backward-compatible default), or when the group reaches `Settings::get('core.multi_module_install_batch_size', 20)` members.

To safely install multiple modules in one pass without false dependency errors, `ConfigInstaller::checkConfigurationToInstall()` now accepts `string|array $name` and iterates across all modules in the list before any installation begins, accumulating a `$previous_config_names` array. This array is passed to `findPreExistingConfiguration(StorageInterface $storage, array $previous_config_names = [])` and `findDefaultConfigWithUnmetDependencies(..., array $previously_checked_config = [])` to prevent false `PreExistingConfigException` and `UnmetDependenciesException` errors when peer modules in the same batch each provide config that the other requires. The new `\Drupal\Core\Config\DefaultConfigMode` enum (cases: `All`, `InstallSimple`, `InstallEntities`, `Optional`, `SiteOptional`) is added as a third parameter to `ConfigInstaller::installDefaultConfig()` and `ConfigInstallerInterface::installDefaultConfig()`.

`ConfigImporter::getNextExtensionOperation()` now returns a slice of up to `core.multi_module_install_batch_size` module names for module-install operations instead of one at a time. `processExtension()` and `setProcessedExtension()` are updated to accept `string|array`. In `install.core.inc`, `install_profile_modules()` applies the same grouping logic (respecting `container_rebuild_required`) when building batch operations, switching from one module per batch operation to up to 20. `_install_module_batch()` is updated to accept `array $modules` and `array $module_names`. `install_recipe_required_modules()` consolidates its required modules into a single batch operation, which also benefits recipe-based installs ([#3498026](https://www.drupal.org/node/3498026)). A long-standing exception in `ThemeRenderer::render()` that blocked theme rendering while modules were loading was also removed, as the new grouped install flow no longer leaves the module handler in a partially-loaded state.

## Upgrade

Modules or contrib packages that implement or decorate `ConfigInstallerInterface` must update their `installDefaultConfig()` and `checkConfigurationToInstall()` signatures:

```php
// Old signatures:
public function installDefaultConfig($type, $name);
public function checkConfigurationToInstall($type, $name);

// New signatures (DefaultConfigMode defaults to All, so existing callers
// that pass only $type and $name are unaffected at runtime):
use Drupal\Core\Config\DefaultConfigMode;

public function installDefaultConfig($type, $name, DefaultConfigMode $mode = DefaultConfigMode::All);
public function checkConfigurationToInstall($type, string|array $name);
```

The protected methods `findPreExistingConfiguration()` and `findDefaultConfigWithUnmetDependencies()` in `ConfigInstaller` have new optional array parameters; subclasses overriding them should add the new signatures to avoid deprecation notices:

```php
protected function findPreExistingConfiguration(StorageInterface $storage, array $previous_config_names = []);
protected function findDefaultConfigWithUnmetDependencies(StorageInterface $storage, array $enabled_extensions, array $profile_storages = [], array $previously_checked_config = []);
```

To allow a module to be installed in a batch with others (no per-module container rebuild), add to its `.info.yml`:

```yaml
container_rebuild_required: false
```

To force a container rebuild immediately after install (for example, the module registers new container services), add:

```yaml
container_rebuild_required: true
```

Omitting the key preserves the old behavior (rebuild after every install) for backward compatibility. A follow-up issue ([#3492235](https://www.drupal.org/node/3492235)) changed the default to `false` for modules that omit the key, extending the benefit to unmodified contrib modules.
