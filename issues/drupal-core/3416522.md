## Summary

Site installation and module enabling in Drupal 11.2 are significantly faster because the dependency injection container is now rebuilt as few times as possible when installing multiple modules, instead of once per module. Modules can declare `container_rebuild_required: false` in their `.info.yml` to opt into batched installation; all core modules already carry this flag. The batch size defaults to 20 modules per operation and is tunable via the `core.multi_module_install_batch_size` setting. Real-world testing showed site-install times drop by roughly 30–50% and peak memory use can fall dramatically (e.g., 252 MB to 70 MB on the standard profile).

## Impact

Affects module maintainers and site owners.

- **Config change:** New `core.multi_module_install_batch_size` entry readable via `Settings::get()` controls how many modules are installed per batch operation.
- **API change:** `ConfigInstallerInterface::installDefaultConfig()` gains a new `DefaultConfigMode $mode` parameter; `checkConfigurationToInstall()` now accepts `string|array $name`; protected `ConfigInstaller` methods `findPreExistingConfiguration()` and `findDefaultConfigWithUnmetDependencies()` have new optional array parameters.
- **New API:** New `\Drupal\Core\Config\DefaultConfigMode` enum controls which phase of default config to install; new `container_rebuild_required` key in module `.info.yml`; new `core.multi_module_install_batch_size` site setting (default 20).

## Technical details

The root problem was that `ModuleInstaller::install()` called `updateKernel()` (a full container rebuild) for every module in the list, even when the module registered no new container services. This dominated install time and inflated peak memory because each rebuild re-parses all service definitions and plugin attributes.

**Grouping in `ModuleInstaller`**

`install()` now reads the `container_rebuild_required` key from each module's parsed `.info.yml` data and partitions the list into sequential groups. A new private helper `doInstall(array $module_list, ...)` processes each group with a single `updateKernel()` call at the start. A group boundary is forced when a module has `container_rebuild_required: true` or when the key is absent (backward-compatible default that forces a rebuild), or when the group reaches `Settings::get('core.multi_module_install_batch_size', 20)` members. `hook_modules_installed()` is now fired once after all groups in the original call complete, not per group. All core (non-test) modules were annotated with `container_rebuild_required: false` in this commit, so they batch immediately.

**Config installation decomposition with `DefaultConfigMode`**

To safely install multiple modules in one pass, the order of config installation steps had to be made explicit. A new PHP enum `\Drupal\Core\Config\DefaultConfigMode` (cases: `All`, `InstallSimple`, `InstallEntities`, `Optional`, `SiteOptional`) is added as a third parameter to `ConfigInstaller::installDefaultConfig()` and `ConfigInstallerInterface::installDefaultConfig()`. Inside `doInstall()`, the method is called in four separate passes for the group: (1) simple config only (`InstallSimple`), (2) config entities (`InstallEntities`), (3) per-module optional config (`Optional`), (4) site-wide optional config from all modules' `config/optional` directories (`SiteOptional`). This ordering prevents config entity creation in one module from failing because data it depends on from a peer module in the same batch has not yet been written.

**Batch-aware config validation**

`ConfigInstaller::checkConfigurationToInstall()` is now called once with the full module list (accepting `string|array`) before any installation begins. It accumulates a `$previous_config_names` array keyed by collection as it iterates, passing it to both `findPreExistingConfiguration(StorageInterface $storage, array $previous_config_names = [])` and `findDefaultConfigWithUnmetDependencies(..., array $previously_checked_config = [])`. This prevents false `PreExistingConfigException` and `UnmetDependenciesException` errors when two modules in the same batch each provide config that the other requires.

**`ConfigImporter` and installer batching**

`ConfigImporter::getNextExtensionOperation()` now returns a slice of up to `core.multi_module_install_batch_size` module names for module-install operations instead of one at a time. `processExtension()` and `setProcessedExtension()` are updated to accept `string|array`. In `install.core.inc`, `install_profile_modules()` applies the same grouping logic (respecting `container_rebuild_required`) when building batch operations, switching from 1 module per batch operation to up to 20. `_install_module_batch()` is updated to accept `array $modules` and `array $module_names`. `install_recipe_required_modules()` consolidates its required modules into a single batch operation.

A follow-up issue [#3492235](https://www.drupal.org/node/3492235) was opened to default `container_rebuild_required` to `false` for modules that omit the key, which would extend the benefit to unmodified contrib modules.

## Upgrade

Modules or contrib packages that implement or decorate `ConfigInstallerInterface` must update their `installDefaultConfig()` and `checkConfigurationToInstall()` signatures:

```php
// Old signatures:
public function installDefaultConfig($type, $name);
public function checkConfigurationToInstall($type, $name);

// New signatures (DefaultConfigMode defaults to All, so existing callers are unaffected):
use Drupal\Core\Config\DefaultConfigMode;

public function installDefaultConfig($type, $name, DefaultConfigMode $mode = DefaultConfigMode::All);
public function checkConfigurationToInstall($type, string|array $name);
```

The protected methods `findPreExistingConfiguration()` and `findDefaultConfigWithUnmetDependencies()` in `ConfigInstaller` also have new optional parameters; subclasses overriding them should add the new signatures.

To allow a module to be installed in a batch with others (no per-module container rebuild), add to its `.info.yml`:

```yaml
container_rebuild_required: false
```

To force a container rebuild immediately after install (e.g., the module registers new container services), add:

```yaml
container_rebuild_required: true
```

Omitting the key preserves the old behavior (rebuild after every install) for backward compatibility.

### Collaboration

The issue spanned nearly a year with a significant design pivot. alexpott opened and drove the initial proof-of-concept, then tested against real distributions (Thunder, Commerce Kickstart), uncovering a conflict with modules that decorate `ConfigInstallerInterface` (notably `config_selector`). At comment #29, catch proposed the `container_rebuild_required` info.yml key as a backward-compatible opt-in: modules that declare `false` batch immediately, everything else keeps the old per-module rebuild. This design change resolved the contrib compatibility risk that had stalled the issue. catch then implemented `ConfigImporter` batching support and the direct `ModuleInstaller` flag handling in the November 2024 sprint. longwave rebased the MR and measured memory on the standard profile dropping from 252 MB to 70 MB, which moved the issue to committed. nicxvan provided exhaustive manual testing (standard profile, Umami, enabling every non-deprecated core module) and created follow-up issues for outstanding inline todos.
