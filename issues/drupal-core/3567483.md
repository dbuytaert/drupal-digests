## Summary

The Update Manager module could crash admin pages when encountering invalid version strings in release data from Drupal.org or contributed modules. This happened when the string "main" appeared in update feeds after Drupal opened its main development branch. Sites running Drupal 8.9 through 10.2 experienced white screens on admin pages because the version parser threw uncaught exceptions. The fix makes the Update Manager resilient to malformed version data by catching exceptions and silently filtering out invalid versions, preventing crashes from external data sources.

## Impact

No upgrade or configuration changes required.

## Technical details

The `ProjectCoreCompatibility` class in the Update module calls `Composer\Semver\VersionParser::normalize()` and `Semver::satisfies()` to parse and compare version strings from external release feeds. When Drupal.org's XML feed included "main" as a version string, these methods threw `UnexpectedValueException` errors that were not caught, crashing any page that invoked `update_page_top()`.

The fix adds try-catch blocks around three points where version parsing can throw exceptions: when normalizing version strings to filter supported versions, when calling `Semver::satisfiedBy()` to find possible core updates, and when checking compatibility ranges with `Semver::satisfies()`. Invalid versions are silently filtered out rather than logged, since site administrators cannot fix upstream release data and logging would create persistent noise for unfixable issues.

A test case was added to verify that invalid version strings like "main" in the release list do not cause exceptions. The fix applies to all three locations where the Composer Semver library interacts with externally sourced data, protecting sites from malformed data whether from accidental errors, hacked sources, or man-in-the-middle attacks.

### Collaboration

This issue was reported when Drupal.org opened the main development branch and inadvertently included "main" in release feeds, immediately crashing older sites. A server-side fix in [#3567494](https://www.drupal.org/project/drupalorg/issues/3567494) filtered out the main branch from feeds, but discussion revealed the root cause was uncaught exceptions in core.

Contributors debated whether to log errors for invalid versions, ultimately deciding against it since site owners cannot fix upstream release data and logging would create permanent noise. Testing revealed the issue also affected sites when contributed modules like Suggestion released versions with invalid version constraints. The maintainers recognized this as a security-sensitive issue: external data from any source (malformed, hacked, or intercepted) should never crash a site. The fix took nearly two months from report to commit as the team carefully considered error handling strategies and tested edge cases.

## Lessons learned

- When processing data from external sources (APIs, feeds, third-party servers), validate or catch exceptions before the data reaches version comparison logic. External sources can be compromised, misconfigured, or return malformed data, and uncaught exceptions from parsing libraries can crash the application. Wrap parser calls in try-catch blocks and fail gracefully rather than propagating exceptions to crash user-facing pages. *(via cmlara, catch, dww, longwave)*
