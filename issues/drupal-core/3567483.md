## Summary

The Update module could crash sites with a white screen when processing malformed version strings from external update feeds. This happened when Drupal.org introduced a "main" branch, but the same issue could occur with any invalid version string from update feeds, whether due to accidental errors, compromised servers, or man-in-the-middle attacks. The Update module now validates version strings before processing them and silently filters out invalid entries instead of crashing. This ensures that unexpected data from external sources cannot take down site administration pages.

## Impact

No upgrade or configuration changes required.

## Technical details

The root cause was uncaught exceptions when the `composer/semver` library encountered invalid version strings like "main" while parsing update XML feeds. The `ProjectCoreCompatibility` class called Composer's semantic version comparison functions without validating input from external sources.

The fix wraps three locations in `ProjectCoreCompatibility` with try-catch blocks for `UnexpectedValueException`. Before comparing versions, the code now validates each version string using `VersionParser::normalize()`. Invalid versions are filtered out early in `getPossibleCoreUpdateVersions()`. The method also catches exceptions when calling `Semver::satisfiedBy()` and `Semver::sort()`, returning an empty array instead of crashing. In `getCompatibilityRanges()`, the code catches exceptions from `Semver::satisfies()` and continues to the next version.

The fix does not log these errors. Since site owners cannot fix malformed data from external feeds, and invalid releases remain in feeds permanently once published, logging would only create noise without actionable information.

A unit test was added to verify that invalid version strings like "main" are properly filtered without throwing exceptions.

### Collaboration

This issue was reported when Drupal.org opened the "main" branch, immediately causing crashes on Drupal 8.9, 9.x, and 10.2 sites. The drupal.org team quickly deployed a server-side fix in [#3567494](https://www.drupal.org/node/3567494) to filter "main" from update feeds, but cmlara correctly argued this only addressed the symptom. External update feeds could provide malformed data through accidents, compromises, or attacks, and Drupal core needed defensive validation.

The discussion clarified that Drupal 10.3+ was already protected by filtering in [#3112962](https://www.drupal.org/node/3112962), but older versions remained vulnerable. dww and catch emphasized that external data should never crash a site. After jastraat confirmed the fix resolved a real-world issue with the Suggestion module's faulty alpha release, the team debated whether to log validation errors. They ultimately decided against logging since site owners cannot fix external feed problems and invalid releases persist forever.

The fix took nearly two months from report to commit, reflecting the careful discussion about proper error handling for external data.

## Lessons learned

- When processing externally-sourced data (from APIs, feeds, or user input), wrap operations that might throw exceptions in try-catch blocks. External data can be malformed accidentally, compromised through attacks, or intercepted, and uncaught exceptions from parsing invalid data should never crash the application. Log errors if actionable by users, or silently skip invalid entries when users cannot fix the problem. *(via cmlara, catch, dww, longwave)*
