## Summary

Drupal core now ships a `user:login` console command (alias `uli`) that generates a one-time login URL for any user account. Site administrators can run `vendor/bin/dr user:login` to instantly get a login link for user 1, or target a specific account with `--uid`, `--name`, or `--mail` options, and optionally redirect the resulting session to a particular path using the positional `path` argument. This is one of the top five most-used Drush commands and is now available in core without requiring Drush.

## Impact

Affects module maintainers.

- **New API:** New `user:login` (alias `uli`) console command to generate a one-time login URL, accepting `--uid`, `--name`, `--mail` options and an optional `path` argument.

## Technical details

The command is implemented as `Drupal\user\Command\LoginCommand` (marked `@internal`), registered via Symfony Console's `#[AsCommand]` attribute with the name `user:login` and alias `uli`. It accepts an optional positional `path` argument and three mutually exclusive options (`--name`, `--uid`, `--mail`) for account selection; when none are provided, it falls back to loading user 1. Account lookup uses `EntityTypeManagerInterface::getStorage('user')->loadByProperties()` and throws `\InvalidArgumentException` for unresolvable identifiers or blocked accounts. The one-time URL is produced by the existing `OneTimeAuthentication::generateOneTimeLoginUrl()` service with `immediate: TRUE`, and the URL string is written to `OutputInterface` with `writeln()` (including a trailing newline so the URL does not run into the shell prompt). This is part of Phase 2 of the CLI in Core community initiative. Test coverage is split: a Functional test verifies that the generated link actually authenticates the user in a real browser session, while a Kernel test covers all failure modes (blocked account, unknown uid/name/mail), cutting overall test time roughly in half compared to running every case in a `BrowserTestBase`.
