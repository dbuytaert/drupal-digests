## Summary

On multilingual sites, setting a path alias to a specific language (or to language-neutral, `LanguageInterface::LANGCODE_NOT_SPECIFIED`) programmatically was silently ignored when saving a node. The alias was always stored with the node translation's language instead. A langcode-only update (same alias text, different language) was also dropped entirely. Both cases are now fixed: the explicitly provided alias langcode is respected on both create and update.

## Impact

No upgrade or configuration changes required.

## Technical details

Two bugs lived in `PathItem::postSave()` inside `core/modules/path/src/Plugin/Field/FieldType/PathItem.php`.

First, the langcode selection used the condition `($langcode && $pid) ? $langcode : $this->getLangcode()`. Because `$pid` is absent when creating a new alias, any explicit langcode supplied at node-creation time was always discarded in favour of the entity's own langcode. The fix simplifies this to `$langcode ?: $this->getLangcode()`, so any non-empty explicit langcode wins for both new and existing aliases.

Second, the update branch only called `$path_alias->save()` when the alias text changed (`$alias != $path_alias->getAlias()`). A change to the langcode alone was never persisted. The fix introduces a `$path_alias_changed` flag that is set when either the alias text or the langcode (`$alias_langcode !== $path_alias->language()->getId()`) differs from the stored value, and save is triggered when either is true.

A third, related fix is in `PathDefaultWidget::formElement()`. The hidden `langcode` form field previously always forwarded `$items[$delta]->langcode`, which for unsaved items resolved to the entity's langcode and would have overridden any programmatic langcode set before the form was rendered. It now passes `NULL` when no `$pid` exists yet (new alias), allowing the save logic to apply the explicitly set value instead.

New kernel test cases in `PathItemTest::testPathItem()` cover: setting `langcode` to `LANGCODE_NOT_SPECIFIED` on a saved node's path field and re-saving (verifies the language-neutral alias is stored and the language-specific alias is gone), and creating a new node with `LANGCODE_NOT_SPECIFIED` in the path data (verifies the alias is created language-neutral from the start).

## Upgrade

No API changes. If your code was previously working around this bug by manually editing the path alias after node save, that workaround can be removed. For example, code like:

```php
// Before: workaround needed because langcode was ignored on save
$node->path->langcode = LanguageInterface::LANGCODE_NOT_SPECIFIED;
$node->save();
// Then manually fix the stored alias…
$alias = $alias_storage->load($pid);
$alias->set('langcode', LanguageInterface::LANGCODE_NOT_SPECIFIED);
$alias->save();
```

can now be simplified to:

```php
// After: langcode is respected on save
$node->path->langcode = LanguageInterface::LANGCODE_NOT_SPECIFIED;
$node->save();
```
