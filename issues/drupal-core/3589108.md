## Summary

Drupal's test deprecation handling has been moved off the `SYMFONY_DEPRECATIONS_HELPER` environment variable and onto structured parameters declared directly in `phpunit.xml`. Anyone who sets `SYMFONY_DEPRECATIONS_HELPER` in their `phpunit.xml` or CI environment will now see a deprecation notice per test; the variable is deprecated in drupal:11.5.0 and removed in drupal:12.0.0. The replacement is an `<extensions>` block in `phpunit.xml` with three named parameters (`enableProjectIgnores`, `projectIgnoreFile`, `enableDebugClassLoader`), and a new `DRUPAL_DEPRECATION_FILTER_CONFIG` JSON env variable for runtime CI overrides. Core's own `phpunit.xml.dist` and the component-level `phpunit.xml` have already been updated to use the new format. This change is a prerequisite for adopting PHPUnit 13, which broke the previous approach in 13.2 and introduces incompatible deprecation-filter APIs in 13.3.

## Impact

Affects module maintainers.

- **Deprecation:** SYMFONY_DEPRECATIONS_HELPER environment variable deprecated in drupal:11.5.0, removed in drupal:12.0.0; replaced by phpunit.xml extension parameters and DRUPAL_DEPRECATION_FILTER_CONFIG env variable

## Technical details

Previously, `DeprecationHandler` was a plain `final class` initialized by ad-hoc code in `bootstrap.php` that read `SYMFONY_DEPRECATIONS_HELPER`, called `DeprecationHandler::init()`, then manually registered `BootstrapErrorHandler` and enabled `DrupalDebugClassLoader`. The class had no PHPUnit lifecycle hooks.

The refactor makes `DeprecationHandler` implement `PHPUnit\Runner\Extension\Extension`. A new `@internal` singleton `Configuration` (`core/tests/Drupal/TestTools/Extension/DeprecationBridge/Configuration.php`) holds all runtime state: `projectIgnoresEnabled`, `projectIgnoreFile`, `debugClassLoaderEnabled`, and `deprecationIgnorePatterns`. Its `load(array $parameters)` method accepts key-value arrays and is called multiple times to layer configuration from different sources.

The initialization entry point is the new static `DeprecationHandler::preBootstrap(PhpUnitConfiguration $configuration)`, called from `bootstrap.php` via `DeprecationHandler::preBootstrap(PhpunitConfigurationRegistry::get())`. It reads the `<parameter>` elements for the extension from PHPUnit's `extensionBootstrappers()` list, then layers any JSON from `DRUPAL_DEPRECATION_FILTER_CONFIG`, then applies a compatibility shim for the legacy `SYMFONY_DEPRECATIONS_HELPER` value (mapping `disabled` to `enableProjectIgnores=false,enableDebugClassLoader=false`, and `ignoreFile=...` to `projectIgnoreFile`). After merging all three sources, it conditionally loads ignore patterns from the file and registers `BootstrapErrorHandler` and `DrupalDebugClassLoader`. The `bootstrap()` method required by the `Extension` interface is left empty because these handlers must be registered before other extensions bootstrap.

The old `isIgnoredDeprecation()` (public) became `isIgnoredByProject()` (private); the public surface visible to `BootstrapErrorHandler` is the new `DeprecationHandler::handle()` static method. `BootstrapErrorHandler::__invoke()` now calls `DeprecationHandler::handle()` instead of `DeprecationHandler::isIgnoredDeprecation()`. The deprecation notice for `SYMFONY_DEPRECATIONS_HELPER` is triggered via a `#[Before]` method `checkLegacySymfonyDeprecationHelperEnvVariable()` added to `UnitTestCase`. The `--suppress-deprecations` CLI flag in the test runner no longer references `SYMFONY_DEPRECATIONS_HELPER` in its description, and the code that set `SYMFONY_DEPRECATIONS_HELPER=disabled` in `test-site.php` has been removed. Related: [#3486376](https://www.drupal.org/node/3486376) (DebugClassLoader cross-module return type reporting) is the upstream cause of the 3.5-4 million deprecations per CI run that make the ignore file essential.

## Upgrade

Remove `SYMFONY_DEPRECATIONS_HELPER` from your `phpunit.xml` and add the extension block instead:

```xml
<!-- Before (deprecated): -->
<env name="SYMFONY_DEPRECATIONS_HELPER" value="ignoreFile=.deprecation-ignore.txt"/>

<!-- After: -->
<extensions>
  <bootstrap class="Drupal\TestTools\Extension\DeprecationBridge\DeprecationHandler">
    <parameter name="enableProjectIgnores" value="true"/>
    <parameter name="projectIgnoreFile" value="core/.deprecation-ignore.txt"/>
    <parameter name="enableDebugClassLoader" value="true"/>
  </bootstrap>
</extensions>
```

To disable all deprecation handling (previously `SYMFONY_DEPRECATIONS_HELPER=disabled`):

```xml
<bootstrap class="Drupal\TestTools\Extension\DeprecationBridge\DeprecationHandler">
  <parameter name="enableProjectIgnores" value="false"/>
  <parameter name="enableDebugClassLoader" value="false"/>
</bootstrap>
```

For CI pipeline overrides without editing `phpunit.xml`, use the new env variable:

```bash
DRUPAL_DEPRECATION_FILTER_CONFIG='{"enableDebugClassLoader":false}' ./vendor/bin/phpunit
```

### Collaboration

mondrake noted in comment [#16609121](https://www.drupal.org/node/16609121) that PHPUnit 13.2 had already broken Drupal's deprecation-ignore approach and that 13.3 would ship `Deprecation Filters` incompatible with `BootstrapErrorHandler`, making this work urgent rather than aspirational. In comment [#16639939](https://www.drupal.org/node/16639939) mondrake wrote a detailed explanation of why initialization must happen before the PHPUnit extension bootstrap phase: the `DebugClassLoader` triggers 3.5-4 million deprecations per CI run starting from class-load time, so the ignore file shield must be in place before any extension can bootstrap and cause class loads. This reasoning directly shaped the decision to keep a `preBootstrap()` static method called from `bootstrap.php` rather than relying solely on the `Extension::bootstrap()` lifecycle hook.
