## Summary

When a user typed a search term with leading or trailing spaces into a Views exposed text filter (for example "foo " instead of "foo"), the view returned no results. The fix trims whitespace from the filter value before it reaches the database query, so those searches now return the expected results. The change was committed to both `main` and `11.x`.

## Impact

No upgrade or configuration changes required.

## Technical details

The bug was in `core/modules/views/src/Plugin/views/filter/FilterString.php`. When an exposed input was accepted, the raw string value (including any surrounding whitespace) was passed directly to the query, causing an exact-match failure against stored values that have no whitespace.

The fix adds an `acceptExposedInput()` override to `FilterString`. After delegating to the parent method, if the parent accepted the input and `$this->value` is a string, it calls `trim($this->value)` in place before returning. This ensures the sanitized value is used in `query()`.

A test case was added to `FilterStringTest::testFilterStringGroupedExposedEqual()` confirming that setting the exposed input to `' Ringo '` (with surrounding spaces) returns the row for `'Ringo'`.

When committing, xjm explicitly evaluated backport safety: the behavior change (returning results that previously returned none) was considered minor-only acceptable, but the new method override in `FilterString` could conflict with contrib or custom subclasses that already override `acceptExposedInput()`. She committed to both `main` and `11.x` anyway, characterizing it as a slightly disruptive bugfix.
