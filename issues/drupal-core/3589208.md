## Summary

Drupal's asset aggregation system now always derives aggregate filename hashes from the actual file contents rather than from library version strings. Previously, core libraries all declared `VERSION` (the global Drupal version constant), so every patch release invalidated every aggregate file even when the underlying CSS or JS files had not changed. Conversely, a contrib or custom module that changed a file without bumping its library version would silently serve stale aggregates. After this fix, aggregate filenames change when and only when file contents change, making the cache more stable under core upgrades and more reliable under file edits.

## Impact

No upgrade or configuration changes required.

## Technical details

The logic lived in `CssCollectionOptimizerLazy->generateHash()`. The old code only replaced the version with a content hash when `$asset['version'] === -1` (meaning no version was declared in the library definition). Libraries that did declare a version used that string directly in the hash, so the aggregate filename was coupled to the version number rather than the file contents.

The fix removes the `-1` check and unconditionally calls `hash_file('xxh64', $asset['data'])` to hash the file contents for every asset item. A `file_exists()` guard was added: when a file does not exist on disk (for example, "stub" library definitions used in testing), the original version value is kept instead of crashing. The switch from `hash('xxh64', file_get_contents(...))` to `hash_file(...)` was a minor optimization suggested during review that avoids reading the entire file into memory.

The related issue [#3505776](https://www.drupal.org/node/3505776) (45-day aggregate retention) is the direct context: because aggregates are now kept for weeks rather than regenerated on every cache clear, a version-string-driven hash that changed with every core release was causing unnecessary aggregate churn, while a stale version string could silently serve outdated content.
