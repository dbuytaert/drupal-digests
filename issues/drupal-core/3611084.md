## Summary

CLI commands run through `vendor/bin/dr` (and tools such as Drush that bootstrap Drupal the same way) now use the compiled, cached container instead of rebuilding it from scratch on every invocation. Previously `DrupalApplication::bootstrap()` always passed `allow_dumping=FALSE` to `DrupalKernel`, forcing a full `ContainerBuilder` build even in production. The fix makes normal CLI use fast while keeping the uncompiled-container behaviour available for kernel tests that need it.

## Impact

Affects module maintainers.

- **New API:** New `DrupalApplicationTesterTrait` in `Drupal\KernelTests` for kernel test classes that need to invoke `vendor/bin/dr` commands, replacing ad-hoc inline helpers.

## Technical details

`DrupalApplication::bootstrap()` previously hardcoded `new DrupalKernel('prod', $this->classloader, FALSE)`, disabling container dumping unconditionally. The original motivation was that `KernelTestBase::register()` mutates the container, so loading a cached copy during kernel tests would reuse the wrong container. The fix replaces the hardcoded values with context lookups: `new DrupalKernel($this->context['kernel.environment'] ?? 'prod', $this->classloader, $this->context['kernel.allow_dumping'] ?? TRUE)`. Production CLI invocations get the compiled container by default; test harnesses opt out by passing `kernel.environment => 'testing'` and `kernel.allow_dumping => FALSE` in the context array.

A new `DrupalApplicationTesterTrait` (in `Drupal\KernelTests`) centralises this opt-out for kernel test classes. The trait provides a private `applicationTester(array $context = [])` method that includes `vendor/bin/dr`, injects the two testing context keys, and returns a Symfony `ApplicationTester`. The duplicate inline `applicationTester()` helpers that existed in `StatusCommandTest` and `ConsoleTest` were removed in favour of the shared trait.
