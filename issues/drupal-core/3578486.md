## Summary

This is a purely internal test improvement with no user-facing effect. The existing `AutowireTest` verified that core services have an alias but never confirmed that an alias actually pointed to a service class implementing the correct interface. A bug in issue [#3572050](https://www.drupal.org/node/3572050) exposed this gap: an icon extractor service was aliased to the wrong interface and no automated test caught it. This change adds that missing assertion. A small cosmetic cleanup in `block_content.services.yml` was also included to remove a leading backslash from a class name, for consistency with the rest of the codebase.

## Impact

No upgrade or configuration changes required.

## Technical details

The new test method `testCoreServiceAliasInterfaces()` in `AutowireTest` parses every core service YAML file via `getCoreServiceFiles()`, splitting entries into two buckets: plain string entries (shortcut alias syntax) go into `$aliases`, and array entries go into `$services`. For each alias it then does one of two checks: if the alias ID is a real PHP interface (detected with `interface_exists()`), the aliased service's class must pass `is_subclass_of()` against it; if the alias ID is a class name instead, the aliased service class must exactly equal that string. Any mismatch is collected and the test fails with a descriptive message listing each offending alias. `block_content.services.yml` had a stray leading backslash on `\Drupal\block_content\BlockContentUuidLookup`, which the new test would have caught as a non-matching class; removing it (making it `Drupal\block_content\BlockContentUuidLookup`) keeps the entry consistent. The commit landed on `main` only; the `11.x` port was reverted by alexpott because `AutowireTest` on that branch was missing prerequisites.
