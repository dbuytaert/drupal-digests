## Summary

Drupal themes can now use object-oriented hook implementations instead of procedural functions in .theme files. This update converts all remaining core themes except Claro, Olivero, and the starter kit to the new pattern. Hook implementations now live in dedicated class files with attributes marking which hooks they implement. This modernizes the codebase while maintaining full backward compatibility for existing custom themes.

## Impact

No upgrade or configuration changes required.

## Technical details

Themes previously implemented hooks as procedural functions in .theme files (e.g. `themename_preprocess_html()`). The new approach places these in a `src/Hook/ThemeNameHooks.php` class with methods decorated by `#[Hook]` attributes.

The conversion moved hook implementations from .theme files to classes for test themes (`test_base_theme`, `test_subtheme`, `test_theme_alter`, etc.), the Umami demo theme, and test themes for Views UI. Each theme now has a hooks class in the `Drupal\themename\Hook` namespace.

For `stable9`, the .theme file retains the procedural functions marked with `#[LegacyHook]` attributes, while also creating the OOP hooks class. This demonstrates the backward compatibility layer: themes can keep existing .theme files working while optionally adopting the new pattern. The `StringTranslationTrait` was added where themes used `t()` function calls, replacing them with `$this->t()`.

The conversion was largely mechanical, performed using rector. Method names in hook classes use camelCase, and double underscores in hook names (common in preprocess hooks with suggestions) convert to StudlyCase (e.g. `preprocess_theme_test_preprocess_suggestions__kitten__flamingo` becomes `preprocessThemeTestPreprocessSuggestionsKittenFlamingo`).
