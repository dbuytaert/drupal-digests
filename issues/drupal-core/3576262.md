## Summary

A critical random test failure in the media module's functional JavaScript test suite was fixed. The test `MediaSourceFileTest::testMediaFileSource` was intermittently failing in CI because assertions were made on page elements immediately after pressing a Save button, without waiting for the page navigation and form submission to fully complete. The fix is purely internal to the test code and has no effect on production behavior.

## Impact

No upgrade or configuration changes required.

## Technical details

The root cause was a race condition in `MediaSourceFileTest::testMediaFileSource` and `MediaSourceTestBase::doTestCreateMediaType`. Calling `$page->pressButton('Save')` initiates a form submission but returns immediately; a few microseconds can pass before the browser has actually navigated to the result page. Assertions made directly afterward (checking `img src` attributes, address bar values, or status messages) could run against the old page or a partially loaded new page, causing intermittent failures like `ElementHtmlException: The text "text--plain.png" was not found in the attribute "src"`.

The fix adds `$assert_session->waitForElement('css', '.messages--status')` after each affected `pressButton('Save')` call. The `.messages--status` element only appears once the server has processed the form and the browser has rendered the new page, making it a reliable synchronization point. Five call sites were fixed: four in `MediaSourceFileTest` and one in `MediaSourceTestBase`.

Additionally, the existing `$assert_session->statusMessageContains()` call (which already has a built-in wait) was reordered to appear before `\Drupal::entityTypeManager()->getStorage('media')->loadUnchanged(1)`, so the entity is fetched from storage only after the save has verifiably completed.

A broader follow-up to provide reusable page-transition-aware infrastructure (discussed as `pressSubmitButton()` or similar) was tracked separately in [#3582376](https://www.drupal.org/node/3582376).
