## Summary

Installing a recipe-based feature through Project Browser or Automatic Updates on a site that had previously unpacked recipes (such as Drupal CMS with the Haven template) would fail with a "directory already exists but is not managed by Composer" error. The `drupal/core-recipe-unpack` Composer plugin intentionally detaches recipe directories from Composer's control, but this confused Package Manager's overwrite validator, which then blocked any subsequent installation that would write to those same directories. The validator now accepts an allow-list of path patterns so that recipe directories can be safely overwritten.

## Impact

Affects site owners.

- **Config change:** New `package_manager.allow_overwrite` container parameter (array of regex strings) in `package_manager.services.yml`; defaults to `['|^/recipes/.|']` to permit overwriting existing recipe directories.

## Technical details

`OverwriteExistingPackagesValidator` (in `package_manager`) fires during `PreApplyEvent` and blocks any new package from being written into a directory that already exists in the active codebase but is not tracked by Composer. This protects against Composer silently clobbering custom modules that happen to share a name with a contrib package.

The `drupal/core-recipe-unpack` plugin breaks this assumption: its purpose is to take a Composer-managed recipe directory and remove it from Composer's awareness while leaving the files in place. A recipe that was previously installed and unpacked therefore sits in the `recipes/` directory without any Composer record. When Package Manager later tries to install a package into that same path, the validator sees an existing, non-Composer-managed directory and raises an error.

The fix adds a `package_manager.allow_overwrite` container parameter (defined in `package_manager.services.yml`) containing an array of regular expressions. The validator checks each new package's relative path against these patterns before doing the directory existence check; a match causes the package to be skipped. The default value `['|^/recipes/.|']` permits anything under `recipes/`. The parameter is injected into `OverwriteExistingPackagesValidator.__construct()` via the `#[Autowire(param: 'package_manager.allow_overwrite')]` attribute.

Contrib modules that implement recipe-like systems can extend this allow-list by adding their own patterns to the container parameter. The approach was suggested by @catch as an alternative to a boolean config flag; using a container parameter requires no configuration schema, no update path, and keeps the logic focused to this one validator.

### Collaboration

The fix approach evolved significantly. @phenaproxima initially proposed either removing the validator entirely or a complex multi-condition skip, then floated a boolean config flag (defaulting to off, with Drupal CMS opting in). @catch suggested a path allow-list passed as a service/container parameter instead, which avoided config schema and an update path. A separate thread clarified terminology: @phenaproxima had implemented it as a constructor argument ("service parameter"), but @catch confirmed they meant a container-level parameter. @phenaproxima then converted it to a container parameter and also had to switch from `fnmatch()` with literal path strings to regular expressions after PHPStan or PHPCS rejected `fnmatch()`.
