## Summary

Drupal 11.4 introduces a dedicated `Theme` extension object and properly deprecates the long-standing `system_region_list()` and `system_default_region()` procedural functions along with the `REGIONS_ALL` and `REGIONS_VISIBLE` constants. Sites that call those functions or use those constants in custom or contrib code should migrate to the new OOP API on the `Theme` object. No user-facing behavior changes.

## Impact

Affects module maintainers.

- **API change:** `ThemeHandlerInterface::addTheme()` now requires a `\Drupal\Core\Extension\Theme` argument instead of the base `Extension`; callers must pass a `Theme` object.
- **Deprecation:** `system_region_list()` (removed drupal:13.0.0), `REGIONS_ALL`/`REGIONS_VISIBLE` global constants and `BlockRepositoryInterface` equivalents (removed drupal:13.0.0), `BlockListBuilder::systemRegionList()` and `BlockDeleteForm::systemRegionList()` (removed drupal:12.0.0), `BlockController::getVisibleRegionNames()` (removed drupal:12.0.0). Use `ThemeHandlerInterface::getTheme()->listAllRegions()`, `->listVisibleRegions()`, or `->getDefaultRegion()` instead.
- **New API:** New `\Drupal\Core\Extension\Theme` class (extends `Extension`, marked `@final`) with `listAllRegions()`, `listVisibleRegions()`, and `getDefaultRegion()` methods; new protected `subClassExtension()` hook in `ExtensionList` for returning typed extension subclasses.

## Technical details

The root problem was twofold. First, `#2807785` deprecated `REGIONS_VISIBLE` and `REGIONS_ALL` in the System module but pointed callers at `\Drupal\block\BlockRepositoryInterface` constants, which was conceptually wrong: regions belong to themes, not blocks. Second, no typed Theme-specific object existed, so region logic lived in procedural helpers (`system_region_list()`, `system_default_region()`) rather than on the extension object itself.

**New `Theme` class (`core/lib/Drupal/Core/Extension/Theme.php`):** Extends `Extension` and is marked `@final` (extending is unsupported). Its constructor accepts `$root`, `$pathname`, `array $info`, and an optional `$filename`. It exposes three new public methods: `listAllRegions()` returns all regions as `TranslatableMarkup[]` keyed by machine name; `listVisibleRegions()` filters out entries listed in `info['regions_hidden']`; `getDefaultRegion()` returns the machine name of the first visible region.

**`ExtensionList::subClassExtension()` hook (`core/lib/Drupal/Core/Extension/ExtensionList.php`):** A new protected `subClassExtension(Extension $extension): Extension` method is added to the base list class and called inside `doList()` after `createExtensionInfo()`. The loop is updated to key-track extensions so the replacement object is stored back. The base implementation is a no-op pass-through; subclasses override it to return a typed subclass.

**`ThemeExtensionList::subClassExtension()` override:** Returns `new Theme($this->root, $extension->getPathname(), $extension->info, $extension->getExtensionFilename())`, converting the raw `Extension` into a `Theme` for every entry in the theme list.

**`ThemeHandlerInterface` and `ThemeHandler`:** `addTheme()` now declares `Theme $theme` (hard break; the old `Extension` type is no longer accepted). `listInfo()`, `rebuildThemeData()`, and `getTheme()` return/yield `Theme` objects. A guard in `ThemeHandler::listInfo()` adds `instanceof Theme` to the existing `isset($list[$theme])` check so stale cache entries containing bare `Extension` objects do not cause errors.

**Deprecations (all drupal:11.4.0, removed drupal:13.0.0 unless noted):**
- `system_region_list()` — removed drupal:13.0.0. Triggers `E_USER_DEPRECATED` and delegates to `Theme::listAllRegions()` or `Theme::listVisibleRegions()`.
- `system_default_region()` — same timeline (mentioned in issue body; not visible in this partial diff).
- `REGIONS_ALL` and `REGIONS_VISIBLE` global constants.
- `BlockRepositoryInterface::REGIONS_ALL` and `BlockRepositoryInterface::REGIONS_VISIBLE` — removed drupal:13.0.0.
- `BlockListBuilder::systemRegionList()` — removed drupal:12.0.0. `ThemeHandlerInterface` injected as optional fifth constructor argument with deprecation shim.
- `BlockDeleteForm::systemRegionList()` — same timeline. `ThemeHandlerInterface` injected via new explicit constructor and `ContainerInterface::create()`.
- `BlockController::getVisibleRegionNames()` — removed drupal:12.0.0.

**Callers updated inside core:** `BlockHooks::rebuild()`, `BlockHooks::themeInitialize()`, `block_theme_initialize()`, `Block::preSave()`, `Block::validateRegion()`, and `BlockListBuilder::buildBlocksForm()` all now resolve regions via `$themeHandler->getTheme($name)->listAllRegions()` or `->listVisibleRegions()` / `->getDefaultRegion()`.

**Tests:** Kernel tests that previously instantiated `BlockHooks` directly (`new BlockHooks()`) are updated to resolve it from the container (`\Drupal::service(BlockHooks::class)`) because it now requires `ThemeHandlerInterface` via constructor injection. Package Manager tests that called `$theme_handler->addTheme($extension)` now wrap the `Extension` in a `new Theme(...)` first.

## Upgrade

Replace calls to the deprecated procedural functions:

```php
// Before
$all     = system_region_list($theme_name);
$visible = system_region_list($theme_name, REGIONS_VISIBLE);
$default = system_default_region($theme_name);

// After
$theme_obj = \Drupal::service('theme_handler')->getTheme($theme_name);
$all       = $theme_obj->listAllRegions();
$visible   = $theme_obj->listVisibleRegions();
$default   = $theme_obj->getDefaultRegion();
```

Replace references to the deprecated constants:

```php
// Before
if ($show === REGIONS_VISIBLE) { ... }
if ($show === \Drupal\block\BlockRepositoryInterface::REGIONS_VISIBLE) { ... }

// After – pass the Theme object directly and call the appropriate method
$visible = $theme_obj->listVisibleRegions();
$all     = $theme_obj->listAllRegions();
```

If your class extends `BlockListBuilder` and overrides `systemRegionList()`, migrate to injecting `ThemeHandlerInterface` and calling `getTheme()->listAllRegions()` or `->listVisibleRegions()` directly.

If you call `ThemeHandlerInterface::addTheme()`, you must now pass a `\Drupal\Core\Extension\Theme` object:

```php
// Before
$theme_handler->addTheme($extension);

// After
use Drupal\Core\Extension\Theme;
$theme = new Theme($app_root, $extension->getPathname(), $info_array, $extension->getExtensionFilename());
$theme_handler->addTheme($theme);
```

### Collaboration

alexpott opened the issue and drove almost all iterations, cycling through three distinct approaches before landing on the `Theme` subclass strategy: first putting methods on `ThemeHandler`, then a `ThemeInfo` value object, then the current `Theme extends Extension` approach. tim.plunkett pushed back on expanding `ThemeHandlerInterface` directly, which contributed to the value-object direction. phenaproxima and markhalliwell (markcarver) both independently recommended marking `Theme` as `@final` or `@internal` to avoid locking in an incomplete API, and the final patch honours that with `@final` and the "no BC for subclasses" note. The issue sat stalled across multiple Drupal minor-version cycles (8.7 through 11) due to repeated re-roll requirements; voleger performed several rerolls that kept it alive and ultimately got it to a committable state.
