## Summary

Drupal core updated its `phpunit/phpunit` development dependency to address CVE-2026-24765, a security vulnerability in PHPUnit. The fix landed on the `main`, `11.x`, `11.3.x`, and `10.6.x` branches. PHPUnit is a development-only dependency: production sites are not affected, but developers who run Drupal's test suite locally or in CI should update their dev dependencies to pick up the patched version.

## Impact

Affects site owners.

- **Security:** CVE-2026-24765 in PHPUnit addressed by updating to 11.5.50 (main/11.x/11.3.x) and 9.6.34 (10.6.x); development-only, no production impact.

## Technical details

The `phpunit/phpunit` version constraint in `core/composer.json` was tightened from `^11.5.42` to `^11.5.50` on the `main`, `11.x`, and `11.3.x` branches, and the constraint in `core/tests/PHPStan/composer.json` was tightened from `^11` to `^11.5.50`. The locked version in `composer.lock` moved from `11.5.44` to `11.5.50`. The 10.6.x branch (which uses PHPUnit 9.x) was similarly updated to `9.6.34`.

Because `phpunit/php-code-coverage` 11.0.12 tightened its own requirement on `nikic/php-parser` to `^5.7.0`, the lock file also pulled in updated versions of `phpunit/php-code-coverage` (11.0.11 to 11.0.12), `nikic/php-parser` (v5.6.2 to v5.7.0), and `sebastian/comparator` (6.3.2 to 6.3.3). These are all dev-only transitive dependencies dragged in by the PHPUnit dependency tree; none touch production code. Running `composer audit` on an unpatched branch would have flagged the vulnerability.

When kentr's initial MR used `composer update -W phpunit/phpunit` and pulled in more packages than expected, smustgrave questioned whether the scope was too broad. mondrake clarified that updating the transitive packages was unavoidable given the dependency hierarchy, and documented the correct three-step process: `composer update phpunit/phpunit -w`, manually bump the constraint to `"^11.5.50"` in both `composer.json` files, then `composer update --lock`. When cherry-picking to `11.x` and `11.3.x` failed, dww recreated the MRs for those branches and also opened a separate MR for `10.6.x` after checking that the locked version there (9.6.29) was within the vulnerable range noted in the advisory.
