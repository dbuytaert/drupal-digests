## Summary

Sites with the Navigation module enabled crashed with a fatal error when any non-existent URL was visited. The error surfaced during page rendering while the Navigation module tried to determine whether the current route is a content-entity route, and the route object was null for 404 pages. This fix prevents the crash so 404 pages render normally. The patch was committed to main, 11.x, and 11.3.x.

## Impact

No upgrade or configuration changes required.

## Technical details

`EntityRouteHelper::isContentEntityRoute()` called `$this->routeMatch->getRouteObject()->getPath()` without first checking whether `getRouteObject()` returns a non-null value. On a 404 route the route object is null, causing the fatal `Call to a member function getPath() on null`. The call chain was: `TopBar::preRenderTopBar` -> `PageActions::build` -> `NavigationRenderer::hasLocalTasks` -> `NavigationRenderer::getLocalTasks` -> `EntityRouteHelper::isContentEntityRoute`.

The fix introduces an early `if (!$route) { return FALSE; }` guard, consistent with the pattern already applied to `getContentEntityFromRoute()` in the same class (see [#3565886](https://www.drupal.org/node/3565886), which fixed the same class for a related method but missed `isContentEntityRoute()`). A kernel test assertion for `isContentEntityRoute()` returning `FALSE` on a non-existent route was added to the existing `NavigationEntityRouteHelperTest`.
