## Summary

Views exposed date filters configured with a default "offset" value (such as "now" or "+2 days") were silently ignored on initial page load. When a visitor arrived at the view without submitting the exposed filter form, the filter produced no results restriction, effectively showing all content regardless of the configured date default. This is now fixed: the default offset is applied automatically so the view filters correctly from the first page load.

## Impact

No upgrade or configuration changes required.

## Technical details

The bug was in `acceptExposedInput()` inside the Views date filter handler (`core/modules/views/src/Plugin/views/filter/Date.php`). When the exposed input arrived empty, the method immediately returned `FALSE` for both the single-value operator path and the between/not-between operator path, bypassing the filter entirely. It never checked whether a default offset had been configured in `$this->options['value']`.

The fix inserts a check just before each early `return FALSE`. If the submitted value is blank, the code now inspects `$this->options['value']['type']`. When that equals `'offset'` and the configured default value is non-empty, the method copies the stored offset (either `value` for single-value operators, or `min`/`max` for range operators) into `$this->value`, sets `$this->value['type']` to `'offset'`, and returns `TRUE` so the filter proceeds to query the backend with the resolved date. If no default offset exists the original `return FALSE` path is preserved.

Two functional tests were added to `testExposedOffsetDefaultAppliedOnPageLoad()` and `testExposedOffsetBetweenDefaultAppliedOnPageLoad()`, covering the single-value `>` operator case and the `between` operator case respectively, including a node placed beyond the upper bound to confirm it is correctly excluded.
