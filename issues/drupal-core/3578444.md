## Summary

Views exposed date filters that are configured with a default offset value (for example "now" or "+2 days") were silently ignored on page load. Because no form had been submitted, the filter's `acceptExposedInput()` method treated the empty input as a no-op and returned `FALSE`, bypassing the configured default entirely. After this fix, a date filter with a default offset type will apply that offset on the initial page load, so visitors see correctly date-filtered results without having to submit the exposed filter form first.

## Impact

No upgrade or configuration changes required.

## Technical details

The bug was in `Date::acceptExposedInput()` in the Views date filter plugin. When the exposed input is empty (no form submission), the method checks whether the resolved value is empty and returns `FALSE` early, preventing the filter from running. Two such early-return guards exist: one for single-value operators (greater than, less than, equals, etc.) checking `$this->value['value']`, and one for range operators (between, not between) checking `$this->value['min']` and `$this->value['max']`.

The fix inserts a check before each `return FALSE`: if `$this->options['value']['type']` is `'offset'` and the corresponding option fields are non-empty, the method copies the configured default offset strings into `$this->value` and returns `TRUE` instead. This causes the filter to proceed and lets the backend resolve the offset strings into concrete dates at query time. Two new functional tests, `testExposedOffsetDefaultAppliedOnPageLoad()` and `testExposedOffsetBetweenDefaultAppliedOnPageLoad()`, verify the single-value and range cases respectively, including a node beyond the upper bound to confirm it is excluded.
