## Summary

Sites that use `hook_locale_translation_projects_alter()` to set a `weight` key on translation projects will see that weight respected again when Drupal processes translation files. The weight was silently ignored in Drupal 11.4.x due to a regression, so any modules (such as `locale_deploy`) relying on this ordering mechanism would have had translation files processed in the wrong order.

## Impact

No upgrade or configuration changes required.

## Technical details

The regression was introduced by the locale modernization refactor in [#3037156](https://www.drupal.org/node/3037156). `LocaleProjectRepository::buildProjects()` constructs `LocaleTranslatableProject` objects using named arguments, but the `weight` named argument was never passed. As a result the weight set by `hook_locale_translation_projects_alter()` implementations was always discarded.

The fix is a one-liner: `weight: $data['weight'] ?? 0` is added to the named-argument list in the `LocaleTranslatableProject` constructor call inside `buildProjects()`. A kernel test was added that sets the weight to 100 via a state flag (exercised through `localeTranslationProjectsAlter()` in the test module) and asserts `getWeight()` returns 100 after a fresh `buildProjects()` call. A baseline assertion confirming the default weight of 0 was also added.
