## Summary

The last four procedural functions in `claro.theme` have been removed and converted to class-based code. Two helper functions became protected methods in Hook classes, and toolbar-specific logic that handled Claro as admin theme moved from system module to toolbar module. The `claro.theme` file is now deleted. This change prepares for future toolbar deprecation work.

## Impact

No upgrade or configuration changes required.

## Technical details

Drupal's Claro theme had four remaining procedural functions in `claro.theme`. The issue removed all four and deleted the file.

Two helper functions became protected methods. `_claro_convert_link_to_action_link` moved to a new `ClaroLinkActionTrait` as `convertLinkToActionLink()`. `_claro_preprocess_file_and_image_widget` became `fileAndImageWidgetHelper()` in `ClaroHooks`. Both `ClaroHooks` and `ClaroFormHooks` use the trait to convert links to action links.

The other two functions handled toolbar rendering when Claro is the admin theme but not the active theme. System module had `hook_library_info_alter()` and `hook_theme_registry_alter()` implementations that checked `_system_is_claro_admin_and_not_active()` and called `claro_system_module_invoked_library_info_alter()` or `claro_system_module_invoked_theme_registry_alter()`. These system hooks are removed. The toolbar-specific logic is now in `ToolbarThemeHooks` with three new hook implementations: `libraryInfoAlter()`, `themeRegistryAlter()`, and `preprocessToolbarForClaro()`. Each checks `isClaroAdminAndNotActive()`, a new protected method that replaces the global `_system_is_claro_admin_and_not_active()` function. `ToolbarThemeHooks` constructor now injects `ThemeHandlerInterface`, `ThemeManagerInterface`, `AccountProxyInterface`, and `ConfigFactoryInterface`. The system module's `preprocessToolbar()` hook that existed only to call Claro preprocessing is also removed.
