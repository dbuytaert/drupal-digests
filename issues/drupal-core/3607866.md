## Summary

When running `composer update` to upgrade from Drupal 11.3.x to 11.4.x, the new `autoload_runtime.php` stub file was sometimes not written to the web root. Because `web/index.php` in 11.4 requires that file, the site immediately returned an HTTP 500 error after the update. The workaround was to run `composer install` a second time after the update completed. This fix eliminates the need for that second invocation.

## Impact

No upgrade or configuration changes required.

## Technical details

Drupal 11.4 introduced `autoload_runtime.php` (a stub that delegates to `vendor/autoload_runtime.php` for the Symfony Runtime integration), and the `drupal/core-composer-scaffold` plugin was updated to generate that file via `Handler::scaffold()`. The problem is a class-loading race inside a single Composer process.

When `composer update` runs, Composer fires `post-package-install`/`post-package-update` events as each package is extracted. If any package that declares `plugin-modifies-downloads` (or any plugin that triggers `postPackage()` on the scaffold plugin) is installed *before* `drupal/core-composer-scaffold` itself is updated on disk, PHP loads and pins the 11.3 version of `Handler` for the remainder of the process. Composer does reload `Plugin.php` itself via `eval()` after the scaffold package is replaced (see `PluginManager::registerPackage()`), but `Handler` was already in PHP's class cache and cannot be reloaded. So when the final `postCmd` event fires `handler()->scaffold()`, the stale 11.3 `Handler` code runs and skips `autoload_runtime.php` generation entirely.

The fix adds `ensureAutoloadRuntimeFile()` directly to `Plugin`, which is always reloaded with the new code. It is called from `postCmd()` immediately after `handler()->scaffold()`. The method uses `ManageOptions`, `AllowedPackages`, `GenerateAutoloadRuntimeReferenceFile`, and `ManageGitIgnore` directly (without going through `Handler`) and is a no-op when the file already exists or when no allowed packages are configured. A `@todo` marks it for removal once upgrades from 11.3 are no longer supported.

The regression test introduces a fixture Composer plugin (`fixtures/composer-plugin-downloads-modifier`) that declares `"plugin-modifies-downloads": true`, which causes Composer's dependency resolver (`Transaction::movePluginsToFront()`) to install it ahead of the scaffold plugin update, reliably reproducing the stale-Handler scenario. The test then asserts both that the stale handler's marker string (`SCAFFOLD_TEST_STALE_HANDLER`) appeared in output and that `autoload_runtime.php` was nonetheless created by the refreshed `Plugin` code.

## Upgrade

No code changes are required in projects. Sites that already ran into the 500 error can recover by running:

```
composer install
```

After this fix ships in `drupal/core-composer-scaffold`, a single `composer update` will correctly produce `autoload_runtime.php`. If your build pipeline copies only tracked Git files, ensure `autoload_runtime.php` is not listed in your web root's `.gitignore` (it is added there by the scaffold plugin and must be included in deployment artifacts).

### Collaboration

longwave diagnosed the root cause through direct instrumentation: they added `debug_print_backtrace()` to the `Handler` constructor and observed that the constructor was called while the 11.3 plugin code was still on disk, confirming that `Handler` was locked in PHP's class cache before the scaffold package was overwritten. That specific debugging session (comments #50 and #51) identified the class-loading race and the `eval()` mechanism Composer uses to reload only the `Plugin` class, which pointed directly to the solution: put the recovery logic in `Plugin` rather than `Handler`.
