## Summary

Sites with many entity bundles, such as those using the Webform or Paragraphs modules, now experience faster page loads when caches are empty or cold. The system now rebuilds field definitions for all bundles at once instead of making hundreds of separate cache operations. This reduces the impact on site performance during cache rebuilds, especially noticeable on sites with dozens or hundreds of webforms or paragraph types. The optimization targets cold cache scenarios where every cache miss previously triggered a separate write operation.

## Impact

No upgrade or configuration changes required.

## Technical details

The `EntityFieldManager::getFieldDefinitions()` method previously cached field definitions one bundle at a time. When a cache miss occurred for one bundle, it would rebuild and cache only that bundle's definitions. On sites with many bundles (100+ webforms, 50+ paragraph types), this resulted in hundreds of individual cache set operations during cold cache requests.

The method now anticipates that when field definitions are requested for one bundle, they will likely be needed for all bundles of that entity type during the same request. When a cache miss occurs, `getFieldDefinitions()` now fetches all bundles from `EntityTypeBundleInfoInterface::getBundleInfo()`, builds field definitions for all of them by calling `buildBundleFieldDefinitions()` in a loop, and writes all cache entries at once using `CacheBackendInterface::setMultiple()`.

The implementation maintains backward compatibility by still supporting field definition requests for non-existent bundles, though this is marked with a TODO for future deprecation. Each bundle still gets its own cache entry with the same cache ID format, but they are now read and written as a batch when any single bundle experiences a cache miss.

Performance test updates show the optimization saves approximately 10 cache gets and sets on Umami, with proportionally larger savings on sites with more bundles (99 operations saved for 100 webforms).

## Lessons learned

- When a system performs identical operations in a loop creating many small cache entries, you can reduce I/O by batching the work and writing all cache entries at once. The cost of loading slightly more data on cache hits is usually negligible compared to the overhead of hundreds of separate cache operations during cold cache scenarios. *(via catch)*
- When optimizing caching for batch operations, consider that cache misses may not always affect all entries simultaneously. Designing for cold-cache scenarios (where all items are missing) may do unnecessary work when only single entries are evicted, though this tradeoff is usually acceptable when items are accessed frequently. *(via berdir)*
