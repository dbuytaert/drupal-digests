## Summary

Ten views procedural functions have been deprecated. Four enable/disable wrapper functions now have direct equivalents on the View entity. The `views_get_view_result` function is replaced by a new static method on the Views class. The `views_embed_view` function is replaced by `'#type' => 'view'` render elements. The `views_set_current_view` and `views_get_current_view` functions are deprecated without replacement because the concept of a "current view" will not work reliably with fibers and asynchronous rendering. Two internal helper functions are deprecated without public replacement.

## Impact

Affects module maintainers.

- **Deprecation:** Ten views functions: `views_view_is_enabled`, `views_view_is_disabled`, `views_enable_view`, `views_disable_view`, `views_set_current_view`, `views_get_current_view`, `views_embed_view`, `views_get_view_result`, `_views_query_tag_alter_condition`, and `views_element_validate_tags`
- **New API:** New `Views::getViewResult()` static method for retrieving view results

## Technical details

Views has accumulated procedural wrappers over the years that are no longer needed. The enable/disable functions (`views_view_is_enabled`, `views_view_is_disabled`, `views_enable_view`, `views_disable_view`) were added in 2011-2012 for use with Drush and as `array_filter` callbacks. These are simple wrappers around methods already available on the View entity. The `views_get_view_result` function is moved to `Views::getViewResult()` to continue the pattern of static methods on the Views helper class. The `views_embed_view` function is replaced by the existing `'#type' => 'view'` render element, which already handles access checks in `preRenderViewElement`. The `views_set_current_view` and `views_get_current_view` functions rely on global statics to track which view is executing. With fibers and lazy builders, single entities may be rendered outside view execution, making the concept of "current view" unreliable. These are deprecated without replacement. In `ViewExecutable::preExecute()` and `postExecute()`, the internal implementation switches to direct `$GLOBALS['_current_view']` access with a follow-up issue to remove the global entirely. The `_views_query_tag_alter_condition` helper is moved to a protected method in `ViewsHooks`. The `views_element_validate_tags` callback is moved to a static method on the Sql query plugin marked `@internal`.

## Upgrade

```php
// Before
if (views_view_is_enabled($view)) { }
if (views_view_is_disabled($view)) { }
views_enable_view($view);
views_disable_view($view);

// After
if ($view->status()) { }
if (!$view->status()) { }
$view->enable()->save();
$view->disable()->save();
```

```php
// Before
$results = views_get_view_result('my_view', 'block_1', $arg1, $arg2);

// After
use Drupal\views\Views;
$results = Views::getViewResult('my_view', 'block_1', $arg1, $arg2);
```

```php
// Before
$output = views_embed_view('my_view', 'block_1', $arg1);

// After
$output = [
  '#type' => 'view',
  '#name' => 'my_view',
  '#display_id' => 'block_1',
  '#arguments' => [$arg1],
];
```

The `views_set_current_view`, `views_get_current_view`, `_views_query_tag_alter_condition`, and `views_element_validate_tags` functions have no public replacement.

### Collaboration

The maintainers initially debated whether to keep the `views_set_current_view` and `views_get_current_view` functions or provide a replacement service. Berdir raised concerns about use cases that rely on knowing which view is rendering an entity, particularly for cache keys and display logic. After discussion, Berdir recognized in comment 35 that with fibers and lazy builders, entities may be rendered outside view execution, making the concept of "current view" unreliable. The group agreed to deprecate without replacement because the functions cannot work as expected in future rendering scenarios.
