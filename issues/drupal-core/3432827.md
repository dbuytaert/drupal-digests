## Summary

The `plugin.cache_clearer` service was marked `lazy: true` in Drupal core, but the lazy proxy it generated never actually deferred instantiation because the `PluginManagerPass` compiler pass called `addCachedDiscovery()` on the proxy during container compilation, which immediately resolved the real service. The lazy proxy was therefore pointless. This change removes the generated proxy class entirely and replaces the method-call pattern with a Symfony tagged iterator: all plugin managers are now tagged with `plugin_manager_cache_clear` and injected into `CachedDiscoveryClearer` via constructor. The `addCachedDiscovery()` method was removed from `CachedDiscoveryClearerInterface`. This is an internal refactor with no user-facing effect, but contrib code that calls `addCachedDiscovery()` or implements the interface will need updating.

## Impact

Affects module maintainers.

- **API change:** `addCachedDiscovery()` removed from `CachedDiscoveryClearerInterface`; services should use the `plugin_manager_cache_clear` tag instead.
- **Deprecation:** `CachedDiscoveryClearer::addCachedDiscovery()` deprecated in drupal:11.1.0 and removed in drupal:12.0.0; use the `plugin_manager_cache_clear` service tag.

## Technical details

The broken lazy-proxy setup had two actors: `core.services.yml` declared `plugin.cache_clearer` with `lazy: true`, causing a proxy class (`core/lib/Drupal/Core/ProxyClass/Plugin/CachedDiscoveryClearer.php`) to be generated. `PluginManagerPass` then registered every `plugin.manager.*` service that implements `CachedDiscoveryInterface` by calling `$cache_clearer_definition->addMethodCall('addCachedDiscovery', [new Reference($service_id)])`. Because those method calls ran on the proxy during container compilation, the proxy immediately resolved the real `CachedDiscoveryClearer` service, defeating the lazy-loading entirely.

The fix converts the service to use Symfony's `#[AutowireIterator]` attribute. `CachedDiscoveryClearer` now has an explicit constructor:

```php
public function __construct(
  #[AutowireIterator(tag: 'plugin_manager_cache_clear')]
  protected \Traversable $cachedDiscoveries,
) {}
```

`PluginManagerPass::process()` no longer touches the `plugin.cache_clearer` definition at all. Instead it walks all service definitions and, for any `plugin.manager.*` service implementing `CachedDiscoveryInterface` that does not already carry the tag, calls `$definition->addTag('plugin_manager_cache_clear')`. Services can also be tagged explicitly in their `services.yml`. The condition was also changed from `||` (service name prefix OR has tag) to `&&` (service name prefix AND does NOT already have tag) to avoid double-tagging.

In `core.services.yml`, `lazy: true` was replaced with `autowire: true` on the `plugin.cache_clearer` definition, and the generated proxy file was deleted.

`addCachedDiscovery()` was removed from `CachedDiscoveryClearerInterface`. In this commit the method was retained on the concrete `CachedDiscoveryClearer` class as a deprecated shim (deprecated in drupal:11.1.0, targeted for removal in drupal:12.0.0), storing callers in a separate `$legacyCachedDiscoveries` list iterated inside `clearCachedDefinitions()`. The current HEAD no longer contains the shim, indicating a subsequent commit completed the removal.

Autoconfigure was considered as an alternative but rejected: enabling it requires `autoconfigure: true` in every `services.yml` that declares a plugin manager, and adding that retroactively to contrib service files is not backward compatible, as noted in comments and in the related issue [#3446026](https://www.drupal.org/node/3446026).

## Upgrade

If your code calls `addCachedDiscovery()` directly, tag the service instead:

```yaml
# In your module's services.yml
services:
  plugin.manager.my_manager:
    class: Drupal\MyModule\Plugin\MyPluginManager
    tags:
      - { name: plugin_manager_cache_clear }
```

Services already following the `plugin.manager.*` naming convention and implementing `CachedDiscoveryInterface` are auto-tagged by `PluginManagerPass` with no manual change needed.

If your code implements `CachedDiscoveryClearerInterface`, remove the `addCachedDiscovery()` method from your implementation; the method no longer exists on the interface.

### Collaboration

Two design decisions were settled through discussion. First, `catch` proposed adding `@deprecated` to the `addCachedDiscovery()` interface method in a 10.3.x MR. `longwave` pushed back: leaving the method on the interface, even deprecated, would force every implementing class to keep its implementation. Instead `longwave` removed the method from the interface and kept a deprecated copy only on the concrete class so that direct callers get a deprecation notice without burdening implementors. Second, `alexpott` asked why autoconfigure was removed from the patch. `longwave` explained that autoconfigure requires `autoconfigure: true` to be set in each `services.yml` that declares a plugin manager, making it not backward compatible for existing contrib code, a constraint independently confirmed by `catch`.
