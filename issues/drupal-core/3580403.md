## Summary

The `WorkspaceSwitchEvent` event now includes a flag to indicate whether a workspace switch is temporary or persistent. Temporary switches happen when code uses `executeInWorkspace()` or `executeOutsideWorkspace()` and are automatically reverted after the callable finishes. Persistent switches happen when calling `setActiveWorkspace()` or `switchToLive()`. Event subscribers can now check this flag to handle the two scenarios differently, which is needed by workspace negotiators in contrib modules like WSE.

## Impact

Affects module maintainers.

- **New API:** `WorkspaceSwitchEvent::isTemporary()` method to distinguish temporary from persistent workspace switches

## Technical details

`WorkspaceManager` has two patterns for switching workspaces. Methods like `setActiveWorkspace()` and `switchToLive()` change the active workspace until explicitly changed again. Methods like `executeInWorkspace()` and `executeOutsideWorkspace()` switch temporarily to run a callable, then switch back.

Both patterns call the protected `doSwitchWorkspace()` method, which dispatches a `WorkspaceSwitchEvent`. Event subscribers previously had no way to tell which pattern triggered the switch. The WSE module needed this distinction to avoid dropping preview cookies during temporary switches (see [#3571367](https://www.drupal.org/node/3571367)).

The change adds a boolean `$isTemporary` parameter to `WorkspaceSwitchEvent` constructor (defaults to FALSE for backward compatibility) and exposes it via `isTemporary()`. The `doSwitchWorkspace()` method now accepts an `$is_temporary` parameter, which `executeInWorkspace()` and `executeOutsideWorkspace()` pass as TRUE. Regular switches via `setActiveWorkspace()` and `switchToLive()` pass FALSE (the default).

Test coverage verifies that persistent switches dispatch events with `isTemporary()` returning FALSE, while temporary switches return TRUE for both the initial switch and the automatic revert.
