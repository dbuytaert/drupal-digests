## Summary

Configuration forms that use the `#config_target` property on a `text_format` (rich text) field threw a `LogicException` on submission: "Two #config_targets both target..." The bug made it impossible to wire a `text_format` element directly to configuration using the standard `ConfigFormBase` mechanism. After this fix the combination works, and the form also correctly re-displays the saved value on subsequent page loads.

## Impact

Affects module maintainers.

- **New API:** ConfigTarget with a fromConfig callback is now the supported pattern for wiring text_format elements to config in ConfigFormBase forms

## Technical details

The `text_format` form element's `TextFormat::processFormat()` builds child elements (`value` textarea and `format` selector) by copying most properties from the parent element. `#config_target` was being copied along with everything else, so `ConfigFormBase::doStoreConfigMap()` encountered the parent element and the `value` child both claiming the same config key and threw a `LogicException`. The fix adds `#config_target` to the `$keys_not_to_copy` list inside `TextFormat::processFormat()`. Because `text_format` stores a compound value (`value` and `format` sub-keys), using `#config_target` on such an element requires a `ConfigTarget` instance with a `fromConfig` callback that extracts only the text portion for the textarea's `#default_value`, while `#format` must be set manually from config. The test fixture in `form_test` demonstrates the intended pattern. A second change in `ConfigFormBase::storeConfigKeyToFormElementMap()` collects cache tags from all editable configs referenced by the map and applies them to the form render array via `CacheableMetadata`, ensuring the form cache is invalidated when the underlying configuration changes. This latter change also caused expected cache tag adjustments in several existing browser tests.

## Upgrade

To use `#config_target` on a `text_format` element, supply a `ConfigTarget` object with a `fromConfig` callback and set `#format` manually:

```php
$form['field'] = [
  '#type' => 'text_format',
  '#title' => $this->t('My field'),
  '#format' => $this->config('mymodule.settings')->get('field.format'),
  '#config_target' => new ConfigTarget(
    'mymodule.settings',
    'field',
    fromConfig: fn (?array $value = NULL): string => $value['value'] ?? '',
  ),
];
```

The `format` sub-key is saved automatically by the normal form submission path; no extra handling is needed for it.

### Collaboration

The issue went through two distinct approaches before landing. The first MR (9014, by VinmayiSwamy) added a `text_format`-specific special case directly inside `ConfigFormBase::doStoreConfigMap()`. catch redirected the effort, noting that hardcoding a specific element type in the config system was the wrong layer and that the exception exists for a reason: if the usage is valid, the system should accommodate it generically. idebr opened MR 10182 with a different strategy, placing `#config_target` in `TextFormat::processFormat()`'s `$keys_not_to_copy` list instead. alexpott then discovered that the fix did not correctly restore the saved value to the textarea on subsequent page loads, pushed a failing test to demonstrate it, and then committed a follow-up that resolved both the default-value display and the cache invalidation problem (idebr had spotted that the form lacked a cache dependency on the config being edited). The final commit combines all three findings.
