## Summary

Configuration forms extending `ConfigFormBase` that used a `text_format` (rich text) element with the `#config_target` property threw a `LogicException` on submission: "Two #config_targets both target..." This made it impossible to wire a rich-text field directly to configuration using the standard config-form mechanism. After this fix the combination works correctly, and the form re-displays the saved value on subsequent page loads. As a side effect, `ConfigFormBase` now also applies the cache tags of every config object it edits to the form's render array, so form caches are properly invalidated when the underlying configuration changes.

## Impact

Affects module maintainers.

- **New API:** Using `#config_target` on a `text_format` element in `ConfigFormBase` now works; requires a `ConfigTarget` with a `fromConfig` callback and a manually set `#format` property.

## Technical details

The `text_format` element's `TextFormat::processFormat()` builds two child elements (the `value` textarea and the `format` selector) by copying most parent properties. `#config_target` was being copied alongside everything else, so `ConfigFormBase::doStoreConfigMap()` encountered both the parent and the `value` child claiming the same config key and raised the exception. The fix adds `#config_target` to the `$keys_not_to_copy` list inside `TextFormat::processFormat()`, keeping it on the parent element only.

Because `text_format` stores a compound value (`value` and `format` sub-keys), using `#config_target` on such an element requires a `ConfigTarget` instance with a `fromConfig` callback that extracts only the text string for the textarea's `#default_value`. The `#format` property must be populated manually from config. The `format` sub-key is saved by the normal form submission path without any extra handling.

A second change in `ConfigFormBase::storeConfigKeyToFormElementMap()` iterates over all config names referenced in the map plus `getEditableConfigNames()`, collects their cache tags via `getCacheTags()`, and applies them to the form render array through `CacheableMetadata`. This corrects a previously undetected gap where the form had no cache dependency on the config objects it edits. Several browser tests required updated expected cache-tag sets (notably `config:system.cron`) as a result.

## Upgrade

To use `#config_target` on a `text_format` element in a `ConfigFormBase` form, supply a `ConfigTarget` with a `fromConfig` callback and set `#format` manually:

```php
use Drupal\Core\Config\Config\ConfigTarget;

$form['field'] = [
  '#type' => 'text_format',
  '#title' => $this->t('My field'),
  '#format' => $this->config('mymodule.settings')->get('field.format'),
  '#config_target' => new ConfigTarget(
    'mymodule.settings',
    'field',
    fromConfig: fn (?array $value = NULL): string => $value['value'] ?? '',
  ),
];
```

The `format` sub-key is saved automatically; no extra handling is needed for it.

### Collaboration

The issue went through two distinct approaches before landing. vinmayiswamy's first MR added a `text_format`-specific special case directly inside `ConfigFormBase::doStoreConfigMap()`. catch redirected the effort, noting that hardcoding a specific element type in the config system was the wrong layer and that if the usage is valid the system should accommodate it generically. idebr opened a second MR placing `#config_target` in `TextFormat::processFormat()`'s `$keys_not_to_copy` list instead. alexpott then found that the fix did not correctly restore the saved value to the textarea on subsequent page loads, pushed a failing test to demonstrate it, and later committed follow-up work. idebr separately spotted that the form lacked a cache dependency on the config being edited, which alexpott folded into the same commit. The final commit combines all three findings.
