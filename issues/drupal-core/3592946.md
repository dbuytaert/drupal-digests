## Summary

On sites using Single Directory Components (SDC) with a subtheme, `ComponentNegotiator::negotiate()` was re-running its full component-definition scan on every call instead of returning a cached result. The issue reports roughly 8× slower negotiation for repeated calls within the same request. The fix is a one-line correction that makes the cache work properly for the common case where no component replacement exists.

## Impact

No upgrade or configuration changes required.

## Technical details

`ComponentNegotiator::negotiate()` stores its result in `$this->cache` keyed by a generated cache key. The previous cache-hit check used `isset($this->cache[$cache_key] ?? NULL)`, which returns `false` when the cached value is `NULL`. Because `doNegotiate()` returns `NULL` when no replacement is found (the normal case for most components under a subtheme), the cache was effectively bypassed every time. `doNegotiate()` runs `array_filter` over the full set of component definitions on each miss, making the cost proportional to the total number of registered components multiplied by the number of `find()` calls per request.

The fix replaces the `isset()` pattern with `array_key_exists($cache_key, $this->cache)`, which correctly treats a stored `NULL` as a valid cache hit. A new kernel test `testNegotiateCachesNullResults` in `ComponentNegotiatorTest` verifies the behavior by injecting a fake replacement definition after the first `negotiate()` call and confirming that the null result is still served from cache on the second call.
