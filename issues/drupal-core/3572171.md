## Summary

During configuration import, Drupal sets a flag to prevent certain automatic actions like creating default blocks when themes are installed. Installing a theme triggers a container rebuild, which previously lost this flag. This caused blocks to be created during config sync even though they should have been skipped. The flag is now preserved across container rebuilds, ensuring that operations that should be skipped during configuration import remain skipped throughout the entire process.

## Impact

No upgrade or configuration changes required.

## Technical details

The `config.installer` service maintains an `isSyncing()` flag that tells Drupal when configuration is being imported. Code like `block_theme_initialize()` checks this flag to skip creating blocks during config sync, since blocks should come from the imported configuration.

The `ModuleInstaller` class already saved and restored this flag when installing modules triggered a container rebuild. However, when themes are installed, the container is also rebuilt through `DrupalKernel::initializeContainer()`, and this path did not preserve the flag.

The fix moves the preservation logic from `ModuleInstaller::updateKernel()` into `DrupalKernel::initializeContainer()`. Now both module and theme installation use the same mechanism. Before rebuilding the container, the syncing status and source storage are captured. After the new container is built, these values are restored.

The test was updated to actually install a theme (`test_theme`) instead of manually invoking hooks with hardcoded values. This provides a more realistic test that catches regressions in the theme installation path.
