## Summary

A stricter type check introduced in the `behat/mink` library caused functional tests that passed HTTP request headers as indexed strings (e.g. `['X-Requested-With: XMLHttpRequest']`) to throw a `TypeError` at runtime, because `Mink\Session::setRequestHeader()` now requires the header name to be a string, not an integer. This issue corrected all such call sites across Drupal core's functional test suite and added deprecation warnings to cover any remaining third-party test code that still uses the old calling convention.

## Impact

Affects module maintainers.

- **Deprecation:** `UiHelperTrait::drupalGet()` now triggers deprecations in drupal:11.1.0 when headers are passed as indexed strings or with null values; both removed in drupal:12.0.0

## Technical details

The problem was in how `UiHelperTrait::drupalGet()` accepts the third `$headers` argument. The documented contract requires an associative array (`['Header-Name' => 'value']`), but many call sites in core were passing flat indexed arrays (`['Header-Name: value']`) or using `null` as a header value. When Mink began enforcing strict types on `setRequestHeader()`, iterating over those indexed arrays passed an `int` as the header name, causing a fatal `TypeError`.

The fix has two parts:

**1. Call-site corrections (all branches including 10.3.x).** Ten test files were updated to use the associative format. The affected files are: `LanguageBrowserDetectionAcceptLanguageTest.php` (`Accept-Language: null` replaced with `''`), `MenuUiTest.php`, `PageCacheTest.php` (`If-None-Match: null` replaced with `''`), `StyleSerializerEntityTest.php`, `FrameworkTest.php`, `ContentNegotiationTest.php`, `StorageTest.php`, `RouterTest.php`, `StackSessionHandlerIntegrationTest.php`, and `ToolbarAdminMenuTest.php`. The `drupalGetAjax()` helper in `FrameworkTest.php` was also corrected: it had been appending `'X-Requested-With: XMLHttpRequest'` to a plain array (`$headers[] = ...`), which was replaced with `$headers = ['X-Requested-With' => 'XMLHttpRequest']`.

**2. Deprecations in `UiHelperTrait::drupalGet()` (11.x and 11.0.x only).** Two `@trigger_error` calls were added to `core/tests/Drupal/Tests/UiHelperTrait.php` inside the `foreach ($headers as $header_name => $header_value)` loop:
- When `is_int($header_name)`: the old colon-delimited string is split via `explode(':', $header_value)` as a compatibility shim, but fires a deprecation targeting `drupal:11.1.0` with removal in `drupal:12.0.0` (change record node 3456178).
- When `is_null($header_value)`: it is coerced to an empty string, but fires a deprecation also targeting `drupal:11.1.0` with removal in `drupal:12.0.0` (change record node 3456233).

The 10.3.x backport was intentionally test-fix-only with no deprecation, to keep all branches aligned while being less disruptive. Alexpott noted in the thread that without the deprecation it would be impossible to prove core had been fully fixed. This issue is related to [#3421105](https://www.drupal.org/node/3421105), which had previously left `@todo` comments in `UiHelperTrait.php` as placeholders for exactly these deprecations.

## Upgrade

If your contributed module or custom code calls `UiHelperTrait::drupalGet()` (directly or via `BrowserTestBase`) with headers as indexed strings or with `null` values, update the calls as follows:

```php
// Old (deprecated, causes TypeError with newer Mink):
$this->drupalGet('/path', $options, ['X-Requested-With: XMLHttpRequest']);
$this->drupalGet('/path', $options, ['Accept-Language' => NULL]);

// New:
$this->drupalGet('/path', $options, ['X-Requested-With' => 'XMLHttpRequest']);
$this->drupalGet('/path', $options, ['Accept-Language' => '']);
```
