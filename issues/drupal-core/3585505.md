## Summary

This is a purely internal cleanup of Drupal core's test code with no user-facing effect. It uses the Rector static analysis tool to automatically add PHP type declarations across all test classes in core and core modules, improving code clarity and helping static analysis tools catch bugs earlier.

## Impact

No upgrade or configuration changes required.

## Technical details

The change applies Rector's `typeDeclarations` prepared set to every test class under `core/tests` and `core/modules/**/tests/src`, covering unit, kernel, functional, and functional-JavaScript tests. The `TypedPropertyFromCreateMockAssignRector` rule was explicitly excluded from the run because it requires manual adjustment beyond what automation can safely handle.

The mechanical changes fall into four repeating patterns visible throughout the diff:

1. Return type declarations added to data-provider methods (the vast majority): untyped `public static function providerFoo()` becomes `public static function providerFoo(): array`, or `: \Iterator` where a generator is returned.
2. Parameter type declarations added to `#[DataProvider]`-driven test methods: previously untyped parameters like `$expected`, `$route`, `$method`, `$state`, etc. are given concrete scalar or class types (e.g. `string`, `bool`, `int`, `array`).
3. Return and parameter types added to anonymous closures and arrow functions used inline, e.g. `function ($v)` becoming `function (ConstraintViolationInterface $v): string`.
4. Return types added to helper and trait methods in test classes, e.g. `: void`, `: bool`, `: string|false`, `: mixed`, `: BigPipeResponseAttachmentsProcessor`.

As a side effect, Rector also alphabetically reordered some `use` import statements and added a few missing `use` declarations (e.g. `NodeElement` in a CKEditor5 test, `ConstraintViolationInterface` in the CKEditor5 unit test). The PHPStan baseline was regenerated after the run to reflect the new type information.

The follow-up issue [#3587327](https://www.drupal.org/node/3587327) was filed to run the same process again once new code has been added.
