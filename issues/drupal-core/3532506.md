## Summary

Passing `$has_trusted_data = TRUE` to `Config::save()` was intended to skip schema-based type coercion (the expensive `castValue()` pass), but it also accidentally skipped the basic sanity check that prevents non-scalar types such as objects and streams from being serialized into the config storage. As a result, calling `$config->set('foo', new stdClass())->save(TRUE)` would silently write PHP-serialized objects to the database. This is now fixed: basic type validation always runs regardless of the `$has_trusted_data` flag, so any unsupported type throws `UnsupportedDataTypeConfigException` at save time. Note that the `$has_trusted_data` parameter itself was deprecated in Drupal 11.4 by [#3347842](https://www.drupal.org/node/3347842), so this fix also guards the remaining lifecycle of that deprecated path.

## Impact

No upgrade or configuration changes required.

## Technical details

The bug was in the conditional inside `Config::save()` in `core/lib/Drupal/Core/Config/Config.php`. The original structure nested both `castValue()` and `validateValue()` inside a single `if (!$has_trusted_data)` guard, meaning the `else` branch that calls `validateValue()` was unreachable when trusted data was passed. The fix moves the condition so that only the schema-based `castValue()` call is guarded by `!$has_trusted_data`: the condition becomes `if (!$has_trusted_data && $this->typedConfigManager->hasConfigSchema($this->name))`, with an unconditional `else` branch that always runs `validateValue()` on every top-level key. This ensures the invariant that config storage only ever contains nested arrays of scalar values, even when the caller opts out of schema enforcement. New test assertions in `ConfigCRUDTest::testTrustedDataDeprecation()` cover both the case where a config schema exists and the case where none is registered, confirming that `UnsupportedDataTypeConfigException` is thrown in both situations.

## Upgrade

If you have code that calls `$config->save(TRUE)` and sets non-scalar values (objects, resources, etc.), that code will now throw `UnsupportedDataTypeConfigException`. Ensure all values are nested arrays of scalars before saving. Additionally, the `$has_trusted_data` parameter is deprecated as of Drupal 11.4 and will be removed in 13.0; migrate to calling `$config->save()` without arguments.
