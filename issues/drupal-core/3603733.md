## Summary

Drupal core's bundled Guzzle HTTP client libraries were updated to address three security advisories: CVE-2026-55766 (CRLF injection in HTTP start-line serialization in `guzzlehttp/psr7`, GHSA-vm85-hxw5-5432), CVE-2026-55567 (dot-only cookie domains matching all hosts in `guzzlehttp/guzzle`, GHSA-cwxw-98qj-8qjx), and CVE-2026-55568 (silent HTTPS-proxy downgrade to cleartext in `guzzlehttp/guzzle`, GHSA-wpwq-4j6v-78m3). Sites using `drupal/core-recommended` will receive the patched package versions when updating Drupal core. Sites that cannot wait for a core release can use Composer version aliases as a temporary workaround (see the issue for the exact command).

## Impact

Affects site owners.

- **Security:** Three medium-severity CVEs in guzzlehttp/psr7 (CRLF injection, CVE-2026-55766) and guzzlehttp/guzzle (dot-only cookie domain matching CVE-2026-55567, HTTPS-proxy downgrade CVE-2026-55568) fixed by updating to guzzlehttp/psr7 2.12.1 and guzzlehttp/guzzle 7.12.1.

## Technical details

The `composer.lock` and `composer/core-recommended/composer.json` pin three packages to their patched versions: `guzzlehttp/guzzle` 7.10.0 → 7.12.1, `guzzlehttp/psr7` 2.10.4 → 2.12.1, and `guzzlehttp/promises` 2.3.0 → 2.5.0 (the latter required because `guzzlehttp/guzzle` 7.12.1 raised its own constraint to `^2.5`).

The `guzzlehttp/psr7` constraint in `core/composer.json` was deliberately relaxed from `^2.10.2` back to `^2.8.0`. This separates the minimum-allowed version (for sites not using `core-recommended`) from the locked version, so the pinned security fix in `core-recommended` does not force a hard floor that blocks `composer install` on older compatible releases.

Two supporting code changes landed alongside the dependency bump. `CookieResourceTestTrait::getAuthenticationRequestOptions()` was tightened to only set the `Cookie` and `X-CSRF-Token` request headers when `$this->sessionCookie` and `$this->csrfToken` are non-null, removing a guzzle 7.11+ deprecation triggered by passing `null` to a headers option. Two PHPStan baseline entries in `core/.phpstan-baseline.php` were also updated: the deprecated-method message for `getConfig()` changed upstream from referencing `Client::getConfig` to `ClientInterface::getConfig`, so the regex patterns were updated to match.
