## Summary

The revisions page for content now displays all relevant revisions correctly. Previously, the page could show no revisions at all when many revisions existed, or incorrectly hide the actual current revision. This happened because the system loaded revisions from the database and then filtered them afterward, which broke pagination and caused confusing displays. The fix moves the filtering into the database query itself, ensuring the pager works correctly and all translation-affected revisions appear as expected.

## Impact

No upgrade or configuration changes required.

## Technical details

The revision overview pages in `NodeController::revisionOverview()` and `VersionHistoryController::loadRevisions()` previously loaded revisions via database queries that returned up to 50 records, then filtered those results in PHP using `hasTranslation()` and `isRevisionTranslationAffected()` checks. This approach caused two problems: pagination broke when many unaffected revisions existed (resulting in empty pages), and revisions with `revision_translation_affected = NULL` were incorrectly hidden.

The fix moves the translation filtering into the entity query itself. Both `NodeController::getRevisionIds()` and `VersionHistoryController::loadRevisions()` now add `langcode` and `revision_translation_affected` conditions directly to the query before pagination occurs. This ensures the pager counts only relevant revisions and all translation-affected revisions display correctly.

The post-query filtering loop was simplified to remove the `hasTranslation()` and `isRevisionTranslationAffected()` checks since the database query now handles this. The `EntityTestMulRev` test entity gained revision route definitions to support testing the generic revision UI.

## Lessons learned

- When filtering large result sets, applying conditions after pagination can create broken pagers and empty pages. Apply filters at the database query level before pagination to ensure consistent page sizes. *(via acbramley, catch)*
- When data is filtered in application code after database retrieval, the view presented to users may not reflect the actual data state. Move filtering logic into database queries to maintain data integrity and display consistency. *(via acbramley, liam morland)*
