## Summary

The revision history page for content could show incorrect pagination or even appear empty when a node had many translation-related revisions. The page loaded 50 revisions from the database and then filtered them in PHP, so if all 50 retrieved records were flagged as not affecting the current language, the displayed page would be empty while the pager still showed multiple pages. The fix moves the translation filtering into the database query so pagination counts only the revisions that will actually be shown, and the page always displays the expected number of entries.

## Impact

No upgrade or configuration changes required.

## Technical details

The revision overview pages load revisions via an entity query and previously applied additional PHP-level filtering to show only revisions where `isRevisionTranslationAffected()` returned `TRUE` for the current language. Because the pager was applied to the raw database query before that filtering, requesting page 2 could return 50 revisions that are all non-affecting for the current language, yielding a completely empty page while the pager remained intact.

The fix moves the translation filtering directly into the entity queries, before pagination is applied.

In `NodeController::getRevisionIds()`, two new conditions are added to the query: `langcode` (matching the current node language) and `revision_translation_affected` (requiring the value `'1'`). The wrapping `if ($revision->hasTranslation($langcode) && $revision->getTranslation($langcode)->isRevisionTranslationAffected())` conditional in `NodeController::revisionOverview()` is removed entirely; all revisions returned by the query are now unconditionally rendered, which also eliminates a level of indentation throughout the method.

In `VersionHistoryController::loadRevisions()`, the same approach is taken: the entity query is built first, and when the entity type is `$translatable`, `langcode` and `revision_translation_affected` conditions are added before calling `->pager(self::REVISIONS_PER_PAGE)->execute()`. The post-load `if` check that called `$revision->hasTranslation()` and `isRevisionTranslationAffected()` is removed from the generator loop.

Test coverage was extended in two places. A new `testNodeRevisionsTabPagerAffectedTranslations()` method in `NodeRevisionsUiTest` creates 50 translation-affecting revisions followed by 50 non-affecting ones and asserts that the page shows all 50 affecting revisions with no pager, then confirms a pager appears and works correctly once a 51st affecting revision is added. The `RevisionVersionHistoryTest::testRevisionsPagination()` test was switched from `EntityTestRev` (which has no `revision_data_table`) to `EntityTestMulRev` (which does), because the new `revision_translation_affected` query condition requires a `revision_data_table` to be present. A post-merge comment by paul121 identified that the change consequently breaks `VersionHistoryController` for translatable custom entity types that do not define a `revision_data_table`, producing a `QueryException` on `revision_translation_affected not found`.
