## Summary

The `default_admin` admin theme (the Gin contrib theme brought into core) has replaced its tangled dual CSS custom-property systems with a single, documented `--admin-*` token namespace. The old `--gin-*` properties are gone. The change also fixes a class of dark-mode bugs where surfaces were hardcoded to white and did not adapt to dark mode.

## Impact

Affects front-end developers.

- **Theme change:** The entire `--gin-*` CSS custom property surface is removed from the `default_admin` theme; all tokens are now in the `--admin-*` namespace with no deprecation aliases.

## Technical details

The theme previously carried two overlapping token systems inherited from its Seven → Claro → Gin lineage: a Claro layer (`--color-*`, `--space-*`, `--font-size-*`, `--input-*`) and a Gin layer (`--gin-*`) that cross-referenced each other with duplicated and conflicting values. This change consolidates everything into one `--admin-*` namespace organised in three tiers: primitive ramps (gray/blue/red scales, not for direct component use), semantic tokens (the public API components consume: text, surfaces, borders, status families, focus, links), and component tokens (per-component values seeded from the semantic layer).

The monolithic `css/base/variables.pcss.css` is split into category files under `css/_variables/`: `variables-color.pcss.css`, `variables-color-accent.pcss.css`, `variables-spacing.pcss.css`, `variables-typography.pcss.css`, `variables-misc.pcss.css`, and `variables-components.pcss.css`. The color token families are `--admin-color-gray-*`, `--admin-color-blue-*`, and `--admin-color-red-*` primitive ramps; semantic roles like `--admin-color-text`, `--admin-color-bg-surface`, `--admin-color-border`, `--admin-color-error`, `--admin-color-focus`; and accent tokens generated via `color-mix(in lch)` from `--accent-base`. Spacing is `--admin-space-{3xs…3xl}` plus a density-aware `--admin-space-density-*` scale controlled by `[data-gin-layout-density]`. Layout geometry is namespaced to `--admin-theme-*` to avoid collisions with core navigation's `--admin-toolbar-*` tokens.

Dark mode is now expressed with a single `light-dark()` declaration per token instead of a separate `.gin--dark-mode` override block. The accent ramp scoped to `[data-gin-accent]` also uses `light-dark()`, eliminating the former `.gin--dark-mode [data-gin-accent]` block entirely. This is what fixed the dark-mode surface bugs: surfaces that previously held a hardcoded white now resolve through `light-dark(white, #2a2a2d)` or similar pairs. The `color-scheme` property drives which half resolves. Component stylesheets were rerouted off raw primitive values and onto the semantic tokens. Dead custom properties were pruned; cross-module contracts were retained.

## Upgrade

Any custom CSS in contrib modules, sub-themes, or site-level stylesheets that references `--gin-*` CSS custom properties must be updated to the new `--admin-*` equivalents. There are no deprecation aliases; the old names are simply gone. Common migrations:

```css
/* OLD → NEW */
--gin-color-primary         → --admin-color-primary
--gin-bg-app                → --admin-color-bg          (defined in variables-color-accent.pcss.css)
--gin-border-s              → --admin-radius-xs
--gin-border-m              → --admin-radius-s
--gin-spacing-xs            → --admin-space-xs
--gin-spacing-s             → --admin-space-s
--gin-spacing-m             → --admin-space-m
--gin-spacing-density-m     → --admin-space-density-m
--gin-font-size-xs          → --admin-font-size-xs
--gin-transition-fast       → --admin-transition        (was misnamed; 0.3s is now --admin-transition-slow)
```

The source-of-truth definitions are in `core/themes/default_admin/css/_variables/*.pcss.css`. Always edit the `.pcss.css` sources, not the compiled `.css` files.
