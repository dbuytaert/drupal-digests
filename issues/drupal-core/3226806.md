## Summary

Nine procedural functions in `filter.module` that backed the core text filter plugins have been deprecated in Drupal 11.4.0 and will be removed in Drupal 13.0.0. Their logic now lives directly inside the corresponding filter plugin classes. Sites and modules that call these underscore-prefixed functions directly will trigger deprecation notices and must migrate before the Drupal 13 removal date. No direct replacement is provided: callers should instead instantiate the relevant plugin through the filter plugin manager.

## Impact

Affects module maintainers.

- **Deprecation:** Nine procedural functions in filter.module (`_filter_url`, `_filter_url_parse_full_links`, `_filter_url_parse_email_links`, `_filter_url_parse_partial_links`, `_filter_url_escape_comments`, `_filter_url_trim`, `_filter_autop`, `_filter_html_escape`, `_filter_html_image_secure_process`) deprecated in drupal:11.4.0, removed in drupal:13.0.0; use the filter plugin manager instead.

## Technical details

The filter system originally converted its filters to plugins ([#1868772](https://www.drupal.org/node/1868772)), but the plugin `process()` methods were thin wrappers that delegated to procedural functions still living in `filter.module`. This change reverses that: the full implementation is now in each plugin class, and the procedural functions become one-line shims that trigger a deprecation notice and delegate back to the plugin via `\Drupal::service('plugin.manager.filter')->createInstance(...)->process(...)`.

The nine deprecated functions and their new homes are: `_filter_url()` and its four sub-callbacks (`_filter_url_parse_full_links`, `_filter_url_parse_email_links`, `_filter_url_parse_partial_links`, `_filter_url_escape_comments`, `_filter_url_trim`) moved into `FilterUrl`; `_filter_autop()` moved into `FilterAutoP::process()`; `_filter_html_escape()` moved into `FilterHtmlEscape::process()`; and `_filter_html_image_secure_process()` moved into `FilterHtmlImageSecure::process()`.

`FilterUrl` and `FilterHtmlImageSecure` now implement `ContainerFactoryPluginInterface` and receive their dependencies through constructor injection. `FilterUrl` receives the `filter_protocols` container parameter via the `#[Autowire(param: 'filter_protocols')]` attribute and stores HTML comment state in `$this->htmlComments` (replacing the static variable in the old `_filter_url_escape_comments()`). The old preg callback tasks that referenced procedural function names by string are replaced with `[static::class . '::methodName', $pattern]` pairs using the new protected methods `parseFullLinks()`, `parseEmailLinks()`, `parsePartialLinks()`, `escapeComments()`, `unescapeComments()`, and `trimUrl()`. `FilterHtmlImageSecure` injects `FileUrlGeneratorInterface`, `ModuleHandlerInterface`, and the `app.root` parameter, replacing the static `\Drupal::service()` calls.

## Upgrade

Callers of the deprecated functions should instantiate the plugin instead:

```php
// Before (deprecated)
$output = _filter_autop($text);

// After
$output = \Drupal::service('plugin.manager.filter')
  ->createInstance('filter_autop')
  ->process($text, \Drupal::languageManager()->getCurrentLanguage()->getId())
  ->getProcessedText();
```

The equivalent plugin IDs for each deprecated function are:
- `_filter_url()` → `filter_url`
- `_filter_autop()` → `filter_autop`
- `_filter_html_escape()` → `filter_html_escape`
- `_filter_html_image_secure_process()` → `filter_html_image_secure`

The sub-callbacks (`_filter_url_parse_full_links`, `_filter_url_parse_email_links`, `_filter_url_parse_partial_links`, `_filter_url_escape_comments`, `_filter_url_trim`) are now protected methods of `FilterUrl` and have no public replacement. Code that called them directly should be refactored to call `FilterUrl::process()` via the plugin manager.

### Collaboration

The issue sat dormant for two years after longwave opened it in 2021. A key design question was whether to expose the logic as public static methods or require callers to go through the plugin manager. In a Slack discussion, longwave and joachim settled on the plugin manager approach, since the functions were documented as private even in Drupal 5/6/7 (the underscore prefix convention). claudiu.cristea later reopened the work by creating a fresh MR after the original branch could not be rebased cleanly. joachim pressed for the deprecated functions to delegate to the plugin rather than duplicate the logic, to avoid a maintenance burden during the deprecation window.
