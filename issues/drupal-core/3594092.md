## Summary

`ContentEntityStorageBase::loadUnchanged()` could return an entity with unsaved in-memory modifications instead of the persisted state when a module such as Workspaces swaps in a non-default revision via `hook_entity_preload()`. This is now fixed so the method always reflects the saved state of the entity.

## Impact

No upgrade or configuration changes required.

## Technical details

`loadUnchanged()` clears the entity-keyed static cache via `parent::resetCache()`, but a second static cache stores revisions keyed by `{entityTypeId}:{id}:revisions`. When `hook_entity_preload()` returns a non-default revision, the reload path goes through `loadMultipleRevisions()` to `getFromStaticRevisionCache()`, which still held the in-memory-modified object.

The fix adds a call to `$this->memoryCache->invalidateTags([$revision_cache_tag])` inside `loadUnchanged()`, using the tag `"{$this->entityTypeId}:{$id}:revisions"`, guarded by a check for `isStaticallyCacheable() && isRevisionable()`. This evicts the revision from the memory cache without touching the persistent cache, so the subsequent reload still benefits from the fast persistent-cache path rather than hitting storage unnecessarily. A new assertion in `ContentEntityStorageBaseTest::testNonDefaultRevision()` covers the scenario directly.
