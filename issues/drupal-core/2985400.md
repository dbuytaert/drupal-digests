## Summary

When code called `Url::createFromRequest()` to build a URL object from a Symfony `Request`, the resulting URL silently dropped any query parameters that were present on the original request. This meant that anything relying on that URL object would generate links without those query string values. The fix makes `createFromRequest()` preserve query parameters from the request, so the returned `Url` object faithfully represents the full original URL including its query string.

## Impact

Affects module maintainers.

- **API change:** `Url::createFromRequest()` now includes the request's query parameters in the returned `Url` object; callers that previously worked around the missing query by setting it manually should remove that workaround.

## Technical details

`Url::createFromRequest()` in `core/lib/Drupal/Core/Url.php` matched the request through the `router.no_access_checks` service to obtain the route name and raw route parameters, but then constructed the new `Url` instance with only those two values, discarding everything in `$request->query`. The fix reads all query parameters via `$request->query->all()` and passes them as the `query` key in the `$options` array to `new static($route_name, $route_parameters, $options)`.

A side effect of always populating the `query` option (even as an empty array when the request has no query string) required updating existing URL generation test fixtures in `core/tests/Drupal/Tests/Core/UrlTest.php`: entries in the URL map that previously used a bare `[]` for options now use `['query' => []]` to match the new consistent shape. A new parameterized test `testUrlFromRequestWithQueryParameters` was added alongside a data provider `providerUrlFromRequestWithQueryParameters`, covering single parameters, multiple parameters, nested array parameters, and an empty parameter set. The three leading-backslash corrections to `@throws` docblock entries (`\Symfony\Component\Routing\Exception\*`) were a minor PHPDoc cleanup bundled in the same commit.

The release managers (larowlan and catch) discussed whether the behavior change warranted a `$withQuery = FALSE` flag to preserve backward compatibility, and concluded the straightforward fix was appropriate for a minor release with a change record.
