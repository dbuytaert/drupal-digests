## Summary

The `StringFormatter` field formatter now supports linking a string field to an entity's edit form, in addition to the existing option to link to its canonical (view) page. When "Link to the entity" is enabled, a new radio group appears letting editors choose between "Link to view" and "Link to edit." If the entity type has only one link template, the radio group stays hidden and the formatter behaves exactly as before. This feature is aimed at CMS-style admin views where it is common to link titles directly to the edit form rather than the view page.

## Impact

Affects site owners.

- **Config change:** New optional `link_rel` key added to `field.formatter.settings.string` config schema; defaults to `canonical` when absent so existing config exports do not require an update path.

## Technical details

Prior to this change, `StringFormatter::viewElements()` and `getEntityUrl()` hardcoded `canonical` (falling back to `revision` for revisioned entities) as the link relationship. There was no way for site builders to choose the link destination from the UI.

The fix introduces a new `link_rel` formatter setting (string, defaults to `'canonical'`) in `StringFormatter::defaultSettings()`. The setting is persisted in config and declared in `core/config/schema/core.entity.schema.yml` under `field.formatter.settings.string` with a `Choice` constraint allowing `canonical` or `edit-form`, and `requiredKey: false` so existing stored config without the key continues to work without an upgrade path.

In `settingsForm()`, a `link_rel` radios element is added beneath the existing `link_to_entity` checkbox. It uses `#states` to only appear when `link_to_entity` is checked. The element's `#access` is set to `FALSE` by default and flipped to `TRUE` only when the entity type has both a `canonical` and an `edit-form` link template, preventing a single-option radio from appearing for entity types that support only one link destination.

In `viewElements()`, the hardcoded `hasLinkTemplate('canonical')` guard is replaced with `hasLinkTemplate($this->getSetting('link_rel'))`, so rendering respects the configured destination.

`getEntityUrl()` is updated so the `revision` fallback only applies when `link_rel` is `canonical`; when it is `edit-form`, the entity is linked directly to its edit-form route.

`settingsSummary()` now uses a `match` expression on `link_rel` with a `default` branch that renders the raw link template name. The `default` branch was added specifically to allow third-party modules to extend the formatter via `hook_config_schema_alter()` and form alters without triggering an `UnhandledMatchError`.

The design decision to add `link_rel` as a separate new key (rather than converting `link_to_entity` into a nullable string) was made by phenaproxima in discussion with catch, explicitly to avoid a complex upgrade path. A follow-up issue [#3584708](https://www.drupal.org/node/3584708) was opened to address how the new setting interacts with the media module's `standalone_url` setting, where both `canonical` and `edit-form` may resolve to the same route.
