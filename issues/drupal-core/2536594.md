## Summary

Drupal now provides a `FilterFormatRepositoryInterface` service for loading filter formats. The service replaces six procedural functions with object-oriented methods. The underlying behavior is unchanged: filter formats are loaded and cached the same way, but the code is now organized as a service that can be dependency-injected.

## Impact

Affects module maintainers.

- **API change:** Constructor signatures changed for `FilterFormatListBuilder`, `Text` views area plugin, and `InputRequired` views exposed form plugin (new optional parameter with BC layer).
- **Deprecation:** Six procedural functions deprecated: `filter_formats()`, `filter_formats_reset()`, `filter_get_roles_by_format()`, `filter_get_formats_by_role()`, `filter_default_format()`, and `filter_fallback_format()`.
- **New API:** New `Drupal\filter\FilterFormatRepositoryInterface` service (`filter.format_repository`) with methods `getAllFormats()`, `getFormatsForAccount()`, `getFormatsByRole()`, `getDefaultFormat()`, `getFallbackFormatId()`, and `resetFormats()`. New `FilterFormatInterface::getRoles()` method.

## Technical details

The filter module had six procedural functions for loading filter formats: `filter_formats()`, `filter_formats_reset()`, `filter_get_roles_by_format()`, `filter_get_formats_by_role()`, `filter_default_format()`, and `filter_fallback_format()`. These functions relied on `drupal_static()` and manual cache management.

The new `FilterFormatRepository` class implements `FilterFormatRepositoryInterface` and provides methods corresponding to five of the six functions. The `filter_formats_reset()` function is replaced by a `resetFormats()` method on the service. The service uses a chained cache backend that combines memory and persistent cache, maintaining the same caching strategy as the procedural code. Filter formats are cached per language, and per-account format lists are cached in memory only.

The repository depends on entity type manager, config factory, current user, and language manager services. It loads formats via `loadByProperties(['status' => TRUE])` from entity storage and sorts them by weight. The service is registered with the ID `filter.format_repository`.

`filter_formats()` previously changed behavior drastically based on whether an account parameter was passed: with no account it returned all formats, with an account it returned only those the account could access. The repository splits this into two methods: `getAllFormats()` and `getFormatsForAccount()`.

A new `FilterFormatInterface::getRoles()` method was added, moving logic from the procedural `filter_get_roles_by_format()` into the entity class. The `FilterFormat` entity annotation now includes `static_cache: TRUE`.

Three classes received constructor changes with backward compatibility: `FilterFormatListBuilder`, the views `Text` area plugin, and the views `InputRequired` exposed form plugin. Each triggers a deprecation error when instantiated without the new parameter.

A deprecation notice was also added to `drupal_static_reset()` when called with `'filter_formats'` as the argument, directing callers to `FilterFormatRepositoryInterface::resetFormats()` instead.

## Upgrade

Replace calls to deprecated functions:

```php
// Before
$formats = filter_formats();
$user_formats = filter_formats($account);
filter_formats_reset();
$roles = filter_get_roles_by_format($format);
$formats = filter_get_formats_by_role($role_id);
$default_format_id = filter_default_format($account);
$fallback_id = filter_fallback_format();

// After
$repository = \Drupal::service('filter.format_repository');
$formats = $repository->getAllFormats();
$user_formats = $repository->getFormatsForAccount($account);
$repository->resetFormats();
$roles = $format->getRoles();
$formats = $repository->getFormatsByRole($role_id);
$default_format_id = $repository->getDefaultFormat($account)->id();
$fallback_id = $repository->getFallbackFormatId();
```

Classes extending `FilterFormatListBuilder`, `\Drupal\views\Plugin\views\area\Text`, or `\Drupal\views\Plugin\views\exposed_form\InputRequired` should add the new `FilterFormatRepositoryInterface` parameter to their constructors.

### Collaboration

The issue was opened in July 2015 and took nearly eleven years to land. The most substantive technical debate centered on caching strategy. After a patch (comment #54) removed the custom cache and delegated entirely to entity storage, berdir raised performance concerns in a detailed review: `loadByProperties()` is not cached and issues a real database query every call, and `loadMultiple()` on config entity storage still triggers an extra query. The decision to restore the original caching approach (a chained memory and persistent cache) was explained in the thread: keeping the existing level of caching avoids needing to measure and justify a performance regression, and the introduction of cache tags makes future removal easier if desired. A follow-up was suggested to explore general persistent caching for config entity wildcard queries, which would allow removing this custom cache.
