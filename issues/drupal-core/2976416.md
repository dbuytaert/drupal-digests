## Summary

Two methods for configuring default values on base field definitions now have complete documentation. Previously, `BaseFieldDefinition::setDefaultValue()` and `BaseFieldDefinition::setDefaultValueCallback()` used placeholder documentation that referenced nonexistent interface methods, making it difficult for developers to understand how to use them properly. The updated documentation explains the accepted formats for default values, the precedence of callbacks over static values, and the callback signature, helping module developers set up entity base fields correctly.

## Impact

No upgrade or configuration changes required.

## Technical details

The `BaseFieldDefinition` class implements `FieldDefinitionInterface`, which is intentionally read-only and contains no setter methods. This design reflects that base field definitions are static after discovery, unlike `FieldConfig` entities which can be modified and saved. The two methods in question inherited `{@inheritdoc}` tags that pointed to no actual parent, leaving them undocumented.

The fix adds explicit docblocks explaining the accepted value formats for `setDefaultValue()`: literals (assigned to the first property of the first item), numerically indexed arrays of items, non-numerically indexed property/value arrays, or NULL/empty array. It documents that `setDefaultValueCallback()` takes precedence when both are set, and specifies the callback signature with two parameters (the entity and field definition) returning a value in the same format.

The type hint for the callback parameter was corrected from `string` to `callable`, aligning with PHPStan documentation standards adopted in core. This reflects that while the current implementation accepts string representations of functions, the semantic meaning is that it's a callable. PHPStan baseline entries for missing return type annotations were also removed.

## Lessons learned

- When a parameter accepts a string that represents a callable (function name, class method, or service notation), document it as 'callable' rather than 'string'. This conveys semantic intent and leverages IDE and static analysis tooling that recognizes the callable type. *(via oily, xjm)*
- When read-only interfaces describe objects that have mutable implementations, setter methods belong on the concrete class without requiring interface changes. Document these methods explicitly rather than using inheritdoc or forcing setters onto interfaces that semantically represent immutable contracts. *(via berdir)*
