## Summary

Two methods for configuring default values on base field definitions now have complete documentation. Previously, `BaseFieldDefinition::setDefaultValue()` and `BaseFieldDefinition::setDefaultValueCallback()` used placeholder `{@inheritdoc}` docblocks that pointed to no real parent, making it difficult for developers to understand how to use them correctly. The updated documentation explains the accepted formats for default values, the precedence of callbacks over static values, and the callback signature, helping module developers set up entity base fields correctly.

## Impact

No upgrade or configuration changes required.

## Technical details

The `BaseFieldDefinition` class implements `FieldDefinitionInterface`, `FieldStorageDefinitionInterface`, and `RequiredFieldStorageDefinitionInterface`, but none of these interfaces define `setDefaultValue()` or `setDefaultValueCallback()`. As berdir explained in the issue, `FieldDefinitionInterface` is intentionally read-only; only specific concrete implementations like `BaseFieldDefinition` or `FieldConfig` expose setters. The two methods therefore had `{@inheritdoc}` tags that pointed to nothing, leaving them effectively undocumented.

The fix replaces those placeholders with explicit docblocks in `core/lib/Drupal/Core/Field/BaseFieldDefinition.php`. For `setDefaultValue()`, the new docs enumerate all accepted formats for `$value`: a literal (assigned to the first property of the first item), a numerically indexed array of items, a non-numerically indexed property/value array (treated as the first item), or `NULL`/`[]` for no default value. A `@see` cross-reference to `setDefaultValueCallback()` is added, and a `@return $this` annotation documents the fluent interface. For `setDefaultValueCallback()`, the new docs describe the two callback arguments (`FieldableEntityInterface $entity` and `FieldDefinitionInterface $definition`), state that `NULL` unsets a previously set callback, note that the callback takes precedence over any static default value, and document the return format as matching `setDefaultValue()`. The `@param` type annotation is documented as `callable|null` rather than `string`, reflecting that the semantic meaning is a callable, in line with the PHPStan doc types adopted as core's canonical reference (see coding standards issue [#3468236](https://www.drupal.org/node/3468236)). The runtime validation in the method body (`!is_string($callback)`) was not changed.

Two PHPStan baseline entries in `core/.phpstan-baseline.php` suppressing missing-return-type warnings for both methods were also removed, since the new `@return $this` annotations satisfy the analyser.
