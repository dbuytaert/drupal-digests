## Summary

Running `drush cr` (cache rebuild) on Drupal 10 sites throws a "Permission denied" error when Drupal tries to delete old aggregated CSS or JS files. The error appears as "Failed to unlink file 'assets://js/...'". This happens because the web server process (for example, running as `www-data`) creates the aggregated asset files and owns them, while the user running Drush (a deployment script or developer account) is a different OS user that lacks permission to delete those files. The problem became more visible after Drupal 10.1.0 introduced a new asset aggregation system. No code change was committed in this issue; it was closed as a support request. The underlying bug is tracked in [#3382589](https://www.drupal.org/node/3382589). Practical workarounds are: add the Drush-running user to the web server group (for example, `www-data`), correct the ownership of `sites/default/files/css` and `sites/default/files/js` to `user:www-data`, or temporarily disable CSS and JS preprocessing.

## Impact

No upgrade or configuration changes required.

## Technical details

The failure surfaces in `FileSystem::unlink()` (`core/lib/Drupal/Core/File/FileSystem.php`), which calls PHP's native `unlink()` on the stream-wrapper-resolved path. When the aggregated asset file was written by the web server user and the OS-level file mode is `0664` (group-writable but not world-writable), a different user running Drush cannot remove the file unless that user shares the owning group.

The error message `FileSystem.php:124` referenced in the report corresponds to the `delete()` method's internal call to `unlink()`, which emits a PHP warning and then throws a `FileException` on failure.

Drupal 10.1.0 rewrote the asset aggregation pipeline. Under the new system, aggregated files are written directly to `assets://js/` and `assets://css/` (stream wrappers that resolve to `sites/default/files/js` and `sites/default/files/css`) during a page request served by the web server. If those directories or files end up owned by the web server user with mode `0664`, a separate Drush user cannot unlink them.

The related open bug report [#3382589](https://www.drupal.org/node/3382589) ("Directory permissions and ownership change after cache:rebuild") tracks the same root cause as a proper bug with steps to reproduce.

## Upgrade

No code changes were made in this issue. If you are hitting this error, the practical fixes are:

1. Add the user that runs Drush to the web server group:
```bash
sudo usermod -aG www-data deploy_user
```

2. Correct ownership of the aggregated asset directories so both users share the group:
```bash
sudo chown -R deploy_user:www-data sites/default/files/css sites/default/files/js
sudo chmod -R g+w sites/default/files/css sites/default/files/js
```

3. Or disable CSS/JS aggregation as a last resort (in `settings.php`):
```php
$config['system.performance']['css']['preprocess'] = FALSE;
$config['system.performance']['js']['preprocess'] = FALSE;
```
