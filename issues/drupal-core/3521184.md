## Summary

Drupal's email validator now rejects email addresses that contain a space immediately before the `@` sign (for example `mmbk @example.org`). Such addresses are technically RFC-compliant but deprecated, and most mail servers refuse to deliver them, causing silent send failures. The fix makes Drupal refuse these addresses at the form-validation stage so users get a clear error instead of a failed delivery.

## Impact

Affects site owners.

- **Config change:** The `drush site-install` command's default seed email addresses changed from `@localhost` to `@example.com` in `InstallCommand.php`

## Technical details

Drupal wraps the `egulias/email-validator` library in `Drupal\Component\Utility\EmailValidator`, which extends the library's `EmailValidator` class and overrides `isValid()`. Previously, `isValid()` delegated to `RFCValidation`, which accepts RFC-deprecated formats (such as folding whitespace and a space before `@`) and merely emits library-level warnings. The fix switches the delegation to `NoRFCWarningsValidation`, a stricter validator from the same library that promotes any RFC warning to a validation failure. The change is a one-line swap in `core/lib/Drupal/Component/Utility/EmailValidator.php`. A corresponding one-line assertion was added to `core/tests/Drupal/Tests/Component/Utility/EmailValidatorTest.php` to cover the `example @example.com` case, and `EmailValidatorTest::testIsValidException()` was updated to pass `NoRFCWarningsValidation` instead of `RFCValidation` to the exception-path test. A separate cleanup in `core/lib/Drupal/Core/Command/InstallCommand.php` changed the hardcoded seed email addresses used by the `drush site-install` command from `drupal@localhost` and `admin@localhost` to `drupal@example.com` and `admin@example.com`, because `localhost` is now invalid under the stricter validator. Related issue [#3061074](https://www.drupal.org/node/3061074) previously addressed a similar problem where spaces in the domain part were accepted due to an older library bug.

## Upgrade

If any custom or contributed code passes `new RFCValidation()` as the second argument to `\Drupal\Component\Utility\EmailValidator::isValid()`, note that this has always thrown a `\BadMethodCallException` (that contract did not change). No caller API change is required. However, if your site deliberately accepts RFC-deprecated email formats (quoted strings, folding whitespace, space before `@`) as valid, they will now be rejected by core validation.
