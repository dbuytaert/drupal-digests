## Summary

Screen reader users navigating the Drupal admin sidebar had no reliable way to understand the structure of the Navigation module's toolbar. Block section headings were `<h2>` elements, conflicting with main-content heading levels, and the `<nav>` landmark had no proper heading to name it. This fix adds a visually-hidden `<h2>` "Administrative sidebar" heading inside the nav and links it with `aria-labelledby`, then demotes all individual navigation block titles from `<h2>` to `<h3>` so there is a clear, consecutive heading hierarchy for screen reader users.

## Impact

No upgrade or configuration changes required.

## Technical details

The Navigation module renders the admin sidebar in two Twig templates. Before this fix, `navigation.html.twig` gave the `<nav>` element a plain `aria-label="Administrative sidebar"` string, and `block--navigation.html.twig` used `<h2>` for every navigation block title, placing them at the same level as main-content headings.

The fix makes two targeted changes:

1. In `core/modules/navigation/layouts/navigation.html.twig`, a unique ID is generated with `random()` (chosen over `clean_unique_id` because that Twig filter can be unreliable with caching), assigned to a new `<h2 class="visually-hidden">{{ 'Administrative sidebar'|t }}</h2>` element placed inside the `<nav>`. The `<nav>` is changed from `aria-label` to `aria-labelledby="{{ nav_id }}"`, linking the landmark to its real heading. This pattern follows WCAG guidance for labelling navigation landmarks with a heading so that the heading text is available to both screen reader users navigating by heading and by landmark.

2. In `core/modules/navigation/templates/block--navigation.html.twig`, both the visible and the visually-hidden `<h2>` block title variants are changed to `<h3>`, creating the hierarchy: `<h2>` administrative sidebar, then `<h3>` per block section. This matches the proposed resolution discussed among `mherchel`, `mgifford`, `rkoller`, and `kat-shaw`.

The change only affects admin markup and was backported to both `11.x` and `11.3.x` as an eligible accessibility fix.

### Collaboration

The issue was opened by `rkoller`, who documented the exact screen reader announcement sequence experienced through VoiceOver's rotor. `katannshaw` identified the three empty, focusable `<h4>` elements in a code comment and proposed giving them text content and dropping the `.focusable` CSS class. A team at the DrupalCon Portland 2024 mentored contribution sprint took it on but ran out of time after consulting `rkoller` directly via Slack to reproduce the issue correctly. Nearly two years later, `mherchel` drove the fix to completion after a Zoom call with `mgifford`, `rkoller`, and `kat-shaw`, settling the final approach: a visually-hidden `<h2>` for the entire sidebar linked via `aria-labelledby`, with all block titles demoted to `<h3>`. `rkoller` tested the result with VoiceOver's rotor and a screen-reader bookmarklet and confirmed it resolved the issue.
