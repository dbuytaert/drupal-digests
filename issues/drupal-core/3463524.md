## Summary

When upgrading from Drupal 10.2.x to 10.3.x or later, sites could fail with a fatal error during database updates if an editor configuration entity referenced a text format that no longer existed. This happened when modules like Webform were uninstalled but left orphaned editor configurations behind. The update process would crash with "Call to a member function getConfigDependencyName() on null," blocking the upgrade. This fix adds defensive checks so the update completes with a warning instead of failing, letting site owners clean up the orphaned configuration later.

## Impact

No upgrade or configuration changes required.

## Technical details

The `editor_post_update_sanitize_image_upload_settings()` post-update hook failed when an `Editor` entity's `format` property pointed to a non-existent `FilterFormat` entity. This happened in `Editor::calculateDependencies()` when calling `$this->getFilterFormat()->getConfigDependencyName()` - if `getFilterFormat()` returned `NULL`, the chained method call triggered a fatal error.

The fix adds a null check in `Editor::calculateDependencies()`. When `getFilterFormat()` returns `NULL`, the code now skips adding the config dependency and emits a `E_USER_WARNING` via `trigger_error()` with a message identifying the orphaned editor configuration. The warning message tells site owners which editor entity is misconfigured and suggests reviewing whether it's still needed.

This scenario occurred when modules like Webform created text formats and editor configurations, then were uninstalled. The text format would be deleted but the editor configuration could remain. The new code handles this gracefully so updates can complete.

A unit test was added to `EditorConfigEntityUnitTest` to verify that when `getFilterFormat()` returns `NULL`, the warning is triggered and no config dependency is added.

### Collaboration

This issue affected multiple sites upgrading to 10.3.x, with the reporter encountering it on three separate instances. The root cause was identified through collaboration between the reporter and maintainers who recognized the pattern from similar issues. The reporter helped diagnose by discovering that orphaned Webform editor configurations were the trigger. The fix was committed to Drupal 10.6.x, 11.x, 11.3.x, and the development branch. A follow-up issue was opened to add additional test coverage for newer branches where PHPUnit's warning expectation API changed.

## Lessons learned

- When config entities reference other entities, the referenced entity may no longer exist (e.g., after module uninstallation). Code calculating dependencies should check for null and handle the missing dependency gracefully with logging rather than fataling, preventing upgrade path failures. *(via catch, alexpott)*
- When a fatal error occurs during database updates on a stable release, bump the issue to critical priority since the upgrade path affects all users updating to that version, especially when triggered by common scenarios like installing and uninstalling popular contributed modules. *(via catch)*
