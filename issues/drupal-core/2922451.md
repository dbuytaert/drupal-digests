# #2922451: [policy no patch] Make it possible to mark plugins as deprecated

## Summary

This issue established the official Drupal policy for how to deprecate a plugin. Before this, developers marking plugin classes with a PHP `@deprecated` docblock caused problems: plugin discovery still instantiates annotated classes at various points (e.g. when rendering a configuration form listing all available plugins), so the deprecation notice would fire on every page that listed plugins, not just when the plugin was genuinely used. The agreed policy is to place `@trigger_error()` in the plugin constructor rather than relying on a class-level deprecation. This ensures the notice fires only when the plugin is actually instantiated, not during discovery or listing operations. The documentation at `https://www.drupal.org/core/deprecation#how-plugin` was updated with a canonical example.

## Impact

No upgrade or configuration changes required.

## Technical details

The core problem was that PHP's process-isolated PHPUnit bridge (landed in [#2870194](https://www.drupal.org/node/2870194)) turns `E_USER_DEPRECATED` into test failures. Because the action module's configuration form iterates `getDefinitions()` and calls `is_subclass_of()` on each class, any plugin class annotated with `@deprecated` at the class level would immediately trigger a failure during discovery-related operations, even in tests that were not testing the deprecated plugin at all.

The policy that was agreed and documented has three parts:

1. Add `@trigger_error('ClassName is deprecated in drupal:X.Y.Z ...', E_USER_DEPRECATED)` as the first statement in the plugin's constructor. If the plugin has no constructor, add one. Placing it in the constructor means it fires only on actual instantiation (`createInstance()`), not during `getDefinitions()` or any other reflection-based discovery call.

2. Add `no_ui = true` to the plugin annotation definition so the deprecated plugin is excluded from UIs that list plugins for selection. The thread noted that `no_ui` must be used on a case-by-case basis: some UIs need to keep showing a plugin that already has active configuration even when new instances should not be created.

3. Add an `@deprecated` and `@see` phpdoc annotation to the plugin class docblock so that static analysis tools (IDEs, PHPStan) can also surface the deprecation.

A second option discussed in the thread, putting the trigger in `PluginManagerBase::createInstance()` or `DefaultPluginFactory` keyed on a plugin annotation property, was considered but not adopted for this policy. The constructor approach was preferred because it parallels how regular classes and methods are deprecated and works regardless of which factory class is in use. A follow-up issue ([#3039240](https://www.drupal.org/node/3039240)) tracks adding a formal `deprecated` annotation key to the plugin system for YAML-defined plugins, which cannot use the constructor approach cleanly.

## Upgrade

To deprecate a plugin, follow this pattern:

```php
/**
 * Plugin implementation of the 'my_old_plugin' type.
 *
 * @deprecated in drupal:10.1.0 and is removed from drupal:11.0.0. Use
 *   \Drupal\mymodule\Plugin\MyType\NewPlugin instead.
 * @see https://www.drupal.org/node/XXXXXXX
 *
 * @MyPluginAnnotation(
 *   id = "my_old_plugin",
 *   label = @Translation("My old plugin (deprecated)"),
 *   no_ui = true
 * )
 */
class MyOldPlugin extends PluginBase {

  /**
   * Constructs a MyOldPlugin object.
   */
  public function __construct(array $configuration, $plugin_id, $plugin_definition) {
    @trigger_error('\Drupal\mymodule\Plugin\MyType\MyOldPlugin is deprecated in drupal:10.1.0 and is removed from drupal:11.0.0. Use \Drupal\mymodule\Plugin\MyType\NewPlugin instead. See https://www.drupal.org/node/XXXXXXX', E_USER_DEPRECATED);
    parent::__construct($configuration, $plugin_id, $plugin_definition);
  }

}
```

Key points: place `@trigger_error()` before `parent::__construct()`, add `no_ui = true` to the annotation, and add `@deprecated` and `@see` to the class docblock. Whether `no_ui` effectively hides the plugin in a given UI depends on that UI's implementation; verify on a case-by-case basis.

### Collaboration

The thread involved a substantive disagreement between EclipseGc and the majority of core maintainers (alexpott, catch, tim.plunkett, berdir). EclipseGc argued that the correct place for the deprecation trigger is `DefaultPluginManager::createInstance()` (or the factory layer), keyed on an introspectable annotation property, because the plugin system is designed around factories and not manual construction. His concern was that the constructor approach cannot support filtering deprecated plugins from UIs and cannot propagate deprecation knowledge through derivatives. The counter-argument, articulated by catch and alexpott, was that putting it in the constructor closely mirrors how regular PHP classes are deprecated, works across all factory implementations (not just `DefaultFactory`), and fires correctly at the point of actual use. The constructor approach was chosen, with the note that UI filtering via `no_ui` should be handled case-by-case and that a formal annotation key for more robust support remains an open follow-up ([#3039240](https://www.drupal.org/node/3039240)).
