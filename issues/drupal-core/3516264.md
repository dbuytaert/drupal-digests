## Summary

When loading CKEditor 5 with multiple text formats on the same page through AJAX operations, Drupal could incorrectly include translation files for all languages instead of just the active interface language. This caused hundreds of unnecessary JavaScript files to be requested, potentially triggering "too many requests" errors from web servers or reverse proxies when JavaScript aggregation was disabled. The fix ensures that translation filtering works correctly during AJAX requests, even when the editor library was already loaded by the initial page request.

## Impact

No upgrade or configuration changes required.

## Technical details

The bug occurred in `Ckeditor5Hooks::jsAlter()`, which filters CKEditor 5 translation libraries to include only the current language. The method relied on a placeholder file (`core/assets/vendor/ckeditor5/translation.js`) being present in the `$javascript` array to trigger filtering logic.

During AJAX operations loading a new text format, the `core/ckeditor5.translations` library was already in the list of loaded libraries, so the placeholder script was excluded from the `$javascript` array. This prevented the filtering logic from running, causing all translation libraries to be included in the response.

The fix checks whether `core/ckeditor5.translations` is present in either the currently loaded libraries or already-loaded libraries by calling `LibraryDependencyResolverInterface::getLibrariesWithDependencies()`. This ensures filtering happens whether or not the placeholder file is present. The method also now accepts `ModuleHandlerInterface` and `LibraryDependencyResolverInterface` as constructor dependencies instead of using `\Drupal::moduleHandler()`.

A new unit test in `CKEditor5Test::testJsAlterHook()` verifies both scenarios: when the placeholder is present and when it has already been loaded. A performance test was added to `AssetAggregationAcrossPagesTest` to detect regressions.
