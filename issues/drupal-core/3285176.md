## Summary

Users who have the "View update notifications" permission (added in [#332796](https://www.drupal.org/node/332796)) but not "Administer site configuration" were shown Update module warning messages containing a link to `/admin/reports/updates`, a page they cannot access (resulting in a 404). The fix checks whether the current user can access that route before including the link. Users without access now see the same warning text without the link. Email notifications are unchanged: they continue to include the absolute URL without access checks, since they can be sent to arbitrary users who may not have a site account.

## Impact

No upgrade or configuration changes required.

## Technical details

The Update module generates warning messages in two contexts: UI pages (via `hook_page_top()` -> `UpdateHooks::pageTop()` -> `UpdateRequirements::requirementCheck()`) and email notifications (via `UpdateHooks::mail()`). Both paths called the shared internal helper `_update_message_text($msg_type, $msg_reason, $langcode)`.

The fix adds a fourth parameter `bool $is_email = FALSE` to `_update_message_text()`. When `$is_email` is `FALSE` (the default, used for UI), the function checks `$url->access()` before returning a linked version of the message. If the current user lacks access, it returns plain text without any link. When `$is_email` is `TRUE` (passed explicitly by `UpdateHooks::mail()`), the access check is skipped and URLs are generated as absolute. This distinction was necessary because email recipients are not necessarily the current user and may not have a site account at all.

Two code paths needed separate fixes:

1. **`_update_message_text()` (`update.module`)**: The `UNKNOWN`/`NOT_CHECKED`/`NOT_FETCHED`/`FETCH_PENDING` switch cases build a "There was a problem checking available updates" sentence. These cases now build a `Url` with `absolute` set from `$is_email`, then return the linked version only when `$url->access() || $is_email`, otherwise returning plain text.

2. **`UpdateRequirements::requirementCheck()` (`src/Hook/UpdateRequirements.php`)**: The trailing "See the available updates page for more information." sentence (appended for non-fetch-error statuses) is now wrapped in `if ($url->access())` so it is omitted entirely for users who cannot reach `update.status`.

The `Url::access()` approach was chosen over checking a specific permission directly, so the code stays correct if the route's access requirements ever change (per comment #11).

A new functional test `testAvailableUpdateLink()` in `UpdateSemverCoreTest` covers both cases: a user with `administer site configuration` sees the link; a user with only `view update notifications` sees the plain text but not the link.
