# #3285176: Warning message links to "available updates" even if user does not have permission for that page

## Summary

After Drupal added a "View update notifications" permission, users with that permission could see security update warnings that included a link to the Available Updates page, even when they lacked permission to view that page. Clicking the link resulted in an access denied error. This fix checks whether the current user can access the Available Updates report before including a link. Users without access now see the warning text without a broken link. Email notifications continue to include the link since they may be sent to users without accounts on the site.

## Impact

No upgrade or configuration changes required.

## Technical details

The Update module generates warning messages in two contexts: on-screen messages via `hook_page_top()` for the current user, and email notifications sent to arbitrary addresses. The `_update_message_text()` function now accepts a `$is_email` parameter that controls whether to perform access checks and use absolute URLs.

For on-screen messages, the code now calls `Url::access()` before including a link to the Available Updates report. If the current user cannot access `update.status` route, the message displays plain text instead of a hyperlink. The `UpdateRequirements::requirementCheck()` method applies similar logic when appending the "See the available updates page" sentence.

For email messages, the new parameter bypasses access checks since recipients may not have accounts on the site. Emails always include absolute URLs to the report. This preserves existing behavior for email notifications while fixing the user experience for users with limited permissions viewing on-screen warnings.

### Collaboration

This issue was opened in June 2022 after the Drupal Usability Meeting noticed the permission gap introduced by [#332796](https://www.drupal.org/node/332796). The fix required careful iteration over multiple code paths. Early patches checked specific permissions, but reviewers pushed for using `Url::access()` to future-proof the solution. The challenge was handling both on-screen messages (where access checks matter) and email notifications (where they don't). After the initial RTBC in July 2022, the issue went dormant when larowlan raised code style concerns about nested conditionals. It took until March 2026 for dww to convert it to a merge request and simplify the logic with early returns. The final version properly separates email from on-screen contexts.

## Lessons learned

- When generating messages that include links, check whether the current user has access to the link destination before including it. For UI messages shown to the current user, use URL access checks to conditionally format messages with or without links. For emails sent to arbitrary recipients where access cannot be determined, include links unconditionally since you cannot reliably check permissions for unknown users. *(via dww)*
