## Summary

The configuration system no longer allows bypassing schema validation and sorting when saving configuration. Previously, the "trusted data" concept let code skip validation for performance reasons during installation, but this created fragility where configuration could be saved in invalid states. This change removes that bypass, ensuring all configuration is consistently validated and sorted according to schema. Performance concerns during site installation are now addressed through multi-module installation optimizations instead.

## Impact

Affects module maintainers.

- **Deprecation:** The `$has_trusted_data` parameter of `Config::save()` and the `ConfigEntityInterface::trustData()` method are deprecated and will be removed in Drupal 13.0.0

## Technical details

The trusted data concept was introduced in [#2432791](https://www.drupal.org/node/2432791) to improve site installation performance by skipping repeated schema cache rebuilds. However, it caused issues like [#3346953](https://www.drupal.org/node/3346953) where Role permissions weren't sorted when saved via the permissions form, because the sorting was removed from the entity's `preSave()` method and moved to schema validation, which was skipped for trusted data.

This change deprecates `Config::save($has_trusted_data)` and `ConfigEntityInterface::trustData()`. All configuration saves now apply schema validation and sorting consistently. The `hasTrustedData()` method is not deprecated yet to ease upgrade paths for contrib.

The original performance concern around slow site installation is now mitigated by multi-module installation, which reduces schema rebuilds. Configuration import and cold cache performance are addressed separately. The change removes calls to `trustData()` throughout core, including in `ConfigInstaller`, module/theme installers, and various tests.

Configuration is now always normalized by schema on save, ensuring consistent sorting and type casting. Tests that needed to write invalid config now use database writes or a new test trait instead of trusted data.

## Upgrade

Remove any calls to `Config::save(TRUE)` and replace them with `Config::save()`. Remove any calls to `ConfigEntityInterface::trustData()`.

If you need to write invalid configuration in tests (for example, to test upgrade paths that fix schema violations), use direct database writes or the new `RawConfigWriterTrait`:

```php
use Drupal\Tests\Traits\Core\Config\RawConfigWriterTrait;

class MyTest extends KernelTestBase {
  use RawConfigWriterTrait;

  public function testUpgrade() {
    // Write config without schema validation.
    $this->writeRawConfig('my_module.settings', ['invalid' => 'data']);
  }
}
```

For config entities, use `writeRawConfigEntity()` instead.

See the change record at [#3348180](https://www.drupal.org/node/3348180) for full details.

## Lessons learned

- When configuration transformation logic (like sorting) exists in both entity code and schema definitions, trusted data paths bypass the entity code, creating dual code paths that are fragile and error-prone. Consolidate transformation logic in one layer—preferably schema—so all save operations produce consistent, canonical output regardless of the code path taken. *(via alexpott)*
- When optimizing configuration operations by skipping validation and transformation for 'trusted' sources, the optimization creates technical debt if data can enter the system through multiple paths. As systems mature and performance bottlenecks shift (e.g., multi-module install reducing schema rebuild frequency), removing the optimization simplifies the codebase even if some performance cost remains. *(via alexpott, catch)*
