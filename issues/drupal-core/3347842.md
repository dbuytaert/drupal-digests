## Summary

The configuration system previously allowed code to bypass schema validation by marking data as "trusted." This optimization was added years ago to speed up site installation, but it created maintenance problems. Code had to check whether data was trusted and handle sorting and validation differently depending on the answer. This made the system fragile and harder to reason about.

With this change, all configuration saves now apply schema validation and normalization. The trusted data parameter is deprecated but still accepted for backward compatibility. The system is now more consistent and predictable. Tests that need to write invalid configuration for upgrade path testing can use a new test trait instead of the deprecated API.

## Impact

No upgrade or configuration changes required.

## Technical details

The `$has_trusted_data` parameter on `Config::save()` and the `trustData()` method on config entities are now deprecated. Internally, the system still respects the flag when present to maintain backward compatibility, but all new code should assume schema validation always runs.

During saves, configuration is now normalized according to schema even when coming from module installation or configuration sync. This ensures data is properly sorted and typed. The original performance concern (repeated schema cache rebuilds during install) has been largely mitigated by multi-module installation improvements that reduce cache rebuilds.

The Views module benefited from a side effect: it no longer loads all action configuration entities just to check if any exist for a given entity type. The bulk form field plugin now registers for all entity types with a base table, not just those that already have actions defined. This reduces query counts on cold cache page loads.

A new `RawConfigWriterTrait` provides a testing API for upgrade tests that need to write configuration without schema validation. It writes directly to storage and invalidates caches, bypassing the entity system entirely.

The `hasTrustedData()` method is not yet deprecated because only three modules use it. It will be deprecated in Drupal 13 when it can only return FALSE, allowing a simpler upgrade path for contrib.

### Collaboration

This issue was opened by alexpott in March 2023 to address a code smell in the configuration system. The "trusted data" concept was originally introduced to improve performance during site installation, but it created fragility and forced workarounds in other parts of the codebase.

The change sat for nearly three years before serious work began. Once active development started in early 2026, it moved quickly through review. Multiple committers and reviewers (wim leers, andypost, bircher, borisson_, catch) contributed to refining the approach. A key decision emerged late in the process: rather than deprecate `hasTrustedData()` immediately, the team decided to wait until Drupal 13 after the ability to trust data was removed, making it easier for contrib modules to upgrade incrementally.

The MR revealed unexpected benefits: removing trusted data eliminated unnecessary action configuration loads in Views, reducing query counts in performance tests. The change was committed to main in March 2026 and backported to 11.x within hours.

## Lessons learned

- When configuration data is normalized and validated on save, duplication of normalization logic across entities becomes fragile. If some code paths skip normalization (via a 'trusted data' flag), each entity must handle normalization separately even when schema already defines the canonical form. Normalize consistently at one layer rather than conditionally at multiple layers. *(via alexpott)*
- When performance optimizations skip validation during bulk operations like installation or import, configuration can become denormalized over time as schemas evolve or relationships change. Even if data was normalized when saved, schema changes or new related entities can introduce noise. Validate and normalize on every save to maintain consistency, accepting the performance cost. *(via alexpott, donquixote)*
