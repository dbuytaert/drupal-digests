## Summary

Drupal's configuration system no longer supports "trusted data" mode, which previously allowed configuration to be saved without schema validation. This mode was originally added to improve installation performance by skipping repeated schema checks, but it created fragile code patterns where entities and configuration had to manually sort data when marked as trusted. The change removes this dual-path complexity, making the configuration system more predictable and easier to maintain. Installation performance is preserved through other optimizations introduced since the trusted data mode was created.

## Impact

No upgrade or configuration changes required.

## Technical details

The trusted data concept allowed calling `Config::save(TRUE)` or `ConfigEntityInterface::trustData()` to skip schema validation and normalization. This was introduced to reduce schema cache rebuilds during site installation, but created inconsistencies where some code paths applied schema sorting (like alphabetizing permissions) while others did not.

The removal involves deprecating `Config::save($has_trusted_data)` and `ConfigEntityInterface::trustData()`. All saves now apply schema validation and normalization consistently. The original performance problem is mitigated by multi-module install (introduced separately), which significantly reduced schema rebuilds during installation.

Throughout the codebase, calls like `$config->save(TRUE)` become `$config->save()`, and `$entity->trustData()->save()` becomes `$entity->save()`. The `ConfigEntityStorage::doSave()` method now always updates entity properties from the normalized config values stored after save. 

For testing scenarios that need to write invalid config, a new `RawConfigWriterTrait` provides `writeRawConfig()` and `writeRawConfigEntity()` methods that bypass the entity system entirely.

The `hasTrustedData()` method is deliberately not deprecated yet, deferred until Drupal 13 when the ability to mark data as trusted is fully removed. This eases the upgrade path for the few contrib modules using it.

## Upgrade

For tests that previously used `trustData()` to save invalid configuration for update path testing, use the new `RawConfigWriterTrait`. Include the trait in your test class and call `writeRawConfig($name, $data)` for simple config or `writeRawConfigEntity($entity)` for config entities. This bypasses schema validation without using the deprecated API.

```php
use Drupal\Tests\Traits\Core\Config\RawConfigWriterTrait;

class MyUpdateTest extends KernelTestBase {
  use RawConfigWriterTrait;

  public function testUpdate() {
    // Write invalid config for testing the update path.
    $this->writeRawConfig('mymodule.settings', [
      'invalid_key' => 'value',
    ]);
  }
}
```

In hook_update_N implementations, remove any calls to `trustData()`. The entity system now always normalizes data on save.

The `ConfigEntityInterface::hasTrustedData()` method remains available until Drupal 13, so contrib code calling it does not need immediate changes.

## Lessons learned

- When performance optimizations bypass normal data processing (like sorting or validation), they create fragility by requiring duplicate logic in multiple code paths. For example, if schema normally sorts configuration but trusted data skips schema processing, entities must sort their own data when marked trusted. This duplication makes the system fragile because changes in one place won't automatically apply to the other. Remove the bypass and accept the performance cost, or find optimizations that don't require skipping normal processing. *(via alexpott)*
- When deprecating an API that consumers check (like hasTrustedData()), delay deprecating the check method until the version after the setter/usage is removed. This gives contrib a single upgrade path: stop calling trustData() in version N, then remove hasTrustedData() checks in N+1 when it only returns false. Deprecating both simultaneously forces contributors to handle two changes at once. *(via alexpott, catch)*
