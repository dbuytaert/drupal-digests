## Summary

A bug in the media system caused mapped field values to be incorrectly overwritten when a media entity was first saved without a source field value and then saved again with one. If custom code or an import process set legitimate values in mapped fields before the source file was attached, those values would be silently replaced by whatever the media source plugin returned for the mapped metadata attributes. The fix tightens the overwrite condition so that a source-field change only triggers a metadata write when both the original and new source field values are non-empty.

## Impact

No upgrade or configuration changes required.

## Technical details

The logic lives in `Media::prepareSave()` in `core/modules/media/src/Entity/Media.php`. The previous condition was: overwrite a mapped field if the field is empty, or if `hasSourceFieldChanged()` returns true. The problem was that `hasSourceFieldChanged()` compares the original entity's source field value against the current one, so a change from NULL to a file reference also returns true, even though a NULL original means we have no prior metadata to justify overwriting an existing value.

The replacement logic distinguishes two scenarios. If the mapped field is currently empty, metadata is written unconditionally (scenario 1, unchanged). If the mapped field already has a value, metadata is only written when `$media_source->getSourceFieldValue($original) !== NULL` (the original source field was non-empty), `$media_source->getSourceFieldValue($translation) !== NULL` (the new source field is also non-empty), and `hasSourceFieldChanged()` is still true (scenario 2). The null check uses `=== NULL` rather than `empty()` because `MediaSourceBase::getSourceFieldValue()` explicitly returns `NULL` when the field is empty, and `empty()` would incorrectly treat a source value of `0` as empty. A new test case in `MediaSourceTest::testMetadataMapping()` covers the NULL-to-non-null transition and asserts the pre-existing mapped value is preserved.

The related issue [#2999370](https://www.drupal.org/node/2999370) tracks a similar overwrite problem on translation saves, which this fix does not address.
