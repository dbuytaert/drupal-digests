## Summary

When saving a media entity, Drupal extracts metadata from the source plugin and writes it into any fields configured in the media type field map. A bug caused those mapped fields to be silently overwritten if a media item was first saved without a source field value and then later saved again with one, even when the mapped fields already held legitimate values. Sites that programmatically create media entities in two passes, or that use tools such as Entity Share to import media, could see mapped field values (captions, alt text, and similar) replaced with raw plugin metadata. This fix tightens the condition so that existing values in mapped fields are only replaced when the source field transitions between two non-empty values.

## Impact

No upgrade or configuration changes required.

## Technical details

The metadata-mapping loop in `Media::prepareSave()` previously allowed two write paths: write if the mapped field is empty, or write if `hasSourceFieldChanged()` returned `TRUE`. Because `hasSourceFieldChanged()` compares the original and current source field values, it returns `TRUE` whenever the original was `NULL`, including the case where the entity simply had no source file on its first save. That made the condition too broad.

The fix introduces a third, stricter condition for the non-empty-field case. `$this->getOriginal()` is called once before the field-map loop. Inside the loop, after confirming the mapped field is not empty, two additional checks are performed: `$media_source->getSourceFieldValue($original) === NULL` to detect an empty original source, and `$media_source->getSourceFieldValue($translation) === NULL` to detect an empty new source. Metadata is now only written when both source values are non-null and `hasSourceFieldChanged()` also returns `TRUE`. The `=== NULL` comparison (rather than `empty()`) was chosen deliberately because `getSourceFieldValue()` already returns `NULL` for any empty field item, and `0` is a legitimate non-empty field value that `empty()` would incorrectly treat as absent.

The related issue [#2999370](https://www.drupal.org/node/2999370) covers a similar overwrite problem in the translation-save code path and remains open.
