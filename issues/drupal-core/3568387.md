## Summary

Drupal core is modernizing its text processing utilities by moving functionality from procedural functions into services. The `text_summary()` function, which generates trimmed summaries of text content, has been deprecated. This function is widely used to create automatic excerpts of content when a manual summary is not provided. The new service-based approach provides better dependency management and follows current Drupal architectural standards. This is part of an ongoing effort to eventually remove the text module's `.module` file entirely by eliminating procedural code.

## Impact

Affects module maintainers.

- **Deprecation:** The `text_summary()` function is deprecated; use the `TextSummary` service instead.

## Technical details

The new `TextSummary` service encapsulates the same logic previously in the `text_summary()` procedural function. The `generate()` method accepts the same parameters: text content (now typed as `string|\Stringable`), an optional text format ID, and a character length limit (now required as `int`, defaulting to 600).

The service is autowired and depends on `EntityTypeManagerInterface` to load filter format entities. The implementation remains identical to the original function: it checks for the `<!--break-->` delimiter, truncates to the specified length, attempts to break at paragraph or sentence boundaries, and normalizes HTML when appropriate filters are enabled.

Core usages in `NodeTokensHooks` and `TextTrimmedFormatter` have been updated to call `\Drupal::service(TextSummary::class)->generate()`. The deprecated `text_summary()` function now serves as a thin wrapper that triggers a deprecation notice and delegates to the service.

The new service uses strict types and proper type hints. The third parameter `$size` is now required as `int` in the service signature, though the deprecated function maintains backward compatibility by reading from `text.settings` configuration when `NULL` is passed.

## Upgrade

Replace calls to `text_summary()` with the `TextSummary` service:

```php
// Before:
$summary = text_summary($text, $format, $size);

// After:
$summary = \Drupal::service(TextSummary::class)->generate($text, $format, $size);
```

For dependency injection:

```php
class MyClass {
  public function __construct(
    protected TextSummary $textSummary,
  ) {}

  public function process($text, $format, $size) {
    return $this->textSummary->generate($text, $format, $size);
  }
}
```

Note that the third parameter `$size` should now be passed as an integer. Previously, passing `NULL` would read from `text.settings` configuration, but the new service defaults to 600 characters if omitted.

### Collaboration

This issue took nearly two months from opening to commit, with 44 comments discussing implementation details. The path was not straightforward, involving several key design decisions.

Early in the process, contributors debated whether to make the new API internal. Berdir argued against it, noting that the function is used in contrib and has value as a public API. The team decided to keep it public.

A significant discussion centered on type hints. Initial attempts to add strict types caused test failures when `FilteredMarkup` objects were passed instead of strings. Berdir suggested accepting `string|\Stringable`, which resolved the issue while properly documenting behavior.

Another debate concerned the `$size` parameter. Berdir proposed making it required in the new service (removing the `NULL` default that read from config) to simplify the API going forward, deferring config cleanup to a separate issue [#3436677](https://www.drupal.org/node/3436677). This approach made the new service cleaner while maintaining backward compatibility in the deprecated function.

Nicxvan made the key contribution of converting from a static helper class to a proper service with dependency injection, improving testability and architectural consistency.
