## Summary

The `ChainedFastBackend` cache class had a gap in its self-healing logic: if the `last_write_timestamp` entry was absent from the persistent (consistent) backend, cache reads would fall through to the persistent backend and write items back to the fast backend, but those fast-backend entries could never be validated because no timestamp existed to compare against. The situation would persist silently until an actual cache miss triggered a genuine `set()`, which finally established the timestamp. The fix writes a fresh timestamp immediately when the missing condition is detected, so the fast backend becomes usable right away.

## Impact

No upgrade or configuration changes required.

## Technical details

In `ChainedFastBackend::getMultiple()`, after the existing call to `$this->getLastWriteTimestamp()`, a new guard `if ($last_write_timestamp === 0)` calls `$this->markAsOutdated()`. `markAsOutdated()` writes a current-time value to the persistent backend under the `last_write_timestamp` key, which makes subsequent fast-backend writes valid immediately. The condition `=== 0` is the correct test because `getLastWriteTimestamp()` returns `0` when no entry exists. Dries caught an early version of the patch that had the condition wrong, and catch corrected it. The missing-timestamp scenario can arise in at least two ways: (1) converting an existing cache bin to chained fast while leaving warm persistent-cache entries in place, so the persistent backend has data but no timestamp; (2) LRU eviction on a backend like Redis or Memcache that removes the timestamp key while retaining other cache items. A new unit test `testLastWriteTimestampMissing()` in `ChainedFastBackendTest` covers the scenario by mocking both backends, asserting that `set()` is called once on the consistent backend (for the timestamp) and once on the fast backend (for the cache item). An existing test `testGetDoesNotHitConsistentBackend()` also gained an assertion that `set()` is never called on the consistent backend in the normal warm-cache path. Discovered while investigating [#3583040](https://www.drupal.org/node/3583040) (Cache the container definition in APCu).
