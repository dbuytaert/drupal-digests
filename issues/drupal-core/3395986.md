## Summary

Six plugin classes and one service class in Drupal core were still using the deprecated `REQUEST_TIME` constant and `time()` function directly instead of the injected `datetime.time` service. This change adds `TimeInterface` as a properly injected dependency to each affected class, completing that part of the broader `REQUEST_TIME` removal effort. Site owners are not directly affected; the change matters to developers who extend these plugin classes.

## Impact

Affects module maintainers.

- **Deprecation:** Calling `__construct()` on `TimestampFormatter`, `datetime\\Plugin\\views\\argument\\Date`, `HistoryUserTimestamp`, `views\\Plugin\\views\\argument\\Date`, `views\\Plugin\\views\\cache\\Time`, and `views\\Plugin\\views\\field\\Date` without the `$time` (`TimeInterface`) argument is deprecated in drupal:10.3.0 and the parameter will be required in drupal:11.0.0.

## Technical details

The `REQUEST_TIME` constant was deprecated in Drupal 8.3.0 and removed in 11.0.0. A series of issues ([#3112298](https://www.drupal.org/node/3112298), [#3112295](https://www.drupal.org/node/3112295), [#3113971](https://www.drupal.org/node/3113971)) progressively replaced its uses across core. This issue covers the remaining plugin classes that had direct container access but had not yet been migrated.

The fix follows a consistent backwards-compatible pattern across all affected plugin constructors: `TimeInterface` is added as an optional trailing parameter (`?TimeInterface $time = NULL`). If the argument is omitted, a deprecation notice fires via `@trigger_error()` pointing to the change record at `https://www.drupal.org/node/3395991`, and the service is retrieved via `\Drupal::service('datetime.time')` as a fallback. The `create()` factory is updated to pass `$container->get('datetime.time')` so normal instantiation from the container never triggers the warning. Once the deprecated parameter path is removed, `$time` will become required.

The seven files touched by commit `86aacdff`:

- `core/lib/Drupal/Core/Field/Plugin/Field/FieldFormatter/TimestampFormatter.php`: Added `TimeInterface $time` parameter; replaced `REQUEST_TIME` in `settingsForm()` with `$this->time->getRequestTime()`.
- `core/modules/datetime/src/Plugin/views/argument/Date.php`: Added `TimeInterface $time` parameter; passes it through to `parent::__construct()`.
- `core/modules/history/src/Plugin/views/filter/HistoryUserTimestamp.php`: This class had no explicit constructor or `create()` factory before; both were added from scratch. `REQUEST_TIME` in `query()` replaced with `$this->time->getRequestTime()`.
- `core/modules/views/src/Plugin/views/argument/Date.php`: Added `TimeInterface $time`; replaced `REQUEST_TIME` in `getDefaultArgument()` with `$this->time->getRequestTime()`.
- `core/modules/views/src/Plugin/views/cache/Time.php`: Added `TimeInterface $time`; replaced `REQUEST_TIME` in `cacheExpire()` with `$this->time->getRequestTime()`.
- `core/modules/views/src/Plugin/views/field/Date.php`: Added `TimeInterface $time`; replaced two `REQUEST_TIME` calls in `buildOptionsForm()` and `render()` with `$this->time->getRequestTime()`.
- `core/modules/update/src/UpdateProcessor.php`: Already had a `$time` property injected; the remaining direct `REQUEST_TIME` calls in `createFetchTask()` and `processFetchTask()` were replaced with `$this->time->getRequestTime()`, and the `time()` call computing elapsed batch time was replaced with `$this->time->getCurrentTime() - $this->time->getRequestTime()`. This stray change was identified by taran2l while reviewing for consistency with [#3113971](https://www.drupal.org/node/3113971).

The `core/phpstan-baseline.neon` suppression entries for `REQUEST_TIME` usage in all six plugin files were removed as part of the same commit.

## Upgrade

If you extend any of the six affected plugin classes and override `__construct()`, add `?TimeInterface $time = NULL` as a trailing parameter and pass it to `parent::__construct()`. Example for a class extending `\Drupal\views\Plugin\views\field\Date`:

```php
use Drupal\Component\Datetime\TimeInterface;

public function __construct(
  array $configuration,
  $plugin_id,
  $plugin_definition,
  DateFormatterInterface $date_formatter,
  EntityStorageInterface $date_format_storage,
  ?TimeInterface $time = NULL,
) {
  parent::__construct($configuration, $plugin_id, $plugin_definition, $date_formatter, $date_format_storage, $time);
}

public static function create(ContainerInterface $container, array $configuration, $plugin_id, $plugin_definition) {
  return new static(
    $configuration,
    $plugin_id,
    $plugin_definition,
    $container->get('date.formatter'),
    $container->get('entity_type.manager')->getStorage('date_format'),
    $container->get('datetime.time'),
  );
}
```

Omitting `$time` triggers a deprecation in Drupal 10.3.0. The parameter becomes required in Drupal 11.0.0. See the change record at https://www.drupal.org/node/3395991.
