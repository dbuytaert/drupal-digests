## Summary

Drupal used to bulk-load all "public" (non-admin GET/HTML) routes into memory on every HTML request through a `RoutePreloader` event subscriber. On larger sites this meant loading 100 to 200 or more serialized route objects, consuming roughly 1.5 MB of memory per request. That subscriber and its associated state key (`routing.non_admin_routes`) have been removed. Routes are now cached individually in a new dedicated `cache.routes` bin backed by the `chainedfast` backend (APCu + persistent storage), and loaded on demand using `getMultiple()` rather than pre-fetched in bulk. The net effect is a significant reduction in memory usage per request and faster cache reads when routes are warm in APCu, with a small increase in the number of cache lookups on cold caches that is outweighed by the cost of the eliminated bulk load.

## Impact

Affects module maintainers and site owners.

- **Database update:** Post-update `system_post_update_remove_non_admin_routes_state()` removes the `routing.non_admin_routes` state key
- **Deprecation:** `RouteProvider::$serializedRoutes` and `RouteProvider::ROUTE_LOAD_CID_PREFIX` deprecated in drupal:11.4.0, removed in drupal:12.0.0; no replacement
- **New API:** New `cache.routes` cache bin (chained fast backend) introduced for per-route caching

## Technical details

The old approach stored all non-admin routes under a single cache item keyed by a `sha512` hash of the route name list (using `ROUTE_LOAD_CID_PREFIX`). These were stored as serialized strings in `cache.data` and held in `RouteProvider::$serializedRoutes` until needed, then deserialized on access. The `RoutePreloader` subscriber populated a `routing.non_admin_routes` state key during route rebuilds and triggered the bulk cache load on `KernelEvents::REQUEST` for every HTML request.

The new approach introduces a `cache.routes` service defined in `core.services.yml` with `default_backend: cache.backend.chainedfast`. `RouteProvider` is now injected with this bin instead of `cache.data`. `preLoadRoutes()` calls `cache->getMultiple()` with the route names as cache IDs, routes not found in cache are fetched from the `router` database table and immediately deserialized, then written back via `cache->setMultiple()` with the `routes` cache tag. The intermediate serialized-string holding area (`$serializedRoutes`) is eliminated; routes enter `$this->routes` as objects directly.

`RouteProvider::$serializedRoutes` and `RouteProvider::ROUTE_LOAD_CID_PREFIX` are deprecated in `drupal:11.4.0` and removed in `drupal:12.0.0` (see change record https://www.drupal.org/node/3589089). The `RoutePreloader` class and `router.route_preloader` service are deleted outright with no BC layer, as event subscribers are explicitly excluded from Drupal's BC policy. A post-update function `system_post_update_remove_non_admin_routes_state()` cleans up the now-unused `routing.non_admin_routes` state entry. A parallel effort to add an explicit TTL to the APCu fast bin to bound memory growth is tracked in [#3586418](https://www.drupal.org/node/3586418).

## Upgrade

If your custom code references `RouteProvider::$serializedRoutes` or `RouteProvider::ROUTE_LOAD_CID_PREFIX`, remove those usages; there is no replacement. If you registered a service that depends on `router.route_preloader`, remove that dependency. The `RoutePreloader` class no longer exists.

```php
// Deprecated — remove any reference to:
RouteProvider::ROUTE_LOAD_CID_PREFIX
$routeProvider->serializedRoutes  // protected, but if subclassed

// Removed service — remove from any service definitions:
// router.route_preloader
```

The new `cache.routes` bin can be overridden like any other cache bin in `settings.php`:
```php
$settings['cache']['bins']['routes'] = 'cache.backend.apcu';
```
Sites running on shared or memory-constrained APCu should review available APCu memory, as individual route objects will now populate it instead of a handful of bulk cache items.

### Collaboration

berdir opened the issue after profiling a production site and finding that route preloading consumed around 1.5 MB per request and triggered dozens of redundant cache lookups. They ran multiple rounds of profiling across anonymous and authenticated page views, explored several caching strategies (fast-only, per-format, combined with `getMultiple()`), and shared detailed query counts and memory measurements from Blackfire to guide the design.

catch independently opened a draft MR (MR !11105) to test the approach without caching, sharing observations about which specific routes drove most lookups (the `is_front` logic, `<current>`, bigpipe). Those experiments fed back into berdir's implementation without replacing it. alexpott's review raised the question of deprecating `preLoadRoutes()` and `PreloadableRouteProviderInterface` outright; berdir explored it, concluded it added complexity that belonged in a follow-up, and opened [#3589340](https://www.drupal.org/node/3589340) for that discussion. longwave provided the final RTBC after reviewing the MR.
