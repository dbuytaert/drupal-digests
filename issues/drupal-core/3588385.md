## Summary

The database-backed key-value store now writes all entries from a `setMultiple()` call inside a single database transaction. Previously, each key was written individually with no transactional protection, so a failure partway through could leave only some keys stored. The change makes bulk writes atomic: either all keys are committed or none are, which is safer during concurrent requests or unexpected errors.

## Impact

No upgrade or configuration changes required.

## Technical details

The base class `KeyValueStoreBase::setMultiple()` loops over its input and calls `$this->set()` for each key. `KeyValueDatabaseStore` did not override this, so there was no transaction wrapping the individual `set()` calls. The fix adds a `setMultiple()` override in `KeyValueDatabaseStore` that opens a transaction via `$this->connection->startTransaction()`, delegates to `parent::setMultiple($data)`, and calls `$transaction->commitOrRelease()` on success or `$transaction->rollBack()` on any exception.

A follow-up in [#3586760](https://www.drupal.org/node/3586760) is exploring replacing this pattern with a single multi-row `UPSERT` query, which would make the transaction unnecessary by making the entire operation a single SQL statement.
