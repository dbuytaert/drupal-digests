## Summary

Sites upgrading from Drupal 11.3.x to 11.4.x that had both the Help and Search modules enabled could encounter a fatal database error ("Table 'help_search_items' doesn't exist") partway through `drush updatedb`, leaving the site in a broken state. The error appeared after the search-related `hook_update_n` functions ran and is fixed in 11.4.2. No manual workaround is needed if you are upgrading to 11.4.2 or later.

## Impact

No upgrade or configuration changes required.

## Technical details

The breakage stemmed from `system_update_11400()`, which unconditionally dropped the `help_search_items` table before installing the new `search_help` module. When the module was then installed and its code immediately queried that table (e.g. during `HelpSearch::findResults()` or `HelpSearch::updateTopicList()`), the table no longer existed and no `hook_schema()` had yet re-created it, causing an `SQLSTATE[42S02]` crash that aborted the entire update batch.

The MR makes two coordinated changes. First, `system_update_11400()` is corrected to only drop the table when Search is **not** installed (meaning `search_help` will not be installed and the table is truly obsolete). When Search is installed, the table drop is skipped entirely because `search_help`'s installation will own the table going forward.

Second, the `hook_schema()` implementation in `search_help.install` is removed entirely. Instead, the schema definition and a lazy `ensureTableExists()` method are moved into the `HelpSearch` plugin class itself, following the pattern established in [#3221051](https://www.drupal.org/node/3221051). `HelpSearch::updateTopicList()` now catches the initial database exception, calls `ensureTableExists()` to create the table on the fly if it is absent, and retries the query. `HelpSearch::findResults()` catches the same exception and returns an empty result set rather than letting the error propagate. This makes the `search_help` module resilient to the table being missing at any point, regardless of update ordering.

## Upgrade

No code changes are required in custom modules. If you are upgrading to 11.4.0 or 11.4.1 and hit this error, the fix is included in 11.4.2. As a temporary workaround on the affected versions you can create the table manually before re-running `drush updatedb`:

```sql
CREATE TABLE IF NOT EXISTS help_search_items (
  sid INT UNSIGNED NOT NULL AUTO_INCREMENT,
  section_plugin_id VARCHAR(255) NOT NULL DEFAULT '',
  permission VARCHAR(255) NOT NULL DEFAULT '',
  topic_id VARCHAR(255) NOT NULL DEFAULT '',
  PRIMARY KEY (sid),
  INDEX section_plugin_id (section_plugin_id),
  INDEX topic_id (topic_id)
) ENGINE=InnoDB;
```
