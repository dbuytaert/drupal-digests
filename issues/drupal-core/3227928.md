# #3227928: Users cannot log in with "Drupal\Core\Http\Exception\CacheableAccessDeniedHttpException: The 'access user profiles' permission is required and the user must be active"

## Summary

Site owners may encounter a misleading "The 'access user profiles' permission is required and the user must be active" access-denied error that blocks login for all users, including administrators, after a Drupal upgrade or a change to domain or SSL configuration. No code change was made to Drupal core. This is a support request that was closed after the root cause was identified as a server-side configuration problem: when a site is reachable under two domain forms (for example, both `site.com` and `www.site.com`) without proper canonical redirects, or when HTTP-to-HTTPS redirects are incomplete, the browser accumulates duplicate session cookies with the same name but different domain scopes. The fix is to configure the web server to redirect all non-canonical domain variants to a single canonical URL before Drupal handles any request, and to clear affected browsers' cookies for the old domain.

## Impact

No upgrade or configuration changes required.

## Technical details

The error is thrown by `AccessAwareRouter->checkAccess()` in `core/lib/Drupal/Core/Routing/AccessAwareRouter.php`. When the access result for a route is not allowed, it throws a `CacheableAccessDeniedHttpException` carrying the reason string from the access result. For the `/user/{uid}` route, `UserAccessControlHandler` in `core/modules/user/src/UserAccessControlHandler.php` checks the `access user profiles` permission (line 54) and sets the reason "The 'access user profiles' permission is required and the user must be active." (line 62) whenever the result is not allowed.

The reason the error appears for authenticated administrators is that the server misconfiguration causes the browser to hold two cookies sharing the same session cookie name but scoped to different domains (for example, one to `.site.com` and one to `.www.site.com`). When the conflicting stale cookie is sent, Drupal resolves an anonymous `AccountProxy` for the request instead of the authenticated user. The access check for viewing the user profile then returns `AccessResultNeutral` for an anonymous account, which is treated as denied, and the misleading reason string is surfaced.

Several workarounds were reported in the thread: clearing all browser cookies for the affected domain, clearing the flood table (via a module such as Flood Control), and using an Incognito or never-before-used browser window (which has no conflicting cookie). The durable fix is a web-server redirect rule that enforces a single canonical domain and protocol before Drupal handles requests, preventing the mismatched cookie from being created in the first place.
