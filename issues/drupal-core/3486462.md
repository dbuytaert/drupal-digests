## Summary

Five module lifecycle hooks that previously only accepted procedural implementations can now be implemented as PHP classes using the `#[Hook]` attribute: `hook_module_preinstall`, `hook_modules_installed`, `hook_module_preuninstall`, `hook_modules_uninstalled`, and `hook_cache_flush`. Modules on Drupal 11.1 or later can move their implementations of these hooks out of `.module` files and into attribute-annotated service classes.

## Impact

Affects module maintainers.

- **New API:** Five hooks (`hook_module_preinstall`, `hook_modules_installed`, `hook_module_preuninstall`, `hook_modules_uninstalled`, `hook_cache_flush`) now accept OOP `#[Hook]` attribute implementations in addition to procedural `.module` file functions.

## Technical details

Drupal's OOP hook system, introduced via the `#[Hook]` attribute, originally excluded several hooks from attribute-based implementations because they are invoked during module install and uninstall, a phase where the module handler's full knowledge of implementations may not be current. `HookCollectorPass::checkForProceduralOnlyHooks()` maintained a `$staticDenyHooks` list that caused an exception if any of the five hooks were declared on a class.

The root cause was in `ModuleInstaller::invokeAll()`, a protected helper that iterated all modules and called the installer's own private `$this->invoke()` per module. That private method only dispatched to procedural `.module` file functions, bypassing the module handler's OOP invocation path entirely. The fix was a one-character-class change: replace `$this->invoke($module, $hook, $args)` with `$this->moduleHandler->invoke($module, $hook, $args)`. Because `$this->moduleHandler->loadAll()` is already called at the top of `invokeAll()`, all modules and their attribute-registered hooks are available by the time each per-module invocation fires, so the OOP path is safe to use.

With the invocation path corrected, the five hooks were removed from the `$staticDenyHooks` array in `HookCollectorPass::checkForProceduralOnlyHooks()` and from the "Install hooks" section of the unsupported-hook lists in `core/core.api.php` and the `#[Hook]` attribute's own docblock. The "Only procedural implementations are supported" warnings were also removed from the four hooks' entries in `core/lib/Drupal/Core/Extension/module.api.php`.

A new test module `respond_install_uninstall_hook_test` was added under `core/modules/system/tests/modules/`, with `OtherModuleInstalledHooks` and `OtherModuleUninstalledHooks` classes using the `#[Hook]` attribute to implement all five hooks. A new kernel test method `testInvokingRespondentHooks()` in `ModuleInstallerTest` installs and uninstalls a test module and asserts that each attribute-based hook fired. Note that `hook_install`, `hook_uninstall`, `hook_schema`, `hook_requirements`, and `hook_post_update_NAME` remain procedural-only because they run before the module being acted on is fully registered with the module handler.

## Upgrade

Modules that implemented any of these five hooks procedurally can now optionally move those implementations into a `#[Hook]`-attributed class. No change is required for existing procedural implementations.

```php
// Before (in mymodule.module):
function mymodule_modules_installed(array $modules, bool $is_syncing): void {
  // ...
}

// After (in src/Hook/MyModuleHooks.php):
use Drupal\Core\Hook\Attribute\Hook;

class MyModuleHooks {
  #[Hook('modules_installed')]
  public function modulesInstalled(array $modules, bool $is_syncing): void {
    // ...
  }
}
```

The same pattern applies to `hook_module_preinstall`, `hook_module_preuninstall`, `hook_modules_uninstalled`, and `hook_cache_flush`.
