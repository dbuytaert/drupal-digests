## Summary

A new `EntityFieldValueTrait` was added to allow entity types to access field values without creating `FieldItemList` objects. Creating `FieldItemList` objects (what happens when calling `$entity->get()`) is relatively expensive and can cause memory leaks due to circular references, particularly when working with large numbers of entities. The User entity now uses this trait internally for methods like `getAccountName()` and `isActive()` to reduce overhead. The trait provides a protected, internal method that is opt-in for custom entity types but not exposed as public API.

## Impact

No upgrade or configuration changes required.

## Technical details

The typed data system creates `FieldItemList` objects when `$entity->get()` is called. Each creation has overhead, and circular references in the field system prevent garbage collection. On a test page loading 500 entities, this caused a 170MB memory leak that could not be reclaimed despite LRU caching.

`EntityFieldValueTrait` provides a protected `getFieldValue()` method that accesses field values directly from the entity's internal `$this->values` array when fields have not been initialized. It checks the active language code and attempts to access values with delta and property, falling back to simpler structures for base fields. If the field has already been initialized, it falls back to the regular `$this->get()` API to maintain consistency.

The method is marked protected and `@internal` because it does not behave consistently for all field types. It does not support computed fields or computed properties. The trait is designed for entity types to wrap with specific public getter methods, as User does for `getAccountName()`, `getTimeZone()`, `isActive()`, and `getLastAccessedTime()`.

The approach was changed during review from a general entity method to an opt-in trait with protected access. This allows User module and contrib to use it for specific performance-critical code paths while discouraging general use until the underlying memory leaks are fixed in [#3572625](https://www.drupal.org/node/3572625) and [#3573982](https://www.drupal.org/node/3573982).
