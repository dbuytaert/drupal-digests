## Summary

Since Drupal 11.3 introduced OOP hook scanning for themes, cache rebuilds on sites with a `node_modules/` directory inside a custom theme began emitting large numbers of `file_get_contents() Permission denied` warnings and could slow down dramatically. This happened because the new hook-collector classes did not respect the `file_scan_ignore_directories` setting. The fix makes `HookCollectorPass` and `ThemeHookCollectorPass` honour that setting, so directories listed there are skipped during hook discovery just as they already were during file and extension scanning.

## Impact

Affects site owners.

- **Config change:** The `file_scan_ignore_directories` setting now also controls which directories are skipped during hook-discovery scanning in `HookCollectorPass` and `ThemeHookCollectorPass`, not only file API and extension scanning.

## Technical details

The 11.3 addition of `ThemeHookCollectorPass` and `HookCollectorPass` each contain a `filterIterator()` method that decides which directories to recurse into when scanning an extension for hook implementations. Those methods had a hardcoded exclusion list (`['tests', 'js', 'css']` for modules; `['tests', 'js', 'css', 'templates']` for themes) but did not consult `Settings::get('file_scan_ignore_directories')`. As a result, `node_modules/` trees inside theme directories were fully traversed, triggering `StaticReflectionParser::parse()` to attempt `file_get_contents()` on every file it encountered, many of which were unreadable directories on the developer's system.

The fix adds `Settings::get('file_scan_ignore_directories', [])` inside `filterIterator()` in both `HookCollectorPass` and `ThemeHookCollectorPass`, merging the result into the hardcoded exclusion array. The comment block for `$settings['file_scan_ignore_directories']` in `default.settings.php` was also updated to document that the setting now governs hook-discovery scanning in addition to the file API and extension discovery. Tests covering both the module-hook and theme-hook cases were added under `core/modules/system/tests/modules/HookCollector/` and `core/modules/system/tests/themes/HookCollector/`.

### Collaboration

The thread had a brief but substantive design debate. berdir's initial read-through suggested the fix should happen higher in the call stack and should respect the scan setting rather than only hardcoding directory names. nicxvan later floated the idea of simply hardcoding `node_modules` and `bower_components` rather than reusing `file_scan_ignore_directories`, reasoning the two concerns were distinct. richard.thomas pushed back by quoting the existing `default.settings.php` documentation, which already describes the setting as preventing "recursive scanning of directories looking for extensions," arguing the intent was identical. nicxvan agreed, and the approach of reading the setting at runtime was kept. The `default.settings.php` comment was then revised to make the broadened scope explicit.
