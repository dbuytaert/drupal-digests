## Summary

When one module decorates a service defined by another module that is not installed, Drupal previously crashed with a hard "dependency on a non-existent service" error. The only workaround was writing a custom service provider class that manually checked for and removed the decorator. Modules can now set `decoration_on_invalid` directly in their `services.yml` to control what happens when the decorated service is absent, eliminating the need for that boilerplate PHP.

## Impact

Affects module maintainers.

- **New API:** Modules can now set `decoration_on_invalid: ignore` or `decoration_on_invalid: null` (bare YAML null) in `services.yml` alongside a `decorates` key to gracefully handle a missing decorated service instead of throwing an exception.

## Technical details

The change is entirely in `YamlFileLoader::parseDefinition()` in `core/lib/Drupal/Core/DependencyInjection/YamlFileLoader.php`. Before the fix, when a service definition contained a `decorates` key, `$definition->setDecoratedService()` was called without the fourth `$invalidBehavior` argument, so Symfony's default (`EXCEPTION_ON_INVALID_REFERENCE`) was always used and there was no YAML-level way to change it.

The fix adds support for the `decoration_on_invalid` key in `services.yml`, porting the feature from Symfony's own `YamlFileLoader` (the reviewer longwave confirmed the code is a 1:1 copy). The `parseDefinition()` method now reads the optional `decoration_on_invalid` key and maps it to a Symfony `ContainerInterface` constant before passing it to `setDecoratedService()`:

- `'exception'` (default when key is absent) → `ContainerInterface::EXCEPTION_ON_INVALID_REFERENCE` — preserves existing behavior.
- `'ignore'` → `ContainerInterface::IGNORE_ON_INVALID_REFERENCE` — the decorator is silently dropped when the target service does not exist.
- YAML bare `null` → `ContainerInterface::NULL_ON_INVALID_REFERENCE` — the decorated service reference resolves to null.

Two helpful validation guards were also added: a `decorates` value beginning with `@` now throws an `InvalidArgumentException` telling the developer to drop the prefix, and the string `"null"` (quoted, a common YAML mistake) throws a specific message suggesting bare `null` instead. Any other unrecognised value throws a general invalid-value error listing the accepted options.

Unit test cases for all three new error conditions were added to `YamlFileLoaderTest::providerTestExceptions()`.

## Upgrade

If you currently work around this problem with a service provider that removes a decorator when its target is absent, you can replace it with a single YAML property. For example, given:

```php
// MyModuleServiceProvider.php
public function alter(ContainerBuilder $container) {
    if (!$container->hasDefinition('modulea.service')) {
        $container->removeDefinition('moduleb.service');
    }
}
```

You can delete that class and instead add `decoration_on_invalid: ignore` to the decorator's definition in `moduleb.services.yml`:

```yaml
services:
  moduleb.service:
    class: Drupal\moduleb\Service\MyDecorator
    decorates: modulea.service
    decoration_on_invalid: ignore
```

Valid values for `decoration_on_invalid` are `exception` (default, keeps existing behavior), `ignore` (silently removes the decorator), and bare YAML `null` (sets the inner service reference to null).
