## Summary

Sites using the Navigation module's admin toolbar had a WCAG 4.1.2 accessibility violation: the mobile sidebar overlay `<div>` was being given an `aria-expanded` attribute that is not permitted on plain `div` elements. Automated tools such as Axe and Accessibility Insights reported this as an error. The fix removes the invalid attribute by reworking how the overlay participates in the toolbar's expand/collapse logic, and the Nightwatch accessibility test now exercises both desktop and mobile viewport sizes so the same class of regression would be caught automatically.

## Impact

No upgrade or configuration changes required.

## Technical details

The Navigation module renders a `<div class="admin-toolbar-overlay">` in `navigation.html.twig` that acts as the clickable backdrop for the mobile modal sidebar. Previously the overlay carried `aria-controls="admin-toolbar"`, which caused `admin-toolbar-wrapper.js` to include it in the `once()` trigger set selected by `[aria-controls="admin-toolbar"]`. The `toggleTriggers()` function then called `setAttribute('aria-expanded', toState)` unconditionally on every element in that set. Because `aria-expanded` is only permitted on elements with specific roles (such as `button`, `combobox`, and others), placing it on a plain `<div>` produced a WCAG 4.1.2 (`aria-allowed-attr`) violation.

Two coordinated changes fix this. First, in `navigation.html.twig` the overlay's `aria-controls="admin-toolbar"` attribute is replaced with a non-ARIA custom attribute `admin-toolbar-trigger`, removing the invalid hook entirely from the markup. Second, in `admin-toolbar-wrapper.js` the `once()` selector is widened to `'[aria-controls="admin-toolbar"], [admin-toolbar-trigger]'` so the overlay is still registered as a trigger (for click-to-close behavior), and `toggleTriggers()` gains an `if (trigger.matches('button'))` guard so `aria-expanded` is set only on `<button>` elements, never on the overlay `<div>`.

The Nightwatch test in `a11yTest.js` was restructured to run Axe at two viewport sizes within a single test flow: desktop at 1220×800 and mobile at 1000×800. The mobile pass explicitly clicks the "Expand sidebar" button and waits for the sidebar content to become visible before calling `axeRun`, ensuring the overlay is in its expanded state during the Axe scan. kentr confirmed locally that the updated test fails against the unfixed `main` branch, providing regression coverage for this class of issue.
