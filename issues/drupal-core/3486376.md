## Summary

When running Drupal's PHPUnit test suite, contrib and custom extensions that override a method from a different Drupal extension (for example, a module class that extends a core class) without adding a native return type will now trigger a deprecation notice, provided the parent method documents its intended return type via a `@return` docblock annotation. This early warning gives extension authors time to add native return types before those types are enforced in interfaces and abstract classes in the next major release, where the absence of a matching return type would cause a fatal error. Production sites are not affected; only test runs are impacted.

## Impact

No upgrade or configuration changes required.

## Technical details

Symfony's `DebugClassLoader` uses the first namespace segment as the vendor boundary for return-type deprecation checks. Because all Drupal code shares `Drupal\` as that first segment, cross-extension warnings between, say, a contrib module and Drupal core were never emitted.

The fix introduces `DrupalDebugClassLoader` in `core/tests/Drupal/TestTools/ErrorHandler/DrupalDebugClassLoader.php` (marked `@internal`), which extends `DebugClassLoader` and overrides `checkAnnotations()`. After calling `parent::checkAnnotations()`, it reads the static `$returnTypes` property from `DebugClassLoader` via `ReflectionProperty` and iterates over the collected return-type data for the class being loaded. A private `getExtensionName()` helper extracts the Drupal extension name from a fully qualified class name: the second namespace segment is used, or the third when `Tests` is the second. When the declaring class and the implementing class belong to different extensions, the loader checks whether the method is actually overridden in the current class, whether it already has a native return type, and whether its docblock contains `@deprecated` or `@return` (any of which suppresses the notice). If none of those conditions are met, a deprecation in standard Symfony format is appended.

The test bootstrap (`core/tests/bootstrap.php`) is updated to call `DrupalDebugClassLoader::enable()` instead of `DebugClassLoader::enable()`. To prevent immediate CI breakage on known existing violations, two broad regex patterns are added to `core/.deprecation-ignore.txt`, one covering all current core modules and one covering test modules and fixture extensions. A handful of violations that could not be suppressed in tests were fixed directly: `TestableSqlContentEntityStorage::getTableMapping()` gains `: TableMappingInterface`, several methods in `core/tests/Drupal/KernelTests/RouteProvider.php` gain native return types, and `ClaroPreRender::trustedCallbacks()` gains `: array`. A unit test `DrupalDebugClassLoaderTest` covers the positive case and five negative cases (same-extension child, child with native return type, child with `@return` annotation, child with `@deprecated`, child that does not override the method). Follow-up work to fix the suppressed violations is tracked in [#3579258](https://www.drupal.org/node/3579258).

## Upgrade

If your test suite shows a new deprecation of the form:

```
Method "Drupal\some_extension\SomeClass::someMethod()" might add "string" as a native
return type declaration in the future. Do the same in child class
"Drupal\your_extension\YourClass" now to avoid errors or add an explicit @return
annotation to suppress this message.
```

You have two options:

**Option 1: Add a native return type (preferred)**
```php
// Before
public function someMethod() {
  return $this->doSomething();
}

// After
public function someMethod(): string {
  return $this->doSomething();
}
```

**Option 2: Add an explicit `@return` annotation to suppress the notice**
```php
/**
 * {@inheritdoc}
 *
 * @return string
 *   Description of the return value.
 */
public function someMethod() {
  return $this->doSomething();
}
```

The notice only fires during test runs. No changes are needed for production code.
