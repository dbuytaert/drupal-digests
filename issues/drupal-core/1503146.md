## Summary

Drupal's front page configuration now accepts and stores path aliases directly, making configuration portable across environments. Previously, entering a path alias like `/home` would automatically convert it to `/node/1` in configuration, creating a hard dependency between site configuration and specific content IDs. This made it difficult to export configuration and deploy it to other environments where the same content might have different node IDs. Now the system stores whatever path you enter, whether it's an alias or an internal path, and correctly recognizes it as the front page in all cases.

## Impact

Affects site owners.

- **Config change:** The front page, 403, and 404 path settings in `system.site` config now store path aliases as entered instead of converting them to internal paths.

## Technical details

The change has two main parts. First, `SiteInformationForm` no longer converts path aliases to system paths when saving. The form previously used `AliasManager::getPathByAlias()` to resolve aliases before storing them in `system.site` config. This conversion has been removed, and the form now uses `#config_target` to store values directly as entered.

Second, the `PathMatcher::isFrontPage()` method was not checking for aliases, so it would fail to recognize the front page when an alias was stored in config. A new `AliasPathMatcher` class now decorates the core `PathMatcherInterface` service in the `path_alias` module. This decorator extends `isFrontPage()` to check both the internal path (via the original implementation) and the alias (via a new `isAliasFrontPage()` method). The alias check uses `AliasManager::getAliasByPath()` to get the current page's alias and compares it to the configured front page path.

This decouples configuration from content IDs, which is important for configuration management workflows where content may have different IDs across environments but maintain consistent aliases. The same approach applies to the 403 and 404 error page settings.

The `SiteInformationForm` was simplified by removing the `AliasManagerInterface` dependency and the `submitForm()` method. The form validation no longer pre-processes paths before validation.

## Upgrade

If you previously relied on `system.site` config always containing internal paths like `/node/123`, you may need to update code that reads this configuration. The front page, 403, and 404 paths may now be either internal paths or aliases. Use the path validator or alias manager to handle both cases.

When you save the site information form after this update, whatever path you enter will be stored as-is in configuration. If you have custom code that sets these values programmatically, you can now use aliases directly instead of needing to resolve them to system paths first.

See the change record for more details: [#3572707](https://www.drupal.org/node/3572707)

## Lessons learned

- When configuration references content by system paths (like /node/123), it creates a hard coupling between configuration and content that breaks modern deployment workflows. Store user-facing aliases instead, and resolve them at runtime so configuration remains portable across environments where content IDs differ. *(via dieterholvoet, gambry)*
- When a form auto-converts user input before saving (like converting aliases to system paths), users cannot see what value is actually stored. Display transformations can hide the stored value, but the underlying data should match user intent to avoid surprising behavior and deployment issues. *(via raphaeltbm, dieterholvoet)*
- When checking if the current page matches a configured path, compare both the direct value and its resolved form. Users may configure either aliases or system paths, and runtime matching must handle both to avoid fragile exact-match comparisons. *(via gambry, dieterholvoet)*
