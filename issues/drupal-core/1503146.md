## Summary

The site information form (admin/config/system/site-information) now stores front page, 403, and 404 paths exactly as entered, instead of silently converting path aliases to internal paths (e.g., `/homepage` to `/node/1`). This means `PathMatcher::isFrontPage()` now correctly recognizes the front page even when the `system.site` config contains a path alias instead of an internal path. Previously, setting the front page via drush, config import, or an installation profile using an alias would cause `isFrontPage()` to return FALSE, breaking front page detection for themes, modules, and redirect behavior.

## Impact

Affects module maintainers and site owners.

- **Config change:** `system.site` `page.front`, `page.403`, and `page.404` now store paths as entered (which may be aliases) rather than converting to internal paths.
- **API change:** `SiteInformationForm` constructor no longer takes an `AliasManagerInterface` parameter; `submitForm()` and `getEditableConfigNames()` methods removed.
- **New API:** New `AliasPathMatcher` service decorator in the `path_alias` module that extends `PathMatcher::isFrontPage()` to also check aliases.

## Technical details

`PathMatcher::isFrontPage()` compares the current route's internal path against the `system.site` `page.front` config value. When `page.front` contained a path alias (e.g., `/homepage` rather than `/node/1`), that comparison always failed. The `SiteInformationForm` masked this by converting aliases to internal paths on save, but config set via drush, config import, or installation profiles could still store aliases, causing the mismatch.

The fix has two parts. First, `SiteInformationForm` no longer converts aliases to internal paths. The `AliasManagerInterface` dependency was removed, the custom `submitForm()` override was removed, and the front page field now uses `#config_target` to save directly to `system.site:page.front`. The same alias-to-path conversion was also removed for the 403 and 404 page fields. The form now uses `RedundantEditableConfigNamesTrait` since all config is handled via `#config_target`.

Second, a new `AliasPathMatcher` class in the `path_alias` module decorates the core `path.matcher` service. Its `isFrontPage()` first delegates to the original `PathMatcher::isFrontPage()`. If that returns FALSE, it additionally resolves the current route's internal path to its alias and compares that against `page.front`. The result is cached because `isFrontPage()` is called often. The decorator lives in the `path_alias` module because core's `PathMatcher` cannot depend on path alias functionality. The decorator is registered in `path_alias.services.yml` using the `decorates` key and `#[AutowireDecorated]`.

## Upgrade

If you extend `SiteInformationForm`, note that the constructor no longer accepts an `AliasManagerInterface` parameter, and `submitForm()` and `getEditableConfigNames()` are no longer overridden.

Previously `system.site` `page.front` always stored an internal path like `/node/1`. It now stores the value as entered, which may be a path alias like `/homepage`. Code that reads `page.front` and assumes it is an internal path may need to resolve aliases.

See change record https://www.drupal.org/node/3572707.

### Collaboration

This issue was open for nearly 14 years, originally filed against Drupal 7. For years, contributors debated whether it was reproducible at all (#12, #56), since the site information form masked the problem by converting aliases to internal paths before saving. The real issue only surfaced when config was set outside the form (drush, config import, installation profiles). DieterHolvoet drove the solution forward starting around 2021, arriving at the decorator pattern in the `path_alias` module after discovering a direct fix would create a dependency on `path_alias` from core. The Drupal CMS project, which was already polyfilling this fix (#90), helped motivate final review and commit. During final review, concerns were raised (#100, #106) that storing aliases instead of internal paths makes the front page break if an alias changes, but the committer (alexpott) concluded after discussion with catch and longwave that this does not make the system more fragile than it already is (#108).
