## Summary

A bug in the Views module caused the `site` and `view` token types to be silently dropped whenever additional token types were passed to `getAvailableGlobalTokens()`. For example, a view using date tokens alongside the expected site tokens would lose all site tokens, producing empty output instead of the intended replacements. The fix ensures that `site` and `view` are always included in the available token set regardless of what other types are requested.

## Impact

No upgrade or configuration changes required.

## Technical details

The method `PluginBase::getAvailableGlobalTokens()` in `core/modules/views/src/Plugin/views/PluginBase.php` used PHP's array union operator (`$types += ['site', 'view']`) to guarantee that `site` and `view` token types are always present. Because `$types` is a numerically-indexed array, the union operator compares by index: index 0 is already occupied by whatever the caller passed, so `'site'` (which would go to index 0) is never appended. The fix replaces the union with `array_merge($types, ['site', 'view'])`, which appends unconditionally and lets `array_intersect_key` downstream select the correct tokens. The proposed `array_unique` wrapper was dropped during review because the token info array is multi-dimensional and `array_unique()` raises an "array to string conversion" error on such structures. A `@DataProvider`-driven unit test was added to `PluginBaseTest` covering combinations of zero, one, and multiple token types to prevent regression.
