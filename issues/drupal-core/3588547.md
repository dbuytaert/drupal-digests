## Summary

Sites using `ChainedFastBackend` (typically APCu as the fast layer in front of a database or Redis persistent layer) will see fewer unnecessary APCu invalidations when cache items are written. Previously, every `set()` or `setMultiple()` call caused the entire fast backend to be marked outdated, forcing subsequent reads to fall back to the persistent backend even when those reads had nothing to do with the items just written. Now, the invalidation only happens when the item being set already existed in the persistent backend, i.e. when it is truly being updated rather than freshly added. During cold cache warmup, where many new items are written in rapid succession, this eliminates a cascade of full-bin invalidations that was making the fast backend effectively useless until warming completed.

No upgrade or configuration changes required.

## Technical details

The key mechanism in `ChainedFastBackend` is a "last write timestamp" stored in the consistent backend. When `markAsOutdated()` is called, it bumps that timestamp, causing `getMultiple()` to treat all APCu hits as stale if they were written before the new timestamp. This timestamp approach is intentionally coarse: it invalidates the whole fast bin rather than individual entries.

Previously, `set()` and `setMultiple()` always called `markAsOutdated()` unconditionally after writing to the consistent backend. The fix changes both methods to first call `consistentBackend->get()` (or `getMultiple()` for the plural form) to check whether the item already exists. If the item is found, `markAsOutdated()` is called as before because existing fast-backend copies of that item are now stale. If the item is not found, no invalidation is needed: the fast backend cannot hold a stale copy of something it has never seen. The comment in the diff acknowledges a narrow race window where another process might write the same key between the `get()` check and the `set()`, but notes that such a process would also have received an empty `get()` result and would therefore not trigger invalidation either, keeping the risk of divergent state extremely low.

Unit tests in `ChainedFastBackendTest` were split from a single `testSet()` into `testSetEmptyConsistent()` (verifying that four writes to new keys produce no `markAsOutdated()` call and exactly four `get()` checks on the consistent backend) and `testSetFullConsistent()` (verifying that writes to pre-existing keys still trigger invalidation).
