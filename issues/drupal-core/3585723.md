## Summary

The `#date_year_range` form property on date elements silently ignored any absolute year value with fewer than four digits (0 through 999), substituting the current year instead. So a range like `0:2050` or `500:1500` would behave as if the minimum were the current year. This is now corrected: absolute years from 0 to 9999 all work as expected.

## Impact

No upgrade or configuration changes required.

## Technical details

The root cause was a too-strict regular expression in `DateElementBase::datetimeRangeYears()`. The absolute-year pattern was `@^[0-9]{4}@`, which requires exactly four digits. Any value with fewer digits (0, 1, 500, 999) failed that match and fell through to the relative-offset branch (`@[\+|\-][0-9]{1,4}@`). Because bare short values like `0` also fail the relative pattern, the method silently fell back to `$this_year`, producing the wrong result with no warning.

The fix changes the pattern to `@^[0-9]{1,4}@`, accepting one to four digits as an absolute year. The updated comment clarifies that current-year-relative notation requires an explicit `+` or `-` prefix (e.g. `+0` for the current year, not `0`), and notes that years BC (negative absolute years) remain unsupported due to the sign-prefix ambiguity.

A new unit test class `DateElementBaseTest` (in `core/tests/Drupal/Tests/Core/Datetime/Element/`) covers 14 data-driven cases: standard four-digit ranges, years below 1000, year zero, single-digit years, mixed absolute and relative bounds, and both-relative ranges.

## Upgrade

If your code passes a `#date_year_range` value with an absolute year below 1000, you no longer need any workaround. Review any range strings that used a bare `0` intending to mean "current year": they now mean year 0 (1 BCE in the proleptic Gregorian calendar). Use `+0` if you want the current year as the boundary.
