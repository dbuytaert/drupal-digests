## Summary

SDC slots can now declare which components or component tags they expect to contain, and how many items they should accommodate. Three new optional keys are available in slot definitions: `expected` (a list of component IDs or tags), `minItems` (minimum number of items), and `maxItems` (maximum number of items). Components can also declare `tags` to categorize themselves. These are declarative features only—they are not enforced at render time. Display building tools like Canvas, Experience Builder, and Display Builder can use this information to guide their UI, such as limiting which components can be added to an accordion group's slot or showing appropriate suggestions.

## Impact

Affects module maintainers.

- **New API:** Slots can declare `expected` (array of component IDs or tags), `minItems`, and `maxItems`. Components can declare `tags`.

## Technical details

SDC needed a way to express relationships between parent and child components, particularly for tightly coupled patterns like accordion groups and items, tab groups and items, or card carousels. Discussion explored several approaches including HTML element-based rules, trait systems, and policy configurations. The team researched how other systems handle this (Oracle Content Management, Builder.io, Storyblok, Magnolia, Gutenberg) and decided on a simple declarative approach.

The implementation adds three properties to the slot schema definition: `expected` (array of strings), `minItems` (integer with minimum 0), and `maxItems` (integer with minimum 1). The `expected` array can contain component IDs (strings with colons) or component tags (strings without colons). A separate `tags` property (array of strings) was added to the component schema to enable tag-based matching.

The changes are limited to the JSON schema files (`metadata-full.schema.json` and `metadata.schema.json`). No enforcement logic was added to the render layer. The decision to keep enforcement at the display builder level rather than the SDC level addresses concerns about flexibility: business needs may require exceptions, and renderables like Views may wrap components in ways that aren't known at placement time. The naming evolved from `allowed` to `expected` to clarify that these are suggestions rather than hard constraints.
