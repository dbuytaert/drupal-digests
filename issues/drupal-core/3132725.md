## Summary

The "Limit list to selected items" checkbox on exposed `in_operator` Views filters (such as Content type) has never functioned correctly. When that option was checked and the exposed filter's default "- Any -" selection was in effect, the result set was not limited to nodes matching the pre-selected filter values. The fix makes the option work as advertised: with "Limit list to selected items" enabled and "- Any -" selected, only records matching the configured filter values are returned.

## Impact

No upgrade or configuration changes required.

## Technical details

The root cause was a key-name mismatch in `InOperator::acceptExposedInput()`. The method guarded its "return TRUE when 'All' is selected and reduce is active" code path with `!empty($this->options['expose']['limit'])`, but the expose form stores this option under the key `'reduce'` (set in `buildExposeForm()` as `$form['expose']['reduce']`). Because no key named `'limit'` ever exists in the expose options, the condition always evaluated to false and the block was never entered, leaving "All" input unhandled by the reduce logic. The fix is a one-character key rename from `'limit'` to `'reduce'` in that condition. A kernel test was added to `FilterInOperatorTest::testFilterExposedReduce()` that configures the `age` filter on the `test_view` with `reduce => TRUE` and `value` restricted to two ages, then asserts the result set contains only matching rows.
