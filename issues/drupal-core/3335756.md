## Summary

Drupal 12 completes the removal of the sequences table, which was deprecated in Drupal 10.2. This table was historically used to generate unique IDs but is no longer needed. The installer now checks for the key_value table instead to determine whether Drupal is already installed. Sites upgrading to Drupal 12 will automatically have the sequences table dropped during the update process. Custom or contributed modules that still relied on this table needed to migrate away before Drupal 12 or recreate the table themselves.

## Impact

Affects site owners.

- **Database update:** The deprecated `sequences` table is removed by `system_update_12000()`

## Technical details

The `sequences` table was originally part of the system schema but deprecated in Drupal 10.2. This change fully removes it from Drupal 12.

The `system_schema()` hook implementation is deleted entirely, as the sequences table was its only remaining definition. The update function `system_update_12000()` drops the sequences table if it exists.

The installer previously checked for the sequences table (the last table in `system_schema()`) to determine if Drupal was already installed. This logic is replaced with checks for the `key_value` table, which is created early in the install process and is a reliable indicator of an existing installation.

The installer's two-phase bootstrap now queries the `key_value` table directly for the `install_task` state entry to determine whether a previous partial install exists. This eliminates the dependency on `system_schema()` while maintaining support for resumed installations.

The change also updates test infrastructure and exception handling to use `key_value` instead of `sequences` for detecting database state. Drush already used `key_value` for this purpose, validating the approach.

## Lessons learned

- When validating whether a system is already installed, check for a table that is guaranteed to exist in that system's core operation and cannot be easily swapped out. Relying on deprecated or optional tables leads to false negatives when those tables are removed. *(via mstrelan, andypost, catch)*
- When removing deprecated functionality that has been used for system bootstrapping or installation validation, verify that alternative database drivers and service replacements still work. Even if modules can swap implementations, they cannot do so before installation, so core tables are safe to check during early bootstrap. *(via catch, andypost)*
