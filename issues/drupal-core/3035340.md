## Summary

Eight procedural helper functions in `core/modules/views_ui/admin.inc` that power Views UI form building and AJAX behavior are deprecated in Drupal 11.4 and scheduled for removal in Drupal 13.0. Their logic has been moved into two new PHP traits in the `Drupal\views` namespace. Contrib modules or custom code that calls any of these functions directly will receive deprecation warnings and should migrate to the equivalent trait methods.

## Impact

Affects module maintainers.

- **Deprecation:** Eight procedural functions in `core/modules/views_ui/admin.inc` deprecated in drupal:11.4.0 and removed in drupal:13.0.0; replaced by methods on `\Drupal\views\ViewsFormAjaxHelperTrait` and `\Drupal\views\ViewsFormHelperTrait`.
- **New API:** Two new traits added to the `views` module: `\Drupal\views\ViewsFormAjaxHelperTrait` (AJAX form helpers) and `\Drupal\views\ViewsFormHelperTrait` (general form helpers).

## Technical details

The eight functions in `admin.inc` were procedural helpers tightly coupled to Views UI form classes. The refactoring moves them into two traits in `core/modules/views/src/`:

`ViewsFormAjaxHelperTrait` carries the AJAX trigger infrastructure: `addAjaxTrigger()`, and the static callbacks `addLimitedValidation()`, `addAjaxWrapper()`, `ajaxUpdateForm()`, and `noJsSubmit()`. A protected static `$ajaxTriggerButtons` property replaces the former `drupal_static()` calls for per-request deduplication state. Service access is via a protected `getElementInfo()` getter that calls `\Drupal::service('plugin.manager.element_info')`.

`ViewsFormHelperTrait` carries `standardDisplayDropdown()`, `buildFormUrl()`, and the static `formButtonWasClicked()`. Service access is via a protected `getConfigFactory()` getter calling `\Drupal::service('config.factory')`.

Seven existing classes now use these traits: `DisplayPluginBase`, `EntityField`, `WizardPluginBase`, `ViewsFormBase`, `ViewAddForm`, `ViewEditForm`, and `ViewUI`. All `\Drupal::moduleHandler()->loadInclude('views_ui', 'inc', 'admin')` calls were removed from those classes and from `TagTest`.

The deprecated functions in `admin.inc` now emit `E_USER_DEPRECATED` and delegate through an internal helper `_views_ui_wrap_views_form_helper_traits()`, which returns an anonymous class that uses both traits and exposes their protected methods as public wrappers to bridge the procedural-to-OOP gap without duplicating logic.

A service-based approach was explored and then reverted. The core problem was that `DisplayPluginBase`, `EntityField`, `WizardPluginBase`, and `ViewsFormBase` each have many subclasses, so injecting the new services would have cascaded constructor changes across a large surface area in core and contrib. Traits with `\Drupal::service()` getters were chosen as the less disruptive path, accepting the `\Drupal::service()` call inside a trait because a trait does not know where it will be consumed.

Some callback array entries that previously held string function names (e.g., `'views_ui_nojs_submit'`) now reference static methods on the consuming class (e.g., `[static::class, 'noJsSubmit']`), which was required because plugin forms cannot reference arbitrary function names as AJAX or submit callbacks.

## Upgrade

Add the appropriate trait to your form class and call the equivalent method instead of the deprecated function.

```php
// Before (all deprecated in drupal:11.4.0, removed in drupal:13.0.0):
views_ui_add_ajax_trigger($wrapping_element, $trigger_key, $refresh_parents);
views_ui_add_limited_validation($element, $form_state);
views_ui_add_ajax_wrapper($element, $form_state);
views_ui_ajax_update_form($form, $form_state);
views_ui_nojs_submit($form, $form_state);

views_ui_standard_display_dropdown($form, $form_state, $section);
views_ui_build_form_url($form_state);
views_ui_form_button_was_clicked($element, $form_state);

// After: add the trait to your class declaration and call the method.
// For AJAX helpers:
use \Drupal\views\ViewsFormAjaxHelperTrait;

$this->addAjaxTrigger($wrapping_element, $trigger_key, $refresh_parents);
static::addLimitedValidation($element, $form_state); // static
static::addAjaxWrapper($element, $form_state);       // static
$this->ajaxUpdateForm($form, $form_state);
static::noJsSubmit($form, $form_state);              // static

// For general form helpers:
use \Drupal\views\ViewsFormHelperTrait;

$this->standardDisplayDropdown($form, $form_state, $section);
$this->buildFormUrl($form_state);
static::formButtonWasClicked($element, $form_state); // static
```

Callback arrays that previously referenced function name strings must be updated to reference static methods on the consuming class:

```php
// Before:
'#submit' => ['views_ui_nojs_submit'],
'#process' => ['views_ui_form_button_was_clicked', ...],

// After:
'#submit' => [[static::class, 'noJsSubmit']],
'#process' => [[static::class, 'formButtonWasClicked'], ...],
```

### Collaboration

There was a substantive design pivot during review. claudiu.cristea explored converting the helpers to proper services to achieve dependency injection purity, but after a discussion with berdir in the MR, reverted that approach. berdir (comment [#16455333](https://www.drupal.org/node/16455333)) clarified that plugin forms have no way to access a plugin instance as a service, so some callbacks must remain static regardless. nicxvan (comment [#16455346](https://www.drupal.org/node/16455346)) noted that DI in traits had no agreed standard in core, and that the service approach produced more problems than it solved. The final design settled on protected service getters inside the traits.

godotislate worked out the removal version in consultation with @xjm (comment [#16564553](https://www.drupal.org/node/16564553)). The AJAX callback functions could arguably be removed in Drupal 12 since they are not documented public API, but consistency with the other deprecated functions in the same file and the proximity of the Drupal 12 release timeline led to setting Drupal 13.0 as the removal target for all eight functions.
