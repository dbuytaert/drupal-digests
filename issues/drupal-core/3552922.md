## Summary

Drupal core now declares `symfony/runtime` as a direct Composer dependency. The Symfony Runtime component provides a standardized, decoupled way to handle application bootstrapping outside of `DrupalKernel`, enabling bootstrap logic to be moved into swappable runtime classes. The dependency was added as a prerequisite for the upcoming Drupal CLI ([#3453474](https://www.drupal.org/node/3453474)) and for future front-controller improvements ([#3313404](https://www.drupal.org/node/3313404)). The component itself is not yet used in this change; it becomes available for those follow-on issues to build on.

## Impact

Affects site owners.

- **Requirements:** New `symfony/runtime ^8.0` (pinned to `~v8.0.8` in core) added to Drupal core; requires PHP 8.4 and Composer plugin API v2, matching existing Drupal Symfony dependencies.

## Technical details

The diff touches several composer configuration files. `core/composer.json` receives `symfony/runtime ~v8.0.8` in its `require` block. `core-recommended/composer.json` receives the same package with the looser `^8.0` constraint. Because `symfony/runtime` ships as a Composer plugin (it registers an autoload bootstrap handler), all affected `composer.json` files must also list it under `allow-plugins`. Accordingly, the package is added with `true` in `core/composer.json`'s `allow-plugins` and in `core-recommended/composer.json`'s equivalent section. `ComposerPluginsValidator::$allowedPlugins` in core is updated to include `'symfony/runtime' => "*"` so Drupal's own plugin validation does not reject it. The fake-site test fixture gains `"symfony/runtime": false` under `allow-plugins` (matching the convention that scaffold and other core plugins are explicitly disallowed there), and the accompanying test assertion in `ComposerPluginsValidatorTest::providerAllowedPlugins()` is updated to expect the new entry. No PHP runtime behavior changes with this commit.
