## Summary

Developers writing access control logic can now provide a reason when `AccessResult::allowedIf()` or `AccessResult::forbiddenIf()` return a neutral result. This improves debugging consistency, as `AccessResult::neutral()` already accepted a reason parameter.

## Impact

Affects module maintainers.

- **API change:** `AccessResult::allowedIf()` and `AccessResult::forbiddenIf()` signatures changed to add optional reason parameters and type hints

## Technical details

The `AccessResult` class provides static factory methods for creating access results. The `neutral()` method has accepted an optional `$reason` parameter since its creation, intended for developer debugging. However, the conditional methods `allowedIf()` and `forbiddenIf()` return `neutral()` when their conditions are false, and there was no way to set a reason for those neutral results.

The change adds a `$reason` parameter to `allowedIf()` that is passed through to `neutral()` when `$condition` is false. For `forbiddenIf()`, a third parameter `$neutralReason` was added (distinct from the existing `$reason` parameter that applies when the result is forbidden) that is passed to `neutral()` when `$condition` is false.

Type hints were added to all parameters: `bool` for conditions, nullable `string` for reasons. The new test `testAccessNeutralReason()` verifies that reasons can be set through all three code paths.
