## Summary

A redundant Fiber was being created on every permission check that ran inside an existing Fiber, even when checking the current user's own permissions. On a typical page render with roughly 1,000 permission checks per request, the vast majority of those checks are for the current user. This fix eliminates those unnecessary Fibers, reducing `Fiber::start` calls by approximately 90% per page and saving roughly 15–25 ms per cold render.

## Impact

No upgrade or configuration changes required.

## Technical details

The Fiber wrapping in `AccessPolicyProcessor::processAccessPolicies()` was introduced by [#3576074](https://www.drupal.org/node/3576074) to isolate `accountSwitcher->switchTo()` calls and prevent them from leaking the switched account identity to code outside the Fiber. However, an account switch only occurs in `doProcessAccessPolicies()` when the requested account differs from the current user. The original guard checked only `!\Fiber::getCurrent()`, so any call from inside a Fiber, regardless of which account was being checked, would spin up a new child Fiber.

The fix adds a second short-circuit condition: `$account->id() === $this->currentUser->id()`. When either condition is true (not in a Fiber, or checking the current user), `doProcessAccessPolicies()` is called directly without creating a `\Fiber` wrapper. A child Fiber is now only created when code is already running inside a Fiber **and** the account being checked is a different user, the only situation where the account-switch isolation actually matters.
