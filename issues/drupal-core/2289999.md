# #2289999: Add an easy way to create HTML on the fly without having to create a theme function / template

## Summary

Drupal 8.0.x gained a new `inline_template` render element that lets developers embed small Twig snippets directly inside render arrays, as an alternative to constructing HTML strings with `SafeMarkup::set()` and manually escaping variables. Because Twig's autoescape applies to the context variables, the approach is secure by default and eliminates a whole class of XSS mistakes that the old pattern invited. It is explicitly a stepping stone: the preferred long-term approach remains moving markup into dedicated template files, but `inline_template` provides a safer, themer-friendlier middle ground for cases where a full template is not yet practical.

## Impact

Affects module maintainers.

- **New API:** New `inline_template` render element with `#template` and `#context` properties, backed by `TwigEnvironment::renderInline()` and the `drupal_render_inline_template()` pre-render callback.

## Technical details

Before this change, generating small HTML fragments in PHP required calling `SafeMarkup::set()` (an internal-only API) and manually escaping every variable with `SafeMarkup::escape()` or `SafeMarkup::check()`. Forgetting the escape call was a silent XSS vulnerability, and the pattern was error-prone even for experienced contributors.

The fix introduces three coordinated pieces:

**1. The `inline_template` render element** (`core/modules/system/system.module`, `system_element_info()`). The element defines `#template` (a static Twig template string), `#context` (an associative array of variables), and a `#pre_render` callback of `drupal_render_inline_template`. The `#template` value must be a literal string constant, not a dynamically constructed one, so that static analysis tools can extract all inline templates for pre-compilation on installations that use read-only `PhpStorage`.

**2. `drupal_render_inline_template()`** (`core/includes/theme.inc`). The `#pre_render` callback retrieves the `twig` service (`TwigEnvironment`) and calls `$environment->renderInline($element['#template'], $element['#context'])`, then stores the result in `$element['#markup']`.

**3. `TwigEnvironment::renderInline()`** (`core/lib/Drupal/Core/Template/TwigEnvironment.php`). The method delegates to `$this->loadTemplate($template_string, NULL, TRUE)->render($context)`, where the new `$inline` boolean flag tells `loadTemplate()` (and the related `getTemplateClass()` and `updateCompiledTemplate()` methods) to use the `Twig_Loader_String` instance stored in `$this->stringLoader` instead of the file-based loader. The cache key is computed with `sha256` so identical template strings reuse their compiled form. A private `needsUpdate()` method was also renamed to `isFresh()` with inverted logic to match the naming convention.

Two real-world conversions were included in the commit. In `core/modules/field_ui/src/DisplayOverviewBase.php`, a manual loop that escaped summary items into a `<div class="field-plugin-summary">` string using `SafeMarkup::set()` was replaced with a single `inline_template` element using Twig's `safe_join` filter. In `core/modules/views_ui/src/ViewUI.php`, several `SafeMarkup::set('<strong>' . t('...') . '</strong>')` calls were replaced with `inline_template` elements using Twig's `{% trans '...' %}` tag.

A kernel test `TwigEnvironmentTest` was added at `core/modules/system/src/Tests/Theme/TwigEnvironmentTest.php` covering both `renderInline()` directly and `drupal_render()` of an `inline_template` element.

## Upgrade

Replace manual SafeMarkup escaping patterns with `inline_template`:

```php
// Before
$summary_escaped = '';
$separator = '';
foreach ($summary as $item) {
  $summary_escaped .= $separator . SafeMarkup::escape($item);
  $separator = '<br />';
}
$build['settings_summary'] = array(
  '#markup' => SafeMarkup::set('<div class="field-plugin-summary">' . $summary_escaped . '</div>'),
);

// After
$build['settings_summary'] = array(
  '#type' => 'inline_template',
  '#template' => '<div class="field-plugin-summary">{{ summary|safe_join("<br />") }}</div>',
  '#context' => array('summary' => $summary),
);
```

```php
// Before
$build['#markup'] = SafeMarkup::set('<span class="x">' . SafeMarkup::escape($var) . '</span>');

// After
$build['x'] = array(
  '#type' => 'inline_template',
  '#template' => '<span class="x">{{ var }}</span>',
  '#context' => array('var' => $var),
);
```

The `#template` string must be a static literal, not a dynamically constructed string, to allow pre-compilation tooling to locate all inline templates.

### Collaboration

Two design decisions during review shaped the final API. catch raised the concern that installations using read-only `PhpStorage` (where templates are pre-compiled during a staging build and deployed as static files) would have no way to discover inline templates unless they were static string literals. fabianx proposed adding a PHPDoc comment to `renderInline()` explaining the constraint, which dawehner implemented in the final patch. Separately, joelpittet suggested renaming the element type from `twig_inline` to `inline_template` and the method from `renderInlineTemplate()` to `renderInline()` for consistency; dawehner adopted both renames immediately. dries asked for concrete real-world conversions to evaluate the DX tradeoff, which led dawehner to add the `field_ui` and `views_ui` conversions that made the benefit tangible.
