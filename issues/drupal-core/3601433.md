## Summary

Multilingual sites upgrading from Drupal 11.3 to 11.4 could encounter errors in the translation update workflow because locale translation source data stored by existing sites (as plain `stdClass` objects) was no longer compatible with the `LocaleTranslationSource` class introduced by the deprecation work in [#3590050](https://www.drupal.org/node/3590050). A new post-update hook clears that stale source data automatically during the database update process, and the deprecated `locale_translation_get_status()` shim is fixed to load its required include file before delegating to the new service.

## Impact

No upgrade or configuration changes required.

## Technical details

The deprecation of `locale_translation_get_status()` (see [#3590050](https://www.drupal.org/node/3590050)) moved locale source tracking to the `LocaleSource` service, which now returns `LocaleTranslationSource` objects. Sites that had already run a translation check stored plain `stdClass` objects for locale project data in the database. After upgrading to 11.4 that stored data was loaded back as `stdClass` but code expected `LocaleTranslationSource`, causing failures.

The fix adds `locale_post_update_clear_sources()` in `locale.post_update.php`, which calls `\Drupal::service(LocaleSource::class)->clearSources()`. Because this source information is non-persistent (it is rebuilt automatically when stale), clearing it on upgrade is safe and sufficient.

A second bug was also fixed in the deprecated `locale_translation_get_status()` shim in `locale.module`: the old function loaded `locale.translation.inc` as a side effect, but the new shim did not. Tools such as Drush's `locale:update` command relied on that side effect to make `_locale_translation_default_update_options()` available. The fix adds an explicit `\Drupal::moduleHandler()->loadInclude('locale', 'inc', 'locale.translation')` call before the service delegation.
