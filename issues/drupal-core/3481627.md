## Summary

The manual user creation form now defaults to emailing new users a one-time login link to set their own password, rather than requiring administrators to manually set an initial password. When the email option is checked, the password and account status fields are hidden since they are not needed. Administrators can still uncheck the email option to manually set a password and choose the account status, supporting scenarios like training accounts or other special cases.

## Impact

No upgrade or configuration changes required.

## Technical details

The `AccountForm` class in the user module has been updated to reorder and enhance the notify checkbox behavior during user creation. The checkbox now appears earlier in the form, is checked by default when an administrator creates a user, and has been relabeled from "Notify user of new account" to "Email user with password setup instructions" for clarity.

When the notify checkbox is checked, the password and status fields are now hidden using Form API `#states`. The password field's `#required` property is removed and both fields get an invisible state triggered by the checked notify checkbox. This prevents administrators from having to enter unnecessary information when the user will set their own password via the emailed one-time login link.

The change only affects the manual user creation form accessed at `/admin/people/create`. Programmatic user creation and other registration workflows are unchanged. Tests were updated to explicitly uncheck the notify field when they need to set passwords directly, and the Nightwatch browser test helper was updated to uncheck notify before entering passwords.

## Lessons learned

- When changing form defaults that affect user-facing operations, provide both release notes and change records even if the API is marked @internal. Form alterations are common, and documentation helps site builders understand behavior changes they need to account for. *(via catch)*
- When a form field's visibility depends on another field's state, hide dependent fields dynamically rather than just changing defaults. This makes the relationship between fields explicit and prevents invalid or contradictory configurations. *(via lauriii, pameeela)*
