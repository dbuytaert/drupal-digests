## Summary

The manual user creation form now defaults to emailing new users with password setup instructions. When this option is checked, the password and account status fields are hidden, since the user will set their own password via a one-time login link. This streamlines the common workflow where administrators create accounts and let users set their own passwords, rather than manually generating and sharing passwords. Administrators can still uncheck the email option to manually set passwords and account status as before.

## Impact

No upgrade or configuration changes required.

## Technical details

The `AccountForm` class was modified to reposition and enhance the notify checkbox for user creation. The checkbox now appears earlier in the form, has been renamed from "Notify user of new account" to "Email user with password setup instructions" for clarity, and defaults to checked when administrators create new accounts.

When the notify checkbox is checked, the password and status fields are now hidden using Form API `#states`. The password field's `#required` property is removed in favor of visibility states, and the status field similarly becomes invisible. This prevents administrators from needing to enter unnecessary information when users will receive an email with a one-time login link to set their own password.

The change only affects the user creation form (`/admin/people/create`) when administrators manually create accounts. It does not change behavior for user registration, programmatic user creation, or any other user creation methods. Tests were updated to explicitly uncheck the notify checkbox when testing scenarios that require manual passwords, and the Nightwatch `drupalCreateUser` command now unchecks the notify field and waits for the password field to become visible before proceeding.

## Collaboration

This issue saw extensive discussion about default values and their implications, with xjm initially concerned about the disruption of changing defaults in a minor release, particularly around accidentally sending emails. After careful consideration and reassurance from catch and longwave that this is an acceptable UI change under the backwards compatibility policy, and recognition that email-based user creation is now the predominant workflow, consensus was reached. The issue evolved from simply changing a default value to hiding fields based on form state, which lauriii proposed with a visual mockup. pameeela drove the issue forward and integrated feedback about hiding both password and status fields. The team agreed to defer related features like automatic password generation and site-wide configuration to follow-up issues, keeping this change focused and manageable.

## Lessons learned

- When changing form defaults, the visibility of the change on the form itself mitigates disruption concerns. A checkbox with a clear label that is checked by default provides an obvious signal to users, making them less likely to unintentionally trigger unwanted behavior. *(via pameeela)*
- When deciding whether a behavior change is appropriate for a minor release versus major, consider the timing of actual adoption: if most sites will upgrade as part of a major version jump due to current version distribution, the practical impact may justify inclusion in the minor release. *(via catch)*
- When proposing opinionated UX defaults, base them on the 80% use case rather than framework flexibility concerns. Core can take a position on common workflows, and the minority use case can be served through configuration or form alters. *(via lauriii, longwave)*
