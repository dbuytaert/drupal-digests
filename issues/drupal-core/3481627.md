## Summary

The manual user creation form now defaults to emailing new users with password setup instructions. When this option is checked, the password and account status fields are automatically hidden, since users will set their own password through a one-time login link sent by email. Administrators can still uncheck the box to manually set passwords and account status when needed. This matches how most modern web platforms handle user invitations and eliminates the need to create temporary passwords that are never used.

## Impact

No upgrade or configuration changes required.

## Technical details

The `AccountForm` class in the user module has been modified to improve the user creation workflow. The notify checkbox has been moved earlier in the form, renamed to "Email user with password setup instructions" for clarity, and now defaults to checked when an administrator is creating a new account (`$admin_create` is true).

When the notify checkbox is checked, the password confirmation fields and status radio buttons are hidden using Form API `#states`. This prevents administrators from needing to enter a password that will never be used, since the user receives a one-time login link to set their own password. The status field is also hidden because notifying a user about a blocked account creates a confusing experience.

The change only affects the manual user creation form at `/admin/people/create`. Programmatic user creation through the API is unchanged. Tests in `UserRolesAssignmentTest` and the Nightwatch `drupalCreateUser` command were updated to explicitly set `notify` to `FALSE` when they need to create users with known passwords. The kernel test `UserAccountFormFieldsTest` was simplified to remove positional index checking that was no longer needed.

## Lessons learned

- When implementing conditional form fields, hiding irrelevant fields based on user choices creates clearer interfaces than showing disabled fields. If a user chooses email-based account setup, hiding manual password and status fields prevents confusion about which values will be used. *(via lauriii, pameeela)*
- When changing default form behavior in minor releases, the visibility of the change on the form itself determines disruption level. A checkbox whose default value changes but remains clearly visible and adjustable is less disruptive than hidden configuration changes, even if the behavioral outcome differs. *(via catch, longwave, pameeela)*
- When proposing behavioral changes that affect different audiences differently, defaulting to the most common use case (80%) is appropriate even when the minority use case has stronger consequences. Document the change prominently so affected users can adapt. *(via lauriii, xjm)*
