## Summary

The `X-Drupal-Dynamic-Cache` response header was showing incorrect or misleading values for 4xx and 5xx error responses. Depending on the request type, it could be absent, show `HIT`, or show `UNCACHEABLE (poor cacheability)` -- all of which were accidental leakage from an internal sub-request used to render the error page, not a reflection of what actually happened with the main request. After this fix, error responses consistently carry an accurate header: `UNCACHEABLE (403)` for JSON/non-HTML responses and `UNCACHEABLE (403, sub-request: MISS|HIT)` for HTML responses where Dynamic Page Cache did cache the underlying error page sub-request. This makes the debug header genuinely useful for diagnosing caching behavior on error pages.

## Impact

Affects module maintainers.

- **API change:** The `X-Drupal-Dynamic-Cache` response header value changes for 4xx/5xx responses: it now always reads `UNCACHEABLE ({status_code})` for non-HTML responses and `UNCACHEABLE ({status_code}, sub-request: MISS|HIT)` for HTML responses, replacing the previous inconsistent values (`HIT`, `UNCACHEABLE (poor cacheability)`, or absent).

## Technical details

Dynamic Page Cache works by attaching a request subscriber and a response subscriber to the Symfony kernel event cycle. For normal requests, the request subscriber fires first and records a policy result in `$this->requestPolicyResults`; the response subscriber then uses that result to decide whether to read from or write to the cache.

For HTML 4xx responses, the router handles the error differently: `AccessAwareRouter::checkAccess()` fires before Dynamic Page Cache's request subscriber and immediately dispatches a sub-request via `DefaultExceptionHtmlSubscriber::on403()` (and its equivalents). The sub-request gets its own full kernel cycle, Dynamic Page Cache's subscriber fires for it normally, and it sets its own `X-Drupal-Dynamic-Cache` header. That sub-request response -- complete with its cache header -- is then returned as the main response, so the main response's header reflects the sub-request's cache state rather than anything meaningful about the main request. For JSON 4xx responses there is no sub-request, so the header was simply absent.

The fix is in `DynamicPageCacheSubscriber::onResponse()`. The existing check `if (!isset($this->requestPolicyResults[$request]))` -- which detects that the request subscriber never ran for this request -- is moved to execute before any other logic. Instead of silently returning with no header, it now explicitly sets the header to `UNCACHEABLE ({status_code})`. If the response already carries an `X-Drupal-Dynamic-Cache` header from a sub-request, that value is appended: `UNCACHEABLE (403, sub-request: HIT)`. For non-HTML requests (no sub-request), only the status code appears: `UNCACHEABLE (403)`.

A test assertion in `StandardTestTrait::testDynamicPageCache()` introduced in [#2429617](https://www.drupal.org/node/2429617) was also removed. It fetched `/user/1` as a non-admin user, but instead of testing the user profile page as its comment claimed, it was testing the 403 access denied response -- and asserting `HIT` on the Dynamic Page Cache header, which was itself the sub-request leakage this issue fixes.

Test coverage was updated across `DynamicPageCacheIntegrationTest`, `BasicAuthTest`, `rest/ResourceTestBase`, `rest/EntityResourceTestBase`, `jsonapi/ResourceTestBase`, and `jsonapi/FileUploadTest` to assert the new explicit header values. Two new always-denied routes (`/dynamic-page-cache-test/cacheable-html/access-denied` and `/dynamic-page-cache-test/cacheable-json/access-denied`) and a new `cacheableJsonResponse()` controller method were added to `dynamic_page_cache_test` to cover the HTML and JSON cases directly.
