## Summary

Drupal 11.3 introduced a bug where permission checks on one user account could unexpectedly change the current user identity in unrelated parts of the page. This happened when render arrays used lazy builders and one checked permissions for a different user than the currently logged-in user. The fix ensures that temporary user switches during permission calculation stay confined to their intended scope and do not leak to other parts of the rendering process.

## Impact

No upgrade or configuration changes required.

## Technical details

The bug occurred in `AccessPolicyProcessor::processAccessPolicies()`, which temporarily switches the current user when calculating permissions for a different account. This is necessary for correct variation cache behavior. The problem arose when fibers—PHP's cooperative multitasking feature used in lazy builder rendering—suspended execution during this user switch. If a fiber suspended while entities were loading (such as roles), execution could return to a different fiber before the user was switched back, causing the wrong user identity to persist.

The fix wraps the permission calculation logic in a child fiber when already running inside a fiber. This child fiber is immediately and repeatedly resumed if it suspends, preventing execution from escaping to the parent fiber until the user switch is complete. The actual permission logic moved to a new protected method `doProcessAccessPolicies()`. When not running in a fiber, the method is called directly without overhead.

A comprehensive kernel test reproduces the issue by creating two lazy builders: one that displays the current user ID and another that checks permissions on a different user. Without the fix, the test fails because the first lazy builder incorrectly shows the second user's ID.

### Collaboration

The issue was reported by claudiu.cristea after encountering the bug in a production environment using a contrib patch that creates dummy users for access checks. The root cause was quickly identified with help from catch, who confirmed this was a core bug related to fiber suspension during user switching. godotislate created a failing test in MR 14937 that clearly demonstrated the issue and initially explored a more complex solution involving a fiber stack to track user changes per fiber. After determining that approach was too complex and not appropriate for a patch release, godotislate implemented the current fix using a child fiber to contain the user switch. The fix was tested and confirmed working on the original production site. The change was committed to 11.3.x, 11.x, and main four days after the issue was opened. A follow-up issue [#3576594](https://www.drupal.org/node/3576594) was created to explore avoiding the user switch entirely.
