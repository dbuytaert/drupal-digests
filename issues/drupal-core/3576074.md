## Summary

A critical bug in Drupal 11.3 caused the current user to unexpectedly switch to a different account during permission checks. This happened when multiple lazy builders ran in fibers and one needed to check permissions for a different user. The bug could cause content meant for one user to be displayed to another, creating serious security and privacy risks. The fix ensures that temporary user switches during permission calculations stay contained within their fiber scope and do not leak into other parts of the rendering process.

## Impact

No upgrade or configuration changes required.

## Technical details

When `AccessPolicyProcessor::processAccessPolicies()` calculates permissions for an account, it temporarily switches the current user to that account to ensure correct cache context data. In Drupal 11.3, PHP fibers were introduced for async rendering. If entity loading caused a fiber to suspend during this user switch, execution could return to the parent fiber before the user was switched back. This left the wrong user as the current user in unrelated code.

The fix wraps the permission calculation in a child fiber when already running inside a fiber. The wrapper immediately resumes the child fiber any time it suspends, preventing execution from escaping to the parent fiber until the user switch is reversed. When not running in a fiber, the code path remains unchanged. The method performing the actual work was extracted to `doProcessAccessPolicies()` and marked protected.

A comprehensive test reproduces the bug using two lazy builders: one checking permissions for user 3, another displaying the current user ID. Without the fix, the second builder incorrectly shows user 3 instead of user 2. A follow-up issue [#3576594](https://www.drupal.org/node/3576594) tracks removing the user switch entirely by improving cache context handling.

## Lessons learned

- When switching global state (like the current user) during operations that may suspend execution through fibers, wrap the state mutation in a child fiber and resume it immediately upon suspension. This prevents execution from leaving scope while global state remains temporarily changed, which would expose other code to the modified state. *(via godotislate, catch)*
- When checking permissions for an account other than the current user, avoid temporarily switching the current user account if possible. If a switch is necessary and entity loading may occur during the check, isolate the switch within a fiber to prevent the changed state from leaking to unrelated code when execution suspends. *(via godotislate, catch)*
