# #3311365: Use PHP attributes for route discovery

- **Project:** Drupal Core
- **Type:** Major task
- **Version:** main

## Summary

Module developers can now define routes directly on controller methods using Symfony's `#[Route]` PHP attribute, instead of maintaining a separate `module.routing.yml` file. The route definition and its controller live side by side in the same file, and the `defaults._controller` value is derived automatically from the annotated method. Site owners see no change; this is a new option for developers building or maintaining modules.

## Impact

Affects module maintainers.

- **Deprecation:** `RouteBuilder::__construct()` 6-argument signature (with `ModuleHandlerInterface` and `ControllerResolverInterface`) deprecated; `RouteBuilder::getRouteDefinitions()` deprecated (moved to `YamlRouteDiscovery`); `InstallerRouteBuilder` deprecated; all removed in Drupal 12.
- **New API:** Symfony's `#[Route]` attribute can now be placed on controller methods (in the `Controller` namespace) to define routes without a YAML file; a new `RoutingEvents::STATIC` event phase and `StaticRouteDiscoveryBase` abstract class were introduced for static route discovery.

## Technical details

Drupal's route building had historically relied on YAML files processed inline inside `RouteBuilder`. This issue introduces PHP attribute-based route discovery as an equally supported alternative, aligned with how modern Symfony projects define routes.

**New infrastructure.** YAML discovery is extracted out of `RouteBuilder` into its own `YamlRouteDiscovery` service. A new `RoutingEvents::STATIC` event phase is introduced; both `YamlRouteDiscovery` and the new `AttributeRouteDiscovery` service subscribe to this event to contribute static routes. A shared `StaticRouteDiscoveryBase` abstract class holds the common plumbing (route creation, `onRouteBuild` listener, `getSubscribedEvents`). The event-based design was chosen for consistency with the existing routing event model rather than a tagged-service approach (per alexpott's comment in the thread).

Two new service definitions appear in `core.services.yml`: `router.builder.attributes` (`AttributeRouteDiscovery`) and `router.builder.yaml` (`YamlRouteDiscovery`). The installer now prevents static route builders from running by clearing their `event_subscriber` tags, replacing the previous pattern of substituting the entire `RouteBuilder` class with `InstallerRouteBuilder`.

**Discovery scope.** `AttributeRouteDiscovery` reflects on all PHP classes found under a module's `src/Controller` directory. Non-class files (traits, interfaces) are silently skipped. Abstract classes are also skipped.

**Attribute usage.** The Symfony `#[Route]` attribute (`\Symfony\Component\Routing\Attribute\Route`) is used directly; no separate Drupal-specific attribute class was added. The attribute can be placed at two levels:
- On the class, to supply a path prefix and shared default values that are merged into every method-level route on that class.
- On individual methods, to define the actual route. The `name` argument is required at the method level; auto-generation of names from the class name was debated and dropped because the resulting strings are not easily discoverable without tooling.

Supported constructor arguments: `path`, `name`, `requirements`, `options`, `defaults`, `host`, `methods`, `schemes`, `priority`, and `alias`. The `condition` and `_locale` arguments are rejected with an `UnsupportedRouteAttributePropertyException`. The `format`, `utf8`, `stateless`, and `env` arguments are silently ignored. Array `path` values (localized paths) are rejected at both class and method level.

**Symfony backports.** The implementation incorporates two features from newer Symfony versions: automatic FQCN-based route aliases for invokable controllers (from Symfony 6.x), and declarative route aliasing via the `alias` argument (from Symfony 7.3). These complement related work in [#3159210](https://www.drupal.org/node/3159210) on route alias support.

**Translation.** POTX can already read PHP attributes, so `_title` strings in `#[Route]` are extractable. Plain string titles are still wrapped in `TranslatableMarkup` at runtime via `TitleResolver::getTitle`, but best practice is to write them as `new TranslatableMarkup('...')` in the attribute so POTX picks them up explicitly. The diff also adds a short-circuit in `TitleResolver::getTitle` so that a `_title` default that is already a `TranslatableMarkup` instance is used directly, skipping the raw-variables substitution path.

**Concrete example.** The `system.401` route in `system.routing.yml` is removed and replaced with a `#[Route]` attribute directly on `Http4xxController::on401()`, serving as the canonical in-tree demonstration.

**Intentional deferrals.** Support for `_form` routes (where the class implements `FormInterface`) was deferred to a follow-up, as was a file-cache layer for the reflection scan and conversion of existing core YAML routes. The `RoutingEvents::STATIC` event is the hook point for the follow-up work tracked in [#3401172](https://www.drupal.org/node/3401172).

## Upgrade

If your code directly instantiates or subclasses `RouteBuilder` and passes `ModuleHandlerInterface` and `ControllerResolverInterface` as constructor arguments (the old 6-argument signature), those arguments are now deprecated and removed in Drupal 12. Move YAML route discovery to `YamlRouteDiscovery` instead:

```php
// Before (deprecated in 11.4, removed in 12):
new RouteBuilder($dumper, $lock, $dispatcher, $moduleHandler, $controllerResolver, $checkProvider);

// After:
$yamlDiscovery = new YamlRouteDiscovery($moduleHandler, $controllerResolver);
// Register $yamlDiscovery as an event subscriber on RoutingEvents::STATIC.
new RouteBuilder($dumper, $lock, $dispatcher, $checkProvider);
```

If your code calls `RouteBuilder::getRouteDefinitions()`, move the logic to a service that subscribes to `RoutingEvents::STATIC` using `StaticRouteDiscoveryBase` as a base class.

`InstallerRouteBuilder` is deprecated in 11.4 and removed in 12 with no replacement.

To add attribute-based routes to your own module, place controllers in `src/Controller`, then annotate methods with Symfony's `#[Route]` attribute. The `name` argument is required at the method level:

```php
use Drupal\Core\StringTranslation\TranslatableMarkup;
use Symfony\Component\Routing\Attribute\Route;

class MyController extends ControllerBase {
  #[Route(
    path: '/my/path',
    name: 'mymodule.my_route',
    requirements: ['_permission' => 'access content'],
    defaults: ['_title' => new TranslatableMarkup('My Page')],
  )]
  public function myPage(): array {
    return ['#markup' => $this->t('Hello!')];
  }
}
```
