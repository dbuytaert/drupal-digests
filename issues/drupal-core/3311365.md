## Summary

Module developers can now define routes directly on controller methods using Symfony's `#[Route]` PHP attribute, instead of maintaining a separate `module.routing.yml` file. The route definition and its controller live side by side in the same file, and the `defaults._controller` value is derived automatically from the annotated method. Site owners see no change; this is a new option for developers building or maintaining modules.

## Impact

Affects module maintainers.

- **New API:** Symfony's `#[Route]` attribute can now be placed on controller methods (in the `Controller` namespace) to define routes without a YAML file; a new `RoutingEvents::STATIC` event phase was introduced for static route discovery.

## Technical details

Drupal's route building has historically relied on YAML files. This issue introduces PHP attribute-based route discovery as an equally supported alternative, aligned with how modern Symfony projects define routes.

**New infrastructure.** YAML discovery is extracted out of `RouteBuilder` into its own service. A new `RoutingEvents::STATIC` event phase is introduced; both the YAML discovery service and the new `AttributeRouteDiscovery` service subscribe to this event to contribute static routes. An `AbstractStaticRouteDiscovery` base class holds the shared plumbing. The event-based design was chosen for consistency with the existing routing event model rather than a tagged-service approach (per alexpott's comment).

**Discovery scope.** `AttributeRouteDiscovery` reflects on all classes in a module's `src/Controller` namespace. Controllers must live in that namespace to be discovered.

**Attribute usage.** The Symfony `#[Route]` attribute (`\Symfony\Component\Routing\Attribute\Route`) is used directly; no separate Drupal-specific attribute class was added. The attribute can be placed at two levels:
- On the class, to supply a path prefix and shared default values that are merged into every method-level route on that class.
- On individual methods, to define the actual route. The `name` argument is required at the method level; auto-generation of names from the class name was considered but dropped because the resulting strings are not easily discoverable without tooling.

Supported constructor arguments: `path`, `name`, `requirements`, `options`, `defaults`, `host`, `methods`, `schemes`, `priority`, and `alias`. The `condition` argument is not supported (an exception is thrown if it is set, since it has no effect even in YAML routing). The `_locale` argument is similarly rejected. The `format`, `utf8`, `stateless`, and `env` arguments are silently ignored.

**Symfony backports.** The implementation incorporates two features from newer Symfony versions: automatic FQCN-based route aliases for invokable controllers (from Symfony 6.x), and declarative route aliasing via the `alias` argument (from Symfony 7.3). These complement related work in [#3159210](https://www.drupal.org/node/3159210) on route alias support.

**Translation.** POTX can already read PHP attributes, so `_title` strings in `#[Route]` are extractable. Plain string titles are still wrapped in `TranslatableMarkup` at runtime, but best practice is to write them as `new TranslatableMarkup('...')` in the attribute so POTX picks them up explicitly.

**Intentional deferrals.** Support for `_form` routes (where the class implements `FormInterface`) was deferred to a follow-up, as was a file-cache layer for the reflection scan and conversion of existing core YAML routes. The `RoutingEvents::STATIC` event introduced here is the hook point for the follow-up work tracked in [#3401172](https://www.drupal.org/node/3401172).

### Collaboration

longwave did the decisive late-stage work on this issue in 2025, rebasing a patch that had been stalled for over a year. In that rebase longwave made the route `name` required at the method level (dropping the debated auto-generation), backported Symfony 7.3 route alias support, added test coverage for all supported attribute arguments, and addressed the `_title` translation concern. The decision to require the name was the result of a multi-contributor debate (alexpott, andypost, joachim, johnpitcairn, donquixote, aaronmchale) over whether auto-generated names from the FQCN would be good DX; the consensus was that undiscoverable magic strings would worsen the developer experience, so explicit naming won.
