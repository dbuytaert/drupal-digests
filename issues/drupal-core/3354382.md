## Summary

PHPUnit 10 requires all `@dataProvider` methods to be static, which breaks test classes that call `$this->prophesize()` inside a provider (since `ProphecyTrait::prophesize()` is an instance method). This change updates 16 test files across Drupal core to remove that incompatibility. Site owners and contrib developers are not affected directly, but the patterns introduced here serve as a reference for making their own test suites PHPUnit 10 compatible.

## Impact

No upgrade or configuration changes required.

## Technical details

PHPUnit 10 deprecated non-static `@dataProvider` methods, meaning any call to `$this->someMethod()` inside a provider triggers a deprecation. The most common blocker in core was `$this->prophesize()`, which is an instance method provided by `ProphecyTrait` (mixed into `UnitTestCase`).

The fix applies two complementary patterns depending on how many mocks a provider needs:

1. **Single-use inline**: `(new Prophet())->prophesize(SomeInterface::class)->reveal()` — used where only one mock object is needed, such as in `BigPipeResponseAttachmentsProcessorTest`, `WorkflowAccessControlHandlerTest`, and `FieldPluginBaseTest`.
2. **Shared local `Prophet` instance**: `$prophet = new Prophet(); $prophet->prophesize(...)` — used where a provider builds several related prophecies, as in `LayoutEntityHelperTraitTest`, `UserMailRequiredValidatorTest`, `DefaultHtmlRouteProviderTest`, and `OptimizedPhpArrayDumperTest`.

Beyond the Prophecy substitution, the commit makes these additional changes required by static context:

- Every affected data provider method is declared `static` (e.g. `nonHtmlResponseProvider`, `providerSectionsWithInlineComponents`, `providerTestValidate`, `getDefinitionsDataProvider`, and many others).
- Helper methods called from within data providers are also promoted to `static`, including `getServiceCall`, `getCollection`, `getParameterCall`, `getPrivateServiceCall`, and `serializeDefinition` in `OptimizedPhpArrayDumperTest` and `PhpArrayDumperTest`, and `createRequest`/`createResponse` in `ResourceResponseValidatorTest`. Call sites within the same providers are updated from `$this->method()` to `static::method()`.
- Protected instance properties used only by data providers are converted to `protected const`: `$missingRequirements` becomes `MISSING_REQUIREMENTS` in `RequirementsExceptionTest`, and `$defaultUrlOptions` becomes `DEFAULT_URL_OPTIONS` in `FieldPluginBaseTest`.
- In `DatetimeElementFormTest`, callable references of the form `[$this, 'methodName']` in a provider are changed to `[static::class, 'methodName']`.
- One `$this->createMock()` call in `FieldPluginBaseTest::providerTestRenderAsLinkWithUrlAndOptions()` is replaced with `(new Prophet())->prophesize(EntityInterface::class)->reveal()` to stay within static context.

The 16 affected files span `big_pipe`, `ckeditor5`, `jsonapi`, `layout_builder`, `migrate`, `system`, `user`, `views`, `workflows`, and core component tests under `core/tests/`. No production code is touched.

## Upgrade

This change touches only test files. No changes are needed to production code or contrib modules. Contrib developers who maintain PHPUnit tests that use `$this->prophesize()` inside a `@dataProvider` method should apply the same pattern: declare the provider `static` and replace `$this->prophesize(...)` with either `(new Prophet())->prophesize(...)` (single mock) or a local `$prophet = new Prophet(); $prophet->prophesize(...)` (multiple mocks).
