## Summary

The SQL entity query layer now generates significantly fewer joins and avoids unnecessary `GROUP BY` clauses in many common scenarios. Previously, querying a base-table property like `uuid` would join the table to itself; querying two fields from the data table would add two independent joins to the same table; following an entity reference with multiple conditions would repeat the reference join for each condition; and any entity type with a data table would unconditionally mark the query as non-simple, adding a `GROUP BY` even when only one row per entity was possible. All of these redundancies are fixed. Sites with large datasets or many translations should see measurable query-time improvements for entity queries, including those driven by JSON:API and Views.

## Impact

Affects module maintainers.

- **API change:** `ConditionInterface` requires two new methods `setQuery()` and `getQuery()`; `Query::getTables()` no longer accepts a `$sql_query` argument (deprecated `drupal:11.4.0`, removed `drupal:13.0.0`).
- **Deprecation:** `Query::getTables($sql_query)` with an argument and passing a `Condition` from a different query into `condition()` are both deprecated in `drupal:11.4.0` and removed in `drupal:13.0.0`.
- **New API:** `LanguageInterface::LANGCODE_DEFAULT` is now a valid langcode argument to entity-query `condition()` calls; it joins data tables on `default_langcode = 1` rather than a literal language match.

## Technical details

The root cause was that `Query::getTables()` created a fresh `Tables` instance on every call, so each nested `Condition` (inside AND/OR groups) got its own join-tracking state and could not deduplicate against joins already added by sibling conditions. The fix introduces `doGetTables()`, which stores and reuses a single `Tables` instance per query. All `Condition` objects that belong to the same query now share one `Tables` instance, so the `entityTables` and `fieldTables` arrays correctly prevent the same join from being added twice.

Five specific join-redundancy bugs are fixed by this shared state:

1. **Base-table self-join**: `prepare()` stores the base table name in query metadata under `entity_query_base_table`. The `Tables` constructor pre-populates `entityTables` with that entry, so a condition on a base-table column (e.g., `uuid`) resolves directly to `base_table` instead of joining the table to itself.

2. **Data-table joins**: `ensureEntityTable()` now keys entries by `$index_prefix . $base_table . '.' . $table . ($langcode ? '.' . $langcode : '')`, properly deduplicating joins regardless of which condition requests them. The old key was incorrect when the base table differed from `'base_table'`.

3. **Entity-reference joins**: `addNextBaseTable()` now caches its result in a `nextBaseTables` array keyed by `$table.$sql_column`, so following the same reference path in two conditions produces one join instead of two.

4. **`simple_query` / `GROUP BY` elimination**: `prepare()` now always starts with `simple_query = TRUE`. Inside `Tables::addField()`, the flag is only set to `FALSE` when the data-table join has no langcode restriction (a langcode-restricted join is 1-to-1, so no fan-out occurs). Queries with a specific language condition or with a single-cardinality field no longer add `GROUP BY`.

5. **Multi-value AND conditions**: A new `tableIndexPrefix` property on `Condition` ensures that each top-level `andConditionGroup()` wrapping a multi-value field gets its own index namespace. This keeps the well-documented behaviour (placing `->andConditionGroup()->condition('field', 'X')` and `->andConditionGroup()->condition('field', 'Y')` as siblings to find entities that have both values) while still deduplicating joins within a single group.

Additionally, `LanguageInterface::LANGCODE_DEFAULT` is now recognised in `addJoin()`. When it is passed for a data table or revision data table, the join condition uses `default_langcode = 1` rather than a literal language-code match, allowing callers to query against the entity's default language without knowing its code.

Two deprecations ship with this change: passing `$sql_query` to `Query::getTables()` is deprecated in `drupal:11.4.0` and removed in `drupal:13.0.0` (use `doGetTables()` or, for custom subclasses, override `doGetTables()`); passing a `Condition` created by a different query instance to another query's `condition()` is also deprecated in `drupal:11.4.0` and removed in `drupal:13.0.0`.

`ConditionInterface` gains two new required methods, `setQuery(QueryInterface $query)` and `getQuery()`, which the base `ConditionBase` class implements by propagating the query reference recursively through nested conditions.

## Upgrade

Implementations of `\Drupal\Core\Entity\Query\ConditionInterface` must add two new methods:

```php
public function setQuery(QueryInterface $query): static {
    $this->query = $query;
    foreach ($this->conditions as $condition) {
        if ($condition['field'] instanceof ConditionInterface) {
            $condition['field']->setQuery($query);
        }
    }
    return $this;
}

public function getQuery(): QueryInterface {
    return $this->query;
}
```

If your code calls `getTables()` with a `$sql_query` argument, remove it:

```php
// Before (deprecated):
$tables = $this->query->getTables($this->sqlQuery);

// After:
$tables = $this->query->doGetTables();
```

If your code reuses a condition group from one query inside a different query, stop doing so:

```php
// Deprecated pattern – $query1 and $query2 are separate instances:
$group = $query1->andConditionGroup()->condition('field', 'value');
$query2->condition($group);

// Correct – always create condition groups from the query you are building:
$group = $query2->andConditionGroup()->condition('field', 'value');
$query2->condition($group);
```
