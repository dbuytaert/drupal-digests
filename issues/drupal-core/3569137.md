## Summary

Drupal's database `Truncate` query class now officially returns nothing from its `execute()` method. Previously, `Truncate::execute()` nominally returned `int|null` (a row count or NULL), but because TRUNCATE is a DDL statement the value was unreliable and database-dependent. The return value has been dropped and the return type is now `void`. A new `QueryExecutionVoidInterface` interface has been introduced to formally express this contract. Developers whose code checked the return value of a truncate query need to remove that usage.

## Impact

Affects module maintainers.

- **API change:** `Truncate::execute()` return type changed from `int|null` to `void`; the returned row count is no longer available.
- **New API:** New `QueryExecutionVoidInterface` declares `public function execute(): void` for DDL/DML queries that produce no return value.

## Technical details

The fix has three parts. First, a new `QueryExecutionVoidInterface` (in `core/lib/Drupal/Core/Database/Query/QueryExecutionVoidInterface.php`) declares `public function execute(): void` for queries whose execution produces no meaningful return value. Second, `Drupal\Core\Database\Query\Truncate` now implements this interface: the `execute()` method gains a native `void` return type, the call to `$statement->rowCount()` is removed, and the fallback `return NULL` is removed. Third, the PostgreSQL-specific `Drupal\Driver\Database\pgsql\Truncate` override is updated the same way: it no longer captures or returns the result of `parent::execute()`. As a side effect, the `abstract protected function execute()` declaration was also removed from the base `Query` class, where it had been documented as returning `StatementInterface|null`; that abstract contract is no longer present.

## Upgrade

Remove any code that reads the return value of `Truncate::execute()`:

```php
// Before
$count = $connection->truncate('my_table')->execute();
if ($count !== NULL) { ... } // unreliable; don't do this

// After
$connection->truncate('my_table')->execute();
// No return value; treat as void
```

If you type-hint against query objects that should not return a value, you may now use `QueryExecutionVoidInterface`.

### Collaboration

The initial approach proposed in the MR introduced a new execution method alongside `execute()`, which drew pushback from catch (#6) who questioned why the existing method's return type couldn't simply be corrected. mondrake explained (#7) that PHP does not allow `void` in a union type, so narrowing the return type on `execute()` in the base `Query` class would be impossible while other implementations still return values. After alexpott (#13) revisited catch's concern and suggested leaving the API docs as-is and returning NULL, mondrake proposed (#14) introducing two distinct interfaces instead. The final MR applied that narrower interface approach only to `Truncate`, which catch then accepted and committed.
