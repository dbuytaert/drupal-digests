## Summary

A bug in entity revision caching caused revisions loaded in the same batch to bleed cache tags across entities. Clearing the revision cache for one entity would unnecessarily invalidate cached revisions of unrelated entities loaded in the same call, producing extra cache misses. The fix ensures each revision cache entry carries only the tags that belong to its own entity.

## Impact

No upgrade or configuration changes required.

## Technical details

The bug lived in `ContentEntityStorageBase::setPersistentRevisionCache()`. That method populates the persistent revision cache via `$this->cacheBackend->setMultiple()`. Before the fix, `$cache_tags` was initialized once to `['entity_field_info']` outside the `foreach` loop and then extended with each entity's `{entityTypeId}:{entityId}:revisions` tag inside the loop. Because the array was never reset between iterations, tags accumulated: the first entity's revision got only its own tag, but the second entity's revision also received the first entity's tag, the third received all previous tags, and so on. Invalidating entity A's revisions therefore also flushed cached revisions of any other entity that happened to be loaded in the same `loadMultipleRevisions()` call.

The fix inlines the tags array directly inside the loop body so that every iteration constructs a fresh two-element array: `['entity_field_info', "{$this->entityTypeId}:{$entity->id()}:revisions"]`. No cross-entity contamination is possible.

`ContentEntityCacheTest::testCacheInvalidationOnSave()` in `core/tests/Drupal/KernelTests/Core/Entity/ContentEntityCacheTest.php` was extended to cover this scenario. The `$assert_cache_exists` helper now accepts an optional `$expected_cache_tag` argument and asserts the exact tag set on both the persistent `entity` cache backend and the `entity.memory_cache` service. A new block resets all caches, calls `loadMultipleRevisions()` with revisions from two different entities, and then checks that each revision's cache entry carries only its own entity-scoped tag. It then saves one revision and verifies the other remains cached and unaffected.

catch independently found the same bug and opened [#3580049](https://www.drupal.org/node/3580049), confirming the issue independently.
