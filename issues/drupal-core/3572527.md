## Summary

Drupal generates a version hash to track when installed packages change. This hash was inconsistent because Composer does not always return packages in the same order. Identical sets of packages could produce different hashes, breaking cache invalidation and causing unnecessary rebuilds. This particularly affected multi-architecture Docker builds and other scenarios where packages are installed multiple times. The fix ensures packages are always sorted before calculating the hash, making the result predictable and reliable.

## Impact

No upgrade or configuration changes required.

## Technical details

The `\Drupal\DrupalInstalled::VERSIONS_HASH` constant is generated by `DrupalInstalledTemplate::getCode()` during Composer operations. This hash is calculated from all installed packages and used to invalidate caches when dependencies change.

The bug occurred because `\Composer\Repository\RepositoryInterface::getPackages()` does not guarantee a consistent sort order. Running `composer remove` and `composer require` on the same package could reorder the list, changing the hash even when the final package set was identical.

The fix adds a `usort()` call in `DrupalInstalledTemplate::getCode()` that sorts packages by their unique name using the spaceship operator before passing them to `array_reduce()`. This ensures the concatenated version string used for hashing always follows the same order.

A new test `DrupalInstalledTest` was added to verify hash consistency. The test installs packages, adds two fixtures, removes one, then re-adds it, confirming the hash returns to its previous value when the package set is restored.
