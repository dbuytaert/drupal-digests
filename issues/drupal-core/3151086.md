## Summary

The `path_alias` module previously used the term "whitelist" for an internal performance optimization that caches the first URL segment of every path that has a known alias. The term was replaced throughout with "prefix list", which more accurately describes what the structure contains. This landed in Drupal 11.1.0 with full backward-compatibility deprecations so existing code continues to work until Drupal 12.0.0.

## Impact

Affects module maintainers.

- **Deprecation:** `AliasWhitelist`, `AliasWhitelistInterface`, service `path_alias.whitelist`, and `AliasManager::pathAliasWhitelistRebuild()` deprecated in drupal:11.1.0; replaced by `AliasPrefixList`, `AliasPrefixListInterface`, service `path_alias.prefix_list`, and `pathAliasPrefixListRebuild()`, removed in drupal:12.0.0

## Technical details

The `path_alias` module maintains a bootstrap-cached list of path prefixes (the first segment of each aliased system path, e.g. "node" for `/node/12345`). Before a database lookup is attempted in `AliasManager::getAliasByPath()`, the prefix of the requested path is checked against this list. If the prefix is absent, the lookup is skipped entirely. This list was previously called the "whitelist" and lived in the `AliasWhitelist` class; the rename to "prefix list" better conveys the purpose.

The commit introduced two new files: `AliasPrefixList` (a `CacheCollector` subclass containing the full implementation, moved verbatim from `AliasWhitelist`) and `AliasPrefixListInterface` (extending `CacheCollectorInterface`). The old `AliasWhitelist` was reduced to `class AliasWhitelist extends AliasPrefixList {}` with a `@deprecated in drupal:11.1.0` docblock, and `AliasWhitelistInterface` became `interface AliasWhitelistInterface extends AliasPrefixListInterface {}` similarly deprecated.

In `path_alias.services.yml` the existing `path_alias.whitelist` service was marked deprecated and a new `path_alias.prefix_list` service was registered pointing to `AliasPrefixList`. The `path_alias.manager` service definition was updated to inject `@path_alias.prefix_list`. The bootstrap cache key used by the collector changed from `path_alias_whitelist` to `path_alias_prefix_list`; this affects the semaphore lock name and the cache entry read by `AliasManager`.

Inside `AliasManager` itself, the constructor parameter `$whitelist: AliasWhitelistInterface` was renamed to `$pathPrefixes: AliasPrefixListInterface`. The protected method `pathAliasWhitelistRebuild()` was deprecated with a `@trigger_error` call and a new `pathAliasPrefixListRebuild()` method was added as the canonical replacement. `cacheClear()` now delegates to `pathAliasPrefixListRebuild()`.

All kernel, unit, and functional tests were updated to use `AliasPrefixList`, the new cache key, and the renamed methods. The `StandardPerformanceTest` expected semaphore query strings were updated to reference `path_alias_prefix_list`.

Early in the thread, neclimdul (comment #4) raised the BC risks in detail: because `AliasWhitelistInterface` is injectable and `AliasManager`'s constructor is typed against it, a simple rename would break any contrib or custom code overriding either the service or implementing the interface. This analysis shaped the final approach of retaining the old names as deprecated thin wrappers rather than removing them.

## Upgrade

Modules or custom code that reference the old names must be updated before Drupal 12.0.0:

```php
// Old (deprecated in 11.1.0):
use Drupal\path_alias\AliasWhitelist;
use Drupal\path_alias\AliasWhitelistInterface;

// New:
use Drupal\path_alias\AliasPrefixList;
use Drupal\path_alias\AliasPrefixListInterface;
```

Service references in YAML or PHP:

```yaml
# Old:
arguments: ['@path_alias.whitelist']

# New:
arguments: ['@path_alias.prefix_list']
```

Subclasses that override `pathAliasWhitelistRebuild()` in `AliasManager` should rename the method:

```php
// Old (deprecated in 11.1.0):
protected function pathAliasWhitelistRebuild($path = NULL) { ... }

// New:
protected function pathAliasPrefixListRebuild($path = NULL) { ... }
```

Cache entries stored under the old key `path_alias_whitelist` will not be read by the new code. The `path_alias_prefix_list` key is used instead; no manual migration is needed as the cache collector rebuilds on a cold start.
