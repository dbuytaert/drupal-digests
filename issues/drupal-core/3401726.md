## Summary

Sites that added contextual filters to the core `media_library` Views display (for example, to restrict the media picker to items owned by the current user) saw those filters correctly applied on the initial widget load, but ignored when switching between the Grid and Table display modes. Clicking those display-mode buttons issued a new AJAX request that bypassed all user-added contextual filter arguments, showing unfiltered results instead. The fix corrects this in all supported branches (10.6.x, 11.x, 11.3.x, main).

## Impact

No upgrade or configuration changes required.

## Technical details

`MediaLibraryUiBuilder::buildMediaLibraryView()` in `core/modules/media_library/src/MediaLibraryUiBuilder.php` is responsible for rendering the media library widget view. It builds a local `$args` array containing only the selected media type ID from `$state->getSelectedTypeId()`, then calls `$view_executable->preExecute($args)` and `$view_executable->execute($display_id)` in sequence. After `execute()`, the view has already resolved all configured contextual filters (including any extras added by site builders), and the full argument set is stored internally in `$view_executable->args`.

The bug was on the final line:

```php
return $view_executable->buildRenderable($display_id, $args, FALSE);
```

Passing the narrow `$args` array (only the entity type ID) to `buildRenderable()` caused it to re-initialize the view's argument handling with that single value, discarding the additional contextual filter arguments that `execute()` had already computed. The initial widget load was unaffected because it fetched results correctly through `execute()`. However, clicking the Grid or Table toggle buttons triggered a fresh AJAX call through the same `buildMediaLibraryView()` path, and the resulting `buildRenderable()` call stripped out the extra filter arguments, returning unfiltered results.

The fix is to pass an empty array instead:

```php
return $view_executable->buildRenderable($display_id, [], FALSE);
```

Because `execute()` has already set `$view_executable->args`, passing `[]` to `buildRenderable()` leaves the computed argument state intact, so all contextual filters are honored across display-mode switches and "Apply filters" submissions.

A new FunctionalJavascript test class `MediaLibraryContextualFilterTest` (in `core/modules/media_library/tests/src/FunctionalJavascript/`) was added to cover the regression. Its `setUp()` uses `View::load('media_library')` and `$executable->addHandler()` to attach a `media_field_data.uid = current_user` contextual argument to both the `widget` and `widget_table` displays programmatically (avoiding a slower HTTP request to the Views UI). The test then logs in as a second user, opens the media library widget, and asserts that the correct count of two items is shown after switching between grid and table displays, after submitting the exposed filters, and on final insert. The related duplicate issue [#3142777](https://www.drupal.org/node/3142777) tracks the same root cause.

### Collaboration

herved had previously hit this bug and worked around it with a service decorator. In comment #18 they diagnosed that any AJAX request inside the modal (including "Apply filters" and "Insert selected") produced a `Drupal.AjaxError` parseerror in the console because the form ID changed, causing an HTML response where JavaScript expected JSON. They also suggested constructing the Views handler programmatically in the test rather than through an HTTP request to the Views UI, a change that reduced test execution time from roughly 22 seconds to 16 seconds and was adopted in the final version.
