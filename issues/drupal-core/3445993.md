## Summary

The navigation module gained a new `NavigationLinkBlock` plugin that lets a single link be placed in the navigation sidebar without the overhead of creating a dedicated menu. The block is intentionally hidden from the general Block layout UI and the Navigation layout UI to prevent a proliferation of one-off link blocks, but it can be placed programmatically. The Help link, which was previously hardcoded in the navigation footer template, was migrated to be an instance of this new block type, removing the special-case logic that had checked for the Help module at render time.

## Impact

Affects module maintainers.

- **New API:** New `navigation_link` block plugin (`NavigationLinkBlock`) for placing a single link in the navigation sidebar, intentionally hidden from the Block and Navigation layout UIs and intended for programmatic placement only.

## Technical details

Before this change, adding a single link to the navigation required creating a full menu, adding a menu item, and configuring a menu block with specific depth settings. The Help link in the navigation footer was handled differently: `UserLazyBuilder::renderNavigationLinks()` passed a `#help` flag (driven by `ModuleHandlerInterface::moduleExists('help')`) into the `menu_region__footer` theme hook, which then conditionally rendered a hardcoded anchor to `help.main` in `menu-region--footer.html.twig`.

The fix introduces `NavigationLinkBlock` (`core/modules/navigation/src/Plugin/Block/NavigationLinkBlock.php`), a `final` class extending `BlockBase` with plugin id `navigation_link`. Its configuration stores three values: `title` (link text), `uri` (stored in Drupal's internal/entity URI format), and `icon_class` (a CSS class name validated against `/^[a-z0-9_-]+$/`). The `build()` method resolves the URI via `Url::fromUri()`, checks access using `$url->access(NULL, TRUE)` to capture cacheable access metadata, and returns a `#theme => 'navigation_menu'` render array when access is granted.

The block's `blockForm()` reuses URI handling logic borrowed directly from `LinkWidget` (the `getUserEnteredStringAsUri()` and `getUriAsDisplayableString()` static helpers), with a `@todo` pointing at [#3450518](https://www.drupal.org/node/3450518) for a future extraction into a shared trait.

The `navigation_link` block id is added to the hidden-block list in `navigation_block_alter()` in `navigation.module`, keeping it out of both the standard Block layout and the Navigation layout UI.

The Help link was migrated to a `navigation_link` block instance placed in the `footer` region with `uri: 'internal:/admin/help'` and `icon_class: help` in `core/modules/navigation/config/install/navigation.block_layout.yml`. The `#help` variable was removed from the `menu_region__footer` theme definition, `UserLazyBuilder` lost its `ModuleHandlerInterface` dependency, and the conditional Help markup was removed from the footer Twig template.

Config schema for `block.settings.navigation_link` was added to `navigation.schema.yml`, including a `Regex` constraint on `icon_class` and a `FullyValidatable` marker, consistent with the work in [#3379725](https://www.drupal.org/node/3379725) to make block config entities fully validatable. Functional test coverage is in `NavigationLinkBlockTest`, which verifies page-cache and dynamic-page-cache behavior and confirms that access control suppresses the link for users without the required permission.
