## Summary

The `Datelist` form element was storing the current timezone inside its cached `getInfo()` return value, meaning the timezone resolved at cache-warm time (which could be UTC when warmed via Drush/CLI, or the site timezone when warmed via the browser) was frozen into the cache and reused for all subsequent requests. This caused date values submitted through a `datelist` widget to be interpreted in a randomly varying timezone depending on how caches had been flushed, leading to silently incorrect stored dates. The fix moves the `#date_timezone` default out of `getInfo()` and into `valueCallback()`, where it is evaluated fresh on every request.

## Impact

No upgrade or configuration changes required.

## Technical details

The `Datelist` element class (`core/lib/Drupal/Core/Datetime/Element/Datelist.php`) extends `DateElementBase` and registers its default render-array properties via `getInfo()`. Because Drupal caches the output of `getInfo()`, any value computed there (such as a call to `date_default_timezone_get()`) is captured at cache-warm time and reused for the lifetime of the cache entry. On a server whose PHP/OS timezone is `UTC` but whose Drupal site timezone is `America/New_York`, warming the cache via Drush (which does not apply the site timezone event listener) would freeze `#date_timezone` as `UTC`, while warming via the web UI would freeze it as `America/New_York`. Downstream code in `valueCallback()` uses `$element['#date_timezone']` to construct `DrupalDateTime` objects, so the mismatch silently corrupted stored dates.

The identical problem was already fixed for the `Datetime` element in [#3087606](https://www.drupal.org/node/3087606), where `#date_timezone` was likewise moved out of `getInfo()` and into `valueCallback()`.

The fix mirrors that prior resolution exactly:
- In `getInfo()`: remove the `'#date_timezone' => date_default_timezone_get()` line and add a comment explaining that the property is intentionally absent because `getInfo()` is cached and the value must vary per-request (and potentially per-user).
- In `valueCallback()`: add `$element += ['#date_timezone' => date_default_timezone_get()];` as the first statement. Using `+=` means an explicitly supplied `#date_timezone` on the element definition is preserved, while the default is applied lazily and correctly at request time.

A kernel test `testTimezoneHandling()` was added to `core/tests/Drupal/KernelTests/Core/Datetime/DatelistElementFormTest.php`. It renders the form once under the site timezone (`Australia/Sydney`), asserts the resolved `#date_timezone`, then calls `date_default_timezone_set('UTC')` to simulate a different runtime context, renders again, and asserts `UTC` is now used. This directly demonstrates the bug (which was reproducible before the fix) and guards against regression.
