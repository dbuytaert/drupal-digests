## Summary

A bug in `RouteProcessorCsrf::processOutbound()` caused REST and JSON:API responses that embed CSRF-protected URLs (for example, a logout link) to be cached without varying by session. Two users sharing the same role or other common cache context could receive each other's CSRF tokens from the cache. The fix is targeted at Drupal 11.4.0 (committed to main and 11.x). Sites on 11.3.x are not yet patched in an official release.

## Impact

Affects site owners.

- **Security:** CSRF tokens generated for non-HTML requests (REST, JSON:API) were not varied by session, allowing a cached response to serve one user's token to another user sharing the same role or other common cache context.

## Technical details

The non-HTML branch of `processOutbound()` generates a real CSRF token via `$this->csrfToken->get($path)` and injects it as a `?token=` query parameter. The behavior of returning a real token (rather than a render placeholder) was introduced by [#3485174](https://www.drupal.org/node/3485174). That change did not also call `$bubbleable_metadata->addCacheContexts(['session'])`, so callers assembling a cacheable REST or JSON:API response never received `session` among the required vary contexts. The HTML branch was already correct: its lazy-builder render callback declared `'#cache' => ['contexts' => ['session']]` (lines 79-81 of `RouteProcessorCsrf`).

The production fix is a single line added immediately after the token is set: `$bubbleable_metadata?->addCacheContexts(['session'])`. The nullsafe operator means the call is a no-op when `$bubbleable_metadata` is null, preserving existing behavior for callers that do not pass metadata.

Test changes mirror the real-world impact. Several REST and JSON:API functional test classes now override `getExpectedCacheContexts()` to include `session`. `LinksetControllerTest::testUserAccountMenu()` adds `session` to expected cacheability because the account menu exposes the CSRF-protected `/user/logout` URL for authenticated users. A new `normalizeHeadersForGetHeadComparison()` hook was introduced in the REST test base and overridden in `BasicAuthResourceTestTrait` to neutralize `?token=` values in `Link` headers: with stateless basic_auth, the CSRF seed is regenerated on every request, so HEAD and GET legitimately produce different token strings even after the fix. The unit test in `RouteProcessorCsrfTest` gained a `CacheContextsManager` stub (required for `BubbleableMetadata::addCacheContexts()` to pass its internal validation) and a direct assertion that `['session']` is present in the returned metadata.
