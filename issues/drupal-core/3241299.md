## Summary

Submitting a `datelist` form element with non-numeric text in any date part (year, month, or day configured via `#date_text_parts`) used to produce a PHP warning instead of a clean validation error. The fix ensures that invalid strings are properly rejected before PHP's `checkdate()` is called, so the expected `InvalidArgumentException` is thrown through the normal validation path. The change is applied to both `main` and `11.x`; a follow-up ([#3566186](https://www.drupal.org/node/3566186)) also made `checkArray()` protected.

## Impact

No upgrade or configuration changes required.

## Technical details

The `DateTimePlus::checkArray()` method in `core/lib/Drupal/Component/Datetime/DateTimePlus.php` validates date parts before creating a `DateTime` object. Prior to this fix, the method passed the year, month, and day values from the input array straight to PHP's `checkdate()` without first verifying they were integers. When the `datelist` form element's `#date_text_parts` option is used, those slots come in as raw strings; `checkdate()` then emitted a "expects parameter 1 to be int, string given" warning, and validation silently fell through instead of returning `FALSE` to trigger the `InvalidArgumentException` in the caller `createFromArray()`.

The fix replaces the single `if (!empty(...))` guard with a `foreach` loop over `['year', 'month', 'day']`. Each key is checked with `empty()` and then with `filter_var($array[$key], FILTER_VALIDATE_INT) === FALSE`. If either condition fails for any key, `checkArray()` returns `FALSE` immediately, before `checkdate()` is ever reached. `createFromArray()` already throws `InvalidArgumentException` on a `FALSE` return, so no changes are needed there.

A subtle behavior difference noted by alexpott: on the old code, passing a non-whole float like `1975.5` would cause a PHP deprecation (implicit int conversion) but still produce a date. With `FILTER_VALIDATE_INT`, such a value returns `FALSE` and throws an exception, while a whole-valued float like `1975.0` still passes. This was considered acceptable since passing a non-integer to day, month, or year makes no sense.

Three new test cases (`'Non-numeric year'`, `'Non-numeric month'`, `'Non-numeric day'`) were added to `providerTestCheckArray()` in `DateTimePlusTest.php`. The test uses `ReflectionMethod` to reach `checkArray()` which was subsequently made `protected` in follow-up [#3566186](https://www.drupal.org/node/3566186).
