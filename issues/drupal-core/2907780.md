## Summary

The procedural field-purging functions in the Field module are deprecated in Drupal 11.2 and replaced by a new `entity_field.purger` service living in core's Entity Field API. The `purge_batch_size` setting is also moved from the `field.settings` configuration object to `system.entity_field_settings`, with an automatic database update to migrate existing values. Cron-triggered purging moves from the Field module to the System module. For most site owners the change is transparent: the update hook handles the config migration and purging continues to run on cron as before.

## Impact

Affects module maintainers and site owners.

- **Database update:** `system_update_11103()` migrates `purge_batch_size` from `field.settings` to `system.entity_field_settings` and deletes the old config object.
- **Config change:** `field.settings` config object removed; `purge_batch_size` moved to `system.entity_field_settings`.
- **Deprecation:** `field_purge_batch()`, `field_purge_field()`, and `field_purge_field_storage()` deprecated in drupal:11.2.0 and removed in drupal:12.0.0; use `FieldPurger::purgeBatch()`, `purgeFieldDefinition()`, and `purgeFieldStorageDefinition()` respectively.
- **New API:** New `entity_field.purger` service (`FieldPurger` implementing `FieldPurgerInterface`) for purging deleted field data, registered in core.

## Technical details

Field purging logic had always lived in the Field module's `field_purge.inc` as three standalone procedural functions (`field_purge_batch()`, `field_purge_field()`, `field_purge_field_storage()`), making it inaccessible to other parts of the Entity Field API that need to purge base fields and bundle fields outside the configurable-field system. The fix introduces `core/lib/Drupal/Core/Field/FieldPurger.php`, implementing the new `FieldPurgerInterface`, and registers it as the `entity_field.purger` service (with `Drupal\Core\Field\FieldPurgerInterface` as an alias in `core.services.yml`). `FieldPurger` is injected with `EntityTypeManagerInterface`, `DeletedFieldsRepositoryInterface`, `ModuleHandlerInterface`, and `LoggerChannelFactoryInterface`. The three procedural functions in `field_purge.inc` are reduced to thin wrappers that emit an `E_USER_DEPRECATED` trigger and forward to the service; they are deprecated in `drupal:11.2.0` and scheduled for removal in `drupal:12.0.0`. `FieldPurger::purgeFieldDefinition()` and `FieldPurger::purgeFieldStorageDefinition()` are marked `protected` so subclasses can override the per-item logic while the public `purgeBatch()` remains the intended entry point. The Field module's `#[Hook('cron')]` implementation that called `field_purge_batch()` was removed entirely; equivalent logic was added to the System module's `SystemHooks::cron()`, reading the batch size from the new `system.entity_field_settings` config key. The `field.settings` config object is removed from Field module config; `system_update_11103()` copies `purge_batch_size` to `system.entity_field_settings` and deletes the old object. `ConfigImporterFieldPurger` was updated to read `purge_batch_size` from `system.entity_field_settings`. `FieldConfigDeleteForm` in `field_ui` was updated to inject `FieldPurgerInterface` via the service container instead of calling the procedural function. Tests throughout core were updated to use `\Drupal::service('entity_field.purger')` directly. A follow-up issue ([#3587460](https://www.drupal.org/node/3587460)) tracks fully removing the now-empty `field.settings` config object in Drupal 13. The migration pattern for config-to-settings follows the approach established in [#3571593](https://www.drupal.org/node/3571593).

## Upgrade

```php
// Before (deprecated in drupal:11.2.0, removed in drupal:12.0.0):
field_purge_batch($batch_size);
field_purge_field($field);
field_purge_field_storage($field_storage);

// After — use the service directly or inject FieldPurgerInterface:
\Drupal::service('entity_field.purger')->purgeBatch($batch_size);
\Drupal::service('entity_field.purger')->purgeFieldDefinition($field);
\Drupal::service('entity_field.purger')->purgeFieldStorageDefinition($field_storage);
```

If your code reads `field.settings:purge_batch_size` from configuration, update it to read `system.entity_field_settings:purge_batch_size`. The `system_update_11103()` update hook migrates the value automatically for existing sites.
