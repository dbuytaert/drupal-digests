## Summary

The global procedural functions `hide()` and `show()` defined in `core/includes/common.inc` are deprecated as of Drupal 10.1.0 and will be removed in Drupal 11.0.0. Code that calls these functions will trigger a deprecation notice. The replacement is to set the render array property directly: use `$element['#printed'] = TRUE` in place of `hide($element)`, and `$element['#printed'] = FALSE` in place of `show($element)`. The one core usage of these functions, in the file field preprocess function, has been updated to use the direct property assignment. The conceptual documentation about how the `#printed` flag controls rendering traversal has been moved from the individual function docblocks into `RendererInterface::render()`, where it more accurately belongs.

## Impact

Affects module maintainers.

## Technical details

Both `hide()` and `show()` in `core/includes/common.inc` simply flip the `#printed` boolean property on a render array element. The `#printed` flag acts as a semaphore: when TRUE, the renderer skips that element and its children on subsequent render passes. Because the functions are nothing more than thin wrappers around a one-line property assignment, the deprecation has no replacement function; callers are directed to inline the assignment. The `@trigger_error()` call in each function points to change record https://www.drupal.org/node/3261271.

The docblock explaining the `#printed` caching and traversal semantics (the "first time render() is called" explanation) was stripped from the two deprecated functions and added to `RendererInterface::render()` in `core/lib/Drupal/Core/Render/RendererInterface.php`, since that is where the behavior is actually implemented.

The sole core callsite, `template_preprocess_file_widget_multiple()` in `core/modules/file/file.field.inc`, was updated to use `$widget[$sub_key]['#printed'] = TRUE` and `$widget['display']['#printed'] = TRUE` in place of `hide()`, and `$operations_elements[$key]['#printed'] = FALSE` in place of `show()`. A new unit test class `ProceduralApiDeprecationTest` in `core/tests/Drupal/Tests/Core/Render/Element/` verifies both deprecation messages fire and that the functions still return the expected array shapes during the transition window.

## Upgrade

Replace every call to the global `hide()` and `show()` functions with a direct property assignment:

```php
// Before
hide($element['some_child']);
show($element['some_child']);

// After
$element['some_child']['#printed'] = TRUE;
$element['some_child']['#printed'] = FALSE;
```

Note: if you are hiding a form element to remove it from output, using `$element['#access'] = FALSE` is often more appropriate, as it also prevents form submission of that element.

### Collaboration

longwave raised a substantive design question late in the review: the original plan was to move `hide()` and `show()` to static methods on `\Drupal\Core\Render\Element`, but longwave noted that with Twig and its `without()` filter now covering the themer use case, and most contrib usage actually being form-element hiding that should use `#access` instead, there was no strong reason to keep the methods as public API at all. andypost picked up the point, updated the MR to deprecate without replacement, and moved the `#printed` traversal documentation to `RendererInterface::render()`. The shift from "move to a class" to "deprecate with no replacement" emerged directly from this exchange.
