## Summary

When contrib modules or event subscribers throw an `AccessDeniedException` before a route has been matched (for example in a `KernelEvents::REQUEST` subscriber), Drupal would produce a fatal "Call to a member function hasRequirement() on null" error instead of a clean 403 response. The fix is now in Drupal 11.x and 11.3.x.

## Impact

No upgrade or configuration changes required.

## Technical details

The crash occurred in `CsrfExceptionSubscriber::on403()` in `core/lib/Drupal/Core/EventSubscriber/CsrfExceptionSubscriber.php`. The method retrieved the route via `RouteMatch::createFromRequest($request)->getRouteObject()`, which returns `null` when no route has been matched, then immediately called `$route->hasRequirement('_csrf_token')` without a null check. The fix adds `is_null($route)` as the first condition in the guard clause so the method returns early when there is no matched route. A new unit test `CsrfExceptionSubscriberTest::testOn403WithNullRouteDoesNothing()` was added in `core/tests/Drupal/Tests/Core/EventSubscriber/CsrfExceptionSubscriberTest.php` to cover this path. During the review, a concern was raised that the fix could create an infinite 403-redirect loop when the CSRF confirm form route itself triggers a 403; the null-route guard addresses the reported crash without touching the redirect logic.

### Collaboration

sourav_paul caught a potential infinite redirect loop before the MR was filed: with a naive earlier patch, a 403 on a CSRF-protected route could redirect to `_csrf_confirm_form_route`, which could trigger another 403, and so on until memory was exhausted. rgpublic proposed adding a `!$event->isMainRequest()` guard to break the cycle. The final MR, opened by sourav_paul, uses the simpler `is_null($route)` guard that fixes the reported null-dereference crash without altering the redirect behaviour.
