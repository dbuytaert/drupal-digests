## Summary

When Drupal processes a text field through a filter format, individual filter plugins can attach JavaScript libraries and CSS via `FilterProcessResult`. If you accessed the processed text via `$entity->text_field->processed` and manually built a render array from it, those attachments were silently lost. Only cache metadata (tags, contexts, max-age) was accessible through `CacheableDependencyInterface`. The `#attachments` were trapped inside the internal `FilterProcessResult` object with no public way to retrieve them. This fix makes `TextProcessed` also implement `AttachmentsInterface`, exposing `getAttachments()`, `addAttachments()`, and `setAttachments()` so callers can now collect and apply filter-added libraries to their render arrays.

## Impact

Affects module maintainers.

- **New API:** `TextProcessed` now implements `AttachmentsInterface`, exposing `getAttachments()`, `addAttachments()`, and `setAttachments()` on the computed processed text typed data object.

## Technical details

`TextProcessed` (in `core/modules/text/src/TextProcessed.php`) now declares `implements CacheableDependencyInterface, AttachmentsInterface` and pulls in `AttachmentsTrait` for the default `addAttachments()` and `setAttachments()` implementations. The key addition is a custom `getAttachments()` override that first calls `$this->getValue()` to trigger the filter pipeline (populating the internal `$this->processed` `FilterProcessResult` object), then calls `$this->addAttachments($this->processed->getAttachments())` to merge the filter's bubbled attachments into the `TextProcessed` instance's own attachment storage before returning it. Without this step, `FilterProcessResult`'s attachments were only reachable by the render pipeline when the text was rendered through a `#type => 'processed_text'` render element; direct access via the computed property provided no route to them. Two tests were added: a unit test (`TextProcessedTest`) verifying `addAttachments()` merges correctly, and a kernel test (`TextProcessedAttachmentsTest`) using the `filter_test_assets` filter to confirm that a real filter's library (`filter/caption`) appears in `getAttachments()` results after processing.

## Upgrade

Modules or themes that manually render formatted text fields can now collect attachments alongside cache metadata:

```php
$processed = $entity->text_field->processed;

$cache = new BubbleableMetadata();
$cache->addCacheableDependency($processed);
// New: collect filter-plugin JS/CSS attachments.
$cache->addAttachments($processed->getAttachments());
$cache->applyTo($a_render_array);
```

The workaround of using a `#type => 'processed_text'` render element (which lets the render pipeline bubble attachments automatically) remains valid and continues to work unchanged.
