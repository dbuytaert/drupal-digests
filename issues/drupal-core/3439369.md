## Summary

Drupal 11 removed several deprecated APIs from the migration system that had been marked for removal since Drupal 9.5 and 10.1. Site owners running custom migration code or contrib modules that used the old `trackLastImported` methods, passed the `$expand` argument to `getMigrationDependencies()`, or called `Sql::getMigrationPluginManager()` will need to update that code before upgrading to Drupal 11.

## Impact

Affects module maintainers.

- **API change:** `MigrationInterface` lost `isTrackLastImported()`, `setTrackLastImported()`, and `getTrackLastImported()`; `Migration::getMigrationDependencies()` no longer accepts an `$expand` parameter and always returns expanded plugin IDs; `Sql::getMigrationPluginManager()` removed and `$migration_plugin_manager` constructor argument is now required.
- **Deprecation:** Three `trackLastImported` methods removed from `Migration`/`MigrationInterface` (no replacement); `Sql::getMigrationPluginManager()` removed (use `$this->migrationPluginManager`); passing any argument to `getMigrationDependencies()` is now deprecated targeting drupal:12.0.0.

## Technical details

The commit cleaned up three files in `core/modules/migrate` by removing code deprecated for drupal:11.0.0, plus handled one case that required a more careful two-major deprecation dance.

**`Sql.php` (id_map, deprecated since drupal:9.5.0):** The null-fallback in `Sql::__construct()` that silently resolved `$migration_plugin_manager` via `\Drupal::service('plugin.manager.migration')` when the argument was omitted is gone; the argument is now required. The protected `getMigrationPluginManager()` accessor method (which existed solely to provide a deprecated call path to `$this->migrationPluginManager`) is also removed. The test file `SqlDeprecationTest.php` and the corresponding unit test method `testGetMigrationPluginManagerDeprecation()` in `MigrateSqlIdMapTest` were deleted. The `getMigrationPluginManager()` override in `TestSqlIdMap` was removed as well.

**`Migration.php` and `MigrationInterface.php` — `trackLastImported` (deprecated since drupal:10.1.0, no replacement):** The `$trackLastImported` property, the constructor check for the `trackLastImported` plugin-definition key, and the three methods `isTrackLastImported()`, `setTrackLastImported()`, and `getTrackLastImported()` are removed from both `Migration` and `MigrationInterface`. The corresponding legacy test `testGetTrackLastImported()` and `testTrackLastImportedDeprecation()` were removed.

**`Migration.php` — invalid `migration_dependencies` format (deprecated since drupal:10.1.0):** The `@trigger_error` warnings for malformed `migration_dependencies` arrays in `__construct()` and `set()` are gone. The validation logic was retained but now throws `InvalidPluginDefinitionException` directly, which was already the behavior inside `getMigrationDependencies()` itself.

**`getMigrationDependencies()` — `$expand` parameter removal (requires two-major dance):** The old signature `getMigrationDependencies(bool $expand = FALSE)` was replaced with `getMigrationDependencies()`. Previously calling without the argument (the default `FALSE`) triggered a drupal:10.1.0 deprecation because it returned non-expanded dependencies; calling with `TRUE` was the correct path. Now the method always expands plugin IDs (the old `$expand = TRUE` path), which made the `$expand` parameter meaningless. However, since PHP does not raise an error for passing extra positional arguments to a function that no longer declares them, `alexpott` pointed out the need for the two-major deprecation dance: the commit adds a `func_num_args() > 0` check that issues a new deprecation targeting drupal:12.0.0 if any argument is passed. All internal callers in `MigrationPluginManager` and `migrate_drupal`'s `ReferenceBase` were updated from `getMigrationDependencies(TRUE)` to `getMigrationDependencies()`. A `@todo` comment in `MigrationPluginManager::createInstances()` notes the loop that sets `migration_dependencies` can be removed once the 12.0.0 deprecation is cleaned up. The related deprecation issues referenced in the thread are [#3282894](https://www.drupal.org/node/3282894) (trackLastImported), [#3277306](https://www.drupal.org/node/3277306) (Sql constructor), and [#3266691](https://www.drupal.org/node/3266691) (migration_dependencies / getMigrationDependencies).

## Upgrade

If your code calls any of the following deprecated APIs, update it as shown:

```php
// REMOVED: trackLastImported methods on MigrationInterface / Migration
// Before:
$migration->setTrackLastImported(TRUE);
$migration->getTrackLastImported();
$migration->isTrackLastImported();
// After: remove these calls entirely — there is no replacement.

// REMOVED: Sql::getMigrationPluginManager()
// Before (in a Sql id_map subclass):
$manager = $this->getMigrationPluginManager();
// After:
$manager = $this->migrationPluginManager;

// REMOVED: Sql::__construct() null fallback for $migration_plugin_manager
// The argument is now required; ensure you pass it when constructing Sql subclasses.

// CHANGED: getMigrationDependencies() always expands plugin IDs now
// Before (passing TRUE to get expanded IDs):
$deps = $migration->getMigrationDependencies(TRUE);
// After:
$deps = $migration->getMigrationDependencies();

// If you were calling getMigrationDependencies() without an argument and
// relying on the non-expanded result, there is no equivalent — the method
// now always returns expanded IDs.
```
