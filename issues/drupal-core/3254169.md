# #3254169: Update aborted by system_post_update_sort_all_config. Drush command terminated abnormally. Output is empty.

## Summary

Upgrading to Drupal 9.3.0 could fail during `drush updatedb` or `update.php` at the `system_post_update_sort_all_config` step, producing errors such as "Drush command terminated abnormally. Output is empty." or a PHP memory exhaustion fatal. The root cause was that the post-update hook loaded and saved every configuration item in a single PHP execution. Sites with hundreds or thousands of config files would exhaust available memory. The fix, committed via [#3254403](https://www.drupal.org/node/3254403), converted the function to process configuration in batches controlled by the `entity_update_batch_size` setting. Some failures could not be resolved by batching alone because individual contrib modules or themes (notably Barrio and Purge) had configuration that caused memory spikes or schema errors during save; the documented workaround in those cases was to temporarily uninstall those extensions and reinstall them after the update completed.

## Impact

No upgrade or configuration changes required.

## Technical details

`system_post_update_sort_all_config()` was introduced in Drupal 9.3.0 in `core/modules/system/system.post_update.php` to re-save all stored configuration so it would be sorted according to its schema. The original implementation called `\Drupal::configFactory()->listAll()` and then iterated through every returned name with `getEditable($name)->save()`, loading every config object into memory before the function returned. On installations with large numbers of config items (one reporter had over 1,900), this caused `TypedConfigManager`-triggered memory exhaustion, which Drush surfaced as "Drush command terminated abnormally. Output is empty." because its subprocess produced no JSON output.

The fix in [#3254403](https://www.drupal.org/node/3254403) added a `&$sandbox` parameter to convert the function into a standard Drupal batched post-update. On the first invocation the sandbox is initialized with the full list of config names and a `max` count. Each subsequent pass processes a slice of `Settings::get('entity_update_batch_size', 50)` names and sets `$sandbox['#finished']` to the fraction complete, allowing the batch API to resume across multiple HTTP requests or Drush subprocess calls. This is the same pattern used by `ConfigEntityUpdater` and `SqlFieldableEntityTypeListenerTrait` elsewhere in core. The fix was committed to 9.3.x, 9.4.x, and 10.0.x. The function was later removed for Drupal 10.0.0 and is now listed in `system_removed_post_updates()`.

Even after batching was applied, some sites continued to fail because a single config object from a contrib module or theme (Barrio, Purge, Gutenberg, link_target) caused a memory spike or a schema validation error when `getEditable()->save()` was called for it. Those were contrib-side issues; the workaround was to uninstall the offending extension before running database updates.
