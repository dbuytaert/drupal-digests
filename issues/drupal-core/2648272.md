## Summary

Drupal's user registration form was setting a plaintext password as a dynamic property directly on the User entity object (`$account->password = $pass`) so that password tokens could be generated in notification emails. Storing plaintext passwords this way is a security risk: the value could leak if the object is serialized or if contrib code handles it incorrectly. This change deprecates reading the `password` property on User entities in Drupal 11.4.0, with removal scheduled for Drupal 12.0.0. Contrib modules that were silently relying on this undocumented property to access the plaintext password at registration time must now find an alternative approach.

## Impact

Affects module maintainers.

- **Deprecation:** `$account->password` (reading the plaintext password dynamic property on `User` entities) is deprecated in Drupal 11.4.0 and removed in Drupal 12.0.0.

## Technical details

The `RegisterForm` (and historically the equivalent Drupal 7 `user.module` code) set `$account->password = $pass` immediately before calling account-save so that password-token replacement in notification emails could access the plaintext value. Drupal 8 dropped the `!password` token that made this meaningful, but the setter and the undocumented dynamic property remained. Several contrib modules (noted in comment #22 and by `mr.baileys`) had grown a silent dependency on `$account->password` being populated at registration time.

The fix adds a `__get($name)` override to `core/modules/user/src/Entity/User.php` (the `User` entity class, which extends `ContentEntityBase`). When any caller reads the `password` property by name it triggers `E_USER_DEPRECATED` with the message: "Getting the password property is deprecated in drupal:11.4.0 and is removed from drupal:12.0.0. See https://www.drupal.org/node/3569185". The actual property value is still returned (via `parent::__get($name)`) so behaviour is unchanged during the deprecation window. Note that only the read path is deprecated here; the setter code that writes `$account->password` is left for the followup.

A kernel test `testPasswordProperty()` was added to `core/modules/user/tests/src/Kernel/UserEntityTest.php`, using the `#[IgnoreDeprecations]` attribute so the test can assert the exact deprecation message without itself failing on it.

Removal of both the setter in `RegisterForm` and the `__get` override was tracked in followup issue [#3571296](https://www.drupal.org/node/3571296), which targets Drupal 12.0.0.

## Upgrade

If your module reads `$account->password` on a User entity to obtain the plaintext password during registration, you will receive a deprecation notice starting in Drupal 11.4.0, and the property will be unavailable in Drupal 12.0.0.

There is no direct replacement exposed by Drupal core. The Registration Password Token contrib module provides a supported mechanism for exposing the password in registration emails if that use case is required.

Example code that will trigger the deprecation:

```php
// Deprecated - triggers E_USER_DEPRECATED in 11.4, breaks in 12.0.
$plaintext = $account->password;
```

Remove any code that reads this property and refactor your module to avoid depending on it.
