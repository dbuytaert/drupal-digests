## Summary

After Drupal's installation process hardens the `sites/default` directory (setting permissions to read-only), subsequent Composer commands that trigger the scaffold plugin would throw a `RuntimeException` and abort, sometimes leaving `composer.json` and `composer.lock` out of sync. The fix makes the scaffold plugin automatically and temporarily loosen permissions on the destination directory before writing a scaffold file, then restore the original permissions when the operation completes. Sites that added `file-mapping` workarounds in `composer.json` to skip scaffolding of `default.settings.php` and `default.services.yml` no longer need those workarounds.

## Impact

No upgrade or configuration changes required.

## Technical details

The fix is applied to `ReplaceOp::process()` in `composer/Plugin/Scaffold/Operations/ReplaceOp.php`. Before removing and replacing a scaffold file, the method now captures the original `fileperms()` values of both the destination directory and the destination file (if they exist). If the destination directory is not writable, it calls the new protected `makeWritable(string $filepath): bool` method, which applies `chmod($path, ($mod & 0777) | 0200)` to add the owner write bit. The actual remove-and-copy (or symlink) operation then runs inside a `try/finally` block that restores the captured original permissions on both the directory and the file, regardless of whether the scaffold write succeeded or failed. If `makeWritable()` itself returns false (the running process lacks sufficient OS privileges to change permissions), a `RuntimeException` is thrown with an interpolated message identifying the path. The symlink path bypasses the post-write file-permission restoration since the symlink inherits source permissions. `makeWritable()` is added as a protected method on the base class shared with `AppendOp`, making it available for future use in other operations.

Test infrastructure in `FixturesBase::io()` gains a `bool $reset = FALSE` parameter to allow flushing the `BufferIO` between assertions in a single test. A new test case in `ReplaceOpTest` verifies that `process()` succeeds when both the target file and its parent directory have been chmod'd to non-writable, that file content is correctly updated, that the restrictive permissions are preserved after the operation, and that the symlink variant does not alter permissions.

## Upgrade

No action required. Sites using the `file-mapping` workaround in `composer.json` to skip scaffolding of `default.settings.php` or `default.services.yml` may remove those entries if the sole purpose was to avoid the permissions failure:

```json
// These workaround entries are no longer necessary:
"file-mapping": {
    "[web-root]/sites/default/default.services.yml": false,
    "[web-root]/sites/default/default.settings.php": false
}
```
