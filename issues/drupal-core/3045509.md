## Summary

Bundle fields defined in code via `hook_entity_bundle_field_info()` were invisible to `EntityFieldManager::getFieldMap()`, which caused subsystems that rely on the field map (Views data building, Layout Builder, JSON:API, and others) to be unaware of those fields. The fix makes the field map authoritative and correct by building it from the actual field definitions rather than a separate, stale key/value store. Sites that define computed or other programmatic bundle fields should see those fields properly recognized across core subsystems without any manual intervention.

## Impact

No upgrade or configuration changes required.

## Technical details

The field map in `EntityFieldManager::getFieldMap()` was built in two passes. The first pass iterated entity type base fields directly. The second pass read a key/value collection (`entity.definitions.bundle_field_map`) that was maintained by `FieldDefinitionListener::onFieldDefinitionCreate()` and `::onFieldDefinitionDelete()`, which only fire for config-backed fields managed by the field module. Bundle fields declared purely in code via `hook_entity_bundle_field_info()` never triggered those listeners, so they were permanently absent from the map.

This design was originally introduced in [#2482295](https://www.drupal.org/node/2482295) specifically to avoid calling `hook_entity_bundle_field_info()` across all entity types and bundles on cold caches, because `field_entity_bundle_field_info()` issued a database query per bundle. Two subsequent optimizations ([#3537863](https://www.drupal.org/node/3537863), [#3538006](https://www.drupal.org/node/3538006)) consolidated those queries so that per-bundle field definition loading is no longer prohibitively expensive.

The diff removes the entire key/value read block from `getFieldMap()` and replaces it with a loop that calls `$this->getFieldDefinitions($entity_type_id, $bundle)` for every bundle of every fieldable entity type, then populates `$this->fieldMap` from the returned definitions. This collapses the two-pass approach into one and ensures all field types, including code-defined bundle fields, appear in the map.

The fix is deliberately narrow so it can be backported to 11.3.x as a patch release. The key/value write path in `FieldDefinitionListener` is intentionally left in place for now; its removal, along with associated constructor argument deprecations, is tracked in the follow-up [#3585986](https://www.drupal.org/node/3585986).

Test changes reflect the structural shift. Two unit tests (`testGetFieldMap` and `testGetFieldMapByFieldType`) were deleted: they tested the key/value retrieval implementation detail directly and required mocking `KeyValueStoreInterface`. The `EntityTestBundleInfo::entityBundleInfo()` helper was extended to include the `entity_test_update` provider so that bundles it manages are actually present when field definitions are resolved, uncovering a latent bug that the key/value layer had been masking. Views integration test assertions were updated for the new bundle iteration order (`article, page` rather than `page, article`). Performance test baseline counts shifted slightly, with a small reduction in database queries and near-neutral change in cache operations, matching catch's xhprof profile showing essentially the same wall time and memory footprint.
