## Summary

When multiple modules are installed at once, Drupal groups them into batches and rebuilds the dependency injection container between batches. A module with `container_rebuild_required = true` in its info file was being placed in the same batch group as the modules that preceded it, meaning the container was only rebuilt after the group finished and not before that specific module ran. This could leave the module installing against a stale container. The fix ensures such modules are always installed in their own isolated batch group, so the container is rebuilt both immediately before and immediately after that module is installed.

## Impact

No upgrade or configuration changes required.

## Technical details

The multi-module install batching logic lives in two places: `ModuleInstaller::install()` in `core/lib/Drupal/Core/Extension/ModuleInstaller.php` and `install_profile_modules()` in `core/includes/install.core.inc`. Both iterate over modules and assign them to numbered groups (`$module_groups[$index]`); each group is processed as one batch step, after which the container is rebuilt. The `container_rebuild_required` info key signals that a module needs the container rebuilt immediately after its installation.

Before this fix, the grouping logic first appended the module to the current group, then incremented `$index` if `container_rebuild_required` was true. This meant a module that needed a rebuild could share its group with previously queued modules, so the container was only rebuilt after all of them installed, never before the problematic module specifically.

The fix adds a pre-increment step: if `container_rebuild_required` is true and the current group is already non-empty (`!empty($module_groups[$index])`), `$index` is incremented before the module is added. The module then lands in a fresh, isolated group. The existing post-increment is retained, so the next module also starts in a new group. Together these two increments guarantee a container rebuild both before and after any module that requires it.

The test data provider `containerRebuildRequiredProvider()` in `ModuleInstallerTest` was updated to reflect the increased rebuild counts that result from this stricter isolation; for example, `[false, true, false]` now produces 3 rebuild groups instead of 2.

This issue is closely related to [#3416522](https://www.drupal.org/node/3416522), which introduced the multi-module batching mechanism in the first place, and to [#3492235](https://www.drupal.org/node/3492235), which changed the default value of `container_rebuild_required` to false so that most modules do not trigger this isolation path at all.
