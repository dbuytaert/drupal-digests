## Summary

Several tests that cover search module functionality were living in other modules (views, node, page_cache, system). This task reorganization moves or extracts those tests into the search module so that search-related test coverage is colocated with the code being tested. There is no change to any production code; this is purely a test file housekeeping improvement.

## Impact

No upgrade or configuration changes required.

## Technical details

The change relocates search-specific tests from several host modules into `core/modules/search/tests/`:

- `Drupal\Tests\views\Functional\SearchIntegrationTest` and `SearchMultilingualTest` were moved to `Drupal\Tests\search\Functional\views\ViewsIntegrationTest` and `ViewsMultilingualTest` respectively, and their `#[Group]` attribute changed from `views` to `search`.
- `testSearchIndexRebuildOnTranslationDeletion` was extracted from `NodeTranslationUITest` (node module) and placed into a new, standalone `SearchNodeTranslationUITest` class under `Drupal\Tests\search\Functional`.
- The `search_page` cacheability data-provider case was removed from `Drupal\Tests\system\Kernel\Entity\EntityOperationsCacheabilityTest` and given its own subclass `Drupal\Tests\search\Kernel\SearchEntityOperationsCacheabilityTest`.
- `NodeSearchTest` was moved from `Drupal\Tests\node\Kernel` to `Drupal\Tests\search\Kernel\node`, with its `#[Group]` attribute updated to `search`.
- `PageCacheTagsIntegrationTest` (page_cache module) had its `search` module dependency, the `search content` permission grant, and the `config:search.settings` and `CACHE_MISS_IF_UNCACHEABLE_HTTP_METHOD:form` cache tag assertions removed, since those were only needed to exercise search coverage that now lives in the search module.
- The PHPStan baseline (`phpstan-baseline.neon`) was updated to reflect the new class fully-qualified names for the two `cronRun()` return-type suppressions.

The initial commits introduced namespace mismatches that were not caught by the MR pipeline but surfaced on 11.x backport runs of a separate issue ([#3390193](https://www.drupal.org/node/3390193)). The change was reverted and recommitted twice before the namespaces were corrected and the patch landed cleanly.
