## Summary

Dot-prefixed container parameters such as `.hook_data` (~45 KB) and `.theme_hook_data` (~10 KB) are used only to pass data between compiler passes during container compilation, but they were never being stripped from the final cached container definition. This caused the serialized container to be unnecessarily large. Registering Symfony's `RemoveBuildParametersPass` in Drupal's compilation pipeline fixes this, reducing the cached container from roughly 549 KB to 482 KB (about 10%) on a standard Umami install, with a corresponding improvement in the time needed to load and unserialize the container on every request.

## Impact

No upgrade or configuration changes required.

## Technical details

In `CoreServiceProvider::register()`, Drupal now adds `new RemoveBuildParametersPass()` with `PassConfig::TYPE_REMOVE`. This is the Symfony-idiomatic mechanism for build-only parameters: any parameter whose name begins with `.` is treated as temporary and stripped at the remove phase of compilation.

The root cause had two layers. First, Drupal's compilation pipeline never registered `RemoveBuildParametersPass` at all. Second, even Symfony's own default `PassConfig` would not have helped here because it registers `RemoveBuildParametersPass(true)`, where the `true` flag tells it to preserve array-valued parameters. Both `.hook_data` and `.theme_hook_data` are arrays (introduced by [#3506930](https://www.drupal.org/node/3506930) and [#3544715](https://www.drupal.org/node/3544715) respectively), so they would have survived the default pass anyway. Drupal's fix uses `new RemoveBuildParametersPass()` without that flag, so array build parameters are also removed.

A new `testBuildParametersRemoved()` kernel test in `DrupalKernelTest` asserts that no dot-prefixed parameters exist in the compiled container, preventing future regressions. nicxvan confirmed the test catches the bug by running the suite without the fix and observing both parameters appear in the failure output.

### Collaboration

The design question of whether to use Symfony's compiler pass or manually delete the two known parameters was settled in the thread. nicxvan argued for the compiler pass on the grounds that it handles any future dot-prefixed parameters automatically, and also required use of `PassConfig::TYPE_REMOVE` instead of a hardcoded priority (-2048) and a regression test before marking the issue RTBC. godotislate then identified why Symfony's own default `RemoveBuildParametersPass` had never cleaned these up: Symfony passes `true` to preserve array-valued parameters, and both `.hook_data` and `.theme_hook_data` are arrays. That observation confirmed the fix needed to omit the `true` argument.
