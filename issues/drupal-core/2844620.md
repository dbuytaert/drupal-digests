## Summary

When `http.response.debug_cacheability_headers` is set to `true` in `development.services.yml`, Drupal emits `X-Drupal-Cache-Tags` and `X-Drupal-Cache-Contexts` response headers listing every cache tag and context that bubbled up to the page. On complex pages with many paragraphs, taxonomy terms, or nodes, these headers could easily exceed 8190 bytes, the hardcoded per-header-line limit in Apache. Apache would silently return a 500 error with no PHP-level exception and no Drupal log entry, producing a blank page that was extremely difficult to diagnose. This fix makes Drupal automatically split the header value across multiple lines with the same header name whenever the value exceeds 8000 bytes, preventing the server error.

## Impact

No upgrade or configuration changes required.

## Technical details

The fix lives in `FinishResponseSubscriber::addDebugCacheHeaders()`, a new protected method that replaces the previous inline `$response->headers->set()` calls for the two debug headers. It sorts the cache tags or contexts, joins them with a single space (the existing separator), and then uses `wordwrap()` to break the resulting string into lines no longer than the new `RESPONSE_HEADER_LINE_MAX_LENGTH = 8000` constant. The resulting array is passed to `$response->headers->set($header, $headers)`, which Symfony's `ResponseHeaderBag` emits as multiple header lines with the same name. A companion constant `RESPONSE_HEADER_CACHE_ITEM_SEPARATOR = ' '` documents the space-separated format (RFC 9110 allows multiple same-name fields, which a receiver is supposed to treat as comma-joined, but these debug headers intentionally use spaces for readability). Apache merges repeated header lines into one with `, ` between them; nginx sends them as separate lines. Either way the server no longer hits the per-line byte limit. The change has no effect when `http.response.debug_cacheability_headers` is `false`, which is the default in production.
