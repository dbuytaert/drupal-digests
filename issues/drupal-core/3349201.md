## Summary

On sites with BigPipe enabled, logged-in users viewing 403 or 404 error pages could see blocks that BigPipe had placeholder-deferred render with wrong context. For example, the language switcher block would show links to the front page instead of correct language-variant error-page URLs. The root cause is that Drupal handles configured error pages via an internal subrequest, but BigPipe resolves its placeholders after that subrequest ends, in the main request context where the error-page routing information is absent. The fix prevents BigPipe from accepting any placeholders when executing inside a subrequest, so those blocks fall back to eager rendering with the correct context.

## Impact

No upgrade or configuration changes required.

## Technical details

Drupal handles configured 403 and 404 error pages by dispatching an internal subrequest via `HttpKernel::handle()`, which pushes an additional entry onto the `RequestStack`. BigPipe's `BigPipeStrategy::processPlaceholders()` collects render placeholders for deferred streaming delivery, but the actual placeholder rendering happens after the full response is assembled, outside any subrequest context. This means a block placeholder collected during a subrequest is later rendered without that subrequest's routing information, producing incorrect output (the language switcher, for instance, consulted the main request's route and generated front-page links).

The fix adds an early-return guard at the top of `BigPipeStrategy::processPlaceholders()` in `core/modules/big_pipe/src/Render/Placeholder/BigPipeStrategy.php`: if `$this->requestStack->getParentRequest()` returns a non-null value, the method immediately returns `[]`, refusing all placeholders. This causes any block rendered inside a subrequest to be rendered eagerly and synchronously while the subrequest's context is still active.

This guard is only correct because of the prior fix in [#2613044](https://www.drupal.org/node/2613044), which eliminated a long-standing bug where every request was pushed onto the `RequestStack` twice but popped only once. Before that correction, `getParentRequest()` would always return a value for ordinary main requests too, and this guard would have completely disabled BigPipe site-wide. As berdir noted in review, this dependency means the fix cannot be backported to 11.3.

`BigPipeStrategyTest::testProcessPlaceholders()` in `core/modules/big_pipe/tests/src/Unit/Render/Placeholder/BigPipeStrategyTest.php` is extended with an optional `bool $sub_request` parameter. The mock for `RequestStack::getParentRequest()` now returns either null or a request object depending on this flag. A new data-provider case, `_no_big_pipe absent, session, no-JS cookie present, sub-request`, verifies that `processPlaceholders()` returns an empty array even when all the other BigPipe preconditions are satisfied.

Before that fix, every main request would have had a parent, and the new guard would have silently disabled BigPipe for all users.
