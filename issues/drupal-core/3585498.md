## Summary

Alter hook implementations defined directly inside Kernel test classes were silently skipped whenever `ModuleHandler::alter()` was called with multiple related hook types (for example `['form', 'form_my_form']`). The bug affected only test code, not production sites. After this fix, Kernel test classes can reliably implement alter hooks alongside regular hooks without needing a separate test module.

## Impact

No upgrade or configuration changes required.

## Technical details

Issue [#3502432](https://www.drupal.org/node/3502432) introduced the ability to write hook implementations as methods on a `KernelTestBase` subclass. Those implementations are registered under the pseudo-module key `'core'`. When `ModuleHandler::alter()` is invoked with more than one hook type, it calls `reOrderModulesForAlter()` to produce a canonical ordering. That method computed `array_intersect(array_keys($this->moduleList), $modules)`, which filtered out any module name not in the installed module list. Because `'core'` is never in `$this->moduleList`, every alter hook implemented on a Kernel test class was silently dropped from the combined listener list and never called.

The fix changes the expression to `array_intersect(array_keys($this->moduleList + ['core' => '']), $modules)`. Adding `'core'` as a trailing entry in the array union means it survives the intersect while still sorting last, matching the order that `KernelTestCompilerPass` uses when registering test hook implementations. A new test class `KernelTestHooksTest` (in `core/tests/Drupal/KernelTests/Core/Hook/`) verifies that both regular and alter hook methods on a Kernel test class are executed correctly, including the multi-type alter case that previously failed.
