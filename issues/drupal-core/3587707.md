## Summary

On cold-cache page requests, `ThemeRegistry` was performing an unnecessary extra cache read and write during its own construction, on top of what the parent `CacheCollector` class already handles. This has been removed. Sites will see slightly fewer database queries and cache operations on cold-cache requests, with no change in visible behavior.

## Impact

No upgrade or configuration changes required.

## Technical details

Before this change, `ThemeRegistry::__construct()` hard-set `$this->cacheLoaded = TRUE` and manually issued a `$this->cache->get()` to populate `$this->storage`, bypassing `CacheCollector`'s lazy-loading mechanism entirely. It also called `$this->updateCache()` at the end of construction, causing an immediate cache write (with a lock acquire and release against the semaphore table) every time the theme registry was instantiated without a warm cache.

The fix replaces the manual constructor logic with a call to `$this->lazyLoadCache()`, the standard `CacheCollector` method. The full custom `updateCache()` override (about 25 lines covering lock acquisition, cache merging, and write) was also removed; `ThemeRegistry::updateCache()` now just calls `parent::updateCache($lock)`. The only logic retained in the constructor is the initialization of `keysToPersist` for each registry key when the storage is empty after the lazy load.

As a side effect, the semaphore lock key for the theme registry changes from `theme_registry:runtime:<theme>:Drupal\Core\Utility\ThemeRegistry` to `theme_registry:runtime:<theme>:Drupal\Core\Cache\CacheCollector`, matching the pattern used by all other `CacheCollector` subclasses. Performance test fixtures in `NodePageCacheTest`, `FrontPageTest`, and the Umami performance tests were updated to reflect the reduced query and cache operation counts (typically 2 fewer queries, 1 fewer cache get, and 1 fewer cache set per cold-cache request).

### Collaboration

catch discovered the redundancy by running `dump($this->getCacheOperations())` against the node cold-cache performance test and observing the theme registry cache entry being read at least three times and written twice per request. He noted that the pre-emptive cache write was added in 2013 and that `CacheCollector` had since accumulated sufficient race-condition protection to make it unnecessary. He also identified that the full `updateCache()` override was superfluous and simplified it to a `parent` call in the same pass.
