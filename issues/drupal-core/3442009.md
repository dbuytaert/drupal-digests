## Summary

Drupal 11.1 ships an attribute-based OOP hook system alongside the existing procedural system. Module developers can now implement hooks as class methods annotated with `#[Hook('hook_name')]` in classes placed under their module's `src/Hook/` directory. These classes are auto-discovered at container build time and registered as autowired services, so no YAML registration is needed in the common case. One module can have multiple OOP hook implementations of the same hook in different classes, which makes it easy to split large `.module` files and inject only the services each implementation needs. Procedural hook implementations continue to work without any changes. Two `ModuleHandlerInterface` methods, `writeCache()` and `getHookInfo()`, are deprecated in drupal:11.1.0 and will be removed in drupal:12.0.0.

## Impact

Affects module maintainers.

- **API change:** `ModuleHandlerInterface` no longer extends `DestructableInterface`; `ModuleHandler` constructor now takes `EventDispatcherInterface` and `array $hookImplementationsMap` instead of `CacheBackendInterface`; `writeCache()` and `getHookInfo()` are deprecated in drupal:11.1.0 and removed in drupal:12.0.0
- **Deprecation:** `ModuleHandlerInterface::writeCache()` and `ModuleHandlerInterface::getHookInfo()` deprecated in drupal:11.1.0, removed in drupal:12.0.0; see https://www.drupal.org/node/3442349
- **New API:** New `#[Hook]` and `#[LegacyHook]` attributes in `Drupal\Core\Hook\Attribute\`, `HookCollectorPass` compiler pass, and `ProceduralCall` service enable OOP hook implementations discovered from a module's `src/Hook/` directory and dispatched via `drupal_hook.$hookname` events

## Technical details

The implementation introduces `HookCollectorPass`, a Symfony DI compiler pass that runs at container build time. It scans each module's `src/Hook/` directory using PHP `ReflectionClass` to find `#[Hook]` attributes, and scans `.module`, `.inc`, `.profile`, and `.install` files using `StaticReflectionParser` to find procedural function names matching the `modulename_hookname` pattern. All collected implementations are registered as `kernel.event_listener` services tagged with the event name `drupal_hook.$hookname`. A `hook_implementations_map` container parameter is written, mapping `[hook][class][method]` to the implementing module name.

The `ProceduralCall` class (`core/lib/Drupal/Core/Extension/ProceduralCall.php`) is an `@internal` wrapper that allows procedural hook functions to participate in the event listener system alongside OOP implementations. At runtime, `ModuleHandler::getHookListeners(string $hook)` queries the event dispatcher for `drupal_hook.$hook` listeners and builds a per-hook `invokeMap` of callables keyed by module. `invokeAllWith()`, `invoke()`, and `alter()` all use this map. A `legacyInvoke()` fallback checks for procedural functions not annotated with `#[LegacyHook]` and calls them directly; this prevents double execution when both a procedural stub and an OOP implementation exist in the same codebase.

`ModuleHandler`'s constructor now takes `EventDispatcherInterface $eventDispatcher` and `array $hookImplementationsMap` instead of `CacheBackendInterface $cache_backend`. The old `getImplementationInfo()`, `buildImplementationInfo()`, `verifyImplementations()`, and the `implementations`/`verified`/`hookInfo`/`alterFunctions` caches are entirely removed. `ModuleHandlerInterface` no longer extends `DestructableInterface`. The `alter()` method's multi-hook logic is simplified through a new `reOrderModulesForAlter()` helper, preserving `hook_module_implements_alter()` compatibility.

Install hooks (`hook_install`, `hook_schema`, `hook_update_N`, `hook_post_update_NAME`, `hook_module_preinstall`, `hook_module_preuninstall`, `hook_modules_installed`, `hook_modules_uninstalled`, `hook_uninstall`, `hook_update_last_removed`) and theme preprocessing hooks (`hook_preprocess_HOOK`, `hook_process_HOOK`) must remain procedural. `ModuleInstaller` now uses its own internal `invoke()` and `invokeAll()` methods for install-time hooks, calling procedural functions directly and bypassing `ModuleHandler` entirely. A small set of hooks may not have multiple OOP implementations per module: `hook_library_build_info`, `hook_mail`, `hook_help`, and `hook_node_update_index`.

## Upgrade

OOP hooks are opt-in. Existing procedural hook implementations require no changes.

**To write a new OOP hook implementation:**

```php
// In modules/mymodule/src/Hook/MyModuleHooks.php
namespace Drupal\mymodule\Hook;

use Drupal\Core\Hook\Attribute\Hook;

class MyModuleHooks {

  #[Hook('user_cancel')]
  public function userCancel(array $edit, object $account, string $method): void {
    // implementation
  }

}
```

The class is auto-discovered and registered as an autowired service. No YAML is needed unless autowire is insufficient.

**To support both Drupal 10 and Drupal 11 simultaneously**, keep the procedural function but mark it `#[LegacyHook]` so Drupal 11 skips it and calls the OOP version instead:

```php
use Drupal\Core\Hook\Attribute\LegacyHook;

#[LegacyHook]
function mymodule_user_cancel(array $edit, object $account, string $method): void {
  \Drupal::service(MyModuleHooks::class)->userCancel($edit, $account, $method);
}
```

**Callers of `ModuleHandlerInterface::writeCache()` or `getHookInfo()`** should remove those calls; both methods are no-ops deprecated in drupal:11.1.0 and removed in drupal:12.0.0.

**Custom `ModuleHandlerInterface` implementations** must update the class to no longer depend on `DestructableInterface` and must update constructor injection: `CacheBackendInterface $cache_backend` is replaced by `EventDispatcherInterface $eventDispatcher` and `array $hookImplementationsMap`.
