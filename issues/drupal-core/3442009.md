## Summary

Drupal 11 gains a new attribute-based OOP hook system that sits alongside the existing procedural hook system. Module developers can now implement hooks as class methods annotated with `#[Hook('hook_name')]` instead of writing magic-named procedural functions. The classes are auto-discovered from the `src/Hook/` directory of each module and automatically registered as autowired services in the container. Procedural hooks continue to work exactly as before, so no existing code breaks. The change also introduces ordering controls so that hook implementations can declare they must run before or after specific others, replacing the notoriously awkward `hook_module_implements_alter()` pattern for most use cases.

## Impact

Affects module maintainers.

- **API change:** `ModuleHandler` now dispatches both OOP (attribute-based) and procedural hooks; `invokeAllWith()`, `invoke()`, and `alter()` transparently include class-method implementations found in `src/Hook/` directories
- **New API:** New `#[Hook]`, `#[LegacyHook]`, `#[ReorderHook]`, `#[RemoveHook]` attributes plus `HookCollectorPass`, `ImplementationList`, `Order`, `OrderAfter`, `OrderBefore`, `OrderInterface`, `ReorderHook`, and `ProceduralCall` classes for OOP hook implementations

## Technical details

**Background.** Drupal's hook system has always been procedural: a module implements `hook_user_cancel()` by defining a function named `mymodule_user_cancel()`. That design makes dependency injection, multiple implementations per module, and relative ordering difficult or impossible.

**Discovery and registration.** `HookCollectorPass` (in `core/lib/Drupal/Core/Hook/HookCollectorPass.php`) is a Symfony DI compiler pass that runs at container build time. It walks every module directory with a `RecursiveDirectoryIterator`. For `.php` files inside `src/Hook/`, it uses PHP `ReflectionClass` to read `HookAttributeInterface` attributes. For `.module`, `.inc`, `.profile`, and `.install` files, it uses `StaticReflectionParser` to scan for procedural function names matching the `modulename_hookname` pattern. The collected data is written to the `hook_data` key-value store by `HookCollectorKeyValueWritePass` and also cached in `cache.bootstrap`.

**Runtime dispatch.** `ModuleHandler::getHookImplementationList()` reads from `hook_data` on first use and builds an `ImplementationList` (in `core/lib/Drupal/Core/Hook/ImplementationList.php`) holding ordered callables and their associated module names. `invokeAllWith()`, `invokeAll()`, `invoke()`, and `alter()` all go through `getHookImplementationList()`. The fallback path for procedural hooks not found in the list is `ModuleHandler::legacyInvoke()`, which calls the `modulename_hookname()` function directly if it exists and is not annotated with `#[LegacyHook]`.

**Attributes.** `Hook` (`core/lib/Drupal/Core/Hook/Attribute/Hook.php`) is the primary attribute. It accepts the hook name (without the `hook_` prefix), an optional method name (defaulting to `__invoke` on a class-level attribute), an optional `module` parameter to implement on behalf of another module, and an optional `order` parameter. It can be placed on a class or on a method, and is repeatable so one method can implement multiple hooks.

`LegacyHook` (`Attribute/LegacyHook.php`) is placed on old procedural functions that have been converted to OOP. The `HookCollectorPass` skips these during procedural scanning, and `legacyInvoke()` also skips them, preventing double execution when both the OOP and procedural implementations are present. This enables contrib modules to ship a single codebase compatible with both Drupal 10 and 11.

`ReorderHook` (`Attribute/ReorderHook.php`) changes the position of an already-existing implementation that lives in a different class. `RemoveHook` (`Attribute/RemoveHook.php`) removes an existing implementation entirely. Both are placed on any class or method and take a target `$class`, `$method`, and the hook name.

**Ordering.** The `order` parameter of `#[Hook]` accepts an `OrderInterface` value. The `Order` enum (`Order::First` / `Order::Last`) moves an implementation to the absolute start or end. `OrderBefore` and `OrderAfter` (`core/lib/Drupal/Core/Hook/Order/`) accept a list of `$class::$method` identifiers to position relative to. `ProceduralCall` (`core/lib/Drupal/Core/Extension/ProceduralCall.php`) is a sentinel class used as the `$class` value when targeting a procedural function with a `ReorderHook` attribute. Order operations are stored as `OrderOperation` objects, applied by `HookCollectorPass::applyOrderOperations()`, and packed for runtime evaluation of alter-hook ordering.

**Scope exclusions.** Install hooks (`hook_install`, `hook_schema`, `hook_update_N`, `hook_post_update_NAME`, etc.), theme hooks, and a small set of single-implementation hooks (`hook_library_info_build`, `hook_mail`, `hook_help`, `hook_node_update_index`) must remain procedural.

**Core adoption.** Core modules have already converted many hooks. For example, `core/modules/node/src/Hook/` contains fourteen classes (`NodeEntityHooks`, `NodeFormHooks`, `NodeUserHooks`, etc.), each injecting only the services it needs.

**Related issues.** Two previously competing OOP-hooks proposals were closed in favour of this approach: [#3368812](https://www.drupal.org/node/3368812) (Hux-style hooks) and [#2233261](https://www.drupal.org/node/2233261) (allow module services to specify hooks).

## Upgrade

OOP hooks are opt-in. Existing procedural hook implementations continue to work without changes.

**To write a new OOP hook implementation:**

```php
// In core/modules/mymodule/src/Hook/MyModuleHooks.php
namespace Drupal\mymodule\Hook;

use Drupal\Core\Hook\Attribute\Hook;

class MyModuleHooks {
  #[Hook('user_cancel')]
  public function userCancel(array $edit, object $account, string $method): void {
    // implementation
  }
}
```

The class is auto-discovered and registered as an autowired service. No YAML registration is needed unless autowire is insufficient.

**To maintain compatibility with Drupal 10 and 11 simultaneously**, keep the old procedural function but mark it with `#[LegacyHook]` so Drupal 11 skips it:

```php
use Drupal\Core\Hook\Attribute\LegacyHook;

#[LegacyHook]
function mymodule_user_cancel(array $edit, object $account, string $method): void {
  \Drupal::service(MyModuleHooks::class)->userCancel($edit, $account, $method);
}
```

**To control execution order**, use the `order` parameter:

```php
use Drupal\Core\Hook\Attribute\Hook;
use Drupal\Core\Hook\Order\OrderAfter;

#[Hook('form_alter', order: new OrderAfter([OtherModule\Hook\OtherHooks::class . '::formAlter']))]
public function formAlter(array &$form, ...) {}
```

**To reorder or remove an implementation defined elsewhere**, use `#[ReorderHook]` or `#[RemoveHook]` on any class or method in your module:

```php
use Drupal\Core\Hook\Attribute\ReorderHook;
use Drupal\Core\Hook\Order\Order;

#[ReorderHook('form_alter', OtherModule\Hook\OtherHooks::class, 'formAlter', Order::Last)]
class MyModuleHooks {}
```
