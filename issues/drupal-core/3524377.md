## Summary

Drupal now provides a `#[HookDependsOnModule]` PHP attribute that module developers can place on an OOP hook class or method to make registration conditional on another module being installed. When the named module is absent, the annotated hook method (or entire class) is skipped at container build time. If every hook method in a class is skipped, the class is not added to the container at all. This solves two problems: it avoids unnecessary service registrations for hooks that can never fire, and it lets developers declare service-container dependencies as required constructor arguments instead of nullable ones, because the hook class will simply never be instantiated when the required module is absent.

## Impact

Affects module maintainers.

- **New API:** New `#[HookDependsOnModule]` attribute for conditionally registering OOP hook classes or methods based on whether a module is installed.

## Technical details

The new `Drupal\Core\Hook\Attribute\HookDependsOnModule` class is a PHP attribute declared with `TARGET_CLASS | TARGET_METHOD | IS_REPEATABLE`, accepts a single `$module` string, and implements `HookAttributeInterface`. The check is performed inside `HookCollectorPass::collectModuleHookImplementations()` against `$this->modules`, the list of installed modules already available in the pass. Class-level attributes are now also serialized into the file cache entry (as `class_attributes`) so the check is fast on warm runs.

The logic runs in two stages: first a loop over class-level `HookDependsOnModule` instances uses `continue 2` to skip the entire file if any dependency is unmet; then a per-method inner loop does the same for method-level instances. A hook class with no surviving method registrations is never added to the container.

Two in-tree examples land in the same commit. `FileViewsHooks` receives `#[HookDependsOnModule('views')]` on the class, which lets the formerly nullable `?FieldViewsDataProvider` constructor argument become required. `MenuUiHooks::menuDelete()` receives `#[HookDependsOnModule('node')]` on the method, removing a runtime `$this->entityTypeManager->hasDefinition('node_type')` guard that was previously needed to handle the case where node module was absent.

## Upgrade

Module developers with OOP hook classes that inject services from an optionally-installed module can now declare those dependencies as required and add the attribute:

```php
use Drupal\Core\Hook\Attribute\Hook;
use Drupal\Core\Hook\Attribute\HookDependsOnModule;

// Class-level: skip the entire class when 'search' is not installed.
#[HookDependsOnModule('search')]
class NodeSearchHooks {
  public function __construct(
    protected readonly SearchIndexInterface $searchIndex, // now required, not nullable
  ) {}

  #[Hook('cron')]
  public function cron(): void { ... }
}

// Method-level: skip only this method when 'node' is not installed.
#[Hook('menu_delete')]
#[HookDependsOnModule('node')]
public function menuDelete(EntityInterface $entity): void { ... }
```

The attribute is repeatable; if any named module is absent the class or method is skipped. Because a non-existent attribute class is not a fatal error (unlike a non-existent named parameter on an existing attribute), contrib modules can safely use `#[HookDependsOnModule]` and simply declare a minimum core version requirement for the class to be recognized.

### Collaboration

berdir opened the issue and drove the implementation. A key design debate in the thread concerned whether to add a named parameter to the existing `#[Hook]` attribute (or a generic `ConditionalRegister` attribute) versus introducing a dedicated class. berdir's argument, confirmed with a live code snippet at https://3v4l.org/31a3W, was that using a non-existent named parameter on an existing attribute is a PHP fatal error, making a separate attribute class the only approach that is safe across the multi-version contrib compatibility window. kristiaanvandeneynde pushed back in favor of a more generic name to leave room for future dependency types (themes, callbacks), and berdir and nicxvan countered that purpose-specific attribute classes are already precedent in the hook system and that any future use case can get its own attribute without BC concerns. acbramley did most of the rebase and cleanup work in the final stretch, renaming the constructor parameter to `$module` for consistency with `#[Hook]` and applying godotislate's doc-review suggestions before the commit landed.
