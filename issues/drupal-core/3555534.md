## Summary

Symfony 7.4 deprecated passing constraint options to parent constructors, requiring Drupal to update how validation constraints are instantiated. This change affects internal validation architecture but is handled transparently. Drupal core now passes validation options as named parameters to constraint constructors and temporarily maintains backward compatibility for existing code that passes options as arrays. Site owners and developers do not need to take any immediate action.

## Impact

No upgrade or configuration changes required.

## Technical details

Symfony's Constraint base class deprecated evaluating options arrays in its constructor. The Symfony-recommended approach requires constraint classes to define constructors with named parameters for each property.

Drupal updated all constraint plugin classes in core to add constructors with named parameters, using the `#[HasNamedArguments]` attribute. Each constructor accepts options as individual named parameters while maintaining backward compatibility for code that still passes options as associative arrays. The `ConstraintManager::create()` method now triggers a deprecation notice when options are passed as a non-associative array or non-array value.

Configuration schema files were updated to pass constraint options as associative arrays. For example, `ValidKeys: '<infer>'` became `ValidKeys: ['allowedKeys' => '<infer>']`. Method signatures for `addConstraint()` throughout the typed data system were annotated to indicate they should accept only associative arrays or NULL, though the actual signature change is deferred to Drupal 12.

The `ConstraintFactory::createInstance()` method now handles both old-style array options and new-style named parameters, using reflection to determine the correct constructor invocation. Special handling was added for constraints like `ComplexDataConstraint` that allow options to be passed without explicit keys.

Entity and field definition classes that use the ComplexData constraint were updated to nest property constraints under a `'properties'` key. The `EntityDataDefinition::getBundles()` method adds temporary fallback logic to handle both old and new constraint structures.

## Collaboration

Seven contributors collaborated over three months to address this Symfony deprecation. The issue involved extensive work to update 68 constraint classes and their usages throughout core. The team carefully evaluated two implementation approaches: introducing a `ConstraintBase` class to centralize parameter handling versus updating individual constructors with named parameters. After discussion in comments #16-28, the team chose individual constructors to align with Symfony's intent of introducing typed parameters, despite the larger diff. The decision prioritized long-term maintainability over short-term convenience. The discussion also surfaced concerns about named parameters becoming implicit API (#27) and potential issues with high volumes of deprecation notices (#61), leading to follow-up issues for Drupal 12.
