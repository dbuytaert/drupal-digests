## Summary

A real access-control bug was discovered in the media module: code in `MediaAccessControlHandler` called `$entity_access->andIf(...)` without capturing the return value, silently discarding the combined access check when a user viewed a non-default media revision. This could allow users who have permission to view all revisions to see revisions of media items they should not have access to. The fix corrects the discarded return value and, on the `main` and `11.x` development branches only, adds the PHP 8.5 `#[\NoDiscard]` attribute to `AccessResultInterface::andIf()` and `AccessResultInterface::orIf()` so that PHP itself will warn developers when they make the same mistake in the future. The `#[\NoDiscard]` attribute was intentionally omitted from the 11.3.x and 10.6.x patch-release backports to avoid flooding logs with runtime warnings for existing code.

## Impact

Affects module maintainers.

- **API change:** `#[\NoDiscard]` added to `AccessResultInterface::andIf()` and `AccessResultInterface::orIf()` on `main`/`11.x`; PHP 8.5 will warn when callers discard the return value of either method.

## Technical details

The root cause is that `AccessResult::andIf()` and `AccessResult::orIf()` return a new object rather than modifying the receiver in place, following an immutable-value pattern. `MediaAccessControlHandler::checkAccess()` in `core/modules/media/src/MediaAccessControlHandler.php` had `$entity_access->andIf($this->access(...))` without reassignment, so the result of combining the revision-access check with the original-entity access check was thrown away, leaving only the revision check in effect for non-default media revisions.

The `main`/`11.x` fix (commit a94c3963) has two parts. First, it corrects the bug: `$entity_access->andIf(...)` becomes `$entity_access = $entity_access->andIf(...)`. Second, it adds `#[\NoDiscard]` to both `orIf()` and `andIf()` in `AccessResultInterface` and `AccessResult`, so PHP 8.5 emits a warning whenever a caller discards the return value of either method. The test in `MediaAccessTest` was also updated: the expected cache context for a non-default revision view changed from `user.permissions` to `user`, reflecting that the now-correct combined result properly carries the `user` context from the per-user ownership check.

The 11.3.x and 10.6.x backports (commits 5c02b602 and 4b3b0b42, respectively) contain only the `MediaAccessControlHandler` one-line fix and the test assertion update. `#[\NoDiscard]` was deliberately left out: PHP 8.5 emits a runtime warning whenever code discards the return value of a `#[\NoDiscard]`-annotated method, and introducing that warning mid-patch-series would be a behavior change that could break existing tests and flood production logs for sites where callers already use the pattern incorrectly. The attribute is appropriate only for the next minor release where callers have been warned in advance.

A related initiative to apply `#[\NoDiscard]` more broadly across utility classes is tracked in [#3575572](https://www.drupal.org/node/3575572).

## Upgrade

If you have custom code that calls `andIf()` or `orIf()` without capturing the return value, update it to always reassign:

```php
// Before (broken — the combined result is silently discarded)
$access_result->andIf($other_result);

// After (correct)
$access_result = $access_result->andIf($other_result);
```

On `main`/`11.x`, PHP 8.5 will emit a warning for the old pattern due to the new `#[\NoDiscard]` attribute.
