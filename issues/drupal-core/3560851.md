## Summary

The `test_theme` testing theme has been converted from procedural hooks in a `.theme` file to the modern object-oriented hook system. This is an internal test infrastructure change with no effect on production sites. The conversion also uncovered and fixed a long-standing issue in the installer test base class, where container dumping was unnecessarily disabled.

## Impact

No upgrade or configuration changes required.

## Technical details

Drupal's hook system supports OOP-style hook classes (using the `#[Hook]` PHP attribute) in addition to procedural functions. This issue applies that pattern to `test_theme`, one of the core functional test themes.

**Main conversion.** The entire `core/modules/system/tests/themes/test_theme/test_theme.theme` file (151 lines of procedural hook functions) was deleted. A new class `TestThemeHooks` was created at `core/modules/system/tests/themes/test_theme/src/Hook/TestThemeHooks.php`. All eleven hook implementations were migrated into methods on this class, each annotated with `#[Hook('...'))]`. The class injects `MessengerInterface` via constructor dependency injection, replacing the previous `\Drupal::messenger()` static calls. Parameter types were also added to all method signatures.

**Removal of the `ThemeClass` class-loading test.** The old `.theme` file included `test_theme_preprocess_theme_test_theme_class()`, which verified that a namespaced class (`ThemeClass`) could be autoloaded from within a `.theme` file. Because OOP hooks are loaded through Drupal's regular PSR-4 class discovery, this concern is moot: the class loader is already active before any hook in a `Hook` class is invoked. The associated `ThemeClass.php`, the `theme_test_theme_class` theme definition (in `ThemeTestHooks::theme()`), the `theme-test-theme-class.html.twig` template, the `ThemeTestController::testThemeClass()` controller method, the `theme_test.test_theme_class` route, and the `testClassLoading()` functional test were all removed.

**Installer test fix.** `InstallerTestBase::setUp()` was calling `DrupalKernel::createFromRequest($request, $class_loader, 'prod', FALSE)`, passing `FALSE` for the `$allow_dumping` parameter. This prevented the container from being serialized to disk. `DrupalKernel::createFromRequest()` defaults `$allow_dumping` to `TRUE`, and the explicit `FALSE` was a relic from the SimpleTest era (traced back to [#2016629](https://www.drupal.org/node/2016629)). OOP hook resolution requires a properly booted container, so that `FALSE` was causing `InstallerTranslationTest` to fail after the conversion. The fix removes the explicit argument, allowing the default `TRUE` to apply.

**Test assertion update.** `ThemeSuggestionsAlterTest::testExecutionOrder()` was updated to expect fully-qualified method references (`Drupal\test_theme\Hook\TestThemeHooks::themeSuggestionsAlter() executed.`) instead of the old procedural function name strings.
