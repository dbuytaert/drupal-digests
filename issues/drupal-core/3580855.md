## Summary

When BigPipe renders blocks as placeholders, each block runs inside a PHP Fiber. A bug in the entity recursive-rendering protection introduced by [#2940605](https://www.drupal.org/node/2940605) caused recursion keys to be shared across Fibers. If two blocks on the same page rendered the same entity, the second Fiber would see the first Fiber's recursion key, treat it as an infinite-loop guard, and suppress the entity's output entirely. The fix scopes recursion keys per Fiber so each block renders independently. Sites using BigPipe with multiple blocks that display the same entity (for example, menu blocks showing node data) were most likely to encounter missing content.

## Impact

No upgrade or configuration changes required.

## Technical details

The entity recursive-rendering protection in `EntityViewBuilder::getRenderRecursionKey()` builds a string key from the entity's class, object ID, and optionally its language. This key is stored in a static tracking array to prevent infinite recursion when an entity renders itself again mid-build. Because PHP Fibers are cooperative coroutines rather than true threads, multiple in-flight Fibers share the same static state. When BigPipe placeholder rendering suspends Fiber A mid-render (for example, while a formatted text field's filter placeholders are resolved) and starts Fiber B for a second block rendering the same entity, Fiber B finds Fiber A's recursion key already present and incorrectly treats it as recursion, returning an empty render array. The fix adds a Fiber-specific component to the key: when `\Fiber::getCurrent()` is non-null, `spl_object_id($fiber)` is appended to `$recursion_keys`, making every Fiber's keys distinct. A new functional test `EntityConcurrentRenderTest` verifies the scenario: two `entity_test_block` blocks (which force `createPlaceholder(): true`) both render the same entity that has a formatted text field; the filter placeholder creation is what causes each Fiber to suspend and allow interleaving. The bug was introduced by [#2940605](https://www.drupal.org/node/2940605) and had not yet shipped in a stable release, so no backport to 11.3.x was required.
