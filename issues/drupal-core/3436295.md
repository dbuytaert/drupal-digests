## Summary

Drupal was unnecessarily loading several services on every page request, even when they were not needed. Services that convert URL parameters (like node IDs) into full objects were marked as lazy but were actually being instantiated immediately. This change fixes the lazy loading behavior, reducing the number of services constructed on each request and improving overall performance. The fix was blocked for nearly two years by a Symfony bug that has now been resolved.

## Impact

No upgrade or configuration changes required.

## Technical details

Services tagged with `paramconverter` were marked as `lazy: true` in service definitions to avoid early instantiation, but the lazy proxy mechanism was not working as intended. The `ParamConverterManager` used a service collector that immediately instantiated all converters when the manager itself was constructed, which happens on every page request.

The solution replaces the service collector pattern with a service locator using Symfony's `#[AutowireLocator]` attribute. Service locators only instantiate services when they are explicitly requested via `get()`, providing true lazy loading. The `ParamConverterManager` constructor now accepts a `ServiceCollectionInterface` instead of collecting services via `addConverter()`.

A Symfony bug initially prevented this fix from working. The `ServiceLocatorTagPass::map()` method called `ksort()` on services, destroying the priority-based ordering required by param converters. This was fixed upstream in Symfony 7.0.9. The change removes proxy class files that are no longer needed and removes the now-unnecessary `lazy: true` declarations from service definitions. It also removes the `addConverter()` method from `ParamConverterManagerInterface`.

New test coverage verifies that converters are only instantiated when actually needed for a specific route parameter conversion.
