## Summary

Drupal entities now expose a formal `getOriginal()` / `setOriginal()` API for accessing the unchanged, pre-save version of an entity. Previously, code inside hooks like `hook_entity_presave` and `hook_entity_update` relied on a magic, undeclared `$entity->original` property, which was invisible to IDEs and static analysis tools, and triggered PHP 8.2 dynamic-property deprecation warnings on config entities. All core code has been updated to use the new methods. Site owners do not see any user-facing change, but module and custom-code developers should replace direct `$entity->original` access with `$entity->getOriginal()`.

## Impact

Affects module maintainers.

- **API change:** `getOriginal(): ?static` and `setOriginal(?EntityInterface $original): static` added to `EntityInterface`; direct access to `$entity->original` is deprecated in favour of these methods.
- **Deprecation:** `$entity->original` magic property access is deprecated; use `EntityInterface::getOriginal()` and `EntityInterface::setOriginal()` instead.

## Technical details

Before this change, the entity save pipeline assigned the pre-save entity to an ad-hoc `$entity->original` property. On content entities this landed silently in `ContentEntityBase::$values` through the `__get`/`__set` magic methods. On config entities, which have no such magic, it created an undeclared dynamic property, causing PHP 8.2 deprecation notices on every save of a config entity.

The fix introduces a properly declared `protected ?EntityInterface $originalEntity = NULL` in `EntityBase`. The property is deliberately named `$originalEntity` rather than `$original` to avoid shadowing the old magic property path that existing contrib/custom code may still use. `EntityInterface` gains two new methods: `getOriginal(): ?static` and `setOriginal(?EntityInterface $original): static`. `EntityBase` implements both by reading from and writing to `$originalEntity`.

`EntityStorageBase::doPreSave()` now calls `$entity->getOriginal()` to check whether the original is already set and `$entity->setOriginal($this->loadUnchanged($id))` to populate it when missing. `EntityStorageBase::doPostSave()` calls `$entity->setOriginal(NULL)` after the hooks run, clearing the reference so the entity can be garbage-collected normally.

`ContentEntityBase::getTranslation()` shares the `$originalEntity` reference across translation objects with `$translation->originalEntity = &$this->originalEntity`, preserving the existing behavior where all translation stubs share the same original entity pointer.

All core modules that previously accessed `$entity->original` directly have been updated: user, node, image, field storage and field config, content_translation, comment, views, media, and `NodeBulkUpdate`/`ViewUI`. Callers that pre-populate the original for efficiency (e.g. bulk update paths) now call `$entity->setOriginal(clone $entity)` instead of assigning to the property.

The old `$entity->original` magic-property path remains functional as a deprecated fallback: on content entities it still routes through `__get`/`__set` (writing to `$this->values`), and `ConfigEntityBase` retains `#[\AllowDynamicProperties]` for now. A follow-up issue ([#3299857](https://www.drupal.org/node/3299857)) will remove that attribute once the deprecated path is fully retired.

## Upgrade

Replace direct property access with the new interface methods:

```php
// Old (deprecated)
$original = $entity->original;
if (isset($entity->original)) { ... }
$entity->original = clone $entity;

// New
$original = $entity->getOriginal();
if ($entity->getOriginal() !== NULL) { ... }
$entity->setOriginal(clone $entity);
```

### Collaboration

The design debate ran for several years and centered on a single question: should the backing store be a `public` property (visible to IDE deprecation scanners but bypassing the setter) or a `protected` one (requiring special magic-method handling on both content and config entities). berdir initially favored `public` for IDE visibility; hchonov pushed back consistently, arguing that a public property would prevent future listener-style optimizations and remove the ability to intercept writes. alexpott supported the `protected` approach, noting it keeps behavior changes narrower. The thread also uncovered that config entities, unlike content entities, have no `__get`/`__set` at all, meaning a protected property on `EntityBase` would silently break config entity access without dedicated magic-method support. That observation, made by berdir, effectively closed the debate: rather than add new magic methods to `EntityBase`, the team chose a separate `$originalEntity` property name that avoids the clash entirely.
