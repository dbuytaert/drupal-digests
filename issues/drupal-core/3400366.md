## Summary

This is a purely internal developer-tooling change with no user-facing effect. The old runtime PHPUnit test listener that checked whether component tests illegally extend Drupal core test base classes has been replaced with a static analysis rule executed by PHPStan. The rule fires earlier, at analysis time rather than test runtime, and is unaffected by the removal of PHPUnit test listeners in PHPUnit 10.

## Impact

Affects module maintainers.

- **Deprecation:** `DrupalComponentTestListenerTrait` (marked `@internal`) and its `componentEndTest()` method are permanently removed from `core/tests/Drupal/Tests/Listeners/`.
- **New API:** New `core/tests/PHPStan/` subtree with its own `composer.json` establishes the pattern for Drupal-core-specific PHPStan rules; first rule is `Drupal\PHPStan\Rules\ComponentTestDoesNotExtendCoreTest`.

## Technical details

The problem originated from the need to migrate away from PHPUnit test listeners, which are removed in PHPUnit 10 ([#3441353](https://www.drupal.org/node/3441353) was a related blocker). The old mechanism was `DrupalComponentTestListenerTrait` in `core/tests/Drupal/Tests/Listeners/DrupalComponentTestListenerTrait.php`, which `DrupalListener` used via the `componentEndTest()` method. At test runtime it checked whether any class under `Drupal\Tests\Component` extended `UnitTestCase`, `KernelTestBase`, or `BrowserTestBase` and failed the test with an `AssertionFailedError` if so.

The replacement is the PHPStan rule `ComponentTestDoesNotExtendCoreTest` added at `core/tests/PHPStan/Rules/ComponentTestDoesNotExtendCoreTest.php`. It implements `Rule<InClassNode>`, inspects every class node via `processNode()`, and returns a `RuleErrorBuilder` error message for any class in the `Drupal\Tests\Component` namespace that is a subclass of `UnitTestCase`, `BuildTestBase`, `KernelTestBase`, or `BrowserTestBase`. The rule is registered in `core/phpstan.neon.dist` under the `rules` key and the fixture files for its tests are excluded from the regular PHPStan scan via the `excludePaths` parameter in the same file.

Testing PHPStan rules requires a different dependency set than Drupal core: PHPStan bundles `nikic/php-parser` inside its PHAR for production use, but its testing infrastructure exposes the real package dependency, which conflicts with the version Drupal's PHPUnit 10 upgrade requires. To keep these dependency trees separate, a new isolated subtree `core/tests/PHPStan/` was introduced with its own `composer.json` (pinning `phpstan/phpstan 1.10.66` and `phpunit/phpunit ^9`), `phpunit.xml.dist`, and `.gitignore`. The rule test class `ComponentTestDoesNotExtendCoreTestTest` extends PHPStan's `RuleTestCase` and calls `analyse()` against a fixture file at `core/tests/PHPStan/fixtures/component-tests.php`. A second test, `EnsurePHPStanVersionsMatchTest`, asserts that the PHPStan version in the local `composer.json` matches the pinned version in `composer/Metapackage/PinnedDevDependencies/composer.json`, preventing version drift.

The root `composer.json` gains an `autoload-dev` PSR-4 entry mapping `Drupal\PHPStan\Rules\` to `core/tests/PHPStan/Rules`. A new CI pipeline job `✅ PHPStan Tests` is added in `.gitlab-ci/pipeline.yml`; it changes into `core/tests/PHPStan`, runs `composer install`, and executes PHPUnit, triggering only when files under `core/tests/PHPStan/` or the pinned-dev-dependencies manifest change. `DrupalComponentTestListenerTrait` and its `componentEndTest()` call in `DrupalListener::endTest()` are fully deleted.

## Upgrade

If any custom or contributed code was referencing the `@internal` `DrupalComponentTestListenerTrait`, remove the `use DrupalComponentTestListenerTrait;` statement and delete any calls to `componentEndTest()`. There is no public replacement; the enforcement is now handled automatically by PHPStan during static analysis of Drupal core.

### Collaboration

The issue sat idle after an initial postpone waiting on phpstan-drupal upstream (#626). When mondrake identified it as being on the critical path to PHPUnit 10, mglaman raised a genuine design question: should the rule live in phpstan-drupal or in Drupal core? He noted that a core-only enforcement paradigm felt wrong for a general-purpose extension. mondrake and alexpott argued it belonged in core because the `Drupal\Tests\Component` namespace constraint is a core-internal concern that has no relevance to contrib or custom projects. mglaman accepted that reasoning and wrote `ComponentTestDoesNotExtendCoreTest`. During review, alexpott proposed the isolated `core/tests/PHPStan/` subdirectory with its own `composer.json`, explaining that PHPStan's PHAR bundles its own copy of `nikic/php-parser` but the testing infrastructure surfaces the real dependency, which conflicted with the `nikic/php-parser` version Drupal needed for PHPUnit 10. That architectural decision is what made the tests runnable without forcing a version conflict on Drupal core's dependency tree. mondrake, longwave, and alexpott collaborated on filling out the `README.md` and wiring up the CI job.
