## Summary

Drupal's `drupal_flush_all_caches()` function used to call `plugin.cache_clearer->clearCachedDefinitions()` after wiping all cache bins and rebuilding the container. Because the container rebuild already replaces all service instances with fresh ones (discarding any in-memory plugin definition state), that explicit plugin cache clear was redundant work. The three lines have been removed, making the cache-rebuild path slightly faster with no change in correctness. The `plugin.cache_clearer` service itself is not removed; it is still used during module and theme install/uninstall, which does not wipe cache bins the same way.

## Impact

No upgrade or configuration changes required.

## Technical details

The removed call in `drupal_flush_all_caches()` (`\Drupal::service('plugin.cache_clearer')->clearCachedDefinitions()`) originally dates to 2013, when the container was not yet rebuilt on cache clear (the `@todo Rebuild the kernel/container` comment in issue [#1892462](https://www.drupal.org/node/1892462) predates the centralized container invalidation added in [#2493665](https://www.drupal.org/node/2493665)). Once `drupal_flush_all_caches()` gained `$kernel->invalidateContainer()` and `$kernel->rebuildContainer()`, all plugin manager service instances are replaced before the `plugin.cache_clearer` line ever runs, so the call clears definitions that no longer exist in the active container. Manual instrumentation by longwave confirmed that by the time the removed call ran in practice, only the entity type manager retained any data, solely because `BlockHooks::rebuild()` (invoked earlier via `hook_rebuild`) had reloaded it. Two follow-up issues were opened as a result of this investigation: [#3593204](https://www.drupal.org/node/3593204) to remove `BlockHooks::rebuild()` and [#3593415](https://www.drupal.org/node/3593415) to deprecate the `plugin.cache_clearer` service entirely.
