## Summary

When a content entity is saved, the `changed` field type automatically updates the timestamp to the current request time whenever actual field changes are detected. This made it impossible to preserve a historically accurate `changed` value when migrating or re-importing content from an external source: every save during a migration run would overwrite the original timestamp with the time of the migration itself. This fix makes `ChangedItem` respect the entity synchronization flag, so callers who set `$entity->setSyncing(TRUE)` before saving (as migration destination plugins now do) will have the `changed` field left at whatever value it was given, not silently replaced by request time.

## Impact

No upgrade or configuration changes required.

## Technical details

The problem lives in `ChangedItem::preSave()` in `core/lib/Drupal/Core/Field/Plugin/Field/FieldType/ChangedItem.php`. When an existing entity translation is saved and its `changed` value matches the original, and `hasTranslationChanges()` returns `TRUE`, the field automatically updates `$this->value` to `\Drupal::time()->getRequestTime()`. There was no way to prevent this without manually setting a future timestamp.

The fix adds a single guard condition around that auto-update block:

```php
if (!$entity instanceof SynchronizableInterface || !$entity->isSyncing()) {
  // … existing auto-update logic …
}
```

`SynchronizableInterface` and its trait `SynchronizableEntityTrait` (with `setSyncing()`/`isSyncing()`) were generalized for all entity types in [#2985297](https://www.drupal.org/node/2985297). All content entities already implement it. When `setSyncing(TRUE)` is set, the block that would overwrite `changed` is simply skipped, so whatever timestamp the caller placed on the field is preserved through save.

The companion issue [#3052115](https://www.drupal.org/node/3052115) ensures that the migrate destination plugins (`EntityContentBase`, `Entity`, `EntityRevision`) call `$entity->setSyncing(TRUE)` when processing rows, so migrations automatically get the correct behavior without any extra code in the migration configuration.

A new `testChangedSyncing()` kernel test in `ContentEntityChangedTest` verifies both sides: that without the sync flag the timestamp advances on an ordinary update, and that with `setSyncing(TRUE)` the timestamp remains frozen after save.

The `ChangedTestItem` test subclass in `entity_test` was retained; earlier patch iterations had proposed deleting it in favor of a `TestTime` service, but the final approach did not require that change.

The design thread considered several alternatives: a dedicated `EntityFrozenInterface` (with `freeze()`/`isFrozen()` methods suggested by wim leers and built out by tstoeckler), a `presetMode` property on `EntityBase`, and a field-level `preserve` property on `ChangedItem` itself. After [#2985297](https://www.drupal.org/node/2985297) landed, hchonov and sam152 converged on reusing the existing `SynchronizableInterface` semantics, noting that from the entity perspective all these cases are logically equivalent: the entity is being synchronized and automatic side-effects should not fire.

### Collaboration

The issue ran for nearly eight years before landing, and the design pivoted twice. In 2014, tstoeckler prototyped a "preset mode" on `EntityBase` and then rebuilt it around a dedicated `EntityFrozenInterface` after wim leers proposed the `freeze()`/`isFrozen()` metaphor (comment [#9154563](https://www.drupal.org/node/9154563)) and dawehner suggested letting `ContentEntityInterface` carry that interface to drop workaround code in ViewUI (comment [#9170485](https://www.drupal.org/node/9170485)). That line of work stalled without a reviewer.

After [#2985297](https://www.drupal.org/node/2985297) generalized `SynchronizableInterface` to all entities in 2018, hchonov reopened the thread with a new patch built entirely on `isSyncing()` (comment [#12701107](https://www.drupal.org/node/12701107)), arguing that migrations and other synchronization scenarios are semantically equivalent: automatic entity side-effects should not fire in any of them. sam152 reinforced that argument (comment [#13153351](https://www.drupal.org/node/13153351)) and split the migration-destination half of the patch into its own issue [#3052115](https://www.drupal.org/node/3052115) to reduce complexity here. wim leers reviewed the final patch in detail (comment [#14295762](https://www.drupal.org/node/14295762)), confirmed it worked locally, called out the single functional line added in `ChangedItem::preSave()`, and filed the change record.
