## Summary

Saving a path alias whose internal path matches a `menu_link_content` entity could throw a fatal `PluginNotFoundException` when that entity's plugin definition was absent from the active menu tree. This happened in scenarios such as importing menu links before the tree was rebuilt, or when a contrib module like `wse_menu` maintains a separate menu tree per workspace so a menu link entity exists in Live storage but has no Live tree definition. The fix makes the alias-save path skip the definition update if the plugin is not currently registered, preventing the crash.

## Impact

No upgrade or configuration changes required.

## Technical details

The problem was in `updatePathAlias()` (previously the procedural `_menu_link_content_update_path_alias()`), which loaded all `menu_link_content` entities whose `link.uri` matched the saved alias path and unconditionally called `$this->menuLinkManager->updateDefinition()` on each. When a plugin ID was absent from the menu link manager's registry, `updateDefinition()` internally called `createInstance()`, which threw `PluginNotFoundException`.

The fix adds a `$this->menuLinkManager->hasDefinition($menu_link->getPluginId())` guard before the `updateDefinition()` call, so definitions that are not present in the active tree are silently skipped. This mirrors the pattern already used in `MenuLinkContent::postSave()`, which calls `getDefinition($id, FALSE)` (the `FALSE` suppresses exceptions) before updating, as introduced in [#2605684](https://www.drupal.org/node/2605684). A new test assertion covers this by explicitly removing the plugin definition via `removeDefinition()` and then verifying that saving a matching path alias neither throws nor re-registers the missing definition.
