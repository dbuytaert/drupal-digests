## Summary

Kernel test classes can now define their own HTTP routes by placing Symfony's `#[Route]` PHP attribute on methods in the test class. Previously, a test author needing a custom route for a kernel test had to create a separate test module just to host that one route. With this change, the test class itself acts as the controller, and test properties are directly accessible in the route method without any database round-trips or mocking.

## Impact

Affects module maintainers.

- **API change:** `AttributeRouteDiscovery::createControllerRouteCollection()` and `getReflectionClass()` widened from `private` to `protected`, enabling subclassing.
- **New API:** Kernel test classes can now place Symfony `#[Route]` attributes on methods to register routes in the test container without a separate test module.

## Technical details

In `KernelTestBase::register()`, a new `KernelTestAttributeRouteDiscovery` service is now registered in the test container, tagged as `event_subscriber`. This `@internal` class extends `AttributeRouteDiscovery` and overrides `collectRoutes()` to scan only the specific kernel test class, which is passed as a constructor string argument. The service is registered with `static::class` so each test class has its own discovery instance.

To enable the subclass, two methods in `AttributeRouteDiscovery` were widened from `private` to `protected`: `createControllerRouteCollection(\ReflectionClass $class)` and `getReflectionClass(string $className)`. These are the only diff-visible changes to `AttributeRouteDiscovery` itself; `AttributeRouteDiscovery` is not marked `@internal`, making this a public API surface change.

The `KernelTestHttpRequestTest` demonstrates the feature: a `$testMarkup` property set in the test method is read by `exampleRoute()`, which is decorated with `#[Route(path: '/kernel-test-example', ...)]`. This proves that test instance state is shared between the test body and the route handler because both run in the same process. Routes meant to be shared across multiple kernel tests can be placed in a trait and used via `use`.

## Upgrade

If you subclass `AttributeRouteDiscovery` and relied on `createControllerRouteCollection()` or `getReflectionClass()` being `private` (e.g. via reflection to access them), they are now `protected` and directly available to subclasses.

To define a route in a kernel test class:

```php
use Symfony\Component\Routing\Attribute\Route;

class MyKernelTest extends KernelTestBase {

  protected string $myValue = '';

  public function testSomething(): void {
    $this->myValue = 'hello';
    $this->drupalGet('my-test-route');
    $this->assertSession()->pageTextContains('hello');
  }

  #[Route(
    path: '/my-test-route',
    name: 'my_test.example',
    requirements: ['_access' => 'TRUE'],
  )]
  public function myController(): array {
    return ['#markup' => $this->myValue];
  }

}
```
