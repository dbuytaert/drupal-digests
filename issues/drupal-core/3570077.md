## Summary

Drupal's `main` branch (targeting Drupal 12) now requires Symfony 8 instead of Symfony 7. All Symfony component constraints in `composer.json` and `core/composer.json` were bumped from `^7.4` to `^8.0`, and `composer.lock` was regenerated to resolve Symfony 8.0.x releases. Because Symfony 8 requires PHP 8.4, that version is now the transitive minimum for any site or build running against `main`. Two non-Symfony packages were also updated to Symfony 8 compatible releases: `asm89/stack-cors` to v2.4.0 and `php-tuf/composer-stager` to v2.1.0. This change only affects the development (`main`) branch; no released Drupal 10 or 11 version is affected.

## Technical details

The version constraints for all 15 runtime Symfony components in `core/composer.json` (`symfony/console`, `symfony/dependency-injection`, `symfony/event-dispatcher`, `symfony/filesystem`, `symfony/finder`, `symfony/http-foundation`, `symfony/http-kernel`, `symfony/mailer`, `symfony/mime`, `symfony/process`, `symfony/psr-http-message-bridge`, `symfony/routing`, `symfony/serializer`, `symfony/validator`, `symfony/yaml`) and all 6 dev-only components in the root `composer.json` were changed from `^7.4` to `^8.0`. The resolved versions in `composer.lock` are Symfony 8.0.6 for most components and 8.0.4 or 8.0.5 for others.

Two prerequisite Drupal issues had to land before this could be unblocked: [#3555532](https://www.drupal.org/node/3555532) replaced uses of the removed `Request::get()` method and [#3555534](https://www.drupal.org/node/3555534) updated custom `Constraint` subclasses to initialize options in their constructors rather than passing them to the parent, both changes required by deprecations introduced in Symfony 7.4 and finalized as removals in Symfony 8. The `#[HasNamedArguments]` cleanup was handled in the sibling issue [#3569092](https://www.drupal.org/node/3569092), which merged immediately before this one.

Symfony 8's `http-foundation` component dropped support for honoring the `X-HTTP-Method-Override` header for safe HTTP methods (GET, HEAD, CONNECT, TRACE): such requests are now treated as POST by Symfony itself rather than being method-overridden. `RequestSanitizer` was left unchanged; only the assertions in `RequestSanitizerTest` were updated to match the new Symfony 8 behavior, including a new test for the `QUERY` method with an explicit return type added during review.

## Upgrade

Drupal 12 (`main` branch) now requires PHP 8.4 as a transitive minimum through Symfony 8. Contrib module authors and anyone maintaining custom code against `main` should:

1. Ensure any direct Symfony package constraints they declare also allow `^8.0`.
2. Test against PHP 8.4.
3. If your code defines custom Symfony `Constraint` subclasses, initialize constraint options in the constructor rather than passing them to the parent. See [#3555534](https://www.drupal.org/node/3555534) for examples of how core updated its own constraints.

```json
// core/composer.json — before
"symfony/http-foundation": "^7.4"

// core/composer.json — after
"symfony/http-foundation": "^8.0"
```

### Collaboration

The issue required coordinating several external blockers before it could merge. longwave tracked the upstream dependency queue and opened pull requests against `php-tuf/composer-stager` and `asm89/stack-cors` to add Symfony 8 support; both eventually released compatible versions (v2.1.0 and v2.4.0 respectively). Gábor Hojtsy checked the remaining blocker, `mglaman/phpstan-drupal`, found it not yet released, and dug into why: PHPStan 2.1.38 had changed stub handling in a way that broke phpstan-drupal, which had to be fixed upstream before a release was possible.

On the test side, andypost identified that `RequestSanitizerTest` was failing because Symfony 8 removed HTTP method override support for safe methods, and proposed updating `RequestSanitizer` itself. longwave determined the correct approach was to update only the test assertions, since Symfony 8 now handles that sanitization internally rather than delegating it to Drupal. The discussion clarified the new behavior: GET, HEAD, CONNECT, and TRACE requests sent with an `X-HTTP-Method-Override` header are treated as POST by Symfony 8, so the page cache no longer sees a cache HIT for those requests, and the test expectations were revised accordingly.
