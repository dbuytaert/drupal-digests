## Summary

Drupal's `main` branch (targeting Drupal 12) has been updated to require Symfony 8. All Symfony components previously pinned to `^7.4` are now pinned to `^8.0` in both `composer.json` and `core/composer.json`, and `composer.lock` has been updated to use the corresponding Symfony 8.0.x releases. Two non-Symfony dependencies were also updated to Symfony 8 compatible versions: `asm89/stack-cors` to v2.4.0 and `php-tuf/composer-stager` to v2.1.0. Because Symfony 8 itself requires PHP 8.4, this change makes PHP 8.4 a transitive minimum for Drupal 12. Tests for `RequestSanitizer` were also updated to reflect Symfony 8's changed behavior around HTTP method override headers. This change is only relevant to sites tracking the `main` development branch.

## Technical details

The change updates all Symfony component version constraints from `^7.4` to `^8.0` in both root `composer.json` (dev-dependencies: `symfony/browser-kit`, `symfony/css-selector`, `symfony/dom-crawler`, `symfony/error-handler`, `symfony/lock`, `symfony/var-dumper`) and `core/composer.json` (runtime dependencies: `symfony/console`, `symfony/dependency-injection`, `symfony/event-dispatcher`, `symfony/filesystem`, `symfony/finder`, `symfony/http-foundation`, `symfony/http-kernel`, `symfony/mailer`, `symfony/mime`, `symfony/process`, `symfony/psr-http-message-bridge`, `symfony/routing`, `symfony/serializer`, `symfony/validator`, `symfony/yaml`). The `composer.lock` was regenerated to resolve Symfony 8.0.6 (or latest 8.0.x patch) for each component. Symfony 8 requires PHP >=8.4, making that a transitive floor for Drupal 12.

Several prerequisite issues and external blockers had to be resolved before this could land. Drupal-side deprecations introduced in Symfony 7.4 were addressed first: `Request::get()` removal ([#3555532](https://www.drupal.org/node/3555532)) and constraint option initialization in constructors ([#3555534](https://www.drupal.org/node/3555534)). The `#[HasNamedArguments]` attribute was removed from Symfony 8 and that cleanup was handled in the sibling issue [#3569092](https://www.drupal.org/node/3569092), which merged immediately before this one. External library support for Symfony 8 was required from `asm89/stack-cors` (bumped to v2.4.0 in the lock), `php-tuf/composer-stager` (bumped to v2.1.0), and `mglaman/phpstan-drupal` (a new release was needed and was the final unblocking step).

On the test side, Symfony 8 removed the ability for `Request` to honor the `X-HTTP-Method-Override` header for safe methods (GET, HEAD, CONNECT, TRACE); those requests are treated as POST. `RequestSanitizer` itself was left unchanged; only the related functional test assertions were updated to match the new Symfony 8 behavior, and a test for the QUERY HTTP method with an explicit return type was also added per reviewer feedback from godotislate.

## Upgrade

Drupal 12 (`main` branch) now requires Symfony 8, which in turn requires PHP 8.4. No code changes are required for sites running Drupal core itself. Contrib module and custom code authors should ensure any direct Symfony dependencies they declare also allow `^8.0`, and should test against PHP 8.4.

If you maintain a custom constraint class, the Symfony 8 change requires constraint options to be initialized in the constructor rather than passed to the parent. See the related fix in [#3555534](https://www.drupal.org/node/3555534) for examples of how core updated its own constraints.
