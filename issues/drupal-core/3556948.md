## Summary

Drupal core now includes a new experimental admin theme called Admin, based on the contributed Gin theme. This provides a modern administrative interface with improved editorial experience, dark mode support, and customizable accent colors. The theme includes two feature flags: one to control sticky action buttons on content forms and another to apply the admin theme to the login page. Site owners can enable and test the new theme but should expect it to evolve as it moves toward stability. The existing Claro admin theme remains available and unchanged.

## Impact

No upgrade or configuration changes required.

## Technical details

The Admin theme is a fork of the Gin 6.x contributed theme, merged into core as an experimental alpha theme. It ships with extensive CSS, JavaScript, templates, and preprocessing hooks to provide a comprehensive admin interface. The theme includes its own library system, overrides many core libraries, and extends others to customize the admin experience.

Two settings control specific behaviors. The `core_admin_theme_use_sticky_action_buttons` setting defaults to `never` but can be set to `content_forms` or `always` to enable sticky save buttons. The `user_login_admin_theme` setting defaults to `TRUE`, applying the admin theme to login pages.

The merge includes a large migration directory containing CSS and JavaScript not yet deduplicated with Claro components. Future work will consolidate these shared components. The theme provides hooks like `hook_admin_content_form_routes()` allowing modules to register routes that should use the admin edit form layout. Many class names and internal references still use "gin" terminology, planned for cleanup after component deduplication.

The theme is marked experimental in `admin.info.yml`, allowing it to evolve without meeting all core gates during the alpha phase. ESLint ignores the migration directory temporarily. A test fix corrected a taxonomy term creation bug dating to 2012.

## Lessons learned

- When merging large experimental features into core, defer non-critical refinements to follow-up issues. The initial merge establishes the foundation; incremental improvements in separate issues allow for focused review and prevent scope creep from blocking the main integration. *(via pameeela, jurgenhaas)*
- When shipping both preprocessed source files (.pcss.css) and their compiled output (.css), the build must produce clean, unchanged output. If source and compiled files diverge, subsequent contributors cannot determine which version is canonical, breaking the build chain. *(via longwave)*
