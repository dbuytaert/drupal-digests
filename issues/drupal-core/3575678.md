## Summary

The locale module now detects changes to local translation files by comparing file content hashes instead of modification timestamps. This fixes issues in Docker-based deployments where file timestamps are not preserved during image builds, which previously caused unnecessary re-imports of unchanged translation files on every deployment. The new approach only re-imports translations when the actual file content changes, improving deployment performance and reliability.

## Technical details

The locale module previously used `filemtime()` to determine whether a `.po` translation file had changed and needed re-import. This was unreliable because file modification timestamps can change without content changing (common in Docker builds using `COPY` commands, file copies, or backup restores), triggering unnecessary re-imports.

The change introduces a content-based approach using xxHash (`hash_file('xxh128', ...)`) to detect actual content changes. A new `hash` column was added to the `{locale_file}` table to store these values. When checking for translation updates, `locale_translation_status_save()` now compares the stored hash against the current file hash for local files. Only content changes trigger re-import.

For backward compatibility with existing installations, the code falls back to timestamp comparison when no hash is stored (legacy pre-update rows). Once a file is imported after the update, its hash is stored and used for all future comparisons. Remote files continue using timestamp comparison from HTTP `Last-Modified` headers since file content is not locally available until download.

Hash computation happens in four places: `LocaleSource::sourceCheckFile()` when scanning local files, `locale_translate_file_create()` when creating file objects for import, `locale_translation_download_source()` after downloading remote files, and the hash is populated in `LocaleSource::sourceBuild()` from file history.
