## Summary

The procedural function `views_add_contextual_links()` in `views.module` has been replaced by a new `Drupal\views\ContextualLinksHelper` service. The function is deprecated in Drupal 11.4.0 and will be removed in 13.0.0. This change is internal infrastructure; the behavior of contextual links on views pages, blocks, and embedded views is unchanged. Module developers who call `views_add_contextual_links()` directly, or subclass `ViewsBlockBase`, need to update their code.

## Impact

Affects module maintainers.

- **API change:** `ViewsBlockBase::__construct()` gains a new `?ContextualLinksHelper $contextual_links = NULL` parameter; omitting it is deprecated in drupal:11.4.0 and the parameter will be required in drupal:13.0.0
- **Deprecation:** `views_add_contextual_links()` deprecated in drupal:11.4.0, removed in drupal:13.0.0; use `Drupal\views\ContextualLinksHelper::addLinks()` instead (see #3382344)
- **New API:** New `Drupal\views\ContextualLinksHelper` service with `addLinks(array &$renderElement, string $location, string $displayId, ?array $viewElement = NULL): void`

## Technical details

The `views_add_contextual_links()` function was called in three places in core: `Element\View::preRenderViewElement()`, `ViewsBlockBase::addContextualLinks()`, and `ViewPageController::handle()`. Unit tests in `ViewsBlockTest` and `ViewPageControllerTest` previously worked around the procedural dependency by declaring namespace-scoped stub functions guarded by `if (!function_exists(...))`, each carrying a `@todo` referencing this issue. Those stubs are now removed.

The new `ContextualLinksHelper` class (`core/modules/views/src/ContextualLinksHelper.php`) takes `ModuleHandlerInterface`, `EntityTypeManagerInterface`, and the `plugin.manager.views.display` plugin manager as constructor dependencies. The display manager is injected using the `#[Autowire(service: 'plugin.manager.views.display')]` attribute because it is not bound by type alone. The service is registered in `views.services.yml` using `~` (null configuration), relying on global autowire defaults.

`ViewPageController` previously had no constructor and did not implement `ContainerInjectionInterface`. It now does both, using `AutowireTrait` to wire `ContextualLinksHelper` automatically.

`ViewsBlockBase::__construct()` gains a new `?ContextualLinksHelper $contextual_links = NULL` parameter. Omitting it triggers a deprecation and falls back to `\Drupal::service()` for BC. `ViewsBlockBase::create()` passes the service from the container. The `contextual_links_locations` normalization logic was simplified in the new service: the original three-branch `if/elseif/else` is replaced by a single `array_filter($plugin['contextual_links_locations'] ?? ['view'])` call.

## Upgrade

Replace direct calls to `views_add_contextual_links()`:

```php
// Before
views_add_contextual_links($element, 'view', $display_id);

// After
use Drupal\views\ContextualLinksHelper;
\Drupal::service(ContextualLinksHelper::class)->addLinks($element, 'view', $display_id);
```

Subclasses of `ViewsBlockBase` that override `__construct()` should add and forward the new parameter:

```php
public function __construct(
  array $configuration,
  $plugin_id,
  $plugin_definition,
  ViewExecutableFactory $executable_factory,
  EntityStorageInterface $storage,
  AccountInterface $user,
  ?ContextualLinksHelper $contextual_links = NULL,
) {
  parent::__construct($configuration, $plugin_id, $plugin_definition, $executable_factory, $storage, $user, $contextual_links);
}

public static function create(ContainerInterface $container, array $configuration, $plugin_id, $plugin_definition) {
  return new static(
    $configuration, $plugin_id, $plugin_definition,
    $container->get('views.executable'),
    $container->get('entity_type.manager')->getStorage('view'),
    $container->get('current_user'),
    $container->get(ContextualLinksHelper::class),
  );
}
```
