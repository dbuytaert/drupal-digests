## Summary

Drupal's Views module uses a text-trimming helper that respects word boundaries when shortening field output. That helper previously called the Oniguruma-based `mb_regex_encoding()` and `mb_ereg()` functions, which PHP 8.6 deprecates as part of an RFC to retire the bundled Oniguruma regex engine. On PHP 8.6 this caused deprecation notices in CI. The fix removes the `mb_ereg()` code path entirely and relies on `preg_match()` with the `/us` flags (Unicode mode + dotall) for all word-boundary trimming, which handles multibyte strings correctly without the deprecated functions.

## Impact

Affects site owners.

- **Requirements:** PHP 8.6 compatibility: `mb_regex_encoding()` and `mb_ereg()` are deprecated in PHP 8.6; the Views word-boundary text trimmer no longer calls them.

## Technical details

The affected code is the static `trimText()` method in `core/modules/views/src/Plugin/views/field/FieldPluginBase.php`. It previously checked `function_exists('mb_ereg')` and, when true, called `mb_regex_encoding('UTF-8')` followed by `mb_ereg($regex, $value, $matches)` for word-boundary detection. The fallback was `preg_match("/$regex/us", $value, $matches)`. The fix removes the entire conditional and uses only the `preg_match` call with the `u` flag, which instructs PCRE to treat the pattern and subject as UTF-8, making it a correct equivalent for multibyte input. An initial approach (MR 15637) simply dropped `mb_ereg` without additional test coverage, which prompted alexpott to raise a multibyte regression concern. The final committed version (MR 15728, from andypost) retained the same `preg_match` replacement but added Hindi-script strings (`हिंदी परीक्षण स्ट्रिंग`) to `testTrimText()` to demonstrate correct truncation at character boundaries.
