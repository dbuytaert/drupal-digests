## Summary

Drupal's update system did not always run system module database updates before other modules' updates, even when no explicit dependencies prevented it. This could cause cryptic failures during major or minor version upgrades, particularly when system updates made foundational changes that other updates depended on. The update dependency graph now ensures system updates execute as early as possible while still respecting explicit dependencies declared via `hook_update_dependencies()`. Sites performing upgrades across Drupal versions with many pending updates will now have more reliable update paths.

## Impact

No upgrade or configuration changes required.

## Technical details

The `update_build_dependency_graph()` function in `core/includes/update.inc` now manipulates the dependency graph to prioritize system module updates. hchonov provided a concrete reproducible case: updating a standard-profile site from 8.4.8 to 8.6.x caused `taxonomy_update_8501` and `taxonomy_update_8502` to run before `system_update_8501` with no dependency requiring that order, because `\Drupal\Component\Graph\Graph::searchAndSort()` assigned lower weights to taxonomy updates based on graph topology rather than module identity.

The fix adds edges to the dependency graph so that updates without any existing path to a system update are given an explicit dependency on the last system update, ensuring all system updates run first. When `hook_update_dependencies()` forces specific non-system updates before a particular system update, the code builds an interim graph to identify those constrained updates, then adds edges from the previous system update rather than the last, preserving the explicit dependency chain while still running as many system updates as early as possible.

The issue body proposed adding a general assertion in `\Drupal\FunctionalTests\Update\UpdatePathTestBase` to verify system updates execute before any non-system update across all update path tests, covering the simple case (no relevant dependencies) and the complex case (some non-system updates must precede a specific system update).
