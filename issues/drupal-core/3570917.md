## Summary

The function that generates the image upload settings form for text editors has been converted to a service. This change supports the ongoing effort to modernize core by eliminating procedural code from .module files. The form itself remains unchanged, so site owners see no difference. Third-party editor plugins that provide image upload configuration should use the new service instead of the deprecated function.

## Impact

Affects module maintainers.

- **Deprecation:** `editor_image_upload_settings_form()` deprecated; use `EditorImageUploadSettings::getForm()` instead

## Technical details

The `editor_image_upload_settings_form()` function previously lived in `editor.admin.inc` and provided a reusable form for configuring image upload settings. Any text editor plugin offering image insertion could call this function to generate the upload settings form, including fields for storage scheme, directory, file size limits, and image dimensions.

This function has been moved to a new `EditorImageUploadSettings` service with a `getForm()` method. The service accepts `ConfigFactoryInterface` and `StreamWrapperManagerInterface` through constructor injection, and uses `StringTranslationTrait` for translations. The form structure and behavior remain identical.

Core plugins like CKEditor5 `Image` and the test `UnicornEditor` plugin now inject this service and call `getForm()` instead of loading the .inc file and calling the procedural function. The deprecated function now triggers an error and delegates to the service.

The entire `editor.admin.inc` file is deprecated in 11.4 and will be removed in 13.0, since it contains only this one function. An unrelated change in the diff moves Olivero theme hooks to a hook class, which appears to be part of the same broader modernization effort.

## Upgrade

Custom text editor plugins that call `editor_image_upload_settings_form()` should inject the `EditorImageUploadSettings` service and call `getForm()` instead.

Before:
```php
public function buildConfigurationForm(array $form, FormStateInterface $form_state) {
  $form_state->loadInclude('editor', 'admin.inc');
  $form['image_upload'] = editor_image_upload_settings_form($form_state->get('editor'));
  return $form;
}
```

After:
```php
public function __construct(
  array $configuration,
  $plugin_id,
  $plugin_definition,
  #[Autowire(service: EditorImageUploadSettings::class)]
  protected EditorImageUploadSettings $editorImageUploadSettings,
) {
  parent::__construct($configuration, $plugin_id, $plugin_definition);
}

public function buildConfigurationForm(array $form, FormStateInterface $form_state) {
  $form['image_upload'] = $this->editorImageUploadSettings->getForm($form_state->get('editor'));
  return $form;
}
```

The plugin must also implement `ContainerFactoryPluginInterface` to support dependency injection.

The initial approach proposed a trait, but after discussion the team decided a service was more appropriate. Contributors noted that injecting services into traits is complex and unsolved, and this function itself uses services internally. The conversion to a service was straightforward.
