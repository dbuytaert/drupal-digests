## Summary

The `js.module.css` stylesheet no longer uses `.js` class-based selectors to control JavaScript-dependent visibility. All visibility logic is now expressed purely through standard CSS `@media (scripting: enabled)` and `@media (scripting: none)` queries, which are well-supported across Drupal's supported browsers. The `.js-hide` rule now carries `display: none !important` to ensure it overrides other display rules regardless of selector specificity. The stylesheet is slightly smaller as a result. There is no user-facing effect on supported browsers.

## Impact

Affects front-end developers.

- **Theme change:** `.js-hide` now uses `display: none !important`; any custom CSS that overrides `.js-hide` without `!important` to reveal an element will no longer take effect.

## Technical details

The original `js.module.css` had two layers of visibility rules, introduced in [#3404218](https://www.drupal.org/node/3404218) to fix a layout-shift regression. The first layer used `.js .js-hide` and `.js .js-show` selectors that relied on JavaScript adding a `.js` class to the `<html>` element. The second layer used `@media (scripting: enabled)` blocks with a doubled `.js-hide.js-hide` selector specifically to out-rank the first layer.

This issue removes the legacy first layer entirely. Since the `@media (scripting: ...)` media feature is now broadly supported across all of Drupal's required browsers, the `.js`-class fallback is no longer needed.

The resulting rules are:
- `@media (scripting: none) { .js-show { display: none !important } }`: prevents JavaScript-only content from showing in no-JS environments.
- `@media (scripting: enabled) { .js-hide { display: none !important } }`: hides no-JS content when scripting is active.

The `display: none !important` is deliberate. An initial commit (f9be36f0) was merged and then immediately reverted after lendude discovered that buttons in the Views wizard were unexpectedly visible. Catch traced the root cause to a CSS specificity conflict: Claro's `.button { display: inline-block }` was winning over the bare `.js-hide` selector once the higher-specificity `.js .js-hide` prefix was gone. The thread debated three remedies: the existing Drupal core convention `.js-hide.js-hide`, the Drupal CMS convention `.js-hide[class]`, and `!important`. After jwilson3 noted that the change simultaneously removed all `display: block` usages from `.js-show`, making `.js-hide`/`.js-show` exclusively about hiding, catch concluded that `!important` is semantically appropriate here and is justified as a real CSS feature rather than a hack.

The `display: block` rule for `.js-show` inside `@media (scripting: enabled)` is also removed. Elements with `.js-show` now simply appear at their natural `display` value when scripting is active, which is more flexible for cases where `flex`, `inline-block`, or other values are needed.

The `StylesheetBytes` assertion in the performance test for anonymous page loads was reduced from 1500 to 1000 to reflect the smaller CSS output. Catch noted that removing the `.js`-class selectors may also eliminate a style recalculation previously triggered when JavaScript added the `.js` class after page load.

A follow-up issue ([#3568172](https://www.drupal.org/node/3568172)) plans to move the `.js-show`/`.js-hide` rules into a separate library loaded only when JavaScript is active.
