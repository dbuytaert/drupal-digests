## Summary

The cron settings form has been modernized to use a more efficient pattern for handling configuration values. The form now extends the proper base class and uses a simpler mechanism to map form fields directly to configuration, eliminating the need for manual save operations. This change is part of a broader effort to clean up and standardize how core forms handle configuration. Site owners will see no difference in how the cron settings form looks or behaves.

## Impact

Affects module maintainers.

- **Deprecation:** `automated_cron_settings_submit()` function is deprecated; configuration now handled automatically by the form.

## Technical details

`CronForm` now extends `ConfigFormBase` instead of `FormBase`. Previously it used `ConfigFormBaseTrait` but did not benefit from the full `ConfigFormBase` functionality. The constructor now uses promoted properties and accepts `TypedConfigManagerInterface` as required by the parent class.

The form's `logging` field now uses `#config_target` property set to `system.cron:logging` instead of manually setting `#default_value` from config. The custom `submitForm()` method that manually saved the configuration value has been removed entirely. The parent class now handles all configuration saving automatically through the `#config_target` mechanism.

Similarly, `AutomatedCronHooks::formSystemCronSettingsAlter()` was updated to use `#config_target` for the `interval` field targeting `automated_cron.settings:interval`. The manual submit handler `automated_cron_settings_submit()` is no longer added to the form's `#submit` array, and the function itself is now deprecated with a trigger error. The `#theme` override is no longer needed because `CronForm` properly extends `ConfigFormBase`.

The change also removes PHPStan baseline entries that were ignoring missing return type specifications that are now properly inherited from the parent class.
