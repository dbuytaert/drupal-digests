## Summary

Since Drupal 8, forms have carried native HTML5 `required` and constraint attributes that cause browsers to intercept form submissions client-side before Drupal's Form API (FAPI) ever runs. The result is generic, browser-localized error messages ("Please fill out this field.") instead of Drupal's descriptive, accessible FAPI messages. Only the first invalid field surfaces an error, and screen readers handle this poorly. This change introduces a new `enable_html5_validation` setting in `settings.php` so site owners can disable browser-native HTML5 validation and rely on Drupal's server-side validation messages instead. In Drupal 11 the setting defaults to enabled, with a status report warning prompting admins to prepare. In Drupal 12 the default changes to disabled; in Drupal 13 the setting will be removed and HTML5 validation will always be off.

## Impact

Affects site owners.

- **Config change:** New `enable_html5_validation` setting in `settings.php` controls whether browser-native HTML5 form validation is active; defaults to `TRUE` in Drupal 11, `FALSE` in Drupal 12, and will be removed in Drupal 13.

## Technical details

The root problem is that the HTML5 `required` attribute (and other constraint attributes like `type="email"`) causes browsers to block form submission before FAPI runs. Browser validation messages are inconsistent: they only report the first invalid field, use the browser's locale rather than Drupal's language, and are handled poorly by screen readers (screen reader testing confirmed JAWS reads only "Alert: Please fill out this field" regardless of field name, while VoiceOver on Safari reads nothing at all). The `#limit_validation_errors` use case was also affected, though the existing `formnovalidate` attribute on affected buttons already partially mitigated that.

The fix adds a `Settings::get('enable_html5_validation', TRUE)` check inside `preprocessForm()` in the form element preprocessor. When the setting is `FALSE`, a `novalidate` attribute is added to the rendered `<form>` element, disabling all browser HTML5 validation globally. A documentation block and commented-out example (`# $settings['enable_html5_validation'] = TRUE;`) are added to `default/settings.php`. The `SystemManager::checkRequirements()` method gains a new status-report check: when the setting is `NULL` (not configured), it warns that Drupal 12 will default to `FALSE` and advises testing; when the setting is `TRUE`, it warns that the setting will be removed in Drupal 13. No warning is shown when the setting is `FALSE`.

Three form validation tests (`testValidateLimitErrors`, `testPatternValidation`, `testCustomRequiredError`) are moved from a non-JS functional test class to a new `FunctionalJavascript` test class (`Drupal\Tests\system\FunctionalJavascript\Form\ValidationTest`). The new class sets `enable_html5_validation` to `FALSE` in `setUp()` via `writeSettings()`, ensuring the tests run in a real browser environment with HTML5 validation disabled so FAPI messages are testable.

## Upgrade

To disable HTML5 validation on all forms and use Drupal's FAPI validation messages, add the following to `settings.php`:

```php
$settings['enable_html5_validation'] = FALSE;
```

To explicitly keep the current behavior (HTML5 validation on) and suppress the status report warning in Drupal 11, set:

```php
$settings['enable_html5_validation'] = TRUE;
```

Note: setting `TRUE` will display a different status report warning reminding you to prepare for removal in Drupal 13. The setting will be removed entirely in Drupal 13, at which point HTML5 validation will always be disabled.
