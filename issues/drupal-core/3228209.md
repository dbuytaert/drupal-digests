## Summary

The `hasRole()` method has been added to `AccountInterface`, making it available on all account objects. Previously, this method was only available on `UserSession` and `AccountProxy` classes, or by loading the full user entity. This change provides a consistent API for checking user roles across different account implementations, eliminating the need to load the full user entity just to check roles. The method was already implemented in the concrete classes, so this change formalizes the interface contract.

## Impact

Affects module maintainers.

- **New API:** `AccountInterface::hasRole()` method added to check if an account has a specific role.

## Technical details

In a previous change ([#2991232](https://www.drupal.org/project/drupal/issues/2991232)), the `hasRole()` method was added to `AccountProxy` and `UserSession` classes but not to `AccountInterface` to maintain backward compatibility with existing implementations. This issue completes that work by moving the method definition to `AccountInterface`.

The change adds `hasRole(string $rid)` to `AccountInterface` and removes the duplicate method definition from `UserInterface` (which extends `AccountInterface`). The `User` entity class signature was updated to include the `string` type hint for the `$rid` parameter to match the interface. TODO comments referencing this issue were removed from `AccountProxy` and `UserSession`.

The implementation uses `in_array()` to check if the role ID exists in the array returned by `getRoles()`. This provides a standard way to check roles without needing to load the full user entity when working with account proxy or session objects.

After initially being committed to Drupal 11.4, the change was briefly reverted due to test failures, then re-committed successfully.

## Upgrade

If you have custom code that implements `AccountInterface`, you must now implement the `hasRole()` method:

```php
public function hasRole(string $rid) {
  return in_array($rid, $this->getRoles(), TRUE);
}
```

Most implementations can follow the same pattern as `UserSession` by checking if the role ID exists in the array returned by `getRoles()`.

See the change record at [https://www.drupal.org/node/3283218](https://www.drupal.org/node/3283218) for more details.

## Lessons learned

- When adding methods to interfaces in a minor release, you can do so without breaking backward compatibility if the interface is not tagged @api and has a 1:1 or similar relationship with implementing classes or base classes/traits. This allows new features to be added while giving module authors the option to implement the method or rely on a base implementation. *(via catch)*
- When a framework maintains separate interfaces for runtime session objects (lightweight, no database dependency) and full entity objects (full data access), moving methods between them creates a tight coupling. If code commonly needs to load the full entity from the session object to access a method, the abstraction boundary may be unclear, but consolidating interfaces requires careful consideration of the dependency implications. *(via berdir)*
