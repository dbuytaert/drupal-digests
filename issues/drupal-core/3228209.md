## Summary

The `hasRole()` method has been added to `AccountInterface`, making it available on all account objects. Previously, this method was only available on `UserSession` and `AccountProxy` classes, or by loading the full user entity. This change provides a consistent API for checking user roles across different account implementations, eliminating the need to load the full user entity just to check roles. The method was already implemented in the concrete classes, so this change formalizes the interface contract.

## Impact

Affects module maintainers.

- **New API:** `AccountInterface::hasRole(string $rid)` added, making role checks available on all account objects without loading the full user entity.

## Technical details

In a previous change ([#2991232](https://www.drupal.org/node/2991232)), the `hasRole()` method was added to `AccountProxy` and `UserSession` classes but not to `AccountInterface` to maintain backward compatibility with existing implementations. This issue completes that work by moving the method definition to `AccountInterface`.

The change adds `hasRole(string $rid)` to `AccountInterface` (in `core/lib/Drupal/Core/Session/AccountInterface.php`) and removes the duplicate method definition (along with its `@todo` forward reference) from `UserInterface` (in `core/modules/user/src/UserInterface.php`). The `User` entity class in `core/modules/user/src/Entity/User.php` had its `hasRole()` signature updated to add the `string` type hint to `$rid` to match the new interface. `@todo` comments referencing this issue were removed from `AccountProxy` and `UserSession`.

`UserSession::hasRole()` checks with `in_array($rid, $this->getRoles(), TRUE)`. `AccountProxy::hasRole()` delegates to `$this->getAccount()->hasRole($rid)`. There was debate during the issue about whether `AccountInterface` is effectively `@api` (which would require a more involved deprecation path for adding methods), but contributors concluded that `AccountInterface` is not tagged `@api` and that the existing concrete implementations in core gave adequate coverage to proceed without a deprecation cycle. A `@return` type hint was deliberately omitted from the interface signature due to BC constraints in Drupal 11.

## Upgrade

If you have custom code that implements `AccountInterface`, you must now add a `hasRole()` method:

```php
public function hasRole(string $rid) {
  return in_array($rid, $this->getRoles(), TRUE);
}
```

Most implementations can follow the same pattern as `UserSession` by checking if the role ID exists in the array returned by `getRoles()`.

See the change record at https://www.drupal.org/node/3283218 for more details.
