## Summary

Drupal's front-controllers (`index.php`, `update.php`) and all core CLI scripts now bootstrap through `symfony/runtime`. Instead of directly instantiating `DrupalKernel`, creating a `Request`, calling `handle`/`send`/`terminate`, and exiting, they now require a generated `autoload_runtime.php` file and return a closure that produces the kernel or console application object. The runtime layer then takes over request handling. The scaffold plugin generates `autoload_runtime.php` in the web root on every `composer install`, alongside the existing `autoload.php`. Existing custom front-controllers using the old pattern continue to work unchanged, confirmed by a new functional test. The long-term payoff is a well-defined seam where environment-setup code currently buried in `DrupalKernel` can be extracted in follow-on issues, and where alternative runtimes (such as FrankenPHP or AMPHP) can be substituted via an environment variable without touching any front-controller.

## Impact

Affects module maintainers and site owners.

- **Requirements:** New Composer dependency on `symfony/runtime ^8.0`; `symfony/runtime` Composer plugin must be listed in `allow-plugins`.
- **API change:** Front-controllers now return a closure instead of bootstrapping directly; `$GLOBALS['autoloader']` is deprecated in `drupal:11.3.0` and removed in `drupal:12.0.0`.
- **Deprecation:** `$GLOBALS['autoloader']` access deprecated via `DeprecatedAutoloadAccess`; callers should require `vendor/autoload.php` directly.
- **New API:** New `DrupalRuntime` class (`core/lib/Drupal/Core/Runtime/DrupalRuntime`); generated `autoload_runtime.php` in the web root; `APP_RUNTIME` environment variable to substitute the runtime class.

## Technical details

The diff adds `symfony/runtime ^8.0` as a Drupal core dependency and introduces two new classes plus a new scaffold-generated file.

`DrupalRuntime` (in `core/lib/Drupal/Core/Runtime/`) extends `SymfonyRuntime`. Its constructor enforces three Drupal-specific defaults before calling `parent::__construct($options)`: it sets `error_handler` to `FALSE` (preserving Drupal's own error handlers), sets `disable_dotenv` to `TRUE` (Drupal does not use dotenv), and defaults the env key (`APP_ENV`) to `'prod'` instead of Symfony's default `'dev'`. It also places a `DeprecatedAutoloadAccess` instance into `$GLOBALS['autoloader']` to preserve backward compatibility for code that reads the global.

`DeprecatedAutoloadAccess` (also in `core/lib/Drupal/Core/Runtime/`) extends `Composer\Autoload\ClassLoader`. Every method override immediately fires an `E_USER_DEPRECATED` error (deprecated in `drupal:11.3.0`, removed in `drupal:12.0.0`) before lazily traversing the filesystem to find and delegate to the real `autoload.php`. This allows the global to remain set without silently perpetuating usage.

The scaffold plugin gains `GenerateAutoloadRuntimeReferenceFile`, which writes `autoload_runtime.php` to the web root (parallel to the existing `autoload.php` generation). The generated file sets `$_ENV['APP_RUNTIME']` to `DrupalRuntime::class` unless overridden by `$_SERVER['APP_RUNTIME']` or `$_ENV['APP_RUNTIME']`, then delegates to `vendor/autoload_runtime.php` (the file generated by the `symfony/runtime` Composer plugin itself). Setting `APP_RUNTIME` to any class implementing `Symfony\Component\Runtime\RuntimeInterface` swaps the entire runtime without modifying any front-controller.

The updated `index.php` and `update.php` now consist of a single `require_once 'autoload_runtime.php'` call followed by `return static function () { return new DrupalKernel('prod', require 'autoload.php'); };`. The same pattern is applied to all core CLI scripts under `core/scripts/`. `symfony/runtime` is added to the scaffold plugin's `ComposerPluginsValidator` allowed-plugin list. A `LegacyFrontControllerTest` functional test and companion test module verify that the pre-existing bootstrap pattern continues to work. Related dependency declaration: [#3552922](https://www.drupal.org/node/3552922).

## Upgrade

**Allow the `symfony/runtime` Composer plugin** in your project's root `composer.json`. Without this, `composer install` will prompt interactively or fail in non-interactive CI environments:

```json
{
  "config": {
    "allow-plugins": {
      "symfony/runtime": true
    }
  }
}
```

**`autoload_runtime.php`** is generated automatically by the scaffold plugin on `composer install`. No manual action is needed for sites managed through the scaffold plugin.

**Custom front-controllers** using the old pattern continue to work. Migrating to the new pattern is encouraged:

```php
// Old pattern (still works):
use Drupal\Core\DrupalKernel;
use Symfony\Component\HttpFoundation\Request;

$autoloader = require_once 'autoload.php';
$kernel = new DrupalKernel('prod', $autoloader);
$request = Request::createFromGlobals();
$response = $kernel->handle($request);
$response->send();
$kernel->terminate($request, $response);

// New pattern:
use Drupal\Core\DrupalKernel;

require_once 'autoload_runtime.php';

return static function () {
  return new DrupalKernel('prod', require 'autoload.php');
};
```

**`$GLOBALS['autoloader']`** is deprecated as of `drupal:11.3.0` and will be removed in `drupal:12.0.0`. Code that reads this global will trigger `E_USER_DEPRECATED`. Replace it by requiring `vendor/autoload.php` directly or passing the autoloader explicitly:

```php
// Deprecated:
$loader = $GLOBALS['autoloader'];

// Replacement:
$loader = require_once __DIR__ . '/vendor/autoload.php';
```

**To select a custom runtime**, set `APP_RUNTIME` in `$_SERVER` (via the web server configuration) or in `$_ENV` before `autoload_runtime.php` is required. The value must be the fully-qualified class name of a class implementing `Symfony\Component\Runtime\RuntimeInterface`.

### Collaboration

The thread contains a substantive design debate that shaped the final approach. `donquixote` challenged whether the MR had clear payoff since it added new code without yet simplifying `DrupalKernel`, and asked for a proof-of-concept showing the benefit. `kingdutch` responded with a detailed comment enumerating `DrupalKernel`'s four conflated responsibilities and explaining that async event-loop work in [#3425210](https://www.drupal.org/node/3425210) made the problem concrete: `EventLoop::run` must live outside the kernel but without a well-defined place, the only option was duplicating code across all front-controllers.

`nicxvan` questioned whether using `symfony/runtime`'s resolver mechanism added unnecessary complexity given that FrankenPHP runtimes in the wild do not use it. `kingdutch` explained that the resolver is preparation for future issues (such as [#2501901](https://www.drupal.org/node/2501901) for env-var-controlled environment names) and declined to collapse it, keeping the issue focused on introducing the component without rewriting anything else.

`ghost of drupal past` proposed a simpler autoload template that skips the closure-returning front-controller pattern and passes the kernel directly to the runtime, and asked why `DrupalKernelRunner` was needed instead of Symfony's `HttpKernelRunner`. `kingdutch` explained that Symfony's `HttpKernelRunner` configures output buffering that Drupal does not currently do, and that `SymfonyRuntime` registers an error handler that would silently change Drupal's behavior, so both were deferred to separate issues via deprecation flow.

`znerol` flagged thread-safety concerns around `putenv`/`getenv` in multi-threaded runtimes like FrankenPHP and recommended exclusive reliance on `$_SERVER`. `kingdutch` acknowledged the `putenv` use (copied from `SymfonyRuntime` for the console path) but pushed back on dropping `$_ENV` support entirely, arguing it would break documented Symfony Runtime behavior and that Drupal's own Docker image uses `EGPCS` on the command line. `catch` confirmed after an in-person discussion in Vienna with `kingdutch` and `alexpott` that the approach is sound and cleared the framework manager review tag.
