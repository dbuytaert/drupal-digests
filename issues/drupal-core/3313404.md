## Summary

Drupal's front-controllers (`index.php`, `update.php`) and core CLI scripts now bootstrap through `symfony/runtime`. A new `autoload_runtime.php` glue file is generated by the scaffold plugin on every `composer install`, replacing the former pattern of directly requiring `autoload.php`, instantiating `DrupalKernel`, creating a `Request`, and calling `handle`/`send`/`terminate` by hand. The runtime layer creates a clean boundary between environment setup and request handling. Existing custom front-controllers using the old pattern continue to work unchanged. The practical payoff comes in follow-on issues: once this foundation is in place, environment-setup code currently buried in `DrupalKernel` (error handlers, site-path discovery, request globals) can be moved into the runtime and made replaceable without touching every front-controller again.

## Impact

Affects module maintainers and site owners.

- **Requirements:** New Composer dependency on `symfony/runtime ~v7.3`; `symfony/runtime` Composer plugin must be allowed in `allow-plugins`.
- **API change:** `DrupalKernel::__construct()` first parameter widened from `string` to `string|\Drupal\Core\AppContext`; passing `AppContext` is the new recommended pattern from the generated front-controllers.
- **New API:** New public classes `DrupalRuntime` (extends `SymfonyRuntime`, bypassing dotenv) and `AppContext` (readonly value object holding runtime options and `getEnvironment()`); new `autoload_runtime.php` glue file generated by the scaffold plugin; `APP_RUNTIME` env var to select the runtime class.

## Technical details

The diff adds `symfony/runtime ~v7.3` as a Drupal core dependency and introduces two new public classes. `DrupalRuntime` (in `core/lib/Drupal/Core/Runtime/`) extends `SymfonyRuntime` but calls `GenericRuntime::__construct()` directly to bypass the dotenv-handling constructor in `SymfonyRuntime`, since Drupal does not use dotenv. `AppContext` (in `core/lib/Drupal/Core/`) is a readonly value object holding runtime options; its `getEnvironment()` reads from an env-var key stored in those options and falls back to `'prod'`.

The scaffold plugin gains a `GenerateAutoloadRuntimeReferenceFile` class that generates `autoload_runtime.php` in the web root (parallel to the existing `autoload.php` generation). This file returns a readonly anonymous class (referred to as "the glue") that on construction loads the autoloader, reads `APP_RUNTIME` from `$_SERVER` or `$_ENV` to select the runtime class (defaulting to `DrupalRuntime`), instantiates the runtime, and creates an `AppContext`. Calling the glue object with any application instance delegates to `$this->runtime->getRunner($app)->run()` and exits. The `APP_RUNTIME` environment variable therefore allows third parties to substitute a different runtime (such as FrankenPHP or AMPHP) without modifying any front-controller.

`DrupalKernel::__construct()` now accepts `string|AppContext` as its first parameter. When an `AppContext` is passed, the kernel extracts the environment string from it via `getEnvironment()`, preserving full backward compatibility for callers that pass a plain string. The updated front-controllers use `$glue->appContext` and `$glue->autoloader` as the constructor arguments. A test module (`legacy_front_controller_test`) and a functional test (`LegacyFrontControllerTest`) confirm that the pre-existing bootstrap pattern continues to work. The `symfony/runtime` Composer plugin is added to the list of plugins explicitly allowed by the scaffold plugin validator. Related: the companion dependency-declaration issue is [#3552922](https://www.drupal.org/node/3552922).

## Upgrade

For most sites managed through the scaffold plugin, no manual changes are needed. `autoload_runtime.php` is generated automatically on `composer install`.

**Allow the symfony/runtime Composer plugin** in your project's root `composer.json` if your project explicitly manages `allow-plugins`:

```json
{
  "config": {
    "allow-plugins": {
      "symfony/runtime": true
    }
  }
}
```

**Custom front-controllers** using the old pattern still work. To migrate a custom front-controller to the new pattern:

```php
// Old pattern:
use Drupal\Core\DrupalKernel;
use Symfony\Component\HttpFoundation\Request;

$autoloader = require_once 'autoload.php';
$kernel = new DrupalKernel('prod', $autoloader);
$request = Request::createFromGlobals();
$response = $kernel->handle($request);
$response->send();
$kernel->terminate($request, $response);

// New pattern:
use Drupal\Core\DrupalKernel;

$glue = require_once 'autoload_runtime.php';
$glue(new DrupalKernel($glue->appContext, $glue->autoloader));
```

**Code that type-hints the first argument of `DrupalKernel::__construct()` as `string`** should be updated to accept `string|\Drupal\Core\AppContext`.

**Selecting a custom runtime** at runtime is possible by setting the `APP_RUNTIME` environment variable to the fully-qualified class name of any class implementing `Symfony\Component\Runtime\RuntimeInterface`. Set it in `$_SERVER` (via the web server) or `$_ENV` before `autoload_runtime.php` is required.

The thread contains a substantive design debate. `donquixote` challenged whether the MR had clear payoff, since it added new code without yet simplifying `DrupalKernel`, and asked for a proof-of-concept showing the benefit. `kingdutch` responded with a detailed walkthrough of `DrupalKernel`'s four conflated responsibilities and explained that the event-loop integration work in [#3425210](https://www.drupal.org/node/3425210) made the problem concrete: `EventLoop::run` must live outside the kernel, but without a well-defined place to put it, the only option was duplicating code across all front-controllers. `nicxvan` questioned whether using `symfony/runtime`'s resolver mechanism added unnecessary complexity given that FrankenPHP runtimes in the wild do not use it, and asked whether `DrupalKernelRunner` was redundant. `kingdutch` explained that the resolver is deliberately prepared for future issues (such as [#2501901](https://www.drupal.org/node/2501901) for env-var-controlled environment names) and that `DrupalKernelRunner` was kept separate from Symfony's `HttpKernelRunner` to avoid silently adopting Symfony's output-buffering defaults and error-handler registration in the same change. `ghost of drupal past` proposed a simpler autoload template that skips the resolver entirely and passes the kernel directly. `znerol` flagged thread-safety concerns around `putenv`/`getenv` in multi-threaded runtimes like FrankenPHP and recommended exclusive reliance on `$_SERVER`. `kingdutch` addressed the `putenv` concern by noting the code is copied from `SymfonyRuntime` for consistency and that the console context in which it appears is not subject to threading, while pushing back on dropping `$_ENV` support as that would break documented Symfony Runtime behavior. `catch` confirmed after an in-person discussion in Vienna with `kingdutch` and `alexpott` that the approach is sound and cleared the framework manager review tag.
