## Summary

The `condition.plugin` config schema in Drupal core had two longstanding errors: it included a `uuid` field that condition plugins do not actually store (they are embedded inside config entities, not config entities themselves), and it required `context_mapping` even though not all condition plugins use it. The fix removes `uuid` from the schema and marks `context_mapping` as optional. The fix was committed to `main` and `11.x`, and was initially backported to `10.6.x` and `11.3.x`, but was reverted from those stable branches after it broke sites using Pathauto, which does write a `uuid` key into its `selection_criteria` condition plugin data. The issue is back to Needs Work to determine whether a core update hook is needed before the change can safely land in stable branches.

## Impact

Affects site owners.

- **Config change:** The `condition.plugin` schema removes the erroneous `uuid` key and marks `context_mapping` as optional; contrib modules that write `uuid` into condition plugin config (e.g. Pathauto) must add their own update hooks to strip that key before this schema change lands on their target branch.

## Technical details

The `condition.plugin` type in `core/config/schema/core.data_types.schema.yml` is the base mapping inherited by all condition plugins. It erroneously declared a `uuid` key (type `uuid`) and listed `context_mapping` as a required key. Condition plugins are not config entities in their own right; they are stored as inline mapping values within other config entities (such as block visibility or pathauto selection criteria). Core never writes a UUID into this data, making the `uuid` schema entry dead weight from an old copy-paste. The missing `requiredKey: false` on `context_mapping` caused schema validation failures for condition plugins that legitimately omit context mapping.

The fix (commit `28b81cc5` on `main`, `fb0632b3` on `11.x`) removes the `uuid` key entirely from the `condition.plugin` mapping and adds `requiredKey: false` to `context_mapping`. `MappingTest` in `core/tests/Drupal/KernelTests/Config/Schema/MappingTest.php` was updated accordingly: `uuid` was dropped from all expected-key lists for `condition.plugin.*` types, and `context_mapping` was added to each test case's list of optional keys.

The issue was originally surfaced by [#3379725](https://www.drupal.org/node/3379725) (making Block config entities `FullyValidatable`), which caused stricter schema validation to run over condition plugin data for the first time.

After the initial merge, it emerged that Pathauto stores a `uuid` key inside its `selection_criteria` condition plugin entries (see comment [#16497591](https://www.drupal.org/node/16497591) for a real config example). This caused `SchemaIncompleteException` failures in CI pipelines with `ConfigSchemaChecker` enabled. The fix was reverted from `10.6.x` (revert `80d562fc`) and `11.3.x` (revert `8b9c5ed1`). The open question is whether a core update hook should strip the `uuid` key from existing condition plugin configuration, or whether each affected contrib module (such as Pathauto, addressed in [#3577286](https://www.drupal.org/node/3577286)) must supply its own update hook.
