## Summary

Drupal core's PHPStan configuration now enforces return type declarations on all new functions and methods. Any new code contributed to core that omits a return type will fail the PHPStan static analysis job. Existing violations at the time of the change are grandfathered into the baseline, so no current code was broken. This makes a long-standing coding standard automatically checkable rather than relying solely on manual review.

## Impact

Affects module maintainers.

- **New API:** Two PHPStan rules now active: `MissingFunctionReturnTypehintRule` and `MissingMethodReturnTypehintRule` enforce return type declarations on all new functions and methods in Drupal core

## Technical details

The change adds two opt-in PHPStan rules to `core/phpstan.neon.dist` without changing the overall analysis level (which remains at level 1): `PHPStan\Rules\Functions\MissingFunctionReturnTypehintRule` and `PHPStan\Rules\Methods\MissingMethodReturnTypehintRule`. The `missingType.iterableValue` identifier is added to the ignored errors list in the same file so that missing iterable value type annotations (a separate concern) do not also trigger failures. All violations that existed at the time of the commit were recorded in `core/.phpstan-baseline.php`, which grew from roughly 2,640 lines (115 KB) to roughly 76,908 lines (3.7 MB). The baseline is the only self-evidently large diff; the meaningful review surface is the three-line change to `phpstan.neon.dist`. After commit, contributors noted edge cases where new code adds an existing, untyped trait or extends an untyped base class method: in those cases the violation must be added to the baseline, or a `@phpstan-return` annotation used on the trait method where adding a native return type would be a BC break. The related issue [#3470913](https://www.drupal.org/node/3470913) tracks a specific case of this covariance problem.

## Upgrade

New functions and methods added to Drupal core (or to contrib modules that adopt `phpstan.neon.dist`) must now include a return type declaration:

```php
// Before (now fails PHPStan)
public function myMethod() {
  return $this->value;
}

// After (required)
public function myMethod(): string {
  return $this->value;
}
```

When a return type genuinely cannot be added (for example, a new implementation of an existing untyped interface method where adding a native return type would be a BC break for downstream extenders), the violation must be added to `core/.phpstan-baseline.php`. The identifier in the baseline will be `missingType.return`.
