## Summary

A bug in `HttpKernelUiHelperTrait::drupalGet()` caused `clickLink()` in kernel tests to navigate to the wrong URL when the clicked link's `href` was a bare query string (for example `?page=1`). The old code prepended `/` to any path not starting with `/`, turning `?page=1` into `/?page=1` and sending the test to the front page instead of the intended page. The fix teaches `drupalGet()` to leave query-string paths alone so they resolve relative to the current page URL as expected.

## Impact

No upgrade or configuration changes required.

## Technical details

In `HttpKernelUiHelperTrait::drupalGet()`, the guard that adds a leading `/` to bare relative paths now also checks whether the path starts with `?`. The original condition `!str_starts_with($path, '/')` was expanded to `!str_starts_with($path, '?') && !str_starts_with($path, '/')`, so paths like `?page=1` pass through unchanged and are resolved relative to the current page URL by the underlying Symfony browser client. The `@param` docblock for `$path` was updated to document this distinction. This trait was introduced in [#3582769](https://www.drupal.org/node/3582769) and `getUrl()` was added in [#3583594](https://www.drupal.org/node/3583594).
