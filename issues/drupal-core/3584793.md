## Summary

Module developers can now declare form routes using a `#[Route]` PHP attribute placed directly on a form class, instead of writing a corresponding entry in a `*.routing.yml` file. This extends the attribute-based route discovery introduced for controllers in [#3311365](https://www.drupal.org/node/3311365) to cover Drupal forms as well. The same standard Symfony `Route` attribute is reused; no separate attribute class is needed. `SiteInformationForm` in the system module was converted as the first real-world example.

## Impact

Affects module maintainers.

- **New API:** Module developers can place `#[Route]` directly on `FormInterface` classes in the `Form` namespace to declare routes, replacing equivalent YAML entries in `*.routing.yml`.

## Technical details

`AttributeRouteDiscovery::collectRoutes()` now iterates over both `Controller` and `Form` subdirectories for every registered namespace. Files containing `#[Route]` in the `Form` tree are handled by a new `createFormRouteCollection()` method rather than the existing `createControllerRouteCollection()`. The form path checks that the reflected class implements `FormInterface`; classes in the `Form` namespace that do not implement it are silently skipped. When the interface is implemented, the route default `_form` is set to the fully-qualified class name (matching the YAML convention) and the class-level attribute is registered as a route. Route attributes on individual methods of a form class are explicitly unsupported: an `\AssertionError` is thrown in assert mode with a descriptive message. When exactly one route is registered for a form class, the class name itself is added as a route alias.

A new `getReflectionClass()` helper was extracted to centralize class existence and abstraction checks; it returns `NULL` and swallows `\Error` when a class cannot be loaded due to a missing parent, interface, or trait. A pre-existing bug in the invokable-controller path was also fixed: the condition `$class->hasMethod('__invoke') === 0` (a boolean compared to an integer, always `FALSE`) was corrected to `$collection->count() === 0 && $class->hasMethod('__invoke')`. `configureRoute()` was removed; the `_controller` default is now set directly on the attribute object before `addRoute()` is called. The `addRoute()` `$method` parameter is now nullable to accommodate form routes, which are class-level rather than method-level.

## Upgrade

To migrate an existing YAML form route to an attribute, remove the YAML entry and add `#[Route]` to the form class. Example from `SiteInformationForm`:

**Before** (`system.routing.yml`):
```yaml
system.site_information_settings:
  path: '/admin/config/system/site-information'
  defaults:
    _form: '\Drupal\system\Form\SiteInformationForm'
    _title: 'Basic site settings'
  requirements:
    _permission: 'administer site configuration'
```

**After** (`SiteInformationForm.php`):
```php
use Drupal\Core\StringTranslation\TranslatableMarkup;
use Symfony\Component\Routing\Attribute\Route;

#[Route(
  path: '/admin/config/system/site-information',
  name: 'system.site_information_settings',
  requirements: [
    '_permission' => 'administer site configuration',
  ],
  defaults: ['_title' => new TranslatableMarkup('Basic site settings')],
)]
class SiteInformationForm extends ConfigFormBase { ... }
```

The form class must implement `FormInterface` and must reside in a `Form` subdirectory/namespace of a registered module namespace. Route attributes on methods inside the form class are not supported and will throw an `\AssertionError`.
