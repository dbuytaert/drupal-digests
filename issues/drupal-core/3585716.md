## Summary

EntityQuery was generating broken SQL for custom entity types that are both translatable and revisionable when the entity type defines a langcode key other than the default `langcode` (for example, `language`). The JOIN condition on the revision data table always referenced the literal column name `langcode`, causing an "Unknown column" SQL error on translation and revision pages. This fix corrects the column name used in those JOIN conditions so it matches the entity type's actual configured langcode key.

## Impact

No upgrade or configuration changes required.

## Technical details

The bug lived in the `addJoin()` method of the SQL entity query backend. When building a JOIN condition that filters by language, the code checked whether the table being joined was the entity's data table (`getDataTable()`) and, if so, fetched the configured langcode key via `$entity_type->getKey('langcode')`. For all other tables, including the revision data table (`getRevisionDataTable()`), it fell back to the hard-coded string `'langcode'`. The revision data table, like the data table, stores per-language row data and must also use the entity type's actual langcode key.

The fix replaces the single ternary with a `match()` expression that maps both `getDataTable()` and `getRevisionDataTable()` to `$entity_type->getKey('langcode')`, leaving the `'langcode'` literal as the `default` case for dedicated field tables (which do hard-code that column name).

The accompanying test modifies the active definition of `entity_test_mulrev` to use `'language'` as its langcode key at runtime, issues an `allRevisions()` query, and asserts the generated SQL contains `"entity_test_mulrev_property_revision"."language"` in the JOIN condition rather than `"entity_test_mulrev_property_revision"."langcode"`. No public API surface changed; only internal SQL generation logic in the entity query system was corrected.
