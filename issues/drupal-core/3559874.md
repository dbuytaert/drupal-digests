## Summary

Drupal's `uuid` entity field type previously applied no format validation to its stored value, meaning any arbitrary string could be saved in a UUID field without complaint. This fix adds the `Uuid` constraint to the `UuidItem` field type so that attempting to save a non-UUID value in a UUID field now produces a validation violation. Any site code or content that stores non-UUID strings in UUID fields will start seeing validation errors after this change.

## Impact

No upgrade or configuration changes required.

## Technical details

The `UuidItem` field type plugin (`core/lib/Drupal/Core/Field/Plugin/Field/FieldType/UuidItem.php`) extends `StringItem` and had only a max-length storage setting of 128 characters but no format constraint on the `value` property. The fix adds a `ComplexData` constraint wrapping a `Uuid` constraint on the `value` property directly in the `#[FieldType]` attribute's `constraints` array, consistent with how other field types such as `TimestampItem` declare their constraints. An earlier revision of the MR placed the constraint in `propertyDefinitions()`; after discussion (raised by idebr in comment #14, confirmed by longwave in comment #15), the cleaner approach of declaring it in the attribute was adopted. The config schema `uuid` type already carried a `Uuid: {}` constraint but nothing ever called `$config_entity->getTypedData()->validate()`, so it had no practical effect. Several test fixtures that used constructed sequential UUIDs (such as `01234567-89ab-cdef-0123-456789abcdef` and `fedcba98-7654-3210-fedc-ba9876543210`) were replaced with properly random UUIDs so they continue to pass the new validation. The REST API validation test was updated to expect two violations when an invalid UUID string is submitted: the new "This is not a valid UUID." message followed by the pre-existing "UUID: may not be longer than 128 characters." message. A new `testInvalidUuid()` test was added to `UuidItemTest` to assert the violation message and property path (`uuid.0.value`) directly.

## Upgrade

If you have custom code or content that stores non-UUID strings in any `uuid` entity field, those entities will now fail validation. Update stored values to be valid UUID v4 strings. No API changes are required for code that uses UUID fields normally.
