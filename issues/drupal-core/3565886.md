## Summary

When the Navigation module is enabled, visiting any URL that does not exist (a 404 page) caused a fatal error: "Call to a member function getPath() on null." This broke the page entirely for logged-in users instead of showing the normal 404 response. The bug is now fixed, so 404 pages display correctly with the Navigation module active.

## Impact

No upgrade or configuration changes required.

## Technical details

The Navigation module's `EntityRouteHelper` service (at `core/modules/navigation/src/EntityRouteHelper.php`) has a method `getContentEntityFromRoute()` that determines which content entity, if any, corresponds to the current route. The original code called `$this->routeMatch->getRouteObject()->getPath()` in a single chained expression. When a request matches no route (a 404), `getRouteObject()` returns `null`, and calling `getPath()` on `null` produced a fatal `Error`.

The fix splits the chain: `getRouteObject()` is now called first and stored in `$route`; if `$route` is falsy the method returns `null` immediately, and `getPath()` is only called on the confirmed non-null object. This also avoids using `null` as an array key into `getContentEntityPaths()`, which would trigger a PHP deprecation notice in addition to being semantically wrong.

A new kernel test, `NavigationEntityRouteHelperTest` (at `core/modules/navigation/tests/src/Kernel/NavigationEntityRouteHelperTest.php`), covers the regression: it issues a request to a non-existent path, asserts a 404 status, and asserts that `getContentEntityFromRoute()` returns `null` cleanly via the `navigation.entity_route_helper` service. The fix was committed to `main`, `11.x`, and `11.3.x`. Issue [#3566389](https://www.drupal.org/node/3566389) is a duplicate of this issue.

### Collaboration

catch pointed out in review that the original proposed fix using the null-safe operator was not quite right: checking for an empty string in `getContentEntityPaths()` would be incorrect behavior. catch suggested explicitly fetching the route object, null-checking it, and only then calling `getPath()`. pjotr.savitski (the issue reporter) agreed, posted the revised code that became the committed solution, and also outlined the two approaches (unit test with mocks or kernel test with a real request) for adding test coverage. That design conversation directly shaped the final implementation.
