## Summary

A regression in `TypedConfigManager` caused config schema definitions to be stored incorrectly in the in-memory cache when the double-bracket (`||`) composite type syntax was used. Once the cache was polluted by a call to `getDefinition()`, subsequent typed config lookups for the same type could resolve against the wrong (incomplete) definition, leading to silent schema mismatches. This bug was severe enough that at least one contributed module (Canvas) had to override the `config.typed` service entirely to work around it.

## Impact

No upgrade or configuration changes required.

## Technical details

The root cause was introduced by [#2663410](https://www.drupal.org/node/2663410). `TypedConfigManager::getDefinitionWithReplacements()` handles a special edge case where a schema type uses the `||` separator (e.g. `wrapping.test.double_brackets.*||test.double_brackets.cat.dog`). When this edge case fires, the method processes the composite type and then unconditionally wrote the result to `$this->definitions[$type]`. The problem: `getDefinition()` calls `getDefinitionWithReplacements()` without passing any `$replacements`, so this unconditional write stored an un-replaced, incomplete definition in the cache. Any subsequent call that did pass replacements would then hit the stale cache entry and skip the replacement logic entirely.

The fix in `TypedConfigManager.php` adds a single guard: `if (!empty($replacements)) { $this->definitions[$type] = $definition; }`. Only calls that actually performed replacements are now allowed to populate the cache entry. Three `clearCachedDefinitions()` workaround calls in `ConfigSchemaTest::testConfigSaveWithWrappingSchemaDoubleBrackets()`, each accompanied by a `@todo Remove in #3400181` annotation, are removed as a direct consequence.
