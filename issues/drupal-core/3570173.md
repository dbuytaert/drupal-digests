## Summary

This is the public follow-up to SA-CORE-2025-005, a security advisory that fixed a cache poisoning vulnerability involving the `X-Http-Method-Override` HTTP header. It adds no new production code: the security fix itself was committed privately. What this issue contributes is the test coverage that was written during the private security process, now published to the main codebase so the fix remains verifiable going forward.

## Technical details

SA-CORE-2025-005 addressed a cache poisoning attack where an authenticated POST request carrying an `X-Http-Method-Override: GET` header could trick Drupal's `RequestSanitizerMiddleware` into serving the response from a cached slot meant only for anonymous GET requests. The fix, committed privately, makes `RequestSanitizer::sanitize()` throw a `BadRequestHttpException` when the override target is a cacheable method (GET, HEAD, OPTIONS, or TRACE).

This issue lands two test additions to verify that fix:

**New functional test** `core/tests/Drupal/FunctionalTests/HttpKernel/RequestSanitizerTest.php` creates `RequestSanitizerTest` extending `BrowserTestBase` with `ApiRequestTrait`. Its `testRequestSanitizer()` method makes a real HTTP POST with `X-Http-Method-Override: GET` to the front page. With the fix in place the first such request (against a warm page cache) returns HTTP 200 and a cache `HIT`, demonstrating the header was honoured while the cache already held a safe response. After `rebuildAll()` flushes the page cache, a second identical POST must return HTTP 400, confirming the middleware blocks the override before a new cache entry can be poisoned.

**New unit test** `testRequestMethodOverride()` added to the existing `core/tests/Drupal/Tests/Core/Security/RequestSanitizerTest.php`. It uses PHPUnit `#[TestWith]` attributes to drive nine method-override cases. Overriding to GET, HEAD, OPTIONS, or TRACE must raise `BadRequestHttpException`; overriding to POST, PUT, DELETE, CONNECT, or PATCH must not.

Reviewer smustgrave validated both tests by reverting the security fix commit locally and confirming: `RequestSanitizerTest` failed with "asserting that 200 is identical to 400" at line 57, and `testRequestMethodOverride` failed for the GET, HEAD, OPTIONS, and TRACE cases, exactly as expected.
