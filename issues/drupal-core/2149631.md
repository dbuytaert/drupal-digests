## Summary

Installing a theme no longer unnecessarily clears the CSS aggregate cache. Previously, every theme install triggered a full deletion of optimized CSS files even though the CSS cache is not theme-specific and its contents are unaffected by which themes are installed. This wasted disk I/O and forced CSS to be regenerated on the next page load for no reason. Removing the call eliminates that overhead.

## Impact

No upgrade or configuration changes required.

## Technical details

The `ThemeInstaller::install()` method in `core/lib/Drupal/Core/Extension/ThemeInstaller.php` called `$this->cssCollectionOptimizer->deleteAll()` after completing each batch of theme installs. The `$cssCollectionOptimizer` property holds the `asset.css.collection_optimizer` service (an `AssetCollectionOptimizerInterface` implementation), whose `deleteAll()` deletes all stored CSS aggregate files from disk. Because CSS aggregation is independent of which themes are installed, this call had no correctness value and only added latency and cache churn to the install process. The fix is a single-line deletion removing that call from `install()`. The corresponding call in `ThemeInstaller::uninstall()` was intentionally left in place, as uninstalling a theme could affect which CSS files are relevant. Reviewer catch noted that [#3505776](https://www.drupal.org/node/3505776) (retain asset aggregates for 45 days by default) would have made the old behavior even less impactful going forward, and that writing a test to assert the absence of the call was not worth the effort.
