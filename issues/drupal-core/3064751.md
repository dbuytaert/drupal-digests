## Summary

When a responsive image field was configured to link to the file or to the content entity, any URL options (such as HTML attributes like CSS classes or `data-*` attributes) set on the link were silently ignored. The generated anchor tag came from a plain `href` attribute in the Twig template, so the URL object's options were never consulted. This fix makes linked responsive images behave the same way the plain image formatter already did after [#2829990](https://www.drupal.org/node/2829990): URL options are now rendered as attributes on the anchor element.

## Impact

Affects module maintainers and front-end developers.

- **Theme change:** `$variables['url']` in `template_preprocess_responsive_image_formatter` is now always a `Url` object; preprocess hooks that treated it as a string for the 'link to file' case must be updated.
- **Deprecation:** `ResponsiveImageFormatter::__construct()` without `$fileUrlGenerator` is deprecated in drupal:11.4.0 and the argument will be required in drupal:12.0.0.

## Technical details

The root problem was two-fold. First, `template_preprocess_responsive_image_formatter()` passed a raw string URL for the "link to file" case (via `$file->createFileUrl()`) rather than a `Url` object, so there was nowhere to attach options. This is replaced with `$this->fileUrlGenerator->generate($file->getFileUri())`, which returns a `Url` object. Second, the Twig template rendered the link as a literal `<a href="{{ url }}">` string interpolation, which cannot output any options. The template now uses the `{{ link(responsive_image, url) }}` Twig function, which consults the `Url` object's options and renders them as HTML attributes.

`ResponsiveImageFormatter` gains a new `FileUrlGeneratorInterface $fileUrlGenerator` constructor parameter. It is optional (nullable, defaulting to `NULL`) for backwards compatibility, with a `@trigger_error` deprecation when omitted; the parameter will become required in Drupal 12.0.0. The `create()` factory already passes `$container->get('file_url_generator')`. The existing constructor parameters were refactored to use PHP constructor property promotion at the same time, removing a block of explicit property assignments.

A new kernel test, `ResponsiveImageFormatterTest`, verifies that attributes set on `$build[...]['#url']` in a preprocess hook appear in the rendered HTML anchor tag.

## Upgrade

Preprocess hooks and other PHP code that reads `$variables['url']` from `template_preprocess_responsive_image_formatter` must now treat it as a `\Drupal\Core\Url` object instead of a plain string for the "link to file" case. Twig templates that just output `{{ url }}` are unaffected because Twig converts `Url` objects to strings automatically.

Code that constructs `ResponsiveImageFormatter` directly without the `$fileUrlGenerator` argument will trigger a deprecation in 11.4.0:

```php
// Deprecated — omitting $fileUrlGenerator triggers E_USER_DEPRECATED.
new ResponsiveImageFormatter($pluginId, $def, $fieldDef, $settings, $label, $viewMode, $thirdParty, $riStorage, $imageStorage, $linkGen, $currentUser);

// Correct — pass the file_url_generator service as the twelfth argument.
new ResponsiveImageFormatter($pluginId, $def, $fieldDef, $settings, $label, $viewMode, $thirdParty, $riStorage, $imageStorage, $linkGen, $currentUser, $fileUrlGenerator);
```
