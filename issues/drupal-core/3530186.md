## Summary

Drupal 12 now uses `argon2id` as the default password hashing algorithm instead of `bcrypt`. PHP 8.4 made argon2 available via openssl, so it no longer requires libsodium or libargon2. Since Drupal 12 requires PHP 8.5, argon2 support can be assumed. If argon2id is not available or the configured algorithm is invalid, the system falls back to PHP's default algorithm (currently bcrypt). Existing passwords are automatically rehashed to argon2id when users next log in. The status report now shows the current hashing algorithm and warns if bcrypt is still in use.

## Impact

Affects module maintainers and site owners.

- **Config change:** New `password.algorithm` (default `argon2id`) and `password.options` (default `[]`) kernel parameters in `core.services.yml` and `default.services.yml`
- **API change:** `PhpPassword::__construct()` `$algorithm` parameter changed from `string` (default `PASSWORD_DEFAULT`) to `?string` (default `NULL`), and now validates against `password_algos()`

## Technical details

The `password` service in Drupal previously used `PhpPassword` with no constructor arguments, which meant it always used `PASSWORD_DEFAULT` (bcrypt). The issue makes the algorithm configurable through two new kernel parameters: `password.algorithm` (set to `argon2id` in `core.services.yml`) and `password.options` (empty array). The `password` service definition now passes these parameters as constructor arguments.

The `PhpPassword` constructor was changed so `$algorithm` is `?string` defaulting to `NULL` instead of `string` defaulting to `PASSWORD_DEFAULT`. The constructor validates the given algorithm against `password_algos()`. If the algorithm is not in the list (unavailable or misspelled), both `$algorithm` and `$options` are reset to `NULL` and `[]` respectively. Passing `NULL` to `password_hash()` uses `PASSWORD_DEFAULT`, so the fallback is always safe.

A `checkPasswordHashing()` method was added to `SystemRequirementsHooks`, called during the `runtime` phase. It reports three tiers: an error if the configured algorithm is not available in PHP, an info message if a non-bcrypt algorithm is active, or a warning if bcrypt is in use (recommending a switch to argon2id, with a link to the change record at node 3530196).

`KernelTestBase::register()` was updated to set the password algorithm via the new kernel parameters instead of directly overriding the service arguments. Tests still use bcrypt with `cost => 4` for speed. A new abstract `PasswordTestBase` class and eight kernel test subclasses were added, covering argon2id, bcrypt, default, and unknown algorithm scenarios for both hashing behavior and requirements output.

An early version included a post-update hook for container rebuild, but this was removed because the default deployment identifier is based on `\Drupal::version`, so updating Drupal already triggers a container rebuild. A dedicated "Password Argon2" module was also built and later dropped in favor of the simpler kernel-parameter approach.

## Upgrade

Sites that override `services.yml` can control the algorithm:

```yaml
parameters:
  password.algorithm: 2y
  password.options:
    cost: 12
```

Setting `password.algorithm: ~` uses PHP's `PASSWORD_DEFAULT` (currently bcrypt).

Code that instantiates `PhpPassword` directly should pass a valid algorithm string or `NULL`; invalid values now silently fall back to `PASSWORD_DEFAULT`.

### Collaboration

The approach went through several significant pivots. The security team triage call (#24) decided to prefer this issue over [#3536662](https://www.drupal.org/node/3536662)'s reintroduction of phpass for long-password handling. A post-update hook for container rebuild was initially added, then removed after catch pointed out the version-based deployment identifier already invalidates the container on update. A dedicated "Password Argon2" module was built (znerol, #46) but benjifisher argued it was less secure than filesystem-based configuration via `services.yml` (#56), and longwave proposed in Slack (#60) the final approach: default to argon2 in core, fall back with validation, and use `hook_requirements` to notify. A companion contrib module (`php_password` 3.0.0) was released by larowlan as a forward-compatibility layer for Drupal 11 stable branches. A follow-up ([#3581966](https://www.drupal.org/node/3581966)) was opened for backporting the kernel parameters to 11.x.
