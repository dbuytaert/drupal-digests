## Summary

Drupal threw a 500 Internal Server Error when a form redirect used a `?destination=` query value that was a short numeric string (for example `?destination=18`). This happened when visiting URLs like `/user/login?destination=18` and then successfully logging in. The bug is now fixed: numeric destination values are handled correctly and Drupal redirects as expected after form submission.

## Impact

No upgrade or configuration changes required.

## Technical details

The root cause is PHP's `parse_url()` interpreting short numeric URI paths differently depending on digit length. `parse_url('base:12345', PHP_URL_SCHEME)` returns `'base'` (correct), but `parse_url('base:1234', PHP_URL_SCHEME)` returns `null` because PHP parses it as a host name plus port rather than a scheme plus path. As a result, the scheme check in `UnroutedUrlAssembler::assemble()` silently failed for short numeric paths, and Drupal fell through to invalid-URI error handling with a fatal 500.

The fix replaces the `parse_url($uri, PHP_URL_SCHEME) === 'base'` check with `strncasecmp($uri, 'base:', 5) === 0 && parse_url($uri, PHP_URL_SCHEME) !== FALSE`. The `strncasecmp` call reliably detects the `base:` scheme prefix regardless of what follows it, while the secondary `parse_url !== FALSE` guard still rejects entirely malformed URIs that PHP cannot parse at all. Two test cases covering `base:1978` were added to `providerTestAssembleWithLocalUri()`.

The fix was backported to 11.4.x and 11.x in addition to the `main` (12.x) branch. A follow-up issue ([#3591074](https://www.drupal.org/node/3591074)) tracks replacing broader usages of `parse_url` across Drupal core with the PHP 8.5 URI extension.

That narrower approach had the important side-effect of making the fix backportable to Drupal 11, which a PHP-8.5-only solution would not have been. steven jones noted the pivot explicitly: "I've tweaked the approach based on excellent feedback from @alexpott, and so this actually becomes possible to backport this fix too."
