## Summary

The docblock for `RowPluginBase::render()` in the Views system has been corrected to match what the method actually does: it returns a render array, not a string, and it expects a `\Drupal\views\ResultRow` object, not a generic `object`. A runtime deprecation notice is now triggered whenever something other than a `ResultRow` is passed. Sites running custom Views row plugins that pass non-`ResultRow` arguments will see deprecation errors on Drupal 11.4 and a hard type error on Drupal 12.

## Impact

Affects module maintainers.

- **Deprecation:** `RowPluginBase::render()` now triggers a deprecation when its `$row` argument is not a `\Drupal\views\ResultRow`; Drupal 12 will enforce this with a type error.

## Technical details

The `render()` method in `RowPluginBase` has returned a render array since at least Drupal 8, but its docblock declared `@return string` and `@param object $row`. The fix corrects both annotations to `@return array` and `@param \Drupal\views\ResultRow $row`, adds the corresponding `use Drupal\views\ResultRow` import, and inserts an `instanceof` guard that calls `@trigger_error(..., E_USER_DEPRECATED)` when `$row` is not a `ResultRow`. The deprecation message reads: "Passing an object other than a ResultRow to `RowPluginBase::render()` is deprecated in drupal:11.4.0 and will throw a type error in drupal:12.0.0." The change record linked from the message is https://www.drupal.org/node/2939099. The method signature itself is unchanged; the actual `ResultRow` type hint is deferred to Drupal 12 to avoid an immediate hard break.

## Upgrade

Any custom Views row plugin whose `render()` override receives, or passes through, an argument that is not a `\Drupal\views\ResultRow` instance must be corrected before upgrading to Drupal 12.

```php
// Before (generic object, triggers deprecation in 11.4):
public function render($row) { ... }

// After (explicit type, compatible with Drupal 12):
public function render(\Drupal\views\ResultRow $row) { ... }
```
