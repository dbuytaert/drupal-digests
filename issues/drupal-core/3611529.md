## Summary

Sites running Content Moderation with two or more moderated bundles could end up with incorrect `moderation_state` field definitions, where one bundle's definition reported the wrong target bundle. This matters because the target bundle is used to derive the associated workflow from a field definition. The fix ensures each bundle receives its own correctly scoped definition.

## Impact

No upgrade or configuration changes required.

## Technical details

The root cause is a shared-reference mutation in `ContentModerationHooks::entityBundleFieldInfo()`. The hook received the shared `$base_field_definitions` array and called `setTargetBundle($bundle)` directly on the `moderation_state` entry before returning it. Because base field definitions are objects, this mutated the shared instance, so the last bundle to be processed would overwrite the target bundle on the same object that earlier bundles had already cached.

This latent bug was surfaced by [#3537962](https://www.drupal.org/node/3537962), which introduced per-bundle caching in `EntityFieldManager::getFieldDefinitions()`. Under that caching model, builds for different bundles reuse the same base field definition objects, making the mutation observable across bundles.

The fix is a one-line change: replace the direct return with `clone $base_field_definitions['moderation_state']`, giving each bundle invocation its own copy. The explicit `setTargetBundle($bundle)` call was also removed because the entity field manager already applies the target bundle to every definition after collecting bundle field info, making the call in the hook redundant. A new kernel test `ModerationStateFieldTest::testFieldDefinitionBundles()` creates three moderated node types and asserts that each bundle's `moderation_state` field definition carries the correct target bundle ID.
