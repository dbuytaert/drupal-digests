## Summary

CKEditor 5 fields that are toggled enabled or disabled at runtime (for example via the Form API `#states` system or the Conditional Fields module) now correctly enter and leave read-only mode as the `disabled` attribute changes. Previously the read-only state was evaluated only once when the editor was initialized, so any subsequent enable/disable had no effect on the editor's editability.

## Impact

No upgrade or configuration changes required.

## Technical details

The original fix in [#3362386](https://www.drupal.org/node/3362386) added a one-time check at editor startup: if `element.hasAttribute('disabled')` was true, `editor.enableReadOnlyMode('ckeditor5_disabled')` was called and nothing more happened. This issue replaces that one-time check in `ckeditor5.js` with a `MutationObserver` that watches for any attribute mutation on the textarea element. On every mutation, a new `readOnlyIfDisabled()` helper checks `element.disabled` and calls either `editor.enableReadOnlyMode('ckeditor5_disabled')` or `editor.disableReadOnlyMode('ckeditor5_disabled')` accordingly. `readOnlyIfDisabled()` is also called immediately after the observer is attached to cover the initial state. The test infrastructure in `CKEditor5ReadOnlyModeTest` is refactored: field creation is extracted into reusable `createCkeditorField()` and `createField()` helpers, and three new test fields (`field_ckeditor5_disable`, `field_ckeditor5_enable`, `field_ckeditor5_toggle`) are added. A new `testReadOnlyModeDynamic()` functional JS test checks and unchecks the toggle and asserts that `contenteditable` flips appropriately on both the "disable on toggle" and "enable on toggle" fields. The `ScriptBytes` expectation in `AssetAggregationAcrossPagesTest` is bumped from 3811015 to 3811582 to account for the slightly larger `ckeditor5.js`.
