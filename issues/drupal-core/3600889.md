## Summary

Sites using `drupal/core-recommended` were regularly blocked from running `composer update` whenever Guzzle, Symfony polyfills, or Twig shipped a security fix in a new minor version, because `core-recommended` pinned those packages to a specific minor. The fix removes the minor-version pins for `guzzlehttp/guzzle`, `guzzlehttp/promises`, `guzzlehttp/psr7`, all `symfony/polyfill-*` packages, `twig/twig`, and `twig/html-extra` from the generated metapackage. Sites can now install security updates for these libraries as soon as they are available, without needing a new Drupal core release. Related security pressure came from issues like [#3599842](https://www.drupal.org/node/3599842) and [#3592065](https://www.drupal.org/node/3592065). A follow-up ([#3606820](https://www.drupal.org/node/3606820)) tracks adding a separate, more-frequent release process for `core-recommended` so that future constraint adjustments can ship independently of core.

No upgrade or configuration changes required.

## Technical details

The `drupal/core-recommended` metapackage is generated from `composer/Plugin/Metapackage/CoreRecommended/src/Builder.php`. That class maintains a `$remove_list`: any package in this list is omitted from the pinned `require` section of the generated `composer/Metapackage/CoreRecommended/composer.json`, meaning Composer will fall back to the broader constraints in `drupal/core`'s own `composer.json` rather than the tighter minor-version pins (`~x.y.z`) that `core-recommended` had previously added.

The commit adds `guzzlehttp/guzzle`, `guzzlehttp/promises`, `guzzlehttp/psr7`, `symfony/polyfill-ctype`, `symfony/polyfill-iconv`, `symfony/polyfill-intl-grapheme`, `symfony/polyfill-intl-idn`, `symfony/polyfill-intl-normalizer`, `symfony/polyfill-mbstring`, `symfony/polyfill-php86`, `twig/html-extra`, and `twig/twig` to that remove list. Each removal is annotated with its rationale: Twig and Guzzle regularly issue security releases only as new minor versions, and the Symfony polyfills are transitive dependencies that follow the same pattern. The generated `composer/Metapackage/CoreRecommended/composer.json` was regenerated after these changes. The earlier approach of removing all minor constraints from the metapackage (MRs !16071, !16072, !16073) was set to draft in favour of this targeted removal; `symfony/yaml` and the full Symfony component stack retain their minor-version pins for now.

### Collaboration

The thread went through a significant design shift. The initial proposal and first set of MRs removed all minor constraints from `core-recommended` entirely. Several contributors, including xjm and joelpittet, pushed back on this as too drastic and potentially confusing, questioning the purpose of the metapackage without any pinning. benjifisher suggested a middle ground: relax constraints only for the specific packages known to publish security-only minor releases, `twig/twig`, `guzzlehttp/*`, and `symfony/polyfill-*`. That targeted approach is what landed. longwave noted during final review that `guzzlehttp/promises` and the PHP 8.4/8.5 polyfills should also be included, and catch added those before the commit.
