## Summary

Forms in Drupal have historically been uncacheable for authenticated users because the CSRF form token carried `max-age=0`, which propagated to any page embedding the form. This issue introduces an opt-in mechanism: a form that explicitly sets `$form['#cache']['max-age']` to any value signals that it manages its own cacheability metadata correctly, and the form token will no longer be forced to `max-age=0`. Forms that do not set this key continue to behave as before (uncacheable), preserving backward compatibility. Entity forms and the node form receive an explicit `max-age=0` to guarantee their caching behavior does not change silently. Follow-up issues ([#3395157](https://www.drupal.org/node/3395157), [#3395524](https://www.drupal.org/node/3395524), [#3395506](https://www.drupal.org/node/3395506), [#3397987](https://www.drupal.org/node/3397987)) track deprecating the default-uncacheable behavior and making specific forms fully cacheable.

## Impact

No upgrade or configuration changes required.

## Technical details

Prior to this change, `FormBuilder::prepareForm()` unconditionally set `$form['form_token']['#cache']['max-age'] = 0` for authenticated users, making any rendered page containing a POST form uncacheable by Dynamic Page Cache. Issue [#2463567](https://www.drupal.org/node/2463567) moved the token into a `#lazy_builder` so the rendered form element itself could be cached, but the `max-age=0` remained, defeating much of the benefit.

The fix is in `FormBuilder::prepareForm()` (lines 839-847 of `core/lib/Drupal/Core/Form/FormBuilder.php`): after building the `form_token` placeholder element, the method checks `if (!isset($form['#cache']['max-age']))`. Only when the top-level form array has no explicit `max-age` key is `$form['form_token']['#cache']['max-age'] = 0` applied. A form that sets `$form['#cache']['max-age']` at the top level (to any value, including `Cache::PERMANENT`) opts in to being cacheable and the token no longer forces uncacheability. A `@todo` comment references [#3395157](https://www.drupal.org/node/3395157) for the eventual deprecation of the default-uncacheable path in a future major version.

Two core forms received explicit `max-age=0` to make the behavior unambiguous and prevent accidental opt-in:

- `EntityFormDisplay::buildForm()` (`core/lib/Drupal/Core/Entity/Entity/EntityFormDisplay.php`, line 221) sets `$form['#cache']['max-age'] = 0` because entity forms already call `$this->renderer->addCacheableDependency($form, $this)`, which would otherwise silently satisfy the opt-in check. A `@todo` points to [#3395524](https://www.drupal.org/node/3395524) to remove this once entity forms are properly audited.
- `NodeForm::form()` (`core/modules/node/src/Form/NodeForm.php`, line 99) sets `$form['#cache']['max-age'] = 0` because the form reads from `PrivateTempStore` on every page load, and TempStore has no cache tags, making invalidation impossible. A `@todo` points to [#3397987](https://www.drupal.org/node/3397987).

The design choice to use the existing `#cache['max-age']` key as the opt-in signal (rather than a new `#form_cacheable` property that was proposed mid-thread) keeps the API surface small and consistent with how cacheability metadata is expressed everywhere else in the render system.
