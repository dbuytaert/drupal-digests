## Summary

When an editor saves a node through the node edit form without making any changes to the associated menu link, Drupal previously saved the `menu_link_content` entity unconditionally, triggering `postSave()` and invalidating the `config:system.menu.MENUNAME` cache tag every time. Because menus are rendered on most pages, this caused broad, unnecessary cache invalidation across CDNs and page caches. After this fix, the menu link entity is only saved when something actually changed, preventing spurious invalidation of menu cache tags on unrelated node saves.

## Impact

No upgrade or configuration changes required.

## Technical details

The fix is in `MenuUiHelper::menuUiNodeSave()` in `menu_ui`. Previously, `$entity->save()` was called unconditionally at the end of the function regardless of whether any menu link fields had changed. The fix splits that single save call into two paths. For the branch that handles updating an existing `menu_link_content` entity (when `$values['entity_id']` is set), `$entity->setOriginal(clone $entity)` is called immediately after loading, which arms the entity's change tracking. For the else branch (pending-revision handling), a `$default_revision_change` flag is computed by comparing `$entity->isDefaultRevision()` with `$node->isDefaultRevision()`, and `$entity->save()` is now gated on `$default_revision_change || $entity->hasTranslationChanges()`. Using `hasTranslationChanges()` alone was not sufficient because it does not detect a default-to-non-default revision transition; that required the explicit separate flag. An earlier approach proposed implementing the skip logic inside `MenuTreeStorage::doSave()`, but that was rejected because the storage layer is not declared internal and because `MenuLinkContent::postSave()` writes only the active-language title into the tree storage, making a purely storage-level comparison unreliable for multilingual sites (see comment #26). The new test coverage in `MenuNodeFormWidgetTest` verifies the `cache_tags.invalidator.checksum` service before and after various scenarios: no-op saves do not increment the checksum; adding, modifying weight, and deleting a menu link do. The fix also uncovered a separate bug in how content moderation and non-default revisions interacted, which was resolved as a prerequisite in [#3499181](https://www.drupal.org/node/3499181).

### Collaboration

berdir drove the final implementation after the original approach (changing `MenuTreeStorage::doSave()` return value, proposed by achap) was found to be a risky behavior change for multilingual sites. In comment #26, berdir explained that `MenuLinkContent::postSave()` only writes the active-language title into the tree storage, so any storage-level equality check would silently drop multilingual data. berdir opened a new MR targeting `menu_ui` instead, then hit a test failure tracing back to a content-moderation bug introduced years earlier (credited partly to berdir's own past work) that allowed saving a default revision as a non-default revision. That was spun out and fixed as a prerequisite in [#3499181](https://www.drupal.org/node/3499181). After that blocker landed, berdir switched the change-detection logic from explicit field comparisons to `hasTranslationChanges()`, then discovered that method misses the default-revision flag, requiring the explicit `$default_revision_change` guard. achap closed the original MR once the community agreed the storage-layer approach was not safe.
