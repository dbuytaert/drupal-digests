# #3567544: Alias lookup by path is not case sensitive but AliasPrefixList and AliasRepository::preloadPathAlias's return value is leading to inconsistencies

## Summary

Drupal's path alias system had an inconsistency where looking up aliases by system path worked correctly regardless of case (treating /node/1 and /NODE/1 the same), but internal components did not consistently honor this case-insensitive behavior. This led to situations where the alias manager would fail to find an alias when given a path with different capitalization, even though Drupal paths are supposed to be case-insensitive. The fix ensures all parts of the path alias system consistently treat paths as case-insensitive, so aliases resolve correctly regardless of how users or code capitalize the path.

## Impact

No upgrade or configuration changes required.

## Technical details

The path alias system uses `AliasPrefixList` to cache which path prefixes have aliases and `AliasRepository::preloadPathAlias()` to bulk-load aliases. The bug was that `AliasPrefixList` performed case-sensitive lookups while `AliasRepositoryInterface::lookupBySystemPath()` performed case-insensitive ones, creating inconsistent behavior depending on which API was used.

The fix modifies `AliasPrefixList` to normalize all paths to lowercase using `mb_strtolower()` in its `get()`, `set()`, `has()`, `resolveCacheMiss()`, and `loadMenuPathRoots()` methods. This ensures the prefix cache correctly handles paths regardless of case.

In `AliasRepository::preloadPathAlias()`, the fix adds logic to track a mapping between user-provided paths and their lowercase equivalents. When the database returns results, if the stored path differs by case from the user-provided path, the method now returns entries for both the stored path and the user-provided path, with the stored path listed first. The interface documentation for `AliasRepositoryInterface::preloadPathAlias()` was updated to document this dual-entry behavior.

New test coverage was added to `AliasTest` to verify case-insensitive behavior in `testLookupPath()` and `testPrefixList()`, including edge cases where different capitalizations are looked up in sequence.
