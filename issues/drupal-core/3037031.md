## Summary

Drupal's locale module previously relied on a set of procedural functions in `locale.compare.inc` and a `locale.project` key-value storage service to track, build, and check the translation status of installed projects. These have been replaced by two new injectable services, `LocaleProjectRepository` and `LocaleProjectChecker`, and a typed `LocaleTranslatableProject` value object. The old functions and the `locale.project` service (`LocaleProjectStorageInterface` / `LocaleProjectStorage`) are now deprecated and will be removed in Drupal 13.0.0. Site behavior is unchanged; this affects developers who call locale translation project APIs directly.

## Impact

Affects module maintainers.

- **API change:** LocaleSource::buildSources() $projects parameter is now required (default [] removed); LocaleSource::sourceBuild() now requires a typed LocaleTranslatableProject argument; LocaleFetch and LocaleFileManager constructor dependencies changed from LocaleProjectStorageInterface to LocaleProjectRepository.
- **Deprecation:** locale_translation_flush_projects(), locale_translation_build_projects(), locale_translation_project_list(), _locale_translation_prepare_project_list(), locale_translation_default_translation_server(), locale_translation_check_projects(), locale_translation_check_projects_local(), LocaleProjectStorageInterface, and LocaleProjectStorage deprecated in drupal:11.4.0, removed in drupal:13.0.0; use LocaleProjectRepository and LocaleProjectChecker instead.
- **New API:** New LocaleProjectRepository service (replaces locale.project / LocaleProjectStorageInterface), new LocaleProjectChecker service, and new LocaleTranslatableProject value object added in locale module.

## Technical details

The core problem was that all project-list logic lived in a single procedural include file (`locale.compare.inc`) with a mix of storage, building, and status-checking concerns, making it hard to inject, test, or extend. The solution splits this into three new symbols in `core/modules/locale/src/`:

`LocaleProjectRepository` is a `readonly` service that owns the `locale.project` key-value store and an in-process `cache.memory` cache (keyed `locale_get_projects`). It provides `getAll()`, `getMultiple()`, `set()`, `deleteMultiple()`, `deleteAll()`, and `buildProjects()`. The latter replaces `locale_translation_build_projects()`: it reads module and theme extension lists via `ModuleExtensionList`/`ThemeExtensionList`, normalises dev-version strings, fires `hook_locale_translation_projects_alter`, and persists `LocaleTranslatableProject` objects. Projects no longer in the extension list are marked disabled rather than deleted.

`LocaleProjectChecker` wraps the two project-checking paths: `checkProjects()` (replaces `locale_translation_check_projects()`) dispatches to a batch if remote sources are enabled, otherwise calls `checkLocalProjects()` (replaces `locale_translation_check_projects_local()`), which uses `LocaleSource::sourceBuild()` and saves the result via `locale_translation_status_save()`.

`LocaleTranslatableProject` is a typed value object with named constructor parameters (`name`, `type`, `core`, `version`, `server_pattern`, `info`, `langcode`, `weight`, `status`). It replaces the bare `stdClass` objects that `locale.compare.inc` previously cast arrays into. It carries `toArray()` and `createFromArray()` for round-tripping through the key-value store.

Three existing services (`LocaleSource`, `LocaleFetch`, `LocaleFileManager`) previously injected `LocaleProjectStorageInterface`; they now inject `LocaleProjectRepository` directly. `LocaleSource::sourceBuild()` now requires a typed `LocaleTranslatableProject $project` instead of a bare `$project` object, and `buildSources()` now requires `array $projects` (the default `= []` was removed). `drupal_static_reset('locale_translation_project_list')` now triggers a deprecation and is a no-op; `drupal_static_reset('locale_translation_get_projects')` was updated to call `cache.memory->delete('locale_get_projects')` instead of the removed `locale.project->resetCache()`.

## Upgrade

Replace deprecated procedural functions and the `locale.project` service as follows:

```php
// locale_translation_flush_projects()
\Drupal::service(\Drupal\locale\LocaleProjectRepository::class)->deleteAll();

// locale_translation_build_projects()
\Drupal::service(\Drupal\locale\LocaleProjectRepository::class)->buildProjects();

// locale_translation_check_projects($projects, $langcodes)
\Drupal::service(\Drupal\locale\LocaleProjectChecker::class)->checkProjects($projects, $langcodes);

// locale_translation_check_projects_local($projects, $langcodes)
\Drupal::service(\Drupal\locale\LocaleProjectChecker::class)->checkLocalProjects($projects, $langcodes);

// \Drupal::service('locale.project')->getProjects() / getAll()
\Drupal::service(\Drupal\locale\LocaleProjectRepository::class)->getAll();

// locale_translation_project_list()
// _locale_translation_prepare_project_list()
// locale_translation_default_translation_server()
// No replacement provided; remove calls to these functions.

// drupal_static_reset('locale_translation_project_list')
// No replacement; remove the call.
```

If you inject `LocaleProjectStorageInterface` or `LocaleProjectStorage` as a constructor dependency, replace the type hint and service reference with `LocaleProjectRepository`:

```php
// Before
public function __construct(
  protected readonly LocaleProjectStorageInterface $localeProjectStorage,
) {}

// After
public function __construct(
  protected readonly \Drupal\locale\LocaleProjectRepository $localeProjectRepository,
) {}
```

`LocaleSource::buildSources()` no longer accepts an empty default for `$projects`; callers that relied on `buildSources()` with no arguments must now pass an explicit array of project names or use `LocaleProjectRepository::getAll()` to obtain the full list first.
