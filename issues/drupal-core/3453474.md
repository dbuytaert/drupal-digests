## Summary

Drupal core now ships a fully-featured CLI tool available as `vendor/bin/dr`. Out of the box it provides `cache:rebuild` (alias `cr`), `system:cron` (alias `cron`), `system:status`, `recipe:apply`, `recipe:info`, and `content:export` commands, covering the most common tasks previously requiring Drush. Module developers can add their own commands by placing a class in their module's `src/Command/` directory and annotating it with `#[AsCommand]`; no service YAML registration is needed. The legacy `core/scripts/drupal` entry point still functions, and `vendor/bin/dr install` / `vendor/bin/dr quick-start` replace the old `php core/scripts/drupal install` invocations documented in INSTALL.md and CI scripts.

## Impact

Affects module maintainers.

- **Deprecation:** `BootableCommandTrait::boot()` deprecated in drupal:11.4.0 and removed in drupal:13.0.0; the new CLI bootstraps Drupal automatically.
- **New API:** New `vendor/bin/dr` CLI entry point backed by `DrupalApplication`; modules auto-discovered via `ConsoleCompilerPass` scanning `src/Command/` for `#[AsCommand]`; built-in commands `cache:rebuild`, `system:cron`, `system:status`, `recipe:apply`, `recipe:info`, and `content:export` added.

## Technical details

The binary is registered in core's `composer.json` as `scripts/dr`, which makes it available as `vendor/bin/dr` after `composer install`. The script returns a Symfony Runtime application, which is now an explicit non-dev dependency (`symfony/runtime ^8.0`).

`DrupalApplication` extends Symfony's `Application` and provides lazy bootstrap: on first command lookup it instantiates `DrupalKernel` with `$allow_dumping=FALSE` (to avoid reusing a cached container), boots the site, pushes a request onto the stack, and then loads module-contributed commands via `DrupalApplication::addDrupalCommands()`. A global `--url` option (also readable from the `DRUPAL_URI` environment variable) controls which site is selected in multi-site setups. Commands that do not need a running site (e.g. `InstallCommand`, `CacheRebuildCommand`) are registered as default commands and run without bootstrapping.

`ConsoleCompilerPass` (a new `@internal` compiler pass) runs at container compile time. It iterates `container.namespaces`, looks inside each module's `src/Command/` directory recursively, and registers any class bearing `#[AsCommand]` as an autowired, public service tagged `console.command`. It also checks the full class hierarchy to skip commands that extend optional third-party base classes (such as Drush commands) whose parents may not be installed. Symfony's own `AddConsoleCommandPass` then processes the tagged services into a lazy `CommandLoaderInterface` (`console.command_loader`). Commands that set their name only via `configure()` (without the attribute) are not auto-discovered and must be declared explicitly in a module's `*.services.yml` with `{ name: console.command }`.

`DrupalConsoleLogger` (a PSR-3 logger) and `ConsoleSubscriber` (a `ConsoleEvents::COMMAND` listener) together route Drupal's internal log messages to the console output during command execution, with notices shown at normal verbosity and info/debug messages gated behind `-v`/`-vvv`.

Several existing `@internal` commands were refactored as part of this change: `ContentExportCommand` moved from `Drupal\Core\DefaultContent` to `Drupal\Core\DefaultContent\Command`; `RecipeCommand` and `RecipeInfoCommand` moved from `Drupal\Core\Recipe` to `Drupal\Core\Recipe\Command`. All three dropped `BootableCommandTrait` in favor of constructor-injected services, since `DrupalApplication` now handles bootstrapping automatically. `QuickStartCommand` was rewritten as an invokable class using `#[Argument]` and `#[Option]` attributes instead of extending `Command`. `DrupalKernel::compileContainer()` was updated to also scan components that expose a `Command/` directory, ensuring core-level commands are discovered.

The full developer API is documented in a new `@defgroup console_commands` section in `module.api.php`, including both the `Command`-extending style and the lighter invokable-class style. Related bootstrap work on Symfony Runtime was tracked in [#3313404](https://www.drupal.org/node/3313404).

## Upgrade

Replace `php core/scripts/drupal` invocations with `vendor/bin/dr`:

```bash
# Before
php core/scripts/drupal install standard
php core/scripts/drupal quick-start --help

# After
vendor/bin/dr install standard
vendor/bin/dr quick-start --help
```

Commands that use `BootableCommandTrait::boot()` must be rewritten to receive their dependencies via constructor injection; `DrupalApplication` now boots Drupal automatically before `execute()` is called:

```php
// Before
class MyCommand extends Command {
  use BootableCommandTrait;
  public function __construct($class_loader) {
    parent::__construct('my:command');
    $this->classLoader = $class_loader;
  }
  protected function execute(InputInterface $input, OutputInterface $output): int {
    $container = $this->boot()->getContainer();
    $service = $container->get(MyService::class);
    // ...
  }
}

// After â€” place in src/Command/MyCommand.php
#[AsCommand(name: 'my:command', description: 'My command.')]
class MyCommand extends Command {
  public function __construct(protected readonly MyService $service) {
    parent::__construct();
  }
  protected function execute(InputInterface $input, OutputInterface $output): int {
    // $this->service is already available; no manual bootstrap needed.
  }
}
```
