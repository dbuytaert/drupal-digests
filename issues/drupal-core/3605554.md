# #3605554: Translations are never loaded or downloaded for a custom profile

## Summary

When running the Drupal installer with a custom install profile, any translatable strings introduced by that profile were never downloaded or loaded. Only Drupal core translations were fetched, leaving custom profile strings untranslated regardless of the selected language. This bug affected any project using a custom install profile, most visibly Drupal CMS, which required dubious workarounds to load its own translations. The fix makes the installer download translation files for the active profile (when it is non-core and has a known version) and makes the early-installer translation loader also read those profile-specific `.po` files.

## Impact

Affects module maintainers.

- **New API:** Optional `$project` and `$version` parameters added to `install_check_translations()`; optional `?string $profile` constructor parameter added to `FileTranslation`.

## Technical details

`install_download_translation()` now calls `install_check_translations()` a second time for the active profile after downloading core translations. This second call is skipped when the profile path starts with `core/` (i.e., it is a core-bundled profile) or when the profile has no declared `version`.

`install_check_translations()` previously hardcoded the project as `'drupal'` and the version as `\Drupal::VERSION`. It now accepts two optional parameters, `string $project = 'drupal'` and `string $version = \Drupal::VERSION`, replacing those hardcoded values. This makes it reusable for any project whose translation files follow the standard localize.drupal.org naming convention.

`FileTranslation::__construct()` gains an optional `?string $profile` parameter (constructor promotion). When a profile is provided, `getTranslationFilesPattern()` widens the filename regex from matching only `drupal-<version>.<langcode>.po` to matching `(drupal|<profile>)-<version>.<langcode>.po`. The version segment of the regex was also relaxed from a strict numeric pattern to `.+` to accommodate the many version string formats used by contributed and custom profiles. `install_begin_request()` now passes `$install_state['parameters']['profile']` to the `FileTranslation` service so it finds both sets of files during the early installer phase.

## Upgrade

No changes required for existing callers: all new parameters are optional and default to their previous hardcoded values. If you added custom code to load profile translations as a workaround for this missing behavior (as Drupal CMS did), that code can now be removed.
