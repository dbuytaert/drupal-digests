## Summary

The `update.fetch.inc` file, the last remnant of the module's old procedural architecture, has been removed along with the `_update_cron_notify()` function it contained. Any code that was including `update.fetch.inc` or calling `_update_cron_notify()` directly will now fail. Neither was ever a public API (the function carried an underscore prefix to signal internal status), but some contrib modules were found to include the file anyway. The cron email notification path is now fully service-based.

## Impact

Affects module maintainers.

- **API change:** `update.fetch.inc` and `_update_cron_notify()` are removed; new `@internal` `Drupal\update\MailHandler` service added with `sendUpdateNotifications()` method.

## Technical details

The `_update_cron_notify()` function's logic is split across two new locations. The notification-triggering logic (checking update requirement severity and deciding whether to send) moves into `UpdateCronHooks::notify()`, a protected method. The email assembly and dispatch loop moves into the new `Drupal\update\MailHandler::sendUpdateNotifications()` method.

`hook_cron()` is extracted from `UpdateHooks` into a dedicated `Drupal\update\Hook\UpdateCronHooks` class. The old implementation used `\Drupal::` static calls and `\Drupal::moduleHandler()->loadInclude('update', 'inc', 'update.fetch')` before invoking `_update_cron_notify()`. The new class injects `ConfigFactoryInterface`, `ModuleHandlerInterface`, `StateInterface`, and `TimeInterface` normally, while `MailHandler`, `UpdateManagerInterface`, and `UpdateProcessorInterface` are injected as lazy closures via `#[AutowireServiceClosure]`. This avoids instantiating those heavier services on every cron run when they are not needed.

`MailHandler` is registered as the service `Drupal\update\MailHandler` in `update.services.yml` and is marked `@internal`. It carries no interface so that subclasses can override only the services they actually need (using `#[Required]` setter injection), matching the pattern adopted in [#3539178](https://www.drupal.org/node/3539178) for `_user_mail_notify()`. This design is intended to support the ongoing symfony mailer migration, where an experimental module could swap the service class to use the new mail API.

There is also a minor behavioral correction: the old code set `update.last_email_notification` inside the per-recipient loop after each successful individual send. The new code sets it once in `cron()` after `notify()` returns `TRUE`, meaning the timestamp is recorded only after the full notification pass completes.

## Upgrade

If any custom or contrib code includes `update.fetch.inc` or calls `_update_cron_notify()` directly, remove those calls. To trigger update notifications programmatically, use the `Drupal\update\MailHandler` service:

```php
// Old (broken after this change):
\Drupal::moduleHandler()->loadInclude('update', 'inc', 'update.fetch');
_update_cron_notify();

// New (internal API, use with caution):
$mailHandler = \Drupal::service(\Drupal\update\MailHandler::class);
$mailHandler->sendUpdateNotifications($recipients, $items);
```

Note that `MailHandler` is marked `@internal` and its signature may change without notice.

### Collaboration

During review, smustgrave raised whether `_update_cron_notify()` needed a formal deprecation before removal. dww argued against it on the grounds that underscore-prefixed functions are explicitly non-API and callers are "doing it wrong." berdir provided a broader view: underscore-prefix functions have been deprecated before when heavily used in contrib (citing `_field_multiple_value_form_sort_helper()`), but a search confirmed `_update_cron_notify()` had effectively zero real usage outside core. nicxvan found a few contrib modules including the file but not using the function, and after consulting alexpott's position from an earlier locale issue (that an immediate white screen is a better signal than a deprecation), concluded no deprecation was needed. The interface question was also resolved by following the approach from [#3539178](https://www.drupal.org/node/3539178): no interface, with `#[Required]` setter injection left available for subclasses.
