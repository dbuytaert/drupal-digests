## Summary

The Navigation module's message display element has been converted from a Twig theme hook to a Single Directory Component (SDC). The old `navigation__message` theme hook is removed; any code that was rendering messages via `#theme => 'navigation__message'` must now use the `navigation:message` SDC component instead. The CSS that was previously declared in a standalone `internal.navigation-messages` library is now bundled inside the SDC, so that library is also gone. As a visual fix, the pointer cursor is now only applied when a message element actually has an `href` attribute, meaning non-linked messages no longer render with a pointer cursor.

## Impact

Affects module maintainers.

- **API change:** The `navigation__message` theme hook is removed; callers must render the `navigation:message` SDC component using `#type => 'component'` instead.
- **Deprecation:** The `internal.navigation-messages` CSS library is removed; remove any dependencies on it.

## Technical details

The `navigation__message` theme hook previously defined in `NavigationThemeHooks::theme()` is removed. In its place, a new SDC component `navigation:message` is introduced, defined in a `.component.yml` file with three typed string props: `type` (enum `error | status | warning`, default `status`), `url` (format `iri-reference`, optional), and `content` (plain string). The component's Twig template uses `{% if url %}` to conditionally render either a link `&lt;a&gt;` or a static `&lt;div&gt;`.

The `internal.navigation-messages` CSS library entry is removed from `navigation.libraries.yml`, and the corresponding `{{ attach_library(...) }}` call is dropped from `navigation__messages.html.twig`, because the component's styles are now declared and scoped inside the SDC itself. The BEM modifier class is corrected from `toolbar-message--type--{type}` to `toolbar-message--{type}`, and `cursor: pointer` is moved from the base `.toolbar-message` rule to a new `.toolbar-message[href]` attribute selector so it only applies to linked messages.

The Umami integration in `NavigationContentTopBlock::navigationContentTop()` is updated to use `#type => 'component'` and `#component => 'navigation:message'` with a `#props` array, replacing the old `#theme => 'navigation__message'` render array. This is the reference implementation for how any other module or profile should render a Navigation message going forward.

## Upgrade

Any code rendering a Navigation message via the old theme hook must switch to the SDC component:

```php
// Before
$element = [
  '#theme' => 'navigation__message',
  '#content' => ['#markup' => $this->t('Some message.')],
  '#url' => $url,
  '#type' => 'warning',
];

// After
$element = [
  '#type' => 'component',
  '#component' => 'navigation:message',
  '#props' => [
    'type' => 'warning',
    'url' => $url,
    'content' => (string) $this->t('Some message.'),
  ],
];
```

Any library dependency on `navigation/internal.navigation-messages` should also be removed, as that library no longer exists.

### Collaboration

pdureau provided detailed design review early in the thread, catching several SDC authoring issues: slots must not carry a `type` property, the BEM class name had a redundant `--type--` segment, and using a slot value as a `data-drupal-tooltip` attribute value is unsafe because slots can contain markup. He recommended making `content` a typed string prop and injecting tooltip attributes from the caller. These suggestions directly shaped the final component definition. Later, alexpott flagged that removing an item from `NavigationThemeHooks::theme()` is a BC break that needs a deprecation or change record, prompting a Slack discussion that concluded with the theme hook being removed outright and a change record added.
