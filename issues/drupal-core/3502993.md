## Summary

The Navigation module's message display is now implemented as a Single Directory Component (SDC) called `navigation:message`, bringing it in line with the rest of the Navigation toolbar components. The old `navigation__message` theme hook and its dedicated internal CSS library have been removed. The component carries its own styles, so no separate library attachment is needed. A small CSS correctness fix is included: `cursor: pointer` is now applied only to `.toolbar-message[href]` rather than every message element, so non-linked messages no longer show a pointer cursor.

## Impact

Affects module maintainers and front-end developers.

- **Theme change:** BEM modifier class changed from `toolbar-message--type--{type}` to `toolbar-message--{type}`; the `internal.navigation-messages` library is removed.
- **API change:** The `navigation__message` theme hook is removed; callers must switch to `#type => 'component'`, `#component => 'navigation:message'` with a `#props` array.
- **New API:** New `navigation:message` SDC component with typed props `type` (enum), `url` (iri-reference), and `content` (string).

## Technical details

The previous implementation used a `navigation__message` theme hook defined in `NavigationThemeHooks::theme()`, a `navigation__message.html.twig` template, and an `internal.navigation-messages` library that loaded `css/components/toolbar-message.css` via `navigation.libraries.yml`. All three are removed.

In their place, a new SDC named `navigation:message` is defined with a `message.component.yml` schema. It exposes three typed props: `type` (string enum of `error`, `status`, `warning`; default `status`), `url` (string, `iri-reference` format), and `content` (string). Because `content` is a plain-string prop rather than a slot, it can safely be used as the value of `data-drupal-tooltip` and `data-drupal-tooltip-class` attributes in the template. The SDC bundles its own CSS, so no external library attach is needed, and `navigation__messages.html.twig` no longer calls `attach_library('navigation/internal.navigation-messages')`.

The call site in `UmamiNavigationBlock::navigationContentTop()` switches from `'#theme' => 'navigation__message'` to `'#type' => 'component'`, `'#component' => 'navigation:message'`, with props passed under `'#props'`. The BEM modifier class is corrected from `toolbar-message--type--warning` to `toolbar-message--warning`. The Twig guard on the optional URL changes from `{% if url is not empty %}` to `{% if url %}` to correctly handle a missing prop. Review feedback from pdureau shaped each of these design decisions, including the choice of typed props over slots and the `iri-reference` format annotation for the URL prop.

## Upgrade

To replace the removed `navigation__message` theme hook, update render arrays like this:

```php
// Before
[
  '#theme' => 'navigation__message',
  '#content' => ['#markup' => $this->t('Some message.')],
  '#url' => $url,
  '#type' => 'warning',
]

// After
[
  '#type' => 'component',
  '#component' => 'navigation:message',
  '#props' => [
    'type' => 'warning',
    'url' => $url,
    'content' => (string) $this->t('Some message.'),
  ],
]
```

Also remove any explicit `attach_library('navigation/internal.navigation-messages')` calls; the SDC bundles its own CSS.

### Collaboration

pdureau provided substantive review feedback that directly shaped the final component design. In the first review round, he flagged that slots must not carry a `type` property per SDC spec, that the BEM modifier was incorrectly doubled (`--type--warning`), and that using a slot as a `data-drupal-tooltip` attribute value is unsafe because slots can contain markup. He recommended converting `content` to a typed string prop and injecting the tooltip attributes from the render array rather than hardcoding them in the template. In a second review round after finnsky moved `content` to a prop, pdureau caught that the `{% if url is not empty %}` guard would not fire when the prop was absent and recommended the simpler `{% if url %}`. alexpott's late comment about needing a deprecation for the removed theme hook triggered a Slack discussion that resulted in the hook being cleanly removed and a change record filed at https://www.drupal.org/node/3559492.
