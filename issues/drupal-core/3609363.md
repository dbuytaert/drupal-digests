## Summary

Sites that install from existing configuration with a non-English default language could lose contrib or custom translation overrides after upgrading to 11.4.x. Core translations would silently "win" over any overriding `.po` file provided by a contrib module (such as `locale_deploy`) or a custom project. This is now fixed and the 11.3.x behavior is restored.

## Impact

No upgrade or configuration changes required.

## Technical details

The 11.4.x locale batch-to-services refactor changed `install_import_translations()` in `core/includes/install.core.inc` to call `\Drupal::service(LocaleFetch::class)->buildUpdateBatch()` with no arguments. That call imports every project's translations in a single "first-wins" pass: `PoDatabaseWriter` uses empty `overwrite_options`, meaning the first import of a string can never be overwritten. Projects are processed in `getAll()` order, so `drupal` is imported first and its strings are locked in before contrib or custom projects run. The subsequent "last-wins" pass in `install_finish_translations()` then finds every file already recorded in `{locale_file}` and re-imports nothing, so custom overrides are permanently lost.

In 11.3.x, `install_import_translations()` explicitly imported only the `drupal` project, deferring all contrib and custom projects to `install_finish_translations()`, which runs with last-wins semantics. The fix restores that contract with a one-character change: `buildUpdateBatch(['drupal'])`. A functional test was also added to `MultilingualInstallerTest`, covering the install-from-config path with two new hidden test modules (`locale_test_additional` and `locale_test_extra`) that supply competing Spanish translations for the same source string; the test asserts that the last-wins override from `locale_test_extra` is the one that survives.
