## Summary

The `image.module` file had two procedural functions (`image_path_flush()`, `image_style_options()`) and a global constant (`IMAGE_DERIVATIVE_TOKEN`) that were the last remaining items blocking removal of `image.module` in a future major version. This change deprecates all three and introduces replacements so the module file can eventually be removed.

## Impact

Affects module maintainers.

- **API change:** `ImageFormatter`, `ImageUrlFormatter`, and `MediaThumbnailFormatter` constructors require a new `ImageDerivativeUtilities` argument as the last parameter
- **Deprecation:** `image_path_flush()`, `image_style_options()`, and `IMAGE_DERIVATIVE_TOKEN` deprecated in drupal:11.4.0, removed in drupal:13.0.0; `ImageFormatter`, `ImageUrlFormatter`, and `MediaThumbnailFormatter` constructors have a new `ImageDerivativeUtilities` parameter appended
- **New API:** New `ImageDerivativeUtilities` service with `pathFlush()` and `styleOptions()` methods; new `ImageStyleInterface::TOKEN` constant

## Technical details

The `image.module` file contained two procedural functions and one `define()` constant that had no service-based equivalents. To allow removing `image.module` in 13.x, this change:

1. Creates `\Drupal\image\ImageDerivativeUtilities` (registered in `image.services.yml` using the class name as the service ID). It has two public methods: `pathFlush(string $path): void`, which loads all image styles and calls `flush()` on each for the given path, and `styleOptions(bool $include_empty = TRUE): array`, which returns an array of image style labels keyed by machine name.

2. Moves `IMAGE_DERIVATIVE_TOKEN` to `ImageStyleInterface::TOKEN`. The issue discussed putting the constant on `ImageStyle` entity or `ImageStyleDownloadController`, but settled on `ImageStyleInterface` because it is already a dependency of the controller and is a more appropriate semantic home. An enum was rejected because constants are simpler to use as array keys.

3. Deprecates `image_path_flush()`, `image_style_options()`, and `IMAGE_DERIVATIVE_TOKEN` in drupal:11.4.0, with removal in drupal:13.0.0. The deprecation trigger errors now delegate directly to the new service.

4. Updates all core callers. `ImageFormatter`, `ImageUrlFormatter`, and `MediaThumbnailFormatter` receive `ImageDerivativeUtilities` as a new constructor parameter. `ImageHooks` uses `#[AutowireServiceClosure]` for lazy injection (avoiding eager service instantiation in hooks). Places that could not use constructor injection (`ImageWidget`, `ImageStyleDeleteForm`, `ResponsiveImageStyleForm`, `ConfigTestForm`) use `\Drupal::service(ImageDerivativeUtilities::class)` directly. Internal uses of `IMAGE_DERIVATIVE_TOKEN` in `ImageStyle::buildUrl()` and `ImageStyleDownloadController::deliver()` now use `ImageStyleInterface::TOKEN`.

The follow-up issue [#3580702](https://www.drupal.org/node/3580702) will address proper dependency injection for `ImageWidget`.

## Upgrade

Replace procedural function calls and the constant:

```php
// Before
image_path_flush($path);
$options = image_style_options();
$token = $request->query->get(IMAGE_DERIVATIVE_TOKEN, '');

// After
\Drupal::service(\Drupal\image\ImageDerivativeUtilities::class)->pathFlush($path);
$options = \Drupal::service(\Drupal\image\ImageDerivativeUtilities::class)->styleOptions();
$token = $request->query->get(\Drupal\image\ImageStyleInterface::TOKEN, '');
```

If you inject the service via a constructor, add it as a new parameter:

```php
use Drupal\image\ImageDerivativeUtilities;

public function __construct(
  // ... existing parameters ...
  protected ImageDerivativeUtilities $imageDerivativeUtilities,
) {}
```

Classes that extend `ImageFormatter`, `ImageUrlFormatter`, or `MediaThumbnailFormatter` must add the `ImageDerivativeUtilities` argument to their constructors and pass it to `parent::__construct()`.

### Collaboration

There was a multi-comment discussion about where to place the `IMAGE_DERIVATIVE_TOKEN` constant. `dcam` suggested `ImageStyle` entity as the most natural home since that is where token-bearing URLs are built. `nicxvan` pushed back, arguing that requiring `ImageStyle` or `ImageStyleDownloadController` just to read a query parameter name felt wrong. `nicxvan` then proposed `ImageStyleInterface`, which `dcam` accepted, noting they had never thought to put constants on interfaces. A single-case backed enum was also floated and rejected for lack of precedent in core and because constants are simpler to use as array keys. Separately, the initial deprecation notices targeted drupal:12.0.0 for removal; a follow-up MR by claudiu.cristea corrected this to drupal:13.0.0.
