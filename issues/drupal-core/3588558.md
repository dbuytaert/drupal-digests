## Summary

The node access rebuild confirmation form (`RebuildPermissionsForm`) declared its injected service property with `readonly`, which conflicted with `DependencySerializationTrait`. Whenever the form was serialized and then deserialized (for example, when form state is stored in a session), the trait's `__wakeup()` method tried to re-initialize the property from the container and PHP threw an error because `readonly` properties cannot be assigned outside the constructor. Removing `readonly` from that property fixes the crash.

## Impact

No upgrade or configuration changes required.

## Technical details

The problem originates from [#3533299](https://www.drupal.org/node/3533299) (Deprecate node access rebuild functions), which introduced `RebuildPermissionsForm` with a constructor-promoted `protected readonly NodeAccessRebuild $nodeAccessRebuild` property. `RebuildPermissionsForm` extends `ConfirmFormBase`, which uses `DependencySerializationTrait`. That trait's `__wakeup()` loops over an object's properties and re-fetches any that match known service IDs from the container, writing them back by name. PHP does not allow assignment to a `readonly` property after construction, so `__wakeup()` raises a fatal error on deserialization.

The fix in MR !15700 is a single-character change in `core/modules/node/src/Form/RebuildPermissionsForm.php`: the `readonly` modifier is dropped from the constructor-promoted property, leaving it `protected NodeAccessRebuild $nodeAccessRebuild`. During review, the question arose of whether `readonly` should also be removed from the `main` branch. `catch` concluded it was acceptable to keep it on `main` because properties on form controllers are `@internal`, so subclass compatibility is not a concern for the public API surface.

The two mysqli-related update-test failures mentioned in the issue body (from [#3581109](https://www.drupal.org/node/3581109) and [#3184242](https://www.drupal.org/node/3184242)) were tracked separately in [#3588702](https://www.drupal.org/node/3588702) and are not part of this merged diff.

### Collaboration

During review, godotislate raised a subtle forward-compatibility concern: if `main` kept the `readonly` modifier, a site upgrading from 11.4 to 12.0 with a subclass that redeclared the property without `readonly` could hit a PHP error. catch replied that form controller properties are `@internal` by convention, so there was no compatibility obligation, and `readonly` could stay on `main`.
