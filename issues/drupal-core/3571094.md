## Summary

Drupal's stream wrapper system now uses a more efficient service locator pattern. Previously, a custom compiler pass collected metadata about stream wrappers during container compilation and injected it separately into the stream wrapper manager. This change removes that duplication by having the manager retrieve the metadata directly from the service locator itself, which already contains all the necessary information. This simplifies the codebase and removes unnecessary code without changing how stream wrappers work for site owners or developers.

## Impact

No upgrade or configuration changes required.

## Technical details

The `RegisterStreamWrappersPass` compiler pass previously collected all services tagged with `stream_wrapper` and called `addStreamWrapper()` on the `StreamWrapperManager` for each one. This populated an internal `$info` array with scheme metadata (class, type, service ID). However, a service locator was already being built and passed to the manager.

This change removes the compiler pass entirely and refactors `StreamWrapperManager` to retrieve metadata directly from the service locator using `getProvidedServices()` and instantiating wrappers on demand. The `#[AutowireLocator]` attribute now handles service locator creation automatically, using the `scheme` tag attribute as the index key.

The public `addStreamWrapper()` method is removed since it was only called by the compiler pass. The `getClass()` method now retrieves the class by instantiating the service and calling `get_class()` on it. The `register()` method iterates over schemes from the service locator rather than the internal info array.

This reduces code complexity and removes duplication since the service locator already contained all necessary information. PHPStan baseline entries for the removed method were also cleaned up.
