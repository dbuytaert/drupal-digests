## Summary

Session management now uses a dedicated repository service instead of having `SessionManager` directly access the database. This makes it easier for modules to replace session storage backends without overriding the entire session manager. The new approach also removes unnecessary checks that prevented deleting sessions in certain contexts, since the repository can delete any user's sessions regardless of the current user context.

## Impact

Affects module maintainers.

- **Deprecation:** `SessionManager::delete()` and `SessionManager::setWriteSafeHandler()` are deprecated; use `UserSessionRepositoryInterface::deleteAll()` instead.
- **New API:** New `UserSessionRepositoryInterface` service provides `deleteAll()` method for managing user sessions.

## Technical details

The `SessionManager::delete()` method previously performed direct database queries to remove user sessions. This was the last remaining database operation in `SessionManager`, making it difficult for modules to swap out session storage backends without extending the entire class.

The responsibility for deleting user sessions now belongs to the new `UserSessionRepository` service implementing `UserSessionRepositoryInterface`. The repository's `deleteAll()` method accepts a user ID and removes all session records for that user.

The old `SessionManager::delete()` guarded deletions with `WriteSafeSessionHandler` checks and CLI detection. These guards are removed in the new repository because `deleteAll()` can target any user's sessions, not just the current session. For example, when blocking a user, Drupal deletes all their sessions regardless of who is currently authenticated.

The `SessionManager` constructor signature changed: the database connection parameter is removed and the session handler parameter moved to match the parent class order. A new `UserSessionRepositoryInterface` parameter was added. Backward compatibility detects the old parameter order and triggers a deprecation error.

Core code in `User::postSave()` now injects and uses `UserSessionRepositoryInterface` instead of calling `SessionManager::delete()`.

## Upgrade

If your code calls `SessionManager::delete()`, inject `UserSessionRepositoryInterface` instead and call `deleteAll()`:

```php
// Before:
$session_manager = \Drupal::service('session_manager');
$session_manager->delete($uid);

// After:
$session_repository = \Drupal::service(UserSessionRepositoryInterface::class);
$session_repository->deleteAll($uid);
```

If you extend `SessionManager` and override the constructor, update your parameter list to match the new signature with the session handler as the second parameter and `UserSessionRepositoryInterface` as the last parameter.

The `SessionManager::setWriteSafeHandler()` method is also deprecated with no replacement needed.

To provide a custom session storage backend, implement `UserSessionRepositoryInterface` and tag your service with `backend_overridable` to override the default implementation.
