## Summary

Drupal core now uses a dedicated service to manage user session data in the database. Previously, the session manager mixed session lifecycle management with direct database queries to delete user sessions. This made it difficult for sites to replace the default database storage with alternative backends like Redis or Memcache. The new user session repository service cleanly separates these concerns. Sites using the default session storage do not need to make any changes.

## Impact

Affects module maintainers.

- **Deprecation:** SessionManager::delete() deprecated in favor of UserSessionRepositoryInterface::deleteAll(); SessionManager::setWriteSafeHandler() deprecated with no replacement; SessionManager constructor second parameter changed from Connection to session handler

## Technical details

The `SessionManager::delete()` method was the last remaining method in `SessionManager` that directly queried the database. This tight coupling required modules providing alternative session storage backends to extend and override `SessionManager` itself.

The responsibility for deleting all sessions belonging to a user has been moved to a new `UserSessionRepositoryInterface` service with a `deleteAll()` method. The default implementation (`UserSessionRepository`) executes the same database query as before. The service is tagged as `backend_overridable` so alternative session storage modules can provide their own implementation.

The new repository is not guarded by `WriteSafeSessionHandler` or CLI checks. This is intentional because `deleteAll()` targets sessions by user ID, not necessarily the current session. For example, when a user is blocked, all their sessions are deleted regardless of which session is currently active.

The `SessionManager` constructor signature changed. The second parameter is now the session handler (previously the sixth parameter), and the sixth parameter is now the user session repository (previously the session handler). Backward compatibility logic detects the old signature and triggers a deprecation warning. The `connection` property and `writeSafeHandler` property are removed from `SessionManager`.

## Upgrade

If you are calling `SessionManager::delete($uid)`, change to `UserSessionRepositoryInterface::deleteAll($uid)`:

```php
// Before:
\Drupal::service('session_manager')->delete($user_id);

// After:
\Drupal::service(UserSessionRepositoryInterface::class)->deleteAll($user_id);
```

If you are extending `SessionManager` to provide custom session storage, implement `UserSessionRepositoryInterface` instead and register it as a service with the `backend_overridable` tag.

If you are manually constructing `SessionManager`, update the constructor arguments to match the new signature (session handler moved from sixth to second parameter).

See the change record for complete details: https://www.drupal.org/node/3570851
