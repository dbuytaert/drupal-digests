## Summary

JSON:API responses now correctly propagate cache tags, cache contexts, and max-age declared at the field item list level for computed fields. Previously, if a computed field's item list class implemented `CacheableDependencyInterface`, that cacheability data was silently dropped during JSON:API normalization. This affected all computed fields including computed entity reference fields, meaning cached JSON:API responses could remain stale even after the computed field's underlying dependencies changed. The fix ensures those cache constraints are honored so that responses are properly invalidated when data changes.

## Impact

No upgrade or configuration changes required.

## Technical details

JSON:API normalizes entity fields in `ResourceObjectNormalizer::serializeField()`. Issue [#2997123](https://www.drupal.org/node/2997123) had already wired up cacheability for individual field item *properties* (the sub-values of each item), but cacheability declared on the *field item list* itself was never collected. Computed fields using `ComputedItemListTrait` and `CacheableDependencyTrait` declare their cache metadata at the list level via `setCacheability()`, which registers tags, contexts, and max-age on the list object. Because JSON:API never checked that level, those constraints were invisible to the normalizer.

The fix adds a six-line guard in `serializeField()`, placed after the existing relationship/attribute branching, that checks whether `$field` (the `FieldItemListInterface` object) implements `CacheableDependencyInterface`. If it does, the normalized result is merged with the field's cacheability via `$normalized_field->withCacheableDependency($field)`, which delegates to `CacheableNormalization::withCacheableDependency()`. This applies uniformly to both entity reference relationship fields and ordinary attribute fields.

The test module's `ComputedTestCacheableIntegerItemList` was adjusted to use the cache context `headers:X-computed_test_cacheable_integer_field=1` instead of the previous `url.query_args:computed_test_cacheable_integer_field`, because the latter is already included in the default JSON:API response cache contexts (as confirmed via `ResourceTestBase::getExpectedCacheContexts()`), which would have caused it to be optimized away and made the test assertion invisible. The `EntityTestComputedFieldTest` functional test now asserts that `computed_test_cacheable_integer_field`'s tags and contexts appear in JSON:API responses. `ResourceTestBase::getExpectedCollectionCacheability()` was also updated to account for field item list cacheability in collection requests.

The fix builds on the test infrastructure introduced in [#3304772](https://www.drupal.org/node/3304772), which established `ComputedTestCacheableIntegerItemList` and verified that the same cacheability bubbles correctly through the render API. That existing setup made it straightforward to add parallel JSON:API coverage.

Multiple merge requests were opened over several years: bradjones1 and duivw worked at the `ResourceObjectNormalizer` level but lacked test coverage or missed cache contexts; jan.stoeckler proposed the item-list-level approach without a new interface; tstoeckler synthesized these, updated the cache context to make test assertions non-trivial, confirmed that no special handling is needed for empty lists (the list's own `CacheableDependencyInterface` implementation covers that case), and drove the issue to RTBC.

## Upgrade

If you have a computed field whose item list class implements `CacheableDependencyInterface` (typically via `CacheableDependencyTrait`), no code changes are needed. JSON:API will now automatically pick up the cache metadata you set with `setCacheability()` inside `computeValue()`. Example pattern that now works correctly out of the box:

```php
use Drupal\Core\Cache\CacheableDependencyInterface;
use Drupal\Core\Cache\CacheableDependencyTrait;
use Drupal\Core\Cache\CacheableMetadata;
use Drupal\Core\Field\FieldItemList;
use Drupal\Core\Field\FieldItemListInterface;

class MyComputedFieldItemList extends FieldItemList implements CacheableDependencyInterface {

  use CacheableDependencyTrait, ComputedItemListTrait;

  protected function computeValue(): void {
    // ... populate $this->list ...

    $cacheability = (new CacheableMetadata())
      ->setCacheTags(['my_custom_tag'])
      ->setCacheContexts(['my_context']);
    $this->setCacheability($cacheability);
  }

}
```

Any workaround that injected cacheability through the `access()` method (as described in comment #31 of the issue) can be removed in favor of the above pattern.
