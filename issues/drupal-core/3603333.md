## Summary

Installing a module that provides a stream wrapper calling `\Drupal::service()` in its constructor would throw a `ServiceNotFoundException` or `ServiceCircularReferenceException` fatal error in Drupal 11.4.x. Affected contrib modules include s3fs, gcsfs, media_avportal, rokka, and flysystem. The fix ensures stream wrapper services are never instantiated during container rebuilds, so all services are safely accessible when stream wrappers are eventually constructed.

## Impact

No upgrade or configuration changes required.

## Technical details

The regression was introduced by [#3571094](https://www.drupal.org/node/3571094) (Remove `RegisterStreamWrappersPass`). After that change, `StreamWrapperManager::getClass()` retrieved the class name by calling `get_class($this->container->get($scheme))`, which instantiated the stream wrapper service immediately. `StreamWrapperManager::register()` calls `getClass()` internally, and `register()` is invoked from `DrupalKernel::initializeContainer()` during a module install — before the new container has replaced the global container. Any stream wrapper whose `__construct()` called `\Drupal::service()` therefore asked the old container for services that only exist in the new one.

The fix reintroduces a dedicated compiler pass, `StreamWrapperClassesPass`, that collects stream wrapper class names from service definitions at container compile time (reading `ContainerDefinition::getClass()`, not instantiating anything) and injects the resulting `$wrapperClasses` array as a constructor argument to `StreamWrapperManager`. `getClass($scheme)` now does a simple array lookup (`$this->wrapperClasses[$scheme] ?? FALSE`), and `register()` iterates `$wrapperClasses` directly. Stream wrapper services are no longer instantiated during the registration phase. A new test module (`stream_wrapper_service_test`) adds a kernel test covering a stream wrapper that calls a service in its constructor during module install, preventing regression.
