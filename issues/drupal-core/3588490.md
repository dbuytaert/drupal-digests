## Summary

Every admin page load was silently invalidating the `cache_bootstrap` cache bin. `SystemRequirementsHooks::checkRequirements()` called `reset()` on the module and theme extension lists before reading them, wiping their cached state on each request. Because system requirements are checked on every admin page, this caused constant cache rebuilds and degraded admin performance across the board. The fix adds a way to retrieve a fresh extension list without discarding the cached copy, eliminating the unnecessary invalidation.

## Impact

Affects module maintainers.

- **New API:** New optional `bool $skip_cache = FALSE` parameter on `ExtensionList::getList()` lets callers retrieve a fresh list without invalidating the cached copy.

## Technical details

The root cause was inherited from Drupal 7's `system_rebuild_module_data()`, which both built and persisted extension data in one call. That pattern was ported forward and survived into `SystemRequirementsHooks::checkRequirements()`, which called `$module_extension_list->reset()` and `$theme_extension_list->reset()` before every call to `getList()`.

`ExtensionList::getList()` now accepts an optional `bool $skip_cache = FALSE` parameter. When `TRUE`, it calls `$this->doList()` directly to bypass the persistent cache, but it still populates `$this->extensions` with the result so that subsequent calls within the same request use the fresh data without another rebuild. The old `reset()` approach cleared that property entirely, forcing every downstream caller in the same request to rebuild from scratch.

`SystemRequirementsHooks::checkRequirements()` drops its two `->reset()` calls and switches to `getList(TRUE)`. `AppController::themesPage()` similarly replaces `$this->themeExtensionList->reset()->getList()` with `$this->themeExtensionList->getList(TRUE)`. The related issue [#3336621](https://www.drupal.org/node/3336621) tracks the broader cost of running `hook_requirements()` on all admin pages.

## Upgrade

If you call `ExtensionList::getList()` or chain `->reset()->getList()` to force a fresh read, switch to the new parameter instead:

```php
// Before
$extensions = $extensionList->reset()->getList();

// After
$extensions = $extensionList->getList(skip_cache: TRUE);
```

Using `getList(TRUE)` fetches fresh data while preserving the cached copy for subsequent callers in the same request, unlike `reset()` which drops the in-memory cache entirely.
