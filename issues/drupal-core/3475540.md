## Summary

When a URL contains non-ASCII characters in a config entity route parameter (for example `/node/add/тест`, `/media/add/öüä`, or `/admin/structure/views/view/тест`), Drupal now returns a 404 instead of an unhandled database exception that produced a white screen of death. This affects all config entity routes in core and contrib. No configuration changes are needed; the fix is applied automatically at route build time.

## Impact

Affects module maintainers.

- **New API:** New `ParamConverterRouteRequirementInterface` with `getRouteRequirement()` lets param converters declare route regex requirements; `EntityConverter` now implements it.

## Technical details

Config entity IDs are stored in the `config` table's `name` column, which uses `varchar_ascii` / `ascii_general_ci` collation. Passing non-ASCII input through a param converter caused a MySQL collation mismatch (`SQLSTATE[HY000]: General error: 1267`) deep in `DatabaseStorage::readMultiple()`, with no handler in the call stack to convert it into a clean 404.

The fix introduces a new interface `ParamConverterRouteRequirementInterface` in `core/lib/Drupal/Core/ParamConverter/`. It declares a single method `getRouteRequirement($definition, $name): ?string` that returns a regex pattern (without delimiters or anchors) suitable for `Route::setRequirement()`, or `NULL` when no requirement can be determined. `EntityConverter` now implements this interface and returns the constant `PRINTABLE_ASCII` (`[\x20-\x7E]+`) for static config entity types (`ConfigEntityTypeInterface`), and `NULL` for content entities and dynamic (`entity:{...}`) entity types that cannot be resolved at route collection build time.

`ParamConverterManager::setRouteParameterConverters()` is updated to check whether the assigned converter implements `ParamConverterRouteRequirementInterface`. If it does and the route has no existing requirement for that parameter, it calls `getRouteRequirement()` and passes the result to `Route::setRequirement()`. This causes the Symfony router itself to reject non-matching URLs with a 404 before any entity loading or database query is attempted.

For dynamic entity type parameters (`entity:{entity_type}`), which cannot have requirements set at build time, `EntityConverter::convert()` now additionally checks whether the resolved entity type implements `ConfigEntityInterface` and, if so, validates the incoming value against `PRINTABLE_ASCII`, returning `NULL` immediately if the check fails. This converts to a 404 via `ParamNotConvertedException` at request time.

## Upgrade

Custom param converter services can now implement `ParamConverterRouteRequirementInterface` to automatically set route requirements:

```php
use Drupal\Core\ParamConverter\ParamConverterInterface;
use Drupal\Core\ParamConverter\ParamConverterRouteRequirementInterface;

class MyConverter implements ParamConverterInterface, ParamConverterRouteRequirementInterface {

  public function getRouteRequirement($definition, string $name): ?string {
    // Return a regex pattern (no delimiters, no anchors).
    return '[a-z0-9_]+';
  }

  // ... convert(), applies() ...
}
```

If `getRouteRequirement()` returns a non-null value and the route has no existing requirement for that parameter, `ParamConverterManager` will call `Route::setRequirement($name, $pattern)` automatically during route collection building.

### Collaboration

The issue sat unresolved for months while the thread debated where to throw the exception (database storage, entity storage, or routing level) and what the end-user response code should be (404 vs 500). The design breakthrough came when alexpott proposed using the routing system itself: a new interface on param converters that lets them declare a regex requirement for their parameters, so the router rejects invalid URLs before any entity loading occurs. alexpott opened a second MR implementing this approach. Dries then joined the effort, wrote kernel and functional tests, caught that `getRouteRequirement()` failed to cover multi-wildcard config schema types (like `entity_view_mode`), and proposed the printable-ASCII fallback. He also spotted that the earlier test for `read()` with non-ASCII input was dead code due to PHPUnit's `expectException()` being a setter that overwrites previous calls, and split the test into separate methods to properly assert each exception. The combination of alexpott's architectural direction and Dries's testing and edge-case analysis shaped the final commit.
