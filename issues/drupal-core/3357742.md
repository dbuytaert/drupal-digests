## Summary

This plan establishes official Drupal.org guidelines for how contributed project maintainers should apply semantic versioning when their code changes its compatibility relationship with Drupal core. During the Drupal 10 readiness effort, many maintainers bumped their module's major version solely to add Drupal 10 support while dropping Drupal 9, which forced site owners to manually update Composer version constraints for every affected module during a core upgrade. The new guidelines reduce that burden by reserving major version increments for cases where a currently-supported Drupal core version is being dropped, and allowing minor or patch releases for less disruptive compatibility changes. The documentation was published at the Drupal.org "Managing branches &amp; releases" guide under the contrib project maintainership section.

## Impact

No upgrade or configuration changes required.

## Technical details

The core problem was that Composer's `^` constraint syntax means a site with `"drupal/foo": "^1.0"` must explicitly change its constraint if `drupal/foo` releases `2.0.0`, even if the only change is dropping an already-end-of-life Drupal version. Because Composer's dependency resolver will refuse to install a release whose `core_version_requirement` no longer matches the site's Drupal version, bumping a major version for that reason is redundant and creates cascading update burdens across dependent modules.

The finalized guidelines, scoped deliberately to only the contrib-to-Drupal-core relationship (not all semver scenarios), define three cases:

**Major version**: the module drops compatibility with a version of Drupal core that is still actively supported, or introduces breaking changes from Symfony or another shared dependency where backward compatibility cannot be maintained (e.g., adding types to method signatures), or creates impossible dependency resolution (e.g., requiring Guzzle 7 while a supported Drupal version requires Guzzle 6).

**Minor version**: the module drops compatibility with a version of Drupal core that has already reached end-of-life. Composer prevents the new release from installing on the unsupported Drupal version, so the semver contract is preserved without a major bump.

**Patch version**: the module adds compatibility with a new Drupal core major version while maintaining full backward compatibility with all previously supported versions.

The decision to classify EOL-Drupal drops as minor rather than major was explicitly influenced by the argument that Drupal core is not an ordinary library dependency but the operating environment for a module, and that signaling a breaking change via a major version is only meaningful when the change actually risks breaking something on a user's running site. Related policy context: the overlap with Drupal core's own long-term support schedule (see [#3238652](https://www.drupal.org/node/3238652), which defined how long major Drupal versions remain supported) directly informs when a drop crosses from "minor" to "major" territory.

### Collaboration

Two contributions were decisive in shaping the final guidelines. japerry (comment #13) made an extended argument against using major version bumps for dropping unsupported Drupal versions, pointing out that a contrib ecosystem where every module cuts a new major on each core transition forces site owners to chase multiple dependency targets simultaneously rather than one (core). The argument that Composer's own resolver already prevents installation on unsupported Drupal versions, making a major bump redundant, gained quick agreement from catch. mark_fullmer then restructured the proposal (comment #17) to narrow its scope explicitly to changes in the module's relationship to Drupal core, removing the attempt to prescribe all semver scenarios and making the guidelines easier to adopt in practice. quietone handled the actual documentation work: drafting the initial text from mglaman's blog post, publishing the page, and later relocating it to the "Managing branches &amp; releases" guide after dww pointed out a better home for it.
