## Summary

A long-standing bug in block rendering caused the `#attributes` from a block plugin's content render array to be incorrectly merged into the outer block wrapper element. For example, a form rendered inside a block would have its `data-drupal-selector` attribute, CSS classes, and other attributes moved from the `<form>` tag to the surrounding block `<div>`, breaking CSS, JavaScript selectors, AJAX behavior, and in some cases causing white-screen errors. The fix corrects `BlockViewBuilder` so that only `#wrapper_attributes` (a separate, explicitly intended key) from the content render array are promoted to the block wrapper. Regular `#attributes` now stay with the content element where they belong. A backward-compatibility preprocess hook is added to the Stable9 theme so that sites using it as a base theme are not immediately broken.

## Impact

Affects module maintainers and front-end developers.

- **Theme change:** Block content `#attributes` no longer bubble up to the block wrapper `<div>`; use `#wrapper_attributes` in `build()` to intentionally target the wrapper. Themes not based on Stable9 may need to update CSS/JS selectors.
- **API change:** `BlockViewBuilder::preRender()` now reads `#wrapper_attributes` (not `#attributes`) from the content render array to populate the block wrapper; block plugins targeting the outer wrapper must switch to `#wrapper_attributes`.

## Technical details

The problem originated in [#2171269](https://www.drupal.org/node/2171269), which added logic to `BlockViewBuilder::preRender()` that moved both `#attributes` and `#contextual_links` from the block plugin's `$content` render array into the top-level `$build` array. While that was correct for `#contextual_links`, applying it to `#attributes` was too broad: any attribute a plugin set for its own content element (a `<form>`, a list, etc.) was silently hijacked by the block wrapper.

The fix in `BlockViewBuilder::preRender()` splits the two properties: `#contextual_links` still bubbles as before, but `#attributes` is no longer touched. Instead, a new convention is established: if a block plugin's `build()` array contains `#wrapper_attributes`, those are merged into `$build['#attributes']` (the outer wrapper). This mirrors the existing `#wrapper_attributes` convention used elsewhere in Drupal's render system.

For backward compatibility, a `hook_preprocess_block()` implementation is added to `Stable9Hooks` in the `stable9` theme. It copies any `#attributes` found in `$variables['content']` into `$variables['attributes']` (the wrapper) and then unsets the content key, replicating the old behavior for themes that extend Stable9. The `TestHtmlBlock` test plugin and its accompanying functional test (`BlockHtmlTest`) are updated to use `wrapper_attributes` through the key-value store, aligning the test fixture with the new convention.

## Upgrade

Block plugins that previously relied on `#attributes` in their `build()` render array to style or identify the **block wrapper** element must switch to `#wrapper_attributes`:

```php
// Before (broken): #attributes on the content was incorrectly merged
// into the block wrapper, not the content element.
public function build(): array {
  return [
    '#theme' => 'item_list',
    '#items' => $items,
    '#attributes' => ['class' => ['my-list']],
  ];
}

// After: use #wrapper_attributes to intentionally target the block wrapper.
public function build(): array {
  return [
    '#theme' => 'item_list',
    '#items' => $items,
    '#attributes' => ['class' => ['my-list']],   // stays on the rendered element
    '#wrapper_attributes' => ['class' => ['my-block-wrapper-class']],
  ];
}
```

Custom or contrib themes that do **not** extend Stable9 and that have CSS or JavaScript selectors depending on content `#attributes` appearing on the block wrapper `<div>` will need to audit those selectors. The Stable9 backward-compatibility hook is only active for themes using Stable9 as a base theme.
