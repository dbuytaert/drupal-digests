## Summary

A regression introduced in [#3311365](https://www.drupal.org/node/3311365) caused route discovery to reject `routing.yml` files that specified `methods` or `schemes` as a plain string (e.g., `methods: 'GET'`) instead of a YAML array. Symfony's own Route constructor has always accepted either form, but Drupal's `StaticRouteDiscoveryBase::createRoute()` was narrowed to require arrays only. The fix restores compatibility: both `methods` and `schemes` now accept a string or an array, matching Symfony's behavior. The Gutenberg contrib module was the first reported breakage.

## Impact

No upgrade or configuration changes required.

## Technical details

The regression originated in [#3311365](https://www.drupal.org/node/3311365), which refactored route discovery to use PHP attributes. That change narrowed the `$methods` and `$schemes` parameters of `StaticRouteDiscoveryBase::createRoute()` from the implicit `string|array` that Symfony's `Route` constructor accepts to a strict `array`. Any `routing.yml` that used a bare string for either key (valid YAML and previously accepted by Symfony) would then fail at route compilation.

The fix widens those two parameter types back to `string|array` in the `protected function createRoute()` signature. The underlying `Route` constructor from Symfony already handles the string-to-array normalization, so no further change is needed in the method body. A new kernel test class `YamlRouteDiscoveryTest` and a companion fixture route `router_test.string_method_scheme` (with `methods: 'GET'` and `schemes: 'https'` as strings) cover the restored behavior.
