## Summary

SDC components (`#type => 'component'`) can now participate in form processing as input elements. Before this change, the `component` render element was not recognized by the Form API as an input element, so form properties like `#default_value`, `#value`, `#required`, and `#element_validate` had no effect and submitted values were not tracked. Site builders and developers can now use an SDC component directly in a `buildForm()` return array, and the Form API will collect its submitted value, respect `#default_value` and `#value`, enforce `#required`, and run `#element_validate` callbacks. The component's Twig template is responsible for rendering the actual HTML control using a `form_state` prop; the PHP layer handles all business logic. Support for `#ajax` and `#htmx` is out of scope and deferred to follow-up issues.

## Impact

Affects module maintainers.

- **New API:** SDC components (`#type => 'component'`) now implement `FormElementInterface` and are recognized as Form API input elements, supporting `#default_value`, `#value`, `#required`, and `#element_validate`.

## Technical details

The `ComponentElement` class in `core/lib/Drupal/Core/Render/Element/ComponentElement.php` previously carried the `#[RenderElement('component')]` attribute and extended `RenderElementBase`. The Form API's `ElementInfoManager` only sets `#input = TRUE` on elements registered with `#[FormElement]`, so `ComponentElement` was invisible to `FormBuilder::handleInputElement()` and submitted values were never collected.

The fix changes the attribute to `#[FormElement('component')]` and adds `implements FormElementInterface` to the class declaration. `ComponentElement` still extends `RenderElementBase` rather than `FormElementBase`, deliberately avoiding the extra static helper methods in `FormElementBase` that are not applicable to components (as noted in the new docblock).

A `valueCallback()` method is added that returns `NULL`. This tells the Form API to fall back to `#default_value` or `#value` for determining the element's stored value, with no server-side transformation. Value rendering is delegated entirely to the Twig template.

In `preRenderComponent()`, a new block checks for a non-empty `#name` (which the Form API sets automatically on `#input` elements). When present, it populates `$props['form_state']['value']['name']` and `$props['form_state']['value']['required']`, making those values available inside the component's Twig template via `form_state.value.name` and `form_state.value.required`. The template (not the PHP class) is responsible for rendering the `<input>` element, its `name` attribute, `value`, and the HTML `required` attribute.

A test fixture component (`sdc_theme_test:input`) is added under the SDC test theme, consisting of a YAML metadata file declaring an `id` prop and a `label` slot, plus a Twig template that uses `form_state.value.name`, `form_state.value.value`, and `form_state.value.required` to render a labeled text input. The kernel test class `ComponentAsFormElementTest` (in `core/tests/Drupal/KernelTests/Components/`) exercises rendering and submission with `#default_value`, `#value`, `#required`, `#slots`, `#props` (including `attributes`), and `#element_validate` callbacks.

The design principle articulated in the thread (comment 16133116 by pdureau) is that SDC components own the UI logic; callbacks like `#element_validate` are added at the PHP/applicative layer by code that returns a `#type => 'component'` element, not declared inside the component YAML itself. `#ajax`/`#htmx` support is explicitly out of scope for this change.
