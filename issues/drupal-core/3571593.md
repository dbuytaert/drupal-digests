## Summary

The interface translations directory path can no longer be configured through the UI or configuration. Site owners who need to customize the path must now set `$settings['locale_translation_path']` in `settings.php` instead. This follows the same pattern used for private file paths and is the last system path to move out of configuration. Sites using the default path (`public://translations`) are unaffected.

## Impact

Affects module maintainers and site owners.

- **Config change:** `locale.settings:translation.path` is deprecated and removed from default configuration
- **Deprecation:** `locale_system_file_system_settings_submit()` function deprecated with no replacement

## Technical details

After private and public file paths moved to settings, `locale.settings:translation.path` was the last system path remaining in configuration. It was also the last usage of the `system_check_directory` function.

The change adds `TranslationsStream::basePath()` as the single source for the translations directory path. This method first checks for the deprecated config value (for backwards compatibility), then falls back to `Settings::get('locale_translation_path', 'public://translations')`. Stream wrappers such as `public://` are resolved to real paths using `FileSystemInterface::realpath()`.

All code that previously read the config directly now calls `basePath()`. This includes `locale_translate_get_interface_translation_files()`, `locale_install()`, and the stream wrapper's `getDirectoryPath()` method.

The file system settings form field changes from `#type: textfield` to `#type: item` (display-only). The form submit handler `locale_system_file_system_settings_submit()` is deprecated and removed from the form's submit handlers.

The post update hook `locale_post_update_clear_translation_path_config()` compares the config value to the new default using `FileSystemInterface::realpath()`. If they match, it clears the config. If they differ, it leaves the config in place and returns a message directing the site owner to move the value to `settings.php`. This preserves customized paths during the update while removing defaults.

## Upgrade

If your site has a custom translations directory path set in configuration, add it to `settings.php`:

```php
$settings['locale_translation_path'] = '/path/to/translations';
```

The update hook will display a message with the exact path to migrate if you have a non-default value.

### Collaboration

The initial approach attempted to migrate config values in the post update hook, but this proved problematic for update path tests where the default `public://translations` path resolves to a dynamic test-specific directory that doesn't match the static fixture. After discussion in Slack, the team decided to conditionally clear the config only when it matches the default, leaving custom values in place with a message. Alex Pott suggested this approach could avoid needing a requirements hook by using an update requirement in a follow-up for the next major version.
