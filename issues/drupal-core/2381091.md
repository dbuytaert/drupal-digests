## Summary

The `example.gitignore` template files shipped with Drupal have been updated to more reliably ignore environment-specific and multisite configuration files. The settings and services wildcard patterns now also match files stored one directory level deeper inside a site folder, covering more real-world multisite layouts. Explicit negation rules are added so that the core-provided `default.settings.php` and `default.services.yml` template files are never accidentally hidden from `git status`. The alternate section of the file, intended for setups where `.gitignore` lives inside the `sites/` folder rather than the repo root, was previously missing any rules for services files; it now mirrors the full set of rules including negations.

## Impact

No upgrade or configuration changes required.

## Technical details

Two files are changed identically: `example.gitignore` in the repo root and `core/assets/scaffold/files/example.gitignore` (the scaffold copy that Composer-based installs receive).

**Pattern depth widened.** The previous patterns `sites/*/settings*.php` and `sites/*/services*.yml` only matched files directly inside a named site directory. They are replaced with `sites/*/*settings*.php` and `sites/*/*services*.yml`, which add one extra directory component and therefore also catch configuration files stored in a subdirectory within a site folder, covering more multisite and environment-specific naming conventions.

**Explicit allow rules added.** Because the broader wildcard would otherwise also suppress the core-tracked template files, two negation lines are added immediately after each ignore rule: `!sites/*/default.settings.php` and `!sites/*/default.services.yml`. These ensure that files already tracked in the Drupal core repository remain visible in `git status` for core contributors, consistent with how git ignores work (tracked files are unaffected, but un-ignore rules clarify intent for tools that display effective ignore rules).

**Alternate `sites/`-based section completed.** The file includes a commented-out block for setups where `.gitignore` is placed inside `sites/` rather than the project root. This section previously contained only `*/settings*.php` with no coverage of services files at all (noted in the original issue as the motivating gap, related to [#2326913](https://www.drupal.org/node/2326913) which first added the services pattern to the main section). It is now updated to `*/*settings*.php` with `!*/default.settings.php`, and gains `*/*services*.yml` with `!*/default.services.yml`, making the alternate section consistent with the primary one.
