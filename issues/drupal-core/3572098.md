## Summary

On sites using JSON:API to serve large numbers of complex entities, the normalization cache could be slower than just re-running the normalization itself, due to I/O cost and deserialization memory overhead. This change makes the JSON:API normalization cacher respect a `max-age` of 0 on `ResourceObject` instances: when max-age is 0 the cacher skips both reading from and writing to the cache entirely. Site owners who build custom JSON:API resources and want to suppress the normalization cache overhead can now do so by setting max-age to 0 on the resource object.

## Impact

No upgrade or configuration changes required.

## Technical details

The JSON:API normalization caching layer, introduced in [#2819335](https://www.drupal.org/node/2819335) to avoid re-normalizing expensive entities on each request, writes partial normalization results to the variation cache at request termination and reads them back on subsequent requests. The read path lives in `ResourceObjectNormalizerCacher::get()` and the write path in `ResourceObjectNormalizerCacher::saveOnTerminate()`.

Neither method previously checked the cache max-age of the `ResourceObject` being processed. For custom JSON:API resources where entities are pre-processed into simple arrays before serialization, the field API overhead is already paid before normalization, making the cache I/O (especially to remote backends like Redis, which also involves deserialization on retrieval) more expensive than re-normalizing from scratch.

The fix adds an early-return guard in both `get()` and `saveOnTerminate()`: if `$object->getCacheMaxAge() === 0` the method returns immediately without touching the variation cache. This aligns with Drupal's standard caching convention that max-age 0 means uncacheable and also prevents uncacheable resource objects from being held in the in-memory `saveOnTerminate` queue for the lifetime of the request.

A new kernel test `testResourceObjectMaxAge0IsHandledByCacher()` in `ResourceObjectNormalizerCacherTest` verifies that a normally-cached resource object can be read back after `saveOnTerminate`, and that after `mergeCacheMaxAge(0)` is applied neither `saveOnTerminate` nor `get` interact with the cache.
