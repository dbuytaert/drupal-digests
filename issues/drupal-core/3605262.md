## Summary

A critical data-integrity regression introduced by the database write optimizations in [#3554579](https://www.drupal.org/node/3554579) has been reverted. When an existing pending revision was promoted to the default revision while field values were also changed in the same save call, the revision table was not updated, leaving it out of sync with the base table. This could cause silent data loss or inconsistency for sites using Content Moderation, Workspaces, or any contributed module that promotes revisions and modifies fields simultaneously. The fix reverts the optimization in `SqlContentEntityStorage::saveToDedicatedTables()` and adds a targeted regression test. The original issue ([#3554579](https://www.drupal.org/node/3554579)) has been re-opened to attempt a corrected optimization.

## Impact

No upgrade or configuration changes required.

## Technical details

The optimization in [#3554579](https://www.drupal.org/node/3554579) introduced three local variables in `saveToDedicatedTables()`: `$save_to_base_table`, `$enforce_save_to_base_table`, and `$save_to_revision_table`. The intent was to skip the revision-table write when promoting a pending revision to default (because only the base table was assumed to need updating). However, the condition set `$save_to_revision_table = false` whenever `$enforce_save_to_base_table` was true, regardless of whether field values had also changed. When field values did change in the same operation, those new values were written only to the base table, leaving the revision table holding stale data.

The revert removes all three variables and restores simpler, always-write logic: the base table is written when `$entity->isDefaultRevision()` returns true; the revision table is written whenever `$this->entityType->isRevisionable()` returns true. The skip-on-no-change guard is simplified to: skip only when not a new revision, the default-revision flag has not changed (`$entity->isDefaultRevision() === $original->isDefaultRevision()`), and `hasFieldValueChanged()` returns false.

The regression test added to `testFieldEntityRevisionWrite()` exercises the precise failing scenario: saving a pending revision with a new value (`baz`), then loading by both `loadUnchanged()` and `loadRevision()` to confirm both the base and revision tables reflect the new value.
