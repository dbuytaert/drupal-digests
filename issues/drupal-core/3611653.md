## Summary

Upgrading from Drupal 11.3 to 11.4 broke database upsert operations where every field passed to the upsert is also part of the key (the "degenerate" case, meaning there are no non-key columns to update on conflict). MySQL returned a syntax error because the generated SQL ended with `ON DUPLICATE KEY UPDATE` and nothing after it. The fix restores this as a supported pattern across all database drivers: the statement now silently ignores duplicates rather than crashing.

## Impact

No upgrade or configuration changes required.

## Technical details

The root cause was an accidental behavioral change introduced by [#2547493](https://www.drupal.org/node/2547493), which added composite key support to `Upsert` queries in 11.4. That commit used `array_combine()` to rekey `$insert_fields` from a zero-based numeric array to a string-keyed associative array. Once the array had string keys, the existing `unset($insert_fields[$key])` loop successfully removed the key columns from the update list. When all fields are key fields, this left `$update` empty, and the MySQL driver emitted `INSERT INTO ... ON DUPLICATE KEY UPDATE` with no assignments, which is a syntax error. In 11.3 the `unset` call was effectively a no-op because the array was still zero-indexed, so the key column remained in the update list and MySQL accepted the statement (performing a no-op update on conflict).

The fix introduces an explicit check for the empty-update case in each driver. In `core/modules/mysql/src/Driver/Database/mysql/Upsert.php`, when `$update_fields` is empty after removing key columns, the query is built as `INSERT IGNORE INTO ...` instead of `INSERT ... ON DUPLICATE KEY UPDATE`. In the PostgreSQL drivers (`core/modules/pgsql` and `core/modules/sqlite`), the same empty check switches from `ON CONFLICT (...) DO UPDATE SET ...` to `ON CONFLICT (...) DO NOTHING`. Three new kernel tests cover the degenerate case: `testDegeneratedSingleValueUpsert`, `testDegeneratedMultiValueUpsert`, and `testDegeneratedCompositeKeyUpsert`.
