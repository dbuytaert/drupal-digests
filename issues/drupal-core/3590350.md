## Summary

Sites using the Workspaces module were exposed to a data-integrity risk: if a PHP `\Error` subclass (such as `\TypeError` or `\AssertionError`) was thrown during a workspace publish or merge operation, the database transaction was not rolled back. This meant the first entities in the save loop could be committed to Live while later ones were not, leaving content in a partially published state with no error logged on the `workspaces` channel. The fix ensures the transaction is always rolled back and the throwable always logged, regardless of whether an `\Exception` or an `\Error` subclass escapes the save loop.

## Impact

No upgrade or configuration changes required.

## Technical details

Both `WorkspacePublisher::publish()` and `WorkspaceMerger::merge()` wrapped their entity-save loops in `try / catch (\Exception $e)` blocks that called `$transaction->rollBack()` and `Error::logException()` before rethrowing. Because PHP's `\Error` hierarchy is not a subclass of `\Exception`, any `\Error` thrown from inside the closure (for example from a `hook_entity_presave()` implementation or a downstream service called during `$entity->save()`) bypassed the catch block entirely. The fix widens both catch clauses to `catch (\Throwable $e)`. `Error::logException()` already accepts `\Throwable`, so no change to the catch body was required. A PHPStan suppression for a false-positive `isset.variable` warning in `WorkspaceMerger.php` was also removed as it became unnecessary. A kernel test (`WorkspacePublisherTest::testPublishThrowableRollback`) was added using a `#[Hook('node_presave')]` implementation on the test class itself to simulate a failure mid-loop; a data provider covers both `\RuntimeException` (previously caught) and `\Error` (previously missed), asserting that neither node's Live revision changes and that the error is recorded on the workspaces logger channel.
