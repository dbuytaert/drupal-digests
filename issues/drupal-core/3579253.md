## Summary

When PHPUnit skips an entire test class because a PHP extension it requires is not installed (e.g. via `#[RequiresPhpExtension('apcu')]`), it exits with a success code and emits "No tests executed!" rather than reporting individual test results. Drupal's `run-tests.sh` wrapper was treating this as a passing test run. The fix makes `run-tests.sh` treat zero collected test results as an error condition, the same way it already treats deprecations or warnings, so developers running the test suite on machines with missing extensions see a clear failure instead of a silent false positive.

## Impact

No upgrade or configuration changes required.

## Technical details

The affected class is `PhpUnitTestRunner`. In `processPhpUnitOnSingleTestClassOutcome`, the condition that triggers error output was extended from `$status > TestStatus::PASS` to also fire when `$phpunit_results === []`, covering the case where PHPUnit produced no result records because it never ran any tests.

Because `summarizeResults` previously built its result map by iterating over `$phpunit_results` and reading `$result['test_class']` from each entry, an empty results array produced an empty summary map with no entry for the class under test. The method now accepts the class name as a required first argument (`string $test_class`) and pre-initializes the summary entry for that class before iterating, ensuring an entry always exists even when results are empty. The test data provider `providerTestSummarizeResults` was also refactored from a plain `array` return to a PHP generator (`\Generator`) to match the new three-argument test method signature.

## Upgrade

If custom code calls `PhpUnitTestRunner::summarizeResults()` directly, note that the method is marked `@internal` but its signature changed: a `string $test_class` argument is now required as the first parameter.

```php
// Before
$summaries = $runner->summarizeResults($phpunit_results);

// After
$summaries = $runner->summarizeResults($test_class, $phpunit_results);
```
