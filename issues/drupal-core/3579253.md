## Summary

When `run-tests.sh` ran a PHPUnit test class whose requirements were not met (for example, a class annotated with `#[RequiresPhpExtension('apcu')]` on a machine without that extension), PHPUnit exited with success and printed "No tests executed!". The test runner accepted that as a pass. After this fix, an empty test result set is treated as an error, surfacing the missed class to the developer just as deprecations and warnings already were.

## Impact

No upgrade or configuration changes required.

## Technical details

The fix is in `PhpUnitTestRunner::processPhpUnitOnSingleTestClassOutcome`. The existing condition that triggered extended error output only fired when `$status > TestStatus::PASS`. A second condition `|| $phpunit_results === []` was added so that a completely empty result set also triggers error reporting.

Because the empty-results path now needs to report a failure for the specific class, `summarizeResults` was refactored to accept the current `$test_class` as a required first argument (`string $test_class`). Previously the method inferred the class name from whichever entries appeared in `$results`; now it initializes the summary entry for `$test_class` upfront before iterating results, ensuring the class is present in the summary even when `$results` is empty. The method is marked `@internal`, so this signature change does not affect the public API. The corresponding test provider was converted from a static array to a `\Generator` and updated to pass the class name as the first argument to `summarizeResults`.
