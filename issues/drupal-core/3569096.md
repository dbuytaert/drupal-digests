## Summary

In preparation for Symfony 8 compatibility in Drupal 12, the legacy `$options` backward-compatibility argument has been removed from the constructors of all built-in Drupal constraint plugin classes. At the same time, the `getDefaultOption()` and `getRequiredOptions()` methods, deprecated in Symfony 7.4 and removed in Symfony 8, have been deleted from every Drupal constraint class that previously overrode them. This change is internal to the validation subsystem and has no user-facing effect, but it does break any custom module that either subclasses a built-in constraint or instantiates one using the old positional-array `$options` calling convention.

## Impact

Affects module maintainers.

- **API change:** The `mixed $options = NULL` first constructor parameter is removed from all built-in Drupal constraint classes; `getDefaultOption()` and `getRequiredOptions()` overrides are also removed, mirroring their removal in Symfony 8.

## Technical details

Symfony's `Constraint` base class historically accepted a catch-all `mixed $options` first argument that let callers pass constraint properties as a positional array. Symfony deprecated this pattern in 7.4 and removes it entirely in Symfony 8. Each affected Drupal constraint class previously forwarded that argument to `parent::__construct($options, $groups, $payload)`.

The change removes `mixed $options = NULL` from the constructor of every Drupal-defined constraint class (roughly 50 classes spanning `core/`, `ckeditor5/`, `file/`, `link/`, `media/`, `user/`, and several test modules). `parent::__construct()` calls are updated to use named arguments: `parent::__construct(groups: $groups, payload: $payload)`. Properties that were previously declared as class-level `public $foo = default` and then assigned conditionally in the constructor body (the `$this->foo = $foo ?? $this->foo` pattern) are collapsed into constructor property promotion. Where a property was previously "required" only through `getRequiredOptions()` but had a nullable default, the constructor parameter becomes genuinely required with no default (e.g., `BundleConstraint::$bundle`, `EntityHasFieldConstraint::$field_name`, `ImmutablePropertiesConstraint::$properties`, `ValidKeysConstraint::$allowedKeys`, `StringEqualsConcatenatedValuesConstraint::$separator` and `$values`, and others).

`ComplexDataConstraint` required special handling because it previously relied on the `$options` array to accept a bare array of properties without a `properties` key. That implicit-key logic is replaced with an explicit check: if `$properties` is `NULL` but variadic `$otherProperties` are present, those become the properties; otherwise an `\InvalidArgumentException` is thrown. `MenuLinkDepthConstraint`, which extended Symfony's `RangeConstraint` and forwarded arguments via `...$args`, is rewritten to explicitly enumerate all parent constructor parameters. `EmailConstraint` and `RangeConstraint` (Drupal subclasses of Symfony built-ins) pass `NULL` explicitly as the first argument rather than forwarding a Drupal-level `$options` param.

The companion issue [#3569092](https://www.drupal.org/node/3569092) covers the related removal of `HasNamedArguments` and the `ConstraintManager::create()` / `addConstraint()` signature updates.

## Upgrade

Custom constraint plugins that subclass any of the affected classes, or that follow the same pattern, must be updated:

1. **Remove the `$options` parameter** from your constructor:
```php
// Before
public function __construct(
    mixed $options = NULL,
    public string $myOption = 'default',
    ?array $groups = NULL,
    mixed $payload = NULL,
) {
    parent::__construct($options, $groups, $payload);
    $this->myOption = $myOption ?? $this->myOption;
}

// After
public function __construct(
    public string $myOption = 'default',
    ?array $groups = NULL,
    mixed $payload = NULL,
) {
    parent::__construct(groups: $groups, payload: $payload);
}
```

2. **Remove `getDefaultOption()` and `getRequiredOptions()` overrides** — these methods no longer exist on the Symfony 8 `Constraint` base class and must be deleted from any subclass.

3. **Update call sites** that passed constraint options as a positional array (the old Symfony style):
```php
// Before — passing options array as first argument
new MyConstraint(['myOption' => 'value']);

// After — pass named arguments directly
new MyConstraint(myOption: 'value');
```
