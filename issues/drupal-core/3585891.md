## Summary

Drupal 11.4 deprecates the fallback that allowed CSRF tokens validated against the old `'rest'` key to pass the `X-CSRF-Token` access check. This key was a leftover from a Drupal 8 migration path and a `@todo` comment in the code targeted its removal as far back as Drupal 8.3. Any module that still generates or validates CSRF tokens using the literal string `'rest'` will now trigger a deprecation notice; the behaviour continues working but will break in Drupal 12. The `user.login.http` endpoint was also updated to generate its CSRF token using the canonical key.

## Impact

Affects module maintainers.

- **Deprecation:** CSRF token validation and generation using the hardcoded 'rest' key is deprecated in 11.4.0; switch callers to CsrfRequestHeaderAccessCheck::TOKEN_KEY before Drupal 12.0.0

## Technical details

In `CsrfRequestHeaderAccessCheck::access()`, the old double-`validate()` call (checking both `self::TOKEN_KEY` and `'rest'`) has been restructured: if the primary check fails and the token validates under `'rest'`, a `@trigger_error()` deprecation is issued but the request is still allowed through. If neither key validates, `AccessResult::forbidden()` is returned as before. The constant `TOKEN_KEY` is `'X-CSRF-Token'`.

`UserAuthenticationController::login()` previously called `$this->csrfToken->get('rest')` to produce the CSRF token included in the login response. It now imports `CsrfRequestHeaderAccessCheck` and calls `$this->csrfToken->get(CsrfRequestHeaderAccessCheck::TOKEN_KEY)`. This was the concrete caller that caused initial test failures when a straight removal was attempted, and it motivated pivoting to a deprecation-first approach rather than an immediate removal.

## Upgrade

Any code that calls `$csrfToken->get('rest')` to produce a CSRF token for REST requests must switch to the constant:

```php
// Before (deprecated):
$token = $this->csrfToken->get('rest');

// After:
use Drupal\Core\Access\CsrfRequestHeaderAccessCheck;
$token = $this->csrfToken->get(CsrfRequestHeaderAccessCheck::TOKEN_KEY);
```

Similarly, any explicit `$csrfToken->validate($value, 'rest')` calls should use `CsrfRequestHeaderAccessCheck::TOKEN_KEY` as the second argument. The `'rest'` key fallback is removed in Drupal 12.0.0.
