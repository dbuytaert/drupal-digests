## Summary

The block "response status code" visibility condition (introduced in [#2245767](https://www.drupal.org/node/2245767) to show or hide blocks based on 200/403/404 response pages) was using the `url.path` cache context. Because 404 pages typically have unique paths, every distinct 404 URL visited generated its own dynamic page cache entry, rapidly bloating the cache. This change replaces that context with a new, coarser `exception_status_code` cache context so that all requests sharing the same HTTP error code (e.g. all 404s) can reuse a single cache entry, dramatically reducing cache table growth on sites with many 404s.

## Impact

Affects module maintainers.

- **New API:** New `exception_status_code` cache context (`ExceptionStatusCodeCacheContext`) for varying cache entries by HTTP exception status code.

## Technical details

The `HttpResponseCodeCondition` plugin's `getCacheContexts()` method previously appended `url.path` to its declared cache contexts. The fix replaces this with `exception_status_code`.

The new `ExceptionStatusCodeCacheContext` class (registered as `cache_context.exception_status_code`) extends `RequestStackCacheContextBase` and reads the `exception` attribute from the current request. `HttpExceptionSubscriberBase` sets this attribute before block rendering, so it is reliably available. If the attribute holds an instance of `HttpExceptionInterface`, the context returns the status code as a string (e.g. `'404'`, `'403'`). If there is no exception or it is not an `HttpExceptionInterface`, it returns `'0'` to represent a normal (typically 200) response. Returning `'0'` rather than `'200'` avoids falsely claiming knowledge of what response subscribers may set.

## Upgrade

No upgrade steps are required for existing code. If you maintain a custom cache context or condition plugin that currently uses `url.path` purely to vary by error status, consider switching to `exception_status_code` instead.
