## Summary

On every cold cache request, the `Datetime` form element class was loading two date format config entities (`html_date` and `html_time`) as part of populating the element info cache, even on pages that never rendered a datetime form element. This added 2 database queries and 2 cache set operations to every cache warmup cycle. The fix moves those loads out of `getInfo()` and defers them to the form processing callbacks that run only when a datetime element is actually being rendered. The performance tests confirm a reduction of 2 queries, 2 cache gets, and 2 cache sets per cold-cache request on both the front page and node pages.

## Impact

No upgrade or configuration changes required.

## Technical details

The `Datetime` element's `getInfo()` method was using `DateFormat::load('html_date')` and `DateFormat::load('html_time')` to populate `#date_date_format` and `#date_time_format` default properties. Because `getInfo()` results populate a shared element info cache, those loads ran on every cold-cache request regardless of whether any datetime element appeared on the page.

The fix removes `#date_date_format` and `#date_time_format` from the `getInfo()` return value entirely. Instead, `valueCallback()` and `processDatetime()` each call `DateFormat::loadMultiple(['html_date', 'html_time'])` at the start and set the two properties via `$element +=` (the `+=` operator preserves any caller-supplied values, keeping the existing override contract). The redundant duplicate `DateFormat::load()` calls that previously existed inside the `datetime-local` branch of `valueCallback()` are also removed, replaced by reuse of the already-loaded `$formats` array.

The `DatetimeElementFormTest` kernel test was missing a call to `installConfig(['system'])`, so it did not have the `html_date` and `html_time` config entities available; this was fixed by adding a `setUp()` override that installs both the entity schema and system config before tests run.

The review thread had a substantive back-and-forth over whether to add NULL checks when `DateFormat::loadMultiple()` returns results. `dcam` flagged inconsistent NULL guarding in the MR, `godotislate` pressed further and noted that `$formats['html_date']` or `$formats['html_time']` could theoretically be NULL. `alexpott` resolved the debate by pointing out that both `html_date` and `html_time` are locked system config entities that cannot be deleted, making the NULL guards incorrect rather than merely unnecessary. `catch` confirmed this by attempting the removal, observing a test failure in `DatetimeElementFormTest`, diagnosing that the test was simply missing `installConfig(['system'])`, and fixing the test rather than restoring the guards.
