## Summary

When site builders add an "Authored by" filter to a Media view in the Views UI, it now appears as a username autocomplete field instead of a plain text box that expected a raw numeric user ID. This brings Media views into parity with Content views at `/admin/content`, where the same filter has long used username autocomplete. Existing saved views are not changed; only filters added after this fix benefit from the improved UX.

## Impact

No upgrade or configuration changes required.

## Technical details

The root cause was that `MediaViewsData` did not expose a dedicated `user_name` virtual field for the media author column, so the only available "Authored by" filter in the Views UI was the raw `uid` numeric field.

The fix is in `MediaViewsData::getViewsData()` (`core/modules/media/src/MediaViewsData.php`). It registers a new virtual field `user_name` inside the `media_field_data` table data. The entry copies the base filter configuration from the `uid` entry (which `EntityViewsData::getViewsData()` already sets to the `user_name` plugin at line 354 for all uid-keyed entities), then overrides the filter plugin `id` to `'user_name'`, sets `real field` to `'uid'` so the underlying SQL still operates on that column, and provides the label "Authored by" and help text "The username of the content author." to match the existing Content view.

During the issue, an upgrade path was developed to migrate existing views from the raw `uid` filter to the new `user_name` plugin via `ViewsConfigUpdater` and a `views_post_update` hook. That approach was dropped after lendude identified that the `ViewsConfigUpdater::updateAll()` call would process far more views than intended (any `uid` filter, not just media ones), that operator mappings could silently destroy `between`-style filter configurations, and that the base-table guard was bypassed in the `updateAll` code path. The team chose the safer approach of adding a new dedicated filter entry and leaving existing views untouched. The previously shipped `media_post_update_media_author_views_filter_update` hook was removed and tombstoned in `media_removed_post_updates()` targeting 12.0.0.

Test coverage was added as `testMediaOverviewAuthorFilter()` in `core/modules/media/tests/src/Functional/MediaOverviewPageTest.php`. The test navigates to the Views filter-picker UI, selects `media_field_data.user_name`, exposes it, saves the view, then submits the filter by username and asserts that only media owned by that user appears in the results.

### Collaboration

lendude's two rounds of detailed review (#39 and #75) were the turning point for the upgrade-path approach. In the first round, lendude identified that hardcoding the plugin ID in the test made it pass regardless of the `MediaViewsData` change, flagged that the post_update hook should use `ViewsConfigUpdater` for broader coverage, and noted a missing base-table guard in `updateAll`. After those were addressed, lendude found in the second review that the `updateAll` call still bypassed the base-table guard and raised the BC risk of silently converting operators like `between` to `in`. That analysis directly led to abandoning the upgrade path in favor of the new virtual-field design, which carries no risk to existing views.
