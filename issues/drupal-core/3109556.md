## Summary

Drupal core's JavaScript build tooling has been upgraded from Yarn 1 ("Yarn Classic") to Yarn 4 for the 11.x development branch. Yarn 1 has been in security-only maintenance mode since February 2020. This change affects contributors and developers who compile Drupal core JavaScript or CSS assets: they must now have Node.js 18 or later installed and run `corepack enable` before using `yarn` in the `core/` directory. Site owners who do not build frontend assets are unaffected.

## Impact

Affects site owners.

- **Requirements:** Node.js 18+ and Corepack (included with Node.js) are now required to run Yarn build tasks for Drupal 11.x core development.

## Technical details

Drupal core ships a `core/package.json` that drives all JS and CSS build tasks via Yarn. Yarn 1 entered maintenance-only mode in 2020 and the upgrade was deferred through Drupal 9 and 10 before landing in 11.x.

**Two approaches were explored:** The first committed the Yarn CLI binary (`yarn-x.x.x.cjs`, roughly 2.6 MB) directly into `core/.yarn/releases/`. The second used Corepack, a Node.js built-in (available since Node.js 16.10) that reads the `packageManager` field in `package.json` and downloads the matching Yarn release automatically. The team chose Corepack because it avoids checking a large binary into the repository.

**Changes in commit `ad10de38`:**

- `core/package.json`: `engines.yarn` minimum raised from `>= 1.6` to `>= 4.1.1`; `packageManager` field updated from `yarn@1.22.22` to `yarn@4.1.1`. Spellcheck script calls that used the deprecated `-s` (silent) flag had that flag removed, since Yarn 4 dropped it.
- `core/.yarnrc.yml` (new file): sets `nodeLinker: node-modules` to continue using the traditional `node_modules` directory layout instead of Yarn's Plug'n'Play mode, which would have required broader ecosystem changes.
- `core/.gitignore`: added `.yarn/*` exclusion per Yarn's own `.gitignore` recommendations, since Corepack manages the binary outside the repo.
- `core/.cspell.json`: added `.yarnrc.yml` and `.yarn/*` to the spellcheck exclusion list.
- `core/misc/cspell/drupal-dictionary.txt`: added "corepack" to the custom CSpell dictionary.
- `.gitlab-ci.yml` and `.gitlab-ci/pipeline.yml`: all Yarn invocations replaced `--cwd=./core` with an explicit `cd core` (the `--cwd` flag was removed in Yarn 4), and each job gained a `corepack enable` step before calling `yarn`.
- `core/scripts/dev/commit-code-check.sh`: the old `yarn check -s` dependency verification block was replaced with `yarn --version && yarn`; all `yarn run -s` calls changed to plain `yarn run`.
- `core/yarn.lock` was regenerated in Yarn 4 format.

A related issue, [#3428122](https://www.drupal.org/node/3428122), backported the `packageManager` key to the 10.x branches so that developers switching between branches do not have Corepack silently rewrite `package.json`.

## Upgrade

Core contributors and developers who build Drupal core JS/CSS assets need to update their local setup:

```bash
# Requires Node.js 18 or later
# Enable Corepack (ships with Node.js, manages Yarn versions)
corepack enable

# Then install dependencies as before
cd core
yarn install

# Build tasks work as before
yarn build:css
yarn build:ckeditor5
```

If you switch between an 11.x branch (Yarn 4) and a 10.x branch (Yarn 1), Corepack will automatically switch Yarn versions based on the `packageManager` field in each branch's `core/package.json`. Review the [change record on Yarn 4](https://www.drupal.org/node/3428571) for troubleshooting tips.
