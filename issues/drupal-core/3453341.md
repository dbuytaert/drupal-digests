## Summary

Drupal 11's functional test HTML output logger is now bootstrapped through `phpunit.xml` as a proper PHPUnit extension, rather than being initialized from `bootstrap.php` via environment variables. Developers with custom `phpunit.xml` files need to add the extension block shown below; otherwise HTML debug output from browser tests will not be generated. The default verbosity is now `true` (all links printed), which is the better default for local development.

## Impact

Affects site owners.

- **Config change:** Custom `phpunit.xml` files must add an `<extensions>` block with `HtmlOutputLogger` and its `outputDirectory` / `verbose` parameters; the old `BROWSERTEST_OUTPUT_DIRECTORY` and `BROWSERTEST_OUTPUT_VERBOSE` `<env>` entries in phpunit.xml are removed from the dist file.

## Technical details

Before this change, `HtmlOutputLogger` was initialized as a singleton from `core/tests/bootstrap.php` via a static `init()` method, controlled entirely by the `BROWSERTEST_OUTPUT_DIRECTORY` and `BROWSERTEST_OUTPUT_VERBOSE` environment variables. This approach was not the canonical PHPUnit extension bootstrap mechanism and broke parallel test runners such as paratest because the singleton state was not compatible with that execution model.

The fix moves bootstrap to `core/phpunit.xml.dist`'s new `<extensions>` block, making `HtmlOutputLogger` implement the PHPUnit `Extension` interface and providing a `bootstrap(Configuration $configuration, Facade $facade, ParameterCollection $parameters)` method. Configuration is now read from two `<parameter>` elements in the XML: `outputDirectory` (defaults to `sites/simpletest/browser_output`) and `verbose` (defaults to `true`). The `BROWSERTEST_OUTPUT_DIRECTORY` and `BROWSERTEST_OUTPUT_VERBOSE` environment variables remain supported as overrides, with `BROWSERTEST_OUTPUT_DIRECTORY=''` providing a way to disable HTML output entirely.

The singleton pattern (`private static ?self $instance`) and `isEnabled()` / `init()` static methods are removed. State is now tracked by a private instance property `$enabled`. In `BrowserHtmlDebugTrait::initBrowserOutputFile()`, the enabled check is changed from calling `HtmlOutputLogger::isEnabled()` to checking whether the `BROWSERTEST_OUTPUT_FILE` environment variable is set, which is the env var that the extension sets at test-runner-start time. The `HtmlOutputLogger` import is removed from `bootstrap.php` entirely.

The `FunctionalTestDebugHtmlOutputTest` integration test is expanded to cover: default XML settings, verbose overridden by XML, verbose overridden by the environment variable, wrong output directory via XML, wrong output directory via env, disabling via `BROWSERTEST_OUTPUT_DIRECTORY=''`, and a missing `outputDirectory` parameter. A private helper `getAlteredPhpunitXmlConfigurationFile()` is added to create temporary modified copies of `phpunit.xml.dist` for these test scenarios. alexpott confirmed in the thread that applying this change to a downstream project fixed paratest compatibility with Drupal 11 (see [#3452524](https://www.drupal.org/node/3452524)).

## Upgrade

Anyone maintaining a custom `phpunit.xml` (copied from `phpunit.xml.dist`) must add the extensions block. Without it, functional test HTML output will not be generated.

```xml
<extensions>
  <!-- Functional tests HTML output logging. -->
  <bootstrap class="Drupal\TestTools\Extension\HtmlLogging\HtmlOutputLogger">
    <parameter name="outputDirectory" value="sites/simpletest/browser_output"/>
    <parameter name="verbose" value="true"/>
  </bootstrap>
</extensions>
```

The `BROWSERTEST_OUTPUT_DIRECTORY` and `BROWSERTEST_OUTPUT_VERBOSE` environment variables continue to work as overrides of the XML parameters. Setting `BROWSERTEST_OUTPUT_DIRECTORY` to an empty string disables HTML output entirely.
