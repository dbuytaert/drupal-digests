## Summary

Five procedural functions remaining in `language.module` have been deprecated as part of the ongoing effort to move Drupal core away from procedural code toward object-oriented hook classes and services. Site owners and module authors who call any of these functions directly will begin receiving deprecation warnings in Drupal 11.4 and should update their code to use the replacement APIs before Drupal 12 or 13, depending on the specific function.

## Impact

Affects module maintainers.

- **Deprecation:** Five procedural `language.module` functions deprecated in 11.4.0: `language_process_language_select` (removed in 12.0.0), `language_configuration_element_submit`, `language_get_default_langcode`, `language_negotiation_url_prefixes_update`, and `language_get_browser_drupal_langcode_mappings` (all removed in 13.0.0); `DefaultLanguageItem::getDefaultLangcode()` also deprecated (removed in 13.0.0); `LanguageMapper` constructor without `ConfigFactoryInterface` deprecated (required in 12.0.0).
- **New API:** New `LanguageConfiguration::submit()` static method on the language configuration form element class; new `LanguageHooks::processLanguageSelect()` instance method.

## Technical details

The issue targeted five procedural functions in `language.module` that had not yet been converted:

1. **`language_process_language_select()`** — deprecated in `drupal:11.4.0`, removed in `drupal:12.0.0`. The logic moved into `LanguageHooks::processLanguageSelect()` as an instance method. The `elementInfoAlter` hook method in `LanguageHooks` now references the callback as `static::class . ':processLanguageSelect'` and also switches the adjacent `Select::processSelect` and `RenderElementBase::processAjaxForm` callbacks from string literals to class-constant form.

2. **`language_configuration_element_submit()`** — deprecated in `drupal:11.4.0`, removed in `drupal:13.0.0`. The logic moved into a new static `LanguageConfiguration::submit()` method on the `Drupal\language\Element\LanguageConfiguration` form element class. All three callers in `ContentEntityForm`, `NodeForm`, and `LanguageConfigurationElement` test form were updated to reference `LanguageConfiguration::class . '::submit'` instead of the function name string.

3. **`language_get_default_langcode()`** — deprecated in `drupal:11.4.0`, removed in `drupal:13.0.0`, with no public replacement. The logic moved into `DefaultLanguageItem::getDefaultLanguageCode()` as a `protected` method. The existing public `DefaultLanguageItem::getDefaultLangcode(EntityInterface $entity)` method is also now deprecated (removed in `drupal:13.0.0`) and simply delegates to the new protected method.

4. **`language_negotiation_url_prefixes_update()`** — deprecated in `drupal:11.4.0`, removed in `drupal:13.0.0`, with no public replacement. The logic moved into `ConfigurableLanguage::updateUrlPrefixes()` as a `protected` method, called from `ConfigurableLanguage::postSave()`. A comment in `LanguageNegotiationUrlSubscriber::onConfigSave()` was updated to reference the new method name.

5. **`language_get_browser_drupal_langcode_mappings()`** — deprecated in `drupal:11.4.0`, removed in `drupal:13.0.0`, with no public replacement. The sole non-test caller, `LanguageMapper::getMappings()`, now reads `language.mappings` configuration directly via a newly injected `ConfigFactoryInterface`. The `LanguageMapper` constructor gains an optional `?ConfigFactoryInterface $config_factory` parameter (with a `@trigger_error` deprecation if omitted, required in `drupal:12.0.0`).

Tests were refactored alongside: the `testDefaultLangcode` test no longer calls the deprecated function; instead it creates an `EntityTest` entity and reads `langcode->value` through `DefaultLanguageItem`. A separate test form class (`LanguageConfigurationElementTest`) and its route were removed as they existed only to test the deprecated function via a browser request; that scenario is now covered by a direct entity assertion.

## Upgrade

If you call any of these functions directly, replace them as follows:

```php
// language_process_language_select() — removed in drupal:12.0.0
// Before:
'language_process_language_select'
// After (inject or retrieve the LanguageHooks service):
\Drupal\language\Hook\LanguageHooks::class . ':processLanguageSelect'

// language_configuration_element_submit() — removed in drupal:13.0.0
// Before:
$form['#submit'][] = 'language_configuration_element_submit';
// After:
$form['#submit'][] = \Drupal\language\Element\LanguageConfiguration::class . '::submit';

// language_get_default_langcode() — removed in drupal:13.0.0
// No public replacement; read the language configuration directly:
$config = \Drupal\language\Entity\ContentLanguageSettings::loadByEntityTypeBundle($entity_type, $bundle);
$langcode = $config->getDefaultLangcode();

// language_negotiation_url_prefixes_update() — removed in drupal:13.0.0
// No public replacement; the update now happens internally in
// ConfigurableLanguage::postSave().

// language_get_browser_drupal_langcode_mappings() — removed in drupal:13.0.0
// No public replacement; read config directly:
$map = \Drupal::config('language.mappings')->get('map');
```

If you extend `LanguageMapper` and rely on its constructor signature, add `ConfigFactoryInterface` as the fourth argument:

```php
// Before (triggers deprecation in 11.4, breaks in 12.0):
new LanguageMapper($persistent_cache, $memory_cache, $moduleHandler);

// After:
new LanguageMapper($persistent_cache, $memory_cache, $moduleHandler, $configFactory);
```

If you call `DefaultLanguageItem::getDefaultLangcode(EntityInterface $entity)` directly, note it is deprecated in `drupal:11.4.0` and removed in `drupal:13.0.0` with no public replacement.

### Collaboration

A discussion arose between nicxvan, berdir, and penyaskito (the new `language.module` maintainer) over two design questions. Berdir raised whether new methods should be consistently targeted at `drupal:13.0.0` for removal. Penyaskito confirmed that the `drupal:12.0.0` deprecation train had already passed, so only functions already slated for 12.0.0 removal kept that target. Penyaskito also suggested making the new helper methods `private` to retain more refactoring freedom before Drupal 13. nicxvan pushed back, citing the BC policy that `protected` methods on plugins are not considered API and that `private` should be reserved for experimental features. Penyaskito agreed that this was not the right issue to change that policy, gave a +1, and the `protected` approach was kept.
