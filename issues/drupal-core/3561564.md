## Summary

Drupal's block system previously assumed that a block plugin would implement at most one of three special interfaces: `MainContentBlockPluginInterface`, `TitleBlockPluginInterface`, or `MessagesBlockPluginInterface`. A custom or contrib block that implemented more than one of these interfaces would only receive the first matching role because the checks used `elseif` chains. This change removes that restriction, allowing a single block to fulfill all three roles at once. The fix is purely permissive, with no impact on existing sites or existing blocks.

## Impact

Affects module maintainers.

- **New API:** A block plugin may now implement any combination of `MainContentBlockPluginInterface`, `TitleBlockPluginInterface`, and `MessagesBlockPluginInterface`; all matching setters are called rather than only the first.

## Technical details

The affected logic lives in `BlockPageVariant::build()` in `core/modules/block/src/Plugin/DisplayVariant/BlockPageVariant.php`. During page assembly, the method iterates over all placed blocks and calls interface-specific setters: `setMainContent()` for `MainContentBlockPluginInterface`, `setTitle()` for `TitleBlockPluginInterface`, and sets a flag for `MessagesBlockPluginInterface`. These three checks were written as a single `if`/`elseif`/`elseif` chain, so only the first matching interface on any given block plugin would be acted upon. The fix changes the two `elseif` keywords to `if`, making each check independent. A new test block class, `TestPageTitleMainContentAndMessagesBlock` (added to the `block_test` module in `core/modules/block/tests/modules/block_test/src/Plugin/Block/`), implements all three interfaces simultaneously. The corresponding functional test `testTitleContentMessagesSingleBlock()` in `core/modules/block/tests/src/Functional/BlockTest.php` asserts that `setTitle()`, `setMainContent()`, and message rendering are all invoked when the block is placed.
