## Summary

More than a dozen places in Drupal core that previously used `merge()` for write-or-update operations now use `upsert()` with composite keys. The practical effect is that each of these operations no longer issues a preliminary `SELECT` to decide whether to `INSERT` or `UPDATE`; instead a single atomic `INSERT ... ON DUPLICATE KEY UPDATE` (or equivalent) is used. Affected paths include session writes, key-value store operations, taxonomy index updates, comment statistics, migration map saves, locale storage, search word weights, shortcut set assignment, inline block usage, and user data storage. Session writes during login drop from 16 to 15 queries per the updated performance tests.

## Impact

No upgrade or configuration changes required.

## Technical details

The `merge()` method in Drupal's database layer performs a preliminary `SELECT` before deciding between `INSERT` and `UPDATE`. The `upsert()` method, whose composite key support was established in [#2547493](https://www.drupal.org/node/2547493), bypasses that SELECT entirely with a native single-statement upsert. This change adopts `upsert()` across the following locations: `SessionHandler::doWrite()` (sessions table), `FileStorage::doWrite()` (config), `KeyValueStore\DatabaseStorage::doSet()` and `doSetMultiple()`, expirable key-value `doSetWithExpire()`, `CommentStatisticsStorage::update()` (comment_entity_statistics), `InlineBlockUsage::addUsage()` (inline_block_usage), `LocaleFileStorage::save()` (locale_file), `LocaleStringDatabaseStorage::dbStringUpdate()` (locales_source / locales_target), `SqlWithCentralizedMessageStorage::saveIdMapping()` (migration map table), `SearchIndexStorage::updateWordWeights()` (search_total), `ShortcutSetStorage::assignUser()` (shortcut_set_users), `taxonomy_build_node_index()` (taxonomy_index), and `UserDataStorage::set()` (users_data).

`KeyValueStore\DatabaseStorage::setMultiple()` received the most substantial rewrite. Previously it delegated to `parent::setMultiple()` inside a transaction, which internally looped over individual `set()` calls. The new implementation builds a single multi-row `upsert` query using `->fields()` for the first row and `->values()` for each additional row, issuing one statement regardless of how many pairs are written. An early-return guard for an empty `$data` array was also added, since `upsert()` does not tolerate an empty value set.

One incidental correctness fix is included: `LocaleFileStorage::save()` now passes `$currentImport->hash ?? ''` because `upsert()->fields()` requires a concrete value for every column while the old `merge()->fields()` silently omitted a `null` hash field.
