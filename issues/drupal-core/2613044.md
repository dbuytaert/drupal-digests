# #2613044: Requests are pushed onto the request stack twice, popped once

## Summary

Drupal's request handling incorrectly pushed every request onto the request stack twice but only popped it once. This caused duplicate entries during the entire request lifecycle and allowed the stack to grow unbounded across main requests and sub-requests. The fix ensures Drupal manages the request stack lifecycle independently from Symfony's kernel by popping main requests during termination and sub-requests immediately after handling, while letting Symfony use its own internal stack. The request stack is now correctly empty after kernel termination, preventing cache poisoning and state leakage issues.

## Impact

No upgrade or configuration changes required.

## Technical details

Symfony's `HttpKernel` unconditionally pushes and pops requests. Drupal's `DrupalKernel::preHandle()` also pushed requests early (needed for service initialization during boot), but nothing popped this second push. The double-push caused problems: duplicate stack entries during request processing, sub-requests compounding the issue, and the main request persisting after termination.

The solution decouples Drupal's request stack from Symfony's. `http_kernel.basic` now receives `null` for the request stack parameter, causing `HttpKernel` to create its own internal stack. Drupal maintains its own stack lifecycle: `DrupalKernel::preHandle()` still pushes early, `KernelPreHandle::handle()` now pops sub-requests in a finally block after handling, and `DrupalKernel::terminate()` pops the main request at the very end after destructible services run. This ensures the request remains available throughout Drupal's entire lifecycle including termination events, while the stack is properly cleared afterward. Test coverage validates correct stack depth during main requests, sub-requests, and after termination.

## Lessons learned

- When integrating with framework components that manage stack-based request lifecycles, different layers may have conflicting expectations about push/pop symmetry. If your application needs resources to remain available longer than the framework's default lifecycle (e.g., during termination handlers), take explicit ownership of the stack lifecycle rather than relying on the framework's automatic cleanup. Use wrapper objects to prevent the framework from modifying shared state while still allowing it to read that state. *(via alexpott, cpj)*
- When handling main requests and subrequests in a stack-based architecture, ensure each request type has a clearly scoped push/pop lifecycle. Main requests may need to persist longer than subrequests (e.g., through termination events), while subrequests should be popped immediately after handling. Asymmetric push/pop patterns are acceptable when they accurately reflect the application's requirements, even if they differ from framework conventions. *(via alexpott, cpj, dawehner)*
