## Summary

This bug fix corrects a long-standing problem where every request was pushed onto Drupal's `RequestStack` twice but only popped once. The duplicate entry could cause cache poisoning: after a sub-request, the route cache context could still point to the sub-request's route, causing a wrong cached response to be served to subsequent visitors. The fix takes full ownership of the request stack lifecycle within Drupal's own code, so the stack now contains exactly one entry per active request and is empty after `terminate()` completes.

## Impact

Affects module maintainers.

- **API change:** `KernelPreHandle::__construct()` now requires a third `RequestStack $requestStack` argument

## Technical details

Drupal's `DrupalKernel::preHandle()` pushes the request onto `request_stack` early so services like `SessionManager` can access it during middleware execution. Symfony's `HttpKernel::handle()` then pushed the same request a second time unconditionally, and only popped its own push in a `finally` block. Drupal never popped the first push, so the stack always held one extra entry per request and was never cleared.

The fix passes `null` as the `request_stack` argument to the `http_kernel.basic` service definition. When `HttpKernel` receives `null` it creates its own internal `RequestStack` and pushes/pops to that, leaving Drupal's `request_stack` service untouched. Drupal then manages the full push/pop lifecycle itself:

- **Push**: `DrupalKernel::preHandle()` pushes every request (unchanged).
- **Pop sub-requests**: `KernelPreHandle::handle()` now receives `@request_stack` as a third constructor argument and pops in a `finally` block after `HttpKernel::handle()` returns, but only when `$type !== MAIN_REQUEST`.
- **Pop main requests**: `DrupalKernel::terminate()` pops after all destructible services have run. The main request is deliberately left on the stack until then because many terminate-time subscribers (JSON:API's `ResourceObjectNormalizationCacher`, `UserRequestSubscriber`, page cache response policy checks) still need access to the current request.

The result is that after `terminate()` the stack is empty, and during a sub-request the stack depth grows by one and returns to its previous depth when the sub-request is done.

`KernelTestBase::tearDown()` was updated: the session cleanup block now guards on `getCurrentRequest() !== NULL` instead of `$this->container` being set, since the stack is now correctly empty after `terminate()`.

`TaxonomyDefaultArgumentTest::initViewWithRequest()` was changed to call `handle($request)` as a main request instead of `HttpKernelInterface::SUB_REQUEST`, because the test was relying on the old behavior where the request stayed on the stack after a sub-request.

A new kernel test `DrupalKernelRequestStackTest` covers the stack depth and identity of main, current, and parent requests across main requests, sub-requests, nested sub-requests, 404 forwarding, and post-terminate state.

## Upgrade

If you extend or instantiate `KernelPreHandle` directly, add the new `RequestStack` argument:

```php
// Before
new KernelPreHandle($http_kernel, $drupal_kernel);

// After
new KernelPreHandle($http_kernel, $drupal_kernel, $request_stack);
```

If your code relies on a request remaining on `request_stack` after a sub-request completes but before `terminate()`, it will now find only the main request on the stack. Any code accessing the request stack after `DrupalKernel::terminate()` will find an empty stack.

### Collaboration

This issue was open for over ten years (filed November 2015, fixed March 2026) with 150 comments and 21 contributors. Early attempts by cpj explored several approaches: a `RequestStackWrapper` that intercepted `push()`/`pop()` calls from `HttpKernel`, a separate `symfony_request_stack` service, and a `StackMiddlewareSubscriber` to pop the request in a low-priority `KernelEvents::TERMINATE` handler. All of these either introduced new classes that were hard to justify, failed tests, or left the request unavailable during termination. alexpott joined late in the issue and identified that passing `null` as the request stack to `http_kernel.basic` was sufficient: Symfony's `HttpKernel` already handles a `null` stack by creating its own internal one, making a wrapper class unnecessary. larowlan reported a real-world cache poisoning bug on a client project that gave the issue renewed urgency.
