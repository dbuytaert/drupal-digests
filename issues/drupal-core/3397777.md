## Summary

This was a support request about the confusing failure that occurs when running `composer require -W drupal/core-recommended:^10` to upgrade from Drupal 9 to Drupal 10 on a site that also has Drush installed. The command fails with an unhelpful error pointing to `psr/log` rather than identifying Drush as the source of the conflict. After discussion, the issue was closed with guidance to seek support through Drupal Slack or the forums, as no Drupal core code change was needed.

## Impact

No upgrade or configuration changes required.

## Technical details

The root cause is a Composer constraint resolution limitation. The `-W` flag (`--with-all-dependencies`) tells Composer to update a named package and all packages that the named package depends on. Because `drush/drush` is not a dependency of `drupal/core-recommended` (the dependency relationship is the reverse, or indirect), Drush and its own dependencies are excluded from the update resolution. Drush 11 shares transitive dependencies with Drupal 10 (such as `psr/log ^3` and various `symfony/*` components at new major versions), but those shared packages are locked by Drush's existing constraints and cannot be updated within the narrow `-W` scope. Composer therefore reports an unsatisfiable requirement citing `psr/log` without naming Drush, which is misleading. Two workarounds were discussed: including `drush/drush` explicitly in the `composer require -W` invocation (e.g. `composer require -W drupal/core-recommended:^10 drush/drush`), which widens the resolution scope to include Drush's dependencies; or editing `composer.json` manually (or with `--no-update`) and then running `composer update` with no package argument, which allows Composer to resolve the entire dependency graph at once. The broader discussion noted that the official upgrade documentation is difficult to discover, and that the `composer update` (no argument) approach is the more robust recommendation for sites with many contributed modules.
