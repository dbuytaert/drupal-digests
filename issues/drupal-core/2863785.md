## Summary

During config synchronization, Drupal's `StorageComparer` was throwing a PHP "Undefined index: uuid" notice whenever a config entity in the active (target) store was missing its `uuid` key. This could happen with old configs migrated from early Drupal 8, contrib module settings files that omit a UUID, or manually created database entries. The fix suppresses the notice and, as a side effect, treats the missing UUID as a mismatch, causing the config system to recreate the item with a proper UUID on the next import.

## Impact

No upgrade or configuration changes required.

## Technical details

The problem was in `StorageComparer::addChangelistUpdate()` in `core/lib/Drupal/Core/Config/StorageComparer.php`. The UUID comparison read `$target_data['uuid']` directly, which triggers a PHP notice when the key is absent. The fix replaces that bare access with a null-coalescing expression: `($target_data['uuid'] ?? NULL)`. Because no real UUID string equals `null`, a missing UUID in the target store is treated as a UUID mismatch. The existing branch for UUID mismatches schedules a recreate (a paired delete and create operation), which self-heals the target store by replacing the UUID-less entry with a properly-generated one on import. A new unit test `testChangelistUuidMissMatch()` in `StorageComparerTest` covers both the missing-UUID case (triggering recreate) and the changed-UUID case (also triggering recreate). Note: alexpott reviewed a draft version that mutated `$target_data['uuid']` in-place and objected to modifying the target data; `neclimdul` reworked the approach to the null-coalesce-only form that landed.
