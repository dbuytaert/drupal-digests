## Summary

The temporary workaround that let plugin attribute discovery handle missing traits without a fatal error has been removed from `AttributeClassDiscovery`. The workaround was introduced in [#3502913](https://www.drupal.org/node/3502913) to paper over a PHP limitation where loading a class that uses a non-existent trait caused an unrecoverable fatal error. PHP 8.5 fixes this by making the same situation throw a catchable `\Error`, so the workaround is no longer needed. The practical effect for most sites is nothing: plugins whose classes depend on uninstalled modules continue to be silently skipped during discovery, just through simpler code.

## Impact

Affects site owners.

- **Requirements:** Relies on PHP 8.5 behavior where loading a class with a missing trait throws a catchable `\Error`; this removal is only safe because Drupal `main` now targets PHP 8.5.

## Technical details

The removed workaround registered `MissingClassDetectionClassLoader` as an `spl_autoload` handler during every `getDefinitions()` call. When PHP tried to load a class with a missing trait, this handler fired last, recorded the missing trait in `$missingTraits`, and aliased it to `StubTrait` so reflection could proceed. `AttributeClassDiscovery` tracked affected classes in a static `$skipClasses` array and re-checked them on subsequent discovery passes, comparing the missing trait's two-level namespace against registered plugin namespaces to decide whether to allow or skip the class. Plugins with missing traits were also excluded from the file cache so they would be re-scanned on every request.

With PHP 8.5, `class_exists()` on a class that uses a non-existent trait throws a catchable `\Error`. The diff deletes `MissingClassDetectionClassLoader` and `StubTrait` entirely, removes the `spl_autoload_register` / `spl_autoload_unregister` calls, drops the `$skipClasses` static property and all associated re-check logic, and reduces the error-handling block inside `getDefinitions()` to a bare `catch (\Error) { continue; }`. File cache writes for successfully resolved plugins are now unconditional.

The one visible behavioral difference, captured in the updated `AttributeClassDiscoveryCachedTest::testGetDefinitionsMissingTrait()`, is that if discovery runs twice in a single request and the missing trait's module is installed between the two passes, the plugin is no longer re-discovered in the second pass. Previously the stub-alias meant PHP would not throw again, so the class appeared valid. Now PHP 8.5 throws an `\Error` on every `class_exists()` call for that class until the next full discovery cycle. The thread also considered restoring a `preg_match('/(Class|Interface|Trait) .* not found$/', $e->getMessage())` guard to prevent silently swallowing unexpected `\Error` exceptions, but contributors decided against it as fragile given how unlikely such exceptions are in practice; the tracking issue [#3520811](https://www.drupal.org/node/3520811) is expected to be closed as won't-do.

## Upgrade

Both removed classes were `@internal` and not part of Drupal's public API. If you referenced them despite that marker, remove those references:

```php
// Remove these imports and any usage:
use Drupal\Component\Discovery\MissingClassDetectionClassLoader;
use Drupal\Component\Discovery\StubTrait;
```

If your code installs a module and then calls `getDefinitions()` again in the same request expecting to find a previously-missing-trait plugin, that plugin will no longer appear until the next request's discovery cycle. Move such code to a fresh request or clear the plugin cache after module installation.
