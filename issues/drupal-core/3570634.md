## Summary

The fallback classloader that worked around PHP's inability to catch fatal errors from missing traits has been removed from the plugin discovery system. This simplifies `AttributeClassDiscovery` significantly. The change is possible because PHP 8.5 converts missing-trait loading into a catchable `\Error` instead of an unrecoverable fatal error. Plugins that use traits from uninstalled optional modules are now always skipped during discovery rather than sometimes being surfaced through a stub-trait aliasing workaround.

## Impact

Affects module maintainers and site owners.

- **Requirements:** Drupal main now relies on PHP 8.5 behavior where missing-trait loading throws a catchable \Error rather than a fatal error, enabling removal of the classloader workaround.
- **API change:** `MissingClassDetectionClassLoader` and `StubTrait` in `Drupal\Component\Discovery` have been deleted; `AttributeClassDiscovery::$skipClasses` static property has been removed.

## Technical details

The original workaround, added in [#3502913](https://www.drupal.org/node/3502913), registered a custom `spl_autoload` handler (`MissingClassDetectionClassLoader`) during each call to `getDefinitions()`. When PHP tried to load a class whose trait was missing, the handler fired last, recorded the missing trait, and aliased it to a `StubTrait` so that reflection could proceed without a fatal error. `AttributeClassDiscovery` then tracked these affected classes in a static `$skipClasses` array and rechecked them on subsequent discovery passes, skipping them if the missing trait's two-level namespace was still absent from the registered plugin namespaces. Classes with missing traits were also excluded from the file cache.

With PHP 8.5 making missing-trait loading throw a catchable `\Error` (see https://github.com/php/php-src/issues/17959), all of that machinery is unnecessary. The diff removes `MissingClassDetectionClassLoader` and `StubTrait` entirely, drops the `spl_autoload_register` / `spl_autoload_unregister` calls, removes the `$skipClasses` static property and all its associated re-check logic, and simplifies the error handling to a bare `catch (\Error) { continue; }`. File cache writes are now unconditional for successfully parsed plugins.

One behavioral difference is visible in the tests: previously, when the namespace of a missing trait was added to the registered plugin namespaces between discovery passes, the plugin would be re-discovered (because the stub alias had been set up). Now that no stub aliasing occurs, PHP 8.5 throws an `\Error` on `class_exists()` for any class using a non-existent trait, and that plugin is always skipped until the module providing the trait is actually installed.

## Upgrade

Both removed classes are `@internal` and were never part of Drupal's public API. If you referenced them directly (e.g. in tests), remove those references:

```php
// Remove these imports and any usage:
use Drupal\Component\Discovery\MissingClassDetectionClassLoader;
// StubTrait was also removed from Drupal\Component\Discovery\StubTrait
```

No changes are needed for standard plugin definitions or plugin discovery consumers. The only behavioral change is that a plugin whose class uses a trait from an uninstalled module is now always skipped during discovery rather than conditionally surfaced via stub aliasing.

### Collaboration

catch, who wrote the original `MissingClassDetectionClassLoader` workaround in [#3502913](https://www.drupal.org/node/3502913), noted in comment [#16577237](https://www.drupal.org/node/16577237): "I think I'm allowed to call this a horrible hack given I introduced it in the first place, good to see the back of it." The thread also surfaced a live design question: godotislate pointed out in comment [#16451771](https://www.drupal.org/node/16451771) that removing the classloader without restoring a regex guard on the `\Error` message meant any unexpected `\Error` from `class_exists()` would be silently swallowed. catch and godotislate discussed whether `preg_match('/(Class|Interface|Trait) .* not found$/', $e->getMessage())` should be added back; they ultimately decided against it, reasoning that such errors are extremely unlikely in practice and that matching on exception message strings is brittle.
