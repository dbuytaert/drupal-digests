## Summary

The `package_manager` module (which underlies Automatic Updates) is alpha-stability and experimental. Normally, alpha experimental modules are stripped from tagged release branches, but that creates a catch-22: a module designed to update sites between releases needs to exist in those releases. The fix keeps `package_manager` in tagged releases while hiding it from the UI and blocking its installation unless a developer explicitly opts in via a `settings.php` flag. This means production sites cannot accidentally enable it, while developers who want to test early adoption can do so by setting one line in their configuration.

## Impact

Affects site owners.

- **Config change:** New `$settings['testing_package_manager']` key in `settings.php` required to install `package_manager`; absence blocks installation with a `REQUIREMENT_ERROR`.

## Technical details

The catch-22 is explained in comment [#15847048](https://www.drupal.org/node/15847048) by catch: removing `package_manager` from tagged release branches (the normal policy for alpha experimental modules) would make it impossible to test or use the module's core purpose of updating a site from one release to the next. The workaround chosen here, pending the completion of [#1537198](https://www.drupal.org/node/1537198) (a formal production/development toggle), is a lightweight arbitrary `$settings` key.

Two changes enforce the gate:

1. `package_manager.info.yml` gains `hidden: true`, removing the module from the Extend UI listing entirely.

2. `package_manager_requirements()` in `package_manager.install` now checks `Settings::get('testing_package_manager', FALSE)` at the top of the function. If the setting is not exactly `TRUE`, the function immediately returns a `REQUIREMENT_ERROR` with the message "Package Manager is available for early testing. To install the module set the value of 'testing_package_manager' to TRUE in your settings.php file." This blocks module installation through every code path that honours `hook_requirements` (UI, Drush, etc.).

Test infrastructure is updated in three places so the existing functional test suite can still install the module. A new abstract `PackageManagerTestBase` class (extending `BrowserTestBase`) is introduced at `core/modules/package_manager/tests/src/Functional/PackageManagerTestBase.php`. Its `setUp()` calls `$this->writeSettings()` to inject `testing_package_manager = TRUE` before each test. `ComposerRequirementTest` and `FailureMarkerRequirementTest` are re-parented from `BrowserTestBase` to `PackageManagerTestBase`, and `TemplateProjectTestBase::installQuickStart()` adds the same setting to the test site's `settings.php` via a heredoc. The `defaultTheme = 'stark'` declaration is also deduplicated into the new base class.

## Upgrade

If you have `package_manager` installed or plan to install it in a development environment, add the following line to your `settings.php`:

```php
$settings['testing_package_manager'] = TRUE;
```

Without this setting, attempting to install `package_manager` will fail with a `REQUIREMENT_ERROR`. The module also no longer appears in the Extend UI, so programmatic or Drush-based installation is required alongside this setting.
