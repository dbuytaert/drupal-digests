## Summary

Kernel tests in Drupal can now make HTTP requests and use Mink assertions via a new `HttpKernelUiHelperTrait`, which is included in `KernelTestBase` by default. This lets developers write tests that call `drupalGet()`, `clickLink()`, and `assertSession()` without spinning up a full browser stack or running the Drupal installer, making many tests significantly faster. A companion `PageCachePolicyTrait` is provided for tests that also need to exercise page-cache or dynamic-page-cache behavior. As a concrete example, `NodeLinksTest` was converted from a functional browser test to a kernel test, dropping its run time from roughly 36 seconds to 10 seconds.

## Impact

Affects module maintainers.

- **New API:** New `HttpKernelUiHelperTrait` (included in `KernelTestBase`) provides `drupalGet()`, `clickLink()`, `getSession()`, and `assertSession()` for kernel tests; new `PageCachePolicyTrait` enables page-cache and dynamic-page-cache testing in kernel tests.

## Technical details

Kernel tests previously had no way to issue HTTP requests and assert against rendered output, forcing those tests to extend `BrowserTestBase` with all its installer and browser overhead. The fix introduces `HttpKernelUiHelperTrait` (`core/tests/Drupal/Tests/HttpKernelUiHelperTrait.php`), which is now used by `KernelTestBase` alongside the existing `BrowserHtmlDebugTrait`.

The implementation uses Symfony's `HttpKernelBrowser` (a BrowserKit client that speaks directly to a `HttpKernelInterface`) wrapped in Mink's `BrowserKitDriver` to create a `Behat\Mink\Session`. No real HTTP server or external browser process is involved. Mink is lazily initialized in `getSession()` on the first call, so kernel tests that never make a request pay no initialization cost.

Key methods on the trait:
- `drupalGet($path)` ensures the path starts with `/` (working around Symfony's relative-path behavior for multi-segment paths), visits the URL, and optionally writes HTML debug output via `BrowserHtmlDebugTrait`.
- `clickLink($label, $index)` uses the Mink page to find the link and delegates to `drupalGet()` so debug output is produced consistently.
- `getSession($name)` is the central lazy initializer; it also calls `initBrowserOutputFile()`.
- `assertSession($name)` returns a `WebAssert` instance backed by the Mink session, giving access to the same assertion vocabulary as browser tests.
- `getDefaultDriverInstance()` constructs the `HttpKernelBrowser` from `$this->container->get('http_kernel')` and wraps it in `BrowserKitDriver`.

For page-caching scenarios, the default `page_cache` and `dynamic_page_cache` request policies reject requests coming from the CLI SAPI. The new `PageCachePolicyTrait` (`core/tests/Drupal/Tests/Traits/Core/Cache/PageCachePolicyTrait.php`) provides an `alter(ContainerBuilder $container)` implementation (matching `ServiceModifierInterface`) that swaps those policies for lightweight kernel-test-friendly variants: `KernelTestPageCacheRequestPolicy` and `KernelTestDynamicPageCacheRequestPolicy`. Tests using this must implement `ServiceModifierInterface`.

Three coverage tests were added: `KernelTestHttpRequestTest` (basic requests and `clickLink`), `KernelTestHttpCacheRequestTest` (page cache MISS/HIT headers), and `KernelTestHttpDynamicCacheRequestTest` (dynamic page cache MISS/HIT headers with an authenticated user). `NodeLinksTest` was converted from a functional test to a kernel test as a real-world proof of concept.

Documented limitations in the trait: no logged-in user by default (use `UserCreationTrait::setUpCurrentUser()`), no active theme by default (blocks require manual theme install and activation), session semantics differ from a real browser request, and page caching requires `PageCachePolicyTrait`. Issue [#2469717](https://www.drupal.org/node/2469717) (Create a KernelWebTestBase) was closed as a duplicate once this approach proved viable.

### Collaboration

larowlan's comment (#8) was the key turning point: rather than simply adding a standalone `drupalGet()` method, larowlan pointed to Symfony's `KernelBrowser` as a model for wiring Mink directly to the HTTP kernel, which would give `submitForm`, `assertSession`, and the full Mink vocabulary essentially for free. joachim initially explored this at DrupalCon Barcelona with longwave (comment #31), who confirmed the approach was viable and that Symfony already provided a BrowserKit implementation for exactly this purpose. joachim then found the wiring needed only a minimal driver setup, described in comment #34 as needing just "a 1-line method override." mstrelan contributed code fixes, performed real conversion attempts across several modules (help, block_content, dynamic_page_cache), and opened follow-up issues ([#3517299](https://www.drupal.org/node/3517299), [#3519393](https://www.drupal.org/node/3519393)) that surfaced practical edge cases including relative-path handling and `AssertContentTrait::getTextContent()` compatibility.
