## Summary

Drupal's cache contexts system became noticeably faster when resolving cache IDs, especially on sites with many cache context checks per request such as those using Group or Access Policy API. The `optimizeTokens()` method in `CacheContextsManager` was the bottleneck: profiling showed it appearing in 1k+ calls per request and consuming a meaningful slice of total time on dynamic-page-cache and static-cache hits. The fix reduces work inside that method so that the expensive container service lookup is skipped whenever it cannot produce a result. Real-world follow-up profiling reported the time spent in `convertTokensToKeys()` dropping roughly in half on a dynamic page cache hit.

## Impact

No upgrade or configuration changes required.

## Technical details

The cache contexts subsystem lives in `CacheContextsManager`, which is called on every `VariationCache` lookup to turn context tokens into a stable cache key. The hot path is `optimizeTokens()`, which removes redundant child contexts when a parent is already present (e.g. drop `user.roles` if `user` is also listed). The method was slow for three reasons.

First, a single token (or empty array) can never be optimized, yet the method looped and called `getService()` anyway. The fix adds an early return when `count($context_tokens) <= 1`.

Second, the ancestor-presence check used `in_array()` (O(n) per iteration). The fix pre-builds `$context_tokens_lookup = array_flip($context_tokens)` and replaces every `in_array()` with `isset()` (O(1)).

Third, and most impactful, `getService()` (a container lookup that resolves the cache context service) was called for every token that had a period or colon, regardless of whether an ancestor was actually present. Calling `getCacheableMetadata()` on the resolved service only makes sense when an ancestor was found, because that is the only situation where the token could be removed. The rewritten loop checks for ancestors with cheap string operations first, and only calls `getService()` when an ancestor is confirmed. A comment in the diff explicitly documents why `getCacheableMetadata()` results are not cached: context values may change during a request.

Files changed: `core/lib/Drupal/Core/Cache/Context/CacheContextsManager.php` (the optimization itself), `core/tests/Drupal/Tests/Core/Cache/Context/CacheContextsManagerTest.php` (added `$expected_container_calls` assertions to each data-provider case using `createMock` + `expects($this->exactly(...))` to lock in the new call-count guarantees), and `core/profiles/standard/tests/src/FunctionalJavascript/StandardPerformanceTest.php` (three `CacheGetCount` assertions each reduced by 3, reflecting fewer upstream cache lookups triggered by context resolution). A complementary static-caching approach explored in [#3582977](https://www.drupal.org/node/3582977) was discussed but kept separate.
