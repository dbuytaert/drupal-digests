## Summary

The `BatchBuilder` class now accepts plain strings as callback specifications in `addOperation()` and `setFinishCallback()`, in addition to the PHP `callable` type it already accepted. This means site code can now use the Drupal-style single-colon static callback notation (e.g. `ClassName:methodName`) with `BatchBuilder` without hitting a `TypeError`.

## Impact

Affects module maintainers.

- **API change:** `BatchBuilder::addOperation()` and `BatchBuilder::setFinishCallback()` parameter type widened from `callable` to `callable|string`; subclasses overriding these methods must update their signatures.

## Technical details

PHP's `callable` type hint only accepts values that are immediately resolvable as callables: closures, `[class, method]` arrays, or strings naming already-loaded functions. Drupal's batch system supports a single-colon string format (`ClassName:methodName`) that the PHP engine does not recognise as callable. When code updated to use that notation (as encouraged by [#3533299](https://www.drupal.org/node/3533299) for the node access rebuild deprecation) was passed to `BatchBuilder::addOperation()` or `BatchBuilder::setFinishCallback()`, PHP threw a `TypeError` before the batch system could interpret the string.

The fix changes both method signatures in `BatchBuilder` from `callable $callback` to `callable|string $callback`, allowing any string through the type gate while the underlying batch processor handles interpretation. No logic inside the methods changed.

Test coverage was expanded in `BatchBuilderTest`: the original single `testSetFinishCallback()` was split into four explicit variants covering the array form `[Class, method]`, the double-colon string `Class::method`, and two single-colon string forms. `testAddOperation()` was similarly extended to verify all three callback styles. The functional test fixture `BatchTestDefinitions` was updated to use the single-colon format (`BatchTestCallbacks::class . ':method'`) for most operations, while a few array-based calls were deliberately left in place to retain coverage of that pre-existing form.

## Upgrade

If you subclass `BatchBuilder` and override `addOperation()` or `setFinishCallback()`, update the type hint to match the new union type:

```php
// Before
public function addOperation(callable $callback, array $arguments = []): static { … }
public function setFinishCallback(callable $callback): static { … }

// After
public function addOperation(callable|string $callback, array $arguments = []): static { … }
public function setFinishCallback(callable|string $callback): static { … }
```

Callers that previously had to use `[ClassName::class, 'methodName']` arrays to satisfy the `callable` check can now also pass a string directly:

```php
// Now accepted by BatchBuilder
$batch_builder->addOperation(MyClass::class . ':myOperation', [$arg]);
$batch_builder->setFinishCallback(MyClass::class . ':finished');
```
