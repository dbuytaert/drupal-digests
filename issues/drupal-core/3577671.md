## Summary

Six procedural functions in the locale module that handled translation file operations have been replaced by a proper service and typed value objects. Site owners with custom code that calls any of the deprecated locale file functions will see deprecation notices starting in Drupal 11.4.0, and must update before Drupal 13.0.0. There is no user-facing behavior change.

## Impact

Affects module maintainers.

- **Deprecation:** Six procedural locale file functions in `locale.bulk.inc` and `locale.batch.inc` deprecated in favor of `LocaleFileManager` service and `LocaleFile` value object, removed in drupal:13.0.0.
- **New API:** New `LocaleFileManager` service, `LocaleFile` typed value object, `RemoteFileInfo` value object, and `RemoteFileStatus` backed enum added under `Drupal\locale\File\`.

## Technical details

The locale module historically passed translation file metadata around as loosely-typed `stdClass` objects and exposed file operations as bare procedural functions in `locale.bulk.inc` and `locale.batch.inc`. This change introduces four new classes under `Drupal\locale\File\` and deprecates the six procedural functions.

**New classes:**
- `LocaleFile` — a typed `final class` value object replacing ad-hoc `stdClass` instances. Its constructor takes `filename`, `uri`, `hash`, `timestamp`, `langcode`, `project`, and `version`. The static factory `LocaleFile::createFromPath($filename, $filepath, $langcodeOverride)` performs the regex-based filename parsing (previously embedded in `locale_translate_file_attach_properties()`) to extract project name, version, and language code from filenames matching `{project}-{version}.{langcode}.po` or `{project}.{langcode}.po`. The regex was explicitly moved to `createFromPath()` rather than the constructor, following a review observation from alexpott that running it on every construction was unnecessary and caused test failures.
- `LocaleFileManager` — a new injectable service (registered in `locale.services.yml` as `Drupal\locale\File\LocaleFileManager`). It takes `LocaleProjectStorageInterface`, `FileSystemInterface`, `ClientFactory`, `ClientInterface`, `LoggerChannelFactoryInterface`, and `MessengerInterface` as constructor dependencies and exposes: `getInterfaceTranslationFiles()`, `deleteTranslationFiles()`, `checkRemoteFileStatus()`, and `downloadTranslationSource()`.
- `RemoteFileInfo` — a value object returned by `checkRemoteFileStatus()` carrying a `RemoteFileStatus $status`, nullable `int $lastModified`, and nullable `string $location` (set when a redirect occurred).
- `RemoteFileStatus` — a backed `int` enum with cases `Success = 1`, `Error = 0`, `Missing = 404`.

**Deprecated functions** (all deprecated in `drupal:11.4.0`, removed in `drupal:13.0.0`, change record `https://www.drupal.org/node/3577675`):
- `locale_translate_get_interface_translation_files()` → `LocaleFileManager::getInterfaceTranslationFiles()`
- `locale_translate_file_create()` → `new LocaleFile()`
- `locale_translate_file_attach_properties()` → `LocaleFile::createFromPath()`
- `locale_translate_delete_translation_files()` → `LocaleFileManager::deleteTranslationFiles()`
- `locale_translation_http_check()` → `LocaleFileManager::checkRemoteFileStatus()`
- `locale_translation_download_source()` → `LocaleFileManager::downloadTranslationSource()`

All six deprecated functions are updated to thin `@trigger_error` wrappers that delegate to the new service or class. Internal callers in `install.core.inc`, `locale.module`, `LocaleSource`, `LocaleEventSubscriber`, and `TranslationStatusForm` are updated to call the service or `LocaleFile` directly.

## Upgrade

Replace calls to the deprecated procedural functions with the new service and value objects:

```php
// Before:
$files = locale_translate_get_interface_translation_files($projects, $langcodes);

// After:
use Drupal\locale\File\LocaleFileManager;
$files = \Drupal::service(LocaleFileManager::class)->getInterfaceTranslationFiles($projects, $langcodes);
```

```php
// Before:
$file = locale_translate_file_create($filepath);

// After:
use Drupal\locale\File\LocaleFile;
use Drupal\locale\LocaleSource;
$hash = hash_file(LocaleSource::LOCAL_FILE_HASH_ALGO, $filepath);
$file = new LocaleFile(basename($filepath), $filepath, $hash, filemtime($filepath));
```

```php
// Before:
$file = locale_translate_file_attach_properties($file, $options);

// After:
use Drupal\locale\File\LocaleFile;
$file = LocaleFile::createFromPath($filename, $uri, $options['langcode'] ?? NULL);
```

```php
// Before:
locale_translate_delete_translation_files($projects, $langcodes);

// After:
use Drupal\locale\File\LocaleFileManager;
\Drupal::service(LocaleFileManager::class)->deleteTranslationFiles($projects, $langcodes);
```

```php
// Before:
$result = locale_translation_http_check($uri);

// After:
use Drupal\locale\File\LocaleFileManager;
use Drupal\locale\File\RemoteFileStatus;
$remoteFileInfo = \Drupal::service(LocaleFileManager::class)->checkRemoteFileStatus($uri);
// $remoteFileInfo->status is a RemoteFileStatus enum: Success, Error, or Missing
// $remoteFileInfo->lastModified and $remoteFileInfo->location replace the old array keys
```

```php
// Before:
$file = locale_translation_download_source($source_file, 'translations://');

// After:
use Drupal\locale\File\LocaleFile;
use Drupal\locale\File\LocaleFileManager;
use Drupal\locale\LocaleSource;
$hash = hash_file(LocaleSource::LOCAL_FILE_HASH_ALGO, $source_file->uri);
$locale_file = new LocaleFile($source_file->filename, $source_file->uri, $hash);
$file = \Drupal::service(LocaleFileManager::class)->downloadTranslationSource($locale_file, 'translations://');
```

Code that reads `$file->info['version']` from a file object returned by these functions must be updated to read `$file->version` directly (a public property on `LocaleFile`).

### Collaboration

berdir provided an early architectural review proposing the value-object approach and a scoping suggestion to keep the change manageable by using public properties. alexpott caught a critical structural issue: the filename-parsing regex had been moved into the `LocaleFile` constructor, causing it to run far more often than it did previously (only two call sites before the refactor, many more after). This introduced a test failure. nicxvan responded by moving the regex to the `createFromPath()` static factory and using plain `new LocaleFile()` construction everywhere else.
