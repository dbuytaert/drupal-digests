## Summary

Drupal core now uses PHPStan 2.1.38 for static code analysis, updating from a version that supported both PHPStan 1.x and 2.x. This change improves code quality checks by fixing issues with duplicate array keys that newer PHPStan versions detect. The update also resolves conflicts between the automated testing bot and the continuous integration system that were causing false failures on ready-to-be-committed issues.

## Impact

No upgrade or configuration changes required.

## Technical details

This change updates the `phpstan/phpstan` requirement in `composer.json` from `^1.12.27 || ^2.1.34` to `^2.1.38`, and `phpstan/phpstan-phpunit` from `^1.4.2 || ^2.0.11` to `^2.0.11`. The update removes support for PHPStan 1.x in Drupal core's main branch (which will become Drupal 12).

The immediate motivation was fixing a duplicate array key error detected in `core/modules/mysql/src/Driver/Database/mysql/Connection.php` where three PDO MySQL constants had the same value (0). PHPStan 2.1.36+ includes updated stubs from JetBrains that properly detect this issue.

The update also resolved a bot testing issue. An automated bot was testing ready-to-be-committed issues using a local environment that had different PHPStan versions than the CI system, causing false failures. The bot has been adjusted to prevent similar conflicts.

The PHPStan baseline was updated to include additional missing type annotations that the newer version now detects, including return types for `validateForm()` methods and generic type specifications for several interfaces.

## Collaboration

Eight contributors worked through 35 comments over two days to resolve this issue. The discussion revealed important infrastructure considerations about maintaining consistency between automated testing tools and CI systems. Contributors identified that the testing bot was running with different dependency versions than the main CI pipeline, causing false negatives on ready-to-be-committed issues. This led to temporarily disabling the bot and discussing improvements to ensure a single source of truth for code quality checks. The conversation also surfaced concerns about contrib module compatibility, as some modules still support Drupal 10.6 which may require PHPStan 1.x, though this change only affects the main branch heading toward Drupal 12.

## Lessons learned

- When automated tooling runs in multiple environments (CI pipelines vs. local bots), inconsistent versions create conflicting sources of truth. Establish a single authoritative source—typically CI—and derive all other checks from its configuration. *(via mondrake, nod_, longwave)*
- When breaking changes to development tools (like PHPStan major versions) land in a core branch, modules supporting multiple core versions must either maintain separate tool configurations per version or constrain analysis to the latest version only. Core should avoid forcing this choice by deferring breaking tool upgrades to major releases. *(via cmlara, alexpott)*
