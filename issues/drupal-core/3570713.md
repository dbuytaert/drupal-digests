## Summary

Drupal core's `main` branch (future Drupal 12) now requires PHPStan 2.1.38 or later for static analysis. PHPStan 1.x support has been removed from this branch entirely. This only affects developers running static analysis on Drupal core itself; there is no change to runtime behavior or production sites. The 11.x and 10.6.x branches were not changed.

## Impact

No upgrade or configuration changes required.

## Technical details

PHPStan 2.1.35 shipped updated JetBrains/phpstorm-stubs that correctly reflect PHP 8.4's PDO MySQL constants. Earlier versions incorrectly reported `Array has 3 duplicate keys with value 0` for `\Pdo\Mysql::ATTR_USE_BUFFERED_QUERY`, `\Pdo\Mysql::ATTR_FOUND_ROWS`, and `\Pdo\Mysql::ATTR_MULTI_STATEMENTS` in `core/modules/mysql/src/Driver/Database/mysql/Connection.php` at line 125.

The diff updates `composer.json` (both the root and the `core/` copy) from `"phpstan/phpstan": "^1.12.27 || ^2.1.34"` to `"^2.1.38"`, and `phpstan/phpstan-phpunit` from `"^1.4.2 || ^2.0.11"` to `"^2.0.11"`, dropping the PHPStan 1.x range entirely. The locked version in `composer.lock` moves from 2.1.34 to 2.1.38. The PHPStan custom-rules test package inside `core/tests/` is also bumped to 2.1.38.

The tighter PHPStan version detected additional issues in core that must be baselined: new `missingType.return` entries for `ContentEntityConfirmFormBase::validateForm()`, `ContentEntityForm::validateForm()`, `contact\MessageForm::validateForm()`, and `taxonomy\TermForm::validateForm()`; new `missingType.generics` entries for `FieldableEntityInterface::get()`, `getFields()`, and `getTranslatableFields()`, and `ItemList::getIterator()`; and a changed baseline entry in `filter\FilterProcessResult.php` that now flags a `renderCallback.invalidKeyValue` identifier instead of the former `missingType.return` entry. All of these are added to `core/.phpstan-baseline.php`.

The immediate trigger was an automated RTBC-verification bot (run by nod_ outside CI) that used PHP 8.4.6 with an older PHPStan version, causing it to incorrectly flip approximately half the RTBC queue to "Needs work" overnight while GitLab CI pipelines continued to pass. The bot was stopped while this fix landed.

## Upgrade

Developers running PHPStan locally against Drupal core's `main` branch must update to PHPStan 2.1.38 or later:

```bash
composer update phpstan/phpstan phpstan/phpstan-phpunit --with-dependencies
```

Contrib modules that declare `"phpstan/phpstan": "^1"` in their `require-dev` and wish to support both Drupal 10.6/11.x and 12.x simultaneously should be aware that core's `main` branch no longer allows PHPStan 1.x; the 11.x and 10.6.x branches still permit it.

### Collaboration

The incident exposed a gap between the bot's local environment and GitLab CI as sources of truth. After mondrake noted the bot was "systematically killing the RTBC queue," nod_ stopped it and proposed moving the verification checks into CI itself rather than running them on a local machine with a potentially different PHP version. cmlara raised a concern that dropping the PHPStan 1.x range from `main` would complicate contrib modules that support Drupal 10.6 through 12 in a single release, since PHPStan 1.x and 2.x have incompatible configuration formats. longwave and alexpott noted the change was scoped to `main` only (future Drupal 12) and that some point of incompatibility was inevitable.
