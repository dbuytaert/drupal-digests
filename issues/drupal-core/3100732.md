## Summary

The JSON:API module previously provided no extension point for custom modules to inject non-standard metadata into resource object or relationship responses. Two new subscribable events now allow any module to attach arbitrary `meta` fields to JSON:API resource objects and their relationships, following the JSON:API specification's `meta` object support. Sites that do not subscribe to these events are unaffected; the feature is entirely opt-in.

## Impact

Affects module maintainers.

- **New API:** `CollectResourceObjectMetaEvent` and `CollectRelationshipMetaEvent` in `Drupal\jsonapi\Events` allow subscribers to attach `meta` fields to JSON:API resource objects and relationship objects.

## Technical details

The JSON:API specification permits a `meta` member on resource objects and relationship objects. The existing normalizers already handled `meta` when the underlying data objects carried it, but `Relationship::createFromEntityReferenceField()` was only called with default (empty) meta and there was no hook or event to supply it from outside.

The fix introduces two new public event classes in `Drupal\jsonapi\Events\`:

`CollectResourceObjectMetaEvent` is dispatched by `ResourceObjectNormalizer::normalize()` once per resource object, after field normalization but before the final `CacheableNormalization::aggregate()` call. If the event's `getMeta()` returns a non-empty array, it is wrapped in a `CacheableNormalization` using the event itself as the cacheable dependency and placed on the `meta` key of the serialized resource.

`CollectRelationshipMetaEvent` is dispatched in two places: inside `ResourceObjectNormalizer::serializeField()` for relationships embedded in a full resource response, and inside `EntityResource::getRelationship()` for dedicated `/relationships/` endpoint responses. In both cases `getMeta()` is passed as the `meta` argument to `Relationship::createFromEntityReferenceField()`. In `EntityResource::getRelationship()` the event is also added as a cacheable dependency directly on the `CacheableResponseInterface` response.

Both event classes implement `RefinableCacheableDependencyInterface` via `RefinableCacheableDependencyTrait`. Subscribers are required to call `addCacheTags()` and `addCacheContexts()` on the event to reflect the cache variance of the data they attach; this bubbles up through the normalization pipeline automatically. Meta values are written via `setMetaValue(array|string $property, mixed $value)`, which delegates to `NestedArray::setValue()` and therefore supports nested keys. Subscribers register using the event class name as the event identifier (e.g. `CollectResourceObjectMetaEvent::class`).

An earlier draft included a `MetaDataEvents` string-constants class (`COLLECT_RESOURCE_OBJECT_META` and `COLLECT_RELATIONSHIP_META`) that was subsequently removed in favor of class-name-based dispatch, following the pattern already used by Symfony and discussed in [#3375453](https://www.drupal.org/node/3375453) (and informed by [#2825358](https://www.drupal.org/node/2825358)). `ResourceObjectNormalizer` gains a new `EventDispatcherInterface` constructor argument; the class is `@internal` so this does not constitute a public API change. `EntityResource` already received `EventDispatcherInterface` via DI and needed only the additional dispatch calls.

A test module `jsonapi_test_meta_events` ships a `MetaEventSubscriber` that covers: adding meta to a specific resource type, filtering by resource ID, varying meta by user role (with `user.roles` cache context), and adding meta to a named relationship field.

## Upgrade

Modules wanting to add `meta` to resource objects should subscribe to `CollectResourceObjectMetaEvent`:

```php
use Drupal\jsonapi\Events\CollectResourceObjectMetaEvent;
use Symfony\Component\EventDispatcher\EventSubscriberInterface;

class MySubscriber implements EventSubscriberInterface {
  public static function getSubscribedEvents(): array {
    return [CollectResourceObjectMetaEvent::class => 'onCollectMeta'];
  }

  public function onCollectMeta(CollectResourceObjectMetaEvent $event): void {
    if ($event->getResourceObject()->getTypeName() !== 'order--order') {
      return;
    }
    $event->setMetaValue('payment_iframe_url', /* ... */);
    $event->addCacheContexts(['user']);
  }
}
```

Modules wanting to add `meta` to a specific relationship should subscribe to `CollectRelationshipMetaEvent` and filter on `getRelationshipFieldName()`:

```php
use Drupal\jsonapi\Events\CollectRelationshipMetaEvent;

public static function getSubscribedEvents(): array {
  return [CollectRelationshipMetaEvent::class => 'onCollectRelationshipMeta'];
}

public function onCollectRelationshipMeta(CollectRelationshipMetaEvent $event): void {
  if ($event->getRelationshipFieldName() !== 'field_tags') {
    return;
  }
  $event->setMetaValue('stock_level', /* ... */);
  $event->addCacheTags(['my_stock_cache_tag']);
}
```

Subscribers **must** declare appropriate cache tags and contexts on the event to ensure correct cache variance.
