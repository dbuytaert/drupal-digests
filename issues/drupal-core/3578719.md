## Summary

This change reduces the CPU resources requested by Drupal's GitLab CI test pipeline, taking advantage of the reduced total test count now that `migrate_drupal` and other modules are being removed from core. The Kubernetes CPU requests for kernel tests dropped from 12 to 10, for functional tests from 16 to 12, and for functional JavaScript tests from 16 to 14. The build test job received more resources (CPU from 12 to 16, concurrency from 6 to 8) and is now first in the YAML so it starts immediately. PHPStan parallelism was also improved so that MRs touching 21 to 79 files now spawn multiple analysis processes instead of one, shaving roughly 20 seconds off that job. These changes have no effect on how tests are written or how Drupal sites behave; they affect only the CI pipeline that runs on drupal.org.

## Technical details

The GitLab CI configuration file was restructured so the build test job appears first (earlier YAML position determines runner start order). `KUBERNETES_CPU_REQUEST` and `CONCURRENCY` were adjusted for four job types: build tests (12 CPUs / 6 concurrency → 16 CPUs / 8 concurrency), kernel tests (12 CPUs → 10), functional tests (16 CPUs / 24 concurrency → 12 CPUs / 20 concurrency), and functional JavaScript tests (16 CPUs → 14). The kernel, functional, and functional JavaScript jobs also gained an optional `needs` dependency on the PHPStan job, re-adding a constraint that was lost when the pipeline was previously flattened.

In `phpstan.neon`, `parallel.minimumNumberOfJobsPerProcess` was lowered from the PHPStan default of 2 to 1. catch diagnosed that with the default of 2, any MR touching 21 to 60 files produced only 3 jobs total but PHPStan would allocate them all to a single process because neither partition met the minimum-jobs-per-process threshold, leaving other CPU cores idle. With a minimum of 1, the same 58-file MR went from 1 spawned process (48 s) to 3-4 spawned processes (20-22 s).

To maintain acceptable wall times at the lower concurrency settings, `#[Group('#slow')]` annotations were rebalanced across roughly 40 test classes. Tests that were genuinely slow (many `package_manager` kernel tests, installer tests, several WebDriver/functional JS tests) gained the annotation so the scheduler prioritises starting them early. Tests that turned out to be fast (several kernel tests in `views`, `workflows`, `system`, and `recipe`) had the annotation removed.

catch diagnosed and fixed a PHPStan parallelism quirk during the MR: with the default `minimumNumberOfJobsPerProcess: 2`, MRs touching 21 to 60 files would always run PHPStan single-threaded because the job count fell below the threshold for spawning a second process.
