## Summary

Drupal now supports mapping query string parameters directly to controller method arguments using the `#[MapQueryParameter]` attribute from Symfony 6.3. This eliminates repetitive validation code that checked for missing query parameters and manually extracted values from the request object. Controllers become shorter and easier to read, with clearer method signatures that show what parameters they expect.

## Impact

Affects module maintainers.

- **New API:** Controllers can now use the `#[MapQueryParameter]` attribute to automatically map query string parameters to method arguments.

## Technical details

This change adds Symfony's `QueryParameterValueResolver` to Drupal's argument resolver chain in `core.services.yml`. The resolver automatically extracts query string parameters and maps them to controller method arguments decorated with the `#[MapQueryParameter]` attribute.

`ThemeController` was refactored to demonstrate this pattern. The three methods `uninstall()`, `install()`, and `setDefaultTheme()` previously accepted a `Request` object, manually extracted the `theme` query parameter, and validated its presence. Now they accept a `string $theme` parameter with the `#[MapQueryParameter]` attribute. The validation code and throws of `AccessDeniedHttpException` were removed.

Error handling changed slightly. When a required query parameter is missing, the response code changed from 403 to 500, with a more helpful error message from Symfony explaining which argument could not be resolved. This is acceptable because these are error conditions that should not occur during normal operation.

Controllers are explicitly excluded from Drupal's backward compatibility policy, so this method signature change does not require deprecation.
