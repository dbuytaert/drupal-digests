## Summary

A new `search_help` core sub-module now owns the Help search plugin and its related database table. Previously, the `help` module contained a search plugin that depended on the optional `search` module, creating an unwanted hard coupling. Sites that rely on searching help topics must install the new `search_help` module to retain that functionality. For most sites the help module itself is unchanged.

## Impact

Affects module maintainers and site owners.

- **Database update:** No schema change at runtime; the `help_search_items` table definition moves from `help_schema()` to `search_help_schema()`, so the table only exists when `search_help` is installed.
- **API change:** `HelpSearch` renamed to `SearchHelpSearch` and moved to `Drupal\search_help\Plugin\Search`; `HelpSectionManager::setSearchManager()` and the `clearCachedDefinitions()` override are removed.
- **New API:** New `search_help` core module provides the `help_search` search plugin and the `help_search_items` table schema, previously owned by the `help` module.

## Technical details

The `HelpSearch` search plugin, the `help_search_items` database table schema, and the hook implementations that kept the search index in sync were all bundled inside the `help` module despite depending on the optional `search` module. This change extracts them into a dedicated `search_help` sub-module that declares explicit dependencies on `drupal:help` and `search:search`.

The plugin class is renamed from `HelpSearch` to `SearchHelpSearch` and moved from `Drupal\help\Plugin\Search` to `Drupal\search_help\Plugin\Search`. The `help_schema()` hook (defining the `help_search_items` table) is removed from `help.install` and replaced by `search_help_schema()` in the new module's install file.

`HelpSectionManager::setSearchManager()` and the `clearCachedDefinitions()` override (which called `markForReindex()` when the search plugin existed) are both removed. The `plugin.manager.help_section` service definition no longer injects the search manager via a `calls` directive. The index-rebuild trigger is instead handled by a new `hook_rebuild` implementation in `SearchHelpHooks`, which uses constructor injection with `#[Autowire]` for `SearchPluginManager` and `CachedDiscoveryClearerInterface` rather than `\Drupal::service()` calls. The hook implementations for `modules_installed`, `modules_uninstalled`, `themes_installed`, and `themes_uninstalled` that updated `updateTopicList()` and `updateIndexState()` are also moved from the help module's hooks class into this new `SearchHelpHooks` class. Several methods in the plugin class (`getType()`, `execute()`, `updateIndex()`, `indexClear()`, `updateTopicList()`, `updateIndexState()`, `markForReindex()`, `indexStatus()`, `removeItemsFromIndex()`) gain explicit return type declarations, removing the corresponding PHPStan baseline suppressions. The constructor is modernized to use promoted properties.

The commit was initially landed, reverted due to apparent CI slowness, then re-landed after the slowness was traced to a separate race-condition issue ([#3496257](https://www.drupal.org/node/3496257)) unrelated to this change.

## Upgrade

If you have the Help search plugin enabled (visible at `/admin/config/search/pages`), install the new `search_help` module to retain that functionality:

```
drush en search_help
```

If your custom code references the old class or namespace, update it:

```php
// Before
use Drupal\help\Plugin\Search\HelpSearch;

// After
use Drupal\search_help\Plugin\Search\SearchHelpSearch;
```

If your code calls `HelpSectionManager::setSearchManager()`, remove those calls. The method no longer exists.
