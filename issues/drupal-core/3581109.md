## Summary

A new core module called `search_help` now provides the `help_search` search plugin, which was previously bundled inside the `help` module. When updating a site that has both the `help` and `search` modules enabled, `system_update_11400()` installs `search_help` automatically and migrates the existing index table. Sites that have `help` but not `search` simply have the `help_search_items` table dropped. Without `search_help` enabled, Help topic content will not appear in site search results.

## Impact

Affects module maintainers and site owners.

- **Database update:** `system_update_11400()` drops the `help_search_items` table previously owned by `help` and installs `search_help` if both `help` and `search` are active.
- **API change:** `Drupal\help\Plugin\Search\HelpSearch` moved and renamed to `Drupal\search_help\Plugin\Search\SearchHelpSearch`; `HelpSectionManager::setSearchManager()` and `HelpSectionManager::clearCachedDefinitions()` removed.
- **New API:** New `search_help` core sub-module at `core/modules/search/modules/search_help` providing the `help_search` plugin via `SearchHelpSearch`.

## Technical details

The `help_search` plugin, the `help_search_items` database schema, and the hook implementations that kept the index current were all living in the `help` module despite requiring the `search` module. The restructuring extracts all search-specific code into a new `search_help` sub-module.

The plugin class `Drupal\help\Plugin\Search\HelpSearch` is replaced by `Drupal\search_help\Plugin\Search\SearchHelpSearch`; the plugin ID `help_search` is unchanged. The `help_search_items` table schema moves from `help_schema()` to `search_help_schema()`. `HelpSectionManager::setSearchManager()` and the overridden `HelpSectionManager::clearCachedDefinitions()` are removed from `help`; index revalidation on cache clear is now handled by `SearchHelpHooks::rebuild()` implementing `hook_rebuild` in `search_help`. Hook implementations for `hook_modules_installed`, `hook_modules_uninstalled`, `hook_themes_installed`, and `hook_themes_uninstalled` move to `Drupal\search_help\Hook\SearchHelpHooks`.

The upgrade path runs as `system_update_11400()` in the `system` module, placing it early enough that the new `help_search` plugin is in place before other module updates execute. It drops the `help_search_items` table previously owned by `help` and installs `search_help` when both `help` and `search` are active. The companion `help_post_update_search_help_dependencies()` post-update function recalculates config entity dependencies, updating the `search.page.help_search` config entity's module dependency from `help` to `search_help`. A follow-up at [#3586821](https://www.drupal.org/node/3586821) corrects additional typos introduced by this change.

## Upgrade

Developers referencing `Drupal\help\Plugin\Search\HelpSearch` must update to `Drupal\search_help\Plugin\Search\SearchHelpSearch`:

```php
// Before
use Drupal\help\Plugin\Search\HelpSearch;
$plugin = HelpSearch::create($container, [], 'help_search', []);

// After
use Drupal\search_help\Plugin\Search\SearchHelpSearch;
$plugin = SearchHelpSearch::create($container, [], 'help_search', []);
```

`HelpSectionManager::setSearchManager()` and `HelpSectionManager::clearCachedDefinitions()` are removed. Remove any direct calls to these methods. The `search_help` module must be listed as a dependency in any module or test that installs it alongside `help` and `search`.

### Collaboration

After the initial commit landed, alexpott identified a critical missing upgrade path: existing sites could not install `search_help` because the `help_search_items` table was already in the database (owned by the old `help` schema), causing a fatal error when running database updates. alexpott opened and landed the fix MR under this same issue, and in the thread discussion with catch determined that the update hook should be numbered `system_update_11400()` rather than `system_update_12002()` so the change could be backported to 11.x.
