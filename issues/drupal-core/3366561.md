## Summary

Drupal's libraries API now supports a first-class `fonts` key in `.libraries.yml` files, allowing themes and modules to declare font preloading directly alongside their CSS and JS. When `preload: true` is set on a font entry, Drupal automatically emits a `<link rel="preload" as="font">` tag in the HTML head. This replaces ad-hoc workarounds such as Olivero's hand-crafted Twig template and Navigation module's manual `html_head_link` attachment. Umami also gains font preloading for the first time. The practical effect is that browsers can begin fetching font files earlier, reducing the rendering delay caused by fonts that previously only started loading after CSS was downloaded and parsed.

## Impact

Affects module maintainers.

- **Deprecation:** `HtmlResponseAttachmentsProcessor` and `BigPipeResponseAttachmentsProcessor` constructors require a new `FileUrlGeneratorInterface` argument; omitting it is deprecated in drupal:11.4.0 and will be required in drupal:12.0.0.
- **New API:** New `AssetResolverInterface::getFontAssets()` method and `fonts:` key in `.libraries.yml` for declaring font preloading.

## Technical details

The problem was that font preloading had no structured place in the libraries system, forcing themes like Olivero to maintain a separate `preload.twig` template (with hardcoded paths) included from `html.html.twig`, and Navigation to manually build `html_head_link` arrays in PHP.

The fix adds `fonts` as a recognized top-level key alongside `js` and `css` in `LibraryDiscoveryParser::buildByExtension()`. The parser now defaults each library to `'fonts' => []` and iterates over `['js', 'css', 'fonts']` uniformly. Each font entry supports a `preload: true` flag.

A new `getFontAssets(AttachedAssetsInterface $assets, ?LanguageInterface $language = NULL): array` method is added to `AssetResolverInterface` and implemented in `AssetResolver`. It follows the same caching pattern as `getCssAssets()` and `getJsAssets()`, keying the persistent cache entry (`library_info` tag) on active theme, language, and the resolved library list via a `fonts:` prefix. Because the font library list is a subset of all attached libraries and typically changes less often than the full CSS set, the cache hit rate is higher.

`HtmlResponseAttachmentsProcessor::processAttachments()` calls `getFontAssets()` and, for each font with `preload: true`, pushes a `<link rel="preload" as="font" type="font/<ext>" crossorigin="anonymous">` entry onto `$attached['html_head_link']`. URL generation uses `FileUrlGeneratorInterface::generate()` so that paths resolve correctly on non-root pages. `FileUrlGeneratorInterface` is injected as a new constructor argument on both `HtmlResponseAttachmentsProcessor` and `BigPipeResponseAttachmentsProcessor`; omitting it is deprecated in drupal:11.4.0.

Concrete conversions in the diff: Olivero's `preload.twig` template and its `{% include %}` line in `html.html.twig` are deleted; four font paths move to a `fonts:` block in `olivero.libraries.yml`. Navigation removes the manual `html_head_link` build from `NavigationBlock::doBuildNavigation()` and declares `inter-var.woff2` in `internal.navigation` library. Umami adds three Source Sans Pro and one Scope One font to the `global` library.

Early-hints (103 responses) support is left for the follow-up issue [#3570909](https://www.drupal.org/node/3570909).

## Upgrade

Any code that constructs `HtmlResponseAttachmentsProcessor` or `BigPipeResponseAttachmentsProcessor` directly (outside of the service container) must add `FileUrlGeneratorInterface` as the last constructor argument. The service definitions in `core.services.yml` and `big_pipe.services.yml` are already updated.

```php
// Before (deprecated since drupal:11.4.0):
new HtmlResponseAttachmentsProcessor(
  $assetResolver, $configFactory, $cssRenderer, $jsRenderer,
  $requestStack, $renderer, $moduleHandler, $languageManager
);

// After (required from drupal:12.0.0):
new HtmlResponseAttachmentsProcessor(
  $assetResolver, $configFactory, $cssRenderer, $jsRenderer,
  $requestStack, $renderer, $moduleHandler, $languageManager,
  $fileUrlGenerator   // \Drupal\Core\File\FileUrlGeneratorInterface
);
```

To declare font preloading in a theme or module library, add a `fonts:` block to the relevant `.libraries.yml` entry:

```yaml
my-library:
  version: VERSION
  css:
    base:
      css/fonts.css: {}
  fonts:
    fonts/my-font-regular.woff2:
      preload: true
    fonts/my-font-bold.woff2:
      preload: true
```

The path is relative to the extension root, matching the URL that appears in the CSS `@font-face` declaration. External URLs are also accepted as keys.

Themes that previously preloaded fonts via a hand-maintained Twig template or a `hook_preprocess_page()` `html_head_link` attachment can remove that code and use the `fonts:` key instead.

### Collaboration

The API shape was debated across several comments. rikki_iki (comment #5) proposed using separate named top-level entries (`preload:`, `preconnect:`) rather than a generic `links:` block to reduce misuse risk, and catch agreed, eventually settling on a `fonts:` key with a `preload: true` flag per entry. berdir raised a practical concern about cache-busting: because the preload URL and the CSS `@font-face` URL must match exactly, and Drupal does not append a version string to font URLs, keeping them in sync is the author's responsibility. grimreaper provided code review that led to at least one test improvement catch had initially declined, and flagged the question of Single Directory Components; that was deferred to follow-up issue [#3586470](https://www.drupal.org/node/3586470). finnsky made the first external commit, fixing PHPCS issues and adding Navigation support, which surfaced the path-resolution bug that led catch to introduce `FileUrlGeneratorInterface`.
