## Summary

The `drupal/core-recommended` package pinned `symfony/polyfill-intl-idn` at `~v1.37.0`, which is affected by CVE-2026-46644 (insecure equivalence in IDN handling for ASCII-only `xn--` labels). Because the pin prevented Composer from resolving a newer version, sites using `drupal/core-recommended` could not update the vulnerable package on their own. This change raises the pin to `~v1.38.1`, the first non-vulnerable release, so a normal `composer update` will pull in the patched version. The fix was backported to 10.5.x, 10.6.x, 11.2.x, 11.3.x, 11.4.x, 11.x, and main.

## Impact

Affects site owners.

- **Security:** CVE-2026-46644: insecure IDN equivalence in symfony/polyfill-intl-idn versions prior to 1.38.1; drupal/core-recommended pin updated to require the patched release.

## Technical details

The entire change is in `composer.json` (the version constraint for `symfony/polyfill-intl-idn` updated from `~v1.37.0` to `~v1.38.1`) and the corresponding `composer.lock` entry (version bumped from `v1.37.0` to `v1.38.1`, git reference updated to `dc21118016c039a66235cf93d96b435ffb282412`). No Drupal PHP code was modified. The vulnerability is in the polyfill's IDN-to-ASCII conversion, which incorrectly treats some `xn--` ACE labels as equivalent when they are not. Sites running PHP with the native `intl` extension enabled are not affected at runtime because the polyfill checks for the extension and defers to it; the vulnerability only triggers when the polyfill's own PHP implementation is used.

## Upgrade

After updating to a Drupal core release that includes this fix, run:

```bash
composer update drupal/core-recommended symfony/polyfill-intl-idn -W
```

Until such a release is available, a temporary workaround is to add an alias in your root `composer.json`:

```json
"require-dev": {
    "symfony/polyfill-intl-idn": "1.38.1 as 1.37.0"
}
```

Or replace `drupal/core-recommended` with `drupal/core` in your root `require` and add `"symfony/polyfill-intl-idn": "^1.38.1"` explicitly. Alternatively, enabling the PHP `intl` extension on your server bypasses the vulnerable polyfill code at runtime without any Composer changes.
