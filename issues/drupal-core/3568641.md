## Summary

The Drupal CI pipeline previously had to invoke PHPStan two to four times per job: once for the GitLab code-quality report (gitlab format), once for JUnit, once for a human-readable table on failure, and once more to regenerate the baseline. This change introduces a custom `MultiplexErrorFormatter` that runs all three output formats in a single PHPStan execution, cutting 10 to 30 seconds of wall time from each MR pipeline. Baseline regeneration remains a separate run only when the scan actually fails. This change affects only the Drupal CI infrastructure and has no impact on sites or their code.

## Impact

No upgrade or configuration changes required.

## Technical details

PHPStan's `--error-format` flag accepts only one formatter at a time, and PHPStan's result cache means re-runs are fast but still incur process-startup and file-I/O overhead for each invocation. The fix introduces two new classes under `core/tests/PHPStan/ErrorFormatter/` (namespace `Drupal\PHPStan\ErrorFormatter\`, autoloaded via both the root `composer.json` and `core/tests/PHPStan/composer.json`):

- `MultiplexErrorFormatter` implements PHPStan's `ErrorFormatter` interface. Its constructor accepts the PHPStan DI `Container`, a `basePath` (optionally read from an environment variable via `::getenv(_ARTIFACTS_DIR)` in the neon config), and an `array<string, string> $outputs` mapping formatter service name to output file path. In `formatErrors()`, it iterates the map, resolves each formatter from the container as `errorFormatter.<name>`, wraps the target file in a `FileOutput` instance, and calls the formatter's own `formatErrors()`, collecting the highest exit code.

- `FileOutput` implements PHPStan's `Output` interface and writes all output to a file handle opened in the constructor. It handles `php://stdout` transparently because `fopen('php://stdout', 'w')` works normally. The constructor also creates the target directory if it is missing. The handle is closed in `__destruct()`.

The `errorFormatter.multiplex` service is registered in `core/phpstan.neon.dist` with three outputs: `gitlab` writing to `phpstan-gitlab.json`, `junit` writing to `phpstan-junit.xml`, and `table` writing to `php://stdout`. The `basePath` is supplied by `::getenv(_ARTIFACTS_DIR)` so files land in the CI artifacts directory automatically; locally the env var is absent and `basePath` is `false`, so filenames are resolved relative to the working directory.

In `.gitlab-ci.yml` the three separate PHPStan invocations are collapsed into one: `php vendor/bin/phpstan -vvv analyze --configuration=$_PHPSTAN_NEON --error-format=multiplex`. The human-readable "plain text for human logs" re-run on failure is also removed, since the `table` formatter now writes to stdout during the main run. The CI artifact for `codequality` is renamed from `phpstan-quality-report.json` to `phpstan-gitlab.json` to match the new output filename. Baseline regeneration on failure is retained as a separate run because it requires `--generate-baseline`, which is unrelated to error formatting.

The related issue [#3575080](https://www.drupal.org/node/3575080) covers further enhancements to this formatter.

longwave's contribution was the decisive design improvement.
