## Summary

When a recipe used the `addToAllBundles` config action without specifying the `label` or `description` properties, PHP emitted "Undefined array key" warnings. The fix enforces that `label` must be provided (throwing a clear `ConfigActionException` if it is missing) and makes `description` optional by defaulting to an empty string. Recipe authors who omit `label` from `addToAllBundles` actions will now get a descriptive error instead of a silent warning.

## Impact

No upgrade or configuration changes required.

## Technical details

The `AddToAllBundles` config action in the recipe system creates field instances on all bundles of a given entity type. Its `apply()` method reads `$value['label']` and `$value['description']` directly from the action's YAML input, with no guards against missing keys.

The initial issue proposed defaulting both to `null`, but `phenaproxima` noted this would produce broken configuration: per `core/config/schema/core.data_types.schema.yml`, `label` is a required field and `description` is of type `text` (non-nullable). Committing null values would cause immediate failures when the recipe system validates against config schema.

The accepted fix uses PHP's null-coalescing-with-throw pattern:
- `'label' => $value['label'] ?? throw new ConfigActionException(sprintf('The \`label\` property must be set for %s.', $id))` makes the missing-label error actionable.
- `'description' => $value['description'] ?? ''` quietly defaults to an empty string, which satisfies the schema.

A new kernel test `testMissingLabel()` in the `AddToAllBundlesTest` class exercises the exception path by running `RecipeRunner::processRecipe()` against a recipe that omits `label` from `addToAllBundles` and asserting that `ConfigActionException` is thrown with the expected message.
