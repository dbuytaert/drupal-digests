## Summary

The form state immutability feature has been removed from Drupal's Form API in 11.x. This feature had been designed to protect against form state poisoning on page-cached pages: when a form was served from page cache to multiple anonymous users, a malicious user could potentially submit a shared form build ID to overwrite another user's form state. The immutability mechanism prevented this by regenerating the form build ID on each cache load. After [#2502785](https://www.drupal.org/node/2502785) removed support for caching forms on safe HTTP methods (GET etc.), the attack vector no longer exists and the feature became dead code. The removal is a pure deletion with no replacement needed.

## Impact

Affects module maintainers.

- **API change:** The `immutable` key in `$form_state->getBuildInfo()` is no longer processed; `FormCache::getCache()` no longer generates a new `#build_id` or sets `#build_id_old`, and `system_form_alter()` in `system.module` has been removed entirely.

## Technical details

The immutability mechanism had two halves. First, `system_form_alter()` in `system.module` checked whether the current request was cacheable by consulting the `page_cache_request_policy` service; if so, it called `$form_state->addBuildInfo('immutable', TRUE)`. Second, `FormCache::getCache()` in `core/lib/Drupal/Core/Form/FormCache.php` checked the `immutable` key in `getBuildInfo()` when loading a cached form; if set, it generated a new random `#build_id` via `Crypt::randomBytesBase64()`, updated the `form_build_id` element's `#value` and `#id`, stored the old ID as `#build_id_old`, and cleared the flag before returning the form. This ensured each user loading a cached page got a unique form build ID even if the rendered HTML was shared.

The commit removes both halves entirely, along with the now-unused `use Drupal\Component\Utility\Crypt` import in `FormCache.php` and `use Drupal\Core\PageCache\RequestPolicyInterface` import in `system.module`. The entire `page_cache_form_test` test module (five files under `core/modules/page_cache/tests/modules/`) is deleted, as it existed solely to verify immutability behavior. Associated test methods are also removed: `testFormImmutability()` in `PageCacheTest.php`, `testImmutableForm()` and `testImmutableFormLegacyProtection()` in `StorageTest.php`, and `testGetCacheImmutableForm()` in `FormCacheTest.php`. A test-only workaround in `FormTestStorageForm.php` that faked form caching via `?immutable=1` and `?cache=1` query parameters is also removed.

One gap remains: the docblock on `$build_info` in `FormState.php` (around line 60) still describes the `immutable` key and its old behavior; this was not updated in this commit.

## Upgrade

If your code sets the immutable flag on form state, those calls can be safely removed:

```php
// Before (no longer has any effect):
$form_state->addBuildInfo('immutable', TRUE);

// After: simply delete the call.
```

Code that inspects `$form_state->getBuildInfo()['immutable']` or relies on `$form['#build_id_old']` being set when loading a cached form should also be updated, as neither is populated anymore.

### Collaboration

alexpott revived this nine-year-old issue in August 2024, noting that the immutability code had broken a commit. He surveyed contrib usage, found only one module (sessionless) using the flag and identified it as a copy-paste of core code that would silently stop doing anything once the flag was removed. He argued directly against deprecation: since the prerequisite [#2502785](https://www.drupal.org/node/2502785) had already eliminated the attack vector, the feature was pointless and a change record was the right path. He opened MR !9340 and catch committed it to 11.x shortly after.
