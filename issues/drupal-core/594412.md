# #594412: Correctly label all site-owning super-admin permissions

- **Project:** Drupal Core
- **Type:** Normal task

## Summary

Some Drupal core permissions that effectively grant full control over a site were missing the standard security warning on the permissions administration page, while others had it. This inconsistency could mislead site builders into granting powerful permissions without realizing the risk. The fix added the missing warning to "Administer content types" and "Administer site configuration," and corrected the "administer software updates" permission, which had been inadvertently renamed to "Run software updates" while every other reference in the codebase still used the original name. A `restrict access` label was added to that permission as well. The result is a consistently labeled set of site-owning permissions matching the guidance in the security team's FAQ.

## Impact

No upgrade or configuration changes required.

## Technical details

The security team's FAQ documented five permissions as effectively granting full site takeover ("administer filters," "administer users," "administer permissions," "administer content types," "administer site configuration"), but the Drupal 7 permissions UI only showed the standard warning text on some of them. David_Rothstein audited the mismatch in comment #5; mrfelton produced a systematic table in comment #12 showing that `administer content types` and `administer site configuration` were the two missing the warning entirely.

Two commits were made to `modules/node/node.module` and `modules/system/system.module`. The first (56e60eb, March 2010, by webchick) added `'description' => drupal_placeholder(array('text' => t('Warning: Give to trusted roles only; this permission has security implications.')))` to the `administer content types` permission in `node_permission()` and to the `administer site configuration` permission in `system_permission()`.

The second commit (1c9b84a, May 2010, by Dries) addressed the `administer software updates` permission: it corrected its display title from `t('Run software updates')` back to `t('Administer software updates')`, which was already the string used in `update.php` user-facing text, code comments in `settings.php`, and elsewhere, and it added `'restrict access' => TRUE`, the clean flag introduced by [#248598](https://www.drupal.org/node/248598) to mark permissions as security-sensitive. The `restrict access` mechanism was also already in use for `administer users`, `administer permissions`, and others; this commit simply extended it to the software-updates permission.
