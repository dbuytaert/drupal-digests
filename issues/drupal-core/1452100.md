## Summary

Private files attached to non-default entity revisions (draft revisions created by Content Moderation, or older revisions accessible via revision history) previously returned "Access Denied" to all users, including administrators. This happened because the file download hook and access handler only checked the default revision for a file reference. If a file had been replaced in the current revision, or existed only in a pending draft, no reference was found and access was denied. The fix makes private file access work correctly: users with the `view revision` permission on the referencing entity can now download private files from non-default revisions, while users without that permission continue to be denied access to files that are not in the current revision.

## Impact

Affects module maintainers.

- **Deprecation:** `file_get_file_references()`, `file_field_find_file_reference_column()`, and `FileAccessControlHandler::getFileReferences()` deprecated in drupal:11.4.0, removed in drupal:13.0.0; use `FileReferenceResolver::getReferences()` instead.
- **New API:** New `FileReferenceResolver` service and `FileReferenceUsage` value object for resolving file references across both default and non-default entity revisions.

## Technical details

The old `file_get_file_references()` function loaded only the current/default revision of each referencing entity. When a file was superseded in the current revision or existed only in a pending draft, no reference was found and access was denied. The `file_file_download()` hook compounded this by also using `FIELD_LOAD_CURRENT` for its early-exit reference check.

The fix introduces `FileReferenceResolver` (registered as a service in `file.services.yml`) and a `FileReferenceUsage` readonly value object. `getReferences(FileInterface $file)` is a generator that implements a two-pass strategy: first it loads the default revision of each entity recorded in `file_usage` and checks all file-reference fields (across all translations); if the file is not found in the default revision and the entity type is revisionable, it queries `allRevisions()` sorted descending to find the most recent non-default revision that references the file. Results are capped at 50 non-default revision lookups per entity type (`REVISION_LOOKUP_LIMIT`). Results are cached in the memory cache under the `file_references` cache tag. Each yielded `FileReferenceUsage` carries the entity type ID, field name, and either the entity ID (default-revision match) or the revision ID (non-default-revision match).

`FileAccessControlHandler::checkAccess()` now calls `$resolver->getReferences()` and `$resolver->loadEntityFromUsage()` instead of the old `getFileReferences()` method. When the loaded entity implements `RevisionableInterface` and `!isDefaultRevision()`, the handler checks `view revision` access rather than `view` access before additionally checking field-level access. `FileDownloadHook::fileDownload()` also uses the resolver for its early-exit check (consuming only the first yielded result).

`file_get_file_references()`, `file_field_find_file_reference_column()`, and `FileAccessControlHandler::getFileReferences()` are all deprecated in drupal:11.4.0 and removed in drupal:13.0.0. Cache invalidation via `drupal_static_reset('file_get_file_references')` or `drupal_static_reset('file_get_file_references:field_columns')` is also deprecated.

## Upgrade

```php
// Before (deprecated in 11.4.0, removed in 13.0.0):
use Drupal\Core\Entity\EntityStorageInterface;
$references = file_get_file_references($file, NULL, EntityStorageInterface::FIELD_LOAD_REVISION, NULL);

// After:
$resolver = \Drupal::service(\Drupal\file\FileReferenceResolver::class);
foreach ($resolver->getReferences($file) as $usage) {
  $entity = $resolver->loadEntityFromUsage($usage);

}
```
