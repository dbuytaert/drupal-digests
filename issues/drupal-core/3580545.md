## Summary

When Drupal's routing system could not find a route for a requested path, the resulting 404 response was incorrectly marked as uncacheable (max-age 0). This was a precondition blocker for [#2352009](https://www.drupal.org/node/2352009), which aims to let the page cache properly respect response cacheability. This fix makes empty route lookups cacheable so that 404 responses now carry max-age -1 (Permanent) in cache debug headers. No additional caching actually occurs from this fix alone, because Dynamic Page Cache operates per-route and cannot act before routing, and the page cache's handling of the max-age is addressed separately in [#2352009](https://www.drupal.org/node/2352009).

## Impact

Affects module maintainers.

- **API change:** The Symfony welcome page is no longer triggered when a controller throws `new NotFoundHttpException(previous: new NoConfigurationException())`; Drupal's regular 404 page is rendered instead.
- **New API:** New `CacheableResourceNotFoundException` in `Drupal\Core\Routing\Exception`; new `Drupal\Core\Http\EventListener\RouterListener` replaces `Symfony\Component\HttpKernel\EventListener\RouterListener` as the `router_listener` service.

## Technical details

The root cause was that Symfony's `ResourceNotFoundException` does not implement `CacheableDependencyInterface`. When `Router::matchRequest()` threw it for an unmatched path, the exception bubbled through `DefaultExceptionHtmlSubscriber::makeSubrequest()`, which read cacheability off the exception and got nothing, effectively contributing a max-age of 0 to the response.

The fix introduces three coordinated changes. First, a new `CacheableResourceNotFoundException` class in `Drupal\Core\Routing\Exception` extends Symfony's `ResourceNotFoundException` and implements `CacheableDependencyInterface` via `CacheableDependencyTrait`. It is constructed with a `CacheableMetadata` carrying no contexts, no tags, and `Cache::PERMANENT` max-age. Second, `Router::matchRequest()` now throws `CacheableResourceNotFoundException` instead of the bare `ResourceNotFoundException` when the initial route collection is empty. Third, a new Drupal-owned `RouterListener` in `Drupal\Core\Http\EventListener` replaces `Symfony\Component\HttpKernel\EventListener\RouterListener` (which is declared `final` in Symfony, so it is copied rather than extended). The key addition in Drupal's `RouterListener::onKernelRequest()` is a dedicated `catch (CacheableResourceNotFoundException)` block that wraps the exception in a `CacheableNotFoundHttpException`, preserving the cacheability metadata so it propagates correctly. The existing `catch (ResourceNotFoundException)` block remains to handle any non-cacheable cases. The Drupal `RouterListener` also omits Symfony's welcome-page logic, logging support, and URL-matcher support, which are not applicable in Drupal.

The `router_listener` service in `core.services.yml` is updated from `Symfony\Component\HttpKernel\EventListener\RouterListener` to `Drupal\Core\Http\EventListener\RouterListener`, removing the previously passed `NULL` logger argument. Tests in `PageCacheTest`, the REST module base test class, and a new `RouterTest` kernel test verify the new behavior and that the thrown exception carries the expected empty cache metadata with permanent max-age.

## Upgrade

If your code catches `Symfony\Component\Routing\Exception\ResourceNotFoundException` thrown by `Router::matchRequest()`, ensure you also handle (or widen the catch to) `Drupal\Core\Routing\Exception\CacheableResourceNotFoundException`, since that is now thrown for the empty-route case.

If you depend on the Symfony welcome page appearing when a controller throws:

```php
throw new NotFoundHttpException(previous: new NoConfigurationException());
```

that behavior is removed. The regular Drupal 404 page is rendered instead.

### Collaboration

tstoeckler's initial MR included significant refactoring of `RouteProvider` to expose `CacheableMetadata` through route collection lookups, including removing internal methods like `getRouteCollectionCacheId()` and injecting `CacheContextsManager`. berdir pointed out in comment #7 that this would cause widespread breakage in contrib modules that override `RouteProvider` (such as domain_source), and asked whether a less invasive intermediate step was possible. After discussion, tstoeckler reverted the `RouteProvider` changes and implemented the simpler approach in the final MR, focusing only on the exception chain. The Dynamic Page Cache header-leakage side effect that emerged during review was spun off to the pre-existing [#3451483](https://www.drupal.org/node/3451483).
