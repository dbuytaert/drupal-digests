## Summary

A SQL injection vulnerability in Drupal's PostgreSQL entity query handler has been fixed. When performing a case-insensitive entity query with an array value (for example, an `IN` query on a field configured as case-insensitive), the operator string was concatenated directly into the SQL `WHERE` clause with no validation. An attacker who could control the operator argument to `->condition()`, for example via a contrib module that exposes this parameter, could inject arbitrary SQL on PostgreSQL databases. The fix restricts allowed operators to `'IN'` and `'NOT IN'` for this specific code path. The vulnerability does not affect MySQL or SQLite, and is not directly exploitable from Drupal core alone.

## Impact

Affects site owners.

- **Security:** SQL injection via unvalidated operator in the PostgreSQL entity query case-insensitive array condition handler in `Drupal\pgsql\EntityQuery\Condition::translateCondition()`.

## Technical details

The vulnerable code is in `Drupal\pgsql\EntityQuery\Condition::translateCondition()`. The method checks `is_array($condition['value']) && $case_sensitive === FALSE` and, if true, constructs a raw SQL fragment by concatenating `$condition['operator']` directly: `'LOWER(' . $sql_query->escapeField($condition['real_field']) . ') ' . $condition['operator'] . ' ('`. While the field name is properly escaped via `escapeField()` and values are parameterized, no validation existed for the operator itself. `ConditionBase::condition()` stores the operator without validation, and the base `Sql\Condition::translateCondition()` returns early for array values, so no upstream guard caught this.

The fix inserts an allowlist check immediately before the SQL construction: if `$condition['operator']` is not `'IN'` or `'NOT IN'`, an `\InvalidArgumentException` is thrown with a descriptive message. Case-sensitive conditions and non-array values are unaffected. The commit also adds a new unit test class `ConditionTest` covering valid operators, the invalid-operator exception path, and the case-sensitive bypass path. This issue was originally a private security report; the Drupal Security Team approved public handling because the `$operator` parameter being an unvalidated sink was already publicly documented since 2015 (issue [#2626026](https://www.drupal.org/node/2626026)).

## Upgrade

No API changes. If your contrib module or custom code passes user-controlled input as the operator argument to entity query conditions on PostgreSQL, it will now receive an `\InvalidArgumentException` for any operator other than `'IN'` or `'NOT IN'` in the case-insensitive array path. Sanitize or validate the operator before passing it to `->condition()`.
