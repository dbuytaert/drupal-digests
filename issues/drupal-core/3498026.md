## Summary

Applying recipes that install many modules was dramatically slower than installing those same modules through the UI or Drush. A recipe installing a large distribution (like Drupal CMS) could take over 45 seconds and use 160 MB of memory, while the equivalent non-recipe install finished in under 20 seconds. The cause was that the recipe runner called `ModuleInstaller::install()` once per module, triggering a full router and container rebuild for each one. This fix makes the recipe runner install modules in configurable batches (default 20 at a time), cutting installation time roughly in half in real-world testing.

## Impact

Affects module maintainers.

- **Deprecation:** `RecipeRunner::installModule()` deprecated in drupal:11.4.0, removed in drupal:13.0.0; replaced by `RecipeRunner::installModules()`
- **New API:** New `RecipeMultipleModulesConfigStorage` class and `RecipeRunner::installModules()` method for batch module installation within recipes

## Technical details

Before this change, `RecipeRunner::processInstall()` looped over `$install->modules` and called `static::installModule()` once per module. `toBatchOperationsInstall()` similarly produced one batch step per module. Both paths caused `ModuleInstaller::install()` to run individually for each module, which rebuilds the router and the service container on every call.

The fix rewrites both paths to collect modules and pass them in chunks to a new `RecipeRunner::installModules()` method. The chunk size is read from `Settings::get('core.multi_module_install_batch_size', 20)`, the same setting used by the rest of the installer subsystem. `ModuleInstaller::install()` is now called once per chunk rather than once per module. The old `RecipeRunner::installModule()` is deprecated in drupal:11.4.0 (removed in drupal:13.0.0) and now delegates to `installModules()`.

The non-trivial part of the change is config storage. During module install the recipe runner puts `ConfigInstaller` into sync mode so that only simple configuration is installed (config entities are deferred to `RecipeRunner::processConfiguration()`). The old single-module path supplied a `RecipeOverrideConfigStorage` wrapping just that one module's `config/install` directory. Installing a batch of modules requires a storage that spans all of their `config/install` directories simultaneously.

The new `RecipeMultipleModulesConfigStorage` class (`core/lib/Drupal/Core/Recipe/RecipeMultipleModulesConfigStorage.php`) is a read-only `StorageInterface` implementation that holds one `FileStorage` per module, keyed by module name. Every read operation (e.g. `read()`, `exists()`, `listAll()`) routes to the correct per-module storage by splitting the config object name on the first dot and matching the prefix to the module name. For example, `system.site` is always served from the `system` module's storage, regardless of whether another module in the batch also ships a `system.site.yml` in its own `config/install` directory. Write operations throw `\BadMethodCallException`. The static factory `RecipeMultipleModulesConfigStorage::createFromModuleList()` builds the map of `FileStorage` instances from a list of module names and a `ModuleExtensionList`.

`RecipeRunner::installModules()` wraps this storage in a `RecipeOverrideConfigStorage` (which layers the recipe's own config directory on top, allowing the recipe to override simple module configuration), sets it as the source storage on the `config.installer` service, then calls `ModuleInstaller::install()` with the full array of modules in the chunk.

A comprehensive unit test for `RecipeMultipleModulesConfigStorage` was added at `core/tests/Drupal/Tests/Core/Recipe/RecipeMultipleModulesConfigStorageTest.php`, covering read isolation, modules with similar name prefixes (e.g. `system` vs `system_test`), read-only enforcement, collection support, and `getAllCollectionNames()` deduplication. The kernel test `RecipeRunnerTest` was updated to reflect the reduced number of batch steps produced by `toBatchOperationsInstall()`.

## Upgrade

If your code calls `RecipeRunner::installModule()` directly, replace it with `RecipeRunner::installModules()`:

```php
// Before (deprecated in drupal:11.4.0):
RecipeRunner::installModule($module, $recipeConfigStorage);

// After:
RecipeRunner::installModules([$module], $recipeConfigStorage);
```

Note that the Recipe API remains `@internal` and `final`; direct calls are discouraged outside of core and contrib maintained against the recipe subsystem.

### Collaboration

alexpott flagged early in the thread that naively combining all modules' config directories would be unsafe: a module can ship a `config/install` file whose name belongs to a different module, which could silently override that module's configuration during a bulk install. alexpott then designed the prefix-based routing that became `RecipeMultipleModulesConfigStorage`, specifically so each config object is read from exactly the module that owns it. phenaproxima tested the patch against a real Drupal CMS install ("Byte") and measured a drop from 45 seconds to 26 seconds, which gave the team a concrete benchmark to cite.
