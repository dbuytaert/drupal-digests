## Summary

This is a policy decision with no immediate code changes. The Drupal core community agreed to move high-level conceptual API documentation out of `*.api.php` files and the Drupal.org handbook into Markdown files stored in the core repository, published via GitLab Pages. The `/docs` folder does not yet exist in core; the concrete implementation steps are tracked in the follow-up issue [#3582087](https://www.drupal.org/node/3582087).

## Impact

No upgrade or configuration changes required.

## Technical details

Drupal core currently stores high-level conceptual API documentation in `*.api.php` files such as `core/core.api.php` (which contains 28 `@defgroup` annotations), `core/lib/Drupal/Core/Entity/entity.api.php`, `core/lib/Drupal/Core/Render/theme.api.php`, and many others. These files use phpdoc syntax and are parsed by the API module to render pages on api.drupal.org, but the re-parse cycle can take up to a week. A parallel body of documentation lives in the Drupal.org handbook, where structural edits are difficult because content is stored in individual nodes, structural changes require elevated permissions, and there is no clear cultural invitation to make large rewrites.

The discussion evaluated four options: moving docs to the handbook, using an RST-based toolchain similar to Symfony's, adding a `/docs` folder with Markdown files published via a static site generator, or retiring the API module in favour of an external tool such as Doctum. The community converged on moving long-form conceptual prose, content not tightly coupled to specific PHP classes or functions, into Markdown files in a new `/docs` folder in core, published via GitLab Pages. Markdown was preferred over RST because it is already used for README files throughout the codebase, making it easy to move content between the two. Replacing the API module itself was explicitly scoped out as a separate and more complex topic.

The agreed path forward, outlined in [#3582087](https://www.drupal.org/node/3582087), is to convert one subsystem's docs first (the cache API was the example given), set up GitLab Pages publishing, refine the format, and then open issues for other subsystems. As conversions happen, the corresponding `*.api.php` high-level topics should be removed and handbook pages replaced with links or redirects, moving from documentation in two places to one.
