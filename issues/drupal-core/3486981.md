## Summary

Drupal recipes can now enable and configure Layout Builder for existing content types through config actions. Previously, attempting to enable Layout Builder via a recipe had no effect because the relevant methods on the view display entity were not exposed as config action targets. This capability is needed for Drupal CMS, which relies on Layout Builder-enabled view modes and per-entity overrides.

## Impact

Affects module maintainers.

- **New API:** Three new config action methods on `LayoutBuilderEntityViewDisplay`: `enableLayoutBuilder`, `disableLayoutBuilder`, and `allowLayoutOverrides` (exposed from `setOverridable`)

## Technical details

Layout Builder's configuration state for a view display is controlled by three methods on `LayoutBuilderEntityViewDisplay` (in `core/modules/layout_builder/src/Entity/LayoutBuilderEntityViewDisplay.php`): `enableLayoutBuilder`, `disableLayoutBuilder`, and `setOverridable`. None of these were exposed to the config action system, so recipe YAML could not invoke them. Calling `setThirdPartySettings` directly on a view display from a recipe also had no effect, leaving Layout Builder setup inaccessible to the recipe system.

The fix adds the `#[ActionMethod]` PHP attribute from `Drupal\Core\Config\Action\Attribute\ActionMethod` to all three methods, registering them with the config action plugin manager:

- `enableLayoutBuilder` is decorated with `#[ActionMethod(adminLabel: new TranslatableMarkup('Enable Layout Builder'), pluralize: FALSE)]` and is reachable under the action name `enableLayoutBuilder`.
- `disableLayoutBuilder` is decorated with `#[ActionMethod(adminLabel: new TranslatableMarkup('Disable Layout Builder'), pluralize: FALSE)]` and is reachable under the action name `disableLayoutBuilder`.
- `setOverridable` is decorated with `#[ActionMethod(adminLabel: new TranslatableMarkup('Toggle overridable layouts'), pluralize: FALSE, name: 'allowLayoutOverrides')]` and is reachable under the custom action name `allowLayoutOverrides` (the `name` parameter overrides the default method-name derivation to produce a more semantically clear action identifier).

A new kernel test class `ConfigActionsTest` in `core/modules/layout_builder/tests/src/Kernel/ConfigActionsTest.php` verifies all three actions by invoking `ConfigActionManager::applyAction()` and asserting the resulting state via `isLayoutBuilderEnabled()` and `isOverridable()`.

## Upgrade

Recipe authors can now enable Layout Builder and allow per-entity overrides on an existing view display using config actions, for example:

```yaml
config:
  actions:
    core.entity_view_display.node.page.default:
      enableLayoutBuilder: []
      allowLayoutOverrides: true
```

No changes are required to existing code outside of recipes.
