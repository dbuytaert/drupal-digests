## Summary

Recipes can now create config entities for every existing bundle of an entity type in a single, generic action. Before this change, doing something like adding content language settings or an image style to every node type required either a custom per-module config action or manual duplication. Two new recipe config actions, `createForEach` and `createForEachIfNotExists`, handle this pattern generically and work with any bundle entity type such as node types, media types, and taxonomy vocabularies. This is marked experimental API.

## Impact

Affects module maintainers.

- **New API:** New `createForEach` and `createForEachIfNotExists` config actions (experimental) allow recipes to create config entities for every bundle of an entity type using `%bundle` and `%label` placeholders.

## Technical details

The recipe system's config action subsystem gained a new plugin, `CreateForEachBundle` (`core/lib/Drupal/Core/Config/Action/Plugin/ConfigAction/CreateForEachBundle.php`), with plugin ID `create_for_each_bundle`. Its companion deriver, `CreateForEachBundleDeriver` (`core/lib/Drupal/Core/Config/Action/Plugin/ConfigAction/Deriver/CreateForEachBundleDeriver.php`), generates two derivative action IDs: `createForEach` (backed by `entity_create:create`) and `createForEachIfNotExists` (backed by `entity_create:createIfNotExists`). The deriver restricts these actions to entity types where `EntityTypeInterface::getBundleOf()` returns a string, that is, entity types that are themselves bundles of another entity type.

In a recipe's `config.actions` block, a wildcard target such as `node.type.*` combined with `createForEach` receives a map of config names to values. The plugin's `apply()` method loads each matched bundle entity, then replaces the `%bundle` placeholder (the bundle's machine ID) and `%label` placeholder (the bundle's human-readable label) throughout the value map before delegating each individual creation to the config action manager. The `%label` placeholder is only substituted in array values, not keys; only `%bundle` is substituted in keys. Each delegated call goes through `entity_create:create` or `entity_create:createIfNotExists`, so the usual config entity validation is enforced on every entity created.

A secondary bug fix was made to `EntityCreate::apply()`: the hardcoded `'id' => $id` was replaced with `$entity_type->getKey('id') => $id`, so the correct ID property key is used for any config entity type rather than always assuming it is named `id`.

The commit was landed on both 11.x (b03d9ab8) and 10.4.x (2013eb87). The 10.4.x backport uses `block_content` in the validation test instead of the `image` module due to a config schema difference between branches. Tests live in `WildcardConfigActionsTest` and cover normal creation, the `%label` placeholder, the non-overwriting `createForEachIfNotExists` variant, conflict errors from `createForEach`, rejection when the target entity type is not itself a bundle type, and a graceful error when no bundle entity types exist at all.

## Upgrade

No upgrade steps are required. This is a new feature addition. Both `createForEach` and `createForEachIfNotExists` are available to recipes in Drupal 10.4 and 11.1. Example recipe usage:

```yaml
config:
  actions:
    node.type.*:
      createForEach:
        language.content_settings.node.%bundle:
          target_entity_type_id: node
          target_bundle: '%bundle'
        image.style.node_%bundle_big:
          label: 'Big images for %label content'
```

The `%bundle` placeholder is replaced with the bundle machine ID in both keys and values. The `%label` placeholder is replaced with the bundle's human-readable label in values only.
