## Summary

The Database Logging module now uses a service for filter operations instead of procedural functions. This modernizes the codebase by moving legacy functions into a proper service class that can be dependency-injected. The old `dblog.admin.inc` include file is no longer needed and will be removed in the next major version. This change makes the code easier to test, maintain, and extend while keeping backward compatibility during the transition period.

## Impact

Affects module maintainers.

- **Deprecation:** Three `dblog` procedural functions moved to `DbLogFilters` service: `_dblog_get_message_types()` and `dblog_filters()` deprecated in Drupal 11.4, removed in Drupal 13; `dblog.admin.inc` file deprecated in 11.4, removed in Drupal 12

## Technical details

Two procedural functions that handled database log filtering have been moved into a new `DbLogFilters` service class. The `_dblog_get_message_types()` function becomes the `getMessageTypes()` method, and the `dblog_filters()` function becomes the `filters()` method. Both methods retain identical functionality.

The new service is registered in `dblog.services.yml` with autowiring enabled, allowing automatic dependency injection of the database connection. It uses `StringTranslationTrait` to handle translatable strings that were previously handled by the `t()` function.

Three classes now inject and use the new service: `DbLogController` injects it through constructor parameters, while `DblogFilterForm` and the `DblogTypes` Views plugin both add constructors to receive the service. The controller no longer needs to load `dblog.admin.inc` via `ModuleHandlerInterface::loadInclude()`.

The deprecated procedural functions now trigger `E_USER_DEPRECATED` errors and immediately delegate to the new service methods via `\Drupal::service()`. The `dblog.admin.inc` file itself triggers a deprecation error when loaded. All deprecated code will be removed in Drupal 13.0 (functions) or 12.0 (include file).

## Upgrade

If your custom code calls `_dblog_get_message_types()` or `dblog_filters()`, inject the `DbLogFilters` service and call the corresponding methods instead:

```php
// Before:
$types = _dblog_get_message_types();
$filters = dblog_filters();

// After:
$types = \Drupal::service(\Drupal\dblog\DbLogFilters::class)->getMessageTypes();
$filters = \Drupal::service(\Drupal\dblog\DbLogFilters::class)->filters();
```

In classes with dependency injection, add the service to your constructor parameters. If you manually include `dblog.admin.inc`, remove those includes and inject the service instead.

See [#3560399](https://www.drupal.org/node/3560399) for full details.

This issue took over two months from opening to commit, with eight contributors collaborating across 36 comments. The discussion explored whether to use a service or a trait for the refactored code, with contributors thoughtfully debating the tradeoffs. The team ultimately chose the service approach as simpler and more practical.

Early feedback questioned whether exposing the previously private `_dblog_get_message_types()` function as public API was appropriate. The group concluded the specific, stable functionality posed minimal risk. Reviewers provided detailed code suggestions that improved constructor formatting and fixed a copy-paste error in deprecation messages. Multiple rounds of environmental test failures required patience before getting clean pipeline results.

## Lessons learned

- When converting underscore-prefixed 'private' functions to services, making them public API is acceptable. Nothing truly prevented their use before, and if they're called by multiple parts of the system, the service pattern provides cleaner dependency injection without adding complexity like traits. *(via nicxvan, smustgrave, mstrelan)*
