## Summary

Entity loading in Drupal now uses one fewer database query per load operation. Previously, `SqlContentEntityStorage` issued a separate query for the shared data or revision table and another query that joined dedicated field tables. These are now merged into a single query, so entities load in at most two or three queries in all cases (down from potentially over a hundred before this and the two preceding issues [#3551308](https://www.drupal.org/node/3551308) and [#3564689](https://www.drupal.org/node/3564689)). Performance tests show 3 to 7 fewer queries per request across Umami front-page, node-page, and JSON:API endpoints, with larger reductions on uncached requests where multiple entity loads occur.

## Impact

Affects module maintainers.

- **API change:** `SqlContentEntityStorage::loadFromDedicatedTables()` gains a required third parameter `array &$translations`; subclasses overriding this method must add it.
- **Deprecation:** `SqlContentEntityStorage::loadFromSharedTables()` deprecated in drupal:11.4.0 and removed in drupal:12.0.0 with no replacement; see https://www.drupal.org/node/3586362.

## Technical details

The core change is in `SqlContentEntityStorage::mapFromStorageRecords()`. The two sequential calls to `loadFromSharedTables()` and `loadFromDedicatedTables()` are replaced by a single call to `loadFromDedicatedTables()`, which now also handles all shared-table loading.

`loadFromDedicatedTables()` receives a new third parameter `array &$translations` so it can track language variants as it processes rows. The method builds its base query from the data table or revision table (whichever is appropriate), now selecting all columns via `->fields($base_table)` rather than only the ID. Because both the base table and dedicated field tables can have a `langcode` column, the base table langcode is aliased as `{table_name}__langcode` (stored in `$base_langcode_alias`) to prevent it being overwritten by dedicated-table columns in the result set. The previous validity filter against a pre-fetched `$langcodes` list from `LanguageManager::getLanguages()` is removed; the langcode of each row is used directly from the aliased column instead.

For entities that have both a `revisionDataTable` and a `dataTable`, the method now detects fields that are not revisioned (data fields) and, when loading a revision, LEFT JOINs the data table into the same query to pick up those values. Single cardinality dedicated fields continue to be LEFT JOINed onto this expanded base query, and the early-return guard that previously skipped the single-cardinality query when no dedicated storage definitions existed is removed, since the query now always runs to retrieve base-table fields.

`loadFromSharedTables()` is deprecated in `drupal:11.4.0` and removed in `drupal:12.0.0` with no replacement, per the change record at https://www.drupal.org/node/3586362.

## Upgrade

If you have a class that overrides `loadFromSharedTables()`, remove the override. The method is deprecated and its logic is now handled inside `loadFromDedicatedTables()`.

If you have a class that overrides `loadFromDedicatedTables()`, add the new third parameter:

```php
// Before
protected function loadFromDedicatedTables(array &$values, $load_from_revision) {

// After
protected function loadFromDedicatedTables(array &$values, $load_from_revision, array &$translations) {
```

If your override calls `parent::loadFromDedicatedTables()`, pass `$translations` through:

```php
parent::loadFromDedicatedTables($values, $load_from_revision, $translations);
```

### Collaboration

The langcode handling was the main difficulty catch noted explicitly: entities can have base, data, revision, and revision-data tables in various combinations, and the default-language and translation logic differs between shared and dedicated tables. Catch worked through several rounds of test failures across MySQL, PostgreSQL, SQLite, and MariaDB before getting everything green, observing that "it's not dramatically more complex than the logic it's replacing but it was hard to get right because there's so many combinations with different entity schemas." alexpott raised a specific concern during review about the `base_langcode` alias potentially clashing with a custom entity field also named `base_langcode`, and suggested a mitigation. Catch resolved it with what they described as "one alias rename" before the issue was committed.
