## Summary

Entity loading in Drupal now uses one fewer database query per load operation. Previously, `SqlContentEntityStorage` issued one query for the shared data/revision tables and a separate query for dedicated field tables. These two queries are now merged into one, reducing entity loading to at most two or three queries in all cases (down from potentially over a hundred before this and the two preceding issues). Performance tests show 3 to 7 fewer queries per request on various pages, with higher gains on uncached requests where multiple entity loads occur.

## Impact

Affects module maintainers.

- **API change:** `SqlContentEntityStorage::loadFromDedicatedTables()` gains a required third parameter `array &$translations`; subclasses overriding this method must add it
- **Deprecation:** `SqlContentEntityStorage::loadFromSharedTables()` deprecated in drupal:11.4.0, removed in drupal:12.0.0 with no replacement (see https://www.drupal.org/node/3586362)

## Technical details

The core change is in `SqlContentEntityStorage::mapFromStorageRecords()`. The two sequential calls to `loadFromSharedTables()` and `loadFromDedicatedTables()` are replaced by a single call to `loadFromDedicatedTables()`, which now also handles the work that `loadFromSharedTables()` previously performed.

`loadFromDedicatedTables()` receives a new third parameter, `array &$translations`, so it can track language variants as it processes rows. The method builds its base query from the data table or revision table (whichever is appropriate), now selecting all columns from that base table via `->fields($base_table)` rather than only the ID. Because the base table and dedicated field tables both have a `langcode` column, the base table langcode is aliased as `{table_name}__langcode` (stored in `$base_langcode_alias`) to prevent it from being overwritten by dedicated table columns in the result set.

For entities that have both a `revisionDataTable` and a `dataTable`, the method now also detects fields that are not revisioned (data fields) and, when loading a revision, LEFT JOINs the data table to pick up those values in the same query.

Single cardinality dedicated fields continue to be LEFT JOINed onto this now-expanded base query. The previous validity filter on `$langcodes` (a pre-fetched list of all language codes) is removed; instead, the langcode of each row is used directly from the aliased column.

`loadFromSharedTables()` is deprecated in `drupal:11.4.0` and removed in `drupal:12.0.0` with no replacement, per the change record at https://www.drupal.org/node/3586362.

This issue is the third in a series: [#3551308](https://www.drupal.org/node/3551308) merged single cardinality dedicated fields into one query, [#3564689](https://www.drupal.org/node/3564689) merged multiple cardinality fields into one query, and this issue eliminates the separate shared-table query entirely.

## Upgrade

If you have a class that overrides `loadFromSharedTables()`, remove the override. The method is deprecated and its logic is now handled inside `loadFromDedicatedTables()`.

If you have a class that overrides `loadFromDedicatedTables()`, add the new third parameter:

```php
// Before
protected function loadFromDedicatedTables(array &$values, $load_from_revision) {

// After
protected function loadFromDedicatedTables(array &$values, $load_from_revision, array &$translations) {
```

If your override calls `parent::loadFromDedicatedTables()`, pass `$translations` through:

```php
parent::loadFromDedicatedTables($values, $load_from_revision, $translations);
```

### Collaboration

The langcode handling was the main difficulty catch called out explicitly: entities can have base, data, revision, and revision-data tables in various combinations, and the default-language/translation logic differs between shared and dedicated tables. Catch worked through several rounds of failures (MariaDB being the last to pass), noting that the result was not dramatically more complex but was hard to get right. alexpott raised a specific concern during review about the `base_langcode` alias potentially clashing with a custom entity field named `base_langcode`, and suggested a mitigation. Catch resolved that with what they described as "one alias rename" before the issue was committed.
