## Summary

Drupal's `Upsert` database query class now supports composite unique or primary key constraints, meaning tables whose uniqueness is defined by more than one column can now use upsert queries. Previously, the class was limited to single-field keys, which prevented it from being used across most of Drupal core. This unblocks replacing slower multi-statement `Merge` queries with single-statement `Upsert` queries in more places, reducing database round-trips. Site owners should see a modest performance improvement wherever core previously fell back to separate SELECT/INSERT/UPDATE sequences.

## Impact

Affects module maintainers.

- **API change:** `Upsert::$key` property type changed from `string` to `string[]`; third-party drivers that access `$this->key` directly as a string must update to treat it as an array.
- **New API:** `Upsert::key()` now accepts `string|array`, enabling composite unique/primary key constraints in upsert queries across all three built-in database drivers (MySQL, PostgreSQL, SQLite).

## Technical details

The `Upsert` base class (`core/lib/Drupal/Core/Database/Query/Upsert.php`) stored the key column as a bare `string $key`. It is now stored as `string[] $key`. The `key()` method signature changed from an untyped `$field` to `string|array $field`, normalized internally with `(array) $field`, so existing single-string callers remain unaffected.

All three built-in driver `__toString()` implementations were updated in parallel. Each now calls `array_combine($insert_fields, $insert_fields)` immediately after merging default and insert fields, making the array addressable by field name so that key columns can be removed with `unset($insert_fields[$key])` inside a `foreach ($this->key as $key)` loop. The PostgreSQL and SQLite drivers build their `ON CONFLICT (...)` clause by mapping each key column through `escapeField()` and joining with commas. The MySQL driver similarly iterates the key array when excluding key columns from the `ON DUPLICATE KEY UPDATE` set list.

The SQLite driver (`core/modules/sqlite/src/Driver/Database/sqlite/Upsert.php`) also received a positional-index correctness fix: `$insert_values = array_values($insert_values)` is prepended to the blob-field processing loop so that numeric offsets align with `$this->insertFields` when values are passed as associative arrays.

A new kernel test `testCompositeKeyUpsert()` in `core/tests/Drupal/KernelTests/Core/Database/UpsertTest.php` exercises insert, conditional update, and same-name-different-age scenarios against a `test_composite_primary` table whose primary key spans `(name, age)`.

Third-party database driver maintainers whose drivers override `Upsert::__toString()` and access `$this->key` directly as a string must update to treat it as `string[]`. A follow-up issue ([#3549762](https://www.drupal.org/node/3549762)) tracks a potential future deprecation of `Merge` queries in favour of `Upsert`.

## Upgrade

Callers that pass a single string to `Upsert::key()` require no change:

```php
// Still works as before.
$connection->upsert('my_table')
  ->key('id')
  ->fields(['id', 'value'])
  ->values([...])
  ->execute();
```

To use composite keys, pass an array:

```php
$connection->upsert('my_table')
  ->key(['col_a', 'col_b'])
  ->fields(['col_a', 'col_b', 'value'])
  ->values([...])
  ->execute();
```

Third-party database drivers that override `Upsert::__toString()` and access `$this->key` as a string must update to iterate it as an array. A typical update looks like:

```php
// Before:
unset($insert_fields[$this->key]);
$query .= ' ON CONFLICT (' . $this->connection->escapeField($this->key) . ') DO UPDATE SET ...';

// After:
foreach ($this->key as $key) {
    unset($insert_fields[$key]);
}
$keys = array_map(fn($k) => $this->connection->escapeField($k), $this->key);
$query .= ' ON CONFLICT (' . implode(', ', $keys) . ') DO UPDATE SET ...';
```

### Collaboration

The issue was opened in 2015 but closed immediately because SQLite lacked real UPSERT support. After the minimum SQLite version was raised, amateescu reopened it in 2020; the patch then sat dormant for years. In 2025, two design discussions shaped the final commit. mondrake proposed overloading `key()` to accept `string|array` rather than adding a separate `keyColumns()` method, arguing this keeps the API simpler for callers who already use single-field upserts; amateescu agreed and updated the MR accordingly. A parallel debate with "ghost of drupal past", who argued that Upsert should be folded into Merge rather than kept as a separate API, was ultimately set aside by amateescu and mondrake, who chose to keep both classes and deferred any Merge evolution to a separate issue. longwave then extracted a minimal MR (MR!15248) focused strictly on the composite-key change, stripping out ancillary refactors, which was the version alexpott committed.
