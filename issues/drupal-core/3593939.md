## Summary

Invokable controllers that place a `#[Route]` attribute only on the class, leaving `__invoke()` without its own attribute, silently failed to register any routes. Visiting the declared path returned a 404 with no error. This fix makes the documented pattern work as expected on all active branches (11.4.x, 11.x, and main).

## Impact

No upgrade or configuration changes required.

## Technical details

The bug was in `AttributeRouteDiscovery::getRoutes()`. The fallback condition that handles invokable controllers read `if ($collection->count() && $class->hasMethod('__invoke') === 0)`. This had two errors. First, `ReflectionClass::hasMethod()` returns `bool`, so `=== 0` (strict comparison with an integer) is always `false`, making the entire block dead code. Second, the polarity of `$collection->count()` was wrong: the block is meant to run when no method-level `#[Route]` attributes were found, so the correct guard is `!$collection->count()`.

The corrected condition is `if (!$collection->count() && $class->hasMethod('__invoke'))`. The code fix was first applied to `main` and `11.x` incidentally as part of [#3584793](https://www.drupal.org/node/3584793), and was then backported explicitly to `11.4.x`. The `main` commit (`b599308d`) covered by this issue adds dedicated test coverage: a new `TestClassAttributeClassOnly` controller in `router_test` and a `testClassOnlyRouteWithInvoke()` unit test in `AttributeRouteDiscoveryTest` that asserts both the route path and its `::__invoke` alias are registered correctly.

## Upgrade

No API changes. If you worked around this bug by duplicating the `#[Route]` attribute on `__invoke()` alongside the class-level attribute, you can remove the redundant method-level attribute. The class-only form now works as documented:

```php
#[Route(
  path: '/my-path',
  name: 'my_route',
  requirements: ['_access' => 'TRUE'],
)]
class MyController extends ControllerBase {
  public function __invoke(): array { /* ... */ }
}
```
