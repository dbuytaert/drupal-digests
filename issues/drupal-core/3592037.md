## Summary

`Settings::get()` previously triggered a deprecation warning every time a deprecated setting was requested, even if the setting was never configured in `settings.php` and had no replacement key. This caused spurious deprecation failures across many unrelated test classes, because simply instantiating a service that internally calls `Settings::get()` for a deprecated setting was enough to fail a test. After this fix, the warning is only emitted when there is actually something actionable: either a replacement key exists (the calling code must be updated) or the setting is present in `settings.php` (the site owner must remove it). Deprecated settings with no replacement that are simply absent from `settings.php` are silently ignored.

## Impact

No upgrade or configuration changes required.

## Technical details

In `Settings::get()`, the unconditional `@trigger_error()` call is now wrapped in a condition: `!empty($deprecation['replacement']) || array_key_exists($name, self::$instance->storage)`. The first branch catches the case where calling code must be updated to use a new key; the second catches the case where a site owner has the legacy setting in `settings.php` and must remove it. Settings that are deprecated with an empty `replacement` and are absent from storage fall through both branches without any warning.

`Settings::handleDeprecations()`, which runs at bootstrap when `settings.php` is loaded, was also hardened with the same `!empty($deprecation['replacement'])` guard before it copies values from the legacy key to the replacement key and before it back-syncs them. Without this guard, a `NULL`/empty replacement key would have been written into `$settings`.

Two tests were added to `SettingsTest`: `testGetNoDeprecationWhenNotConfiguredAndNoReplacement()` verifies no deprecation fires for an absent no-replacement setting, and `testGetDeprecationWhenConfiguredAndNoReplacement()` (annotated `#[IgnoreDeprecations]`) verifies the warning still fires when the setting is present in storage. The practical trigger for this fix was the `update_fetch_with_http_fallback` setting (see related issue [#3591513](https://www.drupal.org/node/3591513)), which is deprecated with no replacement and was polluting the output of every kernel test that instantiated `UpdateFetcher` or `SecurityAdvisoriesFetcher`.
