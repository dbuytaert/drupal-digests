# #3512962: Add validation to field.storage.*.* indexes

## Summary

Field storage index definitions in `field.storage.*.*` configuration previously used `type: ignore` in the config schema, meaning their structure was never validated. This change adds a `FieldStorageIndexes` config schema constraint that validates the shape of the `indexes` array: each entry must have a non-empty string key (the index name), and each column definition within that entry must be either a non-empty string (a bare column name) or a two-element array of `[column_name, positive_integer_length]`. This matches the format required by the Database Schema API and enforced at the database level. Existing valid field storage configs are unaffected.

## Impact

Affects module maintainers.

- **New API:** New `FieldStorageIndexes` config schema constraint validates the structure of `field.storage.*.*` index definitions against the Database Schema API's expected array shape.

## Technical details

The `indexes` key in `field.storage.*.*` (in `field.schema.yml`) had its inner sequence typed as `type: ignore`, providing zero structural validation. The fix adds a `FieldStorageIndexes: ~` constraint declaration directly on the outer `indexes` sequence.

Two new classes are added under `Drupal\field\Plugin\Validation\Constraint`. `FieldStorageIndexesConstraint` extends `SymfonyConstraint` and is tagged with `#[Constraint(id: 'FieldStorageIndexes', type: ['sequence'])]`. Its validator `FieldStorageIndexesConstraintValidator` enforces:

1. NULL or empty array passes (the field is optional).
2. The value must be an array; non-arrays trigger `$constraint->message`.
3. Each key must be a non-empty string (index name); otherwise `invalidIndexNameMessage`.
4. Each value must be a PHP list array (column definitions); otherwise `invalidIndexMessage`.
5. Each column definition is valid if it is a non-empty string, or a two-element list `[column_name, length]` where `column_name` is a non-empty string and `length` is a positive integer; other shapes trigger `invalidColumnMessage` or `invalidColumnLengthMessage`.

This shape exactly mirrors what `Schema::addIndex()` documents and what `mysql\Driver\Database\mysql\Schema::createKeySql()` implements: each column is either a bare name string or a `(name, length)` tuple. The `(string, int)` tuple cannot be expressed declaratively in config schema type definitions alone, which is why a custom constraint is required rather than a purely schema-driven approach (confirmed by longwave in the commit comment).

A kernel test `FieldStorageIndexesValidationTest` (in `Drupal\Tests\field\Kernel\Config`) covers a wide matrix of valid and invalid inputs. Index name and column name content are intentionally not validated against actual database column lists, as that was considered out of scope. The `MappingCollectionConstraint` introduced in [#3535734](https://www.drupal.org/node/3535734) was considered but a more focused custom constraint was chosen for simplicity.
