## Summary

Drupal core templates no longer include redundant WAI-ARIA role attributes on HTML5 semantic elements. Modern browsers automatically expose the correct accessibility roles for elements like `<main>`, `<nav>`, `<aside>`, `<header>`, and `<footer>`, making explicit role attributes unnecessary. This change resolves HTML validation warnings and cleans up the generated markup without affecting accessibility. The attributes were kept for years specifically to support Internet Explorer 11, which Drupal 10+ no longer supports.

## Impact

No upgrade or configuration changes required.

## Technical details

This change removes explicit `role` attributes from HTML5 semantic landmark elements throughout Drupal core's Twig templates. The HTML5 specification defines implicit ARIA landmark roles: `<main>` maps to `role="main"`, `<nav>` to `role="navigation"`, `<aside>` to `role="complementary"`, `<header>` to `role="banner"`, and `<footer>` to `role="contentinfo"`. All modern browsers pass these implicit roles to accessibility APIs, making the explicit attributes redundant markup that also triggers HTML validation warnings.

The attributes were deliberately kept until Drupal dropped IE11 support. As andrewmacpherson documented in the thread, IE11 did not pass implicit ARIA landmark roles to the host OS accessibility APIs, meaning users of IE11 with a screen reader like NVDA would lose landmark navigation without the explicit `role` attributes.

The commit touches 42 files. On the template side, the affected files span system module base templates (`page.html.twig`, `maintenance-page.html.twig`, `install-page.html.twig`, `breadcrumb.html.twig`, `pager.html.twig`, `block--system-menu-block.html.twig`), the toolbar module's `toolbar.html.twig`, the views module's `views-mini-pager.html.twig`, and their overrides in the Claro, Olivero, Umami (demo_umami profile), and starterkit themes. Claro-specific templates also updated include `menu-local-tasks.html.twig`, `maintenance-page--front.html.twig`, and multiple toolbar template variants. Olivero's `menu-local-tasks.html.twig` and both themes' `block--local-tasks-block.html.twig` templates are included as well.

Beyond templates, Claro's `core/themes/claro/js/mobile.install.js` was updated from `document.querySelector('header[role="banner"]')` to `document.querySelector('header')`, and both `core/themes/claro/css/theme/maintenance-page.css` and `maintenance-page.pcss.css` had the CSS attribute selector `header[role="banner"]` replaced with `header`. Test assertions in `BlockTest.php`, `BlockUiTest.php`, and `CommentTypeTest.php` that used XPath or CSS selectors targeting role attributes were also updated.

The `stable9` theme was intentionally excluded. As longwave noted and bnjmnm (a framework manager) confirmed, changing `stable9` could break CSS or JavaScript in custom themes that rely on the role attribute as a CSS or JS selector, and the redundancy does not cause any user-facing or assistive technology regression.

## Upgrade

If your custom theme overrides any of the affected core templates and your CSS or JavaScript selects elements using the now-removed role attributes, update those selectors to use the element name alone:

```css
/* Before */
header[role="banner"] { ... }

/* After */
header { ... }
```

```javascript
// Before
const header = document.querySelector('header[role="banner"]');

// After
const header = document.querySelector('header');
```

The same applies to XPath selectors used in tests, e.g. `//header[@role = "banner"]` becomes `//header`. If your theme extends `stable9`, no change is needed as that theme retains the attributes for backward compatibility.
