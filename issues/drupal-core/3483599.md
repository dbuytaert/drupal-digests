## Summary

All procedural hook implementations in Drupal core have been converted to OOP `#[Hook]` attribute classes, landing in Drupal 11.1. Before the change, core shipped 241 `.module` files and 69 `.inc` files containing hook functions; after, those counts dropped to 99 and 39 respectively. The logic of each hook is unchanged. Sites running only standard modules will see no functional change. Module developers who were directly calling core's procedural hook implementation functions (for example `block_rebuild()` or `announcements_feed_cron()`) will find those functions gone and must update their code.

## Impact

No upgrade or configuration changes required.

## Technical details

Drupal core's OOP hook system, introduced in Drupal 11, allows hook implementations to be declared as methods on plain PHP classes annotated with the `#[Hook('hook_name')]` attribute from `Drupal\Core\Hook\Attribute\Hook`. Before this issue, core itself still delivered nearly all hook implementations as global procedural functions in `.module` and `.inc` files. This issue performed a bulk automated conversion of those functions using a custom Rector rule applied to the entire `core/` directory.

The conversion pattern is uniform across all converted modules. For each module (e.g. `announcements_feed`), a new class is created at `src/Hook/{ModuleNameInPascalCase}Hooks.php` in the namespace `Drupal\{module_name}\Hook`. Each former `function {module}_{hook_name}(...)` becomes a `public function {camelCaseHookName}(...)` method annotated with `#[Hook('{hook_name}')]`. The body of each function is preserved verbatim. `.module` files that contained only hook implementations were deleted entirely; those that also contained non-hook procedural helpers kept only the non-hook code.

Several hooks were explicitly excluded from conversion: `system_page_attachments` was renamed to `_system_page_attachments` (a private-by-convention helper), and `BareHtmlPageRenderer` was updated to call the renamed function, because it is invoked directly during bare-page rendering and Drupal installation. `system_theme` was similarly excluded. `hook_module_implements_alter` must remain procedural because it is consulted during hook discovery build time. Test modules that specifically test procedural hook behavior (`deprecation_test`, `module_test`, `module_handler_test_all1`, `module_handler_test_all2`) and the `empty_file.php.module` fixture were left unchanged.

Several prerequisite issues were merged into this branch before the final commit: [#3482618](https://www.drupal.org/node/3482618) (fix deprecated OOP hook handling in `ModuleHandler::alterDeprecated`), [#3486462](https://www.drupal.org/node/3486462) (support `#[Hook]` for hooks called via `ModuleInstaller::invokeAll()`), [#3484754](https://www.drupal.org/node/3484754) (ordering of hooks and BC for `hook_module_implements_alter`), and documentation corrections in [#3485690](https://www.drupal.org/node/3485690) and [#3485470](https://www.drupal.org/node/3485470). The Drupal BC policy explicitly excludes hook implementation functions from the backwards-compatibility promise, so removing the procedural functions without a deprecation layer was intentional. Retaining them via `#[LegacyHook]` was considered and rejected due to the memory cost of continued hook discovery on `.module` files.

## Upgrade

If your custom or contrib module directly calls any of Drupal core's old procedural hook implementation functions, those functions no longer exist. Replace direct calls with the appropriate hook invocation via the module handler, or call the relevant service directly.

```php
// Before (calling a core hook implementation function directly):
block_rebuild();

// After (invoke the hook properly, or call the class method directly):
\Drupal::moduleHandler()->invokeAll('rebuild');

// Or, as tests now do, instantiate the hook class directly:
use Drupal\block\Hook\BlockHooks;
(new BlockHooks())->rebuild();
```

If you were calling `system_page_attachments($page)` directly, the function is now `_system_page_attachments($page)` (an internal helper; prefer invoking `hook_page_attachments` via the module handler instead).

No changes are required for modules that only *implement* hooks or that invoke hooks through `ModuleHandler`. The OOP hook system is already supported in Drupal 11, and your own `#[Hook]` classes continue to work exactly as before.
