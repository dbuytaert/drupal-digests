## Summary

Three procedural functions that tracked locale translation import history have been deprecated in `locale.module` in favor of the new `CurrentImportStorage` service. The old approach loaded every row in the `locale_file` table into a static cache each time the history was read, and invalidated that entire cache after any single update. On sites with many languages and projects this caused severe performance problems: heddn reported calls to these functions consuming around 96% of execution time, with cron imports taking close to two hours for 8-9 languages. The new service fetches and caches one project/language record at a time and only invalidates individual entries on write, eliminating the full-reload loop. Site owners with large multi-language installations should see a meaningful improvement in translation update cron performance.

## Impact

Affects module maintainers.

- **Deprecation:** Three `locale.module` functions deprecated: `locale_translation_get_file_history()`, `locale_translation_update_file_history()`, and `locale_translation_file_history_delete()`; replaced by the `@internal` `CurrentImportStorage` service.

## Technical details

The root cause was `locale_translation_get_file_history()` using `drupal_static()` with no selectivity, always fetching every row of `locale_file`. `locale_translation_update_file_history()` then called `drupal_static_reset('locale_translation_get_file_history')` after each merge, so a cron run or install with N projects × M languages would reload the full table N×M times.

The replacement is `CurrentImportStorage`, an `@internal` service registered as `Drupal\locale\CurrentImportStorage`. Its `get(string $project, string $langcode)` method queries one row by project+langcode and stores the result in `cache.memory` under the key `locale_current_import:{project}:{langcode}` tagged with `locale_current_import`. `save(CurrentImport $currentImport)` runs a single `merge()` and then writes that same cache entry directly, avoiding any invalidation of unrelated records. `delete()` runs a filtered `DELETE` and then invalidates all entries via the `locale_current_import` cache tag. A new `getOutdatedImports()` method encapsulates the cron query and `last_checked` timestamp update that were previously inlined in `LocaleHooks`.

The cron implementation was extracted from `LocaleHooks` into a dedicated `LocaleCronHooks` class so that `CurrentImportStorage`, `LocaleFetch`, and other dependencies can be properly constructor-injected rather than pulled from the service container procedurally.

The typed `CurrentImport` value object (also `@internal`) replaces the untyped `stdClass` that the functions previously passed around. It carries `project`, `langcode`, `version`, `hash`, `timestamp`, and `last_checked`, and provides three named constructors: `createFromFile(LocaleFile $file)`, `createFromSource(object $source)`, and `createFromArray(array $result)`.

The three deprecated functions (`locale_translation_get_file_history()`, `locale_translation_update_file_history()`, `locale_translation_file_history_delete()`) still exist in `locale.module` emitting `E_USER_DEPRECATED` notices as of drupal:11.4.0 and are removed in drupal:13.0.0. `drupal_static_reset('locale_translation_get_file_history')` is likewise deprecated and redirected to `cache.memory` tag invalidation. The issue body notes there are zero usages of these functions in contrib. Related work: [#3037031](https://www.drupal.org/node/3037031) (converting `locale.compare.inc` to a service) was a prerequisite that landed first.

## Upgrade

If custom code calls any of the deprecated functions, replace as follows. Note that `CurrentImport` and `CurrentImportStorage` are marked `@internal` and may change in future releases.

```php
// Before: load all history, then access by project/langcode
$history = locale_translation_get_file_history();
$record = $history[$project][$langcode];

// After: fetch one record directly
$currentImport = \Drupal::service(\Drupal\locale\CurrentImportStorage::class)
  ->get($project, $langcode);
// Returns a \Drupal\locale\CurrentImport|null
```

```php
// Before:
locale_translation_update_file_history($source);

// After:
\Drupal::service(\Drupal\locale\CurrentImportStorage::class)
  ->save(\Drupal\locale\CurrentImport::createFromSource($source));
```

```php
// Before:
locale_translation_file_history_delete($projects, $langcodes);

// After:
\Drupal::service(\Drupal\locale\CurrentImportStorage::class)
  ->delete($projects, $langcodes);
```

```php
// Before:
drupal_static_reset('locale_translation_get_file_history');

// After:
\Drupal::service('cache.memory')
  ->invalidateTags(['locale_current_import']);
```

berdir and nicxvan pair-programmed the final design. berdir did live profiling on a real distribution with many languages, documenting the load-all → update → invalidate-all → load-all loop in detail (comment #43), and confirmed that per-record fetching and caching would address the main bottleneck.
