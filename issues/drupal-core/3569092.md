## Summary

This change cleans up Drupal's validation constraint system to align with Symfony 8 compatibility. The `#[HasNamedArguments]` attribute, deprecated in Symfony 7.4 and removed in Symfony 8, has been stripped from all ~35 constraint classes across core. At the same time, the `addConstraint()` method signatures throughout the typed data, entity, field config, and context subsystems have been tightened with proper PHP type declarations. Site owners are unaffected directly, but contrib and custom modules that pass non-string constraint names, non-associative arrays as options, or rely on the old `'value'` configuration key in `PluginExistsConstraint` will need to update their code.

## Impact

Affects module maintainers.

- **API change:** `addConstraint()` on `DataDefinitionInterface`, `EntityTypeInterface`, `FieldConfigInterface`, and `ContextDefinitionInterface` now enforces `string $constraint_name` and `?array $options` types; `ConstraintManager::create()` and `ConstraintFactory::createInstance()` now throw `\InvalidArgumentException` for sequential (list) arrays; the deprecated `'value'` configuration key in `PluginExistsConstraint::create()` is removed.
- **Deprecation:** BC layer in `ConstraintFactory` for `HasNamedArguments`-based constructor dispatch removed; BC layer in `PluginExistsConstraint::create()` for the `'value'` configuration key (deprecated drupal:11.4.0) removed.

## Technical details

This issue follows the groundwork laid in [#3555534](https://www.drupal.org/node/3555534), which deprecated passing non-associative arrays to constraint constructors. The work here finalizes that cleanup and removes the BC layer that was bridging to Symfony 8.

**`#[HasNamedArguments]` removal.** Symfony's `HasNamedArguments` attribute was used on constraint constructors to tell Drupal's `ConstraintFactory` to spread an associative configuration array as named constructor arguments. Because this attribute is removed in Symfony 8, all ~35 constraint classes (spread across `core/lib/Drupal/Core/Validation/Plugin/`, `core/lib/Drupal/Core/Entity/Plugin/Validation/`, `core/modules/ckeditor5/`, `core/modules/comment/`, `core/modules/content_moderation/`, and many others) had their `use` import and `#[HasNamedArguments]` annotation dropped.

**`ConstraintFactory` simplification.** `ConstraintFactory::createInstance()` previously used reflection to detect `HasNamedArguments` and conditionally spread or wrap the configuration array. That entire block is replaced by a single `return new $plugin_class(...$configuration)`, which always spreads the (now always-associative) configuration. A guard using `array_is_list($configuration)` is added at the top to throw `\InvalidArgumentException` if a sequential array is mistakenly passed.

**`ConstraintManager::create()` cleanup.** The method signature gains a `string` type declaration for `$name`. The old BC path that accepted any non-array `$options` and wrapped it in `['value' => $options]` is removed. Instead, a `array_is_list()` guard now throws `\InvalidArgumentException` for sequential arrays. `NULL` options are normalized to `[]` before being forwarded to `createInstance()`.

**`addConstraint()` signature finalization.** Four interfaces and their concrete classes had `addConstraint()` signatures with the type declarations commented out pending drupal:12.0.0 (referencing https://www.drupal.org/node/3554746): `DataDefinitionInterface` / `DataDefinition`, `EntityTypeInterface` / `EntityType`, `FieldConfigInterface` / `FieldConfigBase`, and `ContextDefinitionInterface` / `ContextDefinition`. The comment scaffolding (`phpcs:disable`, `@todo`, commented-out `/* ?array */`) is removed and the signatures become `addConstraint(string $constraint_name, ?array $options = NULL): static` in all cases.

**`DataDefinitionInterface` doc example updated.** The inline code example in the `getConstraints()` doc block previously showed a scalar shorthand (`'EntityType' => 'node'`). It is updated to the now-required associative form (`'EntityType' => ['type' => 'node']`).

**`EntityDataDefinition::getEntityTypeId()` simplified.** The fallback branch that read the constraint value directly as a scalar (`$this->definition['constraints']['EntityType'] ?? NULL`) is removed; only the associative `['type']` key is now read.

**`PluginExistsConstraint::create()` BC layer removed.** The code that mapped a `'value'` key in `$configuration` to `'manager'` (deprecated in drupal:11.4.0 per https://www.drupal.org/node/3554746) is dropped.

**`ExtensionNameConstraint` simplified.** The class previously accepted a required `$pattern` parameter but immediately overwrote it with `ExtensionDiscovery::PHP_FUNCTION_PATTERN`. That parameter is removed; the constant is passed directly to `parent::__construct()`.

**`OptionalConstraint` and `RequiredConstraint` cleaned up.** Both had constructors that only called `parent::__construct(...$args)` with `#[HasNamedArguments]`. With the attribute gone and no other logic present, the constructors are removed entirely so the parent constructors are inherited directly.

## Upgrade

If you call `addConstraint()` on any typed data definition, entity type, field config, or context definition object, ensure the first argument is a string and the second is either `NULL` or an associative (non-sequential) array:

```php
// Before (scalar shorthand, no longer supported):
$definition->addConstraint('EntityType', 'node');

// After (associative array required):
$definition->addConstraint('EntityType', ['type' => 'node']);
```

If you call `ConstraintManager::create()` or `ConstraintFactory::createInstance()` with a sequential (list) array, an `\InvalidArgumentException` is now thrown. Pass an associative array or `NULL` instead.

If you relied on the deprecated `'value'` key in `PluginExistsConstraint`'s `create()` configuration, switch to the `'manager'` key:

```php
// Before (deprecated, now removed):
$configuration = ['value' => 'my_plugin_manager'];

// After:
$configuration = ['manager' => 'my_plugin_manager'];
```

Custom constraint classes that carried the `#[HasNamedArguments]` attribute can safely remove the attribute and its `use` import. No other change is needed as long as the constructor already uses named parameters matching the constraint's option array keys.
