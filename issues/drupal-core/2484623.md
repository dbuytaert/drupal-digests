## Summary

All JavaScript files that were sitting loose in core module root folders have been moved into a `js/` subdirectory, matching the convention already used for CSS files. The affected modules include comment, content_translation, file, filter, language, locale, menu_ui, node, path, taxonomy, text, and user, as well as several test support modules. A backwards-compatibility layer in each module's `.libraries.yml` maps old JS file paths to the new ones, so existing themes and modules that reference the old paths via `libraries-override` or `hook_library_info_alter()` will receive a deprecation warning rather than silently break. The old paths are deprecated as of Drupal 11.1.0 and will be removed in Drupal 12.0.0.

## Impact

Affects module maintainers.

- **Deprecation:** Old module-root JS paths (e.g., `comment-entity-form.js`) deprecated in favour of `js/` subdirectory paths (e.g., `js/comment-entity-form.js`) in `drupal:11.1.0`, removed in `drupal:12.0.0`

## Technical details

Before this change, most core modules placed JavaScript files directly in the module root (e.g., `core/modules/comment/comment-entity-form.js`), while CSS had already been consolidated under a `css/` subdirectory. The toolbar module had long used a `js/` subdirectory as the intended model.

The change was originally proposed in 2015 and reached RTBC quickly, but was postponed by core committers because `libraries-override` entries in `.info.yml` files and `hook_library_info_alter()` hooks reference asset paths by string. Moving files would silently break any theme or module that had overridden or replaced one of these library assets. Wim Leers confirmed this concern; catch proposed a backwards-compatibility mapping layer as the unblocking path.

That BC mechanism was implemented separately in [#3432601](https://www.drupal.org/node/3432601). It introduces a `moved_files` key inside library definitions in `.libraries.yml`. Each entry under `moved_files` declares: the old module-relative asset path as the key, the new path in a `js:` sub-map, a `deprecation_version`, a `removed_version`, and a `deprecation_link`. When the asset library system encounters a `libraries-override` or alter that targets a deprecated old path, it resolves the override to the new file and logs a deprecation notice.

The commit (e9a2e9ab, authored 2024-09-13, on 11.x) renames all loose JS files into `js/` subdirectories and updates every corresponding `.libraries.yml` to point to the new path, while adding a `moved_files` block for each. For example, in `core/modules/comment/comment.libraries.yml`:

```yaml
drupal.comment:
  js:
    js/comment-entity-form.js: {}
  moved_files:
    comment/drupal.comment:
      deprecation_version: drupal:11.1.0
      removed_version: drupal:12.0.0
      deprecation_link: https://www.drupal.org/node/3471539
      js:
        comment-entity-form.js: 'js/comment-entity-form.js'
```

Test-only JS files were moved with the same pattern: `core/modules/locale/tests/js/locale_test.js`, `core/modules/system/tests/modules/twig_theme_test/js/twig_theme_test.js`, and `core/modules/views/tests/modules/views_test_data/js/views_cache.test.js`. The corresponding functional test classes (`LocaleJavascriptTranslationTest`, `EngineTwigTest`) and the Twig template `twig_theme_test.file_url.html.twig` were updated to reference the new paths. `core/.eslintignore` was also updated for the moved `locale_test.js`.

## Upgrade

Any theme or module that references a core JS asset by its old module-root path in `libraries-override` or in a `hook_library_info_alter()` implementation must update those references to use the new `js/` subdirectory path. For example:

```yaml
# Before (generates deprecation notice on Drupal 11.1+, breaks on Drupal 12)
libraries-override:
  drupal.comment:
    js:
      comment-entity-form.js: false

# After
libraries-override:
  drupal.comment:
    js:
      js/comment-entity-form.js: false
```

The same pattern applies to all affected modules: comment, content_translation, file, filter, language, locale, menu_ui, node, path, taxonomy, text, and user. Old paths are deprecated as of `drupal:11.1.0` and will be removed in `drupal:12.0.0`. The change record at https://www.drupal.org/node/3471539 lists all affected paths.

### Collaboration

This issue was open for nine years primarily because of a backwards-compatibility obstacle that no one could agree how to resolve. Core committer xjm postponed it to 9.x in 2015 after Wim Leers pointed out that `libraries-override` entries use literal file paths, so moving the files would silently break any theme overriding one of these assets. catch followed up in 2016 and proposed a hardcoded mapping of old paths to new as a BC layer, which he acknowledged would be "ugly" but necessary given the internal-API status of asset paths.

That idea sat dormant until catch opened [#3432601](https://www.drupal.org/node/3432601) in 2024 to implement the `moved_files` deprecation mechanism properly. Once that was in place, quietone converted the long-stalled patch to a merge request (!9251) in August 2024, resolved conflicts in `filter.libraries.yml`, `field_ui.libraries.yml`, `LocaleJavascriptTranslationTest.php`, and `.eslintignore`, and added the `moved_files` deprecation blocks to each `.libraries.yml`. nod_ reviewed in the MR and confirmed that formal test coverage was no longer needed given the existing deprecation process. alexpott committed to 11.x on September 13, 2024.
