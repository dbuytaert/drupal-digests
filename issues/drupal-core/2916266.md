# #2916266: How to fix "non-existent config entity name returned by FieldStorageConfigInterface::getBundles()"

## Summary

Site owners who see the watchdog error "A non-existent config entity name returned by FieldStorageConfigInterface::getBundles()" have a stale entry in the `entity.definitions.bundle_field_map` key-value store. This most commonly occurs after Drupal 6 or 7 to 8/9 migrations, module uninstallation, or configuration import operations that bypass the normal field deletion lifecycle. The issue thread documents the community-developed workaround: a `hook_update_N()` function that removes the orphaned bundle reference from the key-value store. This is a support request with no Drupal core code change; the thread serves as a reference for site owners encountering the error.

## Impact

No upgrade or configuration changes required.

## Technical details

The error message is logged in `FieldViewsDataProvider` (`core/modules/views/src/FieldViewsDataProvider.php`, line 119). It fires during Views field data collection when `FieldStorageConfig::getBundles()` returns a bundle name that no longer resolves to a valid config entity. `getBundles()` reads from the field map via `EntityFieldManager::getFieldMap()`, which is ultimately backed by the `entity.definitions.bundle_field_map` key-value collection.

That collection is maintained by `FieldDefinitionListener` (`core/lib/Drupal/Core/Field/FieldDefinitionListener.php`). Its `onFieldDefinitionCreate()` method adds a bundle entry, and `onFieldDefinitionDelete()` removes it. When these listeners do not fire — for example during a config import that deletes a field storage config, during module uninstallation without a proper field cleanup, or after a D6/D7-to-D8 migration that leaves orphaned field config — the stale bundle names remain in the store permanently and trigger the logged error on every cache rebuild.

Comment #9 (acbramley) identified the specific trigger: `FieldDefinitionListener::onFieldDefinitionDelete` does not fire when field config is deleted through the configuration synchronization pipeline.

Related issue [#2985907](https://www.drupal.org/node/2985907) improved the error message in Drupal 8.6.x to include the entity type in the log entry, making the affected entity discoverable without having to query the database directly.

The `FieldStorageConfigListBuilder::buildRow()` method also calls `getBundles()` when rendering the field list at `/admin/reports/fields`, producing a secondary `Undefined index` PHP notice for the same stale bundles (visible in several comments in the thread).

## Upgrade

To remove a stale bundle reference from the key-value store, add a `hook_update_N()` function to any enabled custom module. Substitute the correct `$entity_type`, `$bundle`, and `$field_name` values from the watchdog log entry:

```php
function mymodule_update_8901() {
  // Fixes: A non-existent config entity name returned by
  // FieldStorageConfigInterface::getBundles(): entity type: node,
  // bundle: old_bundle, field name: field_example
  $entity_type = 'node';
  $bundle = 'old_bundle';
  $field_name = 'field_example';

  /** @var \Drupal\Core\KeyValueStore\KeyValueFactoryInterface $key_value_factory */
  $key_value_factory = \Drupal::service('keyvalue');
  $field_map_kv_store = $key_value_factory->get('entity.definitions.bundle_field_map');
  $map = $field_map_kv_store->get($entity_type);
  unset($map[$field_name]['bundles'][$bundle]);
  $field_map_kv_store->set($entity_type, $map);
}
```

Then run `drush updb` (or visit `/update.php`). Repeat with an incremented `N` for each additional stale entry. The update function number must be higher than any previously run update for that module. On Drupal 8.9.x use `_8901`, on 9.x use `_9001`, etc.
