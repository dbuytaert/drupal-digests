## Summary

Functional tests now run significantly faster by avoiding unnecessary container rebuilds during installation. Previously, the installer would rebuild the dependency injection container 11 times in a simple test, even when no modules or themes had changed. This optimization reduces that to 8 rebuilds by tracking which extensions were compiled and only rebuilding when modules or themes actually change. Test suite execution times improved by 30-50% in optimal conditions, making local development and continuous integration faster.

## Impact

No upgrade or configuration changes required.

## Technical details

The `InstallerKernel` class previously forced a container rebuild every time `initializeContainer()` was called, even when no extensions had changed. Since `allowDumping` is FALSE during installation, no cached container exists, so the parent class would always recompile from scratch.

The fix adds a `compiledExtensions` property that tracks which modules and themes were present during the last compilation. Before triggering a rebuild, the kernel now checks if the current extensions match the compiled ones by comparing counts and keys. If they match and a container already exists, it reuses the existing container. Only when extensions actually differ does it set `containerNeedsRebuild` to TRUE and call the parent initialization.

This reduces container rebuilds from 11 to 8 in a trivial functional test. While actual wall-clock time varies due to hardware availability, the optimization shows measurable improvements in test runtime, with minimum times dropping from 3:34 to 1:47 in some test runs.
