## Summary

Uninstalling a theme now shows a confirmation page before the theme is removed. Previously, clicking the "Uninstall" link on `/admin/appearance` would immediately remove the theme (protected only by a CSRF token). Now it loads a confirmation form at `/admin/appearance/uninstall?theme=THEME_NAME` that warns about data loss and lists any configuration that will be deleted, matching how module uninstall already works. Site owners benefit most when themes have block or other config dependencies that would be silently deleted.

## Impact

Affects module maintainers.

- **API change:** `ThemeController::uninstall()` is removed; the `system.theme_uninstall` route now resolves to `ThemeUninstallConfirmForm` instead of a controller action, and no longer carries a `_csrf_token` route requirement

## Technical details

The appearance admin page generates "Uninstall" links that point to `system.theme_uninstall` with a `?theme=` query parameter. Previously this route used `_controller: ThemeController::uninstall`, which performed the uninstall immediately and relied on `_csrf_token: TRUE` for protection. That controller method is now removed entirely.

The new `ThemeUninstallConfirmForm` (`core/modules/system/src/Form/ThemeUninstallConfirmForm.php`) extends `ConfirmFormBase` and uses `ConfigDependencyDeleteFormTrait::addDependencyListsToForm` to enumerate config entities that depend on the theme. The route now uses `_form` instead of `_controller`, and the `_csrf_token` route requirement is dropped because the form provides its own CSRF protection.

`buildForm()` performs all the same guards the old controller did, but redirects back to `system.themes_page` with an error message rather than proceeding: missing `theme` parameter throws `AccessDeniedHttpException` (HTTP 403); unknown theme name, default theme, admin theme, and base theme with active sub-themes each redirect with an appropriate error message. These checks run before the form is rendered so there is nothing to confirm when the action is blocked.

The form is marked `@internal`. The page title is "Uninstall %theme theme" and the warning reads "The %theme theme will be completely uninstalled from your site, and all data from this theme will be lost!" -- wording settled during a usability review ([#3580386](https://www.drupal.org/node/3580386)).

A new functional test class `ThemeUninstallConfirmTest` covers the main workflow, each guard condition, config dependency display, invalid theme name, and missing parameter handling. Existing tests across `ThemeTest`, `ThemeUiTest`, `ClaroTest`, `OliveroTest`, and `InstallProfileUninstallTest` were updated to add `$this->submitForm([], 'Uninstall')` after clicking the uninstall link.

## Upgrade

Any test or custom code that clicks or follows the theme uninstall link and immediately asserts the result must now submit the confirmation form first:

```php
// Before
$this->clickLink('Uninstall Claro theme');
$this->assertSession()->pageTextContains('The Claro theme has been uninstalled.');

// After
$this->clickLink('Uninstall Claro theme');
$this->submitForm([], 'Uninstall');
$this->assertSession()->pageTextContains('The Claro theme has been uninstalled.');
```
