## Summary

Block config entities (`block.block.*`) are now fully validatable in Drupal 11.1. This matters most to site owners using the Recipes feature, which depends on config validation to safely apply and roll back recipes that place blocks. The change ensures that block configuration is validated against schema before being saved, catching problems like invalid themes, invalid regions, or missing plugin settings early. Two database update paths run automatically on update: one converts any non-integer block weights to integers, and one converts empty-string `page_id` values in search blocks to `NULL`.

## Impact

Affects module maintainers and site owners.

- **Database update:** `block_post_update_make_weight_integer` casts any non-integer block weights to int; `search_post_update_block_with_empty_page_id` converts empty-string `page_id` in search blocks to NULL
- **Config change:** `status`, `info`, and `view_mode` removed from the shared `block_settings` schema type and moved to the new `block.settings.block_content:*` type; `block.block.*` and `block.settings.*` now carry `FullyValidatable` constraints
- **Deprecation:** Non-integer block weights deprecated in `drupal:11.1.0` (removed in 12.0.0); empty-string `page_id` on search blocks deprecated in `drupal:11.1.0` (removed in 12.0.0)

## Technical details

The core change is adding `FullyValidatable: ~` constraints to three schema types in `block.schema.yml` and `core.data_types.schema.yml`:

- `block.block.*` (the config entity itself) gains `FullyValidatable`, plus field-level constraints on `theme` (`NotBlank`, `ExtensionName`, `ExtensionExists: theme`), `region` (`NotBlank`, `Callback: [Block::validateRegion]`), and `provider` (marked `nullable: true` pending deprecation in [#3426278](https://www.drupal.org/node/3426278)).
- `block.settings.*` (the wildcard fallback for block plugin settings with no dedicated schema) gains `FullyValidatable`.
- `block.settings.search_form_block` gains `FullyValidatable`, and its `page_id` key is made `nullable: true` with a `ConfigExists: {prefix: search.page.}` constraint.

A new static method `Block::validateRegion()` is introduced as the callback for the region constraint; it looks up `system_region_list($theme)` using the sibling `theme` value from the validation context root.

**Historic technical debt resolved:** `status`, `info`, and `view_mode` were previously part of the generic `block_settings` type, making them appear to belong to every block plugin. This was an oversight from [#2274175](https://www.drupal.org/node/2274175), when the block plugin schema was split out but the keys specific to `BlockContentBlock` were accidentally left in the shared type. This issue removes those three keys from `block_settings` and introduces a new dedicated `block.settings.block_content:*` type in `block_content.schema.yml` that carries them (with `FullyValidatable: ~` on that type as well), along with a `ConfigExists` constraint on `view_mode`.

**Weight integrity:** `Block::setWeight()` now casts its argument to `(int)`, and the `$weight` property defaults to `0` instead of `null`. `BlockForm` switches from `$request->query->get('weight', 0)` to `$request->query->getInt('weight')` to ensure integer weights from the URL. A deprecation fires in `Block::preSave()` when a non-integer weight is saved (deprecated in `drupal:11.1.0`, removed in `drupal:12.0.0`). The post-update function `block_post_update_make_weight_integer` uses `ConfigEntityUpdater` to cast any persisted non-integer weights.

**Search block `page_id`:** `SearchBlock::defaultConfiguration()` changes the default from `''` to `NULL`. `search_block_presave()` (a new `hook_ENTITY_TYPE_presave()` implementation) normalizes empty-string `page_id` to `NULL` with a deprecation (deprecated in `drupal:11.1.0`). The post-update `search_post_update_block_with_empty_page_id` migrates existing stored values. Olivero's shipped block config YAML files are updated in place.

**Migrations:** Both `d6_block.yml` and `d7_block.yml` now pipe the `weight` source value through an `intval` callback plugin to guarantee integer output.

**Test coverage:** `BlockValidationTest` gains explicit tests for `theme`, `region`, and `weight` validation. `BlockRebuildTest` and `BlockInvalidRegionTest` add `$configSchemaCheckerExclusions` entries for blocks intentionally placed in invalid regions. New update-path tests `BlockWeightUpdateTest` and `SearchBlockPageIdUpdatePathTest` cover both post-update functions. Blockers resolved before landing: [#3426309](https://www.drupal.org/node/3426309) (config validation for weights) and [#3426324](https://www.drupal.org/node/3426324) (`ExistsConstraintValidator` null and `core` handling).

## Upgrade

If you have a **custom block plugin** that defines settings, its schema type (`block.settings.YOUR_PLUGIN_ID`) does not automatically gain `FullyValidatable`. Add it explicitly once you have added constraints for all your settings keys:

```yaml
block.settings.my_custom_block:
  type: block_settings
  label: 'My custom block'
  constraints:
    FullyValidatable: ~
  mapping:
    my_setting:
      type: string
      label: 'My setting'
      constraints:
        NotBlank: []
```

If your module **sets block weight programmatically**, ensure you pass an integer:

```php
// Before (now deprecated in 11.1.0, removed in 12.0.0):
$block->setWeight('5');

// After:
$block->setWeight(5);
```

If you use **search blocks** with `page_id: ''` in config YAML, change it to `page_id: null`. The post-update handles existing database rows automatically, but any YAML files you ship in a module's `config/install` or `config/optional` directory must be updated manually:

```yaml
# Before:
settings:
  page_id: ''

# After:
settings:
  page_id: null
```

If your module's schema or test fixtures assumed that `block_settings` includes `status`, `info`, or `view_mode` keys for non-content-block plugins, remove those references. Those keys now only appear under `block.settings.block_content:*`.

### Collaboration

Wim Leers opened the issue and drove the initial implementation. While working through the MR he discovered in comment #5 that `status`, `info`, and `view_mode` had been accidentally left in the generic `block_settings` type since [#2274175](https://www.drupal.org/node/2274175) — a schema quirk that had confused developers reading exported block YAML for years. Fixing that oversight alone reduced failing tests from 788 to 115, which Wim described as what made the whole effort "suddenly doable." narendraR took over continued development after the groundwork was laid. phenaproxima provided multiple rounds of review, eventually taking direct commits to address remaining feedback. alexpott pushed back on the approach of widening the `weight` type to accept non-integers, arguing that config always casts weight to integer on save and that validation should enforce the integer type rather than widen it; this led to the deprecation approach for non-integer weights. On commit, alexpott also adjusted the deprecation version from `11.0.0` to `11.1.0` to reflect where the release cycle actually stood.
