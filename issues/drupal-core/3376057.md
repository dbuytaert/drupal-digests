## Summary

All Drupal core test files now carry `declare(strict_types=1)` and the rule is enforced by PHP CodeSniffer. Without this declaration, PHP silently coerces argument types, which can mask bugs (for example, a string passed where a `bool` is expected casts to `true` without error). Restricting the change to test files means there is no risk to production sites. Developers contributing tests to Drupal core must add `declare(strict_types=1);` immediately after the opening `<?php` tag in every test file.

## Impact

No upgrade or configuration changes required.

## Technical details

The issue was opened by catch after a silent string-to-bool coercion slipped through a type-hinted function in a test file that lacked `declare(strict_types=1)`. The initial approach of adding the declaration to all files immediately exposed around 3,000 type errors across the test suite, so the work was broken into four sequential phases across many child issues.

**Phase 1: Fix type errors.** Strict type violations were corrected file-by-file across unit tests ([#3397890](https://www.drupal.org/node/3397890)), kernel tests ([#3397905](https://www.drupal.org/node/3397905)), functional tests ([#3399413](https://www.drupal.org/node/3399413)), functional JavaScript tests ([#3399754](https://www.drupal.org/node/3399754)), test traits ([#3399992](https://www.drupal.org/node/3399992)), and test modules ([#3402007](https://www.drupal.org/node/3402007)).

**Phase 2: Agree on the coding standard.** The exact format was settled in the Drupal Coding Standards project at [#3303206](https://www.drupal.org/node/3303206). The final decision is `declare(strict_types=1)` with no spaces around the `=` sign.

**Phase 3: Add the declaration.** `declare(strict_types=1);` was added to every test category: unit ([#3399373](https://www.drupal.org/node/3399373)), kernel ([#3399388](https://www.drupal.org/node/3399388)), functional ([#3399746](https://www.drupal.org/node/3399746)), functional JavaScript ([#3400018](https://www.drupal.org/node/3400018)), test traits ([#3401994](https://www.drupal.org/node/3401994)), test modules ([#3400334](https://www.drupal.org/node/3400334)), build tests ([#3403711](https://www.drupal.org/node/3403711)), and miscellaneous test classes ([#3407793](https://www.drupal.org/node/3407793)). The Rector rule `\Rector\TypeDeclaration\Rector\StmtsAwareInterface\DeclareStrictTypesRector` was used to automate the additions.

**Phase 4: Enforce going forward.** `#3400434` added the `SlevomatCodingStandard.TypeHints.DeclareStrictTypes` sniff to `core/phpcs.xml.dist` with `spacesCountAroundEqualsSign` set to `0`, scoped to `*/tests/*` paths and excluding `*/fixtures/*`. This means any new test file added to core that omits the declaration will fail the phpcs check in CI.

The original monolithic MR (!5195) had to be abandoned because CI tooling (cspell, yarn) hit argument-list limits when more than 1,000 files were touched in a single merge request. Splitting the work into smaller child issues resolved both the CI problem and made individual changes reviewable.

## Upgrade

Any PHP file under a `tests/` directory that is part of Drupal core (or validated against core's `phpcs.xml.dist`) must now include `declare(strict_types=1);` on the line immediately after `<?php`:

```php
<?php

declare(strict_types=1);

namespace Drupal\...;
```

Files under `fixtures/` subdirectories are excluded from this requirement.
