# #3408420: Deleting a workflow can lead to the deletion of unaffected views

- **Project:** Drupal Core
- **Type:** Critical bug report
- **Version:** main
- **Status:** Fixed
- **Discussion:** [#3408420](https://www.drupal.org/node/3408420) · 4 contributors · 17 comments
- **Followers:** 5

## Summary

Deleting a workflow could silently delete views and other configuration entities that had no real dependency on the workflow. The same underlying defect could also cause a user role to lose a permission when an unrelated field was removed from a content type with Layout Builder enabled. Both symptoms were caused by a dependency-ordering bug in the configuration system that made config entities get processed in the wrong sequence during a cascading deletion. The fix ensures that the dependency processing order is preserved correctly, so only configuration that genuinely depends on the deleted item is affected.

## Impact

No upgrade or configuration changes required.

## Technical details

The defect was in `ConfigManager::findConfigEntityDependenciesAsEntities()`. That method calls `findConfigEntityDependencies()` to get an ordered list of config names (ordered so that dependents come before their dependencies), then loads the corresponding entities using `EntityStorageInterface::loadMultiple()` grouped by entity type. The problem: `loadMultiple()` returns entities sorted by their own storage order, not by the dependency order. This silently reordered the returned entity list. `getConfigEntitiesToChangeOnDependencyRemoval()` processes the list from the end, relying on the ordering to ensure a dependency is processed before any entity that depends on it. When the order was wrong, a role could be processed before an entity view display it depended on. Because that display appeared to be in the removal list, the system incorrectly stripped the role's permission.

The fix pre-populates the result array with keys in the correct dependency order using `array_fill_keys(array_keys($dependencies), NULL)`, then fills in each loaded entity by its config dependency name (`$loaded_entity->getConfigDependencyName()`). A final `array_filter()` removes any entries that were not loaded. This preserves the order from `findConfigEntityDependencies()` regardless of how `loadMultiple()` sorts entities internally.

The fix was applied in the related issue [#3504368](https://www.drupal.org/node/3504368) and committed to 10.5.x, 11.1.x, and 11.x. Issue [#3408420](https://www.drupal.org/node/3408420) was subsequently closed because the same ordering fix resolved the workflow-deletion-deletes-views scenario.

### Collaboration

The root cause proved elusive: the alphabetical name of a contributing module changed which config objects were in the static cache on first load versus subsequent loads, altering the order `findConfigEntityDependencies()` returned, which meant the bug was not reliably reproducible in a clean install. godotislate isolated the mechanism by tracing the list order through `findConfigEntityDependenciesAsEntities()` and identifying how grouping by entity type for `loadMultiple()` destroyed the dependency order. godotislate then proposed and implemented the fix in MR 11141, while dpagini confirmed the fix resolved the original project issue.
