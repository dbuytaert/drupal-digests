## Summary

When CSS and JS aggregation is enabled, Drupal now also generates Brotli-compressed (`.br`) versions of those aggregate files alongside the existing gzip (`.gz`) files, provided the PHP `brotli` extension is installed. The Apache `.htaccess` configuration is updated to serve `.br` files to browsers that advertise `br` in their `Accept-Encoding` header, with gzip as the fallback. Brotli typically achieves better compression ratios than gzip, reducing bandwidth and improving page load performance for users on supported browsers. No UI changes are involved; the feature activates automatically when the extension is present and compression is configured.

## Impact

Affects module maintainers and site owners.

- **Config change:** Config keys `system.performance.css.gzip` and `system.performance.js.gzip` are renamed to `css.compress` and `js.compress`; old keys deprecated in 11.4.0, removed in 12.0.0.
- **Deprecation:** `system.performance.css.gzip` and `system.performance.js.gzip` config schema keys replaced by `css.compress` and `js.compress`.

## Technical details

The compression logic lives in `AssetDumper::dumpToUri()`. After writing the `.gz` file via `gzencode()`, the method now additionally calls `brotli_compress($data, 11, BROTLI_TEXT)` and saves the result as `$uri . '.br'` when `extension_loaded('brotli')` is true. The gate condition checks both `$file_extension . '.compress'` (the new key) and `$file_extension . '.gzip'` (the deprecated key) so that sites which have not yet run the post-update hook continue to work.

In `.htaccess`, brotli `RewriteCond`/`RewriteRule` blocks are placed immediately before the existing gzip blocks for both CSS and JS patterns. This ordering ensures a brotli-capable client receives the more efficient encoding. New `FilesMatch` directives for `.css.br` and `.js.br` set `Content-Encoding: br` and `Vary: Accept-Encoding`, and rewrite rules mark those files with `E=no-gzip:1,E=no-brotli:1` to prevent double compression. Nginx users must enable `brotli_static on` in their server configuration separately.

The `system.performance` config schema adds a `compress` boolean key for both `css` and `js` sections and marks the old `gzip` key as deprecated in 11.4.0 for removal in 12.0.0. The post-update hook `system_post_update_migrate_compress_setting()` copies `css.gzip` → `css.compress` and `js.gzip` → `js.compress`, then clears the old keys from active config. The design choice to merge gzip and brotli into a single `compress` toggle (rather than separate per-algorithm flags) was deliberate: the `.htaccess` rules cannot be conditioned on which algorithms are configured, so the toggle is best understood as "enable all available compression" rather than selecting individual algorithms.

## Upgrade

The post-update hook runs automatically during `drush updatedb` or the update UI. Custom code or recipes that read or set the old config keys must be updated:

```php
// Before (deprecated in 11.4.0, removed in 12.0.0)
\Drupal::config('system.performance')->get('css.gzip');
\Drupal::config('system.performance')->get('js.gzip');

\Drupal::configFactory()->getEditable('system.performance')
  ->set('css.gzip', TRUE)
  ->set('js.gzip', TRUE)
  ->save();

// After
\Drupal::config('system.performance')->get('css.compress');
\Drupal::config('system.performance')->get('js.compress');

\Drupal::configFactory()->getEditable('system.performance')
  ->set('css.compress', TRUE)
  ->set('js.compress', TRUE)
  ->save();
```

Nginx users who want to serve the pre-compressed `.br` files must install the `ngx_brotli` module and add `brotli_static on;` to the appropriate server block. Apache sites benefit from the updated `.htaccess` automatically.

### Collaboration

The initial patch from `prudloff` added separate `css.brotli` and `js.brotli` boolean settings independent of the existing `gzip` keys. `andypost` proposed collapsing both into a single `compress` boolean that enables every available algorithm, reasoning that there is no real use case for enabling gzip without brotli, and that a future algorithm (such as zstandard, tracked in [#3346184](https://www.drupal.org/node/3346184)) could be added without another config key. `catch` agreed, noting that the `.htaccess` rules cannot be conditioned on which algorithms are individually selected anyway, making a unified toggle the only practical design. `prudloff` then rewrote the MR around the `compress` key and added the upgrade path. A separate CI blocker required brotli to be present in the test images; `andypost` resolved this by adding it to the DrupalCI environments ([#3512083](https://www.drupal.org/node/3512083)), which was a prerequisite for the tests to pass before commit.
