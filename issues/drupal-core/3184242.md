## Summary

Drupal now generates Brotli-compressed (`.br`) versions of aggregated CSS and JS assets alongside the existing gzip (`.gz`) files, provided the PHP `brotli` extension is installed. The Apache `.htaccess` rules are updated to serve `.br` files with priority over `.gz` when a client advertises brotli support. The `system.performance.css.gzip` and `system.performance.js.gzip` config settings are renamed to `compress`; an automatic upgrade path migrates existing configuration. Sites without the brotli extension see no change in behaviour.

## Impact

Affects module maintainers and site owners.

- **Database update:** system_post_update_migrate_compress_setting() migrates css.gzip and js.gzip to css.compress and js.compress
- **Config change:** system.performance.css.gzip and system.performance.js.gzip renamed to system.performance.css.compress and system.performance.js.compress
- **Deprecation:** system.performance.*.gzip config keys deprecated in drupal:11.4.0, removed in drupal:12.0.0; use system.performance.*.compress instead

## Technical details

The core change is in `AssetDumper::dumpToUri()`. Where it previously checked `$file_extension . '.gzip'` to decide whether to write a `.gz` file, it now checks the new `compress` key first, falling back to the deprecated `gzip` key for backward compatibility. When compression is enabled and `extension_loaded('brotli')` is true, it writes a `.br` file via `brotli_compress($data, 11, BROTLI_TEXT)` after writing the `.gz` file. The `.htaccess` in both `sites/` and `core/` gains `RewriteCond`/`RewriteRule` blocks for `.css.br` and `.js.br`, placed before the gzip rules so brotli is preferred, and matching `FilesMatch` blocks set `Content-Encoding: br` and `Vary: Accept-Encoding`. The `system.performance` config schema (in `system.schema.yml`) gains a new `compress` key under both `css` and `js`; the existing `gzip` key is kept but marked deprecated in `drupal:11.4.0` with removal in `drupal:12.0.0`. The post-update hook `system_post_update_migrate_compress_setting()` copies `css.gzip` → `css.compress` and `js.gzip` → `js.compress` and clears the old keys. A new `AssetCompressUpdatePathTest` verifies the upgrade path. The design choice to use a single `compress` boolean rather than separate per-algorithm flags was settled in the issue thread by `andypost` and `catch`: because `.htaccess` rules always check for `.br` files regardless of a per-algorithm config flag, a finer-grained setting would give a false impression of control while still causing filesystem lookups for non-existent files on servers without brotli.

## Upgrade

If custom code reads or writes the old config keys, update references:

```php
// Before
\Drupal::config('system.performance')->get('css.gzip');
\Drupal::config('system.performance')->get('js.gzip');

// After
\Drupal::config('system.performance')->get('css.compress');
\Drupal::config('system.performance')->get('js.compress');
```

The automatic post-update hook `system_post_update_migrate_compress_setting()` migrates the values in the database. If you ship `system.performance.yml` in a recipe or distribution with `gzip: true`, update it to `compress: true` before targeting Drupal 12.

### Collaboration

The design evolved significantly over the thread. The original patch added separate `system.performance.css.brotli` and `system.performance.js.brotli` boolean keys alongside `gzip`. `andypost` pointed out in comment [#15102385](https://www.drupal.org/node/15102385) that a single `compress` key that enables every available algorithm would be cleaner and more future-proof. `catch` reinforced this in comment [#15102469](https://www.drupal.org/node/15102469), noting that `.htaccess` rules cannot be conditionally applied per-algorithm anyway, so a fine-grained config flag would be misleading. `prudloff` agreed and updated the MR to use the single `compress` key. Testing was blocked until `andypost` filed [#3512083](https://www.drupal.org/node/3512083) to add the brotli PHP extension to the CI Docker images, which unblocked the functional test `prudloff` had written.
