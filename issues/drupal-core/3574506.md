## Summary

If you have a Views table display that includes the "Views result counter" (row number) field and that column was made click-sortable, clicking the column header produced a fatal database error: `ORDER BY "unknown"`. The counter field has no underlying database column to sort on, so the generated SQL was invalid. This is now fixed. The counter field is explicitly marked as non-sortable, so the sort option no longer appears in the table header. A cache rebuild is triggered automatically on update. Note that existing view configuration that already has this column marked as sortable is not automatically corrected; a follow-up issue ([#3588188](https://www.drupal.org/node/3588188)) tracks that cleanup.

## Impact

No upgrade or configuration changes required.

## Technical details

The root cause is that the "Views result counter" field is a computed pseudo-field with no real database column. When a column in a Views table has no `click sortable` constraint set, Views falls back to the literal string `"unknown"` as the ORDER BY value, which MySQL rejects as an invalid column reference.

The fix adds `'click sortable' => FALSE` to the counter field's entry in the `viewsData()` method of the Views counter field plugin. This single key prevents Views from offering or executing click-sort on this field.

Because Views caches its field data definitions, an empty `views_post_update_counter_field_not_sortable()` hook is added to `views.post_update.php`. The empty hook triggers a cache rebuild on `drush updb` / update.php so that the corrected field definition is picked up by existing sites without requiring a manual cache clear. The post-update hook was required even for a patch release because alexpott noted the Views data cache must be invalidated for the fix to take effect on running installations.
