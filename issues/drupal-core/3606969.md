## Summary

On the content language settings form (`admin/config/regional/content-language`), "column group" or subfield sync options for translatable base fields were invisible until the user had already saved the field once as untranslatable. This forced an awkward workaround before the per-column translation controls would appear. The fix makes those controls visible immediately on the first load. Core's own Media entity `thumbnail` field (with its `file`, `alt`, and `title` subfields) was already affected, as were any contributed or custom modules that register translatable base fields with column groups.

## Impact

No upgrade or configuration changes required.

## Technical details

The content translation form alter in `ContentTranslationManager` (specifically `formLanguageContentSettingsFormAlter`) passes a field definition to `FieldSyncWidget::widget()` so it can render per-column translatable checkboxes. Base field definitions (`BaseFieldDefinition`) never carry third-party settings, which is where column group configuration is stored. Before this fix, the code always passed the raw `$definition` object directly, so `FieldSyncWidget::widget()` found no third-party settings and produced no output for base fields unless a `BaseFieldOverride` config entity had already been saved (overrides do carry third-party settings).

The fix adds a branch: when `$definition->getFieldStorageDefinition()->isBaseField()` is true, it calls `$definition->getConfig($bundle)` to obtain the bundle-level config wrapper before passing it to `$this->fieldSyncWidget->widget()`. This wrapper can surface third-party settings even before an explicit `BaseFieldOverride` exists on disk. Bundle fields (non-base fields) continue to use `$definition` unchanged. A new functional test `ContentLanguageBaseFieldSyncWidgetTest` verifies that the `thumbnail` subfield checkboxes (`file`, `alt`, `title`) are present on the first page load for a Media type, before any override has been saved.
