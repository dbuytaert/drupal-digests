## Summary

After upgrading to Drupal 10.1.x, site owners began seeing frequent log entries reading "The theme must be passed as a query argument" from `AssetControllerBase::deliver()`. The errors appear as HTTP 400 responses against aggregated CSS/JS file URLs. Investigation across many sites and environments confirmed this is expected behavior: the errors are produced when clients request aggregated asset URLs that lack the required query parameters, which happens due to stale pages held in reverse-proxy caches (Cloudflare, Varnish), browsers with old tabs open from pre-10.1 visits, bots such as GPTBot crawling old URLs, or certain contrib modules (LiteSpeed Cache, Simple Menu Icons, bandwidth optimization modules) generating or interfering with asset URLs. The issue was closed without a code change. A separately resolved issue ([#3377310](https://www.drupal.org/node/3377310)) had already fixed the related problem of these correct 400 responses being incorrectly logged through the PHP error channel.

## Impact

No upgrade or configuration changes required.

## Technical details

Drupal 10.1.x redesigned asset aggregation so that aggregated CSS/JS files can be regenerated on demand via a controller. `AssetControllerBase::deliver()` requires `theme`, `delta`, `language`, and `include` query parameters to know which aggregate to rebuild. When a request arrives for a path under `sites/default/files/css/` or `sites/default/files/js/` but those parameters are absent or malformed (e.g. double-encoded `&amp;amp;` instead of `&`), the controller throws a `BadRequestHttpException` because it has no way to determine what aggregate to generate. This is intentional: without the parameters the URL is structurally unresolvable. Before 10.1.x, Drupal 9's `AssetResolver::getCssAssets()` derived the active theme directly from `ThemeManager::getActiveTheme()` without needing URL-encoded context, so old D9-era asset URLs naturally lack these parameters. Common real-world sources of these requests include: clients loading pages cached by Varnish or Cloudflare from before the upgrade, browsers pre-fetching assets from cached page state, AI crawlers (GPTBot) following stale links, and contrib modules that store or rewrite asset URLs. Catch confirmed in the thread that a 301 redirect workaround is not feasible because the parameters are the only source of information needed to construct the redirect target. The issue was ultimately closed as "works as designed".
