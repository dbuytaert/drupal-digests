## Summary

Calling `fetchAll()` on a database query result when the fetch mode was set to a custom class (either via the `fetch` query option or via `setFetchMode(FetchAs::ClassObject, ...)`) threw a `TypeError` at runtime: `PDOStatement::fetchAll(): Argument #2 must be of type string, int given`. This is now fixed, and query results are correctly returned as instances of the requested class.

## Impact

No upgrade or configuration changes required.

## Technical details

The bug lived in `PdoResult::fetchAll()`. Its original implementation used a null-coalescing chain to resolve the second argument passed to `clientFetchAll()`:

```php
$this->clientFetchAll($mode, $fetchOptions['column'] ?? $fetchOptions['class'] ?? NULL, ...)
```

`StatementBase` initialises `$fetchOptions['column']` to integer `0` by default. Because `0` is not `null`, the `??` operator short-circuits and the class name stored in `$fetchOptions['class']` is never reached. `clientFetchAll` therefore received `0` (an integer) instead of a class-name string, triggering the `TypeError` from PDO.

The fix replaces the coalescing chain with a `match` expression keyed on the active `FetchAs` enum case:

```php
return $this->clientFetchAll($mode, ...match ($mode) {
  FetchAs::Column => [$fetchOptions['column'] ?? NULL],
  FetchAs::ClassObject => [
    $fetchOptions['class'] ?? NULL,
    $fetchOptions['constructor_args'] ?? NULL,
  ],
  default => [],
});
```

This ensures each fetch mode receives only its own relevant arguments, regardless of whatever default values are present in `$fetchOptions`. Two new kernel tests were added to `FetchTest`: `testQueryFetchAllAsClassInstances` (covering the `fetch` query option path) and `testQueryFetchAllAsFetchModeClassInstances` (covering `setFetchMode` with constructor arguments). The change was also backported to `11.x` and `11.4.x`.

### Collaboration

The issue went through two distinct implementation approaches before landing. The original submitter (pjotr.savitski) identified the root cause and proposed a fix in `StatementBase::setFetchMode()` that would explicitly null out the `column` key when switching to `ClassObject` mode. A second contributor (rhezios) noted that this second approach broke cases with constructor arguments and instead implemented the first proposed option (nulling the column) in a separate MR. Then alexpott suggested moving the fix entirely to `PdoResult::fetchAll()` as a more robust and centralised solution. dpi picked up that suggestion, implemented it, and noted in the thread that the Entity Hierarchy module had already encountered the same breakage as a downstream consequence of [#3522561](https://www.drupal.org/node/3522561). mondrake reviewed and requested minor test consistency improvements, which dpi addressed before the patch was committed.
