## Summary

Starting in Drupal 11.4, the HTTP REST authentication endpoints (login, logout, login status, and password reset) require the `rest` module to be enabled. Previously these routes lived in the `user` module and were always active, which meant third-party modules adding login protection such as OTP or two-factor authentication often secured the form login but left the REST endpoint unguarded (see [SA-CONTRIB-2025-056](https://www.drupal.org/sa-contrib-2025-056)). The URL paths (`/user/login?_format=json`, `/user/logout`, `/user/login_status`, `/user/password`) are unchanged, so external API clients do not need to update their requests as long as the `rest` module is enabled. Any Drupal code that generates URLs using the old route names (`user.login.http`, `user.logout.http`, `user.login_status.http`, `user.pass.http`) needs to be updated to the new names in the `rest` module.

## Impact

Affects module maintainers and site owners.

- **Security:** REST login endpoints are now disabled by default on sites without the `rest` module enabled, reducing exposure when third-party login-protection modules do not cover the REST route.
- **Requirements:** The `rest` module must be enabled for HTTP REST authentication endpoints to function without deprecation warnings; previously only the `user` module was required.
- **API change:** Routes `user.login.http`, `user.logout.http`, `user.login_status.http`, `user.pass.http` deprecated in 11.4.0 and removed in 12.0.0; use `rest.login`, `rest.logout`, `rest.login_status`, `rest.pass` respectively.
- **Deprecation:** `UserAuthenticationController` deprecated in 11.4.0 and removed in 12.0.0; use `RestAuthenticationController` in the `rest` module instead.

## Technical details

The four HTTP authentication routes previously defined in `user.routing.yml` are deprecated in 11.4.0 and will be removed in 12.0.0. Equivalent routes (`rest.login`, `rest.logout`, `rest.login_status`, `rest.pass`) are added to a new `rest.routing.yml` with identical paths, so the HTTP URL surface is stable.

A new `RestAuthenticationController` in `Drupal\rest\Controller` holds all authentication logic (`login`, `logout`, `loginStatus`, `resetPassword`), copied from the now-deprecated `UserAuthenticationController`. The old `UserAuthenticationController` fires a runtime `E_USER_DEPRECATED` from its constructor. The old routes in `user.routing.yml` carry `deprecated` keys pointing to the change record at `https://www.drupal.org/node/3552724`.

The path-preservation mechanism is a `DeprecatedUserRoutesSubscriber` event subscriber (registered via autowiring in `rest.services.yml`) that listens on `RoutingEvents::ALTER`. When the `rest` module is installed, it replaces the `_controller` default on the four user-module routes with the corresponding method on `RestAuthenticationController`, so those routes no longer invoke the deprecated controller. If `rest` is not installed, the user routes remain active but instantiate the deprecated controller. The `rest` module also registers the four new route names with the existing `RouteSubscriber` for format negotiation so `_format=json` continues to work on all of them. The `rest.module` `info.yml` now declares a hard dependency on `drupal:user`.

## Upgrade

Enable the `rest` module if you use the REST login endpoints:

```sh
drush en rest
```

Update any Drupal code that generates URLs to these routes:

```php
// Before (deprecated in 11.4, removed in 12.0)
Url::fromRoute('user.login.http', ['_format' => 'json']);
Url::fromRoute('user.logout.http', ['_format' => 'json']);
Url::fromRoute('user.login_status.http', ['_format' => 'json']);
Url::fromRoute('user.pass.http', ['_format' => 'json']);

// After
Url::fromRoute('rest.login', ['_format' => 'json']);
Url::fromRoute('rest.logout', ['_format' => 'json']);
Url::fromRoute('rest.login_status', ['_format' => 'json']);
Url::fromRoute('rest.pass', ['_format' => 'json']);
```

The URL paths (`/user/login`, `/user/logout`, etc.) are unchanged. No changes to external HTTP clients are required as long as the `rest` module is enabled.

### Collaboration

The key design decision that resolved a significant disagreement came from longwave. The initial MR used different URL paths for the new `rest` module routes, which would have forced all API clients to update their endpoint URLs. longwave pushed back (comment [#16309694](https://www.drupal.org/node/16309694)) and then implemented the path-preservation approach (described as "option #19" in the thread): keeping the old `user.routing.yml` routes at the same paths while using `DeprecatedUserRoutesSubscriber` to swap in the non-deprecated controller when `rest` is installed (comment [#16337846](https://www.drupal.org/node/16337846)). longwave also opened the 11.x backport MR and caught a non-trivial compatibility issue: the backported `RestAuthenticationController` needed to support both `UserAuthInterface` and `UserAuthenticationInterface` because Drupal 11 still allows both (comment [#16570313](https://www.drupal.org/node/16570313)).
