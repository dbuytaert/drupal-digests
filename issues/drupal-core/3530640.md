## Summary

Starting in Drupal 11.4, the HTTP REST authentication endpoints (login, logout, login status, and password reset) require the `rest` module to be enabled. Previously these routes were part of the `user` module and always active, which meant third-party modules adding login protection (such as OTP or two-factor authentication) often secured the form login but left the REST endpoint unguarded. The URL paths (`/user/login?_format=json`, `/user/logout`, `/user/login_status`, `/user/password`) are unchanged, so API clients do not need to update their requests. Any Drupal code that generates URLs using the old route names (`user.login.http`, `user.logout.http`, `user.login_status.http`, `user.pass.http`) needs to be updated to the new names in the `rest` module.

## Impact

Affects module maintainers and site owners.

- **Security:** REST login endpoints are now disabled by default on sites without the `rest` module enabled, reducing exposure when third-party login-protection modules do not cover the REST route.
- **Requirements:** The `rest` module must be enabled for HTTP REST authentication endpoints to function without deprecation warnings; previously only the `user` module was required.
- **API change:** Routes `user.login.http`, `user.logout.http`, `user.login_status.http`, `user.pass.http` deprecated in 11.4.0 and removed in 12.0.0; use `rest.login`, `rest.logout`, `rest.login_status`, `rest.pass` respectively.
- **Deprecation:** `UserAuthenticationController` deprecated in 11.4.0, removed in 12.0.0; use `RestAuthenticationController` in the `rest` module instead.

## Technical details

The four HTTP authentication routes previously defined in `user.routing.yml` are deprecated in 11.4.0 and will be removed in 12.0.0. Equivalent routes with new names (`rest.login`, `rest.logout`, `rest.login_status`, `rest.pass`) are added to a new `rest.routing.yml` section with identical paths, so the HTTP URLs remain stable.

A new `RestAuthenticationController` in `Drupal\rest\Controller` holds all the authentication logic (login, logout, loginStatus, resetPassword), copied from the now-deprecated `UserAuthenticationController`. The old `UserAuthenticationController` fires a runtime `E_USER_DEPRECATED` from its constructor. The old routes in `user.routing.yml` carry `deprecated` keys pointing to the change record at `https://www.drupal.org/node/3552724`.

The path-preservation mechanism is a `DeprecatedUserRoutesSubscriber` event subscriber (registered as an autowired service in `rest.services.yml`) that listens on `RoutingEvents::ALTER`. When the `rest` module is installed, it replaces the `_controller` default on the four user module routes with `RestAuthenticationController`, so those routes no longer trigger the deprecation. If the `rest` module is not installed, the user routes remain active but instantiate the deprecated controller. The `rest` module also registers the four new route names with the existing format-negotiation event subscriber so `_format=json` continues to work on all of them. On the 11.x backport, `RestAuthenticationController` additionally handles the still-supported `UserAuthInterface` alongside `UserAuthenticationInterface` (which deprecates the former), matching the dual-path already present in `UserAuthenticationController`.

## Upgrade

Enable the `rest` module if you use the REST login endpoints:

```sh
drush en rest
```

Update any Drupal code that generates URLs to these routes:

```php
// Before (deprecated in 11.4, removed in 12.0)
Url::fromRoute('user.login.http', ['_format' => 'json']);
Url::fromRoute('user.logout.http', ['_format' => 'json']);
Url::fromRoute('user.login_status.http', ['_format' => 'json']);
Url::fromRoute('user.pass.http', ['_format' => 'json']);

// After
Url::fromRoute('rest.login', ['_format' => 'json']);
Url::fromRoute('rest.logout', ['_format' => 'json']);
Url::fromRoute('rest.login_status', ['_format' => 'json']);
Url::fromRoute('rest.pass', ['_format' => 'json']);
```

The URL paths (`/user/login`, `/user/logout`, etc.) are unchanged. No changes to external HTTP clients are required as long as the `rest` module is enabled.

longwave drove the key design decision that resolved a significant disagreement in the thread. The initial MR used different paths for the new `rest` module routes, which would have forced all API clients to update their URLs. After pushback, longwave implemented the path-preservation approach (option #19 in the thread): keeping the old `user.routing.yml` routes at the same paths but using `DeprecatedUserRoutesSubscriber` to swap in the non-deprecated controller when `rest` is installed. longwave also handled the 11.x backport and caught a non-trivial compatibility issue: the backported `RestAuthenticationController` needed to support both `UserAuthInterface` and `UserAuthenticationInterface` because Drupal 11 still allows both, unlike `main`.
