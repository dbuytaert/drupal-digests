## Summary

Stream wrappers like `public://`, `temporary://`, and `private://` were not being registered after the dependency injection container was rebuilt during module installation. This caused validation errors when installing modules with configuration that references file storage schemes, and could prevent proper file system operations like deleting image style folders during configuration imports.

## Impact

No upgrade or configuration changes required.

## Technical details

`\Drupal\Core\DrupalKernel::resetContainer()` rebuilds the dependency injection container when modules are installed or uninstalled. This creates a new `stream_wrapper_manager` service instance, which lost the information about which stream wrappers had been registered with PHP's stream wrapper system.

The `StreamWrapperManager` service has two key properties: `$info` (stream wrapper metadata from service discovery) and `$wrappers` (actually registered wrappers). After container rebuild, `$info` was populated from the container definition but `$wrappers` was empty until `register()` was called again.

This caused `getNames()` to return an empty array even though wrapper definitions existed, breaking validation constraints and file operations. The issue manifested when installing multiple modules at once ([#3416522](https://www.drupal.org/node/3416522) optimized this to do a single container rebuild) or during recipe application.

The fix detects whether stream wrappers were registered before the container reset by checking if the service was initialized and had wrappers. If so, it calls `register()` again after rebuilding to restore the PHP-level stream wrapper registrations. This mirrors the existing pattern for reloading the module handler service.

A test was added to `DrupalKernelTest::testResetContainer()` to verify that registered stream wrappers persist across container resets. The fix also removed a validation error workaround from `SchemaCheckTrait` that was masking this problem for editor image upload configuration.
