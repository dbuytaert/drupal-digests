## Summary

In Drupal 11, the state key-value store is now always cached using the cache collector pattern. Previously, sites had to explicitly opt in by setting `$settings['state_cache'] = TRUE` in `settings.php`; that setting is now deprecated and ignored. Sites that stored an unusually large number of dynamic keys in state can no longer opt out. The runtime status report warning about the missing setting is also gone.

## Impact

Affects module maintainers.

- **API change:** `State::__construct()` now requires `$cache` and `$lock` arguments; previously they were optional with deprecated container-fallback defaults.
- **Deprecation:** `$settings['state_cache']` deprecated in drupal:11.0.0; remove it from settings.php as it is no longer consulted.

## Technical details

State caching via `CacheCollector` was introduced as an opt-in in issue [#2575105](https://www.drupal.org/node/2575105), where `$settings['state_cache']` controlled whether the state service aggregated all requested keys into a single cache entry. To support that opt-out, the `State` class in `core/lib/Drupal/Core/State/State.php` accepted `$cache` and `$lock` as optional constructor arguments (with deprecated fallbacks that resolved them from the container), and overrode `updateCache()` and `invalidateCache()` to check `Settings::get('state_cache') !== TRUE` before calling the parent `CacheCollector` methods. When the setting was absent or false, `$this->cacheLoaded` was set to `TRUE` at construction time to prevent the cache from ever being populated.

This commit removes all of that scaffolding for Drupal 11:

- `State::__construct()` now requires `$cache` (`CacheBackendInterface`) and `$lock` (`LockBackendInterface`) as non-nullable arguments. The deprecated container-fallback code and the `Settings` import are removed.
- The `updateCache()` and `invalidateCache()` overrides that short-circuited the parent are deleted entirely, so `CacheCollector`'s full caching behavior is unconditionally active.
- The `state_cache_not_set` runtime requirement check in `core/modules/system/system.install` (which emitted a `REQUIREMENT_WARNING` when the setting was `NULL`) is removed.
- `state_cache` is added to the `DEPRECATED_SETTINGS` array in `core/lib/Drupal/Core/Site/Settings.php`, so any site still defining it will trigger a deprecation notice pointing to the change record at https://www.drupal.org/node/3177901.
- The `$settings['state_cache']` block is removed from both `core/assets/scaffold/files/default.settings.php` and `sites/default/default.settings.php`.

## Upgrade

Remove `$settings['state_cache']` from your `settings.php`. The setting is deprecated in Drupal 11.0.0 and no longer has any effect. State caching is now always enabled.

```php
// Remove this line from settings.php:
// $settings['state_cache'] = TRUE;
```

If you instantiate `State` directly (rare outside of core), ensure you pass both a `CacheBackendInterface` and a `LockBackendInterface` to the constructor — those arguments are now required.
