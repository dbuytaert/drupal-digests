## Summary

Drupal 11.4 introduces a new `\Drupal\user\OneTimeAuthentication` service that consolidates all one-time login and account-cancellation URL generation into one place. The four procedural functions `user_pass_rehash()`, `user_pass_reset_url()`, `user_cancel_url()`, and `user_mail_tokens()` are deprecated and will be removed in Drupal 13.0.0. Custom and contrib code that calls any of those functions will start triggering deprecation notices and must be updated before the Drupal 13 upgrade.

## Impact

Affects module maintainers.

- **Deprecation:** user_pass_rehash(), user_pass_reset_url(), user_cancel_url(), user_mail_tokens() deprecated in 11.4.0, removed in 13.0.0; UserController::validatePathParameters() and ServerCommand::getOneTimeLoginUrl() deprecated in 11.4.0, removed in 12.0.0
- **New API:** New final service \Drupal\user\OneTimeAuthentication with generateHmac(), verifyHmac(), generateOneTimeLoginUrl(), and generateCancelConfirmUrl() methods

## Technical details

The new `final readonly class OneTimeAuthentication` lives in `core/modules/user/src/OneTimeAuthentication.php` and is registered as an autowireable service (`Drupal\user\OneTimeAuthentication: ~` in `user.services.yml`). It depends on `TimeInterface` and `LanguageManagerInterface`. Its public non-internal methods are `generateHmac()`, `verifyHmac()`, `generateOneTimeLoginUrl()`, and `generateCancelConfirmUrl()`; the `tokens()` method is marked `@internal` and serves as a drop-in callback for `Token::replace()`.

The procedural functions in `user.module` now emit `E_USER_DEPRECATED` and delegate to the service. `user_pass_reset_url()` and `user_cancel_url()` are removed in 13.0.0; `UserController::validatePathParameters()` (a `protected` helper) is deprecated and removed in 12.0.0 because protected methods carry no BC promise. `ServerCommand::getOneTimeLoginUrl()` is also deprecated in 11.4.0 with no replacement and removed in 12.0.0; `ServerCommand::start()` now calls the service directly.

`UserController` gains `OneTimeAuthentication` as an optional constructor argument (defaulting to a service-container fallback with a deprecation notice when omitted), and its internal `validatePathParameters()` now delegates to `verifyHmac()`.

A key behavioral difference: `generateOneTimeLoginUrl()` returns a `\Drupal\Core\Url` object, not a string. It accepts an `$immediate` bool parameter that selects the `user.reset.login` route (previously callers appended `/login` to the URL string). The old `destination` query-string pattern that was manually concatenated is now handled via `$url->mergeOptions(['query' => ['destination' => ...]])` on the returned object.

## Upgrade

Replace calls to deprecated functions with the new service methods. The service is typehintable as `\Drupal\user\OneTimeAuthentication` for constructor injection, or retrievable via `\Drupal::service(OneTimeAuthentication::class)`.

```php
// Before
$hash = user_pass_rehash($account, $timestamp);
$login_url = user_pass_reset_url($account, $options);         // returns string
$cancel_url = user_cancel_url($account, $options);            // returns string
// In token callbacks:
$token_options = ['callback' => 'user_mail_tokens', ...];
// Deprecated method on UserController:
$valid = $this->validatePathParameters($user, $timestamp, $hash, $timeout);

// After
use Drupal\user\OneTimeAuthentication;
$ota = \Drupal::service(OneTimeAuthentication::class);

$hash      = $ota->generateHmac($account, $timestamp);
$login_url = $ota->generateOneTimeLoginUrl($account, $options)->toString(); // returns Url, call toString()
$cancel_url = $ota->generateCancelConfirmUrl($account, $options)->toString();
// For immediate login (replaces appending '/login' to the URL):
$login_url = $ota->generateOneTimeLoginUrl($account, $options, immediate: TRUE)->toString();
// In token callbacks:
$token_options = ['callback' => $ota->tokens(...), ...];
// In UserController subclasses (if overriding validatePathParameters):
$valid = $this->oneTimeAuthentication->verifyHmac($user, $timestamp, $hash, $timeout);
```

Note: `UserController::validatePathParameters()` is removed in 12.0.0 (not 13.0.0), so subclasses overriding it must migrate sooner.

### Collaboration

A substantive design debate ran through the thread on two points. First, `berdir` argued against adding a PHP interface to the service because the team does not want to support alternative implementations; this keeps it easier to add new methods without BC concerns. Second, whether to mark the class `final` triggered a lengthy discussion referencing a still-open policy issue ([#3488476](https://www.drupal.org/node/3488476)). `alexpott` reviewed the security-critical code paths (confirming `hash_equals()` was preserved) and concluded the class should ship as `final` to actively discourage extension; `catch` agreed for this specific case. `nicxvan` and `benjifisher` drove several rounds of review covering coding standards, the deprecation timeline for the protected `validatePathParameters()` method, and change-record completeness before the committer's sign-off.
