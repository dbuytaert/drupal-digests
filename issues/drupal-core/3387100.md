## Summary

When enabling content translation on entity types such as media, Drupal writes `core.base_field_override.*.*.*` config entities that may carry a `third_party_settings.content_translation` entry. No config schema existed for that key, causing config inspector and strict schema validation to report errors. The fix adds the missing schema and includes a post-update hook that resaves all affected base field override entities so that previously stored values (for example integer `0` instead of string `'0'`) are normalized to match the new schema.

## Impact

Affects site owners.

- **Database update:** New post-update hook `content_translation_post_update_resave_base_field_overrides` resaves base field override config entities that have content translation third-party settings to normalize stored values against the new schema.

## Technical details

The schema file `content_translation.schema.yml` previously defined `field.field.*.*.*.third_party.content_translation` inline. The fix extracts that mapping into a reusable named type `content_translation_field_third_party_settings` and references it from both the existing `field.field` key and the newly added `core.base_field_override.*.*.*.third_party.content_translation` key.

A new `content_translation_post_update_resave_base_field_overrides()` function in `content_translation.post_update.php` iterates all `base_field_override` entities via `ConfigEntityUpdater` and resaves only those that have `content_translation` third-party settings. Resaving forces Drupal to cast stored values through the schema, converting any integer `0` / `1` values to the string type the schema requires. A test assertion was also added to `ContentLanguageBaseFieldSyncWidgetTest::testBaseFieldSyncOptionsVisibleOnFirstLoad()` verifying that saving the content-language settings form produces no config schema errors.

## Upgrade

Run database updates after deploying this change so that existing base field override config entities with content translation settings are resaved and normalized:

```bash
drush updatedb
# or
php core/scripts/drupal.php update
```

No code changes are required in custom modules.

### Collaboration

The thread ran for nearly two years before landing. A long-standing question was whether an update hook was needed or would be too disruptive. Contributor heddn (comment #12) argued that a hook would force unwanted config exports for sites that never exported their base field overrides. Contributor bbu23 later demonstrated concretely (comment #18) that without resaving, integer `0` values stored by older Drupal versions still trigger a type-mismatch error even with the new schema present. Contributor codebymikey then added the post-update hook targeting only base field overrides that already carry `content_translation` third-party settings, limiting the resave footprint. Maintainer alexpott accelerated the final review after the related issue [#3606969](https://www.drupal.org/node/3606969) caused contrib test failures, and longwave committed the result.
