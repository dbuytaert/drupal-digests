## Summary

After deploying new code that changes OOP hook implementations (such as moving a hook to a different class), sites could end up with stale hook data in the key/value store. This happened because PHP's opcache can hold old file bytecode for up to `opcache.revalidate_freq` seconds (default: 2s). If a container cache rebuild started during that window, hook attribute parsing via reflection would read the old class definitions and persist them. Subsequent requests would use this stale hook data until another manual cache rebuild occurred. The fix ensures opcache is explicitly invalidated for each file before it is inspected for hook attributes during a rebuild.

## Impact

No upgrade or configuration changes required.

## Technical details

The hook attribute collector classes (`HookCollectorPass` for modules, and the equivalent for themes) walk PHP source files and parse their `#[Hook]` attributes using reflection. When a file has no valid entry in the file cache, the collector falls through to live reflection. The fix adds a call to `OpCodeCache::invalidate($filename)` at the top of that file-cache-miss branch in both `collectModuleHookImplementations()` and `collectThemeHookImplementations()`. `OpCodeCache::invalidate()` calls PHP's `opcache_invalidate()` under the hood; that function performs an mtime check and drops the cached bytecode for the file if it has changed on disk. This brings the opcache inline with the filesystem before reflection runs, preventing the reflection APIs from seeing stale class definitions. The change is confined to the cache-miss path, so it adds no overhead when the file cache is warm.
