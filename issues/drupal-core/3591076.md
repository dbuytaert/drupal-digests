## Summary

The locale module now stores and serves JavaScript translation files using the `assets://` stream wrapper instead of `public://`. Since Drupal 10.1/11.1 ([#3027639](https://www.drupal.org/node/3027639)), `assets://` has been available as a configurable wrapper so sites can separate generated files (aggregated CSS/JS, etc.) from user-uploaded content. JS translation files were overlooked in that change and continued to use `public://`. On sites where `assets://` is not separately configured (the default), there is no visible difference. On sites that have pointed `assets://` to a separate location, locale JS files will now land there alongside other generated assets.

## Impact

No upgrade or configuration changes required.

## Technical details

Three places in `locale.module` built the translation JS directory path as `'public://' . $config->get('javascript.directory')`: the `locale_uninstall()` cleanup routine, `locale_js_translate()`, and `_locale_rebuild_js()`. Each is now `'assets://'` instead. The `javascript.directory` config key in `locale.settings` is unchanged. Corresponding assertions in `LocaleJavascriptTranslationTest` were updated to the new stream wrapper prefix. A change record was drafted at [#3592731](https://www.drupal.org/node/3592731).

## Upgrade

Sites that have configured `assets://` to a different physical path than `public://` will have locale JS translation files placed in the new location on the next rebuild of JS translations. Any custom code or external tooling that references those files by constructing a `public://` path should be updated:

```php
// Before
$js_file = 'public://' . $config->get('javascript.directory') . '/' . $filename;

// After
$js_file = 'assets://' . $config->get('javascript.directory') . '/' . $filename;
```
