## Summary

The `menu_tree` database table's `url` and `route_param_key` columns were only 255 characters wide, which was too short for real-world use. Trying to save a menu link pointing to a long external URL, or to a route with many parameters (such as a Search API view with many facets), would produce a fatal database error: "SQLSTATE[22001]: String data, right truncated: 1406 Data too long for column." Both columns are now 2048 characters wide. Existing sites need to run `drush updb` (or visit update.php) to apply the schema change; no content is lost. A backport to Drupal 10.4.x/10.5.x is possible via the equivalent-update mechanism introduced in [#3459876](https://www.drupal.org/node/3459876).

## Impact

Affects site owners.

- **Database update:** Both `menu_tree.url` and `menu_tree.route_param_key` columns widened from VARCHAR(255) to VARCHAR(2048); run update.php or `drush updb` to apply.

## Technical details

The `menu_tree` table is managed by `MenuTreeStorage` in `core/lib/Drupal/Core/Menu/MenuTreeStorage.php`. Its static `schemaDefinition()` method defines all columns. Two columns had `'length' => 255`:

- `url`: stores the external path for menu links that do not use a route.
- `route_param_key`: stores a URL-encoded query string built from a link's route parameters (generated via `UrlHelper::buildQuery()`), used for fast keyed lookup. When a route carries many parameters, such as a Search API/Facets view with 40+ facet keys, this encoded string easily exceeds 255 characters.

The fix changes both column definitions to `'length' => 2048` (matching the length already used for the `uri` column in `core/modules/link/src/Plugin/Field/FieldType/LinkItem.php`), and adds a database update hook `system_update_11001()` in `core/modules/system/system.install` that calls `$schema->changeField()` twice to resize both columns on existing installations. A functional upgrade-path test, `MenuTreeStorageSchemaUpdateTest`, was added at `core/modules/system/tests/src/Functional/Update/MenuTreeStorageSchemaUpdateTest.php`; it starts from a Drupal 10.3.0 database fixture, verifies the columns are 255 before running updates, then asserts they are 2048 afterward. The test is skipped on SQLite, which does not expose `INFORMATION_SCHEMA.COLUMNS`. The update hook is numbered 11001 and was committed to the `11.x` and `11.1.x` branches; it has since been rolled past by `system_update_last_removed()` returning `11300` in the current `main` branch, confirming the schema change is present in all subsequent releases.

## Upgrade

Sites upgrading from any Drupal version that predates 11.1 must run the database update to resize the columns:

```bash
drush updb
```

or visit `/update.php` in the browser. The update hook is `system_update_11001` and is non-destructive: it only widens the columns and does not alter any stored data.

For Drupal 10 sites that cannot yet upgrade to 11.x, a community-maintained patch is available on the issue at [#3106205](https://www.drupal.org/node/3106205). Use a safe update hook number for your minor version branch (e.g., `10401` for a 10.4.x site, `10501` for a 10.5.x site) to avoid conflicting with future official core updates. See the related issue [#3459876](https://www.drupal.org/node/3459876) for the equivalent-update mechanism that will eventually enable an official 10.x backport.

### Collaboration

The issue sat unresolved for nearly five years while the update-numbering problem was worked out. The core debate, led by catch and longwave, was whether it was safe to commit the same schema change to multiple branches with different update hook numbers. catch explained (comments #17 and #20) why simply using different numbers in 8.9 and 9.2 was not sufficient: an upgrade path from 8.9 directly to 9.1 would silently skip the update because it would never appear in that branch. This discussion was tracked in the related issue [#3108658](https://www.drupal.org/node/3108658) on handling update path divergence. The practical impact of `route_param_key` being too short was only confirmed later, in comments #37 and #39, when devad and fmb reported real truncation errors from Search API views with 40+ facets, which finally justified expanding that column alongside `url`. plopesc added the upgrade-path test coverage in July 2024 after smustgrave flagged its absence, and smustgrave and pwolanin reviewed the final version before alexpott committed it to `11.x` and `11.1.x` in November 2024.
