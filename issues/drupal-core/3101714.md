## Summary

Link fields configured to display URLs as plain text (the "URL only" or "Show URL as plain text" formatter options) no longer truncate the URL at 80 characters. Previously, the default 80-character trim was applied regardless of display mode, which silently produced broken URLs whenever a URL exceeded that length and had no separate title. The trim remains active for standard hyperlinked display, where truncating the visible text does not break the underlying link. This change affects both new and existing link fields at render time.

## Impact

Affects site owners.

- **Config change:** Existing link fields with `url_only` or `url_plain` enabled and a non-zero `trim_length` will now render full URLs at display time, regardless of the stored `trim_length` value.

## Technical details

In `LinkFormatter::viewElements()`, the call to `Unicode::truncate()` was previously guarded only by `!empty($settings['trim_length'])`. The condition is now `!empty($settings['trim_length']) && empty($settings['url_plain']) && empty($settings['url_only'])`, so trimming fires only when the link is rendered as a standard anchor tag. The same guard was added to `settingsSummary()`, so the "Link text trimmed to N characters" message is also suppressed when it would be inaccurate. In the formatter settings form, the `trim_length` element gains a `#states` rule that hides it in the UI when `url_only` is checked. The default value of `trim_length` (80) is unchanged in `defaultSettings()`; what changed is whether that value is honored at render time. Tests in `LinkFormatterDisplayTest` and `LinkSeparateFormatterDisplayTest` add two new input fixtures: a URL longer than 80 characters with an explicit title, and the same URL with no title. Expected results confirm that plain-text and URL-only display modes now render the full URL while anchor-tag display still applies the trim.

### Collaboration

The thread had been open for over seven years and went through several design pivots. The original report flagged that plain-text URLs were being broken, but the issue stalled partly because the bug was hard to reproduce on a fresh install without a specific template. The key design shift came from alexpott (comment [#15621679](https://www.drupal.org/node/15621679)), who argued that the formatter does two fundamentally different things: rendering an anchor tag (where trimming the visible text is harmless) versus emitting raw text (where trimming produces a broken URL). That framing redirected the fix from simply removing the 80-character default entirely to a targeted bypass that only skips the trim in plain-text display modes, preserving the trim for anchor-tag rendering where it still makes sense.
