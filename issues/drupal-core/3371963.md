## Summary

Drupal core's Nightwatch browser-testing library has been updated from version 2.4.2 to 3.7.0 (targeting Drupal 11.1.x). This keeps the JavaScript testing toolchain current and allows the test suite to run against a modern Selenium and Chrome stack. The update was blocked for over a year by W3C WebDriver compatibility issues that required fixes to existing tests before the version bump could land. The change has no effect on site behaviour; it only affects developers running Nightwatch tests.

## Impact

No upgrade or configuration changes required.

## Technical details

Nightwatch 3.x enables W3C WebDriver mode by default, which introduced two concrete incompatibilities with existing Drupal Nightwatch tests and helpers.

First, the `domain` property in cookie objects is handled differently under W3C mode. The `drupalInstall.js` custom command sets a `SIMPLETEST_USER_AGENT` cookie during installation; the `domain: url.host` field was removed from that cookie object in `core/tests/Drupal/Nightwatch/Commands/drupalInstall.js` to make cookie creation succeed.

Second, the `api.execute()` method signature changed: in Nightwatch 3.x the callback must be passed as the third argument, with an explicit arguments array as the second. Two custom assertion files — `core/tests/Drupal/Nightwatch/Assertions/deprecationErrorExists.js` and `core/tests/Drupal/Nightwatch/Assertions/noDeprecationErrors.js` — called `this.api.execute(fn, callback)` (two arguments), which silently caused tests to hang because Nightwatch treated the callback as the arguments array. Both were updated to `this.api.execute(fn, [], callback)`.

In `core/package.json`, the `nightwatch` dependency moved from the pinned `2.4.2` to `^3.7.0`, and the `resolutions` section that previously pinned `glob/jackspeak` to `2.1.1` and `nightwatch/semver` to `~7.5.2` (workarounds for dependency conflicts in the old version) was removed entirely. The transitive dependency tree updated accordingly: `selenium-webdriver` jumped from 4.3.1 to 4.22.0, `mocha` from 9.2.2 to 10.3.0, `nightwatch-axe-verbose` from 2.0.3 to 2.3.1 (pulling in `axe-core` 4.10.0), and `dotenv` from 10.0.0 to 16.3.1, among many other packages in `core/yarn.lock`.

A prerequisite for this update landing was [#3421202](https://www.drupal.org/node/3421202), which switched the GitLab CI test environment to a W3C-compliant Selenium and Chrome setup. The issue also uncovered that DrupalCI was reporting failing Nightwatch jobs as green (tracked separately as a follow-up).

## Upgrade

If your contrib module or custom code has Nightwatch tests that call `browser.execute()` or `api.execute()` with the callback as the second argument, update those calls to pass an explicit arguments array as the second argument and the callback as the third:

```js
// Before (Nightwatch 2.x)
this.api.execute(function () { return someValue; }, callback);

// After (Nightwatch 3.x)
this.api.execute(function () { return someValue; }, [], callback);
```

If you set cookies with a `domain` property in W3C mode (the default in Nightwatch 3.x), remove the `domain` field or ensure it is W3C-compliant. See the [Nightwatch 3.x upgrade guide](https://nightwatchjs.org/guide/overview/whats-new-in-v3.html) for the full list of breaking changes.
