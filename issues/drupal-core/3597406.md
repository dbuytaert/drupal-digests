## Summary

A regression introduced in [#3453474](https://www.drupal.org/node/3453474) (which added `DrupalApplication` and deprecated `BootableCommandTrait`) accidentally dropped support for the `DRUPAL_DEV_SITE_PATH` environment variable. That variable is the standard way for functional tests to tell the `dr` command which site directory to use. Without it, recipe-related functional tests in projects such as Drupal CMS could no longer isolate `dr` to the correct test site. This fix restores the env-var lookup in `DrupalApplication` and also corrects how `RecipeTestTrait` invokes `dr` so that it works correctly in Composer-based installs (such as `drupal/core-recommended`) rather than only in raw core checkouts.

## Impact

No upgrade or configuration changes required.

## Technical details

In `DrupalApplication::bootstrap()`, the site path resolution now checks `getenv('DRUPAL_DEV_SITE_PATH')` first and falls back to `DrupalKernel::findSitePath()` only when the env var is absent. This mirrors the behaviour that existed in the now-deprecated `BootableCommandTrait::getSitePath()`.

`RecipeTestTrait::runDrupalCommand()` received a related fix: it previously invoked `core/scripts/dr` relative to `$this->root` (the core root). It now uses `vendor/bin/dr` relative to the Composer project root, retrieved via `Composer\InstalledVersions::getRootPackage()['install_path']`. This ensures the correct class loader is in place and makes the trait work in installations where core lives under `vendor/drupal/core` rather than at the repository root.
