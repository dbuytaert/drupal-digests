## Summary

Five procedural functions in the locale module that managed translation status have been deprecated in Drupal 11.4.0 and will be removed in 13.0.0. `locale_translation_get_status`, `locale_translation_status_save`, `locale_translation_status_delete_languages`, `locale_translation_status_delete_projects`, and `locale_translation_clear_status` are replaced by equivalent methods on the `LocaleSource` service. Custom modules or install profiles that call any of these functions must update before upgrading to Drupal 13.

## Impact

Affects module maintainers.

- **Deprecation:** Five locale procedural functions deprecated in 11.4.0 and removed in 13.0.0: `locale_translation_get_status`, `locale_translation_status_save`, `locale_translation_status_delete_languages`, `locale_translation_status_delete_projects`, and `locale_translation_clear_status` replaced by methods on the `LocaleSource` service.
- **New API:** New `LocaleTranslationSource` value object class and new methods `loadSources()`, `loadSource()`, `saveSource()`, `deleteSources()`, `deleteSourcesByLanguage()`, `clearSources()`, `updateLastChecked()`, and `getLastChecked()` added to the `LocaleSource` service.

## Technical details

The `LocaleSource` service gained new public methods: `loadSources()`, `loadSource()`, `saveSource()`, `deleteSources()`, `deleteSourcesByLanguage()`, `clearSources()`, `updateLastChecked()`, and `getLastChecked()`. The key efficiency gain is that `loadSources()` now calls `getMultiple()` on the `locale.translation_status` key-value store for only the requested projects, rather than the unconditional `loadAll()` that the old `locale_translation_get_status()` always performed. Batch operations in `LocaleFetch` and `LocaleProjectChecker` were updated to call `loadSource()` per project/langcode, so each batch item touches only its own data.

A new typed value object `LocaleTranslationSource` (in `Drupal\locale`) replaces the anonymous `stdClass` that `sourceBuild()` previously returned. It carries the same public properties (`project`, `langcode`, `type`, `timestamp`, `hash`, `last_checked`, `files`, `version`, `server_pattern`, `core`, `name`) and is instantiated via `LocaleTranslationSource::fromProject()`. The `last_checked` timestamp previously stored in `\Drupal::state()` under `locale.translation_last_checked` is now stored inside the `locale.translation_status` key-value collection under the key `last-checked`, accessed only through `LocaleSource::getLastChecked()` and `updateLastChecked()`.

The installer was further simplified in the second commit: `_install_prepare_import()` was removed entirely, and `install_import_translations()` now delegates directly to `\Drupal::service(LocaleFetch::class)->buildUpdateBatch()`. Installer test fixtures that previously hardcoded `drupal-8.0.0.*` translation filenames now use `\Drupal::VERSION` so the filename matches the running codebase, removing an entire category of version-matching logic that `_install_prepare_import()` had handled as a special case. `locale_translation_build_sources()` deprecation text was also corrected to point to `sourceBuild()` rather than the removed `buildSources()` method.

## Upgrade

Replace the deprecated procedural calls with the `LocaleSource` service:

```php
use Drupal\locale\LocaleSource;

$localeSource = \Drupal::service(LocaleSource::class);

// locale_translation_get_status($projects, $langcodes)
$sources = $localeSource->loadSources($projects, $langcodes);

// locale_translation_status_save($project, $langcode, $type, $data)
$localeSource->saveSource($project, $langcode, $type, $data);

// locale_translation_status_delete_languages($langcodes)
foreach ($langcodes as $langcode) {
  $localeSource->deleteSourcesByLanguage($langcode);
}

// locale_translation_status_delete_projects($projects)
$localeSource->deleteSources($projects);

// locale_translation_clear_status()
$localeSource->clearSources();
```

The return type of `loadSources()` is `LocaleTranslationSource[][]` rather than `stdClass[][]`. Code that previously assigned properties directly to the returned objects via `$status[$project][$langcode]->type = 'current'` continues to work because `LocaleTranslationSource` uses public typed properties, but the class must be imported.

If you previously read the last-checked timestamp directly from state (`\Drupal::state()->get('locale.translation_last_checked')`), use `$localeSource->getLastChecked()` instead. That state key is no longer written by core.

nicxvan and berdir collaborated closely on this issue. berdir ported analysis notes from the MR thread identifying the layered inefficiency: every call to `locale_translation_get_status()` loaded all project data from key-value regardless of which project was needed, and callers then called `buildSources()` again inside loops, producing redundant work. nicxvan uncovered the tricky installer failure caused by `_install_prepare_import()` relying on version-string side effects: `\Drupal::service(LocaleProjectRepository::class)->set($project)` was being called with the version extracted from the `.po` filename (e.g. `8.0.0`), implicitly updating the project version so subsequent lookups matched. nicxvan diagnosed this by manually tracing the installer steps and confirmed in comment 21 that explicitly setting the file URI and filename on the loaded source and calling `saveSource()` was the correct replacement. berdir summarized in comment 32 that the remaining test failures on the forward-port were due to tests still constructing filenames with the old major-version pattern, and opened follow-up issues for further cleanup.
