## Summary

Drupal kernel tests no longer allow field storage definitions to be created before the target entity's database schema has been installed. This ordering requirement was always implicit but unenforced, and stricter database drivers such as MongoDB made the problem visible with hard failures. A new event subscriber now detects the violation at test time: in Drupal core tests it immediately throws a `LogicException`, while in contrib and custom module tests it triggers a deprecation warning pointing to the Drupal 12.0.0 deadline. Several core kernel tests were also corrected in the same commit to comply with the new rule.

## Impact

Affects module maintainers.

- **Deprecation:** Creating a field storage definition in a kernel test before installing the target entity's schema is deprecated in drupal:11.2.0 and will throw a LogicException in drupal:12.0.0; call `$this->installEntitySchema()` first.

## Technical details

The root cause is that some kernel tests called `$this->installConfig()` or created field storage config entities before calling `$this->installEntitySchema()` for the entity type the field targets. For SQL-based drivers this happened to work because fields are stored in separate tables that can be created independently, but document-oriented drivers like MongoDB embed fields inside the entity document and require the entity schema to exist first.

The fix has two parts.

**New enforcement subscriber.** `core/lib/Drupal/Core/Test/EventSubscriber/FieldStorageCreateCheckSubscriber.php` implements `EventSubscriberInterface` and listens on `FieldStorageDefinitionEvents::CREATE`. In `onFieldStorageDefinitionCreateEvent()` it retrieves the entity type's storage, checks whether the storage is a `SqlEntityStorageInterface`, and then calls `$this->schema->tableExists($base_table)`. If the base table is absent it either throws a `\LogicException` (strict mode) or calls `@trigger_error()` with an `E_USER_DEPRECATED` message: `"Creating the \"<id>\" field storage definition without the entity schema \"<entity_type>\" being installed is deprecated in drupal:11.2.0 and will be replaced by a LogicException in drupal:12.0.0."` The mode is controlled by the constructor parameter `$throwLogicExceptionOrDeprecationWarning`.

**Registration in `KernelTestBase`.** In `KernelTestBase::register()` (in `core/tests/Drupal/KernelTests/KernelTestBase.php`) the subscriber is unconditionally added to the test container. The `$is_core_test` flag — already computed for the config schema checker — is reused: core tests receive `TRUE` (strict, throws), non-core tests receive `FALSE` (deprecation only). The `@todo` in the code points to [#3395099](https://www.drupal.org/node/3395099) for the future policy decision on contrib.

**Existing test corrections.** More than 20 kernel test `setUp()` methods across modules including `comment`, `content_moderation`, `field`, `language`, `layout_builder`, `media`, `menu_ui`, `node`, `text`, `views`, and several core test directories were updated to call `$this->installEntitySchema()` before `$this->installConfig()` or before creating field storage config entities. In `EntityKernelTestBase::setUp()` automatic installation of `taxonomy_term` schema was added when the `taxonomy` module is present in `$modules`.

**New tests.** `FieldStorageCreateCheckTest` verifies the `LogicException` path (strict mode). `FieldStorageCreateCheckDeprecationTest` overrides the container registration to force `$throwLogicExceptionOrDeprecationWarning = FALSE` and then asserts the specific deprecation message is emitted, covering the non-core-test path.

## Upgrade

Contrib and custom module kernel tests that create field storage definitions (via `FieldStorageConfig::create()` or equivalent) must now call `$this->installEntitySchema()` for the target entity type **before** doing so, or they will trigger a deprecation warning in Drupal 11.2 and a `LogicException` in Drupal 12.

```php
// Before (incorrect order):
$this->installConfig(['node']);
// ... creates field storage on node entity ...

// After (correct order):
$this->installEntitySchema('node');
$this->installConfig(['node']);
// ... creates field storage on node entity ...
```
