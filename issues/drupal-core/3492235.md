## Summary

Installing Drupal or enabling many modules at once is significantly faster and uses far less memory in Drupal 11.2. Previously, the extension system assumed every module needed a full dependency-injection container rebuild unless it explicitly said otherwise, which caused one or more expensive rebuilds per module during installation. Now the logic is reversed: no container rebuild happens during module installation unless a module explicitly declares `container_rebuild_required: true` in its `.info.yml`. This change reduced peak memory during a typical install-profile run from around 1.6 GB to around 200 MB. Contrib modules that register services or otherwise modify the container in ways that affect the installation of other modules must add that flag to their `.info.yml`.

## Impact

Affects module maintainers.

- **API change:** `container_rebuild_required` in module `.info.yml` now defaults to `false` when absent; modules that modify the container during install must explicitly set `container_rebuild_required: true`.

## Technical details

This issue is a follow-up to [#3416522](https://www.drupal.org/node/3416522), which introduced batch-based container rebuilding to `ModuleInstaller` and used a `container_rebuild_required` key in module `.info.yml` files to control the behaviour. In that earlier issue the key defaulted to `TRUE` when absent, to preserve backward compatibility with any module that relied on a fresh container being available during install hooks.

The cost of that conservative default is a full attribute-discovery container rebuild for nearly every module installed, including all modules that do not touch the container at all. A memory leak in attribute discovery makes each rebuild expensive.

The fix is a two-character change in two symmetrical places. In `core/includes/install.core.inc` (`install_profile_modules()`) and in `core/lib/Drupal/Core/Extension/ModuleInstaller.php` (`ModuleInstaller::install()`), the old expression:

```php
!isset($module_data[$module]->info['container_rebuild_required']) || $module_data[$module]->info['container_rebuild_required']
```

is replaced with:

```php
!empty($module_data[$module]->info['container_rebuild_required'])
```

This makes an absent key evaluate to `FALSE` instead of `TRUE`.

As a cleanup, the `container_rebuild_required: false` lines that had been explicitly added to approximately 70 core module `.info.yml` files (to document the old default) are all removed, since they are now redundant.

The change was gated on [#3492705](https://www.drupal.org/node/3492705) landing first, which guarantees that any module that does set `container_rebuild_required: true` receives its own isolated rebuild batch, making the new default safe for modules that do need it.

## Upgrade

`container_rebuild_required: true` must be added to the `.info.yml` of any contrib or custom module that:
- Registers services that other modules in the same install batch consume during their `hook_install()`, or
- Uses a `ServiceProvider` class to decorate or alter container services in ways that affect how subsequently-installed modules behave.

```yaml
# mymodule.info.yml
name: My Module
type: module
description: 'Registers a service used by other modules at install time.'
version: VERSION
container_rebuild_required: true
```

Modules that do not modify the container at all need no change; the absence of the key is now the correct state.

### Collaboration

The thread started with a more complex proposal: automatically detect which modules needed a rebuild by inspecting their dependency graph or checking for the presence of a `ServiceProvider` PHP class. catch sketched out both approaches across several comments. mikelutz then observed that the install batch already performs a container rebuild before running any install hooks for that batch, making the scenario where `hook_install()` calls a service from a just-installed dependency much less of a concern than originally thought. That analysis convinced the group that a simple default flip was sufficient, and catch dropped the complex MR (!10494) in favour of the straightforward one-line change (!10495).
