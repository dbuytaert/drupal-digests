## Summary

Drupal controllers that return PSR-7 responses backed by non-seekable stream bodies (such as those produced by the MCP SDK or other third-party streaming SDKs) would previously respond incorrectly because `PsrResponseSubscriber` never told the Symfony bridge to treat the response as streamed. The subscriber now inspects the PSR-7 body's seekability and passes the appropriate flag, so non-seekable streams are converted to a proper Symfony `StreamedResponse`.

## Impact

No upgrade or configuration changes required.

## Technical details

`PsrResponseSubscriber::onKernelView()` previously called `HttpFoundationFactoryInterface::createResponse($controller_result)` with no second argument, meaning `$streamed` always defaulted to `false`. The Symfony bridge's `createResponse` uses that flag to decide whether to produce a `StreamedResponse` (which reads the body lazily) or a regular `Response` (which buffers the body). Non-seekable PSR-7 streams cannot be rewound after reading, so buffering them silently produces an empty or corrupt response body.

The fix replaces the hard-coded call with `$streamed = !$controller_result->getBody()->isSeekable()` and passes that value as the second argument. Seekable bodies continue to produce a regular `Response`; non-seekable bodies now produce a `StreamedResponse`. Two new unit tests cover the two branches using `StreamInterface` mocks with controlled `isSeekable()` return values and mock expectations on `createResponse`.
