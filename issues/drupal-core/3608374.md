## Summary

When loading entity types such as `file` that have no revision table, no data table, and no dedicated field tables, Drupal was issuing a redundant SQL query against the base table on every entity load. The extra query produced identical data to one already fired by `getFromStorage()`, so nothing useful came from it. This fix eliminates that query, cutting database load for content pages that reference files. Performance tests for the Umami demo profile show warm-cache query counts dropping by 5 queries per page request.

## Impact

No upgrade or configuration changes required.

## Technical details

In `SqlContentEntityStorage::loadFromDedicatedTables()`, the method builds a list of field "chunks" to fetch from dedicated and shared tables. When the count of single-cardinality fields does not exceed `FIELD_MINIMUM_CHUNK_SIZE`, the old `else` branch unconditionally set `$chunks = [$single_cardinality_fields]`, even when `$single_cardinality_fields` was empty and the entity type had no `$this->dataTable`, `$this->revisionTable`, or `$this->revisionDataTable`. This produced a `$chunks` value of `[0 => []]`, which caused the `foreach` loop to execute once with an empty fields list. That in turn triggered a `SELECT *` against the base table (e.g. `file_managed`) that duplicated the query already issued by `buildQuery()` in `getFromStorage()`.

The fix initializes `$chunks = []` before the conditional, then changes the `else` to an `elseif` that only creates the single-element chunk array when there is actually something to load: non-empty `$single_cardinality_fields`, or at least one of `$this->dataTable`, `$this->revisionTable`, or `$this->revisionDataTable`. With none of those present, the loop is skipped entirely and the redundant base-table query is never issued.
