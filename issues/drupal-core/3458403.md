## Summary

Visiting any Update Manager page (install modules/themes, run updates) forced the Dynamic Page Cache to treat that page and pages that link to it as entirely uncacheable, causing a measurable performance regression introduced in 10.3. This fix removes the runtime uncacheability by moving the `allow_authorize_operations` settings check from request time to route compile time, so sites that disable the authorize operations feature get a static `_access: FALSE` on those routes, and sites that leave it enabled get only a normal permission check. The change was backported to 10.3.x, 10.4.x, and 11.0.x on the same day it landed on 11.x.

## Impact

Affects module maintainers.

- **Deprecation:** `UpdateManagerAccessCheck` deprecated in drupal:11.1.0 and removed in drupal:12.0.0 with no replacement

## Technical details

The root cause was `UpdateManagerAccessCheck::access()`, a custom route access check registered as `_access_update_manager` on seven Update module routes (`update.report_install`, `update.report_update`, `update.module_install`, `update.module_update`, `update.theme_install`, `update.theme_update`, `update.confirmation_page`). That method called `setCacheMaxAge(0)` on its result because the `allow_authorize_operations` value comes from `Settings`, which is considered changeable at any time. Any page that evaluated this access check, including pages whose local tasks or local actions linked to these routes, became uncacheable. The issue was surfaced in the Admin Toolbar project ([#3456128](https://www.drupal.org/node/3456128)).

The fix replaces the per-request check with a route-build-time check using a new `UpdateRouteSubscriber` (in `core/modules/update/src/Routing/UpdateRouteSubscriber.php`), which extends `RouteSubscriberBase`. Its `alterRoutes()` method reads `allow_authorize_operations` once during route building: if the setting is `FALSE`, it iterates the seven affected routes and calls `$route->setRequirement('_access', 'FALSE')`, permanently denying them at compile time with no runtime overhead. If the setting is `TRUE` (the default), the routes are left unchanged, relying solely on the existing `_permission: 'administer software updates'` requirement, which is fully cacheable.

The `_access_update_manager: 'TRUE'` requirement was removed from all seven routes in `update.routing.yml`. The `update.route_subscriber` service was registered in `update.services.yml` with `@settings` injected.

On 11.x, `UpdateManagerAccessCheck` was deprecated (with `@trigger_error` in `access()`) for removal in `drupal:12.0.0` with no replacement, as documented at `https://www.drupal.org/node/3458658`. The 10.3.x and 10.4.x backports did not include the deprecation trigger, only the new subscriber and routing changes.
