## Summary

A new database-backed cache bin, `cache.file_parsing`, is introduced to store the results of YAML file parsing across cache clears. Unlike the existing `FileCache` (APCu), this bin is intentionally excluded from the cache-clear tag system, so its contents survive `drush cr` and the UI cache-clear form. Invalidation happens via file modification time checks instead. Library definition parsing (`*.libraries.yml`) and route YAML discovery now use this persistent cache, reducing redundant file reads and freeing APCu for higher-frequency data. Sites with multiple web heads or mixed CLI and web processes will benefit most, since APCu is process-local while the new bin is database-backed and shared.

## Impact

Affects module maintainers.

- **Deprecation:** `LibraryDiscoveryParser::__construct()` and `YamlRouteDiscovery::__construct()` require a new `$yamlCacheCollector` argument; omitting it is deprecated in drupal:11.4.0 and will be required in drupal:12.0.0.
- **New API:** New `FileParsingCacheCollectorBase`, `YamlCacheCollector`, and `YamlCacheCollectorDiscovery` classes provide a reusable, persistent file-parse caching layer backed by the new `cache.file_parsing` bin.

## Technical details

The fix introduces an abstract `FileParsingCacheCollectorBase` class in `Drupal\Core\Utility`, extending `CacheCollector`. Each cache item stores an array keyed by absolute file path, where each entry holds the file's `mtime` and the parsed result. On `::get($key)` the collector loads the whole item lazily with one cache read, then validates `filemtime()` per file before returning the stored data; a mtime mismatch falls through to `::resolveCacheMiss()` which re-parses the file. The `::updateCache()` override adds garbage collection: any file entry whose stored mtime is older than 90 days and whose file no longer exists is removed before the parent writes back. `CacheCollector::destruct()` now also calls `$this->reset()` after `updateCache()` to release the in-memory storage.

`YamlCacheCollector` (in `Drupal\Core\Utility`) is the concrete subclass that implements `::parseFile()` via `Yaml::decode()`. `YamlCacheCollectorDiscovery` (in `Drupal\Core\Discovery`) subclasses `YamlDiscovery` and accepts a `CacheCollectorInterface`; it overrides `::findAll()` and `::decode()` to delegate file reads to the collector, then calls `::destruct()` when discovery finishes to release memory immediately.

Two services are wired in `core.services.yml`: `library.parsing_cache` (a `YamlCacheCollector` backed by `cache.file_parsing`, tagged `needs_destruction`) feeds into `library.discovery.parser`, and `router.yaml_cache_collector` feeds into `router.builder.yaml` which now uses `YamlCacheCollectorDiscovery` instead of bare `YamlDiscovery`. The `cache.file_parsing` bin is defined without the `cache.bin` service tag so standard cache invalidation logic skips it entirely.

`LibraryDiscoveryParser::__construct()` gains a new optional `?YamlCacheCollector $yamlCacheCollector` argument and also converts its property declarations to constructor promotion. `YamlRouteDiscovery::__construct()` gains an optional `?CacheCollectorInterface $yamlCacheCollector` argument. Both emit an `E_USER_DEPRECATED` if the argument is omitted, noting the argument will be required in 12.0.0. The second commit (`5768a3d2`) moves `YamlCacheCollectorTest` from `Drupal\Tests\Core\Cache` to `Drupal\Tests\Core\Utility` to match the class location.

## Upgrade

If you directly instantiate `LibraryDiscoveryParser` or `YamlRouteDiscovery`, pass a `YamlCacheCollector` as the final constructor argument to avoid the deprecation warning:

```php
use Drupal\Core\Utility\YamlCacheCollector;

// For library discovery:
$yaml_cache_collector = new YamlCacheCollector(
  'library.parsing_cache',
  $cache_file_parsing_backend, // the cache.file_parsing service
  $lock,
  $time,
);
$parser = new LibraryDiscoveryParser(
  $root, $module_handler, $theme_manager, $stream_wrapper_manager,
  $libraries_directory_file_finder, $extension_path_resolver,
  $component_plugin_manager,
  $yaml_cache_collector,  // new required argument in 12.0.0
);

// For route discovery:
$yaml_cache_collector = new YamlCacheCollector(
  'router.parsing_cache',
  $cache_file_parsing_backend,
  $lock,
  $time,
);
$discovery = new YamlRouteDiscovery(
  $module_handler,
  $controller_resolver,
  $yaml_cache_collector,  // new required argument in 12.0.0
);
```

If you inject these classes as services, Drupal's service container will supply the argument automatically once you update your service definition to match the new signature.

### Collaboration

During the first (abandoned) implementation attempt in late 2024, alexpott raised the concern about the `InstallerExistingSettingsTest` table-creation expectation that had been added to prevent cache pollution during early install phases. catch worked through the issue and concluded that the new bin's mtime-based invalidation makes it safe to create the table in that early installer state, unlike the regular cache backends the test was originally guarding against. alexpott and catch also discussed the database connection handling: after reviewing @daffie's concern about `::isActiveConnection()` vs `::getConnectionInfo()`, they agreed to use `getConnectionInfo('default')` to check availability and then `getConnection('default', 'default')` for explicit key and target, since the class can create database tables and there was evidence FileCache is called by `run-tests.sh` when rendering test results.

In the second (successful) attempt, berdir raised through a Slack thread that the per-file database approach added roughly 1700 database queries on a typical setup, and that for small YAML files the database lookups were slower than re-parsing. That observation was a key factor in the redesign: rather than one database query per file (the first MR's approach), the final implementation uses the `CacheCollector` pattern so all files for a given discovery type are stored in a single cache item, resulting in one database read and one write per collector lifetime.
