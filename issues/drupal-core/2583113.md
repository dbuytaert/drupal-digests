# #2583113: Config import from a different site fails because of shortcut.set.default.yml

- **Project:** Drupal Core

- **Status:** Fixed

## Summary

When importing configuration from one Drupal site to another after a fresh standard-profile install, the config import fails with the message "Entities exist of type Shortcut link and Shortcut set Default. These entities need to be deleted before importing." Each fresh install generates a new UUID for the default shortcut set, which conflicts with the UUID stored in the exported configuration from the original site. No code change was made to Drupal core; this is the expected behavior when using a post-install config import workflow. The correct approach is Drupal's native "install from config" feature, which preserves the staged config's UUIDs during installation and avoids the collision entirely. Workarounds documented in the issue include: deleting all shortcut set entities before importing (`drush entity:delete shortcut_set`), or replacing the UUID in `shortcut.set.default.yml` in the config sync directory with the UUID from the target site's active config. The issue was closed as a support request.

## Impact

No upgrade or configuration changes required.

## Technical details

The error originates in `core/lib/Drupal/Core/Entity/Event/BundleConfigImportValidate.php`. Its `onConfigImporterValidate()` method iterates over every config name in the importer's delete list and checks whether that config entity type is the bundle type for another entity type. If so, it queries for content entities using that bundle and, if any exist, calls `$event->getConfigImporter()->logError()` to block the import.

The collision arises like this: `core/modules/shortcut/config/install/shortcut.set.default.yml` deliberately carries no `uuid` field, so Drupal assigns a fresh UUID every time the shortcut module is installed. The exported `shortcut.set.default.yml` from the source site therefore carries a different UUID than the entity already present on the freshly installed target site. From the config importer's perspective, the staged config is a new entity and the existing one must be deleted before the new one can be created. That proposed deletion triggers `BundleConfigImportValidate`, which finds existing `shortcut` content entities belonging to the `default` bundle (created by `standard.install` as part of the standard profile installation) and blocks the import.

The root issue is the two-step workflow: install with the standard profile (which creates shortcut content), then run `drush config:import`. This forces the importer to delete and re-create every config entity with a differing UUID, including bundle types that already have content entities. Drupal's "install from config" feature (tracked in [#2788777](https://www.drupal.org/node/2788777) and natively supported since Drupal 8.6) sidesteps this entirely by reading the staged config directory during installation, preserving its UUIDs, and never creating conflicting default content.
