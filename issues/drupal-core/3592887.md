## Summary

Sites running modules that include "compatibility shim" controllers, such as TFA 1.x on Drupal 11.4, were seeing a PHP fatal error during cache clear or `drush cr`. The attribute-based route discovery introduced in [#3311365](https://www.drupal.org/node/3311365) scanned every PHP class file in a module and tried to autoload it. If a class had an incompatible method signature relative to its parent, PHP raised a fatal error that was not caught, bringing down the routing system. This fix makes the discovery process skip such classes gracefully instead of crashing.

## Impact

No upgrade or configuration changes required.

## Technical details

Two changes land in `AttributeRouteDiscovery`. First, a fast pre-filter: before autoloading a class, the file's raw contents are read and checked with `str_contains()` for the strings `#[Route` or `Routing\Attribute\Route`. Files that match neither string are skipped entirely, avoiding the autoloader for the vast majority of classes. The approach has acknowledged edge cases (a comment containing `#[Route` is a false positive) but these are an accepted tradeoff since route discovery results are cached. Second, the `class_exists()` call inside `createRouteCollection()` is wrapped in a `try/catch (\Error)` block so that classes which do pass the string filter but still fail to autoload (e.g., a class extending a parent from a missing module) are silently skipped instead of propagating a fatal error. The catch targets `\Error` specifically; the commit comment notes that missing traits only became catchable as `\Error` in PHP 8.5, so the test fixture for this case uses a missing parent class instead, which is catchable on all supported PHP versions. Two intentionally invalid test fixtures are added to `router_test_directory`: `TestInvalidController` (an abstract method on a non-abstract class) and `TestMissingDependency` (a `#[Route]`-annotated class that extends a class from a non-existent module). Both are excluded from PHPStan. The broader challenge of centralizing fatal-error handling across all attribute discovery subsystems is tracked separately in [#3586328](https://www.drupal.org/node/3586328).
