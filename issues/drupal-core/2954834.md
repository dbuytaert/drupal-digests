## Summary

Site owners can now configure a poster image for HTML5 video fields without any custom code or template overrides. By adding an image field to the entity type that holds the video and selecting it in the "Manage Display" settings for the video formatter, the image is rendered as the HTML5 `poster` attribute on the `<video>` element. An image style can optionally be applied to control dimensions. This works on any entity type that has both a video file field and an image field, not only media entities.

## Impact

Affects module maintainers and site owners.

- **Config change:** New `poster` and `poster_image_style` `machine_name` keys added to the `field.formatter.settings.file_video` config schema.
- **New API:** `FileVideoFormatter` adds `poster` and `poster_image_style` formatter settings with supporting `viewElements()`, `calculateDependencies()`, and `onDependencyRemoval()` overrides; subclasses must account for the new constructor parameters.

## Technical details

The `FileVideoFormatter` class in `core/modules/file/src/Plugin/Field/FieldFormatter/FileVideoFormatter.php` gains two new settings: `poster` (the machine name of an image field on the host entity) and `poster_image_style` (an image style ID, optional). The formatter is converted to use constructor injection, adding `AccountInterface`, an `EntityStorageInterface` for image styles, and `EntityFieldManagerInterface` as constructor arguments alongside the existing parent arguments.

In `settingsForm()`, `EntityFieldManagerInterface::getFieldDefinitions()` is called to enumerate image-type fields on the current entity bundle, which are offered as `poster` candidates via a select element. A `poster_image_style` select is conditionally shown (using `#states`) only when a poster field is selected, and includes a "Configure Image Styles" link visible to users with the `administer image styles` permission. The `settingsSummary()` output is extended to include poster field and image style selections when set.

In the new `viewElements()` override, when a poster field is configured and not empty, the code retrieves the first image file item from the host entity, applies a language-aware translation check (`TranslatableInterface::hasTranslation()`), then either calls `ImageStyle::buildUrl()` (if a style is configured) or `File::createFileUrl()` (for the original). A `CacheableMetadata` bubble is applied for both the image style config entity and the file entity, and the resolved URL is merged onto the video element's `#attributes` object via `Attribute::merge()`.

`calculateDependencies()` and `onDependencyRemoval()` are implemented to track the chosen `poster_image_style` as a config dependency and substitute a replacement style if the original is deleted.

Config schema at `field.formatter.settings.file_video` gains two new `machine_name` keys: `poster` and `poster_image_style`, both with `requiredKey: false`. New functional tests in `FileVideoPosterFormatterTest` cover both the no-image-style and image-style paths.

Transcript (`<track>`) support was originally part of this issue but was split into a sibling issue to unblock the more widely needed poster feature. Note that a `settingsSummary()` guard for a `transcript` setting appears in the diff as a leftover; since no `transcript` default or schema key exists, it is inert.

## Upgrade

If you subclass `FileVideoFormatter`, update your constructor to accept and pass through the three new injected services:

```php
public function __construct(
  $plugin_id,
  $plugin_definition,
  FieldDefinitionInterface $field_definition,
  array $settings,
  $label,
  $view_mode,
  array $third_party_settings,
  AccountInterface $currentUser,
  EntityStorageInterface $imageStyleStorage,
  EntityFieldManagerInterface $entityFieldManager,
) {
  parent::__construct(...);
}

public static function create(ContainerInterface $container, ...) {
  return new static(
    ...,
    $container->get('current_user'),
    $container->get('entity_type.manager')->getStorage('image_style'),
    $container->get('entity_field.manager'),
  );
}
```

Existing display configurations using the `file_video` formatter will not have the `poster` or `poster_image_style` keys in their stored config. They default to empty strings at runtime, so existing video rendering is unaffected, but if you export configuration you may see these keys appear when the display is next saved.
