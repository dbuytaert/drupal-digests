## Summary

A fatal PHP error occurred when site builders clicked "Add item" while configuring default values for a multi-valued formatted text field with allowed formats restricted. The error prevented saving any default values for those fields. This bug is now fixed across Drupal 10.6.x, 11.x, 11.3.x, and main.

## Impact

No upgrade or configuration changes required.

## Technical details

The `text` module's `TextFieldItemList::defaultValuesFormValidate()` in `core/modules/text/src/Plugin/Field/FieldType/TextFieldItemList.php` validates that each submitted default value uses a permitted text format from the `allowed_formats` field setting. When a text field has unlimited cardinality, the "Add another item" AJAX button injects an `add_more` entry into the submitted form values array alongside the actual field deltas. That entry's value is a `TranslatableMarkup` object, not an array. The loop over `$submitted_values` then tried to access `$value['format']` on it, producing `Error: Cannot use object of type Drupal\Core\StringTranslation\TranslatableMarkup as array`.

The fix adds a guard immediately after `$submitted_values` is read from the form state: if the `add_more` key is present, it is removed with `unset()` before the `foreach` loop iterates over the deltas and checks their `format` values. The change is minimal and self-contained within `defaultValuesFormValidate()`.

A new unit test class `TextFieldItemListTest` (`core/modules/text/tests/src/Unit/Plugin/Field/FieldType/TextFieldItemListTest.php`) was added. Its single test method `testDefaultValuesFormValidateWithMultipleCardinality()` mocks `FieldDefinitionInterface` and `FormStateInterface` to return a `$submitted_values` array containing only an `add_more` key set to a `TranslatableMarkup` object, confirming that validation completes without error after the fix. The 10.6.x backport uses PHPDoc `@coversClass` / `@group` annotations instead of PHP 8.0 `#[CoversClass]` / `#[Group]` attributes used in the main/11.x version.
