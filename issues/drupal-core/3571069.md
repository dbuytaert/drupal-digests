## Summary

Drupal 12 removes `hook_module_implements_alter()` and all supporting infrastructure. This hook had been the traditional way for modules to change the execution order of other modules' hook implementations. Any contributed or custom module that still implements this hook must migrate to the attribute-based ordering system using the `order` parameter on `#[Hook]` or the `#[ReorderHook]` attribute before upgrading to Drupal 12.

## Impact

Affects module maintainers.

- **API change:** `hook_module_implements_alter()` is removed; the `LegacyModuleImplementsAlter` attribute class is deleted; `HookCollectorPass::addProceduralImplementation()` loses its `\SplFileInfo $fileinfo` parameter; `ModuleHandler::reOrderModulesForAlter()` is removed.
- **Deprecation:** `hook_module_implements_alter()` (deprecated in 11.2.0) is fully removed; replace with `#[Hook(order: Order::Last/First)]` or `#[ReorderHook]`.

## Technical details

The hook system introduced attribute-based hook ordering (`#[Hook(order: Order::First/Last)]`, `#[ReorderHook]`) as the replacement for `hook_module_implements_alter()`, which was deprecated in 11.2.0 and is now fully removed for 12.0.0.

The removal touches several parts of the codebase:

**`ModuleHandler::getCombinedListeners()`** previously called `reOrderModulesForAlter()`, which first sorted the implementing modules by `$this->moduleList` order (module weight then alphabetical) and then invoked `hook_module_implements_alter` to let modules further reorder. Now `reOrderModulesForAlter()` is deleted and the module-weight ordering step is inlined directly with `array_intersect(array_keys($this->moduleList), $modules)`. This preserves expected ordering for alter hooks implemented on behalf of other modules without the extra hook invocation.

**`HookCollectorPass`** loses the `$moduleImplementsAlters` property, the loop in `calculateImplementations()` that called each collected alter function at build time, and special handling in `addProceduralImplementation()` that loaded the implementing file and emitted a deprecation notice. The `$fileinfo` parameter is removed from `addProceduralImplementation()` entirely since it was only needed for that file-include path. `'module_implements_alter'` is removed from the `checkForProceduralOnlyHooks()` deny-list.

**`LegacyModuleImplementsAlter` attribute class** (`core/lib/Drupal/Core/Hook/Attribute/LegacyModuleImplementsAlter.php`) is deleted. This attribute allowed a procedural `hook_module_implements_alter` to be suppressed when running on Drupal 11.2+, providing a compatibility shim for modules that needed to support older core versions alongside attribute-based ordering. It is no longer needed.

**`HookCollectorPass::getOrderOperations()`** gains two small correctness fixes uncovered during this work: the guard condition now also skips operations where the identified implementation's module is not in `$this->modules` (preventing a `TypeError` when a module with `#[ReorderHook]` targeting hooks in another module is uninstalled), and the return value is passed through `array_filter()` to remove hooks whose order-operation list became empty after filtering.

**In-tree consumers** are updated: `common_test`'s hook that was previously moved to the end of the list via `hook_module_implements_alter` now carries `order: Order::Last` directly on the `#[Hook]` attribute; `content_translation` and `layout_builder` have their comments updated accordingly.

**Tests removed:** `ModuleImplementsAlterTest` kernel test, `testProceduralModuleImplementsAlterOrder` in `HookAlterOrderTest`, `testLegacyModuleImplementsAlter` unit test, the `module_implements_alter_test` and `module_implements_alter_test_legacy` test modules, and the `ModuleImplementsAlter` helper class in `aaa_hook_order_test`. Two tests (`testReorderAlterMissingTarget`, `testReorderMissingTarget`) that previously expected a `TypeError` on module uninstall are updated to assert the error no longer occurs, because the `getOrderOperations()` fix resolves it.

This issue is a companion to [#3571067](https://www.drupal.org/node/3571067), which removes `hook_hook_info()` and the `$module.$group.inc` include mechanism in the same release.

## Upgrade

Modules implementing `hook_module_implements_alter()` to reorder hook implementations must switch to the `order` parameter on `#[Hook]` or to `#[ReorderHook]`.

```php
// Before (removed in Drupal 12):
function mymodule_module_implements_alter(array &$implementations, string $hook): void {
  if ($hook === 'form_alter') {
    // Move mymodule to end of the list.
    $group = $implementations['mymodule'];
    unset($implementations['mymodule']);
    $implementations['mymodule'] = $group;
  }
}

// After: declare the desired position directly on the hook attribute.
use Drupal\Core\Hook\Attribute\Hook;
use Drupal\Core\Hook\Order\Order;

#[Hook('form_alter', order: Order::Last)]
public function formAlter(array &$form, FormStateInterface $form_state, string $form_id): void {
  // implementation
}
```

To reorder another module's hook implementation relative to yours, use `#[ReorderHook]`:

```php
use Drupal\Core\Hook\Attribute\ReorderHook;
use Drupal\Core\Hook\Order\OrderBefore;

#[ReorderHook('form_alter', module: 'other_module', method: 'formAlter',
  order: new OrderBefore(['mymodule']))]
```

The `LegacyModuleImplementsAlter` attribute class is also deleted; any reference to `Drupal\Core\Hook\Attribute\LegacyModuleImplementsAlter` must be removed.

### Collaboration

nicxvan opened the issue and did most of the implementation work but hit a series of test failures that required careful debugging. The trickiest problem was that removing `reOrderModulesForAlter()` changed the execution order for hooks implemented on behalf of other modules (such as `#[Hook('...', module: 'block')]`), because that method had been silently re-sorting by module weight before invoking `hook_module_implements_alter`. nicxvan traced this to the `array_intersect` step inside the deleted method and restored it inline in `getCombinedListeners()`. berdir independently identified a separate `TypeError` triggered when a module with `#[ReorderHook]` targeting hooks in a not-yet-enabled module was uninstalled, and contributed the guard condition in `getOrderOperations()` along with the `array_filter()` cleanup that prevents empty operation sets from causing crashes.
