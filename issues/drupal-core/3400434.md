## Summary

All test files in Drupal core now have `declare(strict_types=1)` enforced by the PHP CodeSniffer configuration. This is the final step of the broader effort tracked in [#3376057](https://www.drupal.org/node/3376057) to add strict type declarations to every test file. The change is internal to the development tooling and has no effect on running Drupal sites.

## Impact

No upgrade or configuration changes required.

## Technical details

This issue is the culmination of the "[META] Add declare(strict_types=1) to all tests" effort ([#3376057](https://www.drupal.org/node/3376057)). The key change is in `core/phpcs.xml.dist`, where the `SlevomatCodingStandard.TypeHints.DeclareStrictTypes` sniff had accumulated many narrow `<include-pattern>` entries covering specific subdirectories. Those are replaced by a single broad `<include-pattern>*/tests/*</include-pattern>` with only one `<exclude-pattern>*/fixtures/*</exclude-pattern>`, enforcing the declaration across all test paths at once.

To satisfy this broadened rule, `declare(strict_types=1)` was added to a set of files not previously covered: traits in legacy `src/Tests/` namespaces (`CommentTestTrait.php`, `StubTestTrait.php`, `AssertViewsCacheTagsTrait.php`, `ViewResultAssertionTrait.php`, `ViewTestData.php`, `TestHelperPlugin.php`), test theme files and a theme engine under `core/modules/system/tests/themes/`, test plugin and controller helper classes (`FilterSparkles.php`, `FilterTestStatic.php`, `Routes.php`, `NodeAccessTestAutoBubblingController.php`), and the bootstrapping files `core/tests/bootstrap.php` and `core/modules/contact/tests/drupal-7.contact.database.php`.

Two files required behavioral adjustments to compile cleanly under strict types. In `core/modules/system/tests/http.php` and `https.php`, the loop `foreach ($_SERVER as &$value)` was passing potentially non-string superglobal values to `str_replace()`, so a `if (!is_string($value)) { continue; }` guard was added. In `core/modules/views/src/Tests/ViewResultAssertionTrait.php` and `core/modules/views/tests/src/Kernel/Handler/SortDateTest.php`, uses of `FormattableMarkup` for constructing assertion messages were replaced with `sprintf()`, which returns a plain string compatible with strict type expectations.
