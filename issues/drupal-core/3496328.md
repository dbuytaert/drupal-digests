## Summary

`CacheCollector`'s race condition detection no longer relies on millisecond-precision cache creation timestamps. Instead, it stores a copy of the cache data as it was loaded at the start of the request and computes a content fingerprint at write time to detect whether another process changed the entry in the interim. Two concurrent writes that happen within the same millisecond are now reliably distinguished. `State::setMultiple()` is also reworked to acquire the collector lock before writing to the key-value store and immediately publish the new value to cache within that lock, closing a related race window in `State`.

## Impact

Affects module maintainers.

- **API change:** `CacheCollector::$cacheCreated` (protected) removed on `main` and deprecated on `11.x`; replaced by `$loadedData` (nullable array) and the new protected `fingerprint(array): string` method.
- **Deprecation:** `CacheCollector::$cacheCreated` deprecated on `11.x`; subclasses that reference it should switch to `$loadedData` or rely on the built-in fingerprint comparison in `updateCache()`.

## Technical details

The protected `$cacheCreated` unix-timestamp property is removed from `CacheCollector` and replaced with `$loadedData`, a nullable array that `lazyLoadCache()` populates with a copy of the cache entry's `data` array when the entry exists. PHP's copy-on-write semantics mean this copy only costs extra memory if the data is subsequently modified, making the approach cheap for read-heavy requests.

A new protected `fingerprint(array $data): string` method hashes its argument with `hash('xxh128', serialize($data))`. In `updateCache()`, the fingerprint of `$loadedData` is compared against the fingerprint of the cache item's current content. If they match, no other request has touched the entry and the write proceeds; if they differ (or if the data was invalidated during the request), the method backs out rather than overwriting a concurrent request's work. The fingerprint is computed lazily and only when there are actually keys to persist, so purely read-only requests pay no serialization or hashing cost.

Lock acquisition in `updateCache()` is now decoupled from the safety check: the lock is still attempted to prevent stampedes, but the write decision is governed by the content fingerprint, not by whether the lock was held. When a `State::setMultiple()` call cannot acquire the lock, it immediately writes an empty tombstone cache item (acting as a signal to concurrent requests) and waits up to one second before retrying the lock. `State::set()` now delegates to `setMultiple()` instead of calling `parent::set()` directly.

The test helper `setCacheCreated(int)` in the test stub is replaced by `setLoadedData(array)`. New parameterized unit tests cover concurrent-write scenarios including same-millisecond writes that previously defeated the timestamp comparison. On `11.x`, `$cacheCreated` is restored with a deprecation notice to give subclasses a migration path; on `main` it is removed entirely.

## Upgrade

Subclasses of `CacheCollector` that read or write the protected `$cacheCreated` property must update. On `11.x` the property is deprecated; on `main` it is removed.

```php
// Before (broken on main, deprecated on 11.x):
$this->cacheCreated = $cache->created;
if ($this->cacheCreated !== $otherCreated) { ... }

// After: rely on the new fingerprint-based detection in updateCache().
// If you need to track whether the cache was loaded, check $this->cacheLoaded
// or compare against $this->loadedData instead.
```

Test code that called `setCacheCreated(int)` on the test stub must be updated to call `setLoadedData(array)` with the data array instead of the timestamp.

### Collaboration

longwave revived the issue after it sat idle for months. Their first approach (MR!15932) replaced the timestamp with a random token written alongside each cache entry, but they quickly realized that a content hash requires no extra storage at all and is therefore simpler. In the discussion that followed, catch raised the memory cost of keeping a full copy of the data. longwave pointed to PHP's copy-on-write behavior, demonstrating with a live snippet that the copy only allocates additional memory when one of the arrays is actually modified, making the approach essentially free for the common read-heavy case. That observation settled the design. Later, alexpott noticed that subclasses of the `canvas` module check `$this->cacheCreated` in `lazyLoadCache()`, and catch responded by adding a deprecation for the property on `11.x` (MR!16122) while removing it outright on `main`.
