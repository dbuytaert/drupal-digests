## Summary

Block plugins that return a plain render array (one with no `#type` or `#theme` at the top level) with `#attributes` set will again have those attributes hoisted onto the block wrapper element, restoring behavior that worked correctly up to Drupal 11.3. In 11.4 those attributes were silently dropped, breaking a common workaround pattern used by module developers.

## Impact

No upgrade or configuration changes required.

## Technical details

The regression was introduced by [#2486267](https://www.drupal.org/node/2486267), which reworked how `BlockViewBuilder::preRender()` handles attributes. That change only merged attributes when the plugin's content array carried `#wrapper_attributes`; any other top-level `#attributes` were simply discarded.

The fix adds an `elseif` branch in `BlockViewBuilder::preRender()` immediately after the `#wrapper_attributes` path: when `#wrapper_attributes` is absent and the content render element has neither `#type` nor `#theme`, any `#attributes` on the content are hoisted into `$build['#attributes']` and removed from the child array. This covers the typical "plain build array with custom attributes" pattern that many contrib and custom module block plugins rely on. A kernel test using the existing `test_html` block plugin was added to verify the hoisting behavior and confirm the fix against the regression. A follow-up issue ([#3605585](https://www.drupal.org/node/3605585)) tracks whether this compatibility path should eventually be deprecated.
