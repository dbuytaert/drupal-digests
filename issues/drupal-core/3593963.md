## Summary

Sites with content types that define roughly 60 or more fields can hit a fatal database error when loading entities, because MySQL and MariaDB enforce a hard limit of 61 tables per JOIN and SQLite has a comparable limit. The error surfaced in Drupal 11.4.x after earlier optimizations consolidated field loading into single queries that JOIN each dedicated field storage table. After this fix, entity loading on sites with large numbers of fields works correctly again regardless of how many fields are defined.

## Impact

No upgrade or configuration changes required.

## Technical details

The problem was introduced by [#3564689](https://www.drupal.org/node/3564689) and [#3551308](https://www.drupal.org/node/3551308), which changed `SqlContentEntityStorage::loadFromDedicatedTables()` to load all single-cardinality fields in one query and all multiple-cardinality fields in another, each using a `LEFT JOIN` per field. With ~60 or more fields that use dedicated storage tables, the number of joins exceeds MySQL's documented limit of 61.

The fix adds a constant `FIELD_MINIMUM_CHUNK_SIZE = 25` to `SqlContentEntityStorage` and splits both the single-cardinality and multiple-cardinality field lists into chunks before querying. The chunking logic merges the last chunk into the penultimate one, so 26 fields load as a single chunk of 26 (not 25 + 1) and 51 fields load as chunks of 25 and 26. This keeps each query well under database limits while avoiding an excessively small final batch.

The inline field-loading code was extracted into two new `private` methods, `loadSingleCardinalityFields()` and `loadMultipleCardinalityFields()`, which are called in a `foreach` loop over the chunks. A `$load_shared_table_fields` boolean is passed to `loadSingleCardinalityFields()` and is only `true` for the first chunk, preventing shared base/revision-data-table fields from being fetched and overwritten on subsequent iterations. A new kernel test `testEntityWithManyFieldsLoad()` in `SqlContentEntityStorageTest` creates 71 fields on `entity_test_rev` and asserts all field values survive a save/load and save/loadRevision cycle for both single and unlimited cardinality.
