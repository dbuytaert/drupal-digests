## Summary

Two static helper methods for accessing Views plugin managers are now deprecated. The `Views::pluginManager()` and `Views::handlerManager()` methods were simple wrappers that made it easy to get plugin managers by type, but they prevented proper dependency injection. Core now uses a service locator pattern internally, which provides better performance and cleaner architecture. This change is part of ongoing work to remove static service calls and support dependency injection throughout the Views system.

## Impact

- **Deprecation:** Static methods `Views::pluginManager()` and `Views::handlerManager()` deprecated; use `\Drupal::service('plugin.manager.views.{type}')` or `\Drupal::service('views.plugin_managers')->get($type)` instead

## Technical details

The `Views::pluginManager()` and `Views::handlerManager()` static methods in `\Drupal\views\Views` are now deprecated. These methods previously called `\Drupal::service('plugin.manager.views.' . $type)` to dynamically fetch a plugin manager service by type.

Core has replaced all internal uses of these deprecated methods with direct service calls. A new `views.plugin_managers` service locator was added in `views.services.yml` that provides access to all 19 Views plugin manager services (access, area, argument, argument_default, argument_validator, cache, display, display_extender, exposed_form, field, filter, join, pager, query, relationship, row, sort, style, and wizard).

The implementation now triggers `E_USER_DEPRECATED` errors and delegates to the service locator using `\Drupal::service('views.plugin_managers')->get($type)`. This service locator pattern provides better performance by lazy-loading plugin managers and enables future work to inject dependencies into display plugins and other Views classes.

All 43 call sites in core were updated to use either the specific plugin manager service (e.g., `\Drupal::service('plugin.manager.views.join')`) when the type is known at development time, or the service locator (e.g., `\Drupal::service('views.plugin_managers')->get($handler_type)`) when the type is dynamic. Unit tests were updated to mock the service locator, and two new legacy tests verify the deprecation messages.

## Upgrade

If your custom Views code uses `Views::pluginManager()` or `Views::handlerManager()`, replace these calls with direct service access. For known plugin types, use the specific service:

```php
// Before:
$join = Views::pluginManager('join')->createInstance('standard', $configuration);

// After:
$join = \Drupal::service('plugin.manager.views.join')->createInstance('standard', $configuration);
```

For dynamic types where you determine the plugin type at runtime, use the service locator:

```php
// Before:
$handler = Views::handlerManager($handler_type)->getHandler($item);

// After:
$handler = \Drupal::service('views.plugin_managers')->get($handler_type)->getHandler($item);
```

If your code uses proper dependency injection, inject the specific plugin manager services you need or inject the `views.plugin_managers` service locator. See the [change record](https://www.drupal.org/node/3566982) for full details.
