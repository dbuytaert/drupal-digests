## Summary

During every container build, Drupal's backward-compatibility layer for procedural hooks uses `HookCollectorPass` to recursively scan module directories and parse `.module` and `.inc` files looking for hook functions. For modules that have fully migrated to OOP hooks this scanning is entirely wasted work. This change adds two opt-in mechanisms so modules can signal that the scan should be skipped or cut short, reducing memory and CPU overhead during the container compile phase. The benefit becomes most visible once core modules are also marked as converted (tracked in a follow-up, [#3491275](https://www.drupal.org/node/3491275)).

## Impact

Affects module maintainers.

- **New API:** New `#[StopProceduralHookScan]` attribute (`Drupal\Core\Hook\Attribute\StopProceduralHookScan`) stops procedural hook scanning at the attributed function; new `$module.hooks_converted` container parameter in `services.yml` skips all procedural scanning for a fully converted module.

## Technical details

`HookCollectorPass::collectAllHookImplementations()` now accepts an optional `?ContainerBuilder $container` parameter. For each module it checks `$container->hasParameter("$module.hooks_converted")`; when that parameter is present the module is flagged as `$skip_procedural = true`.

When `collectModuleHookImplementations()` receives `$skip_procedural = true`, it narrows `$dir` to `$dir . '/src/Hook'` (with an `is_dir()` guard) and skips the `include_once` of the `.module`/`.profile` file and the entire `StaticReflectionParser` procedural scan. OOP hook classes in `src/Hook/` are still collected normally.

For modules that still have some procedural hooks, the new `Drupal\Core\Hook\Attribute\StopProceduralHookScan` PHP attribute (a `TARGET_FUNCTION` attribute) can be placed on any function in a scanned file. During the `StaticReflectionParser` loop the code calls `StaticReflectionParser::hasAttribute($attributes, StopProceduralHookScan::class)` and breaks out of the loop as soon as it is found. All procedural hooks to be discovered must appear before the attributed function.

Two test modules cover the two paths: `hook_collector_skip_procedural` (whole-module skip, verified by asserting its `.module` hook function is never called) and `hook_collector_skip_procedural_attribute` (partial skip, verified by asserting hooks before the attribute are collected and those after are not). Both are exercised in `HookCollectorPassTest::testProceduralHooksSkippedWhenConfigured()`. A companion bug fix for a directory-narrowing regression when using this parameter was tracked in [#3491386](https://www.drupal.org/node/3491386).

## Upgrade

To skip all procedural hook scanning for a fully converted module, add to `my_module.services.yml`:

```yaml
parameters:
  my_module.hooks_converted: true
```

To stop scanning partway through a `.module` file, place `#[StopProceduralHookScan]` on the last function that the scanner still needs to see. All procedural hooks that must be discovered must appear before the attributed function:

```php
use Drupal\Core\Hook\Attribute\StopProceduralHookScan;

// Hooks above this point are still discovered by the scanner.

#[StopProceduralHookScan]
function my_module_some_hook(): void {
  // This function and all functions after it are not collected as procedural hooks.
}
```

### Collaboration

nicxvan diagnosed early in the thread that using `info.yml` flags was unworkable because the compiler pass runs before info files are parsed. After confirming that and opening a sibling-file MR, godotislate suggested using container parameters in `services.yml` instead (comment [#15882421](https://www.drupal.org/node/15882421)). nicxvan immediately opened a third MR on a new branch with that approach, confirmed it passed CI, and closed the sibling-file branch. The switch was consequential: the parameter approach is simpler, removes the need for an extra file on disk, and lets a site owner mark a contrib module as converted by overriding its services without waiting for the module maintainer.
