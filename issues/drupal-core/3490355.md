## Summary

The backward-compatibility layer for procedural hooks is expensive: during every container build, `HookCollectorPass` recursively scans module directories and uses `StaticReflectionParser` to find hook functions in `.module` and `.inc` files. This issue added two opt-in mechanisms so that modules that have migrated to OOP hooks can skip that scanning. A module that has fully converted declares a container parameter in its `services.yml` to signal the skip. A module that has only partially converted can place a PHP attribute on the last function that still needs to be scanned, stopping the parser at that point. Both mechanisms are purely internal with no effect on end-user behavior, but they reduce memory and CPU overhead during the container compile phase, which becomes measurable once core modules are also converted (tracked in a follow-up issue).

## Impact

Affects module maintainers.

- **New API:** New `#[ProceduralHookScanStop]` attribute (`Drupal\Core\Hook\Attribute\ProceduralHookScanStop`) stops procedural hook scanning at the attributed function; new `$module.skip_procedural_hook_scan` container parameter skips all procedural scanning for a fully converted module.

## Technical details

The problem is in `HookCollectorPass`, which scans every enabled module's directory tree on each container rebuild to discover procedural hook implementations. For modules that have fully converted to OOP hooks, this scan is entirely wasted work.

Two complementary short-circuit mechanisms were added:

**Whole-module skip via container parameter.** `collectAllHookImplementations()` now reads all container parameters from the `ContainerBuilder` and builds a list of modules for which `"$module.skip_procedural_hook_scan"` is non-empty. A module opts in by adding this to its `services.yml`:

```yaml
parameters:
  my_module.skip_procedural_hook_scan: true
```

When `collectModuleHookImplementations()` receives `$skip_procedural = true`, it narrows `$dir` to `src/Hook/` only, skips the `include_once` of the `.module`/`.profile` file, and bypasses the entire `StaticReflectionParser` procedural scan. The `is_dir()` guard around the iterator handles modules that have no `src/Hook/` directory.

**Partial skip via `#[ProceduralHookScanStop]` attribute.** For modules that still have some procedural hooks (such as `hook_preprocess_*` or `hook_requirements()`), the new `Drupal\Core\Hook\Attribute\ProceduralHookScanStop` attribute (located at `core/lib/Drupal/Core/Hook/Attribute/ProceduralHookScanStop.php`) can be placed on any function in a scanned file. During the `StaticReflectionParser` loop in `collectModuleHookImplementations()`, the code calls `StaticReflectionParser::hasAttribute($attributes, ProceduralHookScanStop::class)` and breaks out of the loop as soon as it is found. All procedural hooks that still need to be discovered must therefore appear before the attributed function.

The commit introduced these under the names `hooks_converted` and `StopProceduralHookScan`; both were subsequently renamed in later commits to `skip_procedural_hook_scan` and `ProceduralHookScanStop` respectively, which are the names present in the current codebase.

Two test modules were added under `core/modules/system/tests/modules/`: `hook_collector_skip_procedural` (whole-module skip, verified by checking that its `.module` file's hook function is never executed) and `hook_collector_skip_procedural_attribute` (partial skip, verified by confirming hooks before the attribute are collected and those after are not). The kernel test `HookCollectorPassTest::testProceduralHooksSkippedWhenConfigured()` covers both cases.

The design choice to use a container parameter rather than a sibling file or an `info.yml` flag was driven by a key constraint: the compiler pass runs before `info.yml` data is available, making the `info.yml` approach unworkable. The `services.yml` parameter approach also allows a site owner to mark a contrib module as converted via a custom services override, without waiting for the module maintainer.

## Upgrade

To skip all procedural hook scanning for a fully converted module, add to `my_module.services.yml`:

```yaml
parameters:
  my_module.skip_procedural_hook_scan: true
```

To stop scanning partway through a `.module` file, place the attribute on the last function that needs OOP-hook collection (all remaining procedural hooks must appear before it):

```php
use Drupal\Core\Hook\Attribute\ProceduralHookScanStop;

// ... procedural hooks that must still be found ...

#[ProceduralHookScanStop]
function my_module_some_unconvertible_hook(): void {
  // hooks after this point are not scanned
}
```
