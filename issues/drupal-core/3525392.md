## Summary

The `response_status` block visibility condition now has a fully validated config schema. Its `status_codes` values are constrained to the three HTTP codes the plugin actually supports: 200, 403, and 404. Config that was previously accepted silently at the schema layer, but contained unsupported status codes, will now produce validation errors. UI-created config is unaffected because the condition's checkbox form already restricts input to those three codes; this change closes the gap for config import, configuration actions, and programmatic edits.

## Impact

Affects site owners.

- **Config change:** The `condition.plugin.response_status` config schema is now FullyValidatable; `status_codes` items are constrained to the Choice [200, 403, 404].

## Technical details

In `core/modules/system/config/schema/system.schema.yml`, the `condition.plugin.response_status` schema type gains two additions: a top-level `FullyValidatable: ~` constraint, which opts the entire type into strict typed-config validation, and a `Choice` constraint on the items of the `status_codes` sequence limiting accepted values to `[200, 403, 404]`, matching exactly what `ResponseStatus::buildConfigurationForm()` exposes. Previously, the schema placed no constraint on the integers stored in `status_codes`, so any integer was accepted even though `ResponseStatus::evaluate()` only branches on those three codes. The related schema corrections to the parent `condition.plugin` type were handled in [#3525391](https://www.drupal.org/node/3525391). A `testStatusCodesValidation` kernel test with a `providerStatusCodesValidation` data provider was added to `ResponseStatusConditionTest` to verify that the three valid codes pass and that an unsupported code (HTTP 418) produces the expected violation on `status_codes.N`.

## Upgrade

If you have configuration (exported YAML or programmatically created) that sets `status_codes` to any integer other than `200`, `403`, or `404` on a `response_status` block visibility condition, typed-config validation will now report a violation:

```
The value you selected is not a valid choice.
```

Correct the stored config to use only the supported codes before running config import or any validation pass.
