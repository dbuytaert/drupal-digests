## Summary

A critical regression introduced in Drupal 11.4 caused entity loading to generate catastrophically large SQL queries when an entity type had more than one unlimited-cardinality (dedicated table) field. Loading a single entity with several well-populated multi-value fields could produce millions of result rows via Cartesian multiplication, exhausting PHP memory and causing 502 errors and high CPU load regardless of how high the memory limit was set. This fix eliminates the Cartesian explosion by loading each multi-cardinality field in its own separate query instead of grouping them together.

## Impact

No upgrade or configuration changes required.

## Technical details

The root cause was a chunking optimization in `SqlContentEntityStorage::loadFromDedicatedTables()`, introduced alongside the related 61-table JOIN limit fix ([#3593963](https://www.drupal.org/node/3593963)). When an entity had more than `FIELD_MINIMUM_CHUNK_SIZE` unlimited-cardinality fields, those fields were grouped into chunks and each chunk was passed to `loadMultipleCardinalityFields()` together. That method builds a single SELECT by joining each field's dedicated table to the base entity table. Joining N multi-value field tables in one query produces a Cartesian product: rows from `field_a` (5 values) × `field_b` (10 values) × `field_c` (4 values) = 200 rows instead of 19. On real content the multiplication was far worse; one confirmed case produced ~19.5 million rows per language per entity load.

The fix removes the chunking loop entirely. `loadFromDedicatedTables()` now iterates over `$multiple_cardinality_fields` one field at a time, calling `loadMultipleCardinalityFields()` with a single-element array on each iteration. This trades fewer queries for a guaranteed linear row count. Simultaneously, the join type inside `loadMultipleCardinalityFields()` was changed from `leftJoin` to `join` (INNER JOIN) for both the translatable and non-translatable code paths. A LEFT JOIN was unnecessary here because an entity that has no rows for a given dedicated field simply produces no result in that field's individual query; an INNER JOIN is correct and slightly more efficient.

The performance test fixtures reflect the new query shape: what was previously one multi-field LEFT OUTER JOIN query becomes N separate INNER JOIN queries, one per unlimited-cardinality field. Query counts in the Umami and standard performance tests increased accordingly (e.g., warm-cache node page rose from 56 to 65 queries). A follow-up issue [#3608184](https://www.drupal.org/node/3608184) was opened to explore loading multiple fields without Cartesian multiplication (e.g., via UNION), which could recover some of that query count increase.
