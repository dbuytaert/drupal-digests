## Summary

When a custom entity type has no label field, visiting its edit or delete page would crash with a TypeError. This happened because the title callbacks expected entity labels to always be strings, but entities without label fields return NULL. The fix ensures these pages display generic "Edit" or "Delete" titles instead of crashing when an entity has no label.

## Impact

No upgrade or configuration changes required.

## Technical details

The `EntityController::editTitle()` and `EntityController::deleteTitle()` methods generate page titles by calling `$entity->label()` and passing the result to translatable strings like "Edit %label" and "Delete %label". When an entity type has no label field, `label()` returns NULL, which caused a TypeError in `Html::escape()` because it expects a string parameter.

The fix splits the logic into explicit checks. First, it verifies the entity exists (returning NULL or empty string as before to maintain backward compatibility). Then it checks if `label()` returns NULL. If so, it returns just "Edit" or "Delete" without the placeholder. If a label exists, it returns the original formatted string with the label included.

A new unit test covers both scenarios: entities with labels (which should display "Edit example" or "Delete example") and entities without labels (which should display just "Edit" or "Delete"). The test uses PHPUnit attributes to parameterize test cases for both callbacks and both label states.

### Collaboration

This issue remained open for over three years before being resolved. The initial patch approach used a ternary operator with an empty string fallback, which would have produced trailing spaces in titles. Community reviewers identified this issue and suggested using proper conditionals instead. 

The path forward became clear when dcam converted the patch to a merge request, added comprehensive unit tests, and discovered the same bug existed in both `editTitle()` and `deleteTitle()` callbacks. This expanded the scope appropriately. Several iterations refined the code structure, with joachim pushing for cleaner conditional logic rather than mixing ternary and if/else patterns. The final version uses straightforward null checks that are easy to understand and maintain.

The issue was committed to Drupal 11.3 and later versions only, as it did not backport cleanly to 10.6.
