# #3584812: ConfigInstaller isSyncing does not persist to hook_modules_installed() when multiple modules are installed

## Summary

A critical regression introduced in Drupal 11.3.6 caused configuration import to behave incorrectly when multiple modules were installed at once during a `drush cim` or equivalent operation. The `isSyncing` flag on the config installer was silently lost mid-process, so `hook_modules_installed()` implementations received `$is_syncing = FALSE` when it should have been `TRUE`. This could lead to default content, default blocks, or other actions normally suppressed during config sync being triggered incorrectly, with the potential for data loss. Sites that imported configuration while running 11.3.6 or 11.3.7 are advised to re-import it and verify no configuration differences remain.

## Impact

No upgrade or configuration changes required.

## Technical details

The root cause was that `DrupalKernel::resetContainer()` did not preserve the `config.installer` syncing state across the container reset, unlike the sibling method `initializeContainer()` which already handled this correctly (added in [#3572171](https://www.drupal.org/node/3572171)). During a multi-module install, `ModuleInstaller::install()` calls `doInstall()` for each module, and when more than one module is in the list it calls `updateKernel([])`, which in turn calls `$this->kernel->resetContainer()`. Because `resetContainer()` was missing the save-and-restore pattern, the `ConfigInstaller::$isSyncing` flag was lost each time the container was reset between installs. When `hook_modules_installed()` was finally fired, `isSyncing()` returned `FALSE`. The fix mirrors the pattern already present in `initializeContainer()`: before calling `$this->container->reset()`, `resetContainer()` now reads `$config_is_syncing` via `isSyncing()` and `$source_storage` via `getSourceStorage()` from the old container, then after the reset restores them on the new container's `config.installer` service via `setSyncing(TRUE)` and `setSourceStorage($source_storage)`. The change is confined to `DrupalKernel::resetContainer()` in `core/lib/Drupal/Core/DrupalKernel.php`.

## Upgrade

Sites that imported configuration while running Drupal 11.3.6 or 11.3.7 should re-import configuration after updating to 11.3.8 or later and verify no unexpected differences remain:

```
drush deploy
```

Sites that updated to 11.3.6 or 11.3.7 without importing configuration do not need to take any action.
