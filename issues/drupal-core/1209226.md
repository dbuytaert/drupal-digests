# #1209226: Avoid slow query for path alias whitelists

- **Project:** Drupal Core
- **Type:** Major bug report
- **Version:** 8.0.x-dev
- **Status:** Fixed
- **Discussion:** [#1209226](https://www.drupal.org/node/1209226) · 20 contributors · 166 comments
- **Followers:** 64

## Summary

On sites with millions of path aliases, Drupal's path alias subsystem used to run a `SELECT SUBSTRING_INDEX(src, '/', 1) ... GROUP BY` query to build a whitelist of valid path prefixes. On a real production site with 4.5 million aliases this query took between 5 and 19 seconds, blocking every page request that called `url()` until it completed, with no locking protection against concurrent stampedes. This fix replaces that single slow query with a lazily-built prefix list that issues small, indexed `LIKE 'foo%' LIMIT 1` queries per path root on demand, reducing lookup time to under 5 ms per prefix and eliminating the blocking rebuild. The change also moves the whitelist out of the Drupal variable system and into the cache, backed by path roots derived from the route/menu system rather than scanning `url_alias` directly.

## Impact

No upgrade or configuration changes required.

## Technical details

The path alias subsystem maintains a prefix list so that `url()` can cheaply skip alias lookups for paths whose root segment has never had any alias. The old approach ran `SELECT SUBSTRING_INDEX(src, '/', 1) AS path FROM url_alias GROUP BY path` which forced a full index scan over every alias row to produce the distinct prefix set; on a large table this could consume many seconds and take locks.

The fix introduces a lazy-loading cache collector class, now called `AliasPrefixList` (in `core/modules/path_alias/src/AliasPrefixList.php`), which extends `CacheCollector`. The class stores three states per path root: `NULL` meaning the root exists in the router but has not yet been checked for aliases, `TRUE` meaning at least one alias exists for that prefix, and `FALSE` meaning no alias exists. This three-state design is critical: paths whose roots are not even in the router are absent from the list entirely, so `url()` calls with arbitrary non-router paths do not trigger any query at all.

On a cold start, `AliasPrefixList::lazyLoadCache()` calls `loadMenuPathRoots()`, which reads the `router.path_roots` state key and pre-populates the internal storage with `NULL` values for each known root. That state key is written by `PathRootsSubscriber` (`core/lib/Drupal/Core/EventSubscriber/PathRootsSubscriber.php`), which listens on `RoutingEvents::ALTER` and `RoutingEvents::FINISHED` to collect the first path segment from every registered route. This means the prefix list is derived from the routing system rather than from `url_alias`, so the expensive scan is never needed.

When `AliasPrefixList::get()` is called with a root whose stored value is `NULL` (known router path, not yet resolved), it calls `resolveCacheMiss()`, which delegates to `AliasRepository::pathHasMatchingAlias()` (`core/modules/path_alias/src/AliasRepository.php`). That method issues `SELECT 1 FROM {path_alias} WHERE path LIKE :prefix LIMIT 1` using `escapeLike()` on the prefix, hitting the existing index in milliseconds rather than seconds. The result is persisted back into the cache collector so subsequent requests within the same cache lifetime pay no database cost for that prefix.

The `AliasPrefixList` is injected into `AliasManager` (`core/modules/path_alias/src/AliasManager.php`) via constructor, typed as `AliasPrefixListInterface`.

The design went through significant iteration. An early approach used `strtok()` on the requested path and a key-shrinking heuristic (starting at a 5-character substring, reducing by one when the list exceeded 100 entries) to bound cache growth when `url()` was called with arbitrary paths. This was later replaced with seeding from `{menu_router}` path roots (the approach that made it into the committed patch), which avoids the growth problem entirely because the set of known path roots is bounded. `chx` suggested storing those roots in a variable/state during menu rebuild so that a cache miss never needs to rebuild the entire menu, which was the final key to making the approach safe for production. The `clear()` method on `AliasPrefixList` calls `loadMenuPathRoots()` to re-seed after a cache wipe so the list is never left empty.

The related issue [#2540154](https://www.drupal.org/node/2540154) tracked a D7 version of the stale-whitelist-on-deletion problem and was eventually closed as outdated once the D8 fix was in.
