# #1209226: Avoid slow query for path alias whitelists

- **Project:** Drupal Core
- **Type:** Major bug report
- **Version:** 8.0.x-dev
- **Status:** Fixed
- **Discussion:** [#1209226](https://www.drupal.org/node/1209226) · 20 contributors · 166 comments

## Summary

When Drupal checks if a URL needs a path alias, it builds a whitelist to avoid unnecessary database lookups. Previously, creating this whitelist on sites with millions of aliases could trigger a slow, poorly-indexed query that took 5 to 19 seconds to complete. During this time, any request calling `url()` would block, potentially halting the entire site. The new approach replaces the single expensive query with smaller, well-indexed queries that run incrementally as needed, eliminating the multi-second freeze and providing flat performance even with millions of aliases.

## Impact

No upgrade or configuration changes required.

## Technical details

The original implementation used `SUBSTRING_INDEX()` with `GROUP BY` to scan the entire `url_alias` table, examining millions of rows to identify all unique path prefixes. This query used temporary tables and filesort, causing 5+ second blocks.

The patch refactors path alias whitelist logic into a new `PathAliasWhitelist` class (later `AliasWhitelist`) that extends `CacheArray`. Instead of one massive query, it builds the whitelist incrementally. When a path root is first encountered, it runs `SELECT alias FROM url_alias WHERE source LIKE 'node%' LIMIT 1` to check if any aliases exist for that prefix. This indexed query completes in milliseconds.

The whitelist now stores three states: `TRUE` (aliases exist), `FALSE` (no aliases), and `NULL` (not yet checked). Path roots are seeded from `menu_router` during menu rebuild and stored in the state system. The class tracks which roots have aliases, clearing and rebuilding only affected roots when aliases change.

For Drupal 8, the implementation was converted to a service with proper dependency injection. The `AliasWhitelist` service is injected into `AliasManager`, with database connection, state, and cache bin provided through the container.

### Collaboration

This issue took over five years from opening to final commit, starting in 2011 for Drupal 6/Pressflow and finally landing in Drupal 8 in 2016. The original reporter (catch) identified the problem on a client site with 4.5 million aliases where the whitelist query appeared at the top of slow query logs.

The approach evolved significantly through discussion. Early patches used `strtok()` on paths but this risked cache pollution from arbitrary paths passed to `url()`. Catch proposed using path substrings with dynamic length adjustment, borrowed from memcache wildcard logic. After extensive IRC discussion with chx, the final design switched to seeding the whitelist from `menu_router` path roots rather than building it entirely from alias queries.

Converting to Drupal 8's new service architecture required multiple re-rolls as the routing system was rewritten. Berdir drove the final push, handling PSR-0 conversion, dependency injection, and test coverage. The patch was blocked temporarily by segmentation faults from service destructors until [#1891980](https://www.drupal.org/node/1891980) landed.

The Drupal 7 backport (arnested in #145, #152) remains unmerged, awaiting comprehensive test coverage beyond the unit tests added for Drupal 8.

## Lessons learned

- When caching data that requires an expensive query to rebuild, store it in the persistent cache rather than the variable system. Variables survive cache clears, which means expensive rebuild queries can run frequently if triggered by cache invalidation events like content updates. *(via catch)*
- When a single expensive query blocks all requests during cache rebuild, replace it with multiple fast indexed queries that run lazily on-demand. This trades a large up-front cost for incremental overhead distributed across requests, preventing a single blocking operation from creating a stampede. *(via catch)*
- When building cache data incrementally from arbitrary user input, limit cache growth by reducing key granularity as the cache size grows. Start with fine-grained keys (longer substrings) and progressively shorten them when the cache exceeds a threshold, preventing unbounded memory usage from pathological input patterns. *(via catch)*
- When a cache needs to distinguish between 'never checked' and 'checked but not found', use three states (NULL for unchecked, FALSE for blacklisted, TRUE for whitelisted) instead of two. This prevents unnecessary lookups for known-absent data while still detecting data that hasn't been evaluated yet. *(via catch)*
