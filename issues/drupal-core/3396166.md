## Summary

All entity type definitions in Drupal core have been converted from Doctrine annotation syntax (`@ContentEntityType`, `@ConfigEntityType`) to native PHP attribute syntax (`#[ContentEntityType(...)]`, `#[ConfigEntityType(...)]`). Annotation-based definitions continue to work for contrib modules via a backward-compatible fallback, but PHP attributes are now the recommended approach for Drupal 11.1 and later. This is part of the broader effort tracked in the meta issue [#3396165](https://www.drupal.org/node/3396165) to move all core plugin types off annotations, which is a prerequisite for eventually deprecating annotation-based discovery tracked in [#3265946](https://www.drupal.org/node/3265946).

## Impact

Affects module maintainers.

- **New API:** Three new attribute classes added: `Drupal\Core\Entity\Attribute\EntityType`, `ContentEntityType`, and `ConfigEntityType`, for use in place of `@EntityType`, `@ContentEntityType`, and `@ConfigEntityType` annotations.

## Technical details

`EntityTypeManager` previously used `AnnotatedClassDiscovery` pointing at `Drupal\Core\Entity\Annotation\EntityType`. It now uses `AttributeDiscoveryWithAnnotations`, which accepts `Drupal\Core\Entity\Attribute\EntityType::class` as the primary attribute class while retaining the annotation class as a fallback for backward compatibility.

Three new attribute classes were added under `Drupal\Core\Entity\Attribute\`: `EntityType` extends `Drupal\Component\Plugin\Attribute\Plugin` and declares all generic entity type properties as typed, named, readonly constructor parameters; its `get()` method collects properties via `get_object_vars()`, augments with `class` and `provider`, and instantiates the entity type definition object via the `$entity_type_class` parameter. `ContentEntityType` extends `EntityType` with content-specific defaults (`static_cache: TRUE`, `render_cache: TRUE`, `group: 'content'`) and adds parameters such as `base_table`, `revision_table`, `revision_data_table`, `data_table`, `translatable`, `show_revision_ui`, and `revision_metadata_keys`. `ConfigEntityType` extends `EntityType` with configuration-entity defaults (`static_cache: FALSE`, `group: 'configuration'`) and adds `config_prefix`, `config_export`, and `lookup_keys`.

A behavioral bug was fixed alongside the conversion: `ConfigEntityType::__construct()` previously checked `if (empty($this->list_cache_tags))` to set the default cache tag, but because the attribute now supplies an explicit empty-array default, that check always evaluated as true and could overwrite a legitimately provided value. The fix changes the guard to `if (empty($definition['list_cache_tags']))`, reading from the incoming definition array instead.

In the converted entity classes, quoted handler class strings (e.g. `"Drupal\block\BlockForm"`) become `::class` constants, `@Translation(...)` becomes `new TranslatableMarkup(...)`, and `@PluralTranslation(singular = "...", plural = "...")` is replaced by a plain array with `'singular'` and `'plural'` keys passed to `label_count:`.

## Upgrade

Annotations (`@ContentEntityType`, `@ConfigEntityType`) continue to work via the `AttributeDiscoveryWithAnnotations` fallback. No immediate action is required. When migrating a custom entity type to attributes, replace the docblock annotation with a PHP attribute:

```php
// Before (annotation style):
/**
 * Defines the Foo entity.
 *
 * @ContentEntityType(
 *   id = "foo",
 *   label = @Translation("Foo"),
 *   label_count = @PluralTranslation(
 *     singular = "@count foo",
 *     plural = "@count foos",
 *   ),
 *   handlers = {
 *     "storage" = "Drupal\Core\Entity\Sql\SqlContentEntityStorage",
 *     "form" = {
 *       "default" = "Drupal\foo\FooForm",
 *     },
 *   },
 *   base_table = "foo",
 *   entity_keys = {
 *     "id" = "id",
 *   },
 * )
 */
class Foo extends ContentEntityBase { ... }

// After (PHP attribute style):
use Drupal\Core\Entity\Attribute\ContentEntityType;
use Drupal\Core\Entity\Sql\SqlContentEntityStorage;
use Drupal\Core\StringTranslation\TranslatableMarkup;
use Drupal\foo\FooForm;

#[ContentEntityType(
  id: 'foo',
  label: new TranslatableMarkup('Foo'),
  label_count: [
    'singular' => '@count foo',
    'plural' => '@count foos',
  ],
  handlers: [
    'storage' => SqlContentEntityStorage::class,
    'form' => [
      'default' => FooForm::class,
    ],
  ],
  base_table: 'foo',
  entity_keys: [
    'id' => 'id',
  ],
)]
class Foo extends ContentEntityBase { ... }
```

The same pattern applies to `@ConfigEntityType` / `#[ConfigEntityType(...)]` using `Drupal\Core\Entity\Attribute\ConfigEntityType`.

### Collaboration

godotislate carried the majority of the rebase work across many months, including debugging a subtle `list_cache_tags` regression that arose because the attribute supplies an explicit empty-array default where the annotation had no default at all (described in detail in comment #26). berdir set the architectural direction early: keep the scope to a strict 1:1 annotation-to-attribute mapping with no API restructuring mixed in, so the change could land quickly. This position was explicitly seconded by longwave and later by catch and larowlan, who closed down several proposals (combining label arrays, splitting Field UI properties into a separate attribute) as out of scope. bbrala contributed an automated Rector-based conversion pass using a development branch of drupal-rector to accelerate the bulk conversion. The issue was blocked for a period on subclass discovery support, resolved upstream in a separate issue before the branch could land.
