## Summary

All entity type definitions in Drupal core have been converted from Doctrine annotation syntax (`@ContentEntityType`, `@ConfigEntityType`) to native PHP attribute syntax (`#[ContentEntityType(...)]`, `#[ConfigEntityType(...)]`). Annotation-based definitions continue to work for existing contrib modules, but attributes are now the recommended approach for Drupal 11.1 and later. This is part of the broader effort tracked in the meta issue [#3396165](https://www.drupal.org/node/3396165) to move all core plugin types off annotations.

## Impact

Affects module maintainers.

- **New API:** Three new attribute classes added: `Drupal\Core\Entity\Attribute\EntityType`, `ContentEntityType`, and `ConfigEntityType`, usable in place of `@EntityType`/`@ContentEntityType`/`@ConfigEntityType` annotations.

## Technical details

The entity type subsystem previously used `AnnotatedClassDiscovery` pointed at `Drupal\Core\Entity\Annotation\EntityType` for plugin discovery. `EntityTypeManager` now uses `AttributeDiscoveryWithAnnotations`, which accepts `Drupal\Core\Entity\Attribute\EntityType::class` as the primary attribute class while retaining the annotation class as a fallback. This preserves backward compatibility for contrib modules that have not yet migrated.

Three new attribute classes were introduced under `Drupal\Core\Entity\Attribute\`:

- `EntityType` extends `Drupal\Component\Plugin\Attribute\Plugin` and declares all generic entity type properties as typed, named, readonly constructor parameters. Its `get()` method calls `get_object_vars()`, augments with `class` and `provider`, and instantiates the entity type definition object via the `$entity_type_class` property (defaulting to `Drupal\Core\Entity\EntityType`).
- `ContentEntityType` extends `EntityType`, overrides defaults appropriate for content entities (e.g. `static_cache: TRUE`, `render_cache: TRUE`, `group: 'content'`), and adds content-specific parameters such as `base_table`, `revision_table`, `revision_data_table`, `data_table`, `translatable`, `show_revision_ui`, and `revision_metadata_keys`.
- `ConfigEntityType` extends `EntityType` with defaults appropriate for config entities (e.g. `static_cache: FALSE`, `group: 'configuration'`) and adds `config_prefix`, `config_export`, and `lookup_keys`.

A behavioral bug was fixed at the same time: `ConfigEntityType::__construct()` previously checked `if (empty($this->list_cache_tags))` to set the default cache tag, but because the attribute now supplies an explicit empty-array default, that check always passed as empty and overwrote a legitimately set value. The fix changes the guard to `if (empty($definition['list_cache_tags']))`, reading from the incoming definition array instead.

Every entity class in core was converted 1:1 from its annotation to the corresponding attribute. Handler class strings that were previously bare quoted strings (e.g. `"Drupal\block\BlockForm"`) become `::class` constants in the attribute arguments. `@Translation(...)` calls become `new TranslatableMarkup(...)` instances. `@PluralTranslation(singular = "...", plural = "...")` is replaced by a plain array passed to `label_count:` with `'singular'` and `'plural'` keys.

The conversion deliberately avoids restructuring the attribute API (for example, grouping label properties or moving `field_ui_base_route` to a separate Field UI attribute) in order to land quickly and unblock annotation deprecation tracked in [#3265946](https://www.drupal.org/node/3265946).

## Upgrade

Annotations (`@ContentEntityType`, `@ConfigEntityType`) continue to work via the `AttributeDiscoveryWithAnnotations` fallback. No immediate action is required. When you are ready to migrate a custom entity type to attributes, replace the docblock annotation with a PHP attribute:

```php
// Before (annotation style):
/**
 * Defines the Foo entity.
 *
 * @ContentEntityType(
 *   id = "foo",
 *   label = @Translation("Foo"),
 *   label_plural = @Translation("Foos"),
 *   label_count = @PluralTranslation(
 *     singular = "@count foo",
 *     plural = "@count foos",
 *   ),
 *   handlers = {
 *     "storage" = "Drupal\Core\Entity\Sql\SqlContentEntityStorage",
 *     "form" = {
 *       "default" = "Drupal\foo\FooForm",
 *     },
 *   },
 *   base_table = "foo",
 *   entity_keys = {
 *     "id" = "id",
 *   },
 * )
 */
class Foo extends ContentEntityBase { ... }

// After (PHP attribute style):
use Drupal\Core\Entity\Attribute\ContentEntityType;
use Drupal\Core\Entity\Sql\SqlContentEntityStorage;
use Drupal\Core\StringTranslation\TranslatableMarkup;
use Drupal\foo\FooForm;

#[ContentEntityType(
  id: 'foo',
  label: new TranslatableMarkup('Foo'),
  label_plural: new TranslatableMarkup('Foos'),
  label_count: [
    'singular' => '@count foo',
    'plural' => '@count foos',
  ],
  handlers: [
    'storage' => SqlContentEntityStorage::class,
    'form' => [
      'default' => FooForm::class,
    ],
  ],
  base_table: 'foo',
  entity_keys: [
    'id' => 'id',
  ],
)]
class Foo extends ContentEntityBase { ... }
```

The same pattern applies to `@ConfigEntityType` / `#[ConfigEntityType(...)]`. Use `Drupal\Core\Entity\Attribute\ConfigEntityType` as the attribute class.
