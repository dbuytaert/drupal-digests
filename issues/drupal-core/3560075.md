## Summary

Four migrate process plugins that were previously tied to Migrate Drupal-specific upgrade paths are now available in the core `migrate` module for use in any migration. Developers writing custom migrations can now use the `link_options`, `link_uri`, `timezone`, and `user_langcode` process plugins without depending on Migrate Drupal. The two plugins that lived in the `menu_link_content` module are deprecated in their old locations in Drupal 11.4 and will be removed in Drupal 13.

## Impact

Affects module maintainers.

- **Deprecation:** Drupal\menu_link_content\Plugin\migrate\process\LinkOptions and LinkUri deprecated in drupal:11.4.0; replaced by equivalents in Drupal\migrate\Plugin\migrate\process

## Technical details

The issue was spun off from [#3502755](https://www.drupal.org/node/3502755), which identified `link_options`, `link_uri`, `timezone`, and `user_langcode` as process plugins worth keeping outside of Migrate Drupal. A complicating factor was that [#3572280](https://www.drupal.org/node/3572280) (Remove Migrate Drupal) had already deleted the `timezone` and `user_langcode` classes from the `main` branch before this issue was resolved, requiring those two to be restored rather than copied.

**Deprecated old classes (two plugins):** `Drupal\menu_link_content\Plugin\migrate\process\LinkOptions` and `Drupal\menu_link_content\Plugin\migrate\process\LinkUri`, which were the original implementations, lose their `#[MigrateProcess]` attribute (removing them from plugin discovery) and gain a `@trigger_error` deprecation notice. Both are deprecated in drupal:11.4.0 and will be removed in drupal:13.0.0, with the replacement pointing to the new `migrate` module namespace.

**New canonical classes (all four plugins):** Four classes are created in `core/modules/migrate/src/Plugin/migrate/process/`: `LinkOptions`, `LinkUri`, `TimeZone`, and `UserLangcode`. Each carries the `#[MigrateProcess]` attribute with the original plugin IDs (`link_options`, `link_uri`, `timezone`, `user_langcode`), preserving backward compatibility for any migration YAML that references them. The old `menu_link_content` classes retain their own logic rather than extending the replacements, keeping the deprecation path clean.

The replacement classes take advantage of modern PHP conventions. `LinkUri` uses constructor property promotion for its `EntityTypeManagerInterface $entityTypeManager` dependency. `UserLangcode` injects `LanguageManager` via the `#[Autowire(service: 'language_manager')]` attribute. `TimeZone` is a straightforward `ProcessPluginBase` with no service dependencies.

A kernel test, `LinkUriTest`, is moved from `Drupal\Tests\menu_link_content\Kernel\Plugin\migrate\process` to `Drupal\Tests\migrate\Kernel\process` and updated to cover `Drupal\migrate\Plugin\migrate\process\LinkUri`.

The initial commit set the deprecation version to `drupal:12.0.0`. catch immediately reverted it and recommitted at `drupal:11.4.0` so that contrib and custom code can rely on these plugins while Drupal 11 is still supported, rather than having to wait until Drupal 12.

## Upgrade

Any migration YAML using `plugin: link_options`, `plugin: link_uri`, `plugin: timezone`, or `plugin: user_langcode` continues to work without changes because the canonical plugin IDs are unchanged in the new classes.

If your PHP code directly instantiates or references the old `menu_link_content` classes, update the namespace:

```php
// Before
use Drupal\menu_link_content\Plugin\migrate\process\LinkOptions;
use Drupal\menu_link_content\Plugin\migrate\process\LinkUri;

// After
use Drupal\migrate\Plugin\migrate\process\LinkOptions;
use Drupal\migrate\Plugin\migrate\process\LinkUri;
```

`TimeZone` and `UserLangcode` have no deprecated predecessor in the current codebase; references to `Drupal\migrate\Plugin\migrate\process\TimeZone` and `Drupal\migrate\Plugin\migrate\process\UserLangcode` are correct as-is.
